## 二、模块一：Java 基础与语法（错题与补强）

### 2.1 错题纠正

**1. Integer 缓存与装箱（你选了 B「说法错误」）**

- **正确**：说法错误的是 **C**。
  - A：-128～127 可调（`-XX:AutoBoxCacheMax=xxx`）✓  
  - B：`Integer a = 10;` 底层是 `Integer.valueOf(10)` ✓  
  - **C：`new Integer(10)` 不会进缓存池**，只有自动装箱才走 valueOf 缓存 ✗  
  - D：在缓存范围内 == 为 true，超出为 false ✓  

**学习要点**：自动装箱用 `valueOf`（有缓存），`new Integer()` 永远 new 新对象；比较用 `equals`，不要用 `==`（除非确定在缓存内）。

---

**2. String 与 intern（你选了 D false false）**

- **正确**：应选 **A：false, true**。
  - `s1 = "abc"` → 常量池引用。  
  - `s2 = new String("abc")` → 堆上新对象，和 s1 不同。  
  - `s3 = s2.intern()` → 把 s2 对应字面量放入常量池（若没有）并**返回常量池中 "abc" 的引用**，即和 s1 指向同一对象。  
  - 所以 `s1 == s2` → false，`s1 == s3` → true。

**学习要点**：`intern()` 返回的是常量池中该字符串的引用；比较内容用 `equals`。

---

**3. final / finally / finalize（你选了 D）**

- **正确**：D 是错的。**final 修饰集合只保证引用不变，集合内部元素可以增删改。**
- 常见正确选项：B——finally 唯一不执行的情况包括 JVM 退出、线程被 kill、无限循环等。

**学习要点**：final 修饰变量 = 引用不可变；final 修饰类 = 不可继承；final 修饰方法 = 不可重写。finalize 已废弃，不依赖它。

---

**4. 方法参数传递（你选了 A）**

- 严谨说法：**Java 只有值传递**。引用类型传的是「引用的副本」（即拷贝了一份引用，指向同一对象），所以不能改变调用方的引用指向，但可以改对象内容。
- 若题目选项里有「只有值传递」，选它更准确；若只有「引用类型传引用本身」，在多数题目里会算对，但要心里清楚本质是「传值的引用副本」。

---

### 2.2 简答补强：不可变对象、分派、HashMap 并发

**不可变对象（4 分要答全）**

- **设计**：类 final、字段 final、无 setter、若字段是引用则防御性拷贝或保证其也不可变。  
- **优点**：线程安全、可做 key/缓存、无副作用。  
- **实现**：构造时赋好所有字段，不暴露可变内部状态。  
- **举例**：String、Integer、BigDecimal、LocalDate 等。

**静态分派 vs 动态分派（对应重载 vs 重写）**

- **静态分派**：编译期确定，看「静态类型」（声明类型）；对应 **重载**——选哪个重载方法在编译期就定了。  
- **动态分派**：运行期确定，看「实际类型」；对应 **重写**——子类重写父类方法，运行时按对象实际类型调用。  
- 一句话：重载 = 编译期多态（静态分派），重写 = 运行期多态（动态分派）。

**HashMap 高并发问题（你已写对，可再精炼）**

- 共同：非线程安全，多线程 put 可能丢数据。  
- JDK 1.7：头插法扩容会产生**环形链表**，导致 CPU 飙高甚至死循环。  
- JDK 1.8：改为尾插法，不再成环，但仍会**覆盖/丢数据**，要用 ConcurrentHashMap。

## 三、模块二：集合框架（错题与补强）

### 3.1 易错与漏选

- **HashMap 1.8**：树化阈值 8、链表化（退化）阈值 6 也是对的，多选若出现要选上。  
- **阻塞队列**：**ArrayBlockingQueue、LinkedBlockingQueue** 等才是阻塞队列；**PriorityQueue 不是**阻塞队列，不要多选。

### 3.2 必须记准的细节

**ArrayList 扩容**

- 初始：**默认容量 0**，第一次 add 后变为 **10**（或按第一次 add 的数量）。  
- 扩容倍数：**1.5 倍**，即 `newCapacity = oldCapacity + (oldCapacity >> 1)`，不是 2 倍+1。  
- 流程：计算新容量 → 拷贝到新数组 → 引用指向新数组。

**ConcurrentHashMap 1.8 的 size**

- 用 **baseCount** + **CounterCell[]** 分桶计数，多线程累加时减少竞争；最后 size = baseCount + 各 CounterCell 之和。

**指定初始容量**

- **原因**：减少扩容次数，避免多次数组拷贝。  
- **计算**：若已知大概 size 为 n，HashMap 建议 `(n / loadFactor) + 1` 再取 2 的幂；ArrayList 直接 `new ArrayList(n)`。

## 四、模块三：JVM（缺失与补强）

### 4.1 你写「不知道」的部分：对象创建完整流程

按顺序能默写即可：

1. **类加载检查**：看常量池能否定位到类、类是否已加载。  
2. **分配内存**：堆上分配——指针碰撞（Serial/ParNew）或空闲列表（CMS）。  
3. **初始化零值**：保证实例字段未显式赋值也有默认值。  
4. **设置对象头**：Mark Word（哈希、GC 年龄、锁信息等）、类型指针。  
5. **执行 `<init>`**：即执行你的构造方法里的代码。

### 4.2 CMS 与 Concurrent Mode Failure

- **流程**：初始标记（STW）→ 并发标记 → 重新标记（STW）→ 并发清除。  
- **优点**：并发、低停顿。  
- **缺点**：碎片多；并发阶段产生新垃圾无法当轮处理。  
- **Concurrent Mode Failure**：并发清理时空间不足，只能退化为 **Serial Old** 做 Full GC，停顿很长。  
- **缓解**：预留足够空间、调大堆或提前触发 CMS。

### 4.3 内存泄漏 vs 内存溢出 + 4 种泄漏场景

- **泄漏**：对象不再用但被引用，GC 回收不了，占着内存。  
- **溢出**：需要的内存超过堆/元空间等上限，OOM。  
- **4 种典型泄漏**：① 静态集合不断加对象不删；② 各种连接（DB、Socket）未关闭；③ 监听器/回调未移除；④ **ThreadLocal 用后未 remove**。

### 4.4 定位 OOM 的完整思路

1. **-XX:+HeapDumpOnOutOfMemoryError -XX:HeapDumpPath=...** 拿到 dump。  
2. **MAT / JVisualVM** 打开 dump，看 **Dominator Tree、Histogram**，找占用大的对象和谁在引用。  
3. **思路**：是否大对象、是否某类对象数量异常多、是否线程栈/本地变量持有大对象、是否堆外/直接内存。

## 五、模块四：多线程与 JUC（错题与手写）

### 5.1 错题纠正

**start() vs run()（你选了 A）**

- **正确**：**start()** 才会「启动线程」（让 JVM 调度 run）；**run()** 只是普通方法，直接调 run() 并不会新起线程。所以「run() 启动线程」是错的，应选 **B：start() 使线程进入就绪**。

**保证原子性的方式（多选你选了 abcd）**

- **volatile 不保证原子性**，只保证可见性+禁止指令重排。保证原子性的选：**CAS、synchronized、AtomicInteger** 等，不要选 volatile。

### 5.2 手写：DCL 单例 + 为什么要 volatile

```java
/**
 * 单例模式-双重检查锁（DCL）标准实现
 * 特点：线程安全、懒加载、性能优异、代码规范
 */
public class Singleton {
    // 1. 私有静态volatile实例：核心，防止指令重排序+保证可见性
    private static volatile Singleton instance;

    // 2. 私有构造方法：禁止外部实例化，保证单例唯一性
    private Singleton() {
        // 可选：防止通过反射破坏单例（增强版，标准基础版可省略）
        if (instance != null) {
            throw new IllegalStateException("单例对象已创建，禁止重复实例化");
        }
    }

    // 3. 公共静态获取实例方法：双重检查锁核心逻辑
    public static Singleton getInstance() {
        // 第一层检查：避免非必要加锁，提升高并发性能
        if (instance == null) {
            // 类级别同步锁：保证多线程下只有一个线程能进入创建逻辑
            synchronized (Singleton.class) {
                // 第二层检查：防止多个线程等待锁后重复创建实例
                if (instance == null) {
                    instance = new Singleton();
                }
            }
        }
        return instance;
    }
}
```

**为什么要 volatile**：  
`instance = new Singleton()` 不是原子操作：① 分配内存 ② 初始化对象 ③ 把引用赋给 instance。JIT 可能重排成 ③→②，其他线程可能拿到未初始化完的对象。**volatile** 禁止指令重排，保证 ② 在 ③ 之前完成。

### 5.3 手写：必然死锁 + 定位与避免

完整可运行代码见文末 **「附录：手写代码汇总」** 中的「必然死锁示例」。

**要点**：两把锁 lockA、lockB；线程1 先拿 A 再拿 B，线程2 先拿 B 再拿 A；中间 sleep 一下让双方都持有一把锁，再等对方释放，形成循环等待。

**定位**：`jstack <pid>` 或 IDEA 线程 dump，搜 **deadlock**；会标出哪两个线程、持有什么锁、在等什么锁。  
**避免**：统一加锁顺序（都先 A 再 B）、超时锁（tryLock）、死锁检测、减少锁粒度。

---

### 6.0 先建立概念：什么是 IO？为什么会有 BIO / NIO / Netty？

#### 6.0.1 什么是 IO？

**IO = Input / Output（输入/输出）**，指的是程序和「外部」交换数据：

- **读文件**：程序从磁盘读入数据（Input）
- **写文件**：程序把数据写到磁盘（Output）
- **网络**：程序通过网卡收发数据，也是 IO（读 = 从网络收包，写 = 向网络发包）

所以「IO」不是一个具体技术，而是一类操作：**凡是和磁盘、网络、设备打交道的数据读写，都叫 IO**。  
Java 里最早做这类事的是 **java.io** 包（流式 API），后来为了更好支持**高并发网络**，又有了 **java.nio**（NIO），再后来大家常用 **Netty** 这类框架在 NIO 之上做网络应用。下面按「是什么 → 有什么问题 → 下一代怎么解决」来串。

---

#### 6.0.2 阻塞 vs 非阻塞、同步 vs 异步（必区分）

- **阻塞 / 非阻塞**：说的是**当前线程**在等 IO 时能不能干别的事。  
  - **阻塞**：调用了 read()，数据没到之前，线程就一直卡在那里等，啥也不干。  
  - **非阻塞**：调用了 read()，数据没到就立刻返回「还没好」，线程可以去处理别的连接或逻辑。
- **同步 / 异步**：说的是「谁」在等 IO 完成。  
  - **同步**：我们的代码要自己等「数据读好了」再继续往下执行（要么阻塞等，要么轮询查「好了没」）。  
  - **异步**：把「读」交给系统/内核，我们只登记一个回调；数据好了之后，系统会通知我们（回调或 Future），我们不用自己傻等。

组合起来就有：**同步阻塞**（BIO）、**同步非阻塞 + 多路复用**（NIO）、**异步**（AIO）。面试里说的「NIO」多数指「多路复用的同步非阻塞」，不是 AIO。

---

#### 6.0.3 BIO 是什么？有什么问题？

**BIO = Blocking IO（阻塞式 IO）**，对应 Java 里传统的 **InputStream / OutputStream、ServerSocket#accept()、Socket#getInputStream().read()** 等。

- **怎么工作**：  
  - 服务端：`accept()` 等一个新连接 → 等到了就为这个连接开一个线程，在这个线程里 `read()` 读请求，读不到就**一直阻塞**；写完响应再读下一次。  
  - 一个连接一个线程，线程里读/写都是阻塞的。
- **问题**：  
  - 连接多了线程就爆炸（一个线程栈大概 1MB，一万连接就 10GB 内存）。  
  - 大部分时间线程在「等 IO」，CPU 没事干，资源浪费。  
  - 所以 BIO 适合**连接数不多、短连接**的场景（例如老一代的 Tomcat 默认就是 BIO 的 Connector）。

**一句话**：BIO = 一个线程盯一个连接，读/写都阻塞，适合低并发。

---

#### 6.0.4 NIO 是什么？解决什么问题？

**NIO = New IO / Non-blocking IO**，在 Java 里指 **java.nio** 包那一套：**Channel、Buffer、Selector**。

- **要解决的问题**：用尽量少的线程，同时管很多个网络连接（读/写不阻塞，有数据再处理）。
- **核心思想**：**多路复用**——一个线程通过 **Selector** 监听很多个 **Channel**（每个 Channel 代表一个连接），哪个 Channel 有数据可读/可写，就只处理那一个，其它不占着线程傻等。
- **和 BIO 的对比**：  
  - BIO：1 个连接 1 个线程，线程阻塞在 read。  
  - NIO：1 个线程 + 1 个 Selector，可以监听成百上千个 Channel；哪个有事件就处理哪个，所以是「非阻塞」+「多路复用」。

**一句话**：NIO = 用 Channel + Buffer + Selector 实现「一个线程管多连接」，适合高并发、长连接。

---

#### 6.0.5 Netty 是什么？和 NIO 什么关系？

**Netty** 是一个**基于 NIO 的网络应用框架**（第三方库，不是 JDK 自带的）。

- **关系**：底层用的就是 Java NIO（Channel、Selector 等），但在上面做了封装和增强。
- **为什么要用 Netty 而不是裸写 NIO**：  
  - 原生 NIO API 难用（要自己管 Buffer 的 flip、clear，要处理粘包拆包、边界情况）。  
  - Netty 把**线程模型**（Boss 接客、Worker 干活）、**编解码、粘包拆包、心跳**等都做好了，写业务逻辑就行。  
  - 性能好：零拷贝、内存池、更合理的 Reactor 模型等。
- **典型用途**：RPC 框架（Dubbo）、消息队列客户端、HTTP 服务器（如某些网关）、游戏/IM 长连接等，凡是「高并发网络 IO」都会用到。

**一句话**：Netty = 在 NIO 之上封装的、易用且高性能的网络框架；面试里「NIO」常和 Netty 一起问。

---

### 6.1 BIO / NIO / AIO 对比（简答 5 分）

在有了上面概念后，答题可以按下面表格 + 三句话概括。

|        | BIO              | NIO                    | AIO                    |
|--------|------------------|------------------------|------------------------|
| 全称/含义 | Blocking IO，阻塞式 | Non-blocking IO，多路复用 | Asynchronous IO，异步 IO |
| 阻塞性 | 阻塞（线程卡在 read/accept） | 非阻塞（select 到再处理） | 异步（OS 完成后回调） |
| 连接模型 | 1 线程 1 连接     | 1 线程多连接（Selector） | 回调/Future，少线程     |
| 线程   | 每连接一线程     | 少线程多 Channel       | 回调，少线程           |
| 场景   | 连接少、短连接   | 高并发、长连接         | 高并发、异步 IO（Linux 上少用） |

- **BIO**：accept/read 都阻塞，一连接一线程；适合连接数不多。  
- **NIO**：Channel + Buffer + Selector 多路复用，一个线程管多个连接；适合高并发、长连接。  
- **AIO**：异步，由 OS 完成 IO 后回调；Java 里用得少，面试常说的「NIO」一般指多路复用 NIO。

**笔试单选常考**：B 是阻塞、N 是多路复用非阻塞、A 是异步；NIO 不是「一个线程一个连接」，是「一个线程多连接」。

---

### 6.2 NIO 三大组件（4 分）

先理解「多路复用」：一个线程怎么同时等很多个连接？答案是 **Selector**——它像一个大管家，负责问内核「哪些连接已经可读/可写了」，然后只把「就绪」的连接交给程序处理。数据从哪读、往哪写？用 **Channel** 和 **Buffer**。

#### 6.2.1 Channel（通道）

- **是什么**：可以理解为「连接」或「管道」，双向的（既可读又可写），对应 BIO 里的 Stream 升级版。  
- **常见类型**：`ServerSocketChannel`（服务端监听）、`SocketChannel`（一个 TCP 连接）、`FileChannel`（文件）。  
- **作用**：把网络或文件的数据读到 Buffer，或把 Buffer 的数据写到网络/文件。

#### 6.2.2 Buffer（缓冲区）

- **是什么**：一块连续内存，用来暂存要读/写的数据；NIO 里读和写都绕不过 Buffer。  
- **重要属性**：`capacity`（容量）、`position`（当前读写位置）、`limit`（可读/写的上界）。  
- **两种模式**：  
  - **写模式**：往 Buffer 里 put 数据，position 往后移。  
  - **读模式**：从 Buffer 里 get 数据，position 往后移。  
- **flip()**：**写模式 → 读模式**。把 limit 设为当前 position（刚写完的位置），position 置 0，这样接下来 get 就从 0 读到 limit。所以「flip 的作用」常考：**写模式切换为读模式**，选「写转读」就对了。

#### 6.2.3 Selector（多路复用器）

- **是什么**：一个对象对应一个线程，可以注册很多个 Channel；Selector 负责问内核「哪些 Channel 就绪了」（可读/可写/可连接）。  
- **工作方式**：调用 `select()` 会阻塞（或超时）直到有 Channel 就绪，然后返回就绪的 Channel 集合，程序只遍历这些 Channel 做 read/write，不会为每个连接都开一个阻塞的 read。  
- **和 Channel 的关系**：Channel 注册到 Selector，并声明「我关心读/写/连接」中的哪些事件；Selector 只返回有事件的 Channel。

**答题要点**：Channel 双向通道、Buffer 读写用 flip 切换、Selector 多路复用；关系是「Channel 注册到 Selector，数据经 Buffer 读写，select() 得到就绪 Channel 再处理」。

---

### 6.3 Java 原生序列化缺点与替代（3 分）

#### 6.3.1 什么是序列化？

**序列化**：把内存里的 Java 对象变成**字节流**，方便存到文件或通过网络发出去。  
**反序列化**：把字节流再变回 Java 对象。  
这样不同进程、不同机器之间才能传「对象」（否则只能传字节，对方不知道结构）。

#### 6.3.2 Java 原生序列化（ObjectOutputStream / ObjectInputStream）

- **是什么**：JDK 自带的，实现 `Serializable`，用 `ObjectOutputStream.writeObject()` 等。  
- **缺点**：只支持 Java；版本兼容差（类增删字段易出问题）；体积大、性能一般；反序列化会执行类型信息里的逻辑，容易被构造恶意数据攻击（反序列化漏洞）。  
- **替代**：JSON（如 Jackson）跨语言、可读；Protobuf 二进制、高性能、跨语言；Hessian 也是二进制、跨语言。实际项目里很少再用 Java 原生序列化，尤其是跨系统、跨语言时。

答题时：**缺点**写「仅 Java、版本兼容差、体积大、性能一般、有安全风险」；**替代**写「JSON、Protobuf、Hessian 等」。

### 6.4 Netty 优势与核心组件（4 分）

#### 6.4.1 为什么有了 NIO 还要用 Netty？

- 原生 NIO 要自己管 Buffer 的 flip/clear、边界；要自己处理**粘包拆包**（一次收到半条或多条消息）；线程模型也要自己写。  
- Netty 把这些都封装好了：**API 简单**、**内置粘包拆包**（如按长度、分隔符）、**主从 Reactor 线程模型**清晰，还有零拷贝、内存池等优化，所以高并发网络几乎都用 Netty 而不是裸 NIO。

#### 6.4.2 主从 Reactor 多线程（常考）

- **主 Reactor（Boss）**：一般一个线程，只负责 `accept()` 新连接，把新连接注册给 **从 Reactor**。  
- **从 Reactor（Worker）**：一个或多个线程，每个负责一批已建立的连接的 read/write。  
- 这样「接客」和「干活」分开，接客不阻塞干活，适合高并发。

#### 6.4.3 核心组件（答简答用）

- **Bootstrap**：启动入口，配置线程组、Channel 类型、参数等。  
- **Channel**：对 NIO Channel 的封装，代表一条连接。  
- **Pipeline**：责任链，一条连接上挂一串 **Handler**（入站、出站），数据进来或出去会依次经过这些 Handler。  
- **Handler**：入站（读到的数据怎么解码、业务处理）、出站（要发的数据怎么编码、写回）。

**答题要点**：Netty 对比 NIO 的优势（API 简单、粘包拆包、线程模型、性能）；模型是主从 Reactor；组件有 Bootstrap、Channel、Pipeline、Handler。

## 九、模块八：Spring（IOC 与循环依赖补强）

### 9.1 IOC 完整流程（你写「不理解题目」）

题目是让你按顺序说出：**扫描 → 解析 → BeanDefinition → 实例化前后 → 依赖注入 → 初始化**。

- **扫描**：根据包路径/注解找到 Bean。  
- **解析**：生成 **BeanDefinition**（元数据）。  
- **实例化**：反射 new 对象（构造器）。  
- **属性填充**：注入 @Autowired 等依赖（此时可能触发依赖 Bean 的创建）。  
- **初始化**：执行 Aware、@PostConstruct、InitializingBean、init-method 等。  
- **销毁**：@PreDestroy、DisposableBean、destroy-method。

### 9.2 循环依赖为什么三级缓存不能解决「构造器循环」

- **Setter/字段注入**：可以先 new 出「半成品」（只执行了构造，未注入属性），放进三级缓存，供对方引用，再回来把属性填完。  
- **构造器注入**：要 new 就必须先拿到构造参数（即依赖的 Bean），而对方也要先拿到你，两边都卡在「构造」这一步，没有「半成品」可暴露，所以**构造器循环依赖无法用三级缓存解决**，只能改设计（如改为 setter/字段注入或拆依赖）。

## 十、模块九：消息队列（你基本未答，需系统补）

> 先搞清「MQ 是什么、在系统里扮演什么角色」，再记优点缺点和可靠性、幂等、堆积等答题点。

---

### 10.0 先建立概念：什么是消息队列（MQ）？

**消息队列（Message Queue，MQ）** 可以理解成一个「**中间人信箱**」：

- **生产者**：业务代码把「消息」（一段数据，比如订单号、用户 ID）发到队列里，不直接调下游。
- **队列/Broker**：中间件负责把消息存下来，按顺序或按规则分给消费者。
- **消费者**：另一个服务从队列里取消息，按自己的节奏处理（下单、发短信、扣库存等）。

这样做的效果是：**发消息的人不用等处理完**（异步）、**发消息的人和处理的模块可以独立扩容和部署**（解耦）、**流量高峰时请求先堆在队列里，下游按能力慢慢消费**（削峰）。  
常见 MQ 有：RocketMQ、Kafka、RabbitMQ 等；校招里 RocketMQ、Kafka 问得多。

---

### 10.1 引入 MQ 优点与缺点（5 分）

- **优点**：异步（提速）、解耦（服务间通过 MQ 通信）、削峰（请求先到 MQ，下游按能力消费）。  
- **缺点**：一致性难（最终一致）、可用性依赖 MQ、消息可能重复/丢失、运维与复杂度增加。

### 10.2 保证消息不丢失（4 分）

- **生产端**：同步发送+回调确认、重试、事务消息（若支持）。  
- **Broker**：持久化（刷盘）、副本（多副本同步）。  
- **消费端**：业务处理成功后再 ack；失败可重试或进死信队列。

### 10.3 重复消费与幂等（4 分）

- **原因**：网络重试、Broker 重投、消费 ack 失败等。  
- **幂等**：业务用唯一 key（如订单号+业务场景）做幂等——先查是否已处理，再写 DB/缓存；或唯一索引防重。

### 10.4 消息堆积原因与处理（5 分）

- **原因**：消费慢于生产、消费者宕机、单线程消费、下游 DB 慢等。  
- **思路**：加消费者实例、增加分区/队列、批量消费、异步处理、排查慢消费与下游瓶颈、必要时先堆积再限流保护下游。

## 十一、模块十：微服务与分布式（你未答，需系统补）

> 先弄清「微服务、注册中心、网关、分布式事务」分别解决什么问题，再背具体方案。

---

### 11.0 先建立概念：微服务与分布式在说什么？

- **微服务**：把一个大系统拆成很多小服务（每个服务一个进程、一个库或一组表），服务之间通过 **HTTP/RPC** 或 **MQ** 通信。好处是独立部署、技术栈可不同；代价是运维复杂、要解决「服务发现、限流、熔断、分布式事务」等问题。
- **服务注册与发现**：每个服务启动时把自己的 IP+端口 注册到**注册中心**（如 Nacos、Eureka）；别的服务要调用它时，先去注册中心查「这个服务有哪些实例」，再选一个发请求。这样实例扩缩容、换机器，调用方不用改配置。
- **网关**：所有外部请求先到一个统一的入口（网关），由网关做路由（转发到哪个微服务）、鉴权、限流、日志等，后面的微服务只关心业务。
- **分布式事务**：一个业务要跨多个服务、多个数据库改数据，要保证「要么全成功要么全失败」或「最终一致」，这就是分布式事务；和单机事务不一样，不能靠一个数据库的 commit/rollback 解决。

下面按「注册发现、雪崩/熔断/限流/降级、分布式事务、幂等、分布式 ID、网关」给答题要点。

---

### 11.1 服务注册发现（3 分）

- 服务启动时向**注册中心**注册自己的地址与元数据；客户端从注册中心**发现**实例列表。  
- **心跳**：定时上报存活；**健康检查**：注册中心或探针检查实例可用性；**故障剔除**：超时未心跳则从列表摘掉。

### 11.2 雪崩、熔断、限流、降级（4 分）

- **雪崩**：一个服务挂了导致调用方大量阻塞/失败，蔓延到整个系统。  
- **熔断**：调用失败率/慢调用超过阈值，熔断器打开，直接失败或走降级，不继续打挂掉的服务。  
- **限流**：控制 QPS/并发数，超过则拒绝或排队（令牌桶、漏桶等）。  
- **降级**：非核心功能关掉或返回默认值，保核心。

### 11.3 分布式事务（5 分）

- **2PC**：协调者两阶段提交，强一致但阻塞、性能差，适合短事务。  
- **TCC**：Try-Confirm-Cancel，业务自己实现补偿，适合对一致性要求高、可补偿的业务。  
- **可靠消息最终一致**：发事务消息/本地消息表，消费者最终一致；适合允许延迟一致的场景。  
- **SAGA**：长事务拆成多个本地事务+补偿；适合长流程、可补偿。

### 11.4 接口幂等（4 分）

- **Token**：先发 token，提交时带 token 且仅能使用一次。  
- **唯一索引**：DB 唯一键防重复插入。  
- **防重表**：先插业务唯一键，插得进再做业务。  
- **分布式锁**：同一业务键加锁，保证只处理一次。

### 11.5 分布式 ID（4 分）

- **UUID**：无规律、长，做 DB 主键影响索引性能。  
- **雪花**：趋势递增、高性能，依赖时钟与机器位。  
- **号段**：DB 取一段 ID，应用内自增，减少 DB 访问。  
- **选型**：要趋势递增用雪花/号段；要简单可用 UUID（或改造雪花）。

### 11.6 网关作用（4 分）

- **路由**：根据 path/host 转发到不同服务。  
- **限流**：按 IP/用户/接口限流。  
- **认证**：统一鉴权、验签。  
- **日志/监控**：访问日志、耗时、状态码。  
- **跨域**：统一配置 CORS 等。

## 十三、模块十二：计网与 OS（你部分对，补全要点）

### 13.1 TCP 三次握手、四次挥手（4 分）

- **三次握手**：C 发 SYN → S 回 SYN+ACK → C 回 ACK；目的：确认双方收发能力、协商初始序列号。  
- **四次挥手**：C 发 FIN → S 回 ACK（可能还有数据）→ S 发 FIN → C 回 ACK；TIME_WAIT 等 2MSL 再关，防止最后的 ACK 丢包导致 S 重发 FIN。

### 13.2 HTTP 状态码（3 分）

- **2xx**：200 OK、201 Created、204 No Content。  
- **3xx**：301/302 重定向、304 Not Modified。  
- **4xx**：400 请求错误、401 未认证、403 无权限、404 未找到。  
- **5xx**：500 服务器错误、502 网关错误、503 不可用。

### 13.3 进程与线程、通信方式、IO 模型（3 分）

- **进程**：资源分配单位；**线程**：调度单位，共享进程资源。  
- **进程通信**：管道、消息队列、共享内存、信号量、Socket。  
- **IO 模型**：阻塞/非阻塞、同步/异步、多路复用（select/poll/epoll）。

```java
    private static final Object lockA = new Object();
    private static final Object lockB = new Object();

    public static void main(String[] args) {
        Thread t1 = new Thread(() -> {
            synchronized (lockA) {
                System.out.println("线程1 持有 lockA，等待 lockB");
                try {
                    Thread.sleep(50);
                } catch (InterruptedException e) {
                    Thread.currentThread().interrupt();
                }
                synchronized (lockB) {
                    System.out.println("线程1 拿到 lockB");
                }
            }
        }, "thread-1");

        Thread t2 = new Thread(() -> {
            synchronized (lockB) {
                System.out.println("线程2 持有 lockB，等待 lockA");
                try {
                    Thread.sleep(50);
                } catch (InterruptedException e) {
                    Thread.currentThread().interrupt();
                }
                synchronized (lockA) {
                    System.out.println("线程2 拿到 lockA");
                }
            }
        }, "thread-2");

        t1.start();
        t2.start();
        // 运行后程序卡住，用 jstack <pid> 可看到 Found one Java-level deadlock
    }
}
```

**避免死锁的写法（统一加锁顺序）**：两个线程都先锁 lockA 再锁 lockB，则不会形成循环等待。

```java
// 正确：统一顺序，先 A 后 B
Thread t1 = new Thread(() -> {
    synchronized (lockA) {
        synchronized (lockB) { /* 业务 */ }
    }
});
Thread t2 = new Thread(() -> {
    synchronized (lockA) {
        synchronized (lockB) { /* 业务 */ }
    }
});
```

### 3. 生产者消费者（阻塞队列版，常考手写）

```java
import java.util.concurrent.ArrayBlockingQueue;
import java.util.concurrent.BlockingQueue;

public class ProducerConsumerDemo {
    private static final BlockingQueue<Integer> queue = new ArrayBlockingQueue<>(10);

    public static void main(String[] args) {
        Thread producer = new Thread(() -> {
            try {
                for (int i = 0; i < 20; i++) {
                    queue.put(i);
                    System.out.println("生产: " + i);
                }
            } catch (InterruptedException e) {
                Thread.currentThread().interrupt();
            }
        });
        Thread consumer = new Thread(() -> {
            try {
                for (int i = 0; i < 20; i++) {
                    Integer v = queue.take();
                    System.out.println("消费: " + v);
                }
            } catch (InterruptedException e) {
                Thread.currentThread().interrupt();
            }
        });
        producer.start();
        consumer.start();
    }
}
```

### 4. 两线程交替打印 1–100（synchronized + wait/notify）

```java
public class AlternatePrint {
    private static int num = 1;
    private static final Object lock = new Object();

    public static void main(String[] args) {
        Thread t1 = new Thread(() -> {
            synchronized (lock) {
                while (num <= 100) {
                    if (num % 2 == 1) {
                        System.out.println(Thread.currentThread().getName() + ": " + num++);
                        lock.notify();
                    } else {
                        try {
                            lock.wait();
                        } catch (InterruptedException e) {
                            Thread.currentThread().interrupt();
                        }
                    }
                }
            }
        }, "A");
        Thread t2 = new Thread(() -> {
            synchronized (lock) {
                while (num <= 100) {
                    if (num % 2 == 0) {
                        System.out.println(Thread.currentThread().getName() + ": " + num++);
                        lock.notify();
                    } else {
                        try {
                            lock.wait();
                        } catch (InterruptedException e) {
                            Thread.currentThread().interrupt();
                        }
                    }
                }
            }
        }, "B");
        t1.start();
        t2.start();
    }
}
```
