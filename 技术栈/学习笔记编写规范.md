# 📐 学习笔记编写规范

> 本规范基于 **MySQL、Redis、Spring Framework、Spring Boot** 等现有学习笔记提炼，适用于技术栈下各模块学习笔记的编写与重构，保证风格统一、便于系统学习与面试速查。

------

## 1️⃣ 文档整体结构

| 部分 | 必须/可选 | 说明 |
|------|-----------|------|
| **封面标题** | 必须 | 手册名 + 一句话说明；若有速查文档则注明「系统学习见下文各章；日常速查见同目录《XXX》」 |
| **章** | 必须 | 按学习顺序组织，每章一个主题，从概述/基础到原理/进阶 |
| **本章小结** | 建议每章必有 | 表格：知识点 \| 面试关键词 \| 实际应用（或「模块 \| 关键点」）；或核心章用一段话收束 |
| **面试内容** | 核心章建议有 | 可为章末「面试常见追问」表格，或节内「面试扩展」「面试官常问」表格/标准回答 |
| **学习要点** | 建议核心章有 | 3～5 条收束，或与小结合并为一段 |
| **附录** | 可选 | 速查表、流程速记、面试答题模板等 |
| **学习建议** | 可选 | 学习顺序、实战练习建议 |
| **版本说明** | 建议 | 文档开头或相关章注明适用版本（如 MySQL 8.0+、JDK 17），便于读者对照环境 |

------

## 2️⃣ 章节布局与标题层级

### （1）两套可选风格（任选其一并全文统一）

**风格 A：数字 + 中文括号（偏理论/框架）**

| 层级 | 格式 | 示例 |
|------|------|------|
| 文档标题 | `# 📘《XXX 学习与实战手册》` 或 `# 🧭 XXX 学习与实战手册` | 仅文首一处 |
| 章 | `# 🌱 第X章：章节名` 或 `# 第X章：章节名` | 每章一个一级标题，章 emoji 可省略 |
| 大节 | `## 1️⃣ 大节标题`、`## 2️⃣ …` | 用数字 emoji（1️⃣ 2️⃣ …） |
| 小节 | `### （1）小节标题`、`### （2）…` | 中文括号数字 |

**风格 B：多级数字 / 数字+emoji（偏命令/配置/实战）**

| 层级 | 格式 | 示例 |
|------|------|------|
| 文档标题 | 同上 | 同上 |
| 章 | `# 第X章：章节名` 或 `# 🧩 第X章：章节名` | 可按章类型使用不同 emoji（如概述🧱、数据结构🧩、持久化💾） |
| 大节 | `## 1.1`、`## 1.2` 或 `## 2.1`、`## 2.2` | 小数点编号 |
| 小节 | `### 1.1.1`、`### 2.1.1` 或 `### 🔹 1.1.1`、`### 🧩 背景` | 可加 emoji 区分小节类型（概念🔹、原理、命令、场景） |

> 💡 **约定**：同一份笔记内只采用一种风格；若已有同系列笔记（如 Spring Framework / Spring Boot），尽量与系列保持一致。

### （2）分隔线

- 章标题下、大节之间、章末小结前使用 **`------`**（六个短横线）。
- 同一小节内一般不插分隔线，避免碎片化。

### （3）附录 / 特殊章

- 附录可用 `# 📎 附录`，其下用 `## 1️⃣ 2️⃣` 或 `## 1.1 1.2` 分块。
- 学习建议可用 `# ✅ 学习建议` 或置于某章末尾。

------

## 3️⃣ 内容深度与组织方式

### （1）单节内容推荐顺序

1. **概念/定义**：一句话或短段说清「是什么」；重要术语可加粗，首次出现可配合表格。
2. **原理/对比**：用表格对比同类概念、优缺点、适用场景；必要时配合示意图（ASCII 或文字描述）。
3. **命令/配置/代码**：给出可直接使用的示例，标明语言或类型（` ```sql `、` ```java `、` ```yaml ` 等）；代码前可加一句说明。
4. **应用场景/实战建议**：表格或列表说明「用在什么场景」「注意点」「与它章关联」。

### （2）反例与避坑（建议适当加入）

- **不推荐写法**：在讲「正确用法」时，可附带 1～2 个「不推荐」或「错误示例」，并简短说明原因（如索引失效、N+1 查询、全表扫描）。
- **常见错误/失效原因**：对易踩坑点（如 SQL 索引失效、配置项误用、循环依赖）用表格或列表列出「原因 \| 示例 \| 正确做法」。
- **生产慎用**：对仅适合开发/测试的配置或命令，用 `> ⚠️` 明确标注「生产慎用/禁用」。

### （3）深度要求

| 类型 | 要求 |
|------|------|
| **基础章** | 概念清晰、命令/配置可照抄、能上手练习 |
| **原理章** | 说清「为什么」「和谁对比」「底层大致怎么实现」；可配合流程图、结构图 |
| **面试高频** | 必有小结表（知识点 \| 面试关键词 \| 实际应用），建议有「面试常见追问」或节内「面试扩展」「标准回答」 |

### （4）内容数量参考

| 项目 | 建议 |
|------|------|
| **每章大节数** | 2～6 个，视主题而定 |
| **每大节小节数** | 2～5 个，避免单节过长 |
| **表格** | 概念对比、命令/API、配置项、小结以表格为主；列数 2～4 列，列名简短 |
| **代码块** | 必须标明语言；以关键片段为主，过长可截取并加注释 |
| **本章小结行数** | 表格时每章约 5～10 行；段落小结时 2～4 句 |

------

## 4️⃣ 内容块约定

### （1）表格

- **概念/对比**：用表格说清术语、对比项、选项说明。
- **列数**：以 2～4 列为宜，列名简短（如「说明」「用法」「注意」「面试关键词」）。
- **对齐**：表头与分隔行按 Markdown 标准（`| --- | --- |`）。

### （2）代码块

- 必须标明语言：` ```java `、` ```sql `、` ```yaml `、` ```xml `、` ```properties ` 等。
- 代码前可加一句说明（如「示例：」「配置：」）；过长可截取关键片段并注释。

### （3）强调与提示（引用块）

统一用引用块（`>`）区分类别，下列标记可任选使用、全文风格一致即可：

| 标记 | 用途 | 示例 |
|------|------|------|
| `> 💬 **一句话总结**` / `> **XXX**` | 全章/全节核心结论 | Spring 解决的是… |
| `> 💡 **重点**` | 易错点、面试必答 | IoC 是思想，DI 是实现。 |
| `> 🔹 **注意**` / `> ✅ **注意**` | 补充说明、与它处关联 | 详见第五章。 |
| `> ⚠️` | 慎用、风险提示 | 生产禁用 root 远程登录。 |

部分笔记使用 **📘 一句话总结**、**✅ 总结**、**🧩** 等与引用块混用，可保留；同一文档内风格统一即可。

### （4）示意图

- 可用 ASCII 图表示架构、流程、数据结构（如 B+树、组件关系、启动流程）。
- 保持简洁，能表达层次与关系即可。

------

## 5️⃣ 章末固定模块

### （1）本章小结

**形式一（推荐用于核心/面试章）**：表格

```markdown
# ✅ 本章小结  或  ## 📖 本章小结

| 知识点 | 面试关键词 | 实际应用 |
| ------ | ---------- | -------- |
| …      | …          | …        |

------
```

- 「面试关键词」便于背诵，「实际应用」便于举例；行数每章约 5～10 行。

**形式二（可用于偏实操/配置章）**：段落

```markdown
**本章小结**：用 2～4 句话概括本章核心（解决了什么问题、关键配置/命令、与它章关系）。

------
```

### （2）面试常见追问 / 面试扩展

**形式一**：章末独立小节

```markdown
## 🎯 面试常见追问

| 面试官提问 | 回答思路 |
| ---------- | -------- |
| …          | …        |

------
```

**形式二**：节内「面试扩展」「面试官常问」「标准回答」

- 在对应原理或概念的小节下，用表格或引用块给出「问题 + 回答要点」或「标准回答」。
- 回答用要点即可，不必长段叙述。

**标准回答建议结构**（便于面试口述）：**定义/是什么** → **原理/流程** → **对比/适用场景** → **注意点/坑**。每段 1～2 句即可，不必逐字背诵。

### （3）学习要点

```markdown
**学习要点**：

- 第一条：概念或原理层面的收束。
- 第二条：实现或配置上的注意点。
- 第三条：与其它章节/实际项目的关联。

------
```

- 3～5 条为宜，每条一句话；可与本章小结合并为一段（如「本章小结：… 学习要点：…」）。

------

## 6️⃣ 格式与细节

| 项目 | 约定 |
|------|------|
| **换行** | 表格、代码块、引用块前后各空一行，便于渲染与编辑。 |
| **中英文** | 中文与英文/数字之间可加空格，提升可读性（如「通过 `@Configuration` 配置」）。 |
| **专有名词** | 首次出现时可加粗（**BeanFactory**），后续可不再加粗。 |
| **术语一致** | 同一概念全文统一说法（如「三级缓存」与「三层缓存」择一）。 |
| **交叉引用** | 用「见第X章」「详见 XXX 节」即可，不必写死具体行号。 |
| **术语与缩写** | 英文缩写首次出现时给出全称（如 **RDBMS**（Relational Database Management System）、**IoC**（Inversion of Control）），便于查词与记忆。 |
| **版本与时效** | 涉及具体版本差异时（如 MySQL 8.0 移除 Query Cache、Spring Boot 3.x 需 JDK 17）在相关小节注明，避免读者按旧版操作出错。 |

以下 **6.1～6.4** 为格式与细节的补充细则，按需查阅即可。

------

## 6.1 可验证性与示例要求

| 项目 | 约定 |
|------|------|
| **代码/配置** | 示例尽量**可运行、可复现**；若依赖特定环境（如 JDK 17、MySQL 8、Redis 6），在示例前或章首用一句话注明。 |
| **命令与路径** | 命令行示例中的路径、端口、库名等可用占位或通用值（如 `localhost:3306`、`your_db`），必要时用注释说明「请替换为实际值」。 |

------

## 6.2 前后关联与学习路径

- **章首可选**：在章标题下用 1～2 句说明「本章前置知识」（如「需先掌握第X章 SQL 基础」）或「学完可继续」（如「下一章将讲事务与锁」），帮助形成学习路径。
- **小结中**：在「实际应用」列或学习要点里点明与它章的关联（如「索引设计见第三章」「与 Spring 事务配合见第X章」）。

------

## 6.3 自测与练习（建议核心章具备）

- **动手练习**：每章或每 2～3 章末尾可给出 1～2 个「实战练习建议」（如「为本项目表设计索引并 EXPLAIN 验证」「用 Redis 实现一个简单分布式锁」），便于巩固。
- **自测题**：核心章可在「面试常见追问」之外，加 1～3 道简答/选择题（不写答案，仅作自测），或在小结后写「自测：能否口述 XXX 流程？」。

------

## 6.4 图表与可读性

- **复杂流程**：超过 4～5 步的流程，用**有序列表**或 **ASCII 流程图**呈现，避免大段纯文字。
- **表格过宽**：单表列数超过 4 列时，考虑拆成两个表或保留最关键的列，避免在窄屏下难以阅读。
- **层级清晰**：同一小节内列表层级不超过 2 层（如 `-` 下再用 `-`），更深时考虑拆成新的小节或子标题。

------

```
## 1️⃣ 大节一  或  ## 1.1 大节一

…

```

------

## 8️⃣ 与其它文档的配合

- 若某技术有 **常用命令/注解/配置速查**，可单独维护《常用XXX与使用场景》类文档，在学习笔记开头说明「系统学习见各章，速查见 XXX」。
- 学习笔记以 **系统学习 + 面试复习** 为主，速查文档以 **按场景查用法** 为主，二者互补即可。

------

## 9️⃣ 质量自检清单（编写/修订后建议核对）

编写或修订完一章/整份笔记后，可快速核对以下项，提升一致性与可用性：

| 检查项 | 说明 |
|--------|------|
| 标题风格 | 全篇是否统一为风格 A 或 B，无混用？ |
| 版本与环境 | 涉及具体技术处是否注明适用版本或环境？ |
| 术语与缩写 | 重要缩写首次出现是否给出全称？ |
| 反例与避坑 | 易错点是否有「不推荐/常见错误/正确做法」？ |
| 小结与面试 | 核心章是否有小结表 + 面试追问或学习要点？ |
| 示例可运行 | 代码/配置/命令是否可直接使用或仅需少量替换？ |
| 前后关联 | 是否在章首或小结中点到前置/后续章节？ |
| 格式细节 | 表格/代码/引用块前后是否空行，术语是否全文一致？ |

------

> 💬 编写新笔记时：先选好 **章节标题风格**（风格 A 或 B），按「单章结构速查」搭好骨架；再按 **概念 → 原理/对比 → 命令/代码/配置 → 场景** 填表格与示例；适当加入 **反例与避坑**、**版本说明**、**自测/练习**；最后补 **本章小结** 与 **面试追问/学习要点**，并用 **质量自检清单** 过一遍，即可与现有 MySQL、Redis、Spring、Spring Boot 等笔记在质量与风格上保持一致。
