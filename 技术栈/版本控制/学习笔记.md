------

## 1.1 版本控制的起源与发展

### 🧩 什么是版本控制

**版本控制（Version Control）** 是一种记录文件内容变化，以便将来查阅特定版本历史的系统。它帮助开发团队：

- **追踪变更历史**：谁、何时、做了什么修改
- **协同开发**：多人同时工作在同一个项目上
- **版本回退**：随时回到之前的稳定版本
- **分支管理**：并行开发不同功能或版本

### 🚀 版本控制系统的发展

| 阶段 | 代表工具 | 年代 | 核心特点 |
|------|----------|------|----------|
| **本地版本控制** | RCS | 1980s | 仅本地存储，无网络协作 |
| **集中式版本控制** | CVS、SVN | 1990s-2000s | 中央服务器，客户端 checkout |
| **分布式版本控制** | Git、Mercurial | 2005+ | 本地完整仓库，支持离线操作 |

### 📜 Git 的诞生

- **2005年**：Linus Torvalds 为管理 Linux 内核开发而创建 Git
- **2008年**：GitHub 上线，推动 Git 普及
- **2014年**：GitLab 开源，提供自托管方案
- **当前**：Git 已成为全球最流行的版本控制工具

> ✅ Git 是当前校招与团队协作的必备技能。

------

## 1.2 版本控制工具对比

### 📚 集中式 vs 分布式

| 对比维度 | 集中式（SVN） | 分布式（Git） |
|----------|--------------|---------------|
| **架构** | 中央服务器，客户端只保存工作副本 | 每个节点都有完整仓库 |
| **离线工作** | 不支持，必须连接服务器 | 完全支持，可离线提交 |
| **分支操作** | 分支是目录拷贝，操作慢 | 分支是指针，操作极快 |
| **存储方式** | 增量存储 | 快照存储 |
| **协作方式** | checkout/commit 到中央服务器 | push/pull 在仓库间同步 |

### 🧩 常见工具对比

| 工具 | 类型 | 优势 | 劣势 | 适用场景 |
|------|------|------|------|----------|
| **Git** | 分布式 | 功能强大、分支轻量、生态完善 | 学习曲线较陡 | 绝大多数项目 |
| **SVN** | 集中式 | 简单易用、权限管理完善 | 依赖中央服务器 | 文档管理、二进制文件 |
| **Mercurial** | 分布式 | 简单易用、跨平台 | 生态相对较小 | 小型项目 |

## 1.3 Git 的核心概念

### 🔹 工作区、暂存区、仓库

Git 管理代码的三个核心区域：

```
┌─────────────────────────────────────────────────────┐
│                    Git 工作模型                      │
├─────────────────────────────────────────────────────┤
│  工作区 (Working Directory)                          │
│  └── 你看到的文件目录（除 .git 外）                   │
│          ↓ git add                                  │
│  暂存区 (Staging Area/Index)                         │
│  └── 下次提交的预备区                                │
│          ↓ git commit                               │
│  本地仓库 (Local Repository)                         │
│  └── .git 目录，存储完整历史                         │
│          ↓ git push                                 │
│  远程仓库 (Remote Repository)                        │
│  └── GitHub/GitLab 等托管平台                        │
└─────────────────────────────────────────────────────┘
```

| 区域 | 说明 | 对应命令 |
|------|------|----------|
| **工作区** | 当前看到的文件目录 | 编辑文件 |
| **暂存区** | 下次提交的预备区 | `git add` |
| **本地仓库** | 存储完整提交历史 | `git commit` |
| **远程仓库** | 托管平台上的仓库 | `git push/pull` |

### 🔹 提交（Commit）

**提交**是 Git 的核心概念，代表项目在某个时间点的快照：

- 包含唯一 SHA-1 哈希标识
- 记录作者、时间、提交信息
- 指向父提交，形成有向无环图（DAG）

```
A --- B --- C --- D (main)
       \
        E --- F (feature)
```

### 🔹 分支（Branch）与 HEAD

| 概念 | 说明 |
|------|------|
| **分支** | 指向某次提交的指针，创建分支只是新建指针 |
| **HEAD** | 指向当前所在分支的指针，代表"当前位置" |
| **默认分支** | 通常为 `main` 或 `master` |

> 💡 **Git 分支的本质**：只是一个指向提交的指针，创建分支成本极低（仅创建指针，不复制文件）。

### 🔹 远程仓库（Remote）

- **origin**：默认远程仓库名称
- **upstream**：上游仓库（fork 场景）
- 常见托管平台：GitHub、GitLab、Gitee、Bitbucket

## 1.4 Git 的核心优势

### 🌟 五大核心优势

| 优势 | 说明 |
|------|------|
| **分布式架构** | 本地完整仓库，无单点故障 |
| **分支轻量** | 分支是指针，创建/切换极快（毫秒级） |
| **离线支持** | 提交、查看历史、分支操作均可离线完成 |
| **内容寻址** | 使用 SHA-1 哈希标识对象，保证完整性 |
| **高性能** | 处理大型项目速度快 |

### 🎯 适用场景

| 场景 | 说明 |
|------|------|
| ✅ 团队协作 | 多人并行开发，代码合并 |
| ✅ 版本管理 | 追踪变更，随时回退 |
| ✅ 分支开发 | 功能开发、Bug修复、版本发布 |
| ✅ 代码审查 | Pull Request / Merge Request |
| ✅ 开源贡献 | Fork、PR 流程 |

## 1.5 面试高频题

### Q1：Git 与 SVN 的主要区别是什么？

> **标准回答**：
> 1. **架构不同**：Git 是分布式，每个节点都有完整仓库；SVN 是集中式，依赖中央服务器。
> 2. **分支操作**：Git 分支是指针，操作极快；SVN 分支是目录拷贝，操作慢。
> 3. **离线支持**：Git 支持离线提交、查看历史；SVN 必须连接服务器。
> 4. **存储方式**：Git 使用快照存储；SVN 使用增量存储。

### Q2：什么是工作区、暂存区和仓库？

> **标准回答**：
> - **工作区**：当前看到的文件目录，编辑文件的地方
> - **暂存区**：`git add` 后文件进入的区域，是下次提交的预备区
> - **本地仓库**：`.git` 目录，存储完整提交历史
> - **关系**：工作区 → `git add` → 暂存区 → `git commit` → 本地仓库

### Q3：Git 的分支是如何实现的？

> **标准回答**：
> Git 分支本质上是一个指向提交的指针（40字节的 SHA-1 哈希值）。创建分支只是新建一个指针，指向当前提交，因此成本极低。切换分支则是移动 HEAD 指针到目标分支。

## 2.1 仓库初始化与克隆

### 🔹 新建仓库

**`git init`**：在当前目录初始化 `.git` 目录，建立空仓库。

```bash
# 在当前目录初始化
git init

# 在指定目录初始化
git init my-project
cd my-project
```

### 🔹 克隆远程仓库

**`git clone <url>`**：克隆远程仓库到本地，包含完整历史。

```bash
# 克隆到当前目录（创建 repository 目录）
git clone https://github.com/username/repository.git

# 克隆到指定目录
git clone https://github.com/username/repository.git my-project

# 克隆指定分支
git clone -b develop https://github.com/username/repository.git
```

## 2.2 状态管理与提交

### 🔹 查看状态

**`git status`**：显示工作区与暂存区状态。

```bash
git status
```

**状态分类**：

| 状态 | 说明 | 处理方式 |
|------|------|----------|
| **Untracked** | 未跟踪的新文件 | `git add` 加入跟踪 |
| **Modified** | 已跟踪文件的修改 | `git add` 加入暂存区 |
| **Staged** | 已加入暂存区 | `git commit` 提交 |
| **Unmodified** | 未修改 | 无需处理 |

### 🔹 暂存修改

**`git add`**：将修改加入暂存区。

```bash
# 暂存单个文件
git add README.md

# 暂存所有修改（包括新文件和修改）
git add .

# 暂存所有修改（不包括新文件）
git add -u

# 交互式选择部分修改（hunk）
git add -p
```

### 🔹 提交修改

**`git commit`**：将暂存区内容提交到本地仓库。

```bash
# 提交暂存区（带提交信息）
git commit -m "feat: 新增用户登录功能"

# 提交并添加所有已跟踪文件的修改（跳过 git add）
git commit -am "fix: 修复登录验证 bug"

# 修改最近一次提交信息
git commit --amend -m "新的提交信息"

# 追加文件到最近一次提交（不改信息）
git commit --amend --no-edit
```

> ⚠️ **注意**：`--amend` 会修改提交历史，已推送到远程的提交慎用。

------

## 2.3 历史查看与比较

### 🔹 查看提交历史

**`git log`**：查看提交历史。

```bash
# 查看完整历史
git log

# 单行简略显示（推荐）
git log --oneline

# 图形化显示分支结构
git log --oneline --graph --all

# 查看某文件的修改历史
git log -p README.md

# 查看某次提交详情
git show <commit-hash>

# 查看最近 5 次提交
git log -5

# 查看提交统计信息
git log --stat
```

### 🔹 差异比较

**`git diff`**：比较不同版本之间的差异。

```bash
# 比较工作区与暂存区
git diff

# 比较暂存区与最近提交
git diff --staged
# 或
git diff --cached

# 比较工作区与指定提交
git diff HEAD

# 比较两个分支
git diff main feature/login

# 比较某文件在不同版本的差异
git diff HEAD~2 HEAD README.md
```

## 2.4 忽略文件

### 🔹 `.gitignore` 文件

在项目根目录创建 `.gitignore` 文件，定义忽略规则。

```gitignore
# 忽略所有 .log 文件
*.log

# 忽略 target 目录（Maven 构建输出）
/target

# 忽略 node_modules 目录（Node.js 依赖）
node_modules/

# 忽略 IDE 配置文件
.idea/
.vscode/
*.iml

# 忽略编译输出
*.class
*.jar
*.war

# 忽略敏感信息
.env
config.ini

# 不忽略 important.log（取反）
!important.log
```

### 🔹 常用忽略规则

| 模式 | 说明 | 示例 |
|------|------|------|
| `*.log` | 忽略所有 `.log` 文件 | `debug.log` |
| `temp/` | 忽略 `temp` 目录 | `temp/file.txt` |
| `/temp` | 仅忽略根目录的 `temp` | `/temp` 但不忽略 `/src/temp` |
| `!important.log` | 不忽略 `important.log` | 取反规则 |
| `doc/*.txt` | 忽略 `doc` 目录下的 `.txt` | `doc/readme.txt` |

### 🔹 已跟踪文件的处理

若文件已被 Git 跟踪，加入 `.gitignore` 后仍会继续跟踪。

```bash
# 从跟踪中移除文件（保留本地文件）
git rm --cached config.ini

# 提交修改
git commit -m "chore: 从跟踪中移除 config.ini"
```

## 2.5 面试高频题

### Q1：`git add`、`git commit`、`git push` 的区别？

> **标准回答**：
> - `git add`：将工作区的修改加入暂存区（Stage）
> - `git commit`：将暂存区的内容提交到本地仓库，生成一次提交记录
> - `git push`：将本地仓库的提交推送到远程仓库

### Q2：`git log` 和 `git show` 的区别？

> **标准回答**：
> - `git log`：查看提交历史列表，可显示多个提交
> - `git show`：查看某次提交的详细信息，包括修改内容（diff）

### Q3：`.gitignore` 对已跟踪的文件生效吗？

> **标准回答**：
> 不生效。`.gitignore` 只对未跟踪的文件有效。若文件已被跟踪，需要先用 `git rm --cached <file>` 从跟踪中移除，再提交，之后 `.gitignore` 才会对该文件生效。

## 3.1 分支管理

### 🔹 查看分支

```bash
# 查看本地分支
git branch

# 查看本地与远程分支
git branch -a

# 查看分支详细信息（最近提交）
git branch -v

# 查看已合并到当前分支的分支
git branch --merged

# 查看未合并到当前分支的分支
git branch --no-merged
```

### 🔹 创建与切换分支

```bash
# 创建分支
git branch feature/login

# 切换分支（传统方式）
git checkout feature/login

# 切换分支（新方式，Git 2.23+）
git switch feature/login

# 创建并切换分支（传统方式）
git checkout -b feature/register

# 创建并切换分支（新方式）
git switch -c feature/register

# 基于指定提交创建分支
git branch hotfix/bug abc1234
```

### 🔹 删除分支

```bash
# 删除已合并的分支
git branch -d feature/login

# 强制删除未合并的分支
git branch -D feature/old

# 删除远程分支
git push origin --delete feature/login
```

------

## 3.2 合并操作

### 🔹 基本合并

**`git merge`**：将指定分支合并到当前分支。

```bash
# 切换到目标分支
git checkout main

# 合并 feature 分支
git merge feature/login
```

### 🔹 快进合并 vs 三方合并

| 合并类型 | 触发条件 | 结果 | 特点 |
|----------|----------|------|------|
| **快进合并（Fast-forward）** | 当前分支无新提交 | 直接移动指针 | 不产生合并提交，历史线性 |
| **三方合并（Three-way merge）** | 当前分支有新提交 | 产生新的合并提交 | 保留分支历史，形成分支结构 |

```
快进合并前：           快进合并后：
A --- B (main)         A --- B --- C --- D (main, feature)
       \
        C --- D (feature)

三方合并前：           三方合并后：
A --- B --- E (main)   A --- B --- E --- M (main)
       \                       \   /
        C --- D (feature)       C --- D (feature)
                              M 是合并提交
```

### 🔹 合并选项

```bash
# 强制产生合并提交（即使可以快进）
git merge --no-ff feature/login

# 压缩合并（将目标分支的多次提交压缩为一次）
git merge --squash feature/login

# 中止合并
git merge --abort
```

## 3.3 变基操作

### 🔹 基本变基

**`git rebase`**：将当前分支的提交"挪到"目标分支之后，历史呈直线。

```bash
# 切换到 feature 分支
git checkout feature/login

# 基于 main 分支变基
git rebase main
```

**变基原理**：

```
变基前：                    变基后：
A --- B --- C (main)        A --- B --- C (main)
       \                          \
        D --- E (feature)          D' --- E' (feature)
                                     （重新应用的提交）
```

### 🔹 交互式变基

**`git rebase -i`**：可合并、重排、修改、删除提交。

```bash
# 交互式变基最近 5 次提交
git rebase -i HEAD~5

# 交互式变基到指定提交
git rebase -i abc1234
```

**交互式变基命令**：

| 命令 | 缩写 | 作用 |
|------|------|------|
| `pick` | `p` | 保留提交 |
| `reword` | `r` | 保留提交，修改提交信息 |
| `squash` | `s` | 合并到上一个提交，保留提交信息 |
| `fixup` | `f` | 合并到上一个提交，丢弃提交信息 |
| `drop` | `d` | 删除提交 |
| `edit` | `e` | 保留提交，暂停修改 |

### 🔹 变基的注意事项

> ⚠️ **黄金法则**：
>  **不要对已推送到远程的公共分支做 rebase**，会改写他人依赖的历史，造成协作混乱。
>  仅对未推送的个人分支使用 rebase。

## 3.4 冲突解决

### 🔹 冲突产生

merge 或 rebase 时，若两方修改了同一文件的同一区域，Git 无法自动合并，会标记冲突。

### 🔹 冲突标记

```java
public class UserService {
<<<<<<< HEAD
    // main 分支的代码
    public void login() {
        System.out.println("Login v1");
    }
=======
    // feature 分支的代码
    public void login() {
        System.out.println("Login v2");
    }
>>>>>>> feature/login
}
```

### 🔹 解决流程

1. **打开冲突文件**：找到 `<<<<<<<`、`=======`、`>>>>>>>` 标记
2. **手动编辑**：保留需要的版本或合并两者，删除冲突标记
3. **标记已解决**：`git add <file>`
4. **完成操作**：
   - merge 时：`git commit` 完成合并
   - rebase 时：`git rebase --continue` 继续，或 `git rebase --abort` 放弃

```bash
# 解决冲突后标记已解决
git add conflicted-file.txt

# merge 时完成合并
git commit

# rebase 时继续
git rebase --continue

# 放弃 rebase
git rebase --abort
```

## 3.5 分支策略

### 🔹 Git Flow

**Git Flow**：经典的分支管理模型，适合有固定发布周期的项目。

| 分支类型 | 命名 | 用途 | 生命周期 |
|----------|------|------|----------|
| **主分支** | `main` / `master` | 生产环境代码 | 长期 |
| **开发分支** | `develop` | 日常开发集成 | 长期 |
| **功能分支** | `feature/*` | 新功能开发 | 临时 |
| **发布分支** | `release/*` | 版本发布准备 | 临时 |
| **热修复分支** | `hotfix/*` | 生产环境紧急修复 | 临时 |

```
main ─────●────────●────────●────────●────────●─────
           \       /         \       /         \
            ●─────●           ●─────●           ●
           release/1.0       hotfix/1.1        release/1.1

develop ───●────●────●────●────●────●────●────●────
                \    /         \    /
                 ●──●           ●──●
               feature/a      feature/b
```

### 🔹 GitHub Flow

**GitHub Flow**：简化模型，适合持续部署的项目。

- `main` 分支始终可部署
- 功能在 `feature/*` 分支开发
- 通过 Pull Request 合并
- 合并后立即部署

```
main ─────●────────●────────●────────●────────●─────
           \       /         \       /         \
            ●─────●           ●─────●           ●
          feature/1         feature/2         feature/3
```

### 🔹 简单分支策略

**小团队常用**：`main` + `feature/xxx`

| 场景 | 分支命名 |
|------|----------|
| 新功能 | `feature/user-login` |
| Bug 修复 | `fix/login-error` |
| 文档更新 | `docs/api-guide` |
| 性能优化 | `perf/query-optimize` |

------

## 3.6 面试高频题

### Q1：merge 与 rebase 的区别是什么？

> **标准回答**：
> 1. **历史记录**：merge 保留分支历史，产生合并提交；rebase 改写历史，呈线性
> 2. **使用场景**：merge 适合公共分支；rebase 适合个人分支整理提交
> 3. **冲突处理**：merge 解决一次冲突；rebase 可能需要解决多次冲突
> 4. **安全性**：merge 不改变提交历史，更安全；rebase 改写历史，对已推送的分支慎用

### Q2：什么是快进合并（Fast-forward）？

> **标准回答**：
> 快进合并是指当前分支没有产生新提交时，直接将分支指针移动到目标分支的最新提交，不产生新的合并提交。这种合并方式历史记录呈线性，但会丢失分支信息。

### Q3：如何解决 Git 冲突？

> **标准回答**：
> 1. 执行 `git status` 查看冲突文件
> 2. 打开冲突文件，找到 `<<<<<<<`、`=======`、`>>>>>>>` 标记
> 3. 手动编辑，保留需要的代码，删除冲突标记
> 4. `git add <file>` 标记冲突已解决
> 5. `git commit`（merge）或 `git rebase --continue`（rebase）完成操作

### Q4：Git Flow 和 GitHub Flow 的区别？

> **标准回答**：
> - **Git Flow**：多分支模型（main、develop、feature、release、hotfix），适合有固定发布周期的项目
> - **GitHub Flow**：简化模型（main + feature），通过 PR 合并，适合持续部署的项目

| 操作 | 命令 | 注意事项 |
|------|------|----------|
| 创建分支 | `git branch`、`git switch -c` | - |
| 切换分支 | `git checkout`、`git switch` | 新推荐 switch |
| 删除分支 | `git branch -d`、`-D` | `-D` 强制删除 |
| 合并 | `git merge`、`--no-ff` | 公共分支用 merge |
| 变基 | `git rebase`、`-i` | 个人分支用 rebase |
| 冲突解决 | 手动编辑 + `git add` | 注意冲突标记 |

## 4.1 远程仓库配置

### 🔹 查看远程仓库

```bash
# 查看远程仓库名称和 URL
git remote -v

# 查看远程仓库详细信息
git remote show origin
```

### 🔹 添加与修改远程仓库

```bash
# 添加远程仓库
git remote add origin https://github.com/username/repository.git

# 修改远程仓库 URL
git remote set-url origin https://github.com/username/repository.git

# 重命名远程仓库
git remote rename origin upstream

# 移除远程仓库
git remote remove origin
```

### 🔹 多远程仓库配置

Fork 工作流中常见的多远程配置：

```bash
# 添加自己的 fork
git remote add origin https://github.com/your-name/repository.git

# 添加上游仓库
git remote add upstream https://github.com/original/repository.git

# 查看配置
git remote -v
# origin  https://github.com/your-name/repository.git (fetch)
# origin  https://github.com/your-name/repository.git (push)
# upstream  https://github.com/original/repository.git (fetch)
# upstream  https://github.com/original/repository.git (push)
```

------

## 4.2 推送与拉取

### 🔹 推送（Push）

```bash
# 推送本地分支到远程
git push origin main

# 首次推送并设置上游（upstream）
git push -u origin feature/login

# 设置上游后，后续可直接推送
git push

# 强制推送（慎用）
git push --force
# 或更安全的强制推送
git push --force-with-lease
```

### 🔹 拉取（Fetch & Pull）

| 命令 | 作用 |
|------|------|
| `git fetch` | 从远程获取最新提交，不合并 |
| `git pull` | `fetch` + `merge`，拉取并合并 |
| `git pull --rebase` | `fetch` + `rebase`，拉取并变基 |

```bash
# 仅获取远程更新（不合并）
git fetch origin

# 查看远程分支状态
git log HEAD..origin/main --oneline

# 拉取并合并
git pull

# 拉取并变基（推荐，保持线性历史）
git pull --rebase

# 拉取指定分支
git pull origin feature/login
```

### 🔹 Fetch 与 Pull 的区别

```
git fetch:                    git pull:
远程: A---B---C               远程: A---B---C
本地: A---B                   本地: A---B---C (merge)
     ↓ fetch                       ↓
本地: A---B                   或 A---B---M (merge commit)
     origin/main = C              C (rebase)
```

## 4.3 跟踪分支与上游

### 🔹 跟踪分支（Tracking Branch）

本地分支可设置上游（upstream）跟踪远程分支，设置后 `git push`、`git pull` 可省略分支名。

```bash
# 查看分支跟踪关系
git branch -vv

# 设置上游
git branch --set-upstream-to=origin/main main

# 或简写
git branch -u origin/main

# 首次推送时设置上游
git push -u origin feature/login
```

### 🔹 上游分支关系

```
本地分支: feature/login  ──────跟踪──────►  远程分支: origin/feature/login
                ↑                                  ↑
                └──────── push/pull ───────────────┘
```

## 4.4 推送被拒绝的处理

### 🔹 原因分析

推送被拒绝通常是因为远程分支已有他人推送的新提交，你的本地历史与远程分叉。

```
远程: A---B---C---D (origin/main)
本地: A---B---E---F (main)
             ↘   ↗ 分叉
```

### 🔹 解决方案

```bash
# 方案 1：拉取并合并（产生合并提交）
git pull
git push

# 方案 2：拉取并变基（推荐，保持线性历史）
git pull --rebase
git push

# 方案 3：仅自己使用分支时，强制推送（慎用）
git push --force
```

> ⚠️ **警告**：`git push --force` 会覆盖远程历史，仅限个人分支或团队明确允许时使用。

## 4.5 Pull Request / Merge Request

### 🔹 概念

**PR（GitHub）/ MR（GitLab）**：从 fork 或分支发起请求，将更改合并到目标分支。Code Review、CI 检查常在此完成。

### 🔹 参与开源流程（Fork 工作流）

```
┌─────────────┐     fork      ┌─────────────┐
│  上游仓库    │ ────────────► │  你的 fork   │
│  original   │               │   origin    │
└──────┬──────┘               └──────┬──────┘
       │                              │
       │        PR/MR                 │
       │◄─────────────────────────────┤
       │                              │
       │    clone、修改、push          │
       └──────────────────────────────┘
```

**具体步骤**：

1. **Fork**：在 GitHub 上 fork 目标仓库到你的账号
2. **Clone**：克隆你的 fork 到本地
   ```bash
   git clone https://github.com/your-name/repository.git
   ```
3. **添加上游**：添加上游仓库地址
   ```bash
   git remote add upstream https://github.com/original/repository.git
   ```
4. **创建分支**：基于 main 创建功能分支
   ```bash
   git checkout -b feature/xxx
   ```
5. **修改提交**：进行修改并提交
   ```bash
   git add .
   git commit -m "feat: xxx"
   ```
6. **推送**：推送到你的 fork
   ```bash
   git push origin feature/xxx
   ```
7. **发起 PR**：在 GitHub 上点击 "New Pull Request"
8. **同步上游**：定期同步上游更新
   ```bash
   git fetch upstream
   git checkout main
   git merge upstream/main
   ```

## 4.6 面试高频题

### Q1：`git fetch` 与 `git pull` 的区别？

> **标准回答**：
> - `git fetch`：从远程获取最新提交，更新远程分支引用，但不合并到当前分支
> - `git pull`：`git fetch` + `git merge`，获取并合并到当前分支
> - `git pull --rebase`：`git fetch` + `git rebase`，获取并变基

### Q2：推送被拒绝怎么办？

> **标准回答**：
> 1. 先执行 `git pull --rebase` 或 `git pull` 同步远程变更
> 2. 如有冲突，解决冲突后 `git add` 和 `git rebase --continue`
> 3. 然后 `git push` 推送
> 4. 仅在个人分支且确定无他人使用时，可考虑 `git push --force`

### Q3：Fork 工作流的完整流程是什么？

> **标准回答**：
> 1. 在 GitHub 上 fork 目标仓库
> 2. 克隆自己的 fork 到本地
> 3. 添加上游仓库 remote
> 4. 创建功能分支进行开发
> 5. 提交修改并推送到自己的 fork
> 6. 在 GitHub 上发起 Pull Request
> 7. 定期同步上游仓库更新

# 第5章：撤销与恢复

------

## 5.1 工作区与暂存区撤销

### 🔹 丢弃工作区修改

**`git restore`**（Git 2.23+）或 **`git checkout --`**：丢弃工作区的修改。

```bash
# 丢弃单个文件的修改
git restore README.md

# 丢弃所有修改
git restore .

# 旧命令（仍可用）
git checkout -- README.md
```

> ⚠️ **警告**：此操作不可恢复，修改将永久丢失。

### 🔹 取消暂存

**`git restore --staged`** 或 **`git reset HEAD`**：将文件从暂存区移除，保留工作区修改。

```bash
# 取消暂存单个文件
git restore --staged README.md

# 取消暂存所有文件
git restore --staged .

# 旧命令（仍可用）
git reset HEAD README.md
```

### 🔹 撤销操作对比

| 场景 | 命令 | 工作区 | 暂存区 |
|------|------|--------|--------|
| 丢弃工作区修改 | `git restore <file>` | 重置 | 不变 |
| 取消暂存 | `git restore --staged <file>` | 保留 | 清空 |
| 同时取消暂存和丢弃修改 | `git restore --staged <file>` + `git restore <file>` | 重置 | 清空 |

------

## 5.2 提交回退（Reset）

### 🔹 reset 的三种模式

| 模式 | 命令 | HEAD | 暂存区 | 工作区 | 适用场景 |
|------|------|------|--------|--------|----------|
| **--soft** | `git reset --soft HEAD~1` | 移动 | 保留 | 保留 | 撤销提交但保留修改 |
| **--mixed** | `git reset --mixed HEAD~1` | 移动 | 清空 | 保留 | 默认模式，重置暂存区 |
| **--hard** | `git reset --hard HEAD~1` | 移动 | 清空 | 清空 | 完全放弃后续修改 |

### 🔹 使用示例

```bash
# 软重置：撤销最近提交，修改保留在暂存区
git reset --soft HEAD~1

# 混合重置：撤销最近提交，修改保留在工作区（默认）
git reset --mixed HEAD~1
# 或简写
git reset HEAD~1

# 硬重置：彻底回到某提交，丢弃所有修改
git reset --hard HEAD~1

# 回到指定提交
git reset --hard abc1234
```

### 🔹 reset 原理

```
--soft:                      --mixed:                    --hard:
A---B---C (HEAD)             A---B---C (HEAD)            A---B---C (HEAD)
    ↑                            ↑                           ↑
  工作区: C修改                 工作区: C修改                工作区: B状态
  暂存区: C修改                 暂存区: 空                   暂存区: 空
```

> ⚠️ **警告**：`git reset --hard` 会丢弃未提交的修改，使用前请确保已备份重要更改。

------

## 5.3 恢复文件版本

### 🔹 从某提交恢复文件

**`git restore --source`** 或 **`git checkout`**：将指定提交中的文件覆盖到工作区。

```bash
# 从指定提交恢复文件
git restore --source=HEAD~2 README.md

# 从指定分支恢复文件
git restore --source=main config.ini

# 旧命令（仍可用）
git checkout HEAD~2 -- README.md
```

### 🔹 恢复已删除的文件

```bash
# 查看文件历史
git log --diff-filter=D --summary | grep delete

# 恢复文件
git restore --source=HEAD~1 deleted-file.txt
```

------

## 5.4 已推送提交的撤销

### 🔹 推荐：revert

**`git revert`**：生成一次"反向"提交，抵消指定提交的改动。不改写历史，适合已推送的提交。

```bash
# 撤销指定提交
git revert abc1234

# 撤销多个提交
git revert abc1234 def5678

# 撤销但不自动提交（手动编辑）
git revert -n abc1234
```

**revert 原理**：

```
原历史:  A---B---C---D (main)
revert C: A---B---C---D---C' (main)
                  ↑         ↑
                 C提交      C'是C的反向提交
```

### 🔹 慎用：reset + force push

**`git reset --hard`** + **`git push --force`**：改写历史，仅限个人分支使用。

```bash
# 重置到指定提交
git reset --hard abc1234

# 强制推送
git push --force
```

### 🔹 revert vs reset 对比

| 方式 | 是否改写历史 | 适用场景 | 安全性 |
|------|-------------|----------|--------|
| **revert** | 否 | 已推送的公共分支 | 高 |
| **reset + force push** | 是 | 个人分支、未推送 | 低 |

------

## 5.5 reflog 与误操作恢复

### 🔹 `git reflog`

**`git reflog`**：记录 HEAD 的移动历史，包括 reset、checkout、merge 等操作。

```bash
# 查看 reflog
git reflog

# 查看指定时间之前的记录
git reflog --before="2024-01-01"
```

**reflog 输出示例**：

```
abc1234 HEAD@{0}: commit: feat: 新增功能
def5678 HEAD@{1}: checkout: moving from main to feature/login
ghi9012 HEAD@{2}: reset: moving to HEAD~1
```

### 🔹 恢复误删分支

```bash
# 1. 查看 reflog 找到分支最后所在提交
git reflog

# 2. 重建分支
git checkout -b feature/login abc1234
```

### 🔹 恢复误 reset

```bash
# 1. 查看 reflog 找到 reset 前的提交
git reflog
# 输出: abc1234 HEAD@{1}: commit: feat: 新增功能

# 2. 恢复到 reset 前
git reset --hard HEAD@{1}
# 或
git reset --hard abc1234
```

### 🔹 恢复误删的提交

```bash
# 查看所有 reflog（包括已删除的）
git reflog show --all

# 基于 reflog 中的提交创建分支
git checkout -b recovery-branch abc1234
```

------

## 5.6 面试高频题

### Q1：`git reset` 与 `git revert` 的区别？

> **标准回答**：
> - `git reset`：移动 HEAD 指针，可改写历史。`--soft` 保留修改，`--hard` 丢弃修改。
> - `git revert`：生成新的反向提交，抵消指定提交的改动，不改写历史。
> - 使用场景：已推送的提交用 `revert`；个人分支未推送的用 `reset`。

### Q2：`git reset` 的三种模式有什么区别？

> **标准回答**：
> - `--soft`：只移动 HEAD，暂存区和工作区保留
> - `--mixed`（默认）：移动 HEAD，清空暂存区，工作区保留
> - `--hard`：移动 HEAD，清空暂存区，工作区也重置

### Q3：如何恢复误删的分支？

> **标准回答**：
> 1. 使用 `git reflog` 查看 HEAD 移动历史
> 2. 找到分支最后所在的提交哈希
> 3. 执行 `git checkout -b <branch-name> <commit-hash>` 重建分支

### Q4：`git reflog` 是什么？有什么作用？

> **标准回答**：
> `git reflog` 记录了 HEAD 和分支引用的移动历史，包括 commit、reset、checkout、merge 等操作。即使提交被 reset 或分支被删除，短期内仍可通过 reflog 找到并恢复。

| 丢弃工作区修改 | `git restore <file>` | 不可恢复 |
| 取消暂存 | `git restore --staged <file>` | 保留工作区修改 |
| 撤销提交（保留修改） | `git reset --soft HEAD~1` | 个人分支 |
| 撤销提交（丢弃修改） | `git reset --hard HEAD~1` | 慎用 |
| 撤销已推送提交 | `git revert <commit>` | 推荐 |

## 6.1 标签（Tag）

### 🔹 标签类型

| 类型 | 命令 | 特点 | 适用场景 |
|------|------|------|----------|
| **轻量标签** | `git tag <name>` | 仅是指向提交的指针 | 临时标记 |
| **附注标签** | `git tag -a <name> -m "msg"` | 包含打标签者、时间、说明 | 正式发布 |

### 🔹 标签操作

```bash
# 创建轻量标签
git tag v1.0.0

# 创建附注标签
git tag -a v1.0.0 -m "版本 1.0.0 发布"

# 查看标签列表
git tag

# 查看标签列表（按版本排序）
git tag -l "v1.*"

# 查看标签详情
git show v1.0.0

# 推送标签到远程
git push origin v1.0.0

# 推送所有标签
git push origin --tags

# 删除本地标签
git tag -d v1.0.0

# 删除远程标签
git push origin --delete v1.0.0

# 检出标签（创建分支）
git checkout -b release-1.0 v1.0.0
```

### 🔹 标签与分支的区别

| 特性 | 标签（Tag） | 分支（Branch） |
|------|-------------|----------------|
| **用途** | 标记特定版本 | 持续开发 |
| **移动** | 固定指向某提交 | 随新提交移动 |
| **命名** | 通常 v1.0.0 | feature/xxx |
| **生命周期** | 长期保留 | 合并后可删除 |

## 6.2 贮藏（Stash）

### 🔹 基本用法

**`git stash`**：将工作区与暂存区改动暂存，清理工作区。

```bash
# 暂存当前改动
git stash

# 带说明暂存
git stash push -m "修复登录功能，待测试"

# 查看贮藏列表
git stash list

# 应用最近的贮藏（不删除）
git stash apply

# 应用指定的贮藏
git stash apply stash@{1}

# 应用并删除最近的贮藏
git stash pop

# 删除最近的贮藏
git stash drop

# 删除所有贮藏
git stash clear

# 查看贮藏内容
git stash show -p
```

### 🔹 适用场景

| 场景 | 说明 |
|------|------|
| **临时切分支** | 当前工作未完成，需要紧急切换到其他分支 |
| **拉取前保存** | 拉取远程更新前保存本地修改 |
| **代码审查** | 临时保存当前工作，审查他人代码 |
| **多任务切换** | 在不同任务间快速切换 |

### 🔹 贮藏原理

```
工作区修改 ──► git stash ──► 贮藏堆栈
                                  │
                                  ▼
                         stash@{0}: WIP on main
                         stash@{1}: WIP on feature
```

------

## 6.3 Cherry-pick

### 🔹 基本用法

**`git cherry-pick`**：将指定提交"摘取"并应用到当前分支。

```bash
# 摘取单个提交
git cherry-pick abc1234

# 摘取多个提交
git cherry-pick abc1234 def5678

# 摘取一系列提交
git cherry-pick abc1234^..def5678

# 摘取但不自动提交
git cherry-pick -n abc1234
```

### 🔹 应用场景

| 场景 | 说明 |
|------|------|
| **移植修复** | 将某分支的 bug 修复应用到主分支 |
| **选择性合并** | 只合并某分支的特定提交，而非整个分支 |
| **整理提交** | 从不同分支挑选提交整合到新分支 |

### 🔹 cherry-pick vs merge

| 方式 | 结果 | 适用场景 |
|------|------|----------|
| **merge** | 合并整个分支历史 | 整合功能分支 |
| **cherry-pick** | 只合并指定提交 | 选择性移植 |

------

## 6.4 Bisect（二分查找）

### 🔹 基本用法

**`git bisect`**：使用二分查找定位引入 bug 的提交。

```bash
# 开始二分查找
git bisect start

# 标记当前提交有 bug（坏）
git bisect bad

# 标记某提交无 bug（好）
git bisect good v1.0.0

# Git 自动切换到中间提交，测试后标记
git bisect good
# 或
git bisect bad

# 重复直到找到引入 bug 的提交

# 结束二分查找
git bisect reset
```

### 🔹 自动化 bisect

```bash
# 使用脚本自动测试
git bisect run ./test-script.sh
```

### 🔹 适用场景

- 不确定哪次提交引入问题
- 需要快速定位 bug 源头
- 提交历史较长，手动查找效率低

## 6.5 子模块（Submodule）

### 🔹 基本用法

**`git submodule`**：在一个 Git 仓库中嵌套另一个 Git 仓库。

```bash
# 添加子模块
git submodule add https://github.com/username/library.git libs/library

# 初始化子模块
git submodule init

# 更新子模块
git submodule update

# 初始化并更新（克隆后执行）
git submodule update --init --recursive

# 克隆包含子模块的仓库
git clone --recurse-submodules https://github.com/username/project.git
```

### 🔹 子模块工作流

```bash
# 进入子模块目录
cd libs/library

# 在子模块中操作（相当于独立仓库）
git checkout v2.0.0
git pull

# 返回主项目
cd ../..

# 提交子模块变更
git add libs/library
git commit -m "更新子模块到 v2.0.0"
```

### 🔹 适用场景

| 场景 | 说明 |
|------|------|
| **引用第三方库** | 公共组件、工具库等 |
| **内部公共模块** | 多个项目共享的代码 |
| **大项目拆分** | 将大项目拆分为多个子模块 |

## 6.6 面试高频题

### Q1：标签和分支有什么区别？

> **标准回答**：
> - **标签**：固定指向某提交，用于标记版本（如 v1.0.0），不会移动
> - **分支**：随新提交移动，用于持续开发
> - **生命周期**：标签长期保留，分支合并后可删除

### Q2：`git stash` 的适用场景是什么？

> **标准回答**：
> - 临时切换到其他分支处理紧急任务
> - 拉取远程更新前保存本地未完成的修改
> - 在不同任务间快速切换上下文

### Q3：`git cherry-pick` 和 `git merge` 的区别？

> **标准回答**：
> - `git merge`：合并整个分支的所有提交
> - `git cherry-pick`：只选择性地合并指定的提交
> - 使用场景：需要移植某个特定修复或功能时使用 cherry-pick

### Q4：什么是 `git bisect`？

> **标准回答**：
> `git bisect` 是 Git 提供的二分查找工具，用于快速定位引入 bug 的提交。通过标记"好"和"坏"的提交，Git 自动切换到中间提交，反复测试后缩小范围，最终找到引入问题的具体提交。

## 7.1 配置管理

### 🔹 配置层级

Git 配置有三个层级，优先级由低到高：

| 层级 | 配置文件路径 | 命令 | 优先级 |
|------|-------------|------|--------|
| **系统** | `/etc/gitconfig` | `git config --system` | 低 |
| **全局** | `~/.gitconfig` | `git config --global` | 中 |
| **仓库** | `.git/config` | `git config --local` | 高 |

### 🔹 常用配置

```bash
# 设置全局用户名和邮箱（必须）
git config --global user.name "Your Name"
git config --global user.email "your.email@example.com"

# 设置默认编辑器
git config --global core.editor "vim"

# 设置默认分支名为 main
git config --global init.defaultBranch main

# 设置别名
git config --global alias.st status
git config --global alias.co checkout
git config --global alias.br branch
git config --global alias.ci commit
git config --global alias.lg "log --oneline --graph --all"

# 设置换行符（Windows）
git config --global core.autocrlf true

# 设置换行符（Mac/Linux）
git config --global core.autocrlf input

# 设置凭据缓存（避免重复输入密码）
git config --global credential.helper cache

# 设置凭据存储（永久保存）
git config --global credential.helper store
```

### 🔹 查看配置

```bash
# 查看所有配置
git config --list

# 查看全局配置
git config --global --list

# 查看仓库配置
git config --local --list

# 查看特定配置项
git config user.name
```

### 🔹 常用别名配置

```bash
# 添加到 ~/.gitconfig
git config --global alias.st status
git config --global alias.co checkout
git config --global alias.br branch
git config --global alias.ci commit
git config --global alias.lg "log --color --graph --pretty=format:'%Cred%h%Creset -%C(yellow)%d%Creset %s %Cgreen(%cr) %C(bold blue)<%an>%Creset' --abbrev-commit"
```

## 7.2 钩子（Hooks）

### 🔹 钩子简介

Git 钩子是在特定事件发生时自动执行的脚本，位于 `.git/hooks/` 目录。

### 🔹 常用钩子

| 钩子 | 触发时机 | 用途 |
|------|----------|------|
| `pre-commit` | 提交前 | 代码检查、格式化 |
| `prepare-commit-msg` | 打开提交信息编辑器前 | 修改默认提交信息 |
| `commit-msg` | 提交信息编辑后 | 验证提交信息格式 |
| `post-commit` | 提交后 | 发送通知 |
| `pre-push` | 推送前 | 运行测试 |
| `pre-receive` | 服务器接收推送前 | 服务器端验证 |

### 🔹 客户端钩子示例

**`.git/hooks/pre-commit`**：

```bash
#!/bin/bash
# 运行代码检查
echo "Running lint..."
npm run lint

if [ $? -ne 0 ]; then
    echo "Lint failed, commit aborted"
    exit 1
fi
```

### 🔹 Husky + lint-staged（团队实践）

```bash
# 安装
npm install --save-dev husky lint-staged

# 初始化 husky
npx husky install

# 添加 pre-commit 钩子
npx husky add .husky/pre-commit "npx lint-staged"
```

**`package.json` 配置**：

```json
{
  "husky": {
    "hooks": {
      "pre-commit": "lint-staged",
      "commit-msg": "commitlint -E HUSKY_GIT_PARAMS"
    }
  },
  "lint-staged": {
    "*.js": ["eslint --fix", "git add"],
    "*.ts": ["eslint --fix", "git add"]
  }
}
```

## 7.3 最佳实践

### 🔹 提交信息规范

**约定式提交（Conventional Commits）**：

```
<type>(<scope>): <subject>

<body>

<footer>
```

**类型（type）**：

| 类型 | 说明 |
|------|------|
| `feat` | 新功能 |
| `fix` | 修复 bug |
| `docs` | 文档更新 |
| `style` | 代码格式调整（不影响功能） |
| `refactor` | 代码重构 |
| `perf` | 性能优化 |
| `test` | 测试相关 |
| `chore` | 构建/依赖/工具相关 |

**示例**：

```
feat(user): 新增用户登录功能

- 实现 JWT Token 认证
- 添加登录接口 /api/login
- 集成 Redis 存储 Token

Closes #123
```

### 🔹 提交粒度

**小步提交原则**：
- 每次提交只做一件事
- 保持提交的原子性
- 便于回滚和代码审查

```bash
# ❌ 不好的提交
git commit -m "更新"

# ✅ 好的提交
git commit -m "feat(auth): 新增用户登录接口"
git commit -m "fix(auth): 修复登录验证逻辑"
git commit -m "docs(api): 更新登录接口文档"
```

### 🔹 分支命名规范

| 类型 | 命名格式 | 示例 |
|------|----------|------|
| 功能 | `feature/<功能描述>` | `feature/user-login` |
| Bug 修复 | `fix/<问题描述>` | `fix/login-error` |
| 文档 | `docs/<文档描述>` | `docs/api-guide` |
| 性能优化 | `perf/<优化描述>` | `perf/query-optimize` |
| 重构 | `refactor/<重构描述>` | `refactor/user-service` |
| 发布 | `release/<版本号>` | `release/v1.0.0` |
| 热修复 | `hotfix/<问题描述>` | `hotfix/critical-bug` |

### 🔹 主分支保护

- **不直接在主分支开发**：所有修改通过 PR/MR 合并
- **强制代码审查**：至少 1-2 人审查通过
- **CI 检查通过**：自动化测试、代码检查通过
- **禁止强制推送**：保护分支历史

### 🔹 大文件与敏感信息处理

**大文件**：
- 使用 Git LFS（Large File Storage）管理大文件
- 或在 `.gitignore` 中忽略

```bash
# 安装 Git LFS
git lfs install

# 追踪大文件
git lfs track "*.psd"
git lfs track "*.zip"
```

**敏感信息**：
- 使用环境变量或配置文件模板
- 将敏感文件加入 `.gitignore`
- 已提交敏感信息需从历史中移除

```bash
# 从历史中移除敏感文件
git filter-branch --force --index-filter \
'git rm --cached --ignore-unmatch config.ini' \
HEAD
```

### 🔹 Git 工作流总结

```
┌─────────────────────────────────────────────────────────────┐
│                     日常开发流程                             │
├─────────────────────────────────────────────────────────────┤
│  1. 更新代码: git pull --rebase                             │
│  2. 创建分支: git checkout -b feature/xxx                   │
│  3. 开发修改: 编辑文件                                       │
│  4. 查看状态: git status                                     │
│  5. 暂存修改: git add .                                      │
│  6. 提交修改: git commit -m "feat: xxx"                     │
│  7. 推送分支: git push -u origin feature/xxx                │
│  8. 发起 PR: 在 GitHub/GitLab 上创建 Pull Request           │
│  9. 代码审查: 团队成员审查代码                               │
│ 10. 合并分支: 审查通过后合并到 main                         │
│ 11. 删除分支: git branch -d feature/xxx                     │
└─────────────────────────────────────────────────────────────┘
```

## 7.4 面试高频题

### Q1：Git 配置的三个层级是什么？优先级如何？

> **标准回答**：
> 1. **系统级**（`--system`）：`/etc/gitconfig`，对所有用户生效
> 2. **全局级**（`--global`）：`~/.gitconfig`，对当前用户生效
> 3. **仓库级**（`--local`）：`.git/config`，对当前仓库生效
> 优先级：仓库级 > 全局级 > 系统级

### Q2：什么是 Git 钩子？常见的有哪些？

> **标准回答**：
> Git 钩子是在特定事件发生时自动执行的脚本。常见的客户端钩子：
> - `pre-commit`：提交前执行，用于代码检查
> - `commit-msg`：提交信息编辑后执行，用于验证格式
> - `pre-push`：推送前执行，用于运行测试

### Q3：什么是约定式提交（Conventional Commits）？

> **标准回答**：
> 约定式提交是一种提交信息规范，格式为 `<type>(<scope>): <subject>`。
> 常见 type：feat（新功能）、fix（修复）、docs（文档）、style（格式）、refactor（重构）、test（测试）、chore（构建）。
> 优点：便于生成 CHANGELOG、自动化版本号、代码审查。

### Q4：如何保护主分支？

> **标准回答**：
> 1. 不直接在主分支开发，通过 PR/MR 合并代码
> 2. 强制代码审查，至少 1-2 人通过
> 3. CI 检查通过（自动化测试、代码检查）
> 4. 禁止强制推送（force push）
> 5. 开启分支保护规则

## 核心概念

| 考点 | 回答要点 |
|------|----------|
| **工作区、暂存区、仓库** | 工作区 → `git add` → 暂存区 → `git commit` → 本地仓库 → `git push` → 远程仓库 |
| **Git 与 SVN 区别** | 分布式 vs 集中式；分支轻量 vs 分支复杂；支持离线 vs 依赖服务器 |
| **分支本质** | 指向提交的指针，创建成本极低 |

## 常用操作

| 考点 | 回答要点 |
|------|----------|
| **merge vs rebase** | merge 保留历史、产生合并提交；rebase 历史线性、改写提交；公共分支用 merge，个人分支用 rebase |
| **fetch vs pull** | fetch 只获取不合并；pull = fetch + merge；pull --rebase = fetch + rebase |
| **reset vs revert** | reset 移动 HEAD、改写历史；revert 生成反向提交、不改写历史；已推送用 revert |
| **reset 三种模式** | --soft 保留暂存区和工作区；--mixed 清空暂存区（默认）；--hard 全部清空 |

## 协作流程

| 考点 | 回答要点 |
|------|----------|
| **冲突解决** | 编辑冲突文件 → 删除冲突标记 → `git add` → `git commit` 或 `git rebase --continue` |
| **Fork 工作流** | fork → clone → 添加 upstream → 创建分支 → 修改 → push → 发起 PR → 同步 upstream |
| **推送被拒绝** | `git pull --rebase` 同步 → 解决冲突 → `git push` |

## 进阶操作

| 考点 | 回答要点 |
|------|----------|
| **标签类型** | 轻量标签（仅指针）和附注标签（含信息）；用于版本发布标记 |
| **贮藏用途** | `git stash` 临时保存工作状态，适合临时切换分支 |
| **cherry-pick** | 选择性移植指定提交，而非合并整个分支 |
| **bisect** | 二分查找定位引入 bug 的提交 |
| **reflog** | 记录 HEAD 移动历史，用于恢复误操作 |

| **主分支保护** | PR 合并、代码审查、CI 检查、禁止 force push |
