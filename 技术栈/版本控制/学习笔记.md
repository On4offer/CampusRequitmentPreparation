# 📘《Git 学习与实战手册》

> 系统学习见下文各章；日常常用命令与使用场景速查见同目录《常用命令与使用场景》。

------

## 第1章：Git 概述

### 1.1 什么是 Git

- **Git** 是**分布式版本控制系统**，由 Linus Torvalds 创建；每个仓库都含完整历史，可离线提交、分支与合并。
- 与 SVN 等集中式对比：无单点、分支轻量、支持离线、基于内容寻址（SHA-1 哈希）；是当前主流版本控制工具，校招与协作必备。

### 1.2 核心概念

- **工作区**：当前看到的文件目录；**暂存区（索引）**：git add 后文件进入此处；**本地仓库**：.git 目录，含提交历史与分支。
- **提交（commit）**：暂存区快照形成一次提交，有唯一 SHA、作者、时间、提交信息；**分支**：指向某次提交的指针，默认 main/master。
- **远程仓库**：如 GitHub、GitLab、Gitee；通过 push/pull/fetch 与本地同步。

------

## 第2章：本地基本操作

1. **初始化与克隆**：git init 新建仓库；git clone &lt;url&gt; 克隆远程仓库（含历史）。
2. **状态与暂存**：git status 查看工作区与暂存区；git add &lt;file&gt; 或 git add . 将修改加入暂存区；git restore &lt;file&gt; 或 git checkout -- &lt;file&gt; 丢弃工作区修改。
3. **提交**：git commit -m "message" 将暂存区提交；git commit --amend 修改最近一次提交（改信息或追加文件）。
4. **查看历史**：git log、git log --oneline、git log -p、git show &lt;commit&gt;；git diff 工作区与暂存区、git diff --staged 暂存区与最近提交。
5. **忽略文件**：.gitignore 书写规则（通配符、目录、取反）；已跟踪文件需先 git rm --cached 再忽略。

------

## 第3章：分支与合并

1. **分支**：git branch 查看；git branch &lt;name&gt; 创建；git checkout &lt;name&gt; 或 git switch &lt;name&gt; 切换；git checkout -b &lt;name&gt; 或 git switch -c &lt;name&gt; 创建并切换。
2. **合并**：git merge &lt;branch&gt; 将指定分支合并到当前分支；**快进合并**（当前分支未前进则直接移动指针）与**三方合并**（产生新提交）；冲突时需手动解决后 git add 再 commit。
3. **变基**：git rebase &lt;branch&gt; 将当前分支的提交“挪到”目标分支之后，历史呈直线；git rebase -i 交互式变基（合并、重排、修改提交）；**不要对已推送的公共分支做 rebase**，避免改写他人依赖的历史。
4. **面试要点**：merge 与 rebase 区别、何时用 rebase、冲突如何解决、分支策略（如 Git Flow、GitHub Flow）。

------

## 第4章：远程协作

1. **远程仓库**：git remote -v 查看；git remote add origin &lt;url&gt; 添加；git remote set-url origin &lt;url&gt; 修改地址。
2. **推送与拉取**：git push origin &lt;branch&gt; 推送本地分支；git push -u origin &lt;branch&gt; 首次推送并设上游；git pull 相当于 fetch + merge 当前分支上游；git fetch origin 只拉取不合并。
3. **跟踪分支**：本地分支可设置上游（upstream）跟踪远程分支；push/pull 时省略分支名即用上游。
4. **拉取请求 / PR**：在 GitHub/GitLab 上从 fork 或分支发起 PR，经审查后合并到目标分支；Code Review 与 CI 常在此完成。
5. **面试要点**：push 被拒绝时（远程更新）先 pull/rebase 再 push、fetch 与 pull 区别、如何参与开源（fork → 改 → PR）。

------

## 第5章：撤销与恢复

1. **工作区**：git restore &lt;file&gt; 或 git checkout -- &lt;file&gt; 丢弃未暂存修改。
2. **暂存区**：git restore --staged &lt;file&gt; 或 git reset HEAD &lt;file&gt; 取消暂存，保留工作区修改。
3. **提交**：git reset --soft &lt;commit&gt; 只移动 HEAD，保留暂存区和工作区；git reset --mixed &lt;commit&gt; 移动 HEAD 并清暂存区，保留工作区；git reset --hard &lt;commit&gt; 彻底回到某提交（**慎用**，未提交修改会丢失）。
4. **恢复某文件**：git checkout &lt;commit&gt; -- &lt;file&gt; 或 git restore --source=&lt;commit&gt; &lt;file&gt; 从某提交取出文件到工作区。
5. **已推送的撤销**：若需“撤销”已 push 的提交，可用 git revert &lt;commit&gt; 生成反向提交（推荐，不改写历史）；或 reset 后 force push（仅个人分支或团队约定允许时）。

------

## 第6章：标签与贮藏

1. **标签**：git tag &lt;name&gt; 轻量标签；git tag -a &lt;name&gt; -m "msg" 附注标签；git push origin &lt;tagname&gt; 推送标签；用于发版标记（如 v1.0.0）。
2. **贮藏**：git stash 将工作区与暂存区改动暂存；git stash list、git stash pop、git stash apply；临时切分支或拉取前常用。
3. **面试要点**：tag 与 branch 区别、stash 适用场景。

------

## 第7章：配置、钩子与最佳实践

1. **配置**：git config --global user.name / user.email；git config --list；.gitconfig 与仓库内 .git/config；凭据存储（credential helper）。
2. **钩子**：.git/hooks/ 下 pre-commit、commit-msg、pre-push 等；可用于提交前检查（lint、测试）、提交信息规范；团队常用 Husky + lint-staged 在客户端执行。
3. **最佳实践**：提交信息规范（如约定式提交 feat/fix/docs）、小步提交、分支命名（feature/xxx、bugfix/xxx、release/xxx）、不直接在主分支开发、慎用 force push。
4. **面试要点**：如何保证提交质量（钩子、CI）、团队分支与发布流程。
