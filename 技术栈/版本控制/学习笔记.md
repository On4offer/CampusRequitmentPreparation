## 1.1 版本控制的起源与发展

### 🔹 1.1.1 什么是版本控制

> **一句话**：**版本控制（Version Control）** 是一种记录文件内容变化，以便将来查阅特定版本历史的系统。
> **为什么需要它**：多人协作时需要追踪谁改了什么、何时改的；需要随时回退到稳定版本；需要并行开发不同功能。没有版本控制，代码会一团糟，无法追溯历史、无法协作。
> **类比**：版本控制像「时光机」——可以随时回到过去的某个版本，也可以看到每次修改的详细记录。

**版本控制**帮助开发团队：

- **追踪变更历史**：谁、何时、做了什么修改
- **协同开发**：多人同时工作在同一个项目上
- **版本回退**：随时回到之前的稳定版本
- **分支管理**：并行开发不同功能或版本

### 🔹 1.1.2 版本控制系统的发展

| 阶段 | 代表工具 | 年代 | 核心特点 |
|------|----------|------|----------|
| **本地版本控制** | RCS | 1980s | 仅本地存储，无网络协作 |
| **集中式版本控制** | CVS、SVN | 1990s-2000s | 中央服务器，客户端 checkout |
| **分布式版本控制** | Git、Mercurial | 2005+ | 本地完整仓库，支持离线操作 |

### 🔹 1.1.3 Git 的诞生

- **2005年**：Linus Torvalds 为管理 Linux 内核开发而创建 Git
- **2008年**：GitHub 上线，推动 Git 普及
- **2014年**：GitLab 开源，提供自托管方案
- **当前**：Git 已成为全球最流行的版本控制工具

> ✅ Git 是当前校招与团队协作的必备技能。

## 1.2 版本控制工具对比

### 🔹 1.2.1 集中式 vs 分布式

| 对比维度 | 集中式（SVN） | 分布式（Git） |
|----------|--------------|---------------|
| **架构** | 中央服务器，客户端只保存工作副本 | 每个节点都有完整仓库 |
| **离线工作** | 不支持，必须连接服务器 | 完全支持，可离线提交 |
| **分支操作** | 分支是目录拷贝，操作慢 | 分支是指针，操作极快 |
| **存储方式** | 增量存储 | 快照存储 |
| **协作方式** | checkout/commit 到中央服务器 | push/pull 在仓库间同步 |

### 🔹 1.2.2 常见工具对比

| 工具 | 类型 | 优势 | 劣势 | 适用场景 |
|------|------|------|------|----------|
| **Git** | 分布式 | 功能强大、分支轻量、生态完善 | 学习曲线较陡 | 绝大多数项目 |
| **SVN** | 集中式 | 简单易用、权限管理完善 | 依赖中央服务器 | 文档管理、二进制文件 |
| **Mercurial** | 分布式 | 简单易用、跨平台 | 生态相对较小 | 小型项目 |

## 1.3 Git 的核心概念

### 🔹 1.3.1 工作区、暂存区、仓库

Git 管理代码的三个核心区域：

```
┌─────────────────────────────────────────────────────┐
│                    Git 工作模型                      │
├─────────────────────────────────────────────────────┤
│  工作区 (Working Directory)                          │
│  └── 你看到的文件目录（除 .git 外）                   │
│          ↓ git add                                  │
│  暂存区 (Staging Area/Index)                         │
│  └── 下次提交的预备区                                │
│          ↓ git commit                               │
│  本地仓库 (Local Repository)                         │
│  └── .git 目录，存储完整历史                         │
│          ↓ git push                                 │
│  远程仓库 (Remote Repository)                        │
│  └── GitHub/GitLab 等托管平台                        │
└─────────────────────────────────────────────────────┘
```

| 区域 | 说明 | 对应命令 |
|------|------|----------|
| **工作区** | 当前看到的文件目录 | 编辑文件 |
| **暂存区** | 下次提交的预备区 | `git add` |
| **本地仓库** | 存储完整提交历史 | `git commit` |
| **远程仓库** | 托管平台上的仓库 | `git push/pull` |

### 🔹 1.3.2 提交（Commit）

**提交**是 Git 的核心概念，代表项目在某个时间点的快照：

- 包含唯一 SHA-1 哈希标识
- 记录作者、时间、提交信息
- 指向父提交，形成有向无环图（DAG）

```
A --- B --- C --- D (main)
       \
        E --- F (feature)
```

### 🔹 1.3.3 分支（Branch）与 HEAD

| 概念 | 说明 |
|------|------|
| **分支** | 指向某次提交的指针，创建分支只是新建指针 |
| **HEAD** | 指向当前所在分支的指针，代表"当前位置" |
| **默认分支** | 通常为 `main` 或 `master` |

> 💡 **Git 分支的本质**：只是一个指向提交的指针，创建分支成本极低（仅创建指针，不复制文件）。

### 🔹 1.3.4 远程仓库（Remote）

- **origin**：默认远程仓库名称
- **upstream**：上游仓库（fork 场景）
- 常见托管平台：GitHub、GitLab、Gitee、Bitbucket

## 1.4 Git 的核心优势

### 🔹 1.4.1 五大核心优势

| 优势 | 说明 |
|------|------|
| **分布式架构** | 本地完整仓库，无单点故障 |
| **分支轻量** | 分支是指针，创建/切换极快（毫秒级） |
| **离线支持** | 提交、查看历史、分支操作均可离线完成 |
| **内容寻址** | 使用 SHA-1 哈希标识对象，保证完整性 |
| **高性能** | 处理大型项目速度快 |

### 🔹 1.4.2 适用场景

| 场景 | 说明 |
|------|------|
| ✅ 团队协作 | 多人并行开发，代码合并 |
| ✅ 版本管理 | 追踪变更，随时回退 |
| ✅ 分支开发 | 功能开发、Bug修复、版本发布 |
| ✅ 代码审查 | Pull Request / Merge Request |
| ✅ 开源贡献 | Fork、PR 流程 |

## 1.5 面试高频题

### Q1：Git 与 SVN 的主要区别是什么？

> **标准回答**：
> 1. **架构不同**：Git 是分布式，每个节点都有完整仓库；SVN 是集中式，依赖中央服务器。
> 2. **分支操作**：Git 分支是指针，操作极快；SVN 分支是目录拷贝，操作慢。
> 3. **离线支持**：Git 支持离线提交、查看历史；SVN 必须连接服务器。
> 4. **存储方式**：Git 使用快照存储；SVN 使用增量存储。

### Q2：什么是工作区、暂存区和仓库？

> **标准回答**：
> - **工作区**：当前看到的文件目录，编辑文件的地方
> - **暂存区**：`git add` 后文件进入的区域，是下次提交的预备区
> - **本地仓库**：`.git` 目录，存储完整提交历史
> - **关系**：工作区 → `git add` → 暂存区 → `git commit` → 本地仓库

### Q3：Git 的分支是如何实现的？

> **标准回答**：
> Git 分支本质上是一个指向提交的指针（40字节的 SHA-1 哈希值）。创建分支只是新建一个指针，指向当前提交，因此成本极低。切换分支则是移动 HEAD 指针到目标分支。

## ⚠️ 常见坑与注意点

| 现象 / 易错点 | 原因 | 怎么改 / 怎么记 |
|---------------|------|-----------------|
| 以为 Git 分支是复制文件 | 分支只是指针，不复制文件 | 记住：**分支=指针**，创建分支=新建指针 |
| 不理解暂存区的作用 | 跳过 git add 直接 commit | 记住：**工作区→暂存区→仓库**，git add 进暂存区 |
| 混淆 HEAD 和分支名 | HEAD 指向当前分支，分支指向提交 | 记住：**HEAD→分支→提交**，HEAD 代表"当前位置" |
| 以为 Git 必须联网才能用 | Git 是分布式，本地有完整仓库 | 记住：**Git 支持离线操作**，push/pull 才需要联网 |

## 2.1 仓库初始化与克隆

### 🔹 2.1.1 新建仓库

**`git init`**：在当前目录初始化 `.git` 目录，建立空仓库。

```bash
# 在当前目录初始化
git init

# 在指定目录初始化
git init my-project
cd my-project
```

### 🔹 2.1.2 克隆远程仓库

**`git clone <url>`**：克隆远程仓库到本地，包含完整历史。

```bash
# 克隆到当前目录（创建 repository 目录）
git clone https://github.com/username/repository.git

# 克隆到指定目录
git clone https://github.com/username/repository.git my-project

# 克隆指定分支
git clone -b develop https://github.com/username/repository.git
```

> 💡 **面试可答**：git clone 会自动完成 git init、git remote add、git fetch、git checkout 等操作，是"一站式"克隆命令。

## 2.2 状态管理与提交

### 🔹 2.2.1 查看状态

**`git status`**：显示工作区与暂存区状态。

```bash
git status
```

**状态分类**：

| 状态 | 说明 | 处理方式 |
|------|------|----------|
| **Untracked** | 未跟踪的新文件 | `git add` 加入跟踪 |
| **Modified** | 已跟踪文件的修改 | `git add` 加入暂存区 |
| **Staged** | 已加入暂存区 | `git commit` 提交 |
| **Unmodified** | 未修改 | 无需处理 |

### 🔹 2.2.2 暂存修改

**`git add`**：将修改加入暂存区。

```bash
# 暂存单个文件
git add README.md

# 暂存所有修改（包括新文件和修改）
git add .

# 暂存所有修改（不包括新文件）
git add -u

# 交互式选择部分修改（hunk）
git add -p
```

### 🔹 2.2.3 提交修改

**`git commit`**：将暂存区内容提交到本地仓库。

```bash
# 提交暂存区（带提交信息）
git commit -m "feat: 新增用户登录功能"

# 提交并添加所有已跟踪文件的修改（跳过 git add）
git commit -am "fix: 修复登录验证 bug"

# 修改最近一次提交信息
git commit --amend -m "新的提交信息"

# 追加文件到最近一次提交（不改信息）
git commit --amend --no-edit
```

> ⚠️ **注意**：`--amend` 会修改提交历史，已推送到远程的提交慎用。

> 💡 **面试可答**：git commit -am 只能提交已跟踪文件的修改，不能提交新文件；新文件必须先用 git add 加入跟踪。

### 🔹 2.2.4 提交信息规范

**Conventional Commits** 规范：

| 类型 | 说明 | 示例 |
|------|------|------|
| `feat` | 新功能 | `feat: 新增用户登录功能` |
| `fix` | 修复 bug | `fix: 修复登录验证失败的问题` |
| `docs` | 文档更新 | `docs: 更新 README 安装说明` |
| `style` | 代码格式调整 | `style: 统一代码缩进` |
| `refactor` | 重构 | `refactor: 重构用户服务层` |
| `perf` | 性能优化 | `perf: 优化查询性能` |
| `test` | 测试相关 | `test: 添加用户登录单元测试` |
| `chore` | 构建/工具相关 | `chore: 更新依赖版本` |

```bash
# ❌ 不推荐：提交信息不清晰
git commit -m "修改代码"

# ✅ 推荐：使用 Conventional Commits 规范
git commit -m "feat: 新增用户登录功能"
git commit -m "fix: 修复登录验证失败的问题"
```

## 2.3 历史查看与比较

### 🔹 2.3.1 查看提交历史

**`git log`**：查看提交历史。

```bash
# 查看完整历史
git log

# 单行简略显示（推荐）
git log --oneline

# 图形化显示分支结构
git log --oneline --graph --all

# 查看某文件的修改历史
git log -p README.md

# 查看某次提交详情
git show <commit-hash>

# 查看最近 5 次提交
git log -5

# 查看提交统计信息
git log --stat
```

### 🔹 2.3.2 差异比较

**`git diff`**：比较不同版本之间的差异。

```bash
# 比较工作区与暂存区
git diff

# 比较暂存区与最近提交
git diff --staged
# 或
git diff --cached

# 比较工作区与指定提交
git diff HEAD

# 比较两个分支
git diff main feature/login

# 比较某文件在不同版本的差异
git diff HEAD~2 HEAD README.md
```

> 💡 **面试可答**：git diff 比较工作区和暂存区；git diff --staged 比较暂存区和最近提交；git diff HEAD 比较工作区和最近提交。

## 2.4 忽略文件

### 🔹 2.4.1 `.gitignore` 文件

在项目根目录创建 `.gitignore` 文件，定义忽略规则。

```gitignore
# 忽略所有 .log 文件
*.log

# 忽略 target 目录（Maven 构建输出）
/target

# 忽略 node_modules 目录（Node.js 依赖）
node_modules/

# 忽略 IDE 配置文件
.idea/
.vscode/
*.iml

# 忽略编译输出
*.class
*.jar
*.war

# 忽略敏感信息
.env
config.ini

# 不忽略 important.log（取反）
!important.log
```

### 🔹 2.4.2 常用忽略规则

| 模式 | 说明 | 示例 |
|------|------|------|
| `*.log` | 忽略所有 `.log` 文件 | `debug.log` |
| `temp/` | 忽略 `temp` 目录 | `temp/file.txt` |
| `/temp` | 仅忽略根目录的 `temp` | `/temp` 但不忽略 `/src/temp` |
| `!important.log` | 不忽略 `important.log` | 取反规则 |
| `doc/*.txt` | 忽略 `doc` 目录下的 `.txt` | `doc/readme.txt` |

### 🔹 2.4.3 已跟踪文件的处理

若文件已被 Git 跟踪，加入 `.gitignore` 后仍会继续跟踪。

```bash
# 从跟踪中移除文件（保留本地文件）
git rm --cached config.ini

# 提交修改
git commit -m "chore: 从跟踪中移除 config.ini"
```

> ⚠️ **注意**：`.gitignore` 只对未跟踪的文件有效。若文件已被跟踪，需要先用 `git rm --cached <file>` 从跟踪中移除，再提交，之后 `.gitignore` 才会对该文件生效。

## 2.5 面试高频题

### Q1：`git add`、`git commit`、`git push` 的区别？

> **标准回答**：
> - `git add`：将工作区的修改加入暂存区（Stage）
> - `git commit`：将暂存区的内容提交到本地仓库，生成一次提交记录
> - `git push`：将本地仓库的提交推送到远程仓库

### Q2：`git log` 和 `git show` 的区别？

> **标准回答**：
> - `git log`：查看提交历史列表，可显示多个提交
> - `git show`：查看某次提交的详细信息，包括修改内容（diff）

### Q3：`.gitignore` 对已跟踪的文件生效吗？

> **标准回答**：
> 不生效。`.gitignore` 只对未跟踪的文件有效。若文件已被跟踪，需要先用 `git rm --cached <file>` 从跟踪中移除，再提交，之后 `.gitignore` 才会对该文件生效。

### Q4：`git commit -am` 和 `git commit -m` 的区别？

> **标准回答**：
> - `git commit -m`：只提交暂存区的内容
> - `git commit -am`：跳过 git add，将所有已跟踪文件的修改加入暂存区并提交，但不能提交新文件

### Q5：`git diff`、`git diff --staged`、`git diff HEAD` 的区别？

> **标准回答**：
> - `git diff`：比较工作区和暂存区
> - `git diff --staged`：比较暂存区和最近提交
> - `git diff HEAD`：比较工作区和最近提交

| 状态管理 | git status、Untracked/Modified/Staged | 查看文件状态，决定下一步操作 |
| 暂存与提交 | git add、git commit、--amend | 将修改提交到本地仓库 |
| 历史查看 | git log、git show、--oneline | 查看提交历史和修改内容 |
| 差异比较 | git diff、--staged、HEAD | 比较不同版本的差异 |
| 忽略文件 | .gitignore、git rm --cached | 排除不需要跟踪的文件 |

------

## ⚠️ 常见坑与注意点

| 现象 / 易错点 | 原因 | 怎么改 / 怎么记 |
|---------------|------|-----------------|
| git commit -am 提交不了新文件 | -am 只能提交已跟踪文件的修改 | 记住：**新文件必须先 git add** |
| .gitignore 不生效 | 文件已被跟踪，.gitignore 只对未跟踪文件有效 | 记住：**已跟踪文件需 git rm --cached** |
| --amend 修改已推送的提交 | --amend 改写历史，会破坏远程仓库的同步 | 记住：**已推送的提交慎用 --amend** |
| 混淆 git diff 的三种用法 | 工作区vs暂存区、暂存区vs最近提交、工作区vs最近提交 | 记住：**git diff（工作区）、--staged（暂存区）、HEAD（工作区vs最近提交）** |
| 提交信息不规范 | 提交信息不清晰，难以追溯历史 | 记住：**使用 Conventional Commits 规范**（feat/fix/docs等） |

------

**学习要点**：

- git init 初始化本地仓库，git clone 克隆远程仓库（包含完整历史）
- git status 查看文件状态，理解 Untracked/Modified/Staged/Unmodified 四种状态
- git add 将修改加入暂存区，git commit 将暂存区内容提交到本地仓库
- git log 查看提交历史，git show 查看某次提交详情，git diff 比较差异
- .gitignore 定义忽略规则，只对未跟踪文件有效；已跟踪文件需 git rm --cached
- 提交信息使用 Conventional Commits 规范，便于追溯历史和自动化处理
- 学完本章可以继续学习**第3章**（分支与合并），学习如何创建分支、合并代码

## 3.1 分支管理

### 🔹 3.1.1 查看分支

```bash
# 查看本地分支
git branch

# 查看本地与远程分支
git branch -a

# 查看分支详细信息（最近提交）
git branch -v

# 查看已合并到当前分支的分支
git branch --merged

# 查看未合并到当前分支的分支
git branch --no-merged
```

### 🔹 3.1.2 创建与切换分支

```bash
# 创建分支
git branch feature/login

# 切换分支（传统方式）
git checkout feature/login

# 切换分支（新方式，Git 2.23+）
git switch feature/login

# 创建并切换分支（传统方式）
git checkout -b feature/register

# 创建并切换分支（新方式）
git switch -c feature/register

# 基于指定提交创建分支
git branch hotfix/bug abc1234
```

> 💡 **面试可答**：git checkout -b = git branch + git switch，一步完成创建和切换；git switch 是 Git 2.23+ 新增的命令，专门用于切换分支，语义更清晰。

### 🔹 3.1.3 删除分支

```bash
# 删除已合并的分支
git branch -d feature/login

# 强制删除未合并的分支
git branch -D feature/old

# 删除远程分支
git push origin --delete feature/login
```

> ⚠️ **注意**：删除分支前确保分支已合并或不再需要；删除远程分支需要 push 权限。

## 3.2 合并操作

### 🔹 3.2.1 基本合并

**`git merge`**：将指定分支合并到当前分支。

```bash
# 切换到目标分支
git checkout main

# 合并 feature 分支
git merge feature/login
```

### 🔹 3.2.2 快进合并 vs 三方合并

| 合并类型 | 触发条件 | 结果 | 特点 |
|----------|----------|------|------|
| **快进合并（Fast-forward）** | 当前分支无新提交 | 直接移动指针 | 不产生合并提交，历史线性 |
| **三方合并（Three-way merge）** | 当前分支有新提交 | 产生新的合并提交 | 保留分支历史，形成分支结构 |

```
快进合并前：           快进合并后：
A --- B (main)         A --- B --- C --- D (main, feature)
       \
        C --- D (feature)

三方合并前：           三方合并后：
A --- B --- E (main)   A --- B --- E --- M (main)
       \                       \   /
        C --- D (feature)       C --- D (feature)
                              M 是合并提交
```

### 🔹 3.2.3 合并选项

```bash
# 强制产生合并提交（即使可以快进）
git merge --no-ff feature/login

# 压缩合并（将目标分支的多次提交压缩为一次）
git merge --squash feature/login

# 中止合并
git merge --abort
```

> 💡 **面试可答**：--no-ff 强制产生合并提交，保留分支历史，便于追溯；--squash 将多次提交压缩为一次，适合整理提交历史。

```bash
# ❌ 不推荐：直接 merge，可能丢失分支信息
git checkout main
git merge feature/login

# ✅ 推荐：使用 --no-ff 保留分支历史
git checkout main
git merge --no-ff feature/login -m "Merge feature/login: 新增用户登录功能"
```

## 3.3 变基操作

### 🔹 3.3.1 基本变基

**`git rebase`**：将当前分支的提交"挪到"目标分支之后，历史呈直线。

```bash
# 切换到 feature 分支
git checkout feature/login

# 基于 main 分支变基
git rebase main
```

**变基原理**：

```
变基前：                    变基后：
A --- B --- C (main)        A --- B --- C (main)
       \                          \
        D --- E (feature)          D' --- E' (feature)
                                     （重新应用的提交）
```

> 💡 **面试可答**：rebase 改写提交历史，使历史呈线性；merge 保留分支历史，产生合并提交。rebase 适合个人分支整理，merge 适合公共分支合并。

### 🔹 3.3.2 交互式变基

**`git rebase -i`**：可合并、重排、修改、删除提交。

```bash
# 交互式变基最近 5 次提交
git rebase -i HEAD~5

# 交互式变基到指定提交
git rebase -i abc1234
```

**交互式变基命令**：

| 命令 | 缩写 | 作用 |
|------|------|------|
| `pick` | `p` | 保留提交 |
| `reword` | `r` | 保留提交，修改提交信息 |
| `squash` | `s` | 合并到上一个提交，保留提交信息 |
| `fixup` | `f` | 合并到上一个提交，丢弃提交信息 |
| `drop` | `d` | 删除提交 |
| `edit` | `e` | 保留提交，暂停修改 |

### 🔹 3.3.3 变基的注意事项

> ⚠️ **黄金法则**：
>  **不要对已推送到远程的公共分支做 rebase**，会改写他人依赖的历史，造成协作混乱。
>  仅对未推送的个人分支使用 rebase。

```bash
# ❌ 危险：对已推送的公共分支 rebase
git checkout main
git rebase feature/login  # 会改写 main 的历史

# ✅ 安全：对未推送的个人分支 rebase
git checkout feature/login
git rebase main  # feature/login 未推送，可以 rebase
```

## 3.4 冲突解决

### 🔹 3.4.1 冲突产生

merge 或 rebase 时，若两方修改了同一文件的同一区域，Git 无法自动合并，会标记冲突。

### 🔹 3.4.2 冲突标记

```java
public class UserService {
<<<<<<< HEAD
    // main 分支的代码
    public void login() {
        System.out.println("Login v1");
    }
=======
    // feature 分支的代码
    public void login() {
        System.out.println("Login v2");
    }
>>>>>>> feature/login
}
```

### 🔹 3.4.3 解决流程

1. **打开冲突文件**：找到 `<<<<<<<`、`=======`、`>>>>>>>` 标记
2. **手动编辑**：保留需要的版本或合并两者，删除冲突标记
3. **标记已解决**：`git add <file>`
4. **完成操作**：
   - merge 时：`git commit` 完成合并
   - rebase 时：`git rebase --continue` 继续，或 `git rebase --abort` 放弃

```bash
# 解决冲突后标记已解决
git add conflicted-file.txt

# merge 时完成合并
git commit

# rebase 时继续
git rebase --continue

# 放弃 rebase
git rebase --abort
```

> 💡 **面试可答**：解决冲突的核心是手动编辑冲突文件，保留需要的代码，删除冲突标记，然后用 git add 标记已解决。merge 时需要 commit，rebase 时需要 continue。

```bash
# ❌ 不推荐：直接删除冲突标记，不检查代码
git add conflicted-file.txt

# ✅ 推荐：仔细对比两方代码，合并或选择最优版本
# 手动编辑 conflicted-file.txt，保留需要的代码
git add conflicted-file.txt
git commit  # 或 git rebase --continue
```

## 3.5 分支策略

### 🔹 3.5.1 Git Flow

**Git Flow**：经典的分支管理模型，适合有固定发布周期的项目。

| 分支类型 | 命名 | 用途 | 生命周期 |
|----------|------|------|----------|
| **主分支** | `main` / `master` | 生产环境代码 | 长期 |
| **开发分支** | `develop` | 日常开发集成 | 长期 |
| **功能分支** | `feature/*` | 新功能开发 | 临时 |
| **发布分支** | `release/*` | 版本发布准备 | 临时 |
| **热修复分支** | `hotfix/*` | 生产环境紧急修复 | 临时 |

```
main ─────●────────●────────●────────●────────●─────
           \       /         \       /         \
            ●─────●           ●─────●           ●
           release/1.0       hotfix/1.1        release/1.1

develop ───●────●────●────●────●────●────●────●────
                \    /         \    /
                 ●──●           ●──●
               feature/a      feature/b
```

### 🔹 3.5.2 GitHub Flow

**GitHub Flow**：简化模型，适合持续部署的项目。

- `main` 分支始终可部署
- 功能在 `feature/*` 分支开发
- 通过 Pull Request 合并
- 合并后立即部署

```
main ─────●────────●────────●────────●────────●─────
           \       /         \       /         \
            ●─────●           ●─────●           ●
          feature/1         feature/2         feature/3
```

### 🔹 3.5.3 简单分支策略

**小团队常用**：`main` + `feature/xxx`

| 场景 | 分支命名 |
|------|----------|
| 新功能 | `feature/user-login` |
| Bug 修复 | `fix/login-error` |
| 文档更新 | `docs/api-guide` |
| 性能优化 | `perf/query-optimize` |

> 💡 **面试可答**：Git Flow 适合有固定发布周期的项目，分支较多但结构清晰；GitHub Flow 适合持续部署，简单高效；小团队可以用 main + feature 的简化策略。

## 3.6 面试高频题

### Q1：merge 与 rebase 的区别是什么？

> **标准回答**：
> 1. **历史记录**：merge 保留分支历史，产生合并提交；rebase 改写历史，呈线性
> 2. **使用场景**：merge 适合公共分支；rebase 适合个人分支整理提交
> 3. **冲突处理**：merge 解决一次冲突；rebase 可能需要解决多次冲突
> 4. **安全性**：merge 不改变提交历史，更安全；rebase 改写历史，对已推送的分支慎用

### Q2：什么是快进合并（Fast-forward）？

> **标准回答**：
> 快进合并是指当前分支没有产生新提交时，直接将分支指针移动到目标分支的最新提交，不产生新的合并提交。这种合并方式历史记录呈线性，但会丢失分支信息。

### Q3：如何解决 Git 冲突？

> **标准回答**：
> 1. 执行 `git status` 查看冲突文件
> 2. 打开冲突文件，找到 `<<<<<<<`、`=======`、`>>>>>>>` 标记
> 3. 手动编辑，保留需要的代码，删除冲突标记
> 4. `git add <file>` 标记冲突已解决
> 5. `git commit`（merge）或 `git rebase --continue`（rebase）完成操作

### Q4：Git Flow 和 GitHub Flow 的区别？

> **标准回答**：
> Git Flow 是经典的分支管理模型，包含 main、develop、feature、release、hotfix 五种分支，适合有固定发布周期的项目；GitHub Flow 是简化模型，只有 main 分支和 feature 分支，适合持续部署的项目。

### Q5：为什么不要对已推送的分支做 rebase？

> **标准回答**：
> rebase 会改写提交历史，如果分支已推送到远程并被他人使用，rebase 会破坏他人的历史，造成协作混乱。黄金法则：仅对未推送的个人分支使用 rebase。

## 4.1 远程仓库管理

### 🔹 4.1.1 查看远程仓库

```bash
# 查看远程仓库
git remote

# 查看远程仓库详细信息
git remote -v

# 查看指定远程仓库信息
git remote show origin
```

### 🔹 4.1.2 添加远程仓库

```bash
# 添加远程仓库（默认命名为 origin）
git remote add origin https://github.com/username/repository.git

# 添加上游仓库（fork 场景）
git remote add upstream https://github.com/original-owner/repository.git
```

### 🔹 4.1.3 删除和重命名远程仓库

```bash
# 删除远程仓库
git remote remove origin

# 重命名远程仓库
git remote rename origin new-origin
```

> 💡 **面试可答**：origin 是默认远程仓库名称，upstream 是上游仓库（fork 场景）；git remote -v 显示远程仓库的 fetch 和 push 地址。

## 4.2 推送与拉取

### 🔹 4.2.1 推送代码

**`git push`**：将本地提交推送到远程仓库。

```bash
# 推送当前分支到远程（需先设置上游分支）
git push

# 推送当前分支到远程，并设置上游分支
git push -u origin main

# 推送指定分支到远程
git push origin feature/login

# 推送所有分支到远程
git push --all

# 推送所有标签到远程
git push --tags

# 强制推送（慎用）
git push -f origin main
```

> ⚠️ **注意**：`git push -f` 会强制覆盖远程仓库，慎用！仅在本地历史需要改写且确认无他人使用时使用。

> 💡 **面试可答**：git push -u 设置上游分支，之后可以直接 git push；git push --all 推送所有分支；git push --tags 推送所有标签。

```bash
# ❌ 不推荐：强制推送，覆盖远程历史
git push -f origin main

# ✅ 推荐：正常推送，或使用 --force-with-lease
git push origin main
git push --force-with-lease origin main  # 更安全，检查远程是否有新提交
```

### 🔹 4.2.2 拉取代码

**`git pull`**：拉取远程更新并合并到当前分支。

```bash
# 拉取当前分支的上游分支（默认）
git pull

# 拉取指定远程分支
git pull origin main

# 拉取时使用 rebase 代替 merge，保持历史线性
git pull --rebase

# 拉取但不合并（仅更新远程分支信息）
git fetch origin
```

> 💡 **面试可答**：git pull = git fetch + git merge；git pull --rebase = git fetch + git rebase，保持历史线性。

```bash
# ❌ 不推荐：直接 git pull，可能产生合并提交
git pull

# ✅ 推荐：使用 --rebase 保持历史线性
git pull --rebase
```

### 🔹 4.2.3 fetch 与 pull 的区别

| 命令 | 说明 | 使用场景 |
|------|------|----------|
| `git fetch` | 拉取远程更新，不合并 | 查看远程更新、谨慎合并 |
| `git pull` | 拉取并自动合并 | 快速同步远程更新 |

> 💡 **面试可答**：fetch 只拉取不合并，可以查看远程更新后再决定是否合并；pull 自动合并，适合快速同步。

```bash
# 查看所有分支（本地+远程）
git branch -a

git branch -r

# 查看远程分支详细信息
git branch -vv
```

### 🔹 4.3.2 跟踪远程分支

```bash
# 基于远程分支创建本地分支（自动跟踪）
git checkout -b feature/login origin/feature/login

# 设置本地分支跟踪远程分支
git branch --set-upstream-to=origin/main main

# 查看跟踪关系

```

### 🔹 4.3.3 删除远程分支

```bash
# 删除远程分支

```

> 💡 **面试可答**：git push origin --delete 删除远程分支；git branch -d 删除本地分支；删除前确认分支已合并或不再需要。

## 4.4 参与开源项目

### 🔹 4.4.1 Fork 流程

1. **Fork**：在 GitHub 上 fork 原仓库到自己的账号
2. **Clone**：克隆自己的 fork 仓库到本地
3. **修改**：在本地修改代码并提交
4. **Push**：推送到自己的 fork 仓库
5. **PR**：在原仓库创建 Pull Request

```bash
# 1. Fork（在 GitHub 网页操作）

# 2. Clone 自己的 fork 仓库
git clone https://github.com/your-username/repository.git
cd repository

# 3. 添加上游仓库
git remote add upstream https://github.com/original-owner/repository.git

# 4. 创建功能分支
git checkout -b feature/new-feature

# 5. 修改代码并提交
git add .
git commit -m "feat: 新增功能"

# 6. 推送到自己的 fork 仓库
git push origin feature/new-feature

# 7. 在 GitHub 上创建 Pull Request
```

### 🔹 4.4.2 同步上游更新

```bash
# 拉取上游更新
git fetch upstream

# 合并上游更新到本地 main 分支
git checkout main
git merge upstream/main

# 推送到自己的 fork 仓库
git push origin main
```

> 💡 **面试可答**：参与开源项目的流程是 Fork → Clone → 修改 → Push → PR；需要定期同步上游更新，避免 PR 冲突。

## 4.5 面试高频题

### Q1：fetch 与 pull 的区别？

> **标准回答**：
> - `git fetch`：拉取远程更新，不合并到当前分支
> - `git pull`：拉取远程更新并自动合并到当前分支
> - `git pull` = `git fetch` + `git merge`

### Q2：如何设置上游分支？

> **标准回答**：
> - `git push -u origin main`：推送并设置上游分支
> - `git branch --set-upstream-to=origin/main main`：设置本地分支跟踪远程分支
> - 设置上游分支后，可以直接使用 `git push` 和 `git pull`

### Q3：什么是 origin 和 upstream？

> **标准回答**：
> - `origin`：默认远程仓库名称，通常是自己 fork 的仓库
> - `upstream`：上游仓库，通常是原仓库（fork 场景）
> - 参与开源项目时，origin 是自己的 fork，upstream 是原仓库

### Q4：如何删除远程分支？

> **标准回答**：
> - `git push origin --delete feature/login`：删除远程分支
> - `git push origin :feature/login`：使用冒号语法删除远程分支

### Q5：什么时候用 git push -f？

> **标准回答**：
> `git push -f` 强制推送，会覆盖远程仓库。仅在以下情况使用：
> 1. 本地历史需要改写（如 rebase）
> 2. 确认远程分支没有被他人使用
> 3. 更安全的做法是使用 `git push --force-with-lease`

## 5.1 撤销工作区修改

### 🔹 5.1.1 丢弃工作区修改

**`git restore`**：丢弃工作区的修改（Git 2.23+）。

```bash
# 丢弃单个文件的修改
git restore README.md

# 丢弃所有文件的修改
git restore .

# 丢弃暂存区和工作区的修改
git restore --staged --worktree README.md
```

**`git checkout --`**：传统方式（仍可用）。

```bash
# 丢弃单个文件的修改
git checkout -- README.md

# 丢弃所有文件的修改
git checkout -- .
```

> 💡 **面试可答**：git restore 是 Git 2.23+ 新增的命令，专门用于恢复文件；git checkout -- 是传统方式，功能更多但语义不清晰。

```bash
# ❌ 不推荐：误用 git restore，可能丢失重要修改
git restore README.md  # 未确认是否需要保留

# ✅ 推荐：先用 git diff 查看修改，再决定是否恢复
git diff README.md  # 查看修改内容
git restore README.md  # 确认不需要后恢复
```

### 🔹 5.1.2 恢复删除的文件

```bash
# 恢复误删的文件（从最近提交恢复）
git restore deleted-file.txt

# 从指定提交恢复文件
git restore --source=abc1234 deleted-file.txt
```

> 💡 **面试可答**：git restore --source 可以从任意提交恢复文件；git checkout 也可以恢复文件，但 git restore 语义更清晰。

## 5.2 取消暂存

### 🔹 5.2.1 取消暂存区修改

**`git restore --staged`**：取消暂存，保留工作区修改（Git 2.23+）。

```bash
# 取消单个文件的暂存
git restore --staged README.md

# 取消所有文件的暂存
git restore --staged .
```

**`git reset HEAD`**：传统方式（仍可用）。

```bash
# 取消单个文件的暂存
git reset HEAD README.md

# 取消所有文件的暂存
git reset HEAD
```

> 💡 **面试可答**：git restore --staged 取消暂存但保留工作区修改；git reset HEAD 也可以取消暂存，但 git restore 语义更清晰。

```bash
# ❌ 不推荐：误用 git reset，可能影响工作区
git reset --hard HEAD  # 会清空工作区，丢失所有修改

# ✅ 推荐：使用 --staged 保留工作区修改
git restore --staged README.md  # 只取消暂存，保留修改
```

## 5.3 回退提交

### 🔹 5.3.1 git reset 三种模式

**`git reset`**：回退到指定提交，有三种模式。

| 模式 | 暂存区 | 工作区 | 说明 |
|------|--------|--------|------|
| `--soft` | 保留 | 保留 | 回退提交，保留修改在暂存区 |
| `--mixed`（默认） | 清空 | 保留 | 回退提交，保留修改在工作区 |
| `--hard` | 清空 | 清空 | 彻底回退，**未提交修改会丢失** |

```bash
# 回退到上一个提交，保留修改在暂存区
git reset --soft HEAD~1

# 回退到上一个提交，保留修改在工作区（默认）
git reset HEAD~1

# 彻底回退到上一个提交，未提交修改会丢失
git reset --hard HEAD~1
```

> ⚠️ **注意**：`--hard` 会清空工作区，未提交的修改会丢失，慎用！

> 💡 **面试可答**：--soft 保留修改在暂存区，--mixed 保留修改在工作区，--hard 清空工作区；已推送的提交慎用 reset。

```bash
# ❌ 危险：对已推送的提交使用 --hard
git reset --hard HEAD~1
git push -f  # 会破坏远程历史

# ✅ 安全：对未推送的提交使用 --soft 或 --mixed
git reset --soft HEAD~1  # 保留修改在暂存区
git commit -m "新的提交信息"  # 重新提交
```

### 🔹 5.3.2 回退到指定提交

```bash
# 回退到指定提交（使用提交哈希）
git reset --hard abc1234

# 回退到指定提交（使用相对引用）
git reset --hard HEAD~3

# 回退到指定提交（使用分支名）
git reset --hard main
```

### 🔹 5.3.3 撤销已推送的提交

> ⚠️ **黄金法则**：
>  **不要对已推送的提交使用 reset**，会破坏他人依赖的历史。
>  若需撤销已推送的提交，使用 `git revert`。

```bash
# ❌ 危险：对已推送的提交使用 reset
git reset --hard HEAD~1
git push -f  # 会破坏远程历史

# ✅ 安全：使用 revert 撤销已推送的提交
git revert HEAD  # 新增一次"反向"提交
git push  # 正常推送，不破坏历史
```

------

## 5.4 反向提交（revert）

### 🔹 5.4.1 基本用法

**`git revert`**：新增一次"反向"提交，撤销指定提交的修改，不改写历史。

```bash
# 撤销最近一次提交
git revert HEAD

# 撤销指定提交
git revert abc1234

# 撤销多次提交
git revert HEAD~3..HEAD
```

> 💡 **面试可答**：revert 新增一次"反向"提交，不改写历史，适合撤销已推送的提交；reset 改写历史，适合撤销未推送的提交。

```bash
# ❌ 不推荐：对已推送的提交使用 reset
git reset --hard HEAD~1
git push -f  # 会破坏远程历史

# ✅ 推荐：使用 revert 撤销已推送的提交
git revert HEAD  # 新增一次"反向"提交
git push  # 正常推送，不破坏历史
```

### 🔹 5.4.2 revert 与 reset 的区别

| 对比项 | git reset | git revert |
|--------|-----------|------------|
| **改写历史** | ✅ 改写 | ❌ 不改写 |
| **适用场景** | 未推送的提交 | 已推送的提交 |
| **安全性** | 危险，破坏历史 | 安全，保留历史 |
| **操作方式** | 移动 HEAD | 新增反向提交 |

> 💡 **面试可答**：reset 改写历史，适合未推送的提交；revert 不改写历史，适合已推送的提交；已推送的提交必须用 revert。

## 5.5 恢复丢失的提交

### 🔹 5.5.1 使用 reflog

**`git reflog`**：查看 HEAD 的移动历史，用于恢复误操作。

```bash
# 查看 reflog
git reflog

# 查看最近 10 条 reflog
git reflog -10

# 恢复到 reflog 中的某个状态
git reset --hard HEAD@{2}
```

> 💡 **面试可答**：reflog 记录 HEAD 的移动历史，可以找回误删的提交、误删的分支；reflog 有过期时间（默认 90 天），及时恢复。

```bash
# 场景：误用 git reset --hard，丢失了重要提交
git reset --hard HEAD~1  # 误操作，丢失了最近一次提交

# 恢复步骤：
git reflog  # 查找丢失的提交
# 输出示例：
# abc1234 HEAD@{0}: reset: moving to HEAD~1
# def5678 HEAD@{1}: commit: 新增重要功能

git reset --hard def5678  # 恢复到丢失的提交
```

### 🔹 5.5.2 恢复误删的分支

```bash
# 场景：误删了分支
git branch -d feature/login  # 误操作，删除了分支

# 恢复步骤：
git reflog  # 查找分支的最后一次提交
# 输出示例：
# abc1234 HEAD@{2}: checkout: moving from main to feature/login

git branch feature/login abc1234  # 基于提交重建分支
```

> 💡 **面试可答**：误删分支后，可以用 git reflog 找到分支的最后一次提交，然后用 git branch 重建分支。

## 6.1 标签管理

### 🔹 6.1.1 标签类型

**轻量标签（Lightweight）**：只是一个指向提交的指针，类似分支但不会移动。

```bash
# 创建轻量标签
git tag v1.0.0
```

**附注标签（Annotated）**：包含标签名、标签信息、日期、作者等完整信息。

```bash
# 创建附注标签
git tag -a v1.0.0 -m "版本 1.0.0 发布"
```

> 💡 **面试可答**：轻量标签只是指针，附注标签包含完整信息；推荐使用附注标签，便于追溯版本信息。

```bash
# ❌ 不推荐：使用轻量标签，信息不完整
git tag v1.0.0

# ✅ 推荐：使用附注标签，包含完整信息
git tag -a v1.0.0 -m "版本 1.0.0 发布"
```

### 🔹 6.1.2 查看标签

```bash
# 列出所有标签
git tag

# 查看标签详细信息
git show v1.0.0

# 查看标签列表（按版本排序）
git tag -l "v*"
```

### 🔹 6.1.3 推送和删除标签

```bash
# 推送指定标签到远程
git push origin v1.0.0

# 推送所有标签到远程
git push --tags

# 删除本地标签
git tag -d v1.0.0

# 删除远程标签
git push origin --delete v1.0.0
```

> 💡 **面试可答**：git push --tags 推送所有标签；git push origin <tag> 推送指定标签；删除远程标签用 --delete。

### 🔹 6.1.4 基于标签创建分支

```bash
# 基于标签创建分支（用于维护旧版本）
git checkout -b hotfix/1.0.0 v1.0.0
```

> 💡 **面试可答**：基于标签创建分支，用于维护旧版本（如 hotfix）；标签指向固定提交，不会移动。

## 6.2 贮藏（Stash）

### 🔹 6.2.1 基本用法

**`git stash`**：临时保存当前工作区的修改，便于切换分支。

```bash
# 贮藏当前修改
git stash

# 贮藏并添加说明
git stash save "工作到一半，切换分支"

# 查看贮藏列表
git stash list

# 应用最近一次贮藏
git stash pop

# 应用指定贮藏
git stash apply stash@{1}

# 删除指定贮藏
git stash drop stash@{1}

# 删除所有贮藏
git stash clear
```

> 💡 **面试可答**：stash 临时保存工作区修改，便于切换分支；stash pop 应用并删除，stash apply 应用但不删除。

```bash
# ❌ 不推荐：直接切换分支，工作区有未提交修改
git checkout main  # 可能失败或丢失修改

# ✅ 推荐：先 stash 再切换分支
git stash  # 保存当前修改
git checkout main  # 切换分支
git stash pop  # 切换回来后恢复修改
```

### 🔹 6.2.2 贮藏选项

```bash
# 贮藏包括未跟踪的文件
git stash -u

# 贮藏包括未跟踪和忽略的文件
git stash -a

# 只贮藏暂存区的修改
git stash --keep-index
```

> 💡 **面试可答**：-u 包括未跟踪文件，-a 包括忽略文件，--keep-index 只贮藏暂存区。

### 🔹 6.2.3 应用贮藏

```bash
# 应用并删除最近一次贮藏
git stash pop

# 应用但不删除最近一次贮藏
git stash apply

# 应用指定贮藏
git stash apply stash@{1}
```

> 💡 **面试可答**：stash pop 应用并删除，stash apply 应用但不删除；可以多次 apply 同一个贮藏。

## 6.3 挑选提交（Cherry-pick）

### 🔹 6.3.1 基本用法

**`git cherry-pick`**：将指定提交应用到当前分支。

```bash
# 挑选单个提交
git cherry-pick abc1234

# 挑选多个提交
git cherry-pick abc1234 def5678

# 挑选范围提交（不包括起始提交）
git cherry-pick abc1234^..def5678
```

> 💡 **面试可答**：cherry-pick 将指定提交应用到当前分支，适合跨分支合并单个提交；可能产生冲突，需要解决。

```bash
# 场景：feature 分支有重要修复，需要合并到 main
git checkout main
git cherry-pick abc1234  # 将 feature 分支的提交应用到 main
```

### 🔹 6.3.2 cherry-pick 选项

```bash
# 只应用提交，不自动提交
git cherry-pick -n abc1234

# 应用提交并编辑提交信息
git cherry-pick -e abc1234

# 放弃 cherry-pick
git cherry-pick --abort
```

> 💡 **面试可答**：-n 只应用不提交，-e 编辑提交信息，--abort 放弃操作。

### 🔹 6.3.3 解决冲突

cherry-pick 可能产生冲突，解决流程与 merge 类似。

```bash
# 1. 解决冲突文件
# 2. 标记已解决
git add conflicted-file.txt

# 3. 继续 cherry-pick
git cherry-pick --continue

# 或放弃 cherry-pick
git cherry-pick --abort
```

> 💡 **面试可答**：cherry-pick 冲突解决流程与 merge 类似：解决冲突 → git add → git cherry-pick --continue。

## 6.4 二分查找（Bisect）

### 🔹 6.4.1 基本用法

**`git bisect`**：通过二分查找定位引入 bug 的提交。

```bash
# 开始二分查找
git bisect start

# 标记当前提交有 bug
git bisect bad

# 标记某个提交无 bug（如上一个版本）
git bisect good v1.0.0

# Git 会自动切换到中间提交，测试后标记
git bisect good  # 或 git bisect bad

# 重复测试，直到找到引入 bug 的提交

# 结束二分查找
git bisect reset
```

> 💡 **面试可答**：git bisect 通过二分查找快速定位引入 bug 的提交；需要标记 good 和 bad 提交，Git 会自动缩小范围。

```bash
# 场景：v1.0.0 正常，v1.1.0 有 bug，定位引入 bug 的提交
git bisect start
git bisect bad  # 当前版本有 bug
git bisect good v1.0.0  # v1.0.0 正常
# Git 自动切换到中间提交
# 测试后标记 good 或 bad
# 重复直到找到引入 bug 的提交
git bisect reset  # 结束查找
```

### 🔹 6.4.2 自动化二分查找

```bash
# 使用脚本自动测试
git bisect run ./test-script.sh
```

> 💡 **面试可答**：git bisect run 可以自动化测试，脚本返回 0 表示 good，非 0 表示 bad。

## 6.5 面试高频题

### Q1：轻量标签和附注标签的区别？

> **标准回答**：
> - **轻量标签**：只是一个指向提交的指针，类似分支但不会移动
> - **附注标签**：包含标签名、标签信息、日期、作者等完整信息
> - 推荐使用附注标签，便于追溯版本信息

### Q2：git stash 的作用？

> **标准回答**：
> - `git stash`：临时保存当前工作区的修改，便于切换分支
> - `git stash pop`：应用并删除最近一次贮藏
> - `git stash apply`：应用但不删除最近一次贮藏

### Q3：git cherry-pick 的作用？

> **标准回答**：
> - `git cherry-pick`：将指定提交应用到当前分支
> - 适合跨分支合并单个提交，可能产生冲突
> - 冲突解决流程：解决冲突 → git add → git cherry-pick --continue

### Q4：git bisect 的作用？

> **标准回答**：
> - `git bisect`：通过二分查找定位引入 bug 的提交
> - 需要标记 good 和 bad 提交，Git 会自动缩小范围
> - 可以用 `git bisect run` 自动化测试

### Q5：如何推送标签到远程？

> **标准回答**：
> - `git push origin <tag>`：推送指定标签
> - `git push --tags`：推送所有标签
> - 删除远程标签：`git push origin --delete <tag>`

## 7.1 Git 配置

### 🔹 7.1.1 配置级别

Git 配置有三个级别，优先级从高到低：

| 级别 | 作用范围 | 配置文件位置 |
|------|----------|--------------|
| `--local` | 当前仓库 | `.git/config` |
| `--global` | 当前用户 | `~/.gitconfig` |
| `--system` | 系统所有用户 | `/etc/gitconfig` |

```bash
# 查看当前配置（显示生效的配置）
git config --list

# 查看指定配置
git config user.name

# 查看本地配置
git config --local --list

# 查看全局配置
git config --global --list

# 查看系统配置
git config --system --list
```

> 💡 **面试可答**：配置优先级：local > global > system；local 只影响当前仓库，global 影响当前用户的所有仓库。

### 🔹 7.1.2 基本配置

```bash
# 设置用户名（必须配置）
git config --global user.name "Your Name"

# 设置邮箱（必须配置）
git config --global user.email "your.email@example.com"

# 设置默认分支名（Git 2.28+）
git config --global init.defaultBranch main

# 设置默认编辑器
git config --global core.editor vim

# 设置差异比较工具
git config --global diff.tool vimdiff
```

> 💡 **面试可答**：user.name 和 user.email 是必须配置的，用于标识提交者；init.defaultBranch 设置默认分支名（如 main）。

```bash
# ❌ 不推荐：不配置用户信息，提交时无法识别
git commit -m "test"

# ✅ 推荐：配置用户信息后再提交
git config --global user.name "Your Name"
git config --global user.email "your.email@example.com"
git commit -m "test"
```

### 🔹 7.1.3 别名配置

```bash
# 设置别名
git config --global alias.st status
git config --global alias.co checkout
git config --global alias.br branch
git config --global alias.ci commit
git config --global alias.unstage 'reset HEAD --'
git config --global alias.last 'log -1 HEAD'
git config --global alias.visual '!gitk'

# 查看别名
git config --global --get-regexp alias

# 删除别名
git config --global --unset alias.st
```

> 💡 **面试可答**：别名可以简化常用命令，提高效率；别名可以执行 shell 命令（用 ! 前缀）。

```bash
# ❌ 不推荐：每次输入完整命令
git status
git checkout main
git branch -a

# ✅ 推荐：使用别名简化命令
git st
git co main
git br -a
```

### 🔹 7.1.4 其他常用配置

```bash
# 设置换行符处理（Windows）
git config --global core.autocrlf true

# 设置换行符处理（Linux/Mac）
git config --global core.autocrlf input

# 设置忽略文件权限变化
git config --global core.fileMode false

# 设置颜色输出
git config --global color.ui true

# 设置合并策略
git config --global merge.ff false

# 设置拉取策略
git config --global pull.rebase true
```

> 💡 **面试可答**：core.autocrlf 处理换行符（Windows 用 true，Linux 用 input）；pull.rebase 设置拉取时是否使用 rebase。

## 7.2 Git 钩子

### 🔹 7.2.1 钩子类型

Git 钩子是在特定事件发生时自动执行的脚本，分为客户端钩子和服务端钩子。

**客户端钩子**：
- `pre-commit`：提交前执行
- `commit-msg`：提交信息编辑后执行
- `pre-push`：推送前执行
- `post-merge`：合并后执行

**服务端钩子**：
- `pre-receive`：接收推送前执行
- `post-receive`：接收推送后执行
- `update`：更新分支前执行

> 💡 **面试可答**：客户端钩子在本地执行，服务端钩子在服务器执行；常用钩子：pre-commit（代码检查）、commit-msg（提交信息规范）。

### 🔹 7.2.2 钩子位置

```bash
# 钩子位置：.git/hooks/
ls -la .git/hooks/

# 查看示例钩子（以 .sample 结尾）
cat .git/hooks/pre-commit.sample
```

> 💡 **面试可答**：钩子位于 .git/hooks/ 目录；示例钩子以 .sample 结尾，需要去掉 .sample 并添加执行权限才能生效。

### 🔹 7.2.3 创建钩子

```bash
# 1. 复制示例钩子
cp .git/hooks/pre-commit.sample .git/hooks/pre-commit

# 2. 编辑钩子内容
vim .git/hooks/pre-commit

# 3. 添加执行权限
chmod +x .git/hooks/pre-commit
```

**示例：pre-commit 钩子（检查代码风格）**

```bash
#!/bin/bash
# pre-commit 钩子：检查代码风格

# 检查暂存区文件
FILES=$(git diff --cached --name-only --diff-filter=ACM | grep '\.js$')

if [ -n "$FILES" ]; then
    echo "Running ESLint..."
    eslint $FILES
    if [ $? -ne 0 ]; then
        echo "ESLint failed. Commit aborted."
        exit 1
    fi
fi

exit 0
```

**示例：commit-msg 钩子（检查提交信息格式）**

```bash
#!/bin/bash
# commit-msg 钩子：检查提交信息格式

MSG_FILE=$1
MSG=$(cat $MSG_FILE)

# 检查提交信息格式（如：feat: add new feature）
if [[ ! $MSG =~ ^(feat|fix|docs|style|refactor|test|chore)(\(.+\))?: .+ ]]; then
    echo "Invalid commit message format."
    echo "Format: type(scope): description"
    echo "Types: feat, fix, docs, style, refactor, test, chore"
    exit 1
fi

exit 0
```

> 💡 **面试可答**：pre-commit 用于代码检查（如 ESLint、Prettier）；commit-msg 用于检查提交信息格式（如 Conventional Commits）。

```bash
# ❌ 不推荐：手动检查代码风格和提交信息
git commit -m "fix bug"

# ✅ 推荐：使用钩子自动检查
git commit -m "fix: fix login bug"
# pre-commit 自动检查代码风格
# commit-msg 自动检查提交信息格式
```

### 🔹 7.2.4 跳过钩子

```bash
# 跳过所有钩子
git commit --no-verify -m "skip hooks"

# 跳过 pre-commit 钩子
git commit --no-verify -m "skip pre-commit"
```

> ⚠️ **注意**：跳过钩子可能导致代码质量下降，慎用。

> 💡 **面试可答**：--no-verify 跳过所有钩子；慎用，可能导致代码质量下降。

## 7.3 提交规范

### 🔹 7.3.1 Conventional Commits

Conventional Commits 是一种提交信息规范，格式为：

```
<type>(<scope>): <subject>

<body>

<footer>
```

**Type 类型**：

| Type | 说明 |
|------|------|
| `feat` | 新功能 |
| `fix` | 修复 bug |
| `docs` | 文档更新 |
| `style` | 代码格式调整（不影响功能） |
| `refactor` | 重构（既不是新功能也不是修复） |
| `test` | 测试相关 |
| `chore` | 构建过程或辅助工具的变动 |

> 💡 **面试可答**：Conventional Commits 规范提交信息格式；常用类型：feat（新功能）、fix（修复 bug）、docs（文档）。

**示例**：

```bash
# 新功能
git commit -m "feat(auth): add login page"

# 修复 bug
git commit -m "fix(api): fix user authentication error"

# 文档更新
git commit -m "docs(readme): update installation guide"

# 代码格式调整
git commit -m "style(ui): adjust button spacing"

# 重构
git commit -m "refactor(core): simplify user validation logic"

# 测试
git commit -m "test(auth): add unit tests for login"

# 构建工具变动
git commit -m "chore(deps): update dependencies"
```

### 🔹 7.3.2 提交信息最佳实践

```bash
# ❌ 不推荐：提交信息不清晰
git commit -m "update"
git commit -m "fix bug"
git commit -m "add stuff"

# ✅ 推荐：提交信息清晰明确
git commit -m "feat(auth): add OAuth2 login support"
git commit -m "fix(api): handle null response from user service"
git commit -m "docs(readme): update installation instructions for Windows"
```

> 💡 **面试可答**：提交信息应该清晰明确，说明做了什么、为什么做；使用 Conventional Commits 规范。

### 🔹 7.3.3 提交频率

```bash
# ❌ 不推荐：一次性提交大量修改
git add .
git commit -m "complete feature"

# ✅ 推荐：小步提交，每次提交一个逻辑单元
git add auth.js
git commit -m "feat(auth): add user authentication"

git add login.js
git commit -m "feat(ui): add login page"

git add api.js
git commit -m "feat(api): add user login endpoint"
```

> 💡 **面试可答**：小步提交，每次提交一个逻辑单元；便于代码审查和回退。

## 7.4 分支策略

### 🔹 7.4.1 Git Flow

Git Flow 是一种经典的分支策略，适用于有明确发布周期的项目。

**分支类型**：

| 分支 | 说明 | 生命周期 |
|------|------|----------|
| `main` | 主分支，只包含发布版本 | 长期 |
| `develop` | 开发分支，集成所有功能 | 长期 |
| `feature/*` | 功能分支，开发新功能 | 临时 |
| `release/*` | 发布分支，准备发布 | 临时 |
| `hotfix/*` | 紧急修复分支，修复线上 bug | 临时 |

**工作流程**：

```bash
# 1. 从 develop 创建功能分支
git checkout develop
git checkout -b feature/login

# 2. 开发功能
git add .
git commit -m "feat(auth): add login page"

# 3. 合并回 develop
git checkout develop
git merge feature/login
git branch -d feature/login

# 4. 从 develop 创建发布分支
git checkout develop
git checkout -b release/1.0.0

# 5. 测试和修复
git commit -m "fix: fix bug found in testing"

# 6. 合并到 main 和 develop，打标签
git checkout main
git merge release/1.0.0
git tag -a v1.0.0 -m "Release 1.0.0"

git checkout develop
git merge release/1.0.0
git branch -d release/1.0.0

# 7. 从 main 创建紧急修复分支
git checkout main
git checkout -b hotfix/1.0.1

# 8. 修复 bug
git commit -m "fix: fix critical bug in production"

# 9. 合并到 main 和 develop，打标签
git checkout main
git merge hotfix/1.0.1
git tag -a v1.0.1 -m "Hotfix 1.0.1"

git checkout develop
git merge hotfix/1.0.1
git branch -d hotfix/1.0.1
```

> 💡 **面试可答**：Git Flow 适用于有明确发布周期的项目；分支：main（主分支）、develop（开发分支）、feature/*（功能分支）、release/*（发布分支）、hotfix/*（紧急修复）。

### 🔹 7.4.2 GitHub Flow

GitHub Flow 是一种简化的分支策略，适用于持续部署的项目。

**分支类型**：

| 分支 | 说明 | 生命周期 |
|------|------|----------|
| `main` | 主分支，始终可部署 | 长期 |
| `feature/*` | 功能分支，开发新功能 | 临时 |

**工作流程**：

```bash
# 1. 从 main 创建功能分支
git checkout main
git checkout -b feature/login

# 2. 开发功能
git add .
git commit -m "feat(auth): add login page"

# 3. 推送到远程
git push -u origin feature/login

# 4. 创建 Pull Request，代码审查

# 5. 合并到 main
git checkout main
git merge feature/login
git branch -d feature/login

# 6. 部署
git push origin main
```

> 💡 **面试可答**：GitHub Flow 适用于持续部署的项目；分支：main（主分支，始终可部署）、feature/*（功能分支）；通过 Pull Request 代码审查。

### 🔹 7.4.3 Git Flow vs GitHub Flow

| 对比项 | Git Flow | GitHub Flow |
|--------|----------|-------------|
| **适用场景** | 有明确发布周期 | 持续部署 |
| **主分支** | main、develop | main |
| **发布流程** | release 分支 | 直接合并到 main |
| **复杂度** | 复杂 | 简单 |
| **团队规模** | 大型团队 | 小型团队 |

> 💡 **面试可答**：Git Flow 适用于有明确发布周期的大型项目；GitHub Flow 适用于持续部署的小型项目。

## 7.5 最佳实践

### 🔹 7.5.1 代码审查

```bash
# 1. 创建功能分支
git checkout -b feature/login

# 2. 开发功能
git add .
git commit -m "feat(auth): add login page"

# 3. 推送到远程
git push -u origin feature/login

# 4. 创建 Pull Request / Merge Request

# 5. 代码审查

# 6. 根据反馈修改
git add .
git commit -m "fix: address review comments"
git push

# 7. 合并到主分支
```

> 💡 **面试可答**：代码审查通过 Pull Request / Merge Request 进行；审查代码质量、逻辑、安全性。

### 🔹 7.5.2 提交前检查

```bash
# 1. 查看修改内容
git diff

# 2. 查看暂存区内容
git diff --staged

# 3. 查看提交历史
git log --oneline -5

# 4. 运行测试
npm test

# 5. 提交
git commit -m "feat(auth): add login page"
```

> 💡 **面试可答**：提交前检查修改内容、运行测试；避免提交错误代码。

### 🔹 7.5.3 分支管理

```bash
# 定期清理已合并的分支
git branch --merged | grep -v 'main\|develop' | xargs git branch -d

# 定期清理远程已删除的分支
git remote prune origin

# 定期清理无用的 stash
git stash clear
```

> 💡 **面试可答**：定期清理已合并的分支和无用的 stash；保持仓库整洁。

### 🔹 7.5.4 安全最佳实践

```bash
# 不要提交敏感信息（密码、密钥等）
# 使用 .gitignore 忽略敏感文件

# 不要提交大文件（超过 100MB）
# 使用 Git LFS 管理大文件

# 不要在提交信息中包含敏感信息
git commit -m "feat: add feature"  # ✅
git commit -m "feat: add API key abc123"  # ❌

# 使用 SSH 密钥而不是 HTTPS 密码
git remote set-url origin git@github.com:user/repo.git
```

> 💡 **面试可答**：不要提交敏感信息；使用 .gitignore 忽略敏感文件；使用 Git LFS 管理大文件；使用 SSH 密钥而不是 HTTPS 密码。

## 7.6 面试高频题

### Q1：Git 配置的三个级别？

> **标准回答**：
> - `--local`：当前仓库，配置文件 `.git/config`
> - `--global`：当前用户，配置文件 `~/.gitconfig`
> - `--system`：系统所有用户，配置文件 `/etc/gitconfig`
> - 优先级：local > global > system

### Q2：常用的 Git 钩子？

> **标准回答**：
> - `pre-commit`：提交前执行，用于代码检查（如 ESLint）
> - `commit-msg`：提交信息编辑后执行，用于检查提交信息格式
> - `pre-push`：推送前执行，用于运行测试
> - `post-receive`：接收推送后执行，用于部署

### Q3：Conventional Commits 规范？

> **标准回答**：
> - 格式：`<type>(<scope>): <subject>`
> - 常用类型：feat（新功能）、fix（修复 bug）、docs（文档）、style（格式）、refactor（重构）、test（测试）、chore（构建工具）
> - 示例：`feat(auth): add login page`

### Q4：Git Flow 和 GitHub Flow 的区别？

> **标准回答**：
> - **Git Flow**：适用于有明确发布周期的项目，分支包括 main、develop、feature/*、release/*、hotfix/*
> - **GitHub Flow**：适用于持续部署的项目，分支包括 main、feature/*
> - Git Flow 复杂，GitHub Flow 简单

### Q5：Git 最佳实践？

> **标准回答**：
> - 小步提交，每次提交一个逻辑单元
> - 使用 Conventional Commits 规范提交信息
> - 代码审查通过 Pull Request / Merge Request
> - 定期清理已合并的分支和无用的 stash
> - 不要提交敏感信息，使用 .gitignore 忽略敏感文件

## ⚠️ 常见坑与注意点

| 现象 / 易错点 | 原因 | 怎么改 / 怎么记 |
|---------------|------|-----------------|
| 不配置用户信息 | 提交时无法识别提交者 | 记住：**必须配置 user.name 和 user.email** |
| 提交信息不清晰 | 不使用 Conventional Commits 规范 | 记住：**使用 type(scope): subject 格式** |
| 一次性提交大量修改 | 不便于代码审查和回退 | 记住：**小步提交，每次提交一个逻辑单元** |
| 提交敏感信息 | 未使用 .gitignore 忽略敏感文件 | 记住：**不要提交密码、密钥等敏感信息** |
| 钩子未生效 | 钩子文件未添加执行权限 | 记住：**chmod +x .git/hooks/* 添加执行权限** |

------

**学习要点**：

- Git 配置有三个级别：local（当前仓库）、global（当前用户）、system（系统所有用户）
- Git 钩子是在特定事件发生时自动执行的脚本，常用钩子：pre-commit（代码检查）、commit-msg（提交信息规范）
- Conventional Commits 规范提交信息格式：type(scope): subject
- Git Flow 适用于有明确发布周期的项目，GitHub Flow 适用于持续部署的项目
- 最佳实践：小步提交、代码审查、定期清理分支、不提交敏感信息
- 学完本章已掌握 Git 的完整使用流程，从个人配置到团队协作的最佳实践
