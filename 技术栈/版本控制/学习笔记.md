# 📘《Git 学习与实战手册》

> 系统学习见下文各章；日常常用命令与使用场景速查见同目录《常用命令与使用场景》。

------

# 🌱 第一章：Git 概述与核心概念

------

## 1.1 什么是 Git

**Git** 是**分布式版本控制系统**，由 Linus Torvalds 于 2005 年创建，用于管理 Linux 内核开发。每个开发者本地都拥有完整仓库副本，可离线提交、分支与合并。

**与 SVN 等集中式版本控制的对比**：

- **无单点故障**：本地即完整仓库，不依赖中央服务器即可工作
- **分支轻量**：分支本质是指针，创建与切换极快
- **支持离线**：提交、查看历史、分支操作均可离线完成
- **基于内容寻址**：使用 SHA-1 哈希标识对象，保证完整性

Git 是当前主流版本控制工具，校招与团队协作必备。

------

## 1.2 核心概念：工作区、暂存区、仓库

**工作区（Working Directory）**：当前看到的文件目录，即项目根目录下除 `.git` 外的所有文件。你对文件的修改首先发生在这里。

**暂存区（Staging Area / Index）**：`git add` 后，文件进入暂存区。暂存区是“下一次提交”的预备区，可选择性加入部分修改。

**本地仓库（Local Repository）**：`.git` 目录，存储完整提交历史、分支、标签等。`git commit` 将暂存区内容固化为一次提交。

**三者关系**：工作区 → `git add` → 暂存区 → `git commit` → 本地仓库。

------

## 1.3 提交与分支

**提交（Commit）**：暂存区的快照形成一次提交，包含唯一 SHA、作者、时间、提交信息。每次提交指向其父提交，形成有向无环图（DAG）。

**分支（Branch）**：指向某次提交的指针。默认分支通常为 `main` 或 `master`。创建分支只是新建一个指针，成本极低。

**HEAD**：指向当前所在分支的指针，即“当前工作位置”。

------

## 1.4 远程仓库

**远程仓库（Remote）**：如 GitHub、GitLab、Gitee 等托管平台上的仓库。通过 `push`、`pull`、`fetch` 与本地同步。

**常见协作流程**：克隆 → 本地修改 → 提交 → 推送；或 fork → 拉取 → 修改 → 发起 Pull Request（PR）。

------

# 🌱 第二章：本地基础操作

------

## 2.1 初始化与克隆

**新建仓库**：`git init` 在当前目录初始化 `.git`，建立空仓库。

**克隆远程仓库**：`git clone <url>` 克隆远程仓库到本地，包含完整历史。可指定目录：`git clone <url> <目录名>`。

------

## 2.2 状态、暂存与提交

**查看状态**：`git status` 显示工作区与暂存区状态，哪些文件未跟踪、已修改、已暂存。

**暂存**：`git add <file>` 或 `git add .` 将修改加入暂存区。`git add -p` 可交互式选择部分修改（hunk）加入。

**提交**：`git commit -m "message"` 将暂存区提交。`git commit --amend` 可修改最近一次提交（改信息或追加文件）；`git commit --amend --no-edit` 仅追加文件，不改信息。

------

## 2.3 查看历史与比较

**提交历史**：`git log` 查看完整历史；`git log --oneline` 单行简略；`git log -p` 带 diff；`git show <commit>` 查看某次提交详情与 diff。

**差异比较**：`git diff` 比较工作区与暂存区；`git diff --staged` 比较暂存区与最近提交；`git diff <branch>` 比较当前分支与指定分支。

------

## 2.4 忽略文件

**`.gitignore`**：书写忽略规则，支持通配符（`*.log`）、目录（`/target`、`node_modules/`）、取反（`!important.log`）。

**已跟踪文件**：若文件已被 Git 跟踪，加入 `.gitignore` 后仍会继续跟踪。需先 `git rm --cached <file>` 再提交，才能从跟踪中移除。

------

# 🌱 第三章：分支与合并

------

## 3.1 分支管理

**查看分支**：`git branch` 列出本地分支；`git branch -a` 列出本地与远程分支。

**创建与切换**：`git branch <name>` 创建；`git checkout <name>` 或 `git switch <name>` 切换；`git checkout -b <name>` 或 `git switch -c <name>` 创建并切换。

**删除分支**：`git branch -d <name>` 删除已合并分支；`git branch -D <name>` 强制删除。

------

## 3.2 合并（Merge）

**基本合并**：`git merge <branch>` 将指定分支合并到当前分支。

**快进合并（Fast-forward）**：若当前分支未产生新提交，则直接将指针移动到目标分支，不产生合并提交。

**三方合并（Three-way merge）**：若当前分支有独立提交，则产生新的合并提交，保留两个分支的历史。

**合并选项**：`git merge --no-ff` 强制产生合并提交，便于保留分支历史；`git merge --squash` 将目标分支的多次提交压缩为一次，再提交。

------

## 3.3 变基（Rebase）

**基本变基**：`git rebase <branch>` 将当前分支的提交“挪到”目标分支之后，历史呈直线，更清晰。

**交互式变基**：`git rebase -i <commit>` 可合并、重排、修改、删除提交，常用于整理提交历史。

**重要原则**：**不要对已推送到远程的公共分支做 rebase**，会改写他人依赖的历史，造成协作混乱。仅对未推送的个人分支使用。

------

## 3.4 冲突解决

**冲突产生**：merge 或 rebase 时，若两方修改了同一文件的同一区域，Git 无法自动合并，会标记冲突。

**解决流程**：
1. 打开冲突文件，找到 `<<<<<<<`、`=======`、`>>>>>>>` 标记
2. 手动编辑，保留需要的版本或合并两者
3. `git add <file>` 标记已解决
4. merge 时：`git commit` 完成合并；rebase 时：`git rebase --continue` 继续，或 `git rebase --abort` 放弃

------

## 3.5 分支策略（面试常考）

**Git Flow**：`main`（生产）、`develop`（开发）、`feature/*`（功能）、`release/*`（发布）、`hotfix/*`（紧急修复）。适合有固定发布周期的项目。

**GitHub Flow**：`main` 始终可部署，功能在 `feature/*` 分支开发，通过 PR 合并。适合持续部署、发布频繁的项目。

**简单策略**：`main` + `feature/xxx`，小团队常用。

------

# 🌱 第四章：远程协作

------

## 4.1 远程仓库配置

**查看远程**：`git remote -v` 查看远程仓库地址。

**添加与修改**：`git remote add origin <url>` 添加；`git remote set-url origin <url>` 修改地址；`git remote remove origin` 移除。

------

## 4.2 推送与拉取

**推送**：`git push origin <branch>` 推送本地分支；`git push -u origin <branch>` 首次推送并设置上游，之后可直接 `git push`。

**拉取**：`git pull` 相当于 `git fetch` + `git merge` 当前分支上游；`git pull --rebase` 用 rebase 代替 merge，保持历史线性。

**仅拉取不合并**：`git fetch origin` 只更新远程引用，不修改工作区。适合先查看再决定是否合并。

------

## 4.3 跟踪分支与上游

**跟踪分支（Tracking Branch）**：本地分支可设置上游（upstream）跟踪远程分支。设置后，`git push`、`git pull` 可省略分支名。

**设置上游**：`git branch --set-upstream-to=origin/<branch> <local-branch>` 或首次 `git push -u origin <branch>`。

------

## 4.4 Push 被拒绝时的处理

**原因**：远程分支已有他人推送的新提交，你的本地历史与远程分叉。

**推荐做法**：`git pull --rebase` 或 `git pull` 后再 `git push`。若已确定仅自己使用该分支，可 `git push --force`（慎用，会覆盖远程历史）。

------

## 4.5 Pull Request / Merge Request

**PR（GitHub）/ MR（GitLab）**：从 fork 或分支发起请求，将更改合并到目标分支。Code Review、CI 检查常在此完成。

**参与开源流程**：fork → clone → 修改 → commit → push → 在平台开 PR。

------

# 🌱 第五章：撤销与恢复

------

## 5.1 工作区与暂存区撤销

**丢弃工作区修改**：`git restore <file>` 或 `git checkout -- <file>`，未暂存的修改会丢失。

**取消暂存**：`git restore --staged <file>` 或 `git reset HEAD <file>`，文件从暂存区移除，工作区修改保留。

------

## 5.2 提交回退（Reset）

**`git reset --soft <commit>`**：只移动 HEAD，暂存区和工作区保留。常用于“撤销提交但保留修改”。

**`git reset --mixed <commit>`**：移动 HEAD 并清空暂存区，工作区保留。默认模式。

**`git reset --hard <commit>`**：彻底回到某提交，暂存区和工作区均被重置。**慎用**，未提交修改会丢失。

------

## 5.3 恢复某文件到某版本

**从某提交取出文件**：`git checkout <commit> -- <file>` 或 `git restore --source=<commit> <file>`，将指定提交中的文件覆盖到工作区。

------

## 5.4 已推送提交的“撤销”

**推荐：revert**：`git revert <commit>` 生成一次“反向”提交，抵消指定提交的改动。不改写历史，适合已推送的提交。

**慎用：reset + force push**：`git reset --hard <commit>` 后 `git push --force`。仅限个人分支或团队明确允许时使用，会改写远程历史。

------

## 5.5 reflog 与误操作恢复

**`git reflog`**：记录 HEAD 的移动历史，包括 reset、checkout、merge 等操作。即使提交“消失”，短期内仍可通过 reflog 找到。

**恢复误删分支**：`git reflog` 找到分支最后所在提交 → `git checkout -b <branch> <commit>` 重建分支。

**恢复误 reset**：`git reflog` 找到 reset 前的提交 → `git reset --hard <commit>` 恢复。

------

# 🌱 第六章：标签、贮藏与进阶

------

## 6.1 标签（Tag）

**轻量标签**：`git tag <name>`，仅是一个指向提交的引用。

**附注标签**：`git tag -a <name> -m "msg"`，包含打标签者、时间、说明等信息，适合发版。

**推送标签**：`git push origin <tagname>` 或 `git push origin --tags` 推送所有标签。

**用途**：发版标记（如 v1.0.0）、重要节点记录。

------

## 6.2 贮藏（Stash）

**基本用法**：`git stash` 将工作区与暂存区改动暂存；`git stash list` 查看；`git stash pop` 应用并移除；`git stash apply` 应用不移除。

**适用场景**：临时切分支、拉取前有未完成修改、需要快速切换上下文时。

**带说明贮藏**：`git stash push -m "说明"` 便于识别。

------

## 6.3 Cherry-pick

**`git cherry-pick <commit>`**：将指定提交“摘取”并应用到当前分支。用于选择性移植某次提交，而非合并整个分支。

**应用场景**：将某分支的单个修复提交应用到主分支；从 feature 分支挑选已完成的提交。

------

## 6.4 Bisect（二分查找）

**`git bisect start`**：开始二分查找，用于定位引入 bug 的提交。

**流程**：标记一个“坏”提交（有 bug）和一个“好”提交（无 bug），Git 自动切换到中间提交，你测试后标记 good/bad，重复直到找到引入 bug 的提交。

**适用场景**：不确定哪次提交引入问题，需要快速定位。

------

## 6.5 子模块（Submodule）简介

**Submodule**：在一个 Git 仓库中嵌套另一个 Git 仓库。常用于引用第三方库或内部公共模块。

**基本命令**：`git submodule add <url>` 添加；`git submodule update --init` 初始化并拉取子模块。克隆含子模块的仓库时，需 `git clone --recurse-submodules` 或单独 `git submodule update --init`。

------

# 🌱 第七章：配置、钩子与最佳实践

------

## 7.1 配置

**用户信息**：`git config --global user.name "xxx"`、`git config --global user.email "xxx"`，提交时使用。

**查看配置**：`git config --list`。配置层级：系统（`/etc/gitconfig`）、全局（`~/.gitconfig`）、仓库（`.git/config`），后者覆盖前者。

**凭据存储**：`git config --global credential.helper store` 或 `cache`，避免重复输入密码。

------

## 7.2 钩子（Hooks）

**位置**：`.git/hooks/` 下的脚本，如 `pre-commit`、`commit-msg`、`pre-push`。

**常见用途**：提交前运行 lint、测试；校验提交信息格式；推送前检查。

**团队实践**：常用 Husky + lint-staged 在客户端统一执行，保证提交质量。

------

## 7.3 最佳实践

**提交信息**：使用约定式提交（Conventional Commits），如 `feat: 新功能`、`fix: 修复 bug`、`docs: 文档更新`。便于生成 CHANGELOG、自动化版本号。

**提交粒度**：小步提交，每次提交只做一件事，便于回滚与 code review。

**分支命名**：`feature/xxx`、`bugfix/xxx`、`release/xxx`、`hotfix/xxx`，语义清晰。

**主分支保护**：不直接在主分支开发，通过 PR 合并；慎用 `force push`。

------

# 🌱 附录：面试要点速查

------

| 考点 | 回答要点 |
|------|----------|
| 工作区、暂存区、仓库 | 工作区 → add → 暂存区 → commit → 仓库；撤销时对应 restore、reset |
| merge 与 rebase 区别 | merge 保留分支历史、产生合并提交；rebase 历史线性、改写提交；不要对已推送公共分支 rebase |
| 冲突解决 | 编辑冲突文件 → add → commit（merge）或 rebase --continue |
| fetch 与 pull | fetch 只拉取不合并；pull = fetch + merge |
| reset 与 revert | reset 移动 HEAD，可改写历史；revert 新增反向提交，不改写历史，适合已推送 |
| 大文件 / 敏感信息 | 大文件用 Git LFS 或加入 .gitignore；敏感信息泄露需从历史中彻底移除（filter-branch / BFG） |
| 参与开源 | fork → clone → 改 → commit → push → 开 PR |
