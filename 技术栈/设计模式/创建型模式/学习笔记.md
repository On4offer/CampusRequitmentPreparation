------

## 第0章：设计原则（SOLID）

本章介绍面向对象设计的**五大原则**，是所有设计模式的理论基础。理解 SOLID 有助于理解"为什么需要这个模式"、"模式解决了什么问题"。面试中常问"设计原则有哪些"、"开闭原则是什么"。

------

### 1️⃣ SOLID 五大原则概览

| 原则 | 全称 | 一句话 | 核心思想 |
|------|------|--------|----------|
| **S**RP | Single Responsibility | 一个类只做一件事 | 职责单一，降低耦合 |
| **O**CP | Open/Closed | 对扩展开放，对修改关闭 | 抽象与多态实现扩展 |
| **L**SP | Liskov Substitution | 子类可替换父类 | 继承的正确使用 |
| **I**SP | Interface Segregation | 接口要小而专 | 避免胖接口 |
| **D**IP | Dependency Inversion | 依赖抽象不依赖具体 | 面向接口编程 |

------

### 2️⃣ 单一职责原则（SRP）

- **定义**：一个类应该**只有一个引起它变化的原因**，即一个类只负责一项职责。
- **为什么需要**：职责过多会导致类臃肿、修改一个职责影响其他职责、难以维护和测试。
- **示例**：一个 `UserManager` 类同时负责用户管理、日志记录、邮件发送 → 违反 SRP。应拆分为 `UserService`、`LogService`、`EmailService`。

```java
// 违反 SRP：一个类做太多事
class UserManager {
    void createUser() { }
    void log(String msg) { }
    void sendEmail() { }
}

// 符合 SRP：职责分离
class UserService { void createUser() { } }
class LogService { void log(String msg) { } }
class EmailService { void sendEmail() { } }
```

- **注意**：职责划分粒度要适度，过度拆分会导致类数量爆炸。

------

### 3️⃣ 开闭原则（OCP）

- **定义**：软件实体应该**对扩展开放，对修改关闭**。即在不修改已有代码的情况下扩展功能。
- **为什么需要**：修改已有代码风险大（可能引入 Bug），扩展新功能应该通过新增代码实现。
- **实现方式**：**抽象与多态**。定义稳定的抽象（接口/抽象类），新功能通过新实现类扩展。
- **示例**：支付系统新增支付方式时不修改原有代码。

```java
// 符合 OCP：新增支付方式只需新增实现类
interface PaymentStrategy { void pay(BigDecimal amount); }

class AlipayPayment implements PaymentStrategy {
    public void pay(BigDecimal amount) { System.out.println("支付宝支付"); }
}

class WechatPayment implements PaymentStrategy {
    public void pay(BigDecimal amount) { System.out.println("微信支付"); }
}

// 新增银行卡支付：只需新增类，不修改原有代码
class BankCardPayment implements PaymentStrategy {
    public void pay(BigDecimal amount) { System.out.println("银行卡支付"); }
}
```

- **设计模式体现**：策略模式、工厂模式、装饰器模式等都体现了 OCP。

------

### 4️⃣ 里氏替换原则（LSP）

- **定义**：**子类对象必须能够替换父类对象**，且程序行为正确。即继承必须保证"是一个"语义。
- **为什么需要**：违反 LSP 会导致多态使用时出现预期外的行为，破坏继承体系。
- **经典反例**：正方形继承矩形。正方形强制长宽相等，但矩形可以独立设置长宽，替换后行为不一致。

```java
// 违反 LSP 的经典例子
class Rectangle {
    protected int width, height;
    void setWidth(int w) { width = w; }
    void setHeight(int h) { height = h; }
    int getArea() { return width * height; }
}

class Square extends Rectangle {
    void setWidth(int w) { width = height = w; }  // 强制长宽相等
    void setHeight(int h) { width = height = h; }
}

// 使用时出问题
Rectangle r = new Square();
r.setWidth(5);
r.setHeight(4);
System.out.println(r.getArea());  // 期望 20，实际 16
```

- **正确做法**：正方形和矩形应该作为两个独立的类，或抽象出共同的父类 `Shape`。

------

### 5️⃣ 接口隔离原则（ISP）

- **定义**：客户端不应该依赖它不需要的接口。**接口要小而专**，不要设计"胖接口"。
- **为什么需要**：胖接口迫使实现类实现不需要的方法，导致耦合和冗余。
- **示例**：一个 `IWorker` 接口包含 `work()` 和 `eat()`，机器人不需要 `eat()` 但被迫实现。

```java
// 违反 ISP：胖接口
interface IWorker {
    void work();
    void eat();  // 机器人不需要
}

class HumanWorker implements IWorker {
    public void work() { System.out.println("工作"); }
    public void eat() { System.out.println("吃饭"); }
}

class RobotWorker implements IWorker {
    public void work() { System.out.println("工作"); }
    public void eat() { }  // 被迫实现空方法
}

// 符合 ISP：接口拆分
interface Workable { void work(); }
interface Eatable { void eat(); }

class HumanWorker implements Workable, Eatable {
    public void work() { System.out.println("工作"); }
    public void eat() { System.out.println("吃饭"); }
}

class RobotWorker implements Workable {
    public void work() { System.out.println("工作"); }
}
```

------

### 6️⃣ 依赖倒置原则（DIP）

- **定义**：高层模块不应依赖低层模块，**二者都应依赖抽象**。抽象不应依赖细节，细节应依赖抽象。
- **为什么需要**：直接依赖具体实现会导致高层模块与低层模块耦合，难以替换和测试。
- **示例**：业务层依赖数据访问接口而非具体实现。

```java
// 违反 DIP：高层直接依赖低层
class OrderService {
    private MySQLOrderDao dao = new MySQLOrderDao();  // 直接依赖具体实现
    void createOrder(Order order) { dao.save(order); }
}

// 符合 DIP：依赖抽象
interface OrderDao { void save(Order order); }

class MySQLOrderDao implements OrderDao {
    public void save(Order order) { System.out.println("保存到 MySQL"); }
}

class OrderService {
    private OrderDao dao;  // 依赖接口
    public OrderService(OrderDao dao) { this.dao = dao; }  // 依赖注入
    void createOrder(Order order) { dao.save(order); }
}
```

- **Spring 体现**：Spring 的依赖注入（DI）就是 DIP 的典型实现，通过 `@Autowired` 或构造器注入抽象类型。

------

### 📖 本章小结

| 原则 | 核心思想 | 设计模式体现 |
|------|----------|--------------|
| SRP | 单一职责、降低耦合 | 外观模式、代理模式 |
| OCP | 扩展开放、修改关闭 | 策略、工厂、装饰器 |
| LSP | 子类可替换父类 | 继承体系设计 |
| ISP | 接口小而专 | 适配器模式、接口拆分 |
| DIP | 依赖抽象不依赖具体 | Spring DI、策略模式 |

------

### ⚠️ 常见坑与注意点

1. **现象**：为了"符合原则"而过度设计。**原因**：原则是指导方针，不是教条。**正确做法**：根据实际需求权衡，小项目简单优先，大项目逐步重构。
2. **现象**：SRP 拆分粒度过细导致类爆炸。**正确做法**：职责划分要结合业务变化点，只有会一起变化的才放一起。
3. **现象**：LSP 只记定义不理解本质。**正确做法**：继承要保证"是一个"语义，子类不能破坏父类的行为契约。

## 第1章：创建型模式概述

本章建立对**创建型模式**的整体认知：解决“对象由谁创建、如何创建”的问题，与结构型（如何组织）、行为型（如何协作）并列。学完可继续第 2 章单例与后续各模式。

------

### 1️⃣ 什么是创建型模式

- **创建型模式**：关注**对象的创建**，将创建与使用解耦，使系统不依赖具体类的实例化方式。**为什么需要**：避免到处 `new`、便于替换实现、统一创建逻辑（如单例、池化、复杂构建）。
- GoF 创建型共五种：**单例**、**工厂方法**、**抽象工厂**、**建造者**、**原型**。校招与 Spring 中**单例、工厂**出现频率最高。

| 模式 | 一句话 | 典型场景 |
|------|--------|----------|
| 单例 | 保证类只有一个实例、全局访问点 | 配置、线程池、Spring 单例 Bean |
| 工厂方法 | 由子类决定创建哪种产品 | JDBC、BeanFactory、LoggerFactory |
| 抽象工厂 | 创建一族相关产品 | UI 主题、数据库访问层 |
| 建造者 | 分步构建复杂对象 | StringBuilder、Lombok @Builder |
| 原型 | 通过复制原型创建新对象 | 高成本对象克隆、状态快照 |

### 2️⃣ 与其它类型模式的关系

- **结构型**：关注对象组合与结构（代理、适配器、装饰器等）。**行为型**：关注对象间的职责与通信（观察者、策略、责任链等）。创建型解决“谁来建、怎么建”，为后续结构与行为提供对象来源。

------

### 📖 本章小结

| 知识点 | 面试关键词 | 实际应用 |
|--------|------------|----------|
| 创建型定义 | 对象创建、解耦、不依赖具体实例化 | 与结构型/行为型区分 |
| 五种模式 | 单例、工厂方法、抽象工厂、建造者、原型 | 能说出各自意图与一个应用 |

------

### ⚠️ 常见坑与注意点

1. **现象**：把“简单工厂”当 GoF 工厂方法。**原因**：简单工厂是 if/switch 创建，不属于 23 种；工厂方法是子类决定创建谁。**正确做法**：面试时说“工厂方法由子类决定实例化哪个类”，简单工厂可顺带对比。
2. **现象**：创建型只记名字不说意图。**原因**：面试要的是“解决什么问题”。**正确做法**：每个模式用“意图 + 一个场景”回答（如单例：保证唯一实例 + 配置管理器）。

------

### 🎯 面试常见追问

| 面试官提问 | 回答思路 |
|------------|----------|
| 创建型模式有哪些？各自解决什么问题？ | 单例（唯一实例）、工厂方法（子类决定创建谁）、抽象工厂（一族产品）、建造者（分步构建）、原型（克隆）。各举一例场景。若追问具体实现见对应章。 |

## 第2章：单例模式（Singleton）

本章解决**如何保证类只有一个实例并提供全局访问点**，是校招手写高频。与 Spring 单例 Bean、线程池等直接相关；实现上涉及线程安全与反射/反序列化防护。

------

### 1️⃣ 意图与适用场景

- **意图**：保证一个类**只有一个实例**，并提供**全局访问点**。**为什么需要**：节省资源、统一状态（如配置、连接池），避免多处 new 导致多实例。
- **应用场景**：线程池、配置管理、**Spring 默认 Bean 作用域**、连接池、日志 Logger。

### 2️⃣ 实现方式对比

| 方式 | 特点 | 线程安全 | 备注 |
|------|------|----------|------|
| 饿汉 | 类加载即创建实例 | 是（类加载保证） | 简单，无法懒加载 |
| 懒汉（同步方法） | 首次 get 时创建，方法加 synchronized | 是 | 每次获取都加锁，性能差 |
| **双重检查锁（DCL）** | 首次 get 时创建，锁内再判空，instance 用 volatile | 是 | 推荐，需 volatile 防指令重排 |
| **静态内部类** | 内部类持有一个静态实例，类加载时懒加载 | 是（JVM 保证） | 推荐，简洁且懒加载 |
| 枚举 | 枚举单例，天然单实例 | 是 | **可防反射破坏**，推荐 |

### 3️⃣ 示例：双重检查锁与静态内部类

```java
// 双重检查锁：volatile 防止指令重排导致“半初始化”对象被看到
public class Singleton {
    private static volatile Singleton instance;
    private Singleton() {}
    public static Singleton getInstance() {
        if (instance == null) {
            synchronized (Singleton.class) {
                if (instance == null) instance = new Singleton();
            }
        }
        return instance;
    }
}

// 静态内部类：懒加载 + 线程安全由类加载保证
public class Singleton {
    private Singleton() {}
    private static class Holder { static final Singleton INSTANCE = new Singleton(); }
    public static Singleton getInstance() { return Holder.INSTANCE; }
}
```

> 💬 **面试怎么说**：DCL 要写 volatile，否则可能拿到未完全初始化的对象；静态内部类利用“类只加载一次”保证安全且懒加载，手写推荐这两种或枚举。

### 4️⃣ 破坏与防护

- **反射**：通过反射调私有构造器可再建实例。**防护**：构造器内若 instance 已非 null 则抛异常；**枚举单例**天然防反射。
- **反序列化**：反序列化会生成新对象。**防护**：实现 `readResolve()` 返回已有 instance。
- **可测性**：硬编码单例不利于单元测试（难以替换）。**正确做法**：业务层尽量用依赖注入拿到“单例”，而不是静态 getInstance，便于 Mock。

------

### 📖 本章小结

| 知识点 | 面试关键词 | 实际应用 |
|--------|------------|----------|
| 单例意图 | 唯一实例、全局访问点 | Spring 单例 Bean、配置、池 |
| 实现方式 | 饿汉、懒汉、DCL、静态内部类、枚举 | 手写 DCL/静态内部类、说清 volatile 作用 |
| 破坏与防护 | 反射、反序列化、readResolve、枚举防反射 | 面试追问时能答 |

------

### ⚠️ 常见坑与注意点

1. **现象**：DCL 不写 volatile。**原因**：`instance = new Singleton()` 非原子，可能先分配内存再赋值，其他线程看到非 null 但对象未初始化完。**正确做法**：instance 必须用 **volatile** 禁止指令重排。
2. **现象**：懒汉只写 `synchronized (this)`。**原因**：getInstance 是静态方法，应锁类对象。**正确做法**：`synchronized (Singleton.class)`。
3. **现象**：单例里持有大量状态导致难以测试。**原因**：全局状态难以隔离。**正确做法**：能注入的尽量通过构造/ setter 注入，单例只做“唯一入口”，内部委托给可替换的依赖。

------

### 🎯 面试常见追问

| 面试官提问 | 回答思路 |
|------------|----------|
| 手写单例（线程安全）？ | 双重检查锁（带 volatile）或静态内部类；枚举单例防反射。能写出一种并说清为什么 volatile/为什么静态内部类安全。 |
| 为什么 DCL 要加 volatile？ | 防止指令重排：new 可能先赋值引用再执行构造器，其他线程可能拿到未初始化完的对象；volatile 保证可见性与禁止重排。若追问反射/反序列化见本章 4️⃣。 |

## 第3章：工厂方法模式（Factory Method）

本章解决**由子类决定创建哪种产品**，使客户端依赖抽象而非具体类，符合开闭原则。与第 4 章抽象工厂（一族产品）对比；Spring BeanFactory、JDBC、LoggerFactory 均为典型应用。

------

### 1️⃣ 意图与结构

- **意图**：定义创建对象的接口，**由子类决定实例化哪一个类**，使实例化推迟到子类。**为什么需要**：解耦创建与使用，新增产品时只需新增工厂子类，符合**开闭原则**。
- **结构**：**抽象产品**、**具体产品**；**抽象工厂**（含 factoryMethod）、**具体工厂**；客户端依赖抽象工厂与抽象产品。

### 2️⃣ 与简单工厂区别

| 维度 | 简单工厂 | 工厂方法 |
|------|----------|----------|
| 创建逻辑 | 工厂类内 if/switch 创建 | 子类实现 factoryMethod，决定创建谁 |
| 开闭原则 | 新增产品要改工厂类 | 新增产品只加具体工厂子类 |
| 典型写法 | 一个工厂类 + 多个分支 | 抽象工厂 + 多个具体工厂 |

### 3️⃣ 应用与示例

- **应用场景**：JDBC Driver、集合 Iterator、**Spring BeanFactory 的 getBean**、日志框架 **LoggerFactory**。
- **示例**：抽象工厂 `LoggerFactory`，子类 `FileLoggerFactory` / `ConsoleLoggerFactory` 分别创建 FileLogger、ConsoleLogger；客户端通过 `LoggerFactory` 拿 Logger，不依赖具体实现。

------

### 📖 本章小结

| 知识点 | 面试关键词 | 实际应用 |
|--------|------------|----------|
| 工厂方法意图 | 子类决定创建谁、推迟实例化、开闭原则 | BeanFactory、LoggerFactory |
| 与简单工厂 | if/switch vs 子类、开闭 | 面试对比答法 |

------

### ⚠️ 常见坑与注意点

1. **现象**：把“简单工厂”说成工厂方法。**正确做法**：工厂方法是“一个抽象工厂 + 多个具体工厂子类”，每个子类只创建一种产品；简单工厂是一个类里分支创建多种。
2. **现象**：认为工厂方法必须返回“产品接口”。**正确做法**：返回抽象产品类型即可，客户端依赖抽象；具体产品由具体工厂创建。

------

### 🎯 面试常见追问

| 面试官提问 | 回答思路 |
|------------|----------|
| 工厂方法和简单工厂区别？ | 简单工厂在工厂类里分支创建；工厂方法把“创建谁”交给子类，符合开闭。若追问抽象工厂见第 4 章。 |

## 第4章：抽象工厂模式（Abstract Factory）

本章解决**创建一族相关或相互依赖的对象**（如一套 UI 组件、一套数据库访问对象），客户端只依赖抽象工厂与抽象产品。与第 3 章工厂方法（一种产品）对比；UI 主题、数据库访问层为典型应用。

------

### 1️⃣ 意图与结构

- **意图**：提供创建**一系列相关或相互依赖对象**的接口，无需指定具体类。**为什么需要**：保证一族产品配套使用（如同一主题的按钮+文本框），避免混用不同风格。
- **结构**：多个**抽象产品**（如 Button、TextBox）、多个**具体产品**；**抽象工厂**（多个 create 方法）、**具体工厂**（一族产品）；一个工厂负责一族。

### 2️⃣ 与工厂方法区别

| 维度 | 工厂方法 | 抽象工厂 |
|------|----------|----------|
| 产品 | 一种产品 | **一族**产品（多个等级/类型） |
| 工厂方法数 | 一个 create | 多个 create（每个产品一个） |

### 3️⃣ 应用场景

UI 主题（按钮+文本框一族）、数据库访问（Connection+Statement 一族）、跨平台 GUI 组件。**示例**：`ThemeFactory` 有 `createButton()`、`createTextBox()`；`DarkThemeFactory` 返回 DarkButton、DarkTextBox。

------

### 📖 本章小结

| 知识点 | 面试关键词 | 实际应用 |
|--------|------------|----------|
| 抽象工厂意图 | 一族产品、配套创建、不指定具体类 | UI 主题、数据库访问层 |
| 与工厂方法 | 一种 vs 一族、多个 create 方法 | 面试对比 |

------

### ⚠️ 常见坑与注意点

1. **现象**：抽象工厂和工厂方法混用。**正确做法**：工厂方法 = 一个工厂生产一种产品；抽象工厂 = 一个工厂生产多种**相关**产品（一族）。
2. **现象**：新增产品类型（如新增“边框”）要改抽象工厂接口。**正确做法**：抽象工厂的缺点就是“增加产品族容易，增加产品等级难”，需改接口与所有具体工厂。

------

### 🎯 面试常见追问

| 面试官提问 | 回答思路 |
|------------|----------|
| 抽象工厂和工厂方法区别？ | 工厂方法一个工厂一种产品；抽象工厂一个工厂一族产品（多个 create），如 UI 主题下按钮+文本框。 |

## 第5章：建造者模式（Builder）

本章解决**分步构建复杂对象**，使构建过程与表示分离，支持链式调用与可选参数。与工厂（关注“创建哪种”）不同，建造者关注“如何一步步建”。StringBuilder、Lombok @Builder、MyBatis Builder 为典型应用。

------

### 1️⃣ 意图与结构

- **意图**：将复杂对象的**构建与表示分离**，同一构建过程可创建不同表示。**为什么需要**：构造器参数过多、可选参数多时，建造者可读性更好且避免 telescoping 构造器。
- **结构**：产品类、**建造者**（定义构建步骤，链式 return this）、**Director**（可选，封装构建顺序）。常见简化：产品内部静态 Builder 类 + 链式 setter。

### 2️⃣ 与工厂区别

| 维度 | 工厂 | 建造者 |
|------|------|--------|
| 关注点 | 创建**哪种**对象 | **分步构建**复杂对象 |
| 典型场景 | 多态、子类决定类型 | 多参数、可选参数、链式调用 |

### 3️⃣ 应用与示例

**应用场景**：StringBuilder、**Lombok @Builder**、MyBatis SqlSessionFactoryBuilder、AlertDialog、复杂配置对象。

```java
// Lombok @Builder 等价于手写：静态内部 Builder、链式方法、build() 构造目标对象
@Builder
public class Config {
    private String host;
    private int port;
    private boolean ssl;
}
// 使用：Config.builder().host("localhost").port(8080).ssl(true).build();
```

------

### 📖 本章小结

| 知识点 | 面试关键词 | 实际应用 |
|--------|------------|----------|
| 建造者意图 | 构建与表示分离、分步构建、链式调用 | @Builder、StringBuilder、复杂配置 |
| 与工厂 | 哪种 vs 怎么建、可选参数 | 面试对比 |

------

### ⚠️ 常见坑与注意点

1. **现象**：建造者每步都 new 新对象。**正确做法**：建造者内部持有一个“待构建对象”或若干字段，build() 时一次性构造并返回；链式调用 return this。
2. **现象**：必填参数未在 build() 里校验。**正确做法**：build() 内对必填项做校验，缺则抛 IllegalStateException。

------

### 🎯 面试常见追问

| 面试官提问 | 回答思路 |
|------------|----------|
| 建造者解决什么问题？ | 复杂对象分步构建、多可选参数、链式调用；与工厂区别是工厂关心“创建哪种”，建造者关心“怎么建”。 |

## 第6章：原型模式（Prototype）

本章解决**通过复制已有对象创建新对象**，适用于创建成本高或需要大量相似对象的场景。与克隆、深拷贝/浅拷贝直接相关；Spring prototype 作用域可类比。

------

### 1️⃣ 意图与实现

- **意图**：用**原型实例**指定创建对象的种类，**通过复制该原型**创建新对象。**为什么需要**：避免重复高成本创建、需要对象副本或状态快照时。
- **实现**：实现 **Cloneable**，重写 **clone()**；注意 **浅拷贝**（只复制引用）与 **深拷贝**（递归或序列化/工具复制内部对象）。Java clone **不调用构造器**，且默认浅拷贝。

### 2️⃣ 深拷贝与浅拷贝

| 类型 | 含义 | 实现 |
|------|------|------|
| 浅拷贝 | 只复制字段值，引用类型仍指向原对象 | Object.clone() 默认 |
| 深拷贝 | 引用类型也递归复制或序列化后反序列化 | 手动递归、JSON 拷贝、BeanUtils、序列化 |

### 3️⃣ 应用与注意点

**应用场景**：对象创建成本高、大量相似对象、状态保存与恢复；Spring **prototype 作用域** Bean 每次 getBean 新建，可类比“按需复制”思路。**注意**：Java clone 有缺陷（浅拷贝、不调构造器）；实际常用 **JSON 拷贝、BeanUtils 或手动拷贝** 替代。

------

### 📖 本章小结

| 知识点 | 面试关键词 | 实际应用 |
|--------|------------|----------|
| 原型意图 | 复制原型创建新对象、避免高成本创建 | 快照、prototype Bean |
| 深/浅拷贝 | clone 默认浅、深拷贝需递归或序列化 | 面试必答、实际用 JSON/BeanUtils 多 |

------

### ⚠️ 常见坑与注意点

1. **现象**：直接 clone() 以为得到深拷贝。**原因**：Object.clone() 默认浅拷贝。**正确做法**：重写 clone() 时对引用类型做深拷贝（递归 clone 或 new 新对象），或改用序列化/JSON/BeanUtils。
2. **现象**：clone 后修改新对象却影响原对象。**原因**：浅拷贝共享引用。**正确做法**：需要独立副本时必须深拷贝。

------

### 🎯 面试常见追问

| 面试官提问 | 回答思路 |
|------------|----------|
| 深拷贝和浅拷贝区别？如何实现深拷贝？ | 浅拷贝只复制引用；深拷贝递归复制或序列化/反序列化、JSON 拷贝、BeanUtils。Java clone 默认浅拷贝。若追问原型模式见本章意图。 |

## 第7章：实战与面试要点

本章把创建型模式与 **Spring** 及面试选择原则串起来，便于从“用”的角度复习。依赖前六章各模式。

------

### 1️⃣ Spring 中的体现

| 模式 | Spring 中的体现 |
|------|-----------------|
| 单例 | 默认 Bean 作用域为 singleton；单例池缓存 Bean |
| 原型 | prototype 作用域，每次 getBean 新建 |
| 工厂方法 | BeanFactory / ApplicationContext 的 getBean；FactoryBean 自定义创建逻辑 |
| 建造者 | Builder 风格配置（如 Security 链式配置） |

### 2️⃣ 选择原则

| 需求 | 选用模式 |
|------|----------|
| 唯一实例、全局访问 | 单例 |
| 解耦创建、子类决定类型 | 工厂方法 / 抽象工厂 |
| 分步构建、多可选参数 | 建造者 |
| 克隆、低成本复制 | 原型 |

### 3️⃣ 面试高频小结

手写单例（DCL、静态内部类、枚举）；工厂方法与抽象工厂区别；建造者解决什么问题；深拷贝与浅拷贝、如何实现深拷贝。

------

### 📖 本章小结

| 知识点 | 面试关键词 | 实际应用 |
|--------|------------|----------|
| Spring 与创建型 | 单例/原型作用域、BeanFactory、FactoryBean、Builder 配置 | 读 Spring 源码、答“Spring 用了哪些创建型” |
| 选择原则 | 唯一→单例、解耦创建→工厂、分步构建→建造者、克隆→原型 | 面试选型题 |

------

### 🎯 面试常见追问

| 面试官提问 | 回答思路 |
|------------|----------|
| Spring 里用到了哪些创建型模式？ | 单例（默认作用域）、原型（prototype）、工厂方法（BeanFactory getBean、FactoryBean）。若追问具体实现见前几章或 Spring 笔记。 |
