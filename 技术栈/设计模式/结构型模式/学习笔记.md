# 📘《结构型模式学习与实战手册》

> 系统学习见下文各章；日常常用概念与使用场景速查见同目录《常用概念与使用场景》。  
> 适用：GoF 结构型七种，面向校招与 Spring/MyBatis 等框架（代理、适配器、装饰器、外观等）。

------

## 第1章：结构型模式概述

本章建立对**结构型模式**的整体认知：解决“对象如何组织、包装、适配、复用”的问题，与创建型（对象从哪来）、行为型（如何协作）并列。校招与框架中**代理、装饰器、适配器**出现最多。

------

### 1️⃣ 什么是结构型模式

- **结构型模式**：关注**对象与类的组合**，形成更大结构，保持灵活与可扩展。**为什么需要**：解耦接口与实现、复用与扩展能力、统一复杂子系统调用。
- GoF 结构型共七种：**适配器**、**桥接**、**组合**、**装饰器**、**外观**、**享元**、**代理**。

| 模式 | 一句话 | 典型场景 |
|------|--------|----------|
| 代理 | 为对象提供代理，控制访问 | AOP、权限、延迟加载、RPC、MyBatis Mapper |
| 适配器 | 将不兼容接口转为客户期望的接口 | 旧接口对接、SLF4J、HandlerAdapter |
| 装饰器 | 动态给对象增加职责，同接口增强 | Java I/O 流、Servlet Wrapper |
| 外观 | 为子系统提供统一高层接口 | 网关、门面服务、SDK 封装 |
| 桥接/组合/享元 | 抽象与实现分离/树形一致/共享细粒度对象 | JDBC、文件树、池化 |

### 2️⃣ 与创建型、行为型的关系

- **创建型**：对象从哪来。**结构型**：对象如何组织、包装、复用。**行为型**：对象如何协作与分工。

------

### 📖 本章小结

| 知识点 | 面试关键词 | 实际应用 |
|--------|------------|----------|
| 结构型定义 | 对象组合、组装/适配/复用 | 与创建型/行为型区分 |
| 七种模式 | 代理、适配器、装饰器、外观等 | 能说出意图与一个应用，尤其代理/装饰器/适配器 |

------

### ⚠️ 常见坑与注意点

1. **现象**：代理和装饰器完全混为一谈。**正确做法**：都持有一个内部对象、实现同一接口；代理侧重**控制访问**（权限、延迟、中介），装饰器侧重**增强功能**（加一层能力）；面试时说清侧重点即可。
2. **现象**：适配器和装饰器混淆。**正确做法**：适配器**改接口**（不兼容→兼容）；装饰器**同接口增强**（不改变接口）。

------

### 🎯 面试常见追问

| 面试官提问 | 回答思路 |
|------------|----------|
| 结构型模式有哪些？代理和装饰器区别？ | 七种：适配器、桥接、组合、装饰器、外观、享元、代理。代理控制访问，装饰器增强功能；都持有一对象、同接口。若追问动态代理见第 2 章。 |

## 第2章：代理模式（Proxy）

本章解决**通过代理控制对真实对象的访问**，是 Spring AOP、MyBatis Mapper、RPC 等的基础。与装饰器（增强功能）对比；静态代理与动态代理（JDK/CGLib）为面试高频。

------

### 1️⃣ 意图与类型

- **意图**：为对象提供**代理**，以**控制对它的访问**；客户端通过代理间接访问真实对象。**为什么需要**：在不改目标类的前提下做权限控制、延迟加载、日志、事务等。
- **类型**：**静态代理**（手写代理类，实现与目标相同接口）；**动态代理**（运行时生成代理：**JDK 基于接口**、**CGLib 基于子类**）。**Spring AOP** 底层即动态代理（默认 JDK，无接口用 CGLib）。

### 2️⃣ JDK 动态代理与 CGLib 区别

| 维度 | JDK 动态代理 | CGLib |
|------|--------------|-------|
| 依据 | 基于**接口**，代理实现同一接口 | 基于**子类**，继承目标类 |
| 限制 | 目标必须有接口 | 目标不能 final、方法不能 final |
| 性能 | 调用略快（生成类更简单） | 生成子类，首次慢、调用略慢（视版本） |

### 3️⃣ 应用场景

权限控制、延迟加载、日志/监控、**RPC 存根**、**Spring 事务与 AOP**、**MyBatis Mapper 接口**（Mapper 无实现类，由代理生成实现）。

### 4️⃣ 与装饰器区别

| 维度 | 代理 | 装饰器 |
|------|------|--------|
| 目的 | **控制访问**（能不能访问、怎么访问） | **增强功能**（加一层能力） |
| 接口 | 同接口 | 同接口 |
| 典型 | 权限、延迟、AOP | I/O 流包装、Wrapper 增强 |

------

### 📖 本章小结

| 知识点 | 面试关键词 | 实际应用 |
|--------|------------|----------|
| 代理意图 | 控制访问、间接访问 | AOP、事务、Mapper、RPC |
| 静态 vs 动态 | 手写代理类 vs 运行时生成 | 能说清 JDK/CGLib 区别 |
| 与装饰器 | 控制访问 vs 增强功能 | 面试对比 |

------

### ⚠️ 常见坑与注意点

1. **现象**：认为 Spring AOP 一定用 CGLib。**正确做法**：有接口时默认用 **JDK 动态代理**，无接口时用 **CGLib**；可配置强制 CGLib。
2. **现象**：CGLib 代理 final 类或 final 方法。**正确做法**：CGLib 通过子类实现，无法代理 final；目标需可被继承、方法需可重写。

------

### 🎯 面试常见追问

| 面试官提问 | 回答思路 |
|------------|----------|
| 静态代理和动态代理区别？ | 静态手写代理类；动态运行时生成代理，JDK 基于接口、CGLib 基于子类。Spring AOP 用动态代理。 |
| JDK 动态代理和 CGLib 区别？ | JDK 基于接口、目标需有接口；CGLib 基于子类、不能 final。若追问 Spring 用哪个见本章 2️⃣。 |

## 第3章：适配器模式（Adapter）

本章解决**将不兼容的接口转换为客户期望的接口**，使旧类与新系统协作。与装饰器（同接口增强）严格区分；SLF4J、InputStreamReader、Spring HandlerAdapter 为典型应用。

------

### 1️⃣ 意图与结构

- **意图**：将**不兼容的接口**转换为客户**期望的接口**，使原本无法一起工作的类可以协作。**为什么需要**：复用已有类但接口与当前系统不一致时，不修改原有类而加一层适配。
- **结构**：**目标接口**（客户依赖）、**适配器**（实现目标接口并持有**被适配者**）、**被适配者**（原有接口）。**类适配器**（继承被适配者）、**对象适配器**（组合被适配者，常用）。

### 2️⃣ 应用场景与示例

**应用场景**：旧接口与新系统对接、**SLF4J** 与各日志实现（统一 API，适配 log4j/logback）、**InputStreamReader**（字节流→字符流）、**Spring MVC HandlerAdapter**（将各种 Controller 适配成统一的 ModelAndView 处理流程）。

> 💬 **面试怎么说**：适配器是“接口转换”，把 A 接口转成 B 接口；装饰器是“同接口加功能”，不换接口。

### 3️⃣ 与装饰器区别

| 维度 | 适配器 | 装饰器 |
|------|--------|--------|
| 目的 | **接口转换**（不兼容→兼容） | **同接口增强**（不改变接口） |
| 客户看到的接口 | 与原来不同（目标接口） | 与原来相同 |

------

### 📖 本章小结

| 知识点 | 面试关键词 | 实际应用 |
|--------|------------|----------|
| 适配器意图 | 接口转换、不兼容→兼容、被适配者 | SLF4J、HandlerAdapter、InputStreamReader |
| 与装饰器 | 换接口 vs 同接口增强 | 面试必对比 |

------

### ⚠️ 常见坑与注意点

1. **现象**：适配器里既转换接口又加很多业务逻辑。**正确做法**：适配器只做“转调”，把目标接口调用转成对被适配者的调用；复杂逻辑应放在别处。
2. **现象**：和装饰器混淆。**正确做法**：适配器**一定改变**客户看到的接口；装饰器与组件**同一接口**。

------

### 🎯 面试常见追问

| 面试官提问 | 回答思路 |
|------------|----------|
| 适配器和装饰器区别？ | 适配器做接口转换，使不兼容变兼容；装饰器同接口增强，不改变接口。若追问代理见第 2 章。 |

------

## 第4章：装饰器模式（Decorator）

本章解决**动态给对象增加职责**，与组件同接口、可层层包装，比继承更灵活。Java I/O 流、Servlet Wrapper 为典型；与代理（控制访问）对比。

------

### 1️⃣ 意图与结构

- **意图**：**动态地**给对象增加职责，比继承更灵活；装饰器与组件**实现同一接口**，可层层包装。**为什么需要**：避免子类爆炸、运行时组合能力而非编译期继承。
- **结构**：**抽象组件**、**具体组件**、**抽象装饰器**（持有一个组件引用，实现同一接口）、**具体装饰器**。Java I/O 中 **InputStream / FilterInputStream** 是典型（FilterInputStream 持有一个 InputStream）。

### 2️⃣ 应用场景

**Java I/O 流**：BufferedInputStream 装饰 InputStream，增加缓冲；**Servlet** 中 HttpServletRequestWrapper 装饰 Request，增强请求/响应。

### 3️⃣ 与代理区别

都持有一个内部对象、同接口；**装饰器**强调“层层加功能”；**代理**强调“控制访问、中介”。边界可模糊，面试时说清侧重点。

------

### 📖 本章小结

| 知识点 | 面试关键词 | 实际应用 |
|--------|------------|----------|
| 装饰器意图 | 同接口增强、层层包装、动态职责 | I/O 流、Wrapper、增强请求 |
| 与代理 | 增强功能 vs 控制访问 | 面试对比 |

------

### ⚠️ 常见坑与注意点

1. **现象**：装饰器里改变了对外的接口。**正确做法**：装饰器必须与组件**同一接口**，否则变成适配器。
2. **现象**：多层装饰时顺序影响结果。**正确做法**：装饰器顺序有意义（如先压缩再加密 vs 先加密再压缩），设计时要考虑顺序。

------

### 🎯 面试常见追问

| 面试官提问 | 回答思路 |
|------------|----------|
| 装饰器和代理区别？ | 都持有一对象、同接口；装饰器侧重增强功能、层层包装；代理侧重控制访问。举 I/O 流为装饰器、AOP 为代理。 |

------

## 第5章：外观模式（Facade）

本章解决**为子系统提供统一高层接口**，降低客户端与子系统的耦合、简化调用。与代理（代表一个对象）、适配器（转换一个接口）区分；网关、门面服务为典型。

------

### 1️⃣ 意图与结构

- **意图**：为**子系统中的一组接口**提供**统一的高层接口**，降低子系统与客户端的耦合，简化调用。**为什么需要**：客户端不需要知道多个子模块的细节，只调门面即可。
- **结构**：**外观类**封装多个子系统的调用，客户端只依赖外观；不改变子系统，只增加一层门面。

### 2️⃣ 应用与对比

**应用场景**：API 网关对外统一入口、封装多个服务调用的门面服务、SDK 对底层多模块的封装。**与代理/适配器**：外观是“**简化一群接口**”；代理是“代表**一个**对象”；适配器是“转换**一个**接口”。

------

### 📖 本章小结

| 知识点 | 面试关键词 | 实际应用 |
|--------|------------|----------|
| 外观意图 | 统一高层接口、简化多子系统调用 | 网关、门面服务、SDK |
| 与代理/适配器 | 一群 vs 一个、简化 vs 代表/转换 | 面试对比 |

------

### 🎯 面试常见追问

| 面试官提问 | 回答思路 |
|------------|----------|
| 外观和代理、适配器区别？ | 外观简化一群接口；代理代表一个对象控制访问；适配器转换一个接口。 |

## 第6章：桥接模式（Bridge）

本章解决**抽象与实现分离**的问题，使二者可独立变化，避免多维度组合导致的类爆炸。与适配器（接口转换）不同，桥接关注"抽象与实现解耦"。JDBC 驱动架构是典型应用。

------

### 1️⃣ 意图与结构

- **意图**：将**抽象部分与实现部分分离**，使它们都可以独立地变化。**为什么需要**：避免多维度组合导致的类爆炸（如形状×颜色×材质 = 大量子类）。
- **结构**：**抽象（Abstraction）** 定义高层接口，持有**实现（Implementor）** 引用；**具体抽象（RefinedAbstraction）** 扩展抽象；**具体实现（ConcreteImplementor）** 实现实现接口。

```
Abstraction ──持有──> Implementor
     ↑                    ↑
RefinedAbstraction    ConcreteImplementor
```

### 2️⃣ 类爆炸问题示例

假设有形状（圆形、矩形）和颜色（红色、蓝色），不用桥接：

```
RedCircle, BlueCircle, RedRectangle, BlueRectangle
// 新增形状×颜色×材质 → 类数量 = 形状数 × 颜色数 × 材质数
```

使用桥接后：形状和颜色独立变化，类数量 = 形状数 + 颜色数。

### 3️⃣ 示例：形状与颜色

```java
// 实现接口：颜色
interface Color {
    void applyColor();
}

class RedColor implements Color {
    public void applyColor() { System.out.println("红色"); }
}

class BlueColor implements Color {
    public void applyColor() { System.out.println("蓝色"); }
}

// 抽象：形状（持有颜色引用）
abstract class Shape {
    protected Color color;  // 桥接点：持有实现引用
    public Shape(Color color) { this.color = color; }
    abstract void draw();
}

class Circle extends Shape {
    public Circle(Color color) { super(color); }
    void draw() {
        System.out.print("圆形，");
        color.applyColor();
    }
}

class Rectangle extends Shape {
    public Rectangle(Color color) { super(color); }
    void draw() {
        System.out.print("矩形，");
        color.applyColor();
    }
}

// 使用：形状和颜色独立变化
Shape redCircle = new Circle(new RedColor());
Shape blueRect = new Rectangle(new BlueColor());
```

### 4️⃣ 应用场景

- **JDBC 架构**：`Connection` 是抽象，各数据库驱动（MySQL、Oracle）是实现；Java 应用通过 JDBC API（抽象）操作，底层驱动（实现）可替换。
- **消息发送**：消息类型（文本、图片）是抽象，发送渠道（邮件、短信、推送）是实现。
- **UI 主题**：控件（按钮、文本框）是抽象，主题风格（亮色、暗色）是实现。
- **跨平台 GUI**：窗口/按钮是抽象，不同平台的渲染是实现。

### 5️⃣ 与适配器区别

| 维度 | 桥接 | 适配器 |
|------|------|--------|
| 目的 | **抽象与实现分离**，独立变化 | **接口转换**，使不兼容变兼容 |
| 使用时机 | 设计阶段，预先解耦 | 事后补救，对接已有类 |
| 结构 | 抽象持有实现引用 | 适配器持有被适配者 |

------

### 📖 本章小结

| 知识点 | 面试关键词 | 实际应用 |
|--------|------------|----------|
| 桥接意图 | 抽象与实现分离、独立变化、避免类爆炸 | JDBC、消息渠道、UI 主题 |
| 结构 | Abstraction 持有 Implementor | 能画出结构图 |
| 与适配器 | 设计阶段解耦 vs 事后接口转换 | 面试对比 |

------

### ⚠️ 常见坑与注意点

1. **现象**：把桥接和适配器混淆。**正确做法**：桥接是**设计阶段**预先解耦抽象与实现；适配器是**事后补救**转换接口。
2. **现象**：桥接过度使用导致设计复杂。**正确做法**：只有当有**多维度独立变化**的需求时才用桥接；单维度变化不需要。

------

### 🎯 面试常见追问

| 面试官提问 | 回答思路 |
|------------|----------|
| 桥接模式解决什么问题？ | 抽象与实现分离，避免多维度组合导致类爆炸。如形状×颜色，不用桥接要 N×M 个类，用桥接只需 N+M 个。 |
| 桥接和适配器区别？ | 桥接是设计阶段预先解耦；适配器是事后补救转换接口。JDBC 用桥接，HandlerAdapter 用适配器。 |

## 第7章：组合模式（Composite）

本章解决**树形结构的一致处理**问题，使客户端可以统一对待单个对象和组合对象。文件系统、菜单树、组织架构为典型应用。

------

### 1️⃣ 意图与结构

- **意图**：将对象组合成**树形结构**以表示"部分-整体"层次，使客户端可以**一致地处理单个对象和组合对象**。**为什么需要**：统一叶子节点和容器节点的操作，简化客户端代码。
- **结构**：**抽象组件（Component）** 定义统一接口；**叶子（Leaf）** 无子节点；**组合（Composite）** 持有子组件列表，实现 add/remove/getChild。

```
        Component（抽象组件）
       /              \
    Leaf（叶子）    Composite（组合）
                        |
                   List<Component>
```

### 2️⃣ 示例：文件系统

```java
// 抽象组件：文件系统节点
abstract class FileSystemNode {
    protected String name;
    public FileSystemNode(String name) { this.name = name; }
    abstract void display(int depth);
    public void add(FileSystemNode node) { throw new UnsupportedOperationException(); }
    public void remove(FileSystemNode node) { throw new UnsupportedOperationException(); }
}

// 叶子：文件
class File extends FileSystemNode {
    public File(String name) { super(name); }
    void display(int depth) {
        System.out.println("-".repeat(depth) + name);
    }
}

// 组合：文件夹
class Folder extends FileSystemNode {
    private List<FileSystemNode> children = new ArrayList<>();
    public Folder(String name) { super(name); }
    
    void display(int depth) {
        System.out.println("-".repeat(depth) + name + "/");
        for (FileSystemNode child : children) {
            child.display(depth + 2);
        }
    }
    
    public void add(FileSystemNode node) { children.add(node); }
    public void remove(FileSystemNode node) { children.remove(node); }
}

// 使用：统一对待文件和文件夹
FileSystemNode root = new Folder("root");
root.add(new File("readme.txt"));
FileSystemNode src = new Folder("src");
src.add(new File("Main.java"));
root.add(src);
root.display(0);
// 输出：
// root/
// --readme.txt
// --src/
// ----Main.java
```

### 3️⃣ 透明式与安全式

| 方式 | 特点 | 优缺点 |
|------|------|--------|
| **透明式** | Component 定义 add/remove，Leaf 抛异常 | 客户端统一，但 Leaf 有无意义方法 |
| **安全式** | 只有 Composite 定义 add/remove | 客户端需判断类型，但更安全 |

上例为透明式：Leaf 继承 add/remove 但抛异常。

### 4️⃣ 应用场景

- **文件系统**：文件和文件夹统一对待
- **菜单树**：菜单项和子菜单统一操作
- **组织架构**：员工和部门统一处理
- **GUI 组件**：容器和控件统一管理

------

### 📖 本章小结

| 知识点 | 面试关键词 | 实际应用 |
|--------|------------|----------|
| 组合意图 | 树形结构、部分-整体、一致对待 | 文件系统、菜单树、组织架构 |
| 结构 | Component、Leaf、Composite | 能画出树形结构 |
| 透明式 vs 安全式 | Component 定义 add/remove vs 仅 Composite | 面试可对比 |

------

### ⚠️ 常见坑与注意点

1. **现象**：Leaf 实现了 add/remove 但不知道怎么处理。**正确做法**：透明式下 Leaf 抛 `UnsupportedOperationException`；安全式下 Leaf 不继承这些方法。
2. **现象**：组合模式用于非树形结构。**正确做法**：只有**部分-整体层次结构**才适合组合模式，普通对象关系不需要。

------

### 🎯 面试常见追问

| 面试官提问 | 回答思路 |
|------------|----------|
| 组合模式解决什么问题？ | 树形结构一致处理：客户端统一对待叶子（单个对象）和组合（容器对象），如文件系统、菜单树。 |
| 透明式和安全式区别？ | 透明式：Component 定义 add/remove，Leaf 抛异常，客户端统一。安全式：只有 Composite 定义，客户端需判断类型。 |

## 第8章：享元模式（Flyweight）

本章解决**共享细粒度对象以节省内存**的问题，通过区分内部状态（共享）和外部状态（传入）实现对象复用。池化技术、Integer 缓存是典型应用。

------

### 1️⃣ 意图与结构

- **意图**：运用**共享技术**有效地支持大量细粒度对象，减少内存占用。**为什么需要**：大量相似对象会消耗大量内存，共享不变部分可大幅减少对象数量。
- **核心概念**：
  - **内部状态（Intrinsic）**：可共享、不随环境变化的部分（如棋子的颜色、类型）
  - **外部状态（Extrinsic）**：不可共享、随环境变化的部分（如棋子的位置）
- **结构**：**享元工厂（FlyweightFactory）** 管理享元对象池；**享元接口（Flyweight）** 定义操作；**具体享元（ConcreteFlyweight）** 包含内部状态。

### 2️⃣ 示例：围棋棋子

```java
// 享元接口
interface ChessPiece {
    void display(int x, int y);  // 位置是外部状态
}

// 具体享元：只保存内部状态（颜色）
class ConcreteChessPiece implements ChessPiece {
    private String color;  // 内部状态：黑或白
    
    public ConcreteChessPiece(String color) { this.color = color; }
    
    public void display(int x, int y) {  // x, y 是外部状态
        System.out.println(color + "棋子位置：(" + x + ", " + y + ")");
    }
}

// 享元工厂：管理对象池
class ChessPieceFactory {
    private static Map<String, ChessPiece> pool = new HashMap<>();
    
    public static ChessPiece getChessPiece(String color) {
        if (!pool.containsKey(color)) {
            pool.put(color, new ConcreteChessPiece(color));
        }
        return pool.get(color);
    }
    
    public static int getPoolSize() { return pool.size(); }
}

// 使用：300 个棋子只需 2 个对象（黑、白各一个）
ChessPiece black = ChessPieceFactory.getChessPiece("黑");
ChessPiece white = ChessPieceFactory.getChessPiece("白");
black.display(1, 1);   // 黑棋位置：(1, 1)
white.display(2, 2);   // 白棋位置：(2, 2)
System.out.println("对象数量：" + ChessPieceFactory.getPoolSize());  // 2
```

### 3️⃣ 应用场景

- **Integer 缓存**：`Integer.valueOf(int)` 缓存 -128~127 的对象
- **String 常量池**：相同字符串共享同一对象
- **线程池、连接池**：复用昂贵对象
- **文本编辑器**：字符字形共享，位置是外部状态

### 4️⃣ Integer 缓存示例

```java
Integer a = Integer.valueOf(127);
Integer b = Integer.valueOf(127);
System.out.println(a == b);  // true，使用缓存

Integer c = Integer.valueOf(128);
Integer d = Integer.valueOf(128);
System.out.println(c == d);  // false，超出缓存范围
```

------

### 📖 本章小结

| 知识点 | 面试关键词 | 实际应用 |
|--------|------------|----------|
| 享元意图 | 共享细粒度对象、节省内存 | 池化、Integer 缓存、String 池 |
| 内部/外部状态 | 内部共享、外部传入 | 能区分两种状态 |
| 享元工厂 | 对象池、getOrCreate | 理解工厂管理逻辑 |

------

### ⚠️ 常见坑与注意点

1. **现象**：把所有状态都当内部状态共享。**正确做法**：只有**不变的部分**才能作为内部状态共享；变化的部分必须作为外部状态传入。
2. **现象**：享元对象是可变的。**正确做法**：享元对象应该是**不可变**的，否则一个地方修改会影响所有共享该对象的地方。
3. **现象**：Integer 缓存范围记错。**正确做法**：默认 -128~127，可通过 `-XX:AutoBoxCacheMax` 调整上限。

------

### 🎯 面试常见追问

| 面试官提问 | 回答思路 |
|------------|----------|
| 享元模式解决什么问题？ | 大量细粒度对象占用内存，通过共享不变部分（内部状态）减少对象数量。如围棋 300 棋子只需 2 个享元对象。 |
| 什么是内部状态和外部状态？ | 内部状态：可共享、不变的部分。外部状态：不可共享、随环境变化的部分，由客户端传入。 |
| Integer 缓存是享元吗？ | 是。`Integer.valueOf(int)` 缓存 -128~127 的对象，复用相同值的 Integer。 |

## 第9章：实战与面试要点

### 1️⃣ Spring 中的体现

| 模式 | Spring 中的体现 |

| 简化多子系统调用 | 外观 |
