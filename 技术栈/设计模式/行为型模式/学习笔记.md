# 📘《行为型模式学习与实战手册》

> 系统学习见下文各章；日常常用概念与使用场景速查见同目录《常用概念与使用场景》。  
> 适用：GoF 行为型十一种，面向校招与 Spring/业务开发（观察者、策略、模板方法、责任链等）。

------

## 第1章：行为型模式概述

本章建立对**行为型模式**的整体认知：解决“对象间如何协作、职责如何划分、流程如何组织”的问题，与创建型（对象从哪来）、结构型（如何组织）并列。校招与业务中最常考**观察者、策略、模板方法、责任链**。

------

### 1️⃣ 什么是行为型模式

- **行为型模式**：关注**对象间的职责划分与通信**，描述“谁做什么、谁通知谁、谁先谁后”，解决算法与职责的分配问题。**为什么需要**：把变化的行为和稳定的流程分离、解耦发送方与接收方、统一多步处理流程等。
- GoF 行为型共十一种：**责任链**、**命令**、**解释器**、**迭代器**、**中介者**、**备忘录**、**观察者**、**状态**、**策略**、**模板方法**、**访问者**。

| 模式 | 一句话 | 典型场景 |
|------|--------|----------|
| 观察者 | 一对多依赖，主题变则通知所有观察者 | 事件、Spring 事件、Vue 响应式 |
| 策略 | 算法族可互换，与使用解耦 | 支付方式、排序比较器、Resource |
| 模板方法 | 骨架固定，步骤推迟到子类 | JdbcTemplate、Servlet、JUnit |
| 责任链 | 请求沿链传递，多对象可处理 | Filter、Interceptor、审批流 |
| 命令/状态/迭代器等 | 封装请求/状态驱动行为/统一遍历 | GUI 命令、订单状态、集合 Iterator |

### 2️⃣ 与创建型、结构型的关系

- **创建型**：提供对象。**结构型**：组织、包装对象。**行为型**：定义对象间的**协作与流程**，是“动”的部分。

------

### 📖 本章小结

| 知识点 | 面试关键词 | 实际应用 |
|--------|------------|----------|
| 行为型定义 | 职责划分、通信、协作与流程 | 与创建型/结构型区分 |
| 高频四种 | 观察者、策略、模板方法、责任链 | 能说出意图与一个应用 |

------

### ⚠️ 常见坑与注意点

1. **现象**：策略和状态混淆。**正确做法**：**策略**由**客户端/外部**在运行时注入或选择；**状态**由**对象内部**根据当前状态切换，状态自己决定下一状态。
2. **现象**：观察者和发布/订阅混为一谈。**正确做法**：观察者里主题直接持有观察者列表；发布/订阅通过**事件总线或 MQ** 解耦，发布者与订阅者互不知对方。

------

### 🎯 面试常见追问

| 面试官提问 | 回答思路 |
|------------|----------|
| 行为型模式有哪些？观察者和发布订阅区别？ | 十一种；观察者主题直接维护观察者并通知；发布/订阅通过中间件解耦，发布者与订阅者不直接依赖。若追问观察者实现见第 2 章。 |

## 第2章：观察者模式（Observer）

本章解决**一对多依赖**：主题状态变化时自动通知所有观察者，是事件驱动、Spring 事件、前端响应式的基础。与发布/订阅（通过中间件解耦）对比。

------

### 1️⃣ 意图与结构

- **意图**：定义**一对多**依赖，多个观察者监听**主题**，主题状态变化时**自动通知**所有观察者。**为什么需要**：解耦“谁变了”和“谁要反应”，便于扩展新观察者而不改主题。
- **结构**：**主题（Subject）** 维护观察者列表，提供 attach/detach/notify；**观察者（Observer）** 提供 update。**推模型**（主题推送数据）与**拉模型**（观察者从主题拉取）。

### 2️⃣ 应用场景

事件监听、消息队列消费、**Vue/React 响应式与订阅**、**Spring 事件**（ApplicationEventPublisher / @EventListener）、GUI 数据绑定。

### 3️⃣ 与发布/订阅区别

| 维度 | 观察者 | 发布/订阅 |
|------|--------|-----------|
| 耦合 | 主题**直接持有**观察者列表 | 通过**事件总线 / MQ**，发布者与订阅者互不知 |
| 典型 | 内存中 Subject 调 Observer.update | 消息中间件、EventBus |

------

### 📖 本章小结

| 知识点 | 面试关键词 | 实际应用 |
|--------|------------|----------|
| 观察者意图 | 一对多、主题通知观察者、推/拉 | Spring 事件、前端响应式 |
| 与发布订阅 | 直接依赖 vs 中间件解耦 | 面试必对比 |

------

### ⚠️ 常见坑与注意点

1. **现象**：notify 时在遍历列表里增删观察者导致并发修改异常。**正确做法**：先拷贝一份列表再遍历通知，或使用 CopyOnWriteArrayList 等。
2. **现象**：观察者里抛异常导致后续观察者收不到通知。**正确做法**：主题 notify 时 try-catch 每个观察者，或约定观察者内部消化异常，避免一个观察者拖垮整体。

------

### 🎯 面试常见追问

| 面试官提问 | 回答思路 |
|------------|----------|
| 观察者和发布订阅区别？ | 观察者主题直接维护并通知观察者；发布/订阅通过事件总线或 MQ，发布者与订阅者解耦。若追问 Spring 事件见本章应用。 |

## 第3章：策略模式（Strategy）

本章解决**算法族可互换、与使用解耦**，客户端通过注入不同策略实现不同行为，避免大量 if/switch。与状态模式（内部切换）、模板方法（继承+步骤）对比；支付方式、排序比较器、Spring Resource 为典型。

------

### 1️⃣ 意图与结构

- **意图**：定义**一系列算法**并封装，使它们**可互换**；将算法与使用算法的客户端**解耦**。**为什么需要**：避免 if/switch 膨胀、便于新增算法（开闭原则）、便于测试与替换。
- **结构**：**策略接口**、多个**具体策略**、**上下文 Context** 持有策略并委托执行；**客户端**选择策略注入 Context。

### 2️⃣ 应用场景

支付方式（支付宝/微信/银行卡）、排序/比较器（Comparator）、路由规则、折扣计算、**Spring Resource**（UrlResource、ClassPathResource 等多种实现）。

### 3️⃣ 与状态模式区别

| 维度 | 策略 | 状态 |
|------|------|------|
| 谁决定用哪个 | **客户端/外部**指定并注入 | **对象内部**根据当前状态切换 |
| 切换 | 外部替换策略 | 状态自己决定下一状态 |

------

### 📖 本章小结

| 知识点 | 面试关键词 | 实际应用 |
|--------|------------|----------|
| 策略意图 | 算法可互换、解耦、避免 if/switch | 支付、比较器、Resource |
| 与状态 | 外部指定 vs 内部切换 | 面试对比 |

------

### ⚠️ 常见坑与注意点

1. **现象**：策略类里依赖大量外部状态导致难以复用。**正确做法**：策略尽量无状态或只依赖入参；若需上下文，通过参数传入而非静态/单例。
2. **现象**：策略和模板方法混淆。**正确做法**：策略是**组合**（Context 持有 Strategy），模板方法是**继承**（子类重写步骤）；策略多用组合少用继承。

------

### 🎯 面试常见追问

| 面试官提问 | 回答思路 |
|------------|----------|
| 策略模式解决什么问题？和 if/switch 比有什么好处？ | 算法族可互换、与使用解耦；新增算法不用改客户端、符合开闭、易测试。若追问与状态/模板方法区别见本章 3️⃣、第 4 章。 |

## 第4章：模板方法模式（Template Method）

本章解决**算法骨架固定、部分步骤由子类实现**，用继承实现“流程不变、步骤可变”。与策略（组合替换算法）对比；JdbcTemplate、Servlet、JUnit 为典型。

------

### 1️⃣ 意图与结构

- **意图**：在**父类**中定义**算法骨架**，将某些**步骤推迟到子类**实现，子类可重写步骤而**不改变流程**。**为什么需要**：复用公共流程、避免重复代码、扩展点清晰。
- **结构**：**抽象类**定义 templateMethod()，其中调用抽象方法 step1()、step2() 等；**子类**实现这些抽象方法（或重写钩子方法）。

### 2️⃣ 应用场景

**JUnit** 的 setUp/tearDown、**Servlet** 的 service 与 doGet/doPost、**Spring JdbcTemplate** 的 execute（固定流程：取连接、执行、释放；可变的是 SQL 与参数）、抽象类定义流程框架。

### 3️⃣ 与策略区别

| 维度 | 模板方法 | 策略 |
|------|----------|------|
| 实现方式 | **继承** + 重写步骤 | **组合** + 替换策略 |
| 扩展点 | 子类重写抽象/钩子方法 | 注入不同策略实现 |

------

### 📖 本章小结

| 知识点 | 面试关键词 | 实际应用 |
|--------|------------|----------|
| 模板方法意图 | 骨架固定、步骤推迟子类、继承 | JdbcTemplate、Servlet、JUnit |
| 与策略 | 继承 vs 组合、步骤 vs 算法 | 面试对比 |

------

### ⚠️ 常见坑与注意点

1. **现象**：子类重写 templateMethod 把流程改乱。**正确做法**：templateMethod 通常为 final 或约定不重写，只重写步骤方法；若需改变流程应考虑策略或责任链。
2. **现象**：模板方法里步骤过多导致子类负担大。**正确做法**：提供默认实现或钩子方法，子类只重写关心的步骤。

------

### 🎯 面试常见追问

| 面试官提问 | 回答思路 |
|------------|----------|
| 模板方法和策略模式区别？ | 模板方法用继承、骨架固定步骤可变；策略用组合、算法可整体替换。JdbcTemplate 是模板方法，支付方式选型是策略。若追问 Spring 见第 7 章。 |

## 第5章：责任链模式（Chain of Responsibility）

本章解决**请求沿链传递、多个对象都有机会处理**，常用于过滤器链、审批流、拦截器。与观察者（一对多通知）不同，责任链是“一条链、依次处理”。

------

### 1️⃣ 意图与结构

- **意图**：**多个对象**都有机会处理请求，请求**沿链传递**直到某对象处理或链结束。**为什么需要**：解耦发送方与多个处理方、可动态调整链顺序、符合单一职责（每个节点只处理一类请求）。
- **结构**：**抽象处理者**（含 **next** 引用与 handleRequest）、**具体处理者**；客户端**组链**并提交请求。处理者中若自己处理则结束，否则 next.handle(request)。

### 2️⃣ 应用场景

**过滤器链**（Servlet Filter、**Spring Interceptor**）、审批流程、异常处理层级、日志/权限链式处理。

### 3️⃣ 注意点

链的**顺序**影响结果；是否**必须有人处理**（否则请求落空）；请求是否可被**多个节点**处理（通常一个处理即结束，也可设计为每节点都执行）。

------

### 📖 本章小结

| 知识点 | 面试关键词 | 实际应用 |
|--------|------------|----------|
| 责任链意图 | 沿链传递、多对象可处理、next | Filter、Interceptor、审批流 |
| 与观察者 | 链式依次处理 vs 一对多通知 | 面试对比 |

------

### ⚠️ 常见坑与注意点

1. **现象**：忘记设置 next 导致链断裂。**正确做法**：组链时明确 setNext；或在构造/配置中把整条链串好。
2. **现象**：链过长或节点职责过重。**正确做法**：每个节点只做一类事；链长度与顺序要在配置/代码里清晰可维护。

------

### 🎯 面试常见追问

| 面试官提问 | 回答思路 |
|------------|----------|
| 责任链怎么实现？Spring 里哪用到了？ | 抽象处理者持 next，handle 里要么处理完要么转 next；Spring 的 Interceptor、Filter 链就是责任链。若追问与观察者区别见本章 3️⃣。 |

## 第6章：命令模式（Command）

本章解决**将请求封装为对象**的问题，使请求的发送者与接收者解耦，支持撤销、排队、日志等操作。GUI 菜单命令、任务队列、事务操作为典型应用。

------

### 1️⃣ 意图与结构

- **意图**：将**请求封装为对象**，从而可用不同的请求对客户进行参数化、对请求排队或记录日志、支持撤销操作。**为什么需要**：解耦调用者与执行者，便于扩展新命令而不修改调用者。
- **结构**：**命令接口（Command）** 定义 execute 方法；**具体命令（ConcreteCommand）** 持有**接收者（Receiver）** 并实现 execute；**调用者（Invoker）** 持有命令并调用 execute；**接收者（Receiver）** 执行实际逻辑。

```
Invoker（调用者）──持有──> Command（命令接口）
                              ↑
                    ConcreteCommand（具体命令）
                              │
                              └──持有──> Receiver（接收者）
```

### 2️⃣ 示例：遥控器控制电灯

```java
// 接收者：电灯
class Light {
    void on() { System.out.println("灯开了"); }
    void off() { System.out.println("灯关了"); }
}

// 命令接口
interface Command {
    void execute();
    void undo();  // 支持撤销
}

// 具体命令：开灯
class LightOnCommand implements Command {
    private Light light;
    public LightOnCommand(Light light) { this.light = light; }
    
    public void execute() { light.on(); }
    public void undo() { light.off(); }  // 撤销开灯 = 关灯
}

// 具体命令：关灯
class LightOffCommand implements Command {
    private Light light;
    public LightOffCommand(Light light) { this.light = light; }
    
    public void execute() { light.off(); }
    public void undo() { light.on(); }  // 撤销关灯 = 开灯
}

// 调用者：遥控器
class RemoteControl {
    private Command command;
    private Command undoCommand;  // 记录上一个命令用于撤销
    
    public void setCommand(Command command) {
        this.command = command;
    }
    
    public void pressButton() {
        command.execute();
        undoCommand = command;  // 记录用于撤销
    }
    
    public void pressUndo() {
        if (undoCommand != null) {
            undoCommand.undo();
        }
    }
}

// 使用
Light light = new Light();
RemoteControl remote = new RemoteControl();
remote.setCommand(new LightOnCommand(light));
remote.pressButton();   // 灯开了
remote.pressUndo();     // 灯关了（撤销）
```

### 3️⃣ 应用场景

- **GUI 菜单/按钮**：菜单项、按钮点击执行对应命令，支持撤销/重做
- **任务队列**：命令对象放入队列，工作线程取出执行（解耦提交与执行）
- **事务操作**：每个操作封装为命令，支持提交/回滚
- **宏命令**：多个命令组合成一个宏，一次性执行

### 4️⃣ 与策略模式区别

| 维度 | 命令 | 策略 |
|------|------|------|
| 目的 | **封装请求**，解耦调用者与执行者 | **封装算法**，可互换 |
| 关注点 | "做什么"（请求本身） | "怎么做"（算法实现） |
| 典型场景 | 撤销、排队、日志 | 算法选择、替代 if/switch |

------

### 📖 本章小结

| 知识点 | 面试关键词 | 实际应用 |
|--------|------------|----------|
| 命令意图 | 请求封装为对象、解耦调用者与执行者 | GUI 命令、任务队列、事务 |
| 结构 | Command、Invoker、Receiver | 能画出结构图 |
| 撤销支持 | undo 方法、记录上一命令 | 面试追问 |

------

### ⚠️ 常见坑与注意点

1. **现象**：命令对象持有过多状态导致内存占用大。**正确做法**：命令对象只保存必要的状态（如接收者引用、参数），避免保存大对象。
2. **现象**：撤销操作实现复杂。**正确做法**：简单撤销可以反向操作（开灯→关灯）；复杂撤销需要保存状态快照（结合备忘录模式）。

------

### 🎯 面试常见追问

| 面试官提问 | 回答思路 |
|------------|----------|
| 命令模式解决什么问题？ | 将请求封装为对象，解耦调用者与执行者，支持撤销、排队、日志。如遥控器按钮、菜单命令。 |
| 命令模式和策略模式区别？ | 命令封装"请求"，关注做什么；策略封装"算法"，关注怎么做。命令支持撤销/排队，策略用于算法选择。 |

## 第7章：状态模式（State）

本章解决**对象根据内部状态改变行为**的问题，将状态相关的行为分散到不同状态类中，避免大量 if/switch 判断当前状态。订单状态机、工作流引擎为典型应用。

------

### 1️⃣ 意图与结构

- **意图**：允许对象在**内部状态改变时改变其行为**，对象看起来好像修改了它的类。**为什么需要**：避免大量 if/switch 判断当前状态，将每个状态的行为封装到独立类中，符合开闭原则。
- **结构**：**上下文（Context）** 持有当前状态；**状态接口（State）** 定义行为；**具体状态（ConcreteState）** 实现对应状态的行为，并可切换到下一状态。

```
Context（上下文）──持有──> State（状态接口）
     ↑                        ↑
  setState()         ConcreteStateA / ConcreteStateB
```

### 2️⃣ 示例：订单状态机

```java
// 状态接口
interface OrderState {
    void pay(OrderContext order);
    void ship(OrderContext order);
    void complete(OrderContext order);
    void cancel(OrderContext order);
}

// 上下文：订单
class OrderContext {
    private OrderState state;
    
    public OrderContext() { state = new NewOrderState(); }  // 初始状态：新订单
    
    public void setState(OrderState state) { this.state = state; }
    
    public void pay() { state.pay(this); }
    public void ship() { state.ship(this); }
    public void complete() { state.complete(this); }
    public void cancel() { state.cancel(this); }
}

// 具体状态：新订单
class NewOrderState implements OrderState {
    public void pay(OrderContext order) {
        System.out.println("支付成功，状态变为已支付");
        order.setState(new PaidState());
    }
    public void ship(OrderContext order) { System.out.println("未支付，不能发货"); }
    public void complete(OrderContext order) { System.out.println("未支付，不能完成"); }
    public void cancel(OrderContext order) {
        System.out.println("订单已取消");
        order.setState(new CancelledState());
    }
}

// 具体状态：已支付
class PaidState implements OrderState {
    public void pay(OrderContext order) { System.out.println("已支付，请勿重复支付"); }
    public void ship(OrderContext order) {
        System.out.println("发货成功，状态变为已发货");
        order.setState(new ShippedState());
    }
    public void complete(OrderContext order) { System.out.println("未发货，不能完成"); }
    public void cancel(OrderContext order) {
        System.out.println("退款成功，订单已取消");
        order.setState(new CancelledState());
    }
}

// 具体状态：已发货
class ShippedState implements OrderState {
    public void pay(OrderContext order) { System.out.println("已支付，请勿重复支付"); }
    public void ship(OrderContext order) { System.out.println("已发货，请勿重复发货"); }
    public void complete(OrderContext order) {
        System.out.println("订单完成");
        order.setState(new CompletedState());
    }
    public void cancel(OrderContext order) { System.out.println("已发货，需退货才能取消"); }
}

// 具体状态：已完成、已取消（省略实现）

// 使用
OrderContext order = new OrderContext();
order.pay();       // 支付成功，状态变为已支付
order.ship();      // 发货成功，状态变为已发货
order.complete();  // 订单完成
```

### 3️⃣ 与策略模式区别

| 维度 | 状态 | 策略 |
|------|------|------|
| 谁决定用哪个 | **对象内部**根据当前状态切换 | **客户端/外部**指定并注入 |
| 切换方式 | 状态自己决定下一状态 | 外部替换策略对象 |
| 状态感知 | 状态间知道彼此存在 | 策略间互不知道 |

### 4️⃣ 应用场景

- **订单状态机**：待支付 → 已支付 → 已发货 → 已完成/已取消
- **TCP 连接状态**：LISTEN → SYN_SENT → ESTABLISHED → FIN_WAIT
- **工作流引擎**：审批流程、请假流程
- **游戏角色状态**：站立 → 跑步 → 跳跃 → 攻击

| 知识点 | 面试关键词 | 实际应用 |
|--------|------------|----------|
| 状态意图 | 内部状态改变行为、状态类封装行为 | 订单状态机、工作流 |
| 结构 | Context、State、ConcreteState | 能画出状态转换图 |
| 与策略 | 内部切换 vs 外部注入 | 面试必对比 |

### ⚠️ 常见坑与注意点

1. **现象**：状态类过多导致类爆炸。**正确做法**：简单状态机可用 if/switch 或枚举实现；复杂状态机才用状态模式。
2. **现象**：状态转换逻辑分散在各状态类中难以维护。**正确做法**：可以用状态转换表集中管理转换规则，状态类只负责行为。

## 第8章：迭代器、中介者与备忘录（简述）

本章简述**迭代器、中介者、备忘录**三种行为型模式，校招出现频率低于观察者/策略/模板方法/责任链/命令/状态，但需知意图与典型应用。

------

### 1️⃣ 迭代器（Iterator）

- **意图**：提供一种方法**顺序访问聚合对象中的元素**，又不暴露该对象的内部表示。**为什么需要**：解耦遍历算法与聚合对象，统一遍历接口。
- **结构**：**迭代器接口（Iterator）** 定义 hasNext/next；**具体迭代器** 实现遍历逻辑；**聚合接口（Aggregate）** 定义 createIterator。
- **应用**：Java 集合的 `Iterator`、`ListIterator`；`for-each` 循环底层用迭代器。
- **注意**：迭代过程中修改集合可能抛 `ConcurrentModificationException`。

```java
List<String> list = Arrays.asList("a", "b", "c");
Iterator<String> it = list.iterator();
while (it.hasNext()) {
    System.out.println(it.next());
}
// for-each 等价于上述代码
```

### 2️⃣ 中介者（Mediator）

- **意图**：用一个**中介对象封装一系列对象的交互**，中介者使各对象不需要显式相互引用，从而降低耦合。**为什么需要**：多对多交互导致网状依赖，中介者将其变为星形结构。
- **结构**：**中介者接口（Mediator）** 定义交互方法；**具体中介者** 协调各同事对象；**同事（Colleague）** 只与中介者交互。
- **应用**：聊天室（中介者转发消息）、MVC 中 Controller（协调 Model 和 View）、GUI 组件交互。

```
无中介者：A ←→ B ←→ C ←→ A（网状）
有中介者：A ←→ Mediator ←→ B
              ↑
              C（星形）
```

### 3️⃣ 备忘录（Memento）

- **意图**：在不破坏封装的前提下**捕获对象的内部状态**，并在之后恢复该状态。**为什么需要**：实现撤销、回滚、快照功能。
- **结构**：**备忘录（Memento）** 存储状态；**原发器（Originator）** 创建和恢复备忘录；**负责人类（Caretaker）** 保管备忘录。
- **应用**：文本编辑器撤销、游戏存档、事务回滚。
- **注意**：备忘录可能暴露内部状态，需控制访问权限。

```java
// 原发器
class Editor {
    private String content;
    public void write(String s) { content = s; }
    public Memento save() { return new Memento(content); }  // 创建备忘录
    public void restore(Memento m) { content = m.getContent(); }  // 恢复
}

// 备忘录
class Memento {
    private String content;
    public Memento(String content) { this.content = content; }
    public String getContent() { return content; }
}

// 负责人
class History {
    private Stack<Memento> stack = new Stack<>();
    public void push(Memento m) { stack.push(m); }
    public Memento pop() { return stack.pop(); }
}
```

------

### 📖 本章小结

| 知识点 | 面试关键词 | 实际应用 |
|--------|------------|----------|
| 迭代器 | 顺序访问、不暴露内部、hasNext/next | Java 集合 Iterator、for-each |
| 中介者 | 封装交互、降低耦合、网状变星形 | 聊天室、MVC Controller |
| 备忘录 | 捕获状态、撤销恢复、Originator/Memento | 编辑器撤销、游戏存档 |
