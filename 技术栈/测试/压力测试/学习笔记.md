## 1.1 压力测试概述

### 🧩 什么是压力测试

**压力测试（Stress Testing）**是通过模拟高并发、大数据量的场景，测试系统在极限负载下的性能表现，发现系统的性能瓶颈和潜在问题。

```
测试金字塔：
      /\
     /  \      E2E测试（端到端测试）
    /____\     少量、慢、贵
   /      \
  /        \   集成测试
 /          \   中等数量、中等速度
/____________\
 单元测试
 大量、快、便宜

性能测试（独立于测试金字塔）：
┌─────────────────────────────┐
│     负载测试（Load Test）    │  正常负载下的性能
├─────────────────────────────┤
│     压力测试（Stress Test）  │  极限负载下的性能 ← 我们在这里
├─────────────────────────────┤
│     容量测试（Volume Test）  │  大数据量下的性能
├─────────────────────────────┤
│     稳定性测试（Soak Test）  │  长时间运行的稳定性
└─────────────────────────────┘
```

**核心特点**：

| 特性 | 描述 |
|------|------|
| **高并发** | 模拟大量用户同时访问 |
| **大数据量** | 模拟大量数据处理 |
| **极限负载** | 测试系统的极限性能 |
| **发现瓶颈** | 找出性能瓶颈和问题 |
| **验证容量** | 验证系统的承载能力 |

------

### 🎯 性能测试类型对比

| 测试类型 | 目标 | 负载级别 | 持续时间 | 预期结果 |
|----------|------|----------|----------|----------|
| **负载测试** | 验证正常负载下的性能 | 正常负载 | 短期 | 响应时间符合预期 |
| **压力测试** | 发现极限负载下的瓶颈 | 极限负载 | 短期 | 发现性能瓶颈 |
| **容量测试** | 验证大数据量下的性能 | 大数据量 | 中期 | 处理大量数据 |
| **稳定性测试** | 验证长时间运行的稳定性 | 正常负载 | 长期 | 系统稳定运行 |

**示例对比**：

```
负载测试：
- 模拟1000个用户同时访问
- 持续10分钟
- 验证响应时间<500ms

压力测试：
- 模拟10000个用户同时访问
- 持续5分钟
- 发现系统崩溃点

容量测试：
- 模拟100个用户，处理100万条数据
- 持续30分钟
- 验证数据处理能力

稳定性测试：
- 模拟1000个用户同时访问
- 持续24小时
- 验证系统稳定性
```

## 1.2 压力测试的核心指标

### 📊 性能指标体系

```
1. 响应时间指标
   ├─ 平均响应时间（Average Response Time）
   ├─ 中位数响应时间（Median Response Time）
   ├─ 90%分位响应时间（90th Percentile）
   ├─ 95%分位响应时间（95th Percentile）
   ├─ 99%分位响应时间（99th Percentile）
   └─ 最大响应时间（Max Response Time）

2. 吞吐量指标
   ├─ 每秒请求数（Requests Per Second, RPS）
   ├─ 每秒事务数（Transactions Per Second, TPS）
   └─ 每秒查询数（Queries Per Second, QPS）

3. 并发用户数指标
   ├─ 虚拟用户数（Virtual Users）
   ├─ 活跃用户数（Active Users）
   └─ 并发连接数（Concurrent Connections）

4. 错误率指标
   ├─ HTTP错误率（HTTP Error Rate）
   ├─ 业务错误率（Business Error Rate）
   └─ 超时率（Timeout Rate）

5. 资源使用率指标
   ├─ CPU使用率（CPU Usage）
   ├─ 内存使用率（Memory Usage）
   ├─ 磁盘I/O（Disk I/O）
   ├─ 网络I/O（Network I/O）
   └─ 数据库连接数（Database Connections）
```

### 🎯 关键指标详解

**1. 响应时间**

```
响应时间分布：
0-100ms:   ████████████████████  40%
100-200ms:  ████████████         25%
200-500ms:  ████████            15%
500-1000ms: ████                 10%
>1000ms:    ██                   10%

关键指标：
- 平均响应时间：所有请求的平均值
- 中位数响应时间：50%的请求小于这个值
- 90%分位：90%的请求小于这个值
- 95%分位：95%的请求小于这个值
- 99%分位：99%的请求小于这个值
```

**2. 吞吐量**

```
吞吐量计算：
TPS = 成功事务数 / 测试时间
QPS = 成功查询数 / 测试时间
RPS = 成功请求数 / 测试时间

示例：
- 测试时间：60秒
- 成功请求数：6000个
- RPS = 6000 / 60 = 100
```

**3. 并发用户数**

```
并发用户数计算：
并发用户数 = RPS × 平均响应时间

示例：
- RPS = 100
- 平均响应时间 = 0.5秒
- 并发用户数 = 100 × 0.5 = 50
```

**4. 错误率**

```
错误率计算：
错误率 = 失败请求数 / 总请求数 × 100%

示例：
- 总请求数：10000个
- 失败请求数：100个
- 错误率 = 100 / 10000 × 100% = 1%
```

### 📈 性能基线

```
性能基线定义：
- 正常负载下的性能指标
- 作为性能回归的参考
- 定期更新以反映系统变化

示例基线：
┌─────────────┬──────────┬──────────┬──────────┐
│   指标      │  目标值  │  警告值  │  严重值  │
├─────────────┼──────────┼──────────┼──────────┤
│ 响应时间    │  <500ms  │  <1000ms │  >2000ms │
│ 吞吐量      │  >1000   │  >500    │  <100    │
│ 错误率      │  <0.1%   │  <1%     │  >5%     │
│ CPU使用率   │  <70%    │  <85%    │  >95%    │
│ 内存使用率  │  <80%    │  <90%    │  >95%    │
└─────────────┴──────────┴──────────┴──────────┘
```

## 1.4 压力测试的挑战

### ⚠️ 常见挑战

| 挑战 | 描述 | 解决方案 |
|------|------|----------|
| **环境一致性** | 测试环境与生产环境不一致 | 使用容器化、配置管理 |
| **数据准备** | 需要准备大量测试数据 | 使用数据生成工具 |
| **测试设计** | 难以设计真实的测试场景 | 分析生产日志、用户行为 |
| **结果分析** | 难以定位性能瓶颈 | 使用监控工具、性能分析工具 |
| **成本高昂** | 需要大量硬件资源 | 使用云服务、共享测试环境 |
| **时间限制** | 测试时间可能很长 | 使用分布式测试、优化测试脚本 |

### 💡 解决方案

**1. 环境一致性**

```yaml
# 使用Docker Compose创建一致的测试环境

    depends_on:
      - postgres
      - redis
  
  postgres:
    image: postgres:14
    environment:
      - POSTGRES_PASSWORD=postgres
    volumes:
      - postgres-data:/var/lib/postgresql/data
  
  redis:
    image: redis:7-alpine
    volumes:
      - redis-data:/data
  
  jmeter:
    image: justb4/jmeter:latest
    volumes:
      - ./tests:/jmeter/tests
      - ./results:/jmeter/results
    command: -n -t /jmeter/tests/test-plan.jmx -l /jmeter/results/results.jtl

volumes:
  postgres-data:
  redis-data:
```

## 1.5 压力测试最佳实践

### 📝 测试设计原则

```
1. 真实性原则
   - 模拟真实的用户行为
   - 使用真实的数据分布
   - 模拟真实的网络环境

2. 可重复性原则
   - 测试结果可重复
   - 测试环境一致
   - 测试数据一致

3. 渐进性原则
   - 从小负载开始
   - 逐步增加负载
   - 找到系统极限

4. 全面性原则
   - 覆盖主要业务场景
   - 覆盖主要接口
   - 覆盖主要数据类型

5. 可分析性原则
   - 记录详细日志
   - 收集性能指标
   - 生成可视化报告
```

### ⚠️ 常见反模式

| 反模式 | 问题 | 正确做法 |
|--------|------|----------|
| **测试生产环境** | 可能影响生产环境 | 使用独立的测试环境 |
| **单一场景测试** | 无法发现全面问题 | 设计多个测试场景 |
| **忽略预热** | 测试结果不准确 | 包含预热阶段 |
| **不监控资源** | 难以定位瓶颈 | 监控CPU、内存、磁盘、网络 |
| **不分析结果** | 测试价值低 | 深入分析测试结果 |
| **不建立基线** | 无法比较性能变化 | 建立性能基线 |

## 2.1 JMeter实战

### 🧩 JMeter简介

**Apache JMeter**是一个开源的Java桌面应用程序，用于负载测试和性能测试。

**特点**：

| 特点 | 描述 |
|------|------|
| **开源免费** | 完全免费，开源协议 |
| **跨平台** | 支持Windows、Linux、macOS |
| **功能强大** | 支持多种协议（HTTP、FTP、JDBC等） |
| **可扩展** | 支持插件扩展 |
| **可视化** | 图形化界面，易于使用 |

### 🎯 JMeter安装与配置

**1. 下载安装**

```bash
# 下载JMeter
```

```scala
import io.gatling.core.Predef._
import io.gatling.http.Predef._
import scala.concurrent.duration._

class ComplexSimulation extends Simulation {
  
  // HTTP配置
  val httpProtocol = http
    .baseUrl("http://localhost:8080")
    .acceptHeader("application/json")
    .contentTypeHeader("application/json")
    .userAgentHeader("Gatling/3.9.5")
  
  // Feeders
  val userFeeder = csv("users.csv").circular

  val loginScenario = scenario("Login Scenario")
    .feed(userFeeder)
    .exec(http("Login")
      .post("/api/auth/login")
      .body(StringBody("""{"username":"${username}","password":"${password}"}"""))
      .check(status.is(200))
      .check(jsonPath("$.token").saveAs("token")))
    .exec(session => {
      println(s"Token: ${session("token").as[String]}")
      session
    })
  
  // 场景2：浏览商品
  val browseScenario = scenario("Browse Scenario")
    .exec(http("Get Products")
      .get("/api/products")
      .check(status.is(200)))
    .pause(1, 3)
    .exec(http("Get Product Detail")
      .get("/api/products/${product_id}")
      .check(status.is(200)))
    .pause(1, 2)
  
  // 场景3：下单
  val orderScenario = scenario("Order Scenario")
    .feed(userFeeder)
    .exec(http("Add to Cart")
      .post("/api/cart/add")
      .header("Authorization", "Bearer ${token}")
      .body(StringBody("""{"product_id":"${product_id}","quantity":1}"""))
      .check(status.is(200)))
    .pause(1, 2)
    .exec(http("Create Order")
      .post("/api/orders")
      .header("Authorization", "Bearer ${token}")
      .body(StringBody("""{"cart_id":"${cart_id}"}"""))
      .check(status.is(200)))
  
  // 设置
  setUp(
    loginScenario.inject(
      rampUsers(10) during (5 seconds)
    ),
    browseScenario.inject(
      rampUsers(50) during (10 seconds)
    ),
    orderScenario.inject(
      rampUsers(30) during (10 seconds)
    )
  ).protocols(httpProtocol)
}
```

**2. 复杂脚本**

```javascript
import http from 'k6/http';
import { check, sleep } from 'k6';
import { Rate } from 'k6/metrics';

// 自定义指标
let errorRate = new Rate('errors');

export let options = {
  stages: [
    { duration: '1m', target: 10 },   // 1分钟内增加到10个用户
    { duration: '2m', target: 50 },   // 2分钟内增加到50个用户
    { duration: '5m', target: 100 },  // 5分钟内增加到100个用户
    { duration: '2m', target: 50 },   // 2分钟内减少到50个用户
    { duration: '1m', target: 0 },    // 1分钟内减少到0个用户
  ],
  thresholds: {
    http_req_duration: ['p(95)<500'], // 95%的请求响应时间<500ms
    errors: ['rate<0.01'],            // 错误率<1%
  },
};

export function setup() {
  // 初始化数据
  let users = [];
  for (let i = 0; i < 1000; i++) {
    users.push({
      username: `user${i}`,
      password: 'password123',
    });
  }
  return { users };
}

export default function (data) {
  // 随机选择用户
  let user = data.users[Math.floor(Math.random() * data.users.length)];
  
  // 登录
  let loginRes = http.post('http://localhost:8080/api/auth/login', JSON.stringify({
    username: user.username,
    password: user.password,
  }), {
    headers: { 'Content-Type': 'application/json' },
  });
  
  let loginSuccess = check(loginRes, {
    'login status is 200': (r) => r.status === 200,
  });
  
  errorRate.add(!loginSuccess);
  
  if (loginSuccess) {
    let token = loginRes.json('token');
    
    // 浏览商品
    let productsRes = http.get('http://localhost:8080/api/products', {
      headers: { 'Authorization': `Bearer ${token}` },
    });
    
    let productsSuccess = check(productsRes, {
      'products status is 200': (r) => r.status === 200,
    });
    
    errorRate.add(!productsSuccess);
    
    if (productsSuccess) {
      let products = productsRes.json();
      let productId = products[0].id;
      
      // 下单
      let orderRes = http.post('http://localhost:8080/api/orders', JSON.stringify({
        product_id: productId,
        quantity: 1,
      }), {
        headers: {
          'Content-Type': 'application/json',
          'Authorization': `Bearer ${token}`,
        },
      });
      
      let orderSuccess = check(orderRes, {
        'order status is 200': (r) => r.status === 200,
      });
      
      errorRate.add(!orderSuccess);
    }
  }
  
  sleep(1);
}

export function teardown(data) {
  // 清理数据
  console.log('Test completed');
}
```

## 3.1 性能瓶颈识别

### 🧩 常见性能瓶颈

| 瓶颈类型 | 症状 | 原因 | 解决方案 |
|----------|------|------|----------|
| **CPU瓶颈** | CPU使用率高，响应时间长 | 计算密集型操作、线程阻塞 | 优化算法、减少计算量、使用异步处理 |
| **内存瓶颈** | 内存使用率高，频繁GC | 内存泄漏、对象创建过多 | 优化内存使用、使用对象池、修复内存泄漏 |
| **磁盘I/O瓶颈** | 磁盘I/O高，响应时间长 | 频繁磁盘读写、文件操作 | 使用缓存、优化文件操作、使用SSD |
| **网络I/O瓶颈** | 网络I/O高，响应时间长 | 网络延迟、带宽不足 | 优化网络配置、减少网络请求、使用CDN |
| **数据库瓶颈** | 数据库响应慢，连接数高 | 慢查询、锁竞争、索引不当 | 优化SQL、添加索引、使用连接池 |
| **应用瓶颈** | 应用响应慢，CPU/内存使用正常 | 代码逻辑复杂、同步调用 | 优化代码、使用异步处理、微服务化 |

### 🔍 瓶颈识别流程

```
1. 执行压力测试
   - 模拟真实用户场景
   - 记录性能指标

2. 分析性能指标
   - 响应时间
   - 吞吐量
   - 错误率
   - 资源使用率

3. 定位瓶颈
   - 查看CPU、内存、磁盘、网络使用情况
   - 查看应用日志和错误信息
   - 使用性能分析工具分析代码执行情况

4. 验证瓶颈
   - 执行针对性测试
   - 确认瓶颈位置

5. 实施优化
   - 根据瓶颈类型实施优化
   - 重新执行测试验证优化效果
```

```java
// 使用Redis缓存
@Service
public class UserService {
    @Autowired
    private StringRedisTemplate redisTemplate;
    
    @Autowired
    private UserRepository userRepository;
    
    public User getUserById(Long id) {
        String key = "user:" + id;
        String userJson = redisTemplate.opsForValue().get(key);
        
        if (userJson != null) {
            return JSON.parseObject(userJson, User.class);
        }
        
        User user = userRepository.findById(id).orElse(null);
        if (user != null) {
            redisTemplate.opsForValue().set(key, JSON.toJSONString(user), 30, TimeUnit.MINUTES);
        }
        
        return user;
    }
}
```

### 🎯 案例1：数据库查询优化

**问题**：用户列表查询响应时间超过2秒

**分析**：

```sql
-- 原始SQL
SELECT * FROM users WHERE age > 30 ORDER BY created_at DESC;

-- 执行计划
EXPLAIN SELECT * FROM users WHERE age > 30 ORDER BY created_at DESC;
+----+-------------+-------+------------+------+---------------+------+---------+------+---------+----------+-----------------------------+
| id | select_type | table | partitions | type | possible_keys | key  | key_len | ref  | rows    | filtered | Extra                       |
+----+-------------+-------+------------+------+---------------+------+---------+------+---------+----------+-----------------------------+
|  1 | SIMPLE      | users | NULL       | ALL  | NULL          | NULL | NULL    | NULL | 1000000 |    33.33 | Using where; Using filesort |
+----+-------------+-------+------------+------+---------------+------+---------+------+---------+----------+-----------------------------+
```

**优化**：

```sql
-- 添加索引
CREATE INDEX idx_users_age_created_at ON users(age, created_at);

-- 优化后执行计划
EXPLAIN SELECT * FROM users WHERE age > 30 ORDER BY created_at DESC;
+----+-------------+-------+------------+-------+---------------------------+---------------------------+---------+------+-------+----------+--------------------------+
| id | select_type | table | partitions | type  | possible_keys             | key                       | key_len | ref  | rows  | filtered | Extra                    |
+----+-------------+-------+------------+-------+---------------------------+---------------------------+---------+------+-------+----------+--------------------------+
|  1 | SIMPLE      | users | NULL       | range | idx_users_age_created_at  | idx_users_age_created_at  | 5       | NULL | 333333 |   100.00 | Using where; Using index |
+----+-------------+-------+------------+-------+---------------------------+---------------------------+---------+------+-------+----------+--------------------------+

-- 优化后响应时间：0.2秒
```

### 🎯 案例2：应用代码优化

**问题**：用户登录响应时间超过1秒

**分析**：

```java
// 原始代码
public User login(String username, String password) {
    // 1. 验证用户名密码
    User user = userRepository.findByUsername(username);
    if (user == null) {
        throw new RuntimeException("用户不存在");
    }
    if (!passwordEncoder.matches(password, user.getPassword())) {
        throw new RuntimeException("密码错误");
    }
    
    // 2. 生成令牌
    String token = jwtService.generateToken(user.getId());
    user.setToken(token);
    
    // 3. 更新登录时间
    user.setLastLoginTime(new Date());
    userRepository.save(user);
    
    // 4. 记录登录日志

    return user;
}
```

**优化**：


## 4.1 压力测试最佳实践

### 📝 测试设计最佳实践

**1. 真实场景模拟**

```
- 分析用户行为：使用生产日志分析真实用户行为
- 模拟真实数据：使用接近生产环境的数据
- 模拟真实网络：模拟不同网络延迟和带宽
- 模拟真实用户分布：不同用户类型和行为模式
```

**2. 测试数据准备**

```
- 数据量：使用接近生产环境的数据量
- 数据分布：模拟真实的数据分布
- 数据多样性：包含各种类型的数据
- 数据一致性：确保测试数据的一致性

# 数据生成工具
- Faker：生成随机测试数据
- DBUnit：数据库测试数据管理
- JMeter CSV Data Set：CSV数据文件
```

**3. 测试执行最佳实践**

```
- 预热阶段：在正式测试前进行预热
- 逐步增加负载：从小负载开始，逐步增加
- 持续时间：根据测试目的设置合适的持续时间
- 监控：实时监控系统状态
- 重复测试：多次执行测试以确保结果可重复

# 测试执行流程
1. 环境准备
2. 数据准备
3. 预热测试
4. 正式测试
5. 结果分析
6. 报告生成
```

**4. 结果分析最佳实践**


### ⚠️ 常见错误与解决方案

| 错误 | 原因 | 解决方案 |
|------|------|----------|
| **测试环境与生产环境不一致** | 硬件配置、软件版本、网络环境不同 | 使用容器化，确保环境一致性 |
| **测试数据不真实** | 数据量不足、数据分布不合理 | 使用真实数据或生成接近真实的数据 |
| **测试场景不完整** | 只测试部分场景 | 覆盖主要业务场景和接口 |
| **忽略预热阶段** | 测试结果不准确 | 包含预热阶段，让系统达到稳定状态 |
| **不监控系统资源** | 难以定位瓶颈 | 实时监控CPU、内存、磁盘、网络 |
| **不分析测试结果** | 测试价值低 | 深入分析测试结果，提出优化建议 |
| **测试执行不稳定** | 结果波动大 | 多次执行测试，取平均值 |
| **不建立性能基线** | 无法比较性能变化 | 建立性能基线，定期更新 |

## 4.4 压力测试案例分析

### 🎯 案例1：电商系统压力测试

**项目背景**：
- 电商平台，日活跃用户100万
- 峰值QPS 10000
- 主要业务：用户登录、商品浏览、下单支付

**测试目标**：
- 验证系统在峰值负载下的性能
- 发现系统瓶颈
- 提供容量规划建议

**测试场景**：
1. **登录场景**：10000用户/分钟
2. **商品浏览场景**：50000用户/分钟
3. **下单场景**：10000用户/分钟
4. **支付场景**：5000用户/分钟

**测试结果**：

| 场景 | 响应时间 | 吞吐量 | 错误率 |
|------|---------|--------|--------|
| 登录 | 450ms | 167 TPS | 0.1% |
| 商品浏览 | 280ms | 833 TPS | 0.05% |
| 下单 | 850ms | 167 TPS | 0.2% |
| 支付 | 1200ms | 83 TPS | 0.1% |

**瓶颈分析**：
- 数据库连接池不足
- 商品详情页缓存未命中
- 支付接口响应慢

**优化建议**：
1. 增加数据库连接池大小
2. 优化商品详情页缓存策略
3. 优化支付接口性能
4. 水平扩展应用服务器

**优化效果**：
- 登录响应时间：450ms → 250ms
- 商品浏览响应时间：280ms → 150ms
- 下单响应时间：850ms → 450ms
- 支付响应时间：1200ms → 700ms

### 🎯 案例2：金融系统压力测试

**项目背景**：
- 银行核心系统

| 转账 | 950ms | 167 TPS | 0.1% |
