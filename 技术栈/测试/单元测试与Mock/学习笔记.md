## 1.1 单元测试概述

### 🧩 什么是单元测试

**单元测试（Unit Testing）**是对软件中最小可测试单元（通常是函数、方法或类）进行验证的测试方式。

```
测试金字塔：
      /\
     /  \      E2E测试（端到端测试）
    /____\     少量、慢、贵
   /      \
  /        \   集成测试
 /          \   中等数量、中等速度
/____________\
 单元测试
 大量、快、便宜
```

**核心特点**：

| 特性 | 描述 |
|------|------|
| **粒度小** | 测试单个函数或方法 |
| **执行快** | 毫秒级完成 |
| **独立性强** | 不依赖外部系统 |
| **可重复** | 每次执行结果一致 |
| **自动化** | 可集成到CI/CD |

------

### 🎯 为什么要写单元测试

**1. 提升代码质量**

```java
// 没有单元测试的代码
public int divide(int a, int b) {
    return a / b;  // 可能抛出除零异常
}

// 有单元测试的代码
@Test(expected = ArithmeticException.class)
public void testDivideByZero() {
    Calculator calculator = new Calculator();
    calculator.divide(10, 0);  // 预期抛出异常
}
```

**2. 重构的保障**

```
重构流程：
1. 编写单元测试（通过）
2. 修改代码
3. 运行单元测试
4. 如果测试通过，重构成功
5. 如果测试失败，回滚代码
```

**3. 活的文档**

```java
// 测试用例就是最好的文档
@Test
public void testCalculateDiscount() {
    // 当用户等级为VIP，消费金额超过1000时
    // 享受8折优惠
    User user = new User("VIP");
    BigDecimal discount = discountService.calculate(user, new BigDecimal("1500"));
    assertEquals(new BigDecimal("1200.00"), discount);
}
```

**4. 快速反馈**

```
开发流程对比：

传统开发：
写代码 → 提交 → 测试人员测试 → 发现Bug → 修复 → 重新测试
时间：2-3天

单元测试驱动：
写代码 → 写测试 → 运行测试 → 发现Bug → 修复 → 重新运行测试
时间：几分钟
```

## 1.2 单元测试的核心原则

### 🔹 FIRST原则

| 原则 | 含义 | 说明 |
|------|------|------|
| **F**ast | 快速 | 单元测试应该在毫秒级完成 |
| **I**ndependent | 独立 | 测试之间不依赖，可任意顺序执行 |
| **R**epeatable | 可重复 | 多次执行结果一致，不依赖环境 |
| **S**elf-Validating | 自验证 | 测试自动通过或失败，无需人工判断 |
| **T**imely | 及时 | 在编写代码之前或同时编写测试 |

### 📋 测试覆盖率

**覆盖率类型**：

```
1. 行覆盖率（Line Coverage）
   - 执行的代码行数 / 总代码行数

2. 分支覆盖率（Branch Coverage）
   - 执行的分支数 / 总分支数

3. 函数覆盖率（Function Coverage）
   - 被调用的函数数 / 总函数数

4. 语句覆盖率（Statement Coverage）
   - 执行的语句数 / 总语句数
```

**覆盖率目标**：

| 项目类型 | 目标覆盖率 | 说明 |
|----------|-----------|------|
| 核心业务逻辑 | 80%+ | 关键路径必须覆盖 |
| 工具类 | 90%+ | 逻辑简单，应尽量覆盖 |
| 配置类 | 50%+ | 主要是配置，覆盖率要求低 |
| 整体项目 | 70%+ | 平衡成本与收益 |

> ⚠️ 注意：覆盖率不是越高越好，关键是有意义的测试。

## 1.4 测试驱动开发（TDD）

### 🔄 TDD开发流程

```
TDD循环（红-绿-重构）：

1. 🔴 红色：编写一个失败的测试
   - 先写测试，再写代码
   - 测试应该失败（因为代码还没写）

2. 🟢 绿色：编写最简单的代码让测试通过
   - 写最少的代码
   - 不考虑代码质量，只让测试通过

3. 🔵 重构：优化代码
   - 提取方法、消除重复
   - 保持测试通过

4. 重复上述步骤
```

### 💡 TDD实战示例

```java
// 步骤1：红色 - 编写失败的测试
@Test
public void testAdd() {
    Calculator calculator = new Calculator();
    int result = calculator.add(2, 3);
    assertEquals(5, result);  // 测试失败，因为Calculator类还不存在
}

// 步骤2：绿色 - 编写最简单的代码
public class Calculator {
    public int add(int a, int b) {
        return a + b;  // 最简单的实现
    }
}

// 步骤3：重构 - 优化代码（如果需要）
// 当前代码已经足够简洁，无需重构
```

### ✅ TDD的优势

| 优势 | 说明 |
|------|------|
| **设计驱动** | 从使用者的角度设计API |
| **文档同步** | 测试就是最新的文档 |
| **减少Bug** | 在开发早期发现问题 |
| **重构信心** | 有测试保障，敢于重构 |
| **代码质量** | 测试驱动的代码更易测试 |

## 1.5 单元测试最佳实践

### 📝 命名规范

```java
// 推荐的测试命名格式
@Test
public void test<方法名>_<场景>_<预期结果>()

// 示例
@Test
public void testCalculateDiscount_VIPUser_80PercentDiscount() {
    // 测试VIP用户享受8折优惠
}

@Test
public void testCalculateDiscount_NormalUser_NoDiscount() {
    // 测试普通用户无折扣
}

@Test
public void testLogin_InvalidPassword_ThrowException() {
    // 测试密码错误抛出异常
}
```

### 🎯 测试结构（AAA模式）

```java
@Test
public void testTransferMoney_Success() {
    // Arrange（准备）：准备测试数据
    Account fromAccount = new Account(1000);
    Account toAccount = new Account(500);
    TransferService service = new TransferService();

    // Act（执行）：执行被测试的方法
    service.transfer(fromAccount, toAccount, 300);

    // Assert（断言）：验证结果
    assertEquals(700, fromAccount.getBalance());
    assertEquals(800, toAccount.getBalance());
}
```

### ⚠️ 常见反模式

| 反模式 | 问题 | 正确做法 |
|--------|------|----------|
| **测试私有方法** | 私有方法应该通过公共方法测试 | 测试公共方法的行为 |
| **测试实现细节** | 测试过于依赖内部实现 | 测试对外行为 |
| **测试异常处理** | 只测试正常流程 | 同时测试正常和异常流程 |
| **测试数据库** | 依赖外部系统，速度慢 | 使用Mock模拟 |
| **测试包含逻辑** | 测试代码本身有if/else | 每个测试只验证一个场景 |

## 2.1 Mock技术概述

### 🧩 什么是Mock

**Mock（模拟）**是在测试中用模拟对象替换真实依赖的技术。

```
真实场景：
UserService → Database → 获取用户数据

测试场景：
UserService → Mock Database → 返回模拟数据
```

**为什么要使用Mock**：

| 原因 | 说明 |
|------|------|
| **隔离依赖** | 不依赖外部系统（数据库、网络、文件系统） |
| **提高速度** | Mock对象响应快，不等待网络或IO |
| **可控性** | 可以模拟各种场景（成功、失败、超时） |
| **确定性** | 每次测试结果一致，不受外部环境影响 |

### 📊 Mock vs Stub vs Spy

| 类型 | 定义 | 用途 |
|------|------|------|
| **Mock** | 完全模拟对象，可以验证调用 | 验证方法调用、参数 |
| **Stub** | 提供预定义的返回值 | 提供测试数据 |
| **Spy** | 部分模拟，可以调用真实方法 | 监控真实对象行为 |

```java
// Mock示例
UserService mockUserService = Mockito.mock(UserService.class);
when(mockUserService.getUserById(1)).thenReturn(new User("张三"));

// Stub示例
User user = new User("张三");
when(mockUserService.getUserById(1)).thenReturn(user);

// Spy示例
List<String> list = new ArrayList<>();
List<String> spyList = Mockito.spy(list);
spyList.add("张三");  // 调用真实方法
```

## 2.2 Mockito框架详解

### 🔧 Mockito基础用法

**1. 创建Mock对象**

```java
// 方式1：使用Mockito.mock()
UserService userService = Mockito.mock(UserService.class);

// 方式2：使用@Mock注解
@ExtendWith(MockitoExtension.class)
public class UserServiceTest {
    @Mock
    private UserService userService;
}

// 方式3：使用@MockBean（Spring Boot）
@SpringBootTest
public class UserServiceTest {
    @MockBean
    private UserRepository userRepository;
}
```

**2. 设置行为（when-then）**

```java
// 设置返回值
when(userService.getUserById(1)).thenReturn(new User("张三"));

// 设置异常
when(userService.getUserById(-1))
    .thenThrow(new IllegalArgumentException("用户ID不能为负数"));

// 设置多次调用返回不同值
when(userService.getUserById(anyInt()))
    .thenReturn(new User("第一次"))
    .thenReturn(new User("第二次"))
    .thenReturn(new User("第三次"));

// 设置连续调用
when(userService.getUserById(anyInt()))
    .thenAnswer(invocation -> {
    Integer id = invocation.getArgument(0);
    return new User("用户" + id);
});
```

**3. 验证调用（verify）**

```java
// 验证方法被调用
userService.getUserById(1);
verify(userService).getUserById(1);

// 验证方法从未被调用
verify(userService, never()).getUserById(2);

// 验证方法调用次数
verify(userService, times(1)).getUserById(1);
verify(userService, atLeastOnce()).getUserById(1);
verify(userService, atMost(2)).getUserById(1);

// 验证调用顺序
InOrder inOrder = inOrder(userService, orderService);
inOrder.verify(userService).getUserById(1);
inOrder.verify(orderService).createOrder(any());
```

### 🎯 参数匹配器

```java
// 精确匹配
when(userService.getUserById(1)).thenReturn(user);

// 任意匹配
when(userService.getUserById(anyInt())).thenReturn(user);
when(userService.findUser(anyString())).thenReturn(user);

// 特定条件匹配
when(userService.findUser(contains("张"))).thenReturn(user);
when(userService.findUser(startsWith("李"))).thenReturn(user);
when(userService.findUser(endsWith("王"))).thenReturn(user);
when(userService.findUser(matches(".*三.*"))).thenReturn(user);

// 自定义匹配器
when(userService.findUser(argThat(user -> user.getAge() > 18)))
    .thenReturn(user);

// 多个参数匹配
when(userService.createUser(anyString(), anyInt()))
    .thenReturn(user);
```

### 📦 Mockito高级特性

**1. Spy（部分模拟）**

```java
// 创建Spy对象
List<String> list = new ArrayList<>();
List<String> spyList = Mockito.spy(list);

// 调用真实方法
spyList.add("张三");
assertEquals(1, spyList.size());

// 模拟特定方法
when(spyList.size()).thenReturn(100);
assertEquals(100, spyList.size());

// 注意：不能使用when(spyList.get(0)).thenReturn("李四")
// 应该使用doReturn
doReturn("李四").when(spyList).get(0);
```

**2. 静态方法Mock（Mockito 3.4+）**

```java
// Mock静态方法
try (MockedStatic<StringUtils> mocked = mockStatic(StringUtils.class)) {
    mocked.when(() -> StringUtils.isEmpty(anyString())).thenReturn(true);
    
    assertTrue(StringUtils.isEmpty("test"));
}
```

**3. 构造函数Mock（Mockito 3.5+）**

```java
// Mock构造函数
try (MockedConstruction<User> mocked = mockConstruction(User.class)) {
    User user = new User("张三");
    when(user.getName()).thenReturn("模拟用户");
    
    assertEquals("模拟用户", user.getName());
}
```

**4. Answer接口**

```java
// 使用Answer自定义返回值
when(userService.getUserById(anyInt()))
    .thenAnswer(invocation -> {
    Integer id = invocation.getArgument(0);
    if (id < 0) {
        throw new IllegalArgumentException("用户ID不能为负数");
    }
    return new User("用户" + id);
});
```

## 2.3 断言框架

### 📊 AssertJ vs JUnit Assert

| 特性 | JUnit Assert | AssertJ |
|------|--------------|---------|
| **可读性** | 一般 | 优秀 |
| **链式调用** | 不支持 | 支持 |
| **错误信息** | 简单 | 详细 |
| **类型安全** | 弱 | 强 |

### 🎯 JUnit断言

```java
// 基本断言
assertEquals(5, calculator.add(2, 3));
assertNotEquals(4, calculator.add(2, 3));
assertTrue(calculator.add(2, 3) == 5);
assertFalse(calculator.add(2, 3) == 4);

// 对象断言
User user1 = new User("张三");
User user2 = new User("张三");
assertEquals(user1, user2);  // 需要重写equals方法
assertSame(user1, user1);   // 同一个对象
assertNotSame(user1, user2);

// 数组断言
int[] expected = {1, 2, 3};
int[] actual = calculator.generateArray(3);
assertArrayEquals(expected, actual);

// 异常断言
assertThrows(IllegalArgumentException.class, () -> {
    calculator.divide(10, 0);
});

// 超时断言
assertTimeout(Duration.ofSeconds(1), () -> {
    calculator.longRunningTask();
});
```

### 🚀 AssertJ断言

```java
// 基本断言
assertThat(calculator.add(2, 3))
    .isEqualTo(5)
    .isGreaterThan(4)
    .isLessThan(10);

// 对象断言
assertThat(user)
    .isNotNull()
    .hasFieldOrProperty("name")
    .hasFieldOrPropertyWithValue("name", "张三");

// 集合断言
assertThat(users)
    .isNotEmpty()
    .hasSize(3)
    .extracting("name")
    .contains("张三", "李四", "王五");

// 字符串断言
assertThat(text)
    .isNotEmpty()
    .startsWith("Hello")
    .endsWith("World")
    .contains("Java");

// 异常断言
assertThatThrownBy(() -> calculator.divide(10, 0))
    .isInstanceOf(ArithmeticException.class)
    .hasMessageContaining("by zero");
```

## 2.4 测试异常场景

### 🔹 异常测试

```java
// JUnit方式
@Test(expected = IllegalArgumentException.class)
public void testDivideByZero() {
    calculator.divide(10, 0);
}

// JUnit 5方式
@Test
public void testDivideByZero() {
    IllegalArgumentException exception = assertThrows(
        IllegalArgumentException.class,
        () -> calculator.divide(10, 0)
    );
    assertEquals("除数不能为零", exception.getMessage());
}

// AssertJ方式
@Test
public void testDivideByZero() {
    assertThatThrownBy(() -> calculator.divide(10, 0))
        .isInstanceOf(ArithmeticException.class)
        .hasMessage("/ by zero");
}
```

### 🔹 边界条件测试

```java
@Test
public void testBoundaryConditions() {
    // 测试最小值
    assertEquals(0, calculator.add(0, 0));
    
    // 测试最大值
    assertEquals(Integer.MAX_VALUE, calculator.add(Integer.MAX_VALUE, 0));
    
    // 测试溢出
    assertThrows(ArithmeticException.class, () -> {
        calculator.add(Integer.MAX_VALUE, 1);
    });
    
    // 测试负数
    assertEquals(-5, calculator.add(-2, -3));
    
    // 测试混合
    assertEquals(0, calculator.add(5, -5));
}
```

### 🔹 空值测试

```java
@Test
public void testNullHandling() {
    // 测试null输入
    assertThrows(NullPointerException.class, () -> {
        userService.getUserById(null);
    });
    
    // 测试null返回
    when(userService.getUserById(999)).thenReturn(null);
    User user = userService.getUserById(999);
    assertNull(user);
    
    // 使用Optional避免null
    when(userService.findUserById(999)).thenReturn(Optional.empty());
    Optional<User> optionalUser = userService.findUserById(999);
    assertFalse(optionalUser.isPresent());
```

## 3.1 参数化测试

### 📊 什么是参数化测试

**参数化测试**是一种使用不同参数集执行相同测试逻辑的测试方法。

```
参数化测试流程：
1. 定义测试方法
2. 提供测试数据
3. 对每组数据执行测试
```

## 3.2 测试生命周期管理

### 🔹 JUnit 5生命周期注解

| 注解 | 作用 | 执行时机 |
|------|------|----------|
| `@BeforeAll` | 所有测试方法执行前 | 只执行一次 |
| `@AfterAll` | 所有测试方法执行后 | 只执行一次 |
| `@BeforeEach` | 每个测试方法执行前 | 每个测试执行前 |
| `@AfterEach` | 每个测试方法执行后 | 每个测试执行后 |

```java
    }
    
    @Test

    }
    
    @AfterEach

        // 每个测试方法执行后清理

    }
```

## 3.3 测试替身策略

### 🧩 测试替身类型

| 类型 | 用途 | 适用场景 |
|------|------|----------|

| 复杂依赖模拟 | Fake | 提供简单实现，避免外部依赖 |

## 4.5 单元测试与集成测试的边界

### 🔹 测试类型划分

| 测试类型 | 范围 | 速度 | 依赖 | 用途 |
|----------|------|------|------|------|
| **单元测试** | 单个方法/类 | 毫秒级 | 无外部依赖 | 验证逻辑正确性 |
| **集成测试** | 多个组件 | 秒级 | 外部依赖 | 验证组件协作 |
| **E2E测试** | 整个系统 | 分钟级 | 完整环境 | 验证系统功能 |

### 🎯 测试策略制定

**1. 测试金字塔应用**

```
测试金字塔策略：
- 底层：单元测试（70%）
  - 大量、快速、独立
  - 验证核心逻辑

- 中层：集成测试（20%）
  - 中等数量、中等速度
  - 验证组件协作

- 顶层：E2E测试（10%）
  - 少量、慢速、昂贵
  - 验证系统功能
```

**2. 测试边界确定**

| 场景 | 推荐测试类型 | 原因 |
|------|--------------|------|
| 业务逻辑验证 | 单元测试 | 快速、独立、聚焦 |
| 数据库交互 | 集成测试 | 验证SQL、事务 |
| API接口 | 集成测试 | 验证接口契约 |
| 外部服务调用 | 集成测试 | 验证服务集成 |
| 完整业务流程 | E2E测试 | 验证端到端流程 |

```yaml
# .github/workflows/ci.yml
name: CI

on:
  push:
    branches: [ main, develop ]
  pull_request:
    branches: [ main, develop ]

jobs:
  build:
    runs-on: ubuntu-latest
    
    steps:
    - uses: actions/checkout@v2
    
    - name: Set up JDK 11
      uses: actions/setup-java@v2
      with:
        java-version: '11'
        distribution: 'adopt'
    
    - name: Cache Maven packages
      uses: actions/cache@v2
      with:
        path: ~/.m2
        key: ${{ runner.os }}-m2-${{ hashFiles('**/pom.xml') }}
        restore-keys: ${{ runner.os }}-m2
    
    - name: Build with Maven
      run: mvn clean compile
    
    - name: Run unit tests
      run: mvn test -Dgroups=unit
    
    - name: Run integration tests
      run: mvn test -Dgroups=integration
    
    - name: Generate coverage report
      run: mvn jacoco:report
    
    - name: Upload coverage report
      uses: codecov/codecov-action@v2
      with:
        file: ./target/site/jacoco/jacoco.xml
    
    - name: Deploy to staging
      if: github.ref == 'refs/heads/develop'
      run: mvn deploy -DskipTests
    
    - name: Deploy to production
      if: github.ref == 'refs/heads/main'
      run: mvn deploy -DskipTests
```
