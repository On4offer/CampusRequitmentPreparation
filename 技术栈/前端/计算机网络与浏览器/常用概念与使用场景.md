## 一、HTTP 状态码

### 📖 状态码分类与含义

| 分类 | 常见码 | 说明 | 适用场景 |
|------|--------|------|----------|
| **2xx 成功** | 200 | OK - 请求成功 | 常规请求成功返回 |
| | 201 | Created - 创建成功 | POST请求创建资源 |
| | 204 | No Content - 无内容 | DELETE请求成功 |
| **3xx 重定向** | 301 | Moved Permanently - 永久重定向 | 域名迁移、URL规范化 |
| | 302 | Found - 临时重定向 | 临时跳转、促销活动 |
| | 304 | Not Modified - 协商缓存命中 | 资源未修改，使用本地缓存 |
| | 307 | Temporary Redirect - 临时重定向（保持方法） | 同302，但保持请求方法 |
| | 308 | Permanent Redirect - 永久重定向（保持方法） | 同301，但保持请求方法 |
| **4xx 客户端错误** | 400 | Bad Request - 请求错误 | 参数错误、格式不正确 |
| | 401 | Unauthorized - 未认证 | 缺少认证信息 |
| | 403 | Forbidden - 禁止访问 | 认证通过但无权限 |
| | 404 | Not Found - 资源未找到 | URL错误、资源不存在 |
| | 405 | Method Not Allowed - 方法不允许 | 使用了不支持的HTTP方法 |
| | 429 | Too Many Requests - 请求过多 | 限流保护 |
| **5xx 服务器错误** | 500 | Internal Server Error - 服务器内部错误 | 后端代码异常 |
| | 502 | Bad Gateway - 网关错误 | 上游服务异常 |
| | 503 | Service Unavailable - 服务不可用 | 服务器维护、过载 |
| | 504 | Gateway Timeout - 网关超时 | 上游服务响应超时 |

### 💬 面试问答

> **问：301和302的区别是什么？**
> **答**：301是永久重定向，浏览器会缓存重定向关系，下次直接访问新URL；302是临时重定向，浏览器不会缓存，每次都需要经过原URL。SEO角度：301会传递权重，302不会。

> **问：什么时候会返回304状态码？**
> **答**：当使用协商缓存时，浏览器发送请求并携带If-Modified-Since或If-None-Match头，服务器对比后发现资源未修改，返回304 Not Modified，浏览器使用本地缓存。

## 二、常用 HTTP 头

### 📖 请求头

| 头字段 | 说明 | 示例/场景 |
|--------|------|-----------|
| **Content-Type** | 请求体类型 | `application/json`（JSON数据）、`application/x-www-form-urlencoded`（表单）、`multipart/form-data`（文件上传） |
| **Accept** | 期望的响应类型 | `text/html,application/json` |
| **Authorization** | 认证信息 | `Bearer token123`（JWT）、`Basic base64(username:password)` |
| **User-Agent** | 客户端标识 | `Mozilla/5.0 (Windows NT 10.0; Win64; x64) AppleWebKit/537.36` |
| **Referer** | 来源页面 | `https://example.com/search?q=keyword` |
| **Origin** | 跨域请求来源 | `https://example.com`（只包含协议+域名+端口） |
| **Cache-Control** | 缓存指令 | `max-age=0, no-cache`（强制协商缓存） |
| **If-Modified-Since** | 协商缓存时间戳 | `Wed, 21 Oct 2024 07:28:00 GMT` |
| **If-None-Match** | 协商缓存ETag | `"abc123"` |

### 📖 响应头

| 头字段 | 说明 | 示例/场景 |
|--------|------|-----------|
| **Content-Type** | 响应体类型 | `application/json; charset=utf-8` |
| **Content-Length** | 响应体大小 | `1234` |
| **Cache-Control** | 缓存策略 | `public, max-age=31536000`（1年强缓存） |
| **Expires** | 过期时间 | `Wed, 21 Oct 2025 07:28:00 GMT` |
| **Last-Modified** | 最后修改时间 | `Wed, 21 Oct 2024 07:28:00 GMT` |
| **ETag** | 资源唯一标识 | `"abc123"` |
| **Set-Cookie** | 设置Cookie | `sessionId=abc123; HttpOnly; Secure; SameSite=Lax` |
| **Access-Control-Allow-Origin** | CORS允许的源 | `*`（任意）或具体域名 |
| **Access-Control-Allow-Methods** | CORS允许的方法 | `GET, POST, PUT, DELETE, OPTIONS` |
| **Access-Control-Allow-Headers** | CORS允许的头 | `Content-Type, Authorization` |
| **Access-Control-Allow-Credentials** | CORS是否允许凭证 | `true`（允许携带Cookie） |
| **Location** | 重定向目标URL | `https://example.com/new-path` |

### 🔍 实用配置示例

**Nginx缓存配置**：
```nginx
# 静态资源强缓存
location ~* \.(js|css|png|jpg|jpeg|gif|ico|svg|woff|woff2)$ {
    expires 1y;
    add_header Cache-Control "public, immutable";
    add_header Vary "Accept-Encoding";
}

# API接口协商缓存
location /api {
    add_header Cache-Control "no-cache";
    proxy_pass http://backend;
}
```

**Express CORS配置**：
```javascript
const cors = require('cors');
app.use(cors({
    origin: 'https://example.com',
    methods: ['GET', 'POST', 'PUT', 'DELETE'],
    allowedHeaders: ['Content-Type', 'Authorization'],
    credentials: true,
    maxAge: 86400 // 预检缓存1天
}));
```

## 三、缓存策略

### 📖 缓存类型对比

| 类型 | 相关头 | 行为 | 性能 | 适用场景 |
|------|--------|------|------|----------|
| **强缓存** | Expires、Cache-Control | 未过期直接用本地，不发请求 | 高 | 静态资源（JS、CSS、图片） |
| **协商缓存** | Last-Modified / If-Modified-Since | 发请求，服务器比较时间 | 中 | 频繁更新的资源 |
| **协商缓存** | ETag / If-None-Match | 发请求，比较摘要 | 中 | 动态内容 |
| **Service Worker缓存** | 自定义缓存策略 | 完全控制缓存逻辑 | 高 | PWA、离线应用 |

### 🔍 缓存流程

```
浏览器请求资源
    ↓
检查强缓存是否命中
    ↓ 是
使用本地缓存（200 from cache）
    ↓ 否
发送HTTP请求（携带协商缓存头）
    ↓
服务器检查资源是否修改
    ↓ 未修改
返回304 Not Modified，使用本地缓存
    ↓ 已修改
返回200 OK + 新资源
    ↓
更新本地缓存
```

### 📖 缓存策略选择

| 资源类型 | 推荐策略 | 配置示例 |
|----------|----------|----------|
| **第三方库** | 强缓存（长过期） | `Cache-Control: public, max-age=31536000` |
| **静态资源** | 强缓存（哈希文件名） | `app.abc123.js` + `max-age=31536000` |
| **API接口** | 协商缓存或禁用 | `Cache-Control: no-cache` |
| **HTML页面** | 协商缓存 | `Cache-Control: no-cache` |

### 💬 面试问答

> **问：强缓存和协商缓存的区别？**
> **答**：强缓存直接使用本地缓存，不发HTTP请求，性能更高；协商缓存需要发请求到服务器验证资源是否修改，性能稍低但更灵活。强缓存通过Expires和Cache-Control控制，协商缓存通过Last-Modified/ETag控制。

> **问：ETag和Last-Modified的区别？**
> **答**：Last-Modified基于时间戳，精度为秒，可能存在同一秒内修改的文件无法检测的问题；ETag基于文件内容生成的唯一标识，更准确但计算开销更大。ETag优先级高于Last-Modified。

## 四、跨域解决方案

### 📖 跨域概念

**同源**：协议 + 域名 + 端口 三者完全一致。

**跨域场景**：
- `http://a.com` → `https://a.com`（协议不同）
- `http://a.com` → `http://b.com`（域名不同）
- `http://a.com:8080` → `http://a.com:3000`（端口不同）

### 📖 跨域解决方案对比

| 方案 | 原理 | 优点 | 缺点 | 适用场景 |
|------|------|------|------|----------|
| **CORS** | 服务端设置响应头 | 标准方案，支持所有方法 | 需要服务端配合 | 生产环境、API调用 |
| **JSONP** | 动态创建script标签 | 兼容旧浏览器 | 只支持GET请求 | 第三方接口、旧项目 |
| **代理服务器** | 同源代理转发 | 开发环境简单 | 增加服务器负担 | 开发环境、前后端分离 |
| **postMessage** | 窗口间通信API | 跨域窗口通信 | 复杂度较高 | iframe通信、多窗口 |
| **WebSockets** | 全双工通信协议 | 实时通信 | 服务器需要支持 | 实时应用、聊天 |

### 🔍 实用配置示例

**CORS服务端配置（Node.js）**：
```javascript
// Express
app.use((req, res, next) => {
    res.setHeader('Access-Control-Allow-Origin', 'https://example.com');
    res.setHeader('Access-Control-Allow-Methods', 'GET, POST, PUT, DELETE, OPTIONS');
    res.setHeader('Access-Control-Allow-Headers', 'Content-Type, Authorization');
    res.setHeader('Access-Control-Allow-Credentials', 'true');
    
    if (req.method === 'OPTIONS') {
        return res.status(204).end();
    }
    next();
});
```

**开发环境代理配置（Vite）**：
```javascript
// vite.config.js
export default defineConfig({
    server: {
        proxy: {
            '/api': {
                target: 'http://localhost:3000',
                changeOrigin: true,
                rewrite: (path) => path.replace(/^\/api/, '')
            }
        }
    }
});
```

**JSONP实现**：
```javascript
// 前端
function jsonp(url, callback) {
    const script = document.createElement('script');
    script.src = `${url}?callback=${callback}`;
    document.body.appendChild(script);
}

// 调用
jsonp('https://api.example.com/data', 'handleData');

function handleData(data) {
    console.log('JSONP数据:', data);
}

// 后端（Node.js）
app.get('/data', (req, res) => {
    const callback = req.query.callback;
    const data = { message: 'Hello JSONP' };
    res.send(`${callback}(${JSON.stringify(data)})`);
});
```

### 💬 面试问答

> **问：CORS的预检请求（OPTIONS）是什么？**
> **答**：当发送非简单请求（如PUT/DELETE方法、自定义头、Content-Type为application/json）时，浏览器会先发送OPTIONS预检请求，检查服务器是否允许该请求。服务器返回CORS相关响应头后，浏览器才会发送实际请求。

> **问：开发环境如何解决跨域问题？**
> **答**：使用代理服务器，如Vite/Webpack的proxy配置，将前端请求代理到后端，避免浏览器的跨域限制。例如Vite配置`/api`代理到`http://localhost:3000`，前端请求`/api/data`会被代理到`http://localhost:3000/data`。

## 五、Web 安全

### 📖 常见安全威胁

| 威胁 | 原理 | 防护措施 | 代码示例 |
|------|------|----------|----------|
| **XSS（跨站脚本）** | 注入恶意脚本 | 1. 输入输出转义<br>2. 使用CSP<br>3. Cookie设HttpOnly<br>4. 避免innerHTML | `// 转义示例
const safeHtml = escapeHtml(userInput);
div.textContent = safeHtml;` |
| **CSRF（跨站请求伪造）** | 利用用户身份执行未授权操作 | 1. CSRF Token<br>2. 校验Referer/Origin<br>3. Cookie SameSite | `// 后端生成Token
const token = crypto.randomBytes(32).toString('hex');
// 前端提交
<form><input type="hidden" name="_csrf" value="${token}"></form>` |
| **Clickjacking（点击劫持）** | 透明iframe诱导用户点击 | 1. X-Frame-Options头<br>2. Content-Security-Policy | `// Nginx配置
add_header X-Frame-Options "SAMEORIGIN";` |
| **SQL注入** | 注入恶意SQL语句 | 1. 参数化查询<br>2. ORM框架<br>3. 输入验证 | `// 安全查询
const query = 'SELECT * FROM users WHERE id = ?';
db.query(query, [userId]);` |
| **敏感信息泄露** | 日志、错误信息暴露 | 1. 生产环境隐藏错误详情<br>2. 敏感信息加密<br>3. 日志脱敏 | `// 密码加密
const hashedPassword = bcrypt.hashSync(password, 10);` |

### 📖 CSP（内容安全策略）

**配置示例**：
```nginx
# Nginx配置
add_header Content-Security-Policy "default-src 'self'; script-src 'self' https://trusted-cdn.com; style-src 'self' 'unsafe-inline'; img-src 'self' data:;";
```

**常用指令**：
- `default-src`：默认资源来源
- `script-src`：脚本来源
- `style-src`：样式来源
- `img-src`：图片来源
- `connect-src`：AJAX请求来源
- `frame-src`：iframe来源

### 🔍 安全头配置

| 头字段 | 说明 | 配置示例 |
|--------|------|----------|
| **X-Content-Type-Options** | 防止MIME类型嗅探 | `nosniff` |
| **X-Frame-Options** | 防止点击劫持 | `SAMEORIGIN` |
| **X-XSS-Protection** | 启用XSS过滤 | `1; mode=block` |
| **Strict-Transport-Security** | 强制HTTPS | `max-age=31536000; includeSubDomains` |
| **Referrer-Policy** | 控制Referer头 | `strict-origin-when-cross-origin` |

### 💬 面试问答

> **问：XSS和CSRF的区别？**
> **答**：XSS是注入恶意脚本到用户浏览器执行，目标是获取用户信息（如Cookie）；CSRF是利用用户的已认证状态，诱导用户执行非授权操作（如转账）。XSS关注的是客户端脚本执行，CSRF关注的是跨站请求伪造。

> **问：如何防止XSS攻击？**
> **答**：1) 输入输出转义（如HTML实体编码）；2) 使用CSP（内容安全策略）限制资源来源；3) Cookie设置HttpOnly属性，防止JavaScript读取；4) 避免使用innerHTML插入不可信内容；5) 使用现代前端框架的自动转义机制。

## 六、浏览器存储

### 📖 存储方案对比

| 存储方案 | 容量 | 生命周期 | 作用域 | 特点 | 适用场景 |
|----------|------|----------|--------|------|----------|
| **Cookie** | 约4KB | 可设置过期 | 同源（可跨子域） | 自动携带到服务端 | 登录态、Session ID |
| **LocalStorage** | 约5-10MB | 持久化 | 同源 | 需手动清理 | 用户偏好、主题设置 |
| **SessionStorage** | 约5-10MB | 标签页关闭 | 同源（标签页隔离） | 页面刷新保留 | 表单草稿、临时数据 |
| **IndexedDB** | 较大（磁盘限制） | 持久化 | 同源 | 异步、支持索引 | 离线应用、大量数据 |
| **Service Worker Cache** | 较大 | 持久化 | 同源 | 网络请求缓存 | PWA、离线缓存 |
| **内存存储** | 无限制（内存大小） | 页面刷新丢失 | 页面级 | 速度快 | 临时状态、React/Vue状态 |

### 🔍 实用代码示例

**LocalStorage操作**：
```javascript
// 存储对象（需序列化）
const user = { id: 1, name: 'John' };
localStorage.setItem('user', JSON.stringify(user));

// 读取
const savedUser = JSON.parse(localStorage.getItem('user'));

// 移除
localStorage.removeItem('user');

// 清空
localStorage.clear();
```

**Cookie操作**：
```javascript
// 设置Cookie
function setCookie(name, value, days) {
    const date = new Date();
    date.setTime(date.getTime() + (days * 24 * 60 * 60 * 1000));
    const expires = "expires=" + date.toUTCString();
    document.cookie = name + "=" + value + ";" + expires + ";path=/;HttpOnly;Secure;SameSite=Lax";
}

// 读取Cookie
function getCookie(name) {
    const value = `; ${document.cookie}`;
    const parts = value.split(`; ${name}=`);
    if (parts.length === 2) return parts.pop().split(';').shift();
}
```

**IndexedDB基础**：
```javascript
// 打开数据库
const request = indexedDB.open('MyApp', 1);

// 创建存储对象
request.onupgradeneeded = (event) => {
    const db = event.target.result;
    const store = db.createObjectStore('products', { keyPath: 'id' });
    store.createIndex('name', 'name', { unique: false });
};

// 存储数据
request.onsuccess = (event) => {
    const db = event.target.result;
    const transaction = db.transaction(['products'], 'readwrite');
    const store = transaction.objectStore('products');
    store.add({ id: 1, name: 'Product 1', price: 100 });
};
```

### 💬 面试问答

> **问：LocalStorage和SessionStorage的区别？**
> **答**：1) 生命周期：LocalStorage持久保存，除非手动清除；SessionStorage标签页关闭后清除。2) 作用域：LocalStorage同源窗口共享；SessionStorage同一标签页内。3) 应用场景：LocalStorage用于用户偏好设置，SessionStorage用于临时表单数据。

> **问：前端存储敏感信息的最佳实践？**
> **答**：敏感信息（如Token、用户凭证）优先使用HttpOnly Cookie，防止XSS攻击。如果必须使用前端存储，应：1) 加密存储（但密钥管理困难）；2) 设置短有效期；3) 考虑使用内存存储（页面刷新丢失）；4) 避免存储密码等核心敏感信息。

## 七、浏览器渲染与性能

### 📖 渲染流程

1. **解析HTML** → 构建DOM树
2. **解析CSS** → 构建CSSOM树
3. **合并DOM和CSSOM** → 渲染树
4. **布局（Layout/Reflow）** → 计算几何信息
5. **绘制（Paint）** → 生成绘制指令
6. **合成（Composite）** → 分层渲染到屏幕

### 📖 重排与重绘

| 操作 | 触发条件 | 性能消耗 | 优化策略 |
|------|----------|----------|----------|
| **重排** | 尺寸、位置、内容变化 | 高 | 1. 批量读写<br>2. 使用transform<br>3. 脱离文档流<br>4. DocumentFragment |
| **重绘** | 颜色、背景、阴影变化 | 中 | 减少不必要的样式修改 |
| **合成** | transform、opacity变化 | 低 | 优先使用transform做动画 |

### 🔍 性能优化技巧

**1. 减少阻塞**
```html
<!-- 关键CSS内联 -->
<style>/* 首屏样式 */</style>

<!-- 非关键CSS异步加载 -->
<link rel="preload" href="styles.css" as="style" onload="this.onload=null;this.rel='stylesheet'">

<!-- JavaScript延迟加载 -->
<script src="app.js" defer></script>
```

**2. 图片优化**
```html
<!-- 懒加载 -->
<img loading="lazy" src="image.jpg" alt="">

<!-- 响应式图片 -->
<img srcset="image-480w.jpg 480w, image-800w.jpg 800w" sizes="(max-width: 600px) 480px, 800px" src="image-800w.jpg" alt="">

<!-- WebP格式 -->
<picture>
    <source srcset="image.webp" type="image/webp">
    <img src="image.jpg" alt="">
</picture>
```

**3. 代码分割**
```javascript
// 路由懒加载
const Home = () => import('./Home.vue');
const About = () => import('./About.vue');

// 组件懒加载
const HeavyComponent = defineAsyncComponent(() => import('./HeavyComponent.vue'));
```

**4. 防抖与节流**
```javascript
// 防抖
function debounce(fn, delay) {
    let timer;
    return function(...args) {
        clearTimeout(timer);
        timer = setTimeout(() => fn.apply(this, args), delay);
    };
}

// 节流
function throttle(fn, limit) {
    let inThrottle;
    return function(...args) {
        if (!inThrottle) {
            fn.apply(this, args);
            inThrottle = true;
            setTimeout(() => inThrottle = false, limit);
        }
    };
}

// 使用
window.addEventListener('scroll', throttle(handleScroll, 100));
```

### 📖 性能指标

| 指标 | 含义 | 目标值 | 测量工具 |
|------|------|--------|----------|
| **LCP** | 最大内容绘制 | ≤ 2.5s | Lighthouse、Web Vitals |
| **INP** | 交互到下一次绘制 | ≤ 200ms | Web Vitals |
| **CLS** | 累积布局偏移 | ≤ 0.1 | Lighthouse |

| **TTFB** | 首字节时间 | ≤ 600ms | Chrome DevTools |

### 💬 面试问答

> **问：如何优化首屏加载速度？**
> **答**：1) 减少HTTP请求（合并资源、内联关键CSS）；2) 启用压缩（Gzip/Brotli）；3) 使用CDN；4) 浏览器缓存策略；5) 懒加载非关键资源；6) 代码分割和路由懒加载；7) 预加载关键资源；8) 服务端渲染（SSR）。

> **问：为什么transform性能更好？**
> **答**：transform的变化只影响合成阶段，浏览器会将元素提升到独立的合成层，由GPU直接处理，避免了CPU的布局和绘制计算。其他属性（如width、left）的变化会触发重排或重绘，性能开销更大。

## 八、从 URL 到页面展示

### 📖 完整流程

```
1. DNS解析 → 域名 → IP地址
   - 浏览器缓存 → OS缓存 → 本地DNS → 根DNS → 顶级域DNS → 权威DNS

2. TCP三次握手 → 建立连接
   - 客户端SYN → 服务端SYN+ACK → 客户端ACK

3. TLS握手（HTTPS） → 建立加密通道
   - ClientHello → ServerHello+证书 → 客户端验证证书 → 密钥交换 → 加密通信

4. 发送HTTP请求
   - 请求行（方法、URL、版本）
   - 请求头（Host、Accept、Cookie等）
   - 请求体（POST数据等）

5. 服务器处理请求
   - 路由匹配 → 业务逻辑 → 数据库操作 → 生成响应

6. 浏览器接收响应
   - 检查状态码（200/301/304等）
   - 检查缓存策略
   - 处理重定向

7. 解析HTML → 构建DOM树
   - 词法分析 → 语法分析 → DOM节点构建
   - 遇到<script>会阻塞解析（除非defer/async）

8. 解析CSS → 构建CSSOM树
   - CSS是渲染阻塞的
   - 并行构建，不阻塞DOM解析

9. 合并DOM和CSSOM → 渲染树
   - 只包含可见元素
   - 计算每个节点的样式

10. 布局（Layout/Reflow）
    - 计算元素位置和尺寸

11. 绘制（Paint）
    - 生成绘制指令

12. 合成（Composite）
    - 分层渲染到屏幕

13. 执行JavaScript
    - 可能触发重排重绘

14. 页面可交互
```

### 🔍 优化关键点

| 阶段 | 优化手段 | 效果 |
|------|----------|------|
| **DNS解析** | DNS预解析、HTTPDNS | 减少解析时间 |
| **TCP连接** | Keep-Alive、HTTP/2多路复用 | 减少连接建立时间 |
| **TLS握手** | TLS 1.3、Session Resumption | 减少握手时间 |
| **HTTP传输** | 压缩、缓存、CDN | 减少传输时间 |
| **渲染** | 关键CSS内联、defer/async、懒加载 | 减少渲染阻塞 |

### 💬 面试问答

> **问：从输入URL到页面展示的完整流程？**
> **答**：1) DNS解析域名得到IP地址；2) TCP三次握手建立连接；3) HTTPS的TLS握手建立加密通道；4) 发送HTTP请求；5) 服务器处理并返回响应；6) 浏览器检查缓存和状态码；7) 解析HTML构建DOM树；8) 解析CSS构建CSSOM树；9) 合并为渲染树；10) 布局计算几何信息；11) 绘制生成指令；12) 合成渲染到屏幕；13) 执行JavaScript；14) 页面可交互。

> **问：为什么CSS会阻塞渲染？**
> **答**：因为渲染树需要CSSOM才能构建。如果CSS未加载完成，浏览器不知道元素应该如何显示，只能等待。但CSS下载不会阻塞DOM解析（HTML解析和CSS下载并行），只是阻塞渲染树的构建和后续流程。

## 九、网络协议与分层

### 📖 OSI七层模型

| 层级 | 名称 | 功能 | 协议 | 设备 |
|------|------|------|------|------|
| 7 | 应用层 | 应用程序接口 | HTTP、HTTPS、FTP、DNS | 应用程序 |
| 6 | 表示层 | 数据格式转换 | JPEG、MPEG、ASCII | 终端设备 |
| 5 | 会话层 | 会话管理 | NetBIOS、RPC | 终端设备 |
| 4 | 传输层 | 端到端传输 | TCP、UDP | 防火墙 |
| 3 | 网络层 | 路由寻址 | IP、ICMP、ARP | 路由器 |
| 2 | 数据链路层 | 帧传输 | Ethernet、Wi-Fi | 交换机 |
| 1 | 物理层 | 物理传输 | 双绞线、光纤 | 网卡 |

### 📖 TCP/IP四层模型

| 层级 | 对应OSI | 功能 | 协议 |
|------|----------|------|------|
| 应用层 | 7-5层 | 应用程序接口 | HTTP、HTTPS、DNS、FTP |
| 传输层 | 4层 | 端到端传输 | TCP、UDP |
| 网络层 | 3层 | 路由寻址 | IP、ICMP、ARP |
| 网络接口层 | 2-1层 | 物理传输 | Ethernet、Wi-Fi |

### 📖 TCP vs UDP

| 特性 | TCP | UDP | 适用场景 |
|------|------|------|----------|
| **连接** | 面向连接 | 无连接 | TCP：HTTP、FTP；UDP：视频、语音 |
| **可靠性** | 可靠（重传、校验） | 不可靠 | TCP：文件传输；UDP：实时通信 |
| **速度** | 较慢 | 较快 | TCP：邮件；UDP：游戏 |
| **拥塞控制** | 有 | 无 | TCP：Web浏览；UDP：流媒体 |
| **首部开销** | 20-60字节 | 8字节 | TCP：可靠性要求高；UDP：速度要求高 |

### 💬 面试问答

> **问：TCP三次握手的过程？**
> **答**：1) 客户端发送SYN包（seq=x）到服务器，进入SYN_SENT状态；2) 服务器收到SYN包，发送SYN+ACK包（seq=y, ack=x+1），进入SYN_RECV状态；3) 客户端收到SYN+ACK包，发送ACK包（ack=y+1），双方进入ESTABLISHED状态，连接建立。

> **问：TCP四次挥手的过程？**
> **答**：1) 客户端发送FIN包（seq=u），进入FIN_WAIT_1状态；2) 服务器收到FIN包，发送ACK包（ack=u+1），进入CLOSE_WAIT状态；3) 服务器发送FIN包（seq=v），进入LAST_ACK状态；4) 客户端收到FIN包，发送ACK包（ack=v+1），进入TIME_WAIT状态，等待2MSL后关闭。

## 十二、校招面试高频题

### 📖 必背面试题

| 问题 | 核心要点 | 回答思路 |
|------|----------|----------|
| **HTTP和HTTPS的区别** | 加密、端口、性能 | HTTPS是HTTP的安全版本，使用TLS加密，端口443，HTTP端口80。HTTPS需要证书，性能稍低但更安全。 |
| **TCP三次握手的原因** | 防止过期连接、同步序列号 | 三次握手确保双方都了解对方的发送和接收能力，防止过期连接请求导致错误，同步初始序列号。 |
| **强缓存和协商缓存的区别** | 是否发请求、性能 | 强缓存直接使用本地缓存，不发请求，性能高；协商缓存需要发请求验证，性能稍低但更灵活。 |
| **CORS的预检请求** | 非简单请求、OPTIONS | 非简单请求（PUT/DELETE、自定义头、application/json）会先发OPTIONS预检请求，检查服务器是否允许。 |
| **XSS和CSRF的防护** | 输入转义、Token、SameSite | XSS：输入输出转义、CSP、HttpOnly；CSRF：Token、Referer校验、SameSite Cookie。 |
| **浏览器渲染流程** | DOM、CSSOM、渲染树、布局、绘制、合成 | HTML解析→DOM树，CSS解析→CSSOM树，合并→渲染树，布局计算几何信息，绘制生成指令，合成渲染到屏幕。 |
| **性能优化策略** | 网络、渲染、代码 | 网络：压缩、缓存、CDN；渲染：关键CSS内联、defer/async、懒加载；代码：分割、防抖节流、虚拟列表。 |
| **从URL到页面展示** | 完整流程、各阶段优化 | DNS解析→TCP连接→TLS握手→HTTP请求→响应→解析HTML→解析CSS→渲染树→布局→绘制→合成→执行JS。 |

### 📖 进阶面试题

| 问题 | 核心要点 | 回答思路 |
|------|----------|----------|
| **HTTP/2的特性** | 多路复用、头部压缩、服务器推送 | 多路复用：一个连接多个请求；头部压缩：HPACK算法；服务器推送：主动推送相关资源。 |
| **Service Worker的作用** | 离线缓存、后台同步、推送通知 | 拦截网络请求、实现PWA、离线访问、后台数据同步、推送通知。 |
| **IndexedDB vs LocalStorage** | 容量、异步、索引 | IndexedDB：大容量、异步API、支持索引和事务；LocalStorage：小容量、同步API、简单键值对。 |
| **WebAssembly的优势** | 性能、跨平台 | 接近原生性能、跨浏览器兼容、可由C/C++/Rust等编译。 |
| **浏览器的事件循环** | 宏任务、微任务、渲染时机 | 事件循环由宏任务和微任务组成，微任务在当前宏任务执行后立即执行，渲染在微任务执行完成后进行。 |
