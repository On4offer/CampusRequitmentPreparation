### 1.1 前端开发的演进与痛点

#### （1）演进历程

前端开发经历了几个重要的历史阶段，工程化正是为了解决各个阶段暴露出的问题而逐渐成熟的：

| 阶段 | 特征与开发模式 | 核心痛点 | 典型代表 |
|------|----------------|----------|----------|
| **古典时期（刀耕火种）** | 纯静态页面，HTML/CSS/JS 分离，通过 `<script>` 标签手动引入。 | **全局变量污染**、依赖关系不透明、文件加载顺序全靠人工维护、加载慢。 | jQuery、原生 DOM 操作 |
| **工具库时期（模块化萌芽）** | 引入 AJAX 实现前后端分离，使用 RequireJS（AMD）、SeaJS（CMD）实现初步模块化。 | 模块化标准不统一、配置繁琐、仍然需要在浏览器端动态加载解析模块。 | RequireJS、SeaJS |
| **Node.js 时代（工程化基石）**| Node.js 的诞生带来了 **npm** 与 **CommonJS** 规范，前端可以使用后端思维管理包。出现 Grunt、Gulp 等任务流工具。 | 只能做任务自动化，缺乏对现代框架（如 React/Vue）的高级语法（JSX、SFC）的深度编译支持。 | Grunt、Gulp |
| **现代化（组件化与全链路）** | Webpack 一统天下，提出“一切皆模块”。配合 Babel、TS、ESLint 形成完整的编译构建流水线。 | Webpack 随着项目膨胀，冷启动与热更新极慢，配置极其复杂。 | Webpack、Babel |
| **云原生与提效（下一代）** | 浏览器原生支持 **ES Modules (ESM)**，催生了以 **Vite** 为代表的无包（No-bundle）开发工具，Rust/Go 编写的底层工具（SWC/esbuild）极大提升了性能。 | 工具链多且杂，要求开发者对底层有一定了解。 | Vite、esbuild、pnpm |

#### （2）为什么需要前端工程化（核心价值）

> 💬 **一句话**：前端工程化就是通过**工具和规范**，将前端开发从“手工作坊”升级为“自动化流水线”，解决代码难维护、协作成本高、性能体验差的问题。

- **解决依赖与污染**：通过模块化机制，消灭全局变量冲突，清晰管理成百上千个第三方包。
- **提升开发体验**：支持 TypeScript、Sass/Less、JSX 等高级语法（编译转换）；支持热更新（HMR），修改代码毫秒级在浏览器生效。
- **保证代码质量**：通过 ESLint/Prettier 统一代码风格，通过 TypeScript 提供类型安全，通过 Husky/CI 在提交或部署前拦截低级错误。
- **优化产物性能**：自动压缩代码、Tree-shaking（摇树优化去死代码）、代码分包、图片压缩，提升首屏加载速度。

### 1.2 工程化的四大核心支柱

前端工程化可以归纳为四个“化”：

| 支柱 | 核心概念 | 典型实践与工具 |
|------|----------|----------------|
| **模块化** | 将复杂代码拆分为独立、可复用的模块。 | JS 模块化（ESM、CommonJS）、CSS 模块化（CSS Modules、Sass）。 |
| **组件化** | 在 UI 层面上，将页面拆分为结构（HTML）、样式（CSS）、逻辑（JS）高内聚的独立组件。 | React/Vue 组件、Element Plus 等组件库。 |
| **规范化** | 制定并强制执行统一的标准，降低协作阻力。 | 目录结构规范、ESLint（逻辑规则）、Prettier（代码格式）、Commitlint（提交信息）。 |
| **自动化** | 用脚本和工具替代人工重复劳动。 | 自动化构建（npm run build）、自动化测试（Vitest）、自动化部署（CI/CD 流水线）。 |

### 1.3 现代前端常见工具链生态全景

在面试和实际架构选型中，我们需要从全局视角了解当前工具链的分布与对比。

#### （1）构建工具（Bundler & Build Tool）

| 工具 | 定位与原理 | 优缺点与适用场景 |
|------|------------|------------------|
| **Webpack** | 老牌霸主，基于 Node.js，将一切资源视为模块，分析依赖图，**全量打包**。 | **优点**：生态极其成熟，loader/plugin 丰富，能处理各种极度复杂的场景。<br>**缺点**：项目大时冷启动和热更新极慢、配置复杂（“配置工程师”）。 |
| **Vite** | 新一代工具，开发环境基于浏览器原生 **ESM** 按需编译（No-bundle），生产环境用 **Rollup** 打包。 | **优点**：冷启动极快，热更新（HMR）速度不受项目规模影响。<br>**缺点**：开发与生产环境机制不同（可能产生细微差异），生态不如 Webpack 老练。<br>**适用**：目前 Vue/React 新项目首选。 |
| **Rollup** | 专注于 JS 库的打包，最早提出 Tree-shaking。 | **适用**：打包第三方库（如 Vue 源码、各种 npm 包），不适合打复杂的业务应用（业务应用更适合 Vite/Webpack）。 |
| **esbuild / SWC** | 底层编译工具，使用 Go (esbuild) / Rust (SWC) 编写。 | 速度比基于 Node.js 的 Babel 快几十倍到上百倍。目前多作为 Vite 等上层工具的底层引擎使用。 |

#### （2）包管理工具（Package Manager）

| 工具 | 核心机制 | 优缺点 |
|------|----------|--------|
| **npm** | Node 自带。v3 后采用扁平化 node_modules 结构。 | 原生支持，但下载慢、产生幽灵依赖、多项目重复占用磁盘。 |
| **yarn** | 早期为解决 npm 的慢和无 lock 文件而生。 | 速度较快，但依然基于扁平化，无法彻底解决幽灵依赖。 |
| **pnpm** | 基于**全局存储 + 符号链接（Symlink）**，非扁平化结构。 | **优点**：安装极快、极大地节约磁盘空间、严格的依赖树结构（彻底解决幽灵依赖）。<br>**适用**：现代项目与 Monorepo 的首选。 |

#### （3）规范与质量工具

- **ESLint**：检查 JavaScript/TypeScript 的语法与逻辑错误（如变量未定义）。
- **Prettier**：专业的代码格式化工具（处理缩进、引号等）。
- **Husky + lint-staged**：Git 钩子工具，在执行 `git commit` 时拦截并强制执行检查。

### 2.1 包管理工具对比（npm vs yarn vs pnpm）

| 工具 | 原理/特点 | 优缺点 |
|------|-----------|--------|
| **npm** | Node 默认自带；v3 以前嵌套层级深，v3 以后扁平化。 | 原生自带，但安装速度相对较慢，磁盘占用大。 |
| **yarn** | 引入缓存、并行下载、锁文件机制。 | 速度比老版 npm 快，但依然是扁平化结构。 |
| **pnpm** | 基于**全局存储 + 硬链接/符号链接**。项目下的 node_modules 不占额外物理空间，且不是扁平化。 | **推荐**：安装极快、省磁盘、严格的依赖隔离（解决幽灵依赖）。 |

#### （1）幽灵依赖（Phantom Dependencies）

- **什么是幽灵依赖**：package.json 里没有声明，但代码里却能 `import` 成功的包。
- **原因**：npm/yarn 为了避免重复安装，把依赖的依赖**打平**提升到了 node_modules 根目录。
- **危害**：一旦第三方包去掉了这个依赖，你的项目就会突然报错（找不到模块）。
- **解决**：使用 **pnpm**，它通过符号链接（symlink）保持严格的依赖树结构，非 package.json 声明的包无法直接引入。

------

### 2.2 依赖类型与 package.json

| 类型 | 说明 | 安装命令示例（npm / pnpm） | 场景 |
|------|------|----------------------------|------|
| **dependencies** | 生产环境依赖，打包时需要包含在最终产物中 | `npm i axios` / `pnpm add axios` | vue, react, axios, lodash |
| **devDependencies** | 开发环境依赖，仅在开发与构建时需要，不打入最终产物 | `npm i -D eslint` / `pnpm add -D eslint` | vite, eslint, typescript |
| **peerDependencies** | 同级依赖，要求宿主环境安装指定的包 | 手动配在 package.json 中 | UI 组件库要求项目安装 Vue 或 React |

------

### 2.3 锁文件与版本管理

- **锁文件**：`package-lock.json`（npm）、`pnpm-lock.yaml`（pnpm）。
- **作用**：记录每一次安装时每个包的**确切版本号**和**下载地址**，确保团队所有人（及 CI/CD 环境）安装的依赖版本绝对一致。
- **版本号符号（SemVer 规范）**：
  - `^1.2.3`：允许次版本号更新（>=1.2.3 且 <2.0.0）。
  - `~1.2.3`：仅允许修订号更新（>=1.2.3 且 <1.3.0）。
  - `1.2.3`：固定版本。

> 💡 **重点**：永远**不要把锁文件加到 .gitignore 中**，必须提交到 Git 仓库。

------

### 2.4 镜像源与 .npmrc 配置

- 默认源在国外，下载慢。常用淘宝/阿里云镜像。
- **配置方式**：项目根目录建 `.npmrc` 文件（推荐，跟代码走，团队统一）。

```ini
# .npmrc 示例
registry=https://registry.npmmirror.com/
# pnpm 开启 shamelss-hoist 可兼容幽灵依赖（不推荐，仅作遗留项目过渡）
# shamefully-hoist=true
```

------

### 2.5 常见错误写法 vs 正确写法

| 场景 | 错误写法 / 现象 | 正确写法 / 做法 |
|------|-----------------|-----------------|
| **锁文件管理** | 把 `pnpm-lock.yaml` 放进 `.gitignore` | 锁文件必须**提交到 Git**，保证环境一致。 |
| **幽灵依赖引用** | `import xxx` 能跑，但没写进 dependencies | 使用 **pnpm** 拦截，或手动检查 package.json。 |
| **切换包管理器** | 项目里同时存在 `package-lock.json` 和 `pnpm-lock.yaml` | 团队**统一包管理器**，只保留一种锁文件，使用 `npm/pnpm/yarn` 中明确规定的一种。 |
| **环境依赖放错位置** | 把 React 放到了 devDependencies | 运行时需要的核心库放 **dependencies**，构建/测试工具放 **devDependencies**。 |

### 3.1 Vite 原理简述

- **传统打包（Webpack）**：冷启动时，把所有模块解析、编译、打包成 bundle 后才启动开发服务器；项目越大越慢。
- **Vite 开发阶段（No-bundle）**：利用现代浏览器的 **原生 ESM**（`<script type="module">`）能力。浏览器请求哪个文件，Vite 服务器就即时编译返回（如把 Vue/TS 编译成 JS），**启动极快，与项目大小无关**。
- **Vite 生产阶段**：使用 **Rollup** 对代码进行 Tree-shaking、懒加载拆分与压缩打包，保证生产环境的性能与兼容性。

------

### 3.2 典型项目目录结构

```
my-app/
├── public/           # 静态资源（直接复制到 dist，不经 Vite 处理），引用如 /favicon.ico
├── src/
│   ├── assets/       # 需构建的静态资源（图片、样式），经 Vite 处理，如 import img from '@/assets/img.png'
│   ├── components/   # 公共组件
│   ├── views/        # 页面组件
│   ├── utils/        # 工具函数
│   ├── main.js/ts    # 项目入口
│   └── App.vue/jsx   # 根组件
├── index.html        # Vite 的入口文件，包含 <script type="module" src="/src/main.js"></script>
├── vite.config.js    # Vite 配置文件
├── package.json      # 依赖与脚本
└── .env              # 环境变量
```

### 3.3 vite.config 常用配置

```javascript
import { defineConfig } from 'vite'
import vue from '@vitejs/plugin-vue'
import path from 'path'

export default defineConfig({
  plugins: [vue()], // React 则用 @vitejs/plugin-react
  
  resolve: {
    // 别名配置：把 @ 映射到 src 目录
    alias: {
      '@': path.resolve(__dirname, 'src')
    }
  },
  
  server: {
    port: 3000,
    open: true,
    // 开发代理：解决本地联调跨域问题
    proxy: {
      '/api': {
        target: 'http://localhost:8080',
        changeOrigin: true, // 改变请求的 origin，欺骗后端
        rewrite: (path) => path.replace(/^\/api/, '') // 根据后端需不需要 /api 前缀来重写
      }
    }
  },
  
  build: {
    outDir: 'dist', // 打包输出目录
    sourcemap: false, // 生产不开启 sourcemap 减小体积
  }
})
```

### 3.4 环境变量与多环境

#### （1）环境文件

Vite 默认加载项目根目录下的 `.env` 文件，可使用 mode 区分：

- `.env`：所有环境生效
- `.env.development`：执行 `vite`（开发）时加载
- `.env.production`：执行 `vite build`（生产）时加载

#### （2）变量定义与使用

- **定义**：只有以 **`VITE_`** 开头的变量才会暴露给客户端（防泄密）。

```ini
# .env.development
VITE_API_BASE=/api
VITE_APP_TITLE=测试版

# .env.production
VITE_API_BASE=https://api.example.com
VITE_APP_TITLE=正式版
```

- **使用**：在代码中通过 `import.meta.env.VITE_API_BASE` 读取。

> 💬 **面试怎么说**：多环境通过 `.env.xxx` 文件配 `VITE_` 变量；开发时配 proxy 走本地代理，生产构建后请求线上真实域名，由 axios 根据环境变量自动拼前缀。

### 3.5 构建常见错误 vs 正确写法

| 场景 | 错误写法 / 现象 | 正确写法 / 做法 |
|------|-----------------|-----------------|
| **引 public 目录图片** | `import img from '../../public/logo.png'` 导致打包双份或报错 | 直接用绝对路径 `/logo.png` 或不经 import 走 url；需要处理的图放 `src/assets`。 |
| **环境变量不生效** | 变量名叫 `API_BASE`，客户端取不到 | 必须加 **VITE_** 前缀：`VITE_API_BASE`；修改 env 文件后**需重启 dev server**。 |
| **部署后跨域或 404** | 线上请求仍是 `http://localhost/api/...` | proxy 仅在本地开发生效；生产代码应请求完整线上域名（环境变量），或通过 Nginx 做反向代理。 |
| **别名 TS 报错** | vite.config 配了 `@`，但 TS 代码里报错找不到模块 | 在 `tsconfig.json` 的 `compilerOptions.paths` 里也要配 `"@/*": ["./src/*"]`。 |

## 第4章：代码规范与质量

------

> **本章在整体中解决什么问题**：第三章解决了“代码怎么跑起来和打包”，本章解决**“代码质量怎么保证”**——如何通过 **ESLint** 检查语法与逻辑错误，通过 **Prettier** 统一代码格式，并通过 **Husky + lint-staged** 在 Git 提交时强制自动校验，避免“垃圾代码”合入代码库。掌握后能独立搭建团队的前端规范检查流。

------

### 4.1 ESLint 与 Prettier

#### （1）分工与定位

- **ESLint**：侧重**代码质量与逻辑**检查（如：变量声明未使用、不能用 `==` 只能用 `===`、Hooks 调用顺序报错）。
- **Prettier**：侧重**代码格式化**（如：缩进是 2 还是 4、单引号还是双引号、行尾是否加分号）。
- **冲突解决**：当 ESLint 的格式规则和 Prettier 冲突时，使用 `eslint-config-prettier` 关闭 ESLint 中所有与格式相关的规则，**把格式化完全交给 Prettier**。

#### （2）核心配置文件

- **ESLint**：`.eslintrc.cjs`（或 `eslint.config.js` 新版扁平化配置）。常用继承：`extends: ['plugin:vue/vue3-recommended', 'prettier']`。
- **Prettier**：`.prettierrc`。

```json
// .prettierrc 示例
{
  "semi": false,
  "singleQuote": true,
  "trailingComma": "none",
  "printWidth": 100
}
```

------

### 4.2 Husky 与 lint-staged

#### （1）为什么要用

如果只配了 ESLint，开发者忘记运行 `npm run lint` 直接 commit，依然会有不规范代码进入仓库。
通过 **Git Hooks** 可以在特定操作（如 commit、push）触发前执行脚本强制拦截。

#### （2）工具介绍

- **Husky**：让配置 Git Hooks 变得极其简单。
- **lint-staged**：只针对**当前通过 `git add` 暂存的文件**进行 lint 检查，而不是全量检查，速度极快。

#### （3）工作流（面试可答）

1. 开发者执行 `git commit -m "xxx"`。
2. 触发 Husky 的 `pre-commit` 钩子。
3. `pre-commit` 执行 `npx lint-staged` 命令。
4. `lint-staged` 对暂存区的文件执行 `eslint --fix` 和 `prettier --write`。
5. 校验通过则 commit 成功；报错则中断 commit，强制开发者修改。

------

### 4.3 提交规范（Commitlint）

- 规范 commit message，便于生成 Changelog 和回溯历史。
- 常用规范（Angular 规范）：`<type>(<scope>): <subject>`。
- **type 类型**：
  - `feat`: 新功能
  - `fix`: 修复 bug
  - `docs`: 文档修改
  - `style`: 代码格式修改（不影响逻辑）
  - `refactor`: 重构
  - `test`: 测试代码
  - `chore`: 构建过程或辅助工具变动
- **工具实现**：`@commitlint/cli` + Husky 的 `commit-msg` 钩子。

------

### 4.4 规范化常见错误 vs 正确写法

| 场景 | 错误写法 / 现象 | 正确写法 / 做法 |
|------|-----------------|-----------------|
| **ESLint 与 Prettier 冲突** | 保存时格式化一会儿单引号一会儿双引号 | 安装 `eslint-config-prettier`，在 eslintrc 的 extends **最后** 加入 `'prettier'`。 |
| **全量 lint 极慢** | 每次 pre-commit 跑 `npm run lint`（检查全部文件） | 使用 **lint-staged**，在 package.json 配置仅对 `*.{js,ts,vue}` 暂存文件执行 lint。 |
| **绕过 Git Hooks** | 同事用 `git commit -n` 强行绕过校验 | CI 流水线中增加代码校验节点（MR/PR 时），本地绕过也无法合入主干。 |

### 5.1 前端自动化测试简介

- **单元测试（Unit Test）**：测试一个函数、一个组件。Vite 项目推荐使用 **Vitest**（API 兼容 Jest，速度极快）。
- **组件测试**：测试组件的 DOM 渲染与事件。Vue 常用 `Vue Test Utils`，React 常用 `React Testing Library`。
- **端到端测试（E2E Test）**：模拟真实用户在浏览器中的点击、输入流程。主流工具：**Cypress**、**Playwright**。
  
> **面试/实战建议**：校招一般不要求手写极复杂的单测，但要知道「Vitest 测逻辑，Cypress 测流程」。

------

### 5.2 构建产物（Build）

- 执行 `npm run build`，Vite 会使用 Rollup 进行打包，默认输出到 `dist` 目录。
- **产物内容**：一个 `index.html`，以及包含 js、css、图片等压缩后文件的 `assets` 目录。文件名通常带有 hash（如 `main-8a7b6c.js`）用于长期缓存。

------

### 5.3 Nginx 部署与单页应用（SPA）配置

#### （1）静态资源托管

将打包后的 `dist` 文件夹上传到服务器，通过 Nginx 托管：

```nginx
server {
    listen       80;
    server_name  example.com;
    
    # 指向打包后的前端产物目录
    root   /usr/share/nginx/html/dist;
    index  index.html;

    # 解决 SPA 路由刷新 404 问题（重要！）
    location / {
        try_files $uri $uri/ /index.html;
    }

    # （可选）线上跨域反向代理
    location /api/ {
        proxy_pass http://backend-server:8080/;
    }
}
```

#### （2）为什么 SPA 刷新会 404？

- **原因**：Vue/React 用的 history 路由（如 `/user/profile`）是前端用 JS 模拟的。如果用户直接在浏览器刷新该路径，Nginx 会去服务器物理目录下找 `user/profile` 这个文件，找不到就会报 404。
- **解决**：配置 `try_files $uri $uri/ /index.html;`。意思是如果找不到这个文件，就把请求重定向给 `index.html`，然后由前端路由接管并渲染对应的组件。

------

### 5.4 部署常见错误 vs 正确写法

| 场景 | 错误写法 / 现象 | 正确写法 / 做法 |
|------|-----------------|-----------------|
| **SPA 路由刷新 404** | 没配 `try_files`，一刷新就页面丢失 | Nginx 增加 `try_files $uri $uri/ /index.html;` 让所有未知路径回退到入口页。 |
| **静态资源缓存未更新** | 页面更新了但用户看到的还是旧版 | 打包时静态文件（JS/CSS）名自带 hash；确保 Nginx 对 `index.html` 不设强缓存（配置 `Cache-Control: no-cache`）。 |
| **接口线上跨域** | 线上代码依然请求 `http://localhost:8080/api` | 开发用 Vite proxy，生产用 **环境变量** 记录线上真实域名，或用 Nginx 的 `proxy_pass` 转发。 |

## 第6章：Monorepo 与进阶

------

> **本章在整体中解决什么问题**：前面的章节解决了“单体项目”的工程化问题；本章收束**大型项目架构与性能进阶**——认识 **Monorepo** 多包同仓架构、前端性能优化的构建手段（懒加载、Tree-shaking），并对整个前端工程化面试高频考点进行总结。

------

### 6.1 Monorepo 多包架构简介

- **Polyrepo（多仓库）**：每个项目（如组件库、后台系统、前台H5）各建一个 Git 仓库。缺点：依赖升级繁琐、跨仓库代码复用极难、发版不同步。
- **Monorepo（单体仓库）**：把相关的多个项目放在**同一个 Git 仓库**里。
  - **优势**：极易复用公共组件和 utils 工具包；依赖管理集中；只需提一次 MR 就能改多处关联代码。
  - **工具**：**pnpm workspace**（最常用，通过 `pnpm-workspace.yaml` 配置）、**Turborepo**（Vercel 出品，极大优化 Monorepo 的构建缓存与提速）。
  - **应用场景**：Vue 3 源码、React 源码、大厂内部的中后台微前端矩阵、组件库开发等。

------

### 6.2 前端性能与体积优化（构建层面）

工程化不仅是跑起来，还要“跑得快”。在构建层可做以下优化：

| 优化手段 | 原理与说明 |
|----------|------------|
| **路由/组件懒加载** | 利用动态 `import()` 拆分 JS bundle，避免首屏加载所有代码。 |
| **Tree-shaking** | 依赖于 ES Module 的静态分析，打包时“摇掉”未引用的僵尸代码（导出但没使用的函数），减小体积。 |
| **代码分包（Split Chunks）** | 把第三方库（如 Vue, lodash）单独打成一个 vendor.js 包。业务代码常变，但库不常变，便于浏览器长期缓存库文件。 |
| **图片压缩与 CDN** | 小图转 base64 减少 HTTP 请求，大图用 WebP 格式或走 CDN 分发。 |
| **Gzip / Brotli 压缩** | 在 Vite 开启压缩插件，Nginx 上也开启 gzip，大幅缩小网络传输文件体积。 |

------

### 6.3 面试要点：工程化的全局理解

面试官常问宏观问题：“谈谈你对前端工程化的理解”。推荐**按生命周期**作答，体现全局视野：

> “前端工程化是为了提升效率和保证质量，涵盖了项目全生命周期：
> 1. **创建与包管理**：用 Vite 快速构建，用 pnpm 解决幽灵依赖与提速。

## 🎯 面试常见追问

| 面试官提问 | 回答思路 |
|------------|----------|
| 什么是 Tree-shaking？有什么前提？ | 顾名思义是把没用到的死代码“摇掉”减小体积。前提是代码必须是 **ES Module** 模块（依赖 import/export 静态语法分析），CommonJS（require）无法精确分析，做不了。 |
| 多项目代码共享怎么做？ | 传统可以用 npm 私有包发版；现代大型应用更推荐使用 **Monorepo** 架构，把基础组件库和多个子应用放一个 Git 库里，用 pnpm workspace 直接本地软链，改了核心包其它应用直接生效。 |

------

### 常见坑与注意点

| 现象 / 易错点 | 原因 | 怎么改 / 怎么记 |
|---------------|------|-----------------|
| Tree-shaking 失效 | 引入了 CommonJS 的库或写了带有“副作用”的代码 | 尽量使用支持 ESM 的库（如用 `lodash-es` 代替 `lodash`）；在 package.json 声明 `"sideEffects": false` 辅助构建工具优化。 |
