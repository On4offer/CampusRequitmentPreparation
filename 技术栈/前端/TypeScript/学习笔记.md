### 1.1 TypeScript 简介

#### （1）定义与定位

**TypeScript** 是由微软开发并开源的一种编程语言，它是 JavaScript 的**超集（Superset）**。
- **超集意味着什么**：任何合法的 JavaScript 代码都是合法的 TypeScript 代码。TS 在 JS 的基础上增加了**静态类型系统**和**最新的 ECMAScript 特性**。
- **编译时运行**：TS 不能直接在浏览器或 Node.js 中运行，必须经过编译器（`tsc`）或打包工具（Vite/Babel/SWC）**编译（擦除类型）**为纯 JavaScript 后才能执行。

#### （2）传统 JS 开发的痛点（为什么需要 TS）

| 痛点 | 说明 |
|------|------|
| **运行时才发现错误** | JS 是动态弱类型语言，拼写错误、对象属性不存在、传参类型错误等，只有在代码运行到那一小段时才会报错（常常在生产环境引发白屏）。 |
| **重构就像排雷** | 在大型项目中修改一个基础对象的结构，很难知道究竟影响了多少个文件，搜索替换容易遗漏。 |
| **开发体验差** | 缺乏类型推导，IDE 无法准确提示对象有哪些属性，函数需要什么参数，全靠翻文档或看源码。 |

- **TS 的应对**：**静态类型检查**（把错误从运行时提前到**编译时**甚至**敲代码时**）；**极佳的智能提示**（IDE 可无缝跳转、补全）；**代码即文档**（接口的定义就是最好的参数说明）。

------

### 1.2 环境与配置 (tsconfig.json)

#### （1）基本安装与编译

在前端工程（如 Vite 创建的项目）中，TS 通常已经配置好。若需从零手动使用：

```bash
# 全局或局部安装
npm i typescript -D

# 初始化配置文件 tsconfig.json
npx tsc --init

# 编译 ts 文件为 js 文件
npx tsc index.ts
```

#### （2）tsconfig.json 核心配置解析

`tsconfig.json` 是 TS 项目的灵魂，控制着编译行为和严格程度。

```json
{
  "compilerOptions": {
    "target": "ESNext",         // 编译产物的 JS 版本（ES5、ES6、ESNext 等）
    "module": "ESNext",         // 模块化规范（CommonJS 或 ESNext）
    "strict": true,             // 开启所有严格的类型检查（推荐开启，极其重要！）
    "noImplicitAny": true,      // 不允许隐式的 any 类型
    "strictNullChecks": true,   // 严格的 null/undefined 检查
    "baseUrl": ".",             // 解析非相对模块的基准目录
    "paths": {                  // 路径别名（需与 Vite/Webpack 的 alias 保持一致）
      "@/*": ["src/*"]
    }
  },
  "include": ["src/**/*.ts", "src/**/*.vue"], // 需要编译的文件
  "exclude": ["node_modules", "dist"]         // 排除的文件
}
```

## 🎯 面试常见追问

| 面试官提问 | 回答思路 |
|------------|----------|
| TypeScript 和 JavaScript 的区别是什么？ | TS 是 JS 的超集，增加了静态类型检查。JS 是动态弱类型，运行时才暴露类型错误；TS 在编译阶段（编写时）就能发现错误。且 TS 编译后会变成纯 JS 运行。 |

## 第2章：基础类型与推断

------

> **本章在整体中解决什么问题**：本章是 TS 语法的入门——掌握常用的**基础类型**（number、string 等）、**数组与元组**的区别、以及**联合类型与字面量类型**。同时深入理解 TS 的**类型推断**机制和**类型断言**的使用场景。掌握后，**第三章**会接着讲更复杂的**对象与接口**。

------

### 2.1 基础数据类型

TS 涵盖了 JS 的所有基础类型，并提供了显式的类型注解（`: Type`）。

```typescript
// 基本类型
let age: number = 25;
let userName: string = "Alice";
let isDone: boolean = false;
let u: undefined = undefined;
let n: null = null;
let sym: symbol = Symbol("id");
let big: bigint = 100n; // 需 target 为 ES2020
```

> 🔹 **注意**：在 TS 中，`null` 和 `undefined` 是所有类型的子类型（如果不开启 `strictNullChecks`），但**强烈建议开启 strict 模式**，开启后它们只能赋值给自己或 `any/unknown`。

------

### 2.2 数组与元组（Tuple）

#### （1）数组

有两种声明数组的方式：`类型[]` 或泛型 `Array<类型>`。

```typescript
let list1: number[] = [1, 2, 3];
let list2: Array<string> = ["a", "b", "c"];

// 如果数组里既有数字又有字符串呢？使用联合类型：
let mixed: (number | string)[] = [1, "a", 2];
```

#### （2）元组（Tuple）

**元组**是 TS 特有的类型，表示**已知元素数量和类型的数组**，各元素的类型可以不同。常用于 React 的 Hook 返回值（如 `useState`）。

```typescript
let user: [number, string] = [1, "Alice"];
// user[0] 明确推断为 number，user[1] 为 string
// user[2] = "error" // 报错：长度超出
```

------

### 2.3 联合类型与字面量类型

#### （1）联合类型（Union Types）

表示一个值可以是**几种类型之一**。用 `|` 分隔。

```typescript
let status: string | number;
status = "success";
status = 200;
```

#### （2）字面量类型

不仅可以限制类型，还可以**限制具体的值**。通常与联合类型配合，实现类似枚举的效果。

```typescript
// direction 只能是这三个字符串之一
let direction: 'left' | 'right' | 'top' | 'bottom';
direction = 'left';  // OK
// direction = 'up'; // Error: 不能将类型“"up"”分配给类型“"left" | "right" | "top" | "bottom"”
```

------

### 2.4 any vs unknown vs void

| 类型 | 说明 | 适用场景 / 注意点 |
|------|------|-------------------|
| **any** | 任意类型，**放弃类型检查**。 | 极度不推荐。过度使用会让 TS 变成 AnyScript。仅在老项目迁移或极特殊情况使用。 |
| **unknown** | 任意类型，但**类型安全**。 | 推荐作为 any 的替代。unknown 类型的值在进行操作前，**必须先进行类型校验或断言**。 |
| **void** | 没有任何类型，通常用于**没有返回值的函数**。 | `function log(): void { console.log('hi'); }` |
| **never** | 永不存在的值的类型。 | 抛出异常的函数、死循环，或者在高级类型条件判断中作为底线兜底。 |

------

### 2.5 类型推断（Type Inference）

TS 并不是每个地方都需要写类型。如果没写类型注解，TS 会根据赋的**初始值**自动推断出类型。

```typescript
let age = 25; // TS 会自动推断 age 为 number
// age = "abc"; // Error: 不能将类型“string”分配给类型“number”

// 复杂推断
let arr = [1, "hello"]; // 推断为 (number | string)[]
```

> 💬 **最佳实践**：**能让 TS 自动推断的，就尽量不手写类型**（如简单变量初始化），保持代码简洁；在函数参数、复杂的对象和返回值处尽量显式声明。

------

### 2.6 类型断言（Type Assertion）

当你比 TS 编译器**更清楚某个值的确切类型**时，可以用断言强制指定。
**语法**：`值 as 类型` （推荐，兼容 JSX） 或 `<类型>值`。

```typescript
// 假设有个不知道类型的变量
let someValue: unknown = "this is a string";

// 我们明确知道它是字符串，想获取长度
let strLength: number = (someValue as string).length;
```

**非空断言（`!`）**：告诉 TS “这个值绝对不会是 null 或 undefined”。

```typescript
const el = document.getElementById("app"); // el 被推断为 HTMLElement | null
el!.style.color = "red"; // 加 ! 告诉 TS el 一定存在
```

### 2.7 常见错误写法 vs 正确写法

| 场景 | 错误写法 / 现象 | 正确写法 / 做法 |
|------|-----------------|-----------------|
| **滥用 any** | `let data: any = res.data; data.push(1)` | 改用 **interface** 定义确切结构，或用 **unknown** 并在使用前做类型缩小。 |
| **非空断言滥用** | 在运行时可能为 null 的地方写 `obj!.name`，导致生产环境抛错 | 尽量用**可选链（Optional Chaining）** `obj?.name`，TS 既不报错，运行时也安全。 |
| **多此一举写类型** | `let num: number = 10;` | 简单赋值直接 `let num = 10;`，让**类型推断**发挥作用。 |

| any 和 unknown 有什么区别？ | 两者都可以接收任何类型的值。但 any 放弃了类型检查，你可以对 any 调用任何属性和方法（不安全）；unknown 是安全的 any，在对 unknown 类型的值执行操作前，必须通过类型收窄（如 typeof 判断或断言）确定其具体类型，否则 TS 会报错。 |

## 第3章：对象、接口与类型别名

------

> **本章在整体中解决什么问题**：第二章解决了基本数据类型；本章解决**复杂结构（对象）**的类型定义。你将学习 **Interface（接口）** 和 **Type（类型别名）** 的语法、它们之间的**核心差异**，以及日常开发必备的**交叉类型**与**内置工具类型**（如 Partial、Pick 等）。这是 TS 中使用频率最高的章节。

------

### 3.1 接口（Interface）

**接口** 是定义对象结构的核心方式。它不仅可以描述对象的属性，还能描述函数类型和类。

#### （1）基本语法与可选/只读属性

```typescript
interface User {
  id: number;           // 必填属性
  name: string;
  age?: number;         // 可选属性（可为 number 或 undefined）
  readonly role: string;// 只读属性（初始化后不可修改）
}

const user: User = {
  id: 1,
  name: "Alice",
  role: "admin"
};
// user.role = "user"; // Error: 无法分配到 "role" ，因为它是只读属性。
```

#### （2）索引签名（动态属性）

当你不知道对象会有哪些具体属性，但知道它们的类型时，使用索引签名。

```typescript
interface StringArray {
  // 索引必须是 string，值必须是 string
  [key: string]: string;
}

const dict: StringArray = {
  "foo": "bar",
  "baz": "qux"
};
```

#### （3）接口继承（extends）

接口可以通过 `extends` 继承一个或多个接口，实现属性复用。

```typescript
interface Animal {
  name: string;
}
interface Dog extends Animal {
  breed: string;
}
// Dog 自动拥有 name 和 breed 属性
```

------

### 3.2 类型别名（Type Alias）

**Type** 给一个类型起个新名字。它不仅能定义对象，还能定义基本类型、联合类型、元组等。

```typescript
// 定义基本类型、联合类型
type ID = string | number;

// 定义对象类型
type Point = {
  x: number;
  y: number;
};
```

#### （1）交叉类型（&）

类型别名通过 `&`（交叉类型）来实现类似接口的“继承”效果，即**取类型的并集**（包含所有属性）。

```typescript
type Name = { name: string };
type Age = { age: number };

// Person 同时拥有 name 和 age
type Person = Name & Age; 
```

------

### 3.3 Interface 与 Type 的对比（高频面试题）

| 对比项 | `interface` (接口) | `type` (类型别名) |
|--------|---------------------|-------------------|
| **扩展方式** | 使用 `extends` | 使用交叉类型 `&` |
| **能定义什么** | 只能定义**对象/函数/类**的形状 | 能定义**任意类型**（含联合类型、基本类型等） |
| **同名合并** | **支持声明合并**：定义多个同名 interface 会自动合并属性 | **不支持**：同名 type 会报错 |
| **映射与高级类型**| 不能使用 `in` 关键字等映射高级类型 | 适合用来写复杂的高级类型、条件类型 |

> 💬 **团队选型建议**：通常建议在定义**对象结构、API 返回值、组件 Props** 时首选 `interface`；当需要使用**联合类型、交叉类型或复杂的泛型计算**时，使用 `type`。

------

### 3.4 常用内置工具类型（Utility Types）

TS 提供了一套全局可用的泛型工具类型，用于**基于已有类型快速派生新类型**，面试和实战极高频。

| 工具类型 | 作用 | 示例场景 |
|----------|------|----------|
| **`Partial<T>`** | 将 `T` 的所有属性变为**可选**（`?`） | 用于表单的局部更新接口（如 `updateUser` 传部分字段）。 |
| **`Required<T>`** | 将 `T` 的所有属性变为**必选** | 消除可选状态。 |
| **`Readonly<T>`** | 将 `T` 的所有属性变为**只读** | 保护传入的数据不被意外篡改。 |
| **`Pick<T, K>`** | 从 `T` 中**提取**一部分属性 `K` 组成新类型 | 后端返回字段很多，组件只需其中几个时。 |
| **`Omit<T, K>`** | 从 `T` 中**剔除**一部分属性 `K` 组成新类型 | 定义新建用户的类型（Omit 掉 id 和 createdAt）。 |
| **`Record<K, V>`**| 构建一个键为 `K`、值为 `V` 的对象类型 | 定义字典/映射表：`Record<string, User>`。 |

```typescript
interface Todo {
  id: number;
  title: string;
  completed: boolean;
}

// 场景 1：更新 Todo，只需要部分字段
type UpdateTodoDto = Partial<Todo>; 

// 场景 2：列表中只需要 id 和 title
type TodoPreview = Pick<Todo, 'id' | 'title'>;
```

------

### 3.5 常见错误 vs 正确写法

| 场景 | 错误写法 / 现象 | 正确写法 / 做法 |
|------|-----------------|-----------------|
| **扩展对象结构** | 用多个重复结构的 interface 复制粘贴属性 | 使用 `extends` (接口) 或 `&` (Type) 实现复用。 |
| **提取部分字段** | 重新手写一个新 interface 包含那几个字段 | 使用 **`Pick<T, K>`** 或 **`Omit<T, K>`**，保证数据源头单一，同步更新。 |
| **同名类型冲突** | 重复定义两个同名 `type` 导致报错 | 对象扩展建议使用 `interface`，它会自动执行声明合并。 |

| 面试官提问 | 回答思路 |
|------------|----------|
| interface 和 type 的区别？ | 1. 扩展方式不同（extends vs &）；2. 适用范围不同（interface 仅限对象，type 可接任何类型）；3. 同名合并（interface 会自动合并属性，type 会报错）。 |
| Pick 和 Omit 分别是用来干什么的？ | 都是从已有类型中派生新类型。Pick 提取指定的几个键；Omit 排除指定的几个键。在实际业务中做 DTO 或 Props 类型裁剪时极常用。 |

### 常见坑与注意点

| 现象 / 易错点 | 原因 | 怎么改 / 怎么记 |
|---------------|------|-----------------|
| 索引签名与明确属性冲突 | 索引签名规定了所有属性的值类型，明确属性如果类型不符会报错。 | 如 `[key: string]: string; id: number` 会报错。应改为 `[key: string]: string | number`。 |

## 第4章：函数与泛型

------

> **本章在整体中解决什么问题**：本章解决**动态与复用**的类型问题。函数不仅有入参和出参，有时还需要**重载**来处理多种传参组合；**泛型**（Generics）相当于类型的“变量”，是 TS 最强大但也最难懂的特性。掌握本章能让你写出像公共组件、通用请求封装等高复用性代码。

------

### 4.1 函数类型定义

#### （1）声明参数与返回值

可以同时定义输入参数类型和返回值类型。如果没写返回值类型，TS 也会自动推断。

```typescript
// 声明式
function add(x: number, y: number): number {
  return x + y;
}

// 箭头函数 / 函数表达式
const multiply = (x: number, y: number): number => x * y;
```

#### （2）可选参数、默认参数与剩余参数

- **可选参数**（`?`）：必须放在必需参数的**最后面**。
- **剩余参数**（`...rest`）：类型通常为数组。

```typescript
function buildName(first: string, last?: string): string {
  return last ? `${first} ${last}` : first;
}

function sum(...nums: number[]): number {
  return nums.reduce((a, b) => a + b, 0);
}
```

------

### 4.2 函数重载（Overload）

**场景**：同一个函数，根据传参类型或数量的不同，返回不同类型的值。
**实现**：先写多个**重载签名**（没有函数体），最后写一个**实现签名**（包含完整逻辑，参数类型必须兼容上面所有的重载签名）。

```typescript
// 1. 重载签名 1
function reverse(x: string): string;
// 2. 重载签名 2
function reverse(x: number[]): number[];

// 3. 实现签名（注意：实现签名对外部是不可见的，外部只能调用上面的两种形式）
function reverse(x: string | number[]): string | number[] {
  if (typeof x === "string") {
    return x.split("").reverse().join("");
  } else {
    return x.slice().reverse();
  }
}

reverse("hello"); // 正确，返回 string
reverse([1, 2]);  // 正确，返回 number[]
// reverse(123);  // 报错：没有与此调用匹配的重载。
```

### 4.3 泛型（Generics）入门

> **一句话**：泛型就是**把类型当作参数传递**，让一段代码可以支持多种数据类型，同时保持**严格的类型安全**。

#### （1）为什么需要泛型

如果我们想写一个 `identity` 函数，原样返回入参：
- 用具体类型：`function id(x: number): number`，只能传数字。
- 用 any：`function id(x: any): any`，失去了类型检查（传入字符串，TS 不知道返回的也是字符串）。
- **用泛型 `<T>`**：捕获用户传入的类型。

```typescript
function identity<T>(arg: T): T {
  return arg;
}

// 明确指定类型为 string
let output1 = identity<string>("myString");

// 类型推断（推荐）：TS 会自动推断 T 为 string
let output2 = identity("myString"); 
```

#### （2）泛型接口与泛型类

泛型不仅用在函数上，也广泛用于接口（如后端分页返回数据）和类。

```typescript
// 泛型接口：T 代表具体数据的类型
interface ResponseData<T> {
  code: number;
  message: string;
  data: T;
}

// 实际使用：用具体的 User 替换 T
interface User { name: string; }
const res: ResponseData<User> = {
  code: 200,
  message: "success",
  data: { name: "Alice" }
};
```

------

### 4.4 泛型约束（extends）与 keyof

有时候泛型太“宽泛”了，我们需要它具有某些特定属性（比如必须有 `length` 属性）。可以使用 `extends` 进行**泛型约束**。

#### （1）基本约束

```typescript
interface Lengthwise {
  length: number;
}

// T 必须包含 Lengthwise 接口定义的属性（即必须有 length）
function loggingIdentity<T extends Lengthwise>(arg: T): T {
  console.log(arg.length);  // 现在不会报错了
  return arg;
}

loggingIdentity({ length: 10, value: 3 }); // OK
// loggingIdentity(3); // Error: number 没有 length 属性
```

#### （2）在泛型约束中使用类型参数（keyof）

`keyof` 操作符会提取一个对象类型的所有键，返回一个**字符串字面量联合类型**。

```typescript
// K 必须是 T 的属性名之一
function getProperty<T, K extends keyof T>(obj: T, key: K): T[K] {
  return obj[key];
}

const obj = { a: 1, b: "hello" };
getProperty(obj, "a"); // OK，返回 number
// getProperty(obj, "c"); // Error: 类型“"c"”的参数不能赋给类型“"a" | "b"”的参数
```

### 5.1 类（Class）与访问修饰符

TS 对 ES6 的 Class 进行了增强，增加了类型注解和控制外部访问权限的修饰符。

| 修饰符 | 说明 | 适用范围 |
|--------|------|----------|
| **public** | 公有的（默认）。 | 类的内部、子类、实例都能访问。 |
| **private** | 私有的。 | 只能在**声明它的类的内部**访问（子类与实例均不可访问）。 |
| **protected** | 受保护的。 | 只能在**类的内部及其子类**中访问（实例不可访问）。 |
| **readonly** | 只读属性。 | 只能在声明时或构造函数（`constructor`）中被初始化。 |

```typescript
class Animal {
  // 参数属性写法：直接在构造函数参数前加修饰符，等同于声明并赋值属性
  constructor(public name: string, private age: number) {}

  public greet() {
    console.log(`Hi, I am ${this.name}, ${this.age} years old.`);
  }
}

const dog = new Animal("Buddy", 3);
dog.greet();
// dog.age; // 报错：属性“age”为私有属性，只能在类“Animal”中访问。
```

> **抽象类（abstract class）**：不能被实例化，主要用于被子类继承并要求子类**必须实现**其抽象方法。

------

### 5.2 进阶类型：条件类型与 infer（类型体操）

#### （1）条件类型（Conditional Types）

类似于 JS 的三元表达式，语法为 `T extends U ? X : Y`。意思是：**如果泛型 T 能够赋值给 U（兼容 U），那么类型为 X，否则类型为 Y**。

```typescript
// 判断一个类型是否为 string
type IsString<T> = T extends string ? true : false;

type A = IsString<string>; // true
type B = IsString<number>; // false
```

#### （2）`infer` 关键字（推断）

`infer` 只能在条件类型的 `extends` 子句中使用，用于**声明一个类型变量并且对它进行提取（推断）**。常用于提取函数的返回值类型、Promise 内部的类型等。

```typescript
// 提取 Promise 的内部类型
type Unpromisify<T> = T extends Promise<infer U> ? U : T;

type C = Unpromisify<Promise<string>>; // string
type D = Unpromisify<number>;          // number（走 false 分支）
```

> 💬 **一句话总结**：条件类型 `extends ? :` 负责判断流转，`infer` 负责在判断成功时**把未知的类型抠出来**拿给结果用。官方内置的 `ReturnType<T>` 就是用 `infer` 实现的。

------

### 5.3 声明文件（.d.ts）与 declare

当我们在 TS 中引入用原生 JS 写的第三方库（如 `lodash`）时，TS 编译器不知道它的类型，会报错。

- **解决方式 1（推荐）**：安装社区维护的类型包 `npm i -D @types/lodash`。
- **解决方式 2（自己写声明文件）**：在项目根目录新建一个 `global.d.ts` 或 `typings.d.ts`，使用 `declare` 关键字为它声明类型。

```typescript
// typings.d.ts
// 为全局变量声明类型（如引入了某 SDK 的全局对象）
declare const wx: any;

// 为没有类型包的第三方 npm 模块声明（消除报错）
declare module "some-untyped-module" {
  export function doSomething(): void;
}
```

------

### 5.4 进阶常见错误 vs 正确写法

| 场景 | 错误写法 / 现象 | 正确写法 / 做法 |
|------|-----------------|-----------------|
| **JS库报类型缺失** | `import _ from 'lodash'` 报错：找不到模块或其相应的类型声明。 | 优先运行 `npm i -D @types/lodash`；若无社区类型，则在 `env.d.ts` 中写 `declare module 'lodash';` 兜底（类型为 any）。 |
| **类的私有属性约束** | JS 原生 `#age` 与 TS 的 `private age` 混用。 | JS 运行时的私用字段是 `#`（ES2022），TS `private` 只在编译时做检查（编译后就没有 private 了），两者选其一即可，推荐 TS 语法保持统一。 |

### 6.1 TypeScript 在 Vue 3 中的应用

Vue 3 原生对 TS 支持极好，通常结合 `<script setup lang="ts">` 使用。

#### （1）响应式数据 ref 与 reactive

```vue
<script setup lang="ts">
import { ref, reactive } from 'vue'

// 1. 类型推断（简单值推荐）
const count = ref(0) // 推断为 Ref<number>

// 2. 显式指定类型（复杂对象、初始可能为空时推荐）
interface User { id: number; name: string }
const user = ref<User | null>(null)

// 3. reactive 的类型
const state = reactive<User>({ id: 1, name: "Alice" })
</script>
```

#### （2）定义 Props 与 Emits（推荐基于类型的声明）

利用编译宏 `defineProps` 和 `defineEmits`，直接传递泛型参数。

```vue
<script setup lang="ts">
// 定义 Props
// 若需默认值，推荐使用 withDefaults 宏
interface Props {
  title: string;
  count?: number;
}
const props = withDefaults(defineProps<Props>(), {
  count: 0
});

// 定义 Emits
// 参数 (e: 事件名, 载荷变量名: 类型)
const emit = defineEmits<{
  (e: 'update', id: number): void;
  (e: 'delete', name: string): void;
}>();

const handleUpdate = () => emit('update', 1);
</script>
```

### 6.2 TypeScript 在 React 中的应用

React 的 Hooks 在搭配 TS 时也非常直观。

#### （1）组件与 Props (FC)

有两种常见写法：普通的函数声明（推荐），或使用 `React.FC`。

```tsx
import React, { useState } from 'react';

interface ButtonProps {
  title: string;
  onClick?: () => void;
  children?: React.ReactNode; // 声明 children 类型
}

// 写法 1：普通函数（推荐）
export default function Button({ title, onClick, children }: ButtonProps) {
  return <button onClick={onClick}>{title}{children}</button>;
}

// 写法 2：React.FC（老项目常见，自带 children，但 React 18 已移除默认的 children）
export const ButtonFC: React.FC<ButtonProps> = ({ title, onClick }) => {
  return <button onClick={onClick}>{title}</button>;
};
```

#### （2）useState 与 useRef

```tsx
// useState 自动推断或泛型指定
const [user, setUser] = useState<User | null>(null);

// useRef 获取 DOM 元素：需明确指定 DOM 元素类型（如 HTMLInputElement），并初始为 null
const inputRef = useRef<HTMLInputElement>(null);

const focusInput = () => {
  inputRef.current?.focus();
};
```

#### （3）DOM 事件类型

内联事件函数 TS 会自动推断事件对象 `e`，但如果把事件函数提取出来，需要显式声明。

```tsx
import React, { ChangeEvent, MouseEvent } from 'react';

function Form() {
  // input 的 change 事件：ChangeEvent<元素类型>
  const handleChange = (e: ChangeEvent<HTMLInputElement>) => {
    console.log(e.target.value);
  };

  // button 的 click 事件：MouseEvent<元素类型>
  const handleClick = (e: MouseEvent<HTMLButtonElement>) => {
    e.preventDefault();
  };

  return (
    <form>
      <input type="text" onChange={handleChange} />
      <button onClick={handleClick}>提交</button>
    </form>
  );
}
```

### 6.3 框架实战常见错误 vs 正确写法

| 场景 | 错误写法 / 现象 | 正确写法 / 做法 |
|------|-----------------|-----------------|
| **Vue ref 获取组件实例** | `const child = ref(null); child.value.fn()` 报错找不到属性 | 需要导入子组件类型并泛型约束：`const child = ref<InstanceType<typeof Child>>(null);` 并在调用前使用 `?` 判空。 |
| **React useRef 报错** | `useRef()` 没给初始值 null | 获取 DOM 的 useRef 必须写 `useRef<HTMLElement>(null)`。 |
| **React onClick 事件参数** | `const click = (e: Event) => ...` 报错类型不匹配 | React 使用的是合成事件，需使用 `React.MouseEvent<HTMLElement>`。 |

| 面试官提问 | 回答思路 |
|------------|----------|
| 你们项目中为什么要用 TypeScript？有什么收益？ | 1. **错误前置**：在编译期拦截大部分拼写错误和类型错误，减少线上 bug；2. **代码即文档**：复杂的接口数据结构一目了然，不需要频繁看 YApi；3. **IDE 体验**：智能提示与安全重构，大幅提升开发与重构效率。 |
| 遇到很复杂的 JSON 数据怎么在代码里写类型？ | 不要手写！使用工具（如 VSCode 插件 `Paste JSON as Code`）或在线网站（如 `json2ts`）一键生成 Interface，再在此基础上进行微调或 Pick/Omit。 |
| 什么是 AnyScript？如何避免？ | 指项目中滥用 `any` 导致失去了 TS 的类型保护意义。避免：1. tsconfig 开启 `strict: true` 和 `noImplicitAny: true`；2. 必须要跳过类型检查时，优先使用 `unknown`；3. 第三方包缺少类型时，自己写 `.d.ts` 补充，而不是直接用 any。 |
