# 📘《自动化构建与部署学习与实战手册》

> 系统学习见下文各章；日常常用概念与使用场景速查见同目录《常用概念与使用场景》。  
> 适用：构建与部署实操（Maven/Gradle、前端构建、Docker、部署方式），面向校招与后端/运维，与《持续集成与持续部署》衔接。

------

## 第1章：自动化构建概述

本章建立对**自动化构建与部署**的整体认知：构建产出什么、与部署的关系，为后续后端构建（第 2 章）、前端构建（第 3 章）、Docker（第 4 章）、部署方式（第 5 章）打基础。与《持续集成与持续部署》中流水线阶段对应。

------

### 1️⃣ 什么是自动化构建

- **自动化构建**：通过**脚本或流水线**一键完成**依赖解析、编译/打包、静态检查、测试、生成制品**，无需人工逐条执行命令。**为什么需要**：可重复、可追溯、与环境一致；与 CI 结合时每次提交或合并触发构建，保证主干始终可构建。
- **目标**：产出**制品**（jar、war、前端 dist、Docker 镜像等）且版本可追溯；构建环境一致（如统一 JDK、Node 版本）。

### 2️⃣ 构建与部署的关系

- **构建**：产出**制品**（jar、前端 dist、**Docker 镜像**等）。**部署**：将制品放到**目标环境**并启动（如上传 jar 并执行、拉取镜像并 run、K8s 更新 Deployment）。流水线中通常**先构建、再部署**；部署可自动化（CD）或人工触发。
- **衔接**：同一流水线内，构建阶段生成带**版本/tag** 的制品，部署阶段用该版本拉取并部署，保证“部署的就是刚构建的那一版”。

------

### 📖 本章小结

| 知识点 | 面试关键词 | 实际应用 |
|--------|------------|----------|
| 自动化构建 | 脚本/流水线、依赖→编译→测试→制品 | 与 CI 结合、主干可构建 |
| 构建与部署 | 制品、先构建再部署、版本追溯 | 流水线阶段设计 |

------

### ⚠️ 常见坑与注意点

1. **现象**：本地能构建、流水线失败。**原因**：环境不一致（JDK/Node 版本、依赖未锁）。**正确做法**：流水线固定 JDK/Node 版本；Maven 用同一 settings、前端用 lockfile（npm ci）。
2. **现象**：部署时拿到的不是本次构建的制品。**正确做法**：构建与部署在同一流水线内，用**构建号或 Git SHA** 作为制品版本/tag，部署阶段显式指定该版本。
3. **现象**：构建产物不归档、无法回滚。**正确做法**：制品上传**制品库**或镜像仓库，并打**版本 tag**，便于回滚时拉取历史版本。

------

### 🎯 面试常见追问

| 面试官提问 | 回答思路 |
|------------|----------|
| 自动化构建和部署的关系？ | 构建产出制品（jar/镜像/dist），部署把制品放到目标环境并启动；流水线先构建再部署，用同一版本号保证一致性。若追问具体命令见第 2～5 章。 |

## 第2章：后端构建（Maven / Gradle）

本章解决** Java/后端项目如何自动化构建、产出制品**，与第 4 章 Docker（jar 可打镜像）、第 5 章部署（jar 或镜像部署）衔接。校招常考：Maven 生命周期、跳过测试、多模块构建。

------

### 1️⃣ Maven 常用命令与生命周期

- **常用命令**：`mvn clean install`（清理+打包+装到本地仓库）、`mvn clean package`（清理+打包，产出在 target/）。**生命周期**：**compile** → **test** → **package** → **install** → **deploy**；前一阶段会先执行。
- **跳过测试**：`-DskipTests`（跳过执行测试，仍编译测试代码）、`-Dmaven.test.skip=true`（连编译测试都跳过）。**CI 中**：一般**不跳过测试**，用门禁；紧急时可临时 skipTests 并注明原因。

```bash
# 打包并跳过测试（慎用，CI 建议跑测试）
mvn clean package -DskipTests
```

### 2️⃣ Gradle 简述

- **命令**：`./gradlew build`、`./gradlew clean build`；**任务依赖与增量构建**，未变更可跳过。**配置**：`build.gradle`、`settings.gradle`；与 Maven 仓库兼容，可发布到 Nexus。**制品**：通常在 `build/libs/`。

### 3️⃣ 多模块与制品

- **Maven 多模块**：父 pom 用 `<modules>`，子模块用 `<parent>`；**reactor** 按依赖顺序构建。**制品**：jar/war 在 `target/` 或 `build/libs/`；可上传 **Nexus**；**版本号**与 **Git tag** 或 **build number** 关联便于追溯。

| 工具 | 打包命令 | 产物位置 | 跳过测试 |
|------|----------|----------|----------|
| Maven | mvn clean package | target/*.jar | -DskipTests |
| Gradle | ./gradlew build | build/libs/*.jar | -x test 或配置 |

------

### 📖 本章小结

| 知识点 | 面试关键词 | 实际应用 |
|--------|------------|----------|
| Maven | 生命周期、clean package/install、skipTests | 流水线中执行 mvn |
| Gradle | gradlew build、增量构建 | 与 Maven 二选一或共存 |
| 制品与版本 | target/build、Nexus、版本与 tag 关联 | 上传制品库、追溯 |

------

### ⚠️ 常见坑与注意点

1. **现象**：CI 里用 -Dmaven.test.skip=true 导致从不跑测试。**正确做法**：CI 默认**跑测试**；只有明确例外时才 skipTests，并记录原因。
2. **现象**：多模块构建顺序错导致失败。**正确做法**：Maven reactor 会按依赖顺序构建；确保父 pom 的 modules 与子模块 parent 一致，依赖关系正确。
3. **现象**：未指定 Java 版本，本地与 CI 行为不一致。**正确做法**：pom 中指定 `maven.compiler.source/target` 或 Gradle 中 `sourceCompatibility`，CI 使用对应 JDK。

------

### 🎯 面试常见追问

| 面试官提问 | 回答思路 |
|------------|----------|
| Maven 生命周期有哪些阶段？ | compile、test、package、install、deploy；前一阶段会先执行。package 产出 jar，install 装到本地仓库。若追问跳过测试见本章 1️⃣。 |
| 多模块构建顺序？ | Maven 按 reactor 依赖顺序构建；父 pom 定义 modules，子模块用 parent 引用。 |

## 第3章：前端构建与静态资源

本章解决**前端项目如何自动化构建、多环境变量、产物如何部署**，与第 5 章部署（静态资源放 Nginx/OSS）、第 6 章流水线衔接。校招常考：npm ci 与 npm install、多环境变量、SPA 部署。

------

### 1️⃣ npm / pnpm 与构建命令

- **构建**：`npm run build`、`pnpm build`；脚本在 **package.json** 的 **scripts** 中定义。**依赖安装**：**npm ci**（按 **lock 文件**安装，与 CI 一致、更快）、**npm install**（可能更新 lock）。**CI 中建议用 npm ci** 保证依赖一致。
- **产物**：通常为 **dist/** 或 **build/**，可配置；为静态文件（HTML/CSS/JS），可部署到 Nginx、**对象存储（OSS）+ CDN**。

| 命令 | 说明 | CI 建议 |
|------|------|---------|
| npm install | 安装并可能更新 lock | 本地开发 |
| npm ci | 按 package-lock.json 安装，不改 lock | **CI 推荐** |

### 2️⃣ 构建工具与环境变量

- **构建工具**：**Vite**、**Webpack** 等；**环境变量**（如 `VITE_API_BASE`、`REACT_APP_API`）在**构建时**注入。**多环境**：`.env.development`、`.env.production` 或 CI 中通过 **env** 传入；构建时读取对应变量打进产物。
- **面试怎么说**：前端多环境靠**构建时**注入不同 env；不是运行时读配置文件，所以不同环境需**重新构建**一份 dist。

### 3️⃣ 静态部署与 SPA

- **部署**：dist 上传到 **Nginx** 目录或 **OSS**，配置 CDN 加速。**SPA**：单页应用路由在前端，需 Nginx 配置 **try_files** 或 **fallback** 到 index.html，否则刷新子路径会 404。

------

### 📖 本章小结

| 知识点 | 面试关键词 | 实际应用 |
|--------|------------|----------|
| npm/pnpm | npm ci、npm run build、lock 一致 | CI 中装依赖与构建 |
| 环境变量 | 构建时注入、.env.*、CI 传入 | 多环境 API 地址等 |
| 静态部署 | dist、Nginx、OSS+CDN、SPA fallback | 前端上线 |

------

### ⚠️ 常见坑与注意点

1. **现象**：CI 用 npm install 导致每次依赖版本漂移。**正确做法**：CI 用 **npm ci**，保证与 lock 一致、可重复构建。
2. **现象**：生产构建用了开发环境变量。**正确做法**：CI 部署生产时指定 **NODE_ENV=production** 或对应 `.env.production`，避免把 dev API 打进生产。
3. **现象**：SPA 部署后刷新子路径 404。**正确做法**：Nginx 配置 `try_files $uri $uri/ /index.html;`（或等价），使未匹配静态文件时回退到 index.html。

------

### 🎯 面试常见追问

| 面试官提问 | 回答思路 |
|------------|----------|
| npm ci 和 npm install 区别？ | npm ci 按 lock 安装、不修改 lock、适合 CI；npm install 可能更新 lock。CI 推荐 npm ci 保证一致。若追问多环境见本章 2️⃣。 |
| 前端多环境变量如何注入？ | 构建时通过 .env.* 或 CI 环境变量注入（如 VITE_*、REACT_APP_*）；不同环境打不同构建产物。 |
| 构建产物如何部署？ | dist 上传 Nginx 或 OSS，配 CDN；SPA 需配置 fallback 到 index.html。 |

## 第4章：Docker 镜像构建与推送

本章解决**如何用 Dockerfile 构建镜像、多阶段构建、如何推送到镜像仓库**，与第 5 章部署（Docker/K8s 拉取镜像）、第 6 章流水线衔接。校招常考：多阶段构建、减小镜像体积、tag 与版本追溯。

------

### 1️⃣ Dockerfile 基础

- **常见指令**：**FROM** 基础镜像、**WORKDIR**、**COPY/ADD**、**RUN**、**EXPOSE**、**CMD/ENTRYPOINT**。**多阶段构建**：先在一阶段**编译/构建**，再在另一阶段 **COPY 产物**到精简运行镜像，**减小最终镜像体积**。
- **为什么多阶段**：构建需要 JDK/Node 等，运行可能只需要 JRE 或静态文件；多阶段可只把运行时需要的放进最终镜像。

```dockerfile
# 多阶段示例：先构建 jar，再只保留 JRE + jar 运行
FROM maven:3-eclipse-temurin-17 AS builder
WORKDIR /app
COPY pom.xml ./
RUN mvn dependency:go-offline -B
COPY src ./src
RUN mvn package -DskipTests -B

FROM eclipse-temurin:17-jre
WORKDIR /app
COPY --from=builder /app/target/*.jar app.jar
EXPOSE 8080
ENTRYPOINT ["java","-jar","app.jar"]
```

### 2️⃣ 构建与推送

- **构建**：`docker build -t name:tag .`；CI 中由流水线执行；**版本**可通过 `--build-arg` 传入（如 VERSION、BUILD_DATE）。**推送**：`docker push registry/name:tag`；需先 **docker login**；registry 可为 **Docker Hub**、**Harbor**、云厂商镜像仓库、**GHCR**。
- **最佳实践**：**多阶段**减小体积、**非 root 用户**运行、**.dockerignore** 减少构建上下文、**固定基础镜像 tag**（不用 latest）便于复现。

### 3️⃣ 镜像 tag 与追溯

- **tag**：与 **Git tag** 或 **CI build number** 一致，便于“哪次构建上的生产”的追溯与回滚。例如：`myapp:1.2.3`、`myapp:git-abc1234`。

------

### 📖 本章小结

| 知识点 | 面试关键词 | 实际应用 |
|--------|------------|----------|
| Dockerfile | FROM、COPY、RUN、多阶段 | 镜像构建、减小体积 |
| 构建与推送 | docker build、docker push、registry | 流水线中构建并推仓库 |
| 最佳实践 | 多阶段、.dockerignore、固定 base tag | 安全与可重复 |

------

### ⚠️ 常见坑与注意点

1. **现象**：镜像体积很大。**正确做法**：用**多阶段构建**，只把运行时需要的放进最终镜像；用 **.dockerignore** 排除无关文件；选**精简基础镜像**（如 alpine、distroless）。
2. **现象**：基础镜像用 latest，某天构建突然坏。**正确做法**：**固定 tag**（如 `eclipse-temurin:17-jre`），必要时再升级并测试。
3. **现象**：镜像内用 root 运行。**正确做法**：用 **USER** 指定非 root 用户，减少风险。

------

### 🎯 面试常见追问

| 面试官提问 | 回答思路 |
|------------|----------|
| 多阶段构建的目的是什么？ | 构建阶段需要编译环境（JDK/Node），运行阶段只需运行时或静态文件；多阶段只把产物拷进最终镜像，减小体积、提高安全。若追问如何减小体积见本章 2️⃣。 |
| 如何保证镜像与版本追溯？ | 镜像 tag 用 Git tag 或 CI build number；部署时拉取同一 tag，便于追溯与回滚。 |

## 第5章：部署方式概览

本章解决**制品/镜像如何部署到目标环境**（脚本、Docker、K8s、云厂商），与第 2～4 章制品来源、第 6 章流水线中的部署阶段衔接。校招常考：几种部署方式适用场景、零停机与回滚。

------

### 1️⃣ 部署方式对比

| 方式 | 思路 | 优点 | 适用 |
|------|------|------|------|
| **脚本部署** | SSH 登录目标机，拉取制品或 scp 上传，执行启动/重启脚本 | 简单、直接 | 少量机器、无容器 |
| **Docker 部署** | 目标机 **docker pull** 镜像并 **docker run** 或 **docker-compose up** | 环境一致、易回滚（换 tag） | 单机/多机容器化 |
| **K8s 部署** | **Deployment、Service** 等；**kubectl set image**、**helm upgrade** 或 **GitOps**（Argo CD、Flux） | 多实例、滚动、灰度 | 云原生、多实例 |
| **云厂商** | 云效、CODING、或云主机/容器服务控制台或 API | 与云集成、托管 | 上云项目 |

### 2️⃣ 零停机与回滚

- **零停机**：**滚动更新**（K8s 默认）、**蓝绿**（先起新再切流量）、**金丝雀**（部分流量新版本）；避免一次性停掉所有实例。**回滚**：保留上一版本镜像或制品；K8s 可 **kubectl rollout undo**；脚本部署需有上一版包或镜像可拉。
- **面试怎么说**：零停机靠滚动/蓝绿/灰度；回滚靠保留上一版本并有一键或标准化回滚流程（见《持续集成与持续部署》第 3 章）。

------

### 📖 本章小结

| 知识点 | 面试关键词 | 实际应用 |
|--------|------------|----------|
| 部署方式 | 脚本、Docker、K8s、云厂商 | 选型与流水线部署步骤 |
| 零停机与回滚 | 滚动、蓝绿、灰度、保留上一版 | 发布与故障恢复 |

------

### ⚠️ 常见坑与注意点

1. **现象**：脚本部署直接 kill 再启动，有短暂不可用。**正确做法**：多实例时先起新再停旧，或用 systemd/supervisor 做平滑重启；或改用 Docker/K8s 滚动。
2. **现象**：K8s 更新镜像未指定 tag 导致用默认 latest。**正确做法**：**显式指定镜像 tag**（如 `myapp:1.2.3`），与构建产出一致，避免拉错版本。
3. **现象**：没有回滚预案。**正确做法**：部署前保留上一版本；K8s 用 rollout undo；脚本或 Docker 保留上一镜像 tag 可拉。

------

### 🎯 面试常见追问

| 面试官提问 | 回答思路 |
|------------|----------|
| 脚本、Docker、K8s 部署各适合什么场景？ | 脚本：少量机器、简单发布。Docker：环境一致、易回滚、单机或小规模。K8s：多实例、滚动与灰度、云原生。若追问零停机见本章 2️⃣。 |
| 如何做零停机发布与回滚？ | 零停机：滚动更新、蓝绿、金丝雀。回滚：保留上一版本镜像/制品，K8s rollout undo 或切回蓝绿/收口灰度。 |

## 第6章：与 CI/CD 流水线的衔接

本章解决**构建与部署如何在流水线中被调用、版本如何一致、环境如何隔离**，与《持续集成与持续部署》直接对应。校招常考：流水线如何调用构建与部署、如何保证部署的是“刚构建的那一版”。

------

### 1️⃣ 流水线中的构建与部署

- **构建阶段**：流水线中执行 **mvn/gradle/npm** 命令；产物上传**制品库**或打 **Docker 镜像**并 **push**。**部署阶段**：执行 **SSH 命令**、**docker pull + run**、或 **kubectl/helm**；凭据由 CI 平台 **Secrets** 提供（SSH 密钥、kubeconfig、镜像仓库登录）。
- **面试怎么说**：构建阶段跑 mvn package 或 docker build，产出带版本号的 jar 或镜像并推送；部署阶段用同一版本号拉取并部署，保证“部署即刚构建的版本”。

### 2️⃣ 版本与追溯

- **构建号或 Git SHA** 写入制品名或镜像 **tag**；部署时使用**同一 tag**，便于“哪次构建上了生产”的追溯与回滚。例如：镜像 tag 用 `git-${COMMIT_SHA}` 或 `v${BUILD_NUMBER}`。

### 3️⃣ 环境隔离

- **不同环境**用不同**配置文件**或**环境变量**（如 Spring profile、前端 .env）；流水线按**分支/tag** 选择部署目标与配置（如 main→测试环境，tag→生产）。

------

### 📖 本章小结

| 知识点 | 面试关键词 | 实际应用 |
|--------|------------|----------|
| 流水线调用 | 构建阶段 mvn/docker、部署阶段 SSH/kubectl、Secrets | 与《持续集成与持续部署》一致 |
| 版本追溯 | 构建号/Git SHA 作 tag、部署用同一版本 | 可追溯、可回滚 |
| 环境隔离 | profile、env、分支/tag 决定目标环境 | 多环境发布 |

------

### ⚠️ 常见坑与注意点

1. **现象**：部署阶段未显式指定版本，拉成旧制品或 latest。**正确做法**：部署步骤**显式传版本/tag**（从构建阶段输出或环境变量读取），禁止默认 latest。
2. **现象**：测试与生产共同一套配置。**正确做法**：用 **profile** 或**环境变量**区分；流水线按环境注入对应配置或密钥。
3. **现象**：Secrets 写在脚本里并提交。**正确做法**：密钥只放在 CI **Secrets**，脚本中通过环境变量引用，不提交仓库。

------

### 🎯 面试常见追问

| 面试官提问 | 回答思路 |
|------------|----------|
| 流水线如何调用构建与部署？ | 构建阶段执行 mvn/gradle/npm、docker build 并 push；部署阶段用 Secrets 凭据执行 SSH、docker pull+run 或 kubectl；版本由构建阶段产出，部署阶段使用同一版本。若追问《持续集成与持续部署》见同目录该手册。 |
| 如何保证部署的是“刚构建出的那一版”？ | 构建与部署在同一流水线；制品或镜像 tag 用构建号或 Git SHA；部署步骤显式指定该 tag/版本拉取并部署。 |

## 第7章：K8s 部署实战

本章深入讲解 **Kubernetes** 部署的完整流程，与第 5 章部署方式、第 6 章流水线衔接形成完整闭环。校招常考：Deployment/Service 配置、滚动更新、Helm 基础、ConfigMap/Secret 管理配置。

------

### 1️⃣ K8s 部署基础概念

- **Pod**：K8s 最小部署单元，包含一个或多个容器。**Deployment**：管理 Pod 的副本数、滚动更新、回滚。**Service**：提供稳定访问入口，将流量负载均衡到 Pod。
- **部署流程**：构建镜像 → 编写 K8s YAML → kubectl apply → 验证部署。

```
┌─────────────┐    ┌─────────────┐    ┌─────────────┐
│   Docker    │ → │  K8s YAML   │ → │   kubectl   │
│   镜像构建   │    │  (Deployment│    │   apply     │
│   & 推送    │    │  /Service)  │    │             │
└─────────────┘    └─────────────┘    └─────────────┘
```

------

### 2️⃣ Deployment 配置详解

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: myapp-deployment
  labels:
    app: myapp
spec:
  replicas: 3                    # 副本数
  selector:
    matchLabels:
      app: myapp
  template:
    metadata:
      labels:
        app: myapp
    spec:
      containers:
      - name: myapp
        image: myregistry/myapp:v1.2.3   # 镜像版本
        ports:
        - containerPort: 8080
        env:                         # 环境变量
        - name: SPRING_PROFILES_ACTIVE
          value: "prod"
        resources:                   # 资源限制
          requests:
            memory: "256Mi"
            cpu: "250m"
          limits:
            memory: "512Mi"
            cpu: "500m"
        livenessProbe:               # 存活探针
          httpGet:
            path: /actuator/health
            port: 8080
          initialDelaySeconds: 30
          periodSeconds: 10
        readinessProbe:              # 就绪探针
          httpGet:
            path: /actuator/health
            port: 8080
          initialDelaySeconds: 10
          periodSeconds: 5
```

**关键字段说明**：
| 字段 | 说明 |
|------|------|
| `replicas` | Pod 副本数，保证高可用 |
| `image` | 镜像地址，**必须带版本 tag**，不用 latest |
| `resources` | CPU/内存请求与限制，防止资源耗尽 |
| `livenessProbe` | 检测容器是否存活，失败则重启 |
| `readinessProbe` | 检测是否可接收流量，失败则从 Service 摘除 |

### 3️⃣ Service 与访问暴露

```yaml
apiVersion: v1
kind: Service
metadata:
  name: myapp-service
spec:
  selector:
    app: myapp                   # 匹配 Pod 标签
  ports:
  - port: 80                     # Service 端口
    targetPort: 8080             # Pod 端口
    protocol: TCP
  type: ClusterIP                # 集群内访问
```

**Service 类型对比**：
| 类型 | 说明 | 适用场景 |
|------|------|----------|
| **ClusterIP** | 默认，仅集群内访问 | 内部服务调用 |
| **NodePort** | 每个节点开放端口 | 开发测试 |
| **LoadBalancer** | 云厂商负载均衡 | 生产环境暴露服务 |
| **Ingress** | 七层路由 | 多服务统一入口 |

### 4️⃣ ConfigMap 与 Secret 管理配置

**ConfigMap**：存储非敏感配置
```yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: myapp-config
data:
  application.yml: |
    server:
      port: 8080
    logging:
      level: INFO
```

**Secret**：存储敏感信息（base64 编码）
```yaml
apiVersion: v1
kind: Secret
metadata:
  name: myapp-secrets
type: Opaque
data:
  db-password: cGFzc3dvcmQxMjM=    # echo -n 'password123' | base64
```

**在 Deployment 中引用**：
```yaml
spec:
  template:
    spec:
      containers:
      - name: myapp
        image: myapp:v1.2.3
        envFrom:                     # 全部注入为环境变量
        - configMapRef:
            name: myapp-config
        env:                         # 单独注入 Secret
        - name: DB_PASSWORD
          valueFrom:
            secretKeyRef:
              name: myapp-secrets
              key: db-password
        volumeMounts:                # 挂载 ConfigMap 为文件
        - name: config-volume
          mountPath: /app/config
      volumes:
      - name: config-volume
        configMap:
          name: myapp-config
```

### 5️⃣ 滚动更新与回滚

**更新镜像版本**：
```bash
# 方式1：直接修改 YAML 后 apply
kubectl apply -f deployment.yaml

# 方式2：命令行更新（适合 CI/CD）
kubectl set image deployment/myapp-deployment myapp=myregistry/myapp:v1.2.4

# 查看更新状态
kubectl rollout status deployment/myapp-deployment
```

**回滚**：
```bash
# 查看历史版本
kubectl rollout history deployment/myapp-deployment

# 回滚到上一版本
kubectl rollout undo deployment/myapp-deployment

# 回滚到指定版本
kubectl rollout undo deployment/myapp-deployment --to-revision=2
```

**更新策略**：
```yaml
spec:
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxSurge: 25%          # 更新时最多多出 25% 的 Pod
      maxUnavailable: 25%    # 更新时最多不可用 25% 的 Pod
```

### 6️⃣ Helm 基础（包管理）

- **Helm**：K8s 的包管理工具，类似 apt/yum。将一组 K8s 资源打包为 **Chart**，支持参数化配置。

```bash
# 安装 Helm
brew install helm

# 添加仓库
helm repo add stable https://charts.helm.sh/stable

# 搜索 Chart
helm search repo nginx

# 安装 Chart
helm install my-nginx stable/nginx-ingress

# 查看已安装
helm list

# 升级
helm upgrade my-nginx stable/nginx-ingress

# 卸载
helm uninstall my-nginx
```

**Chart 目录结构**：
```
mychart/
├── Chart.yaml          # Chart 元数据
├── values.yaml         # 默认配置值
├── templates/          # K8s YAML 模板
│   ├── deployment.yaml
│   ├── service.yaml
│   └── _helpers.tpl    # 辅助模板
└── README.md
```

**模板示例（deployment.yaml）**：
```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: {{ include "mychart.fullname" . }}
spec:
  replicas: {{ .Values.replicaCount }}
  template:
    spec:
      containers:
      - name: {{ .Chart.Name }}
        image: "{{ .Values.image.repository }}:{{ .Values.image.tag }}"
        resources:
          {{- toYaml .Values.resources | nindent 12 }}
```

**values.yaml**：
```yaml
replicaCount: 3

image:
  repository: myapp
  tag: "v1.2.3"

resources:
  limits:
    cpu: 500m
    memory: 512Mi
```

### ⚠️ 常见坑与注意点

1. **现象**：镜像使用 latest，无法确定部署的是哪个版本。**正确做法**：**显式指定镜像 tag**（如 Git SHA 或版本号），便于追溯与回滚。
2. **现象**：Pod 启动后立即接收流量，导致请求失败。**正确做法**：配置 **readinessProbe**，确保应用就绪后再加入 Service 负载均衡。
3. **现象**：Secret 数据未 base64 编码导致创建失败。**正确做法**：Secret 的 `data` 字段值必须是 base64 编码；或使用 `stringData` 字段直接写明文（K8s 自动编码）。
4. **现象**：资源限制设置不当导致 OOMKill。**正确做法**：根据应用实际内存使用设置 `limits.memory`，并留一定余量。

| K8s 如何实现零停机发布？ | 通过 Deployment 的 **RollingUpdate** 策略，逐步替换 Pod；配合 **readinessProbe** 确保新 Pod 就绪后再接收流量；旧 Pod 处理完请求后终止。 |

| Helm 是什么？解决什么问题？ | Helm 是 K8s 的包管理工具，将一组 K8s 资源打包为 Chart。解决：① 资源复用（一个 Chart 多次安装）；② 参数化配置（values.yaml）；③ 版本管理（安装/升级/回滚）。 |
