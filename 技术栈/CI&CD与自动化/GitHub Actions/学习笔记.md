# 📘《GitHub Actions 学习与实战手册》

> 系统学习见下文各章；日常常用配置与使用场景速查见同目录《常用配置与使用场景》。

------

## 第1章：GitHub Actions 概述

### 1.1 什么是 GitHub Actions

- **GitHub Actions** 是 GitHub 提供的 **CI/CD 与自动化** 能力，通过仓库内的 **YAML 工作流** 定义：在 push、PR、tag、定时等事件触发时，在 GitHub 托管的 runner 或自托管 runner 上执行任务。
- 特点：与仓库一体、配置即用、公开仓库免费额度、生态丰富（Actions 市场）；适合开源项目与使用 GitHub 的团队。

### 1.2 核心概念

- **Workflow**：一个 YAML 文件，定义何时运行、运行什么；放在 .github/workflows/ 下。
- **Event**：触发条件，如 push、pull_request、workflow_dispatch（手动）、schedule（cron）。
- **Job**：工作流中的一组 Step，在同一 runner 上顺序执行；多 Job 可并行或设依赖。
- **Step**：单步操作，可以是 run（shell 命令）或 uses（使用 Action）。
- **Runner**：执行环境，GitHub 提供 Ubuntu/Windows/macOS；也可自托管。

------

## 第2章：工作流语法基础

1. **name、on、jobs**：name 为工作流名称；on 指定触发事件与分支/路径过滤；jobs 下为多个 job，每个 job 有 runs-on 与 steps。
2. **runs-on**：ubuntu-latest、windows-latest、macos-latest；或自托管标签。
3. **steps**：每个 step 有 name、uses 或 run；run 下写 shell 命令（默认 bash）；uses 引用 action（ 仓库/路径@版本 ）。
4. **环境变量与输出**：env 在 job 或 step 级设置；step 间用 outputs 传递；${{ env.XXX }}、${{ steps.xxx.outputs.yyy }}。
5. **条件与矩阵**：if 条件控制 step/job 是否执行；strategy.matrix 多维度组合运行（如多版本 Java、多 Node 版本）。

------

## 第3章：常用事件与触发

1. **push**：推送到分支；可限定 branches、paths（仅某路径变更触发）。
2. **pull_request**：PR 打开、更新、合并等；常用于 PR 时跑测试与检查。
3. **workflow_dispatch**：在 Actions 页手动触发；可配 inputs 输入参数。
4. **schedule**：cron 表达式定时运行；如 "0 * * * *" 每小时。
5. **release**：发布 release 时；tag push 也可单独写 on: push: tags: - 'v*'。
6. **面试要点**：如何只对某分支或某目录变更触发、如何手动触发。

------

## 第4章：常用 Action 与步骤

1. **checkout**：actions/checkout@v4，拉取仓库代码；必用第一步（除纯定时任务外）。
2. **Java**：actions/setup-java@v4，指定 JDK 版本；配合 mvn、gradle。
3. **Node**：actions/setup-node@v4，指定 Node 版本；配合 npm/pnpm、前端构建。
4. **缓存**：actions/cache，按 key 缓存目录（如 Maven .m2、npm node_modules）；加速后续运行。
5. **上传/下载制品**：actions/upload-artifact、download-artifact，job 间传递构建产物。
6. **Secrets**：敏感信息存仓库 Settings → Secrets，在 YAML 中用 ${{ secrets.XXX }}，不写明文。
7. **面试要点**：如何缓存依赖、如何传递制品、Secrets 如何使用。

------

## 第5章：多 Job 与矩阵

1. **多 Job**：jobs 下多个 job；默认并行；用 needs 指定依赖，被依赖的 job 通过后才会跑当前 job。
2. **矩阵**：strategy.matrix 定义多组参数（如 java: [8, 17]、os: [ubuntu, windows]）；每个组合跑一次，适合多版本测试。
3. **条件**：job 或 step 的 if；常用 github.ref、github.event_name、success()、failure() 等。
4. **面试要点**：如何做多版本构建与测试、如何控制某 step 仅在主分支执行。

------

## 第6章：与部署的衔接

1. **部署到服务器**：用 SSH、SCP 或部署脚本；密钥存 Secrets，在 step 中执行命令。
2. **部署到云**：使用云厂商官方 Action（如 AWS、阿里云）或 SDK/CLI；凭据用 Secrets。
3. **Docker 构建与推送**：docker/build-push-action；镜像推送到 Docker Hub 或 GHCR（GitHub Container Registry）；需登录与权限。
4. **K8s 部署**：kubectl 或 Helm；kubeconfig 或 token 存 Secrets；注意生产环境权限与审批策略。
5. **面试要点**：流水线中如何安全使用密钥、如何构建并推送镜像。
