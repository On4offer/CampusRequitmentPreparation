# 📘《GitHub Actions 学习与实战手册》

> 系统学习见下文各章；日常常用配置与使用场景速查见同目录《常用配置与使用场景》。  
> 适用：GitHub Actions 工作流配置，面向校招与使用 GitHub 的 CI/CD（Workflow、事件、Job/Step、常用 Action、部署衔接）。

------

## 第1章：GitHub Actions 概述

本章建立对 **GitHub Actions** 的整体认知：是什么、核心概念有哪些，为后续语法（第 2 章）、事件与 Action（第 3～4 章）、多 Job 与部署（第 5～6 章）打基础。与《持续集成与持续部署》中“托管 CI”对应。

------

### 1️⃣ 什么是 GitHub Actions

- **GitHub Actions**：GitHub 提供的 **CI/CD 与自动化** 能力，通过仓库内的 **YAML 工作流** 定义：在 **push、PR、tag、定时**等事件触发时，在 GitHub 托管的 **runner** 或**自托管 runner** 上执行任务。**为什么用**：与仓库一体、**配置即用**、公开仓库有免费额度、**Actions 市场**生态丰富；适合开源与使用 GitHub 的团队。
- **特点**：工作流文件放在 **.github/workflows/** 下，**Pipeline as Code**；无需自建 Jenkins，与 Git 事件天然集成。

### 2️⃣ 核心概念

| 概念 | 说明 |
|------|------|
| **Workflow** | 一个 YAML 文件，定义**何时运行、运行什么**；放在 `.github/workflows/` 下 |
| **Event** | 触发条件：**push**、**pull_request**、**workflow_dispatch**（手动）、**schedule**（cron）、**release** 等 |
| **Job** | 工作流中的一组 **Step**，在同一 runner 上**顺序**执行；多 Job 可**并行**或通过 **needs** 设依赖 |
| **Step** | 单步操作：**run**（shell 命令）或 **uses**（使用 Action，如 actions/checkout@v4） |
| **Runner** | 执行环境：GitHub 提供 **ubuntu-latest**、**windows-latest**、**macos-latest**；也可**自托管** |

------

### 📖 本章小结

| 知识点 | 面试关键词 | 实际应用 |
|--------|------------|----------|
| Actions 定位 | 托管 CI/CD、YAML 工作流、与仓库一体 | 与 Jenkins 对比（见《持续集成与持续部署》） |
| 核心概念 | Workflow、Event、Job、Step、Runner | 读/写 YAML、口述流程 |

------

### ⚠️ 常见坑与注意点

1. **现象**：工作流文件放错位置或命名非 YAML。**正确做法**：必须放在 **.github/workflows/** 下，后缀 **.yml** 或 **.yaml**；文件名任意。
2. **现象**：把 Job 和 Step 混为一谈。**正确做法**：**Job** 是“一组步骤在一个 runner 上跑”；**Step** 是 Job 内的一条条操作（run 或 uses）。
3. **现象**：敏感信息写在 YAML 里并提交。**正确做法**：用 **Secrets**（仓库 Settings → Secrets），在 YAML 里用 `${{ secrets.XXX }}`，不写明文。

------

### 🎯 面试常见追问

| 面试官提问 | 回答思路 |
|------------|----------|
| GitHub Actions 是什么？ | GitHub 提供的 CI/CD，用仓库里 .github/workflows 下的 YAML 定义工作流；由 push/PR/tag 等事件触发，在 runner 上执行 Job 和 Step。若追问与 Jenkins 区别见《持续集成与持续部署》第 5 章。 |
| Workflow、Job、Step 的关系？ | Workflow 是一个 YAML 文件；里面有一个或多个 Job；每个 Job 里有多个 Step 顺序执行。多 Job 可并行或 needs 依赖。 |

## 第2章：工作流语法基础

本章解决**如何写一个最小可用的工作流**：name、on、jobs、runs-on、steps，以及环境变量、输出、条件与矩阵的常用写法。与第 3 章事件、第 4 章 Action 衔接。校招常考：最小结构、runs-on、uses 与 run、表达式。

------

### 1️⃣ 最小结构

- **name**：工作流名称，在 Actions 页显示。**on**：触发事件（如 push、pull_request）。**jobs**：一个或多个 job；每个 job 有 **runs-on**（ runner 类型）和 **steps**（步骤列表）。

```yaml
name: CI
on: push
jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: echo "Hello"
```

> 💬 **面试怎么说**：最少要有 on（触发）、jobs.xxx.runs-on（跑在哪）、jobs.xxx.steps（先 checkout 再 run 命令）；checkout 一般是第一步（除纯定时任务外）。

### 2️⃣ runs-on 与 steps

- **runs-on**：**ubuntu-latest**、**windows-latest**、**macos-latest**；或自托管 runner 的 **labels**。**steps**：每个 step 可有 **name**（日志中显示）、**uses**（引用 Action，格式 `owner/repo@ref`）、**run**（shell 命令，默认 bash）；**uses** 与 **run** 二选一或组合（先 uses 再 run）。
- **环境变量与输出**：**env** 可在 **job** 或 **step** 级设置；step 间用 **outputs** 传递（id + outputs），在后续 step 用 `${{ steps.xxx.outputs.yyy }}`。表达式：**${{ env.XXX }}**、**${{ secrets.XXX }}**、**${{ github.ref }}** 等。

### 3️⃣ 条件与矩阵

- **if**：在 **step** 或 **job** 上写 `if: ...`，条件为真才执行；常用 **github.ref**（分支/tag）、**github.event_name**、**success()**、**failure()**。**strategy.matrix**：多维度组合运行，如多版本 Java、多 Node、多 OS；每个组合跑一次，适合**多版本构建/测试**。

------

### 📖 本章小结

| 知识点 | 面试关键词 | 实际应用 |
|--------|------------|----------|
| 最小结构 | name、on、jobs、runs-on、steps | 新建一个最小 workflow |
| uses 与 run | checkout、setup-java、run 命令 | 拉代码、装环境、执行脚本 |
| 表达式 | env、secrets、outputs、if、matrix | 多环境、多版本、条件执行 |

------

### ⚠️ 常见坑与注意点

1. **现象**：Action 引用不写版本（如 actions/checkout）。**正确做法**：建议**固定版本**（如 @v4、@v3.6.0），避免主版本更新导致行为变化；可用 @v4 表示大版本。
2. **现象**：run 里多行命令未用 `|` 导致只执行第一行。**正确做法**：多行用 **run: \|** 或 **run: |-**，下面每行一条命令。
3. **现象**：矩阵组合爆炸（如 5×5×5）。**正确做法**：控制 matrix 维度或用 **exclude** 排除部分组合，避免时间与额度浪费。

------

### 🎯 面试常见追问

| 面试官提问 | 回答思路 |
|------------|----------|
| 一个最小的 workflow 要写哪些？ | name、on（触发）、jobs.xxx.runs-on、jobs.xxx.steps（至少 checkout + run）。若追问表达式见本章 2️⃣。 |
| 如何做多版本构建与测试？ | 用 strategy.matrix 定义多组参数（如 java: [8, 17]），每个组合跑一次 job。 |

## 第3章：常用事件与触发

本章解决**何时触发工作流**：push、PR、手动、定时、release、tag 等，以及**分支/路径过滤**。与第 2 章 on 语法、第 6 章部署（如仅 tag 触发部署）衔接。校招常考：只对某分支或某目录变更触发、如何手动触发。

------

### 1️⃣ 常用事件

| 事件 | 说明 | 典型用途 |
|------|------|----------|
| **push** | 推送到分支 | 合并后构建与测试、部署到测试环境 |
| **pull_request** | PR 打开/更新/同步 | PR 时跑测试与 Lint，合并前门禁 |
| **workflow_dispatch** | **手动**在 Actions 页触发 | 临时跑一次、带 inputs 参数 |
| **schedule** | **cron** 定时 | 定时巡检、日报、清理 |
| **release** | 发布 Release | 打 release 时构建与发布 |
| **push: tags** | 推送 tag（如 v*） | tag 触发打镜像或部署生产 |

### 2️⃣ 分支与路径过滤

- **只对某分支**：`on.push.branches: [main]` 或 `on.pull_request.branches: [main]`。**只对某目录变更**：`on.push.paths: ['src/**', 'pom.xml']`，仅这些路径变更才触发，避免无关文件改动跑流水线。
- **面试怎么说**：用 **on.push.branches** 和 **on.push.paths** 控制“谁推、改了什么才触发”；手动用 **workflow_dispatch**，可配 **inputs** 传参。

------

### 📖 本章小结

| 知识点 | 面试关键词 | 实际应用 |
|--------|------------|----------|
| 常用事件 | push、pull_request、workflow_dispatch、schedule、release、tags | 按场景选触发 |
| 过滤 | branches、paths | 减少无效运行、节省额度 |

------

### ⚠️ 常见坑与注意点

1. **现象**：push 任何文件都触发，文档改动也跑全量构建。**正确做法**：用 **paths** 过滤，仅源码或构建相关路径变更时触发。
2. **现象**：schedule 的 cron 时区搞错。**正确做法**：GitHub Actions 的 schedule 使用 **UTC**；若要北京时间需减 8 小时或查文档。
3. **现象**：PR 来自 fork 时权限受限。**正确做法**：fork 的 PR 默认只读权限、不能写 Secrets；若需写仓库需在 Settings 里放宽（有安全权衡）。

------

### 🎯 面试常见追问

| 面试官提问 | 回答思路 |
|------------|----------|
| 如何只对某分支或某目录变更触发？ | on.push.branches: [main] 限定分支；on.push.paths: ['src/**'] 限定路径，只有这些变更才触发。 |
| 如何手动触发？ | 使用 on.workflow_dispatch；在 Actions 页选该 workflow 点 Run workflow；可配 inputs 输入参数。 |

## 第4章：常用 Action 与步骤

本章解决**拉代码、装环境、缓存、制品传递、Secrets** 等常用步骤，与第 2 章 steps、第 6 章部署（需凭据）衔接。校招常考：checkout、setup-java/node、缓存、制品、Secrets。

------

### 1️⃣ 必用：checkout 与 setup

- **actions/checkout@v4**：**拉取仓库代码**，绝大多数工作流**第一步**（除纯定时不依赖代码外）。**actions/setup-java@v4**：安装 **JDK**；`with: java-version: '17', distribution: 'temurin'`。**actions/setup-node@v4**：安装 **Node**；`with: node-version: '20', cache: 'npm'`（可顺带缓存 npm）。
- **面试怎么说**：第一步通常 **checkout**，然后按项目用 **setup-java** 或 **setup-node**；再用 **run** 执行 mvn/npm 等。

### 2️⃣ 缓存与制品

- **actions/cache@v4**：按 **key** 缓存目录（如 Maven **.m2**、npm **node_modules**）；**key** 可含 **hashFiles('pom.xml')** 等，依赖变则 key 变、自动失效。**加速**后续运行的依赖安装。
- **actions/upload-artifact@v4**、**download-artifact@v4**：**job 间传递**构建产物；上传时指定 **name** 和 **path**，下载时指定 **name**。用于“构建 job 产出 jar/dist，部署 job 下载后部署”。

### 3️⃣ Secrets

- **敏感信息**（API Key、SSH 私钥、镜像仓库密码等）存 **仓库 Settings → Secrets and variables → Actions**；在 YAML 中用 **${{ secrets.XXX }}**，**不会**在日志中明文显示。**正确做法**：绝不把密码写进 YAML 或代码；生产凭据用 Secrets，必要时用**环境 Secrets** 或**审批**。

------

### 📖 本章小结

| 知识点 | 面试关键词 | 实际应用 |
|--------|------------|----------|
| checkout 与 setup | 第一步 checkout、setup-java/node | 任何构建型 workflow |
| 缓存 | actions/cache、path、key（含 hash） | 加速 Maven/npm 依赖 |
| 制品 | upload-artifact、download-artifact | Job 间传 jar/dist |
| Secrets | ${{ secrets.XXX }}、不写明文 | 密钥、凭据 |

------

### ⚠️ 常见坑与注意点

1. **现象**：缓存 key 不变导致依赖升级后仍用旧缓存。**正确做法**：key 里包含 **hashFiles('lock 文件或 pom.xml')**，依赖变则 key 变，自动失效并重建缓存。
2. **现象**：上传制品后忘记下载就部署。**正确做法**：部署 job 用 **needs** 依赖构建 job，第一步 **download-artifact** 再执行部署命令。
3. **现象**：在 run 里 echo ${{ secrets.XXX }} 导致泄露。**正确做法**：不要 **echo** 或 **print** Secrets；GitHub 会尝试掩码，但脚本里避免输出。

------

### 🎯 面试常见追问

| 面试官提问 | 回答思路 |
|------------|----------|
| 如何缓存依赖？ | 用 actions/cache，path 指到 .m2 或 node_modules，key 含 hashFiles 使依赖变时失效；下一步装依赖会命中缓存。若追问制品见本章 2️⃣。 |
| 如何传递制品？ | 构建 job 用 upload-artifact 上传 name+path；部署 job 用 needs 依赖构建 job，用 download-artifact 下载同一 name。 |
| Secrets 如何使用？ | 在仓库 Settings 里配置 Secrets；YAML 里用 ${{ secrets.XXX }} 引用；不要 echo 或写进文件提交。 |

## 第5章：多 Job 与矩阵

本章解决**多 Job 编排（并行与依赖）**以及**矩阵多维度运行**，与第 2 章 jobs 语法、第 4 章制品（job 间传产物）衔接。校招常考：needs、matrix、条件控制某 step 仅主分支执行。

------

### 1️⃣ 多 Job 与 needs

- **jobs** 下多个 **job** 默认**并行**执行。**needs**：指定依赖的 job 列表，**被依赖的 job 成功**后当前 job 才运行；用于“先构建、再部署”或“先测试、再发布”。
- **面试怎么说**：构建 job 产出 artifact，部署 job 写 `needs: [build]`，先 download-artifact 再部署；保证顺序与产物传递。

### 2️⃣ 矩阵（matrix）

- **strategy.matrix**：定义多组参数（如 **java: [8, 17]**、**os: [ubuntu-latest, windows-latest]**），每个**组合**跑一次当前 job；适合**多版本构建/测试**（如“在 Java 8 和 17 上各跑一遍测试”）。
- **exclude** / **include**：可从矩阵中排除或追加组合，控制运行范围。

### 3️⃣ 条件（if）

- **job** 或 **step** 的 **if**：条件为真才执行。常用：**github.ref == 'refs/heads/main'**（仅 main 分支）、**github.event_name == 'push'**、**success()**（上一步成功）、**failure()**（上一步失败，用于通知）。**面试怎么说**：例如“只有 main 合并后才部署生产”可写 deploy job 的 `if: github.ref == 'refs/heads/main'`。

------

### 📖 本章小结

| 知识点 | 面试关键词 | 实际应用 |
|--------|------------|----------|
| 多 Job | needs、并行与顺序、制品传递 | 构建→部署、测试→发布 |
| 矩阵 | strategy.matrix、多版本/多 OS | 多版本构建与测试 |
| 条件 | if、github.ref、success() | 仅主分支部署、失败通知 |

------

### ⚠️ 常见坑与注意点

1. **现象**：needs 的 job 失败后当前 job 被跳过但未显式标注。**正确做法**：理解“needs 依赖失败则后续 job 不跑”；若需要“失败也跑”（如通知），可用 **if: always()** 等，并理解状态。
2. **现象**：矩阵组合太多导致超时或额度耗尽。**正确做法**：用 **exclude** 减组合，或只在关键分支/ tag 跑全矩阵。
3. **现象**：if 条件写错导致该跑的没跑。**正确做法**：**github.ref** 是完整 ref（如 refs/heads/main）；比较时用字符串或 endsWith，查文档确认表达式。

------

### 🎯 面试常见追问

| 面试官提问 | 回答思路 |
|------------|----------|
| 如何控制某 step 仅在主分支执行？ | 在该 step 或 job 上写 if: github.ref == 'refs/heads/main'（或 endsWith('/main')）。若追问多版本见本章 2️⃣。 |
| 多 Job 如何保证顺序？ | 用 needs 指定依赖；例如 deploy 的 needs: [build]，则 build 成功后才跑 deploy。 |

## 第6章：与部署的衔接

本章解决**在 GitHub Actions 中如何执行部署**：SSH 到服务器、Docker 构建推送、K8s 部署等，以及**密钥安全**。与《自动化构建与部署》第 5 章、《持续集成与持续部署》衔接。校招常考：流水线中如何安全用密钥、如何构建并推送镜像。

------

### 1️⃣ 部署到服务器

- **方式**：用 **SSH**、**SCP** 或远程执行脚本；**密钥**存 **Secrets**（如 SSH 私钥、主机地址），在 step 中配置 SSH 后执行命令或 scp 上传制品。**注意**：生产建议**仅 tag 或 release 触发**，或加**环境审批**（GitHub Environments）。
- **面试怎么说**：把 SSH 私钥、主机、用户放进 Secrets；step 里用 **run** 或专用 Action（如 appleboy/ssh-action）连上服务器执行部署命令或拉取制品。

### 2️⃣ Docker 构建与推送

- **docker/build-push-action@v5**：在 workflow 中**构建并推送**镜像；可指定 **context**、**push**（是否 push）、**tags**；登录凭据用 **Secrets**（如 **DOCKER_USERNAME**、**DOCKER_PASSWORD** 或 **GITHUB_TOKEN** 推 **GHCR**）。**镜像 tag** 建议用 **Git SHA** 或 **Git tag**，便于追溯与回滚。
- **面试怎么说**：用 build-push-action 在 CI 里 build 并 push；登录信息放 Secrets；tag 用 ${{ github.sha }} 或 github.ref_name 保证唯一与可追溯。

### 3️⃣ K8s 与云厂商

- **K8s**：在 step 里用 **kubectl** 或 **helm**；**kubeconfig** 或 **token** 存 Secrets，配置 KUBECONFIG 或 **run: kubectl set image ...**。**云厂商**：用官方 Action（如 AWS、阿里云）或 **run: aws/cli**；凭据用 Secrets。**注意**：生产部署建议**权限最小化**、**仅 tag 触发**或**环境审批**。

------

### 📖 本章小结

| 知识点 | 面试关键词 | 实际应用 |
|--------|------------|----------|
| 部署到服务器 | SSH、Secrets、仅 tag/审批 | 传统主机部署 |
| Docker | build-push-action、Secrets、tag 追溯 | 镜像构建与推 GHCR/仓库 |
| K8s/云 | kubectl、helm、云 Action、Secrets | 云原生与云上发布 |

------

### ⚠️ 常见坑与注意点

1. **现象**：生产部署由任意 push 触发。**正确做法**：生产仅由 **tag** 或 **release** 触发，或使用 **Environments** 配置**审批**与**保护规则**。
2. **现象**：kubeconfig 或云 AK 写进 YAML。**正确做法**：全部放进 **Secrets**，在 step 里用 `env` 或临时文件注入，用完即不落盘（或仅限 runner 临时目录）。
3. **现象**：镜像推送到公共仓库未限制权限。**正确做法**：用 **GITHUB_TOKEN** 推 **GHCR** 时注意仓库可见性；若用 Docker Hub 等，用 Secrets 且不 echo。

------

### 🎯 面试常见追问

| 面试官提问 | 回答思路 |
|------------|----------|
| 流水线中如何安全使用密钥？ | 密钥存仓库/组织 Secrets；YAML 里用 ${{ secrets.XXX }}；不 echo、不写进代码；生产可用 Environments 审批。若追问构建并推送镜像见本章 2️⃣。 |
| 如何构建并推送镜像？ | 用 docker/build-push-action；登录凭据放 Secrets（或 GITHUB_TOKEN 推 GHCR）；tag 用 git sha 或 tag 便于追溯。 |

## 第7章：Reusable Workflows（可复用工作流）

本章讲解 GitHub Actions 的**可复用工作流**特性，解决多仓库/多项目重复配置的问题。与第 2～6 章基础语法衔接，实现 DRY（Don't Repeat Yourself）原则。校招常考：如何复用工作流、输入输出参数、使用场景。

------

### 1️⃣ 什么是 Reusable Workflows

- **可复用工作流**：将通用的 workflow 逻辑提取到**单独的仓库**或**单独文件**，供其他工作流**调用**。类似编程中的**函数/模块复用**。
- **为什么需要**：① 多项目使用相同的构建/测试流程；② 统一维护，一处修改全局生效；③ 减少重复代码，降低维护成本。

```
┌─────────────────────────────────────────────────────────────┐
│                    可复用工作流架构                           │
├─────────────────────────────────────────────────────────────┤
│  ┌──────────────────┐      ┌──────────────────────────────┐ │
│  │  .github/workflows/       │      │  调用方工作流 (caller.yml)   │ │
│  │  └── reusable-build.yml   │ ←──  │  uses: org/repo/.github/... │ │
│  │      (可复用工作流)        │      │  with:                      │ │
│  └──────────────────┘      │          java-version: '17'   │ │
│         ↑                  │          node-version: '20'   │ │
│         │ 被调用            │      └──────────────────────────────┘ │
│  ┌──────────────────┐      │                                         │
│  │  其他仓库调用      │ ←───┘                                         │
│  │  uses: org/...   │                                               │
│  └──────────────────┘                                               │
└─────────────────────────────────────────────────────────────────────┘
```

------

### 2️⃣ 创建可复用工作流

**文件位置**：`.github/workflows/reusable-build.yml`

**必须包含 `workflow_call` 触发器**：

```yaml
name: Reusable Build

on:
  workflow_call:                    # 声明为可复用工作流
    inputs:                         # 定义输入参数
      java-version:
        description: 'Java 版本'
        required: true
        type: string
        default: '17'
      node-version:
        description: 'Node 版本'
        required: false
        type: string
        default: '20'
      run-tests:
        description: '是否运行测试'
        required: false
        type: boolean
        default: true
    
    secrets:                        # 定义需要的 secrets
      SONAR_TOKEN:
        description: 'SonarQube Token'
        required: false
    
    outputs:                        # 定义输出
      build-status:
        description: '构建状态'
        value: ${{ jobs.build.outputs.status }}
      artifact-name:
        description: '制品名称'
        value: ${{ jobs.build.outputs.artifact }}

jobs:
  build:
    runs-on: ubuntu-latest
    outputs:
      status: ${{ steps.build-step.outcome }}
      artifact: ${{ steps.upload.outputs.artifact-name }}
    
    steps:
      - uses: actions/checkout@v4
      
      - name: Setup Java
        uses: actions/setup-java@v4
        with:
          java-version: ${{ inputs.java-version }}
          distribution: 'temurin'
      
      - name: Setup Node
        if: inputs.node-version != ''
        uses: actions/setup-node@v4
        with:
          node-version: ${{ inputs.node-version }}
      
      - name: Build
        id: build-step
        run: mvn clean package -DskipTests
      
      - name: Run Tests
        if: inputs.run-tests
        run: mvn test
      
      - name: SonarQube Analysis
        if: secrets.SONAR_TOKEN != ''
        run: mvn sonar:sonar -Dsonar.token=${{ secrets.SONAR_TOKEN }}
      
      - name: Upload Artifact
        id: upload
        uses: actions/upload-artifact@v4
        with:
          name: app-${{ inputs.java-version }}
          path: target/*.jar
```

**关键要素**：
| 要素 | 说明 |
|------|------|
| `workflow_call` | 声明此工作流可被调用 |
| `inputs` | 调用方传入的参数（支持 string/number/boolean） |
| `secrets` | 调用方传入的敏感信息 |
| `outputs` | 向调用方返回的数据 |

------

### 3️⃣ 调用可复用工作流

**同一仓库内调用**：
```yaml
name: CI

on: push

jobs:
  call-reusable:
    uses: ./.github/workflows/reusable-build.yml    # 相对路径
    with:
      java-version: '21'
      node-version: '18'
      run-tests: false
    secrets:
      SONAR_TOKEN: ${{ secrets.SONAR_TOKEN }}
```

**跨仓库调用**（推荐：集中管理）：
```yaml
name: CI

on: push

jobs:
  call-reusable:
    uses: my-org/shared-workflows/.github/workflows/java-build.yml@main
    with:
      java-version: '17'
    secrets: inherit                         # 继承所有 secrets（需权限）
    # 或显式传递：
    # secrets:
    #   SONAR_TOKEN: ${{ secrets.SONAR_TOKEN }}

  use-outputs:
    needs: call-reusable
    runs-on: ubuntu-latest
    steps:
      - name: Use Build Status
        run: |
          echo "构建状态: ${{ needs.call-reusable.outputs.build-status }}"
          echo "制品名称: ${{ needs.call-reusable.outputs.artifact-name }}"
```

**调用语法**：
```yaml
uses: owner/repo/path/to/workflow.yml@ref
```
- `owner/repo`：仓库名
- `path`：工作流文件路径（必须在 `.github/workflows/` 下）
- `ref`：分支、tag 或 commit SHA

------

### 4️⃣ 使用场景与最佳实践

**场景1：组织级统一构建流程**
```yaml
# my-org/shared-workflows/.github/workflows/standard-ci.yml
# 所有 Java 项目统一使用

name: Standard Java CI
on:
  workflow_call:
    inputs:
      java-version: { required: true, type: string }

jobs:
  quality-gate:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-java@v4
        with:
          java-version: ${{ inputs.java-version }}
          distribution: 'temurin'
      - run: mvn verify
      - run: mvn sonar:sonar
      - uses: advanced-security/maven-dependency-submission-action@v3
```

**场景2：多环境部署模板**
```yaml
# 可复用的部署工作流
name: Deploy to Environment
on:
  workflow_call:
    inputs:
      environment: { required: true, type: string }
      version: { required: true, type: string }

jobs:
  deploy:
    runs-on: ubuntu-latest
    environment: ${{ inputs.environment }}    # GitHub Environment 审批
    steps:
      - name: Deploy ${{ inputs.version }} to ${{ inputs.environment }}
        run: ./deploy.sh ${{ inputs.environment }} ${{ inputs.version }}
```

**最佳实践**：
| 实践 | 说明 |
|------|------|
| **版本控制** | 使用 tag（如 `@v1`）而非分支，避免破坏性更新 |
| **输入验证** | 使用 `required` 和 `type` 约束输入 |
| **文档说明** | 在可复用工作流中添加注释说明用途和参数 |
| **权限最小化** | 不要滥用 `secrets: inherit`，显式传递更安全 |

------

### 📖 本章小结

| 知识点 | 面试关键词 | 实际应用 |
|--------|------------|----------|
| 可复用工作流 | workflow_call、inputs/secrets/outputs | 多项目复用 |
| 调用方式 | uses: owner/repo/workflow.yml@ref | 本地/跨仓库调用 |
| 场景 | 统一构建、多环境部署模板 | 组织级标准化 |

------

### ⚠️ 常见坑与注意点

1. **现象**：修改可复用工作流后，所有调用方都受影响。**正确做法**：使用 **tag 版本**（如 `@v1.2.3`）而非 `@main`；重大变更时升级主版本（`@v2`）。
2. **现象**：secrets 传递失败。**正确做法**：调用方需显式传递或使用 `secrets: inherit`；被调用方需在 `workflow_call.secrets` 中声明。
3. **现象**：outputs 获取不到。**正确做法**：被调用方的 job 必须声明 `outputs`，且调用方使用 `needs.job-id.outputs.name` 语法获取。

### 🎯 面试常见追问

| 面试官提问 | 回答思路 |
|------------|----------|
| 什么是 Reusable Workflows？解决什么问题？ | GitHub Actions 的可复用工作流特性，通过 `workflow_call` 触发器定义。解决多项目重复配置问题，实现"一处定义，多处复用"，便于统一维护和更新。 |
| 如何调用其他仓库的可复用工作流？ | 使用 `uses: owner/repo/.github/workflows/file.yml@ref` 语法；可以传 `inputs` 和 `secrets`；使用 `secrets: inherit` 继承所有 secrets（需权限）。 |
| 可复用工作流如何返回数据给调用方？ | 在 `workflow_call.outputs` 中定义输出；job 中使用 `outputs` 映射步骤结果；调用方通过 `needs.job-id.outputs.name` 获取。 |

## 第8章：Self-hosted Runner（自托管运行器）

本章讲解 GitHub Actions 的**自托管 Runner**，解决内网部署、特殊硬件需求、合规要求等场景。与第 1 章 Runner 概念、第 6 章部署衔接。校招常考：自托管 Runner 适用场景、与 GitHub-hosted 区别、安全风险。

------

### 1️⃣ 什么是 Self-hosted Runner

- **自托管 Runner**：由用户自己提供和维护的执行环境（物理机、虚拟机、容器），而非 GitHub 托管的 runner。**为什么需要**：
  1. **内网资源访问**：访问内网数据库、私有 Nexus、内部 API
  2. **特殊硬件**：GPU、ARM 架构、大内存机器
  3. **合规要求**：数据不能出内网
  4. **自定义环境**：预装特定软件、License

| 对比项 | GitHub-hosted Runner | Self-hosted Runner |
|--------|---------------------|-------------------|
| **维护方** | GitHub | 用户自己 |
| **环境** | 标准化（ubuntu/windows/macos） | 完全自定义 |
| **网络** | 公网 | 可访问内网资源 |
| **费用** | 公开仓库免费，私有仓库按分钟计费 | 机器成本自负 |
| **启动时间** | 秒级 | 需提前启动并注册 |
| **安全性** | 每次全新环境 | 需自行保证隔离和清理 |

------

### 2️⃣ Runner 安装与配置

**系统要求**：
- 操作系统：Linux、Windows、macOS
- 硬件：至少 2 核 CPU、4GB 内存、20GB 磁盘
- 网络：能访问 GitHub（https://github.com）

**安装步骤（Linux 示例）**：

```bash
# 1. 创建目录
mkdir actions-runner && cd actions-runner

# 2. 下载最新 runner（x64 示例）
curl -o actions-runner-linux-x64-2.311.0.tar.gz \
  -L https://github.com/actions/runner/releases/download/v2.311.0/actions-runner-linux-x64-2.311.0.tar.gz

# 3. 解压
tar xzf ./actions-runner-linux-x64-2.311.0.tar.gz

# 4. 配置（从 GitHub 获取 token）
./config.sh --url https://github.com/OWNER/REPO --token YOUR_TOKEN

# 5. 安装为系统服务（推荐）
sudo ./svc.sh install
sudo ./svc.sh start

# 或直接运行
./run.sh
```

**获取 Token**：
- 仓库级：Settings → Actions → Runners → New self-hosted runner
- 组织级：Organization Settings → Actions → Runners

------

### 3️⃣ 工作流中使用自托管 Runner

**通过标签（labels）指定**：

```yaml
name: CI with Self-hosted Runner

on: push

jobs:
  build:
    # 使用自托管 runner
    runs-on: self-hosted           # 所有自托管 runner
    # 或指定标签
    # runs-on: [self-hosted, linux, gpu]
    
    steps:
      - uses: actions/checkout@v4
      
      - name: Build with GPU
        run: |
          nvidia-smi                    # 使用 GPU
          python train.py
      
      - name: Access Internal Service
        run: |
          curl http://internal-nexus:8081  # 访问内网 Nexus
```

**标签策略**：
| 标签示例 | 用途 |
|---------|------|
| `self-hosted, linux, x64` | 通用 Linux 构建 |
| `self-hosted, windows, gpu` | Windows + GPU |
| `self-hosted, arm64` | ARM 架构构建 |
| `self-hosted, production` | 生产部署专用 |

------

### 4️⃣ Runner 管理与安全

**管理命令**：
```bash
# 查看状态
./config.sh --check

# 移除 runner
./config.sh remove --token YOUR_TOKEN

# 服务管理
sudo ./svc.sh stop
sudo ./svc.sh start
sudo ./svc.sh status
```

**安全最佳实践**：

| 风险 | 防护措施 |
|------|---------|
| **Runner 被滥用** | 仅对受信仓库启用；使用 Environment 保护规则 |
| **残留数据** | 每次 job 后清理工作目录；使用 `--ephemeral` 一次性 runner |
| **权限过大** | Runner 机器最小权限；不存储长期凭据 |
| **网络暴露** | 限制 runner 出站规则；使用防火墙 |

**一次性 Runner（推荐用于敏感环境）**：
```bash
# 配置时添加 --ephemeral，执行一个 job 后自动注销
./config.sh --url https://github.com/OWNER/REPO --token TOKEN --ephemeral
```

------

### 5️⃣ 容器化 Runner（高级）

使用 Docker 运行 runner，便于管理和扩展：

```dockerfile
FROM ubuntu:22.04

# 安装依赖
RUN apt-get update && apt-get install -y \
    curl sudo git jq \
    && rm -rf /var/lib/apt/lists/*

# 创建用户
RUN useradd -m runner && usermod -aG sudo runner

WORKDIR /home/runner

# 下载并配置 runner
RUN curl -o runner.tar.gz -L https://github.com/actions/runner/releases/download/v2.311.0/actions-runner-linux-x64-2.311.0.tar.gz \
    && tar xzf runner.tar.gz \
    && rm runner.tar.gz

COPY --chown=runner:runner entrypoint.sh .
RUN chmod +x entrypoint.sh

USER runner

ENTRYPOINT ["./entrypoint.sh"]
```

```bash
# entrypoint.sh
#!/bin/bash
./config.sh --url "$REPO_URL" --token "$TOKEN" --ephemeral --unattended
./run.sh
```

**K8s 中运行 Runner**：
```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: github-runner
spec:
  replicas: 2
  selector:
    matchLabels:
      app: github-runner
  template:
    metadata:
      labels:
        app: github-runner
    spec:
      containers:
      - name: runner
        image: my-registry/github-runner:latest
        env:
        - name: REPO_URL
          value: "https://github.com/my-org/my-repo"
        - name: TOKEN
          valueFrom:
            secretKeyRef:
              name: github-runner-token
              key: token
```

| 知识点 | 面试关键词 | 实际应用 |
|--------|-----------|---------|
| 自托管 Runner | 内网访问、特殊硬件、合规 | 私有环境构建部署 |
| 安装配置 | config.sh、token、标签 | 机器注册与启动 |
| 使用方式 | runs-on: self-hosted/标签 | 工作流指定 runner |
| 安全 | --ephemeral、权限最小化 | 防止数据泄露和滥用 |

------

### ⚠️ 常见坑与注意点

1. **现象**：Runner 执行多个 job 后磁盘满了。**正确做法**：配置 **自动清理** 或使用 `--ephemeral` 一次性 runner；定期监控磁盘使用。
2. **现象**：Runner 机器被攻破，所有仓库受影响。**正确做法**：使用 **Repository 级 runner** 而非 Organization 级；敏感操作用 Environment 保护规则。
3. **现象**：Runner 离线但 GitHub 仍显示在线。**正确做法**：检查网络连接；使用 `./svc.sh status` 查看服务状态；必要时重新注册。
4. **现象**：工作流卡在等待 runner。**正确做法**：检查标签是否匹配；确认 runner 已启动并注册成功；查看 runner 日志排查问题。

### 🎯 面试常见追问

| 面试官提问 | 回答思路 |
|-----------|---------|
| 什么时候需要用 Self-hosted Runner？ | ① 访问内网资源；② 特殊硬件需求（GPU/ARM）；③ 合规要求数据不出内网；④ 需要预装特定软件或 License。 |
| 自托管 Runner 和 GitHub-hosted 有什么区别？ | 自托管由用户维护，可访问内网、自定义环境，但需自行保证安全和清理；GitHub-hosted 由 GitHub 维护，环境标准化，每次全新，但只能访问公网。 |
| 如何保证自托管 Runner 的安全性？ | 使用 `--ephemeral` 一次性 runner；每次 job 后清理工作目录；runner 机器最小权限；对敏感操作使用 Environment 保护规则；限制网络出站。 |
