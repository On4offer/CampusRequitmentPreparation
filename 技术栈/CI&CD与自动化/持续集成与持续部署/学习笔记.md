# 📘《持续集成与持续部署学习与实战手册》

> 系统学习见下文各章；日常常用概念与使用场景速查见同目录《常用概念与使用场景》。

------

## 第1章：CI/CD 概述

### 1.1 什么是持续集成（CI）

- **持续集成**指开发者**频繁**将代码合并到主干（或共享分支），每次合并后**自动触发构建与测试**，尽早发现集成错误与回归。
- 核心：**自动化构建**、**自动化测试**（单元、集成）、**快速反馈**；减少“集成地狱”，保证主干可部署。

### 1.2 什么是持续交付/部署（CD）

- **持续交付（Continuous Delivery）**：每次通过流水线的内容都可**随时**安全发布，发布动作由人工触发；保证“能发布”而非“必须自动发布”。
- **持续部署（Continuous Deployment）**：通过流水线后**自动**发布到生产（或目标环境）；从提交到上线全自动。
- 关系：CI 是基础；CD 在 CI 之上增加自动化发布与环境推进；多数团队先做 CI + 持续交付，再视情况做持续部署。

## 第2章：流水线阶段与质量门禁

1. **典型阶段**：**拉取代码** → **安装依赖** → **构建/编译** → **静态检查（Lint）** → **单元测试** → **集成测试** → **构建制品** → **部署到环境**（如测试/预发/生产）。
2. **质量门禁**：某一阶段失败则流水线失败，后续阶段不执行或不可发布；如测试不通过不能部署、覆盖率不达标可配置为失败或警告。
3. **并行与缓存**：独立任务可并行（如多模块并行构建）；依赖安装、构建缓存可加速流水线。
4. **面试要点**：CI 与 CD 区别、流水线一般包含哪些阶段、质量门禁的作用。

------

## 第3章：环境与发布策略

1. **环境**：开发（dev）、测试（test）、预发/预生产（staging）、生产（prod）；配置与数据隔离，流水线按分支或标签推进到不同环境。
2. **发布策略**：**蓝绿**（两套环境切换）、**金丝雀/灰度**（部分流量新版本）、**滚动**（逐步替换实例）；回滚预案（保留上一版本、一键回滚）。
3. **分支与流水线**：main 合并触发构建与测试；tag 或 release 分支触发部署；feature 分支仅跑测试不部署；按团队约定划分。
4. **面试要点**：蓝绿与灰度区别、如何做回滚、环境如何隔离。

------

## 第4章：与版本控制、构建、部署的衔接

1. **版本控制**：流水线由 **push、merge、tag** 等事件触发；需要代码权限、密钥管理（如 Secrets）、避免敏感信息进仓库。
2. **构建**：Maven/Gradle/npm/pnpm 等在流水线中执行；产物（jar、镜像、前端 dist）上传到制品库或镜像仓库。
3. **部署**：流水线调用部署脚本、SSH、K8s API、或云厂商 API，将制品部署到目标环境；需凭据与权限最小化。
4. **面试要点**：CI/CD 如何与 Git 配合、制品是什么、流水线中的密钥如何管理。

------

## 第5章：常见工具与选型

1. **云原生/托管**：**GitHub Actions**、**GitLab CI**、**Jenkins X**、云厂商流水线（阿里云效、腾讯 CODING、AWS CodePipeline）；与代码托管一体，配置即用。
2. **自建**：**Jenkins**（插件丰富、可自托管）、**Tekton**（K8s 原生）；适合内网或定制化强的场景。
3. **选型考量**：与 Git 平台是否一体、是否支持自托管、与现有构建/部署工具集成、团队学习成本。
4. **面试要点**：Jenkins 与 GitHub Actions 区别、为什么需要 CI/CD。
