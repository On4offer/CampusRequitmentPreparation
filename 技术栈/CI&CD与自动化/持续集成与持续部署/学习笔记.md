# 📘《持续集成与持续部署学习与实战手册》

> 系统学习见下文各章；日常常用概念与使用场景速查见同目录《常用概念与使用场景》。  
> 适用：CI/CD 概念与流水线设计，面向校招与后端/运维（CI/CD 区别、流水线阶段、环境与发布策略、常见工具）。

------

## 第1章：CI/CD 概述

本章建立对**持续集成（CI）与持续交付/部署（CD）**的整体认知：是什么、解决什么问题、三者关系，为后续流水线（第 2 章）、环境与发布（第 3 章）、工具选型（第 5 章）打基础。校招常从“CI 和 CD 区别”问起。

------

### 1️⃣ 什么是持续集成（CI）

- **持续集成（Continuous Integration）**：开发者**频繁**将代码合并到主干（或共享分支），每次合并后**自动触发构建与测试**，尽早发现集成错误与回归。**为什么需要**：避免长期不合并导致的“集成地狱”、保证主干随时可构建可测。
- **核心**：**自动化构建**、**自动化测试**（单元、集成）、**快速反馈**；减少手工集成、保证主干可部署。

### 2️⃣ 持续交付与持续部署（CD）

- **持续交付（Continuous Delivery）**：通过流水线的内容**随时可安全发布**，**发布动作由人工触发**；保证“能发布”而非“必须自动发布”。**为什么**：业务需要控制上线节奏、审批或择时发布。
- **持续部署（Continuous Deployment）**：通过流水线后**自动**发布到生产（或目标环境）；从提交到上线**全自动**。**为什么**：追求极致效率、适合可快速回滚与灰度场景。
- **关系**：**CI 是基础**（先有自动构建与测试）；**CD 在 CI 之上**增加自动化发布与环境推进；多数团队先做 **CI + 持续交付**，再视情况做**持续部署**。

| 概念 | 含义 | 关键点 |
|------|------|--------|
| CI | 频繁合并 + 自动构建与测试 | 主干可构建、快速反馈 |
| 持续交付 | 可随时发布，人工触发发布 | “能发布” |
| 持续部署 | 通过流水线即自动发布 | “自动发布” |

------

### 📖 本章小结

| 知识点 | 面试关键词 | 实际应用 |
|--------|------------|----------|
| CI | 频繁合并、自动构建与测试、快速反馈 | 避免集成地狱、主干可部署 |
| 持续交付 vs 持续部署 | 人工发布 vs 自动发布、能发布 vs 必须自动 | 团队节奏与风险控制 |
| 三者关系 | CI 是基础、CD 在 CI 之上 | 实施顺序：先 CI 再持续交付再持续部署 |

------

### ⚠️ 常见坑与注意点

1. **现象**：把“持续交付”和“持续部署”混用。**原因**：都叫 CD。**正确做法**：持续交付 = 可随时发布、人工点发布；持续部署 = 通过流水线即自动发布到生产；面试时说清“我们做的是持续交付还是持续部署”。
2. **现象**：只做构建不做测试就称 CI。**正确做法**：CI 应包含**自动化测试**（至少单元测试），否则只是自动构建，无法保证“合并后主干可靠”。
3. **现象**：认为上了工具就是 CI/CD。**正确做法**：CI/CD 是**实践与流程**，工具是支撑；要有约定（如合并前必须通过流水线、主干保护等）。

------

## 第2章：流水线阶段与质量门禁

本章解决**流水线一般包含哪些阶段**以及**质量门禁如何卡住不合格构建**，与第 1 章 CI/CD、第 4 章构建与部署衔接。校招常考：流水线阶段、质量门禁、并行与缓存。

------

### 1️⃣ 典型流水线阶段

- **流水线（Pipeline）**：从代码变更到可部署（或已部署）的**自动化流程**，由一系列**阶段（Stage）**和**步骤（Step）**组成。**为什么需要**：把“拉代码→构建→测试→部署”标准化、可重复、可追溯。
- **典型阶段顺序**：**拉取代码** → **安装依赖** → **构建/编译** → **静态检查（Lint）** → **单元测试** → **集成测试** → **构建制品**（jar/镜像/dist） → **部署到环境**（测试/预发/生产）。不同项目可增删阶段（如安全扫描、E2E 测试）。

```
  [代码仓库] → 拉取 → 依赖 → 构建 → Lint → 单测 → 集成测 → 制品 → 部署
       ↑                                                              ↓
       └──────────────── 质量门禁：任一步失败则流水线失败 ─────────────┘
```

> 💬 **面试怎么说**：流水线一般包括拉代码、装依赖、构建、静态检查、单元测试、集成测试、打制品、部署；某阶段失败则整条流水线失败，后续不执行或不可发布。

### 2️⃣ 质量门禁

- **质量门禁**：某一阶段**失败则流水线失败**，后续阶段不执行或**不可发布**。例如：测试不通过不能部署、**覆盖率**不达标可配置为失败或警告、Lint 不通过可阻断。
- **作用**：保证只有“通过门禁”的代码才能进入下一环境或发布；避免坏代码上线。

### 3️⃣ 并行与缓存

- **并行**：相互独立的任务可**并行**执行（如多模块并行构建、多任务并行测试），缩短总耗时。
- **缓存**：**依赖安装**（如 Maven 本地库、npm node_modules）、**构建产物**（如增量编译）可缓存，下次流水线复用，**加速**执行。

------

### 📖 本章小结

| 知识点 | 面试关键词 | 实际应用 |
|--------|------------|----------|
| 流水线阶段 | 拉代码、依赖、构建、Lint、单测、集成测、制品、部署 | 设计或描述一条流水线 |
| 质量门禁 | 阶段失败即流水线失败、不可发布 | 保证只有合格构建才能发布 |
| 并行与缓存 | 并行缩短时间、缓存加速 | 优化流水线耗时 |

------

### ⚠️ 常见坑与注意点

1. **现象**：流水线没有测试阶段就部署。**正确做法**：至少要有**单元测试**门禁；集成测试与 E2E 视项目而定，关键路径要有门禁。
2. **现象**：每次流水线都全量装依赖、全量构建。**正确做法**：用**缓存**（依赖目录、构建缓存）；并行能拆则拆，缩短反馈时间。
3. **现象**：质量门禁过松（如覆盖率 0 也过）。**正确做法**：根据团队约定设合理阈值（覆盖率、Lint 零错误等），并逐步提高，避免形同虚设。

------

## 第3章：环境与发布策略

本章解决**有哪些环境、如何隔离**以及**如何发布与回滚**（蓝绿、灰度、滚动），与第 2 章流水线（推进到不同环境）、第 4 章部署衔接。校招常考：环境划分、蓝绿与灰度区别、回滚。

------

### 1️⃣ 环境划分

- **常见环境**：**开发（dev）**、**测试（test）**、**预发/预生产（staging）**、**生产（prod）**。**配置与数据**隔离（不同库、不同配置中心命名空间）；流水线按**分支或标签**推进到不同环境（如 main→test，tag→prod）。
- **为什么需要多环境**：开发与测试不影响生产；预发尽量贴近生产，做上线前验证。

### 2️⃣ 发布策略对比

| 策略 | 思路 | 优点 | 缺点/注意 |
|------|------|------|-----------|
| **蓝绿** | 两套**完整环境**，发布时**切换流量**到新版本 | 切换快、回滚即切回 | 资源双倍、需兼容数据与配置 |
| **金丝雀/灰度** | **部分流量**到新版本，逐步放量 | 风险可控、可观察再全量 | 需流量调度与监控 |
| **滚动** | **逐步替换**实例（如 K8s 滚动更新） | 资源不双倍、平滑 | 发布期间新旧并存、需兼容 |

> 💬 **面试怎么说**：蓝绿是两套环境切换；灰度是部分流量新版本、逐步放量；滚动是逐步替换实例。回滚要预留上一版本或镜像，支持一键回滚或快速切回。

### 3️⃣ 回滚

- **回滚预案**：保留**上一版本**制品或镜像；支持**一键回滚**（切回蓝绿旧环境、或灰度切回全量旧版、或 K8s 回滚 Deployment）。**为什么重要**：发布异常时可快速恢复，减少故障时长。

### 4️⃣ 分支与流水线触发

- **常见约定**：**main 合并**触发构建与测试；**tag 或 release 分支**触发部署到预发/生产；**feature 分支**仅跑测试不部署。按团队约定划分，保证“谁合到哪触发什么”清晰。

------

### 📖 本章小结

| 知识点 | 面试关键词 | 实际应用 |
|--------|------------|----------|
| 环境 | dev/test/staging/prod、配置与数据隔离 | 多环境推进与隔离 |
| 发布策略 | 蓝绿、灰度、滚动、回滚 | 选型与风险控制 |
| 分支与触发 | main 构建测试、tag 部署、feature 仅测 | 与 Git 配合 |

------

### ⚠️ 常见坑与注意点

1. **现象**：预发与生产配置不一致导致“预发没问题、生产出问题”。**正确做法**：预发尽量与生产**配置一致**（仅数据可脱敏）；配置用同一套管理（如配置中心不同 namespace）。
2. **现象**：没有回滚预案就上线。**正确做法**：发布前保留上一版本制品/镜像，并演练回滚步骤；蓝绿要能立刻切回，灰度要能收口。
3. **现象**：蓝绿切换时数据库或缓存未考虑。**正确做法**：蓝绿若共库，要保证**数据与 schema 兼容**；若双库，要有数据同步或切换方案。

------

### 🎯 面试常见追问

| 面试官提问 | 回答思路 |
|------------|----------|
| 蓝绿和灰度有什么区别？ | 蓝绿是两套完整环境、切换流量；灰度是部分流量到新版本、逐步放量。蓝绿回滚快，灰度风险更可控。若追问回滚见本章 3️⃣。 |
| 如何做回滚？ | 保留上一版本制品/镜像；蓝绿切回旧环境、灰度收口到旧版、K8s 回滚 Deployment；要有一键或标准化回滚流程。 |
| 环境如何隔离？ | 不同环境用不同配置（配置中心 namespace）、不同数据库/库名、不同域名；流水线按分支或 tag 推到对应环境。 |

## 第4章：与版本控制、构建、部署的衔接

本章解决**流水线如何与 Git、构建工具、部署方式**衔接，以及**密钥与权限**管理。与第 2 章流水线阶段、第 5 章工具（Jenkins、GitHub Actions 等）关联。校招常考：触发方式、制品、密钥管理。

------

### 1️⃣ 版本控制与触发

- **触发**：流水线由 **push、merge、tag**、**PR 事件**等触发；在 Git 平台（GitHub/GitLab）或 CI 工具中配置** webhook / 轮询**。**权限**：需要代码**读权限**（拉取）、必要时 **tag 或保护分支**权限；**敏感信息**不能进仓库，用 **Secrets** 或环境变量注入。
- **为什么需要**：代码是流水线输入源；触发要准确（如仅 main 合并触发生产部署）、密钥不能泄露。

### 2️⃣ 构建与制品

- **构建**：在流水线中执行 **Maven/Gradle/npm/pnpm** 等；产物（**jar、镜像、前端 dist**）称为**制品**。**制品库**：制品上传到**制品库**（如 Nexus、Harbor 镜像仓库、对象存储），便于版本管理与部署时拉取。
- **面试怎么说**：流水线里执行 mvn package 或 docker build，产物上传到 Nexus 或镜像仓库；部署阶段从制品库拉取指定版本部署。

### 3️⃣ 部署方式与权限

- **部署**：流水线调用**部署脚本、SSH、K8s API、或云厂商 API**，将制品部署到目标环境。**凭据**：用 **Secrets** 存 SSH 密钥、K8s kubeconfig、云 AK/SK 等；**权限最小化**，流水线仅具备部署所需权限。
- **注意**：生产部署建议有人工审批或仅 tag 触发，避免误操作。

------

### 📖 本章小结

| 知识点 | 面试关键词 | 实际应用 |
|--------|------------|----------|
| 触发 | push/merge/tag、webhook、分支策略 | 何时跑流水线、何时部署 |
| 制品 | jar/镜像/dist、制品库、版本管理 | 构建产出与部署输入 |
| 密钥与权限 | Secrets、不进仓库、权限最小化 | 安全与合规 |

------

### ⚠️ 常见坑与注意点

1. **现象**：密钥写在代码或配置文件里并提交仓库。**正确做法**：敏感信息用 **Secrets** 或环境变量，在流水线中注入；仓库只放非敏感配置。
2. **现象**：流水线账号权限过大。**正确做法**：**最小权限**：只能拉指定仓库、只能推指定制品库、只能部署指定项目/命名空间。
3. **现象**：任意分支 push 都能触发生产部署。**正确做法**：生产部署仅由 **tag 或 release 分支**触发，或加**人工审批**；main 合并可只触发测试环境。

------

### 🎯 面试常见追问

| 面试官提问 | 回答思路 |
|------------|----------|
| CI/CD 如何与 Git 配合？ | 流水线由 push/merge/tag 触发；按分支策略决定跑构建测试还是部署；代码从 Git 拉取，密钥用 Secrets 不進仓库。若追问制品见本章 2️⃣。 |
| 制品是什么？流水线中的密钥如何管理？ | 制品是构建产物（jar、镜像、前端包），存制品库；部署时拉取指定版本。密钥用 Secrets 存储，在流水线中注入，不写进代码与仓库。 |

## 第5章：常见工具与选型

本章解决**CI/CD 常见工具有哪些、如何选型**，与第 2～4 章（流水线、环境、构建部署）对应到具体工具。校招常考：Jenkins 与 GitHub Actions 区别、为什么需要 CI/CD。

------

### 1️⃣ 常见工具对比

| 类型 | 典型 | 特点 | 适用 |
|------|------|------|------|
| **云原生/托管** | **GitHub Actions**、**GitLab CI**、云效、CODING、AWS CodePipeline | 与代码托管一体、**配置即用**、按量计费 | 开源/云上项目、快速上手 |
| **自建** | **Jenkins**、Tekton（K8s 原生） | **插件丰富**、可**自托管**、定制强 | 内网、定制多、已有 Jenkins 资产 |
| **K8s 原生** | Tekton、Argo CD（偏 GitOps） | 以 K8s 资源定义流水线/应用 | 云原生栈、K8s 内运行 |

### 2️⃣ Jenkins 与 GitHub Actions 区别（面试常问）

| 维度 | Jenkins | GitHub Actions |
|------|----------|----------------|
| **部署** | 多为**自建**、自己维护 | **托管**在 GitHub，无需维护 |
| **配置** | 界面 + Pipeline 脚本（Groovy） | **YAML** 文件放在仓库 `.github/workflows` |
| **触发** | 需配置 webhook 或轮询 | 与 **Git 事件**天然集成（push、PR、tag） |
| **生态** | **插件**极多、可对接各种系统 | 市场 Actions、与 GitHub 生态紧 |

> 💬 **面试怎么说**：Jenkins 自建、插件多、适合内网与深度定制；GitHub Actions 托管、配置即用、与 GitHub 深度集成，适合开源与云上项目。若追问 GitLab CI 可说“和 Actions 类似，和 GitLab 一体”。

### 3️⃣ 选型考量

- **与 Git 平台是否一体**：同平台（GitHub/GitLab）用自带 CI 省事。**是否支持自托管**：内网或合规要求时选 Jenkins/Tekton。**与现有构建/部署工具集成**：如已有 Jenkins 流水线、K8s、云厂商，看插件或 API 支持。**团队学习成本**：YAML 配置易上手，Jenkins 脚本稍重。

------

### 📖 本章小结

| 知识点 | 面试关键词 | 实际应用 |
|--------|------------|----------|
| 工具类型 | 托管（Actions、GitLab CI）、自建（Jenkins）、K8s 原生（Tekton） | 选型依据 |
| Jenkins vs Actions | 自建 vs 托管、插件 vs YAML、与 Git 集成 | 面试对比 |
| 选型考量 | Git 一体、自托管、集成、学习成本 | 实际选型 |

------

### ⚠️ 常见坑与注意点

1. **现象**：盲目上 Jenkins 却无人维护。**正确做法**：自建要有**维护与升级**能力；小团队或开源项目可优先托管 CI（Actions、GitLab CI）。
2. **现象**：认为“用上 Jenkins 就是 CI/CD”。**正确做法**：工具是支撑，**流程与约定**才是关键（分支策略、门禁、谁负责修红）。
3. **现象**：流水线配置全部写死在界面。**正确做法**：**Pipeline as Code**（Jenkinsfile、GitHub Actions YAML）进仓库，可版本管理与复用。

------

### 🎯 面试常见追问

## 第6章：Jenkins Pipeline 详解

本章深入讲解 **Jenkins Pipeline** 的配置与使用，与第 5 章工具选型、同目录《GitHub Actions》形成对比。校招常考：Jenkinsfile 结构、声明式与脚本式 Pipeline、阶段与步骤、凭据管理。

------

### 1️⃣ Jenkins Pipeline 概述

- **Pipeline**：Jenkins 中定义持续交付流水线的**一套插件**，将构建、测试、部署等步骤代码化。**Pipeline as Code**：将流水线配置写在 **Jenkinsfile** 中，随代码一起版本管理。
- **两种语法**：**声明式（Declarative）** 结构化、易读，适合大多数场景；**脚本式（Scripted）** 基于 Groovy，更灵活但复杂。校招与生产推荐先用**声明式**。

| 特性 | 声明式 Pipeline | 脚本式 Pipeline |
|------|-----------------|-----------------|
| **语法** | 结构化、固定块（pipeline/agent/stages） | Groovy 脚本、自由编程 |
| **可读性** | 高，像 YAML | 中，像代码 |
| **灵活性** | 满足 90% 场景 | 复杂逻辑、动态生成阶段 |
| **校验** | 进入流水线前语法校验 | 运行时才发现错误 |
| **推荐** | ✅ 校招与生产首选 | 特殊复杂场景 |

------

### 2️⃣ 声明式 Pipeline 结构

```groovy
pipeline {
    agent any                    // 在任何可用节点运行
    
    environment {                // 环境变量
        MAVEN_HOME = '/opt/maven'
        IMAGE_NAME = 'myapp'
    }
    
    options {                    // 流水线选项
        buildDiscarder(logRotator(numToKeepStr: '10'))
        timeout(time: 30, unit: 'MINUTES')
    }
    
    triggers {                   // 触发器
        pollSCM('H/5 * * * *')   // 每5分钟轮询代码库
    }
    
    stages {                     // 阶段集合
        stage('Build') {
            steps {
                sh 'mvn clean package -DskipTests'
            }
        }
        
        stage('Test') {
            steps {
                sh 'mvn test'
            }
            post {
                always {
                    junit 'target/surefire-reports/*.xml'
                }
            }
        }
        
        stage('Deploy') {
            when {
                branch 'main'      // 只在 main 分支执行
            }
            steps {
                sh './deploy.sh'
            }
        }
    }
    
    post {                       // 后置处理
        always {
            cleanWs()            // 清理工作区
        }
        success {
            echo 'Pipeline 成功！'
        }
        failure {
            echo 'Pipeline 失败！'
        }
    }
}
```

------

### 3️⃣ 核心概念详解

#### **Agent（执行节点）**

| 配置 | 说明 |
|------|------|
| `agent any` | 任意可用节点 |
| `agent none` | 不在根级别指定，各 stage 自行指定 |
| `agent { label 'linux && jdk17' }` | 指定标签（标签在节点配置中设置） |
| `agent { docker 'maven:3.9' }` | 在 Docker 容器中运行 |

#### **Stage（阶段）与 Step（步骤）**

- **Stage**：逻辑分组，如 Build、Test、Deploy，在 Jenkins UI 中可视化展示。
- **Step**：具体动作，如 `sh`（执行 shell）、`git`（拉代码）、`archiveArtifacts`（归档制品）。

```groovy
stage('Build') {
    steps {
        git url: 'https://github.com/example/repo.git', branch: 'main'
        sh 'mvn clean package'
        archiveArtifacts artifacts: 'target/*.jar', fingerprint: true
    }
}
```

#### **Post（后置处理）**

| 条件 | 触发时机 |
|------|----------|
| `always` | 无论结果都执行 |
| `success` | 成功时 |
| `failure` | 失败时 |
| `unstable` | 不稳定（如测试失败但继续） |
| `changed` | 状态变化时（成功→失败 或 失败→成功） |

### 4️⃣ 凭据与 Secrets 管理

Jenkins 提供 **Credentials** 插件管理敏感信息：

```groovy
pipeline {
    agent any
    
    environment {
        // 从 Jenkins 凭据注入
        DOCKER_USER = credentials('docker-username')
        DOCKER_PASS = credentials('docker-password')
        // 或 SSH 私钥
        SSH_KEY = credentials('deploy-ssh-key')
    }
    
    stages {
        stage('Deploy') {
            steps {
                // 使用凭据
                sh 'echo $DOCKER_PASS | docker login -u $DOCKER_USER --password-stdin'
                
                // 或使用 withCredentials 临时注入
                withCredentials([string(credentialsId: 'api-key', variable: 'API_KEY')]) {
                    sh 'curl -H "Authorization: $API_KEY" ...'
                }
            }
        }
    }
}
```

**凭据类型**：
| 类型 | 用途 |
|------|------|
| **Username with password** | 用户名密码 |
| **SSH Username with private key** | SSH 密钥 |
| **Secret file** | 配置文件（如 kubeconfig） |
| **Secret text** | API Key、Token |

### 5️⃣ 多分支流水线（Multibranch Pipeline）

自动为 Git 仓库的每个分支创建流水线：

1. **新建 Item** → **Multibranch Pipeline**
2. **Branch Sources** 添加 Git 仓库地址
3. **Build Configuration** 指定 Jenkinsfile 路径（默认根目录 `Jenkinsfile`）
4. Jenkins 自动扫描分支，每个分支独立构建

```groovy
// Jenkinsfile 中根据分支做不同处理
stage('Deploy') {
    when {
        anyOf {
            branch 'main'
            branch 'release/*'
        }
    }
    steps {
        script {
            if (env.BRANCH_NAME == 'main') {
                sh './deploy-staging.sh'
            } else if (env.BRANCH_NAME.startsWith('release/')) {
                sh './deploy-prod.sh'
            }
        }
    }
}
```

### ⚠️ 常见坑与注意点

1. **现象**：Jenkinsfile 语法错误导致流水线无法启动。**正确做法**：使用 **声明式** 减少错误；Jenkins 提供 **Pipeline Syntax** 生成器辅助编写；提交前用 `Replay` 功能测试。
2. **现象**：凭据在日志中泄露。**正确做法**：使用 `credentials()` 或 `withCredentials()`，Jenkins 会自动打码；**绝不用 `echo` 打印敏感信息**。
3. **现象**：长时间运行的流水线占用节点。**正确做法**：设置 `options { timeout(...) }`；使用 `agent none` 让各 stage 按需分配节点。
