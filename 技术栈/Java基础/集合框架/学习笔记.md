# 🧩 **第一章：集合框架概述**

> **目标：**
>  掌握 Java 集合框架（JCF, Java Collections Framework）整体设计思想、层级结构、核心接口与分类，建立从“容器”角度理解数据结构的体系思维。

## 🧠 1. 什么是集合（Collection）

### 📘 概念

在 Java 中，**集合（Collection）** 是用于存储、操作和传输对象的 **容器对象（Container）**。
 它用于替代数组，提供了更灵活的**动态扩容、元素查找、排序、去重、并发访问控制**等功能。

简单来说：

> **集合 = 可变长度的对象容器**

------

### 📦 与数组的区别

| 特性     | 数组（Array）    | 集合（Collection）                               |
| -------- | ---------------- | ------------------------------------------------ |
| 长度     | 固定             | 可变                                             |
| 元素类型 | 基本类型 / 对象  | 仅对象（引用类型）                               |
| 操作支持 | 仅索引访问       | 增删改查、遍历、排序、过滤等                     |
| 线程安全 | 否               | 部分集合支持（如 `Vector`、`ConcurrentHashMap`） |
| 泛型支持 | 无（需要包装类） | 有泛型，类型安全                                 |

------

### ✅ 示例

```java
// 数组
String[] arr = {"A", "B", "C"};

// 集合
List<String> list = new ArrayList<>();
list.add("A");
list.add("B");
list.add("C");
```

集合相比数组，更易于**动态扩展和操作对象集合**。

------

## 🎯 2. 集合框架的设计目标与优势

### 📌 设计目标

1. **统一接口**：为各种集合类型提供标准化接口（List、Set、Map等）。
2. **可扩展性**：允许自定义实现（如自定义HashMap、LRU缓存）。
3. **高性能**：内部采用高效数据结构（如哈希表、红黑树、链表）。
4. **算法与数据分离**：算法通过 `Collections` 工具类独立实现。
5. **类型安全与泛型支持**：从 JDK1.5 开始，集合全面引入泛型。

------

### 🌟 集合框架的优势

| 优势           | 说明                                             |
| -------------- | ------------------------------------------------ |
| 🧱 统一性       | 所有集合遵循统一的接口规范（Collection / Map）   |
| 🔄 可扩展性     | 自定义实现接口即可创建新集合                     |
| ⚡ 高效性       | HashMap、ArrayList 等经过优化的算法实现          |
| 🔒 安全性       | 支持同步包装（`Collections.synchronizedList()`） |
| 🧩 泛型支持     | 保证类型安全，避免强制类型转换                   |
| 🔍 丰富算法支持 | 排序、搜索、打乱、同步化、不可变集合等           |

## ⚖️ 4. Collection 与 Map 的区别

| 对比项       | Collection                     | Map                                 |
| ------------ | ------------------------------ | ----------------------------------- |
| 存储结构     | 单值元素集合                   | 键值对集合                          |
| 元素访问     | 通过迭代器或下标               | 通过 key 访问                       |
| 接口层级     | List、Set、Queue               | HashMap、TreeMap、LinkedHashMap     |
| 是否允许重复 | List 允许，Set 不允许          | key 不可重复，value 可重复          |
| 典型实现类   | ArrayList、HashSet、LinkedList | HashMap、TreeMap、ConcurrentHashMap |

------

### 💡 小例子对比

```java
// Collection 示例：List
List<String> names = new ArrayList<>();

names.add("Jerry");
System.out.println(names.get(0)); // Tom

// Map 示例：Key-Value 存储
Map<String, Integer> score = new HashMap<>();
score.put("Tom", 90);
score.put("Jerry", 85);
System.out.println(score.get("Tom")); // 90
```

## 🧠 6. JCF（Java Collections Framework）的设计思想

### ✳️ 设计核心：**接口 + 实现 + 算法**

| 模块                          | 说明                   | 示例                                      |
| ----------------------------- | ---------------------- | ----------------------------------------- |
| **接口（Interfaces）**        | 定义行为规范           | Collection、List、Set、Map                |
| **实现类（Implementations）** | 提供具体的数据结构     | ArrayList、HashMap、TreeSet               |
| **算法（Algorithms）**        | 通过工具类实现操作逻辑 | Collections.sort()、Collections.shuffle() |

------

### 💡 示例：统一算法操作不同实现类

```java
List<Integer> nums = new ArrayList<>(List.of(3, 1, 2));
Collections.sort(nums);   // ArrayList 排序

Set<Integer> set = new TreeSet<>(nums);
Collections.reverse(nums); // 同样适用于 List
```

> 同一个算法可作用于不同集合实现类 → **统一接口设计的强大威力！**

## 🧠 2.1 **Collection 总览**

### 📘 基本功能

Java 中的 **Collection** 接口是所有集合类的父接口，定义了常见的集合操作方法，包括：

- **增**：`add(E e)` / `addAll(Collection<? extends E> c)`
- **删**：`remove(Object o)` / `clear()`
- **改**：`replaceAll(UnaryOperator<E> operator)`
- **查**：`contains(Object o)` / `containsAll(Collection<?> c)` / `size()`
- **迭代**：`iterator()` / `forEach(Consumer<? super E> action)`
- **批量操作**：`retainAll(Collection<?> c)` / `removeAll(Collection<?> c)`

------

### 🧩 **Collection 常见子接口**

1. **List**：有序，可重复，支持按索引访问元素
2. **Set**：无序，不可重复
3. **Queue**：先进先出（FIFO）数据结构
4. **Deque**：双端队列，支持两端插入和删除元素

------

## 🧳 2.2 **List 接口与实现类**

### 📌 **1. ArrayList**

#### 🧠 底层结构

- `ArrayList` 是基于动态数组实现的集合。
- 它的底层是一个对象数组，通过 `Object[] elementData` 存储元素。

#### 📈 扩容机制与时间复杂度

- 默认初始容量为 10，当元素个数超过容量时，`ArrayList` 会扩容，扩容为原数组大小的 1.5 倍。
- 扩容的时间复杂度是 **O(n)**，因为需要创建新的数组并将旧数组元素复制过来。

#### 🔒 线程安全问题

- `ArrayList` 不是线程安全的。
- 如果需要在多线程环境下使用线程安全的列表，可以使用 `CopyOnWriteArrayList`，它在每次修改时都会复制一份新的列表。

#### 🏗️ 示例代码

```java
List<String> list = new ArrayList<>();
list.add("Java");
list.add("Python");
System.out.println(list.get(0)); // 输出: Java
```

------

### 📌 **2. LinkedList**

#### 🧠 底层结构

- `LinkedList` 是基于双向链表实现的集合，每个元素包含前后指针。
- 提供 O(1) 时间复杂度的元素插入和删除操作。

#### ⚙️ 实现队列与栈的功能

- `LinkedList` 实现了 **Queue** 和 **Deque** 接口，可以作为队列或双端队列使用。

#### 🏗️ 示例代码

```java
Queue<String> queue = new LinkedList<>();
queue.add("A");
queue.add("B");
System.out.println(queue.poll()); // 输出: A

Deque<String> deque = new LinkedList<>();
deque.addFirst("C");
deque.addLast("D");
System.out.println(deque.removeFirst()); // 输出: C
```

------

### 📌 **3. Vector（旧）**

#### 🧠 底层结构

- `Vector` 类似于 `ArrayList`，底层也是数组。
- 不同之处在于，`Vector` 是 **同步的**，适用于线程安全的场景。

#### ⚡ 性能缺陷

- 由于同步开销，`Vector` 的性能较 `ArrayList` 差，且不再推荐使用。

#### 🏗️ 示例代码

```java
Vector<String> vector = new Vector<>();
vector.add("Old");
vector.add("Vector");
System.out.println(vector.get(0)); // 输出: Old
```

> 🚨 **注意**：建议使用 `ArrayList` 代替 `Vector`，如果需要线程安全，可以考虑使用 `CopyOnWriteArrayList`。

------

## 🧳 2.3 **Set 接口与实现类**

### 📌 **1. HashSet**

#### 🧠 底层结构

- `HashSet` 基于 `HashMap` 实现，存储元素时通过计算元素的 **hashCode** 决定元素位置。
- `HashSet` 不保证元素的顺序。

#### ⚙️ HashCode 与 equals 的关系

- 插入元素时，`HashSet` 会根据 **hashCode** 来定位桶，如果发生哈希冲突，则通过 **equals** 方法进一步比较元素是否相等。

#### 🏗️ 示例代码

```java
Set<String> set = new HashSet<>();
set.add("Java");
set.add("Python");
set.add("Java"); // 重复元素不会加入
System.out.println(set); // 输出: [Java, Python]
```

------

### 📌 **2. LinkedHashSet**

#### 🧠 底层结构

- `LinkedHashSet` 基于 `HashMap` 实现，除了存储元素外，还维护一个双向链表，用于保持插入顺序。
- 保证插入元素的顺序。

#### ⚙️ 应用场景：LRU 缓存

- 因为元素是按插入顺序存储的，`LinkedHashSet` 可以用于实现 LRU 缓存。

#### 🏗️ 示例代码

```java
Set<String> linkedSet = new LinkedHashSet<>();
linkedSet.add("A");
linkedSet.add("B");
linkedSet.add("C");
System.out.println(linkedSet); // 输出: [A, B, C]
```

------

### 📌 **3. TreeSet**

#### 🧠 底层结构

- `TreeSet` 基于 **红黑树** 实现，元素是 **有序** 的。
- 支持 **自然顺序** 或 **Comparator** 排序。

#### ⚙️ 可排序与自定义 Comparator

- 可以通过 `Comparator` 自定义排序规则。

#### 🏗️ 示例代码

```java
Set<Integer> treeSet = new TreeSet<>();
treeSet.add(5);
treeSet.add(1);
treeSet.add(3);
System.out.println(treeSet); // 输出: [1, 3, 5]
```

------

## 🧳 2.4 **Queue / Deque 接口与实现类**

### 📌 **1. Queue 与优先级队列 PriorityQueue**

#### 🧠 底层结构

- `Queue` 是先进先出（FIFO）数据结构，用于存储等待处理的元素。
- `PriorityQueue` 是一个 **基于堆** 的优先队列，按照元素的自然顺序或自定义比较器顺序排序。

#### 🏗️ 示例代码

```java
Queue<Integer> priorityQueue = new PriorityQueue<>();
priorityQueue.add(3);
priorityQueue.add(1);
priorityQueue.add(2);
System.out.println(priorityQueue.poll()); // 输出: 1（优先级最低的）
```

------

### 📌 **2. Deque 与 ArrayDeque**

#### 🧠 底层结构

- `Deque` 是双端队列，支持从两端插入和删除元素。
- `ArrayDeque` 是基于数组实现的 **可扩展** 双端队列。

#### 🏗️ 示例代码

```java
Deque<String> deque = new ArrayDeque<>();
deque.addFirst("A");
deque.addLast("B");
System.out.println(deque.removeFirst()); // 输出: A
System.out.println(deque.removeLast()); // 输出: B
```

------

### 📌 **3. BlockingQueue（并发包中的阻塞队列）**

#### 🧠 主要实现类

1. **LinkedBlockingQueue**：一个基于链表的阻塞队列，支持容量限制。
2. **ArrayBlockingQueue**：一个基于数组的阻塞队列，支持容量限制。
3. **DelayQueue**：支持延迟任务的队列，元素在到期前不可消费。
4. **PriorityBlockingQueue**：优先级队列，按照元素的优先级排序。

#### 🏗️ 示例代码

```java
BlockingQueue<Integer> queue = new LinkedBlockingQueue<>();
queue.put(1);  // 阻塞方法，直到队列有空间
System.out.println(queue.take());  // 阻塞方法，直到队列有元素
```

------

## 📚 **小结**

- **List**：有序，可重复，支持按索引访问的集合。典型实现：`ArrayList`、`LinkedList`。
- **Set**：无序，不可重复，元素不能有重复。典型实现：`HashSet`、`TreeSet`。
- **Queue**：先进先出，支持任务排队。典型实现：`PriorityQueue`、`LinkedList`。
- **Deque**：双端队列，支持两端操作。典型实现：`ArrayDeque`、`LinkedList`。

# 🧩 **第3章：Map 接口体系**

> **目标：**
>  通过学习本章内容，掌握 Java 中 **Map 接口及其实现类**，理解其底层结构、扩容机制、性能特点、线程安全及应用场景。

------

## 🧠 1. **Map 接口概览**

### 📘 **键值对存储**

- `Map` 是一种 **键值对** 数据结构，其中每个元素由 **键（key）** 和 **值（value）** 组成，且 **key** 是唯一的。
- 常见应用：缓存、字典、计数器等。

### 📈 **常见操作与遍历方式**

1. **增**：`put(K key, V value)`
2. **查**：`get(Object key)`
3. **删**：`remove(Object key)`
4. **修改**：`replace(K key, V oldValue, V newValue)`
5. **包含检查**：`containsKey(Object key)` / `containsValue(Object value)`
6. **遍历方式**：
   - `entrySet()`：返回 `Set<Map.Entry<K,V>>`，用于遍历键值对。
   - `keySet()`：返回 `Set<K>`，用于遍历键。
   - `values()`：返回 `Collection<V>`，用于遍历值。

#### 🏗️ 示例代码

```java
Map<String, Integer> map = new HashMap<>();
map.put("Apple", 1);
map.put("Banana", 2);
map.put("Cherry", 3);

System.out.println(map.get("Apple")); // 输出: 1
map.remove("Banana");
for (Map.Entry<String, Integer> entry : map.entrySet()) {
    System.out.println(entry.getKey() + ": " + entry.getValue());
}
```

------

## 🧳 2. **HashMap**

### 📘 **底层数据结构：数组 + 链表 + 红黑树**

- `HashMap` 底层由 **数组** 和 **链表** 组成。当发生哈希冲突时，多个元素会被存储在同一个数组位置的 **链表** 中。
- 如果链表长度超过一定阈值（默认为 8），则会 **树化**，转为红黑树，以提升查找效率。

### ⚡ **扩容机制与负载因子**

- 默认负载因子：0.75，当当前容量达到 `threshold = capacity * loadFactor` 时，`HashMap` 会扩容，通常扩容为原来的两倍。
- 扩容操作是 **O(n)**，因为需要将原数组中的元素重新计算哈希并复制到新数组。

### 🔄 **hash 冲突与链表树化**

- 当多个元素的哈希值相同（即哈希冲突），`HashMap` 会将它们存储在同一个桶（bucket）中，使用 **链表** 或 **红黑树** 处理冲突。
- 树化机制：当冲突的链表长度超过阈值（默认为 8），链表会转换为红黑树，时间复杂度降低为 **O(log n)**。

#### 🏗️ 示例代码

```java
Map<String, Integer> hashMap = new HashMap<>();
hashMap.put("Java", 10);
hashMap.put("Python", 20);
hashMap.put("C", 30);
System.out.println(hashMap.get("Java"));  // 输出: 10
```

------

## 🧳 3. **LinkedHashMap**

### 📘 **访问顺序与 LRU 缓存实现**

- `LinkedHashMap` 继承自 `HashMap`，并额外维护了一个 **双向链表**，以保证插入元素的顺序或访问顺序（按 `accessOrder`）。
- 默认按 **插入顺序** 排列元素，但可以通过设置 `accessOrder = true` 来按访问顺序排列元素。

### ⚡ **应用场景：LRU 缓存**

- `LinkedHashMap` 可以通过重写 `removeEldestEntry()` 方法实现 **LRU（Least Recently Used）** 缓存，即在缓存超过一定容量时，自动移除最少使用的元素。

#### 🏗️ 示例代码

```java
Map<String, Integer> linkedHashMap = new LinkedHashMap<>(16, 0.75f, true);
linkedHashMap.put("Java", 10);
linkedHashMap.put("Python", 20);
linkedHashMap.put("C", 30);
linkedHashMap.put("JavaScript", 40);

// 访问某个元素
linkedHashMap.get("Java");
linkedHashMap.put("Ruby", 50);

// 删除最旧的元素
linkedHashMap.entrySet().removeIf(entry -> entry.getValue() == 10); // 通过条件删除
System.out.println(linkedHashMap);
```

------

## 🧳 4. **TreeMap**

### 📘 **基于红黑树实现的有序 Map**

- `TreeMap` 是 **有序的** `Map` 实现，基于 **红黑树** 数据结构。
- 默认情况下，`TreeMap` 按照 **键的自然顺序** 排序，或者可以通过传入自定义的 `Comparator` 实现排序。

### ⚙️ **自定义排序规则**

- 使用 `Comparator` 允许对 `TreeMap` 进行自定义排序，例如按 **降序** 排序。

#### 🏗️ 示例代码

```java
Map<Integer, String> treeMap = new TreeMap<>();
treeMap.put(1, "Java");
treeMap.put(3, "Python");
treeMap.put(2, "C");

System.out.println(treeMap); // 输出: {1=Java, 2=C, 3=Python}

// 自定义排序：按键降序
Map<Integer, String> descendingMap = new TreeMap<>((a, b) -> b - a);
descendingMap.put(1, "Java");
descendingMap.put(3, "Python");
descendingMap.put(2, "C");

System.out.println(descendingMap); // 输出: {3=Python, 2=C, 1=Java}
```

------

## 🧳 5. **Hashtable 与 ConcurrentHashMap**

### 📘 **同步机制比较**

1. **Hashtable**：
   - `Hashtable` 是 **线程安全** 的 `Map`，但其所有方法都被 `synchronized` 修饰，导致性能较差。
   - 不推荐使用，通常使用 `ConcurrentHashMap` 作为替代。
2. **ConcurrentHashMap**：
   - `ConcurrentHashMap` 提供了更高效的线程安全机制，通过 **分段锁**（Segment）来减少锁的竞争。
   - 在读操作时，**读操作无需加锁**，只有在写操作时才进行锁定。

### ⚡ **分段锁与 CAS 优化**

- `ConcurrentHashMap` 使用 **分段锁** 技术，将数据分成多个段，每个段独立加锁，保证高并发情况下的性能。
- 通过 **CAS（Compare and Swap）** 实现非阻塞更新操作，减少了传统锁机制的性能开销。

#### 🏗️ 示例代码

```java
Map<String, Integer> concurrentMap = new ConcurrentHashMap<>();
concurrentMap.put("Java", 10);
concurrentMap.put("Python", 20);

System.out.println(concurrentMap.get("Java"));  // 输出: 10
```

> 🚨 **注意**：对于多线程高并发环境，推荐使用 `ConcurrentHashMap`，而不推荐使用 `Hashtable`。

------

## 📚 **小结**

| 类别                  | 特点                                                   | 典型实现类          |
| --------------------- | ------------------------------------------------------ | ------------------- |
| **HashMap**           | 无序，键值对存储，非线程安全，底层数组 + 链表 + 红黑树 | `HashMap`           |
| **LinkedHashMap**     | 有序，支持插入顺序或访问顺序，支持 LRU 缓存            | `LinkedHashMap`     |
| **TreeMap**           | 有序，基于红黑树，支持自定义排序                       | `TreeMap`           |
| **Hashtable**         | 线程安全，性能较差，已过时                             | `Hashtable`         |
| **ConcurrentHashMap** | 高效线程安全，分段锁 + CAS 优化                        | `ConcurrentHashMap` |

------

## 📚 延伸阅读与思考

- **HashMap 和 ConcurrentHashMap 的区别**：分别适用于单线程和多线程场景。
- **TreeMap 与 HashMap 的选择**：如何根据需求选择有序与无序的映射实现类。

# 🧩 **第4章：迭代器与遍历机制**

> **目标：**
>  理解 Java 中集合的遍历机制，掌握 `Iterator`、`ListIterator`、`foreach` 等遍历方式的使用，了解并解决 `ConcurrentModificationException` 错误，学习如何在多线程环境下安全地遍历集合。

------

## 🧠 1. **Iterator 接口与 fail-fast 原理**

### 📘 **Iterator 接口概览**

- `Iterator` 是 Java 集合框架中用于 **遍历** 集合的标准接口。
- 它提供了三种基本方法：
  - `boolean hasNext()`：判断是否还有下一个元素。
  - `E next()`：返回下一个元素。
  - `void remove()`：删除当前元素（可选操作）。

### 🔐 **fail-fast 原理**

- **fail-fast** 是一种机制，当集合在遍历过程中被结构性修改（例如元素被添加或删除）时，`Iterator` 会在下一次操作时抛出 **`ConcurrentModificationException`** 异常。***与其让程序在不确定状态下继续运行，不如立即失败***
- 该机制的目的是防止遍历过程中发生不一致的修改，确保遍历和修改操作的安全性。

#### 🏗️ 示例代码

```java
List<String> list = new ArrayList<>();
list.add("Java");
list.add("Python");
list.add("C");

Iterator<String> iterator = list.iterator();
while (iterator.hasNext()) {
    String element = iterator.next();
    if ("Python".equals(element)) {
        list.remove(element);  // 修改集合，触发 ConcurrentModificationException
    }
}
```

在上面的代码中，`list.remove(element)` 会导致 `ConcurrentModificationException` 异常，因为我们在使用 `Iterator` 遍历时修改了集合。

------

## 🧳 2. **ListIterator 特性与双向遍历**

### 📘 **ListIterator 概览**

- `ListIterator` 是 `Iterator` 的子接口，专门用于 **`List`** 类型的集合。
- 它支持双向遍历，可以向前和向后遍历元素。
- `ListIterator` 提供了以下方法：
  - `boolean hasNext()`：判断是否还有下一个元素。
  - `E next()`：返回下一个元素。
  - `boolean hasPrevious()`：判断是否还有前一个元素。
  - `E previous()`：返回前一个元素。
  - `void add(E e)`：在当前元素前插入元素。
  - `void set(E e)`：替换当前元素。

### 🔄 **双向遍历的优势**

- `ListIterator` 支持 **双向遍历**，不仅可以遍历列表的元素，还可以在遍历过程中修改元素或进行插入操作。

#### 🏗️ 示例代码

```java
List<String> list = new ArrayList<>();
list.add("Java");
list.add("Python");
list.add("C");

ListIterator<String> iterator = list.listIterator();
while (iterator.hasNext()) {
    String element = iterator.next();
    System.out.println(element); // 向前遍历
}

while (iterator.hasPrevious()) {
    String element = iterator.previous();
    System.out.println(element); // 向后遍历
}
```

------

## 🧳 3. **foreach 与 Iterable**

### 📘 **foreach 与 Iterable 接口**

- `foreach` 是 Java 5 引入的语法糖，简化了集合的遍历操作。
- 通过 `Iterable` 接口，所有实现了 `Iterable` 接口的集合类（如 `List`、`Set`）可以使用 `foreach` 进行遍历。

#### 🏗️ 示例代码

```java
List<String> list = new ArrayList<>();
list.add("Java");
list.add("Python");
list.add("C");

// 使用 foreach 遍历集合
list.forEach(element -> System.out.println(element));
```

- `foreach` 的好处是简洁、易于理解，并且可以与 Lambda 表达式结合使用，实现灵活的遍历操作。

------

## 🧳 4. **ConcurrentModificationException 触发机制**

### 📘 **触发机制概览**

- `ConcurrentModificationException` 异常会在以下情况下被触发：
  1. 在使用 `Iterator` 遍历集合时，集合被其他线程修改（例如，删除或添加元素）。
  2. 在单线程操作时，遍历集合时集合结构被修改（例如，使用 `for-each` 遍历时修改集合）。

### ⚡ **为什么会抛出这个异常**

- Java 的集合类在遍历过程中会保存一个 **modCount** 变量，表示集合结构的修改次数。每当结构发生变化时，`modCount` 会自增。
- 当 `Iterator` 在遍历过程中发现 `modCount` 的值和它创建时保存的值不一致时，就会抛出 `ConcurrentModificationException`。
- 任何遍历过程中对原集合的结构性修改都会触发异常；与是否多线程无关，单线程自己修改也会触发
- 只有通过迭代器自身的修改方法才是安全的（`iterator.remove(); list.removeIf(item -> item.equals("B"))`）

------

## 🧳 5. **安全遍历的几种方式**

### 📘 **1. CopyOnWriteArrayList**

- `CopyOnWriteArrayList` 是一种线程安全的列表实现，适用于 **读多写少** 的场景。
- 它通过 **写时复制** 的策略来实现线程安全：每次写操作都会复制一份新的数组，这样不会影响正在进行的遍历操作。

#### 🏗️ 示例代码

```java
List<String> list = new CopyOnWriteArrayList<>();
list.add("Java");
list.add("Python");

Iterator<String> iterator = list.iterator();
while (iterator.hasNext()) {
    String element = iterator.next();
    list.add("C"); // 在遍历时安全地修改集合
}
```

- `CopyOnWriteArrayList` 适用于需要频繁遍历且修改较少的场景，例如缓存、监听器集合等。

### 📘 **2. ConcurrentHashMap**

- `ConcurrentHashMap` 是线程安全的 `Map`，它支持并发读操作，不会在读操作时加锁。对于写操作（如 `put` 或 `remove`），它通过 **分段锁**（Segment Locking）来提高并发性能。
- `ConcurrentHashMap` 通过 **CAS**（Compare and Swap）优化操作，避免了锁的竞争。

#### 🏗️ 示例代码

```java
Map<String, Integer> map = new ConcurrentHashMap<>();
map.put("Java", 10);
map.put("Python", 20);

map.forEach((key, value) -> System.out.println(key + ": " + value)); // 安全遍历
```

- `ConcurrentHashMap` 适用于多线程环境中，需要在遍历的同时执行写操作的场景。

### 📘 **3. 使用 `synchronized` 块同步集合**

- 如果不使用 `CopyOnWriteArrayList` 或 `ConcurrentHashMap`，可以使用 **同步块** 来保证线程安全。在遍历集合时，使用 `synchronized` 来控制对集合的访问。

#### 🏗️ 示例代码

```java
List<String> list = new ArrayList<>();
list.add("Java");
list.add("Python");

synchronized (list) {
    for (String element : list) {
        System.out.println(element);
    }
}
```

- `synchronized` 可以保证在遍历过程中不会发生集合结构的修改，避免触发 `ConcurrentModificationException`。

------

## 📚 **小结**

| 遍历方式                 | 说明                                       | 示例代码                                                     |
| ------------------------ | ------------------------------------------ | ------------------------------------------------------------ |
| **Iterator**             | 标准的集合遍历方式，支持 `remove()` 操作   | `Iterator<String> it = list.iterator();`                     |
| **ListIterator**         | 双向遍历 `List`，支持插入和替换元素        | `ListIterator<String> it = list.listIterator();`             |
| **foreach**              | 简化的遍历方式，支持 Lambda 表达式         | `list.forEach(e -> System.out.println(e));`                  |
| **CopyOnWriteArrayList** | 适用于读多写少的场景，写时复制保证线程安全 | `CopyOnWriteArrayList<String> list = new CopyOnWriteArrayList<>();` |
| **ConcurrentHashMap**    | 高并发场景下安全遍历 `Map`                 | `map.forEach((k, v) -> System.out.println(k + ": " + v));`   |

------

## 📚 延伸阅读与思考

- **Iterator 和 ListIterator 的选择**：在单向遍历 `List` 时使用 `Iterator`，双向遍历时使用 `ListIterator`。
- **CopyOnWriteArrayList 与 ConcurrentHashMap 的性能差异**：什么时候使用 `CopyOnWriteArrayList`，什么时候使用 `ConcurrentHashMap`？
- **多线程环境中的安全遍历**：除了 `CopyOnWriteArrayList` 和 `ConcurrentHashMap`，还可以使用 `synchronized` 和 `ReentrantLock` 来保证线程安全。

## 🧳 3. **不可变集合与只读视图**

### 📘 **不可变集合概念**

不可变集合是指集合内容在创建后不能被修改。它们是 **线程安全** 的，因此非常适合在多线程环境下使用。

### 🔧 **`Collections.unmodifiableList()`**

- 返回一个不可修改的集合视图。如果对其进行修改，抛出 `UnsupportedOperationException` 异常。

#### 🏗️ 示例代码

```java
List<String> list = new ArrayList<>();
list.add("Java");
list.add("Python");

List<String> unmodifiableList = Collections.unmodifiableList(list);
unmodifiableList.add("C");  // 会抛出 UnsupportedOperationException
```

### 🔧 **`List.of()` 与 `Set.of()`（JDK9+）**

- 从 JDK9 开始，`List.of()` 和 `Set.of()` 方法可以直接创建不可变集合。
- 这些集合是 **不可修改** 的，不能添加、删除或修改元素。

#### 🏗️ 示例代码

```java
List<String> immutableList = List.of("Java", "Python", "C");
Set<String> immutableSet = Set.of("Java", "Python", "C");

immutableList.add("JavaScript");  // 会抛出 UnsupportedOperationException
immutableSet.remove("Java");     // 会抛出 UnsupportedOperationException
```

### ⚡ **不可变集合的优点**

1. **线程安全**：由于集合内容无法改变，不存在多线程环境下的数据竞争。
2. **防止不小心修改**：不允许对集合进行任何修改，保证数据一致性。
3. **性能优化**：不可变集合通常会进行优化（如缓存）。

# 🧩 **第6章：泛型与集合类型安全**

> **目标：**
>  了解 Java 泛型的使用及其在集合中的作用，掌握类型擦除、通配符、边界限定、原始类型集合的风险等关键概念，确保集合的类型安全。

------

## 🧠 **1. 泛型在集合中的作用**

### 📘 **泛型概述**

- 泛型（Generics）是 Java 5 引入的特性，它允许在定义类、接口或方法时使用类型参数，从而使得代码更加通用和类型安全。
- 泛型的主要作用是 **提供类型安全**，在编译时就能检查出类型错误，避免了运行时类型转换错误。

### 🔧 **泛型在集合中的应用**

- 泛型使得集合可以存储特定类型的对象，确保集合中的元素类型一致，避免了强制类型转换。

#### 🏗️ 示例代码

```java
List<String> list = new ArrayList<>();
list.add("Java");
list.add("Python");
// list.add(10);  // 编译时错误，不能添加非 String 类型的元素
System.out.println(list);  // 输出: [Java, Python]
```

- 通过泛型，`List<String>` 确保了集合只能包含 `String` 类型的元素，这样就避免了类型转换异常。

------

## 🧳 **2. 类型擦除机制**

### 📘 **类型擦除概念**

- Java 泛型是通过 **类型擦除（Type Erasure）** 实现的，编译器会在编译时将泛型类型转换为原始类型。
- 在运行时，泛型不再存在，只留下原始类型的集合，所有泛型信息在编译后都会被擦除。

### ⚡ **类型擦除的具体实现**

- 例如，`List<String>` 和 `List<Integer>` 在编译后都会变成 `List`，运行时无法知道元素的实际类型。

#### 🏗️ 示例代码

```java
List<String> list1 = new ArrayList<>();
List<Integer> list2 = new ArrayList<>();
// 运行时，list1 和 list2 都是 List 类型，编译器无法区分它们
System.out.println(list1.getClass() == list2.getClass());  // 输出: true
```

- 类型擦除机制使得 **泛型代码兼容旧版本 Java**，但也导致了运行时无法获取泛型的类型信息。

------

## 🧳 **3. 通配符与边界限定**

### 📘 **通配符（Wildcard）概述**

- Java 泛型提供了通配符 `?`，表示 **未知类型**。它常用于方法中，表示可以接受不同类型的集合。

#### 🧑‍💻 **上界通配符 `? extends T`**

- `? extends T` 表示接受任何 `T` 或其子类的类型。

#### 🧑‍💻 **下界通配符 `? super T`**

- `? super T` 表示接受任何 `T` 或其父类的类型。

### ⚡ **边界限定的应用**

- 上界通配符常用于 **只读取** 集合元素的场景。
- 下界通配符常用于 **写入** 集合元素的场景。

#### 🏗️ 示例代码

```java
// 上界通配符：只读操作
public static void printList(List<? extends Number> list) {
    for (Number n : list) {
        System.out.println(n);
    }
}

// 下界通配符：写入操作
public static void addIntegers(List<? super Integer> list) {
    list.add(1);  // 只允许添加 Integer 或其子类元素
    // list.add(1.5);  // 编译时错误，不能添加 Double 类型
}
```

- 使用通配符时，尽量避免在方法中修改集合的元素，尤其是当集合的类型是 `? extends T` 时。

------

## 🧳 **4. 原始类型集合的风险**

### 📘 **原始类型集合概述**

- 在使用泛型时，如果没有指定具体的类型参数，就会导致原始类型的集合。
- 原始类型集合会导致 **类型安全** 问题，因为它会绕过泛型检查，使得元素类型不受限制。

#### 🧑‍💻 **原始类型集合的示例**

```java
List list = new ArrayList();
list.add("Java");
list.add(10);  // 添加了不同类型的元素，编译时不会报错

String str = (String) list.get(0);  // 强制类型转换，可能导致 ClassCastException
```

### ⚡ **原始类型集合的风险**

- 使用原始类型集合时，编译器无法检测到类型不匹配，可能导致运行时的类型转换异常（如 `ClassCastException`）。

### 🔒 **避免原始类型集合的建议**

- 总是使用泛型来定义集合，避免使用原始类型集合。

#### 🏗️ 示例代码

```java
List<String> list = new ArrayList<>();
list.add("Java");
// list.add(10);  // 编译时错误，类型不匹配
```

------

## 🧳 **5. 泛型数组与集合协变问题**

### 📘 **泛型数组问题**

- 在 Java 中，**泛型数组** 是不允许创建的，因为泛型的 **协变性**（子类可以替代父类）会导致类型不匹配问题。

#### 🧑‍💻 **泛型数组的问题示例**

```java
List<String>[] listArray = new List<String>[10];  // 编译错误
```

- 这是因为泛型类型的信息在运行时会被擦除，无法确定元素类型，导致无法创建泛型数组。

### ⚡ **解决方案**

- 可以使用 **`ArrayList`** 或 **`List`** 来代替泛型数组：

```java
List<List<String>> listOfLists = new ArrayList<>();
```

- 另外，还可以使用 **`Object[]`** 来创建数组，并通过类型转换来实现类型安全，但这会丧失泛型带来的类型检查优势。

------

## 📚 **小结**

| 主题                       | 关键内容                                                     |
| -------------------------- | ------------------------------------------------------------ |
| **泛型在集合中的作用**     | 提供类型安全，避免强制类型转换错误，增强代码的通用性         |
| **类型擦除机制**           | 在编译时擦除泛型信息，运行时无法获得类型参数的实际类型       |
| **通配符与边界限定**       | 上界通配符（`? extends T`）用于读取，下界通配符（`? super T`）用于写入 |
| **原始类型集合的风险**     | 使用原始类型集合会丧失类型安全，容易导致 `ClassCastException` |
| **泛型数组与集合协变问题** | 泛型数组不可创建，使用 `List<List<T>>` 替代                  |

## 🧠 **1. ArrayList 源码结构分析**

### 📘 **底层数据结构：elementData**

- `ArrayList` 是基于 **动态数组** 实现的集合，底层存储数据的数组称为 `elementData`。
- 初始化时，`ArrayList` 会创建一个长度为 10 的空数组，当元素超过当前容量时，`ArrayList` 会扩容。

#### 🧑‍💻 **构造函数与扩容机制**

```java
// 默认构造函数，初始化容量为 10
public ArrayList() {
    this(DEFAULT_CAPACITY);
}

// 指定容量构造函数
public ArrayList(int initialCapacity) {
    if (initialCapacity > 0) {
        this.elementData = new Object[initialCapacity];
    } else if (initialCapacity == 0) {
        this.elementData = EMPTY_ELEMENTDATA;
    } else {
        throw new IllegalArgumentException("Illegal Capacity: " + initialCapacity);
    }
}
```

- 默认容量为 10，`ArrayList` 会根据元素的个数动态扩容。扩容是通过 `ensureCapacity` 方法进行的。

#### 🧑‍💻 **扩容逻辑**

- 当 `ArrayList` 的容量达到 `elementData.length` 时，`ArrayList` 会进行扩容，通常扩容为原容量的 1.5 倍。

```java
private void grow(int minCapacity) {
    // 增长因子为 1.5 倍
    int oldCapacity = elementData.length;
    int newCapacity = oldCapacity + (oldCapacity >> 1);
    if (newCapacity - minCapacity < 0)
        newCapacity = minCapacity;
    elementData = Arrays.copyOf(elementData, newCapacity);
}
```

#### ⚡ **扩容的时间复杂度**

- 扩容操作是 **O(n)**，因为需要创建新的数组并将旧数组的元素复制到新数组中。

## 🧳 **2. HashMap 源码分析**

### 📘 **底层数据结构：数组 + 链表 + 红黑树**

- `HashMap` 底层是一个数组，数组的每个位置是一个 **链表** 或 **红黑树**，用于处理哈希冲突。
- `HashMap` 使用 **哈希表** 结构存储键值对，每个键值对通过计算哈希值定位到桶数组的位置。

#### 🧑‍💻 **put() 方法**

- `put()` 方法用于插入键值对，首先计算键的哈希值，然后查找对应的桶位置。如果该位置没有元素，则直接插入；如果已经有元素，则需要通过 **链表** 或 **红黑树** 处理冲突。

```java
public V put(K key, V value) {
    if (key == null)
        return putForNullKey(value);
    int hash = hash(key);
    int i = indexFor(hash, table.length);
    // 插入新元素或更新已有元素
    for (Entry<K,V> e = table[i]; e != null; e = e.next) {
        if (e.hash == hash && (e.key == key || key.equals(e.key))) {
            V oldValue = e.value;
            e.value = value;
            return oldValue;
        }
    }
    // 插入新的元素
    modCount++;
    addEntry(hash, key, value, i);
    return null;
}
```

#### ⚡ **resize() 和 treeify() 流程**

- 当 **链表** 长度超过一定阈值（默认为 8）时，`HashMap` 会将链表转换为 **红黑树**，以提高查询性能。

```java
private void treeifyBin(Node<K,V>[] tab, int hash) {
    // 将链表转换为红黑树
    // 树化操作
}
```

- `resize()` 方法用于扩容，当 `HashMap` 容量达到负载因子的阈值时，会进行扩容操作。

```java
void resize() {
    // 扩容操作
    int newCapacity = oldCapacity << 1;
    threshold = (int)(newCapacity * loadFactor);
    transfer(newTable);
}
```

## 🧳 **3. ConcurrentHashMap 源码分析**

### 📘 **高并发优化：分段锁与 CAS**

- `ConcurrentHashMap` 采用了 **分段锁（Segment）** 技术，将数据分为多个段，每个段维护独立的锁，从而提高并发性能。
- `ConcurrentHashMap` 的写操作是通过 **CAS（Compare and Swap）** 实现的，保证操作的 **原子性**，避免锁的竞争。

#### 🧑‍💻 **Segment → Node → CAS → synchronized**

```java
final Segment<K,V> segment = segments[hash & (segments.length - 1)];
synchronized (segment) {
    // CAS 操作
    segment.put(key, value);
}
```

- `ConcurrentHashMap` 使用 **分段锁** 来对不同段的数据加锁，因此可以并发修改不同段的元素，从而提升性能。

#### ⚡ **并发优化的核心**

- 通过分段锁和 CAS，`ConcurrentHashMap` 能够在高并发场景下提供良好的性能，避免了整个 `Map` 锁定的瓶颈。

## 🧳 **4. LinkedHashMap LRU 实现机制**

### 📘 **底层数据结构：双向链表**

- `LinkedHashMap` 在 `HashMap` 的基础上，增加了 **双向链表**，用于保持元素的插入顺序或访问顺序。

#### 🧑‍💻 **removeEldestEntry() 方法**

- `LinkedHashMap` 通过重写 `removeEldestEntry()` 方法实现 LRU 缓存的功能。该方法在每次插入新元素后被调用，决定是否移除最老的元素。

```java
protected boolean removeEldestEntry(Map.Entry<K,V> eldest) {
    return size() > MAX_ENTRIES;
}
```

- `removeEldestEntry()` 方法可以根据 `Map` 的大小或其他条件，自动删除最旧的元素，常用于实现 **LRU（Least Recently Used）** 缓存。

#### 🧑‍💻 **LRU 缓存示例**

```java
Map<String, Integer> lruCache = new LinkedHashMap<>(16, 0.75f, true);
lruCache.put("A", 1);
lruCache.put("B", 2);
lruCache.put("C", 3);
lruCache.get("A");  // 访问 "A"
lruCache.put("D", 4); // 自动移除 "B"（最少使用）
```

- 设置 `accessOrder` 为 `true`，`LinkedHashMap` 会按 **访问顺序** 排序元素，常用于实现 LRU 缓存。

| **LinkedHashMap**     | 基于双向链表的 `HashMap`，支持按插入顺序或访问顺序维护元素，常用于实现 LRU 缓存 | `removeEldestEntry()`, `accessOrder` |

## 📚 延伸阅读与思考

- **`HashMap` 中为什么要使用红黑树？**
   红黑树的引入可以将链表操作的时间复杂度从 **O(n)** 降低到 **O(log n)**，大大提升查询效率。
- **`ConcurrentHashMap` 的分段锁机制如何提高性能？**
   通过对数据进行分段，每个段使用独立的锁，避免了全表锁的性能瓶颈，提高了并发性能。
- **如何选择使用 `LinkedHashMap`？**
   在需要维持元素顺序（如 LRU 缓存）时，`LinkedHashMap` 是更好的选择，而普通的 `HashMap` 适用于不关心元素顺序的场景。
