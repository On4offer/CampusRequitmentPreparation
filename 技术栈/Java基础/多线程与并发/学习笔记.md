## 🧠 1. 什么是多线程与并发

### 📘 概念

- **多线程**：在同一进程中并发执行多个线程。每个线程是独立的执行路径，拥有自己的程序计数器、栈和局部变量，但**共享进程的堆内存**。线程是 **CPU 调度的基本单位**。
- **并发（Concurrency）**：多个任务在**同一时间段内**交替执行，不要求同一时刻同时执行；单核 CPU 上通过时间片轮转实现。
- **并行（Parallelism）**：多个任务在**同一时刻**同时执行，通常需要多核 CPU。

| 概念     | 说明                         |
| -------- | ---------------------------- |
| 多线程   | 同一进程内多条执行流         |
| 并发     | 时间段内交替执行             |
| 并行     | 同一时刻同时执行（多核）     |

### ⚡ 小结

- 多核 CPU 上，多线程可**并行**执行；单核上多线程是**并发**（时间片轮转）。

| 场景           | 描述                                   |
| -------------- | -------------------------------------- |
| Web 服务器    | 每请求一线程或线程池处理，高并发       |

## 🧳 3. 进程与线程

### 📘 进程模型

- 进程是**操作系统分配资源的基本单位**，有独立地址空间和资源。
- 进程间通信（IPC）较复杂；创建与销毁开销大。

### 📘 线程模型

- 线程**共享进程的地址空间和资源**，线程间通过共享内存通信简单。
- 创建与切换开销比进程小；同一进程内线程崩溃可能影响整个进程。

### ⚖️ 对比表

| 特性     | 进程                 | 线程                         |
| -------- | -------------------- | ---------------------------- |
| 定义     | 资源分配的基本单位   | CPU 调度的基本单位           |
| 内存     | 独立地址空间         | 共享进程内存                 |
| 通信     | IPC 复杂             | 共享变量即可                 |
| 创建/切换 | 开销大               | 开销小                       |

## 🧳 4. 为什么需要并发编程

| 原因           | 说明                           |
| -------------- | ------------------------------ |
| 提高资源利用率 | 多核 CPU 可被多线程充分利用    |
| 提高响应性     | 避免单线程阻塞导致界面卡死     |
| 缩短任务时间   | 大任务拆分并行，减少总耗时     |
| 应对高并发     | Web/中间件等需同时处理大量请求 |

**挑战**：线程安全、死锁、竞态、可见性等问题，需同步与设计规范。

## 🧠 1. 线程的生命周期与状态

### 📘 Java 线程状态（Thread.State）

| 状态            | 说明                                           |
| --------------- | ---------------------------------------------- |
| **NEW**         | 已创建未调用 `start()`                         |
| **RUNNABLE**    | 可运行（就绪或正在运行），等待或占用 CPU       |
| **BLOCKED**     | 等待获取**监视器锁**（如等进入 synchronized）  |
| **WAITING**     | 无限期等待，如 `wait()`、`join()` 无参         |
| **TIMED_WAITING** | 限时等待，如 `sleep()`、`wait(time)`、`join(time)` |
| **TERMINATED**  | 执行结束或异常退出                             |

### 📈 常见转换

- `start()` → NEW 到 RUNNABLE。
- 获取不到锁 → RUNNABLE 到 BLOCKED；获取到锁 → BLOCKED 到 RUNNABLE。
- `wait()` → WAITING；`notify()`/`notifyAll()` → 回到 RUNNABLE（需重新争锁）。
- `run()` 结束或异常 → TERMINATED。

## 🧳 2. 创建线程的方式

### 📘 （1）继承 Thread 类

- 重写 `run()`，`new MyThread().start()` 启动。
- **缺点**：Java 单继承，无法再继承其他类。

#### 🧑‍💻 示例代码

```java
class MyThread extends Thread {
    @Override
    public void run() {
        System.out.println(Thread.currentThread().getName() + " running");
    }
}
// 使用：new MyThread().start();
```

### 📘 （2）实现 Runnable 接口

- 实现 `run()`，将 `Runnable` 传给 `Thread` 构造器再 `start()`。
- **优点**：可多实现接口，任务与线程解耦，便于线程池等复用。

#### 🧑‍💻 示例代码

```java
Runnable task = () -> System.out.println(Thread.currentThread().getName());
new Thread(task).start();
```

### 📘 （3）使用 Callable + Future

- `Callable<V>` 带返回值、可抛异常；通过 `ExecutorService.submit(callable)` 提交，返回 `Future<V>`，用 `get()` 获取结果（阻塞）。

#### 🧑‍💻 示例代码

```java
ExecutorService es = Executors.newSingleThreadExecutor();
Future<String> f = es.submit(() -> "result");
String s = f.get();  // 阻塞直到完成
es.shutdown();
```

> 💡 **面试**：推荐实现 **Runnable** 或 **Callable**，便于与线程池、CompletableFuture 等结合；继承 Thread 不利于扩展。

------

## 🧳 3. start() 与 run() 的区别

| 方法      | 作用                         |
| --------- | ---------------------------- |
| **start()** | 启动新线程，由系统调度执行 `run()` |
| **run()**   | 线程体；直接调用 `run()` 只在当前线程执行，不会启动新线程 |

------

## 🧳 4. sleep() 与 yield()

| 方法                | 作用                     | 是否释放锁 |
| ------------------- | ------------------------ | ---------- |
| **sleep(ms)**       | 当前线程暂停指定时间     | 不释放     |
| **yield()**         | 提示调度器让出 CPU，不保证一定让出 | 不释放     |

------

## 🧳 5. 线程的中断与安全停止

### 📘 interrupt()

- `interrupt()` 设置线程的**中断标志**，不会立刻终止线程。
- 若线程在 `sleep()`、`wait()`、`join()` 等阻塞中，会抛出 `InterruptedException` 并清除中断标志。
- 线程内可通过 `Thread.interrupted()` 或 `currentThread().isInterrupted()` 检查标志并退出循环。

### 📘 安全停止（非阻塞任务）

- 用 **volatile boolean** 或 **AtomicBoolean** 作为“停止标志”，在循环中判断后退出。

#### 🧑‍💻 示例代码

```java
volatile boolean running = true;
public void stop() { running = false; }
@Override
public void run() {
    while (running) {
        // 做任务
    }
}
```

## 🧠 1. 线程安全的概念

### 📘 定义

- **线程安全**：多线程并发访问时，程序行为正确，数据一致、无竞态。
- 常见**不安全**表现：共享变量被多线程同时读写导致结果不确定（如 i++ 非原子）。

### ⚡ 实现方式概览

| 方式           | 说明                     |
| -------------- | ------------------------ |
| 互斥同步       | synchronized、Lock       |
| 非阻塞         | CAS、原子类              |
| 不可变         | final、不可变对象        |
| 线程封闭       | 局部变量、ThreadLocal    |

------

## 🧳 2. synchronized 关键字

### 📘 用法

- **同步实例方法**：锁对象为 **this**，同一实例串行执行。
- **同步静态方法**：锁对象为 **当前类的 Class**，所有实例共享一把类锁。
- **同步块**：`synchronized (obj) { ... }`，锁对象为 **obj**。

### 📘 特点

- **可重入**：同一线程可多次获取同一把锁（锁计数器 +1/-1）。
- **不可中断**：等待锁时一般不响应 interrupt（Lock 可中断）。
- **非公平**：不保证先等待的线程先获得锁（ReentrantLock 可选公平）。

#### 🧑‍💻 示例代码

```java
// 同步方法
public synchronized void increment() { count++; }

// 同步块：缩小锁范围，提高并发
public void add(int v) {
    synchronized (this) {
        count += v;
    }
}
```

### ⚡ 同步方法 vs 同步块

| 对比     | 同步方法       | 同步块           |
| -------- | -------------- | ---------------- |
| 锁范围   | 整个方法       | 仅代码块         |
| 灵活性   | 低             | 高（可锁任意对象） |
| 性能     | 易锁粒度过大   | 可细粒度         |

------

## 🧳 3. volatile 关键字

### 📘 作用

- **可见性**：写 volatile 变量后，其他线程能立即看到最新值（禁止线程本地缓存）。
- **有序性**：禁止指令重排序（单线程语义不变前提下，多线程下避免重排导致问题）。
- **不保证原子性**：如 `i++` 仍不是原子操作，多线程下仍需 synchronized 或原子类。

### 📘 适用场景

- 状态标志位（如 `volatile boolean flag`），一写多读。
- 不适用于“读-改-写”复合操作（如计数），需用锁或原子类。

#### 🧑‍💻 示例代码

```java
private volatile boolean running = true;
public void stop() { running = false; }
public void run() {
    while (running) { /* 工作 */ }
}
```

------

## 🧳 4. wait() 与 notify() / notifyAll()

### 📘 要点

- 必须在 **synchronized** 块内调用；否则抛出 `IllegalMonitorStateException`。（ReentrantLock 等显式锁 → 必须用 `Condition`）
- **wait()**：释放锁，当前线程进入该对象的等待集，直到被 notify 或中断。（`wait()` 释放的是 synchronized 内置监视器锁）
- **notify()**：唤醒该对象等待集上的**一个**线程（不指定哪个）；**notifyAll()** 唤醒全部。
- 被唤醒后线程需**重新竞争锁**，获得锁后才从 `wait()` 返回。

### 📘 模板（生产者-消费者）

```java
synchronized (lock) {
    while (!条件满足) {  // 必须用 while，防止虚假唤醒
        lock.wait();
    }
    // 业务
    lock.notifyAll();  // 或 notify()
}
```

## 🧳 5. 死锁与预防

### 📘 死锁四个必要条件

1. **互斥**：资源只能被一个线程占用。
2. **占有且等待**：持有至少一个资源并等待其他资源。
3. **不可剥夺**：资源不能被强制剥夺。
4. **循环等待**：存在线程等待环，如 A 等 B，B 等 A。

### ⚡ 预防思路

- **固定加锁顺序**：所有线程按同一顺序获取多把锁（如先 A 后 B），避免循环等待。
- **tryLock**：使用 `ReentrantLock.tryLock()`，获取不到则释放已持有锁并重试或退避。
- **超时**：`tryLock(timeout)`，避免无限等待。

#### 🧑‍💻 示例代码（tryLock 避免死锁）

```java
if (lockA.tryLock()) {
    try {
        if (lockB.tryLock()) {
            try {
                // 业务
            } finally { lockB.unlock(); }
        }
    } finally { lockA.unlock(); }
}
```

| synchronized 和 volatile 区别？ | synchronized 保证可见性+原子性+有序性；volatile 只保证可见性+有序性，不保证复合操作原子性。 |

### ⚠️ 常见坑与注意点

| 现象 / 易错点 | 原因 | 正确做法 / 怎么记 |
|---------------|------|-------------------|
| 锁错了对象（如锁方法却锁在 null 或不同实例） | 不同线程拿到不同锁，起不到互斥作用 | 多线程要互斥必须锁**同一个对象**；实例方法锁 this，静态方法锁 Class；同步块明确锁同一 lock 对象。 |
| 用 if 判断条件再 wait() | 虚假唤醒时条件可能又不满足，直接往下执行会出错 | 必须用 **while(条件)** 包住 wait()，被唤醒后再次检查条件。 |
| 以为 volatile 能保证 i++ 线程安全 | i++ 是读-改-写三步，volatile 只保证单次读写可见 | 计数、复合操作用 synchronized 或原子类（AtomicInteger）；volatile 只用于状态标志等一写多读。 |
| 多把锁加锁顺序不一致 | 易形成循环等待，死锁 | 所有线程按**同一顺序**获取多把锁（如先 A 后 B）；或使用 tryLock 失败则释放已持有的锁。 |
| synchronized 锁在包装类型或 String 常量 | 包装类型缓存、字符串常量池可能导致多段代码误用同一锁 | 锁对象用 **private final Object lock = new Object()** 或专用锁实例，避免与别人锁到同一对象。 |

# 🧩 第四章：Lock 与 ReentrantLock

> **目标：** 掌握 Lock 接口与 ReentrantLock 的用法，理解与 synchronized 的差异及 AQS 简要原理。

> **本章在整体中解决什么问题**：第三章讲了 synchronized；本章讲 **Lock 接口与 ReentrantLock**——可中断、tryLock、公平锁、多 Condition，以及底层 AQS 简要。学完本章后**第五章**讲并发集合与原子类；面试常对比「synchronized 和 Lock 区别」。

------

## 🧠 1. Lock 接口

### 📘 常用方法

| 方法          | 说明                     |
| ------------- | ------------------------ |
| `lock()`      | 获取锁，阻塞直到获得     |
| `unlock()`    | 释放锁，必须在 finally 中调用 |
| `tryLock()`   | 尝试获取，不阻塞，返回是否成功 |
| `tryLock(time)` | 在指定时间内尝试获取     |
| `lockInterruptibly()` | 可响应中断的获取方式   |

### 📘 使用模板

```java
Lock lock = new ReentrantLock();
lock.lock();
try {
    // 临界区
} finally {
    lock.unlock();  // 必须释放
}
```

------

## 🧳 2. ReentrantLock 与 synchronized 对比

| 对比项     | synchronized              | ReentrantLock                    |
| ---------- | ------------------------- | -------------------------------- |
| 使用方式   | 关键字，自动加锁/解锁     | API，需手动 lock/unlock          |
| 公平锁     | 非公平                    | 可选公平/非公平构造              |
| 可中断     | 等待锁时一般不响应中断   | lockInterruptibly() 可响应       |
| 尝试获取   | 无                        | tryLock() / tryLock(time)        |
| 条件队列   | 单条件（wait/notify）     | 多条件（Condition）              |
| 底层       | JVM 管程（monitor）       | AQS（AbstractQueuedSynchronizer）|

------

## 🧳 3. Condition 条件变量

- `Lock.newCondition()` 得到 **Condition**，替代 wait/notify；可多个条件，语义更清晰。
- `await()` / `signal()` / `signalAll()` 对应 `wait()` / `notify()` / `notifyAll()`，同样必须在持有锁时调用。

#### 🧑‍💻 示例代码

```java
Lock lock = new ReentrantLock();
Condition notFull = lock.newCondition();
Condition notEmpty = lock.newCondition();
// 生产者：notFull.await(); notEmpty.signal();
// 消费者：notEmpty.await(); notFull.signal();
```

------

## 🧳 4. AQS 简述（面试向）

- **AQS**：AbstractQueuedSynchronizer，是很多同步器的基础（ReentrantLock、Semaphore、CountDownLatch、ReentrantReadWriteLock 等）。
- 内部维护 **state** 和 **FIFO 队列**；子类通过 CAS 改 state，失败则入队阻塞，释放时唤醒后继。
- ReentrantLock：state 表示重入次数；独占模式，公平/非公平由入队与抢锁策略决定。

> 💡 校招常问“ReentrantLock 和 synchronized 区别”“AQS 是什么”，能说出上表 + “基于 AQS、state+队列”即可。

### ⚠️ 常见坑与注意点

| 现象 / 易错点 | 原因 | 正确做法 / 怎么记 |
|---------------|------|-------------------|
| lock() 后未在 finally 中 unlock() | 异常跳出时锁不释放，导致死锁或后续永远拿不到锁 | **必须在 finally 里调用 unlock()**，且只释放当前线程持有的锁。 |
| 在未持有锁时调用 unlock() | 会抛 IllegalMonitorStateException | 只有当前线程 hold 住该锁时才能 unlock；用 tryLock 时若未获取成功不要 unlock。 |

| Condition 的 await/signal 未在持有锁时调用 | 会抛 IllegalMonitorStateException | 与 wait/notify 一样，必须在 **lock.lock()** 与 **lock.unlock()** 之间调用 await/signal。 |

# 🧩 第五章：并发集合与原子类

> **目标：** 掌握常用并发容器（ConcurrentHashMap、CopyOnWriteArrayList、BlockingQueue）与原子类（AtomicInteger 等）及 CAS 概念。

> **本章在整体中解决什么问题**：第三、四章讲的是「锁」；本章讲**无锁或细粒度锁**的并发容器与原子类，避免处处 synchronized。学完本章后**第六章**讲线程池（任务提交与队列）；面试常问 ConcurrentHashMap、CAS、ABA 问题。

------

## 🧠 1. 并发集合概览

| 类                    | 说明                           |
| --------------------- | ------------------------------ |
| **ConcurrentHashMap** | 线程安全 HashMap，分段/CAS+syn  |
| **CopyOnWriteArrayList** | 写时复制，读多写少场景       |
| **BlockingQueue**     | 阻塞队列，支持 put/take 阻塞   |
| **ConcurrentLinkedQueue** | 非阻塞并发队列，CAS 实现   |

------

## 🧳 2. ConcurrentHashMap

### 📘 要点

- JDK 8 前：分段锁（Segment）；JDK 8：**数组 + 链表 + 红黑树**，锁**桶头节点**（synchronized）+ CAS，粒度更细。
- 读多无锁（volatile 读），写时锁单桶或 CAS 插入空桶。
- 不支持 null key/value（与 HashMap 设计区分，避免二义性）。

### ⚡ 与 Hashtable 对比

- Hashtable 全表 syn，并发差；ConcurrentHashMap 锁粒度小，高并发下性能好。

------

## 🧳 3. CopyOnWriteArrayList

### 📘 原理

- **写时复制**：写操作时复制新数组，在新数组上修改，再替换引用；读无锁。
- **适用**：读多写少（如监听器列表）；写多时复制成本高，不适用。

------

## 🧳 4. BlockingQueue 与常用实现

| 实现类                | 特点                         |
| --------------------- | ---------------------------- |
| **ArrayBlockingQueue**  | 有界，数组，一把锁           |
| **LinkedBlockingQueue** | 可选有界，链表，两把锁（put/take 可并行） |
| **LinkedTransferQueue** | 可 transfer（等消费）       |
| **SynchronousQueue**  | 无容量，直接交付             |

- **put(e)**：满则阻塞；**take()**：空则阻塞。常用于**生产者-消费者**。

------

## 🧳 5. 原子类与 CAS

### 📘 原子类（java.util.concurrent.atomic）

- **AtomicInteger**、**AtomicLong**、**AtomicReference** 等，提供 `get`、`set`、`getAndIncrement`、`compareAndSet` 等**无锁**更新。
- 底层依赖 **CAS**（Compare-And-Swap）：比较并交换，CPU 指令级支持，保证单变量读-改-写原子性。

### 📘 CAS 简述

- 三个操作数：内存位置 V、期望值 A、新值 B。仅当 V 当前等于 A 时才把 V 更新为 B，否则不更新；返回是否成功。
- **ABA 问题**：值从 A 变 B 再变回 A，CAS 无法感知中间变化。可用 **AtomicStampedReference**（带版本号）缓解。

#### 🧑‍💻 示例代码

```java
AtomicInteger count = new AtomicInteger(0);
count.incrementAndGet();
count.getAndAdd(10);
```

# 🧩 第六章：线程池

> **目标：** 理解线程池的作用、核心参数、常见类型、拒绝策略及使用建议。

> **本章在整体中解决什么问题**：前面讲的是单线程与同步；本章讲**如何管理大量任务与线程**——线程池参数、执行流程、拒绝策略。校招常问核心线程数怎么设、为什么不建议 Executors；学完本章后**第八章**会强调「用线程池代替裸 new Thread」等实践。

------

## 🧠 1. 为什么使用线程池

- **降低开销**：复用线程，避免频繁创建/销毁。
- **控制并发**：限制同时运行的线程数，防止资源耗尽。
- **管理任务**：队列排队、定时/周期任务（ScheduledExecutorService）。

------

## 🧳 2. ThreadPoolExecutor 核心参数

| 参数             | 含义                         |
| ---------------- | ---------------------------- |
| **corePoolSize** | 核心线程数，常驻（可配置允许超时回收） |
| **maximumPoolSize** | 最大线程数                 |
| **keepAliveTime**   | 非核心线程空闲存活时间     |
| **unit**            | 时间单位                   |
| **workQueue**       | 任务队列（有界/无界）      |
| **threadFactory**   | 创建线程的工厂             |
| **handler**         | 拒绝策略（队列满且线程满时） |

### 📘 执行流程简述

1. 任务提交后，若运行线程数 < corePoolSize，新建核心线程执行。
2. 否则放入 workQueue。
3. 若队列满且运行线程数 < maximumPoolSize，新建非核心线程执行。
4. 若队列满且线程数已达 maximum，执行 **拒绝策略**。

------

## 🧳 3. 常见线程池类型（Executors）

| 方法                         | 实现特点                         | 注意                     |
| ---------------------------- | -------------------------------- | ------------------------ |
| **newFixedThreadPool(n)**    | 固定 n 线程，无界队列            | 队列无界可能 OOM         |
| **newCachedThreadPool()**    | 无核心，可扩容，60s 回收         | 任务多时线程数可能很大   |
| **newSingleThreadExecutor()** | 单线程，无界队列                | 顺序执行                 |
| **newScheduledThreadPool(n)** | 可延迟/周期执行                 | 定时任务                 |

> ⚠️ **阿里规范**：不建议直接使用 `Executors`，推荐 **ThreadPoolExecutor** 显式指定队列大小和拒绝策略，避免无界队列导致 OOM。

------

## 🧳 4. 拒绝策略

| 策略                  | 行为                           |
| --------------------- | ------------------------------ |
| **AbortPolicy**（默认） | 抛 RejectedExecutionException  |
| **CallerRunsPolicy**  | 由提交任务的线程执行           |
| **DiscardPolicy**     | 静默丢弃                       |
| **DiscardOldestPolicy** | 丢弃队列最老任务，再提交当前 |

------

## 🧳 5. 使用建议

- 根据任务类型设 **corePoolSize**（CPU 密集 ≈ 核数，IO 密集可更大）。
- 使用**有界队列**并指定**拒绝策略**，避免无界队列与默认 Abort 导致问题。
- 关闭时：**shutdown()** 不再接受新任务，**shutdownNow()** 尝试中断正在执行的任务；可用 **awaitTermination** 等待结束。

### ❌ 错误配置 vs ✅ 正确配置（面试常问）

| 场景 | 错误做法 | 正确做法 |
|------|----------|----------|
| 高吞吐任务、任务量不可控 | `Executors.newFixedThreadPool(10)` 或 `newCachedThreadPool()` | 使用 **ThreadPoolExecutor** 构造，指定**有界队列**（如 `new ArrayBlockingQueue<>(200)`）和**拒绝策略**（如 CallerRunsPolicy 或自定义），避免无界队列 OOM、Cached 线程数爆炸。 |
| 核心线程数 | 拍脑袋写 10、100 | CPU 密集约 **CPU 核数**或核数+1；IO 密集可 **核数 * 2** 或更大，结合压测；最大线程数按队列与拒绝策略设上限。 |
| 拒绝策略 | 不设置，默认 Abort 直接抛异常丢任务 | 根据业务选择：**CallerRunsPolicy** 让提交者执行（背压）；**DiscardOldestPolicy** 丢最老任务；或自定义记录日志、入库补偿。 |

```java
// 不推荐：无界队列，任务暴增时 OOM
ExecutorService bad = Executors.newFixedThreadPool(10);

// 推荐：有界队列 + 拒绝策略
ExecutorService good = new ThreadPoolExecutor(
    4, 16, 60L, TimeUnit.SECONDS,
    new ArrayBlockingQueue<>(200),
    new ThreadPoolExecutor.CallerRunsPolicy()
);
```

------

### ⚠️ 常见坑与注意点

| 现象 / 易错点 | 原因 | 正确做法 / 怎么记 |
|---------------|------|-------------------|
| 直接用 Executors.newFixedThreadPool 且任务量很大 | 无界 LinkedBlockingQueue，任务堆积导致 OOM | 用 ThreadPoolExecutor + 有界队列（ArrayBlockingQueue 等）+ 拒绝策略。 |

# 🧩 第七章：JMM 与可见性、有序性

> **目标：** 理解 Java 内存模型（JMM）中的主内存与工作内存、happens-before、以及 volatile 与 synchronized 在 JMM 中的语义。

> **本章在整体中解决什么问题**：第三章用了 volatile/synchronized；本章讲**为什么有效**——JMM、主存与工作内存、happens-before、可见性与有序性。校招常问「volatile 为什么不能保证 i++ 安全」「DCL 为什么要加 volatile」；与《JVM与内存模型》笔记中的内存结构可对照。

------

## 🧠 1. JMM 简要

- **JMM**：定义线程与主内存的交互规则，屏蔽底层 CPU 缓存、指令重排等差异，保证“正确同步”的程序具有一致性语义。
- **主内存**：所有线程共享的变量（堆等）。
- **工作内存**：每条线程的私有副本（可类比 CPU 缓存/寄存器），线程对变量的读写先在工作内存，再与主内存交互。

------

## 🧳 2. 原子性、可见性、有序性

| 性质   | 含义                         | 保障手段举例           |
| ------ | ------------------------------ | ---------------------- |
| 原子性 | 操作不可被拆散                | synchronized、原子类   |
| 可见性 | 某线程写后其他线程能读到最新值 | volatile、synchronized |
| 有序性 | 指令不因重排导致多线程下错误  | volatile、synchronized、happens-before |

------

## 🧳 3. happens-before（HB）

- **happens-before**：描述两个操作之间的偏序关系；若 A happens-before B，则 A 对 B 可见。
- 常见规则：程序顺序、锁的 unlock happens-before 后续 lock、volatile 写 happens-before 后续 volatile 读、线程 start happens-before 其 run、线程结束 happens-before join 返回等。
- volatile 与 synchronized 都能在语义上形成 happens-before，从而保证可见性与一定有序性。

------

## 🧳 4. volatile 在 JMM 中的语义

- 写 volatile：写后立即**刷新到主内存**。
- 读 volatile：每次从**主内存**读，并使后续读写不能重排到该读之前。
- 因此可保证**可见性**与**禁止重排**，不保证复合操作**原子性**。

### ❌ 可见性错误示例 vs ✅ 正确示例

**错误：** 共享变量无 volatile，一个线程修改后另一线程可能一直看不到（工作内存未刷新）。

```java
// 错误：running 无 volatile，主线程调用 stop() 后，子线程可能一直看不到 true -> false
static boolean running = true;
public static void main(String[] args) throws Exception {
    new Thread(() -> { while (running) { } }).start();
    Thread.sleep(100);
    running = false;  // 子线程可能永远不退出
}
```

**正确：** 用 volatile 保证写后对其它线程可见。

```java
// 正确：volatile 保证可见性，stop() 后子线程能及时看到
static volatile boolean running = true;
public static void main(String[] args) throws Exception {
    new Thread(() -> { while (running) { } }).start();
    Thread.sleep(100);
    running = false;  // 子线程能退出
}
```

> **面试怎么说**：无 volatile 时，写线程可能只更新了工作内存/缓存，读线程一直读自己的副本；volatile 写会刷新到主内存并令其他线程副本失效，读时从主内存读，从而保证可见性。

------

# ✅ 本章小结

| 知识点     | 面试关键词               | 实际应用     |
| ---------- | ------------------------- | ------------ |
| JMM        | 主内存、工作内存、交互规则 | 理解可见性   |
| 三性       | 原子性、可见性、有序性    | 选 syn/volatile/原子类 |
| happens-before | 偏序、可见性传递       | 理解 syn/volatile 语义 |
| volatile   | 可见、禁重排、不保证原子  | 状态标志等   |

------

## 🎯 面试常见追问

| 面试官提问           | 回答思路 |
| -------------------- | -------- |
| JMM 和 JVM 内存结构区别？ | JMM 是规范（主存/工作内存、HB）；JVM 内存是运行时堆、栈等分区。 |
| volatile 为什么不能保证 i++ 线程安全？ | i++ 是读-改-写，非单次读写；volatile 只保证单次读/写可见与有序，不保证复合原子性。 |

------

### ⚠️ 常见坑与注意点

| 现象 / 易错点 | 原因 | 正确做法 / 怎么记 |
|---------------|------|-------------------|
| 用 volatile 做计数器或 i++ | 读-改-写不是原子操作，多线程会丢更新 | 用 **AtomicInteger** 等原子类，或 synchronized/Lock。 |
| 以为 synchronized 块内不会重排 | 单线程内可重排，但 synchronized 保证块内对块外的可见性与有序性 | 理解 happens-before：unlock 之前的写对后续 lock 之后的读可见。 |
| 混淆 JMM 与 JVM 内存布局 | JMM 是抽象规范（主存/工作内存），JVM 是堆、栈等具体分区 | 面试时区分：JMM 管可见性、有序性规则；JVM 内存是运行时结构。 |

# 🧩 第八章：实践与面试要点

> **目标：** 掌握双重检查锁单例、ThreadLocal 要点、常见坑与最佳实践。

> **本章在整体中解决什么问题**：在前面各章「概念+同步+线程池+JMM」基础上，本章汇总**实战模式**（DCL 单例、ThreadLocal）与**最佳实践**。若面试官追问底层原理，可回到**第三、四章**（synchronized/Lock）、**第七章**（JMM/volatile）；追问线程池见**第六章**；追问并发容器见**第五章**。

------

## 🧠 1. 双重检查锁（DCL）单例

- 懒汉单例中，若只 syn 方法则每次获取都加锁；若只判空再 new 则可能多实例。**DCL**：在 syn 块内外各判空一次，且单例引用用 **volatile**，避免指令重排导致“半初始化”对象被看到。

#### 🧑‍💻 示例代码

```java
private static volatile Singleton instance;
public static Singleton getInstance() {
    if (instance == null) {
        synchronized (Singleton.class) {
            if (instance == null) {
                instance = new Singleton();
            }
        }
    }
    return instance;
}
```

> 💡 若可接受类加载时初始化，**静态内部类**单例更简单且线程安全。

------

## 🧳 2. ThreadLocal

### 📘 作用

- 每个线程持有**本线程专属**的变量副本，线程隔离，无共享则无竞态。
- 典型用途：连接、Transaction、用户上下文等“线程级”数据。

### 📘 注意

- **内存泄漏**：线程池中线程长期存活，ThreadLocal 不 remove 会导致 entry 与 value 无法回收（key 为弱引用会回收，value 强引用可能一直存在）。使用完应 **remove()**。

------

## 🧳 3. 最佳实践与常见坑

| 建议               | 说明                           |
| ------------------ | ------------------------------ |
| 用线程池代替裸 new Thread | 复用、控并发、管理生命周期   |
| 有界队列 + 拒绝策略 | 避免无界队列 OOM、任务丢失有策略 |
| 锁范围尽量小       | 减少竞争，提高吞吐             |
| 避免死锁           | 顺序加锁、tryLock、超时        |
| ThreadLocal 用完 remove | 尤其在线程池中               |
| 不依赖线程优先级   | 仅作提示                       |

# 📎 附录：面试题速查

1. **线程的几种状态？** NEW、RUNNABLE、BLOCKED、WAITING、TIMED_WAITING、TERMINATED。
2. **synchronized 和 Lock 区别？** 见第四章对比表；Lock 可中断、可 tryLock、可公平、多 Condition。
3. **synchronized 和 volatile 区别？** syn 保证原子+可见+有序；volatile 只可见+有序，不保证原子。
4. **线程池核心参数与流程？** core、max、队列、拒绝策略；先核心→队列→非核心→拒绝。
5. **ConcurrentHashMap 原理？** JDK8 数组+链表+红黑树，锁桶+CAS，读多无锁。
6. **如何避免死锁？** 统一加锁顺序、tryLock、超时。
7. **ThreadLocal 原理与注意？** 每线程一个 map 存变量；用完 remove，防泄漏。
