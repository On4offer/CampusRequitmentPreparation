## 🧠 1. 什么是多线程与并发

### 📘 概念

- **多线程**：在同一进程中并发执行多个线程。每个线程是独立的执行路径，拥有自己的程序计数器、栈和局部变量，但**共享进程的堆内存**。线程是 **CPU 调度的基本单位**。
- **并发（Concurrency）**：多个任务在**同一时间段内**交替执行，不要求同一时刻同时执行；单核 CPU 上通过时间片轮转实现。
- **并行（Parallelism）**：多个任务在**同一时刻**同时执行，通常需要多核 CPU。

| 概念     | 说明                         |
| -------- | ---------------------------- |
| 多线程   | 同一进程内多条执行流         |
| 并发     | 时间段内交替执行             |
| 并行     | 同一时刻同时执行（多核）     |

### ⚡ 小结

- 多核 CPU 上，多线程可**并行**执行；单核上多线程是**并发**（时间片轮转）。

## 🧳 3. 进程与线程

### 📘 进程模型

- 进程是**操作系统分配资源的基本单位**，有独立地址空间和资源。
- 进程间通信（IPC）较复杂；创建与销毁开销大。

### 📘 线程模型

- 线程**共享进程的地址空间和资源**，线程间通过共享内存通信简单。
- 创建与切换开销比进程小；同一进程内线程崩溃可能影响整个进程。

### ⚖️ 对比表

| 特性     | 进程                 | 线程                         |
| -------- | -------------------- | ---------------------------- |
| 定义     | 资源分配的基本单位   | CPU 调度的基本单位           |
| 内存     | 独立地址空间         | 共享进程内存                 |
| 通信     | IPC 复杂             | 共享变量即可                 |
| 创建/切换 | 开销大               | 开销小                       |

| 原因           | 说明                           |
| -------------- | ------------------------------ |
| 提高资源利用率 | 多核 CPU 可被多线程充分利用    |
| 提高响应性     | 避免单线程阻塞导致界面卡死     |
| 缩短任务时间   | 大任务拆分并行，减少总耗时     |
| 应对高并发     | Web/中间件等需同时处理大量请求 |

**挑战**：线程安全、死锁、竞态、可见性等问题，需同步与设计规范。

## 🧠 1. 线程的生命周期与状态

### 📘 Java 线程状态（Thread.State）

| 状态            | 说明                                           |
| --------------- | ---------------------------------------------- |
| **NEW**         | 已创建未调用 `start()`                         |
| **RUNNABLE**    | 可运行（就绪或正在运行），等待或占用 CPU       |
| **BLOCKED**     | 等待获取**监视器锁**（如等进入 synchronized）  |
| **WAITING**     | 无限期等待，如 `wait()`、`join()` 无参         |
| **TIMED_WAITING** | 限时等待，如 `sleep()`、`wait(time)`、`join(time)` |
| **TERMINATED**  | 执行结束或异常退出                             |

### 📈 常见转换

- `start()` → NEW 到 RUNNABLE。
- 获取不到锁 → RUNNABLE 到 BLOCKED；获取到锁 → BLOCKED 到 RUNNABLE。
- `wait()` → WAITING；`notify()`/`notifyAll()` → 回到 RUNNABLE（需重新争锁）。
- `run()` 结束或异常 → TERMINATED。

### 📊 线程状态转换图

```
┌───────────┐     start()     ┌────────────┐
│   NEW     │ ──────────────> │  RUNNABLE  │
└───────────┘                 └────────────┘
                                   │
        ┌──────────────────────────┘
        │ 无法获取锁
        ▼
┌───────────┐ <──┐  获取锁  ┌────────────┐
│  BLOCKED  │ ───┘          │  RUNNABLE  │
└───────────┘               └────────────┘
                                   │
        ┌──────────────────────────┘
        │ 调用 wait()/join()/park()
        ▼
┌───────────┐ <──┐  notify()/   ┌────────────┐
│  WAITING  │ ───┘  unpark()    │  RUNNABLE  │
└───────────┘                   └────────────┘
                                   │
        ┌──────────────────────────┘
        │ 调用 sleep()/wait(time)/
        │ join(time)/parkNanos()
        ▼
┌──────────────────┐ <──┐  时间到   ┌────────────┐
│ TIMED_WAITING    │ ───┘          │  RUNNABLE  │
└──────────────────┘                └────────────┘
                                   │
        ┌──────────────────────────┘
        │ 任务完成或异常
        ▼
┌───────────┐
│ TERMINATED │
└───────────┘
```

### 🧳 2.1 守护线程（Daemon Thread）

- **定义**：守护线程是为其他线程提供服务的线程，当所有非守护线程结束时，JVM 会自动退出，无论守护线程是否执行完毕。
- **设置**：`thread.setDaemon(true)`，必须在 `start()` 前设置。
- **应用场景**：垃圾回收器、日志记录、监控等后台任务。
- **注意**：守护线程不能持有任何需要关闭的资源（如文件句柄），因为它可能会在任何时候被终止。

```java
Thread daemonThread = new Thread(() -> {
    while (true) {
        System.out.println("Daemon thread running");
        try { Thread.sleep(1000); } catch (InterruptedException e) {}
    }
});
daemonThread.setDaemon(true);
daemonThread.start();

// 主线程结束后，守护线程也会被终止
System.out.println("Main thread ending");
```

### 🧳 2.2 ThreadLocal

- **定义**：ThreadLocal 为每个线程提供独立的变量副本，线程间互不影响。
- **原理**：内部维护 `ThreadLocalMap`，键为 ThreadLocal 实例，值为线程局部变量。
- **应用场景**：数据库连接、Session 管理、事务上下文等。
- **内存泄漏风险**：ThreadLocalMap 中的键是弱引用，值是强引用。如果线程池中的线程长期存在，可能导致内存泄漏。
- **正确使用**：使用完毕后调用 `remove()` 清理，避免内存泄漏。

```java
ThreadLocal<User> userThreadLocal = new ThreadLocal<>();

// 设置值
userThreadLocal.set(new User("Alice"));

// 获取值
User user = userThreadLocal.get();

// 清理
userThreadLocal.remove();
```

## 🧳 2. 创建线程的方式

### 📘 （1）继承 Thread 类

- 重写 `run()`，`new MyThread().start()` 启动。
- **缺点**：Java 单继承，无法再继承其他类。

#### 🧑‍💻 示例代码

```java
class MyThread extends Thread {
    @Override
    public void run() {
        System.out.println(Thread.currentThread().getName() + " running");
    }
}
// 使用：new MyThread().start();
```

### 📘 （2）实现 Runnable 接口

- 实现 `run()`，将 `Runnable` 传给 `Thread` 构造器再 `start()`。
- **优点**：可多实现接口，任务与线程解耦，便于线程池等复用。

#### 🧑‍💻 示例代码

```java
Runnable task = () -> System.out.println(Thread.currentThread().getName());
new Thread(task).start();
```

### 📘 （3）使用 Callable + Future

- `Callable<V>` 带返回值、可抛异常；通过 `ExecutorService.submit(callable)` 提交，返回 `Future<V>`，用 `get()` 获取结果（阻塞）。

#### 🧑‍💻 示例代码

```java
ExecutorService es = Executors.newSingleThreadExecutor();
Future<String> f = es.submit(() -> "result");
String s = f.get();  // 阻塞直到完成
es.shutdown();
```

> 💡 **面试**：推荐实现 **Runnable** 或 **Callable**，便于与线程池、CompletableFuture 等结合；继承 Thread 不利于扩展。

## 🧳 3. start() 与 run() 的区别

| 方法      | 作用                         |
| --------- | ---------------------------- |
| **start()** | 启动新线程，由系统调度执行 `run()` |
| **run()**   | 线程体；直接调用 `run()` 只在当前线程执行，不会启动新线程 |

| ------------------- | ------------------------ | ---------- |
| **sleep(ms)**       | 当前线程暂停指定时间     | 不释放     |
| **yield()**         | 提示调度器让出 CPU，不保证一定让出 | 不释放     |

## 🧳 5. 线程的中断与安全停止

### 📘 interrupt()

- `interrupt()` 设置线程的**中断标志**，不会立刻终止线程。
- 若线程在 `sleep()`、`wait()`、`join()` 等阻塞中，会抛出 `InterruptedException` 并清除中断标志。
- 线程内可通过 `Thread.interrupted()` 或 `currentThread().isInterrupted()` 检查标志并退出循环。

### 📘 安全停止（非阻塞任务）

- 用 **volatile boolean** 或 **AtomicBoolean** 作为“停止标志”，在循环中判断后退出。

#### 🧑‍💻 示例代码

```java
volatile boolean running = true;
public void stop() { running = false; }
@Override
public void run() {
    while (running) {
        // 做任务
    }
}
```

## 🧠 1. 线程安全的概念

### 📘 定义

- **线程安全**：多线程并发访问时，程序行为正确，数据一致、无竞态。
- 常见**不安全**表现：共享变量被多线程同时读写导致结果不确定（如 i++ 非原子）。

### ⚡ 实现方式概览

| 方式           | 说明                     |
| -------------- | ------------------------ |
| 互斥同步       | synchronized、Lock       |
| 非阻塞         | CAS、原子类              |
| 不可变         | final、不可变对象        |
| 线程封闭       | 局部变量、ThreadLocal    |

## 🧳 2. synchronized 关键字

### 📘 用法

- **同步实例方法**：锁对象为 **this**，同一实例串行执行。
- **同步静态方法**：锁对象为 **当前类的 Class**，所有实例共享一把类锁。
- **同步块**：`synchronized (obj) { ... }`，锁对象为 **obj**。

### 📘 特点

- **可重入**：同一线程可多次获取同一把锁（锁计数器 +1/-1）。
- **不可中断**：等待锁时一般不响应 interrupt（Lock 可中断）。
- **非公平**：不保证先等待的线程先获得锁（ReentrantLock 可选公平）。

#### 🧑‍💻 示例代码

```java
// 同步方法
public synchronized void increment() { count++; }

// 同步块：缩小锁范围，提高并发
public void add(int v) {
    synchronized (this) {
        count += v;
    }
}
```

### ⚡ 同步方法 vs 同步块

| 对比     | 同步方法       | 同步块           |
| -------- | -------------- | ---------------- |
| 锁范围   | 整个方法       | 仅代码块         |
| 灵活性   | 低             | 高（可锁任意对象） |
| 性能     | 易锁粒度过大   | 可细粒度         |

### 📘 synchronized 原理

- **Monitor**：每个对象都有一个 Monitor（监视器），synchronized 基于 Monitor 实现互斥。
- **对象头**：64位 JVM 中，对象头包含 Mark Word（锁信息、哈希码、GC 年龄）、Class Pointer（类元数据指针）。
- **锁升级**：
  1. **无锁**：对象刚创建时，无锁状态。
  2. **偏向锁**：第一个线程获取锁时，记录线程 ID，避免 CAS 操作。
  3. **轻量级锁**：其他线程竞争时，偏向锁升级为轻量级锁，使用 CAS 操作。
  4. **重量级锁**：竞争激烈时，升级为重量级锁，使用操作系统互斥量。

### ⚡ 锁升级过程

| 锁状态 | 特点 | 适用场景 |
| ------ | ---- | -------- |
| 无锁 | 无竞争 | 初始状态 |
| 偏向锁 | 单线程访问 | 无竞争场景 |
| 轻量级锁 | 短时间竞争 | 线程交替执行 |
| 重量级锁 | 长时间竞争 | 多线程同时竞争 |

## 🧳 3. volatile 关键字

### 📘 内存语义

- **可见性**：写 volatile 变量时，会将线程本地缓存刷新到主内存；读 volatile 变量时，会从主内存读取最新值。
- **有序性**：禁止指令重排序，确保代码执行顺序与预期一致。
- **内存屏障**：编译器和 CPU 会在 volatile 操作前后插入内存屏障：
  - **写操作**：StoreStore 屏障（禁止前面的写与当前写重排） → StoreLoad 屏障（禁止当前写与后面的读写重排）
  - **读操作**：LoadLoad 屏障（禁止后面的读与当前读重排） → LoadStore 屏障（禁止后面的写与当前读重排）

### 📘 与 synchronized 的区别

| 特性 | volatile | synchronized |
| ---- | -------- | ------------ |
| 原子性 | 不保证（如 i++） | 保证 |
| 可见性 | 保证 | 保证 |
| 有序性 | 保证 | 保证（串行执行） |
| 开销 | 低（无锁） | 高（可能阻塞） |

## 🧳 4. wait()/notify()/notifyAll()

### 📘 正确使用模式

- **必须在 synchronized 块内调用**：wait()/notify()/notifyAll() 必须在持有对象监视器锁时调用。
- **等待条件**：使用 while 循环检查条件，而非 if，防止虚假唤醒。

```java
// 生产者-消费者模式
class SharedResource {
    private boolean ready = false;
    
    public synchronized void produce() {
        while (ready) {
            try { wait(); } catch (InterruptedException e) {}
        }
        // 生产数据
        ready = true;
        notifyAll(); // 通知消费者
    }
    
    public synchronized void consume() {
        while (!ready) {
            try { wait(); } catch (InterruptedException e) {}
        }
        // 消费数据
        ready = false;
        notifyAll(); // 通知生产者
    }
}
```

### ⚡ notify() vs notifyAll()

| 方法 | 作用 | 适用场景 |
| ---- | ---- | -------- |
| notify() | 唤醒一个等待线程 | 只有一个等待线程时 |
| notifyAll() | 唤醒所有等待线程 | 多个等待线程时，避免信号丢失 |

> 💡 **推荐**：优先使用 `notifyAll()`，避免因信号丢失导致线程永久等待。

## 🧳 5. 死锁

### 📘 死锁的四个必要条件

1. **互斥条件**：资源不能被多个线程同时使用。
2. **请求与保持条件**：线程持有资源的同时，继续请求其他资源。
3. **不剥夺条件**：线程获得的资源不能被强制剥夺。
4. **循环等待条件**：多个线程形成循环等待资源的链。

### 📘 死锁示例

```java
class DeadlockDemo {
    private static final Object lockA = new Object();
    private static final Object lockB = new Object();
    
    public static void main(String[] args) {
        // 线程1：先获取 lockA，再获取 lockB
        new Thread(() -> {
            synchronized (lockA) {
                System.out.println("Thread 1 holding lockA");
                try { Thread.sleep(100); } catch (InterruptedException e) {}
                synchronized (lockB) {
                    System.out.println("Thread 1 holding lockB");
                }
            }
        }).start();
        
        // 线程2：先获取 lockB，再获取 lockA
        new Thread(() -> {
            synchronized (lockB) {
                System.out.println("Thread 2 holding lockB");
                try { Thread.sleep(100); } catch (InterruptedException e) {}
                synchronized (lockA) {
                    System.out.println("Thread 2 holding lockA");
                }
            }
        }).start();
    }
}
```

### 📘 死锁预防

- **破坏互斥条件**：使用共享资源（如读写锁）。
- **破坏请求与保持条件**：一次性获取所有资源。
- **破坏不剥夺条件**：允许超时放弃资源。
- **破坏循环等待条件**：按顺序获取资源。

### 📘 死锁检测

- **jstack**：使用 `jstack <pid>` 查看线程堆栈，分析阻塞状态。
- **VisualVM**：通过 JDK 自带的 VisualVM 查看线程状态。

------

## 🧳 6. 线程安全的其他实现方式

### 📘 不可变对象

- **定义**：对象创建后状态不可修改。
- **实现**：
  - 使用 `final` 修饰字段
  - 不提供 setter 方法
  - 字段为引用类型时，确保引用的对象也是不可变的
- **优势**：天然线程安全，无需同步。

### 📘 线程封闭

- **栈封闭**：局部变量存储在栈中，线程私有。
- **ThreadLocal**：每个线程独立的变量副本。

### 📘 无锁编程

- **CAS**：Compare-And-Swap，乐观锁，通过硬件指令实现原子操作。
- **原子类**：如 `AtomicInteger`、`AtomicReference`，基于 CAS 实现。

## 🧳 4. wait() 与 notify() / notifyAll()

### 📘 要点

- 必须在 **synchronized** 块内调用；否则抛出 `IllegalMonitorStateException`。（ReentrantLock 等显式锁 → 必须用 `Condition`）
- **wait()**：释放锁，当前线程进入该对象的等待集，直到被 notify 或中断。（`wait()` 释放的是 synchronized 内置监视器锁）
- **notify()**：唤醒该对象等待集上的**一个**线程（不指定哪个）；**notifyAll()** 唤醒全部。
- 被唤醒后线程需**重新竞争锁**，获得锁后才从 `wait()` 返回。

### 📘 模板（生产者-消费者）

```java
synchronized (lock) {
    while (!条件满足) {  // 必须用 while，防止虚假唤醒
        lock.wait();
    }
    // 业务
    lock.notifyAll();  // 或 notify()
}
```

## 🧳 5. 死锁与预防

### 📘 死锁四个必要条件

1. **互斥**：资源只能被一个线程占用。
2. **占有且等待**：持有至少一个资源并等待其他资源。
3. **不可剥夺**：资源不能被强制剥夺。
4. **循环等待**：存在线程等待环，如 A 等 B，B 等 A。

### ⚡ 预防思路

- **固定加锁顺序**：所有线程按同一顺序获取多把锁（如先 A 后 B），避免循环等待。
- **tryLock**：使用 `ReentrantLock.tryLock()`，获取不到则释放已持有锁并重试或退避。
- **超时**：`tryLock(timeout)`，避免无限等待。

#### 🧑‍💻 示例代码（tryLock 避免死锁）

```java
if (lockA.tryLock()) {
    try {
        if (lockB.tryLock()) {
            try {
                // 业务
            } finally { lockB.unlock(); }
        }
    } finally { lockA.unlock(); }
}
```

### ⚠️ 常见坑与注意点

| 现象 / 易错点 | 原因 | 正确做法 / 怎么记 |
|---------------|------|-------------------|
| 锁错了对象（如锁方法却锁在 null 或不同实例） | 不同线程拿到不同锁，起不到互斥作用 | 多线程要互斥必须锁**同一个对象**；实例方法锁 this，静态方法锁 Class；同步块明确锁同一 lock 对象。 |

| 以为 volatile 能保证 i++ 线程安全 | i++ 是读-改-写三步，volatile 只保证单次读写可见 | 计数、复合操作用 synchronized 或原子类（AtomicInteger）；volatile 只用于状态标志等一写多读。 |
| 多把锁加锁顺序不一致 | 易形成循环等待，死锁 | 所有线程按**同一顺序**获取多把锁（如先 A 后 B）；或使用 tryLock 失败则释放已持有的锁。 |
| synchronized 锁在包装类型或 String 常量 | 包装类型缓存、字符串常量池可能导致多段代码误用同一锁 | 锁对象用 **private final Object lock = new Object()** 或专用锁实例，避免与别人锁到同一对象。 |

# 🧩 第四章：Lock 与 ReentrantLock

> **目标：** 掌握 Lock 接口与 ReentrantLock 的用法，理解与 synchronized 的差异及 AQS 简要原理。

> **本章在整体中解决什么问题**：第三章讲了 synchronized；本章讲 **Lock 接口与 ReentrantLock**——可中断、tryLock、公平锁、多 Condition，以及底层 AQS 简要。学完本章后**第五章**讲并发集合与原子类；面试常对比「synchronized 和 Lock 区别」。

## 🧠 1. Lock 接口

### 📘 常用方法

| 方法          | 说明                     |
| ------------- | ------------------------ |
| `lock()`      | 获取锁，阻塞直到获得     |
| `unlock()`    | 释放锁，必须在 finally 中调用 |
| `tryLock()`   | 尝试获取，不阻塞，返回是否成功 |
| `tryLock(time)` | 在指定时间内尝试获取     |
| `lockInterruptibly()` | 可响应中断的获取方式   |

### 📘 使用模板

```java
Lock lock = new ReentrantLock();
lock.lock();
try {
    // 临界区
} finally {
    lock.unlock();  // 必须释放
}
```

------

## 🧳 2. ReentrantLock 与 synchronized 对比

| 对比项     | synchronized              | ReentrantLock                    |
| ---------- | ------------------------- | -------------------------------- |
| 使用方式   | 关键字，自动加锁/解锁     | API，需手动 lock/unlock          |
| 公平锁     | 非公平                    | 可选公平/非公平构造              |
| 可中断     | 等待锁时一般不响应中断   | lockInterruptibly() 可响应       |
| 尝试获取   | 无                        | tryLock() / tryLock(time)        |
| 条件队列   | 单条件（wait/notify）     | 多条件（Condition）              |
| 底层       | JVM 管程（monitor）       | AQS（AbstractQueuedSynchronizer）|

------

## 🧳 3. Condition 条件变量

- `Lock.newCondition()` 得到 **Condition**，替代 wait/notify；可多个条件，语义更清晰。
- `await()` / `signal()` / `signalAll()` 对应 `wait()` / `notify()` / `notifyAll()`，同样必须在持有锁时调用。

#### 🧑‍💻 示例代码

```java
Lock lock = new ReentrantLock();
Condition notFull = lock.newCondition();
Condition notEmpty = lock.newCondition();
// 生产者：notFull.await(); notEmpty.signal();
// 消费者：notEmpty.await(); notFull.signal();
```

## 🧳 4. AQS 简述（面试向）

- **AQS**：AbstractQueuedSynchronizer，是很多同步器的基础（ReentrantLock、Semaphore、CountDownLatch、ReentrantReadWriteLock 等）。
- 内部维护 **state** 和 **FIFO 队列**；子类通过 CAS 改 state，失败则入队阻塞，释放时唤醒后继。
- ReentrantLock：state 表示重入次数；独占模式，公平/非公平由入队与抢锁策略决定。

> 💡 校招常问“ReentrantLock 和 synchronized 区别”“AQS 是什么”，能说出上表 + “基于 AQS、state+队列”即可。

### ⚠️ 常见坑与注意点

| 现象 / 易错点 | 原因 | 正确做法 / 怎么记 |
|---------------|------|-------------------|
| lock() 后未在 finally 中 unlock() | 异常跳出时锁不释放，导致死锁或后续永远拿不到锁 | **必须在 finally 里调用 unlock()**，且只释放当前线程持有的锁。 |
| 在未持有锁时调用 unlock() | 会抛 IllegalMonitorStateException | 只有当前线程 hold 住该锁时才能 unlock；用 tryLock 时若未获取成功不要 unlock。 |
| 把 Lock 声明在方法内部 | 每次调用 new 一个锁，多线程锁的不是同一把锁 | 锁对象应作为**成员变量**或**类静态变量**，保证多线程竞争的是同一把锁。 |
| Condition 的 await/signal 未在持有锁时调用 | 会抛 IllegalMonitorStateException | 与 wait/notify 一样，必须在 **lock.lock()** 与 **lock.unlock()** 之间调用 await/signal。 |

## 🧳 5. 读写锁与 StampedLock（补充）

### 📘 ReentrantReadWriteLock

- **读锁（ReadLock）**：共享锁，多个线程可同时读
- **写锁（WriteLock）**：独占锁，写时禁止其他读写
- **锁降级**：写锁可降级为读锁（获取写锁→获取读锁→释放写锁），保证看到最新值

```java
ReentrantReadWriteLock rwLock = new ReentrantReadWriteLock();
// 读操作
rwLock.readLock().lock();
try { /* 读取 */ } finally { rwLock.readLock().unlock(); }
// 写操作
rwLock.writeLock().lock();
try { /* 写入 */ } finally { rwLock.writeLock().unlock(); }
```

> ⚠️ 不支持锁升级（读锁→写锁），会导致死锁

### 📘 StampedLock（JDK 8）

- **三种模式**：写锁（独占）、悲观读锁（共享）、乐观读（无锁）
- **乐观读**：读时记录版本戳，读完后验证，如被修改则转为悲观读
- **优势**：读多写少时，乐观读完全无锁，性能远高于 ReentrantReadWriteLock

```java
StampedLock lock = new StampedLock();
long stamp = lock.tryOptimisticRead();  // 乐观读
// 读取数据...
if (!lock.validate(stamp)) {  // 验证版本
    stamp = lock.readLock();  // 被修改，转为悲观读
    try { /* 重新读取 */ } finally { lock.unlockRead(stamp); }
}
```

## 🧳 2. ConcurrentHashMap

### 📘 JDK 8 实现原理

- **数据结构**：数组 + 链表 + 红黑树
- **并发控制**：
  - **读操作**：无锁，使用 `volatile` 保证可见性
  - **写操作**：锁桶头节点（synchronized）+ CAS
- **特点**：不支持 null key/value，避免二义性

### 📘 put 操作流程

1. **计算哈希**：对 key 进行哈希计算，获取桶索引
2. **检查数组**：若数组未初始化，CAS 初始化
3. **检查桶头**：
   - 若桶为空，CAS 插入新节点
   - 若桶不为空，synchronized 锁桶头节点
4. **处理节点**：
   - 若为链表，遍历找到位置插入
   - 若为红黑树，按红黑树规则插入
5. **扩容检查**：若当前桶元素数超过阈值，触发扩容

### 📘 get 操作流程

1. **计算哈希**：对 key 进行哈希计算，获取桶索引
2. **检查数组**：若数组未初始化或桶为空，返回 null
3. **遍历桶**：
   - 若桶头节点匹配，返回对应 value
   - 若为红黑树，按红黑树查找
   - 若为链表，遍历查找
4. **返回结果**：找到则返回 value，否则返回 null

### 📘 扩容机制

- **触发条件**：当桶元素数超过阈值（默认 0.75）
- **扩容过程**：
  1. 计算新容量（原容量 * 2）
  2. 创建新数组
  3. 并发迁移：每个线程负责一部分桶的迁移
  4. 迁移完成后，替换数组引用

### 📘 size() 实现

- **计数器**：维护多个 counter，减少竞争
- **计算方式**：
  1. 尝试无锁方式累加所有 counter
  2. 若发现并发修改，使用加锁方式重新计算

### ⚡ 与 Hashtable 对比

| 特性 | ConcurrentHashMap | Hashtable |
| ---- | ----------------- | --------- |
| 锁粒度 | 桶级锁（synchronized + CAS） | 全表锁（synchronized） |
| 并发性能 | 高（多线程可同时操作不同桶） | 低（同一时间只能一个线程操作） |
| null 支持 | 不支持 | 支持 |
| 迭代器 | 弱一致性（可能看到旧值） | 快速失败（并发修改抛异常） |

## 🧳 3. CopyOnWriteArrayList

### 📘 原理

- **写时复制**：写操作时复制新数组，在新数组上修改，再替换引用；读无锁。
- **适用**：读多写少（如监听器列表）；写多时复制成本高，不适用。

### 📘 核心方法

- **add(E e)**：创建新数组，复制原数组元素，添加新元素，替换引用
- **get(int index)**：直接从原数组读取，无锁
- **remove(int index)**：创建新数组，复制除指定位置外的元素，替换引用

### ⚡ 优缺点

| 优点 | 缺点 |
| ---- | ---- |
| 读操作无锁，性能高 | 写操作需要复制数组，成本高 |
| 读操作不需要同步，不会阻塞 | 内存占用较大（同时存在新旧数组） |
| 迭代器不会抛出 ConcurrentModificationException | 写操作之间需要同步，可能阻塞 |

## 🧳 4. BlockingQueue 与常用实现

### 📘 核心方法

| 方法 | 描述 | 满/空时行为 |
| ---- | ---- | ----------- |
| put(E e) | 插入元素 | 满时阻塞 |
| take() | 取出并移除元素 | 空时阻塞 |
| offer(E e) | 尝试插入 | 满时返回 false |
| poll() | 尝试取出 | 空时返回 null |
| offer(E e, long timeout, TimeUnit unit) | 限时尝试插入 | 超时返回 false |
| poll(long timeout, TimeUnit unit) | 限时尝试取出 | 超时返回 null |

### 📘 常用实现对比

| 实现类 | 特点 | 适用场景 |
| ------ | ---- | -------- |
| **ArrayBlockingQueue** | 有界，数组，一把锁 | 固定容量场景，如线程池任务队列 |
| **LinkedBlockingQueue** | 可选有界，链表，两把锁（put/take 可并行） | 生产者-消费者模式，吞吐量较高 |
| **LinkedTransferQueue** | 无界，支持 transfer（等待消费者） | 需要确保元素被消费的场景 |
| **SynchronousQueue** | 无容量，直接交付 | 线程间直接传递，如 Executors.newCachedThreadPool |
| **DelayQueue** | 支持延迟获取 | 定时任务，如 ScheduledThreadPoolExecutor |
| **PriorityBlockingQueue** | 无界，优先级排序 | 任务优先级处理 |

### 📘 应用场景

- **生产者-消费者模式**：使用 put/take 阻塞方法
- **线程池**：作为任务队列，如 LinkedBlockingQueue
- **消息传递**：线程间安全传递数据

## 🧳 5. 原子类与 CAS

### 📘 原子类（java.util.concurrent.atomic）

- **AtomicInteger**、**AtomicLong**、**AtomicReference** 等，提供 `get`、`set`、`getAndIncrement`、`compareAndSet` 等**无锁**更新。
- 底层依赖 **CAS**（Compare-And-Swap）：比较并交换，CPU 指令级支持，保证单变量读-改-写原子性。

### 📘 CAS 原理

- **CAS 操作**：`compareAndSet(expectedValue, newValue)`
  1. 读取当前值 `currentValue`
  2. 比较 `currentValue` 与 `expectedValue`
  3. 若相等，将值更新为 `newValue`，返回 true
  4. 若不相等，不更新，返回 false

- **特点**：无锁、乐观、自旋

### 📘 原子类分类

| 类型 | 代表类 | 用途 |
| ---- | ------ | ---- |
| 基本类型 | AtomicInteger、AtomicLong、AtomicBoolean | 原子更新基本类型 |
| 引用类型 | AtomicReference、AtomicStampedReference | 原子更新引用类型 |
| 数组类型 | AtomicIntegerArray、AtomicLongArray | 原子更新数组元素 |
| 字段更新器 | AtomicIntegerFieldUpdater | 原子更新对象字段 |
| 累加器 | LongAdder、LongAccumulator | 高并发计数 |

### 📘 LongAdder 与 LongAccumulator

- **LongAdder**：分段累加，将计数器分为多个段，减少 CAS 竞争
- **LongAccumulator**：支持自定义累加器函数
- **优势**：高并发下性能远高于 AtomicLong

```java
// LongAdder 示例
LongAdder counter = new LongAdder();
// 多线程并发调用
counter.increment();
// 获取最终结果
long result = counter.sum();
```

### 📘 ABA 问题

- **ABA 问题**：线程 1 读取值为 A，线程 2 将值改为 B 后又改回 A，线程 1 执行 CAS 时发现值仍为 A，会误以为没有发生过修改
- **解决方案**：使用 `AtomicStampedReference`，增加版本号

```java
// AtomicStampedReference 解决 ABA 问题
AtomicStampedReference<String> ref = new AtomicStampedReference<>("A", 0);
int[] stampHolder = new int[1];
String currentValue = ref.get(stampHolder);
int currentStamp = stampHolder[0];
// 比较时同时比较值和版本号
ref.compareAndSet(currentValue, "B", currentStamp, currentStamp + 1);
```

### 📘 Unsafe 类

- **Unsafe**：提供底层 CAS 操作、内存操作等能力
- **注意**：不建议直接使用，属于 JDK 内部 API




```java
AtomicInteger count = new AtomicInteger(0);
count.incrementAndGet();
count.getAndAdd(10);
```

# 🧩 第六章：线程池

> **目标：** 理解线程池的作用、核心参数、常见类型、拒绝策略及使用建议。

> **本章在整体中解决什么问题**：前面讲的是单线程与同步；本章讲**如何管理大量任务与线程**——线程池参数、执行流程、拒绝策略。校招常问核心线程数怎么设、为什么不建议 Executors；学完本章后**第八章**会强调「用线程池代替裸 new Thread」等实践。

------

## 🧠 1. 为什么使用线程池

- **降低开销**：复用线程，避免频繁创建/销毁。
- **控制并发**：限制同时运行的线程数，防止资源耗尽。
- **管理任务**：队列排队、定时/周期任务（ScheduledExecutorService）。

------

## 🧳 2. ThreadPoolExecutor 核心参数

| 参数             | 含义                         |
| ---------------- | ---------------------------- |
| **corePoolSize** | 核心线程数，常驻（可配置允许超时回收） |
| **maximumPoolSize** | 最大线程数                 |
| **keepAliveTime**   | 非核心线程空闲存活时间     |
| **unit**            | 时间单位                   |
| **workQueue**       | 任务队列（有界/无界）      |
| **threadFactory**   | 创建线程的工厂             |
| **handler**         | 拒绝策略（队列满且线程满时） |

### 📘 执行流程简述

1. 任务提交后，若运行线程数 < corePoolSize，新建核心线程执行。
2. 否则放入 workQueue。
3. 若队列满且运行线程数 < maximumPoolSize，新建非核心线程执行。
4. 若队列满且线程数已达 maximum，执行 **拒绝策略**。

------

## 🧳 3. 常见线程池类型（Executors）

| 方法                         | 实现特点                         | 注意                     |
| ---------------------------- | -------------------------------- | ------------------------ |
| **newFixedThreadPool(n)**    | 固定 n 线程，无界队列            | 队列无界可能 OOM         |
| **newCachedThreadPool()**    | 无核心，可扩容，60s 回收         | 任务多时线程数可能很大   |
| **newSingleThreadExecutor()** | 单线程，无界队列                | 顺序执行                 |
| **newScheduledThreadPool(n)** | 可延迟/周期执行                 | 定时任务                 |

> ⚠️ **阿里规范**：不建议直接使用 `Executors`，推荐 **ThreadPoolExecutor** 显式指定队列大小和拒绝策略，避免无界队列导致 OOM。

------

## 🧳 4. 拒绝策略

| 策略                  | 行为                           |
| --------------------- | ------------------------------ |
| **AbortPolicy**（默认） | 抛 RejectedExecutionException  |
| **CallerRunsPolicy**  | 由提交任务的线程执行           |
| **DiscardPolicy**     | 静默丢弃                       |
| **DiscardOldestPolicy** | 丢弃队列最老任务，再提交当前 |

------

## 🧳 5. 使用建议

- 根据任务类型设 **corePoolSize**（CPU 密集 ≈ 核数，IO 密集可更大）。
- 使用**有界队列**并指定**拒绝策略**，避免无界队列与默认 Abort 导致问题。
- 关闭时：**shutdown()** 不再接受新任务，**shutdownNow()** 尝试中断正在执行的任务；可用 **awaitTermination** 等待结束。

### ❌ 错误配置 vs ✅ 正确配置（面试常问）

| 场景 | 错误做法 | 正确做法 |
|------|----------|----------|
| 高吞吐任务、任务量不可控 | `Executors.newFixedThreadPool(10)` 或 `newCachedThreadPool()` | 使用 **ThreadPoolExecutor** 构造，指定**有界队列**（如 `new ArrayBlockingQueue<>(200)`）和**拒绝策略**（如 CallerRunsPolicy 或自定义），避免无界队列 OOM、Cached 线程数爆炸。 |
| 核心线程数 | 拍脑袋写 10、100 | CPU 密集约 **CPU 核数**或核数+1；IO 密集可 **核数 * 2** 或更大，结合压测；最大线程数按队列与拒绝策略设上限。 |
| 拒绝策略 | 不设置，默认 Abort 直接抛异常丢任务 | 根据业务选择：**CallerRunsPolicy** 让提交者执行（背压）；**DiscardOldestPolicy** 丢最老任务；或自定义记录日志、入库补偿。 |

```java
// 不推荐：无界队列，任务暴增时 OOM
ExecutorService bad = Executors.newFixedThreadPool(10);

// 推荐：有界队列 + 拒绝策略
ExecutorService good = new ThreadPoolExecutor(
    4, 16, 60L, TimeUnit.SECONDS,
    new ArrayBlockingQueue<>(200),
    new ThreadPoolExecutor.CallerRunsPolicy()
);
```

### ⚠️ 常见坑与注意点

| 现象 / 易错点 | 原因 | 正确做法 / 怎么记 |
|---------------|------|-------------------|
| 直接用 Executors.newFixedThreadPool 且任务量很大 | 无界 LinkedBlockingQueue，任务堆积导致 OOM | 用 ThreadPoolExecutor + 有界队列（ArrayBlockingQueue 等）+ 拒绝策略。 |
| 核心线程数设得过大（如 100）且任务多为 CPU 密集 | 过多线程上下文切换，反而变慢 | CPU 密集约等于核数；IO 密集可适当放大，需压测。 |
| 不设拒绝策略，队列满就抛 RejectedExecutionException | 任务丢失且可能未记录 | 显式指定 handler（CallerRunsPolicy、自定义打日志/入库等）。 |
| 线程池不关闭，进程退出时线程未结束 | 资源泄漏、优雅停机失败 | 使用 shutdown() 或 shutdownNow()，必要时 awaitTermination 等待。 |

## 🧳 6. 任务队列详解

线程池的任务队列是缓冲任务的核心组件，选择合适的队列对性能和稳定性至关重要。

### 📘 常见队列对比

| 队列类型 | 特点 | 适用场景 | 注意事项 |
| -------- | ---- | -------- | -------- |
| **SynchronousQueue** | 无容量，直接交付 | 高吞吐、任务处理快 | 不存储任务，必须有空闲线程立即处理 |
| **LinkedBlockingQueue** | 链表，可选有界/无界 | 通用场景 | 无界时可能 OOM |
| **ArrayBlockingQueue** | 数组，有界 | 需要限制队列大小 | 一把锁，put/take 串行 |
| **PriorityBlockingQueue** | 优先级队列 | 任务有优先级 | 无界，可能 OOM |
| **DelayQueue** | 延迟队列 | 定时/延迟任务 | 元素需实现 Delayed 接口 |
| **LinkedTransferQueue** | 可 transfer | 生产者等待消费者 | 比 SynchronousQueue 更灵活 |

### 📘 SynchronousQueue

- **特点**：不存储元素，每个插入操作必须等待另一个线程的移除操作，反之亦然。
- **使用场景**：任务处理速度极快，不需要缓冲，直接交给工作线程。
- **示例**：`Executors.newCachedThreadPool()` 内部使用 SynchronousQueue。

```java
// 使用 SynchronousQueue 的线程池
ExecutorService executor = new ThreadPoolExecutor(
    0, Integer.MAX_VALUE, 60L, TimeUnit.SECONDS,
    new SynchronousQueue<>()
);
```

### 📘 LinkedBlockingQueue vs ArrayBlockingQueue

| 特性 | LinkedBlockingQueue | ArrayBlockingQueue |
| ---- | ------------------- | ------------------ |
| 实现 | 链表 | 数组 |
| 有界性 | 可选有界/无界 | 必须有界 |
| 锁 | 两把锁（put/take 可并行） | 一把锁 |
| 内存 | 每个节点额外开销 | 预分配，无节点开销 |
| 性能 | 高并发下更好 | 内存更紧凑 |

> 💡 **推荐**：一般使用 ArrayBlockingQueue，内存更可控；高并发且内存充足可用 LinkedBlockingQueue。

### 📘 DelayQueue

- 元素必须实现 `Delayed` 接口，指定延迟时间。
- 只有延迟时间到了才能从队列取出。
- 常用于**定时任务调度**、**缓存过期**等场景。

```java
// DelayQueue 示例
DelayQueue<DelayedTask> queue = new DelayQueue<>();
queue.put(new DelayedTask("task1", 5000)); // 5秒后执行
DelayedTask task = queue.take(); // 阻塞等待
```

## 🧳 7. CompletableFuture 异步编程

`CompletableFuture` 是 JDK 8 引入的异步编程工具，支持链式调用、组合多个异步任务、异常处理等。

### 📘 创建 CompletableFuture

```java
// 无返回值异步任务
CompletableFuture<Void> future1 = CompletableFuture.runAsync(() -> {
    System.out.println("异步任务");
});

// 有返回值异步任务
CompletableFuture<String> future2 = CompletableFuture.supplyAsync(() -> {
    return "result";
});

// 指定线程池
ExecutorService executor = Executors.newFixedThreadPool(4);
CompletableFuture<String> future3 = CompletableFuture.supplyAsync(() -> "result", executor);
```

### 📘 链式操作

| 方法 | 说明 |
| ---- | ---- |
| `thenApply` | 对结果进行转换（有返回值） |
| `thenAccept` | 消费结果（无返回值） |
| `thenRun` | 任务完成后执行（不依赖结果） |
| `exceptionally` | 异常处理 |
| `whenComplete` | 无论成功失败都执行 |
| `handle` | 统一处理正常结果和异常 |

```java
CompletableFuture<String> future = CompletableFuture.supplyAsync(() -> "Hello")
    .thenApply(s -> s + " World")
    .thenApply(String::toUpperCase)
    .whenComplete((result, ex) -> {
        if (ex != null) {
            System.err.println("Error: " + ex);
        } else {
            System.out.println("Result: " + result);
        }
    });
```

### 📘 组合多个 Future

| 方法 | 说明 |
| ---- | ---- |
| `thenCompose` | 串行组合（前一个结果作为后一个输入） |
| `thenCombine` | 并行组合（两个任务都完成后合并结果） |
| `allOf` | 等待所有任务完成 |
| `anyOf` | 任意一个任务完成即可 |

```java
// 串行组合
CompletableFuture<String> result = getUserId()
    .thenCompose(userId -> getUserName(userId));

// 并行组合
CompletableFuture<String> result = getUser()
    .thenCombine(getOrder(), (user, order) -> user + ":" + order);

// 等待所有
CompletableFuture<Void> all = CompletableFuture.allOf(future1, future2, future3);
all.join(); // 等待全部完成
```

### 📘 异常处理

```java
CompletableFuture<String> future = CompletableFuture.supplyAsync(() -> {
    if (true) throw new RuntimeException("error");
    return "success";
}).exceptionally(ex -> {
    // 异常时返回默认值
    return "default";
});

// 或者使用 handle 统一处理
CompletableFuture<String> future2 = CompletableFuture.supplyAsync(() -> "result")
    .handle((result, ex) -> {
        if (ex != null) return "error: " + ex.getMessage();
        return result;
    });
```

### 📘 超时控制（JDK 9+）

```java
CompletableFuture<String> future = CompletableFuture.supplyAsync(() -> {
    try {
        Thread.sleep(5000);
    } catch (InterruptedException e) {
        Thread.currentThread().interrupt();
    }
    return "result";
}).orTimeout(3, TimeUnit.SECONDS)  // 3秒超时抛异常
  .exceptionally(ex -> "timeout");
```

## 🧳 8. ForkJoinPool 与分治任务

`ForkJoinPool` 是 JDK 7 引入的线程池，专为**分治算法**设计，采用**工作窃取（Work-Stealing）**算法。

### 📘 核心概念

- **Fork**：将大任务拆分为小任务。
- **Join**：等待子任务完成并合并结果。
- **工作窃取**：空闲线程从其他线程的队列尾部窃取任务，提高 CPU 利用率。

### 📘 使用方式

#### RecursiveTask（有返回值）

```java
class SumTask extends RecursiveTask<Long> {
    private static final int THRESHOLD = 10000;
    private long[] array;
    private int start, end;
    
    public SumTask(long[] array, int start, int end) {
        this.array = array;
        this.start = start;
        this.end = end;
    }
    
    @Override
    protected Long compute() {
        if (end - start <= THRESHOLD) {
            // 直接计算
            long sum = 0;
            for (int i = start; i < end; i++) {
                sum += array[i];
            }
            return sum;
        }
        
        // 拆分任务
        int mid = (start + end) / 2;
        SumTask left = new SumTask(array, start, mid);
        SumTask right = new SumTask(array, mid, end);
        
        left.fork();  // 异步执行左任务
        long rightResult = right.compute();  // 同步执行右任务
        long leftResult = left.join();  // 等待左任务结果
        
        return leftResult + rightResult;
    }
}

// 使用
ForkJoinPool pool = new ForkJoinPool();
long result = pool.invoke(new SumTask(array, 0, array.length));
```

#### RecursiveAction（无返回值）

```java
class PrintTask extends RecursiveAction {
    private int start, end;
    
    @Override
    protected void compute() {
        if (end - start <= 100) {
            // 直接执行
            for (int i = start; i < end; i++) {
                System.out.println(i);
            }
        } else {
            int mid = (start + end) / 2;
            invokeAll(
                new PrintTask(start, mid),
                new PrintTask(mid, end)
            );
        }
    }
}
```

### 📘 适用场景

- **大数据计算**：数组求和、排序、搜索。
- **并行流**：`Stream.parallel()` 底层使用 ForkJoinPool。
- **递归任务**：树遍历、文件处理等可拆分任务。

> ⚠️ **注意**：任务拆分到足够小才执行，避免过度拆分导致线程切换开销大于计算收益。

## 🧳 9. ScheduledThreadPoolExecutor 定时任务

`ScheduledThreadPoolExecutor` 是定时任务线程池，支持延迟执行和周期执行。

### 📘 创建方式

```java
// 不推荐：无界队列可能 OOM
ScheduledExecutorService bad = Executors.newScheduledThreadPool(10);

// 推荐：自定义线程池
ScheduledThreadPoolExecutor executor = new ScheduledThreadPoolExecutor(
    5,
    new ThreadPoolExecutor.DiscardPolicy()
);
```

### 📘 调度方法

| 方法 | 说明 |
| ---- | ---- |
| `schedule(Runnable, delay, unit)` | 延迟执行 |
| `scheduleAtFixedRate(Runnable, initialDelay, period, unit)` | 固定频率执行（可能堆积） |
| `scheduleWithFixedDelay(Runnable, initialDelay, delay, unit)` | 固定间隔执行（上次结束+间隔） |

```java
// 延迟 5 秒执行
executor.schedule(() -> System.out.println("delayed"), 5, TimeUnit.SECONDS);

// 初始延迟 1 秒，之后每 3 秒执行一次
executor.scheduleAtFixedRate(() -> {
    System.out.println("fixed rate");
}, 1, 3, TimeUnit.SECONDS);

// 上次执行结束后等待 2 秒再执行
executor.scheduleWithFixedDelay(() -> {
    System.out.println("fixed delay");
}, 0, 2, TimeUnit.SECONDS);
```

### 📘 注意事项

- `scheduleAtFixedRate`：如果任务执行时间超过周期，会堆积任务，可能 OOM。
- `scheduleWithFixedDelay`：更可控，推荐用于需要稳定间隔的场景。
- 任务抛出异常会终止后续执行，需自行捕获处理。

# 🧩 第七章：JMM 与可见性、有序性

> **目标：** 理解 Java 内存模型（JMM）中的主内存与工作内存、happens-before、以及 volatile 与 synchronized 在 JMM 中的语义。

> **本章在整体中解决什么问题**：第三章用了 volatile/synchronized；本章讲**为什么有效**——JMM、主存与工作内存、happens-before、可见性与有序性。校招常问「volatile 为什么不能保证 i++ 安全」「DCL 为什么要加 volatile」；与《JVM与内存模型》笔记中的内存结构可对照。

------

## 🧠 1. JMM 简要

- **JMM**：定义线程与主内存的交互规则，屏蔽底层 CPU 缓存、指令重排等差异，保证“正确同步”的程序具有一致性语义。
- **主内存**：所有线程共享的变量（堆等）。
- **工作内存**：每条线程的私有副本（可类比 CPU 缓存/寄存器），线程对变量的读写先在工作内存，再与主内存交互。

------

## 🧳 2. 原子性、可见性、有序性

| 性质   | 含义                         | 保障手段举例           |
| ------ | ------------------------------ | ---------------------- |
| 原子性 | 操作不可被拆散                | synchronized、原子类   |
| 可见性 | 某线程写后其他线程能读到最新值 | volatile、synchronized |
| 有序性 | 指令不因重排导致多线程下错误  | volatile、synchronized、happens-before |

------

## 🧳 3. happens-before（HB）

- **happens-before**：描述两个操作之间的偏序关系；若 A happens-before B，则 A 对 B 可见。
- 常见规则：程序顺序、锁的 unlock happens-before 后续 lock、volatile 写 happens-before 后续 volatile 读、线程 start happens-before 其 run、线程结束 happens-before join 返回等。
- volatile 与 synchronized 都能在语义上形成 happens-before，从而保证可见性与一定有序性。

------

## 🧳 4. volatile 在 JMM 中的语义

- 写 volatile：写后立即**刷新到主内存**。
- 读 volatile：每次从**主内存**读，并使后续读写不能重排到该读之前。
- 因此可保证**可见性**与**禁止重排**，不保证复合操作**原子性**。

### 📘 内存屏障

- **内存屏障**：编译器和 CPU 指令，用于禁止特定类型的指令重排序
- **volatile 内存屏障策略**：
  - **写操作**：StoreStore 屏障（禁止前面的写与当前写重排） → StoreLoad 屏障（禁止当前写与后面的读写重排）
  - **读操作**：LoadLoad 屏障（禁止后面的读与当前读重排） → LoadStore 屏障（禁止后面的写与当前读重排）

### ❌ 可见性错误示例 vs ✅ 正确示例

**错误：** 共享变量无 volatile，一个线程修改后另一线程可能一直看不到（工作内存未刷新）。

```java
// 错误：running 无 volatile，主线程调用 stop() 后，子线程可能一直看不到 true -> false
static boolean running = true;
public static void main(String[] args) throws Exception {
    new Thread(() -> { while (running) { } }).start();
    Thread.sleep(100);
    running = false;  // 子线程可能永远不退出
}
```

**正确：** 用 volatile 保证写后对其它线程可见。

```java
// 正确：volatile 保证可见性，stop() 后子线程能及时看到
static volatile boolean running = true;
public static void main(String[] args) throws Exception {
    new Thread(() -> { while (running) { } }).start();
    Thread.sleep(100);
    running = false;  // 子线程能退出
}
```

> **面试怎么说**：无 volatile 时，写线程可能只更新了工作内存/缓存，读线程一直读自己的副本；volatile 写会刷新到主内存并令其他线程副本失效，读时从主内存读，从而保证可见性。

------

## 🧳 5. final 语义

### 📘 final 域的内存语义

- **写 final 域**：构造函数中对 final 域的写入，与随后把该对象引用赋值给其他变量，这两个操作之间不能重排序
- **读 final 域**：初次读一个包含 final 域的对象的引用，与随后初次读这个 final 域，这两个操作之间不能重排序

### 📘 final 域的可见性

- 一旦对象被正确构造（构造函数返回），其 final 域的值对所有线程可见，无需同步
- 非 final 域不保证这点，可能看到默认值

```java
class FinalFieldExample {
    final int finalField;
    int nonFinalField;
    
    public FinalFieldExample() {
        finalField = 42;      // 写 final 域
        nonFinalField = 100;  // 写非 final 域
    }
}

// 线程 A 构造对象
FinalFieldExample obj = new FinalFieldExample();

// 线程 B 读取
System.out.println(obj.finalField);   // 一定是 42
System.out.println(obj.nonFinalField); // 可能是 0（默认值）
```

## 🧳 6. DCL（Double-Checked Locking）问题

### 📘 DCL 模式

```java
// 错误的 DCL 实现
class Singleton {
    private static Singleton instance;
    
    private Singleton() {}
    
    public static Singleton getInstance() {
        if (instance == null) {  // 第一次检查
            synchronized (Singleton.class) {
                if (instance == null) {  // 第二次检查
                    instance = new Singleton();  // 问题：指令重排
                }
            }
        }
        return instance;
    }
}
```

### 📘 问题分析

- **指令重排**：`instance = new Singleton()` 不是原子操作，可分解为：
  1. 分配内存
  2. 初始化对象
  3. 赋值给 instance
- **重排风险**：可能重排为 1 → 3 → 2，此时 instance 不为 null，但对象未初始化完成
- **后果**：其他线程看到非 null 的 instance，使用时可能发生异常

### 📘 解决方案

- **使用 volatile**：禁止指令重排，保证初始化完成后才赋值

```java
// 正确的 DCL 实现
class Singleton {
    private static volatile Singleton instance;  // 加 volatile
    
    private Singleton() {}
    
    public static Singleton getInstance() {
        if (instance == null) {
            synchronized (Singleton.class) {
                if (instance == null) {
                    instance = new Singleton();
                }
            }
        }
        return instance;
    }
}
```

> **面试怎么说**：DCL 加 volatile 是为了禁止指令重排，确保对象初始化完成后再赋值给 instance，避免其他线程看到半初始化的对象。

## 🧳 7. 深入理解 happens-before

### 📘 常见 happens-before 规则

1. **程序顺序规则**：在一个线程内，按照代码顺序，前面的操作 happens-before 后面的操作
2. **监视器锁规则**：unlock 操作 happens-before 后续对同一锁的 lock 操作
3. **volatile 变量规则**：对一个 volatile 变量的写操作 happens-before 后续对该变量的读操作
4. **线程启动规则**：Thread.start() 操作 happens-before 线程内的任何操作
5. **线程终止规则**：线程内的任何操作 happens-before 其他线程检测到该线程终止
6. **线程中断规则**：对线程 interrupt() 的调用 happens-before 被中断线程检测到中断事件
7. **传递性**：如果 A happens-before B，且 B happens-before C，则 A happens-before C

### 📘 happens-before 与可见性

- 若 A happens-before B，则 A 的操作结果对 B 可见
- 这是 JMM 保证的核心语义，volatile 和 synchronized 都通过建立 happens-before 关系来保证可见性

## 🧠 1. 双重检查锁（DCL）单例

- 懒汉单例中，若只 syn 方法则每次获取都加锁；若只判空再 new 则可能多实例。**DCL**：在 syn 块内外各判空一次，且单例引用用 **volatile**，避免指令重排导致“半初始化”对象被看到。

#### 🧑‍💻 示例代码

```java
private static volatile Singleton instance;
public static Singleton getInstance() {
    if (instance == null) {
        synchronized (Singleton.class) {
            if (instance == null) {
                instance = new Singleton();
            }
        }
    }
    return instance;
}
```

> 💡 若可接受类加载时初始化，**静态内部类**单例更简单且线程安全。

## 🧳 2. ThreadLocal

### 📘 作用

- 每个线程持有**本线程专属**的变量副本，线程隔离，无共享则无竞态。
- 典型用途：连接、Transaction、用户上下文等“线程级”数据。

### 📘 注意

- **内存泄漏**：线程池中线程长期存活，ThreadLocal 不 remove 会导致 entry 与 value 无法回收（key 为弱引用会回收，value 强引用可能一直存在）。使用完应 **remove()**。

| 用线程池代替裸 new Thread | 复用、控并发、管理生命周期   |

| ThreadLocal 用完 remove | 尤其在线程池中               |
| 不依赖线程优先级   | 仅作提示                       |
