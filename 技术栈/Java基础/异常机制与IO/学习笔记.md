## 1️⃣ 异常的定义与分类

### （1）异常的定义

**异常**（Exception）是程序运行过程中发生的、打断正常执行流程的**异常事件**。Java 通过 **Throwable** 及其子类对这类事件建模，并提供捕获、传递、封装等机制。

> 💡 **重点**：异常是一种**对象**，携带类型、消息、原因（cause）和堆栈信息，可用于日志与排查。

### （2）受检异常与未受检异常

| 类型 | 说明 | 是否强制处理 | 典型示例 |
| --- | --- | --- | --- |
| **受检异常**（Checked Exception） | 编译器要求必须处理或声明抛出 | 是（否则编译不通过） | `IOException`、`SQLException`、`ClassNotFoundException` |
| **未受检异常**（Unchecked Exception） | 继承自 `RuntimeException`，多表示逻辑错误 | 否 | `NullPointerException`、`IllegalArgumentException`、`ArrayIndexOutOfBoundsException` |

- **受检异常**：多与外部环境相关（文件、网络、DB），调用方应显式处理或继续 `throws`。
- **未受检异常**：多为编程错误（空指针、非法参数、越界），修复代码比到处 catch 更合适。

### （3）Error 与 Exception

| 类型 | 说明 | 是否建议捕获 | 典型示例 |
| --- | --- | --- | --- |
| **Error** | JVM 或系统级严重问题，程序通常无法恢复 | 一般不捕获 | `OutOfMemoryError`、`StackOverflowError`、`NoClassDefFoundError` |
| **Exception** | 程序可处理的异常，含受检与未受检 | 按需捕获或声明 | 见上表 |

> ⚠️ **注意**：捕获 `Error` 或 `Throwable` 容易掩盖严重问题，生产代码中应避免。

------

## 2️⃣ 异常类层次结构

### （1）类图

```
Throwable
  ├── Error
  │     ├── OutOfMemoryError
  │     ├── StackOverflowError
  │     └── ...
  └── Exception
        ├── IOException (checked)
        ├── SQLException (checked)
        ├── ClassNotFoundException (checked)
        └── RuntimeException (unchecked)
              ├── NullPointerException
              ├── IllegalArgumentException
              ├── ArrayIndexOutOfBoundsException
              └── ...
```

### （2）异常对象的创建与传递

- **创建**：`new IOException("文件不存在")`、`new IllegalArgumentException("id 不能为负")`。
- **传递**：方法内 `throw` 抛出后，沿调用栈向上传播，直到被某层 `catch` 或导致线程终止；若方法声明 `throws XxxException`，则调用者必须处理或继续声明。

## 3️⃣ 异常的处理

### （1）try-catch-finally

**语法与执行顺序**：

1. 执行 `try` 块；若无异常，跳过所有 `catch`，再执行 `finally`（若有）。
2. 若发生异常，匹配第一个类型相符的 `catch`，执行后同样会执行 `finally`。
3. `finally` 通常用于释放资源（关闭流、连接等），**即使 try/catch 中有 return 也会执行**（除非 JVM 退出）。

示例：

```java
try {
    // 可能发生异常的代码
} catch (IOException e) {
    // 处理 IO 异常
} catch (SQLException e) {
    // 处理 SQL 异常
} finally {
    // 始终执行，常用于关闭资源
}
```

> 🔹 **注意**：`catch` 顺序应**从子类到父类**，否则子类 catch 永远不可达，编译报错。

### （2）throw 与 throws

| 关键字 | 作用 | 位置 |
| --- | --- | --- |
| **throw** | 抛出一个异常对象 | 方法体内 |
| **throws** | 声明方法可能抛出的受检异常 | 方法签名 |

示例：

```java
// 抛出异常
throw new IOException("File not found");

// 声明抛出
public void readFile(String path) throws IOException {
    // ...
}
```

### （3）异常链与多重捕获

- **异常链**：包装原始异常，便于保留根因，便于日志与排查。

```java
try {
    // ...
} catch (IOException e) {
    throw new BusinessException("读取配置失败", e);  // 将 e 作为 cause
}
```

- **多重捕获**（Java 7+）：一个 `catch` 处理多种类型，且捕获变量为 effectively final。

```java
try {
    // ...
} catch (IOException | SQLException e) {
    log.error("操作失败", e);
}
```

## 4️⃣ 自定义异常

### （1）继承 Exception 或 RuntimeException

| 继承类 | 得到类型 | 使用场景 |
| --- | --- | --- |
| **Exception** | 受检异常 | 调用方必须处理的业务/外部异常 |
| **RuntimeException** | 未受检异常 | 参数不合法、状态错误等逻辑异常 |

示例：

```java
// 受检异常：调用方必须 try-catch 或 throws
public class ResourceNotFoundException extends Exception {
    public ResourceNotFoundException(String message) { super(message); }
    public ResourceNotFoundException(String message, Throwable cause) { super(message, cause); }
}

// 未受检异常：多用于参数校验、状态校验
public class InvalidParamException extends RuntimeException {
    public InvalidParamException(String message) { super(message); }
}
```

### （2）设计原则

- 命名清晰，见名知意（如 `DuplicateOrderException`）。
- 提供带 message、带 cause 的构造函数，便于链式传递与日志。
- 避免过度自定义：能复用 JDK 异常则复用（如 `IllegalArgumentException`）。

## 🎯 面试常见追问

| 面试官提问 | 回答思路 |
| ---------- | -------- |
| 受检异常和运行时异常的区别？ | 受检异常编译期检查、必须处理或声明；运行时异常继承 RuntimeException，不强制。受检多用于外部依赖异常，运行时多用于逻辑错误。 |
| finally 一定会执行吗？ | 一般会执行；除非 JVM 退出（如 System.exit）、线程被 kill、或 finally 前发生死循环/未捕获 Error。 |
| 为什么不建议 catch Exception 或 Throwable？ | 容易吞掉不该处理的异常（如 OOM、NPE），不利于定位；应尽量捕获具体类型，顶层再统一兜底并打日志。 |

------

### ⚠️ 常见坑与注意点

| 现象 / 易错点 | 原因 | 正确做法 / 怎么记 |
|---------------|------|-------------------|
| 用异常做正常流程控制（如「未找到」就抛异常再 catch 当分支） | 异常开销大、语义混乱、难维护 | 用 **Optional**、返回结果对象或 null，在业务层判断；异常只表示「异常情况」。 |
| catch 顺序写反（先父类后子类） | 子类 catch 永远不可达，编译报错 | **从子类到父类**：先 catch IOException，再 catch Exception。 |
| 捕获后不记录、直接吞掉 | 排查时无堆栈、无法定位 | 至少 **log.error("上下文", e)**；若重新抛出，用 **cause** 保留原异常。 |
| 捕获 Error 或 Throwable | 会吞掉 OOM、StackOverflow 等严重问题 | 生产代码避免 catch Error/Throwable；顶层兜底也只 catch Exception 并打日志后考虑退出或降级。 |

### ❌ 错误捕获 vs ✅ 正确捕获示例

```java
// 错误：吞掉异常，无日志，且 catch 过于宽泛
try {
    doSomething();
} catch (Exception e) {
    // 空块或只 e.printStackTrace()，生产可能关掉控制台
}

// 正确：捕获具体类型 + 日志带异常对象 + 必要时包装再抛
try {
    doSomething();
} catch (IOException e) {
    log.error("IO 操作失败, path={}", path, e);
    throw new BusinessException("操作失败", e);  // 保留 cause
} catch (IllegalArgumentException e) {
    log.warn("参数不合法: {}", e.getMessage());
    throw e;  // 或转换为统一业务异常
}
```

# 🌱 第二章：异常处理最佳实践

> **本章在整体中解决什么问题**：第一章讲了异常分类与基本处理；本章规范**何时抛、如何捕、如何记日志、如何避免性能坑**。学完本章后**第三、六章**会在 IO 场景中用到「资源关闭 + 异常」；面试常问「异常怎么设计」「日志怎么打」。

------

## 1️⃣ 异常的规范化处理

### （1）什么时候抛出异常

- **违反前置条件**：参数不合法 → 抛 `IllegalArgumentException` 或自定义未受检异常。
- **违反业务规则**：如重复下单、余额不足 → 抛业务异常（受检或未受检视团队约定）。
- **外部依赖失败**：文件不存在、网络超时、DB 异常 → 受检异常或包装为业务异常。

> 💡 **重点**：不要用异常做正常流程控制（如用异常表示「未找到」并作为分支），应返回 Optional 或结果对象。

### （2）如何选择异常类型

| 场景 | 推荐做法 |
| --- | --- |
| 参数/状态不合法 | `IllegalArgumentException`、`IllegalStateException` 或自定义 RuntimeException |
| 资源不存在 | 自定义 `XxxNotFoundException`（受检或未受检均可） |
| 外部 IO/DB/网络 | 保留原始受检异常或包装为业务异常并保留 cause |
| 需要调用方强制处理 | 受检异常或明确声明的业务受检异常 |

------

## 2️⃣ 捕获异常的最佳方式

### （1）捕获特定异常

- 优先捕获**具体类型**，避免 `catch (Exception e)` 吞掉其他异常。
- 顶层（如 Controller、统一异常处理器）可做兜底，记录日志并返回统一错误响应。

### （2）避免空 catch 与吞异常

不推荐：

```java
try {
    doSomething();
} catch (Exception e) {
    // 空实现，异常被吞掉
}
```

推荐：至少打日志，必要时再抛出或包装。

```java
try {
    doSomething();
} catch (IOException e) {
    log.warn("IO 操作失败，使用默认值", e);
    return getDefault();
}
```

### （3）捕获并重新抛出

需要转换或包装时，保留 **cause**，便于排查：

```java
try {
    parseConfig(path);
} catch (IOException e) {
    throw new ConfigException("配置解析失败: " + path, e);
}
```

------

## 3️⃣ 日志记录与异常处理

### （1）记录什么

- **异常对象**：`log.error("操作失败", e)`，不要只打 `e.getMessage()`，否则丢失堆栈。
- **上下文**：请求 ID、关键参数、操作类型，便于排查。

### （2）错误示例与正确示例

| 不推荐 | 推荐 |
| --- | --- |
| `log.error(e.getMessage())` | `log.error("描述信息", e)` |
| 多处重复打同一异常日志 | 在边界层（Controller/统一异常处理）打一次，内层可只抛不记 |
| 用 `printStackTrace()` 到控制台 | 用日志框架按级别输出到日志文件 |

------

## 4️⃣ 异常的性能与注意点

### （1）异常的开销

- 创建异常会**填充堆栈**，成本高于普通返回；**不要用异常做正常分支**。
- 在热点路径上避免频繁抛受检异常（如循环内每轮都抛），可改为返回错误码或 Optional。

### （2）建议

- 正常流程用返回值/Optional；异常仅用于「异常情况」。
- 需要频繁校验时（如参数校验），用校验工具或提前 return，减少不必要的 throw。

------

------

### ⚠️ 常见坑与注意点

| 现象 / 易错点 | 原因 | 正确做法 / 怎么记 |
|---------------|------|-------------------|
| 到处 throws Exception | 调用方无法区分具体异常，不利于处理 | 尽量声明**具体受检异常**（如 throws IOException），或包装为业务异常并保留 cause。 |
| 在循环或热路径里抛/捕异常 | 异常创建与栈收集有开销，影响性能 | 用条件判断提前返回或 Optional，避免用异常做分支；热路径内少抛少捕。 |
| 只打 e.getMessage() 不打堆栈 | 丢失堆栈信息，难以排查 | 使用 **log.error("msg", e)** 或 **log.error("msg", (Throwable) e)**，保证堆栈输出。 |

### ❌ 错误捕获 vs ✅ 正确捕获示例

```java
// 错误：只打 message，丢堆栈；且捕获过宽
catch (Exception e) {
    log.error("失败: " + e.getMessage());  // 无堆栈
}

// 正确：带异常对象打日志，保留堆栈；按类型分别处理
catch (IOException e) {
    log.error("读取文件失败, file={}", path, e);  // 第三个参数 e 会打出堆栈
    throw new ServiceException("读取失败", e);
}
```

# 🌱 第三章：Java IO 概述

> **本章在整体中解决什么问题**：建立 IO 流「字节/字符、输入/输出、常用类」的体系，为**第四、五章** NIO 与性能优化打基础；**第六章**会结合异常讲流关闭与 try-with-resources。学完本章能正确选用字节流/字符流、指定编码、使用缓冲流。

------

## 1️⃣ IO 流的基础概念

### （1）按数据单位：字节流与字符流

| 类型 | 单位 | 抽象基类 | 典型场景 |
| --- | --- | --- | --- |
| **字节流** | 字节（8 bit） | `InputStream` / `OutputStream` | 二进制文件、网络、图片、任意格式 |
| **字符流** | 字符（char，多与编码相关） | `Reader` / `Writer` | 文本文件、需要编码解码的场景 |

> 💡 **重点**：字符流底层仍是字节，但会做**编解码**（如 UTF-8）；处理文本时用字符流可避免乱码与手动编解码。

### （2）按方向：输入流与输出流

- **输入流**：`InputStream`、`Reader` — 从数据源读入程序。
- **输出流**：`OutputStream`、`Writer` — 从程序写出到数据源。

### （3）常用流类型概览

| 类型 | 说明 | 典型类 |
| --- | --- | --- |
| 文件流 | 基于文件的读写 | `FileInputStream`、`FileOutputStream`、`FileReader`、`FileWriter` |
| 缓冲流 | 带缓冲区，减少系统调用 | `BufferedInputStream`、`BufferedReader`、`BufferedWriter` |
| 数据流 | 读写基本类型与 String | `DataInputStream`、`DataOutputStream` |
| 转换流 | 字节↔字符，可指定编码 | `InputStreamReader`、`OutputStreamWriter` |
| 打印流 | 方便输出格式化和自动 flush | `PrintStream`、`PrintWriter` |

------

## 2️⃣ Java IO 类层次结构

### （1）字节流

```
InputStream (抽象)
  ├── FileInputStream
  ├── BufferedInputStream
  ├── DataInputStream
  └── ...
OutputStream (抽象)
  ├── FileOutputStream
  ├── BufferedOutputStream
  ├── DataOutputStream
  └── ...
```

### （2）字符流

```
Reader (抽象)
  ├── InputStreamReader (字节→字符，可指定 Charset)
  ├── FileReader (默认使用系统编码，建议用 InputStreamReader 指定 UTF-8)
  ├── BufferedReader
  └── ...
Writer (抽象)
  ├── OutputStreamWriter
  ├── FileWriter
  ├── BufferedWriter
  ├── PrintWriter
  └── ...
```

> 🔹 **注意**：`FileReader`/`FileWriter` 使用系统默认编码，跨环境易乱码；**推荐**用 `InputStreamReader`/`OutputStreamWriter` + `Charset.forName("UTF-8")` 或 JDK 7+ 的 `StandardCharsets.UTF_8`。

------

## 3️⃣ 基本 IO 操作

### （1）文本文件读取（带编码与缓冲）

示例：按行读取 UTF-8 文本文件。

```java
try (BufferedReader reader = new BufferedReader(
        new InputStreamReader(new FileInputStream("file.txt"), StandardCharsets.UTF_8))) {
    String line;
    while ((line = reader.readLine()) != null) {
        System.out.println(line);
    }
}
```

### （2）文本文件写入

示例：写入 UTF-8 文本。

```java
try (BufferedWriter writer = new BufferedWriter(
        new OutputStreamWriter(new FileOutputStream("out.txt"), StandardCharsets.UTF_8))) {
    writer.write("Hello");
    writer.newLine();
    writer.write("世界");
}
```

### （3）字节流复制文件

示例：使用缓冲字节流复制。

```java
try (InputStream in = new BufferedInputStream(new FileInputStream("src.dat"));
     OutputStream out = new BufferedOutputStream(new FileOutputStream("dest.dat"))) {
    byte[] buf = new byte[8192];
    int n;
    while ((n = in.read(buf)) != -1) {
        out.write(buf, 0, n);
    }
}
```

### （4）常见错误

| 不推荐 | 原因 |
| --- | --- |
| 不指定编码直接用 FileReader/FileWriter | 跨环境乱码 |
| 忘记关闭流或未用 try-with-resources | 资源泄漏；见第六章 |
| 用字节流逐字节 read/write 大文件 | 性能差，应使用缓冲或 NIO |

------

| 缓冲流 | BufferedXxx、减少系统调用 | 文件读写、网络读写 |
| 转换流 | InputStreamReader、OutputStreamWriter、Charset | 指定编码、避免乱码 |
| 基本操作 | read/write、按行读、按块复制 | 配置文件、日志、文件拷贝 |

------

**学习要点**：

- 能区分字节流与字符流及适用场景；能说明为什么文本推荐指定编码。
- 能写出「带缓冲、带编码、正确关闭」的读/写/复制示例。
- 知道 FileReader/FileWriter 的编码陷阱，以及用 NIO.2（Path/Files）做简单读写的替代方式（见第四章）。

------

### 本章小结表（速查）

| 知识点 | 面试关键词 | 实际应用 |
| ------ | ---------- | -------- |
| 字节流 vs 字符流 | InputStream/OutputStream、Reader/Writer、编码 | 文本用字符流+UTF-8，二进制用字节流 |
| 缓冲流 | BufferedXxx、减少系统调用 | 文件/网络读写默认加缓冲 |
| 转换流与编码 | InputStreamReader、OutputStreamWriter、Charset | 指定 UTF-8，避免 FileReader/FileWriter 默认编码乱码 |

# 🌱 第四章：高级 IO 与 NIO

> **本章在整体中解决什么问题**：第三章讲了传统流式 IO；本章介绍 **NIO** 的模型、Channel/Buffer/Selector 及与传统 IO 的对比，便于理解高并发、网络编程与后续 Netty。学完本章后**第五章**会讲 NIO 在性能优化（零拷贝、内存映射）中的应用；面试常问 BIO vs NIO、Selector 作用。

------

## 1️⃣ NIO 的引入与优势

### （1）传统 IO 与 NIO 对比

| 维度 | 传统 IO（BIO） | NIO |
| --- | --- | --- |
| 模型 | 面向流（Stream）、阻塞 | 面向通道（Channel）、缓冲区（Buffer），可非阻塞 |
| 读写方式 | 流式，单向 | Channel 可读可写，Buffer 读写分离 |
| 多路复用 | 多线程一连接（或线程池） | 单线程多 Channel（Selector） |
| 典型场景 | 简单文件、小并发 | 大并发、网络服务器、大文件 |

### （2）NIO 核心组件

| 组件 | 说明 |
| --- | --- |
| **Channel** | 双向通道，替代流的单向；如 `FileChannel`、`SocketChannel`、`ServerSocketChannel` |
| **Buffer** | 一块内存，读写前 flip/clear 等，替代流式 read/write |
| **Selector** | 多路复用器，一个线程监听多个 Channel 的 IO 事件（读/写/连接/接受） |

### （3）Buffer 基本用法

```java
ByteBuffer buf = ByteBuffer.allocate(1024);
channel.read(buf);   // 写入 buffer
buf.flip();          // 切换为读模式
channel.write(buf);  // 从 buffer 读出
buf.clear();         // 清空，可再次写入
```

### 📍 BIO vs NIO 模型简图（面试可画）

```
BIO（阻塞 IO）：
  线程1 ──read()阻塞──→ 连接1
  线程2 ──read()阻塞──→ 连接2
  线程3 ──read()阻塞──→ 连接3
  … 一连接一线程，高并发时线程数爆炸

NIO（多路复用）：
              ┌── 连接1 ──→ Channel
              ├── 连接2 ──→ Channel
  Selector ───┼── 连接3 ──→ Channel   ← 单线程轮询哪些 Channel 就绪
              └── …
  一个线程通过 Selector 监听多个 Channel，就绪的才读写，非阻塞
```

> **面试怎么说**：BIO 是面向流、阻塞的，一个连接一个线程；NIO 是面向 Channel 和 Buffer，可设成非阻塞，用 **Selector** 一个线程监听多个 Channel 的读/写/连接事件，实现多路复用，适合高并发。NIO 三件套：Channel、Buffer、Selector。

------

## 2️⃣ 文件与目录操作（NIO.2）

JDK 7 引入 **NIO.2**（`java.nio.file`），提供 **Path**、**Files**、**Paths**，简化文件与目录操作。

### （1）Path 与 Files

- **Path**：表示路径，替代 `File`；通过 `Paths.get("...")` 或 `Path.of("...")`（JDK 11+）获取。
- **Files**：静态方法读写、复制、移动、删除、遍历等。

示例：读全部文本、写文本。

```java
Path path = Paths.get("config.txt");
String content = Files.readString(path, StandardCharsets.UTF_8);  // JDK 11+

Files.writeString(path, "hello", StandardCharsets.UTF_8);  // 覆盖写入
Files.write(path, lines, StandardCharsets.UTF_8, StandardOpenOption.APPEND);  // 追加
```

### （2）文件复制与遍历

```java
Files.copy(source, target, StandardCopyOption.REPLACE_EXISTING);
Files.walkFileTree(startPath, visitor);  // 递归遍历
```

------

## 3️⃣ 网络通信与多路复用（简述）

- **SocketChannel / ServerSocketChannel**：可设置为非阻塞，配合 **Selector** 实现单线程多连接（多路复用）。
- **DatagramChannel**：UDP。  
详细网络编程与 Netty 可单独成章，此处仅建立「NIO = Channel + Buffer + Selector」的概念。

| ------ | ---------- | -------- |

| NIO 三件套 | Channel、Buffer、Selector | 大文件、网络服务器、多路复用 |

### ⚠️ 常见坑与注意点

| 现象 / 易错点 | 原因 | 正确做法 / 怎么记 |
|---------------|------|-------------------|
| Buffer 读完后未 flip 就写、或写完后未 clear 就读 | Buffer 有 position/limit/capacity，读写模式切换靠 flip/clear | **读前 flip**（limit=position, position=0），**写前 clear**（position=0, limit=capacity）。 |
| 把 NIO 当银弹到处用 | 简单文件、低并发用传统流更简单，NIO 代码更复杂 | 简单 IO 用流或 NIO.2（Path/Files）；高并发、多连接用 NIO Selector 或 Netty。 |
| DirectBuffer 或 MappedByteBuffer 未及时释放 | 占堆外内存，回收依赖 GC，长期持有导致占用高 | 用完后尽快解除引用，不要长时间持有；大块映射可分段映射、用完即关。 |

# 🌱 第五章：IO 性能优化

> **本章在整体中解决什么问题**：在第三、四章「会用流与 NIO」基础上，本章讲**性能优化**——缓冲、NIO 零拷贝与内存映射、多线程与异步。与第四章 NIO、第六章资源关闭结合，可形成「正确 + 高效 + 安全关闭」的完整实践。

------

## 1️⃣ 缓冲区与缓冲流

### （1）为什么用缓冲

- 减少系统调用次数：每次 read/write 都可能触发内核调用；先读入内存缓冲区，再批量读写，可显著降低调用次数。
- **BufferedInputStream / BufferedOutputStream**：包装字节流，默认 8KB 缓冲。
- **BufferedReader / BufferedWriter**：包装字符流，适合按行或大块文本。

### （2）缓冲大小建议

- 默认 8KB 对多数场景够用；大文件或高吞吐可适当增大（如 32KB～64KB），需结合实测。

------

## 2️⃣ 使用 NIO 提高 IO 性能

### （1）FileChannel 与直接缓冲区

- **FileChannel**：可替代部分 `FileInputStream`/`FileOutputStream`，支持 **transferTo/transferFrom**（零拷贝思路），减少用户态拷贝。
- **直接缓冲区**（DirectBuffer）：堆外内存，与内核/设备交互时可少一次拷贝，适合大块 IO；创建和回收成本较高，适合复用或大文件。

### （2）内存映射文件（Memory Mapped Files）

- **FileChannel.map**：把文件一段映射到进程地址空间，适合**大文件随机读**或**超大文件顺序读**。
- 注意：映射区域受 OS 与 JVM 限制；关闭通道前不要长期持有 MappedByteBuffer 引用，以免影响回收。

------

## 3️⃣ IO 与多线程、异步

- **多线程**：大文件可分段由多线程读/写，注意顺序与合并；或使用线程池处理大量小文件。
- **异步 IO**：JDK 7 的 **AsynchronousFileChannel** 提供异步读写的回调或 Future 方式，适合高并发、低阻塞场景。

------

| 知识点 | 面试关键词 | 实际应用 |
| ------ | ---------- | -------- |
| 缓冲流 | BufferedXxx、减少系统调用 | 文件/网络读写默认加一层缓冲 |
| NIO 性能 | FileChannel、transferTo、DirectBuffer、内存映射 | 大文件拷贝、大文件随机/顺序读 |
| 多线程与异步 | 分段读写、AsynchronousFileChannel | 大文件处理、高并发 IO |

------

**学习要点**：

- 理解「缓冲」降低系统调用的原理；能选用 BufferedXxx 并知道默认大小量级。
- 知道 NIO 在「零拷贝」「直接缓冲区」「内存映射」上的性能优势及适用场景。
- 了解多线程与异步 IO 的适用边界（大文件、高并发）。

------

### ⚠️ 注意点

| 注意点 | 说明 |
|--------|------|
| 缓冲流默认 8KB | 多数场景够用；大文件或高吞吐可调大（如 32KB～64KB），需实测。 |
| transferTo/transferFrom | 减少用户态拷贝，适合大文件复制；底层可能用 sendfile 等零拷贝。 |

# 🌱 第六章：Java IO 与异常机制结合

> **本章在整体中解决什么问题**：把**异常与 IO** 串起来——IO 中常见异常、资源关闭规范、**try-with-resources** 的用法。第一章的异常处理、第二章的实践与本章的「流必须关闭」形成闭环；面试常问「流为什么要关」「try-with-resources 和 finally close 区别」。

------

## 1️⃣ IO 操作中的异常与资源管理

### （1）常见 IO 异常

| 异常 | 说明 |
| --- | --- |
| **FileNotFoundException** | 文件不存在或路径不可访问（继承自 IOException） |
| **IOException** | 通用 IO 错误（读写出错、流关闭后仍操作等） |
| **UnsupportedEncodingException** | 不支持的字符编码（继承自 IOException） |

### （2）资源关闭的规范

- 所有实现 **AutoCloseable** 的流、Channel、连接，都必须**在 finally 中关闭**，或使用 **try-with-resources**，否则会产生资源泄漏（文件句柄、连接数等）。

不推荐：只 try-catch 不关闭流。

```java
InputStream in = new FileInputStream("a.txt");
// 若中间抛异常，in 未关闭
in.read();
in.close();
```

推荐：try-with-resources（见下节）。

------

## 2️⃣ try-with-resources 管理流

### （1）自动资源管理（ARM）

**try-with-resources**（JDK 7+）：在 try 的括号中声明实现了 **AutoCloseable** 的资源，块结束时**自动调用 close()**，若有异常会先关闭再抛出，且可保留 **suppressed 异常**。

语法：

```java
try (ResourceType res = ...) {
    // 使用 res
}  // 自动 res.close()，若有异常会 suppress 到主异常
```

### （2）在 IO 流中的应用

示例：多资源、带 catch。

```java
try (InputStream in = new FileInputStream("a.txt");
     OutputStream out = new FileOutputStream("b.txt")) {
    byte[] buf = new byte[8192];
    int n;
    while ((n = in.read(buf)) != -1) {
        out.write(buf, 0, n);
    }
} catch (IOException e) {
    log.error("复制失败", e);
}
```

> 💡 **重点**：从 JDK 9 起，可以在 try 外先声明变量再在 try 中引用（变量要为 effectively final），便于在 catch 中区分不同资源异常。

------

# ✅ 本章小结

| 知识点 | 面试关键词 | 实际应用 |
| ------ | ---------- | -------- |
| IO 常见异常 | FileNotFoundException、IOException | 文件不存在、编码错误、流关闭后操作 |
| 资源关闭 | AutoCloseable、finally、try-with-resources | 所有流、Channel、连接必须关闭 |
| try-with-resources | 自动 close、suppressed 异常 | 替代手写 finally close，避免泄漏 |

------

**学习要点**：

- 能列举 IO 中常见异常及含义。
- 牢固掌握 try-with-resources 的写法与执行顺序（先关资源，再抛主异常；suppressed 可被 getSuppressed() 拿到）。
- 与第二章结合：关闭资源时若 close 也抛异常，会作为主异常的 suppressed，在日志中应关注完整堆栈。

------

## 🎯 面试常见追问

| 面试官提问 | 回答思路 |
| ---------- | -------- |
| IO 流为什么要关闭？不关会怎样？ | 释放文件句柄、连接等系统资源；不关会导致句柄泄漏，高并发下「Too many open files」或连接耗尽。 |
| try-with-resources 和 finally 里 close 的区别？ | try-with-resources 自动 close，且若 try 和 close 都抛异常，主异常会带 suppressed（close 的异常），不会丢；finally 里手写 close 要自己处理 close 抛出的异常。 |

------

### ⚠️ 注意点

| 注意点 | 说明 |
|--------|------|
| 实现 AutoCloseable 的都要关 | 流、Channel、Socket、Connection 等；用 try-with-resources 或 finally，避免泄漏。 |
| close 抛异常时 | try-with-resources 会将其作为主异常的 suppressed，getSuppressed() 可拿到；日志要打完整堆栈。 |
| JDK 9+ try 可引用外部变量 | 变量需 effectively final，便于在 catch 中区分是哪个资源出错。 |
