## 1️⃣ 异常的定义与分类

### （1）异常的定义

**异常**（Exception）是程序运行过程中发生的、打断正常执行流程的**异常事件**。Java 通过 **Throwable** 及其子类对这类事件建模，并提供捕获、传递、封装等机制。

> 💡 **重点**：异常是一种**对象**，携带类型、消息、原因（cause）和堆栈信息，可用于日志与排查。

### （2）受检异常与未受检异常

| 类型 | 说明 | 是否强制处理 | 典型示例 |
| --- | --- | --- | --- |
| **受检异常**（Checked Exception） | 编译器要求必须处理或声明抛出 | 是（否则编译不通过） | `IOException`、`SQLException`、`ClassNotFoundException` |
| **未受检异常**（Unchecked Exception） | 继承自 `RuntimeException`，多表示逻辑错误 | 否 | `NullPointerException`、`IllegalArgumentException`、`ArrayIndexOutOfBoundsException` |

- **受检异常**：多与外部环境相关（文件、网络、DB），调用方应显式处理或继续 `throws`。
- **未受检异常**：多为编程错误（空指针、非法参数、越界），修复代码比到处 catch 更合适。

### （3）Error 与 Exception

| 类型 | 说明 | 是否建议捕获 | 典型示例 |
| --- | --- | --- | --- |
| **Error** | JVM 或系统级严重问题，程序通常无法恢复 | 一般不捕获 | `OutOfMemoryError`、`StackOverflowError`、`NoClassDefFoundError` |
| **Exception** | 程序可处理的异常，含受检与未受检 | 按需捕获或声明 | 见上表 |

> ⚠️ **注意**：捕获 `Error` 或 `Throwable` 容易掩盖严重问题，生产代码中应避免。

## 2️⃣ 异常类层次结构

### （1）类图

```
Throwable
  ├── Error
  │     ├── OutOfMemoryError
  │     ├── StackOverflowError
  │     └── ...
  └── Exception
        ├── IOException (checked)
        ├── SQLException (checked)
        ├── ClassNotFoundException (checked)
        └── RuntimeException (unchecked)
              ├── NullPointerException
              ├── IllegalArgumentException
              ├── ArrayIndexOutOfBoundsException
              └── ...
```

### （2）异常对象的创建与传递

- **创建**：`new IOException("文件不存在")`、`new IllegalArgumentException("id 不能为负")`。
- **传递**：方法内 `throw` 抛出后，沿调用栈向上传播，直到被某层 `catch` 或导致线程终止；若方法声明 `throws XxxException`，则调用者必须处理或继续声明。

## 3️⃣ 异常的处理

### （1）try-catch-finally

**语法与执行顺序**：

1. 执行 `try` 块；若无异常，跳过所有 `catch`，再执行 `finally`（若有）。
2. 若发生异常，匹配第一个类型相符的 `catch`，执行后同样会执行 `finally`。
3. `finally` 通常用于释放资源（关闭流、连接等），**即使 try/catch 中有 return 也会执行**（除非 JVM 退出）。

示例：

```java
try {
    // 可能发生异常的代码
} catch (IOException e) {
    // 处理 IO 异常
} catch (SQLException e) {
    // 处理 SQL 异常
} finally {
    // 始终执行，常用于关闭资源
}
```

> 🔹 **注意**：`catch` 顺序应**从子类到父类**，否则子类 catch 永远不可达，编译报错。

### （2）throw 与 throws

| 关键字 | 作用 | 位置 |
| --- | --- | --- |
| **throw** | 抛出一个异常对象 | 方法体内 |
| **throws** | 声明方法可能抛出的受检异常 | 方法签名 |

示例：

```java
// 抛出异常
throw new IOException("File not found");

// 声明抛出
public void readFile(String path) throws IOException {
    // ...
}
```

### （3）异常链与多重捕获

- **异常链**：包装原始异常，便于保留根因，便于日志与排查。

```java
try {
    // ...
} catch (IOException e) {
    throw new BusinessException("读取配置失败", e);  // 将 e 作为 cause
}
```

- **多重捕获**（Java 7+）：一个 `catch` 处理多种类型，且捕获变量为 effectively final。

```java
try {
    // ...
} catch (IOException | SQLException e) {
    log.error("操作失败", e);
}
```

## 4️⃣ 自定义异常

### （1）继承 Exception 或 RuntimeException

| 继承类 | 得到类型 | 使用场景 |
| --- | --- | --- |
| **Exception** | 受检异常 | 调用方必须处理的业务/外部异常 |
| **RuntimeException** | 未受检异常 | 参数不合法、状态错误等逻辑异常 |

### （2）自定义异常的完整示例

```java
// 受检异常：调用方必须 try-catch 或 throws
public class ResourceNotFoundException extends Exception {
    private final String resourceId;
    private final String resourceType;
    
    public ResourceNotFoundException(String message) {
        super(message);
        this.resourceId = null;
        this.resourceType = null;
    }
    
    public ResourceNotFoundException(String message, Throwable cause) {
        super(message, cause);
        this.resourceId = null;
        this.resourceType = null;
    }
    
    public ResourceNotFoundException(String resourceType, String resourceId) {
        super(resourceType + " not found: " + resourceId);
        this.resourceId = resourceId;
        this.resourceType = resourceType;
    }
    
    // Getters
    public String getResourceId() { return resourceId; }
    public String getResourceType() { return resourceType; }
}

// 未受检异常：多用于参数校验、状态校验
public class InvalidParamException extends RuntimeException {
    private final String paramName;
    private final Object paramValue;
    
    public InvalidParamException(String message) {
        super(message);
        this.paramName = null;
        this.paramValue = null;
    }
    
    public InvalidParamException(String paramName, Object paramValue, String message) {
        super(message + ": " + paramName + " = " + paramValue);
        this.paramName = paramName;
        this.paramValue = paramValue;
    }
    
    // Getters
    public String getParamName() { return paramName; }
    public Object getParamValue() { return paramValue; }
}

// 业务异常基类
public class BusinessException extends Exception {
    private final int errorCode;
    
    public BusinessException(int errorCode, String message) {
        super(message);
        this.errorCode = errorCode;
    }
    
    public BusinessException(int errorCode, String message, Throwable cause) {
        super(message, cause);
        this.errorCode = errorCode;
    }
    
    public int getErrorCode() { return errorCode; }
}

// 具体业务异常
public class OrderNotFoundException extends BusinessException {
    public OrderNotFoundException(String orderId) {
        super(404, "Order not found: " + orderId);
    }
}

public class InsufficientBalanceException extends BusinessException {
    public InsufficientBalanceException(double balance, double required) {
        super(400, "Insufficient balance: " + balance + " < " + required);
    }
}
```

### （3）设计原则

1. **命名清晰**：见名知意，如 `DuplicateOrderException`、`InsufficientBalanceException`
2. **提供丰富的构造函数**：支持带 message、cause、业务参数的构造器
3. **包含业务上下文**：如资源ID、参数名、错误码等，便于排查
4. **避免过度自定义**：能复用 JDK 异常则复用（如 `IllegalArgumentException`、`IllegalStateException`）
5. **统一异常体系**：建立业务异常基类，统一错误码和处理逻辑

### （4）自定义异常的使用场景

| 场景 | 推荐异常类型 | 示例 |
| --- | --- | --- |
| 参数校验失败 | RuntimeException | `InvalidParamException` |
| 业务规则违反 | 受检或未受检 | `InsufficientBalanceException` |
| 资源不存在 | 受检或未受检 | `ResourceNotFoundException` |
| 外部依赖失败 | 受检异常 | `ThirdPartyServiceException` |
| 系统内部错误 | RuntimeException | `SystemInternalException` |

### （5）异常链的正确使用

```java
// 正确：包装原始异常，保留 cause
try {
    callThirdPartyService();
} catch (ThirdPartyApiException e) {
    // 包装为业务异常，保留原始异常作为 cause
    throw new BusinessException(500, "Third party service failed", e);
}

// 错误：丢失原始异常
try {
    callThirdPartyService();
} catch (ThirdPartyApiException e) {
    // 错误：只传递 message，丢失堆栈
    throw new BusinessException(500, "Third party service failed: " + e.getMessage());
}
```

## 🎯 面试常见追问

| 面试官提问 | 回答思路 |
| ---------- | -------- |
| 受检异常和运行时异常的区别？ | 受检异常编译期检查、必须处理或声明；运行时异常继承 RuntimeException，不强制。受检多用于外部依赖异常，运行时多用于逻辑错误。 |
| finally 一定会执行吗？ | 一般会执行；除非 JVM 退出（如 System.exit）、线程被 kill、或 finally 前发生死循环/未捕获 Error。 |
| 为什么不建议 catch Exception 或 Throwable？ | 容易吞掉不该处理的异常（如 OOM、NPE），不利于定位；应尽量捕获具体类型，顶层再统一兜底并打日志。 |

------

### ⚠️ 常见坑与注意点

| 现象 / 易错点 | 原因 | 正确做法 / 怎么记 |
|---------------|------|-------------------|
| 用异常做正常流程控制（如「未找到」就抛异常再 catch 当分支） | 异常开销大、语义混乱、难维护 | 用 **Optional**、返回结果对象或 null，在业务层判断；异常只表示「异常情况」。 |
| catch 顺序写反（先父类后子类） | 子类 catch 永远不可达，编译报错 | **从子类到父类**：先 catch IOException，再 catch Exception。 |
| 捕获后不记录、直接吞掉 | 排查时无堆栈、无法定位 | 至少 **log.error("上下文", e)**；若重新抛出，用 **cause** 保留原异常。 |
| 捕获 Error 或 Throwable | 会吞掉 OOM、StackOverflow 等严重问题 | 生产代码避免 catch Error/Throwable；顶层兜底也只 catch Exception 并打日志后考虑退出或降级。 |

### ❌ 错误捕获 vs ✅ 正确捕获示例

```java
// 错误：吞掉异常，无日志，且 catch 过于宽泛
try {
    doSomething();
} catch (Exception e) {
    // 空块或只 e.printStackTrace()，生产可能关掉控制台
}

// 正确：捕获具体类型 + 日志带异常对象 + 必要时包装再抛
try {
    doSomething();
} catch (IOException e) {
    log.error("IO 操作失败, path={}", path, e);
    throw new BusinessException("操作失败", e);  // 保留 cause
} catch (IllegalArgumentException e) {
    log.warn("参数不合法: {}", e.getMessage());
    throw e;  // 或转换为统一业务异常
}
```

# 🌱 第二章：异常处理最佳实践

> **本章在整体中解决什么问题**：第一章讲了异常分类与基本处理；本章规范**何时抛、如何捕、如何记日志、如何避免性能坑**。学完本章后**第三、六章**会在 IO 场景中用到「资源关闭 + 异常」；面试常问「异常怎么设计」「日志怎么打」。

------

## 1️⃣ 异常的规范化处理

### （1）什么时候抛出异常

- **违反前置条件**：参数不合法 → 抛 `IllegalArgumentException` 或自定义未受检异常。
- **违反业务规则**：如重复下单、余额不足 → 抛业务异常（受检或未受检视团队约定）。
- **外部依赖失败**：文件不存在、网络超时、DB 异常 → 受检异常或包装为业务异常。

> 💡 **重点**：不要用异常做正常流程控制（如用异常表示「未找到」并作为分支），应返回 Optional 或结果对象。

### （2）如何选择异常类型

| 场景 | 推荐做法 |
| --- | --- |
| 参数/状态不合法 | `IllegalArgumentException`、`IllegalStateException` 或自定义 RuntimeException |
| 资源不存在 | 自定义 `XxxNotFoundException`（受检或未受检均可） |
| 外部 IO/DB/网络 | 保留原始受检异常或包装为业务异常并保留 cause |
| 需要调用方强制处理 | 受检异常或明确声明的业务受检异常 |

------

## 2️⃣ 捕获异常的最佳方式

### （1）捕获特定异常

- 优先捕获**具体类型**，避免 `catch (Exception e)` 吞掉其他异常。
- 顶层（如 Controller、统一异常处理器）可做兜底，记录日志并返回统一错误响应。

### （2）避免空 catch 与吞异常

不推荐：

```java
try {
    doSomething();
} catch (Exception e) {
    // 空实现，异常被吞掉
}
```

推荐：至少打日志，必要时再抛出或包装。

```java
try {
    doSomething();
} catch (IOException e) {
    log.warn("IO 操作失败，使用默认值", e);
    return getDefault();
}
```

### （3）捕获并重新抛出

需要转换或包装时，保留 **cause**，便于排查：

```java
try {
    parseConfig(path);
} catch (IOException e) {
    throw new ConfigException("配置解析失败: " + path, e);
}
```

------

## 3️⃣ 日志记录与异常处理

### （1）记录什么

- **异常对象**：`log.error("操作失败", e)`，不要只打 `e.getMessage()`，否则丢失堆栈。
- **上下文**：请求 ID、关键参数、操作类型，便于排查。

### （2）错误示例与正确示例

| 不推荐 | 推荐 |
| --- | --- |
| `log.error(e.getMessage())` | `log.error("描述信息", e)` |
| 多处重复打同一异常日志 | 在边界层（Controller/统一异常处理）打一次，内层可只抛不记 |
| 用 `printStackTrace()` 到控制台 | 用日志框架按级别输出到日志文件 |

## 4️⃣ 异常的性能与注意点

### （1）异常的开销

- **创建异常的成本**：创建异常会填充堆栈，涉及方法调用栈的遍历和字符串拼接，成本远高于普通返回
- **异常处理的成本**：try-catch 会影响 JVM 优化（如内联），在热点路径上可能导致性能下降
- **堆栈填充**：异常对象创建时会捕获当前线程的完整调用栈，这是主要性能开销来源

### （2）性能优化建议

1. **避免用异常做正常流程控制**
   - 错误：用异常表示「未找到」「空结果」等正常业务场景
   - 正确：用 `Optional`、返回结果对象或 `null`，在业务层判断

2. **热点路径减少异常使用**
   - 循环、高频调用的方法中避免抛受检异常
   - 可改为返回错误码、状态对象或使用校验工具

3. **合理使用异常类型**
   - 性能敏感场景：优先使用 `RuntimeException`（减少编译期检查开销）
   - 业务场景：根据需要选择受检或未受检异常

4. **异常对象重用**（仅适用于无状态异常）
   ```java
   // 可重用的异常对象（无状态）
   public class CommonExceptions {
       public static final IllegalArgumentException ILLEGAL_ARGUMENT = 
           new IllegalArgumentException("Illegal argument");
       public static final IllegalStateException ILLEGAL_STATE = 
           new IllegalStateException("Illegal state");
   }
   
   // 使用
   if (condition) {
       throw CommonExceptions.ILLEGAL_ARGUMENT;
   }
   ```

5. **异常粒度控制**
   - 粗粒度：顶层统一捕获，减少 try-catch 嵌套
   - 细粒度：关键业务逻辑精确捕获，便于针对性处理

### （3）异常性能测试示例

```java
public class ExceptionPerformanceTest {
    private static final int ITERATIONS = 10_000_000;
    
    public static void main(String[] args) {
        testNormalReturn();
        testExceptionThrow();
    }
    
    private static void testNormalReturn() {
        long start = System.nanoTime();
        int count = 0;
        for (int i = 0; i < ITERATIONS; i++) {
            if (i % 2 == 0) {
                count++;
            }
        }
        long end = System.nanoTime();
        System.out.println("Normal return: " + (end - start) / 1_000_000 + "ms, count: " + count);
    }
    
    private static void testExceptionThrow() {
        long start = System.nanoTime();
        int count = 0;
        for (int i = 0; i < ITERATIONS; i++) {
            try {
                if (i % 2 == 0) {
                    throw new RuntimeException();
                }
            } catch (RuntimeException e) {
                count++;
            }
        }
        long end = System.nanoTime();
        System.out.println("Exception throw: " + (end - start) / 1_000_000 + "ms, count: " + count);
    }
}
```

### （4）生产环境中的异常处理策略

| 场景 | 策略 | 示例 |
| --- | --- | --- |
| 高频接口 | 使用 Optional 或状态对象，避免异常 | `return Optional.empty()` 代替 `throw NotFoundException` |
| 业务逻辑 | 合理使用异常，保留业务上下文 | `throw new BusinessException("订单创建失败", e)` |
| 外部依赖 | 包装异常，统一处理 | `try { httpClient.execute() } catch (IOException e) { throw new ServiceException("HTTP请求失败", e) }` |
| 系统启动 | 捕获异常，确保启动成功 | 配置加载失败时使用默认配置，记录警告 |
| 批处理任务 | 捕获异常，继续处理其他任务 | 单个任务失败不影响整体流程 |

### ⚠️ 常见坑与注意点

| 现象 / 易错点 | 原因 | 正确做法 / 怎么记 |
|---------------|------|-------------------|
| 到处 throws Exception | 调用方无法区分具体异常，不利于处理 | 尽量声明**具体受检异常**（如 throws IOException），或包装为业务异常并保留 cause。 |
| 在循环或热路径里抛/捕异常 | 异常创建与栈收集有开销，影响性能 | 用条件判断提前返回或 Optional，避免用异常做分支；热路径内少抛少捕。 |
| 只打 e.getMessage() 不打堆栈 | 丢失堆栈信息，难以排查 | 使用 **log.error("msg", e)** 或 **log.error("msg", (Throwable) e)**，保证堆栈输出。 |

### ❌ 错误捕获 vs ✅ 正确捕获示例

```java
// 错误：只打 message，丢堆栈；且捕获过宽
catch (Exception e) {
    log.error("失败: " + e.getMessage());  // 无堆栈
}

// 正确：带异常对象打日志，保留堆栈；按类型分别处理
catch (IOException e) {
    log.error("读取文件失败, file={}", path, e);  // 第三个参数 e 会打出堆栈
    throw new ServiceException("读取失败", e);
}
```

# 🌱 第三章：Java IO 概述

> **本章在整体中解决什么问题**：建立 IO 流「字节/字符、输入/输出、常用类」的体系，为**第四、五章** NIO 与性能优化打基础；**第六章**会结合异常讲流关闭与 try-with-resources。学完本章能正确选用字节流/字符流、指定编码、使用缓冲流。

## 1️⃣ IO 流的基础概念

### （1）按数据单位：字节流与字符流

| 类型 | 单位 | 抽象基类 | 典型场景 |
| --- | --- | --- | --- |
| **字节流** | 字节（8 bit） | `InputStream` / `OutputStream` | 二进制文件、网络、图片、任意格式 |

- **输入流**：`InputStream`、`Reader` — 从数据源读入程序。
- **输出流**：`OutputStream`、`Writer` — 从程序写出到数据源。

### （3）常用流类型概览

| 类型 | 说明 | 典型类 |
| --- | --- | --- |
| 文件流 | 基于文件的读写 | `FileInputStream`、`FileOutputStream`、`FileReader`、`FileWriter` |
| 缓冲流 | 带缓冲区，减少系统调用 | `BufferedInputStream`、`BufferedReader`、`BufferedWriter` |
| 数据流 | 读写基本类型与 String | `DataInputStream`、`DataOutputStream` |
| 转换流 | 字节↔字符，可指定编码 | `InputStreamReader`、`OutputStreamWriter` |
| 打印流 | 方便输出格式化和自动 flush | `PrintStream`、`PrintWriter` |

# 🌱 第四章：高级 IO 与 NIO

> **本章在整体中解决什么问题**：第三章讲了传统流式 IO；本章介绍 **NIO** 的模型、Channel/Buffer/Selector 及与传统 IO 的对比，便于理解高并发、网络编程与后续 Netty。学完本章后**第五章**会讲 NIO 在性能优化（零拷贝、内存映射）中的应用；面试常问 BIO vs NIO、Selector 作用。

------

## 1️⃣ NIO 的引入与优势

### （1）传统 IO 与 NIO 对比（New I/O 或 Non-blocking I/O）

| 维度 | 传统 IO（BIO） | NIO |
| --- | --- | --- |
| **模型** | 面向流（Stream）、阻塞 | 面向通道（Channel）、缓冲区（Buffer），可非阻塞 |
| **读写方式** | 流式，单向 | Channel 可读可写，Buffer 读写分离 |
| **多路复用** | 多线程一连接（或线程池） | 单线程多 Channel（Selector） |
| **典型场景** | 简单文件、小并发 | 大并发、网络服务器、大文件 |
| **数据处理** | 流式处理，逐字节/字符 | 缓冲区批量处理，减少系统调用 |
| **适用范围** | 低并发、简单 IO 操作 | 高并发网络、大文件处理 |

### （2）NIO 核心组件详解

| 组件 | 说明 | 主要实现类 |
| --- | --- | --- |
| **Channel** | 双向通道，替代流的单向 | `FileChannel`、`SocketChannel`、`ServerSocketChannel`、`DatagramChannel` |
| **Buffer** | 内存缓冲区，读写分离 | `ByteBuffer`、`CharBuffer`、`IntBuffer`、`DoubleBuffer` 等 |
| **Selector** | 多路复用器，事件监听 | `Selector`（SelectorProvider.open() 创建） |
| **SelectorKey** | 通道与选择器的关联 | 包含通道、兴趣事件、就绪事件等信息 |
| **Pipe** | 线程间通信管道 | `Pipe.SinkChannel` 和 `Pipe.SourceChannel` |

### （3）Buffer 深度解析

#### Buffer 的核心属性
- **capacity**：缓冲区容量，创建后固定
- **position**：当前读写位置
- **limit**：读写限制位置
- **mark**：标记位置（可选）

#### Buffer 操作流程
1. **分配缓冲区**：`ByteBuffer buf = ByteBuffer.allocate(1024);`（堆内存）或 `ByteBuffer.allocateDirect(1024);`（直接内存）
2. **写入数据**：`buf.put(byte)` 或 `channel.read(buf)`
3. **切换为读模式**：`buf.flip()`（limit=position, position=0）
4. **读取数据**：`buf.get()` 或 `channel.write(buf)`
5. **清空/压缩**：`buf.clear()`（重置 position=0, limit=capacity）或 `buf.compact()`（将未读数据移至缓冲区开头）

#### Buffer 类型与适用场景
| Buffer 类型 | 适用场景 | 特点 |
| --- | --- | --- |
| `ByteBuffer` | 二进制数据 | 最常用，支持直接内存 |
| `CharBuffer` | 字符数据 | 支持字符编解码 |
| `IntBuffer` | 整型数据 | 适合处理整型数组 |
| `DoubleBuffer` | 浮点数据 | 适合处理浮点数组 |

### （4）Channel 深度解析

#### 主要 Channel 类型
| Channel 类型 | 用途 | 特点 |
| --- | --- | --- |
| `FileChannel` | 文件操作 | 支持文件读写、内存映射、零拷贝 |
| `SocketChannel` | TCP 客户端 | 可非阻塞，支持连接、读写 |
| `ServerSocketChannel` | TCP 服务器 | 可非阻塞，支持接受连接 |
| `DatagramChannel` | UDP | 支持无连接通信 |

#### FileChannel 核心方法
- **read/write**：读写文件
- **transferTo/transferFrom**：零拷贝文件传输
- **map**：内存映射文件
- **position/size/truncate**：文件位置和大小操作

### （5）Selector 深度解析

#### 工作原理
1. **注册通道**：`channel.register(selector, SelectionKey.OP_READ)`
2. **轮询事件**：`selector.select()` 阻塞直到有事件就绪
3. **处理事件**：遍历 `selector.selectedKeys()` 处理就绪事件
4. **移除已处理的键**：`iterator.remove()`

#### 事件类型
- `SelectionKey.OP_READ`：可读
- `SelectionKey.OP_WRITE`：可写
- `SelectionKey.OP_CONNECT`：连接完成
- `SelectionKey.OP_ACCEPT`：接受连接

### （6）NIO 示例：文件复制（使用 FileChannel）

```java
public static void copyFile(String sourcePath, String targetPath) throws IOException {
    try (FileChannel sourceChannel = new FileInputStream(sourcePath).getChannel();
         FileChannel targetChannel = new FileOutputStream(targetPath).getChannel()) {
        // 使用 transferTo 实现零拷贝
        long position = 0;
        long size = sourceChannel.size();
        while (position < size) {
            position += sourceChannel.transferTo(position, size - position, targetChannel);
        }
    }
}
```

### （7）NIO 示例：简单的 Echo 服务器

```java
public class NioEchoServer {
    public static void main(String[] args) throws IOException {
        Selector selector = Selector.open();
        
        ServerSocketChannel serverChannel = ServerSocketChannel.open();
        serverChannel.bind(new InetSocketAddress(8080));
        serverChannel.configureBlocking(false);
        serverChannel.register(selector, SelectionKey.OP_ACCEPT);
        
        System.out.println("Server started on port 8080");
        
        while (true) {
            selector.select(); // 阻塞直到有事件
            
            Iterator<SelectionKey> iterator = selector.selectedKeys().iterator();
            while (iterator.hasNext()) {
                SelectionKey key = iterator.next();
                iterator.remove();
                
                if (key.isAcceptable()) {
                    // 处理连接
                    ServerSocketChannel server = (ServerSocketChannel) key.channel();
                    SocketChannel client = server.accept();
                    client.configureBlocking(false);
                    client.register(selector, SelectionKey.OP_READ);
                    System.out.println("Client connected: " + client.getRemoteAddress());
                } else if (key.isReadable()) {
                    // 处理读取
                    SocketChannel client = (SocketChannel) key.channel();
                    ByteBuffer buffer = ByteBuffer.allocate(1024);
                    int bytesRead = client.read(buffer);
                    
                    if (bytesRead == -1) {
                        client.close();
                        System.out.println("Client disconnected");
                    } else if (bytesRead > 0) {
                        buffer.flip();
                        client.write(buffer); // Echo back
                        buffer.clear();
                    }
                }
            }
        }
    }
}
```

### 📍 BIO vs NIO 模型简图（面试可画）

```
BIO（阻塞 IO）：
  线程1 ──read()阻塞──→ 连接1
  线程2 ──read()阻塞──→ 连接2
  线程3 ──read()阻塞──→ 连接3
  … 一连接一线程，高并发时线程数爆炸

NIO（多路复用）：
              ┌── 连接1 ──→ Channel
              ├── 连接2 ──→ Channel
  Selector ───┼── 连接3 ──→ Channel   ← 单线程轮询哪些 Channel 就绪
              └── …
  一个线程通过 Selector 监听多个 Channel，就绪的才读写，非阻塞
```

> **面试怎么说**：BIO 是面向流、阻塞的，一个连接一个线程；NIO 是面向 Channel 和 Buffer，可设成非阻塞，用 **Selector** 一个线程监听多个 Channel 的读/写/连接事件，实现多路复用，适合高并发。NIO 三件套：Channel、Buffer、Selector。

# 🌱 第五章：IO 性能优化

> **本章在整体中解决什么问题**：在第三、四章「会用流与 NIO」基础上，本章讲**性能优化**——缓冲、NIO 零拷贝与内存映射、多线程与异步。与第四章 NIO、第六章资源关闭结合，可形成「正确 + 高效 + 安全关闭」的完整实践。

------

## 1️⃣ 缓冲区与缓冲流

### （1）为什么用缓冲

- 减少系统调用次数：每次 read/write 都可能触发内核调用；先读入内存缓冲区，再批量读写，可显著降低调用次数。
- **BufferedInputStream / BufferedOutputStream**：包装字节流，默认 8KB 缓冲。
- **BufferedReader / BufferedWriter**：包装字符流，适合按行或大块文本。

### （2）缓冲大小建议

- 默认 8KB 对多数场景够用；大文件或高吞吐可适当增大（如 32KB～64KB），需结合实测。

------

## 2️⃣ 使用 NIO 提高 IO 性能

### （1）零拷贝技术

#### 什么是零拷贝
- **零拷贝**（Zero Copy）：减少数据在用户态和内核态之间的拷贝次数，提高 IO 性能
- **传统 IO 拷贝**：用户空间 → 内核空间 → 硬件设备（多次拷贝）
- **零拷贝**：直接在内核空间完成数据传输，减少用户态参与

#### FileChannel 的零拷贝方法
- **transferTo/transferFrom**：利用操作系统的零拷贝机制，适合大文件传输
- **底层实现**：在 Linux 上使用 `sendfile()` 系统调用，在 Windows 上使用 `TransmitFile()`

#### 零拷贝适用场景
- 大文件复制
- 网络文件传输
- 视频流传输
- 任何需要高性能数据传输的场景

### （2）直接缓冲区（DirectBuffer）

#### 直接缓冲区 vs 堆缓冲区
| 特性 | 直接缓冲区 | 堆缓冲区 |
| --- | --- | --- |
| **内存位置** | 堆外内存 | JVM 堆内 |
| **创建成本** | 较高 | 较低 |
| **访问速度** | 较快（减少一次拷贝） | 较慢（需要拷贝到堆外） |
| **回收** | 依赖 GC，可能延迟 | 由 GC 管理 |
| **适用场景** | 大文件、高频 IO | 小文件、低频 IO |

#### 直接缓冲区的使用
```java
// 分配直接缓冲区
ByteBuffer directBuffer = ByteBuffer.allocateDirect(1024 * 1024); // 1MB

// 使用直接缓冲区
FileChannel channel = new FileInputStream("large.file").getChannel();
channel.read(directBuffer);
directBuffer.flip();
// 处理数据...

// 释放引用，便于 GC 回收
directBuffer.clear();
channel.close();
```

### （3）内存映射文件（Memory Mapped Files）

#### 内存映射原理
- 将文件的一部分或全部映射到进程的虚拟内存地址空间
- 读写内存就相当于读写文件，由操作系统负责同步
- 适合大文件的随机访问和顺序读取

#### 内存映射的优势
- **随机访问**：可直接访问文件任意位置，无需 seek
- **性能**：比传统 IO 快，尤其对于大文件
- **简化代码**：无需手动管理缓冲区

#### 内存映射的使用
```java
public static void readLargeFile(String filePath) throws IOException {
    try (FileChannel channel = new FileInputStream(filePath).getChannel()) {
        // 映射整个文件到内存
        MappedByteBuffer buffer = channel.map(FileChannel.MapMode.READ_ONLY, 0, channel.size());
        
        // 直接读取内存映射的数据
        while (buffer.hasRemaining()) {
            byte b = buffer.get();
            // 处理数据...
        }
        
        // 注意：MappedByteBuffer 没有 close 方法，需要释放引用
        buffer = null;
        System.gc(); // 建议在使用完毕后触发 GC
    }
}
```

#### 内存映射的注意事项
- **内存限制**：映射大小受 JVM 和操作系统限制
- **资源管理**：MappedByteBuffer 不会自动释放，需要手动解除引用
- **同步**：修改后需要调用 `force()` 方法同步到磁盘
- **异常处理**：映射过程可能抛出 `IOException`

### （4）NIO.2 的 AsynchronousFileChannel

#### 异步文件 IO
- JDK 7 引入的异步文件通道
- 支持回调和 Future 两种方式
- 适合高并发场景，避免阻塞主线程

#### 异步文件 IO 的使用
```java
public static void asyncReadFile(String filePath) throws Exception {
    Path path = Paths.get(filePath);
    AsynchronousFileChannel channel = AsynchronousFileChannel.open(path, StandardOpenOption.READ);
    
    ByteBuffer buffer = ByteBuffer.allocate(1024);
    
    // 使用 Future 方式
    Future<Integer> future = channel.read(buffer, 0);
    while (!future.isDone()) {
        // 可以做其他事情
        System.out.println("Doing other work...");
        Thread.sleep(100);
    }
    
    int bytesRead = future.get();
    System.out.println("Read " + bytesRead + " bytes");
    
    // 使用回调方式
    channel.read(buffer, 0, buffer, new CompletionHandler<Integer, ByteBuffer>() {
        @Override
        public void completed(Integer result, ByteBuffer attachment) {
            System.out.println("Read completed: " + result + " bytes");
            attachment.flip();
            // 处理数据...
        }
        
        @Override
        public void failed(Throwable exc, ByteBuffer attachment) {
            System.err.println("Read failed: " + exc.getMessage());
        }
    });
    
    channel.close();
}
```

### （5）性能对比测试

#### 测试场景：复制 1GB 文件
| 方法 | 平均时间 | 优势 | 劣势 |
| --- | --- | --- | --- |
| 传统 IO 流 | ~15s | 简单 | 慢 |
| 缓冲流 | ~8s | 较快 | 仍有多次拷贝 |
| FileChannel.transferTo | ~3s | 零拷贝，快 | 实现复杂 |
| 内存映射 | ~2s | 最快 | 内存占用大 |

#### 测试代码
```java
public class FileCopyPerformanceTest {
    private static final String SOURCE = "large.file";
    private static final String TARGET = "copy.file";
    
    public static void main(String[] args) throws Exception {
        System.out.println("Testing file copy performance...");
        
        testTraditionalIO();
        testBufferedIO();
        testFileChannel();
        testMemoryMapped();
    }
    
    private static void testTraditionalIO() throws Exception {
        long start = System.currentTimeMillis();
        try (InputStream in = new FileInputStream(SOURCE);
             OutputStream out = new FileOutputStream(TARGET)) {
            byte[] buf = new byte[8192];
            int n;
            while ((n = in.read(buf)) != -1) {
                out.write(buf, 0, n);
            }
        }
        long end = System.currentTimeMillis();
        System.out.println("Traditional IO: " + (end - start) + "ms");
    }
    
    private static void testBufferedIO() throws Exception {
        long start = System.currentTimeMillis();
        try (InputStream in = new BufferedInputStream(new FileInputStream(SOURCE));
             OutputStream out = new BufferedOutputStream(new FileOutputStream(TARGET))) {
            byte[] buf = new byte[8192];
            int n;
            while ((n = in.read(buf)) != -1) {
                out.write(buf, 0, n);
            }
        }
        long end = System.currentTimeMillis();
        System.out.println("Buffered IO: " + (end - start) + "ms");
    }
    
    private static void testFileChannel() throws Exception {
        long start = System.currentTimeMillis();
        try (FileChannel source = new FileInputStream(SOURCE).getChannel();
             FileChannel target = new FileOutputStream(TARGET).getChannel()) {
            long position = 0;
            long size = source.size();
            while (position < size) {
                position += source.transferTo(position, size - position, target);
            }
        }
        long end = System.currentTimeMillis();
        System.out.println("FileChannel: " + (end - start) + "ms");
    }
    
    private static void testMemoryMapped() throws Exception {
        long start = System.currentTimeMillis();
        try (FileChannel source = new FileInputStream(SOURCE).getChannel();
             FileChannel target = new FileOutputStream(TARGET).getChannel()) {
            long size = source.size();
            MappedByteBuffer sourceBuffer = source.map(FileChannel.MapMode.READ_ONLY, 0, size);
            MappedByteBuffer targetBuffer = target.map(FileChannel.MapMode.READ_WRITE, 0, size);
            targetBuffer.put(sourceBuffer);
            targetBuffer.force(); // 同步到磁盘
        }
        long end = System.currentTimeMillis();
        System.out.println("Memory Mapped: " + (end - start) + "ms");
    }
}
```

# 🌱 第六章：Java IO 与异常机制结合

> **本章在整体中解决什么问题**：把**异常与 IO** 串起来——IO 中常见异常、资源关闭规范、**try-with-resources** 的用法。第一章的异常处理、第二章的实践与本章的「流必须关闭」形成闭环；面试常问「流为什么要关」「try-with-resources 和 finally close 区别」。

------

## 1️⃣ IO 操作中的异常与资源管理

### （1）常见 IO 异常

| 异常 | 说明 |
| --- | --- |
| **FileNotFoundException** | 文件不存在或路径不可访问（继承自 IOException） |
| **IOException** | 通用 IO 错误（读写出错、流关闭后仍操作等） |
| **UnsupportedEncodingException** | 不支持的字符编码（继承自 IOException） |

### （2）资源关闭的规范

- 所有实现 **AutoCloseable** 的流、Channel、连接，都必须**在 finally 中关闭**，或使用 **try-with-resources**，否则会产生资源泄漏（文件句柄、连接数等）。

不推荐：只 try-catch 不关闭流。

```java
InputStream in = new FileInputStream("a.txt");
// 若中间抛异常，in 未关闭
in.read();
in.close();
```

推荐：try-with-resources（见下节）。

------

## 2️⃣ try-with-resources 管理流

### （1）自动资源管理（ARM）

**try-with-resources**（JDK 7+）：在 try 的括号中声明实现了 **AutoCloseable** 的资源，块结束时**自动调用 close()**，若有异常会先关闭再抛出，且可保留 **suppressed 异常**。

语法：

```java
try (ResourceType res = ...) {
    // 使用 res
}  // 自动 res.close()，若有异常会 suppress 到主异常
```

### （2）在 IO 流中的应用

示例：多资源、带 catch。

```java
try (InputStream in = new FileInputStream("a.txt");
     OutputStream out = new FileOutputStream("b.txt")) {
    byte[] buf = new byte[8192];
    int n;
    while ((n = in.read(buf)) != -1) {
        out.write(buf, 0, n);
    }
} catch (IOException e) {
    log.error("复制失败", e);
}
```

> 💡 **重点**：从 JDK 9 起，可以在 try 外先声明变量再在 try 中引用（变量要为 effectively final），便于在 catch 中区分不同资源异常。

| 知识点 | 面试关键词 | 实际应用 |
| ------ | ---------- | -------- |
| IO 常见异常 | FileNotFoundException、IOException | 文件不存在、编码错误、流关闭后操作 |
| 资源关闭 | AutoCloseable、finally、try-with-resources | 所有流、Channel、连接必须关闭 |
| try-with-resources | 自动 close、suppressed 异常 | 替代手写 finally close，避免泄漏 |

------

**学习要点**：

- 能列举 IO 中常见异常及含义。
- 牢固掌握 try-with-resources 的写法与执行顺序（先关资源，再抛主异常；suppressed 可被 getSuppressed() 拿到）。
- 与第二章结合：关闭资源时若 close 也抛异常，会作为主异常的 suppressed，在日志中应关注完整堆栈。

## 🎯 面试常见追问

| 面试官提问 | 回答思路 |
| ---------- | -------- |
| IO 流为什么要关闭？不关会怎样？ | 释放文件句柄、连接等系统资源；不关会导致句柄泄漏，高并发下「Too many open files」或连接耗尽。 |
| try-with-resources 和 finally 里 close 的区别？ | try-with-resources 自动 close，且若 try 和 close 都抛异常，主异常会带 suppressed（close 的异常），不会丢；finally 里手写 close 要自己处理 close 抛出的异常。 |

------

### ⚠️ 注意点

| 注意点 | 说明 |
|--------|------|
| 实现 AutoCloseable 的都要关 | 流、Channel、Socket、Connection 等；用 try-with-resources 或 finally，避免泄漏。 |
| close 抛异常时 | try-with-resources 会将其作为主异常的 suppressed，getSuppressed() 可拿到；日志要打完整堆栈。 |
| JDK 9+ try 可引用外部变量 | 变量需 effectively final，便于在 catch 中区分是哪个资源出错。 |
