## 🧠 1. JVM 的定义与作用

### 📘 定义

**JVM（Java Virtual Machine）** 是 Java 程序运行的**抽象计算机模型**，负责：

- 加载字节码（.class）
- 验证与解释/编译执行
- 管理内存（分配、回收）
- 提供跨平台支持（一次编译，处处运行）

👉 JVM 是 **Java 程序与操作系统之间的中间层**，屏蔽平台差异。

------

### ⚙️ 作用

| 作用类别     | 说明                                                         |
| ------------ | ------------------------------------------------------------ |
| 跨平台执行   | 各系统安装对应 JVM 即可运行相同字节码（Write Once, Run Anywhere） |
| 内存管理     | 自动分配与垃圾回收（GC）                                     |
| 多线程支持   | 线程模型与同步机制                                           |
| 安全与隔离   | 类加载 + 字节码校验                                          |
| 性能优化     | JIT、逃逸分析、内联等                                        |

------

## 🧳 2. JVM 的发展与 HotSpot

| 阶段     | 实现        | 特点                                   |
| -------- | ----------- | -------------------------------------- |
| 初代     | Classic VM  | 纯解释，无 JIT                         |
| 主流     | **HotSpot** | JIT + 热点探测，C1/C2 编译器           |
| 替代选择 | OpenJ9      | 启动快、占内存小，适合容器             |
| 低延迟   | Zing        | C4 无停顿 GC，金融等场景               |
| 多语言   | GraalVM     | 多语言、AOT                            |

📌 **HotSpot** 随 OpenJDK/Oracle JDK 发布，服务端开发最常用；通过**热点探测**将频繁执行的代码 JIT 编译为本地机器码。

------

## 🧳 3. JVM 的主要组成部分

### 📘 整体结构

```
.java → javac → .class → 类加载子系统 → 运行时数据区
                          ↓
                    执行引擎（解释 + JIT）+ GC + JNI
```

| 组成部分       | 作用                           |
| -------------- | ------------------------------ |
| **类加载子系统** | 加载、验证、准备、解析、初始化 .class |
| **运行时数据区** | 堆、方法区、虚拟机栈、本地方法栈、程序计数器 |
| **执行引擎**   | 解释器 + JIT，执行字节码       |
| **JNI**        | Java 调用本地（C/C++）代码    |
| **GC 系统**    | 自动回收堆中不可达对象         |

------

### 📘 运行时数据区简表

| 区域           | 是否线程共享 | 主要内容               | 说明                  |
| -------------- | ------------ | ---------------------- | --------------------- |
| **堆**         | 是           | 对象实例、数组         | GC 主要区域           |
| **方法区**     | 是           | 类信息、常量、静态变量 | JDK8 起为 Metaspace   |
| **虚拟机栈**   | 否           | 局部变量、操作数栈     | 每线程一份            |
| **本地方法栈** | 否           | JNI 方法调用           | 每线程一份            |
| **程序计数器** | 否           | 当前指令地址           | 唯一不会 OOM 的区域   |

------

### 📘 类加载与双亲委派

- **加载** → **验证** → **准备**（静态变量赋零值）→ **解析**（符号引用→直接引用）→ **初始化**（静态赋值、执行静态块）。
- **双亲委派**：请求先交给父加载器，父能加载则父加载，否则子加载；保证核心类由 Bootstrap 等加载，防止被篡改。
- 常见加载器：Bootstrap（rt.jar）→ Extension（ext）→ Application（classpath）。

------

## 🧳 4. 执行流程概览

1. **javac** 将 .java 编译为 .class（字节码）。
2. **类加载器** 加载 .class 到方法区，生成 Class 对象。
3. **执行引擎** 解释执行或 JIT 编译执行。
4. **GC** 在堆上回收不可达对象。

# 🧩 第二章：类加载子系统

> **目标：** 掌握类加载生命周期（加载→验证→准备→解析→初始化）、双亲委派机制及常见面试题。

> **本章在整体中解决什么问题**：第一章提到了类加载；本章细化**五阶段**与**双亲委派**，是面试高频题。学完本章后**第三章**讲运行时数据区（类信息在方法区）；**第七章**调优会涉及类加载与 Metaspace。

------

## 🧠 1. 类加载的生命周期

### 📘 五阶段

| 阶段     | 说明                                           |
| -------- | ---------------------------------------------- |
| **加载** | 读入 .class，生成 Class 对象，可自定义类加载器 |
| **验证** | 格式、元数据、字节码、符号引用等校验           |
| **准备** | 为静态变量分配内存并赋**零值**（非终值）       |
| **解析** | 将符号引用解析为直接引用（可发生在初始化之后） |
| **初始化** | 执行 &lt;clinit&gt;：静态变量赋值、静态块      |

> 💡 解析阶段可在初始化之后（如用到时才解析），但准备一定在初始化之前。

------

## 🧳 2. 双亲委派模型

### 📘 机制

- 类加载请求先交给**父加载器**；父能加载则父加载，否则子加载器再尝试。
- **目的**：保证核心类（如 java.lang.*）由 Bootstrap 等顶层加载，避免应用类替换 JDK 核心类。

### 📘 类加载器层次（典型）

| 加载器              | 加载范围           |
| ------------------- | ------------------ |
| Bootstrap           | rt.jar 等核心库    |
| Extension           | ext 目录           |
| Application         | classpath 应用类   |
| 自定义 ClassLoader  | 自定义来源         |

### ⚡ 破坏双亲委派

- 如 JDBC 等 SPI 需要**父委派类去加载子加载器路径下的类**，会通过 **线程上下文类加载器（Context ClassLoader）** 打破双亲委派；Tomcat、OSGi 等也有自己的类加载体系。

# 🧩 第三章：运行时数据区详解

> **目标：** 理解堆、方法区（Metaspace）、虚拟机栈、本地方法栈、程序计数器的作用与异常（OOM、StackOverflow）。

> **本章在整体中解决什么问题**：第一章给出了数据区简表；本章详解**各区域**的内容、是否共享、常见 OOM。学完本章后**第四章**讲 JMM（与主存/工作内存的对应）；**第五章**讲堆内部分代与对象分配；校招常要求画「运行时数据区」图。

------

## 🧠 1. 堆（Heap）

### 📘 作用

- 存放**对象实例与数组**，几乎所有对象都在堆上分配（逃逸分析后可能栈上分配）。
- **线程共享**，是 **GC 管理的主要区域**。
- 可进一步分为新生代（Eden + Survivor）、老年代；JDK8 后方法区在 Metaspace（元空间），不在堆内。

### ⚠️ 异常

- **OutOfMemoryError: Java heap space**：堆不足，可调大 -Xmx、排查泄漏或大对象。

------

## 🧳 2. 方法区（Method Area / Metaspace）

### 📘 作用

- 存**类元信息、常量、静态变量、JIT 编译后的代码**等。
- JDK8 起用 **Metaspace（元空间）** 实现，使用**本地内存**，不再占用堆。
- **线程共享**。

### ⚠️ 异常

- **OutOfMemoryError: Metaspace**：类元数据过多，可调 -XX:MaxMetaspaceSize、排查类加载泄漏或大量动态类。

------

## 🧳 3. 虚拟机栈（JVM Stack）

### 📘 作用

- 每线程独有，存**栈帧**：局部变量表、操作数栈、方法出口等。
- 方法调用对应栈帧入栈，返回则出栈。

### ⚠️ 异常

- **StackOverflowError**：栈深度超限（如递归过深、栈帧过大）。
- **OutOfMemoryError**：扩展栈时无法申请更多内存（少见）。

------

## 🧳 4. 本地方法栈与程序计数器

- **本地方法栈**：为 JNI 本地方法服务，每线程独有。
- **程序计数器（PC）**：存当前线程执行的**字节码指令地址**，唯一规范中**不会 OOM** 的区域。

------

# ✅ 本章小结

| 知识点     | 面试关键词                     | 实际应用     |
| ---------- | ------------------------------ | ------------ |
| 堆         | 对象、数组、GC、OOM            | -Xmx、泄漏排查 |
| 方法区     | 类元信息、Metaspace、常量池    | MaxMetaspaceSize |
| 虚拟机栈   | 栈帧、局部变量、SOF            | 递归、栈深度 |
| PC         | 当前指令、唯一不 OOM           | 多线程切换   |

------

**学习要点**：能说出各区域存什么、是否共享、常见异常及对应参数。

------

# 🧩 第四章：Java 内存模型（JMM）与线程安全

> **目标：** 理解 JMM 的主内存与工作内存、可见性/有序性、happens-before 及 volatile、synchronized 在 JMM 中的语义。

> **本章在整体中解决什么问题**：第三章讲的是 JVM 运行时「物理」分区；本章讲 **JMM**——规范层面的主存/工作内存、可见性与有序性，与《多线程与并发》中的 JMM 章互补。学完本章后**第五、六章**回到堆与 GC；面试常问「JMM 和 JVM 内存区别」。

------

## 🧠 1. JMM 简介

### 📘 设计目标

- **JMM** 定义线程与主内存的交互规则，屏蔽 CPU 缓存、指令重排等差异，保证**正确同步**的程序具有一致性语义。
- **主内存**：共享变量（可类比堆）。
- **工作内存**：每条线程的私有副本（可类比 CPU 缓存/寄存器），线程对变量的读写先在工作内存，再与主内存交互。

------

## 🧳 2. 可见性、有序性与原子性

| 性质   | 含义                         | 保障手段举例           |
| ------ | ---------------------------- | ---------------------- |
| 可见性 | 某线程写后其他线程能读到最新 | volatile、synchronized |
| 有序性 | 指令不因重排导致错误         | volatile、synchronized、happens-before |
| 原子性 | 操作不可被拆散               | synchronized、原子类   |

------

## 🧳 3. happens-before（HB）

- **happens-before** 描述两个操作之间的偏序关系；若 A happens-before B，则 A 对 B 可见。
- 常见规则：程序顺序、锁的 unlock happens-before 后续 lock、volatile 写 happens-before 后续 volatile 读、线程 start happens-before 其 run、线程结束 happens-before join 返回等。
- volatile 与 synchronized 能在语义上形成 happens-before，从而保证可见性与一定有序性。

------

## 🧳 4. volatile 与 synchronized 在 JMM 中

- **volatile**：写后刷新到主内存，读从主内存读；禁止重排与 volatile 相关的指令；**不保证复合操作原子性**。
- **synchronized**：锁的获取与释放对应主内存与工作内存的同步，保证块内可见性与有序性，且保证块内操作原子性（互斥）。

------

| 知识点     | 面试关键词               | 实际应用     |
| ---------- | ------------------------- | ------------ |
| JMM        | 主内存、工作内存、交互规则 | 理解可见性   |
| 三性       | 可见性、有序性、原子性    | 选 volatile/syn |
| happens-before | 偏序、可见性传递       | 理解语义     |
| volatile   | 可见、禁重排、不保证原子  | 状态标志等   |

------

**学习要点**：能区分 JMM 与 JVM 内存结构（JMM 是规范，JVM 是运行时分区）；能说清 volatile 与 synchronized 在 JMM 上的作用。

# 🧩 第五章：堆内存结构与对象分配

> **目标：** 理解堆的分代（新生代/老年代）、对象分配与晋升、对象内存布局、TLAB 与逃逸分析。

> **本章在整体中解决什么问题**：第三章讲了堆是「对象与数组」所在；本章讲堆**内部分代**、对象从分配到晋升的路径，为**第六章 GC** 打基础。校招常问「对象什么时候进老年代」「对象一定在堆上吗」。

------

## 🧠 1. 堆的分代结构

### 📘 典型划分（以分代收集为例）

| 区域     | 说明                         |
| -------- | ---------------------------- |
| **新生代** | Eden + 两个 Survivor（S0/S1），新对象大多在此分配 |
| **老年代** | 存活较久的对象，晋升而来     |
| **Metaspace** | 类元数据，不属于堆，在本地内存 |

- 大部分对象朝生夕死，新生代采用**复制算法**；老年代采用**标记-清除**或**标记-整理**（视收集器而定）。

------

## 🧳 2. 对象分配与晋升

- **新对象**：优先在 **Eden** 分配；Eden 满则触发 **Minor GC**，存活对象复制到 Survivor，年龄增加，达到阈值（如 15）则**晋升老年代**。
- **大对象**：可能直接进老年代（取决于 JVM 与参数）。
- **TLAB（Thread Local Allocation Buffer）**：在 Eden 中为线程划一小块私有空间，减少同步，提高分配效率。

### 📘 TLAB 详解

- **问题背景**：对象分配需要**原子性**（CAS 或加锁），多线程竞争会影响性能。
- **TLAB 解决方案**：在 Eden 区为每个线程预分配一块**线程私有**的缓冲区，线程在此区域内分配对象无需加锁。
- **工作流程**：
  1. 线程首次分配对象时，向 Eden 区申请一块 TLAB（大小由 -XX:TLABSize 决定，默认约 Eden 的 1%）
  2. 线程在 TLAB 内分配对象（只需要移动指针，极快）
  3. TLAB 满或剩余空间不足以分配新对象时，废弃当前 TLAB，重新申请
- **TLAB 优点**：
  - 无锁分配，极大提高分配速度
  - 减少 Eden 区的内存碎片
- **TLAB 参数**：
  - `-XX:+UseTLAB`：启用 TLAB（默认开启）
  - `-XX:TLABSize`：TLAB 大小
  - `-XX:TLABWasteTargetPercent`：TLAB 允许浪费的目标比例
- **注意事项**：
  - TLAB 只影响对象**分配**，不影响对象**访问**
  - 即使使用 TLAB，对象仍然可能在 Survivor 或老年代

------

## 🧳 3. 对象内存布局

- **对象头**：Mark Word（哈希、GC 年龄、锁状态等）+ 类型指针（类元数据）。
- **实例数据**：各实例字段。
- **对齐填充**：保证 8 字节对齐。

### 📘 对象头详解（Mark Word）

在 64 位 JVM 中，**Mark Word** 占用 8 字节（64 bits），包含对象的哈希码、GC 年龄、锁状态等信息。根据锁状态不同，Mark Word 的内容也不同：

| 锁状态       | Mark Word 内容（64位）                           |
| ------------ | ------------------------------------------------ |
| **无锁**     | 对象哈希码（31位）+ 分代年龄（4位）+ 偏向锁位（1位）+ 锁标志位（01） |
| **偏向锁**   | 线程ID（54位）+ Epoch（2位）+ 分代年龄（4位）+ 锁标志位（01） |
| **轻量级锁** | 指向锁记录（Lock Record）的指针（62位）+ 锁标志位（00） |
| **重量级锁** | 指向互斥量（ObjectMonitor）的指针（62位）+ 锁标志位（10） |
| **GC 标记**  | 空（00）+ 锁标志位（11）                         |

> 💡 偏向锁默认在 JDK 15 后被禁用（`-XX:+UseBiasedLocking` 默认 false），因为自旋锁优化已足够。

### 📘 锁升级过程（锁粗化/锁升级）

synchronized 的锁是**逐步升级**的，**不可降级**（HotSpot 实现），过程如下：

```
无锁 → 偏向锁 → 轻量级锁 → 重量级锁
```

1. **偏向锁**：
   - 第一次获取锁时，将 Mark Word 中的线程ID设为当前线程ID。
   - 后续该线程进入同步块时，只需检查 Mark Word 中是否为偏向锁且线程ID是自己。
   - 如果有其他线程尝试获取，则撤销偏向锁，升级为轻量级锁。

2. **轻量级锁**：
   - 在当前线程栈帧中创建**锁记录（Lock Record）**，将 Mark Word 复制到锁记录。
   - 使用 **CAS（Compare-And-Swap）** 将对象头的 Mark Word 替换为指向锁记录的指针。
   - 如果 CAS 成功，则获取轻量级锁；如果失败（其他线程竞争），自旋等待，超过阈值后膨胀为重量级锁。

3. **重量级锁**：
   - 依赖底层的 **ObjectMonitor**（互斥量），需要 OS 介入，用户态到内核态切换，成本高。
   - 未获取锁的线程会被阻塞，存在线程上下文切换开销。

### 📘 synchronized 底层原理

synchronized 编译后会生成 **monitorenter** 和 **monitorexit** 字节码指令，依赖 **ObjectMonitor** 实现：

1. **monitorenter**：尝试获取对象的 monitor（锁），计数器+1。
2. **monitorexit**：释放 monitor，计数器-1。

**ObjectMonitor 关键结构**：
```
ObjectMonitor() {
    _owner = NULL;           // 持有锁的线程
    _WaitSet = NULL;         // 等待队列（wait()的线程）
    _EntryList = NULL;       // 阻塞队列（竞争锁失败的线程）
    _count = 0;              // 计数器
}
```

- 获取锁时：线程加入 _EntryList，被选中的线程设为 _owner。
- wait() 时：线程进入 _WaitSet，释放锁。
- notify() 时：从 _WaitSet 唤醒一个线程到 _EntryList。

> **面试怎么说**：synchronized 底层依赖 ObjectMonitor，锁会升级（偏向锁→轻量级锁→重量级锁）；Mark Word 在不同锁状态下内容不同；轻量级锁用 CAS+自旋，重量级锁用 ObjectMonitor 需要 OS 阻塞线程。

### 📘 类型指针（Klass Pointer）

- 对象头中还有一个指向**类元数据（Klass）的指针**（开启压缩指针后为 4 字节，否则 8 字节），用于确定对象属于哪个类。
- JDK 7 及之前，类型指针位于对象头；JDK 8+ 开启指针压缩后，类型指针仍在对象头。

### 📘 对齐填充

- JVM 要求对象大小必须是 **8 字节的倍数**（或 16 字节，取决于 GC 收集器）。
- 如果实例数据不足，则通过填充补齐，减少内存访问碎片，提高 CPU 缓存命中率。

## 🧳 4. 逃逸分析

- JVM 会分析对象**是否逃出方法/线程**；若未逃出，可能**栈上分配**或**标量替换**，减轻堆与 GC 压力。
- 实际 HotSpot 默认未开启栈上分配，多以标量替换优化为主。

### 📍 对象分配与晋升流程简图（面试可画）

```
new 对象
    │
    ├─ 1. 若开启逃逸分析且未逃逸 → 可能栈上分配或标量替换（HotSpot 默认栈上分配未开）
    ├─ 2. 大对象 → 可能直接进老年代（视 -XX:PretenureSizeThreshold 等）
    └─ 3. 一般对象 → 优先在 Eden 分配（或 TLAB 内）
             │
             ├─ Eden 有空间 → 直接分配
             └─ Eden 满 → 触发 Minor GC
                     │
                     ├─ 存活对象 → 复制到 Survivor（S0/S1），年龄 +1
                     ├─ 年龄达到阈值（如 15）→ 晋升老年代
                     ├─ Survivor 放不下 → 直接晋升老年代
                     └─ 老年代满 → 触发 Full GC
```

> **面试怎么说**：新对象优先在 Eden 分配；Eden 满触发 Minor GC，存活对象进 Survivor 并增加年龄，年龄到阈值或 Survivor 不够则晋升老年代；大对象可能直接进老年代；逃逸分析未逃出可能栈上分配或标量替换，但 HotSpot 默认栈上分配未开。

## 🧠 1. 判断对象是否可回收

### 📘 可达性分析

- 从 **GC Roots**（栈中引用、静态变量、常量、JNI 引用等）出发，不可达的对象可被回收。
- 与“引用计数”相比，可达性分析能解决循环引用问题。

### 📘 引用类型

| 类型     | 说明                         | 回收时机 | 常见用途 |
| -------- | ---------------------------- | -------- | -------- |
| 强引用   | 普通 new，不回收             | 永不回收 | 普通对象 |
| 软引用   | 内存不足时可能回收           | 内存不足时 | 缓存 |
| 弱引用   | 下次 GC 一般会回收           | GC 时 | 缓存、防止内存泄漏 |
| 虚引用   | 用于跟踪回收，不能取到对象   | 任何时候 | 对象生命周期监听、泄漏追踪 |

#### 软引用（SoftReference）

- 内存不足时会被回收，用于实现内存敏感的高速缓存。
- 适合缓存数据，当堆内存紧张时可释放缓存。
- 示例：`SoftReference<byte[]> cache = new SoftReference<>(data);`

#### 弱引用（WeakReference）

- GC 运行时必然被回收（不管内存是否充足），比软引用更"弱"。
- 常见用途：
  - **缓存**：如 `WeakHashMap`，key 无其他引用时可被回收
  - **防止内存泄漏**：如 `ThreadLocal` 内部使用弱引用持有 ThreadLocalMap 的 Entry
- 示例：`WeakReference<Object> ref = new WeakReference<>(obj);`

#### 虚引用（PhantomReference）

- **无法通过虚引用获取对象**，必须配合 `ReferenceQueue` 使用。
- 主要用途：
  - **对象生命周期跟踪**：GC 回收对象时，将虚引用放入队列，可用于追踪对象何时被回收
  - **泄漏排查与资源清理**：在对象被回收前执行清理工作（如关闭文件、释放资源）
  - **NIO 直接内存管理**：`DirectByteBuffer` 使用虚引用追踪堆外内存
- 示例：
```java
ReferenceQueue<Object> queue = new ReferenceQueue<>();
PhantomReference<Object> phantom = new PhantomReference<>(obj, queue);
// 对象被 GC 回收后，phantom 会被放入 queue
```

> 💡 **虚引用在泄漏排查中的应用**：通过监听虚引用队列，可以知道对象何时被回收，常用于排查无法被回收的泄漏对象。

## 🧳 2. 常见 GC 算法

| 算法       | 思路                     | 优缺点                     |
| ---------- | ------------------------ | -------------------------- |
| 标记-清除 | 标记可达对象，清除未标记 | 简单，易产生碎片           |
| 复制       | 存活对象复制到另一块     | 无碎片，占空间             |
| 标记-整理 | 标记后移动对象到一端     | 无碎片，移动有成本         |
| 分代收集   | 新生代复制、老年代标记整理等 | 结合各算法，常用           |

## 🧳 3. 典型垃圾收集器

| 收集器   | 特点                           | 适用场景         |
| -------- | ------------------------------ | ---------------- |
| Serial   | 单线程，STW                    | 客户端、小堆     |
| ParNew   | 多线程版 Serial，配合 CMS 新生代 | CMS 搭配         |
| CMS      | 并发标记清除，减少 STW，有碎片 | 老年代，低延迟   |
| G1       | 分区、可预测停顿、兼顾吞吐与延迟 | 大堆、通用       |
| ZGC      | 低延迟、并发多，JDK11+         | 大堆、超低延迟   |

### 📘 CMS（Concurrent Mark Sweep）

- **定位**：老年代收集器，以获取**最短回收停顿时间**为目标。
- **工作流程**（四步）：
  1. **初始标记（Initial Mark）**：STW，仅标记 GC Roots 直接引用的对象，速度快。
  2. **并发标记（Concurrent Mark）**：与用户线程并发，从 GC Roots 出发遍历整个对象图，标记所有存活对象。
  3. **重新标记（Remark）**：STW，修正并发标记期间因用户程序运行产生的变化（浮动垃圾），时间比初始标记长但远小于并发标记。
  4. **并发清除（Concurrent Sweep）**：与用户线程并发，清除已死亡对象。

- **优点**：并发收集、低停顿。
- **缺点**：
  - **浮动垃圾**：并发标记阶段产生的垃圾（浮动垃圾）无法在本次清除，只能等下次 GC。
  - **空间碎片**：采用标记-清除算法，会产生内存碎片，导致提前触发 Full GC。
  - **并发失败（Concurrent Mode Failure）**：如果并发阶段老年代空间不足，会退化为 Serial Old 收集器，停顿变长。
- **相关参数**：
  - `-XX:+UseConcMarkSweepGC`：启用 CMS
  - `-XX:CMSInitiatingOccupancyFraction`：老年代使用率阈值（默认 92%），达到则触发 CMS
  - `-XX:+UseCMSCompactAtFullCollection`：Full GC 时进行碎片整理
  - `-XX:+CMSScavengeBeforeRemark`：Remark 前先触发一次 Minor GC，减少 Remark 时间

### 📘 G1（Garbage First）

- **定位**：面向服务端应用的垃圾收集器，JDK9+ 默认，取代 CMS。
- **核心思想**：将堆划分为**多个大小相等的 Region**（1MB~32MB），每个 Region 可独立作为 Eden、Survivor 或 Old 区。
- **重要概念**：
  - **CSet（Collection Set）**：本次 GC 需要回收的 Region 集合。
  - **RSet（Remembered Set）**：每个 Region 记录其他 Region 对本 Region 中对象的引用，用于 GC 时快速找到引用关系。
  - **SATB（Snapshot-At-The-Beginning）**：并发标记阶段使用的算法，标记开始时活着的对象作为"存活"基准，即使后续变死也不影响。
- **工作流程**：
  1. **年轻代收集（Young GC）**：回收年轻代 Region，存活对象复制到 Survivor Region，年龄达到阈值则晋升到 Old 区。
  2. **混合收集（Mixed GC）**：回收所有年轻代 + 部分老年代 Region。
  3. **并发标记周期（Concurrent Marking）**：
     - **初始标记（Initial Mark）**：标记 GC Roots 关联的 Region，依赖 Young GC 完成。
     - **根区域扫描（Root Region Scan）**：扫描 Survivor Region 中对老年代的引用。
     - **并发标记（Concurrent Mark）**：SATB 并发标记，可被 Young GC 打断。
     - **重新标记（Remark）**：STW，完成全部标记，使用 SATB 算法，处理日志缓冲区。
     - **清理（Cleanup）**：STW，统计存活对象，清除空 Region。
- **可预测停顿**：通过 `-XX:MaxGCPauseMillis` 设置目标停顿时间（默认 200ms），G1 会尽量满足。
- **优点**：可预测停顿、分区无碎片、并发收集、适合大堆（6GB+）。
- **缺点**：内存占用高（RSet 开销）、每个 Region 都需要 RSet，Region 越多内存开销越大。
- **相关参数**：
  - `-XX:+UseG1GC`：启用 G1
  - `-XX:MaxGCPauseMillis`：目标停顿时间
  - `-XX:G1HeapRegionSize`：Region 大小（必须是 2 的幂，1MB~32MB）
  - `-XX:InitiatingHeapOccupancyPercent`：触发并发标记周期阈值（默认 45%）
  - `-XX:G1NewSizePercent`：年轻代最小比例（默认 5%）
  - `-XX:G1MaxNewSizePercent`：年轻代最大比例（默认 60%）

### 📘 ZGC（Z Garbage Collector）

- **定位**：JDK11+ 引入的低延迟垃圾收集器，目标**停顿时间不超过 10ms**，且能处理 TB 级堆。
- **核心技术**：
  - **着色指针（Colored Pointers）**：在 64 位指针中用几位作为"着色位"标记对象状态（Marked0、Marked1、Remapped），GC 标记时只修改指针而非对象本身。
  - **读屏障（Load Barrier）**：读取对象时检查着色指针，如需要则修正引用（自愈），保证访问到正确状态的对象。
  - **并发标记与移动**：标记和压缩阶段与用户线程并发，减少 STW。
- **工作流程**：
  1. **暂停-初始化（Pause Start）**：STW，初始化标记，等待所有 GC 线程就绪。
  2. **并发标记（Concurrent Mark）**：遍历对象图，标记存活对象到着色指针。
  3. **并发准备重映射（Concurrent Prepare for Relocation）**：决定哪些 Region 需要压缩，整理内存。
  4. **并发重映射（Concurrent Relocate）**：将存活对象复制到新 Region，更新引用（利用着色指针+读屏障自愈）。
  5. **暂停-重映射（Pause Relocate）**：STW，完成剩余的引用更新。
- **优点**：停顿时间极短（<10ms）、可处理 TB 级堆、吞吐量高、与堆大小无关的停顿时间。
- **缺点**：吞吐量略低于 G1（因为读屏障开销）、内存占用略高（着色指针、RSet）、JDK 版本要求高（11+，生产推荐 15+）。
- **相关参数**：
  - `-XX:+UseZGC`：启用 ZGC
  - `-XX:ConcGCThreads`：并发工作线程数（默认 CPU 核数/4）
  - `-XX:MaxGCPauseMillis`：目标停顿时间（ZGC 会尽量满足）

### 📘 Shenandoah（OpenJDK 特有）

- **定位**：非 Oracle（Red Hat）开发的低延迟收集器，与 ZGC 竞争。
- **核心特点**：
  - **非连接分代（Non-generational）**：不分区代，所有 Region 统一管理。
  - **Brooks 指针**：每个对象有一个额外指针指向新位置，读取时通过指针转发，GC 时更新。
  - **并发标记与压缩**：与用户线程并发。
- **与 ZGC 对比**：
  - ZGC 用着色指针，Shenandoah 用 Brooks 指针。
  - ZGC 需要 OS 支持着色指针（Linux/x86 支持），Shenandoah 兼容性更好。
  - 实际生产 ZGC 用得更多。

### 📘 各收集器组合（典型）

| 新生代收集器 | 老年代收集器 | 说明 |
| ------------ | ------------ | ---- |
| Serial | Serial Old | 客户端默认，简单高效 |
| ParNew | CMS | 经典组合，并发收集 |
| ParNew | Serial Old | CMS 失败时后备 |
| Serial | G1 | 小堆兼容 G1 |
| ParNew | G1 | JDK9+ 默认 |
| G1 | G1 | 全堆 G1，JDK9+ 默认 |

> **面试怎么说**：CMS 是并发标记清除，有初始/并发/重新/并发清除四步，优点是并发低停顿，缺点是有空间碎片和浮动垃圾；G1 是分区收集器，把堆分成 Region，可设停顿目标，用 SATB 标记和 RSet 记录引用关系；ZGC 用着色指针实现亚毫秒级停顿，可处理 TB 级堆；Shenandoah 用 Brooks 指针，OpenJDK 特有。

## 🧳 4. Full GC 与 Minor GC

- **Minor GC**：回收新生代，较频繁，停顿较短。
- **Full GC**：回收整个堆（及部分 Metaspace），停顿长；触发条件包括老年代满、Metaspace 不足、System.gc()（建议）等。
- 调优目标之一：减少 Full GC 频率与停顿时间。

### 📍 GC 流程简图与面试话术

```
GC 触发（Eden 满 / 老年代满 / Metaspace 不足 / System.gc 等）
    │
    ├─ Minor GC（新生代）
    │      │
    │      ├─ 从 GC Roots 出发，标记存活对象
    │      ├─ 存活对象复制到 Survivor（复制算法），年龄+1
    │      ├─ 年龄达阈值或 Survivor 满 → 晋升老年代
    │      └─ 清空 Eden、已用 Survivor
    │
    └─ Full GC（整堆 + 部分 Metaspace）
           │
           ├─ 老年代：标记-清除 或 标记-整理（视收集器）
           ├─ 新生代：同上
           └─ 停顿时间较长，需尽量避免频繁 Full GC
```

> **面试怎么说**：判断可回收用可达性分析，从 GC Roots 不可达则可回收；新生代用复制算法，老年代用标记-清除或标记-整理；Minor GC 收新生代，Full GC 收整堆；Full GC 触发条件有老年代满、Metaspace 不足、System.gc 等；G1 分区、可设停顿目标，ZGC 超低延迟。

|---------------|------|-------------------|
| 未设 -Xmx 或设得过大 | 默认堆可能很小或占满物理内存，导致 OOM 或抢其他进程资源 | 生产环境显式设 -Xms/-Xmx（可相同以减少动态扩容），根据机器与业务定值。 |
| Full GC 频繁、停顿长 | 老年代满、Metaspace 不足、大对象直接进老年代、内存泄漏 | 看 GC 日志与 jstat，调大堆或老年代、查泄漏与大对象；避免无意义 System.gc()。 |
| 混淆「引用计数」与「可达性分析」 | Java 用可达性分析，不用引用计数 | 面试答「可达性分析，从 GC Roots 不可达则回收」；引用计数无法解决循环引用。 |
| 以为 Minor GC 只收 Eden | 会收新生代整体（Eden + 当前使用的 Survivor） | Minor GC 是新生代 GC，存活对象复制到另一块 Survivor。 |
| Metaspace 无限增长导致 Full GC | 类加载过多、未卸载（如动态代理、大量类） | 设 -XX:MaxMetaspaceSize；排查类加载泄漏（重复加载、未回收的 ClassLoader）。 |

## 🧠 1. 解释器与 JIT 编译器

### 📘 解释器（Interpreter）

- 逐条读取字节码，解释执行，每次运行都需要解释转换。
- **优点**：启动快，不需要等待编译。
- **缺点**：执行慢，同样的代码每次都要解释。

### 📘 JIT 编译器（Just-In-Time Compiler）

- 将热点代码（Hot Spot）**即时编译**为本地机器码，缓存起来，后续直接执行机器码。
- **优点**：执行效率高，减少重复解释开销。
- **缺点**：编译需要时间，启动初期可能较慢。

> HotSpot 名字来源于"热点探测"——它会统计方法调用次数和循环回边次数，找出热点代码进行 JIT 编译。

### 📘 热点探测

- **方法调用计数器**：统计方法被调用次数，超过阈值（如 `-XX:CompileThreshold=10000`）触发编译。
- **回边计数器**：统计循环体执行次数，用于栈上替换（On-Stack Replacement，OSR）。
- **热度衰减**：计数器会随时间衰减，避免代码长期不执行仍占用资源。

### 📘 分层编译（Tiered Compilation）

JDK 8+ 默认开启，分层执行：

| 层级 | 编译器 | 特点 | 触发条件 |
| ---- | ------ | ---- | -------- |
| 0 | 解释器 | 不编译，纯解释 | 程序启动 |
| 1 | C1（Client） | 快速编译，优化少 | 方法调用计数 > 1500 |
| 2 | C1 | 带完整日志的快速编译 | -XX:+TieredCompilation |
| 3 | C2（Server） | 激进优化，编译慢 | 方法调用计数 > 10000 |

- 启动时先用 C1 快速编译，保证响应速度；随着热度上升，升级到 C2 进行深度优化。
- 分层编译参数：`-XX:+TieredCompilation`（JDK 8+ 默认开启）

### 📘 C1 与 C2 对比

| 特性 | C1（Client） | C2（Server） |
| ---- | ------------ | ------------- |
| 编译速度 | 快 | 慢 |
| 优化程度 | 基础优化 | 激进优化 |
| 目标 | 快速启动，响应速度 | 峰值性能 |
| 适用 | 客户端、小型应用 | 服务端、高吞吐 |

C2 的优化技术包括：**逃逸分析**、**标量替换**、**锁消除**、**锁粗化**、**内联**等。

------

## 🧳 2. 编译优化技术

### 📘 方法内联（Method Inlining）

- 将小方法（通常 < 35 字节，-XX:MaxInlineSize 可调）的调用直接替换为方法体，避免调用开销。
- 最重要的优化：内联后才能进行更多优化（如逃逸分析）。

**内联条件**：
- 方法调用次数多（热点）
- 方法体不太大
- 虚函数（如多态）也能内联，但需要**类型推断**（CHA：Class Hierarchy Analysis）

### 📘 逃逸分析（Escape Analysis）

- 分析对象的**作用域**：是否逃出方法或线程。
- **不逃逸**的对象可优化：
  - **栈上分配**：在栈上分配内存，方法结束自动回收，无需 GC（HotSpot 默认未开启栈上分配）。
  - **标量替换**：将对象拆解为多个局部变量（标量），直接存在栈上，不创建对象。
  - **锁消除（Lock Elision）**：如果对象不逃出线程，同步块可取消。

### 📘 锁消除与锁粗化

- **锁消除**：通过逃逸分析，判断同步块锁定的对象不会逃逸，则取消锁。
- **锁粗化**：如果一系列操作反复加锁解锁，JVM 会扩大锁范围（如合并为一次加锁）。

### 📘 逆优化（Deoptimization）

- JIT 编译的代码可能在某些情况下不如预期，需要**回退**到解释执行或较低优化级别。
- 触发场景：类加载变化、 CHA 信息变化、代码执行路径未覆盖等。
- 常见场景：多态内联后，子类被加载，类型信息变化。

------

## 🧳 3. AOT 编译与 GraalVM

### 📘 AOT（Ahead-Of-Time）编译

- 在**运行前**编译为本地机器码（如 GraalVM native-image），无需 JIT。
- **优点**：启动极快，无预热，适合短生命周期任务、无 JVM 启动开销。
- **缺点**：无法享受 JIT 的运行时优化、动态类加载受限。

### 📘 GraalVM

- Oracle 的多语言虚拟机，支持 Java、JS、Python、Ruby 等。
- 提供 **GraalVM Compiler**，比 C2 更激进的优化。
- **Substrate VM**：AOT 编译的运行时，极小镜像，适合容器/无 JVM 环境。

## 🧠 1. 常用诊断工具详解

### 📘 jstat：GC 统计

```bash
# 查看 GC 统计（每 1 秒刷新一次）
jstat -gc <pid> 1000

# 查看类加载统计
jstat -class <pid>

# 查看 JIT 编译统计
jstat -compiler <pid>

# 查看 GC 容量信息
jstat -gccapacity <pid>
```

**关键指标解读**：
- **S0C / S1C / EC / OC / MC**：各区容量（Capacity）
- **S0U / S1U / EU / OU / MU**：各区使用量（Used）
- **YGC / YGCT**：年轻代 GC 次数 / 耗时
- **FGC / FGCT**：Full GC 次数 / 耗时
- **GCT**：总 GC 耗时
- **CCSC / CCSU**：压缩类空间

> **正常范围**：FGC 接近 0 或很少，YGCT 远小于 GCT，说明 GC 健康。

### 📘 jcmd：多功能诊断

```bash
# 查看可用命令
jcmd <pid> help

# 查看 VM 信息
jcmd <pid> VM.version

# 查看系统属性
jcmd <pid> VM.system_properties

# 查看线程栈
jcmd <pid> Thread.print

# 查看堆信息
jcmd <pid> GC.heap_info

# 触发 Full GC（仅用于调试！）
jcmd <pid> GC.run

# 查看 GC 原因统计
jcmd <pid> GC.class_histogram
```

### 📘 jinfo：运行时参数

```bash
# 查看所有参数
jinfo -flags <pid>

# 查看指定参数
jinfo -flag MaxHeapSize <pid>

# 动态设置参数（仅限可修改的参数）
jinfo -flag MaxGCPauseMillis=200 <pid>
```

### 📘 arthas（阿尔萨斯）

- 阿里开源的 Java 诊断工具，比命令行更强大。
- **dashboard**：查看整体运行状态
- **thread**：查看线程 CPU、阻塞情况
- **watch**：方法调用观测
- **trace**：方法链路追踪
- **jad**：反编译

> 生产环境推荐 arthas，能够在线上不停服诊断问题。

- 低开销的性能分析工具，支持 CPU、内存、锁分析。

## 🧳 2. GC 日志解读

### 📘 Serial GC 日志

```

[Full GC (Metadata GC Threshold)  41K->35K(57K), 0.0078900 secs]
```

- **GC / Full GC**：GC 类型
- **Allocation Failure / Metadata GC Threshold**：触发原因
- **16K->6K(57K)**：年轻代回收前使用 -> 回收后使用（容量）
- **0.0036780 secs**：耗时

### 📘 G1 GC 日志

```
[GC pause (G1 Evacuation Pause) young (initial-mark), 0.0123456 secs]
   [Root Region Scan Time: 0.0001234 secs]
   [Parallel Time: 0.0110000 secs]
      [GC Worker Start Time (ms): Min: 1.0, Avg: 1.5, Max: 2.0, Diff: 1.0]
      [Ext Root Scanning (ms): Min: 0.5, Avg: 0.8, Max: 1.2, Diff: 0.7]
      ...
   [Code Root Fixup Time: 0.0001000 secs]
   [Code Root Purge Time: 0.0000500 secs]
   [Clear CT Time: 0.0002000 secs]
   [Other Time: 0.0010000 secs]
[GC concurrent-root-region-scan-start]
[GC concurrent-mark-start]
```

- **G1 Evacuation Pause**：年轻代或混合回收
- **(initial-mark)**：并发标记的初始标记阶段
- **young**：年轻代收集
- **Root Region Scan Time**：根区域扫描时间
- **Parallel Time**：并行工作时间

[2024-01-01T12:00:00.456+0800][info][gc] GC(0) Pause Mark Start 0.123ms
[2024-01-01T12:00:00.567+0800][info][gc] GC(0) Concurrent Mark 10.234ms
[2024-01-01T12:00:00.678+0800][info][gc] GC(0) Pause Relocate Start 0.456ms
[2024-01-01T12:00:00.789+0800][info][gc] GC(0) Concurrent Relocate 5.678ms

- **Pause Mark Start**：STW 初始标记
- **Concurrent Mark**：并发标记
- **Pause Relocate Start**：STW 重定位
- **Concurrent Relocate**：并发重定位
- **15M(0.1%)->12M(0.08%)**：堆使用量变化

### 📘 GC 日志分析工具

- **GCViewer**：分析 GC 日志，生成图表
- **GCEasy**：在线 GC 日志分析
- **GCLogViewer**：IDEA 插件
- **Prometheus + Grafana**：接入 JMX，监控 GC

### 📘 常见 GC 问题

| 现象 | 原因 | 解决方案 |
| ---- | ---- | -------- |
| Young GC 频繁 | Eden 太小、对象分配快 | 调大 -Xmn 或 -XX:NewRatio |
| Full GC 频繁 | 老年代满、Metaspace 满、内存泄漏 | 调大老年代/元空间、查泄漏 |
| GC 停顿长 | 堆太大、收集器不适合 | 换 G1/ZGC、调停顿目标 |
| 对象晋升失败 | Survivor 太小、对象过大 | 调大 Survivor、调大阈值 |

## 🧳 3. OOM 排查完整流程

### 📘 常见 OOM 类型

| OOM 类型 | 原因 | 排查重点 |
| -------- | ---- | -------- |
| Java heap space | 堆不足、内存泄漏 | heap dump + MAT |
| Metaspace | 类加载过多 | -XX:MaxMetaspaceSize、类加载泄漏 |
| Direct memory | NIO 堆外内存 | -XX:MaxDirectMemorySize、ByteBuffer |
| Unable to create new native thread | 线程数超限 | 线程泄漏 |
| GC overhead limit exceeded | GC 效率低、内存几乎满 | 内存泄漏 |

### 📘 排查步骤

1. **保留现场**：
   - 开启 `-XX:+HeapDumpOnOutOfMemoryError -XX:HeapDumpPath=/tmp/heap.hprof`
   - OOM 后会自动生成 dump

2. **查看日志**：
   - GC 日志分析（时间、次数、耗时）
   - 触发原因

3. **抓取实时 dump**（如果应用未崩溃）：
   ```bash
   jmap -dump:format=b,file=heap.hprof <pid>
   ```

4. **分析 dump**：
   - 用 **MAT** 打开 heap.hprof
   - 查看 **Histogram**：按对象数量/内存占用排序
   - 查看 **Dominator Tree**：找出占用内存的大对象
   - 查看 **GC Roots 引用链**：找出谁在持有泄漏对象

5. **定位泄漏**：
   - 常见泄漏点：静态集合（HashMap/ArrayList 只增不减）、未关闭的资源（File/DB/Connection）、ThreadLocal 泄漏、监听器未移除

6. **复现修复**：
   - 本地模拟场景
   - 修复代码，重新上线观察

### 📘 MAT 快速上手

- **Leak Suspects**：MAT 自动分析可能泄漏的报告
- **Histogram**：列出所有类对象，按数量/内存排序
- **Dominator Tree**：支配树，找出内存占用最大的对象路径
- **Path to GC Roots**：到 GC Roots 的引用链，用于判断对象是否应该被回收
- **Regex Filter**：支持正则过滤类名

### 📘 常见泄漏场景案例

**案例1：静态集合**
```java
// 错误示例
static List<byte[]> list = new ArrayList<>();
// 不断往里加对象，从不清理
```

**案例2：ThreadLocal 泄漏**
```java
// 线程池复用线程，但 ThreadLocal 未清理
threadLocal.set(data);
// 线程结束时需要 remove()
```

**案例3：监听器未移除**
```java
// 注册监听器后未取消
component.addListener(listener);
// 页面销毁时未 remove
```

**案例4：单例持有对象引用**
```java
// 单例持有 Activity/Context 引用，导致 Activity 无法回收
```

## 🧳 4. 死锁与 CPU 高排查

### 📘 死锁排查

```bash
# jstack 检测死锁
jstack -l <pid> | grep -A 10 "deadlock"

# 查看阻塞线程
jstack <pid> | grep "BLOCKED" -A 5
```

**典型死锁场景**：
1. **互相等待**：A 持有锁 1 等锁 2，B 持有锁 2 等锁 1
2. **循环等待**：A→B→C→A
3. **数据库死锁**：多事务交叉操作同一批记录
4. **文件锁死锁**：多线程竞争文件锁

**解决思路**：
- 统一加锁顺序（先 A 后 B）
- 使用 tryLock 超时
- 减少锁粒度
- 拆分锁为多个锁

### 📘 CPU 高排查

1. **top -H -p <pid>**：查看高 CPU 线程
2. **jstack <pid>**：导出线程栈
3. **将线程 ID 转十六进制**：printf '%x\n' <tid>
4. **在 jstack 中搜索该十六进制 ID**：找到对应的栈
5. **分析代码**：找到热点方法

### 📘 常见 CPU 高原因

| 原因 | 特征 | 解决方案 |
| ---- | ---- | -------- |
| 死循环 | 线程 RUNNABLE，CPU 100% | 排查代码逻辑 |
| 频繁 GC | GC 线程占用高 | 优化内存使用 |
| 锁竞争 | 大量线程 BLOCKED | 减少锁竞争 |
| 序列化 | IO 线程高 | 优化 IO |

## 🧳 5. jstack / jmap 命令示例与解读

### jstack：线程栈与死锁

```bash
# 查看 Java 进程
jps -l

# 导出线程栈（pid 为进程 ID）
jstack <pid> > thread.txt
```

**解读**：
- 打开 `thread.txt`，搜索 **"deadlock"**，若有会直接提示 `Found one Java-level deadlock`，并列出涉及线程与锁。
- 看线程状态：**RUNNABLE** 为运行中，**BLOCKED** 为等锁，**WAITING** 为 wait()/park；结合栈帧中的类与方法名定位业务代码。
- **CPU 高**：在 Linux 上用 `top -H -p <pid>` 得到高 CPU 的线程 id（十进制），转为十六进制后到 thread.txt 里搜该 id，对应栈即为热点。

**示例片段（死锁）**：
```
Found one Java-level deadlock:
  Thread-1: waiting to lock 0x00000000a1b2c3d4 (Object A), held 0x00000000a1b2c5e6 (Object B)
  Thread-2: waiting to lock 0x00000000a1b2c5e6 (Object B), held 0x00000000a1b2c3d4 (Object A)
```

### jmap：堆信息与 heap dump

```bash
# 堆概要（堆配置、各代容量与使用）
jmap -heap <pid>

# 导出堆快照（OOM 排查、泄漏分析）
jmap -dump:format=b,file=heap.hprof <pid>
```

**解读**：
- **-heap**：看堆大小、新生代/老年代容量与使用率，初步判断是否堆不足或某代过满。
- **-dump**：生成 heap.hprof，用 **MAT（Eclipse Memory Analyzer）** 或 VisualVM 打开；看 **Histogram** 找占用多的类，看 **Dominator Tree** 与 **GC Roots 引用链** 找谁在持有大对象或泄漏对象（如未关闭的连接、只增不减的 List）。

**OOM 时自动 dump**（推荐在启动参数中加上）：
```bash
-XX:+HeapDumpOnOutOfMemoryError -XX:HeapDumpPath=/tmp/heap.hprof
```
OOM 发生后会自动生成 dump，便于事后用 MAT 分析。

# 📎 附录：面试题速查

1. **JVM 是什么？** 运行 Java 字节码的虚拟机，负责类加载、执行、内存管理与 GC。
2. **为什么 Java 能跨平台？** 字节码由各平台 JVM 执行，与平台无关。
3. **JVM 内存分区？** 堆、方法区（Metaspace）共享；虚拟机栈、本地方法栈、PC 私有。
4. **类加载过程？** 加载→验证→准备→解析→初始化；准备赋零值，初始化赋真值。
5. **双亲委派？** 先父加载器加载，保证核心类不被替换。
6. **JMM 和 JVM 内存？** JMM 是规范（主存/工作内存）；JVM 内存是运行时堆/栈/方法区等。
7. **堆如何分代？** 新生代（Eden+Survivor）+ 老年代；Metaspace 在本地内存。
8. **如何判断可回收？** 可达性分析，从 GC Roots 不可达则可回收。
9. **G1 特点？** 分区、可设停顿目标、适合大堆。
10. **OOM 怎么排查？** HeapDumpOnOOM + MAT 看引用链与泄漏；jstat 看 GC。
