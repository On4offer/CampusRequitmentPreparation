## 🔧 常用JVM参数速查

### 内存设置

```bash
# 堆内存设置
-Xms512m                   # 初始堆大小
-Xmx2g                     # 最大堆大小
-Xmn256m                   # 新生代大小
-XX:NewRatio=2             # 老年代/新生代比例（默认2，即老年代占2/3）
-XX:SurvivorRatio=8        # Eden/Survivor比例（默认8，即Eden占8/10）

# 元空间（JDK 8+）
-XX:MetaspaceSize=128m     # 初始元空间大小
-XX:MaxMetaspaceSize=256m  # 最大元空间大小

# 栈内存
-Xss1m                     # 每个线程的栈大小（默认1MB）

# 直接内存
-XX:MaxDirectMemorySize=512m  # 最大直接内存（NIO使用）
```

### GC设置

```bash
# 选择垃圾收集器
-XX:+UseSerialGC           # 串行收集器（单线程，适合小应用）
-XX:+UseParallelGC         # 并行收集器（吞吐量优先）
-XX:+UseParallelOldGC      # 并行老年代收集器
-XX:+UseConcMarkSweepGC    # CMS收集器（低延迟，JDK 9废弃）
-XX:+UseG1GC               # G1收集器（JDK 9+默认，平衡吞吐和延迟）
-XX:+UseZGC                # ZGC（超低延迟，JDK 11+）
-XX:+UseShenandoahGC       # Shenandoah GC（低延迟，OpenJDK）

# G1收集器调优
-XX:MaxGCPauseMillis=200   # 最大GC停顿时间目标（默认200ms）
-XX:G1HeapRegionSize=16m   # G1区域大小
-XX:InitiatingHeapOccupancyPercent=45  # 触发并发GC的堆占用率

# GC日志（JDK 9+统一日志）
-Xlog:gc*:file=gc.log:time,uptime,level,tags:filecount=5,filesize=100m

# GC日志（JDK 8）
-XX:+PrintGCDetails
-XX:+PrintGCDateStamps
-Xloggc:gc.log
-XX:+UseGCLogFileRotation
-XX:NumberOfGCLogFiles=5
-XX:GCLogFileSize=100M
```

### 调试与诊断

```bash
# OOM时生成堆转储
-XX:+HeapDumpOnOutOfMemoryError
-XX:HeapDumpPath=/path/to/dumps/

# 发生Full GC前生成堆转储
-XX:+HeapDumpBeforeFullGC

# 打印类加载信息
-XX:+TraceClassLoading
-XX:+TraceClassUnloading

# 打印GC原因
-XX:+PrintGCCause

# 禁用显式GC（防止System.gc()触发Full GC）
-XX:+DisableExplicitGC

# 逃逸分析优化
-XX:+DoEscapeAnalysis        # 开启逃逸分析（默认开启）
-XX:+EliminateAllocations    # 标量替换优化

# 大对象直接进入老年代阈值
-XX:PretenureSizeThreshold=1m  # 仅对Serial和ParNew有效

# 晋升老年代年龄阈值
-XX:MaxTenuringThreshold=15    # 默认15，在Survivor区交换15次后晋升
```

### 性能优化

```bash
# JIT编译器优化
-XX:CompileThreshold=10000   # 方法调用次数阈值（触发C1编译）
-XX:ReservedCodeCacheSize=256m  # 代码缓存大小

# 启用压缩指针（64位JVM默认开启，堆小于32G时有效）
-XX:+UseCompressedOops
-XX:+UseCompressedClassPointers

# 关闭压缩指针（堆大于32G时需要）
-XX:-UseCompressedOops

# 启用字符串去重（G1收集器）
-XX:+UseStringDeduplication

# 大页内存（提升TLB命中率）
-XX:+UseLargePages
```

## 📊 常用诊断命令

### jps - 查看Java进程

```bash
# 基本用法
jps
# 输出示例：
# 12345 JpsDemo
# 12346 SpringBootApplication

# 显示完整包名
jps -l
# 输出示例：
# 12345 sun.tools.jps.Jps
# 12346 com.example.Application

# 显示传递给main方法的参数
jps -m

# 显示JVM参数
jps -v

# 组合使用
jps -lmv
```

### jstat - JVM统计监控

```bash
# 查看GC统计（每1秒输出一次，共10次）
jstat -gc <pid> 1000 10

# 输出字段说明：
# S0C/S1C: Survivor0/1区容量
# S0U/S1U: Survivor0/1区已使用
# EC/EU: Eden区容量/已使用
# OC/OU: 老年代容量/已使用
# MC/MU: 元空间容量/已使用
# CCSC/CCSU: 压缩类空间容量/已使用
# YGC/YGCT: Young GC次数/时间
# FGC/FGCT: Full GC次数/时间
# GCT: 总GC时间

# 查看类加载统计
jstat -class <pid>

# 查看编译统计
jstat -compiler <pid>

# 查看内存统计（百分比）
jstat -gccapacity <pid>

# 查看GC原因（JDK 8+）
jstat -gccause <pid>

# 查看新生代统计
jstat -gcnew <pid>

# 查看老年代统计
jstat -gcold <pid>
```

### jmap - 内存分析

```bash
# 查看堆内存概况
jmap -heap <pid>

# 查看堆中对象的统计信息
jmap -histo <pid>
# 按内存占用排序显示类名和实例数

# 查看堆中对象的统计信息（包含live对象）
jmap -histo:live <pid>

# 生成堆转储文件
jmap -dump:format=b,file=heap.hprof <pid>

# 仅dump存活对象
jmap -dump:live,format=b,file=heap.hprof <pid>

# 查看finalizer队列
jmap -finalizerinfo <pid>

# 查看clstats（类加载器统计）
jmap -clstats <pid>
```

### jstack - 线程分析

```bash
# 打印线程堆栈
jstack <pid>

# 打印线程堆栈并检测死锁
jstack -l <pid>

# 打印纯堆栈（不包含额外信息）
jstack -m <pid>

# 结合grep查找特定线程
jstack <pid> | grep "java.lang.Thread.State" | sort | uniq -c

# 输出到文件
jstack <pid> > thread_dump.txt

# 检测CPU高的线程
# 1. 找到CPU高的进程PID
ps -ef | grep java
# 2. 找到CPU高的线程TID
top -Hp <pid>
# 3. 将TID转为16进制
printf "%x\n" <tid>
# 4. 查看线程堆栈
jstack <pid> | grep -A 30 <hex_tid>
```

### jcmd - 综合诊断工具（JDK 8+）

```bash
# 列出所有可用命令
jcmd <pid> help

# 查看JVM版本

# 查看JVM命令行参数

# 查看GC信息

jcmd <pid> GC.run_finalization  # 触发finalization

# 生成堆转储

# 查看类直方图

# 查看PerfCounter

# 查看VM原生内存

# 飞行记录（JFR）
jcmd <pid> JFR.start name=myrec duration=60s filename=recording.jfr
jcmd <pid> JFR.check
jcmd <pid> JFR.stop name=myrec
```

## 🎯 常用诊断场景

### 场景1：应用OOM排查

```bash
# 1. 配置OOM时自动生成堆转储
java -Xms1g -Xmx1g \
     -XX:+HeapDumpOnOutOfMemoryError \
     -XX:HeapDumpPath=/logs/heapdump.hprof \
     -jar application.jar

# 2. OOM发生后，使用MAT分析堆转储
# - 查看Dominator Tree找到最大对象
# - 查看Path to GC Roots找到引用链
# - 查看Leak Suspects报告

# 3. 常见OOM类型及解决
# java.lang.OutOfMemoryError: Java heap space
#   → 堆内存不足，增加-Xmx或检查内存泄漏
#
# java.lang.OutOfMemoryError: GC overhead limit exceeded
#   → GC回收效率太低，检查大对象或增加堆内存
#
# java.lang.OutOfMemoryError: Metaspace
#   → 元空间不足，增加-XX:MaxMetaspaceSize
#
# java.lang.OutOfMemoryError: Unable to create new native thread
#   → 线程数过多，检查线程泄漏或减少线程池大小
#
# java.lang.OutOfMemoryError: Direct buffer memory
#   → 直接内存不足，增加-XX:MaxDirectMemorySize
```

### 场景2：CPU过高排查

```bash
# 步骤1：找到CPU高的Java进程
jps -l
# 或
top -c -p $(pgrep -d',' java)

# 步骤2：找到CPU高的线程
top -Hp <pid>
# 记录CPU高的线程ID（如：12345）

# 步骤3：线程ID转16进制
printf "%x\n" 12345
# 输出：3039

# 步骤4：查看线程堆栈
jstack <pid> | grep -A 50 "0x3039" 
# 或查看所有线程状态
jstack <pid> | grep "java.lang.Thread.State" | sort | uniq -c

# 步骤5：分析代码
# - 如果是RUNNABLE，查看执行的代码逻辑
# - 如果是BLOCKED，查看等待的锁
# - 如果是WAITING/TIMED_WAITING，查看等待的条件
```

### 场景3：Full GC频繁排查

```bash
# 1. 开启GC日志
java -Xlog:gc*:file=gc.log:time,uptime:filecount=5,filesize=100m -jar app.jar

# 2. 分析GC日志
# 使用gceasy.io或手动分析

# 3. 常见原因及解决
# - 堆内存设置过小 → 增加-Xmx
# - 内存泄漏 → 分析堆转储
# - System.gc()调用 → 使用-XX:+DisableExplicitGC
# - 元空间不足 → 增加-XX:MaxMetaspaceSize
# - 大对象过多 → 优化代码，避免大对象

# 4. 使用jstat实时监控
jstat -gcutil <pid> 1000
# 关注FGC列（Full GC次数）和GCT列（GC总时间）
```

### 场景4：应用卡顿/响应慢排查

```bash
# 1. 查看线程状态
jstack <pid> > thread_dump.txt

# 2. 分析线程dump
# - 查找BLOCKED状态的线程（锁竞争）
# - 查找WAITING状态的线程（等待条件）
# - 查找长时间RUNNABLE的线程（计算密集型）

# 3. 使用arthas进一步诊断
# 安装arthas
curl -O https://arthas.aliyun.com/arthas-boot.jar
java -jar arthas-boot.jar

# 常用命令
thread -n 10                    # 查看CPU占用前10的线程
thread <tid>                    # 查看指定线程堆栈
thread -b                       # 查找阻塞其他线程的线程
trace com.example.Service getOrder '#cost>100'  # 追踪耗时>100ms的方法
watch com.example.Service getOrder '{params,returnObj,throwExp}'  # 观察方法入参和返回值
profiler start                  # 启动性能分析
profiler stop --file /tmp/output.html  # 生成火焰图
```

### 场景5：类加载问题排查

```bash
# 1. 查看类加载情况
jcmd <pid> VM.classloader_stats

# 2. 查看已加载的类
jcmd <pid> GC.class_histogram | head -50

# 3. 开启类加载日志
java -XX:+TraceClassLoading -XX:+TraceClassUnloading -jar app.jar

# 4. 常见类加载问题
# - ClassNotFoundException → 缺少依赖jar包
# - NoClassDefFoundError → 类存在但初始化失败
# - ClassCastException → 不同类加载器加载的相同类
# - Metaspace OOM → 动态生成类过多（如反射、CGLIB）
```

### 场景6：内存泄漏排查

```bash
# 1. 生成堆转储
jmap -dump:live,format=b,file=heap.hprof <pid>

# 2. 使用MAT分析
# - 打开Dominator Tree
# - 按Retained Heap排序
# - 查看最大的对象
# - 查看Path to GC Roots（排除弱引用）

# 3. 常见内存泄漏场景
# - 静态集合持有对象引用
# - 未关闭的资源（流、连接）
# - 监听器未移除
# - ThreadLocal未remove
# - 缓存无过期策略

# 4. 使用arthas实时追踪
# 追踪对象创建
vmtool --action getInstances --className com.example.LeakClass --limit 10
```

## ⚙️ JVM启动脚本模板

### 生产环境推荐配置

```bash
#!/bin/bash

APP_NAME="myapp"
APP_JAR="myapp.jar"
LOG_PATH="/var/log/${APP_NAME}"
DUMP_PATH="/var/log/${APP_NAME}/dumps"

# JVM内存配置
JAVA_OPTS="-server"
JAVA_OPTS="${JAVA_OPTS} -Xms4g -Xmx4g"
JAVA_OPTS="${JAVA_OPTS} -Xmn1536m"
JAVA_OPTS="${JAVA_OPTS} -XX:MetaspaceSize=256m -XX:MaxMetaspaceSize=512m"
JAVA_OPTS="${JAVA_OPTS} -Xss512k"

# GC配置（G1收集器）
JAVA_OPTS="${JAVA_OPTS} -XX:+UseG1GC"
JAVA_OPTS="${JAVA_OPTS} -XX:MaxGCPauseMillis=200"
JAVA_OPTS="${JAVA_OPTS} -XX:InitiatingHeapOccupancyPercent=35"
JAVA_OPTS="${JAVA_OPTS} -XX:+ParallelRefProcEnabled"

# GC日志
JAVA_OPTS="${JAVA_OPTS} -Xlog:gc*:file=${LOG_PATH}/gc.log:time,uptime:filecount=10,filesize=100m"

# OOM处理
JAVA_OPTS="${JAVA_OPTS} -XX:+HeapDumpOnOutOfMemoryError"
JAVA_OPTS="${JAVA_OPTS} -XX:HeapDumpPath=${DUMP_PATH}"

# 性能优化
JAVA_OPTS="${JAVA_OPTS} -XX:+UseStringDeduplication"
JAVA_OPTS="${JAVA_OPTS} -XX:+AlwaysPreTouch"
JAVA_OPTS="${JAVA_OPTS} -XX:+DisableExplicitGC"

# 故障排查
JAVA_OPTS="${JAVA_OPTS} -XX:+PreserveFramePointer"
JAVA_OPTS="${JAVA_OPTS} -XX:ErrorFile=${LOG_PATH}/hs_err_%p.log"

# 启动应用
exec java ${JAVA_OPTS} -jar ${APP_JAR} "$@"
```

### 容器环境配置

```bash
#!/bin/bash

# 容器环境自动感知内存限制

# 其他配置...

exec java ${JAVA_OPTS} -jar app.jar
```

## ⚠️ 常见坑点速查

| 坑点 | 说明 | 正确做法 |
|------|------|----------|
| `-Xmx` 设置过大 | 超过物理内存导致OOM Killer | 留足系统和其他进程内存 |
| 忽略 `Metaspace` | 动态代理类过多导致OOM | 设置 `-XX:MaxMetaspaceSize` |
| 使用 `System.gc()` | 强制Full GC影响性能 | 使用 `-XX:+DisableExplicitGC` |
| 堆转储文件过大 | 大堆内存生成dump很慢 | 使用 `live` 选项只dump存活对象 |
| 生产环境用 `jmap -dump` | 会触发Full GC | 使用 `jcmd GC.heap_dump` 或配置自动dump |
| GC日志不轮转 | 日志文件无限增长 | 配置 `filecount` 和 `filesize` |
| 忽略线程栈大小 | 线程数多导致内存不足 | 设置 `-Xss` 或限制线程池大小 |
| 32G堆内存陷阱 | 超过32G压缩指针失效 | 要么<32G，要么>48G |

## 📝 面试常见问题

### 1. 如何排查线上OOM？

```
1. 配置-XX:+HeapDumpOnOutOfMemoryError自动生成dump
2. 使用MAT分析dump文件
3. 查看Dominator Tree找最大对象
4. 查看Path to GC Roots找引用链
5. 结合代码分析泄漏原因
6. 常见原因：静态集合、未关闭资源、ThreadLocal未remove
```

### 2. 如何排查CPU 100%？

```
1. top找到CPU高的进程
2. top -Hp找到CPU高的线程
3. 线程ID转16进制
4. jstack查看线程堆栈
5. 分析代码找出热点
```

### 3. JVM调优经验

```
1. 先监控再优化，不要过早优化
2. 设置合理的堆大小（经验值：老年代保存所有存活对象*2）
3. 选择合适的GC收集器
   - 低延迟：ZGC/Shenandoah
   - 吞吐量：Parallel GC
   - 平衡：G1
4. 开启GC日志，持续监控
5. 避免大对象和内存泄漏
```
