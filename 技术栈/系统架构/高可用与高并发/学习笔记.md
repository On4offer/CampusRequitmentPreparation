# 📘《高可用与高并发学习与实战手册》

> 系统学习见下文各章；日常常用概念与使用场景速查见同目录《常用概念与使用场景》。  
> 适用：高可用与高并发设计，面向校招与后端/架构（限流、熔断降级、负载均衡、冗余、MQ、容量与压测）。

------

## 第1章：高可用与高并发概述

本章建立对**高可用（HA）与高并发**的整体认知：指标是什么、常用手段有哪些，为后续限流（第 2 章）、熔断降级（第 3 章）、负载均衡（第 4 章）等打基础。校招常从“如何保证高可用、如何支撑高并发”问起。

------

### 1️⃣ 高可用（HA）

- **高可用**：系统在约定时间内**持续可用**，通常用**可用率**（如 99.9%、99.99%）和**故障恢复时间 RTO**（Recovery Time Objective）衡量。**为什么重要**：故障不可避免，通过设计减少不可用时间与影响范围。
- **手段**：**冗余**（多副本、多机房）、**故障转移**（主从切换、自动摘除故障节点）、**熔断降级**、**限流**、**监控与告警**。

### 2️⃣ 高并发

- **高并发**：单位时间内**大量请求**同时到达，系统需在**响应时间（RT）**与**吞吐量（QPS/TPS）**可接受下稳定处理。**为什么重要**：大促、热点事件下不宕机、不超时。
- **手段**：**水平扩容**、**缓存**、**异步与消息队列**、**限流**、**数据库与连接池优化**、**CDN 与静态资源分离**。

| 维度 | 高可用侧重 | 高并发侧重 |
|------|------------|------------|
| 目标 | 少宕机、快恢复 | 高吞吐、低延迟 |
| 典型手段 | 冗余、故障转移、熔断 | 扩容、缓存、异步、限流 |

------

### 📖 本章小结

| 知识点 | 面试关键词 | 实际应用 |
|--------|------------|----------|
| 高可用 | 可用率、RTO、冗余、故障转移、熔断限流 | 设计“不挂、挂了能恢复” |
| 高并发 | QPS、RT、扩容、缓存、异步、限流 | 设计“扛住流量、不拖垮” |

------

### ⚠️ 常见坑与注意点

1. **现象**：只堆机器不做了限流熔断。**原因**：流量突增或下游故障仍会打垮系统。**正确做法**：限流保护自己、熔断保护下游，与扩容、缓存一起用。
2. **现象**：把可用率当唯一指标忽略 RT。**正确做法**：可用率是“能响应”，还要看 RT 和错误率；慢也是变相不可用。

------

### 🎯 面试常见追问

| 面试官提问 | 回答思路 |
|------------|----------|
| 如何保证高可用？ | 冗余无单点、故障转移、熔断降级、限流、监控告警；各举一例。若追问限流/熔断见第 2、3 章。 |
| 如何支撑高并发？ | 水平扩容、缓存、异步/MQ、限流、DB/连接池优化、CDN；说清“先限流保护，再扩容+缓存+异步提升能力”。 |

## 第2章：限流

本章解决**如何控制请求速率、保护系统与下游**，与第 1 章高可用、第 3 章熔断（限流防过载、熔断防故障扩散）配合。校招常考：限流与熔断区别、令牌桶与漏桶、单机与分布式限流。

------

### 1️⃣ 目的与场景

- **目的**：**保护系统与下游**，防止过载导致雪崩；**公平性**或按用户/接口分配资源。**为什么需要**：流量突增或恶意请求时，限流先挡一层，避免 DB、下游被打垮。
- **场景**：接口 QPS 上限、单用户频率限制、秒杀防超卖（限流+库存）。

### 2️⃣ 限流算法对比

| 算法 | 思路 | 特点 | 缺点 |
|------|------|------|------|
| **固定窗口计数** | 单位时间内计数，超则拒绝 | 简单 | 窗口边界突发（如 1s 内前 0.5s 打满、后 0.5s 又打满） |
| **滑动窗口** | 按当前时间往前滑动窗口内计数 | 平滑、边界问题缓解 | 实现与存储略复杂 |
| **漏桶** | 请求进桶，以**恒定速率**流出 | 输出平滑、无突发 | 无法应对合理突发流量 |
| **令牌桶** | 以恒定速率**放令牌**，请求取到令牌才通过 | **允许一定突发**（桶满时），常用 | 需选好桶大小与速率 |

> 💬 **面试怎么说**：令牌桶允许突发（桶里有令牌就能取），漏桶严格恒定输出；实际多用令牌桶（如 Guava RateLimiter、Sentinel）。

### 3️⃣ 实现层级

| 层级 | 典型 | 适用 |
|------|------|------|
| **网关** | Nginx limit_req、Spring Cloud Gateway | 全局限流、按 IP/路由 |
| **应用内** | Sentinel、Guava RateLimiter | 单机、按接口/资源 |
| **分布式** | Redis + Lua（计数/令牌桶） | 多实例共享配额 |

### 4️⃣ 与熔断区别

- **限流**：**主动**控制流量上限，正常下游也会限；**防过载**。  
- **熔断**：**被动**响应下游故障/慢调用，快速失败、不再请求；**防雪崩**。两者常配合：限流先控量，熔断在异常时断路。

------

### 📖 本章小结

| 知识点 | 面试关键词 | 实际应用 |
|--------|------------|----------|
| 限流目的 | 保护系统与下游、防过载、公平 | 接口 QPS、用户限频、秒杀 |
| 算法 | 固定/滑动窗口、漏桶、令牌桶、突发 | 选型：令牌桶常用 |
| 层级 | 网关、应用内、分布式（Redis+Lua） | Sentinel、Gateway、Redis |
| 与熔断 | 限流控量、熔断断路 | 面试必区分 |

------

### ⚠️ 常见坑与注意点

1. **现象**：固定窗口在窗口边界产生双倍通过量。**正确做法**：用滑动窗口或令牌桶；或窗口足够小使边界影响可接受。
2. **现象**：多实例只做单机限流，总 QPS 仍超。**正确做法**：要全局限流时用**分布式限流**（Redis+Lua 等），共享总配额。
3. **现象**：限流阈值拍脑袋。**正确做法**：结合压测与容量规划定阈值，并配合监控与动态配置。

------

### 🎯 面试常见追问

| 面试官提问 | 回答思路 |
|------------|----------|
| 限流和熔断区别？ | 限流是主动控量、防过载；熔断是下游异常时快速失败、防雪崩。可配合使用。若追问算法见本章 2️⃣。 |
| 令牌桶和漏桶区别？ | 漏桶恒定速率输出、无突发；令牌桶按速率放令牌、允许桶内令牌的突发。实际常用令牌桶。 |
| 单机限流和分布式限流？ | 单机限流每实例独立，总 QPS=单机×实例数；分布式用 Redis 等共享配额，全局限流。 |

## 第3章：熔断与降级

本章解决**下游故障时如何快速失败（熔断）**与**如何保证核心链路可用（降级）**，与第 2 章限流、第 5 章故障转移、微服务章节 Sentinel/Hystrix 关联。校招常考：熔断三态、熔断与限流区别、降级策略设计。

------

### 1️⃣ 熔断

- **熔断**：下游**故障或慢调用比例高**时，**快速失败**、不再请求下游，避免雪崩与资源耗尽。**为什么需要**：一个下游挂掉会导致调用方线程池打满、级联故障；熔断后直接返回或走降级。
- **三态**：**关闭**（正常请求）→ 失败/慢调用超阈值 → **打开**（直接失败、不请求下游）→ 一段时间后 → **半开**（放少量请求探测，成功则关闭，失败则再打开）。详见微服务章节 Sentinel/Hystrix。

### 2️⃣ 降级

- **降级**：**非核心功能**关闭或返回**默认/缓存**数据，保证**核心链路**可用。可与熔断、**开关**（如配置中心、大促开关）配合。
- **场景**：大促关推荐、关评论；依赖超时则返回兜底数据；读多写少场景读缓存、写失败可稍后重试或提示“稍后再试”。

### 3️⃣ 熔断与限流、降级的配合

| 手段 | 目的 | 触发 |
|------|------|------|
| **限流** | 控量、防过载 | 请求量超阈值 |
| **熔断** | 断路、防雪崩 | 下游故障/慢调用超比例 |
| **降级** | 保核心、舍非核心 | 熔断后、或主动开关、或超时 |

------

### 📖 本章小结

| 知识点 | 面试关键词 | 实际应用 |
|--------|------------|----------|
| 熔断 | 快速失败、三态（关/开/半开）、防雪崩 | 下游故障时保护调用方 |
| 降级 | 非核心关/兜底、核心可用、开关 | 大促、超时、熔断后 |
| 配合 | 限流控量、熔断断路、降级兜底 | 一起设计 |

------

### ⚠️ 常见坑与注意点

1. **现象**：熔断打开后一直不探测。**正确做法**：必须有**半开**状态，定期放少量请求探测下游是否恢复，成功则关闭熔断。
2. **现象**：降级策略只有“返回 null”。**正确做法**：返回默认值、缓存、或友好提示（如“推荐暂不可用”），避免前端报错或空白。
3. **现象**：所有接口共用一个熔断器。**正确做法**：按**下游资源/接口**划分熔断器，避免一个下游挂掉拖垮所有调用。

------

### 🎯 面试常见追问

| 面试官提问 | 回答思路 |
|------------|----------|
| 熔断三态是什么？ | 关闭→失败超阈值→打开（直接失败）→一段时间后半开（探测）→成功则关闭。若追问与限流区别见第 2 章。 |
| 如何设计降级策略？ | 区分核心与非核心；非核心可关、返回兜底或缓存；配合熔断（熔断后走降级）、开关（大促关功能）；保证核心链路可用。 |

## 第4章：负载均衡

本章解决**如何将流量分散到多台实例**，提高吞吐与可用性，与第 5 章冗余（多实例是前提）配合。校招常考：四层与七层区别、一致性哈希、Nginx 与 LVS 定位。

------

### 1️⃣ 目的与场景

- **目的**：将流量**分散**到多台实例，**提高吞吐与可用性**，避免单点过载。**为什么需要**：单机能力有限，多实例+负载均衡才能水平扩展。
- **场景**：应用多实例、读从库分流、网关入口分流。

### 2️⃣ 层级对比

| 层级 | 典型 | 特点 | 适用 |
|------|------|------|------|
| **DNS** | 域名解析到不同 IP | 粗粒度、有缓存、切换慢 | 多机房/多区域调度 |
| **四层** | LVS、F5、云 SLB（TCP/UDP） | 基于 IP+端口，性能高、不解 HTTP | 高并发转发、数据库/Redis 代理 |
| **七层** | Nginx、Gateway、HTTP | 基于 URL、Header、可做路由、限流、SSL | 应用层路由、灰度、限流 |

> 💬 **面试怎么说**：四层只看到 IP 和端口，七层能看到 HTTP 内容；四层性能更高，七层功能更多（路由、限流、重写）；Nginx 做七层，LVS 做四层。

### 3️⃣ 常见策略

| 策略 | 思路 | 适用 |
|------|------|------|
| 轮询 / 加权轮询 | 依次或按权重分配 | 实例能力相近或按权重 |
| 随机 | 随机选后端 | 简单 |
| 最少连接 | 选当前连接数最少的 | 长连接、连接耗时不均 |
| **一致性哈希** | 同一 key（如 userId）固定到同一后端 | **缓存**场景（同一用户命中同一节点）、会话保持 |

### 4️⃣ 一致性哈希解决什么问题

- **问题**：普通取模在**扩容/缩容**时，大量 key 会映射到不同节点，导致**缓存大面积失效**。**一致性哈希**：对节点做哈希环，key 顺时针找最近节点；扩容时只影响相邻一段，减少失效范围。可加**虚拟节点**使分布更均匀。

------

### 📖 本章小结

| 知识点 | 面试关键词 | 实际应用 |
|--------|------------|----------|
| 层级 | DNS、四层（LVS）、七层（Nginx）、区别 | 选型、面试对比 |
| 策略 | 轮询、最少连接、一致性哈希 | 缓存用一致性哈希、长连接用最少连接 |
| 一致性哈希 | 扩容少失效、虚拟节点 | 缓存、会话保持 |

------

### ⚠️ 常见坑与注意点

1. **现象**：七层负载做大量 body 解析导致性能差。**正确做法**：只做必要解析（如 URI、Header）；大 body 转发可考虑四层或只转发不解析。
2. **现象**：一致性哈希节点少时分布不均。**正确做法**：加**虚拟节点**（一个物理节点对应多个虚拟节点在环上），使 key 分布更均匀。
3. **现象**：DNS 做细粒度负载均衡。**正确做法**：DNS 有缓存、切换慢，细粒度用四层/七层；DNS 适合多机房/多区域调度。

------

### 🎯 面试常见追问

| 面试官提问 | 回答思路 |
|------------|----------|
| 四层和七层负载均衡区别？ | 四层看 IP+端口、性能高、不解应用层；七层看 HTTP、可路由/限流/改写。Nginx 七层、LVS 四层。若追问一致性哈希见本章 4️⃣。 |
| 一致性哈希解决什么问题？ | 扩容/缩容时减少缓存失效范围；key 顺时针找最近节点，只影响相邻一段。可加虚拟节点使分布均匀。 |

## 第5章：冗余与故障转移

本章解决**如何消除单点、发现故障并转移**，是高可用的核心手段，与第 1 章高可用、第 4 章负载均衡（多实例前提）关联。校招常考：无单点设计、主从切换、多活与一致性。

------

### 1️⃣ 无单点（冗余）

- **原则**：关键组件**多副本**，单点故障不导致整体不可用。**应用**：多实例+负载均衡；**DB** 主从/集群；**Redis** 哨兵/集群；**MQ** 集群；**注册中心** 集群（如 Eureka 多节点）。
- **为什么需要**：任何节点都会故障，冗余是可用性的基础。

### 2️⃣ 故障发现

| 方式 | 说明 | 典型 |
|------|------|------|
| **健康检查** | 定期 HTTP/TCP 探活 | K8s liveness/readiness、负载均衡健康检查 |
| **心跳** | 客户端或服务端上报存活 | 注册中心（Eureka、Nacos） |
| **注册中心剔除** | 心跳超时则从注册表摘除 | 流量不再打到故障实例 |

### 3️⃣ 故障转移

- **主从切换**：DB、Redis 主挂了由**哨兵/协调者**触发从提升为主；应用无状态则只需负载均衡摘除故障实例。
- **流量摘除**：健康检查失败或熔断后，负载均衡/网关**不再把流量**打到该节点；恢复后再加回。
- **熔断**：下游故障时调用方熔断，不再请求该下游（见第 3 章）。

### 4️⃣ 多机房/多活

- **同城双活、异地多活**：多机房同时提供服务，**数据同步**与**一致性**是难点（最终一致、冲突处理）。**流量调度**：DNS 解析到不同机房、或网关按路由/灰度到不同机房。
- **面试怎么说**：多活要解决数据同步（延迟、冲突）和流量调度；同城延迟小易做，异地要接受最终一致与更复杂冲突处理。

------

### 📖 本章小结

| 知识点 | 面试关键词 | 实际应用 |
|--------|------------|----------|
| 无单点 | 多副本、应用/DB/Redis/MQ 集群 | 高可用前提 |
| 故障发现与转移 | 健康检查、心跳、摘除、主从切换 | 自动恢复、少人工 |
| 多活 | 同城/异地、数据同步、流量调度 | 更高可用、容灾 |

------

### ⚠️ 常见坑与注意点

1. **现象**：主从切换后应用仍连旧主。**正确做法**：应用通过**域名或 VIP** 连 DB/Redis，主从切换时改域名指向或 VIP 漂移；或使用支持自动感知的客户端（如 Redis 哨兵客户端）。
2. **现象**：健康检查过于敏感导致误摘。**正确做法**：连续失败 N 次再摘除、配合就绪检查（如刚启动不参与流量）；避免短暂抖动导致实例被摘。
3. **现象**：多活数据强一致。**正确做法**：异地多活通常**最终一致**，需业务能接受短暂不一致或做冲突解决（如 last-write-wins、业务合并）。

------

### 🎯 面试常见追问

| 面试官提问 | 回答思路 |
|------------|----------|
| 如何保证无单点？ | 应用多实例、DB 主从/集群、Redis 哨兵/集群、MQ 集群、注册中心集群；关键组件都要有冗余。若追问故障转移见本章 2️⃣3️⃣。 |
| 主从切换后应用怎么感知？ | 通过 VIP/域名访问，切换时改指向；或客户端配合哨兵/注册中心自动发现新主。 |
| 多活要注意什么？ | 数据同步延迟与一致性（多最终一致）、冲突处理、流量调度（DNS/网关）。 |

## 第6章：异步、消息队列与缓存

本章解决**如何用异步与 MQ 提升吞吐、用缓存降低 DB 压力**，与分布式基础中缓存、MQ 事务与幂等衔接。校招常考：为什么用 MQ、消息不丢与幂等、多级缓存。

------

### 1️⃣ 异步

- **思路**：**耗时操作**异步处理，请求快速返回，避免阻塞线程。**实现**：线程池、CompletableFuture、**消息队列**（把任务发到 MQ，另侧消费）。**为什么需要**：同步等待下游或 IO 会占满线程，异步可提高吞吐与响应时间。

### 2️⃣ 消息队列

- **作用**：**削峰填谷**（峰值写入 MQ，消费端按能力消费）、**解耦**（生产与消费独立扩容）、**异步**（发完即返回）。
- **选型**：**Kafka** 高吞吐、日志/大数据场景；**RabbitMQ** 功能丰富、消息模型多；RocketMQ 事务消息、顺序消息。
- **注意**：**堆积**（消费慢要扩容或降级）、**顺序**（单分区/单队列有序）、**幂等**（消费端必做，见分布式基础）。

### 3️⃣ 缓存

- **思路**：**多级缓存**（本地缓存 + Redis）、**热点数据**前置，减少 DB 压力与 RT。注意**穿透/击穿/雪崩**（见《分布式基础》第 6 章）；缓存与 DB 一致用 Cache-Aside、过期兜底。

### 4️⃣ 消息不丢与幂等（面试常问）

- **不丢**：生产者**确认**（ack）、Broker **持久化**、消费者**先处理再 ack**（或至少一次+幂等）；事务消息则按半消息+确认流程。
- **幂等**：消费端用**业务唯一键**去重（DB 唯一索引或 Redis），重复消息只处理一次。

------

### 📖 本章小结

| 知识点 | 面试关键词 | 实际应用 |
|--------|------------|----------|
| 异步 | 耗时异步、线程池、MQ | 提高吞吐、快速响应 |
| MQ | 削峰填谷、解耦、异步、Kafka/RabbitMQ | 秒杀、日志、事件驱动 |
| 消息不丢与幂等 | 生产者 ack、持久化、消费 ack、唯一键去重 | 面试必答 |
| 缓存 | 多级缓存、热点、穿透/击穿/雪崩 | 见分布式基础 |

------

### ⚠️ 常见坑与注意点

1. **现象**：消费端先 ack 再处理。**原因**：处理失败则消息已 ack，会丢。**正确做法**：**先处理成功再 ack**；若用自动 ack 则需保证处理幂等与重试。
2. **现象**：MQ 只当“发出去就行”不关心 ack。**正确做法**：生产者要处理 **ack/失败重试**，否则网络抖动会导致消息丢失。
3. **现象**：缓存和 DB 双写顺序错。**正确做法**：Cache-Aside 写时**先更新 DB 再删缓存**；避免先删缓存再写 DB（并发下可能把旧数据又写回缓存）。

| 面试官提问 | 回答思路 |
|------------|----------|
| 为什么用 MQ？ | 削峰填谷、解耦、异步；各举一例（秒杀削峰、订单与库存解耦）。若追问不丢与幂等见本章 4️⃣。 |
| 如何保证消息不丢？ | 生产者 ack 与重试、Broker 持久化、消费者先处理再 ack；事务消息按对应流程。 |
| 消费端为什么要幂等？ | 消息可能重复投递、重试；不幂等会重复扣款/加积分。用业务唯一键去重。 |

## 第7章：容量规划与压测

本章解决**如何评估系统能扛多少流量、如何压测与扩容**，与第 1 章高并发、第 2 章限流（阈值可参考压测）关联。校招常考：如何评估 QPS、压测指标、容量预留与弹性伸缩。

------

### 1️⃣ 容量规划

- **思路**：根据**目标 QPS**、**响应时间 RT**、**单机能力**（压测得出）估算**实例数**；**留冗余**应对峰值（如 1.5～2 倍）。公式可简化为：实例数 ≥ 目标 QPS / 单机可支撑 QPS，并考虑 RT 与冗余。
- **为什么需要**：避免流量上来才临时扩容、心里有数“加多少机器能扛住”。

### 2️⃣ 压测

- **目的**：模拟高并发，找到**瓶颈**与**单机能力**。**工具**：JMeter、ab、wrk 等。**关注指标**：**QPS/TPS**、**RT**（响应时间、P99）、**错误率**、**CPU/内存/IO**；瓶颈常在 DB、Redis、下游、线程池、连接池。
- **面试怎么说**：压测要逼近真实流量、逐步加压；看 QPS 与 RT 曲线，找到拐点与瓶颈点；单机能力用于容量规划。

### 3️⃣ 扩容方式

| 方式 | 说明 | 适用 |
|------|------|------|
| **水平扩容** | 加机器、加实例 | 无状态应用、首选 |
| **垂直扩容** | 升配（CPU/内存） | 单机瓶颈、有上限 |
| **弹性伸缩** | 按负载自动扩缩（K8s HPA、云 SLB） | 流量波动大、节省成本 |

### 4️⃣ 容量预留

- 日常按**峰值一定比例**预留（如 1.5 倍）；大促前**提前压测与扩容**；配合**限流**防止超过容量导致雪崩。

------

### 📖 本章小结

| 知识点 | 面试关键词 | 实际应用 |
|--------|------------|----------|
| 容量规划 | QPS、RT、单机能力、实例数、冗余 | 大促前评估、采购/弹性 |
| 压测 | JMeter、QPS/RT/错误率、找瓶颈 | 上线前、扩容依据 |
| 扩容 | 水平、垂直、弹性伸缩 | 日常与大促 |

------

### ⚠️ 常见坑与注意点

1. **现象**：压测环境与生产差异大（数据量、网络、依赖）。**正确做法**：尽量贴近生产（数据量级、依赖 mock 或真实）；至少单机能力要在相近环境测。
2. **现象**：只压接口不压下游。**正确做法**：全链路压测或对下游做压测，找到真正瓶颈（常是 DB、Redis、第三方）。
3. **现象**：没有限流就按压测 QPS 当容量。**正确做法**：容量要配合限流使用，防止超预期流量打进来；限流阈值可参考压测拐点。

------

### 🎯 面试常见追问

| 面试官提问 | 回答思路 |
|------------|----------|
| 如何评估系统能扛多少 QPS？ | 压测得到单机 QPS/RT，按目标 QPS 和冗余算实例数；瓶颈在 DB/Redis 时要先提升下游或加缓存。若追问压测指标见本章 2️⃣。 |
| 压测关注哪些指标？ | QPS、RT（含 P99）、错误率、CPU/内存；找到瓶颈（DB、Redis、线程池等）与拐点。 |
| 如何做容量预留？ | 按峰值留 1.5～2 倍冗余；大促前压测+提前扩容；配合限流防止超载。 |

## 第8章：监控与告警

本章解决**如何监控系统健康、发现问题并告警**，是高可用运维的核心能力，与第 1 章高可用、第 7 章压测联动。校招常考：监控指标、Prometheus + Grafana 方案、告警设计。

------

### 1️⃣ 为什么需要监控

- **目的**：**发现问题**（故障、性能下降）、**定位问题**（哪个服务/组件）、**容量规划**（资源使用趋势）、**业务洞察**（QPS、转化率）。
- **无监控的后果**：故障被动发现（用户投诉）、无法快速定位、不知道系统瓶颈在哪。

### 2️⃣ 监控指标分类

| 类型 | 指标 | 说明 |
|------|------|------|
| **系统指标** | CPU、内存、磁盘、网络 IO | 基础资源使用情况 |
| **应用指标** | QPS、RT（平均/P99/P95）、错误率、线程池状态 | 服务健康度 |
| **中间件指标** | DB 连接数、Redis 内存/命中率、MQ 堆积 | 依赖组件状态 |
| **业务指标** | 订单量、支付成功率、用户活跃数 | 业务健康度 |

> 💬 **黄金指标**：**延迟（Latency）**、**流量（Traffic）**、**错误（Errors）**、**饱和度（Saturation）**——Google SRE 四大黄金信号。

### 3️⃣ Prometheus + Grafana 架构

```
┌─────────────────────────────────────────────────────────────┐
│                    监控架构                                  │
├─────────────────────────────────────────────────────────────┤
│  数据采集              数据存储              可视化          │
│  ├── Exporter    ──→  ├── Prometheus    ──→  Grafana       │
│  │   ├── Node         │   ├── TSDB            ├── Dashboard│
│  │   ├── MySQL        │   ├── 告警规则        └── 告警通知  │
│  │   ├── Redis        │   └── PromQL                       │
│  │   └── App          └── 远程存储（可选）                   │
│  └── Pushgateway                                          │
└─────────────────────────────────────────────────────────────┘
```

#### 核心组件

| 组件 | 作用 |
|------|------|
| **Prometheus** | 时序数据库、数据采集与存储、告警规则评估 |
| **Exporter** | 暴露指标接口（如 node_exporter 采集主机指标、mysqld_exporter 采集 MySQL 指标） |
| **Grafana** | 可视化面板、多数据源支持、告警通知 |
| **Alertmanager** | 告警去重、分组、路由、静默、通知（邮件/钉钉/Slack） |

### 4️⃣ Prometheus 核心概念

#### （1）数据模型

- **指标名**：如 `http_requests_total`、`process_cpu_seconds_total`
- **标签**：维度信息，如 `method="GET"`、`status="200"`、`service="order"`
- **时序**：`指标名{标签}` → 一条时间序列

```
http_requests_total{method="GET",status="200",service="order"} 12345
http_requests_total{method="POST",status="500",service="order"} 67
```

#### （2）PromQL 常用查询

| 场景 | PromQL 示例 |
|------|-------------|
| 查询 QPS | `rate(http_requests_total[5m])` |
| 查询 P99 延迟 | `histogram_quantile(0.99, rate(http_request_duration_seconds_bucket[5m]))` |
| 查询错误率 | `rate(http_requests_total{status=~"5.."}[5m]) / rate(http_requests_total[5m])` |
| 查询 CPU 使用率 | `rate(process_cpu_seconds_total[1m])` |
| 按服务聚合 | `sum by (service) (rate(http_requests_total[5m]))` |

### 5️⃣ Grafana Dashboard 设计

| 面板 | 内容 |
|------|------|
| **概览面板** | 总 QPS、平均 RT、错误率、在线实例数 |
| **服务面板** | 各服务 QPS、RT 分布、错误明细 |
| **资源面板** | CPU、内存、磁盘、网络趋势图 |
| **中间件面板** | DB 连接数、Redis 命中率、MQ 堆积 |

### 6️⃣ 告警设计

#### （1）告警规则示例

```yaml
groups:
  - name: service-alerts
    rules:
      - alert: HighErrorRate
        expr: rate(http_requests_total{status=~"5.."}[5m]) / rate(http_requests_total[5m]) > 0.05
        for: 5m
        labels:
          severity: critical
        annotations:
          summary: "服务错误率过高"
          description: "服务 {{ $labels.service }} 错误率 {{ $value | humanizePercentage }}"
      
      - alert: HighLatency
        expr: histogram_quantile(0.99, rate(http_request_duration_seconds_bucket[5m])) > 1
        for: 5m
        labels:
          severity: warning
        annotations:
          summary: "P99 延迟过高"
```

#### （2）告警分级

| 级别 | 含义 | 响应时间 | 通知方式 |
|------|------|----------|----------|
| **P0 Critical** | 服务不可用、核心功能故障 | 立即响应（5 分钟内） | 电话 + 短信 + IM |
| **P1 Warning** | 性能下降、资源告警 | 30 分钟内响应 | 短信 + IM |
| **P2 Info** | 非关键告警、趋势告警 | 工作时间处理 | IM + 邮件 |

#### （3）告警最佳实践

- **避免告警风暴**：相同告警**分组**、**去重**、设置**静默期**。
- **告警可操作**：告警内容包含**定位信息**（服务、实例、错误详情）和**处理建议**。
- **分级响应**：不同级别不同响应速度，避免 P2 告警淹没 P0。
- **定期复盘**：告警太多要收敛、误报要优化规则、漏报要补充规则。

### 7️⃣ Spring Boot 集成 Prometheus

```xml
<dependency>
    <groupId>io.micrometer</groupId>
    <artifactId>micrometer-registry-prometheus</artifactId>
</dependency>
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-actuator</artifactId>
</dependency>
```

```yaml
management:
  endpoints:
    web:
      exposure:
        include: prometheus,health,info,metrics
  metrics:
    tags:
      application: ${spring.application.name}
```

访问 `/actuator/prometheus` 即可暴露指标，Prometheus 配置抓取：

```yaml
scrape_configs:
  - job_name: 'spring-boot'
    metrics_path: '/actuator/prometheus'
    static_configs:
      - targets: ['localhost:8080']
```

### ⚠️ 常见坑与注意点

2. **现象**：告警风暴导致麻木。**正确做法**：告警**分组去重**、设置静默期；分级响应；定期清理无效告警。
3. **现象**：只监控应用不监控中间件。**正确做法**：DB、Redis、MQ 等中间件也要监控，它们往往是瓶颈。
