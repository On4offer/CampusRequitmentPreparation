# 📘《微服务架构学习与实战手册》

> 系统学习见下文各章；日常常用概念与使用场景速查见同目录《常用概念与使用场景》。  
> 适用：微服务架构与 Spring Cloud 生态，面向校招与后端（注册发现、网关、配置中心、调用、熔断限流、分布式事务与链路追踪）。

------

## 第1章：微服务架构概述

本章建立对**微服务**的整体认知：是什么、与单体对比、核心要素有哪些，为后续注册发现（第 2 章）、网关（第 3 章）、配置与调用（第 4～5 章）、熔断与事务（第 6～7 章）打基础。校招常从“微服务和单体区别”问起。

------

### 1️⃣ 什么是微服务

- **微服务**：将单体应用拆分为一组**小服务**，每个服务**围绕单一业务能力**、**独立部署与扩展**、使用**轻量通信**（HTTP/RPC、MQ），可**独立技术选型与数据库**。**为什么需要**：单体膨胀后发布耦合、扩展困难、技术栈单一；微服务可独立迭代、按需扩缩、技术多样。
- **与单体对比**：单体**简单**、无分布式问题，但扩展与发布**耦合**、改一处牵一发动全身；微服务**灵活**、可独立部署，但带来**分布式复杂度**（网络、一致性、运维、链路追踪）。

| 维度 | 单体 | 微服务 |
|------|------|--------|
| 部署 | 整体发布 | 服务独立发布 |
| 扩展 | 整体扩或按模块难 | 按服务扩缩 |
| 技术 | 统一技术栈 | 可异构 |
| 复杂度 | 无分布式 | 网络、事务、治理 |

### 2️⃣ 核心要素

- **服务拆分**：按**业务域/限界上下文**划分，避免过大或过细。**独立部署**：每个服务可单独发版。**轻量通信**：REST、gRPC、消息队列。**去中心化**：数据与治理可分散，通过**网关**与**注册中心**统一入口与发现。

------

### 📖 本章小结

| 知识点 | 面试关键词 | 实际应用 |
|--------|------------|----------|
| 微服务定义 | 小服务、单一职责、独立部署、轻量通信 | 与单体对比、选型 |
| 核心要素 | 拆分、独立部署、网关、注册中心 | 画微服务整体图 |

------

### ⚠️ 常见坑与注意点

1. **现象**：拆得过细导致调用链过长、运维爆炸。**正确做法**：按业务域拆，先粗后细；服务间调用要有熔断与超时，避免级联故障。
2. **现象**：认为微服务一定比单体好。**正确做法**：小团队、业务简单时单体更省事；微服务适合业务复杂、团队多、需要独立迭代与扩展的场景。
3. **现象**：每个服务各搞一套配置与注册。**正确做法**：用**配置中心**与**注册中心**统一管理，避免配置散落与地址写死。

------

### 🎯 面试常见追问

| 面试官提问 | 回答思路 |
|------------|----------|
| 微服务和单体有什么区别？ | 单体整体部署、扩展耦合；微服务按业务拆成小服务、独立部署与扩展、轻量通信，可独立技术选型，但带来分布式与运维复杂度。若追问拆分见本章 2️⃣。 |
| 微服务有哪些核心组件？ | 注册中心、网关、配置中心、服务间调用（REST/gRPC/MQ）、熔断限流、链路追踪；各说一句作用。若追问具体见第 2～7 章。 |

## 第2章：服务注册与发现

本章解决**服务实例如何被调用方发现**，避免写死 IP、支持动态扩缩，是微服务治理的基础。与第 5 章服务调用（需先发现再调用）、第 4 章配置中心（Nacos 可二合一）关联。校招常考：为什么需要、流程、与 Spring Cloud 集成。

------

### 1️⃣ 为什么需要

- **问题**：服务实例**动态扩缩**、**IP 变动**（如 K8s Pod），调用方无法写死地址。**解决**：通过**注册中心**，实例启动时**注册**，调用方**发现**实例列表并负载均衡调用。
- **流程**：① 服务启动时向注册中心**注册**（IP、端口、服务名、元数据）；② 调用方**拉取或订阅**实例列表；③ 注册中心**健康检查**或心跳超时**剔除**不可用实例；④ 调用方按负载均衡策略选实例请求。

### 2️⃣ 常见组件对比

| 组件 | 特点 | 备注 |
|------|------|------|
| **Nacos** | 注册+配置中心、AP/CP 可切换、国内常用 | Spring Cloud Alibaba 首选 |
| **Eureka** | AP、简单、Netflix 开源 | 2.x 停更，仍有人用 |
| **Consul** | 服务发现+配置、健康检查、多数据中心 | 与 Nacos 类似 |
| **Zookeeper** | CP、强一致、原多用于协调 | 做注册中心要配合客户端 |

### 3️⃣ 与 Spring Cloud 集成

- **依赖**：如 `spring-cloud-starter-alibaba-nacos-discovery`；**注解**：`@EnableDiscoveryClient`（或 2020+ 自动装配）。**调用**：**RestTemplate** 或 **OpenFeign** 配合 **LoadBalancer**（原 Ribbon）从注册中心取实例列表并做**客户端负载均衡**（轮询、随机等）。

> 💬 **面试怎么说**：服务启动注册到 Nacos，调用方通过 OpenFeign 或 RestTemplate 调服务名，底层从注册中心拉实例列表并负载均衡；实例下线或心跳超时会被剔除。

------

### 📖 本章小结

| 知识点 | 面试关键词 | 实际应用 |
|--------|------------|----------|
| 为什么需要 | 动态扩缩、IP 变动、写死地址不可行 | 微服务前提 |
| 流程 | 注册、发现、健康检查、剔除 | 口述或画图 |
| 组件 | Nacos、Eureka、Consul、ZK | 选型、Spring Cloud 用 Nacos 多 |

------

### ⚠️ 常见坑与注意点

1. **现象**：调用方拿到的实例列表过期。**正确做法**：拉取模式要**定时刷新**；订阅模式（Nacos 推送）更及时；客户端要有**本地缓存+增量更新**。
2. **现象**：实例宕机未及时剔除导致调用失败。**正确做法**：注册中心做**健康检查**或**心跳**，超时剔除；客户端可配合**熔断**快速失败。
3. **现象**：多环境（dev/test/prod）实例混在一个命名空间。**正确做法**：用 **namespace** 或 **group** 隔离环境，调用方只发现同环境实例。

------

### 🎯 面试常见追问

| 面试官提问 | 回答思路 |
|------------|----------|
| 服务注册与发现的流程？ | 服务启动注册 IP+端口到注册中心；调用方拉取或订阅实例列表；健康检查剔除不可用；调用方负载均衡选实例。若追问 Nacos 见本章 2️⃣。 |
| Nacos 和 Eureka 区别？ | Nacos 支持注册+配置、AP/CP 可配、国内生态好；Eureka 仅注册、AP、已停更。Spring Cloud Alibaba 用 Nacos。 |

## 第3章：API 网关

本章解决**统一入口、路由、鉴权、限流等治理能力**，前端/客户端只认网关，后端服务可隐藏与变更。与第 2 章注册发现（网关可基于服务名路由）、第 6 章熔断限流（网关可做全局限流）关联。校招常考：网关职责、与 Nginx 区别、Gateway 特点。

------

### 1️⃣ 职责

- **统一入口**：前端/客户端只访问网关，**路由转发**到后端服务；后端可隐藏、扩容、换地址。**治理**：**鉴权**（统一校验 token）、**限流**、**熔断**、**日志**、**协议转换**（如 HTTP→gRPC）。**为什么需要**：避免每个服务各自鉴权限流、前端要对多个地址；网关集中做横切关注点。

### 2️⃣ 路由与服务发现

- **路由**：按 **path/host** 等将请求转发到不同服务；可配合**注册中心**做**服务发现**（按服务名解析到实例列表再转发）。**断言 + 过滤器**：Spring Cloud Gateway 用 Predicate 匹配请求、Filter 做前置/后置处理（鉴权、限流、改 Header 等）。

### 3️⃣ 常见组件

| 组件 | 特点 | 适用 |
|------|------|------|
| **Spring Cloud Gateway** | 响应式（WebFlux）、断言与过滤器、与 Spring Cloud 集成、限流等 | Java 技术栈、Spring Cloud |
| **Zuul** | Netflix、1.x 同步、2.x 基于 Netty | 老项目、逐步被 Gateway 替代 |
| **Kong** | 基于 Nginx/OpenResty、插件丰富、高性能 | 多语言、对性能要求高 |
| **Nginx** | 七层负载、反向代理、可做简单网关 | 偏流量与静态、少业务逻辑 |

### 4️⃣ 网关与 Nginx 区别（面试常问）

| 维度 | 网关（如 Gateway） | Nginx |
|------|--------------------|-------|
| 定位 | **业务与治理**：鉴权、限流、路由到服务、聚合 | **流量与静态**：反向代理、负载、静态资源、缓存 |
| 集成 | 与注册中心、配置中心、服务发现深度集成 | 一般独立、配置静态或配合 Lua |
| 动态 | 可从注册中心动态拉服务列表 | 多需改配置或配合 API 动态更新 |

> 💬 **面试怎么说**：网关偏业务侧——统一鉴权、限流、按服务名路由、可与注册中心联动；Nginx 偏流量侧——负载、反向代理、静态资源。网关能做的：鉴权、限流、路由、聚合、协议转换。

------

### 📖 本章小结

| 知识点 | 面试关键词 | 实际应用 |
|--------|------------|----------|
| 网关职责 | 统一入口、路由、鉴权、限流、熔断、日志 | 画微服务图时标出网关 |
| 组件 | Gateway、Zuul、Kong、Nginx | 选型、Gateway 与 Spring Cloud |
| 与 Nginx 区别 | 业务治理 vs 流量与静态、与注册中心 | 面试必区分 |

------

### ⚠️ 常见坑与注意点

1. **现象**：所有流量都经网关，网关成瓶颈。**正确做法**：网关可**水平扩展**；限流与熔断保护网关自身；内部服务间调用可直连（经注册中心），不必再经网关。
2. **现象**：鉴权只在网关做一层，下游服务不校验。**正确做法**：敏感操作下游也应校验 token 或传递用户身份，防止绕过网关直连（内网隔离不足时）。
3. **现象**：Gateway 与 Zuul 混用概念。**正确做法**：Gateway 基于 WebFlux 响应式、性能更好；Zuul 1.x 同步阻塞，2.x 基于 Netty；面试说“我们常用 Gateway”即可。

------

### 🎯 面试常见追问

| 面试官提问 | 回答思路 |
|------------|----------|
| 网关能做什么？ | 统一入口、路由转发、鉴权、限流、熔断、日志、协议转换；可与注册中心做服务发现。若追问与 Nginx 区别见本章 4️⃣。 |
| 网关和 Nginx 区别？ | 网关偏业务与治理（鉴权、限流、按服务路由）；Nginx 偏流量与反向代理、静态资源。网关常与注册中心集成。 |

## 第4章：配置中心

本章解决**配置集中管理、动态刷新、多环境隔离**，避免改配置重启、配置散落各机器。与第 2 章（Nacos 可注册+配置二合一）关联。校招常考：为什么需要、流程、与 Spring Cloud 动态刷新。

------

### 1️⃣ 为什么需要

- **问题**：配置**散落**各机器、改配置要**重启**、多环境（dev/test/prod）难隔离。**解决**：**配置中心**集中存储，应用**拉取或订阅**；支持**动态刷新**（如 @RefreshScope）无需重启；**多环境**用 namespace/profile 隔离。

### 2️⃣ 流程

- 配置存于**配置中心**（按应用、环境、key 管理）。应用**启动时拉取**，或**订阅**变更（长轮询/推送）。**动态生效**：Spring Cloud 支持 **@RefreshScope**，配置变更后调用 refresh 或推送后 Bean 重建，无需重启进程。

### 3️⃣ 常见组件

| 组件 | 特点 | 备注 |
|------|------|------|
| **Nacos Config** | 与 Nacos 注册一体、简单、国内常用 | Spring Cloud Alibaba |
| **Spring Cloud Config** | 与 Git 集成、版本天然 | 需配合 Bus 做动态刷新 |
| **Apollo** | 携程开源、权限与灰度强、界面完善 | 企业级、功能多 |

### 4️⃣ 注意点

- **敏感信息**：加密存储或仅内网访问。**版本与回滚**：配置中心支持历史版本与回滚。**多环境**：dev/test/prod 用 **namespace** 或 **profile** 隔离，避免混用。

------

### 📖 本章小结

| 知识点 | 面试关键词 | 实际应用 |
|--------|------------|----------|
| 为什么需要 | 集中管理、动态刷新、多环境、免重启 | 微服务标配 |
| 流程 | 存储、拉取/订阅、动态生效、@RefreshScope | 与 Spring 集成 |
| 组件 | Nacos Config、Config、Apollo | 选型 |

------

### ⚠️ 常见坑与注意点

1. **现象**：配置改了不生效。**正确做法**：确认是否**订阅/拉取**到新配置；@RefreshScope 的 Bean 要在 refresh 触发后才会重建；部分配置需重启才读（如数据源）。
2. **现象**：生产配置被误改。**正确做法**：权限控制、审批流程；配置中心支持版本与**回滚**；敏感配置加密。
3. **现象**：多环境配置串了。**正确做法**：用 **namespace/profile** 严格隔离；应用启动指定 profile，只拉对应环境配置。

------

### 🎯 面试常见追问

| 面试官提问 | 回答思路 |
|------------|----------|
| 为什么需要配置中心？ | 配置集中管理、动态刷新免重启、多环境隔离；避免配置散落与改配置重启。若追问动态刷新见本章 2️⃣。 |
| 配置如何动态生效？ | 应用订阅配置中心变更；Spring Cloud 用 @RefreshScope，refresh 后重建 Bean 读新配置。 |

## 第5章：服务间调用与通信

本章解决**服务之间如何调用**（同步 REST/gRPC、异步 MQ）以及**负载均衡**，与第 2 章注册发现（先发现再调用）、第 6 章熔断（调用失败要熔断）关联。校招常考：同步与异步选型、OpenFeign、负载均衡策略。

------

### 1️⃣ 同步与异步

| 方式 | 典型 | 特点 | 适用 |
|------|------|------|------|
| **同步** | **REST**（OpenFeign）、**gRPC** | 简单直接、实时响应 | 强依赖、需要立即结果；需**超时、重试、熔断** |
| **异步** | **MQ**（RabbitMQ、Kafka） | 解耦、削峰、最终一致 | 事件驱动、日志、非实时 |

### 2️⃣ OpenFeign

- **是什么**：**声明式 HTTP 客户端**，定义接口即可调用远程服务；集成 **负载均衡**（LoadBalancer）与**熔断**（Sentinel/Resilience4j）。**为什么用**：不用手写 RestTemplate、自动从注册中心取实例并负载均衡，代码简洁。
- **使用**：`@FeignClient(name = "service-name")` + 接口方法；配合 **@EnableFeignClients**；可配置超时、重试、熔断降级。

### 3️⃣ 负载均衡

- **位置**：**客户端**（Ribbon、Spring Cloud **LoadBalancer**）或**网关**。**策略**：轮询、随机、权重、最少连接；OpenFeign 默认用 LoadBalancer 的轮询（可换）。
- **面试怎么说**：调用方从注册中心拿到实例列表，在**客户端**做负载均衡选一个实例发请求；Gateway 则在网关层做负载均衡转发。

------

### 📖 本章小结

| 知识点 | 面试关键词 | 实际应用 |
|--------|------------|----------|
| 同步 vs 异步 | REST/gRPC、MQ、超时重试熔断 | 选型：要实时用同步、解耦用 MQ |
| OpenFeign | 声明式、负载均衡、熔断 | Spring Cloud 常用 |
| 负载均衡 | 客户端/网关、轮询随机权重 | 与注册发现配合 |

------

### ⚠️ 常见坑与注意点

1. **现象**：Feign 未配超时导致长时间阻塞。**正确做法**：配置 **connectTimeout**、**readTimeout**；配合熔断避免下游慢拖垮调用方。
2. **现象**：Feign 调用本服务方法走 HTTP 绕一圈。**正确做法**：同类内部调用不经过 Feign，否则失去本地调用语义；若要通过 Feign 调自己需注意循环与负载。
3. **现象**：异步 MQ 不关心消费失败与幂等。**正确做法**：消费端**重试与死信**、**幂等**（见《分布式基础》）；否则消息丢或重复处理。

------

### 🎯 面试常见追问

| 面试官提问 | 回答思路 |
|------------|----------|
| 同步和异步调用怎么选？ | 需要立即结果、强依赖用同步（REST/gRPC），配超时与熔断；解耦、削峰、最终一致用 MQ。若追问 OpenFeign 见本章 2️⃣。 |
| OpenFeign 做了什么？ | 声明式 HTTP 客户端，从注册中心发现服务并做负载均衡调用；可集成熔断与超时配置。 |

## 第6章：熔断、限流与降级

本章解决**下游故障与流量过载时的保护手段**，与《高可用与高并发》中限流熔断一致，在微服务里用 **Sentinel** 等落地。校招常考：熔断与限流区别、Sentinel 规则、如何防雪崩。

------

### 1️⃣ 熔断

- **熔断**：**下游故障或慢调用比例高**时，**快速失败**、不再请求下游，避免雪崩与线程池打满。**三态**：关闭→打开（直接失败）→半开（探测恢复）。**组件**：**Sentinel**、Resilience4j；Hystrix 已停更。
- **为什么需要**：一个下游挂掉会导致调用方大量线程阻塞、级联故障；熔断后直接返回或走降级，保护调用方。

### 2️⃣ 限流

- **限流**：控制 **QPS/并发数**，保护系统与下游；**令牌桶、漏桶、计数器**；可在**网关**或**应用内**（Sentinel、Guava RateLimiter）。与《高可用》第 2 章一致。
- **熔断 vs 限流**：**限流**是主动控量、防过载；**熔断**是下游异常时断路、防雪崩。

### 3️⃣ 降级

- **降级**：**非核心功能**关闭或返回**默认/兜底**数据，保证**核心链路**可用；与熔断配合（熔断后走降级逻辑）。**场景**：大促关推荐、下游超时返回默认值。

### 4️⃣ Sentinel 规则类型（简述）

- **流控**：QPS/并发数限流。**熔断**：慢调用比例/异常比例/异常数触发熔断。**热点**：针对某参数值限流（如某商品 ID）。**系统**：按系统负载（CPU、RT）限流。**授权**：黑白名单。校招能说出“流控、熔断、热点”即可。

------

### 📖 本章小结

| 知识点 | 面试关键词 | 实际应用 |
|--------|------------|----------|
| 熔断 | 快速失败、三态、防雪崩、Sentinel | 下游故障时保护调用方 |
| 限流 | QPS/并发、令牌桶、网关/应用内 | 与熔断配合、见高可用章 |
| 降级 | 非核心关、兜底数据、与熔断配合 | 大促、超时 |
| Sentinel | 流控、熔断、热点、系统规则 | Spring Cloud 常用 |

------

### ⚠️ 常见坑与注意点

1. **现象**：只限流不熔断。**正确做法**：限流控量、熔断在**下游异常**时断路，两者都要；下游挂了光限流仍会堆积请求。
2. **现象**：熔断后没有半开探测。**正确做法**：必须有**半开**状态定期放少量请求探测，成功则关闭熔断，否则继续打开。
3. **现象**：Sentinel 规则只配在网关。**正确做法**：网关可做全局限流；**应用内**也要配熔断与限流，保护本服务与下游。

------

### 🎯 面试常见追问

| 面试官提问 | 回答思路 |
|------------|----------|
| 熔断和限流区别？ | 限流控量、防过载；熔断在下游故障时快速失败、防雪崩。可配合：先限流保护自己，熔断保护下游。若追问 Sentinel 见本章 4️⃣。 |
| 如何防止雪崩？ | 熔断（下游故障快速失败）、限流（控量）、降级（非核心关）、超时与重试控制；保证核心链路可用。 |

## 第7章：分布式事务与链路

本章解决**微服务下的分布式事务**与**链路追踪**，是微服务带来的典型挑战。与《分布式基础》第 5 章分布式事务、系统设计中的一致性取舍一致。校招常考：Seata 与最终一致选型、链路追踪作用、Spring Cloud 与 Dubbo 对比。

------

### 1️⃣ 分布式事务

- **问题**：跨服务、跨库的多个操作需要一致性；单机事务无法跨进程。**方案**：**Seata**（AT/TCC/SAGA）、**本地消息表+MQ**、**最终一致性**；校招需知**强一致 vs 最终一致**取舍（见《分布式基础》）。
- **Seata 简述**：**AT** 模式自动回滚、对业务侵入小；**TCC** 需业务实现 Try/Confirm/Cancel，灵活但开发成本高；**SAGA** 长事务多步+补偿。实际很多场景**最终一致+幂等**即可。

### 2️⃣ 链路追踪

- **问题**：请求**跨多服务**，出问题难排查、性能难分析。**解决**：**traceId** 贯穿整条调用链，每层打日志或上报 **Span**；**Sleuth + Zipkin**、**SkyWalking**、**Jaeger** 等做采集与展示。
- **作用**：看一次请求经过了哪些服务、每段耗时、哪里慢或报错；便于排查与优化。

#### （1）核心概念

| 概念 | 说明 |
|------|------|
| **Trace** | 一次完整请求的调用链，由多个 Span 组成，用 **traceId** 唯一标识 |
| **Span** | 调用链中的一个基本单元（如一次 RPC 调用、一次 DB 查询），有 **spanId**、开始/结束时间、父 Span 引用 |
| **Annotation** | Span 中的事件标记（如 cs 客户端发送、sr 服务端接收、ss 服务端发送、cr 客户端接收） |
| **Tag** | Span 的元数据标签（如 http.url、db.statement、error=true） |

```
请求流程示例：
Client → Service A → Service B → DB

Trace (traceId: abc123)
 ├── Span A (spanId: 1, 服务A处理)
 │    └── Span B (spanId: 2, parentSpanId: 1, 调用服务B)
 │         └── Span C (spanId: 3, parentSpanId: 2, DB查询)
```

#### （2）主流工具对比

| 工具 | 特点 | 适用场景 |
|------|------|----------|
| **SkyWalking** | 国产、APM 全功能（链路+指标+告警）、无侵入（Java Agent）、UI 完善 | 国内主流、生产推荐 |
| **Zipkin** | Twitter 开源、轻量、与 Spring Cloud Sleuth 集成好 | Spring Cloud 生态、中小规模 |
| **Jaeger** | Uber 开源、CNCF 项目、云原生友好、支持多种存储 | Kubernetes 环境、云原生 |
| **Pinpoint** | 韩国开源、应用拓扑图直观、无侵入 | Java 应用、需要可视化拓扑 |

#### （3）SkyWalking 核心架构

```
┌─────────────────────────────────────────────────────────────┐
│                      SkyWalking 架构                         │
├─────────────────────────────────────────────────────────────┤
│  Agent（探针）          OAP Server              UI          │
│  ├── Java Agent   ──→  ├── 接收数据         ──→  展示       │
│  ├── .NET Agent        ├── 聚合分析              ├── 拓扑图  │
│  └── Go Agent          └── 存储                  ├── 链路   │
│         ↓                   ↓                      └── 指标 │
│    上报 Trace          Elasticsearch/H2                     │
└─────────────────────────────────────────────────────────────┘
```

- **Agent**：无侵入字节码增强，自动采集 HTTP、RPC、DB、MQ 调用；通过 gRPC/HTTP 上报。
- **OAP Server**：接收、聚合、存储链路数据；支持 Elasticsearch、H2、MySQL 等存储。
- **UI**：展示服务拓扑、调用链、慢查询、错误统计；支持告警规则。

#### （4）Spring Cloud 集成方式

**方式一：Sleuth + Zipkin（经典）**

```xml
<dependency>
    <groupId>org.springframework.cloud</groupId>
    <artifactId>spring-cloud-starter-sleuth</artifactId>
</dependency>
<dependency>
    <groupId>org.springframework.cloud</groupId>
    <artifactId>spring-cloud-sleuth-zipkin</artifactId>
</dependency>
```

```yaml
spring:
  zipkin:
    base-url: http://localhost:9411
  sleuth:
    sampler:
      probability: 1.0  # 采样率，生产可降低
```

**方式二：SkyWalking Agent（推荐生产）**

```bash
# 启动时挂载 Agent
java -javaagent:/path/to/skywalking-agent.jar \
     -Dskywalking.agent.service_name=my-service \
     -Dskywalking.collector.backend_service=localhost:11800 \
     -jar app.jar
```

#### （5）链路追踪最佳实践

- **采样率**：生产环境不必 100% 采样，可按流量设置（如 10%～50%），降低存储与性能开销。
- **TraceId 透传**：跨线程、异步调用时需手动透传 TraceId（如 MDC、ThreadLocal）；MQ 消息可在 Header 中携带。
- **慢调用阈值**：设置慢调用告警阈值（如 RT > 500ms），及时发现性能瓶颈。
- **错误链路**：关注错误率高的 Span，配合日志快速定位。
- **存储成本**：链路数据量大，需设置 TTL（如保留 7 天），或按重要性分级存储。

### 3️⃣ 微服务带来的挑战（面试常问）

- **分布式事务**：跨服务一致难，多用最终一致+幂等。**链路追踪**：调用链长，需 traceId 串联。**运维**：服务多、部署与监控复杂；需 CI/CD、注册中心、配置中心、日志与监控。**网络与可用性**：网络抖动、下游故障，需熔断、超时、重试。

### 4️⃣ Spring Cloud 与 Dubbo 简要对比

| 维度 | Spring Cloud | Dubbo |
|------|--------------|-------|
| **通信** | HTTP（REST）、OpenFeign | **RPC**（默认 Dubbo 协议） |
| **注册** | Nacos、Eureka 等 | Nacos、Zookeeper 等 |
| **生态** | 网关、Config、Bus、全家桶 | 偏 RPC 与服务治理 |
| **适用** | 多语言、HTTP 友好、云原生 | Java 体系、高性能 RPC、国内历史项目多 |

> 💬 **面试怎么说**：Spring Cloud 是 HTTP 全家桶、组件多、与 Spring 集成好；Dubbo 是 RPC 框架、性能好、国内很多公司用。可提“Spring Cloud Alibaba 里也有 Dubbo 兼容”。

------

### 📖 本章小结

| 知识点 | 面试关键词 | 实际应用 |
|--------|------------|----------|
| 分布式事务 | Seata AT/TCC/SAGA、本地消息表、最终一致 | 选型：能最终一致就最终一致 |
| 链路追踪 | traceId、Span、Zipkin、SkyWalking | 排查与性能分析 |
| 微服务挑战 | 事务、链路、运维、网络 | 面试总结题 |
| Spring Cloud vs Dubbo | HTTP vs RPC、生态 | 选型与对比 |

------

### ⚠️ 常见坑与注意点

1. **现象**：所有跨服务都上 Seata 强一致。**正确做法**：强一致代价大（性能与可用性）；很多业务可接受**最终一致+幂等**，先考虑本地消息表/MQ，再考虑 Seata。
2. **现象**：没有链路追踪，线上问题只能猜。**正确做法**：至少**traceId** 打日志、能串起一次请求；有条件上 SkyWalking/Zipkin 看调用链与耗时。
3. **现象**：把 Spring Cloud 和 Dubbo 对立。**正确做法**：可组合用（如 Spring Cloud + Dubbo RPC）；Spring Cloud Alibaba 支持 Dubbo，按场景选 HTTP 或 RPC。

## 第8章：服务网格（Service Mesh）

本章介绍**服务网格**这一云原生时代的服务治理方案，与前面各章的服务治理方案形成对比。校招常考：Service Mesh 是什么、与 Spring Cloud 区别、Istio 核心组件。

------

### 1️⃣ 什么是 Service Mesh

> **一句话**：Service Mesh 是**基础设施层**，通过**Sidecar 代理**实现服务间通信的统一治理（流量管理、安全、可观测），**业务代码零侵入**。

#### （1）演进历程

```
单体 → 微服务（Spring Cloud/Dubbo） → Service Mesh

单体：
  所有功能在一个进程内

微服务（Spring Cloud）：
  服务A ──→ 服务B
    ↑        ↑
  SDK 层（服务发现、负载均衡、熔断...）
  问题：SDK 升级需改代码、多语言重复实现

Service Mesh：
  服务A ──→ Sidecar ──→ Sidecar ──→ 服务B
            ↑                    ↑
         Envoy/Istio          Envoy/Istio
  治理逻辑下沉到 Sidecar，业务无感知
```

#### （2）核心特点

| 特点 | 说明 |
|------|------|
| **零侵入** | 业务代码无需改动，治理能力下沉到 Sidecar |
| **语言无关** | Sidecar 代理处理通信，支持任意语言服务 |
| **统一治理** | 流量管理、安全、可观测集中配置 |
| **独立升级** | Sidecar 可独立升级，不影响业务 |

#### （3）典型架构

```
┌─────────────────────────────────────────────────────────────┐
│                    Service Mesh 架构                         │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   控制平面（Control Plane）                                  │
│   ┌─────────────────────────────────────────┐               │
│   │  Istiod（Pilot + Citadel + Galley）      │               │
│   │  ├── Pilot：服务发现、流量管理配置        │               │
│   │  ├── Citadel：证书管理、安全认证          │               │
│   │  └── Galley：配置验证与分发               │               │
│   └─────────────────────────────────────────┘               │
│                         │                                   │
│                         ▼ xDS API（Envoy Discovery Service）│
│   数据平面（Data Plane）                                     │
│   ┌─────────────┐    ┌─────────────┐    ┌─────────────┐    │
│   │  服务 A      │    │  服务 B      │    │  服务 C      │    │
│   │  ┌───────┐  │    │  ┌───────┐  │    │  ┌───────┐  │    │
│   │  │ Envoy │  │◄──►│  │ Envoy │  │◄──►│  │ Envoy │  │    │
│   │  │Sidecar│  │    │  │Sidecar│  │    │  │Sidecar│  │    │
│   │  └───────┘  │    │  └───────┘  │    │  └───────┘  │    │
│   └─────────────┘    └─────────────┘    └─────────────┘    │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

------

### 2️⃣ Istio 核心组件

> **Istio** 是目前最流行的 Service Mesh 实现，基于 Envoy 代理。

#### （1）数据平面 - Envoy

| 功能 | 说明 |
|------|------|
| **流量代理** | 拦截所有进出流量，实现负载均衡、路由 |
| **TLS 加密** | 自动 mTLS，服务间通信加密 |
| **指标收集** | 自动采集延迟、流量、错误等指标 |
| **故障注入** | 支持延迟、错误注入，用于混沌测试 |

#### （2）控制平面 - Istiod

| 组件 | 职责 |
|------|------|
| **Pilot** | 服务发现、流量管理配置下发（VirtualService、DestinationRule） |
| **Citadel** | 证书签发、身份认证、mTLS 管理 |
| **Galley** | 配置验证、分发、与 K8s 集成 |

------

### 3️⃣ 核心功能

#### （1）流量管理

```yaml
# VirtualService：定义路由规则
apiVersion: networking.istio.io/v1beta1
kind: VirtualService
metadata:
  name: reviews
spec:
  hosts:
  - reviews
  http:
  - match:
    - headers:
        end-user:
          exact: jason
    route:
    - destination:
        host: reviews
        subset: v2
  - route:
    - destination:
        host: reviews
        subset: v1
```

| 功能 | 说明 |
|------|------|
| **路由** | 基于权重、Header、URI 的路由 |
| **灰度发布** | 按流量比例或用户特征金丝雀发布 |
| **超时/重试** | 统一配置超时、重试策略 |
| **熔断** | 自动熔断异常服务 |

#### （2）安全

| 功能 | 说明 |
|------|------|
| **mTLS** | 服务间自动双向 TLS 加密 |
| **身份认证** | 基于 Service Account 的身份识别 |
| **访问控制** | AuthorizationPolicy 控制谁能访问谁 |

#### （3）可观测性

| 功能 | 说明 |
|------|------|
| **指标** | 自动采集 Prometheus 指标 |
| **日志** | 访问日志统一收集 |
| **链路追踪** | 自动集成 Jaeger/Zipkin |
| **拓扑图** | Kiali 展示服务调用拓扑 |

------

### 4️⃣ Service Mesh vs Spring Cloud

| 维度 | Spring Cloud | Service Mesh（Istio） |
|------|--------------|----------------------|
| **侵入性** | 需引入 SDK、写代码 | **零侵入**，Sidecar 代理 |
| **语言** | 主要 Java | **语言无关** |
| **治理能力** | 代码实现 | 声明式配置（YAML） |
| **性能** | 无额外代理开销 | 有 Sidecar 代理开销 |
| **复杂度** | 应用层复杂 | 基础设施层复杂 |
| **适用** | 传统微服务、Java 为主 | 云原生、多语言、K8s 环境 |

> 💬 **面试怎么说**：Spring Cloud 是应用层 SDK 方案，Java 生态成熟；Service Mesh 是基础设施层方案，零侵入、多语言、适合云原生。两者可共存，Service Mesh 更适合 K8s 环境下的多语言微服务。

------

### 5️⃣ 适用场景与选型

| 场景 | 推荐方案 |
|------|----------|
| Java 单体转微服务 | Spring Cloud |
| 多语言微服务 | Service Mesh |
| 已上 K8s | Service Mesh |
| 对性能极度敏感 | Spring Cloud（无代理开销） |
| 需要快速迭代治理能力 | Service Mesh（无需改代码） |

------

### 📖 本章小结

| 知识点 | 面试关键词 | 实际应用 |
|--------|------------|----------|
| Service Mesh | Sidecar、零侵入、基础设施层 | 云原生服务治理 |
| Istio | Envoy、Pilot、mTLS、流量管理 | K8s 环境下的服务网格 |
| 核心功能 | 流量路由、安全、可观测 | 灰度发布、服务加密、统一监控 |
| 与 Spring Cloud 对比 | SDK vs Sidecar、Java vs 多语言 | 按场景选型 |

------

### ⚠️ 常见坑与注意点

1. **现象**：所有微服务都上 Service Mesh。**正确做法**：Service Mesh 有 Sidecar 性能开销（延迟增加 2-5ms），对性能极度敏感的场景慎用；Java 单体转微服务可先用 Spring Cloud。
2. **现象**：忽略 K8s 依赖。**正确做法**：Istio 深度依赖 Kubernetes，无 K8s 环境部署复杂；需评估团队 K8s 能力。
3. **现象**：期望 Service Mesh 解决所有问题。**正确做法**：Service Mesh 解决的是通信层面的治理，业务逻辑问题仍需业务解决。

## 第9章：DDD 领域驱动设计

本章介绍**领域驱动设计（DDD）**，一种应对复杂业务系统的架构设计方法，与微服务拆分、中台建设密切相关。校招常考：DDD 核心概念、分层架构、聚合与实体区别。

------

### 1️⃣ 为什么需要 DDD

> **一句话**：DDD 是**应对复杂业务**的设计方法，通过**统一语言**和**领域模型**让技术实现与业务逻辑对齐，指导微服务合理拆分。

#### （1）传统开发的问题

| 问题 | 表现 |
|------|------|
| **贫血模型** | 只有 getter/setter，业务逻辑散落在 Service 中 |
| **上帝类** | 一个类几千行，职责混乱 |
| **烟囱系统** | 各系统重复建设，无法复用 |
| **需求失真** | 开发理解的与业务想要的不一致 |

#### （2）DDD 解决什么

- **统一语言（Ubiquitous Language）**：开发、产品、业务用同一套术语沟通。
- **领域模型**：将业务知识沉淀为代码，业务变化映射到模型变化。
- **边界划分**：识别限界上下文，指导微服务拆分。

------

### 2️⃣ DDD 核心概念

#### （1）战略设计（宏观）

| 概念 | 说明 | 示例 |
|------|------|------|
| **领域（Domain）** | 业务问题的范围 | 电商领域、物流领域 |
| **子域（Subdomain）** | 领域的细分 | 核心域（订单）、支撑域（库存）、通用域（用户） |
| **限界上下文（Bounded Context）** | 业务边界，一个上下文内术语一致 | 订单上下文、支付上下文 |
| **上下文映射（Context Map）** | 多个上下文之间的关系 | 订单上下文依赖支付上下文 |

**子域分类**：

```
电商领域
├── 核心域（Core Domain）：订单、支付（竞争力所在，投入最好资源）
├── 支撑域（Supporting Domain）：库存、物流（重要但非核心）
└── 通用域（Generic Domain）：用户、消息（可采购或外包）
```

**限界上下文与微服务**：
- 一个限界上下文通常对应一个微服务。
- 上下文边界就是服务边界，边界内高内聚，边界间低耦合。

#### （2）战术设计（微观）

| 概念 | 说明 | 代码体现 |
|------|------|----------|
| **实体（Entity）** | 有唯一标识，状态可变 | 订单（orderId 唯一，状态可变） |
| **值对象（Value Object）** | 无唯一标识，不可变，属性相等即相等 | 地址、金额、手机号 |
| **聚合（Aggregate）** | 一组相关对象的集合，有根实体，外部只能通过根访问 | 订单聚合（订单+订单项） |
| **聚合根（Aggregate Root）** | 聚合的入口实体 | Order（包含 OrderItems） |
| **领域服务（Domain Service）** | 跨实体的业务逻辑 | 订单金额计算、库存校验 |
| **领域事件（Domain Event）** | 领域内发生的有意义的事 | OrderCreatedEvent |
| **仓储（Repository）** | 聚合的持久化抽象 | OrderRepository |
| **工厂（Factory）** | 复杂对象的创建逻辑 | OrderFactory |

**实体 vs 值对象**：

| 特性 | 实体（Entity） | 值对象（Value Object） |
|------|----------------|------------------------|
| **唯一标识** | 有（id） | 无 |
| **相等判断** | 按 id | 按属性值 |
| **可变性** | 可变 | 不可变（创建后不能改） |
| **生命周期** | 长 | 短，可随意创建替换 |
| **示例** | 订单、用户、商品 | 地址、金额、时间范围 |

**聚合设计原则**：
- 聚合内强一致性，聚合间最终一致。
- 聚合根负责维护聚合内规则。
- 一个事务只能修改一个聚合。

### 3️⃣ DDD 分层架构

```
┌─────────────────────────────────────────────────────────────┐
│                    DDD 分层架构                              │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   ┌─────────────────────────────────────────────────────┐  │
│   │  用户接口层（User Interface）                        │  │
│   │  - Controller、DTO、VO                               │  │
│   │  - 处理用户请求、参数校验、返回视图                   │  │
│   └──────────────────────┬──────────────────────────────┘  │
│                          │                                  │
│   ┌──────────────────────▼──────────────────────────────┐  │
│   │  应用层（Application）                               │  │
│   │  - ApplicationService                                │  │
│   │  - 编排领域对象、事务控制、事件发布                   │  │
│   │  - 无业务逻辑，只协调                                │  │
│   └──────────────────────┬──────────────────────────────┘  │
│                          │                                  │
│   ┌──────────────────────▼──────────────────────────────┐  │
│   │  领域层（Domain）【核心】                            │  │
│   │  - Entity、ValueObject、Aggregate、DomainService    │  │
│   │  - DomainEvent、Repository Interface                │  │
│   │  - 业务逻辑的核心，最纯净                           │  │
│   └──────────────────────┬──────────────────────────────┘  │
│                          │                                  │
│   ┌──────────────────────▼──────────────────────────────┐  │
│   │  基础设施层（Infrastructure）                        │  │
│   │  - RepositoryImpl、DAO、Mapper                       │  │
│   │  - 外部服务调用、消息发送、缓存                      │  │
│   │  - 为领域层提供技术能力                              │  │
│   └─────────────────────────────────────────────────────┘  │
│                                                             │
│   依赖方向：上层依赖下层，领域层不依赖其他层                 │
│   （通过依赖倒置，领域层定义接口，基础设施层实现）           │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

| 层级 | 职责 | 禁止 |
|------|------|------|
| **用户接口层** | 接收请求、参数校验、返回响应 | 不写业务逻辑 |
| **应用层** | 编排领域对象、事务边界、发布事件 | 不实现业务规则 |
| **领域层** | 核心业务逻辑、领域模型 | 不依赖框架、不直接操作数据库 |
| **基础设施层** | 数据持久化、外部调用、技术实现 | 不含有业务逻辑 |

### 4️⃣ 代码示例

#### （1）领域层（Domain）

```java
// 值对象：金额（不可变）
@Value
public class Money {
    private final BigDecimal amount;
    private final String currency;
    
    public Money add(Money other) {
        // 校验币种一致
        return new Money(this.amount.add(other.amount), this.currency);
    }
}

// 实体：订单项
@Entity
public class OrderItem {
    @Id
    private Long id;
    private Long productId;
    private int quantity;
    private Money unitPrice;
    
    public Money calculateSubtotal() {
        return unitPrice.multiply(quantity);
    }
}

// 聚合根：订单
@AggregateRoot
public class Order {
    @Id
    private OrderId orderId;
    private Long userId;
    private List<OrderItem> items;
    private Money totalAmount;
    private OrderStatus status;
    private List<DomainEvent> domainEvents;
    
    // 工厂方法创建订单
    public static Order create(Long userId, List<OrderItem> items) {
        Order order = new Order();
        order.orderId = OrderId.generate();
        order.userId = userId;
        order.items = items;
        order.totalAmount = calculateTotal(items);
        order.status = OrderStatus.CREATED;
        order.domainEvents = new ArrayList<>();
        order.domainEvents.add(new OrderCreatedEvent(order.orderId));
        return order;
    }
    
    // 领域方法：支付
    public void pay(Money amount) {
        if (status != OrderStatus.CREATED) {
            throw new IllegalStateException("订单状态错误");
        }
        if (!amount.equals(totalAmount)) {
            throw new IllegalArgumentException("支付金额不匹配");
        }
        this.status = OrderStatus.PAID;
        this.domainEvents.add(new OrderPaidEvent(orderId));
    }
    
    // 领域方法：发货
    public void ship() {
        if (status != OrderStatus.PAID) {
            throw new IllegalStateException("订单未支付");
        }
        this.status = OrderStatus.SHIPPED;
        this.domainEvents.add(new OrderShippedEvent(orderId));
    }
    
    public List<DomainEvent> getDomainEvents() {
        return Collections.unmodifiableList(domainEvents);
    }
    
    public void clearDomainEvents() {
        domainEvents.clear();
    }
}
```

#### （2）应用层（Application）

```java
@Service
@Transactional
public class OrderApplicationService {
    
    @Autowired
    private OrderRepository orderRepository;
    
    @Autowired
    private DomainEventPublisher eventPublisher;
    
    public OrderDTO createOrder(CreateOrderCommand command) {
        // 1. 创建聚合
        List<OrderItem> items = command.getItems().stream()
            .map(item -> new OrderItem(item.getProductId(), item.getQuantity(), item.getPrice()))
            .collect(Collectors.toList());
        
        Order order = Order.create(command.getUserId(), items);
        
        // 2. 保存聚合
        orderRepository.save(order);
        
        // 3. 发布领域事件
        order.getDomainEvents().forEach(eventPublisher::publish);
        order.clearDomainEvents();
        
        return OrderDTO.from(order);
    }
    
    public void payOrder(Long orderId, Money amount) {
        // 1. 加载聚合
        Order order = orderRepository.findById(new OrderId(orderId))
            .orElseThrow(() -> new OrderNotFoundException(orderId));
        
        // 2. 执行业务操作
        order.pay(amount);
        
        // 3. 保存聚合
        orderRepository.save(order);
        
        // 4. 发布事件
        order.getDomainEvents().forEach(eventPublisher::publish);
        order.clearDomainEvents();
    }
}
```

#### （3）基础设施层（Infrastructure）

```java
@Repository
public class OrderRepositoryImpl implements OrderRepository {
    
    @Autowired
    private OrderMapper orderMapper;
    
    @Autowired
    private OrderItemMapper orderItemMapper;
    
    @Override
    public void save(Order order) {
        // 保存订单
        OrderDO orderDO = OrderConverter.toDO(order);
        orderMapper.insertOrUpdate(orderDO);
        
        // 保存订单项
        order.getItems().forEach(item -> {
            OrderItemDO itemDO = OrderItemConverter.toDO(item);
            orderItemMapper.insertOrUpdate(itemDO);
        });
    }
    
    @Override
    public Optional<Order> findById(OrderId orderId) {
        OrderDO orderDO = orderMapper.selectById(orderId.getValue());
        if (orderDO == null) {
            return Optional.empty();
        }
        
        List<OrderItemDO> itemDOs = orderItemMapper.selectByOrderId(orderId.getValue());
        Order order = OrderConverter.toDomain(orderDO, itemDOs);
        return Optional.of(order);
    }
}
```

### 5️⃣ DDD 与微服务

| DDD 概念 | 微服务对应 |
|----------|-----------|
| **限界上下文** | 微服务边界 |
| **聚合** | 服务内的核心业务单元 |
| **领域事件** | 服务间异步通信（MQ） |
| **防腐层（ACL）** | 服务间调用适配 |

**微服务拆分原则**：
- 按**限界上下文**拆分，一个上下文一个服务。
- 上下文间通过**领域事件**异步解耦。
- 跨上下文调用通过**防腐层**隔离外部变化。

### 6️⃣ DDD 适用场景

| 适用 | 不适用 |
|------|--------|
| 业务复杂、领域概念多 | 简单 CRUD、无复杂业务 |
| 需要长期演进、业务变化快 | 短期项目、一次性交付 |
| 团队有一定技术能力 | 团队对 OOP 理解不深 |
| 中台建设、大型系统 | 小型应用、原型系统 |

### ⚠️ 常见坑与注意点

1. **现象**：所有系统都用 DDD，简单 CRUD 也搞四层架构。**正确做法**：DDD 适合复杂业务；简单系统用贫血模型+事务脚本更高效。
2. **现象**：聚合设计过大，一个聚合包含几十个实体。**正确做法**：聚合尽量小，一个事务只改一个聚合；大聚合拆分成多个小聚合，用领域事件协作。
3. **现象**：领域层依赖 Spring、数据库。**正确做法**：领域层最纯净，只依赖语言标准库；通过依赖倒置让基础设施层实现领域层接口。
4. **现象**：忽视统一语言，开发、产品各说各话。**正确做法**：从需求分析开始就建立统一语言，代码中的类名、方法名要体现业务术语。
