# 📌 微服务架构常用概念与使用场景速查

> 配合《学习笔记》使用，侧重**注册发现、网关、配置、熔断限流**，便于速查与校招面试。

---

## 一、服务注册与发现

| 组件 | 特点 | 常用场景 |
|------|------|----------|
| **Nacos** | 注册+配置、AP/CP 可切换、国内生态好 | Spring Cloud Alibaba、多环境配置 |
| **Eureka** | 注册中心、AP、Netflix 系 | Spring Cloud Netflix（已较少新用） |
| **Consul** | 服务发现+健康检查+KV | 多语言、云原生 |
| **Zookeeper** | CP、强一致 |  Dubbo、Kafka 等依赖 ZK 的场景 |

- **流程**：服务启动 → 注册到中心；调用方 → 拉取/订阅实例列表 → 负载均衡选实例 → 发起调用。

## 二、API 网关

| 能力 | 说明 |
|------|------|
| **路由** | 按 path/host 转发到后端服务 |
| **鉴权** | 统一校验 Token、放行/拒绝 |
| **限流** | 全局限流、按 path/用户限流 |
| **熔断/降级** | 下游异常时快速失败或返回兜底 |
| **日志/监控** | 统一记录请求、耗时、状态码 |

| 组件 | 说明 |
|------|------|
| **Spring Cloud Gateway** | 基于 WebFlux、断言+过滤器、与 Spring 生态集成 |
| **Kong** | 基于 Nginx/OpenResty、插件丰富 |
| **Nginx** | 可做反向代理与简单路由，偏流量层 |

---

## 三、配置中心

| 能力 | 说明 |
|------|------|
| **集中管理** | 配置存中心，应用拉取或订阅 |
| **动态刷新** | 改配置不重启（如 @RefreshScope） |
| **多环境** | dev/test/prod 命名空间或分组 |

- **Nacos Config**：bootstrap 中配置 spring.cloud.nacos.config；支持 dataId、group、namespace。
- **注意**：敏感配置加密；大促前可锁定配置避免误改。

---

## 四、服务调用

| 方式 | 典型 | 特点 |
|------|------|------|
| **REST + OpenFeign** | Spring Cloud OpenFeign | 声明式、集成负载均衡与熔断 |
| **gRPC** | 跨语言、高性能 | 二进制、流式，适合内部高性能调用 |
| **消息队列** | RabbitMQ、Kafka | 异步、解耦、削峰、最终一致 |

- **负载均衡**：Spring Cloud LoadBalancer（轮询等）；Ribbon 已进入维护，新项目用 LoadBalancer。

## 五、熔断、限流、降级

| 概念 | 目的 | 典型实现 |
|------|------|----------|
| **熔断** | 下游故障时快速失败，避免雪崩 | Sentinel 熔断规则、Resilience4j CircuitBreaker |
| **限流** | 控制 QPS/并发，保护系统 | Sentinel 流控、网关限流、Guava RateLimiter |
| **降级** | 非核心关掉或返回默认，保核心 | 熔断后返回 fallback、开关关闭非核心功能 |

- **Sentinel**：流控规则（QPS/并发数）、熔断规则（慢调用比例/异常比例/异常数）、热点参数限流、系统自适应。

## 六、分布式事务与链路追踪（概念速查）

| 方案 | 思路 | 适用 |
|------|------|------|
| **Seata AT** | 自动回滚、一阶段提交二阶段异步/同步回滚 | 一般业务、对性能要求高 |
| **TCC** |  Try-Confirm-Cancel，业务自己实现补偿 | 强一致、可接受改业务代码 |
| **本地消息表** | 本地事务+发消息、定时扫表补偿 | 最终一致、实现简单 |
| **MQ 事务消息** | 半消息+本地事务+确认/回滚 | 与 MQ 强绑定的场景 |

### 链路追踪

| 概念 | 说明 |
|------|------|
| **Trace** | 一次完整请求的调用链，由多个 Span 组成，用 traceId 标识 |
| **Span** | 调用链中的基本单元（如一次 RPC、DB 查询），有 spanId 和时间戳 |
| **工具** | SkyWalking（国产、APM 全功能）、Zipkin（轻量、Sleuth 集成）、Jaeger（云原生） |

| 工具 | 特点 | 适用 |
|------|------|------|
| **SkyWalking** | 无侵入 Agent、APM 全功能、UI 完善 | 国内主流、生产推荐 |
| **Zipkin** | 轻量、与 Spring Cloud Sleuth 集成好 | Spring Cloud 生态 |
| **Jaeger** | CNCF 项目、云原生友好 | Kubernetes 环境 |

## 七、Service Mesh 速查

### 核心概念

| 概念 | 说明 |
|------|------|
| **Service Mesh** | 基础设施层，Sidecar 代理实现服务治理，业务零侵入 |
| **Sidecar** | 与业务容器一起运行的代理容器（如 Envoy） |
| **数据平面** | 处理实际流量（Envoy 代理） |
| **控制平面** | 管理配置下发（Istiod） |

### Istio 组件

| 组件 | 职责 |
|------|------|
| **Envoy** | Sidecar 代理，拦截流量、负载均衡、mTLS |
| **Pilot** | 服务发现、流量管理配置 |
| **Citadel** | 证书管理、安全认证 |

### 核心功能

| 功能 | 说明 |
|------|------|
| **流量管理** | 路由、灰度发布、超时重试、熔断 |
| **安全** | mTLS 自动加密、身份认证、访问控制 |
| **可观测** | 指标、日志、链路追踪、拓扑图 |

### Service Mesh vs Spring Cloud

| 维度 | Spring Cloud | Service Mesh |
|------|--------------|--------------|
| 侵入性 | SDK 侵入 | **零侵入** |
| 语言 | Java 为主 | **多语言** |
| 性能 | 无代理开销 | 有 Sidecar 开销 |
| 适用 | 传统微服务 | **云原生/K8s** |

## 八、面试一句话总结

- **注册发现**：服务自注册、调用方从中心拉实例，实现动态扩缩与故障剔除。
- **网关**：统一入口，做路由、鉴权、限流、熔断、日志。
- **配置中心**：配置集中、动态刷新、多环境，避免改配置重启。
- **熔断**：下游挂了快速失败；**限流**：控 QPS/并发；**降级**：非核心关掉保核心。
- **微服务挑战**：分布式事务、链路追踪、运维与部署复杂度。
- **Service Mesh**：Sidecar 代理实现治理，零侵入、多语言、适合云原生。
- **DDD**：领域驱动设计，战略设计（子域、限界上下文）+ 战术设计（实体、值对象、聚合），指导微服务拆分。

## 九、DDD 速查

### 核心概念

| 概念 | 说明 |
|------|------|
| **领域** | 业务问题的范围 |
| **子域** | 核心域、支撑域、通用域 |
| **限界上下文** | 业务边界，通常对应一个微服务 |
| **实体** | 有唯一标识、可变 |
| **值对象** | 无标识、不可变 |
| **聚合** | 一组相关对象，有聚合根 |
| **领域事件** | 领域内发生的有意义的事 |

### 分层架构

| 层级 | 职责 |
|------|------|
| **用户接口层** | 接收请求、参数校验 |
| **应用层** | 编排领域对象、事务控制 |
| **领域层** | 核心业务逻辑（最纯净） |
| **基础设施层** | 数据持久化、外部调用 |

### DDD vs 贫血模型

| 特性 | DDD（充血模型） | 传统（贫血模型） |
|------|-----------------|------------------|
| 业务逻辑 | 在领域对象中 | 在 Service 中 |
| 可维护性 | 高 | 低 |
| 适用 | 复杂业务 | 简单 CRUD |
