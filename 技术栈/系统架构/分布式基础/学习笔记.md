# 📘《分布式基础学习与实战手册》

> 系统学习见下文各章；日常常用概念与使用场景速查见同目录《常用概念与使用场景》。  
> 适用：分布式系统基础概念，面向校招与后端开发（CAP/BASE、共识、分布式 ID/锁/事务、缓存与分库分表）。

------

## 第1章：分布式概述

本章建立对**分布式系统**的整体认知：是什么、带来哪些核心问题（CAP、共识、事务/ID/锁），为后续各章（一致性协议、分布式 ID、锁、事务、缓存、分库分表）打基础。校招常从 CAP、BASE 问起。

------

### 1️⃣ 什么是分布式系统

- **分布式系统**：由**多台计算机**协同完成同一目标，通过**网络**通信。**为什么需要**：单机容量与可用性有限，分布式可扩展、容错、负载分散。
- **特点**：**无全局时钟**、**节点故障独立**、**网络延迟与分区**（部分节点间不可达），因此带来一致性、可用性、共识等问题。
- **典型问题**：**一致性**（多副本何时一致）、**可用性**（部分故障时是否仍可服务）、**分区容错**（CAP）；**共识**（多节点对同一值达成一致）；**分布式事务**、**分布式 ID**、**分布式锁**等。

### 2️⃣ CAP 与 BASE

- **CAP**：在分布式系统中，**C**onsistency（强一致性，读总能读到最新写）、**A**vailability（可用性，请求总能得到非错误响应）、**P**artition tolerance（分区容错，网络分区时系统仍能工作）三者**最多同时满足两个**。实际中**P 必须接受**（网络分区客观存在），故在 **C 与 A 间权衡**：CP（如 Etcd、Zookeeper 选主时）与 AP（如多数 NoSQL、Eureka）。
- **BASE**：**B**asically **A**vailable（基本可用）、**S**oft state（软状态）、**E**ventually consistent（**最终一致**）；用**最终一致性**替代强一致，**提高可用性**，是互联网系统常用取舍。

| 取舍 | 含义 | 典型 |
|------|------|------|
| CP | 分区时保一致、牺牲可用 | Zookeeper、Etcd 选主、强一致存储 |
| AP | 分区时保可用、牺牲强一致 | Eureka、Cassandra、多数缓存 |

------

### 📖 本章小结

| 知识点 | 面试关键词 | 实际应用 |
|--------|------------|----------|
| 分布式定义与特点 | 多机协同、网络、无全局时钟、分区 | 理解为什么有 CAP、共识等问题 |
| CAP | C/A/P 最多满足两个、实际 P 必选、C 与 A 权衡 | 选型：CP 还是 AP、举例 |
| BASE | 基本可用、软状态、最终一致 | 互联网系统常用、与强一致对比 |

------

### ⚠️ 常见坑与注意点

1. **现象**：说“我们系统同时满足 CAP”。**原因**：CAP 定理是“最多两选一”。**正确做法**：理解是“分区发生时 C 与 A 二选一”；平时无分区时可以既可用又一致。
2. **现象**：把 BASE 和 CAP 的 A 等同。**原因**：BASE 是设计思想（接受最终一致换可用），不是定理。**正确做法**：CAP 是定理；BASE 是“用最终一致换高可用”的实践。
3. **现象**：认为 CP 系统完全不可用。**原因**：CP 指**分区时**为保一致可能拒绝写或读。**正确做法**：无分区时 CP 系统也可用；分区时可能少数节点不可用或只读。

------

### 🎯 面试常见追问

| 面试官提问 | 回答思路 |
|------------|----------|
| 什么是 CAP？为什么只能满足两个？ | C 一致、A 可用、P 分区容错；分区时若仍要一致则需等待或拒绝，无法同时保证所有节点可用，故最多两选一。实际 P 必选，在 C 与 A 间权衡。 |
| CP 和 AP 各举一例？ | CP：Zookeeper、Etcd（选主、配置）。AP：Eureka（注册中心）、Cassandra。若追问 BASE 见本章 2️⃣。 |

## 第2章：一致性协议与共识

本章解决**多副本如何一致**（强一致 vs 最终一致）以及**多节点如何对同一值达成一致**（共识算法），与第 1 章 CAP、第 5 章分布式事务、第 6 章缓存一致直接相关。校招常考：强一致 vs 最终一致、Raft 选主与日志复制、为什么需要共识。

------

### 1️⃣ 强一致与最终一致

- **强一致**：读**总能读到最新写**；代价高（需同步复制、阻塞或协调），多用于单机或小范围（如单主多从同步复制）。
- **最终一致**：允许**一段时间不一致**，**最终**达到一致；常用**异步复制**、**消息队列**、**版本号/向量时钟**。互联网系统大多采用最终一致 + 业务幂等与补偿。

| 类型 | 含义 | 典型实现 |
|------|------|----------|
| 强一致 | 读必见最新写 | 同步复制、2PC、Paxos/Raft 读主 |
| 最终一致 | 一段时间后一致 | 异步复制、MQ、版本号、CRDT |

### 2️⃣ 共识算法

- **共识**：多节点对**同一值**（如谁当主、某条日志是否提交）达成一致。**为什么需要**：选主、配置一致、分布式锁与协调，避免脑裂。
- **典型算法**：**Paxos**（理论经典，难工程化）、**Raft**（易理解，**Etcd、Consul** 用）、**ZAB**（**Zookeeper**）。用于**注册中心**、**配置中心**、分布式协调。

#### （1）Paxos 算法详解

> **一句话**：Paxos 是分布式共识的**理论基础**，通过**两阶段提交**让多数节点对同一值达成一致。**难点**：理解复杂、工程实现难；**实际**：Chubby、Google Spanner 等使用。

**三种角色**：

| 角色 | 职责 |
|------|------|
| **Proposer（提议者）** | 提出提案（proposal），包含提案编号 N 和值 V |
| **Acceptor（接受者）** | 对提案进行投票，多数接受则提案通过 |
| **Learner（学习者）** | 学习已达成一致的值 |

**两阶段流程**：

```
阶段一：Prepare（准备）
1. Proposer 选择提案编号 N，向多数 Acceptor 发送 Prepare(N) 请求
2. Acceptor 收到 Prepare(N)：
   - 若 N > 已响应的最大编号，承诺不再接受编号 < N 的提案
   - 返回已接受的最大编号提案（若有）

阶段二：Accept（接受）
1. Proposer 收到多数 Acceptor 响应后：
   - 若响应中有已接受的值，选择最大编号对应的值
   - 若无，使用自己的值 V
   - 发送 Accept(N, V) 给多数 Acceptor
2. Acceptor 收到 Accept(N, V)：
   - 若未承诺过更大的编号，接受该提案
   - 返回 Accepted
3. 多数 Acceptor 接受后，提案通过
```

**关键约束**：

- **提案编号全局唯一且递增**：保证提案有序。
- **多数派原则**：只有被多数 Acceptor 接受的提案才生效。
- **一旦提案被多数接受，后续提案不能覆盖**：保证一致性。

**面试怎么说**：Paxos 分两阶段——Prepare 阶段获取承诺和已接受值，Accept 阶段提交值；多数派接受即达成共识。实际工程中 Raft 更常用，因为更易理解和实现。

#### （2）Raft 算法详解（面试常问）

> **一句话**：Raft 是**工程友好**的共识算法，将共识问题分解为**选主**和**日志复制**两个子问题，比 Paxos 更易理解和实现。

**三种角色**：

| 角色 | 职责 |
|------|------|
| **Leader** | 处理所有写请求、日志复制、心跳维护 |
| **Follower** | 接收 Leader 日志和心跳、投票 |
| **Candidate** | 选主过程中的临时角色，发起投票 |

**选主流程**：

```
1. Follower 超时未收到 Leader 心跳 → 变为 Candidate
2. Candidate 自增任期号，向其他节点发起投票请求（RequestVote）
3. 其他节点在一个任期内只能投一票，先到先得
4. Candidate 收到多数票 → 成为 Leader
5. Leader 立即发送心跳确认地位
```

**日志复制流程**：

```
1. 客户端写请求 → Leader
2. Leader 将操作追加到本地日志（未提交状态）
3. Leader 并行发送 AppendEntries 给所有 Follower
4. 多数 Follower 复制成功 → Leader 提交日志并应用状态机
5. Leader 通知 Follower 提交（心跳或追加日志时携带 commitIndex）
```

**安全性保证**：

- **日志匹配**：相同 index 和 term 的日志内容相同。
- **Leader 完整性**：已提交的日志不会丢失。
- **状态机安全**：所有节点按相同顺序应用日志。

**面试怎么说**：Raft 通过选主（超时变 Candidate 拉票，多数同意成 Leader）和日志复制（Leader 写本地并复制到多数 Follower，多数持久化后提交）保证一致性；只有 Leader 处理写，读可走 Leader 或 Follower。

#### （3）ZAB 协议简述

> **一句话**：ZAB（Zookeeper Atomic Broadcast）是 Zookeeper 使用的共识协议，类似 Raft，强调**崩溃恢复**和**消息广播**。

| 模式 | 说明 |
|------|------|
| **消息广播** | Leader 接收写请求，广播给 Follower，多数确认后提交 |
| **崩溃恢复** | Leader 崩溃后，通过选举选出新 Leader，同步日志 |

**与 Raft 区别**：
- ZAB 用 **epoch**（纪元）区分 Leader 任期，Raft 用 **term**。
- ZAB 强调**原子广播**，Raft 强调**日志复制**。
- 实现细节不同，核心思想相似。

### 3️⃣ Raft 简述（面试常问）

- **选主（Leader Election）**：节点有三种角色：Leader、Follower、Candidate。超时未收到 Leader 心跳则变 Candidate 发起投票，得多数票成为 Leader。
- **日志复制**：Leader 接收写请求、写本地日志并复制到多数 Follower；多数持久化成功后 Leader 提交并应用，再通知 Follower 提交。
- **面试怎么说**：Raft 通过选主 + 日志多数派复制保证一致；只有 Leader 处理写，读可走 Leader 或 Follower（读 Follower 可能是最终一致）。

------

### 📖 本章小结

| 知识点 | 面试关键词 | 实际应用 |
|--------|------------|----------|
| 强一致 vs 最终一致 | 读必见最新 vs 最终一致、异步复制、版本号 | 选型、缓存与 DB 一致（第 6 章） |
| 共识算法 | Paxos/Raft/ZAB、选主、日志复制 | Etcd、Zookeeper、注册中心 |
| Raft | 选主、日志多数派、Leader 写 | 能口述选主与复制流程 |

------

### ⚠️ 常见坑与注意点

1. **现象**：认为 Raft 读 Follower 一定是强一致。**原因**：Follower 可能落后。**正确做法**：强一致读需读 Leader 或线性一致读（Raft 的 ReadIndex/LeaseRead）；读 Follower 是最终一致或会话一致。
2. **现象**：把“共识”和“强一致”混用。**正确做法**：共识解决“多节点对同一值达成一致”（如谁当主）；强一致是“读见最新写”，共识可用来实现强一致（如 Raft 写主）。
3. **现象**：说 Paxos 和 Raft 是一回事。**正确做法**：Paxos 是协议族，Raft 是易理解的等价实现；面试说“Raft 是工程友好的共识算法，Etcd/ZK 用 Raft 或类 Raft”即可。

------

|------------|----------|

## 第3章：分布式 ID

本章解决**多节点/分库分表下如何生成全局唯一 ID**，与第 7 章分库分表直接相关。校招常考：为什么需要、雪花算法结构、号段思路、选型。

------

### 1️⃣ 为什么需要分布式 ID

- **场景**：分库分表、多实例写库时，**主键不能重复**；单机**自增**无法保证全局唯一。**要求**：全局唯一、趋势递增（利于索引与页分裂）、高可用、尽量短。

### 2️⃣ 常见方案对比

| 方案 | 优点 | 缺点 | 适用 |
|------|------|------|------|
| **UUID** | 简单、无依赖、全局唯一 | 无序、占空间（36 字符）、不利于索引 | 简单场景、不要求有序 |
| **雪花算法 Snowflake** | 趋势递增、64 位、高性能 | 依赖时钟、机器位需分配 | 高并发、要有序、可接受时钟回拨处理 |
| **号段/步长** | 有序、可读、DB 压力小（一次取一段） | 依赖 DB 或 Redis | 中高并发、要可读性（如 Leaf） |
| **Redis 自增** | 简单、快 | 需持久化与高可用、非严格递增（多实例） | 可接受非严格递增时 |
| **数据库号段表** | 有序、可控、可多种号段 | 依赖 DB、需防并发取段 | 如美团 Leaf 号段模式 |

### 3️⃣ 雪花算法结构（64 位）

- **1** 位符号（0）+ **41** 位时间戳（毫秒）+ **10** 位机器 ID（5 位机房+5 位机器）+ **12** 位序列（同毫秒内序号）。**趋势递增**、**高性能**；需注意**时钟回拨**（回拨时可等待或抛异常，或预留位做扩展）。
- **面试怎么说**：1+41+10+12，时间戳保证趋势递增，机器位保证多机不撞，序列位保证同毫秒不撞；时钟回拨要处理。

### 4️⃣ 号段如何减少 DB 压力

- 每次从 DB（或 Redis）**取一段**（如 1～1000），应用内存内自增分配；用完后再取下一段。**压力**：从“每次生成都访问 DB”变为“每段只访问一次 DB”。

------

### 📖 本章小结

| 知识点 | 面试关键词 | 实际应用 |
|--------|------------|----------|
| 为什么需要 | 分库分表、全局唯一、单机自增不够 | 选型前提 |
| 雪花 | 1+41+10+12、趋势递增、时钟回拨 | 高并发主键、订单号 |
| 号段 | 一次取一段、减少 DB 访问 | Leaf、可读 ID |

------

### ⚠️ 常见坑与注意点

1. **现象**：多机雪花机器位配成相同。**原因**：机器位重复会导致同毫秒内可能重复。**正确做法**：机器 ID 必须全局唯一（机房+机器编号或配置中心分配）。
2. **现象**：雪花时钟回拨不处理。**原因**：回拨后时间戳变小，可能生成重复或乱序 ID。**正确做法**：检测到回拨则等待或抛异常、或关闭机器位分配新段，视业务接受度选择。
3. **现象**：号段用尽瞬间大量请求同时取下一段。**正确做法**：双 buffer 预取、或单线程/分布式锁取段，避免同一段被多实例重复取。

------

### 🎯 面试常见追问

| 面试官提问 | 回答思路 |
|------------|----------|
| 雪花算法结构？ | 1 符号+41 时间+10 机器+12 序列；趋势递增、多机不撞、同毫秒靠序列。需处理时钟回拨。若追问号段见本章 4️⃣。 |
| 号段为什么能减少 DB 压力？ | 一次取一段在内存里分配，用完后才再访问 DB；从“每 ID 一查”变成“每段一查”。 |

## 第4章：分布式锁

本章解决**多实例下同一资源只能被一个实例处理**的问题，避免重复扣减、重复发奖等。与第 2 章共识（ZK）、第 6 章缓存（Redis）关联。校招常考：Redis vs ZK 实现、Redisson 看门狗、防误删。

------

### 1️⃣ 为什么需要分布式锁

- **场景**：多实例部署时，**同一资源**（如库存、优惠券）只能被**一个实例**处理；单机 JVM 的锁无法跨进程。**要求**：互斥、防死锁（过期或释放）、尽量高可用、可重入（视业务）。

### 2️⃣ 实现方式对比

| 方式 | 思路 | 优点 | 缺点 |
|------|------|------|------|
| **Redis** | SET key value NX EX 过期、Lua 保证“判断+删”原子、**Redisson 看门狗**续期 | 性能好、实现简单 | 非 CP、主从切换可能丢锁 |
| **Zookeeper** | **临时顺序节点**，**最小节点**获锁；监听前一个节点 | 强一致、无过期、自动释放（会话断开） | 性能与可用性不如 Redis、依赖 ZK |
| **数据库** | 唯一索引插入或乐观锁 | 实现简单、利用现有 DB | 性能差、锁表压力大 |

### 3️⃣ Redis 实现要点

- **加锁**：`SET key unique_value NX EX 30`（NX 不存在才设、EX 过期）；**value 用唯一标识**（如 UUID+线程 ID），便于**只删自己的锁**防误删。
- **解锁**：Lua 脚本里**先判断 value 是否自己的锁再 DEL**，保证“判断+删”原子，防止删了别人的锁。
- **续期**：业务执行超过过期时间时，**Redisson 看门狗**在后台定时（如 1/3 过期时间）续期，避免业务未执行完锁被释放。

### 4️⃣ 防误删与可重入

- **误删**：A 的锁过期被释放，B 加锁成功，A 执行完又去删锁会把 B 的锁删掉。**正确做法**：value 存唯一标识，删前比对；删用 Lua 保证原子。
- **可重入**：同一线程多次加锁；Redis 可用 Hash 存 threadId→count，加锁 +1 解锁 -1，0 时删 key；Redisson 已支持可重入。

------

### 📖 本章小结

| 知识点 | 面试关键词 | 实际应用 |
|--------|------------|----------|
| 为什么需要 | 多实例互斥、单机锁不够 | 扣库存、发奖、定时任务 |
| Redis 锁 | NX EX、value 唯一、Lua 删锁、看门狗 | 高并发、可接受非 CP 时 |
| ZK 锁 | 临时顺序节点、最小获锁、会话断则释锁 | 强一致、不能丢锁场景 |
| 防误删 | value 标识、Lua 原子判断再删 | 面试必答 |

------

### ⚠️ 常见坑与注意点

1. **现象**：只 SET NX 不设过期时间。**原因**：实例宕机或异常未释放则死锁。**正确做法**：必须设 **EX**（过期时间），或用 Redisson 看门狗续期。
2. **现象**：删锁时不用 value 校验。**原因**：可能删掉别人刚加的锁（自己锁过期后别人加锁成功）。**正确做法**：value 存唯一标识，Lua 里 if value==当前 then DEL。
3. **现象**：主从切换导致锁丢失。**原因**：Redis 主从异步复制，主宕机时从可能还未复制锁。**正确做法**：强一致锁用 Redlock（多主）或 ZK；或业务接受极小概率重复。

------

### 🎯 面试常见追问

| 面试官提问 | 回答思路 |
|------------|----------|
| Redis 和 ZK 做分布式锁优缺点？ | Redis：性能好、实现简单；主从切换可能丢锁。ZK：强一致、临时节点断会话即释锁；性能与可用性不如 Redis。若追问 Redisson 见本章 3️⃣。 |
| Redisson 看门狗做什么？ | 加锁后后台定时续期（如 1/3 过期时间续一次），避免业务执行超过过期时间导致锁被释放；业务结束会主动释锁并取消续期。 |
| 如何防止误删别人的锁？ | 加锁时 value 存唯一标识（UUID+线程）；删锁时用 Lua 先判断 value 是否当前线程再 DEL，保证原子。 |

## 第5章：分布式事务

本章解决**跨服务、跨库的多个操作如何保证要么全成功要么全失败（或最终一致）**，与第 2 章一致性、第 6 章缓存、微服务章节 Seata 关联。校招常考：强一致 vs 最终一致、本地消息表、MQ 事务消息、幂等。

------

### 1️⃣ 场景与难点

- **场景**：**跨服务、跨库**的多个操作（如下单+扣库存+加积分）需要一致性；单机事务无法跨进程与跨库。**难点**：网络与节点故障、部分成功部分失败、回滚与补偿复杂。

### 2️⃣ 强一致方案（少用）

- **2PC/3PC**：**协调者** + **参与者**；两阶段提交（准备+提交）存在阻塞与单点，**性能与可用性差**，实际少用。
- **Seata AT/TCC**：AT 自动回滚、TCC 需业务写 Try/Confirm/Cancel；适合微服务内跨库/跨服务，见微服务章节。

#### （1）Seata 核心概念

| 概念 | 说明 |
|------|------|
| **TC（Transaction Coordinator）** | 事务协调者，维护全局事务状态，驱动全局提交或回滚 |
| **TM（Transaction Manager）** | 事务管理器，定义全局事务范围，开启/提交/回滚全局事务 |
| **RM（Resource Manager）** | 资源管理器，管理分支事务资源，与 TC 通信注册分支事务并报告状态 |

```
┌─────────────────────────────────────────────────────────────┐
│                    Seata 事务流程                            │
├─────────────────────────────────────────────────────────────┤
│  TM（业务发起方）          TC（Seata Server）       RM（参与者）│
│       │                         │                        │   │
│       │  1. 开启全局事务         │                        │   │
│       │ ─────────────────────→  │                        │   │
│       │                         │                        │   │
│       │  2. 注册分支事务         │  ←─────────────────────│   │
│       │                         │                        │   │
│       │  3. 执行业务 SQL         │                        │   │
│       │                         │                        │   │
│       │  4. 全局提交/回滚        │                        │   │
│       │ ─────────────────────→  │ ───────────────────→  │   │
└─────────────────────────────────────────────────────────────┘
```

#### （2）Seata AT 模式（推荐入门）

> **一句话**：AT 模式对业务**无侵入**，自动记录前后镜像，失败自动回滚。**适用**：大多数业务场景，对性能要求高。

**原理**：
- 一阶段：执行业务 SQL，记录**前后镜像**到 undo_log，释放本地锁。
- 二阶段提交：异步删除 undo_log。
- 二阶段回滚：根据 undo_log 反向生成 SQL 还原数据。

**配置示例（Spring Cloud）**：

```yaml
seata:
  enabled: true
  application-id: order-service
  tx-service-group: my_tx_group
  service:
    vgroup-mapping:
      my_tx_group: default
  registry:
    type: nacos
    nacos:
      server-addr: localhost:8848
      namespace: ""
      group: SEATA_GROUP
  config:
    type: nacos
    nacos:
      server-addr: localhost:8848
      namespace: ""
      group: SEATA_GROUP
```

**使用方式**：

```java
@GlobalTransactional(name = "create-order", rollbackFor = Exception.class)
public void createOrder(OrderDTO orderDTO) {
    orderMapper.insert(order);
    inventoryService.deductStock(orderDTO.getProductId(), orderDTO.getQuantity());
    accountService.deductBalance(orderDTO.getUserId(), orderDTO.getAmount());
}
```

**AT 模式注意点**：
- 需要**每个数据库**创建 `undo_log` 表。
- 写隔离：全局锁保证不同全局事务不会同时修改同一行。
- 读隔离：默认读未提交，可通过 `@GlobalLock` 实现读已提交。

#### （3）Seata TCC 模式

> **一句话**：TCC 模式需要业务实现 **Try-Confirm-Cancel** 三个接口，灵活但开发成本高。**适用**：强一致要求高、需要细粒度控制的场景。

**三个阶段**：

| 阶段 | 作用 | 示例（扣库存） |
|------|------|----------------|
| **Try** | 资源预留/冻结 | 冻结库存（stock - frozen_stock） |
| **Confirm** | 确认执行，真正扣减 | 扣减冻结库存 |
| **Cancel** | 取消执行，释放预留 | 释放冻结库存 |

**代码示例**：

```java
@LocalTCC
public interface InventoryTCCService {
    
    @TwoPhaseBusinessAction(name = "deductStock", commitMethod = "commit", rollbackMethod = "rollback")
    boolean deductStock(@BusinessActionContextParameter(paramName = "productId") Long productId,
                        @BusinessActionContextParameter(paramName = "quantity") Integer quantity);
    
    boolean commit(BusinessActionContext context);
    
    boolean rollback(BusinessActionContext context);
}
```

**TCC 注意点**：
- **空回滚**：Try 未执行但收到 Cancel，需判断是否已预留。
- **悬挂**：Cancel 先于 Try 执行，Try 需判断是否已回滚。
- **幂等**：Confirm/Cancel 可能重复调用，需幂等处理。

#### （4）Seata 模式对比

| 模式 | 侵入性 | 性能 | 一致性 | 适用场景 |
|------|--------|------|--------|----------|
| **AT** | 无侵入 | 高 | 最终一致 | 大多数业务、快速接入 |
| **TCC** | 高侵入 | 中 | 强一致 | 资金类、需细粒度控制 |
| **SAGA** | 中侵入 | 高 | 最终一致 | 长事务、跨服务编排 |

#### （5）Seata Server 部署

```yaml
server:
  port: 7091

seata:
  store:
    mode: db
    db:
      datasource: druid
      db-type: mysql
      driver-class-name: com.mysql.cj.jdbc.Driver
      url: jdbc:mysql://localhost:3306/seata?rewriteBatchedStatements=true
      user: root
      password: root
  registry:
    type: nacos
    nacos:
      server-addr: localhost:8848
      namespace: ""
      group: SEATA_GROUP
```

> 💬 **面试怎么说**：Seata 是分布式事务框架，AT 模式无侵入、自动回滚，适合大多数场景；TCC 模式需业务实现三接口，适合强一致场景。核心组件有 TC（协调者）、TM（管理器）、RM（资源管理器）。

### 3️⃣ 最终一致方案（常用）

| 方案 | 思路 | 优点 | 注意 |
|------|------|------|------|
| **本地消息表** | 本地事务同时写**业务表+消息表**，定时任务扫消息表发 MQ，消费端**幂等** | 不依赖 MQ 事务、实现清晰 | 消息表与业务同库、消费端必须幂等 |
| **MQ 事务消息** | **半消息**→执行业务→**确认/回滚**；MQ 根据结果投递或丢弃 | 与 MQ 深度结合、减少扫表 | 依赖 RocketMQ 等支持事务消息 |
| **Saga** | 长事务拆成多步，每步有**补偿**；顺序执行或编排，失败则逆序补偿 | 适合长流程、无大锁 | 补偿需幂等、可能看到中间态 |

### 4️⃣ 幂等的必要性

- **为什么必须幂等**：消息可能**重复投递**、**重试**、**网络抖动**导致重复请求；消费端不幂等会导致重复扣款、重复加积分等。**做法**：唯一业务键（如订单号+操作类型）去重、状态机（已处理则跳过）、 token 表或 Redis 去重。

> 💬 **面试怎么说**：分布式事务常用最终一致；本地消息表+MQ+消费幂等是一套成熟做法；强一致用 2PC/Seata 代价大，能最终一致就最终一致。

------

### 📖 本章小结

| 知识点 | 面试关键词 | 实际应用 |
|--------|------------|----------|
| 强一致 vs 最终一致 | 2PC/Seata vs 本地消息表/MQ 事务/Saga | 选型：能最终一致就最终一致 |
| 本地消息表 | 业务+消息同库、定时发 MQ、消费幂等 | 跨服务下单、扣库存 |
| MQ 事务消息 | 半消息、本地事务、确认/回滚 | RocketMQ 等 |
| 幂等 | 唯一键、状态机、去重表 | 消费端必做 |

------

### ⚠️ 常见坑与注意点

1. **现象**：只发 MQ 不写本地消息表，本地事务成功但发 MQ 失败。**正确做法**：本地事务里同时写消息表，发 MQ 由定时任务或异步补发，保证“本地成功则消息一定可被发出”。
2. **现象**：消费端不幂等。**正确做法**：用业务唯一键（订单号+操作）做幂等（DB 唯一索引或 Redis），重复请求直接返回成功。
3. **现象**：强一致场景滥用 2PC。**正确做法**：2PC 阻塞与协调者单点问题大；优先考虑最终一致+幂等+补偿，强一致仅核心资金类再考虑 Seata/2PC。

------

### 🎯 面试常见追问

| 面试官提问 | 回答思路 |
|------------|----------|
| 分布式事务有哪些方案？强一致和最终一致怎么选？ | 强一致：2PC、Seata。最终一致：本地消息表+MQ、MQ 事务消息、Saga。能接受延迟选最终一致+幂等；强一致代价大。若追问 Seata 见微服务章。 |
| 本地消息表怎么保证不丢消息？ | 消息与业务同一本地事务，业务成功则消息一定落库；定时任务或异步发 MQ，失败可重试；消费端幂等防重复。 |
| 为什么消费端必须幂等？ | 消息可能重复投递、重试、网络重复；不幂等会重复扣款/加积分。用唯一业务键或状态机做幂等。 |

## 第6章：分布式缓存与 Session

本章解决**缓存与 DB 一致、缓存三大问题（穿透/击穿/雪崩）、分布式 Session**，与第 2 章最终一致、第 4 章 Redis 锁关联。校招常考：Cache-Aside、穿透/击穿/雪崩及应对、Session 共享。

------

### 1️⃣ 缓存一致性

- **Cache-Aside**：读先查缓存，无则查 DB 并回写缓存；**写先更新 DB 再删缓存**（而非先删再写 DB，避免并发下旧数据写回缓存）。**延迟双删**：更新 DB 后删缓存、延迟再删一次，降低“读请求把旧数据写入缓存”的概率。**兜底**：给缓存设**过期时间**，强一致难，多以**最终一致+过期**为主。

### 2️⃣ 缓存穿透、击穿、雪崩

| 问题 | 含义 | 应对 |
|------|------|------|
| **穿透** | 查**不存在**的数据，缓存没有、请求直打 DB | **布隆过滤器**挡不存在的 key、**空值缓存**（短过期） |
| **击穿** | **热点 key** 过期瞬间大量请求同时打 DB | **互斥锁**（只有一个请求查 DB 并回写）、**逻辑过期**（不设过期，由逻辑判断是否异步更新） |
| **雪崩** | **大量 key 同时过期**或**缓存宕机**，请求压垮 DB | **过期时间打散**（加随机值）、**多级缓存**、**限流降级**、集群高可用 |

### 3️⃣ 分布式 Session

- **问题**：多实例无状态部署，Session 存本机则用户下次请求到另一实例会丢 Session。**方案**：**Session 存 Redis**，多实例共享；或 **JWT** 无状态（token 自带信息，服务端不存）。

------

### 📖 本章小结

| 知识点 | 面试关键词 | 实际应用 |
|--------|------------|----------|
| 缓存一致 | Cache-Aside、先更新 DB 再删缓存、延迟双删、过期兜底 | 写流程设计、接受最终一致 |
| 穿透/击穿/雪崩 | 布隆、空值/互斥锁、逻辑过期/打散、多级缓存、限流 | 面试必答三种区别与应对 |
| 分布式 Session | Redis 存 Session、JWT 无状态 | 多实例登录态 |

------

### ⚠️ 常见坑与注意点

1. **现象**：写时“先删缓存再更新 DB”。**原因**：并发下可能读请求在 DB 更新前把旧数据又写入缓存。**正确做法**：**先更新 DB 再删缓存**；或延迟双删降低概率。
2. **现象**：热点 key 不设互斥或逻辑过期。**原因**：过期瞬间大量请求同时查 DB。**正确做法**：加锁只放一个请求查 DB 并回写，或逻辑过期（value 带逻辑时间，异步刷新）。
3. **现象**：所有 key 过期时间相同。**原因**：同时过期易雪崩。**正确做法**：过期时间加**随机值**（如 30min + random(0, 5min)）打散。

------

### 🎯 面试常见追问

| 面试官提问 | 回答思路 |
|------------|----------|
| 缓存和 DB 一致怎么保证？ | Cache-Aside、写时先更新 DB 再删缓存；可延迟双删、设过期兜底；强一致难，多用最终一致+过期。若追问穿透/击穿/雪崩见本章 2️⃣。 |
| 穿透、击穿、雪崩区别和解决？ | 穿透：查不存在→布隆、空值缓存。击穿：热点 key 过期→互斥锁、逻辑过期。雪崩：大量过期或宕机→打散过期、多级缓存、限流。 |
| 分布式 Session 怎么做？ | Session 存 Redis 多实例共享；或 JWT 无状态。 |

## 第7章：分库分表与读写分离

本章解决**数据量过大时的水平拆分**与**读多写少时的读写分离**，与第 3 章分布式 ID、第 5 章分布式事务关联。校招常考：何时分库分表、分片键选择、读写分离延迟、中间件。

------

### 1️⃣ 分库分表

- **为什么需要**：单库单表**数据量/并发**过大，需**水平拆分**（按库、按表分片）。**分片方式**：按 **user_id、order_id** 等**取模**或**范围**分片；取模均匀但扩容要迁移；范围便于按时间等归档。
- **带来的问题**：**跨库 join**（避免或业务层拼）、**分布式事务**（见第 5 章）、**全局唯一 ID**（见第 3 章）、**跨库分页/排序**（需汇总或二次查询）。

#### （1）分库分表时机

| 指标 | 建议阈值 | 说明 |
|------|----------|------|
| **单表数据量** | 1000万～5000万行 | InnoDB 单表千万级性能开始下降 |
| **单库连接数** | 接近最大连接数（如 1000） | 连接池耗尽，新请求无法连接 |
| **单库磁盘** | 磁盘使用率 > 80% | 影响写入性能，需扩容 |
| **查询 RT** | P99 > 500ms | 索引失效或数据量过大 |

> 💬 **面试怎么说**：单表千万级、单库连接/IO 压力大时考虑分库分表；先优化索引、SQL、缓存，再考虑拆分。

#### （2）分库 vs 分表

| 方式 | 场景 | 说明 |
|------|------|------|
| **分库** | 单库连接数/IO 压力大 | 按业务或取模拆到多个数据库实例 |
| **分表** | 单表数据量大 | 单库内按取模/范围拆成多张表 |
| **分库分表** | 数据量大且并发高 | 先分库再分表，如 4 库 × 8 表 |

#### （3）分片策略对比

| 策略 | 优点 | 缺点 | 适用 |
|------|------|------|------|
| **取模** | 数据均匀、简单 | 扩容需迁移（rehash） | 数据量稳定、均匀分布 |
| **范围** | 扩容无需迁移、便于归档 | 可能热点（如时间范围） | 按时间分表、冷热分离 |
| **一致性哈希** | 扩容迁移少 | 实现复杂 | 缓存分片、节点动态变化 |
| **混合策略** | 取模+范围结合 | 复杂 | 大规模系统 |

```
取模分表示例（user_id % 8）：
user_id = 1001 → 1001 % 8 = 1 → user_1 表
user_id = 1002 → 1002 % 8 = 2 → user_2 表

范围分表示例（按月份）：
order_202401、order_202402、order_202403...
```

#### （4）分库分表带来的挑战

| 挑战 | 解决方案 |
|------|----------|
| **跨库 join** | 避免跨库 join；业务层拼装；冗余字段；用搜索引擎（ES） |
| **分布式事务** | 最终一致（本地消息表/MQ）；强一致用 Seata（见第 5 章） |
| **全局唯一 ID** | 雪花算法、号段模式（见第 3 章） |
| **跨库分页/排序** | 全局汇总（性能差）；限制返回条数；二次查询 |
| **数据迁移** | 双写迁移；增量同步；灰度切换 |

### 2️⃣ 分片键选择

- **原则**：尽量让**高频查询带分片键**，避免跨库；**均匀**分布（避免热点）；**业务语义**清晰（如 user_id、order_id）。**避免**：用易变的、或与查询无关的字段做分片键导致扫全库。

#### （1）分片键选择原则

| 原则 | 说明 | 示例 |
|------|------|------|
| **高频查询带分片键** | 80% 以上的查询能直接路由到分片 | 订单表按 user_id 分片，查用户订单高效 |
| **数据均匀** | 避免热点分片 | 用户 ID 取模比手机号更均匀 |
| **业务语义** | 与业务逻辑相关，易于理解 | user_id、order_id、tenant_id |
| **不可变性** | 分片键值不应变化 | 避免用状态、时间等易变字段 |

#### （2）常见分片键选择

| 表 | 推荐分片键 | 说明 |
|------|-----------|------|
| **用户表** | **user_id** | 查用户信息、个人主页都带 user_id |
| **订单表** | **user_id** 或 **order_id** | 按用户查订单用 user_id；按订单号查用 order_id |
| **商品表** | **商品 ID** | 商品 ID 取模均匀 |
| **日志表** | **时间** | 按时间范围分表，便于归档 |

#### （3）非分片键查询解决方案

| 场景 | 方案 |
|------|------|
| 按手机号查用户（手机号非分片键） | **冗余表**：手机号→user_id 映射表；或**ES 搜索** |
| 按订单号查订单（订单号非分片键） | **基因法**：订单号包含 user_id 信息；或**全局索引** |
| 分页查询（不带分片键） | **限制返回条数**；或**二次查询**：先查索引表再查详情 |

```
基因法示例：
订单号 = user_id 后 4 位 + 时间戳 + 序列号
订单号 1234567890123456 → 提取 user_id 后 4 位 3456 → 路由到对应分片
```

### 3️⃣ 读写分离

- **思路**：**写主库**，**读从库**（一主多从）；降低主库压力、提高读吞吐。**主从复制延迟**导致**读不一致**（刚写的读从库可能读不到）。**处理**：**强制读主**（写后读走主）、**接受短暂延迟**（如 100ms 内读主）、或**关键读走主、其它走从**。

#### （1）读写分离架构

```
┌─────────────────────────────────────────────────────────────┐
│                    读写分离架构                              │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   应用层                                                     │
│      │                                                       │
│      ▼                                                       │
│   ┌─────────────┐                                            │
│   │  路由层     │  写请求 → 主库                              │
│   │ (Sharding- │  读请求 → 从库（负载均衡）                   │
│   │  Sphere/   │                                            │
│   │  MyCat)    │                                            │
│   └──────┬──────┘                                            │
│          │                                                   │
│   ┌──────┴──────┐                                            │
│   │    主库     │◄───────────────────────────────────────────┤
│   │   (写)      │         主从复制（binlog）                  │
│   └──────┬──────┘                                            │
│          │                                                   │
│   ┌──────┴──────┐                                            │
│   │    从库 1   │                                            │
│   │   (读)      │                                            │
│   └─────────────┘                                            │
│          │                                                   │
│   ┌──────┴──────┐                                            │
│   │    从库 2   │                                            │
│   │   (读)      │                                            │
│   └─────────────┘                                            │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

#### （2）主从复制延迟处理

| 方案 | 说明 | 适用 |
|------|------|------|
| **强制读主** | 写后读走主库 | 对一致性要求高的场景 |
| **延迟读** | 写后 100ms 内读主，之后读从 | 可接受短暂延迟 |
| **关键读主** | 关键查询走主，普通查询走从 | 大部分读走从，关键读走主 |
| **缓存补偿** | 写主库同时写缓存，读时先查缓存 | 读多写少 |

#### （3）主从复制原理（MySQL）

```
1. 主库写 binlog（二进制日志）
2. 从库 IO 线程读取 binlog 写入 relay log
3. 从库 SQL 线程重放 relay log 到从库

延迟原因：
- 网络延迟
- 从库单线程重放（MySQL 5.6+ 支持并行复制）
- 从库压力大
```

### 4️⃣ 中间件

- **ShardingSphere、MyCat** 等：做**分片路由**（根据分片键路由到库表）、**读写分离**（写主、读从）；校招需知概念与适用场景，不必深究配置。

#### （2）ShardingSphere JDBC vs Proxy

| 特性 | ShardingSphere-JDBC | ShardingSphere-Proxy |
|------|---------------------|----------------------|
| **接入方式** | 应用依赖，直连数据库 | 独立服务，应用连 Proxy |
| **性能** | 高（无网络转发） | 中（多一层转发） |
| **异构语言** | 仅 Java | 任意语言 |
| **运维成本** | 低（应用内） | 高（独立部署） |
| **适用** | Java 应用、高性能 | 多语言、统一管控 |

```yaml
spring:
  shardingsphere:
    datasource:
      names: ds0, ds1
      ds0:
        type: com.zaxxer.hikari.HikariDataSource

        jdbc-url: jdbc:mysql://localhost:3306/db0

        password: root
      ds1:
        type: com.zaxxer.hikari.HikariDataSource

        username: root
        password: root
    
    rules:
      sharding:
        tables:
          t_order:
            actual-data-nodes: ds$->{0..1}.t_order_$->{0..7}

              standard:

              standard:
                sharding-column: user_id

            type: INLINE
```

### ⚠️ 常见坑与注意点

1. **现象**：分片键选错导致大量跨库查询。**正确做法**：高频查询条件尽量带分片键；若必须按非分片键查，考虑冗余或搜索引擎，避免全库扫描。
2. **现象**：写后立刻读从库发现读不到。**原因**：主从延迟。**正确做法**：写后短时间内的读走主库，或接受短暂不一致并提示用户。
3. **现象**：分库分表后仍用自增主键。**正确做法**：必须用**分布式 ID**（雪花、号段等），否则多库会重复。

| 面试官提问 | 回答思路 |
|------------|----------|
| 什么时候要分库分表？ | 单表数据量过大（如千万级）、单库连接/IO 压力大、需要水平扩展时。会带来跨库 join、分布式事务、全局 ID 等问题。 |
