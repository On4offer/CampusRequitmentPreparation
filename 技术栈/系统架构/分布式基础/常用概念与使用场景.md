# 📌 分布式基础常用概念与使用场景速查

> 配合《学习笔记》使用，侧重**CAP、分布式 ID、锁、事务、缓存与分库分表**，便于速查与校招面试。

---

## 一、CAP 与 BASE

| 概念 | 说明 |
|------|------|
| **C 一致性** | 所有节点同一时刻看到同一份数据 |
| **A 可用性** | 非故障节点在合理时间内返回响应 |
| **P 分区容错** | 网络分区时系统仍能继续工作 |
| **取舍** | 实际中 P 必须接受，故在 C 与 A 间权衡；CP（如 ZK）vs AP（如 Eureka） |
| **BASE** | 基本可用、软状态、最终一致；用最终一致换高可用 |

---

## 二、分布式 ID

| 方案 | 特点 | 适用 |
|------|------|------|
| **UUID** | 无序、占空间、无依赖 | 对有序无要求 |
| **雪花 Snowflake** | 1+41+10+12 位，趋势递增、高性能 | 订单/日志等需要有序 |
| **号段/Leaf** | DB 或 Redis 取一段，减少 IO | 需要可控、可读时可考虑 |
| **Redis INCR** | 简单，需保证高可用与持久化 | 量不大、简单场景 |

- **雪花**：符号位 1 + 时间戳 41 + 机器 ID 10 + 序列 12；同一毫秒内序列区分，多机机器 ID 不同。

## 三、分布式锁

| 实现 | 要点 | 优缺点 |
|------|------|--------|
| **Redis SET NX EX** | 原子设值+过期；Lua 保证“判断+删”原子 | 实现简单；需处理续期、误删、单点 |
| **Redisson** | 看门狗自动续期、可重入、Lua 脚本 | 生产常用，需 Redis 高可用 |
| **Zookeeper** | 临时顺序节点，最小获锁；会话超时释放 | 强一致、无过期；性能与依赖 ZK |
| **DB 唯一索引** | 插入成功即获锁，失败则未获锁 | 实现简单；压力在 DB、无自动释放需定时清理 |

- **防误删**：锁的 value 存唯一标识（如 UUID），删除时先 get 再 del，或 Lua 脚本“仅当 value 匹配时 del”。

---

## 四、分布式事务（速查）

| 类型 | 方案 | 特点 |
|------|------|------|
| **强一致** | Seata AT/TCC、2PC | 一致性好；性能与复杂度高 |
| **最终一致** | 本地消息表、MQ 事务消息、Saga | 可用性高、实现相对简单；需幂等与补偿 |

- **幂等**：同一请求多次执行效果与一次相同；用唯一业务键、状态机、Token 等保证。

---

## 五、缓存问题

| 问题 | 原因 | 常见方案 |
|------|------|----------|
| **穿透** | 查不存在的数据，绕过缓存打 DB | 布隆过滤器、空值缓存（短 TTL） |
| **击穿** | 热点 key 过期瞬间大量请求打 DB | 互斥锁（只放一个请求查）、逻辑过期 |
| **雪崩** | 大量 key 同时过期或缓存宕机 | 过期时间加随机、多级缓存、限流降级 |
| **一致性** |  DB 与缓存不一致 | 先更新 DB 再删缓存、延迟双删、过期兜底 |

## 六、分库分表与读写分离

| 概念 | 说明 |
|------|------|
| **分库** | 按库拆分，分散连接与存储 |
| **分表** | 单库内表水平拆成多张（如 order_0..order_15） |
| **分片键** | 取模或范围路由；选好避免热点与跨库 join |
| **读写分离** | 写主、读从；主从延迟可读主或接受短暂不一致 |

- **何时分**：单表千万级、读写压力大、冷热分离等；先考虑索引、归档、读写分离再分库分表。

---

## 七、面试一句话总结

- **CAP**：分区下只能在 C 与 A 间权衡；BASE 用最终一致换可用。
- **分布式 ID**：雪花趋势递增、号段减 DB 压力；UUID 简单无序。
- **分布式锁**：Redis（NX+EX+Lua/Redisson）或 ZK；注意续期与误删。
- **分布式事务**：强一致用 Seata，最终一致用 MQ+幂等。
- **缓存**：穿透用布隆/空值；击穿用互斥/逻辑过期；雪崩用打散过期+多级。
- **分库分表**：数据与压力大时水平拆；分片键与读写分离延迟要考虑。
