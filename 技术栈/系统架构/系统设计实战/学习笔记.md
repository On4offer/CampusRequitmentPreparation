> 系统学习见下文各章；日常常用概念与使用场景速查见同目录《常用概念与使用场景》。  
> 适用：校招/社招系统设计面试，面向秒杀、短链、Feed、IM、缓存与存储等经典题及答题方法。

------

## 第1章：系统设计面试概述

本章建立对**系统设计面试**的整体认知：考什么、怎么答、通用步骤是什么，为后续各章（秒杀、短链、Feed、IM、缓存等）提供答题框架。与分布式基础、高可用高并发、微服务等知识联动。

------

### 1️⃣ 考察目的

- **系统设计**考察：**需求理解**、**抽象能力**、**技术选型**、**权衡**（一致性/可用性/成本）、**扩展性与落地思路**。不要求实现细节，要求**思路清晰、能拆问题、能说 trade-off**；常与高并发、分布式、缓存、MQ 结合。
- **为什么考**：看候选人能否从业务抽象出系统、在约束下做合理取舍、与面试官沟通推进。

### 2️⃣ 答题步骤（通用）

| 步骤 | 内容 | 面试怎么说 |
|------|------|------------|
| 1. 澄清需求 | 功能（核心/次要）、用户量/QPS、读写比、时延、一致性 | “先确认一下：日活多少、峰值 QPS、读写比、能否最终一致？” |
| 2. 估算 | QPS、存储量、带宽等（**量级**即可，如 1k/10w/1亿） | “假设日活 1000 万、每人 10 次，约 1k+ QPS，峰值按 10 倍留” |
| 3. 顶层设计 | 画框图：客户端、网关、服务、缓存、DB、MQ；数据流与职责 | “整体分这几层：网关限流、服务层、缓存、DB、MQ 削峰……” |
| 4. 核心链路深挖 | 如秒杀：防超卖、限流、削峰；短链：生成、跳转、统计 | 面试官追问哪就展开哪，说清“为什么这么设计” |
| 5. 扩展与优化 | 分库分表、多级缓存、异步、降级；瓶颈与改进 | “后续可加读写分离、本地缓存、降级开关” |
| 6. 小结 | 方案总结与可替代方案（如强一致 vs 最终一致） | “这里选最终一致是优先可用性；若强一致可上分布式事务” |

> 💬 **重点**：**先总后分**——先给整体框图，再按模块或链路展开；**主动说 trade-off**，体现权衡思维。

------

### 📖 本章小结

| 知识点 | 面试关键词 | 实际应用 |
|--------|------------|----------|
| 考察目的 | 需求理解、抽象、选型、权衡、扩展 | 知道面试官在考什么 |
| 答题步骤 | 澄清→估算→顶层设计→深挖→扩展→小结 | 按步骤答不慌、不遗漏 |
| 技巧 | 先总后分、说 trade-off | 口述时条理清晰 |

------

### ⚠️ 常见坑与注意点

1. **现象**：一上来就画细节。**原因**：没先澄清需求与顶层。**正确做法**：先问清楚 QPS、一致性、核心功能，再画整体框图，最后深挖面试官关心的点。
2. **现象**：只说“用 Redis、用 MQ”不说为什么。**正确做法**：每个选型说一句“为什么”（如“用 Redis 扣库存是为了抗住瞬时 QPS，再异步落库保证最终一致”）。
3. **现象**：估算完全不会。**正确做法**：记几个量级（1k、1w、10w、100w、1亿）；日活×人均次数≈QPS 量级；存储按条数×单条大小粗算。

------

### 🎯 面试常见追问

| 面试官提问 | 回答思路 |
|------------|----------|
| 系统设计一般怎么答？ | 先澄清需求与量级、再顶层设计画框图、然后核心链路深挖、最后扩展与小结；先总后分、说 trade-off。若追问具体题见第 2～6 章。 |
| 估算不会怎么办？ | 给量级即可：日活/DAU、每人每天几次、峰值倍数；QPS=DAU×次数/86400×峰值系数；存储=条数×单条大小。 |

## 第2章：秒杀系统

本章解决**秒杀场景下的高并发、防超卖、削峰与限流**设计，是系统设计面试高频题。与分布式基础（分布式锁、缓存）、高可用（限流、降级）、第 6 章缓存与存储联动。

------

### 1️⃣ 特点与挑战

- **特点**：**瞬时高并发**、读多写多、**库存有限**、必须**防超卖**、防黄牛/刷单。
- **挑战**：流量远大于库存、DB 扛不住直接扣减、超卖与重复下单。

### 2️⃣ 整体思路（分层）

| 层级 | 手段 | 目的 |
|------|------|------|
| **前端** | 限流、防重复提交（按钮置灰、token） | 减少无效请求 |
| **网关** | 全局限流、按 IP/用户限流 | 保护下游 |
| **服务** | 多级缓存（页面静态化、商品信息 Redis）、**库存预扣减** | 少打 DB、抗读与扣减 |
| **库存** | **Redis 预扣减** + 扣减成功发 **MQ**，消费端写订单与 DB 扣库存 | **削峰**、DB 异步写、防超卖 |
| **DB** | 乐观锁 `update ... where stock > 0` 或唯一约束防重复 | 最后一道防超卖与重复 |

### 3️⃣ 防超卖（面试必问）

- **方案一**：**Redis 预扣减**（DECR 或 Lua 保证“判断+扣减”原子），扣减成功再发 MQ；消费端写订单并在 DB 用 **乐观锁** 扣库存（`update product set stock=stock-1 where id=? and stock>0`），失败则回滚 Redis（或补回）+ 取消订单。
- **方案二**：不用 Redis，直接 DB 乐观锁扣减，并发高时 DB 压力大，一般配合 Redis 预扣减 + 异步落库。
- **面试怎么说**：Redis 抗瞬时 QPS，扣减成功才进 MQ；DB 用乐观锁做最终一致与防超卖，超卖关键在“扣减原子性+库存校验”。

### 4️⃣ 削峰、限流与降级

- **削峰**：请求先进 **MQ**，消费端按能力消费，避免瞬时打垮 DB。**限流**：网关+应用层限流，多余请求直接返回“已抢光”或排队。**降级**：非核心功能关（如推荐、评论），保证下单链路；可返回静态页或排队页。

------

### 📖 本章小结

| 知识点 | 面试关键词 | 实际应用 |
|--------|------------|----------|
| 分层设计 | 前端/网关/服务/Redis/MQ/DB | 画框图答秒杀 |
| 防超卖 | Redis 预扣减+Lua、MQ、DB 乐观锁 | 必答链路 |
| 削峰与限流 | MQ 削峰、网关限流、降级 | 与高可用章节结合 |

------

### ⚠️ 常见坑与注意点

1. **现象**：只答 Redis 扣减不说 DB 最终一致。**正确做法**：Redis 是预扣减、抗并发；**订单与库存落库**要在 DB 做，并用乐观锁防超卖，Redis 与 DB 之间靠 MQ 异步+补偿。
2. **现象**：忘记幂等。**正确做法**：同一用户同一商品同一场次只能下一单，用**唯一键**（用户+商品+场次）或 token 防重复提交。
3. **现象**：不提前预热缓存。**正确做法**：秒杀开始前把**商品信息、库存**预热到 Redis，避免开场瞬间打 DB。

------

### 🎯 面试常见追问

| 面试官提问 | 回答思路 |
|------------|----------|
| 为什么用 Redis 扣库存？ | 抗瞬时 QPS，内存操作快；扣减成功再异步落库，DB 用乐观锁做最终一致与二次校验。若追问不超卖见本章 3️⃣。 |
| 如何保证不超卖？ | Redis 扣减用 Lua 保证原子；DB 扣减用乐观锁 update where stock>0；MQ 消费失败可回滚 Redis 或补库存。 |
| 如何削峰？ | 请求进 MQ，消费端按能力消费；配合限流把多余请求挡在网关/应用层。 |

## 第3章：短链服务

本章解决**长 URL 转短链、高并发跳转与统计**的设计，是系统设计面试常见题。与分布式基础（分布式 ID、缓存）联动。

------

### 1️⃣ 需求澄清

- **核心**：长 URL → 短链（如 t.cn/xxx）；跳转时根据**短码查长链**并 **302 重定向**。**可选**：统计 PV/UV、过期时间、同一长链复用同一短链（防重）。
- **量级**：读远多于写（跳转多、生成少）；短码需**全局唯一**、尽量短。

### 2️⃣ 短码生成

| 方案 | 思路 | 优点 | 缺点 |
|------|------|------|------|
| **发号器** | 用**分布式 ID**（雪花、号段）取唯一 ID，转 **62 进制/Base62** 得短码 | 唯一、可控长度、无冲突 | 依赖发号器 |
| **Hash** | MD5/SHA 取前 N 位 | 同长链可同短链（去重） | **冲突**需处理（加盐重试或后缀） |

> 💬 **面试怎么说**：发号器方案更稳，ID 唯一则短码唯一；62 进制能把数字压短。若要去重（同一长链同一短链）可先查 DB 是否已有，没有再用发号器生成。

### 3️⃣ 存储与跳转

- **存储**：**短码 → 长链**；**Redis**（热数据、高并发读）+ **DB**（持久化）。读多写少，**缓存为主**，可设过期时间。
- **跳转**：GET 短链 → 查 **Redis**（无则查 DB 并回写 Redis）→ **302 Location: 长链**。**统计**：PV/UV 可**异步**写 MQ 或定时批量落库，避免阻塞跳转。

### 4️⃣ 防重与过期

- **防重**：同一长链是否复用短链？若复用则生成前先查 DB（长链→短码）；否则每次发号器新 ID。**过期**：Redis 与 DB 都可存过期时间，跳转时校验；过期可物理删除或标记无效。

------

### 📖 本章小结

| 知识点 | 面试关键词 | 实际应用 |
|--------|------------|----------|
| 短码生成 | 发号器+62 进制、Hash+冲突处理 | 唯一、短、可选的同长链复用 |
| 存储与跳转 | 短码→长链、Redis+DB、302 | 高并发读、缓存为主 |
| 统计与过期 | 异步 PV/UV、过期时间 | 不阻塞主链路 |

------

### ⚠️ 常见坑与注意点

1. **现象**：Hash 方案不处理冲突。**正确做法**：冲突时加盐重试或拼后缀，保证唯一；或直接用发号器省事。
2. **现象**：跳转链路同步写统计。**正确做法**：统计**异步**（MQ 或队列），避免拖慢 302 响应。
3. **现象**：短码可枚举被遍历。**正确做法**：短码足够长（如 7 位 62 进制空间很大）、或加鉴权/风控；敏感链可设过期。

------

### 🎯 面试常见追问

| 面试官提问 | 回答思路 |
|------------|----------|
| 短码如何保证唯一？ | 发号器（分布式 ID）保证 ID 唯一，转 62 进制得短码即唯一；或 Hash+冲突处理。若追问高并发读见本章 3️⃣。 |
| 如何支撑高并发跳转？ | 短码→长链存 Redis，读多写少以缓存为主；查 Redis 无则查 DB 并回写；统计异步不阻塞。 |

## 第4章：点赞、计数与 Feed 流

本章解决**点赞与计数的高并发**以及 **Feed 流（关注人动态时间线）**的推/拉设计，是社交与内容类系统设计常考。与分布式基础（缓存、MQ）、高可用（限流）联动。

------

### 1️⃣ 点赞

- **特点**：读多写多、需防**重复点赞**（同一用户同一内容只加一次）。**存储**：**Redis Set** 存“某内容被谁点赞”，**计数**用 Redis 或异步汇总；**DB** 做持久与最终一致（定时或 MQ 同步）。**防重**：Set 天然去重，或 DB 唯一约束（用户+内容）。

### 2️⃣ 计数

- **问题**：高并发下直接 **update count++** 打爆 DB。**方案**：**Redis 计数**（INCR），定时或批量同步 DB；或 **MQ 异步**写 DB；读时以 Redis 为准或 Redis+DB 合并。

### 3️⃣ Feed 流：推 vs 拉

| 模式 | 思路 | 写/读特点 | 适用 |
|------|------|-----------|------|
| **推（写扩散）** | 发帖时写进**每个粉丝的 Feed 收件箱** | 写多读少，读只查自己收件箱 | **粉丝少**的普通用户 |
| **拉（读扩散）** | 读时**聚合关注人的最新动态** | 读多写少，写只写自己发件箱 | **大 V**（粉丝太多写扩散爆炸） |
| **推拉结合** | 普通用户用推、**大 V 用拉** | 折中 | 实际常用 |

> 💬 **面试怎么说**：大 V 发帖若推给所有粉丝，写放大会很大；所以大 V 用拉模式，读时再聚合；普通用户用推，读简单。存储上：发件箱表（用户发了什么）、收件箱表（用户收到的 Feed）、关注关系表。

### 4️⃣ 时序与去重

- **时序**：按**时间戳**排序；多源聚合时按时间归并。**去重**：同一动态只出现一次（发件箱唯一、收件箱按 feed_id 去重）。**分页**：按时间游标（上次最小时间戳）拉取下一批。

------

### 📖 本章小结

| 知识点 | 面试关键词 | 实际应用 |
|--------|------------|----------|
| 点赞与计数 | Redis Set、计数 INCR、异步落库、防重 | 高并发点赞、计数 |
| Feed 推拉 | 写扩散/读扩散、大 V 用拉、收件箱/发件箱 | 时间线、关注流 |
| 时序与去重 | 时间戳、游标分页、去重 | 不重复、顺序正确 |

------

### ⚠️ 常见坑与注意点

1. **现象**：大 V 也用推。**正确做法**：大 V 粉丝量级大，推会导致写放大（一条帖子写几百万条收件箱），必须用**拉**或推拉结合。
2. **现象**：计数只写 Redis 不落库。**正确做法**：Redis 可丢，需**定时或异步同步 DB** 做持久与恢复；或接受重启后从 DB 重建计数。
3. **现象**：Feed 读扩散时每次都查所有关注人。**正确做法**：可缓存关注列表、只查最近活跃用户的动态，或按时间范围+限制条数减少查询。

------

### 🎯 面试常见追问

| 面试官提问 | 回答思路 |
|------------|----------|
| 推模式和拉模式区别？怎么选？ | 推：发帖写进粉丝收件箱，读只读自己；拉：读时聚合关注人动态。粉丝少用推、大 V 用拉，常推拉结合。若追问存储见本章 3️⃣。 |
| 如何保证 Feed 时序与去重？ | 按时间戳排序；多源按时间归并；去重按 feed_id。分页用时间游标。 |

## 第5章：即时通讯（IM）与消息队列

本章解决**IM 的架构要点**（长连接、消息不丢、顺序与扩展）以及**与 MQ 的区别**，是系统设计面试中“设计一个 IM”或“IM 和 MQ 区别”的答题依据。

------

### 1️⃣ IM 架构简述

- **连接**：客户端与服务器**长连接**（WebSocket、TCP），便于双向推送。**消息路径**：消息经**服务端转发**（可存可不存）；多端同步、未读计数、**离线消息**（存库，上线拉取）、群聊与推送。
- **为什么长连接**：服务端可主动推（消息、状态）；若用短轮询则延迟与开销大。

### 2️⃣ 消息不丢与顺序

- **不丢**：**客户端 ACK**（收到后回 ack）、**服务端持久化**（写 DB 或 MQ 再落库）、**重试与幂等**（重复消息按 msgId 去重）。**离线消息**：存 DB，用户上线后按时间拉取。
- **顺序**：单聊可按**会话+序号**保证；群聊可单通道或按会话队列保证顺序；多端已读状态可**同步协议**（已读回执）。

### 3️⃣ 高并发与扩展

- **连接层**：**水平扩展**（多机扛连接），通过 **Router/负载均衡** 把同一用户路由到同一节点（或会话保持）。**消息**：写 **MQ** 异步落库，消费端写 DB、推送给在线连接。**在线状态**：存 **Redis**（userId→节点/连接），便于转发与状态查询。

### 4️⃣ IM 与 MQ 区别（面试常问）

| 维度 | IM | MQ |
|------|----|-----|
| **场景** | **端到端**（人-人、人-端） | **系统间**（服务-服务） |
| **模型** | 长连接、实时推送、会话 | 生产-消费、队列/主题 |
| **持久** | 可选（离线消息要存） | 一般持久、可重放 |
| **顺序** | 按会话/通道保证 | 按分区/队列保证 |

> 💬 **面试怎么说**：IM 是给人用的实时通信，长连接+推送+离线消息；MQ 是系统间异步解耦、削峰，不关心“谁在线”。两者可结合：IM 服务端把消息写 MQ 再落库与推送。

------

### 📖 本章小结

| 知识点 | 面试关键词 | 实际应用 |
|--------|------------|----------|
| IM 架构 | 长连接、服务端转发、离线消息、在线状态 | 画框图答“设计一个 IM” |
| 不丢与顺序 | ACK、持久化、幂等、会话序号 | 消息可靠与有序 |
| 与 MQ 区别 | 端到端 vs 系统间、实时 vs 异步 | 面试必区分 |

------

### ⚠️ 常见坑与注意点

1. **现象**：长连接不设心跳。**正确做法**：**心跳**检测连接存活，超时断连并清理状态；否则假连接占满资源。
2. **现象**：群消息给每人写一份收件箱。**正确做法**：可**读扩散**（存一份群消息，读时按群拉取）或写扩散按需，避免大群写放大。
3. **现象**：IM 和 MQ 混为一谈。**正确做法**：IM 面向“人”、实时、长连接；MQ 面向“系统”、异步、队列。面试时明确区分。

------

### 🎯 面试常见追问

| 面试官提问 | 回答思路 |
|------------|----------|
| 长连接和心跳的作用？ | 长连接便于服务端主动推送；心跳检测存活、超时断连并清理，避免假连接占资源。若追问消息不丢见本章 2️⃣。 |
| IM 和 MQ 区别？ | IM 是端到端、给人用、长连接实时推送；MQ 是系统间、异步解耦、队列。可结合：IM 服务端用 MQ 异步落库。 |

## 第6章：缓存与存储设计

本章解决**何时用缓存、存储选型**以及**缓存一致性、三大问题**在系统设计中的表述，与分布式基础（缓存、分库分表）直接联动。面试中“为什么用 Redis/MySQL/ES”常从这里答。

------

### 1️⃣ 何时用缓存

- **适用**：**读多写少**、**热点明显**、可接受**短暂不一致**。**手段**：**多级缓存**（本地 + Redis）、更新策略（**Cache-Aside**、过期兜底）。**为什么**：降低 DB 压力、降低 RT。
- **不适用**：写多读少、强一致要求高、数据几乎无热点。

### 2️⃣ 缓存一致性、穿透/击穿/雪崩（简述）

- **一致性**：**先更新 DB 再删缓存**；延迟双删、过期兜底；强一致难，多用最终一致（见《分布式基础》第 6 章）。
- **穿透**：布隆、空值缓存。**击穿**：互斥锁、逻辑过期。**雪崩**：过期打散、多级缓存、限流（见《分布式基础》《高可用》）。

### 3️⃣ 存储选型

| 类型 | 典型 | 适用场景 |
|------|------|----------|
| **关系型** | MySQL | 事务、复杂查询、结构化、强一致 |
| **KV** | Redis | 缓存、会话、计数、简单结构 |
| **搜索引擎** | Elasticsearch | 全文检索、日志检索、复杂条件 |
| **时序/日志** | InfluxDB、ES、Kafka | 监控指标、日志、事件流 |
| **分库分表/读写分离** | 见《分布式基础》 | 单表/单库过大、读多写少 |

> 💬 **面试怎么说**：选型看“事务是否需要、查询是否复杂、是否要全文检索、量级与一致性”；Redis 做缓存与高速读写，MySQL 做持久与事务，ES 做搜索。

------

### 📖 本章小结

| 知识点 | 面试关键词 | 实际应用 |
|--------|------------|----------|
| 何时用缓存 | 读多写少、热点、可接受不一致、多级 | 设计时说明“这里加缓存因为…” |
| 一致性与三大问题 | Cache-Aside、穿透/击穿/雪崩应对 | 与分布式基础结合答 |
| 存储选型 | MySQL/Redis/ES、事务/查询/搜索 | 为什么选某某存储 |

------

### ⚠️ 常见坑与注意点

1. **现象**：所有读都走缓存。**正确做法**：写多或强一致场景慎用缓存；或缓存只做旁路、DB 为准。
2. **现象**：存储选型只记名字不说场景。**正确做法**：每个选型说“因为要事务/复杂查询/全文检索/高并发读所以用 MySQL/ES/Redis”。
3. **现象**：设计里不提分库分表与读写分离。**正确做法**：数据量大、读多写少时在“扩展与优化”里提分库分表、读写分离（见分布式基础）。

------

### 🎯 面试常见追问

| 面试官提问 | 回答思路 |
|------------|----------|
| 为什么用 Redis/MySQL/ES？ | Redis：缓存、高并发读、简单 KV。MySQL：事务、复杂查询、持久。ES：全文检索、日志检索、复杂条件。按场景说。若追问缓存一致见《分布式基础》。 |
| 缓存穿透/击穿/雪崩怎么解决？ | 穿透：布隆、空值缓存。击穿：互斥锁、逻辑过期。雪崩：过期打散、多级、限流。简要答即可，细节见分布式基础。 |

## 第7章：综合与面试技巧

本章收束**答题技巧**与**常见小题型**，并点明与分布式、高可用、微服务等前面章节的联动，便于系统设计面试时成体系作答。

------

### 1️⃣ 答题技巧

- **先总后分**：先给**整体框图**（客户端、网关、服务、缓存、DB、MQ），再按模块或**核心链路**展开；面试官追问哪就深挖哪。
- **说 trade-off**：每个选型说一句“为什么”，例如：“这里用**最终一致**是因为可用性更重要”“用 **Redis 扣库存**是为了抗住瞬时 QPS，再异步落库保证最终一致”。
- **量级意识**：估算时给**量级**（1k/10w/1亿），不纠结精确数；说明“峰值按 N 倍留”“存储按 X 年估算”。

### 2️⃣ 常见小题型（速记）

| 场景 | 要点 |
|------|------|
| **延迟队列** | 定时任务扫表、MQ 延迟消息（RocketMQ）、Redis 过期回调或 ZSet 按时间戳；扫表要分片与索引 |
| **搜索建议（前缀匹配）** | Trie 树、ES 前缀查询、Redis 或 DB like 前缀；热词放缓存 |
| **排行榜** | **Redis ZSet**（score 为分数、member 为 ID），按范围取；实时更新 ZADD |
| **分布式 Session** | Session 存 **Redis**，多实例共享；或 **JWT** 无状态（见《分布式基础》《高可用》） |

### 3️⃣ 与其它章节联动

- **微服务**：网关、注册发现、熔断限流（见微服务、高可用章节）。**分布式**：分布式 ID、锁、事务、缓存一致、分库分表（见《分布式基础》）。**高可用**：限流、熔断、负载均衡、冗余、压测与容量（见《高可用与高并发》）。系统设计题会综合用到，按题目需要抽取。

------

### 📖 本章小结

| 知识点 | 面试关键词 | 实际应用 |
|--------|------------|----------|
| 技巧 | 先总后分、说 trade-off、量级 | 面试时条理清晰、有取舍 |
| 小题型 | 延迟队列、搜索建议、排行榜、Session | 被问到能快速搭框架 |
| 联动 | 分布式、高可用、微服务 | 综合题时调用前面知识 |

------

### 🎯 面试常见追问

| 面试官提问 | 回答思路 |
|------------|----------|
| 设计一个延迟队列？ | 方案一：定时任务扫“执行时间≤当前”的记录。方案二：MQ 延迟消息（RocketMQ）。方案三：Redis ZSet 按执行时间戳作 score，轮询取。说 trade-off（精度、吞吐、实现成本）。 |
| 设计一个排行榜？ | Redis ZSet，score 为分数，member 为 ID；更新用 ZADD，取 TopN 用 ZREVRANGE；可按时间维度分 key（日榜/周榜）。若追问分布式见《分布式基础》。 |

## 第8章：微博/社交系统设计

本章解决**微博类社交系统的核心设计**：发布与查看动态、关注与粉丝、Feed 流、热点与热搜，是系统设计面试高频综合题。与第 4 章 Feed 流、分布式基础（缓存、分库分表）联动。

------

### 1️⃣ 需求澄清

- **核心功能**：用户发微博、查看首页 Feed、查看个人主页、关注/取关、点赞/评论/转发、热搜榜。
- **量级**：日活千万～亿级、发帖量级、Feed 读远多于写。**一致性**：发帖可见可接受短暂延迟（最终一致）。

### 2️⃣ 整体架构

| 层级 | 组件 | 职责 |
|------|------|------|
| **客户端** | App/Web | 展示 Feed、发帖、互动 |
| **网关** | Gateway/Nginx | 鉴权、限流、路由 |
| **服务层** | 用户服务、微博服务、关系服务、互动服务 | 业务逻辑 |
| **缓存** | Redis | 用户信息、微博内容、计数、热点 |
| **存储** | MySQL（分库分表） | 持久化 |
| **MQ** | Kafka | 异步通知、Feed 分发 |
| **搜索** | Elasticsearch | 全文检索、热搜 |

### 3️⃣ Feed 流设计（核心）

**推拉结合方案**（见第 4 章）：

| 用户类型 | 方案 | 原因 |
|----------|------|------|
| **普通用户**（粉丝少） | **推模式**：发帖时写入所有粉丝的收件箱 | 读简单、写可控 |
| **大 V**（粉丝多） | **拉模式**：发帖只写自己发件箱，读时聚合 | 避免写放大 |

**存储设计**：

```
发件箱表（user_post）：
user_id | post_id | create_time

收件箱表（user_feed）：
user_id | post_id | author_id | create_time

关注关系表（follow）：
user_id | follow_user_id | create_time
```

### 4️⃣ 发布微博流程

1. 写入**微博内容表**（分库分表，按 user_id 分片）。
2. 写入**发件箱**（user_post）。
3. 判断用户类型：
   - 普通用户：**MQ 异步**写入所有粉丝的收件箱（推）。
   - 大 V：不推，粉丝读时拉取（拉）。
4. 更新**计数**（微博数、互动数）到 Redis。

### 5️⃣ 查看 Feed 流程

1. 从**收件箱**拉取关注人的动态（推模式用户）。
2. 同时从**大 V 发件箱**拉取（拉模式用户）。
3. **按时间归并**，返回分页结果。
4. 热点微博内容从**缓存**读取。

### 6️⃣ 热搜与热点

- **热搜榜**：实时统计**搜索词频率**，用 Redis ZSet 或 Flink 实时计算；定时或实时更新榜单。
- **热点微博**：按**互动量**（点赞、评论、转发）排序，定时任务或流计算更新；结果缓存。

### 7️⃣ 分库分表策略

| 表 | 分片键 | 说明 |
|------|--------|------|
| 微博内容 | **user_id** | 同用户微博同库，查询个人主页高效 |
| 关注关系 | **user_id** | 查"我关注谁"高效；反向表存粉丝 |
| 收件箱 | **user_id** | 查"我的 Feed"高效 |

------

### 📖 本章小结

| 知识点 | 面试关键词 | 实际应用 |
|--------|------------|----------|
| Feed 推拉 | 推适合普通用户、拉适合大 V、推拉结合 | 面试必答 |
| 发布流程 | 写内容→写发件箱→MQ 推粉丝→计数 | 画流程图 |
| 查看 Feed | 收件箱拉取 + 大 V 发件箱聚合 | 时间归并 |
| 热搜热点 | 搜索词频率统计、互动量排序 | Redis ZSet / Flink |
| 分库分表 | 按 user_id 分片、避免跨库 | 与分布式基础联动 |

------

### ⚠️ 常见坑与注意点

1. **现象**：所有用户都用推模式。**正确做法**：大 V 粉丝多，推会导致写放大（一条微博写百万条收件箱），必须用拉或推拉结合。
2. **现象**：Feed 查询每次都查所有关注人。**正确做法**：可缓存关注列表、限制拉取条数、按时间范围优化；收件箱已推的不再拉。
3. **现象**：热搜被刷榜。**正确做法**：加**风控**（频率限制、异常检测）、敏感词过滤、人工干预。

------

### 🎯 面试常见追问

| 面试官提问 | 回答思路 |
|------------|----------|
| 微博 Feed 流怎么设计？ | 推拉结合：普通用户发帖推给粉丝（写收件箱）；大 V 发帖只写发件箱，粉丝读时拉取聚合。存储用发件箱/收件箱表，按时间归并。若追问推拉细节见第 4 章。 |
| 热搜怎么实现？ | 实时统计搜索词频率（Redis 计数或 Flink），按热度排序；定时或实时更新榜单；加风控防刷榜。 |
| 如何处理大 V 发帖？ | 大 V 用拉模式，不发到粉丝收件箱；粉丝读 Feed 时从大 V 发件箱拉取并聚合。避免写放大。 |

## 第9章：电商/订单系统设计

本章解决**电商订单系统的核心设计**：商品浏览、下单、库存、支付、订单状态流转，是系统设计面试经典综合题。与第 2 章秒杀、分布式基础（分布式事务、锁）联动。

------

### 1️⃣ 需求澄清

- **核心功能**：商品浏览、加入购物车、下单、支付、订单管理、库存扣减、物流跟踪。
- **量级**：日订单量级、峰值 QPS（大促时）。**一致性**：库存与订单强一致或最终一致、支付结果必须准确。

### 2️⃣ 整体架构

| 层级 | 组件 | 职责 |
|------|------|------|
| **客户端** | App/Web/小程序 | 商品浏览、下单、支付 |
| **网关** | Gateway/Nginx | 鉴权、限流、路由 |
| **服务层** | 商品服务、订单服务、库存服务、支付服务、物流服务 | 业务逻辑 |
| **缓存** | Redis | 商品信息、库存、Session |
| **存储** | MySQL（分库分表） | 订单、商品、用户数据 |
| **MQ** | RocketMQ/Kafka | 订单事件、异步通知 |
| **搜索** | Elasticsearch | 商品搜索 |

### 3️⃣ 下单核心流程

```
用户下单 → 锁库存 → 创建订单 → 支付 → 扣库存 → 通知
         ↓
      库存不足 → 返回失败
         ↓
      超时未支付 → 释放库存（回滚）
```

**详细步骤**：

1. **校验**：商品是否存在、是否上架、用户是否合法。
2. **锁库存**：Redis 预扣或 DB 乐观锁（`update stock=stock-1 where stock>0`）。
3. **创建订单**：写订单表（状态：待支付），设置**支付超时时间**。
4. **支付**：调用第三方支付（支付宝/微信），回调更新订单状态。
5. **扣库存**：支付成功后**真正扣减**库存（或下单时已扣）。
6. **异步通知**：MQ 发送订单事件，通知物流、积分、通知等服务。

### 4️⃣ 库存扣减方案

| 方案 | 思路 | 优缺点 |
|------|------|--------|
| **DB 乐观锁** | `update ... where stock>0` | 简单、强一致；高并发时 DB 压力大 |
| **Redis 预扣** | DECR/Lua 原子扣减，异步落库 | 高并发友好；需处理 Redis 与 DB 一致 |
| **预扣+回滚** | 下单锁库存、超时未支付释放 | 防超卖、支持库存回滚 |

**防超卖要点**：
- 扣减操作**原子性**（Redis Lua 或 DB 乐观锁）。
- **幂等**：同一订单号只能扣一次，用唯一键或状态机。
- **超时回滚**：未支付订单超时后释放库存。

### 5️⃣ 订单状态机

```
待支付 → 已支付 → 已发货 → 已收货 → 已完成
   ↓
已取消（超时/用户取消）
```

| 状态 | 说明 |
|------|------|
| 待支付 | 订单创建，等待用户支付 |
| 已支付 | 支付成功，等待发货 |
| 已发货 | 商家发货，物流在途 |
| 已收货 | 用户确认收货 |
| 已完成 | 订单结束，可评价 |
| 已取消 | 超时未支付或用户取消 |

**状态机实现**：订单表 `status` 字段 + 状态流转校验；MQ 事件驱动状态变更。

### 6️⃣ 分布式事务处理

| 场景 | 方案 |
|------|------|
| 下单+扣库存 | **本地消息表**：订单服务写订单+消息表，MQ 通知库存服务扣减 |
| 支付+订单状态 | **MQ 事务消息**：支付成功发消息，订单服务消费更新状态 |
| 跨服务一致性 | **最终一致+幂等**：各服务消费消息时幂等处理 |

### 7️⃣ 分库分表策略

| 表 | 分片键 | 说明 |
|------|--------|------|
| 订单表 | **user_id** 或 **order_id** | 按用户查订单高效；或按订单号哈希 |
| 订单明细 | **order_id** | 与订单同库 |
| 商品表 | **商品 ID** | 按商品分片 |
| 库存表 | **商品 ID** | 与商品同库，扣减高效 |

------

### 📖 本章小结

| 知识点 | 面试关键词 | 实际应用 |
|--------|------------|----------|
| 下单流程 | 校验→锁库存→创建订单→支付→扣库存→通知 | 画流程图 |
| 库存扣减 | DB 乐观锁、Redis 预扣、防超卖 | 与秒杀联动 |
| 订单状态机 | 待支付→已支付→已发货→已收货→已完成 | 状态流转校验 |
| 分布式事务 | 本地消息表、MQ 事务消息、最终一致 | 与分布式基础联动 |
| 分库分表 | 按 user_id 或 order_id 分片 | 查询与扩容 |

------

### ⚠️ 常见坑与注意点

1. **现象**：下单直接扣库存，用户不支付导致库存被占。**正确做法**：下单时**锁库存**（预扣），设置支付超时；超时未支付则**释放库存**。
2. **现象**：支付回调重复导致订单状态异常。**正确做法**：支付回调**幂等**处理，用订单号去重或状态机校验（只有"待支付"才能变"已支付"）。
3. **现象**：高并发下 DB 扣库存成为瓶颈。**正确做法**：用 **Redis 预扣**抗并发，异步落库；或分库分表分散压力。

------

### 🎯 面试常见追问

| 面试官提问 | 回答思路 |
|------------|----------|

| 如何防止超卖？ | 扣减原子性（Redis Lua 或 DB 乐观锁）、幂等（同一订单只扣一次）、超时回滚（未支付释放库存）。若追问秒杀见第 2 章。 |
| 订单状态机怎么设计？ | 待支付→已支付→已发货→已收货→已完成；超时或取消走已取消；状态流转需校验、幂等。 |
| 支付成功但订单状态未更新怎么办？ | 支付回调幂等+重试；本地消息表或 MQ 事务消息保证最终一致；定时任务补偿扫描异常订单。 |

## 第10章：支付系统设计

本章解决**支付系统的核心设计**：多渠道支付、幂等性、对账、安全性，是电商系统中最关键、最复杂的子系统。与第 9 章电商订单、分布式事务、幂等设计深度关联。

------

### 1️⃣ 需求澄清

- **核心功能**：收银台（展示支付方式）、支付路由、支付处理、回调通知、退款、对账、风控。
- **支付渠道**：微信支付、支付宝、银联、Apple Pay 等。
- **一致性要求**：**支付结果必须准确**（钱不能多扣、不能少扣），**幂等性**必须保证。
- **安全性**：敏感信息加密、防篡改、防重放攻击。

------

### 3️⃣ 支付核心流程

```
用户下单 → 创建支付订单 → 选择渠道 → 调渠道支付 → 用户支付 → 渠道回调
    │           │             │           │           │         │
    │           │             │           │           │         ▼
    │           │             │           │           │    处理回调
    │           │             │           │           │    幂等校验
    │           │             │           │           │    更新状态
    │           │             │           │           │    通知业务
    │           │             │           │           │         │
    │           │             │           │           ▼         │
    │           │             │           │      支付结果页 ◄───┘
    │           │             │           │
    │           │             │           ▼
    │           │             │      渠道收银台（H5/小程序/APP）
    │           │             │
    │           │             ▼
    │           │        支付路由（选最优渠道）
    │           │
    │           ▼
    │      写支付订单表（状态：待支付）
    │      写支付流水表
    │
    ▼
  订单系统调用支付系统
```

**详细步骤**：

1. **创建支付订单**：订单系统调用支付系统，创建支付订单（唯一流水号）。
2. **风控检查**：检查用户风险等级、限额限次。
3. **支付路由**：根据渠道费率、可用性、用户偏好选择最优渠道。
4. **调起支付**：组装渠道参数，调渠道支付接口（微信/支付宝）。
5. **用户支付**：用户在渠道侧完成支付（输入密码/指纹/人脸）。
6. **渠道回调**：渠道异步回调支付系统，通知支付结果。
7. **处理回调**：幂等校验 → 更新支付订单状态 → 记账 → 通知订单系统。
8. **查询补偿**：若回调丢失，主动查询渠道支付结果（定时任务）。

### 4️⃣ 幂等性设计（核心）

> **支付系统幂等性至关重要**：同一笔业务只能扣一次钱，网络重试、用户重复点击、渠道重复回调都不能导致重复支付。

#### （1）幂等键设计

| 场景 | 幂等键 | 存储 |
|------|--------|------|
| **创建支付订单** | 业务订单号（order_no） | MySQL 唯一索引 |
| **支付请求** | 支付流水号（trade_no） | MySQL 唯一索引 |
| **渠道回调** | 渠道流水号 + 渠道标识 | MySQL 唯一索引 |
| **退款请求** | 退款流水号（refund_no） | MySQL 唯一索引 |

#### （2）幂等实现流程

```
接收请求 → 获取幂等键 → 查询是否已处理
    │                        │
    │                        ├─ 已处理 → 返回上次结果
    │                        │
    ▼                        ▼
执行业务（扣款/退款） ◄── 未处理 → 加分布式锁 → 执行业务
    │                                            │
    ▼                                            ▼
记录处理结果（幂等表）◄───────────────────────────┘
```

#### （3）代码示例

```java
@Service
public class PaymentService {
    
    @Transactional
    public PayResult pay(PayRequest request) {
        String idempotencyKey = request.getOrderNo();
        
        // 1. 先查幂等表
        PayRecord record = payRecordDao.selectByIdempotencyKey(idempotencyencyKey);
        if (record != null) {
            // 已处理，直接返回结果
            return buildResult(record);
        }
        
        // 2. 加分布式锁（防并发）
        try (AutoCloseableLock lock = redisLock.tryLock("pay:" + idempotencyKey, 10)) {
            // 双检
            record = payRecordDao.selectByIdempotencyKey(idempotencyKey);
            if (record != null) {
                return buildResult(record);
            }
            
            // 3. 执行业务
            record = executePay(request);
            
            // 4. 记录幂等（唯一索引防并发插入）
            payRecordDao.insert(record);
            
            return buildResult(record);
        }
    }
}
```

### 5️⃣ 支付状态机

```
待支付 → 支付中 → 支付成功
   │        │         │
   │        ▼         ▼
   │     支付失败   已退款
   │        │         │
   ▼        ▼         ▼
已关闭（超时/取消）  退款中 → 退款成功
                        │
                        ▼
                     退款失败
```

| 状态 | 说明 | 可流转至 |
|------|------|----------|
| **待支付** | 支付订单创建，等待用户支付 | 支付中、已关闭 |
| **支付中** | 用户已在渠道侧发起支付 | 支付成功、支付失败 |
| **支付成功** | 渠道确认扣款成功 | 已退款 |
| **支付失败** | 渠道返回失败或超时 | 待支付（可重试） |
| **已关闭** | 超时未支付或用户取消 | - |
| **已退款** | 已完成退款 | - |

**状态流转校验**：
- 支付成功只能由支付中流转，不能从待支付直接跳转（防篡改）。
- 每次状态变更前校验当前状态是否合法。

### 6️⃣ 对账系统设计

> **对账**：确保支付系统与渠道（微信/支付宝）的账目一致，发现差异及时处理。

#### （1）对账流程

```
T 日交易                    T+1 日对账
    │                            │
    ▼                            ▼
支付系统记录 ──→ 生成对账文件    渠道对账文件
（订单/流水）    （T 日 24:00）   （T+1 日早上）
                      │              │
                      └──────┬───────┘
                             ▼
                      对账系统拉取两方文件
                             │
                             ▼
                      逐笔比对（流水号、金额、状态）
                             │
              ┌──────────────┼──────────────┐
              ▼              ▼              ▼
           长款（多账）   短款（少账）   金额/状态不一致
              │              │              │
              ▼              ▼              ▼
         支付成功未通知   通知成功未记录   人工核查
         补发通知         调渠道查询       差异处理
```

#### （2）对账差异类型

| 差异类型 | 说明 | 处理 |
|----------|------|------|
| **长款** | 支付系统有，渠道无 | 可能是测试数据，需核查 |
| **短款** | 渠道有，支付系统无 | 补录流水，调渠道查询 |
| **金额不一致** | 双方金额不同 | 人工核查，极少发生 |
| **状态不一致** | 一方成功一方失败 | 以渠道为准，修正状态 |

#### （3）对账表设计

```sql
CREATE TABLE reconciliation (
    id BIGINT PRIMARY KEY,
    channel VARCHAR(32),          -- 渠道（微信/支付宝）
    trade_date DATE,              -- 交易日期
    channel_trade_no VARCHAR(64), -- 渠道流水号
    our_trade_no VARCHAR(64),     -- 我方流水号
    amount DECIMAL(16,2),         -- 金额
    channel_status VARCHAR(16),   -- 渠道状态
    our_status VARCHAR(16),       -- 我方状态
    diff_type VARCHAR(32),        -- 差异类型（长款/短款/一致）
    status VARCHAR(16),           -- 对账状态（待处理/已处理）
    created_at TIMESTAMP
);
```

### 7️⃣ 安全性设计

| 安全措施 | 说明 |
|----------|------|
| **HTTPS/TLS** | 全链路加密传输 |
| **签名验签** | 请求参数签名，防篡改；RSA/SM2 非对称加密 |

| **限流熔断** | 防刷、防 DDoS，保护渠道和自身 |
| **风控拦截** | 异常 IP、设备、行为拦截 |

### 8️⃣ 退款设计

```
用户申请退款 → 校验（是否可退、金额）→ 创建退款单 → 调渠道退款
                                                  │
                    ┌─────────────────────────────┘
                    ▼
              渠道返回结果
                    │
      ┌─────────────┼─────────────┐
      ▼             ▼             ▼
  退款成功      退款失败      处理中
      │             │             │
      ▼             ▼             ▼
  更新订单      可重试        定时查询

```

**注意点**：
- 退款必须**原路退回**（微信付退微信，支付宝付退支付宝）。
- 退款金额不能超过支付金额。
- 退款也**必须幂等**。

### ⚠️ 常见坑与注意点

1. **现象**：没有幂等保护，用户双击导致重复支付。**正确做法**：支付请求必须幂等，用唯一流水号 + 分布式锁 + 唯一索引三重保护。
2. **现象**：渠道回调丢失，订单一直显示待支付。**正确做法**：回调 + 主动查询双保险；定时任务扫描"支付中"状态订单，主动查询渠道结果。
3. **现象**：对账发现差异不知如何处理。**正确做法**：以渠道为准；长款核查、短款补录、状态不一致人工确认；建立差异处理 SOP。
4. **现象**：退款不退手续费。**正确做法**：退款金额计算需考虑渠道手续费规则；部分渠道退款不退手续费，需产品明确。
