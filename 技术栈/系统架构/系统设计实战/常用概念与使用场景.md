# 📌 系统设计实战常用概念与使用场景速查

> 配合《学习笔记》使用，侧重**答题步骤与秒杀、短链、Feed、IM**等场景要点，便于速查与校招面试。

---

## 一、答题步骤（通用）

| 步骤 | 要点 |
|------|------|
| **1. 澄清需求** | 功能范围、用户量/QPS、读写比、延迟与一致性要求 |
| **2. 估算** | QPS、存储、带宽量级（不必精确） |
| **3. 顶层设计** | 画框图：客户端→网关→服务→缓存→DB/MQ；标出数据流 |
| **4. 核心深挖** | 针对瓶颈与核心逻辑（如秒杀防超卖、短链生成与跳转） |
| **5. 扩展优化** | 分库分表、多级缓存、异步、限流降级 |
| **6. 小结** | 方案总结、替代方案与 trade-off |

---

## 二、秒杀

| 环节 | 要点 |
|------|------|
| **前端** | 限流、防重复提交、按钮置灰、静态化页面 |
| **网关** | 全局限流、按 path 限流 |
| **服务** | 多级缓存（商品信息）、Redis 预扣库存 |
| **库存** | Redis 扣减 + MQ 异步写 DB；或 DB 乐观锁 `where stock > 0` 防超卖 |
| **下单** | MQ 消峰、幂等（订单号/用户+商品+场次） |
| **降级** | 非核心关掉、排队页、返回“稍后再试” |

- **防超卖**：Redis 扣减原子（DECR/Lua）；扣成功再发 MQ，消费者扣 DB 库存；或仅 DB 乐观锁。

---

## 三、短链

| 环节 | 要点 |
|------|------|
| **生成** | 发号器（分布式 ID）→ Base62 得短码；或 Hash 取前 N 位（需处理冲突） |
| **存储** | 短码 → 长链：Redis（热）+ DB（持久）；读多写少 |
| **跳转** | GET 短链 → 查 Redis/DB → 302 Location: 长链 |
| **统计** | 异步写 MQ 或日志，再统计 PV/UV；或 Redis 计数+定时落库 |

- **高并发读**：缓存为主、DB 做持久；短码唯一由发号器保证。

---

## 四、点赞与计数

| 场景 | 要点 |
|------|------|
| **点赞** | Redis Set 存“谁赞了”、计数 Redis 或异步汇总；防重复（同一用户同一内容只加一次）；DB 持久与最终一致 |
| **计数** | 高并发下 Redis 计数，定时/批量同步 DB；或 MQ 异步写 |

---

## 五、Feed 流

| 模式 | 思路 | 适用 |
|------|------|------|
| **推（写扩散）** | 发帖时写入每个粉丝的 Feed 收件箱 | 粉丝量不大、读多 |
| **拉（读扩散）** | 读时按关注列表拉取动态再聚合 | 大 V、粉丝多 |
| **推拉结合** | 普通用户推、大 V 拉 | 折中 |

- **存储**：用户时间线表（userId + 动态 ID 列表/流）、收件箱表；按时间排序（时间戳或 ID 有序）。

---

## 六、IM 要点

| 点 | 说明 |
|------|------|
| **长连接** | WebSocket 或 TCP；心跳保活；多实例需路由到同一连接（如 userId 绑定机器） |
| **消息不丢** | 服务端持久化、客户端 ACK、重试与幂等 |
| **离线消息** | 存 DB/缓存，登录后拉取；未读计数 Redis |
| **扩展** | 连接层水平扩展、消息写 MQ 异步落库、状态与会话存 Redis |

---

## 七、缓存与存储选型（速记）

| 需求 | 可选 |
|------|------|
| **高并发读、可接受最终一致** | Redis、多级缓存 |
| **事务、复杂查询** | MySQL 等关系型 |
| **全文/搜索** | Elasticsearch |
| **大表、高写** | 分库分表、读写分离 |
| **消息、削峰** | RabbitMQ、Kafka |

## 八、微博/社交系统

| 环节 | 要点 |
|------|------|
| **Feed 推拉** | 普通用户推（写粉丝收件箱）、大 V 拉（读时聚合）；推拉结合 |
| **发布流程** | 写内容→写发件箱→MQ 推粉丝→更新计数 |
| **查看 Feed** | 收件箱拉取 + 大 V 发件箱聚合 + 时间归并 |
| **热搜** | 搜索词频率统计（Redis ZSet/Flink）、风控防刷榜 |
| **存储** | 发件箱/收件箱/关注关系表，按 user_id 分片 |

---

## 九、电商/订单系统

| 环节 | 要点 |
|------|------|
| **下单流程** | 校验→锁库存→创建订单→支付→扣库存→异步通知 |
| **库存扣减** | DB 乐观锁、Redis 预扣、预扣+超时回滚 |
| **防超卖** | 扣减原子性、幂等、超时回滚 |
| **订单状态机** | 待支付→已支付→已发货→已收货→已完成；已取消 |
| **分布式事务** | 本地消息表、MQ 事务消息、最终一致+幂等 |
| **分库分表** | 订单按 user_id/order_id、商品按商品 ID |

---

## 十、支付系统

| 环节 | 要点 |
|------|------|
| **架构** | 收银台→网关→路由→核心→渠道适配→账务→风控 |
| **核心流程** | 创建订单→风控→路由→调渠道→回调→记账→通知 |
| **幂等性** | 唯一流水号+幂等表+分布式锁+唯一索引；先查再处理 |
| **状态机** | 待支付→支付中→支付成功/失败→已退款 |
| **对账** | T+1 对账、长款/短款/不一致、差异处理 |
| **安全** | HTTPS、签名验签、敏感信息加密、防重放、风控 |
| **退款** | 原路退回、幂等、定时查询 |

## 十、面试一句话总结

- **秒杀**：限流+多级缓存+Redis 扣库存+MQ 异步下单+防超卖（Redis 原子或 DB 乐观锁）。
- **短链**：发号器生成短码→Redis+DB 存映射→跳转 302；读多写少以缓存为主。
- **Feed**：推适合粉丝少、拉适合大 V；推拉结合折中；时间线/收件箱表+时序。
- **IM**：长连接+心跳、消息持久与 ACK、离线拉取、连接层与存储分层扩展。
- **微博**：Feed 推拉结合、热搜频率统计、大 V 用拉模式避免写放大。
- **电商**：下单锁库存+支付扣库存+超时回滚；状态机管理订单；分布式事务最终一致。
- **支付**：分层架构+幂等设计+对账系统+安全合规，资金安全是核心。
- **系统设计**：澄清→估算→画图→核心深挖→扩展与 trade-off，结合分布式与高可用知识。
