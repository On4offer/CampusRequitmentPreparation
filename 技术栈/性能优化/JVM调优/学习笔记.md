## 1.1 JVM内存结构

### 🔹 1.1.1 JVM运行时数据区

> **一句话**：JVM运行时数据区是JVM在运行Java程序时使用的内存区域,分为线程私有和线程共享两部分。
> **为什么需要了解**：理解JVM内存结构是JVM调优的基础,不同区域的内存分配和回收机制不同,需要针对性地优化。
> **类比**：JVM内存结构就像一个工厂,有不同的车间(方法区、堆)和工人的工作台(栈、程序计数器),各司其职。

**JVM运行时数据区**由以下几部分组成：

```
┌─────────────────────────────────────────────────────────┐
│                    JVM运行时数据区                        │
├─────────────────────────────────────────────────────────┤
│  线程私有区                                               │
│  ├─ 程序计数器(Program Counter Register)                 │
│  ├─ Java虚拟机栈(Java Virtual Machine Stack)              │
│  └─ 本地方法栈(Native Method Stack)                       │
├─────────────────────────────────────────────────────────┤
│  线程共享区                                               │
│  ├─ 堆(Heap)                                             │
│  ├─ 方法区(Method Area)                                   │
│  └─ 直接内存(Direct Memory)                               │
└─────────────────────────────────────────────────────────┘
```

### 🔹 1.1.2 程序计数器

**定义**：程序计数器是一块较小的内存空间,可以看作是当前线程所执行的字节码的行号指示器。

**特点**：
- 线程私有,每个线程都有一个独立的程序计数器
- 如果线程正在执行Java方法,计数器记录的是正在执行的虚拟机字节码指令的地址
- 如果线程正在执行Native方法,计数器值为空(Undefined)
- 唯一一个在Java虚拟机规范中没有规定任何OutOfMemoryError情况的区域

**作用**：
- 字节码解释器通过改变程序计数器的值来选取下一条需要执行的字节码指令
- 分支、循环、跳转、异常处理、线程恢复等基础功能都需要依赖程序计数器

```java
public class CounterDemo {
    public static void main(String[] args) {
        int sum = 0;
        for (int i = 0; i < 100; i++) {
            sum += i;
        }
        System.out.println(sum);
    }
}
```

**程序计数器的作用**：
- 记录当前执行到哪一行字节码指令
- 线程切换后能够恢复到正确的执行位置

### 🔹 1.1.3 Java虚拟机栈

**定义**：Java虚拟机栈是Java方法执行的内存模型,每个方法在执行的同时都会创建一个栈帧(Stack Frame)。

**特点**：
- 线程私有,每个线程都有自己的虚拟机栈
- 栈的生命周期与线程相同
- 栈帧用于存储局部变量表、操作数栈、动态链接、方法出口等信息
- 每一个方法从调用直至执行完成的过程,就对应着一个栈帧在虚拟机栈中从入栈到出栈的过程

**栈帧结构**：

```
┌─────────────────────────────────┐
│         栈帧(Stack Frame)       │
├─────────────────────────────────┤
│  局部变量表(Local Variable Table)│
├─────────────────────────────────┤
│  操作数栈(Operand Stack)        │
├─────────────────────────────────┤
│  动态链接(Dynamic Linking)     │
├─────────────────────────────────┤
│  返回地址(Return Address)       │
└─────────────────────────────────┘
```

**局部变量表**：
- 存储方法参数和方法内部定义的局部变量
- 基本数据类型、引用类型、returnAddress类型
- 32位以内的类型占用一个slot,64位的类型占用两个slot

**操作数栈**：
- 后进先出(LIFO)栈
- 方法执行过程中,各种字节码指令往操作数栈中写入和提取数据
- 用于进行算术运算、类型转换、方法调用等

**动态链接**：
- 每个栈帧都包含一个指向运行时常量池中该栈帧所属方法的引用
- 支持方法调用过程中的动态链接

**返回地址**：
- 方法正常退出或异常退出时,需要返回到方法被调用的位置
- 正常退出时,返回地址是调用者的PC计数器的值
- 异常退出时,返回地址通过异常处理表确定

**栈溢出**：

```java
public class StackOverflowDemo {
    private static int depth = 0;

    public static void recursion() {
        depth++;
        recursion();
    }

    public static void main(String[] args) {
        try {
            recursion();
        } catch (Throwable e) {
            System.out.println("递归深度: " + depth);
            e.printStackTrace();
        }
    }
}
```

**异常**：
- `StackOverflowError`：线程请求的栈深度大于JVM所允许的深度
- `OutOfMemoryError`：如果JVM栈可以动态扩展,扩展时无法申请到足够的内存

### 🔹 1.1.4 本地方法栈

**定义**：本地方法栈为虚拟机使用到的Native方法服务。

**特点**：
- 线程私有
- 与Java虚拟机栈类似,区别在于Java虚拟机栈为Java方法服务,本地方法栈为Native方法服务
- HotSpot虚拟机中,Java虚拟机栈和本地方法栈合二为一

**Native方法示例**：

```java
public class NativeMethodDemo {
    // 声明一个本地方法
    private native void nativeMethod();

    // 加载本地库
    static {
        System.loadLibrary("NativeMethodDemo");
    }

    public static void main(String[] args) {
        NativeMethodDemo demo = new NativeMethodDemo();
        demo.nativeMethod();
    }
}
```

### 🔹 1.1.5 堆(Heap)

**定义**：堆是JVM中最大的一块内存区域,被所有线程共享,在虚拟机启动时创建。

**特点**：
- 存放对象实例
- 几乎所有的对象实例都在这里分配内存
- 垃圾收集器管理的主要区域
- 可以处于物理上不连续的内存空间中,只要逻辑上是连续的即可

**堆内存结构**：

```
┌─────────────────────────────────────────────────────────┐
│                      堆(Heap)                          │
├─────────────────────────────────────────────────────────┤
│  新生代(Young Generation)                               │
│  ├─ Eden区(Eden Space)                                  │
│  ├─ Survivor0区(From Survivor Space)                     │
│  └─ Survivor1区(To Survivor Space)                       │
├─────────────────────────────────────────────────────────┤
│  老年代(Old Generation)                                 │
└─────────────────────────────────────────────────────────┘
```

**新生代(Young Generation)**：
- 存放新创建的对象
- 大部分对象都会在Eden区创建
- Eden区满后触发Minor GC
- 存活的对象移动到Survivor区
- 经过多次GC仍然存活的对象移动到老年代

**老年代(Old Generation)**：
- 存放生命周期较长的对象
- 大对象直接进入老年代
- 经过多次GC仍然存活的对象进入老年代
- 老年代满后触发Full GC

**内存分配策略**：

```java
public class MemoryAllocationDemo {
    public static void main(String[] args) {
        byte[] allocation1, allocation2, allocation3, allocation4;
        allocation1 = new byte[2 * _1MB];
        allocation2 = new byte[2 * _1MB];
        allocation3 = new byte[2 * _1MB];
        allocation4 = new byte[4 * _1MB];
    }

    private static final int _1MB = 1024 * 1024;
}
```

**堆内存异常**：
- `OutOfMemoryError: Java heap space`：堆中没有足够的内存完成实例分配,并且无法再扩展

### 🔹 1.1.6 方法区

**定义**：方法区用于存储已被虚拟机加载的类信息、常量、静态变量、即时编译器编译后的代码等数据。

**特点**：
- 线程共享
- 虚拟机启动时创建
- 可以选择固定大小或动态扩展
- 不需要连续的内存空间

**方法区内容**：
- 类信息(类的版本、字段、方法、接口等)
- 常量池
- 静态变量
- 即时编译器编译后的代码

**JDK 8变化**：
- JDK 8之前：永久代(Permanent Generation)
- JDK 8及之后：元空间(Metaspace)
- 元空间使用本地内存,不再使用JVM内存

**方法区异常**：
- `OutOfMemoryError: PermGen space` (JDK 8之前)
- `OutOfMemoryError: Metaspace` (JDK 8及之后)

### 🔹 1.1.7 直接内存

**定义**：直接内存不是JVM运行时数据区的一部分,也不是Java虚拟机规范中定义的内存区域。

**特点**：
- 使用Native函数库直接分配堆外内存
- 通过`ByteBuffer.allocateDirect()`方法分配
- 避免在Java堆和Native堆中来回复制数据,提高性能
- 不受Java堆大小限制,但受本机总内存限制

**直接内存示例**：

```java
public class DirectMemoryDemo {
    public static void main(String[] args) {
        // 分配直接内存
        ByteBuffer buffer = ByteBuffer.allocateDirect(1024 * 1024);
        
        // 使用直接内存
        buffer.put((byte) 1);
        buffer.flip();
        System.out.println(buffer.get());
    }
}
```

**直接内存异常**：
- `OutOfMemoryError: Direct buffer memory`

## 1.2 垃圾回收基础

### 🔹 1.2.1 垃圾回收概述

> **一句话**：垃圾回收(Garbage Collection, GC)是自动内存管理机制,自动回收不再使用的对象,避免内存泄漏和内存溢出。
> **为什么需要GC**：手动管理内存容易出错,忘记释放内存会导致内存泄漏,重复释放会导致程序崩溃。GC自动管理内存,提高开发效率和程序稳定性。
> **类比**：GC就像一个清洁工,定期打扫房间,清理不再使用的物品,保持房间整洁。

**GC的作用**：
- 自动回收不再使用的对象
- 避免内存泄漏
- 防止内存溢出
- 提高开发效率

**GC要解决的问题**：
- 哪些内存需要回收？
- 什么时候回收？
- 如何回收？

### 🔹 1.2.2 对象存活判定

**引用计数法(Reference Counting)**：

```java
public class ReferenceCountingDemo {
    private Object instance = null;
    private static final int _1MB = 1024 * 1024;

    private byte[] bigSize = new byte[2 * _1MB];

    public static void main(String[] args) {
        ReferenceCountingDemo a = new ReferenceCountingDemo();
        ReferenceCountingDemo b = new ReferenceCountingDemo();
        a.instance = b;
        b.instance = a;
        
        a = null;
        b = null;
        
        System.gc();
    }
}
```

**原理**：
- 给对象添加一个引用计数器
- 当对象被引用时,计数器加1
- 当引用失效时,计数器减1
- 计数器为0的对象就是不再使用的对象

**优点**：
- 实现简单
- 判定效率高

**缺点**：
- 无法解决循环引用问题
- 需要额外的空间存储引用计数

**可达性分析算法(Reachability Analysis)**：

```
GC Roots
  ├─ 虚拟机栈中引用的对象
  ├─ 方法区中类静态属性引用的对象
  ├─ 方法区中常量引用的对象
  ├─ 本地方法栈中JNI引用的对象
  └─ Java虚拟机内部的引用
```

**原理**：
- 从GC Roots开始向下搜索
- 搜索走过的路径称为引用链
- 如果一个对象到GC Roots没有任何引用链相连,则证明该对象是不可用的
- 可作为GC Roots的对象：
  - 虚拟机栈中引用的对象
  - 方法区中类静态属性引用的对象
  - 方法区中常量引用的对象
  - 本地方法栈中JNI引用的对象

**可达性分析示例**：

```java
public class ReachabilityAnalysisDemo {
    public static void main(String[] args) {
        // GC Root
        Object root = new Object();
        
        // 可达对象
        Object reachable = new Object();
        root = reachable;
        
        // 不可达对象
        Object unreachable = new Object();
        unreachable = null;
        
        System.gc();
    }
}
```

### 🔹 1.2.3 引用类型

**强引用(Strong Reference)**：

```java
Object obj = new Object();
```

- 最常见的引用类型
- 只要强引用存在,垃圾收集器永远不会回收被引用的对象

**软引用(Soft Reference)**：

```java
Object obj = new Object();
SoftReference<Object> softRef = new SoftReference<>(obj);
obj = null;
```

- 内存足够时,垃圾收集器不会回收软引用对象
- 内存不足时,垃圾收集器会回收软引用对象
- 适用于缓存场景

**弱引用(Weak Reference)**：

```java
Object obj = new Object();
WeakReference<Object> weakRef = new WeakReference<>(obj);
obj = null;
```

- 无论内存是否充足,垃圾收集器都会回收弱引用对象
- 适用于缓存场景

**虚引用(Phantom Reference)**：

```java
Object obj = new Object();
ReferenceQueue<Object> queue = new ReferenceQueue<>();
PhantomReference<Object> phantomRef = new PhantomReference<>(obj, queue);
obj = null;
```

- 无法通过虚引用获取对象实例
- 虚引用的唯一目的是在对象被回收时收到系统通知
- 用于跟踪对象被垃圾回收的活动

**引用类型对比**：

| 引用类型 | 被GC回收时间 | 用途 | 生存时间 |
|----------|--------------|------|----------|
| **强引用** | 永不回收 | 通用对象 | JVM停止运行 |
| **软引用** | 内存不足时 | 缓存 | 内存不足时 |
| **弱引用** | 下次GC时 | 缓存 | 下次GC时 |
| **虚引用** | 任何时候 | 跟踪GC | 任何时候 |

### 🔹 1.2.4 垃圾回收算法

**标记-清除算法(Mark-Sweep)**：

```
标记阶段:
  ┌───┬───┬───┬───┐
  │ A │ B │ C │ D │
  └───┴───┴───┴───┘
  ✓   ✓   ✗   ✓

清除阶段:
  ┌───┬───┬───┬───┐
  │ A │ B │   │ D │
  └───┴───┴───┴───┘
```

**过程**：
1. 标记：从GC Roots开始,标记所有可达对象
2. 清除：清除所有未标记的对象

**优点**：
- 实现简单
- 不需要移动对象

**缺点**：
- 效率不高(标记和清除两个过程都需要遍历所有对象)
- 会产生大量不连续的内存碎片

**复制算法(Copying)**：

```
复制前:
  ┌───┬───┬───┬───┐
  │ A │ B │ C │ D │
  └───┴───┴───┴───┘
  ✓   ✓   ✗   ✓

复制后:
  ┌───┬───┬───┬───┐
  │ A │ B │ D │   │
  └───┴───┴───┴───┘
```

**过程**：
1. 将内存分为两块,每次只使用其中一块
2. GC时,将存活的对象复制到另一块内存
3. 清空当前使用的内存

**优点**：
- 没有内存碎片
- 实现简单,运行高效

**缺点**：
- 内存利用率低(只有一半的内存可用)
- 对象存活率高时,复制效率低

**标记-整理算法(Mark-Compact)**：

```
标记阶段:
  ┌───┬───┬───┬───┐
  │ A │ B │ C │ D │
  └───┴───┴───┴───┘
  ✓   ✓   ✗   ✓

整理阶段:
  ┌───┬───┬───┬───┐
  │ A │ B │ D │   │
  └───┴───┴───┴───┘
```

**过程**：
1. 标记：从GC Roots开始,标记所有可达对象
2. 整理：将所有存活对象向一端移动,清除端边界的内存

**优点**：
- 没有内存碎片
- 内存利用率高

**缺点**：
- 需要移动对象,效率较低

**分代收集算法(Generational Collection)**：

```
┌─────────────────────────────────────────────────────────┐
│                      堆(Heap)                          │
├─────────────────────────────────────────────────────────┤
│  新生代(Young Generation)                               │
│  ├─ Eden区(复制算法)                                    │
│  ├─ Survivor0区(复制算法)                                │
│  └─ Survivor1区(复制算法)                                │
├─────────────────────────────────────────────────────────┤
│  老年代(Old Generation)                                 │
│  └─ 标记-整理算法                                       │
└─────────────────────────────────────────────────────────┘
```

**原理**：
- 根据对象存活周期的不同,将内存划分为几块
- 新生代：对象存活时间短,使用复制算法
- 老年代：对象存活时间长,使用标记-整理算法

**优点**：
- 针对不同特点的对象使用不同的收集算法
- 提高GC效率

### 🔹 1.2.5 GC类型

**Minor GC**：
- 发生在新生代
- 触发条件：Eden区满
- 速度快
- 复制算法

**Major GC**：
- 发生在老年代
- 触发条件：老年代满
- 速度慢
- 标记-整理算法

**Full GC**：
- 整个堆的GC,包括新生代和老年代
- 触发条件：
  - 老年代满
  - 方法区满
  - 显式调用System.gc()
- 速度最慢
- 应该尽量避免

**GC类型对比**：

| GC类型 | 发生区域 | 触发条件 | 速度 | 算法 |
|--------|----------|----------|------|------|
| **Minor GC** | 新生代 | Eden区满 | 快 | 复制算法 |
| **Major GC** | 老年代 | 老年代满 | 慢 | 标记-整理算法 |
| **Full GC** | 整个堆 | 老年代满、方法区满、System.gc() | 最慢 | 标记-整理算法 |

## 1.3 常见GC算法

### 🔹 1.3.1 Serial GC

**特点**：
- 单线程GC
- 新生代和老年代都是单线程
- 适合小内存应用
- 简单高效

**工作原理**：
- 新生代：复制算法
- 老年代：标记-整理算法

**配置参数**：

```bash
-XX:+UseSerialGC
```

**适用场景**：
- 客户端应用
- 小型Web应用
- 内存<2GB

**优点**：
- 实现简单
- 没有线程交互开销
- 单核CPU效率高

**缺点**：
- 多核CPU利用率低
- GC停顿时间长

**GC日志示例**：

```
[GC (Allocation Failure) [DefNew: 62912K->9520K(62912K), 0.0123456 secs] 62912K->9520K(125952K), 0.0123456 secs] [Times: user=0.01 sys=0.00, real=0.01 secs]
```

### 🔹 1.3.2 Parallel GC

**特点**：
- 多线程GC
- 新生代和老年代都是多线程
- 吞吐量高
- 适合计算密集型应用

**工作原理**：
- 新生代：多线程复制算法
- 老年代：多线程标记-整理算法

**配置参数**：

```bash
-XX:+UseParallelGC
-XX:ParallelGCThreads=8
```

**适用场景**：
- 批处理应用
- 后台任务
- 计算密集型应用

**优点**：
- 多核CPU利用率高
- 吞吐量高

**缺点**：
- GC停顿时间长
- 对响应时间敏感的应用不友好

**GC日志示例**：

```
[GC (Allocation Failure) [PSYoungGen: 62912K->9520K(62912K)] 62912K->9520K(125952K), 0.0123456 secs] [Times: user=0.03 sys=0.00, real=0.01 secs]
```

### 🔹 1.3.3 CMS GC

**特点**：
- 低停顿
- 并发收集
- 适合响应时间敏感的应用
- JDK 9标记废弃,JDK 14移除

**工作原理**：
- 新生代：多线程复制算法
- 老年代：并发标记-清除算法

**CMS收集过程**：

```
1. 初始标记(STW)
   ↓
2. 并发标记
   ↓
3. 重新标记(STW)
   ↓
4. 并发清除
```

**配置参数**：

```bash
-XX:+UseConcMarkSweepGC
-XX:+UseCMSInitiatingOccupancyOnly
-XX:CMSInitiatingOccupancyFraction=70
```

**适用场景**：
- Web应用
- 响应时间敏感的应用
- 内存<8GB

**优点**：
- 低停顿
- 并发收集,对应用影响小

**缺点**：
- 对CPU资源敏感
- 产生内存碎片
- 无法处理浮动垃圾
- JDK 9标记废弃,JDK 14移除

**GC日志示例**：

```

[CMS-concurrent-preclean-start]

```

### 🔹 1.3.4 G1 GC

**特点**：
- 可预测停顿
- 分代收集
- 适合大内存应用
- JDK 9+默认GC

**工作原理**：
- 堆被划分为多个Region
- 新生代和老年代都是Region的集合
- 跟踪每个Region的垃圾数量
- 优先回收垃圾最多的Region

**G1 Region结构**：

```
┌─────────────────────────────────────────────────────────┐
│                      G1 Heap                          │
├─────────────────────────────────────────────────────────┤
│  Region 1 (Eden)    Region 2 (Eden)    Region 3 (Eden) │
│  Region 4 (Eden)    Region 5 (Survivor) Region 6 (Old) │
│  Region 7 (Old)     Region 8 (Old)     Region 9 (Old) │
└─────────────────────────────────────────────────────────┘
```

**G1 GC过程**：

```
1. 初始标记(STW)
   ↓
2. 并发标记
   ↓
3. 最终标记(STW)
   ↓
4. 筛选回收(STW)
```

**配置参数**：

```bash
-XX:+UseG1GC
-XX:MaxGCPauseMillis=200
-XX:G1HeapRegionSize=16m
```

**适用场景**：
- 大内存应用(>6GB)
- 需要可控停顿的应用
- JDK 9+默认GC

**优点**：
- 可预测停顿
- 分代收集
- 不会产生内存碎片
- 适合大内存应用

**缺点**：
- 内存占用较高
- 小内存应用性能不如Parallel GC

**GC日志示例**：

### 🔹 1.3.5 ZGC

**特点**：
- 极低停顿(<10ms)
- 支持TB级内存
- 并发收集
- JDK 15+生产可用

**工作原理**：
- 基于Region的内存布局
- 读屏障、染色指针、负载屏障
- 并发标记、并发整理、并发重定位

**配置参数**：

```bash
-XX:+UseZGC
```

**适用场景**：
- 超大内存应用(>16GB)
- 对停顿时间要求极高的应用
- JDK 15+生产可用

**优点**：
- 极低停顿(<10ms)
- 支持TB级内存
- 并发收集,对应用影响小

**缺点**：
- 内存占用较高
- JDK 15+才生产可用
- 小内存应用性能不如G1 GC

**GC日志示例**：


### 🔹 1.3.6 GC算法对比

| GC算法 | 特点 | 适用场景 | 停顿时间 | JDK版本 |
|--------|------|----------|----------|---------|
| **Serial GC** | 单线程,简单 | 小内存应用 | 长 | 所有版本 |
| **Parallel GC** | 多线程,吞吐量高 | 计算密集型 | 中 | 所有版本 |
| **CMS GC** | 低停顿,并发收集 | 响应时间敏感 | 短 | JDK 8及之前 |
| **G1 GC** | 可预测停顿,分代收集 | 大内存应用 | 可控 | JDK 7+ |
| **ZGC** | 极低停顿,支持TB级内存 | 超大内存应用 | 极短(<10ms) | JDK 15+ |

**选择建议**：
- 小内存(<2GB)：Serial GC
- 计算密集型：Parallel GC
- 响应时间敏感：G1 GC
- 大内存(>6GB)：G1 GC
- 超大内存(>16GB)：ZGC

## 2.1 JVM参数分类

## 2.2 堆内存调优

### 🔹 2.2.1 堆内存大小设置

**基础参数**：

```bash
-Xms<size>    # 初始堆内存大小
-Xmx<size>    # 最大堆内存大小
```

**设置原则**：

```
-Xms = -Xmx
避免运行时动态调整堆内存大小,提高性能
```

**示例**：

```bash
java -Xms4g -Xmx4g com.example.Main
```

**堆内存大小计算**：

```
堆内存大小 = (存活对象大小 * 3) + 预留空间

示例:
  - 存活对象大小: 1GB
  - 堆内存大小: 1GB * 3 = 3GB
  - 预留空间: 3GB * 0.2 = 0.6GB
  - 总堆内存: 3.6GB
```

**实战计算**：

```bash
# 1. 观察应用运行情况
jstat -gc pid 1000 10

# 2. 分析GC日志
# 老年代使用率稳定在70%左右

# 3. 计算堆内存大小
# 老年代: 2GB
# 新生代: 1GB
# 总堆内存: 3GB

# 4. 配置JVM参数
java -Xms3g -Xmx3g com.example.Main
```

### 🔹 2.2.2 新生代调优

**基础参数**：

```bash
-Xmn<size>          # 新生代大小
-XX:NewRatio=<n>    # 新生代:老年代 = 1:n
-XX:SurvivorRatio=<n>  # Eden:Survivor = n:1
```

**设置原则**：

```
新生代大小 = 存活对象大小 * 2

Survivor区大小 = Eden区大小 / 8
```

**示例**：

```bash
# 方式1: 直接设置新生代大小
java -Xms4g -Xmx4g -Xmn2g com.example.Main

# 方式2: 通过比例设置
java -Xms4g -Xmx4g -XX:NewRatio=2 com.example.Main

# 设置Survivor区比例
java -Xms4g -Xmx4g -Xmn2g -XX:SurvivorRatio=8 com.example.Main
```

**验证**：

```bash
# 观察GC频率
jstat -gc pid 1000 10

# 理想情况:
# - Minor GC频率: 每分钟1-2次
# - Survivor区使用率: 50%-80%
```

### 🔹 2.2.3 老年代调优

**基础参数**：

```bash
-XX:NewRatio=<n>    # 新生代:老年代 = 1:n
-XX:MaxTenuringThreshold=<n>  # 对象晋升老年代的年龄阈值
```

**设置原则**：

```
老年代大小 = 长期存活对象大小 * 1.5

老年代使用率: 70%-80%
```

**示例**：

```bash
# 设置新生代:老年代 = 1:3
java -Xms4g -Xmx4g -XX:NewRatio=3 com.example.Main

# 设置对象晋升老年代的年龄阈值
java -Xms4g -Xmx4g -XX:MaxTenuringThreshold=15 com.example.Main
```

**验证**：

```bash
# 观察Full GC频率
jstat -gc pid 1000 10

# 理想情况:
# - Full GC频率: 每天或更少
# - 老年代使用率: 70%-80%
```

### 🔹 2.2.4 堆内存调优实战

**场景1: 频繁Minor GC**

```bash
# 问题: Minor GC频繁,每次GC时间短
# 原因: 新生代太小

# 优化前
java -Xms2g -Xmx2g -Xmn512m com.example.Main

# 优化后
java -Xms4g -Xmx4g -Xmn2g -XX:SurvivorRatio=8 com.example.Main
```

**场景2: 频繁Full GC**

```bash
# 问题: Full GC频繁,每次GC时间长
# 原因: 老年代太小或内存泄漏

# 优化前
java -Xms2g -Xmx2g -Xmn1g com.example.Main

# 优化后
java -Xms4g -Xmx4g -Xmn1g -XX:NewRatio=3 com.example.Main

# 如果是内存泄漏,需要排查内存泄漏
```

**场景3: GC停顿时间长**

```bash
# 问题: GC停顿时间长,影响用户体验
# 原因: 堆内存太大或GC算法不合适

# 优化前
java -Xms8g -Xmx8g -XX:+UseParallelGC com.example.Main

# 优化后
java -Xms8g -Xmx8g -XX:+UseG1GC -XX:MaxGCPauseMillis=200 com.example.Main
```

## 2.3 GC算法调优

### 🔹 2.3.1 GC算法选择

**选择原则**：

```
小内存(<2GB): Serial GC
计算密集型: Parallel GC
响应时间敏感: G1 GC
大内存(>6GB): G1 GC
超大内存(>16GB): ZGC
```

**配置示例**：

```bash
# Serial GC
java -XX:+UseSerialGC com.example.Main

# Parallel GC
java -XX:+UseParallelGC com.example.Main

# G1 GC
java -XX:+UseG1GC com.example.Main

# ZGC
java -XX:+UseZGC com.example.Main
```

### 🔹 2.3.2 G1 GC调优

**基础参数**：

```bash
-XX:+UseG1GC                    # 使用G1 GC
-XX:MaxGCPauseMillis=<n>        # 最大GC停顿时间
-XX:G1HeapRegionSize=<size>      # G1 Region大小
-XX:InitiatingHeapOccupancyPercent=<n>  # 触发并发标记的堆占用百分比
```

**示例**：

```bash
java -XX:+UseG1GC \
     -XX:MaxGCPauseMillis=200 \
     -XX:G1HeapRegionSize=16m \
     -XX:InitiatingHeapOccupancyPercent=45 \
     com.example.Main
```

**调优建议**：

```
- MaxGCPauseMillis: 根据业务需求设置,通常200ms-500ms
- G1HeapRegionSize: 根据堆内存大小设置,通常16m-32m
- InitiatingHeapOccupancyPercent: 根据应用特点设置,通常40%-50%
```

### 🔹 2.3.3 ZGC调优

**基础参数**：

```bash
-XX:+UseZGC                      # 使用ZGC
-XX:ZCollectionInterval=<n>       # GC间隔时间
```

**示例**：

```bash
java -XX:+UseZGC -XX:ZCollectionInterval=5 com.example.Main
```

**调优建议**：

```
- ZCollectionInterval: 根据业务需求设置,通常5-10秒
- ZGC会自动调整GC频率,一般不需要手动调优
```

### 🔹 2.3.4 GC算法调优实战

**场景1: Web应用调优**

```bash
# 应用特点:
# - 大量短生命周期对象
# - 响应时间敏感
# - 并发量大

# JVM参数配置
java -Xms4g -Xmx4g \
     -Xmn2g -XX:SurvivorRatio=8 \
     -XX:+UseG1GC -XX:MaxGCPauseMillis=200 \
     -Xlog:gc*:file=gc.log:time,uptime:filecount=5,filesize=10m \
     com.example.Main
```

**场景2: 批处理应用调优**

```bash
# 应用特点:
# - 大量长生命周期对象
# - 吞吐量优先
# - 响应时间不敏感

# JVM参数配置
java -Xms8g -Xmx8g \
     -Xmn2g -XX:SurvivorRatio=8 \
     -XX:+UseParallelGC -XX:ParallelGCThreads=8 \
     -Xlog:gc*:file=gc.log:time,uptime:filecount=5,filesize=10m \
     com.example.Main
```

**场景3: 微服务调优**

```bash
# 应用特点:
# - 内存占用小
# - 启动速度快
# - 容器化部署

# JVM参数配置
java -Xms1g -Xmx1g \
     -Xmn512m -XX:SurvivorRatio=8 \
     -XX:+UseG1GC -XX:MaxGCPauseMillis=100 \
     -XX:+UseContainerSupport \
     -XX:MaxRAMPercentage=75.0 \
     -Xlog:gc*:file=gc.log:time,uptime:filecount=5,filesize=10m \
     com.example.Main
```

## 3.1 JVM监控指标

### 🔹 3.1.1 内存监控指标

**堆内存使用情况**：

```
堆内存使用率 = 已使用堆内存 / 堆内存总大小

正常范围:
- 新生代使用率: 50%-80%
- 老年代使用率: 70%-80%
- 元空间使用率: 60%-80%
```

**监控命令**：

```bash
# 查看堆内存使用情况
jstat -gc pid 1000 10

# 输出示例:
#  S0C    S1C    S0U    S1U      EC       EU       OC         OU       MC     MU    CCSC   CCSU   YGC     YGCT    FGC    FGCT     GCT
# 5120.0 5120.0  0.0   4096.0 40960.0  20480.0  81920.0    57344.0  5120.0 4096.0 512.0  256.0   10      0.234   1      0.123    0.357

# 字段说明:
# S0C/S1C: Survivor0/Survivor1容量(KB)
# S0U/S1U: Survivor0/Survivor1使用量(KB)
# EC: Eden区容量(KB)
# EU: Eden区使用量(KB)
# OC: 老年代容量(KB)
# OU: 老年代使用量(KB)
# MC: 元空间容量(KB)
# MU: 元空间使用量(KB)
# YGC: Young GC次数
# YGCT: Young GC总时间(s)
# FGC: Full GC次数
# FGCT: Full GC总时间(s)
# GCT: GC总时间(s)
```

**监控示例**：

```java
public class MemoryMonitor {
    public static void main(String[] args) {
        Runtime runtime = Runtime.getRuntime();
        
        long totalMemory = runtime.totalMemory();
        long freeMemory = runtime.freeMemory();
        long maxMemory = runtime.maxMemory();
        long usedMemory = totalMemory - freeMemory;
        
        System.out.println("总内存: " + totalMemory / 1024 / 1024 + "MB");
        System.out.println("已用内存: " + usedMemory / 1024 / 1024 + "MB");
        System.out.println("空闲内存: " + freeMemory / 1024 / 1024 + "MB");
        System.out.println("最大内存: " + maxMemory / 1024 / 1024 + "MB");
        System.out.println("内存使用率: " + (usedMemory * 100.0 / totalMemory) + "%");
    }
}
```

## 4.1 内存泄漏排查

### 🔹 4.1.1 内存泄漏概述

> **一句话**：内存泄漏是指程序中已动态分配的堆内存由于某种原因程序未释放或无法释放,造成系统内存的浪费,导致程序运行速度减慢甚至系统崩溃等严重后果。
> **为什么需要排查**：内存泄漏会导致内存使用率持续增长,最终导致OOM,影响系统稳定性。
> **类比**：内存泄漏就像家里的水龙头没关紧,水一直在流,最终会淹没整个房间。

**内存泄漏原因**：

```
1. 静态集合类
   - 静态集合类持有对象引用
   - 对象无法被GC回收

2. 未关闭的资源
   - 数据库连接未关闭
   - 文件流未关闭
   - 网络连接未关闭

3. 监听器未移除
   - 监听器注册后未移除
   - 监听器持有对象引用

4. ThreadLocal未清理
   - ThreadLocal未清理
   - ThreadLocal持有对象引用

5. 缓存未清理
   - 缓存未设置过期时间
   - 缓存对象无法被GC回收

6. 内部类持有外部类引用
   - 内部类持有外部类引用
   - 外部类无法被GC回收
```

```java
    }
```

- 导致内存泄漏

```java
public class StaticCollectionLeak {

        list.remove(obj);
    }

    public void clear() {
        list.clear();
    }

    public static void main(String[] args) {
        StaticCollectionLeak leak = new StaticCollectionLeak();
        for (int i = 0; i < 1000000; i++) {
            Object obj = new Object();
            leak.addObject(obj);
            leak.removeObject(obj);
        }
        leak.clear();
    }
}
```

## 4.2 内存溢出排查

### 🔹 4.2.1 内存溢出概述

> **一句话**：内存溢出(OutOfMemoryError)是指程序在申请内存时,没有足够的内存空间供其使用,导致程序崩溃。
> **为什么需要排查**：内存溢出会导致程序崩溃,影响系统稳定性。
> **类比**：内存溢出就像家里的水桶满了,再加水就会溢出来。

**内存溢出类型**：

```
1. Java heap space
   - 堆内存不足
   - 原因: 堆内存太小、内存泄漏、大对象

2. Metaspace
   - 元空间不足
   - 原因: 元空间太小、类加载过多

3. Direct buffer memory
   - 直接内存不足
   - 原因: 直接内存太小、NIO使用过多

4. StackOverflowError
   - 栈内存不足
   - 原因: 递归深度过深、栈帧过大

5. GC overhead limit exceeded
   - GC时间过长
   - 原因: 内存泄漏、堆内存太小
```

## 4.3 GC问题排查

### 🔹 4.3.1 GC问题概述

> **一句话**：GC问题是指GC频繁、GC停顿时间长、GC效率低等问题,影响应用性能。
> **为什么需要排查**：GC问题会导致应用性能下降,影响用户体验。
> **类比**：GC问题就像清洁工打扫房间太频繁或打扫时间太长,影响正常生活。

**GC问题类型**：

```
1. GC频繁
   - Minor GC频繁
   - Full GC频繁
   - 原因: 堆内存太小、内存泄漏、对象创建过多

2. GC停顿时间长
   - Minor GC停顿时间长
   - Full GC停顿时间长
   - 原因: 堆内存太大、GC算法不合适

3. GC效率低
   - GC回收效率低
   - 原因: 对象存活率高、内存碎片多
```

## 4.4 线程问题排查

### 🔹 4.4.1 线程问题概述

> **一句话**：线程问题是指线程数过多、线程阻塞、死锁等问题,影响应用性能。
> **为什么需要排查**：线程问题会导致应用性能下降,甚至导致应用崩溃。
> **类比**：线程问题就像工厂里的工人太多或工人都在等待,导致生产效率下降。

**线程问题类型**：

```
1. 线程数过多
   - 线程数持续增长
   - 原因: 线程泄漏、线程池配置不当

2. 线程阻塞
   - 大量线程处于BLOCKED状态
   - 原因: 锁竞争、IO阻塞

3. 死锁
   - 线程互相等待
   - 原因: 锁顺序不当
```

### 🔹 4.4.2 线程问题排查步骤

**第一步: 查看线程数量**

```bash
# 1. 查看线程数量
jstack pid | grep "java.lang.Thread.State" | wc -l

watch -n 1 "jstack pid | grep 'java.lang.Thread.State' | wc -l"

# 3. 观察线程数量是否持续增长
# 如果线程数持续增长,可能存在线程泄漏
```

**第二步: 查看线程状态**

```bash
# 1. 导出线程转储文件
jstack pid > thread_dump.txt

# 2. 查看线程状态
grep "java.lang.Thread.State" thread_dump.txt | sort | uniq -c

# 3. 观察线程状态是否正常
# - 如果大量线程处于BLOCKED状态,可能存在锁竞争
# - 如果大量线程处于WAITING状态,可能存在线程阻塞
```

**第三步: 分析线程问题**

```bash

# 3. 分析线程问题
# - 根据线程堆栈定位到代码位置
# - 分析为什么线程阻塞
# - 修复线程问题
```

```bash
# - 关闭线程池
```

```java
    static {
        for (int i = 0; i < locks.length; i++) {
            locks[i] = new Object();
        }
    }

    public static void main(String[] args) {
        for (int i = 0; i < 1000; i++) {
            final int index = i;
            new Thread(() -> {
                synchronized (locks[index % locks.length]) {
                    try {
                        Thread.sleep(1000000);
                    } catch (InterruptedException e) {
                        e.printStackTrace();
                    }
                }
            }).start();
        }
    }
}
```

## 6.1 JVM调优原则

### 🔹 6.1.1 调优原则

**原则1: 不要过早优化**

```
过早优化是万恶之源

不要在开发阶段就进行过度的JVM调优
应该先保证功能正确,再进行性能优化
应该基于实际性能问题进行优化,而不是凭感觉优化
```

**原则2: 建立性能基线**

```
没有基线,就没有对比

在优化前,先建立性能基线
记录关键性能指标: 响应时间、吞吐量、GC频率、GC停顿时间
优化后,对比基线,验证优化效果
```

**原则3: 小步快跑**

```
不要一次性做太多改动

每次只调整一个参数
每次改动后,验证效果
如果效果不好,立即回滚
```

**原则4: 持续监控**

```
JVM调优是一个持续的过程

建立监控体系
持续监控系统性能
及时发现和解决问题
```

**原则5: 数据驱动**

```
不要凭感觉调优

基于监控数据进行调优
基于GC日志进行分析
基于堆转储文件进行分析
```

### 🔹 6.1.2 调优流程

**标准调优流程**：

```
1. 建立基线
   - 记录当前性能指标
   - 建立监控体系
   - 确定优化目标

2. 识别瓶颈
   - 分析监控数据
   - 分析GC日志
   - 分析堆转储文件
   - 定位性能瓶颈

3. 制定方案
   - 根据瓶颈制定优化方案
   - 评估优化成本和收益
   - 制定实施计划
   - 准备回滚方案

4. 实施优化
   - 按计划实施优化
   - 小步快跑,逐步优化
   - 每次改动后验证效果

5. 验证效果
   - 对比优化前后的性能指标
   - 验证是否达到优化目标
   - 收集用户反馈

6. 持续优化
   - 建立新的性能基线
   - 持续监控系统性能
   - 定期进行性能评估
```

### 🔹 6.1.3 调优检查清单

**JVM参数检查清单**：

```
□ 堆内存大小是否合理
  - -Xms和-Xmx是否设置相同
  - 堆内存大小是否满足应用需求
  - 是否有足够的预留空间

□ 新生代大小是否合理
  - -Xmn或-XX:NewRatio是否设置合理
  - Survivor区大小是否合理
  - 对象晋升年龄是否合理

□ GC算法是否合适
  - 是否根据应用特点选择了合适的GC算法
  - GC参数是否设置合理
  - GC停顿时间是否满足要求

□ 元空间大小是否合理
  - -XX:MetaspaceSize和-XX:MaxMetaspaceSize是否设置合理
  - 是否有足够的预留空间

□ 线程栈大小是否合理
  - -Xss是否设置合理
  - 是否存在栈溢出

□ 直接内存大小是否合理
  - -XX:MaxDirectMemorySize是否设置合理
  - 是否存在直接内存溢出

□ GC日志是否开启
  - 是否开启了GC日志
  - GC日志格式是否正确
  - GC日志文件大小是否合理

□ OOM处理是否配置
  - 是否开启了HeapDumpOnOutOfMemoryError
  - 堆转储文件路径是否设置
```

**代码优化检查清单**：

```
□ 是否存在内存泄漏
  - 静态集合类是否清理
  - 资源是否关闭
  - 监听器是否移除
  - ThreadLocal是否清理
  - 缓存是否设置过期时间

□ 是否存在大对象
  - 大对象是否可以优化
  - 是否可以使用对象池
  - 是否可以使用流式处理

□ 是否存在频繁的对象创建
  - 是否可以使用对象池
  - 是否可以使用缓存
  - 是否可以复用对象

□ 是否存在不必要的对象引用
  - 是否可以及时释放引用
  - 是否可以使用弱引用
  - 是否可以使用软引用
```

## 6.2 不同场景下的JVM配置模板

### 🔹 6.2.1 电商系统配置模板

## 6.3 JVM调优常见误区

### 🔹 6.3.1 常见误区

**误区1: 堆内存越大越好**

```
错误观点: 堆内存越大,性能越好

实际情况:
- 堆内存越大,GC停顿时间越长
- 堆内存越大,GC频率越低
- 需要根据应用特点选择合适的堆内存大小

正确做法:
- 根据应用内存需求设置堆内存大小
- 监控堆内存使用情况
- 根据监控数据调整堆内存大小
```

**误区2: GC停顿时间越短越好**

```
错误观点: GC停顿时间越短,性能越好

实际情况:
- GC停顿时间越短,吞吐量越低
- 需要根据应用特点选择合适的GC停顿时间
- 响应时间敏感的应用需要短的GC停顿时间
- 吞吐量优先的应用可以接受较长的GC停顿时间

正确做法:
- 根据应用特点选择合适的GC停顿时间
- 平衡吞吐量和停顿时间
- 监控GC情况,根据监控数据调整
```

**误区3: 新生代越大越好**

```
错误观点: 新生代越大,Minor GC越少,性能越好

实际情况:
- 新生代越大,老年代越小
- 新生代越大,Minor GC越少,但每次GC时间越长
- 需要根据应用特点选择合适的新生代大小

正确做法:
- 根据应用对象生命周期特点设置新生代大小
- 监控GC情况,根据监控数据调整
- 平衡新生代和老年代的大小
```

**误区4: G1 GC一定比Parallel GC好**

```
错误观点: G1 GC是最先进的GC算法,一定比Parallel GC好

实际情况:
- G1 GC适合响应时间敏感的应用
- Parallel GC适合吞吐量优先的应用
- 需要根据应用特点选择合适的GC算法

正确做法:
- 根据应用特点选择合适的GC算法
- 测试不同GC算法的性能
- 选择最适合应用的GC算法
```

**误区5: 只调整JVM参数,不优化代码**

```
错误观点: JVM调优就是调整JVM参数

实际情况:
- JVM参数优化只能解决部分性能问题
- 代码优化往往能带来更大的性能提升
- JVM参数优化和代码优化要结合

正确做法:
- 先优化代码,再调整JVM参数
- JVM参数优化和代码优化要结合
- 持续监控和优化
```

### 🔹 6.3.2 避免误区的方法

**方法1: 建立性能基线**

```
在优化前,先建立性能基线
记录关键性能指标
优化后,对比基线,验证优化效果
```

**方法2: 小步快跑**

```
每次只调整一个参数
每次改动后,验证效果
如果效果不好,立即回滚
```

**方法3: 持续监控**

```
建立监控体系
持续监控系统性能
及时发现和解决问题
```

**方法4: 数据驱动**

```
基于监控数据进行调优
基于GC日志进行分析
基于堆转储文件进行分析
```

**方法5: 学习和总结**

```
学习JVM调优的最佳实践
总结调优经验
避免重复踩坑
```
