## 5.1 批量操作优化

### 🔹 5.1.1 批量插入优化

**问题**：

```sql
-- 慢查询：逐条插入
INSERT INTO user (name, age) VALUES ('张三', 20);
INSERT INTO user (name, age) VALUES ('李四', 21);
INSERT INTO user (name, age) VALUES ('王五', 22);
-- ... 10000条

-- 执行时间：100秒
-- 问题：
-- 1. 每次插入都需要网络往返
-- 2. 每次插入都需要事务提交
-- 3. 性能极差
```

**优化1：批量插入**。

```sql
-- 优化：批量插入
INSERT INTO user (name, age) VALUES 
('张三', 20),
('李四', 21),
('王五', 22),
-- ... 一次插入1000条
;

-- 执行时间：1秒
-- 性能提升：100倍
```

**优化2：使用LOAD DATA**。

```sql
-- 优化：使用LOAD DATA
LOAD DATA INFILE '/path/to/user.csv'
INTO TABLE user
FIELDS TERMINATED BY ','
ENCLOSED BY '"'
LINES TERMINATED BY '\n'
(name, age);

-- 执行时间：0.5秒
-- 性能提升：200倍
```

**优化3：关闭自动提交**。

```sql
-- 优化：关闭自动提交
SET autocommit = 0;

INSERT INTO user (name, age) VALUES 
('张三', 20),
('李四', 21),
('王五', 22);
-- ... 一次插入10000条

COMMIT;
SET autocommit = 1;

-- 执行时间：2秒
-- 性能提升：50倍
```

### 🔹 5.1.2 批量更新优化

**问题**：

```sql
-- 慢查询：逐条更新
UPDATE user SET status = 1 WHERE id = 1;
UPDATE user SET status = 1 WHERE id = 2;
UPDATE user SET status = 1 WHERE id = 3;
-- ... 10000条

-- 执行时间：100秒
```

**优化1：使用CASE WHEN**。

```sql
-- 优化：使用CASE WHEN批量更新
UPDATE user SET status = CASE id
    WHEN 1 THEN 1
    WHEN 2 THEN 1
    WHEN 3 THEN 1
    -- ...
END
WHERE id IN (1, 2, 3);

-- 执行时间：1秒
-- 性能提升：100倍
```

**优化2：使用JOIN更新**。

```sql
-- 优化：使用JOIN更新
UPDATE user u
JOIN (
    SELECT id, 1 AS new_status FROM user WHERE id IN (1, 2, 3)
) tmp ON u.id = tmp.id
SET u.status = tmp.new_status;

-- 执行时间：0.5秒
-- 性能提升：200倍
```

**优化3：使用临时表**。

```sql
-- 优化：使用临时表
CREATE TEMPORARY TABLE tmp_user_status (
    id BIGINT PRIMARY KEY,
    status TINYINT
);

INSERT INTO tmp_user_status VALUES (1, 1), (2, 1), (3, 1);

UPDATE user u
JOIN tmp_user_status tmp ON u.id = tmp.id
SET u.status = tmp.status;

DROP TEMPORARY TABLE tmp_user_status;

-- 执行时间：0.3秒
-- 性能提升：300倍
```

## 5.2 锁优化

### 🔹 5.2.1 锁的类型

**MySQL锁分类**：

```
1. 按粒度分类
   - 全局锁：锁定整个数据库
   - 表级锁：锁定整个表
   - 行级锁：锁定某一行

2. 按功能分类
   - 共享锁（S锁）：读锁，允许多个事务同时读取
   - 排他锁（X锁）：写锁，只允许一个事务写入

3. 按使用方式分类
   - 乐观锁：通过版本号控制
   - 悲观锁：通过数据库锁控制
```

**InnoDB行锁**：

```sql
-- 共享锁（S锁）
SELECT * FROM user WHERE id = 1 LOCK IN SHARE MODE;

-- 排他锁（X锁）
SELECT * FROM user WHERE id = 1 FOR UPDATE;

-- 注意：
-- 1. 行锁只在索引上生效
-- 2. 没有索引会退化为表锁
-- 3. 尽量使用索引查询
```

### 🔹 5.2.2 锁优化技巧

**优化1：减少锁持有时间**。

```sql
-- 问题：事务持有锁时间过长
BEGIN;
SELECT * FROM user WHERE id = 1 FOR UPDATE;
-- 执行业务逻辑（10秒）
UPDATE user SET status = 1 WHERE id = 1;
COMMIT;

-- 优化：减少锁持有时间
-- 执行业务逻辑（10秒）
BEGIN;
SELECT * FROM user WHERE id = 1 FOR UPDATE;
UPDATE user SET status = 1 WHERE id = 1;
COMMIT;

-- 锁持有时间从10秒减少到0.1秒
```

**优化2：使用乐观锁**。

```sql
-- 问题：悲观锁性能差
BEGIN;
SELECT * FROM user WHERE id = 1 FOR UPDATE;
UPDATE user SET balance = balance - 100 WHERE id = 1;
COMMIT;

-- 优化：使用乐观锁
-- 添加版本号字段
ALTER TABLE user ADD COLUMN version INT DEFAULT 0;

-- 使用乐观锁更新
UPDATE user SET balance = balance - 100, version = version + 1 
WHERE id = 1 AND version = 0;

-- 检查更新结果
-- 如果影响行数为0，说明数据被修改，需要重试
```

**优化3：降低隔离级别**。

```sql
-- 问题：使用串行化隔离级别，性能差
SET SESSION TRANSACTION ISOLATION LEVEL SERIALIZABLE;

-- 优化：使用读已提交隔离级别
SET SESSION TRANSACTION ISOLATION LEVEL READ COMMITTED;

-- 隔离级别对比：
-- READ UNCOMMITTED：性能最好，可能出现脏读
-- READ COMMITTED：性能较好，可能出现不可重复读
-- REPEATABLE READ：性能中等，可能出现幻读（MySQL默认）
-- SERIALIZABLE：性能最差，完全串行化
```

**优化4：避免死锁**。

```sql
-- 问题：死锁
-- 事务1
BEGIN;
UPDATE user SET status = 1 WHERE id = 1;
UPDATE user SET status = 1 WHERE id = 2;
COMMIT;

-- 事务2
BEGIN;
UPDATE user SET status = 1 WHERE id = 2;
UPDATE user SET status = 1 WHERE id = 1;
COMMIT;

-- 优化：按固定顺序访问资源
-- 事务1
BEGIN;
UPDATE user SET status = 1 WHERE id = 1;
UPDATE user SET status = 1 WHERE id = 2;
COMMIT;

-- 事务2
BEGIN;
UPDATE user SET status = 1 WHERE id = 1;
UPDATE user SET status = 1 WHERE id = 2;
COMMIT;

-- 避免死锁的方法：
-- 1. 按固定顺序访问资源
-- 2. 减少事务持有锁的时间
-- 3. 使用乐观锁
-- 4. 设置锁超时时间
```

### 🔹 5.2.3 锁监控与排查

**查看锁信息**：

```sql
-- 查看当前锁
SHOW ENGINE INNODB STATUS;

-- 查看锁等待
SELECT * FROM information_schema.INNODB_LOCK_WAITS;

-- 查看锁信息
SELECT * FROM information_schema.INNODB_LOCKS;

-- 查看事务
SELECT * FROM information_schema.INNODB_TRX;
```

**查看锁等待**：

```sql
-- 查看锁等待的SQL
SELECT 
    r.trx_id waiting_trx_id,
    r.trx_mysql_thread_id waiting_thread,
    r.trx_query waiting_query,
    b.trx_id blocking_trx_id,
    b.trx_mysql_thread_id blocking_thread,
    b.trx_query blocking_query
FROM information_schema.innodb_lock_waits w
INNER JOIN information_schema.innodb_trx b ON b.trx_id = w.blocking_trx_id
INNER JOIN information_schema.innodb_trx r ON r.trx_id = w.requesting_trx_id;
```

## 5.3 事务优化

### 🔹 5.3.1 事务的基本概念

**ACID特性**：

```
A - Atomicity（原子性）：事务是一个不可分割的工作单位
C - Consistency（一致性）：事务执行前后，数据库从一个一致状态变为另一个一致状态
I - Isolation（隔离性）：多个事务并发执行时，互不干扰
D - Durability（持久性）：事务一旦提交，对数据库的改变是永久的
```

**事务隔离级别**：

| 隔离级别 | 脏读 | 不可重复读 | 幻读 | 性能 |
|----------|------|------------|------|------|
| READ UNCOMMITTED | 可能 | 可能 | 可能 | 最好 |
| READ COMMITTED | 不会 | 可能 | 可能 | 较好 |
| REPEATABLE READ | 不会 | 不会 | 可能 | 中等 |
| SERIALIZABLE | 不会 | 不会 | 不会 | 最差 |

### 🔹 5.3.2 事务优化技巧

**优化1：控制事务大小**。

```sql
-- 问题：大事务
BEGIN;
-- 插入10000条记录
INSERT INTO user (name, age) VALUES ...;
COMMIT;

-- 优化：拆分小事务
BEGIN;
-- 插入1000条记录
INSERT INTO user (name, age) VALUES ...;
COMMIT;

BEGIN;
-- 插入1000条记录
INSERT INTO user (name, age) VALUES ...;
COMMIT;

-- 优点：
-- 1. 减少锁持有时间
-- 2. 减少回滚成本
-- 3. 提高并发性能
```

**优化2：合理使用保存点**。

```sql
-- 使用保存点
BEGIN;

INSERT INTO user (name, age) VALUES ('张三', 20);
SAVEPOINT sp1;

INSERT INTO user (name, age) VALUES ('李四', 21);
SAVEPOINT sp2;

-- 如果出错，回滚到保存点
ROLLBACK TO sp2;

COMMIT;

-- 优点：
-- 1. 部分回滚，不需要回滚整个事务
-- 2. 提高事务灵活性
```

**优化3：避免长事务**。

```sql
-- 问题：长事务
BEGIN;
-- 查询数据
SELECT * FROM user WHERE id = 1;
-- 执行业务逻辑（10秒）
-- 更新数据
UPDATE user SET status = 1 WHERE id = 1;
COMMIT;

-- 优化：缩短事务时间
-- 执行业务逻辑（10秒）
BEGIN;
-- 查询数据
SELECT * FROM user WHERE id = 1;
-- 更新数据
UPDATE user SET status = 1 WHERE id = 1;
COMMIT;

-- 优点：
-- 1. 减少锁持有时间
-- 2. 提高并发性能
-- 3. 减少死锁概率
```

### 🔹 5.3.3 分布式事务

**分布式事务解决方案**：

```
1. XA协议
   - 两阶段提交（2PC）
   - 三阶段提交（3PC）
   - 强一致性，性能较差

2. TCC（Try-Confirm-Cancel）
   - _try阶段：预留资源_
   - _confirm阶段：确认执行_
   - _cancel阶段：取消执行_
   - 最终一致性，性能较好

3. Saga模式
   - 长事务拆分多个本地事务
   - 每个本地事务有对应的补偿操作
   - 最终一致性，性能最好

4. 本地消息表
   - 将分布式事务转换为本地事务
   - 通过消息队列异步执行
   - 最终一致性，性能最好
```

**TCC示例**：

```java
// Try阶段：预留资源
@Transactional
public boolean tryDeduct(Long userId, BigDecimal amount) {
    // 检查余额
    User user = userMapper.selectById(userId);
    if (user.getBalance().compareTo(amount) < 0) {
        return false;
    }
    
    // 冻结金额
    userMapper.freezeAmount(userId, amount);
    return true;
}

// Confirm阶段：确认执行
@Transactional
public boolean confirmDeduct(Long userId, BigDecimal amount) {
    // 扣除冻结金额
    userMapper.deductFrozenAmount(userId, amount);
    return true;
}

// Cancel阶段：取消执行
@Transactional
public boolean cancelDeduct(Long userId, BigDecimal amount) {
    // 解冻金额
    userMapper.unfreezeAmount(userId, amount);
    return true;
}
```

## 5.4 连接池优化

### 🔹 5.4.1 连接池原理

**为什么要使用连接池**：

```
1. 减少连接创建开销
   - 创建数据库连接需要TCP三次握手
   - 创建连接需要认证
   - 创建连接开销大

2. 提高响应速度
   - 复用已有连接
   - 减少连接创建时间
   - 提高系统性能

3. 控制连接数量
   - 避免连接过多导致数据库崩溃
   - 合理分配连接资源
   - 提高系统稳定性
```

**连接池工作流程**：

```
1. 初始化：创建初始连接数
2. 获取连接：从连接池获取连接
3. 使用连接：执行SQL操作
4. 归还连接：将连接归还连接池
5. 销毁连接：关闭空闲连接
```

### 🔹 5.4.2 HikariCP连接池配置

**HikariCP配置**：

```yaml
# application.yml
spring:
  datasource:
    type: com.zaxxer.hikari.HikariDataSource
    hikari:
      # 连接池名称
      pool-name: MyHikariCP
      
      # 最小空闲连接数
      minimum-idle: 10
      
      # 最大连接数
      maximum-pool-size: 50
      
      # 连接空闲超时时间（毫秒）
      idle-timeout: 600000
      
      # 连接最大存活时间（毫秒）
      max-lifetime: 1800000
      
      # 连接超时时间（毫秒）
      connection-timeout: 30000
      
      # 自动提交
      auto-commit: true
      
      # 连接测试查询
      connection-test-query: SELECT 1
```

**连接池参数优化**：

```
1. minimum-idle（最小空闲连接数）
   - 建议：10-20
   - 作用：保持最小空闲连接，避免频繁创建连接

2. maximum-pool-size（最大连接数）
   - 建议：CPU核心数 * 2 + 有效磁盘数
   - 作用：限制最大连接数，避免数据库崩溃

3. idle-timeout（空闲超时时间）
   - 建议：10分钟
   - 作用：关闭空闲连接，节省资源

4. max-lifetime（最大存活时间）
   - 建议：30分钟
   - 作用：定期更换连接，避免连接失效

5. connection-timeout（连接超时时间）
   - 建议：30秒
   - 作用：避免获取连接超时
```

## 5.5 其他优化技巧

### 🔹 5.5.1 使用缓存

**缓存策略**：

```
1. 本地缓存
   - Caffeine
   - Guava Cache
   - 适用于：读多写少，数据量小

2. 分布式缓存
   - Redis
   - Memcached
   - 适用于：读多写少，数据量大

3. 多级缓存
   - 本地缓存 + 分布式缓存
   - 适用于：高并发场景
```

**缓存使用示例**：

```java
// 使用Redis缓存
@Service
public class UserService {
    @Autowired
    private StringRedisTemplate redisTemplate;
    
    @Autowired
    private UserMapper userMapper;
    
    public User getUserById(Long id) {
        // 先从缓存获取
        String key = "user:" + id;
        String userJson = redisTemplate.opsForValue().get(key);
        
        if (userJson != null) {
            return JSON.parseObject(userJson, User.class);
        }
        
        // 缓存未命中，从数据库获取
        User user = userMapper.selectById(id);
        
        if (user != null) {
            // 放入缓存
            redisTemplate.opsForValue().set(key, JSON.toJSONString(user), 30, TimeUnit.MINUTES);
        }
        
        return user;
    }
}
```

### 🔹 5.5.2 使用读写分离

**读写分离架构**：

```
写操作 -> 主库
读操作 -> 从库

主从复制：
1. 主库记录binlog
2. 从库读取binlog
3. 从库重放binlog
4. 实现数据同步
```

**读写分离实现**：


### 🔹 5.5.3 使用搜索引擎

**搜索引擎对比**：

| 特性 | MySQL | ElasticSearch |
|------|-------|---------------|
| 查询速度 | 慢 | 快 |
| 全文搜索 | 支持 | 支持 |
| 分词 | 支持 | 支持 |

| 实时性 | 实时 | 近实时 |
| 适用场景 | 事务数据 | 搜索数据 |

**ElasticSearch使用示例**：

```java
// 同步数据到ES
@Scheduled(cron = "0 0 2 * * ?")
public void syncDataToES() {
    // 查询数据
    List<Product> products = productMapper.selectAll();
    
    // 同步到ES
    for (Product product : products) {
        IndexRequest request = new IndexRequest("products")
            .id(product.getId().toString())
            .source(JSON.toJSONString(product), XContentType.JSON);
        
        esClient.index(request, RequestOptions.DEFAULT);
    }
}

// 从ES搜索
public List<Product> searchProducts(String keyword) {
    SearchRequest request = new SearchRequest("products");
    
    SearchSourceBuilder sourceBuilder = new SearchSourceBuilder();
    sourceBuilder.query(QueryBuilders.multiMatchQuery(keyword, "name", "description"));
    
    request.source(sourceBuilder);
    
    SearchResponse response = esClient.search(request, RequestOptions.DEFAULT);
    
    // 解析结果
    List<Product> products = new ArrayList<>();
    for (SearchHit hit : response.getHits().getHits()) {
        products.add(JSON.parseObject(hit.getSourceAsString(), Product.class));
    }
    
    return products;
}
```

- 锁优化需要根据实际情况选择乐观锁或悲观锁，避免死锁
- 事务优化需要控制事务大小，避免长事务
- 连接池优化需要合理配置参数，监控连接池状态
- 缓存、读写分离、搜索引擎等架构优化可以进一步提升性能

## 6.1 高可用架构

### 🔹 6.1.1 主从复制

**主从复制原理**：

```
1. 主库记录binlog
   - 主库将所有写操作记录到binlog
   - binlog是二进制日志，记录SQL语句或数据变更

2. 从库读取binlog
   - 从库的IO线程连接主库
   - 读取主库的binlog
   - 写入从库的relay log

3. 从库重放binlog
   - 从库的SQL线程读取relay log
   - 重放SQL语句
   - 实现数据同步
```

**主从复制配置**：

```ini
# 主库配置 my.cnf
[mysqld]
server-id = 1
log-bin = mysql-bin
binlog-format = ROW

# 从库配置 my.cnf
[mysqld]
server-id = 2
relay-log = mysql-relay-bin
read-only = 1
```

```sql
-- 主库创建复制用户
CREATE USER 'repl'@'%' IDENTIFIED BY 'password';
GRANT REPLICATION SLAVE ON *.* TO 'repl'@'%';

-- 查看主库状态
SHOW MASTER STATUS;

-- 从库配置主库信息
CHANGE MASTER TO
    MASTER_HOST = 'master_host',
    MASTER_USER = 'repl',
    MASTER_PASSWORD = 'password',
    MASTER_LOG_FILE = 'mysql-bin.000001',
    MASTER_LOG_POS = 154;

-- 启动从库复制
START SLAVE;

-- 查看从库状态
SHOW SLAVE STATUS;
```

### 🔹 6.1.2 主从复制模式

**异步复制**：

```
特点：
- 主库不等待从库确认
- 性能最好
- 可能丢数据

适用场景：
- 对性能要求高
- 可以容忍少量数据丢失
```

**半同步复制**：

```ini
# 主库配置
plugin-load = rpl_semi_sync_master=semisync_master.so
rpl_semi_sync_master_enabled = 1
rpl_semi_sync_master_timeout = 1000

# 从库配置
plugin-load = rpl_semi_sync_slave=semisync_slave.so
rpl_semi_sync_slave_enabled = 1
```

```
特点：
- 主库等待至少一个从库确认
- 性能较好
- 不会丢数据（除非所有从库都挂了）

适用场景：
- 对数据一致性要求高
- 可以容忍一定性能损失
```

**组复制**：

```
特点：
- 多主复制
- 自动故障切换
- 数据一致性高

适用场景：
- 高可用要求高
- 需要自动故障切换
```

### 🔹 6.1.3 高可用方案

**MMM（Master-Master Replication Manager）**：

```
架构：
```

## 6.2 读写分离架构

### 🔹 6.2.1 读写分离原理

**读写分离架构**：

```
应用层
    ↓
代理层（ShardingSphere-Proxy、MyCat）
    ↓
写操作 -> 主库
读操作 -> 从库1、从库2
```

**读写分离优势**：

```
1. 提高读性能
   - 读操作分散到多个从库
   - 提高读吞吐量
   - 支持更大并发

2. 提高可用性
   - 主库故障，从库可以接管
   - 从库故障，其他从库可以继续服务
   - 提高系统可用性

3. 数据备份
   - 从库可以作为数据备份
   - 支持在线备份
   - 不影响主库性能
```

### 🔹 6.2.3 读写分离问题

**主从延迟**：

```
问题：
- 主从复制有延迟
- 写入主库后，从库可能还没同步

3. 缓存补偿
   - 写入后更新缓存
   - 读取时先查缓存
   - 缺点：增加系统复杂度
```

## 6.3 分库分表架构

### 🔹 6.3.1 分库分表策略

**水平分片策略**：

```
1. 范围分片
   - 按ID范围：1-100万在库0，100万-200万在库1
   - 按时间范围：2023年在库0，2024年在库1
   - 优点：扩容简单，数据迁移方便
   - 缺点：数据分布不均匀，热点问题

2. 哈希分片
   - 按ID哈希：id % 2 = 0在库0，id % 2 = 1在库1
   - 按用户ID哈希：user_id % 2 = 0在库0，user_id % 2 = 1在库1
   - 优点：数据分布均匀，无热点问题
   - 缺点：扩容复杂，数据迁移困难

3. 标签分片
   - 按地区分片：华东在库0，华南在库1
   - 按业务分片：订单在库0，用户在库1
   - 优点：符合业务特点，查询性能好
   - 缺点：数据分布可能不均匀
```

**垂直分片策略**：

```
1. 按字段使用频率分片
   - 高频字段在一个表
   - 低频字段在另一个表

2. 按字段大小分片
   - 大字段在一个表
   - 小字段在另一个表

3. 按业务逻辑分片
   - 核心字段在一个表
   - 扩展字段在另一个表
```

### 🔹 6.3.2 分库分表问题

**分布式ID**：

```
问题：
- 分库分表后，自增ID会重复
- 需要全局唯一ID

解决方案：
1. UUID
   - 优点：简单，全局唯一
   - 缺点：长度长，无序，索引性能差

2. 雪花算法
   - 优点：趋势递增，性能高
   - 缺点：依赖时钟，时钟回拨会重复

3. 数据库号段
   - 优点：趋势递增，性能高
   - 缺点：需要额外数据库
```

**雪花算法实现**：

```java
public class SnowflakeIdGenerator {
    
    // 起始时间戳
    private final long twepoch = 1288834974657L;
    
    // 机器ID位数
    private final long workerIdBits = 5L;
    
    // 数据中心ID位数
    private final long datacenterIdBits = 5L;
    
    // 序列号位数
    private final long sequenceBits = 12L;
    
    // 机器ID最大值
    private final long maxWorkerId = -1L ^ (-1L << workerIdBits);
    
    // 数据中心ID最大值
    private final long maxDatacenterId = -1L ^ (-1L << datacenterIdBits);
    
    // 序列号最大值
    private final long sequenceMask = -1L ^ (-1L << sequenceBits);
    
    // 机器ID左移位数
    private final long workerIdShift = sequenceBits;
    
    // 数据中心ID左移位数
    private final long datacenterIdShift = sequenceBits + workerIdBits;
    
    // 时间戳左移位数
    private final long timestampLeftShift = sequenceBits + workerIdBits + datacenterIdBits;
    
    private long workerId;
    private long datacenterId;
    private long sequence = 0L;
    private long lastTimestamp = -1L;
    
    public SnowflakeIdGenerator(long workerId, long datacenterId) {
        if (workerId > maxWorkerId || workerId < 0) {
            throw new IllegalArgumentException("worker Id can't be greater than %d or less than 0");
        }
        if (datacenterId > maxDatacenterId || datacenterId < 0) {
            throw new IllegalArgumentException("datacenter Id can't be greater than %d or less than 0");
        }
        this.workerId = workerId;
        this.datacenterId = datacenterId;
    }
    
    public synchronized long nextId() {
        long timestamp = timeGen();
        
        if (timestamp < lastTimestamp) {
            throw new RuntimeException("Clock moved backwards. Refusing to generate id");
        }
        
        if (lastTimestamp == timestamp) {
            sequence = (sequence + 1) & sequenceMask;
            if (sequence == 0) {
                timestamp = tilNextMillis(lastTimestamp);
            }
        } else {
            sequence = 0L;
        }
        
        lastTimestamp = timestamp;
        
        return ((timestamp - twepoch) << timestampLeftShift)
                | (datacenterId << datacenterIdShift)
                | (workerId << workerIdShift)
                | sequence;
    }
    
    private long tilNextMillis(long lastTimestamp) {
        long timestamp = timeGen();
        while (timestamp <= lastTimestamp) {
            timestamp = timeGen();
        }
        return timestamp;
    }
    
    private long timeGen() {
        return System.currentTimeMillis();
    }
}
```

### 🔹 6.3.3 分库分表扩容

**平滑扩容方案**：

```
1. 双倍扩容法
   - 原分片数：2
   - 新分片数：4
   - 数据迁移：50%的数据需要迁移

2. 一致性哈希
   - 使用一致性哈希算法
   - 扩容时只影响部分数据
   - 数据迁移量少

3. 虚拟节点
   - 每个物理节点对应多个虚拟节点
   - 扩容时只影响部分虚拟节点
   - 数据分布更均匀
```

**双倍扩容实现**：

```
原分片：
- 库0：id % 2 = 0
- 库1：id % 2 = 1

扩容后：
- 库0：id % 4 = 0
- 库1：id % 4 = 1
- 库2：id % 4 = 2（原库0的数据）
- 库3：id % 4 = 3（原库1的数据）

数据迁移：
- 库0中 id % 4 = 2 的数据迁移到库2
- 库1中 id % 4 = 3 的数据迁移到库3
```

## 1.2 SQL性能评估

### 🔹 1.2.1 执行计划分析

**什么是执行计划**：

```
执行计划是数据库优化器生成的SQL执行路径
包含了SQL语句的执行步骤、访问方式、连接方式等信息
通过分析执行计划，可以找到SQL性能瓶颈
```

**MySQL执行计划**：

```sql
-- 查看执行计划
EXPLAIN SELECT * FROM user WHERE id = 1;

-- 输出字段说明：
-- id: 查询序列号
-- select_type: 查询类型
-- table: 表名
-- partitions: 分区信息
-- type: 访问类型（重要）
-- possible_keys: 可能使用的索引
-- key: 实际使用的索引
-- key_len: 使用索引的长度
-- ref: 索引比较的列
-- rows: 扫描的行数（重要）
-- filtered: 过滤的行数比例
-- Extra: 额外信息（重要）
```

**type字段详解**：

```
type字段表示访问类型，从好到坏依次为：

1. system: 表只有一行记录（系统表）
2. const: 通过主键或唯一索引查找，最多返回一行
3. eq_ref: 通过唯一索引扫描，每个索引对应一条记录
4. ref: 通过非唯一索引扫描，返回匹配的多行
5. range: 范围查询
6. index: 索引全扫描
7. ALL: 全表扫描（最差）

优化目标：至少达到ref级别，最好达到const或eq_ref
```

**Extra字段详解**：

```
1. Using index: 使用了覆盖索引，不需要回表（好）
2. Using where: 使用了WHERE过滤
3. Using temporary: 使用了临时表（性能差）
4. Using filesort: 使用了文件排序（性能差）
5. Using index condition: 使用了索引下推
6. Using join buffer: 使用了连接缓冲区

优化目标：避免Using temporary和Using filesort
```

### 🔹 1.2.2 慢查询分析

**慢查询日志**：


```bash
# 参数说明：

#   r: 按返回记录数排序
#   c: 按访问次数排序

```

**慢查询分析步骤**：


## 1.3 SQL优化基础

### 🔹 1.3.1 SQL执行流程

**SQL执行流程**：

```
1. 客户端发送SQL
   ↓
2. 连接器接收SQL
   ↓
3. 查询缓存（MySQL 8.0已移除）
   ↓
4. 分析器解析SQL
   ↓
5. 预处理器预处理
   ↓
6. 优化器生成执行计划
   ↓
7. 执行器执行SQL
   ↓
8. 存储引擎访问数据
   ↓
9. 返回结果给客户端
```

**优化器的作用**：

```
优化器负责生成最优的执行计划

优化器考虑的因素：
1. 统计信息
   - 表的行数
   - 列的基数（不同值的数量）
   - 数据分布

2. 索引信息
   - 可用的索引
   - 索引的选择性

3. 成本估算
   - I/O成本
   - CPU成本
   - 内存成本

4. 优化规则
   - 索引选择规则
   - 连接顺序规则
   - 子查询优化规则
```

### 🔹 1.3.2 成本估算

**成本模型**：

```
MySQL使用基于成本的优化器（CBO）

成本 = I/O成本 + CPU成本

I/O成本：
- 全表扫描：rows * page_size
- 索引扫描：index_rows * page_size

CPU成本：
- 记录比较：rows * cpu_cost
- 排序：rows * log(rows) * cpu_cost
- 连接：rows1 * rows2 * cpu_cost
```

**统计信息**：

```sql
-- 查看表的统计信息
SHOW TABLE STATUS LIKE 'table_name';

-- 更新统计信息
ANALYZE TABLE table_name;

-- 查看列的统计信息
SHOW INDEX FROM table_name;

-- 查看列的基数
SELECT COUNT(DISTINCT column_name) FROM table_name;
```

### 🔹 1.3.3 优化提示

**使用索引提示**：

```sql
-- 强制使用索引
SELECT * FROM table_name USE INDEX (index_name) WHERE condition;

-- 忽略索引
SELECT * FROM table_name IGNORE INDEX (index_name) WHERE condition;

-- 强制使用特定索引
SELECT * FROM table_name FORCE INDEX (index_name) WHERE condition;
```

**使用连接提示**：

```sql
-- STRAIGHT_JOIN强制连接顺序
SELECT * FROM table1 STRAIGHT_JOIN table2 ON table1.id = table2.id;

-- 相当于：
SELECT * FROM table1 JOIN table2 ON table1.id = table2.id;
-- 但强制先访问table1，再访问table2
```

# 🧱 第2章：索引优化

------

> **本章在整体中解决什么问题**：深入理解索引的原理和使用方法，掌握索引设计和优化的核心技巧。学完本章后，你将能够设计高效的索引，解决大部分SQL性能问题。

------

## 2.1 索引概述

### 🔹 2.1.1 什么是索引

**索引的定义**：

```
索引是数据库表中一列或多列值的排序结构
类似于书的目录，可以快速定位到数据
索引可以大大提高查询速度，但会增加写入成本
```

**索引的作用**：

```
1. 提高查询速度
   - 通过索引快速定位数据
   - 避免全表扫描
   - 查询速度提升10-100倍

2. 加速排序
   - 索引本身是有序的
   - 利用索引可以避免额外排序
   - 提高ORDER BY性能

3. 加速分组
   - 利用索引可以加速GROUP BY
   - 避免临时表
   - 提高聚合查询性能

4. 加速连接
   - 利用索引加速表连接
   - 提高JOIN性能
   - 减少连接成本
```

**索引的代价**：

```
1. 空间成本
   - 索引占用存储空间
   - 通常占用表空间的10-30%
   - 需要权衡空间和性能

2. 写入成本
   - 插入数据需要更新索引
   - 删除数据需要更新索引
   - 更新数据需要更新索引
   - 写入性能下降10-50%

3. 维护成本
   - 索引需要定期维护
   - 索引碎片需要整理
   - 统计信息需要更新
```

### 🔹 2.1.2 索引的数据结构

**B+树索引**：

```
B+树是MySQL最常用的索引数据结构

B+树的特点：
1. 多路平衡树
2. 所有数据存储在叶子节点
3. 叶子节点通过指针连接
4. 高度通常为3-4层
5. 查询时间复杂度为O(logN)

B+树的优势：
1. 查询效率高
2. 范围查询效率高
3. 磁盘I/O次数少
4. 适合存储大量数据
```

**B+树结构示例**：

```
        [根节点]
       /    |    \
   [节点1] [节点2] [节点3]
   /   \   /   \   /   \
 [叶1] [叶2] [叶3] [叶4] [叶5] [叶6]
  ↓     ↓     ↓     ↓     ↓     ↓
 [数据] [数据] [数据] [数据] [数据] [数据]

查询过程：
1. 从根节点开始
2. 根据索引值选择子节点
3. 递归查找，直到叶子节点
4. 在叶子节点中找到数据

时间复杂度：O(logN)
空间复杂度：O(N)
```

**哈希索引**：

```
哈希索引基于哈希表实现

哈希索引的特点：
1. 查询时间复杂度为O(1)
2. 只支持等值查询
3. 不支持范围查询
4. 不支持排序

哈希索引的适用场景：
1. 等值查询
2. 数据量较小
3. 查询频率高

MySQL中Memory引擎使用哈希索引
```

**全文索引**：

```
全文索引用于文本搜索

全文索引的特点：
1. 基于倒排索引
2. 支持中文分词
3. 支持相关性排序
4. 查询性能好

全文索引的适用场景：
1. 文本搜索
2. 关键词查询
3. 内容检索

使用示例：
SELECT * FROM articles WHERE MATCH(title, content) AGAINST('关键词');
```

### 🔹 2.1.3 索引的类型

**主键索引**：

```sql
-- 创建主键索引
CREATE TABLE user (
    id INT PRIMARY KEY,
    name VARCHAR(100),
    age INT
);

-- 或
CREATE TABLE user (
    id INT,
    name VARCHAR(100),
    age INT,
    PRIMARY KEY (id)
);

-- 或
ALTER TABLE user ADD PRIMARY KEY (id);

特点：
1. 唯一且非空
2. 一个表只能有一个主键
3. 自动创建聚簇索引
4. 查询性能最好
```

**唯一索引**：

```sql
-- 创建唯一索引
CREATE UNIQUE INDEX idx_user_name ON user(name);

-- 或
CREATE TABLE user (
    id INT PRIMARY KEY,
    name VARCHAR(100) UNIQUE,
    age INT
);

特点：
1. 唯一但允许NULL
2. 一个表可以有多个唯一索引
3. 插入时会检查唯一性
4. 查询性能好
```

**普通索引**：

```sql
-- 创建普通索引
CREATE INDEX idx_user_age ON user(age);

-- 或
ALTER TABLE user ADD INDEX idx_user_age (age);

特点：
1. 不要求唯一
2. 一个表可以有多个普通索引
3. 可以重复值
4. 查询性能好
```

**联合索引**：

```sql
-- 创建联合索引
CREATE INDEX idx_user_name_age ON user(name, age);

-- 或
ALTER TABLE user ADD INDEX idx_user_name_age (name, age);

特点：
1. 多个字段组合的索引
2. 遵循最左前缀原则
3. 查询性能取决于查询条件
4. 需要合理设计字段顺序
```

**全文索引**：

```sql
-- 创建全文索引
CREATE FULLTEXT INDEX idx_article_content ON articles(title, content);

特点：
```

**空间索引**：

```sql
4. 查询性能好
```

## 2.2 索引设计原则

### 🔹 2.2.1 最左前缀原则

**最左前缀原则**：

```
联合索引遵循最左前缀原则
只有查询条件包含索引的最左列，索引才会生效
```

**示例**：

```sql
-- 创建联合索引
CREATE INDEX idx_user_name_age ON user(name, age);

-- 索引生效的查询：
SELECT * FROM user WHERE name = '张三';
SELECT * FROM user WHERE name = '张三' AND age = 20;
SELECT * FROM user WHERE name = '张三' AND age > 20;

-- 索引不生效的查询：
SELECT * FROM user WHERE age = 20;
SELECT * FROM user WHERE age = 20 AND name = '张三';
```

**最左前缀原则详解**：

```
联合索引 idx(a, b, c)

查询条件                索引使用情况
WHERE a = 1             使用索引a
WHERE a = 1 AND b = 2   使用索引a, b
WHERE a = 1 AND b = 2 AND c = 3  使用索引a, b, c
WHERE b = 2             不使用索引
WHERE b = 2 AND c = 3   不使用索引
WHERE c = 3             不使用索引
WHERE a = 1 AND c = 3   使用索引a（跳过了b）
```

### 🔹 2.2.2 索引选择性

**索引选择性定义**：

```
索引选择性 = 不同值的数量 / 总行数

选择性越高，索引效果越好
选择性越低，索引效果越差

经验值：
- 选择性 > 0.1：适合建索引
- 选择性 < 0.1：不适合建索引
```

**计算索引选择性**：

```sql
-- 计算列的选择性
SELECT
    COUNT(DISTINCT column_name) / COUNT(*) AS selectivity
FROM table_name;

-- 示例
SELECT
    COUNT(DISTINCT name) / COUNT(*) AS name_selectivity,
    COUNT(DISTINCT age) / COUNT(*) AS age_selectivity,
    COUNT(DISTINCT gender) / COUNT(*) AS gender_selectivity
FROM user;

-- 结果示例：
-- name_selectivity: 0.95（适合建索引）
-- age_selectivity: 0.05（不适合建索引）
-- gender_selectivity: 0.002（不适合建索引）
```

**索引选择性的应用**：

```
1. 高选择性列适合建索引
   - 主键、唯一键
   - 手机号、身份证号
   - 订单号、流水号

2. 低选择性列不适合建索引
   - 性别、状态
   - 类型、分类
   - 标记位

3. 联合索引的选择性
   - 将高选择性列放在前面
   - 将低选择性列放在后面
   - 提高联合索引的效率
```

### 🔹 2.2.3 覆盖索引

**覆盖索引定义**：

```
覆盖索引：查询的列都包含在索引中
不需要回表查询，性能最好
```

**覆盖索引示例**：

```sql
-- 创建联合索引
CREATE INDEX idx_user_name_age ON user(name, age);

-- 覆盖索引查询（不需要回表）
SELECT name, age FROM user WHERE name = '张三';

-- 非覆盖索引查询（需要回表）
SELECT * FROM user WHERE name = '张三';

-- 查看执行计划
EXPLAIN SELECT name, age FROM user WHERE name = '张三';
-- Extra: Using index（使用了覆盖索引）

EXPLAIN SELECT * FROM user WHERE name = '张三';
-- Extra: （需要回表）
```

**覆盖索引的优势**：

```
1. 避免回表
   - 不需要访问表数据
   - 只需要访问索引
   - 减少I/O操作

2. 减少锁竞争
   - 不需要访问表数据
   - 减少锁的持有时间
   - 提高并发性能

3. 减少内存占用
   - 不需要加载表数据
   - 只需要加载索引
   - 减少内存使用
```

### 🔹 2.2.4 索引列选择

**选择索引列的原则**：

```
1. WHERE条件列
   - 经常作为查询条件的列
   - 优先建立索引
   - 提高查询性能

2. ORDER BY列
   - 经常排序的列
   - 建立索引可以避免排序
   - 提高排序性能

3. GROUP BY列
   - 经常分组的列
   - 建立索引可以加速分组
   - 提高聚合性能

4. JOIN列
   - 连接表的列
   - 建立索引可以加速连接
   - 提高JOIN性能

5. 避免索引的列
   - 区分度低的列
   - 频繁更新的列
   - 大文本列
```

**索引列顺序**：

```
联合索引的列顺序很重要

原则：
1. 将高选择性列放在前面
2. 将经常查询的列放在前面
3. 将范围查询列放在后面

示例：
CREATE INDEX idx_user_name_age_gender ON user(name, age, gender);

查询条件                索引使用情况
WHERE name = '张三'     使用索引name
WHERE name = '张三' AND age = 20  使用索引name, age
WHERE name = '张三' AND age > 20  使用索引name, age
WHERE age = 20          不使用索引
WHERE gender = '男'     不使用索引
```

## 2.3 索引优化技巧

### 🔹 2.3.1 避免索引失效

**索引失效的场景**：

```sql
-- 1. 使用函数
SELECT * FROM user WHERE YEAR(create_time) = 2023;
-- 索引失效，建议改为：
SELECT * FROM user WHERE create_time >= '2023-01-01' AND create_time < '2024-01-01';

-- 2. 使用运算
SELECT * FROM user WHERE age + 1 = 21;
-- 索引失效，建议改为：
SELECT * FROM user WHERE age = 20;

-- 3. 使用OR
SELECT * FROM user WHERE name = '张三' OR age = 20;
-- 索引可能失效，建议改为：
SELECT * FROM user WHERE name = '张三'
UNION ALL
SELECT * FROM user WHERE age = 20;

-- 4. 使用LIKE前缀模糊查询
SELECT * FROM user WHERE name LIKE '%张三';
-- 索引失效，建议改为：
SELECT * FROM user WHERE name LIKE '张三%';
-- 或使用全文索引

-- 5. 类型转换
SELECT * FROM user WHERE phone = 12345678901;
-- 索引失效（隐式类型转换），建议改为：
SELECT * FROM user WHERE phone = '12345678901';

-- 6. 使用NOT
SELECT * FROM user WHERE age != 20;
-- 索引失效，建议改为：
SELECT * FROM user WHERE age < 20 OR age > 20;

-- 7. 使用IS NULL或IS NOT NULL
SELECT * FROM user WHERE name IS NULL;
-- 索引可能失效，建议使用默认值代替NULL
```

### 🔹 2.3.2 索引下推

**索引下推定义**：

```
索引下推：在索引遍历过程中，对索引中包含的字段进行过滤
减少回表次数，提高查询性能
```

**索引下推示例**：

```sql
-- 创建联合索引
CREATE INDEX idx_user_name_age ON user(name, age);

-- 查询
SELECT * FROM user WHERE name LIKE '张%' AND age > 20;

-- 没有索引下推：
-- 1. 使用索引找到name LIKE '张%'的所有记录
-- 2. 回表查询所有记录
-- 3. 过滤age > 20的记录

-- 有索引下推：
-- 1. 使用索引找到name LIKE '张%'的所有记录
-- 2. 在索引中过滤age > 20的记录
-- 3. 只回表满足条件的记录

-- 查看执行计划
EXPLAIN SELECT * FROM user WHERE name LIKE '张%' AND age > 20;
-- Extra: Using index condition（使用了索引下推）
```

**索引下推的优势**：

```
1. 减少回表次数
   - 在索引中过滤数据
   - 只回表满足条件的记录
   - 减少I/O操作

2. 提高查询性能
   - 减少数据传输量
   - 减少CPU计算量
   - 提高查询速度

3. 减少锁竞争
   - 减少回表次数
   - 减少锁的持有时间
   - 提高并发性能
```

### 🔹 2.3.3 索引维护

**索引碎片整理**：

```sql
-- 查看索引碎片
SELECT
    table_name,
    index_name,
    data_free / 1024 / 1024 AS fragment_size_mb
FROM information_schema.tables
WHERE table_schema = 'database_name';

-- 整理索引碎片
OPTIMIZE TABLE table_name;

-- 或
ALTER TABLE table_name ENGINE = InnoDB;

-- 或
ANALYZE TABLE table_name;
```

**索引统计信息更新**：

```sql
-- 更新表的统计信息
ANALYZE TABLE table_name;

-- 更新数据库的统计信息
ANALYZE TABLE database_name.table_name;

-- 查看统计信息
SHOW INDEX FROM table_name;
```

**删除冗余索引**：

```sql
-- 查看冗余索引
SELECT
    table_name,
    index_name,
    GROUP_CONCAT(column_name ORDER BY seq_in_index) AS columns
FROM information_schema.statistics
WHERE table_schema = 'database_name'
GROUP BY table_name, index_name
HAVING COUNT(*) > 1;

-- 删除冗余索引
DROP INDEX idx_redundant ON table_name;
```

**删除未使用的索引**：

```sql
-- 查看未使用的索引（MySQL 5.7+）
SELECT
    object_schema AS database_name,
    object_name AS table_name,
    index_name
FROM performance_schema.table_io_waits_summary_by_index_usage
WHERE index_name IS NOT NULL
AND count_star = 0
AND object_schema = 'database_name'
ORDER BY object_schema, object_name;

-- 删除未使用的索引
DROP INDEX idx_unused ON table_name;
```

## 2.4 索引实战案例

### 🔹 2.4.1 案例一：慢查询优化

**问题**：

```sql
-- 慢查询
SELECT * FROM orders WHERE user_id = 12345 ORDER BY create_time DESC LIMIT 10;

EXPLAIN SELECT * FROM orders WHERE user_id = 12345 ORDER BY create_time DESC LIMIT 10;
```

```sql
-- 创建联合索引
CREATE INDEX idx_user_id_create_time ON orders(user_id, create_time);

-- rows: 100（扫描100行）
-- Extra: （不需要排序）

性能提升：500倍
```

## 3.1 SELECT优化

### 🔹 3.1.1 避免SELECT *

**问题**：

```sql
-- 慢查询
SELECT * FROM user WHERE id = 1;

-- 执行计划
EXPLAIN SELECT * FROM user WHERE id = 1;
-- type: const
-- Extra: （需要回表）

-- 问题：
-- 1. 查询了所有列，可能包含不需要的列
-- 2. 增加了网络传输量
-- 3. 增加了I/O操作
-- 4. 无法使用覆盖索引
```

**优化**：

```sql
-- 优化后
SELECT id, name, age FROM user WHERE id = 1;

-- 创建覆盖索引
CREATE INDEX idx_id_name_age ON user(id, name, age);

-- 查看执行计划
EXPLAIN SELECT id, name, age FROM user WHERE id = 1;
-- type: const
-- Extra: Using index（覆盖索引，不需要回表）

-- 性能提升：5-10倍
```

### 🔹 3.1.2 避免重复查询

**问题**：

```sql
-- 慢查询
SELECT * FROM user WHERE id = 1;
SELECT * FROM user WHERE id = 1;
SELECT * FROM user WHERE id = 1;

-- 问题：
-- 1. 重复查询相同的数据
-- 2. 增加了数据库负载
-- 3. 响应时间增加
```

**优化**：

```sql
-- 优化方案1：使用变量
SELECT @id := id, @name := name, @age := age FROM user WHERE id = 1;
SELECT @id, @name, @age;

-- 优化方案2：使用缓存
-- 在应用层缓存查询结果

-- 优化方案3：使用JOIN
SELECT u1.*, u2.*, u3.*
FROM user u1
JOIN user u2 ON u1.id = u2.id
JOIN user u3 ON u1.id = u3.id
WHERE u1.id = 1;
```

### 🔹 3.1.3 合理使用LIMIT

**问题**：

```sql
-- 慢查询
SELECT * FROM user LIMIT 1000000, 10;

-- 执行计划
EXPLAIN SELECT * FROM user LIMIT 1000000, 10;
-- type: ALL（全表扫描）
-- rows: 1000000（扫描100万行）
-- Extra: （需要扫描前100万行）

-- 执行时间：5秒

-- 问题：
-- 1. 大偏移量性能差
-- 2. 需要扫描大量数据
-- 3. 响应时间长
```

**优化**：

```sql
-- 优化方案1：使用延迟关联
SELECT * FROM user u
JOIN (SELECT id FROM user LIMIT 1000000, 10) tmp
ON u.id = tmp.id;

-- 优化方案2：使用游标
SELECT * FROM user WHERE id > last_id ORDER BY id LIMIT 10;

-- 优化方案3：使用覆盖索引
CREATE INDEX idx_id ON user(id);
SELECT * FROM user u
JOIN (SELECT id FROM user ORDER BY id LIMIT 1000000, 10) tmp
ON u.id = tmp.id;

-- 性能提升：10-100倍
```

## 3.2 JOIN优化

### 🔹 3.2.1 JOIN类型选择

**INNER JOIN**：

```sql
-- INNER JOIN：只返回匹配的行
SELECT * FROM orders o
INNER JOIN user u ON o.user_id = u.id;

-- 适用场景：
-- 1. 只需要匹配的数据
-- 2. 两边数据都重要
-- 3. 性能较好
```

**LEFT JOIN**：

```sql
-- LEFT JOIN：返回左表所有行，右表不匹配的返回NULL
SELECT * FROM orders o
LEFT JOIN user u ON o.user_id = u.id;

-- 适用场景：
-- 1. 需要左表所有数据
-- 2. 右表数据可选
-- 3. 性能较差（需要扫描左表所有行）
```

**RIGHT JOIN**：

```sql
-- RIGHT JOIN：返回右表所有行，左表不匹配的返回NULL
SELECT * FROM orders o
RIGHT JOIN user u ON o.user_id = u.id;

-- 适用场景：
-- 1. 需要右表所有数据

```

### 🔹 3.2.2 JOIN优化技巧

**优化1：小表驱动大表**

```sql
-- 问题

SELECT * FROM small_table s

-- 原理：
-- 1. 小表作为驱动表，减少循环次数
```

**优化2：确保JOIN字段有索引**

```sql
-- Extra: Using where（WHERE过滤）

CREATE INDEX idx_user_id ON orders(user_id);
CREATE INDEX idx_id ON user(id);

-- 查看执行计划

-- type: ref（索引查找）
```

## 3.3 子查询优化

### 🔹 3.3.1 子查询类型

**标量子查询**：

```sql

-- 适用场景：

-- 2. 子查询只返回一行一列
-- 3. 性能较好
```

**列子查询**：

```sql
-- 列子查询：返回单列多行
SELECT * FROM user WHERE id IN (SELECT user_id FROM orders);

-- 适用场景：
-- 1. 需要多值作为条件
-- 2. 子查询返回多行一列
-- 3. 性能中等
```

**行子查询**：

```sql
-- 行子查询：返回单行多列
SELECT * FROM user WHERE (name, age) = (SELECT name, age FROM user WHERE id = 1);

-- 适用场景：
-- 1. 需要多列作为条件
-- 2. 子查询返回一行多列
-- 3. 性能中等
```

**表子查询**：

```sql
-- 表子查询：返回多行多列
SELECT * FROM (SELECT id, name, age FROM user) AS tmp WHERE age > 20;

-- 适用场景：
-- 1. 需要临时表
-- 2. 子查询返回多行多列
-- 3. 性能较差
```

### 🔹 3.3.2 子查询优化技巧

**优化1：使用JOIN替代子查询**

```sql
-- 问题
SELECT * FROM user WHERE id IN (SELECT user_id FROM orders WHERE amount > 1000);

-- 执行计划
EXPLAIN SELECT * FROM user WHERE id IN (SELECT user_id FROM orders WHERE amount > 1000);
-- type: range
-- Extra: Using where; Using temporary（使用临时表）

-- 执行时间：5秒

-- 优化：使用JOIN
SELECT DISTINCT u.* FROM user u
JOIN orders o ON u.id = o.user_id
WHERE o.amount > 1000;

-- 执行计划
EXPLAIN SELECT DISTINCT u.* FROM user u JOIN orders o ON u.id = o.user_id WHERE o.amount > 1000;
-- type: ref
-- Extra: （使用索引）

-- 执行时间：0.5秒

-- 性能提升：10倍
```

**优化2：使用EXISTS替代IN**

```sql
-- 问题
SELECT * FROM user WHERE id IN (SELECT user_id FROM orders WHERE amount > 1000);

-- 优化：使用EXISTS
SELECT * FROM user u
WHERE EXISTS (
    SELECT 1 FROM orders o
    WHERE o.user_id = u.id AND o.amount > 1000
);

-- 性能提升：2-5倍

-- 原理：
-- 1. EXISTS只要找到一条记录就返回
-- 2. IN需要扫描所有记录
-- 3. EXISTS性能更好
```

**优化3：使用临时表**

```sql
-- 问题
SELECT * FROM user WHERE id IN (
    SELECT user_id FROM orders WHERE amount > 1000
    UNION
    SELECT user_id FROM refunds WHERE amount > 1000
);

-- 执行时间：10秒

-- 优化：使用临时表
CREATE TEMPORARY TABLE temp_user_ids AS
SELECT user_id FROM orders WHERE amount > 1000
UNION
SELECT user_id FROM refunds WHERE amount > 1000;

SELECT * FROM user WHERE id IN (SELECT user_id FROM temp_user_ids);

DROP TEMPORARY TABLE temp_user_ids;

-- 执行时间：2秒

-- 性能提升：5倍
```

## 3.5 分页优化

### 🔹 3.5.1 传统分页优化

**问题**：

```sql

-- 3. 不需要扫描前面的数据
```

## 4.1 数据类型选择

### 🔹 4.1.1 整数类型

**整数类型对比**：

| 类型 | 字节数 | 范围 | 适用场景 |
|------|--------|------|----------|
| TINYINT | 1 | -128~127 | 状态、标记位 |
| SMALLINT | 2 | -32768~32767 | 小范围整数 |
| MEDIUMINT | 3 | -8388608~8388607 | 中等范围整数 |
| INT | 4 | -2147483648~2147483647 | 通用整数 |
| BIGINT | 8 | -9223372036854775808~9223372036854775807 | 大整数 |

**选择原则**：

```sql
-- 1. 根据数据范围选择合适的类型
CREATE TABLE user (
    id BIGINT PRIMARY KEY,           -- 主键使用BIGINT
    age TINYINT UNSIGNED,            -- 年龄使用TINYINT（0-255）
    status TINYINT DEFAULT 0,        -- 状态使用TINYINT
    score SMALLINT,                  -- 分数使用SMALLINT
    balance INT                      -- 余额使用INT
);

-- 2. 使用UNSIGNED扩大范围
CREATE TABLE user (
    age TINYINT UNSIGNED,            -- 0-255
    status TINYINT UNSIGNED          -- 0-255
);

-- 3. 使用ZEROFILL填充前导零
CREATE TABLE user (
    code INT ZEROFILL                -- 001, 002, 003
);
```

### 🔹 4.1.2 浮点数类型

**浮点数类型对比**：

| 类型 | 字节数 | 精度 | 适用场景 |
|------|--------|------|----------|
| FLOAT | 4 | 单精度 | 近似值、科学计算 |
| DOUBLE | 8 | 双精度 | 近似值、科学计算 |
| DECIMAL(M,D) | M+2 | 精确值 | 金额、精确计算 |

**选择原则**：

```sql
-- 1. 金额使用DECIMAL
CREATE TABLE orders (
    id BIGINT PRIMARY KEY,
    amount DECIMAL(10, 2)            -- 金额使用DECIMAL，精确到分
);

-- 2. 科学计算使用FLOAT或DOUBLE
CREATE TABLE data (
    id BIGINT PRIMARY KEY,
    temperature FLOAT,                -- 温度使用FLOAT
    weight DOUBLE                    -- 重量使用DOUBLE
);

-- 3. 避免使用FLOAT和DOUBLE存储金额
-- 错误示例
CREATE TABLE orders (
    amount FLOAT                      -- 错误：金额不能使用FLOAT
);

-- 正确示例
CREATE TABLE orders (
    amount DECIMAL(10, 2)             -- 正确：金额使用DECIMAL
);
```

### 🔹 4.1.3 字符串类型

**字符串类型对比**：

| 类型 | 最大长度 | 特点 | 适用场景 |
|------|----------|------|----------|
| CHAR(M) | 255 | 固定长度 | 固定长度字符串 |
| VARCHAR(M) | 65535 | 可变长度 | 可变长度字符串 |
| TEXT | 65535 | 大文本 | 长文本 |
| MEDIUMTEXT | 16777215 | 中等文本 | 中等长度文本 |
| LONGTEXT | 4294967295 | 长文本 | 超长文本 |

**选择原则**：

```sql
-- 1. 固定长度使用CHAR
CREATE TABLE user (
    id BIGINT PRIMARY KEY,
    phone CHAR(11),                  -- 手机号使用CHAR
    id_card CHAR(18)                 -- 身份证号使用CHAR
);

-- 2. 可变长度使用VARCHAR
CREATE TABLE user (
    id BIGINT PRIMARY KEY,
    name VARCHAR(100),               -- 姓名使用VARCHAR
    address VARCHAR(500)             -- 地址使用VARCHAR
);

-- 3. 长文本使用TEXT
CREATE TABLE article (
    id BIGINT PRIMARY KEY,
    title VARCHAR(200),
    content TEXT                     -- 内容使用TEXT
);

-- 4. 避免过大的VARCHAR
-- 错误示例
CREATE TABLE user (
    remark VARCHAR(10000)            -- 错误：VARCHAR过大
);

-- 正确示例
CREATE TABLE user (
    remark TEXT                      -- 正确：使用TEXT
);
```

## 4.2 表设计规范

### 🔹 4.2.1 命名规范

**表命名规范**：

```
1. 表名使用小写字母
   - 正确：user、orders、order_item
   - 错误：User、Orders、OrderItem

2. 表名使用下划线分隔
   - 正确：order_item、user_address
   - 错误：orderitem、useraddress

3. 表名使用复数形式
   - 正确：users、orders、products
   - 错误：user、order、product

4. 表名具有描述性
   - 正确：user_order、order_item
   - 错误：t1、t2、table1
```

**字段命名规范**：

```
1. 字段名使用小写字母
   - 正确：user_id、order_no
   - 错误：UserId、OrderNo

2. 字段名使用下划线分隔
   - 正确：user_id、create_time
   - 错误：userid、createtime

3. 字段名具有描述性
   - 正确：user_name、order_amount
   - 错误：name、amount

4. 布尔字段使用is_前缀
   - 正确：is_deleted、is_active
   - 错误：deleted、active
```

### 🔹 4.2.2 主键设计

```
   - 缺点：索引效率低

3. 使用雪花算法
   - 优点：分布式场景适用
   - 优点：插入性能好
   - 优点：索引效率高
```

### 🔹 4.2.3 字段设计

```
   - 优点：查询性能好
```

### 🔹 4.2.4 反范式化设计

**范式化与反范式化**：

```
范式化：
- 优点：减少数据冗余
- 优点：避免数据不一致
- 缺点：需要多表关联
- 缺点：查询性能差

反范式化：
- 优点：减少表关联
- 优点：查询性能好
- 缺点：数据冗余
- 缺点：可能数据不一致
```

**反范式化示例**：

```sql
-- 范式化设计
CREATE TABLE user (
    id BIGINT PRIMARY KEY,
    name VARCHAR(100) NOT NULL
);

CREATE TABLE orders (
    id BIGINT PRIMARY KEY,
    user_id BIGINT NOT NULL,
    order_no VARCHAR(50) NOT NULL,
    amount DECIMAL(10, 2) NOT NULL,
    create_time DATETIME NOT NULL,
    INDEX idx_user_id (user_id)
);

-- 查询订单和用户信息
SELECT o.*, u.name
FROM orders o
JOIN user u ON o.user_id = u.id;

-- 反范式化设计
CREATE TABLE orders (
    id BIGINT PRIMARY KEY,
    user_id BIGINT NOT NULL,
    user_name VARCHAR(100) NOT NULL,  -- 冗余用户姓名
    order_no VARCHAR(50) NOT NULL,
    amount DECIMAL(10, 2) NOT NULL,
    create_time DATETIME NOT NULL,
    INDEX idx_user_id (user_id)
);

-- 查询订单和用户信息
SELECT * FROM orders;

-- 性能提升：2-5倍
```

**反范式化原则**：

```
1. 冗余不经常变化的字段
   - 用户姓名
   - 商品名称
   - 分类名称

2. 冗余查询频繁的字段
   - 订单用户姓名
   - 订单商品名称
   - 订单分类名称

3. 冗余统计字段
   - 订单数量
   - 订单金额
   - 评论数量

4. 定期同步冗余数据
   - 使用定时任务
   - 使用消息队列
   - 使用触发器
```

## 4.3 分区表

### 🔹 4.3.1 分区表概述

**什么是分区表**：

```
分区表是将一个大表分成多个小表
逻辑上是一个表，物理上是多个表
可以提高查询性能，便于数据管理
```

**分区表的优势**：

```
1. 提高查询性能
   - 分区裁剪，只查询相关分区
   - 减少扫描的数据量
   - 提高查询速度

2. 便于数据管理
   - 可以单独删除分区
   - 可以单独备份分区
   - 可以单独归档分区

3. 提高并发性能
   - 不同分区可以并行查询
   - 提高并发能力
   - 提高系统吞吐量
```

**分区表的劣势**：

```
1. 增加管理成本
   - 需要设计分区策略
   - 需要维护分区
   - 需要监控分区

2. 限制较多
   - 主键必须包含分区键
   - 外键不支持
   - 全文索引不支持

3. 不适合所有场景
   - 小表不适合分区
   - 频繁跨分区查询不适合
   - 频繁更新分区键不适合
```

### 🔹 4.3.2 分区类型

**RANGE分区**：

```sql
-- 按时间范围分区
CREATE TABLE orders (
    id BIGINT,
    order_no VARCHAR(50),
    amount DECIMAL(10, 2),
    create_time DATETIME,
    PRIMARY KEY (id, create_time)
) PARTITION BY RANGE (YEAR(create_time)) (
    PARTITION p2020 VALUES LESS THAN (2021),
    PARTITION p2021 VALUES LESS THAN (2022),
    PARTITION p2022 VALUES LESS THAN (2023),
    PARTITION p2023 VALUES LESS THAN (2024),
    PARTITION pmax VALUES LESS THAN MAXVALUE
);

-- 查询2023年的订单
SELECT * FROM orders WHERE YEAR(create_time) = 2023;
-- 只查询p2023分区，性能提升10-100倍
```

**LIST分区**：

```sql
-- 按地区分区
CREATE TABLE orders (
    id BIGINT,
    order_no VARCHAR(50),
    amount DECIMAL(10, 2),
    region VARCHAR(20),
    PRIMARY KEY (id, region)
) PARTITION BY LIST (region) (
    PARTITION p_east VALUES IN ('北京', '上海', '江苏'),
    PARTITION p_south VALUES IN ('广东', '福建', '海南'),
    PARTITION p_west VALUES IN ('四川', '重庆', '云南'),
    PARTITION p_north VALUES IN ('北京', '天津', '河北')
);

-- 查询华东地区的订单
SELECT * FROM orders WHERE region IN ('北京', '上海', '江苏');
-- 只查询p_east分区，性能提升10-100倍
```

**HASH分区**：

```sql
-- 按用户ID哈希分区
CREATE TABLE orders (
    id BIGINT,
    user_id BIGINT,
    order_no VARCHAR(50),
    amount DECIMAL(10, 2),
    PRIMARY KEY (id, user_id)
) PARTITION BY HASH (user_id) PARTITIONS 10;

-- 查询指定用户的订单
SELECT * FROM orders WHERE user_id = 12345;
-- 只查询一个分区，性能提升10倍
```

## 4.4 分库分表

### 🔹 4.4.1 分库分表概述

**什么是分库分表**：

```
分库分表是将一个大表拆分成多个小表
分散到多个数据库中
可以提高性能，提高可用性
```

**分库分表的优势**：

```
1. 提高性能
   - 分散数据到多个数据库
   - 分散负载到多个服务器
   - 提高并发能力

2. 提高可用性
   - 单个数据库故障不影响整体
   - 提高系统可用性
   - 提高数据可靠性

3. 便于扩展
   - 可以水平扩展
   - 可以垂直扩展
   - 提高系统容量
```

**分库分表的劣势**：

```
1. 增加复杂度
   - 需要设计分片策略
   - 需要实现路由逻辑
   - 需要处理跨分片查询

2. 事务复杂
   - 分布式事务
   - 数据一致性
   - 事务回滚

3. 运维复杂
   - 需要管理多个数据库
   - 需要监控多个数据库
   - 需要备份多个数据库
```

### 🔹 4.4.2 分库分表策略

**水平分表**：

```
水平分表：将一个表的数据按照某个规则拆分到多个表中

分片规则：
1. 按范围分片
   - 按ID范围：1-100万在表1，100万-200万在表2
   - 按时间范围：2023年在表1，2024年在表2

2. 按哈希分片
   - 按ID哈希：id % 2 = 0在表1，id % 2 = 1在表2
   - 按用户ID哈希：user_id % 2 = 0在表1，user_id % 2 = 1在表2

3. 按地理位置分片
   - 按地区分片：华东在表1，华南在表2
   - 按省份分片：北京在表1，上海在表2
```

**垂直分表**：

```
垂直分表：将一个表的字段拆分到多个表中

分片规则：
1. 按字段使用频率分片
   - 高频字段在一个表
   - 低频字段在另一个表

2. 按字段大小分片
   - 大字段在一个表
   - 小字段在另一个表

3. 按业务逻辑分片
   - 核心字段在一个表
   - 扩展字段在另一个表
```

**水平分库**：

```
水平分库：将一个数据库的数据按照某个规则拆分到多个数据库中

分片规则：
1. 按范围分库
   - 按ID范围：1-100万在库1，100万-200万在库2
   - 按时间范围：2023年在库1，2024年在库2

2. 按哈希分库
   - 按ID哈希：id % 2 = 0在库1，id % 2 = 1在库2
   - 按用户ID哈希：user_id % 2 = 0在库1，user_id % 2 = 1在库2

3. 按地理位置分库
   - 按地区分库：华东在库1，华南在库2
   - 按省份分库：北京在库1，上海在库2
```

## 4.5 表结构优化案例

### 🔹 4.5.1 案例一：电商订单表优化

**优化前**：

```sql
CREATE TABLE orders (
    id VARCHAR(50) PRIMARY KEY,
    order_no VARCHAR(100),
    user_id VARCHAR(50),
    user_name VARCHAR(100),
    product_id VARCHAR(50),
    product_name VARCHAR(100),
    amount VARCHAR(20),
    status VARCHAR(20),
    create_time VARCHAR(20),
    update_time VARCHAR(20),
    remark TEXT
);

-- 问题：
-- 1. 主键使用VARCHAR
-- 2. 字段类型不合理
-- 3. 字段长度过大
-- 4. 缺少索引
-- 5. 没有分区
```

**优化后**：

```sql
CREATE TABLE orders (
    id BIGINT PRIMARY KEY,
    order_no VARCHAR(50) NOT NULL COMMENT '订单号',
    user_id BIGINT NOT NULL COMMENT '用户ID',
    user_name VARCHAR(100) NOT NULL COMMENT '用户姓名',
    product_id BIGINT NOT NULL COMMENT '商品ID',
    product_name VARCHAR(100) NOT NULL COMMENT '商品名称',
    amount DECIMAL(10, 2) NOT NULL COMMENT '订单金额',
    status TINYINT NOT NULL DEFAULT 0 COMMENT '订单状态',
    create_time DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
    update_time DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',
    remark TEXT COMMENT '备注',
    INDEX idx_user_id (user_id),
    INDEX idx_order_no (order_no),
    INDEX idx_create_time (create_time),
    INDEX idx_status (status)
) PARTITION BY RANGE (YEAR(create_time)) (
    PARTITION p2020 VALUES LESS THAN (2021),
    PARTITION p2021 VALUES LESS THAN (2022),
    PARTITION p2022 VALUES LESS THAN (2023),
    PARTITION p2023 VALUES LESS THAN (2024),
    PARTITION pmax VALUES LESS THAN MAXVALUE
);

-- 优化点：
-- 1. 主键使用BIGINT
-- 2. 字段类型合理
-- 3. 字段长度合理
-- 4. 添加索引
-- 5. 添加分区

-- 空间节省：40%
-- 性能提升：10-100倍
```

## 🎯 面试常见追问

| 面试官提问 | 回答思路 |
|------------|----------|
| 如何选择数据类型？ | 根据数据范围选择整数类型，金额使用DECIMAL，固定长度使用CHAR，可变长度使用VARCHAR，日期使用DATE，日期时间使用DATETIME |
| 表设计规范有哪些？ | 表名使用小写字母和下划线，字段名使用小写字母和下划线，主键使用BIGINT，字段尽量使用NOT NULL，字段设置默认值和注释 |
| 什么是反范式化？ | 反范式化是指为了提高查询性能，在表中冗余一些字段，减少表关联，但会增加数据冗余，需要定期同步冗余数据 |
| 什么是分区表？ | 分区表是将一个大表分成多个小表，逻辑上是一个表，物理上是多个表，可以提高查询性能，便于数据管理 |
| 分区表有哪些类型？ | RANGE分区（按范围）、LIST分区（按列表）、HASH分区（按哈希） |
| 什么是分库分表？ | 分库分表是将一个大表拆分成多个小表，分散到多个数据库中，可以提高性能，提高可用性 |
| 分库分表有哪些策略？ | 水平分表（按规则拆分数据）、垂直分表（按字段拆分）、水平分库（按规则拆分数据库） |
| 如何实现分库分表？ | 使用ShardingSphere、MyCAT等中间件实现分库分表，配置分片规则和数据源 |
