# 🧱 第1章：Celery 概述与核心思想

------

## **1.1 Celery 简介**

### 🧩 背景

- **Celery** 是 Python 生态中广泛使用的**分布式任务队列**，用于异步执行耗时任务（发邮件、报表、数据处理等），避免阻塞 Web 请求。
- 通常需要 **Broker**（如 Redis、RabbitMQ）传递任务消息，可选 **Backend**（如 Redis、数据库）存储结果。

------

### 🔹 核心概念

| 概念 | 说明 |
|------|------|
| **Broker** | 消息中间件，Worker 从其中取任务（Redis、RabbitMQ 等） |
| **Worker** | 执行任务的进程/节点 |
| **Backend** | 存储任务状态与结果，供查询（可选） |
| **Task** | 用 `@app.task` 装饰的函数，可异步调用 |
| **Beat** | 定时任务调度（类似 cron） |

------

## **1.2 Celery 与 Java MQ 的类比**

| Java 生态 | Celery |
|-----------|--------|
| RabbitMQ / Kafka（消息队列） | Broker（Redis/RabbitMQ） |
| 消费者 Worker | Celery Worker |
| 定时任务（Quartz、Spring @Scheduled） | Celery Beat |
| 异步解耦、削峰填谷 | 异步任务、定时任务 |

------

### 🧠 关键词总结

> Celery 是 **“Broker + Worker + Task”** 的分布式任务队列，适合异步任务、定时任务与解耦耗时逻辑，校招常考其与 Redis/RabbitMQ 的搭配。

------

### 📘 面试延展题

**问：Celery 选 Redis 还是 RabbitMQ 做 Broker？**

**答：** Redis 配置简单、部署方便，适合任务量不大、对可靠性要求不是极高的场景；RabbitMQ 消息持久化与可靠性更强，适合生产环境高可靠要求。若已有 Redis，可先用 Redis；大规模或关键业务可上 RabbitMQ。
