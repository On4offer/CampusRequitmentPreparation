# 📚 第1章：pip 与虚拟环境概述

## 1️⃣ 背景与痛点

在 Python 项目开发中，依赖管理是一个关键挑战：

- **依赖冲突**：不同项目可能需要不同版本的同一个库（如项目 A 需要 Django 3.2，项目 B 需要 Django 4.0）
- **环境污染**：全局安装依赖会影响系统 Python，可能破坏系统工具
- **可复现性差**：没有锁文件时，不同机器安装的依赖版本可能不一致
- **部署困难**：生产环境与开发环境依赖不一致导致部署失败

**示例场景**：
```bash
# 全局安装不同版本的库会导致冲突
pip install django==3.2  # 为项目 A 安装
pip install django==4.0  # 为项目 B 安装，会覆盖之前的版本
```

## 2️⃣ 核心概念

| 概念 | 含义 | 作用 |
|------|------|------|
| **pip** | Python 官方包管理工具 | 从 PyPI 下载并安装包，管理依赖版本 |
| **venv** | Python 3.3+ 内置虚拟环境模块 | 创建独立的 Python 解释器环境 |
| **virtualenv** | 第三方虚拟环境工具 | 功能类似 venv，支持更多 Python 版本 |
| **requirements.txt** | 依赖声明文件 | 记录项目依赖及版本，便于环境复现 |
| **pyproject.toml** | 现代项目配置文件 | PEP 518/621 标准，声明依赖、构建与元数据 |
| **site-packages** | 包安装目录 | 存储已安装的 Python 包 |

## 3️⃣ venv 与 virtualenv 对比

| 特性 | venv | virtualenv |
|------|------|------------|
| **来源** | 标准库（Python 3.3+） | 第三方工具 |
| **安装方式** | 无需额外安装 | `pip install virtualenv` |
| **创建速度** | 较快 | 略慢（功能更丰富） |
| **Python 版本** | 仅使用当前解释器 | 可指定不同 Python 版本 |
| **依赖隔离** | 完全隔离 | 完全隔离 |
| **企业使用** | 推荐，无额外依赖 | 仍有大量项目使用 |

**使用场景选择**：
- **venv**：Python 3.3+ 项目，追求简洁和标准
- **virtualenv**：需要跨 Python 版本，或项目有特殊需求

## 4️⃣ 依赖管理的工作原理

**依赖管理的核心流程**：
1. **环境隔离**：创建独立的虚拟环境
2. **依赖声明**：记录项目所需的包和版本
3. **依赖安装**：根据声明安装指定版本的包
4. **依赖锁定**：生成精确的依赖版本记录
5. **环境复现**：在其他机器上还原相同的环境

**依赖解析**：
- **直接依赖**：项目直接声明的包
- **间接依赖**：依赖包所依赖的其他包
- **版本约束**：指定包的版本范围（如 `>=1.0.0,<2.0.0`）

## 5️⃣ 与 Java Maven 的类比

| Java (Maven) | Python | 说明 |
|--------------|--------|------|
| `pom.xml` | `requirements.txt` / `pyproject.toml` | 依赖声明文件 |
| `mvn install` | `pip install -r requirements.txt` | 安装依赖 |
| `~/.m2/repository` | `venv/lib/site-packages` | 依赖存储目录 |
| `mvn dependency:tree` | `pip show <package>` | 查看依赖树 |
| 多模块项目 | 单项目多虚拟环境 | 模块化管理 |

## 6️⃣ 企业级最佳实践

- **强制使用虚拟环境**：每个项目使用独立的虚拟环境
- **使用固定版本**：在依赖文件中指定具体版本号
- **生成锁定文件**：使用 `pip freeze` 或现代工具生成锁定文件
- **定期更新依赖**：及时更新依赖以修复安全漏洞
- **CI/CD 集成**：在 CI 中自动化依赖安装和验证

## ⚠️ 常见坑与注意点

1. **现象**：全局安装依赖导致系统 Python 被污染。**原因**：直接使用 `pip install` 安装到系统 Python。**正确做法**：始终使用虚拟环境，或使用 `--user` 选项。

2. **现象**：虚拟环境激活失败。**原因**：路径错误或激活脚本未执行。**正确做法**：确保路径正确，使用正确的激活命令（Windows: `venv\Scripts\activate`, Linux/Mac: `source venv/bin/activate`）。

3. **现象**：依赖版本冲突。**原因**：不同包依赖同一库的不同版本。**正确做法**：使用锁定文件，或手动解决版本冲突。

4. **现象**：环境复现失败。**原因**：依赖文件未包含所有间接依赖。**正确做法**：使用 `pip freeze > requirements.txt` 生成完整依赖文件。

# 📚 第2章：venv 虚拟环境的创建与管理

## 1️⃣ 虚拟环境的创建

**基本语法**：
```bash
# 创建虚拟环境
python -m venv <虚拟环境名称>

# 示例：创建名为 venv 的虚拟环境
python -m venv venv
```

**参数说明**：
- `<虚拟环境名称>`：虚拟环境的目录名称，通常使用 `venv`、`.venv` 或 `env`
- `--system-site-packages`：继承系统 Python 的 site-packages（不推荐）
- `--without-pip`：不安装 pip（不推荐）

**创建过程**：
1. 新建目录（如 `venv`）
2. 复制 Python 解释器到目录中
3. 创建 `Lib/site-packages` 目录
4. 安装 pip 到虚拟环境

## 2️⃣ 虚拟环境的激活

### Windows 系统
```bash
# 激活虚拟环境
venv\Scripts\activate

# 验证激活状态
python --version
pip list
```

### Linux/Mac 系统
```bash
# 激活虚拟环境
source venv/bin/activate

# 验证激活状态
python --version
pip list
```

**激活效果**：
- 命令提示符前会显示虚拟环境名称（如 `(venv)`）
- `python` 和 `pip` 命令指向虚拟环境中的版本
- 安装的包会存储在虚拟环境的 `site-packages` 目录

## 3️⃣ 虚拟环境的退出

```bash
# 退出虚拟环境
deactivate
```

**退出效果**：
- 命令提示符恢复正常
- `python` 和 `pip` 命令恢复指向系统 Python

## 4️⃣ 虚拟环境的目录结构

**典型结构**：
```
venv/
├── Scripts/          # Windows 脚本目录
│   ├── activate      # 激活脚本
│   ├── activate.bat  # Windows 批处理激活脚本
│   ├── python.exe    # Python 解释器
│   └── pip.exe       # pip 工具
├── Lib/              # Python 库目录
│   └── site-packages/  # 包安装目录
└── pyvenv.cfg        # 虚拟环境配置文件
```

**pyvenv.cfg 配置文件**：
```
home = C:\\Python39  # 系统 Python 路径
executable = C:\\Python39\\python.exe
base-prefix = C:\\Python39
base-executable = C:\\Python39\\python.exe
```

## 5️⃣ 虚拟环境的管理

### 删除虚拟环境
```bash
# Windows
rd /s /q venv

# Linux/Mac
rm -rf venv
```

### 复制虚拟环境
```bash
# 方法1：创建新环境并重新安装依赖
python -m venv new_venv
new_venv\Scripts\activate
pip install -r requirements.txt

# 方法2：使用 virtualenv-clone（需要安装）
pip install virtualenv-clone
virtualenv-clone venv new_venv
```

### 升级虚拟环境
```bash
# 升级虚拟环境中的 Python 版本
# 注意：通常建议重新创建虚拟环境
python -m venv --upgrade venv
```

## 6️⃣ 虚拟环境的最佳实践

### 项目结构建议
```
project/
├── venv/            # 虚拟环境目录
├── requirements.txt  # 依赖文件
├── setup.py         # 项目配置
└── src/             # 源代码
```

### .gitignore 配置
```gitignore
# 虚拟环境目录
venv/
.venv/
env/

# IDE 配置
.vscode/
.idea/

# 依赖锁文件
Pipfile.lock
```

### 多项目管理
- **每个项目独立虚拟环境**：避免依赖冲突
- **虚拟环境目录名统一**：便于脚本和工具识别
- **使用环境变量**：通过 `VIRTUAL_ENV` 环境变量识别当前虚拟环境

## 7️⃣ 虚拟环境的使用场景

### 开发环境
```bash
# 创建虚拟环境
python -m venv venv

# 激活并安装依赖
venv\Scripts\activate
pip install -r requirements.txt

# 安装开发依赖
pip install pytest black flake8
```

### 测试环境
```bash
# 创建测试专用虚拟环境
python -m venv venv_test

# 激活并安装测试依赖
venv_test\Scripts\activate
pip install -r requirements-test.txt
```

### 生产环境
```bash
# 创建生产虚拟环境
python -m venv venv_prod

# 激活并安装生产依赖
venv_prod\Scripts\activate
pip install -r requirements-prod.txt
```

## ⚠️ 常见坑与注意点

1. **现象**：激活脚本执行失败。**原因**：路径错误或权限问题。**正确做法**：检查路径是否正确，确保有执行权限。

2. **现象**：虚拟环境中的 Python 版本与系统不一致。**原因**：创建虚拟环境时使用的 Python 版本不同。**正确做法**：使用指定版本的 Python 创建虚拟环境（如 `python3.8 -m venv venv`）。

3. **现象**：虚拟环境中安装的包不生效。**原因**：虚拟环境未激活或激活失败。**正确做法**：确保虚拟环境已正确激活，检查命令提示符前是否显示虚拟环境名称。

4. **现象**：虚拟环境体积过大。**原因**：安装了过多依赖或包含了不必要的包。**正确做法**：定期清理未使用的依赖，使用 `pip uninstall` 移除不需要的包。

5. **现象**：在 IDE 中无法识别虚拟环境。**原因**：IDE 未配置虚拟环境路径。**正确做法**：在 IDE 中设置 Python 解释器为虚拟环境中的 Python 可执行文件。

# 📚 第3章：pip 包管理详解

## 1️⃣ pip 的安装与升级

### 检查 pip 版本
```bash
# 检查 pip 版本
pip --version

# 检查 pip3 版本（Python 3）
pip3 --version
```

### 升级 pip
```bash
# 升级 pip
pip install --upgrade pip

# 或使用 python -m pip
python -m pip install --upgrade pip
```

### 安装 pip（如果未安装）
```bash
# 从官方脚本安装
curl https://bootstrap.pypa.io/get-pip.py -o get-pip.py
python get-pip.py

# 或使用包管理器（Linux）
sudo apt install python3-pip  # Ubuntu/Debian
sudo yum install python3-pip  # CentOS/RHEL
```

## 2️⃣ pip 的基本命令

### 安装包
```bash
# 安装最新版本
pip install <package>

# 安装指定版本
pip install <package>==<version>

# 安装版本范围
pip install "<package>>=1.0.0,<2.0.0"

# 从 requirements.txt 安装
pip install -r requirements.txt

# 安装开发版本（从 Git）
pip install git+https://github.com/username/repository.git
```

### 卸载包
```bash
# 卸载包
pip uninstall <package>

# 卸载多个包
pip uninstall <package1> <package2>

# 自动确认卸载
pip uninstall -y <package>
```

### 列出已安装的包
```bash
# 列出所有已安装的包
pip list

# 列出过期的包
pip list --outdated

# 以冻结格式列出（用于 requirements.txt）
pip freeze
```

### 查看包信息
```bash
# 查看包信息
pip show <package>

# 查看包的详细信息（包括依赖）
pip show -v <package>
```

### 搜索包
```bash
# 搜索 PyPI 上的包
pip search <keyword>

# 注意：pip search 已被弃用，建议使用 PyPI 网站或其他工具
```

## 3️⃣ 包的版本管理

### 版本号格式
- **主版本.次版本.修订号**（如 `1.2.3`）
- **预发布版本**：如 `1.2.3a1`（alpha）、`1.2.3b1`（beta）、`1.2.3rc1`（release candidate）
- **开发版本**：如 `1.2.3.dev1`

### 版本约束符
| 约束符 | 含义 | 示例 |
|--------|------|------|
| `==` | 等于 | `==1.2.3` |
| `>=` | 大于等于 | `>=1.2.3` |
| `<=` | 小于等于 | `<=1.2.3` |
| `>` | 大于 | `>1.2.3` |
| `<` | 小于 | `<1.2.3` |
| `~=` | 兼容版本 | `~=1.2.3` 相当于 `>=1.2.3,<1.3.0` |

### 升级包
```bash
# 升级指定包
pip install --upgrade <package>

# 升级所有过期的包
pip install --upgrade $(pip list --outdated | awk 'NR>2 {print $1}')
```

## 4️⃣ pip 的配置

### 配置文件位置
- **Windows**：`%APPDATA%\pip\pip.ini`
- **Linux/Mac**：`~/.config/pip/pip.conf` 或 `~/.pip/pip.conf`

### 常用配置选项
```ini
[global]
index-url = https://pypi.org/simple/
extra-index-url = https://pypi.python.org/simple/
trusted-host = pypi.org files.pythonhosted.org pypi.python.org
cache-dir = ~/.cache/pip
```

### 命令行配置
```bash
# 使用镜像源
pip install -i https://pypi.tuna.tsinghua.edu.cn/simple/ <package>

# 永久设置镜像源
pip config set global.index-url https://pypi.tuna.tsinghua.edu.cn/simple/
```

### 常用镜像源
- **清华大学**：`https://pypi.tuna.tsinghua.edu.cn/simple/`
- **阿里云**：`https://mirrors.aliyun.com/pypi/simple/`
- **豆瓣**：`https://pypi.douban.com/simple/`
- **华为云**：`https://mirrors.huaweicloud.com/repository/pypi/simple/`

## 5️⃣ pip 的高级功能

### 导出依赖
```bash
# 导出所有依赖（包括间接依赖）
pip freeze > requirements.txt

# 仅导出直接依赖
pip list --format=freeze | grep -E '^[^#].+' > requirements.txt
```

### 安装 editable 模式
```bash
# 以开发模式安装本地包
pip install -e .

# 这会创建一个符号链接，修改代码后无需重新安装
```

### 缓存管理
```bash
# 查看缓存目录
pip cache dir

# 清理缓存
pip cache purge

# 查看缓存信息
pip cache info
```

### 检查依赖冲突
```bash
# 检查依赖冲突
pip check

# 输出详细的依赖树
pip show -v <package>
```

## 6️⃣ 企业级 pip 使用最佳实践

### 私有 PyPI 源
- **搭建私有源**：使用 PyPI Server、Artifactory 或 Nexus
- **配置方法**：在 pip.conf 中设置 index-url

### 依赖审计
```bash
# 检查安全漏洞
pip install safety
safety check

# 或使用 pip-audit
pip install pip-audit
pip-audit
```

### 包的哈希校验
```bash
# 安装时验证哈希
pip install --require-hashes -r requirements.txt

# 生成包含哈希的 requirements.txt
```

## ⚠️ 常见坑与注意点

1. **现象**：pip 安装速度慢。**原因**：默认使用 PyPI 官方源，网络延迟高。**正确做法**：使用国内镜像源，如清华、阿里云镜像。

2. **现象**：pip 安装失败，提示权限错误。**原因**：没有写入权限或使用系统 Python。**正确做法**：使用虚拟环境，或添加 `--user` 选项。

3. **现象**：依赖冲突导致安装失败。**原因**：不同包依赖同一库的不同版本。**正确做法**：使用 `pip check` 检查冲突，手动指定兼容版本。

4. **现象**：pip 版本过低。**原因**：系统默认安装的 pip 版本较旧。**正确做法**：定期升级 pip 到最新版本。

5. **现象**：包安装后无法导入。**原因**：包未安装到当前环境或路径问题。**正确做法**：确保虚拟环境已激活，检查 Python 路径。

6. **现象**：pip 命令找不到。**原因**：pip 未添加到系统路径。**正确做法**：将 Python 的 Scripts 目录添加到 PATH 环境变量。

## 3️⃣ 版本约束的最佳实践

### 固定版本
- **适用场景**：生产环境、需要严格控制版本

- **缺点**：无法自动获得安全补丁

```

```

### 版本范围
- **适用场景**：开发环境、需要兼容多个版本
- **优点**：可以获得补丁和小版本升级
- **缺点**：可能引入不兼容变更

```
Django>=3.2.0,<4.0.0
requests>=2.28.0
```

### 兼容版本
- **适用场景**：希望获得小版本升级但避免大版本变更
- **优点**：平衡稳定性和安全性
- **缺点**：可能错过重要的新特性

```
Django~=3.2.0  # 相当于 >=3.2.0,<4.0.0
```

## ⚠️ 常见坑与注意点

# 📚 第5章：pyproject.toml 与现代依赖管理

## 1️⃣ pyproject.toml 的基本介绍

### 历史背景
- **PEP 518**：2018年提出，定义了构建系统接口
- **PEP 621**：2021年提出，标准化项目元数据
- **目的**：取代 setup.py 和 requirements.txt 的部分功能，提供统一的项目配置标准

### 核心功能
- **项目元数据**：名称、版本、作者、描述等
- **构建系统**：指定构建工具和依赖
- **依赖管理**：声明项目依赖
- **工具配置**：各种工具的配置（如 pytest、black 等）

## 2️⃣ pyproject.toml 的语法结构

### 基本结构
```toml
[build-system]
requires = ["setuptools>=61.0", "wheel"]
build-backend = "setuptools.build_meta"

[project]
name = "my-project"
version = "0.1.0"
authors = [
    { name = "Author Name", email = "author@example.com" }
]
description = "A sample Python project"
readme = "README.md"
requires-python = ">=3.8"
classifiers = [
    "Programming Language :: Python :: 3",
    "License :: OSI Approved :: MIT License",
    "Operating System :: OS Independent",
]

[project.dependencies]
requests = ">=2.28.0"
django = "~=3.2.0"

[project.optional-dependencies]
dev = [
    "pytest>=7.0.0",
    "black>=22.0.0",
]
test = [
    "pytest>=7.0.0",
    "pytest-cov>=3.0.0",
]

[tool.pytest.ini_options]
testpaths = ["tests"]
addopts = "-v"

[tool.black]
line-length = 88
target-version = ["py38"]
```

### 主要 sections
- **[build-system]**：构建系统配置
- **[project]**：项目元数据和依赖
- **[project.dependencies]**：核心依赖
- **[project.optional-dependencies]**：可选依赖（如开发、测试）
- **[tool.*]**：各种工具的配置

## 3️⃣ 依赖管理配置

1. **现象**：pyproject.toml 配置错误导致构建失败。**原因**：语法错误或字段缺失。**正确做法**：使用 `poetry check` 或 `flit check` 验证配置。

3. **现象**：Poetry 虚拟环境管理混乱。**原因**：多个项目使用不同版本的 Poetry。**正确做法**：使用 `poetry env use` 指定 Python 版本，或使用 `poetry config virtualenvs.in-project true`。

4. **现象**：pyproject.toml 与 setup.py 冲突。**原因**：同时存在两个配置文件。**正确做法**：优先使用 pyproject.toml，移除或简化 setup.py。

5. **现象**：依赖安装速度慢。**原因**：默认使用 PyPI 官方源。**正确做法**：在 pyproject.toml 中配置镜像源，或使用 `pip config` 设置全局镜像。

6. **现象**：可选依赖安装失败。**原因**：依赖名称错误或版本不兼容。**正确做法**：使用 `poetry add --extras` 或 `pip install "package[extra]"` 安装。

# 📚 第7章：CI/CD 与依赖管理

## 1️⃣ CI/CD 基本概念

### CI/CD 定义
- **CI（持续集成）**：频繁地将代码集成到主干分支，每次集成都自动运行测试，确保代码质量
- **CD（持续部署/交付）**：将集成后的代码自动部署到测试或生产环境
- **核心价值**：减少手动操作，提高代码质量，加快交付速度

### 常见 CI/CD 平台
- **GitHub Actions**：与 GitHub 无缝集成，支持多种操作系统
- **GitLab CI/CD**：集成在 GitLab 中，功能强大
- **Jenkins**：开源，高度可定制，支持丰富的插件
- **CircleCI**：云原生，配置简单，性能优异
- **Travis CI**：专注于 GitHub 项目，配置简洁

## 2️⃣ 依赖管理在 CI/CD 中的重要性

### 挑战与痛点
- **环境一致性**：确保 CI/CD 环境与本地开发环境一致
- **依赖速度**：依赖安装速度直接影响 CI/CD 流水线执行时间
- **依赖安全**：避免使用有安全漏洞的依赖
- **版本控制**：确保依赖版本的可追踪性和可复现性

### 核心原则
1. **可复现性**：使用锁定文件确保每次构建使用相同的依赖版本
2. **速度**：使用缓存加速依赖安装
3. **安全性**：定期扫描依赖的安全漏洞
4. **可靠性**：处理依赖安装失败的情况

## 3️⃣ GitHub Actions 配置示例

### 基本工作流
```yaml
# .github/workflows/ci.yml
name: CI

on:
  push:
    branches: [ main ]
  pull_request:
    branches: [ main ]

jobs:
  test:
    runs-on: ubuntu-latest
    strategy:
      matrix:
        python-version: [3.8, 3.9, '3.10']

    steps:
    - uses: actions/checkout@v3
    
    - name: Set up Python ${{ matrix.python-version }}
      uses: actions/setup-python@v4
      with:
        python-version: ${{ matrix.python-version }}
    
    - name: Cache dependencies
      uses: actions/cache@v3
      with:
        path: ~/.cache/pip
```

## 6️⃣ 依赖管理最佳实践

### 1. 使用依赖锁定文件
- **pip + requirements.txt**：使用 `pip freeze` 生成锁定文件
- **pip-tools**：使用 `pip-compile` 生成包含哈希的锁定文件
- **Poetry**：使用 `poetry.lock` 文件
- **PDM**：使用 `pdm.lock` 文件

### 2. 缓存策略
- **GitHub Actions**：使用 `actions/cache` 缓存 `~/.cache/pip` 和虚拟环境
- **GitLab CI**：使用 `cache` 关键字缓存依赖
- **Jenkins**：使用 `cache` 步骤或插件缓存依赖

### 3. 安全扫描
- **集成安全扫描工具**：在 CI/CD 流水线中添加 `safety` 或 `pip-audit`
- **定期扫描**：设置定时任务定期扫描依赖漏洞
- **自动更新**：使用 Dependabot 或类似工具自动更新有漏洞的依赖

### 4. 多环境管理
- **分离依赖文件**：为开发、测试、生产环境使用不同的依赖文件
- **环境变量**：使用环境变量控制依赖版本
- **容器化**：使用 Docker 容器确保环境一致性

### 5. 性能优化
- **并行安装**：使用 `pip install --parallel`（如果支持）
- **预构建依赖**：使用预构建的依赖镜像
- **增量安装**：只安装变更的依赖

## ⚠️ 常见坑与注意点

1. **现象**：CI/CD 流水线中依赖安装时间过长。**原因**：未配置缓存或缓存策略不当。**正确做法**：配置合理的缓存策略，使用预构建依赖。

2. **现象**：依赖版本在不同环境不一致。**原因**：未使用锁定文件或锁定文件未提交。**正确做法**：使用锁定文件并提交到版本控制系统。

3. **现象**：安全扫描工具提示依赖漏洞。**原因**：依赖版本过旧，存在已知漏洞。**正确做法**：及时更新依赖，或使用安全补丁。

4. **现象**：CI/CD 环境与本地环境不一致。**原因**：CI/CD 环境配置与本地不同。**正确做法**：使用容器化部署，确保环境一致性。

5. **现象**：依赖安装失败，提示网络错误。**原因**：CI/CD 环境网络连接问题或镜像源不可用。**正确做法**：配置可靠的镜像源，增加超时时间。

6. **现象**：构建过程中内存不足。**原因**：依赖安装过程中内存使用过高。**正确做法**：增加 CI/CD 环境内存，或优化依赖安装过程。
