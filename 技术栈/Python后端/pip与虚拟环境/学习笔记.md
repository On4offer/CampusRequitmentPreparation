# 🧱 第1章：pip 与虚拟环境概述

------

## **1.1 背景与痛点**

在 Python 项目开发中常见问题：

- **依赖冲突**：项目 A 需要 Django 3.2，项目 B 需要 Django 4.0，全局安装无法共存。
- **环境污染**：`pip install` 装到系统 Python，影响其他项目或系统工具。
- **可复现性差**：没有锁文件时，不同机器、不同时间安装的依赖版本可能不一致。

👉 **虚拟环境（Virtual Environment）** 与 **依赖管理（requirements / pyproject.toml）** 是 Python 后端工程化的基础。

------

## **1.2 核心概念**

| 概念 | 含义 |
|------|------|
| **pip** | Python 官方推荐的包安装与管理工具，从 PyPI 下载并安装包。 |
| **venv** | Python 3.3+ 内置的轻量虚拟环境模块，创建独立解释器与 site-packages。 |
| **virtualenv** | 第三方虚拟环境工具，功能与 venv 类似，兼容更多 Python 版本。 |
| **requirements.txt** | 记录项目依赖及版本，便于 `pip install -r` 复现环境。 |
| **pyproject.toml** | PEP 518/621 标准，现代项目用其声明依赖、构建与元数据（Poetry、Flit 等）。 |

------

## **1.3 venv 与 virtualenv 对比**

| 特性 | venv | virtualenv |
|------|------|------------|
| **来源** | 标准库（Python 3.3+） | 第三方 |
| **创建速度** | 较快 | 略慢（功能更多） |
| **跨版本** | 仅当前解释器 | 可指定不同 Python 版本 |
| **企业使用** | 推荐，无额外依赖 | 仍有大量项目使用 |

------

### 🧠 关键词总结

> pip 负责**装包**，venv/virtualenv 负责**隔离环境**，requirements.txt 或 pyproject.toml 负责**锁定依赖**，三者配合实现可复现的 Python 项目构建。

------

## **1.4 与 Java Maven 的类比**

| Java (Maven) | Python |
|--------------|--------|
| `pom.xml` 依赖声明 | `requirements.txt` / `pyproject.toml` |
| `mvn install` 拉取依赖 | `pip install -r requirements.txt` |
| 本地仓库 `~/.m2/repository` | 虚拟环境 `venv/lib/site-packages` 或 全局 `site-packages` |
| 多模块 parent POM | 单项目多 requirements 或 monorepo 多 venv |

------

### 📘 面试延展题

**问：为什么 Python 项目要使用虚拟环境？**

**答：** 隔离项目依赖，避免版本冲突；便于在不同机器或 CI 中复现相同环境；不污染系统 Python，保证系统工具（如 yum/apt 依赖的 Python）稳定。
