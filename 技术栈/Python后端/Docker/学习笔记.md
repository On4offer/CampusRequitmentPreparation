# 🧱 第1章：Python 项目与 Docker 概述

------

## **1.1 为什么 Python 项目要容器化**

### 🧩 背景

- **环境一致**：开发、测试、生产使用相同 Python 版本与依赖，避免「在我机器上能跑」。
- **依赖隔离**：镜像内自带虚拟环境或系统级依赖，不污染宿主机。
- **部署简单**：一次构建镜像，到处运行（含 CI/CD、云主机、K8s）。

------

### 🔹 Python 镜像常见选择

| 基础镜像 | 说明 | 适用 |
|----------|------|------|
| `python:3.11-slim` | 官方精简版，体积较小 | 生产推荐 |
| `python:3.11-alpine` | 更小，但部分 C 扩展需额外装依赖 | 对体积敏感 |
| `python:3.11` | 完整版，体积大 | 开发或需大量系统库时 |

------

## **1.2 Python Dockerfile 要点**

| 要点 | 说明 |
|------|------|
| **不把 venv 打进镜像** | 在镜像内用系统 Python 或单独创建 venv，避免路径与宿主机绑定 |
| **先装依赖再拷代码** | 利用层缓存，改代码时只需重算 COPY 及以后层 |
| **多阶段构建** | 构建阶段装编译工具，运行阶段只保留运行时，减小最终镜像 |
| **非 root 用户运行** | 安全起见，用普通用户起进程 |

------

### 🧠 关键词总结

> Python 项目容器化：选合适基础镜像、**依赖层与代码层分离**以利用缓存、生产用 slim 并考虑多阶段与非 root 运行。

------

### 📘 面试延展题

**问：Docker 里跑 Python 要不要用 venv？**

**答：** 容器本身已是隔离环境，理论上可直接用镜像内 Python；但部分团队仍习惯在镜像内建 venv，便于与本地开发习惯一致或多版本并存。若镜像内只有一个 Python，不用 venv 也可接受。
