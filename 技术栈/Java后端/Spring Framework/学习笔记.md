## 3️⃣ Spring 的核心思想

### （1）IoC（控制反转）

> **概念**：把对象的创建与依赖的管理交给容器，而不是在代码中手动 `new`。
>  **核心接口**：`BeanFactory`、`ApplicationContext`

**示例**：

```java
@Component
public class UserService {
    @Autowired
    private UserRepository userRepository;
}
```

👉 容器负责创建 `UserService` 和 `UserRepository`，并完成依赖注入。

**优势**：

- 降低代码耦合
- 提高可维护性与可测试性
- 支持热替换、易于扩展

------

### （2）DI（依赖注入）

> **Dependency Injection**：依赖对象由容器“注入”到当前类中。
>  **注入方式**：
>
> - 构造器注入（推荐）
> - Setter 注入
> - 字段注入（常用于简单项目）

**示例**：

```java
@Service
public class OrderService {
    private final PaymentService paymentService;
    public OrderService(PaymentService paymentService) {
        this.paymentService = paymentService;
    }
}
```

**面试要点**：

- IoC 是思想，DI 是实现。
- IoC 控制反转的是“对象的创建权”，DI 实现的是“依赖关系的注入”。

------

### （3）AOP（面向切面编程）

> **思想**：将“横切关注点”（如日志、安全、事务）从主业务逻辑中剥离出来。

**核心术语**：

| 名称      | 含义                                     |
| --------- | ---------------------------------------- |
| JoinPoint | 方法执行点                               |
| PointCut  | 切入点，定义要织入的规则                 |
| Advice    | 通知，定义执行的逻辑（前置、后置、环绕） |
| Aspect    | 切面，由切入点 + 通知组成                |
| Proxy     | 代理对象（JDK 动态代理或 CGLIB）         |

**示例**：

```java
@Aspect
@Component
public class LogAspect {
    @Before("execution(* com.example.service.*.*(..))")
    public void beforeLog() {
        System.out.println("方法调用前记录日志");
    }
}
```

**优点**：

- 解耦业务与非业务逻辑
- 提高代码复用率与可维护性

------

### （4）面向接口编程与解耦思想

> **本质**：通过接口定义行为规范，不依赖具体实现类。

**示例：**

```java
public interface MessageService {
    void send(String msg);
}
public class EmailService implements MessageService {
    public void send(String msg) { ... }
}
```

然后通过 IoC 注入接口：

```java
@Autowired
private MessageService messageService;
```

好处：

- 可替换实现类（如 Email → SMS）
- 提高可测试性（可注入 Mock 对象）

### （2）ApplicationContext 启动流程概述

```
① 解析配置源（XML / 注解 / JavaConfig）
② 加载 BeanDefinition（Bean 元信息）
③ 初始化 IoC 容器（DefaultListableBeanFactory）
④ 扫描组件并注册（ClassPathBeanDefinitionScanner）
⑤ 实例化 Bean（反射创建 + 注入依赖）
⑥ 执行 BeanPostProcessor（AOP织入、属性填充）
⑦ 发布事件，容器可用（ContextRefreshedEvent）
```

**源码入口示例（AnnotationConfigApplicationContext）**

```java
ApplicationContext ctx = new AnnotationConfigApplicationContext(AppConfig.class);
MyService myService = ctx.getBean(MyService.class);
```

> **关键类**：
>
> - `BeanFactory`：最基础的 Bean 容器
> - `ApplicationContext`：扩展版容器，支持事件、国际化、AOP
> - `BeanDefinitionReader`：解析配置文件并注册 Bean
> - `BeanPostProcessor`：在 Bean 初始化前后做增强（如 AOP、代理）

# 🌱 第二章：IoC 容器原理与 Bean 管理机制（核心）

------

## 1️⃣ IoC（控制反转）概念与原理

### （1）控制反转 vs 传统工厂模式

| 模式     | 传统工厂模式                 | 控制反转（IoC）                  |
| -------- | ---------------------------- | -------------------------------- |
| 定义     | 通过工厂类创建对象           | 容器控制对象的生命周期与依赖注入 |
| 对象创建 | 工厂负责创建对象并返回       | 容器负责创建对象并自动注入依赖   |
| 扩展性   | 需要手动编写工厂类，难以扩展 | 高度灵活，扩展性强               |
| 维护     | 工厂类与业务逻辑耦合紧密     | 通过容器解耦，易于维护           |

**通俗理解**：

- **传统工厂模式**：你自己用 `new` 来创建对象。
- **控制反转（IoC）**：你告诉容器怎么做，容器帮你创建和管理对象。

> 💡 **重点**：
>  IoC 是 Spring 框架的核心思想之一，Spring 容器通过控制对象的创建、管理和生命周期，避免了传统工厂模式的硬编码和高耦合。

------

### （2）IoC 的作用与优点

| 作用                     | 说明                                                         |
| ------------------------ | ------------------------------------------------------------ |
| **解耦**                 | 通过依赖注入（DI），降低类之间的耦合性，增强系统的灵活性     |
| **提高可测试性**         | 依赖注入使得组件之间独立，方便单元测试                       |
| **提升扩展性**           | 容器统一管理，容易扩展和替换组件                             |
| **自动管理对象生命周期** | IoC 容器负责对象的创建、初始化、销毁，开发者不用关心生命周期 |
| **简化配置**             | Spring 提供了 XML、注解、JavaConfig 等多种配置方式，极大简化了配置过程 |

------

## 2️⃣ BeanFactory 与 ApplicationContext 的区别

### （1）BeanFactory 延迟加载机制

- **BeanFactory** 是 Spring 的最基础容器，通常只会在调用 `getBean()` 方法时才创建 Bean，这种方式称为 **懒加载**（Lazy Initialization）。
- 适用于资源有限的环境，减少内存消耗。

**示例：**

```java
BeanFactory factory = new XmlBeanFactory(new ClassPathResource("beans.xml"));
MyBean myBean = factory.getBean(MyBean.class);
```

> 🔹 **延迟加载**的好处是：
>  对于单例 Bean，容器会延迟创建，只有在实际需要时才加载。

------

### （2）ApplicationContext 的扩展功能

- **ApplicationContext** 继承自 `BeanFactory`，是 Spring 的高级容器，提供了更强大的功能。
- 除了支持 BeanFactory 的基本功能外，`ApplicationContext` 还支持：
  - **事件发布**（例如：`ApplicationEvent`）
  - **国际化支持**（通过 `MessageSource`）
  - **AOP 集成**（通过 `ProxyFactoryBean` 等方式）

```java
ApplicationContext context = new AnnotationConfigApplicationContext(AppConfig.class);
context.publishEvent(new CustomEvent(this));
```

------

## 3️⃣ Bean 的生命周期

### （1）创建 → 属性注入 → 初始化 → 销毁

**流程概览**：

1. **创建**：Spring 容器根据 Bean 配置信息（通过反射）创建 Bean 实例。
2. **属性注入**：通过 DI（构造器、Setter 或字段注入）注入依赖对象。
3. **初始化**：
   - 调用 `@PostConstruct` 或实现 `InitializingBean.afterPropertiesSet()`。
   - 调用自定义的初始化方法。
4. **销毁**：
   - 调用 `@PreDestroy` 或实现 `DisposableBean.destroy()`。
   - 容器关闭时执行销毁操作。

### （2）生命周期回调接口

| 接口               | 作用                                  |
| ------------------ | ------------------------------------- |
| `InitializingBean` | 定义初始化方法 `afterPropertiesSet()` |
| `DisposableBean`   | 定义销毁方法 `destroy()`              |
| `@PostConstruct`   | 标记初始化方法                        |
| `@PreDestroy`      | 标记销毁方法                          |

**示例**：

```java
@Component
public class MyBean implements InitializingBean, DisposableBean {
    @Override
    public void afterPropertiesSet() throws Exception {
        System.out.println("初始化方法");
    }

    @Override
    public void destroy() throws Exception {
        System.out.println("销毁方法");
    }
}
```

------

### （3）BeanPostProcessor 扩展点

- **BeanPostProcessor** 是 Spring 提供的扩展接口，用于在 Bean 初始化前后进行自定义操作。

**使用场景**：可以通过该接口实现 AOP、日志、性能监控等功能。

**示例**：

```java
@Component
public class MyBeanPostProcessor implements BeanPostProcessor {
    @Override
    public Object postProcessBeforeInitialization(Object bean, String beanName) throws BeansException {
        System.out.println("初始化前处理：" + beanName);
        return bean;
    }

    @Override
    public Object postProcessAfterInitialization(Object bean, String beanName) throws BeansException {
        System.out.println("初始化后处理：" + beanName);
        return bean;
    }
}
```

## 4️⃣ BeanDefinition 解析流程

### （1）XML / 注解方式注册 Bean 的底层原理

- **XML 配置方式**：
   Spring 通过 `XmlBeanDefinitionReader` 类解析 XML 配置文件，将每个 `<bean>` 标签转化为 `BeanDefinition` 对象，描述了 Bean 的配置信息。
- **注解方式**：
   使用 `@ComponentScan` 自动扫描指定包下的类，找到带有 `@Component` 注解的类，将其注册为 Bean。

**底层原理**：
 Spring 在容器启动时会扫描 Bean 配置，并把它们封装成 `BeanDefinition` 对象，然后交给容器管理。

------

### （2）BeanDefinitionReader 与 BeanFactoryPostProcessor

- **BeanDefinitionReader** 负责读取配置并将 Bean 定义加载到容器中。
- **BeanFactoryPostProcessor** 可以修改容器中的 Bean 定义，在容器启动时进行干预。

**示例**：

```java
@Configuration
public class AppConfig implements BeanFactoryPostProcessor {
    @Override
    public void postProcessBeanFactory(ConfigurableListableBeanFactory beanFactory) throws BeansException {
        // 在此可以修改 Bean 定义
    }
}
```

------

### （3）refresh() 解析核心流程

1. **解析配置**：`BeanDefinitionReader` 读取 XML 文件或注解配置。
2. **加载 BeanDefinition**：将每个 Bean 转化为 `BeanDefinition`。
3. **注册 BeanDefinition**：将所有 BeanDefinition 放入 `BeanFactory` 中。
4. **初始化 Bean**：在调用 `getBean()` 时，Spring 会创建 Bean。

------

## 5️⃣ 依赖注入（DI）实现机制

### （1）构造函数注入 vs Setter 注入

| 类型             | 说明                             | 示例                        |
| ---------------- | -------------------------------- | --------------------------- |
| **构造函数注入** | 通过构造器传入依赖关系，强制依赖 | `@Autowired` on constructor |
| **Setter 注入**  | 通过 Setter 方法注入，灵活       | `@Autowired` on setter      |

**构造函数注入优点**：

- 强制依赖，保证必需的依赖不会为 null。

**Setter 注入优点**：

- 可选依赖，适合可选或可变依赖。

------

### （2）@Autowired、@Qualifier、@Resource 区别

| 注解           | 说明           | 适用场景                                     |
| -------------- | -------------- | -------------------------------------------- |
| **@Autowired** | 自动按类型注入 | 自动注入依赖                                 |
| **@Qualifier** | 指定 Bean 名称 | 配合 `@Autowired` 用于多个同类型 Bean 的选择 |
| **@Resource**  | 按名称自动注入 | 比 `@Autowired` 更依赖 Bean 名称             |

------

### （3）循环依赖的三层缓存解决机制

1. **singletonFactories**：记录正在创建的 BeanFactory，避免重复创建。
2. **earlySingletonObjects**：保存早期实例，用于解决 AOP 等代理问题。
3. **singletonObjects**：存放已完全初始化的单例 Bean。

> 💡 **三层缓存** 是为了解决 **单例 Bean 的循环依赖**，Spring 会利用这三层缓存机制保证 **对象不会因为依赖关系的循环**而导致死循环。

------

## 6️⃣ Bean 作用域与加载方式

### （1）singleton / prototype / request / session

| 作用域        | 说明                       |
| ------------- | -------------------------- |
| **singleton** | 整个应用中只有一个实例     |
| **prototype** | 每次请求都会创建一个新实例 |
| **request**   | 每个 HTTP 请求对应一个实例 |
| **session**   | 每个 HTTP 会话对应一个实例 |

------

### （2）懒加载（@Lazy）与预加载机制

- **@Lazy**：当 Bean 使用时才创建，避免容器启动时立即创建所有单例 Bean。

```java
@Lazy
@Bean
public MyBean myBean() {
    return new MyBean();
}
```

> **懒加载的好处**：
>  避免启动时加载所有 Bean，节省资源和时间。

------

## 7️⃣ Bean 的装配方式

### （1）XML 配置

```xml

```

### （2）注解装配

```java

```

### （3）基于 JavaConfig 的配置方式


## 1️⃣ AOP 概述

### （1）AOP 解决的问题与核心思想

- **AOP（Aspect-Oriented Programming）** 是一种编程范式，解决的是 **横切关注点** 的问题。
- **横切关注点**：与核心业务逻辑无关，但在多个地方都需要关注的功能，比如日志记录、安全校验、事务管理等。

#### 核心思想：

- **AOP** 允许你将横切关注点**从业务逻辑中分离出来**，通过 **切面（Aspect）** 的方式，在不修改源代码的情况下，自动在指定方法的前后加入额外功能。

**典型应用场景：**

- **日志记录**：不必每个方法都写日志，而是通过 AOP 自动执行。
- **事务管理**：自动开启、提交、回滚事务。
- **权限校验**：对特定方法或类自动进行权限检查。

------

### （2）动态代理与横切关注点

- **动态代理**：AOP 实现的核心技术之一，Spring 通过动态代理技术对方法进行增强。
- **横切关注点**：那些跨越多个模块或方法的功能，如事务管理、日志记录等。

**代理的方式：**

- **JDK 动态代理**：基于接口创建代理类，只能代理接口实现类的方法。
- **CGLIB 代理**：通过字节码增强生成目标类的子类，能够代理没有接口的类。

------

## 2️⃣ AOP 术语与结构

### （1）JoinPoint、Pointcut、Advice、Aspect、Weaving

| 术语          | 说明                                                 |
| ------------- | ---------------------------------------------------- |
| **JoinPoint** | 方法执行的时机（如方法调用时）                       |
| **Pointcut**  | 定义哪些方法需要增强，使用表达式进行匹配             |
| **Advice**    | 切面的具体操作（方法前、后执行的功能）               |
| **Aspect**    | 切面，由 Pointcut 和 Advice 组成                     |
| **Weaving**   | 切面与目标对象的绑定过程，发生在编译、类加载或运行时 |

------

### （2）Spring AOP vs AspectJ 对比

| 特性         | Spring AOP                           | AspectJ                          |
| ------------ | ------------------------------------ | -------------------------------- |
| **代理方式** | 只支持基于代理的方式（JDK 或 CGLIB） | 支持编译时、类加载时、运行时织入 |
| **注解支持** | 基于注解（如 `@Before`, `@After`）   | 完全基于注解                     |
| **功能范围** | 适用于方法级的切面                   | 支持方法、构造器、字段等各种切面 |
| **复杂度**   | 简单，适合大部分应用                 | 强大，适合复杂的 AOP 场景        |
| **性能**     | 较低，代理的性能开销较大             | 更高效                           |

## 3️⃣ Spring AOP 底层原理

### （1）JDK 动态代理实现

- **JDK 动态代理**：Spring AOP 默认通过 JDK 动态代理来创建代理对象，代理对象实现了目标类的接口，通过 `Proxy.newProxyInstance()` 创建。

**实现过程**：

1. 创建接口的代理类；
2. 代理类实现目标接口，并在每个方法调用时执行增强逻辑；
3. 执行目标方法并返回结果。

**示例**：

```java
public class MyInvocationHandler implements InvocationHandler {
    private Object target;

    public MyInvocationHandler(Object target) {
        this.target = target;
    }

    @Override
    public Object invoke(Object proxy, Method method, Object[] args) throws Throwable {
        System.out.println("Before method " + method.getName());
        Object result = method.invoke(target, args);
        System.out.println("After method " + method.getName());
        return result;
    }
}
```

------

### （2）CGLIB 字节码增强实现

- **CGLIB 代理**：CGLIB 是通过字节码增强的方式生成目标类的子类，能够代理没有实现接口的类。Spring 会通过 CGLIB 创建目标类的子类来进行代理。

**实现过程**：

1. 使用 CGLIB 动态生成目标类的子类；
2. 在子类中覆盖目标方法，加入增强逻辑；
3. 执行目标方法并返回结果。

**示例**：

```java
Enhancer enhancer = new Enhancer();
enhancer.setSuperclass(TargetClass.class);
enhancer.setCallback(new MyMethodInterceptor());
TargetClass proxy = (TargetClass) enhancer.create();
```

------

### （3）ProxyFactoryBean 与 AopProxy 接口

- **ProxyFactoryBean** 是 Spring 用来创建代理对象的工具类。
- **AopProxy** 是 Spring 提供的接口，`JdkDynamicAopProxy` 和 `CglibAopProxy` 都实现了该接口，负责执行代理操作。

------

## 4️⃣ 通知类型与执行顺序

### （1）@Before、@After、@AfterReturning、@AfterThrowing、@Around

| 通知类型            | 说明                                           |
| ------------------- | ---------------------------------------------- |
| **@Before**         | 在目标方法执行前执行                           |
| **@After**          | 在目标方法执行后执行，不论是否抛出异常         |
| **@AfterReturning** | 在目标方法成功执行后执行                       |
| **@AfterThrowing**  | 在目标方法抛出异常时执行                       |
| **@Around**         | 在目标方法前后都执行，可以控制是否执行目标方法 |

------

### （2）执行顺序与异常处理机制

- **执行顺序**：
   `@Before` → `@Around`（前置） → 执行目标方法 → `@Around`（后置） → `@AfterReturning` 或 `@After`（成功） → `@AfterThrowing`（异常）
- **异常处理机制**：
   AOP 提供了强大的异常处理机制，允许你在 `@AfterThrowing` 通知中捕获并处理异常。

**示例**：

```java
@Around("execution(* com.example.service.*.*(..))")
public Object around(ProceedingJoinPoint joinPoint) throws Throwable {
    try {
        return joinPoint.proceed(); // 执行目标方法
    } catch (Exception e) {
        System.out.println("Error: " + e.getMessage());
        throw e;
    }
}
```

## 5️⃣ 基于注解与基于 XML 的 AOP 配置

### （1）基于注解的 AOP 配置

Spring 支持使用注解来配置 AOP：

- **@EnableAspectJAutoProxy**：启用基于注解的 AOP 支持。
- **@Aspect**：标记切面类。
- **@Before**、`@After`、`@Around` 等：定义切点和通知。

**示例**：

```java
@EnableAspectJAutoProxy
@Configuration
public class AppConfig {}

@Aspect
@Component
public class LoggingAspect {

    @Before("execution(* com.example.service.*.*(..))")
    public void logBefore(JoinPoint joinPoint) {
        System.out.println("Method " + joinPoint.getSignature().getName() + " is about to be called.");
    }
}
```

------

### （2）基于 XML 的 AOP 配置

Spring 也支持基于 XML 的 AOP 配置：

- **[aop:config](aop:config)**：定义 AOP 配置。
- **[aop:aspect](aop:aspect)**：定义切面。
- **[aop:before](aop:before)**、**[aop:after](aop:after)**：定义通知。

**示例**：

```xml
<aop:config>
    <aop:aspect ref="loggingAspect">
        <aop:before method="logBefore" pointcut="execution(* com.example.service.*.*(..))" />
    </aop:aspect>
</aop:config>
```

## 1️⃣ Resource 抽象与加载机制

### （1）Resource、ResourceLoader、ResourcePatternResolver

Spring 提供了强大的 **资源加载** 支持，主要通过 **Resource** 接口、`ResourceLoader` 和 `ResourcePatternResolver` 来处理各种资源的读取。

| 组件                        | 说明                                                         |
| --------------------------- | ------------------------------------------------------------ |
| **Resource**                | Spring 资源的根接口，支持对文件、URL、Classpath 等资源的抽象管理。 |
| **ResourceLoader**          | 资源加载器接口，定义了 `getResource()` 方法，用于加载资源。  |
| **ResourcePatternResolver** | 扩展自 `ResourceLoader`，支持通过路径模式（如 Ant 风格的通配符）加载多个资源。 |

#### 典型实现类：

- **FileSystemResource**：读取文件系统中的资源。
- **ClassPathResource**：读取类路径中的资源。
- **UrlResource**：读取 URL 资源。
- **InputStreamResource**：读取流资源。

------

### （2）文件 / URL / Classpath 资源解析流程

Spring 会根据不同的资源类型，通过 `ResourceLoader` 加载资源。例如：

- **加载文件资源**：

```java
Resource resource = new FileSystemResource("/path/to/resource.txt");
```

- **加载类路径资源**：

```java
Resource resource = new ClassPathResource("com/example/config.properties");
```

- **加载 URL 资源**：

```java
Resource resource = new UrlResource("https://example.com/resource.txt");
```

**资源加载流程**：

1. **获取资源**：通过 `ResourceLoader.getResource()` 获取资源，底层根据资源路径判断加载类型（文件、URL、Classpath 等）。
2. **读取内容**：加载的资源可以通过 `InputStream` 读取其内容，如文件流或 URL 流。

**常见应用场景**：

- 配置文件加载
- 加载模板文件
- 静态文件访问

## 2️⃣ PropertySource 属性源机制

### （1）@PropertySource 加载外部配置文件

`@PropertySource` 注解用于加载外部配置文件（如 `.properties` 或 `.yml` 文件），使得文件中的属性能够注入到 Spring 环境中。

**示例：**

```java
@Configuration
@PropertySource("classpath:application.properties")
public class AppConfig {

}
```

- `@PropertySource` 通过指定文件路径来加载配置文件。
- 支持 `classpath:`、`file:` 等前缀来指定资源路径。

**多个文件加载：**

```java
@Configuration
@PropertySource({"classpath:config1.properties", "classpath:config2.properties"})
public class AppConfig {
    @Value("${property.value}")
    private String propertyValue;
}
```

### （2）Environment 与 PropertyResolver

- **Environment**：Spring 环境接口，提供了对外部配置（如属性源、系统属性、环境变量）的访问。
- **PropertyResolver**：`Environment` 接口的一个扩展，提供了读取属性值的功能。

#### 获取属性值：

```java
@Autowired
private Environment environment;

public void getProperty() {
    String value = environment.getProperty("property.key");
    System.out.println(value);
}
```

#### `@Value` 注解：

- 可以直接在 Bean 中使用 `@Value` 注解来注入属性值。
- 支持 SpEL 表达式，可以动态计算属性值。

```java
@Value("${app.name}")
private String appName;
```

## 3️⃣ Profile 环境隔离机制

### （1）@Profile 使用方式

`@Profile` 用于根据不同的环境配置不同的 Bean 实现。Spring 会根据当前激活的 profile 来选择性地加载 Bean。

#### 定义 Profile：

```java
@Configuration
@Profile("dev")
public class DevConfig {
    @Bean
    public DataSource dataSource() {
        return new HikariDataSource();  // 开发环境的数据库配置
    }
}

@Configuration
@Profile("prod")
public class ProdConfig {
    @Bean
    public DataSource dataSource() {
        return new DataSource();  // 生产环境的数据库配置
    }
}
```

#### 激活 Profile：

- **通过 `application.properties` 配置**：

```properties
spring.profiles.active=dev
```

- **通过命令行参数**：

```bash
-Dspring.profiles.active=prod
```

**应用场景**：

- 不同的数据库配置（开发环境与生产环境的数据库不同）
- 不同的日志级别（开发环境使用调试级别，生产环境使用信息级别）

------

### （2）多环境配置与自动切换

Spring 提供了多环境配置与自动切换的支持，基于 Profile 机制，Spring 能够在不同环境中切换不同的配置文件和 Bean 实现。

#### 示例：通过 `@Profile` 配置不同的数据源：

```java
@Configuration
public class DataSourceConfig {

    @Bean
    @Profile("dev")
    public DataSource devDataSource() {
        return new HikariDataSource(); // 开发环境数据库配置
    }

    @Bean
    @Profile("prod")
    public DataSource prodDataSource() {
        return new DataSource(); // 生产环境数据库配置
    }
}
```

**动态切换**：

- 配置 `application-dev.properties` 和 `application-prod.properties`，不同环境使用不同配置。

```properties
# application-dev.properties
spring.datasource.url=jdbc:h2:mem:testdb
# application-prod.properties
spring.datasource.url=jdbc:mysql://localhost:3306/prod_db
```

# 🌱 第五章：事件驱动模型（ApplicationEvent 机制）

------

## 1️⃣ ApplicationEvent 事件发布机制

### （1）Spring 事件驱动模型概述

Spring 提供了 **ApplicationEvent** 机制来实现事件驱动编程。这使得不同组件之间可以通过事件进行松耦合的通信。
 当某个事件发生时，系统通过 **事件发布机制** 将事件广播出去，所有感兴趣的监听器都会接收到这个事件并进行处理。

Spring 事件的核心是：

- **事件**：通过 `ApplicationEvent` 类及其子类定义。
- **事件发布**：通过 `ApplicationContext` 或 `ApplicationEventPublisher` 发布事件。
- **事件监听器**：通过 `ApplicationListener` 或 `@EventListener` 接收事件并响应。

------

### （2）事件发布过程

1. **事件类**：继承自 `ApplicationEvent`。
   - 定义事件的类型和内容。
2. **事件发布**：调用 `ApplicationContext.publishEvent()` 或 `ApplicationEventPublisher.publishEvent()`。
3. **监听器接收**：实现 `ApplicationListener` 接口或使用 `@EventListener` 注解接收事件。

**发布事件示例**：

```java
@Component
public class MyEventPublisher {

    @Autowired
    private ApplicationContext applicationContext;

    public void publishEvent() {
        MyEvent event = new MyEvent(this, "事件内容");
        applicationContext.publishEvent(event);
    }
}
```

------

## 2️⃣ 事件监听器 ApplicationListener

### （1）ApplicationListener 接口

`ApplicationListener` 是 Spring 提供的接口，用于监听事件并进行处理。
 实现该接口并注册到 Spring 容器中，容器会在事件发布时自动调用监听器的 `onApplicationEvent()` 方法。

**示例**：

```java
@Component
public class MyEventListener implements ApplicationListener<MyEvent> {

    @Override
    public void onApplicationEvent(MyEvent event) {
        System.out.println("接收到事件: " + event.getMessage());
    }
}
```

- `onApplicationEvent()` 方法会在事件发布时被调用，处理接收到的事件。

### （2）泛型支持

`ApplicationListener` 支持泛型，可以根据事件的类型来限定监听的事件类型。

```java
public class MyEventListener implements ApplicationListener<MyEvent> {
    @Override
    public void onApplicationEvent(MyEvent event) {
        // 处理事件
    }
}
```

### （3）事件监听器注册方式

- **自动注册**：使用 `@Component` 注解，Spring 会自动扫描并注册监听器。
- **手动注册**：可以通过 `AnnotationConfigApplicationContext` 或 XML 配置文件注册监听器。

```java
ApplicationContext context = new AnnotationConfigApplicationContext(MyConfig.class);
context.addApplicationListener(new MyEventListener());
```

------

## 3️⃣ 自定义事件发布与监听

### （1）定义自定义事件

Spring 提供了 `ApplicationEvent` 类作为事件的基类。
 可以通过继承该类，定义自己的事件。

**示例**：

```java
public class MyEvent extends ApplicationEvent {
    
    private String message;

    public MyEvent(Object source, String message) {
        super(source);
        this.message = message;
    }

    public String getMessage() {
        return message;
    }
}
```

- **source**：事件源对象，表示事件的发起者。
- **message**：自定义的事件内容。

### （2）发布自定义事件

事件发布者使用 `ApplicationContext.publishEvent()` 方法发布自定义事件。

```java
@Component
public class MyEventPublisher {
    
    @Autowired
    private ApplicationContext context;

    public void publishCustomEvent() {
        MyEvent event = new MyEvent(this, "Custom event message");
        context.publishEvent(event);
    }
}
```

### （3）监听自定义事件

使用 `ApplicationListener` 接口或 `@EventListener` 注解来监听自定义事件。

**方式一：实现 ApplicationListener 接口**：

```java
@Component
public class MyEventListener implements ApplicationListener<MyEvent> {

    @Override
    public void onApplicationEvent(MyEvent event) {
        System.out.println("Received event: " + event.getMessage());
    }
}
```

**方式二：使用 @EventListener 注解**：

```java
@Component
public class MyEventListener {

    @EventListener
    public void handleMyEvent(MyEvent event) {
        System.out.println("Received event: " + event.getMessage());
    }
}
```

- `@EventListener` 注解使得方法能够响应指定类型的事件。

## 4️⃣ Spring 事件传播的线程安全与异步处理

### （1）事件传播的线程安全

Spring 的事件传播是 **线程安全** 的，事件的发布和监听机制是基于单线程处理的。
 但是，在实际应用中，有时我们希望事件能够异步处理，以避免阻塞主线程。

### （2）异步事件处理

Spring 提供了 **异步事件处理** 的支持，可以通过配置实现事件处理的异步化，避免事件处理过程中阻塞主线程。

#### 启用异步处理

1. 在 Spring 配置类中添加 `@EnableAsync` 注解启用异步处理：

```java
@Configuration
@EnableAsync
public class AppConfig {
    // 配置类
}
```

1. 在事件监听器方法上使用 `@Async` 注解：

```java
@Component
public class MyAsyncEventListener {

    @EventListener
    @Async
    public void handleMyEventAsync(MyEvent event) {
        // 异步处理事件
        System.out.println("Processing event asynchronously: " + event.getMessage());
    }
}
```

- 使用 `@Async` 注解后，事件处理方法会在新的线程中异步执行。

------

### （3）异步事件处理的应用场景

- **减少主线程压力**：如果事件的处理不依赖于主流程，可以将其异步执行，减轻主线程的负担。
- **提高性能**：例如，发送通知、记录日志等操作不需要阻塞主业务逻辑，应该异步执行。
- **提高用户体验**：在 Web 应用中，可以将时间消耗较长的操作（如邮件发送、数据同步）移到异步线程中执行。

------

## 5️⃣ 总结

| 知识点                  | 面试关键词            | 实际应用                   |
| ----------------------- | --------------------- | -------------------------- |
| **ApplicationEvent**    | 事件发布机制、事件源  | 事件驱动编程，解耦组件     |
| **ApplicationListener** | 事件监听器、事件处理  | 监听和处理特定事件         |
| **@EventListener**      | 注解方式监听事件      | 更简洁、灵活的事件监听方式 |
| **事件异步处理**        | @Async、线程安全      | 提高性能、减少主线程阻塞   |
| **自定义事件**          | MyEvent、自定义事件类 | 实现业务相关的事件处理     |

------

**学习要点**：

- Spring 通过 `ApplicationEvent` 和 `ApplicationListener` 提供了松耦合的事件发布与监听机制。
- `@EventListener` 注解使得事件监听更加简洁和灵活。

# 🌱 第六章：Spring 事务管理（TX 模块）

------

## 1️⃣ 事务的概念与 ACID 特性

### （1）事务的概念

- **事务**：是对一组操作的集合，它们要么完全成功，要么完全失败，不会处于半成功半失败的中间状态。
- 事务是数据库管理系统（DBMS）的一种重要特性，保证数据的一致性和可靠性。

### （2）ACID 特性

ACID 是事务的四个基本属性：

| 特性                      | 说明                                                         |
| ------------------------- | ------------------------------------------------------------ |
| **原子性（Atomicity）**   | 事务中的所有操作要么全部执行成功，要么全部失败回滚。         |
| **一致性（Consistency）** | 事务执行前后，数据从一个一致的状态转换到另一个一致的状态。   |
| **隔离性（Isolation）**   | 一个事务的执行不受其他事务的干扰，确保事务间的独立性。       |
| **持久性（Durability）**  | 一旦事务提交，其对数据库的更改将是永久性的，系统崩溃也不会丢失。 |

------

## 2️⃣ Spring 声明式事务原理

### （1）@Transactional 注解解析

- `@Transactional` 是 Spring 中声明式事务的核心注解。
- 通过在方法上加上 `@Transactional` 注解，Spring 会自动管理该方法的事务，而无需编写显式的事务控制代码。

#### 使用示例：

```java
@Service
public class UserService {

    @Transactional
    public void transferMoney(Long fromAccountId, Long toAccountId, Double amount) {
        // 执行转账操作
    }
}
```

Spring 会自动为 `transferMoney()` 方法开启事务，并在方法执行完毕后提交或回滚事务。

### （2）事务传播行为（7 种传播类型）

事务传播行为定义了事务在不同方法之间的传播规则，Spring 提供了 7 种传播行为：

| 传播行为          | 说明                                                         |
| ----------------- | ------------------------------------------------------------ |
| **REQUIRED**      | 如果当前存在事务，加入当前事务；如果没有事务，则创建一个新的事务（默认行为）。 |
| **REQUIRES_NEW**  | 总是创建一个新的事务，如果当前存在事务，则挂起当前事务。     |
| **SUPPORTS**      | 如果当前存在事务，加入当前事务；如果没有事务，则以非事务方式执行。 |
| **NOT_SUPPORTED** | 如果当前存在事务，挂起当前事务；如果没有事务，则以非事务方式执行。 |
| **MANDATORY**     | 如果当前存在事务，加入当前事务；如果没有事务，则抛出异常。（adj. 强制的） |
| **NEVER**         | 如果当前存在事务，则抛出异常；如果没有事务，则以非事务方式执行。 |
| **NESTED**        | 如果当前存在事务，则创建一个嵌套事务；如果没有事务，则类似于 `REQUIRED`。 |

**示例：**

```java
@Transactional(propagation = Propagation.REQUIRES_NEW)
public void transferMoney() {
    // 事务逻辑
}
```

------

### （3）隔离级别与超时设置

- **事务隔离级别**：定义了不同事务之间如何相互隔离。Spring 提供了以下 4 种隔离级别：
  - **READ_UNCOMMITTED**：允许读取未提交的数据，可能会导致脏读。
  - **READ_COMMITTED**：只能读取已提交的数据，避免脏读，但可能会出现不可重复读。
  - **REPEATABLE_READ**：确保在事务期间，读取的数据是可重复的，但可能会导致幻读。
  - **SERIALIZABLE**：最高的隔离级别，完全串行化的执行事务，避免脏读、不可重复读和幻读，但性能最差。
- **事务超时设置**：指定事务的最大执行时间，超时会自动回滚。

**示例**：

```java
@Transactional(isolation = Isolation.REPEATABLE_READ, timeout = 30)
public void transferMoney() {
    // 事务逻辑
}
```

- **timeout**：设置事务的超时时间，单位为秒，超过指定时间自动回滚事务。

------

## 3️⃣ 编程式事务管理

### （1）编程式事务管理概述

编程式事务管理是通过显式代码来控制事务的开始、提交和回滚。Spring 提供了 `PlatformTransactionManager` 接口和 `TransactionTemplate` 类来实现编程式事务。

#### 使用 `PlatformTransactionManager` 管理事务：

```java
@Autowired
private PlatformTransactionManager transactionManager;

public void transferMoney() {
    // 创建事务定义
    DefaultTransactionDefinition def = new DefaultTransactionDefinition();
    def.setIsolationLevel(TransactionDefinition.ISOLATION_READ_COMMITTED);
    def.setPropagationBehavior(TransactionDefinition.PROPAGATION_REQUIRED);

    // 开始事务
    TransactionStatus status = transactionManager.getTransaction(def);
    try {
        // 执行事务操作
        // commit 操作
        transactionManager.commit(status);
    } catch (Exception e) {
        // 出现异常回滚事务
        transactionManager.rollback(status);
    }
}
```

- **TransactionStatus**：表示当前事务的状态。
- `transactionManager.getTransaction(def)`：获取当前事务或开始一个新的事务。
- `transactionManager.commit(status)`：提交事务。
- `transactionManager.rollback(status)`：回滚事务。

## 5️⃣ 事务失效的常见原因

### （1）同类中方法调用

当一个类中的方法通过 `@Transactional` 注解时，如果该方法内部调用了同一个类中的另一个 `@Transactional` 方法，Spring 默认情况下不会新开事务，而是会直接在当前事务中执行，导致事务失效。

**解决方案**：将内部方法提取到其他类中，或使用 AOP 的代理方式来避免直接调用。

### （2）private 方法

如果 `@Transactional` 注解的方法是 `private`，那么 Spring 事务管理器无法为其创建代理，事务也不会生效。

**解决方案**：确保被事务管理的方法是 `public` 或至少是 `protected`。

### （3）异常未抛出

Spring 默认只会对 `RuntimeException` 或 `Error` 进行回滚，而对 `CheckedException`（非运行时异常）不会自动回滚。如果方法中抛出了 `CheckedException`，事务会提交，而不是回滚。

**解决方案**：配置 `@Transactional` 注解的 `rollbackFor` 属性，显式指定需要回滚的异常类型。

```java
@Transactional(rollbackFor = SQLException.class)
public void transferMoney() throws SQLException {
    // 事务逻辑
}
```

## 1️⃣ JdbcTemplate 使用与原理

### （1）JdbcTemplate 概述

`JdbcTemplate` 是 Spring 提供的一个简化 JDBC 操作的工具类。
 它封装了大量常见的 JDBC 操作（如建立连接、执行查询、释放资源等），并提供了更加简洁和一致的 API，使得开发者能以更少的代码完成数据库操作。

#### JdbcTemplate 的功能：

- 自动管理数据库连接。
- 简化 SQL 执行和结果映射。
- 支持事务管理和错误处理。

### （2）JdbcTemplate 使用示例

**1. 配置 JdbcTemplate：**

```java
@Bean
public DataSource dataSource() {
    return new DriverManagerDataSource("jdbc:mysql://localhost:3306/mydb", "user", "password");
}

@Bean
public JdbcTemplate jdbcTemplate(DataSource dataSource) {
    return new JdbcTemplate(dataSource);
}
```

**2. 执行 SQL 查询：**

```java
@Autowired
private JdbcTemplate jdbcTemplate;

public void queryData() {
    String sql = "SELECT * FROM users WHERE id = ?";
    User user = jdbcTemplate.queryForObject(sql, new Object[]{1}, new BeanPropertyRowMapper<>(User.class));
    System.out.println(user);
}
```

**3. 执行 SQL 更新：**

```java
public int updateData() {
    String sql = "UPDATE users SET name = ? WHERE id = ?";
    return jdbcTemplate.update(sql, "John", 1);
}
```

**4. 执行批量操作：**

```java
public void batchInsert() {
    String sql = "INSERT INTO users (name, age) VALUES (?, ?)";
    List<Object[]> batchArgs = Arrays.asList(
            new Object[]{"John", 25},
            new Object[]{"Jane", 30}
    );
    jdbcTemplate.batchUpdate(sql, batchArgs);
}
```

### （3）JdbcTemplate 的原理

- `JdbcTemplate` 通过 **JDBC 模板设计模式**，将数据库操作的模板部分封装起来，开发者只需提供 SQL 和参数即可，剩下的事务管理、连接释放等由 Spring 处理。
- 它内部使用 `Connection`、`Statement`、`ResultSet` 来执行 SQL 查询，并通过回调接口 (`PreparedStatementSetter`、`ResultSetExtractor` 等) 来提取查询结果。

## 4️⃣ 事务传播与数据库连接管理

### （1）事务传播行为

Spring 提供了 7 种事务传播行为，以控制多个事务方法的执行方式：

- **REQUIRED**：当前有事务，则加入当前事务；否则，创建一个新事务。

**示例**：

```java
@Transactional(propagation = Propagation.REQUIRES_NEW)
public void transferMoney() {
```

# 🌱 第八章：Spring 模块扩展机制

## 1️⃣ BeanFactoryPostProcessor 与 BeanPostProcessor

### （1）扩展点执行时机与作用

Spring 提供了两个重要的扩展点：**BeanFactoryPostProcessor** 和 **BeanPostProcessor**，它们分别在不同的生命周期阶段介入 Bean 的创建与初始化过程。

#### **BeanFactoryPostProcessor**

- **作用**：主要用于在 Bean 定义信息加载后、Bean 实例化之前修改 Bean 配置（如修改 `BeanDefinition`）。
- **执行时机**：在 Spring 容器启动时，所有的 `BeanFactoryPostProcessor` 会在 Bean 实例化之前执行。

**常见用途**：

- 修改 Bean 的属性。
- 修改 Bean 的作用域、依赖关系等配置。

**示例**：

```java
@Component
public class CustomBeanFactoryPostProcessor implements BeanFactoryPostProcessor {

    @Override
    public void postProcessBeanFactory(ConfigurableListableBeanFactory beanFactory) throws BeansException {
        BeanDefinition beanDefinition = beanFactory.getBeanDefinition("myBean");
        beanDefinition.setScope("prototype");  // 修改 Bean 的作用域
    }
}
```

#### **BeanPostProcessor**

- **作用**：在 Bean 实例化后、初始化前和初始化后进行回调，用于对 Bean 进行处理。
- **执行时机**：Bean 实例化后、属性注入后，但在调用 `@PostConstruct` 或初始化方法之前执行（`@PostConstruct`、`InitializingBean`）。

**常见用途**：

- 对 Bean 的属性进行动态修改。
- 为 Bean 增加 AOP 功能。
- 在初始化前后执行某些逻辑。

**示例**：

```java
@Component
public class CustomBeanPostProcessor implements BeanPostProcessor {

    @Override
    public Object postProcessBeforeInitialization(Object bean, String beanName) throws BeansException {
        if (bean instanceof MyBean) {
            System.out.println("Before initialization: " + beanName);
        }
        return bean;  // 必须返回 Bean 对象
    }

    @Override
    public Object postProcessAfterInitialization(Object bean, String beanName) throws BeansException {
        if (bean instanceof MyBean) {
            System.out.println("After initialization: " + beanName);
        }
        return bean;  // 必须返回 Bean 对象
    }
}
```

### （2）@Configuration 与代理增强机制

- **@Configuration**：当一个类被标注为 `@Configuration` 时，Spring 会自动为该类创建代理，并将其作为配置类。
- **代理机制**：`@Configuration` 的类会通过 **CGLIB 代理** 增强，确保配置方法（如 `@Bean` 注解的方法）能正确执行并且能被 Spring 容器识别。

#### 代理的作用：

- **方法增强**：`@Configuration` 的方法会被 CGLIB 代理，使得方法在执行时能触发代理类的增强逻辑。

**示例**：

```java
@Configuration
public class AppConfig {

    @Bean
    public MyBean myBean() {
        return new MyBean();
    }
}
```

Spring 会为 `AppConfig` 类创建代理，确保 `myBean()` 方法可以被调用并正确注册为 Bean。

## 2️⃣ FactoryBean 与自动代理机制

### （1）FactoryBean

`FactoryBean` 是 Spring 提供的一个用于创建 Bean 的工厂接口。
 通过实现 `FactoryBean` 接口，开发者可以控制 Bean 的创建过程，甚至可以返回一个代理对象。

#### 使用方式：

- **接口定义**：

```java
public interface FactoryBean<T> {
    T getObject() throws Exception;
    Class<?> getObjectType();
    boolean isSingleton();
}
```

#### 典型使用场景：

- **自定义 Bean 创建**：例如，通过 `FactoryBean` 返回一个动态生成的 Bean，或一个代理 Bean。

**示例**：

```java
@Component
public class MyBeanFactory implements FactoryBean<MyBean> {

    @Override
    public MyBean getObject() throws Exception {
        return new MyBean();  // 自定义 Bean 创建逻辑
    }

    @Override
    public Class<?> getObjectType() {
        return MyBean.class;
    }

    @Override
    public boolean isSingleton() {
        return true;  // 该 FactoryBean 返回的对象是单例
    }
}
```

#### 配置使用：

- `FactoryBean` 注册到 Spring 容器后，Spring 会自动调用 `getObject()` 方法来获取实际的 Bean 实例。

## 3️⃣ 自定义注解 + ImportSelector 实现自动装配

### （1）自定义注解

Spring 提供了强大的注解支持，利用 `@Import` 和 `ImportSelector`，可以通过自定义注解实现自动装配。

#### 使用 `@Import` 注解：

`@Import` 可以引入其他配置类或 Bean 定义。

```java
@Target(ElementType.TYPE)
@Retention(RetentionPolicy.RUNTIME)
@Import(MyBeanImportSelector.class)
public @interface EnableMyFeature {
}
```

### （2）ImportSelector 实现自动装配

`ImportSelector` 接口用于选择要导入的配置类或 Bean。

```java
public class MyBeanImportSelector implements ImportSelector {

    @Override
    public String[] selectImports(AnnotationMetadata importingClassMetadata) {
        return new String[] { "com.example.MyBean" };  // 返回要导入的类
    }
}
```

**使用示例**：

```java
@EnableMyFeature  // 激活自动装配
public class MyApplication {
}
```

当 `@EnableMyFeature` 注解应用于类上时，`MyBeanImportSelector` 会根据逻辑自动导入 `MyBean` 配置。

### （1）@EnableAsync

## 1️⃣ 从 `AnnotationConfigApplicationContext` 启动讲起

`AnnotationConfigApplicationContext` 是 Spring 提供的一个用于基于注解配置的 `ApplicationContext` 实现。
 它是 Spring 容器的入口之一，主要用于基于 Java 配置类初始化 Spring 容器。

### 启动过程简述：

1. **创建 `AnnotationConfigApplicationContext` 实例**：
    `AnnotationConfigApplicationContext` 通过反射和配置类的注册，初始化 Spring 容器。

2. **传入配置类**：
    在创建 `AnnotationConfigApplicationContext` 时，可以传入一个或多个配置类，Spring 容器会扫描并加载这些类中定义的 Bean。

   ```java
   AnnotationConfigApplicationContext context = new AnnotationConfigApplicationContext(AppConfig.class);
   ```

3. **启动容器**：
    在调用 `refresh()` 方法时，Spring 容器正式启动并加载 Bean、触发初始化过程。

------

## 2️⃣ `refresh()` 方法核心阶段解析

`refresh()` 方法是 Spring 容器启动的关键方法，包含多个核心阶段。
 该方法的调用顺序和每个阶段的作用如下：

### （1）prepareRefresh()

- **作用**：准备容器刷新，初始化一些基本设置（如当前时间戳、容器状态）。
- **关键点**：通过 `prepareRefresh()`，Spring 设置了容器的启动时间、环境信息，并标记容器为“刷新中”状态。

**源码**：

```java
protected void prepareRefresh() {
    // 初始化容器状态
    this.startTime = System.currentTimeMillis();
    this.environment = createEnvironment();
    this.propertySources = this.environment.getPropertySources();
    this.beanFactory = createBeanFactory();
    this.beanFactory.setBeanClassLoader(getClassLoader());
    this.beanFactory.setEnvironment(this.environment);
}
```

------

### （2）obtainFreshBeanFactory()

- **作用**：创建或获取新的 `BeanFactory`，并进行初始化。
   这个阶段会实例化容器的 `BeanFactory`，并加载相关配置。
- **关键点**：Spring 默认使用 `DefaultListableBeanFactory` 作为 BeanFactory，它是 Spring 最常用的 Bean 工厂实现，负责管理 Bean 的定义与实例。

**源码**：

```java
protected final void obtainFreshBeanFactory() {
    DefaultListableBeanFactory beanFactory = createBeanFactory();
    beanFactory.setSerializationId(getId());
    this.beanFactory = beanFactory;
}
```

------

### （3）invokeBeanFactoryPostProcessors()

- **作用**：执行所有注册到容器中的 `BeanFactoryPostProcessor`，用于修改 `BeanFactory` 中的 Bean 定义。
   例如，`@PropertySource` 配置的 `BeanFactoryPostProcessor` 会在此阶段进行处理。
- **关键点**：此阶段调用 `BeanFactoryPostProcessor` 实现的 `postProcessBeanFactory()` 方法，允许开发者修改或增强 Bean 的定义。

**源码**：

```java
protected void invokeBeanFactoryPostProcessors() {
    // 遍历并执行 BeanFactoryPostProcessor
    for (BeanFactoryPostProcessor postProcessor : getBeanFactoryPostProcessors()) {
        postProcessor.postProcessBeanFactory(this.beanFactory);
    }
}
```

------

### （4）registerBeanPostProcessors()

- **作用**：注册 `BeanPostProcessor`，这些处理器会在 Bean 初始化前后进行自定义处理。
   `BeanPostProcessor` 是 Spring AOP、事务等功能的基础。
- **关键点**：`BeanPostProcessor` 用于实现 Bean 的初始化增强，比如 AOP 的织入、初始化后的回调等。

**源码**：

```java
protected void registerBeanPostProcessors() {
    // 注册 BeanPostProcessor
    for (String beanName : this.beanFactory.getBeanDefinitionNames()) {
        if (this.beanFactory.isTypeMatch(beanName, BeanPostProcessor.class)) {
            this.beanFactory.addBeanPostProcessor(beanName);
        }
    }
}
```

------

### （5）onRefresh() → finishRefresh()

- **作用**：执行容器的进一步初始化工作。
   通过 `onRefresh()` 方法，Spring 会进行一些自定义初始化操作，如 Bean 的初始化、事件监听器的注册等。
   `finishRefresh()` 方法则是容器启动的最后一步，表示容器完全刷新并可以开始提供服务。

**源码**：

```java
protected void onRefresh() {
    // 执行刷新后的操作，例如注册事件监听器
    registerListeners();
}

protected void finishRefresh() {
    // 容器初始化完成，标记容器为已初始化
    this.finished = true;
    publishEvent(new ContextRefreshedEvent(this));
}
```

## 3️⃣ 容器启动全过程时序图

以下是 Spring 容器启动的主要时序步骤：

```plaintext
+-------------------+        +------------------------+
| AnnotationConfig  |        | prepareRefresh()        |
| ApplicationContext|        | - 初始化容器状态       |
+--------+----------+        +------------------------+
         |
         v
+-------------------------+  +--------------------------+
| obtainFreshBeanFactory() |  | invokeBeanFactoryPostProcessors() |
| - 创建 DefaultListableBeanFactory | - 执行 BeanFactoryPostProcessor |
+-------------------------+  +--------------------------+
         |
         v
+--------------------------+    +--------------------------+
| registerBeanPostProcessors()|  | onRefresh()               |
| - 注册 BeanPostProcessor  |   | - 注册事件监听器         |
+--------------------------+    +--------------------------+
         |
         v
+--------------------------+
| finishRefresh()           |
| - 容器初始化完成，开始服务|
+--------------------------+
```

**关键节点**：

- **prepareRefresh()**：初始化容器。
- **obtainFreshBeanFactory()**：获取 BeanFactory 并初始化。
- **invokeBeanFactoryPostProcessors()**：执行 Bean 定义修改操作。
- **registerBeanPostProcessors()**：注册 AOP 和事件监听器等增强功能。
- **onRefresh()**：进一步初始化，注册事件监听器等。
- **finishRefresh()**：标志容器启动完成，发布 `ContextRefreshedEvent`。

## 4️⃣ 常见扩展接口在启动流程中的执行顺序

### （1）BeanFactoryPostProcessor

- **执行时机**：在 `refresh()` 方法中，`invokeBeanFactoryPostProcessors()` 阶段执行。
- **作用**：修改 Bean 定义，通常用于动态修改 Bean 配置或引入外部配置。

### （2）BeanPostProcessor

- **执行时机**：在 `refresh()` 方法中，`registerBeanPostProcessors()` 阶段执行。
- **作用**：对 Bean 进行初始化前后处理，最常见的用途是 AOP 和事务的集成。

### （3）ApplicationListener

- **执行时机**：在 `onRefresh()` 方法中，`registerListeners()` 阶段执行。
- **作用**：注册事件监听器，Spring 容器中的所有事件监听器会在该阶段注册。

### （4）@Configuration 配置类的代理增强

- **执行时机**：在容器初始化的过程中，`@Configuration` 注解的类会通过 CGLIB 生成代理，以确保 `@Bean` 方法能够正确执行并且保持单例。

## **第十章：Spring 与 Spring Boot 的关系**

### 10.1 Spring Boot 对 Spring 的封装与增强

- Spring Boot 基于 Spring Framework，提供**自动装配**、**内嵌容器**、**起步依赖（Starter）**、**外部化配置**，减少 XML/Java 配置量。
- 核心仍为 IoC 容器（ApplicationContext）、AOP、事务等，只是默认配置与约定更完善。

### 10.2 @SpringBootApplication 底层原理

- `@SpringBootApplication` 是组合注解：`@SpringBootConfiguration`（等价 @Configuration）+ `@EnableAutoConfiguration` + `@ComponentScan`。
- `@EnableAutoConfiguration` 通过 `SpringFactoriesLoader` 加载 `META-INF/spring.factories` 中的自动配置类，按条件（@Conditional）决定是否生效。

### 10.3 SpringFactoriesLoader 与自动装配

- **SpringFactoriesLoader**：从 classpath 下所有 jar 的 `META-INF/spring.factories` 读取 key 为 `EnableAutoConfiguration` 的配置类全限定名，统一注册。
- 自动装配类通常带 `@ConditionalOnClass`、`@ConditionalOnProperty` 等，只有条件满足才创建 Bean，实现“引入依赖即生效”。

### 10.4 Spring Boot 的优缺点与适用场景

- **优点**：开箱即用、与微服务/云原生生态结合好、社区活跃。
- **缺点**：封装多、问题排查需理解自动配置；大型单体可考虑按需禁用部分自动配置。
- **适用**：新项目、微服务、快速交付；对极致轻量或老项目渐进迁移可继续用纯 Spring。

## **第十一章：Spring 高级特性与扩展**

### 11.1 异步与任务调度

- **@Async**：在配置类上启用 `@EnableAsync`，在方法上加 `@Async`，该方法在独立线程池中执行；返回值可用 `Future` 或 `CompletableFuture`。
- **@Scheduled**：在配置类上启用 `@EnableScheduling`，在方法上加 `@Scheduled(cron = "0 0 2 * * ?")` 等，实现定时任务。

### 11.2 国际化（MessageSource）

- 通过 `MessageSource` 根据 Locale 获取文案，配合 `ResourceBundleMessageSource` 或 `ReloadableResourceBundleMessageSource` 加载 properties。

### 11.3 校验机制（JSR-303 + Validator）

- 使用 `javax.validation`（或 `jakarta.validation`）的 `@NotNull`、`@Size`、`@Valid` 等注解；在 Controller 方法参数上加 `@Valid` 或 `@Validated`，由 Spring 调用 Validator 做校验并返回 BindingResult。

### 11.4 事件监听与 ApplicationEventPublisher

- 自定义事件继承 `ApplicationEvent`，通过 `ApplicationEventPublisher.publishEvent()` 发布；监听器实现 `ApplicationListener<MyEvent>` 或用 `@EventListener`；可配合 `@Async` 实现异步监听。

## **第十二章：Spring 性能与内存优化**

### 12.1 Bean 加载优化

- 非必要 Bean 使用 `@Lazy` 延迟初始化；按模块拆分 `@Configuration`，避免单次扫描过多类。
- 排除不需要的自动配置类（`exclude` / `excludeName`），减少无用 Bean 与后处理器。

### 12.2 延迟初始化与按需加载

- `@Lazy` 使 Bean 在首次被注入或获取时才创建；在大型应用中可降低启动时间和内存占用。

### 12.3 循环依赖优化与排查

- Spring 通过**三级缓存**解决单例 Bean 的 setter/字段注入循环依赖；构造器注入的循环依赖无法解决，需改为 setter 或拆出中间 Bean。排查时看启动报错中的 Bean 名称链。

### 12.4 大型项目启动优化

- 减少 `@ComponentScan` 范围；使用 `spring.main.lazy-initialization=true` 全局延迟初始化（慎用，可能掩盖配置错误）；合理设置线程池与连接池，避免启动时过多连接。

## **第十三章：Spring 常见问题与面试高频题**

| 问题 | 回答要点 |
|------|----------|
| IoC 容器初始化流程？ | 加载配置/扫描 → 解析 BeanDefinition → 实例化 Bean → 依赖注入 → 初始化回调（BeanPostProcessor）→ 容器就绪；核心在 `refresh()`。 |
| BeanFactory 与 ApplicationContext？ | BeanFactory 是基础容器、延迟加载；ApplicationContext 继承并扩展，支持事件、国际化、资源加载，默认预加载单例 Bean。 |
| AOP 底层实现？ | 基于动态代理：有接口用 JDK 动态代理，无接口用 CGLIB；通过 ProxyFactory 创建代理，在调用前后织入 Advice。 |
| @Transactional 不生效？ | 同类内部调用、方法非 public、异常被吞未抛出、未走代理（未通过 Spring 管理的 Bean 调用）等；需通过代理对象调用且异常对事务可见。 |
| 如何解决循环依赖？ | 单例 + setter/字段注入时，三级缓存暴露早期引用；构造器循环依赖无法解决，需改设计。 |
| 常见设计模式？ | 单例（Bean 默认）、工厂（BeanFactory）、模板方法（JdbcTemplate）、代理（AOP）、观察者（事件）。 |
| @Component / @Bean / @Import？ | @Component 声明组件被扫描；@Bean 在配置类中显式定义 Bean；@Import 导入配置类或 ImportSelector 做条件装配。 |
| 线程安全？ | 无状态 Bean 单例本身线程安全；有状态需用 ThreadLocal 或改为 prototype/每次 new。 |

## **第十四章：Spring 在项目中的实战应用**

- **分层与解耦**：Controller 只做参数校验与转发，Service 写业务，Mapper/Repository 做数据访问；通过接口注入，便于测试与替换实现。
- **拦截器 + ThreadLocal**：在拦截器中解析 token 将用户信息放入 ThreadLocal，在 Service 层按需获取，请求结束清理，实现无侵入鉴权。
- **声明式缓存**：使用 `@Cacheable`、`@CacheEvict`、`@CachePut` 配合 CacheManager（如 Redis），将缓存从业务代码中剥离。
- **定时任务**：用 `@Scheduled` 做对账、统计、过期数据清理等；注意单机重复执行问题，可配合分布式锁或调度中心。
- **配置分离**：数据库、Redis、开关等放在 `application-{profile}.yml` 或配置中心，通过 `@Value`、`@ConfigurationProperties` 注入，避免硬编码。
