## 1. 什么是 Spring MVC

### 1.1 MVC 设计模式简介

**MVC** 是一种经典的软件设计模式，将应用分为三层：

| 组件                     | 全称   | 职责                                                   |
| ------------------------ | ------ | ------------------------------------------------------ |
| **Model（模型）**        | 数据层 | 封装业务数据与逻辑，如实体类、Service、DAO 等。        |
| **View（视图）**         | 展示层 | 负责将数据渲染为用户可见的界面（如 HTML、JSON、XML）。 |
| **Controller（控制器）** | 控制层 | 负责接收请求、调用业务逻辑、返回结果。                 |

> ✅ 优点：解耦视图与业务逻辑，便于维护和扩展。
>  🧠 思考：MVC 不仅限于 Web，也可用于桌面、移动端架构设计。

------

### 1.2 Spring MVC 的作用与特点

Spring MVC 是 **Spring Framework 提供的基于 MVC 模式的 Web 框架**。
 它简化了 Servlet 的开发流程，实现了从请求到响应的全链路控制。

**主要特点：**

1. 基于 **Servlet** 架构，兼容性强；
2. **注解驱动**（如 `@Controller`、`@RequestMapping`）；
3. 支持 **RESTful 风格接口**；
4. 支持 **数据绑定与参数校验**；
5. 支持 **国际化与异常处理机制**；
6. 与 Spring 容器深度整合，**天然支持 IoC 与 AOP**。

------

### 1.3 Spring MVC 与 Servlet 的关系

| 项目     | Servlet                  | Spring MVC                        |
| -------- | ------------------------ | --------------------------------- |
| 核心思想 | 面向请求响应（低层 API） | 基于 MVC 模式的封装               |
| 入口     | `HttpServlet`            | `DispatcherServlet`（前端控制器） |
| 配置方式 | web.xml 手动注册         | Spring Boot 自动配置              |
| 参数处理 | 手动解析请求参数         | 自动绑定到对象                    |
| 响应处理 | 需手动拼接 HTML/JSON     | 可返回视图名或对象（自动序列化）  |

> 💡 **结论**：
>  Spring MVC 的本质仍是基于 Servlet，它的核心控制器 DispatcherServlet 本身继承自 `HttpServlet`。
>  它是对 Servlet 模型的高度抽象与解耦。

## 2. Spring MVC 的工作流程总览

### 2.1 核心组件结构图

```
客户端（Browser）
       │
       ▼
[DispatcherServlet] —— 前端控制器（统一入口）
       │
       ├── [HandlerMapping] —— 负责查找处理器映射
       │
       ├── [HandlerAdapter] —— 负责执行 Controller 方法
       │
       ├── [Controller] —— 业务控制器
       │
       ├── [ModelAndView] —— 返回数据和视图名
       │
       └── [ViewResolver] —— 视图解析器（如 Thymeleaf、FreeMarker）
       ▼
   返回视图或 JSON 响应
```

------

### 2.2 核心组件职责详解

| 组件                  | 职责                                     | 举例                                               |
| --------------------- | ---------------------------------------- | -------------------------------------------------- |
| **DispatcherServlet** | 前端控制器，接收所有请求，调度其他组件。 | 统一入口，类似总控中枢。                           |
| **HandlerMapping**    | 查找哪个控制器方法能处理该请求。         | 依据 `@RequestMapping("/user/list")`。             |
| **HandlerAdapter**    | 调用具体的控制器方法。                   | 负责参数封装、方法调用。                           |
| **Controller**        | 控制层核心逻辑。                         | 如 `UserController.list()`。                       |
| **ModelAndView**      | 封装模型数据和视图信息。                 | 既可返回页面，也可返回 JSON。                      |
| **ViewResolver**      | 将视图名解析成实际视图对象。             | 如解析 `"userList"` → `/templates/userList.html`。 |

------

### 2.3 请求处理时序流程（重要）

一次典型的 HTTP 请求在 Spring MVC 中的处理流程如下：

#### 🌐 Step-by-step 时序图

```
[Client]
   |
   | ① 发送请求 /user/list
   ▼
[DispatcherServlet]
   |
   | ② 查找对应的 Handler（通过 HandlerMapping）
   ▼
[HandlerExecutionChain]
   |
   | ③ 调用对应的 Controller（由 HandlerAdapter 适配）
   ▼
[Controller Method]
   |
   | ④ 执行业务逻辑，返回 ModelAndView 或数据对象
   ▼
[ViewResolver]
   |
   | ⑤ 解析视图名（或返回 JSON 数据）
   ▼
[DispatcherServlet]
   |
   | ⑥ 响应结果返回给客户端
   ▼
[Client]
```

#### 🌟 简化记忆口诀：

> **前端控制器收请求 → 查找映射器 → 执行控制器 → 视图解析器出结果。**

------

### 2.4 典型实战示例

```java
@Controller
@RequestMapping("/user")
public class UserController {

    @GetMapping("/list")
    @ResponseBody
    public List<User> list() {
        // 模拟业务逻辑
        return userService.findAll();
    }
}
```

**解释：**

- `@Controller`：标识该类是一个控制器组件；
- `@RequestMapping("/user")`：指定访问路径；
- `@ResponseBody`：返回的数据自动序列化为 JSON；
- 由 Spring MVC 自动完成参数绑定、类型转换、异常处理。

## 3. 与其他框架的对比

### 3.1 Spring MVC vs Struts2

| 对比项     | Spring MVC         | Struts2                    |
| ---------- | ------------------ | -------------------------- |
| 架构模式   | 基于 Servlet       | 基于 Filter                |
| 控制器对象 | 单例（性能高）     | 多例（每次请求创建新对象） |
| 参数绑定   | 自动封装到方法参数 | 通过 ValueStack 封装       |
| 拦截机制   | HandlerInterceptor | 拦截器链                   |
| 性能       | 高（轻量、单例）   | 相对低（对象频繁创建）     |
| 配置       | 注解 + JavaConfig  | XML 配置繁琐               |
| 生态整合   | 与 Spring 天然融合 | 需额外整合配置             |

> ✅ **总结**：Spring MVC 是 Struts2 的进化版本，更轻量、更高效、与 Spring 集成更自然。

------

### 3.2 Spring MVC vs Spring WebFlux

| 对比项   | Spring MVC                   | Spring WebFlux                       |
| -------- | ---------------------------- | ------------------------------------ |
| 编程模型 | 阻塞式（Servlet API）        | 非阻塞式（Reactive Streams）         |
| 底层容器 | Tomcat、Jetty 等             | Netty、Undertow                      |
| 响应方式 | 同步线程阻塞                 | 响应式流（异步）                     |
| 适用场景 | 传统 Web、管理后台           | 高并发、实时流式系统                 |
| 常用注解 | @Controller、@RestController | @Controller、@RestController（相同） |
| 性能     | 中等（依赖线程池）           | 高（非阻塞 I/O）                     |

> 💬 面试延伸：
>
> - **问：Spring MVC 能否处理 10 万并发请求？**
>    答：传统 MVC 是线程阻塞模型（一个请求一个线程），并发上限取决于线程池大小。WebFlux 则使用 Reactor 模型（事件驱动、非阻塞），更适合高并发 IO 场景。

## ⚠️ 常见坑与注意点

| 问题场景                | 原因                                        | 解决方案                                         |
| ----------------------- | ------------------------------------------- | ------------------------------------------------ |

## 第二章：Spring MVC 的核心组件

### 2.1 DispatcherServlet（前端控制器）

- **职责**：统一接收 HTTP 请求，委托 HandlerMapping 查找处理器、HandlerAdapter 执行处理器，再通过 ViewResolver 解析视图或直接写响应。
- **初始化**：在 web.xml 中配置 servlet 与 url-pattern（如 `/`），或 Spring Boot 中由自动配置注册；启动时加载 Spring 容器（根容器 + Web 容器），完成 Bean 与组件初始化。

### 2.2 HandlerMapping（处理器映射器）

- 根据请求 URL、Method 等将请求映射到对应的 **Handler**（通常是 `@RequestMapping` 方法）；常用实现为 `RequestMappingHandlerMapping`，维护注解解析后的映射表。

### 2.3 HandlerAdapter（处理器适配器）

- 负责**调用** Handler：参数解析（HandlerMethodArgumentResolver）、执行方法、返回值处理（HandlerMethodReturnValueHandler）；`RequestMappingHandlerAdapter` 是注解驱动下的默认适配器。

### 2.4 Controller 与 @Controller / @RestController

- **@Controller**：标识控制器，方法返回值默认作为视图名，需配合 ViewResolver；若方法上加 **@ResponseBody** 则直接写响应体。
- **@RestController** = @Controller + @ResponseBody，类下所有方法默认返回 JSON 等响应体，用于 RESTful API。

### 2.5 Model、ModelMap、ModelAndView

- **Model / ModelMap**：向视图传递数据的键值容器，可作方法参数由框架注入。
- **ModelAndView**：同时封装视图名与 Model，可一次返回；前后端分离时多用 @ResponseBody 返回对象，无需视图名。

### 2.6 ViewResolver（视图解析器）

- 将逻辑视图名解析为实际 View（如 HTML 模板、JSON）；常见有 `InternalResourceViewResolver`（JSP）、ThymeleafViewResolver；返回 @ResponseBody 或 ResponseEntity 时不经过 ViewResolver，由 HttpMessageConverter 写 body。

## 第三章：Spring MVC 核心注解与请求映射

### 3.1 控制层与请求映射

- **@RequestMapping**：可标注类与方法，指定 path（value）、method、params、headers 等；**@GetMapping / @PostMapping / @PutMapping / @DeleteMapping** 为 method 的简写。
- **@Controller**：视图层控制器；**@RestController**：类级别等价于“所有方法带 @ResponseBody”，用于接口层。

### 3.2 参数绑定

- **@RequestParam**：绑定查询参数或表单字段，可指定 name、required、defaultValue。
- **@PathVariable**：绑定 URI 模板变量，如 `/user/{id}` 中 id。
- **@RequestBody**：将请求体反序列化为对象（通常 JSON），依赖 HttpMessageConverter。
- **@RequestHeader**：绑定请求头。**@CookieValue**：绑定 Cookie。

### 3.3 返回值处理

- **@ResponseBody**：方法返回值由 HttpMessageConverter 序列化后写入响应体（如 JSON）。
- **ResponseEntity<T>**：可设置状态码、响应头、body，适合需精确控制响应的场景。

### 3.4 实战要点

- GET 参数用 @RequestParam 或 POJO 直接接收（属性名与参数名一致）；POST JSON 用 @RequestBody；REST 风格路径用 @PathVariable；文件上传用 MultipartFile 参数。

------

## 第四章：数据绑定与类型转换机制

### 4.1 数据绑定流程

- **WebDataBinder**：将请求参数绑定到方法参数或命令对象；内置类型转换（String→Integer、日期等），可注册自定义 Converter 或 Formatter。

### 4.2 自定义类型转换器

- 实现 **Converter<S, T>** 或 **Formatter<T>**，在配置类中通过 **WebMvcConfigurer.addFormatters** 或 **@InitBinder**（局部）注册。

### 4.3 表单与复杂类型绑定

- 支持嵌套对象（如 order.items）、日期、枚举、集合；日期格式可通过 @DateTimeFormat 或全局配置统一。

### 4.4 数据校验（JSR-303 / JSR-380）

- 在 DTO 上使用 **@NotNull、@Size、@Email、@Valid** 等；Controller 方法参数加 **@Valid** 或 **@Validated** 触发校验，校验结果进入 **BindingResult**；与全局异常处理配合可统一返回 400 与错误信息。

------

## 第五章：模型数据传递与作用域管理

### 5.1 Model、ModelMap、ModelAndView

- 方法参数中声明 **Model** 或 **ModelMap**，向其中 put 数据，返回视图名即可在模板中访问；**ModelAndView** 同时设置 viewName 与 model，适合需要明确指定视图时。

### 5.2 Session 与 @SessionAttributes

- **@SessionAttributes(names)**：将 Model 中指定 name 的数据放入 Session；用 **SessionStatus.setComplete()** 清除本次 Session 中由该注解管理的属性。前后端分离下更多用 token 存登录态，少用 Session。

### 5.3 统一响应封装（Result）

- 前后端分离常用：定义 **Result<T>**（code、msg、data），Controller 统一返回 `Result.success(data)` 或 `Result.fail(code, msg)`；配合 **@ControllerAdvice** 全局异常处理，将异常也封装为 Result。

------

## 第六章：拦截器（HandlerInterceptor）

### 6.1 作用与接口

- **HandlerInterceptor**：在 Handler 执行前后插入逻辑，用于**登录校验、权限、日志、耗时统计**等；三个方法：**preHandle**（返回 false 则中断）、**postHandle**（视图渲染前）、**afterCompletion**（请求结束后，可做清理）。

### 6.2 配置与执行顺序

- 通过 **WebMvcConfigurer.addInterceptors** 注册，指定 **addPathPatterns / excludePathPatterns**；多个拦截器按注册顺序执行 preHandle，逆序执行 postHandle 与 afterCompletion。

### 6.3 与 Filter、AOP 区别

- **Filter**：Servlet 规范，最早执行，可修改请求/响应；**Interceptor**：Spring MVC 层，可访问 Handler 与 Model；**AOP**：可切 Controller 方法，但拿不到 HttpServletRequest 等，一般请求前后用拦截器更合适。登录态常放在拦截器里解析 token 并写入 ThreadLocal。

------

## 第七章：异常处理机制

### 7.1 局部与全局异常

- **@ExceptionHandler**：在单个 Controller 内处理指定异常。
- **@ControllerAdvice** + **@ExceptionHandler**：全局统一处理异常，可配合 **@ResponseBody** 或类上 **@RestControllerAdvice** 直接返回 JSON。

### 7.2 异常分类与统一响应

- 定义业务异常（如 **BusinessException**）与错误码；在 **@ExceptionHandler** 中按异常类型映射为 HTTP 状态码与统一 **Result** 结构（code、msg），便于前端统一处理。

### 7.3 实战要点

- 全局类上 **@RestControllerAdvice**，方法上 **@ExceptionHandler(MethodArgumentNotValidException.class)** 处理 @Valid 校验失败，**@ExceptionHandler(Exception.class)** 兜底；日志记录异常，对外返回友好提示。

## 第八章：文件上传与下载

### 8.1 MultipartResolver 与 MultipartFile

- 请求 **Content-Type: multipart/form-data** 时，**MultipartResolver** 解析请求，将文件封装为 **MultipartFile**；Spring Boot 默认使用 **StandardServletMultipartResolver**，需配置 **spring.servlet.multipart**（max-file-size、max-request-size 等）。

### 8.2 上传实现

- Controller 方法参数为 **MultipartFile file** 或 **MultipartFile[] files**；调用 **file.getOriginalFilename()、getInputStream()、transferTo(File)** 等保存；注意**文件名安全**（避免路径穿越、重命名为 UUID 或时间戳）。

### 8.3 文件下载

- 返回 **ResponseEntity<Resource>** 或直接写 **InputStreamResource**，设置 **Content-Disposition: attachment; filename="xxx"**（文件名需编码）；或 **HttpServletResponse** 写流并设置头。

## 第九章：Spring MVC 与 JSON 数据交互

### 9.1 HttpMessageConverter

- **RequestMappingHandlerAdapter** 使用 **HttpMessageConverter** 将 @RequestBody 反序列化、@ResponseBody 序列化；Jackson 默认注册 **MappingJackson2HttpMessageConverter**，依赖 **ObjectMapper**。

### 9.2 序列化控制

- 实体上 **@JsonIgnore**（忽略字段）、**@JsonProperty("name")**（重命名）、**@JsonFormat**（日期格式）；循环引用可用 **@JsonManagedReference / @JsonBackReference** 或 **@JsonIgnore** 打断一方。

### 9.3 全局 ObjectMapper

- 通过 **WebMvcConfigurer.configureMessageConverters** 或 **Bean 注入 ObjectMapper** 统一配置日期格式、空值策略等。

## 第十章：拦截器、过滤器与 AOP 对比

### 10.1 执行顺序与范围

- **Filter**：Servlet 容器层，最早；可做编码、CORS、日志；无法直接拿到 Handler。
- **Interceptor**：Spring MVC 层，在 DispatcherServlet 内、Handler 前后；可拿到 Handler、Model，适合登录、权限。
- **AOP**：可切 Controller 方法，适合与业务无关的日志、耗时；拿不到 HttpServletRequest 需通过 RequestContextHolder 或方法参数传递。

### 10.2 协同使用

- **跨域**：Filter 或 **CorsFilter** / **WebMvcConfigurer.addCorsMappings**；**登录校验**：Interceptor；**请求日志/耗时**：Interceptor 或 AOP。

## 第十一章：Spring MVC 与前后端分离

### 11.1 RESTful 设计

- URL 表示资源（名词），HTTP Method 表示操作（GET/POST/PUT/DELETE）；状态码 2xx/4xx/5xx 与业务 code 结合；统一 **Result** 结构（code、msg、data）。

### 11.2 跨域（CORS）

- 方法或类上 **@CrossOrigin**；或 **WebMvcConfigurer.addCorsMappings** 配置允许的 origin、method、header；生产建议用网关或 Nginx 统一处理 CORS。

### 11.3 调试与测试

- **Postman / Apifox** 调试接口；**Swagger / Knife4j** 生成文档与在线调试；**MockMvc** 做 Controller 层单元测试，模拟请求与校验响应。

------

## 第十二章：Spring MVC 源码与执行流程解析

### 12.1 DispatcherServlet 初始化

- 加载 Web 应用上下文（子容器），可配合 **ContextLoaderListener** 加载根容器；**onRefresh()** 中初始化各策略组件（HandlerMapping、HandlerAdapter、ViewResolver 等）。

### 12.2 doDispatch() 核心流程

- 根据请求查 **HandlerMapping** 得到 **HandlerExecutionChain**（含拦截器）；**HandlerAdapter** 适配并执行 Handler（解析参数、调用方法、处理返回值）；若返回 ModelAndView 则 **ViewResolver** 解析并渲染；任一环节异常交 **HandlerExceptionResolver** 处理。

### 12.3 参数与返回值解析

- **HandlerMethodArgumentResolver** 解析方法参数（如 @RequestBody、@RequestParam）；**HandlerMethodReturnValueHandler** 处理返回值（如 @ResponseBody、ModelAndView）；AOP 对 Controller 的代理在 Handler 执行前已生效，因此可拦截到 Controller 方法调用。

## 第十三章：Spring MVC 项目实战案例

- **分层**：Controller 只做参数接收、校验、调用 Service、封装返回；业务与事务在 Service；Mapper/Repository 只做数据访问。
- **统一响应**：Result 封装；**全局异常**：@RestControllerAdvice + @ExceptionHandler 将异常转为 Result；**登录校验**：拦截器解析 token 写入 ThreadLocal，未登录返回 401 或统一错误码。
- **文件上传**：MultipartFile 接收，校验类型与大小，重命名存储，返回访问 URL；**参数校验**：@Valid + 业务校验，校验失败由全局异常统一返回。

## 第十四章：Spring MVC 常见问题与调优

### 14.1 常见错误

- **404**：路径或 method 不匹配、未扫描到 Controller。**405**：Method 不允许（如 GET 接口用 POST 调）。**415**：Content-Type 不支持（如未传 application/json 却用 @RequestBody）。
- **@RequestBody 无法解析**：Content-Type 正确、body 格式与 DTO 一致；**中文乱码**：配置 **CharacterEncodingFilter** 或 **server.servlet.encoding** 为 UTF-8。

### 14.2 性能与监控

- 静态资源走默认静态路径或 CDN，避免经 DispatcherServlet；接口层可配合缓存、异步；拦截器或 Filter 中记录请求耗时、TraceId，便于链路追踪与慢请求排查。

## 第十五章：Spring MVC 面试题与延伸

| 问题 | 要点 |
|------|------|
| 执行流程？ | DispatcherServlet → HandlerMapping 查 Handler → HandlerAdapter 执行 → 返回 ModelAndView 或写 Body → ViewResolver / HttpMessageConverter。 |
| DispatcherServlet 作用？ | 前端控制器，统一入口，调度 Mapping/Adapter/ViewResolver/ExceptionResolver。 |
| @Controller 与 @RestController？ | @RestController = @Controller + @ResponseBody，类下方法默认写响应体。 |
| @RequestBody / @ResponseBody 实现？ | 通过 HttpMessageConverter 反序列化/序列化，默认 Jackson。 |
| 如何支持 JSON？ | 依赖 Jackson，注册 MappingJackson2HttpMessageConverter；可自定义 ObjectMapper。 |
| 统一返回与全局异常？ | 定义 Result + @RestControllerAdvice + @ExceptionHandler，将正常返回与异常都封装为 Result。 |
| 跨域？ | @CrossOrigin 或 WebMvcConfigurer.addCorsMappings，或网关/Nginx 配置。 |
