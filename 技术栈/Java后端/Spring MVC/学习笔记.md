# 🧩 第一章：Spring MVC 概述与架构原理

------

> 本章建立 Spring MVC 的整体认知：它在 Web 开发中的定位、与 MVC 和 Servlet 的关系、核心组件与一次请求的完整流程，以及和 Struts2、WebFlux 的对比，为后续深入各组件打下基础。

------

## 1️⃣ 什么是 Spring MVC

### （1）MVC 设计模式

**MVC**（Model-View-Controller）把应用拆成三层，各司其职：

| 组件 | 职责 |
|------|------|
| **Model（模型）** | 承载业务数据和业务逻辑，如实体、Service、DAO。 |
| **View（视图）** | 负责展示，把数据变成用户可见的界面（HTML、JSON、XML 等）。 |
| **Controller（控制器）** | 接收请求、调用 Model、决定用哪个 View 返回结果。 |

这样做的目的是**把“怎么展示”和“业务逻辑”分开**，便于维护和扩展；MVC 不限于 Web，桌面、移动端同样适用。

------

### （2）Spring MVC 的定位与特点

Spring MVC 是 **Spring 家族里基于 MVC 的 Web 框架**，建立在 Servlet 之上，对“请求进来到响应出去”的整条链路做了统一抽象和扩展。

| 特点 | 说明 |
|------|------|
| **基于 Servlet** | 跑在任意 Servlet 容器（Tomcat、Jetty 等）上，入口是唯一的 DispatcherServlet。 |
| **注解驱动** | 用 `@Controller`、`@RequestMapping`、`@ResponseBody` 等声明控制器与映射，配置少、可读性好。 |
| **RESTful 友好** | 支持 REST 风格 URL、HTTP 方法、以及 JSON 等请求/响应体。 |
| **数据绑定与校验** | 请求参数自动绑定到方法参数或对象，并可配合 JSR-303/380 做校验。 |
| **可扩展** | 提供视图解析、异常处理、国际化、拦截器等扩展点。 |
| **与 Spring 一体** | 直接使用 IoC、AOP，和 Spring 容器无缝集成。 |

------

### （3）Spring MVC 与 Servlet 的关系

可以理解为：**Spring MVC 是在 Servlet 之上的一层“编排层”**。

| 维度 | 原生 Servlet | Spring MVC |
|------|--------------|------------|
| **入口** | 自己写 `HttpServlet`，在 web.xml 里配 url-pattern | 统一用 `DispatcherServlet`（本身也是 HttpServlet） |
| **配置** | web.xml 里注册 servlet | Spring Boot 自动配置或 JavaConfig 注册 |
| **参数** | 自己 `request.getParameter()`、拼对象 | 根据方法签名自动绑定参数或对象 |
| **响应** | 自己拼 HTML 或写 JSON 流 | 返回视图名或对象，由 ViewResolver / HttpMessageConverter 负责渲染或序列化 |

**结论**：Spring MVC 没有替换 Servlet，而是用 **一个 DispatcherServlet** 作为唯一入口，在内部按 MVC 思路调度 HandlerMapping、HandlerAdapter、ViewResolver 等组件，把“找谁处理、怎么调、怎么出结果”都标准化了。

------

## 2️⃣ 核心组件与请求处理流程

### （1）组件总览

一次 HTTP 请求在 Spring MVC 中的大致走向：

```
  客户端
     │
     ▼
[DispatcherServlet]   ← 前端控制器（唯一入口，继承 HttpServlet）
     │
     ├─ HandlerMapping    → 根据 URL、Method 等找到对应的 Handler（通常是某个 Controller 方法）
     ├─ HandlerAdapter    → 解析请求参数、调用 Handler、处理返回值
     ├─ Controller        → 业务控制器，返回视图名或数据对象
     ├─ ModelAndView      → 可选：封装模型数据 + 逻辑视图名
     └─ ViewResolver      → 把逻辑视图名解析成具体 View（JSP、Thymeleaf 等）；若走 @ResponseBody 则不经过此处
     ▼
  响应返回客户端
```

------

### （2）各组件职责

| 组件 | 职责 |
|------|------|
| **DispatcherServlet** | 前端控制器：接收所有匹配的请求，依次调用 HandlerMapping、HandlerAdapter，再根据返回值走 ViewResolver 或直接写响应。 |
| **HandlerMapping** | 根据请求的 URL、HTTP 方法等，找到能处理该请求的 Handler（如带 `@RequestMapping` 的 Controller 方法）。 |
| **HandlerAdapter** | 适配器：负责把 HttpServletRequest/Response 转成 Handler 需要的入参、调用 Handler、再把返回值交给 DispatcherServlet 处理。 |
| **Controller** | 开发者写的控制层代码，返回视图名（String）、ModelAndView 或普通对象（配合 @ResponseBody）。 |
| **ModelAndView** | 可选的返回值类型，同时带“模型数据”和“逻辑视图名”；若直接返回对象且带 @ResponseBody，则不使用此类。 |
| **ViewResolver** | 将逻辑视图名（如 `"user/list"`）解析为具体的 View 实现，由 View 负责渲染；**返回 @ResponseBody 时不会走 ViewResolver**。 |

------

### （3）请求处理流程（步骤）

1. **客户端** 发起请求，例如 `GET /user/list`。
2. **DispatcherServlet** 收到请求，调用 **HandlerMapping** 查找能处理该 URL 和 Method 的 Handler（通常对应一个 Controller 方法）。
3. **HandlerAdapter** 根据 Handler 的方法签名解析请求参数（Query、Body、Path 等），调用 **Controller** 方法，得到返回值（视图名、ModelAndView 或对象）。
4. **若返回的是视图**：DispatcherServlet 用 **ViewResolver** 把逻辑视图名解析成 View，再渲染并写入响应；**若方法带 @ResponseBody**：返回值由 **HttpMessageConverter** 序列化（如转 JSON）后直接写回响应体。
5. **DispatcherServlet** 将响应返回客户端。

**记忆口诀**：**收请求 → 找 Handler（Mapping）→ 调 Handler（Adapter）→ 出结果（ViewResolver 或 MessageConverter）**。

------

### （4）代码示例

```java
@Controller
@RequestMapping("/user")
public class UserController {

    @GetMapping("/list")
    @ResponseBody
    public List<User> list() {
        return userService.findAll();
    }
}
```

- `@Controller`：把该类注册为 Spring MVC 的控制器 Bean。
- `@RequestMapping("/user")`：类上的路径前缀，与下面方法上的路径拼接。
- `@GetMapping("/list")`：处理 `GET /user/list`。
- `@ResponseBody`：返回值不经过 ViewResolver，由 HttpMessageConverter（如 Jackson）序列化为 JSON 写回。

## 3️⃣ 与其他框架的对比

### （1）Spring MVC vs Struts2

| 对比项 | Spring MVC | Struts2 |
|--------|------------|---------|
| **底层** | 基于 Servlet | 基于 Filter |
| **控制器实例** | 单例（线程安全需自己保证） | 多例，每次请求一个 Action |
| **参数** | 直接绑定到方法参数 | 通过 ValueStack、OGNL 等 |
| **配置方式** | 注解 + JavaConfig 为主 | 传统上以 XML 为主 |
| **与 Spring** | 同属 Spring 生态，天然集成 | 需单独整合 |

目前企业级 Java Web 开发以 Spring（含 Spring MVC）为主流，Struts2 使用较少。

------

### （2）Spring MVC vs Spring WebFlux

| 对比项 | Spring MVC | Spring WebFlux |
|--------|------------|----------------|
| **编程模型** | 同步、阻塞（一个请求占用一个线程） | 异步、非阻塞（Reactor 响应式） |
| **运行容器** | 基于 Servlet（Tomcat、Jetty 等） | 基于 Netty、Undertow 等非 Servlet 容器 |
| **适用场景** | 常规 Web、管理后台、一般并发 | 高并发、大量 IO、流式、长连接等 |

**面试常见题**：*“Spring MVC 能支撑 10 万并发吗？”*  
答：受 Tomcat 线程池和阻塞模型限制，单机线程数有限；若追求极高并发、IO 密集，可考虑 WebFlux（非阻塞）。多数业务场景下 Spring MVC 配合合理线程池和集群即可。

## 第二章：Spring MVC 的核心组件

------

> 本章按请求处理链路拆解六大核心组件：前端控制器、映射器、适配器、控制器与注解、模型与视图封装、视图解析器；弄清各自职责与常用实现，便于理解第一章流程并在面试中说清「谁做了什么」。

------

### 2.1 DispatcherServlet（前端控制器）

#### （1）职责与定位

**DispatcherServlet** 是 Spring MVC 的**唯一入口**，继承 `HttpServlet`，在 Servlet 容器中通常映射为 `/`，所有匹配的 HTTP 请求都会先进入它。

| 职责 | 说明 |
|------|------|
| 接收请求 | 作为前端控制器，统一接收请求并决定后续流程。 |
| 查找 Handler | 委托 **HandlerMapping** 根据 URL、Method 等找到能处理的 Handler（Controller 方法）。 |
| 执行 Handler | 委托 **HandlerAdapter** 解析参数、调用方法、处理返回值。 |
| 渲染或写回 | 若返回视图则交给 **ViewResolver** 解析并渲染；若为 @ResponseBody 则用 **HttpMessageConverter** 写响应体。 |

> 💬 **一句话**：DispatcherServlet 不干具体业务，只负责「找谁处理 → 交给谁执行 → 结果怎么出」。

#### （2）初始化与配置

- **传统 Web**：在 `web.xml` 中配置 `<servlet>` 与 `<servlet-mapping>`，`url-pattern` 一般为 `/`；启动时加载 Spring 的 Web 应用上下文（子容器），可配合 **ContextLoaderListener** 加载根容器（业务 Bean）。
- **Spring Boot**：通过 **DispatcherServletAutoConfiguration** 自动注册 `DispatcherServlet`，默认映射 `/`；容器启动时在 `onRefresh()` 中初始化各策略组件（HandlerMapping、HandlerAdapter、ViewResolver 等）。

| 配置方式 | 说明 |
|----------|------|
| web.xml | 指定 servlet-class、init-param（contextConfigLocation）、url-pattern。 |
| JavaConfig | 实现 `WebApplicationInitializer` 或继承 `AbstractAnnotationConfigDispatcherServletInitializer` 注册。 |
| Spring Boot | 无需手写，自动配置；可通过 `spring.mvc.*` 等调整。 |

------

### 2.2 HandlerMapping（处理器映射器）

#### （1）作用与实现

**HandlerMapping** 负责根据当前请求（URL、HTTP 方法、params、headers 等）找到能处理该请求的 **Handler**（在注解驱动下通常是「Controller + 方法」封装成的 **HandlerMethod**）。

| 常用实现 | 说明 |
|----------|------|
| **RequestMappingHandlerMapping** | 解析 `@RequestMapping`、`@GetMapping` 等，维护「请求条件 → HandlerMethod」的映射表，注解驱动下默认使用。 |
| BeanNameUrlHandlerMapping | 根据 Bean 名称与 URL 映射，较少用。 |

查找结果封装为 **HandlerExecutionChain**（包含 Handler 与匹配到的 **HandlerInterceptor** 列表），供 DispatcherServlet 后续调用。

#### （2）面试要点

- 映射表在应用启动时由 `RequestMappingHandlerMapping` 扫描 `@Controller` 并解析注解生成；请求进来时按条件匹配，得到唯一 Handler（或无则 404）。

------

### 2.3 HandlerAdapter（处理器适配器）

#### （1）为什么需要适配器

Handler 可能是 `@RequestMapping` 方法、`Controller` 接口实现、Servlet 等不同类型；**HandlerAdapter** 屏蔽差异，统一完成「解析请求参数 → 调用 Handler → 处理返回值」三步。

| 步骤 | 对应组件 / 行为 |
|------|------------------|
| 参数解析 | **HandlerMethodArgumentResolver** 链：将 `HttpServletRequest`、Body、Header 等解析为方法参数（支持 @RequestParam、@RequestBody、@PathVariable 等）。 |
| 调用方法 | 通过反射执行 Controller 方法。 |
| 返回值处理 | **HandlerMethodReturnValueHandler** 链：将返回值转为 ModelAndView 或直接写响应（如 @ResponseBody 走 HttpMessageConverter）。 |

#### （2）常用实现

| 实现 | 说明 |
|------|------|
| **RequestMappingHandlerAdapter** | 注解驱动下的默认适配器，配合 `RequestMappingHandlerMapping` 使用；内部注册多种 ArgumentResolver 与 ReturnValueHandler。 |

> 💡 **重点**：参数绑定、类型转换、@RequestBody / @ResponseBody 的读写，都在 HandlerAdapter 这一层完成，而不是在 DispatcherServlet 里。

------

### 2.4 Controller 与 @Controller / @RestController

#### （1）概念与区别

| 注解 | 含义 | 方法返回值默认行为 |
|------|------|---------------------|
| **@Controller** | 标识为 Spring MVC 控制器 Bean，可被 HandlerMapping 扫描。 | 返回值作为**逻辑视图名**，由 ViewResolver 解析；若方法上再加 **@ResponseBody** 则直接写响应体。 |
| **@RestController** | 等价于 @Controller + 类上默认 @ResponseBody。 | 返回值直接写响应体（通常由 HttpMessageConverter 序列化为 JSON），**不经过** ViewResolver。 |

#### （2）使用场景与示例

- **@Controller**：传统 MVC、服务端渲染页面（JSP、Thymeleaf），方法返回视图名。
- **@RestController**：前后端分离、RESTful API，统一返回 JSON。

```java
// 传统视图：返回视图名 "user/list"，由 ViewResolver 解析
@Controller
public class UserPageController {
    @GetMapping("/user/list")
    public String list(Model model) {
        model.addAttribute("users", userService.findAll());
        return "user/list";
    }
}

// RESTful API：直接返回对象，序列化为 JSON
@RestController
@RequestMapping("/api/user")
public class UserApiController {
    @GetMapping("/list")
    public List<User> list() {
        return userService.findAll();
    }
}
```

------

### 2.5 Model、ModelMap、ModelAndView

#### （1）作用与区别

| 类型 | 作用 | 典型用法 |
|------|------|----------|
| **Model / ModelMap** | 向视图传递数据的键值容器，可作方法参数由框架注入。 | 在方法里 `model.addAttribute("key", value)`，返回视图名后模板中通过 key 访问。 |
| **ModelAndView** | 同时封装「逻辑视图名」与「Model 数据」。 | 一次 return 既指定视图又带数据；也可先 new ModelAndView()，再 setViewName、addObject。 |

二者本质都是把数据放到请求域（request scope），供 View 渲染时使用。

#### （2）与前后端分离的关系

> 🔹 **注意**：前后端分离下接口通常用 **@ResponseBody** 或 **@RestController** 直接返回 DTO/Result，不经过视图名，因此很少再用 Model/ModelAndView；模型数据更多体现在「返回值对象」上。

------

### 2.6 ViewResolver（视图解析器）

#### （1）职责与常见实现

**ViewResolver** 将**逻辑视图名**（如 `"user/list"`）解析为具体的 **View** 实现，由 View 负责渲染（如生成 HTML、转发 JSP）。

| 实现 | 说明 |
|------|------|
| **InternalResourceViewResolver** | 将视图名解析为 JSP 或内部转发路径（prefix + viewName + suffix）。 |
| **ThymeleafViewResolver** | 配合 Thymeleaf 模板引擎，解析为 ThymeleafView。 |
| **ContentNegotiatingViewResolver** | 根据请求的 Accept 或扩展名协商使用哪个 ViewResolver。 |

#### （2）何时不经过 ViewResolver

- 方法带 **@ResponseBody** 或类为 **@RestController**：返回值由 **HttpMessageConverter** 直接写响应体，**不经过** ViewResolver。
- 返回 **ResponseEntity** 且 body 非视图名时，同样由 MessageConverter 或直接写流处理。

> 💡 **面试常问**：返回 JSON 时还会走 ViewResolver 吗？答：不会，走 HttpMessageConverter。

## ✅ 本章小结

| 知识点 | 面试关键词 | 实际应用 |
|--------|------------|----------|
| DispatcherServlet | 前端控制器、唯一入口、调度 Mapping/Adapter/ViewResolver | 说明一次请求从哪里进、谁在协调 |
| HandlerMapping | 根据 URL/Method 找 Handler、RequestMappingHandlerMapping、映射表 | 解释「请求如何找到对应方法」 |
| HandlerAdapter | 参数解析、调用方法、返回值处理；ArgumentResolver、ReturnValueHandler | 说明参数绑定与 @ResponseBody 在哪儿完成 |
| @Controller / @RestController | 视图名 vs 直接写 body；@RestController = @Controller + @ResponseBody | 选型：页面用 @Controller，接口用 @RestController |
| Model / ModelAndView | 传数据给视图、请求域；前后端分离少用 | 服务端渲染时向模板传数据 |
| ViewResolver | 逻辑视图名 → View；返回 JSON 不经过 ViewResolver | 区分「渲染页面」与「写 JSON」的路径 |

------

**学习要点**：

- DispatcherServlet 是唯一入口，只做调度，不写业务；初始化时加载 Web 容器并初始化各策略 Bean。
- HandlerMapping 负责「找 Handler」，HandlerAdapter 负责「解析参数、执行方法、处理返回值」；二者配合完成从请求到方法调用的桥梁。
- @Controller 与 @RestController 区别在于返回值是否作为视图名；前后端分离接口多用 @RestController，模型数据通过返回值体现。
- ViewResolver 只处理「返回视图名」的分支；返回 @ResponseBody 或 ResponseEntity 时由 HttpMessageConverter 写 body，不经过 ViewResolver。

|------------|----------|
| DispatcherServlet 做了什么？ | 前端控制器：接收请求 → 交给 HandlerMapping 查 Handler → 交给 HandlerAdapter 执行（参数解析、方法调用、返回值处理）→ 根据返回值走 ViewResolver 渲染视图或 HttpMessageConverter 写 body。 |
| HandlerMapping 和 HandlerAdapter 的区别？ | HandlerMapping 负责「根据请求找哪个 Handler」；HandlerAdapter 负责「怎么调这个 Handler」（解析参数、执行、处理返回值）。一个管映射，一个管执行。 |
| @Controller 和 @RestController 的区别？ | @RestController = @Controller + 类级别 @ResponseBody；前者方法默认返回视图名走 ViewResolver，后者默认直接写响应体走 HttpMessageConverter，用于 RESTful API。 |
| 返回 JSON 时还经过 ViewResolver 吗？ | 不经过。方法带 @ResponseBody 或类为 @RestController 时，返回值由 HttpMessageConverter（如 Jackson）序列化后直接写响应体，不经过 ViewResolver。 |

## 第三章：Spring MVC 核心注解与请求映射

### 3.1 控制层与请求映射

- **@RequestMapping**：可标注类与方法，指定 path（value）、method、params、headers 等；**@GetMapping / @PostMapping / @PutMapping / @DeleteMapping** 为 method 的简写。
- **@Controller**：视图层控制器；**@RestController**：类级别等价于“所有方法带 @ResponseBody”，用于接口层。

### 3.2 参数绑定

- **@RequestParam**：绑定查询参数或表单字段，可指定 name、required、defaultValue。
- **@PathVariable**：绑定 URI 模板变量，如 `/user/{id}` 中 id。
- **@RequestBody**：将请求体反序列化为对象（通常 JSON），依赖 HttpMessageConverter。
- **@RequestHeader**：绑定请求头。**@CookieValue**：绑定 Cookie。

### 3.3 返回值处理

- **@ResponseBody**：方法返回值由 HttpMessageConverter 序列化后写入响应体（如 JSON）。
- **ResponseEntity<T>**：可设置状态码、响应头、body，适合需精确控制响应的场景。

### 3.4 实战要点

- GET 参数用 @RequestParam 或 POJO 直接接收（属性名与参数名一致）；POST JSON 用 @RequestBody；REST 风格路径用 @PathVariable；文件上传用 MultipartFile 参数。

------

## 第四章：数据绑定与类型转换机制

### 4.1 数据绑定流程

- **WebDataBinder**：将请求参数绑定到方法参数或命令对象；内置类型转换（String→Integer、日期等），可注册自定义 Converter 或 Formatter。

### 4.2 自定义类型转换器

- 实现 **Converter<S, T>** 或 **Formatter<T>**，在配置类中通过 **WebMvcConfigurer.addFormatters** 或 **@InitBinder**（局部）注册。

### 4.3 表单与复杂类型绑定

- 支持嵌套对象（如 order.items）、日期、枚举、集合；日期格式可通过 @DateTimeFormat 或全局配置统一。

### 4.4 数据校验（JSR-303 / JSR-380）

- 在 DTO 上使用 **@NotNull、@Size、@Email、@Valid** 等；Controller 方法参数加 **@Valid** 或 **@Validated** 触发校验，校验结果进入 **BindingResult**；与全局异常处理配合可统一返回 400 与错误信息。

------

## 第五章：模型数据传递与作用域管理

### 5.1 Model、ModelMap、ModelAndView

- 方法参数中声明 **Model** 或 **ModelMap**，向其中 put 数据，返回视图名即可在模板中访问；**ModelAndView** 同时设置 viewName 与 model，适合需要明确指定视图时。

### 5.2 Session 与 @SessionAttributes

- **@SessionAttributes(names)**：将 Model 中指定 name 的数据放入 Session；用 **SessionStatus.setComplete()** 清除本次 Session 中由该注解管理的属性。前后端分离下更多用 token 存登录态，少用 Session。

### 5.3 统一响应封装（Result）

- 前后端分离常用：定义 **Result<T>**（code、msg、data），Controller 统一返回 `Result.success(data)` 或 `Result.fail(code, msg)`；配合 **@ControllerAdvice** 全局异常处理，将异常也封装为 Result。

------

## 第六章：拦截器（HandlerInterceptor）

### 6.1 作用与接口

- **HandlerInterceptor**：在 Handler 执行前后插入逻辑，用于**登录校验、权限、日志、耗时统计**等；三个方法：**preHandle**（返回 false 则中断）、**postHandle**（视图渲染前）、**afterCompletion**（请求结束后，可做清理）。

### 6.2 配置与执行顺序

- 通过 **WebMvcConfigurer.addInterceptors** 注册，指定 **addPathPatterns / excludePathPatterns**；多个拦截器按注册顺序执行 preHandle，逆序执行 postHandle 与 afterCompletion。

### 6.3 与 Filter、AOP 区别

- **Filter**：Servlet 规范，最早执行，可修改请求/响应；**Interceptor**：Spring MVC 层，可访问 Handler 与 Model；**AOP**：可切 Controller 方法，但拿不到 HttpServletRequest 等，一般请求前后用拦截器更合适。登录态常放在拦截器里解析 token 并写入 ThreadLocal。

------

## 第七章：异常处理机制

### 7.1 局部与全局异常

- **@ExceptionHandler**：在单个 Controller 内处理指定异常。
- **@ControllerAdvice** + **@ExceptionHandler**：全局统一处理异常，可配合 **@ResponseBody** 或类上 **@RestControllerAdvice** 直接返回 JSON。

### 7.2 异常分类与统一响应

- 定义业务异常（如 **BusinessException**）与错误码；在 **@ExceptionHandler** 中按异常类型映射为 HTTP 状态码与统一 **Result** 结构（code、msg），便于前端统一处理。

### 7.3 实战要点

- 全局类上 **@RestControllerAdvice**，方法上 **@ExceptionHandler(MethodArgumentNotValidException.class)** 处理 @Valid 校验失败，**@ExceptionHandler(Exception.class)** 兜底；日志记录异常，对外返回友好提示。

------


## 第八章：文件上传与下载

### 8.1 MultipartResolver 与 MultipartFile

- 请求 **Content-Type: multipart/form-data** 时，**MultipartResolver** 解析请求，将文件封装为 **MultipartFile**；Spring Boot 默认使用 **StandardServletMultipartResolver**，需配置 **spring.servlet.multipart**（max-file-size、max-request-size 等）。

### 8.2 上传实现

- Controller 方法参数为 **MultipartFile file** 或 **MultipartFile[] files**；调用 **file.getOriginalFilename()、getInputStream()、transferTo(File)** 等保存；注意**文件名安全**（避免路径穿越、重命名为 UUID 或时间戳）。

### 8.3 文件下载

- 返回 **ResponseEntity<Resource>** 或直接写 **InputStreamResource**，设置 **Content-Disposition: attachment; filename="xxx"**（文件名需编码）；或 **HttpServletResponse** 写流并设置头。

------

## 第九章：Spring MVC 与 JSON 数据交互

### 9.1 HttpMessageConverter

- **RequestMappingHandlerAdapter** 使用 **HttpMessageConverter** 将 @RequestBody 反序列化、@ResponseBody 序列化；Jackson 默认注册 **MappingJackson2HttpMessageConverter**，依赖 **ObjectMapper**。

### 9.2 序列化控制

- 实体上 **@JsonIgnore**（忽略字段）、**@JsonProperty("name")**（重命名）、**@JsonFormat**（日期格式）；循环引用可用 **@JsonManagedReference / @JsonBackReference** 或 **@JsonIgnore** 打断一方。

### 9.3 全局 ObjectMapper

- 通过 **WebMvcConfigurer.configureMessageConverters** 或 **Bean 注入 ObjectMapper** 统一配置日期格式、空值策略等。

------

## 第十章：拦截器、过滤器与 AOP 对比

### 10.1 执行顺序与范围

- **Filter**：Servlet 容器层，最早；可做编码、CORS、日志；无法直接拿到 Handler。
- **Interceptor**：Spring MVC 层，在 DispatcherServlet 内、Handler 前后；可拿到 Handler、Model，适合登录、权限。
- **AOP**：可切 Controller 方法，适合与业务无关的日志、耗时；拿不到 HttpServletRequest 需通过 RequestContextHolder 或方法参数传递。

### 10.2 协同使用

- **跨域**：Filter 或 **CorsFilter** / **WebMvcConfigurer.addCorsMappings**；**登录校验**：Interceptor；**请求日志/耗时**：Interceptor 或 AOP。

------

## 第十一章：Spring MVC 与前后端分离

### 11.1 RESTful 设计

- URL 表示资源（名词），HTTP Method 表示操作（GET/POST/PUT/DELETE）；状态码 2xx/4xx/5xx 与业务 code 结合；统一 **Result** 结构（code、msg、data）。

### 11.2 跨域（CORS）

- 方法或类上 **@CrossOrigin**；或 **WebMvcConfigurer.addCorsMappings** 配置允许的 origin、method、header；生产建议用网关或 Nginx 统一处理 CORS。

### 11.3 调试与测试

- **Postman / Apifox** 调试接口；**Swagger / Knife4j** 生成文档与在线调试；**MockMvc** 做 Controller 层单元测试，模拟请求与校验响应。

------

## 第十二章：Spring MVC 源码与执行流程解析

### 12.1 DispatcherServlet 初始化

- 加载 Web 应用上下文（子容器），可配合 **ContextLoaderListener** 加载根容器；**onRefresh()** 中初始化各策略组件（HandlerMapping、HandlerAdapter、ViewResolver 等）。

### 12.2 doDispatch() 核心流程

- 根据请求查 **HandlerMapping** 得到 **HandlerExecutionChain**（含拦截器）；**HandlerAdapter** 适配并执行 Handler（解析参数、调用方法、处理返回值）；若返回 ModelAndView 则 **ViewResolver** 解析并渲染；任一环节异常交 **HandlerExceptionResolver** 处理。

### 12.3 参数与返回值解析

- **HandlerMethodArgumentResolver** 解析方法参数（如 @RequestBody、@RequestParam）；**HandlerMethodReturnValueHandler** 处理返回值（如 @ResponseBody、ModelAndView）；AOP 对 Controller 的代理在 Handler 执行前已生效，因此可拦截到 Controller 方法调用。

------

## 第十三章：Spring MVC 项目实战案例

- **分层**：Controller 只做参数接收、校验、调用 Service、封装返回；业务与事务在 Service；Mapper/Repository 只做数据访问。
- **统一响应**：Result 封装；**全局异常**：@RestControllerAdvice + @ExceptionHandler 将异常转为 Result；**登录校验**：拦截器解析 token 写入 ThreadLocal，未登录返回 401 或统一错误码。
- **文件上传**：MultipartFile 接收，校验类型与大小，重命名存储，返回访问 URL；**参数校验**：@Valid + 业务校验，校验失败由全局异常统一返回。

------

## 第十四章：Spring MVC 常见问题与调优

### 14.1 常见错误

- **404**：路径或 method 不匹配、未扫描到 Controller。**405**：Method 不允许（如 GET 接口用 POST 调）。**415**：Content-Type 不支持（如未传 application/json 却用 @RequestBody）。
- **@RequestBody 无法解析**：Content-Type 正确、body 格式与 DTO 一致；**中文乱码**：配置 **CharacterEncodingFilter** 或 **server.servlet.encoding** 为 UTF-8。

### 14.2 性能与监控

- 静态资源走默认静态路径或 CDN，避免经 DispatcherServlet；接口层可配合缓存、异步；拦截器或 Filter 中记录请求耗时、TraceId，便于链路追踪与慢请求排查。

## 第十五章：Spring MVC 面试题与延伸

| 问题 | 要点 |
|------|------|
| 执行流程？ | DispatcherServlet → HandlerMapping 查 Handler → HandlerAdapter 执行 → 返回 ModelAndView 或写 Body → ViewResolver / HttpMessageConverter。 |
| DispatcherServlet 作用？ | 前端控制器，统一入口，调度 Mapping/Adapter/ViewResolver/ExceptionResolver。 |
| @Controller 与 @RestController？ | @RestController = @Controller + @ResponseBody，类下方法默认写响应体。 |
| @RequestBody / @ResponseBody 实现？ | 通过 HttpMessageConverter 反序列化/序列化，默认 Jackson。 |
| 如何支持 JSON？ | 依赖 Jackson，注册 MappingJackson2HttpMessageConverter；可自定义 ObjectMapper。 |
| 统一返回与全局异常？ | 定义 Result + @RestControllerAdvice + @ExceptionHandler，将正常返回与异常都封装为 Result。 |
| 跨域？ | @CrossOrigin 或 WebMvcConfigurer.addCorsMappings，或网关/Nginx 配置。 |
