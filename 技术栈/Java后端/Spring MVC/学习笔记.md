## 1️⃣ 什么是 Spring MVC

### （1）MVC 设计模式

> **一句话**：**MVC** 把应用拆成 **Model（数据与业务）、View（展示）、Controller（接收请求并协调 M 和 V）** 三层，各层只干自己的事。
> **为什么**：若页面、业务、请求处理混在一起，改一处牵全身；分层后「怎么展示」和「业务逻辑」解耦，便于维护和多人协作。
> **类比**：Controller 是前台接待（接单），Model 是后厨（做菜），View 是出餐口（把菜端给用户）。

**MVC**（Model-View-Controller）把应用拆成三层，各司其职：

| 组件 | 职责 |
|------|------|
| **Model（模型）** | 承载业务数据和业务逻辑，如实体、Service、DAO。 |
| **View（视图）** | 负责展示，把数据变成用户可见的界面（HTML、JSON、XML 等）。 |
| **Controller（控制器）** | 接收请求、调用 Model、决定用哪个 View 返回结果。 |

这样做的目的是**把“怎么展示”和“业务逻辑”分开**，便于维护和扩展；MVC 不限于 Web，桌面、移动端同样适用。

------

### （2）Spring MVC 的定位与特点

Spring MVC 是 **Spring 家族里基于 MVC 的 Web 框架**，建立在 Servlet 之上，对“请求进来到响应出去”的整条链路做了统一抽象和扩展。

| 特点 | 说明 |
|------|------|
| **基于 Servlet** | 跑在任意 Servlet 容器（Tomcat、Jetty 等）上，入口是唯一的 DispatcherServlet。 |
| **注解驱动** | 用 `@Controller`、`@RequestMapping`、`@ResponseBody` 等声明控制器与映射，配置少、可读性好。 |
| **RESTful 友好** | 支持 REST 风格 URL、HTTP 方法、以及 JSON 等请求/响应体。 |
| **数据绑定与校验** | 请求参数自动绑定到方法参数或对象，并可配合 JSR-303/380 做校验。 |
| **可扩展** | 提供视图解析、异常处理、国际化、拦截器等扩展点。 |
| **与 Spring 一体** | 直接使用 IoC、AOP，和 Spring 容器无缝集成。 |

------

### （3）Spring MVC 与 Servlet 的关系

可以理解为：**Spring MVC 是在 Servlet 之上的一层“编排层”**。

| 维度 | 原生 Servlet | Spring MVC |
|------|--------------|------------|
| **入口** | 自己写 `HttpServlet`，在 web.xml 里配 url-pattern | 统一用 `DispatcherServlet`（本身也是 HttpServlet） |
| **配置** | web.xml 里注册 servlet | Spring Boot 自动配置或 JavaConfig 注册 |
| **参数** | 自己 `request.getParameter()`、拼对象 | 根据方法签名自动绑定参数或对象 |
| **响应** | 自己拼 HTML 或写 JSON 流 | 返回视图名或对象，由 ViewResolver / HttpMessageConverter 负责渲染或序列化 |

**结论**：Spring MVC 没有替换 Servlet，而是用 **一个 DispatcherServlet**（前端控制器，本质是一个 **HttpServlet**）作为**唯一入口**，在内部按 MVC 思路调度 HandlerMapping、HandlerAdapter、ViewResolver 等组件，把“找谁处理、怎么调、怎么出结果”都标准化了。

## 2️⃣ 核心组件与请求处理流程

### （1）组件总览

一次 HTTP 请求在 Spring MVC 中的大致走向：

```
  客户端
     │
     ▼
[DispatcherServlet]   ← 前端控制器（唯一入口，继承 HttpServlet）
     │
     ├─ HandlerMapping    → 根据 URL、Method 等找到对应的 Handler（通常是某个 Controller 方法）
     ├─ HandlerAdapter    → 解析请求参数、调用 Handler、处理返回值
     ├─ Controller        → 业务控制器，返回视图名或数据对象
     ├─ ModelAndView      → 可选：封装模型数据 + 逻辑视图名
     └─ ViewResolver      → 把逻辑视图名解析成具体 View（JSP、Thymeleaf 等）；若走 @ResponseBody 则不经过此处
     ▼
  响应返回客户端
```

------

### （2）各组件职责

| 组件 | 职责 |
|------|------|
| **DispatcherServlet** | 前端控制器：接收所有匹配的请求，依次调用 HandlerMapping、HandlerAdapter，再根据返回值走 ViewResolver 或直接写响应。 |
| **HandlerMapping** | 根据请求的 URL、HTTP 方法等，找到能处理该请求的 Handler（如带 `@RequestMapping` 的 Controller 方法）。 |
| **HandlerAdapter** | 适配器：负责把 HttpServletRequest/Response 转成 Handler 需要的入参、调用 Handler、再把返回值交给 DispatcherServlet 处理。 |
| **Controller** | 开发者写的控制层代码，返回视图名（String）、ModelAndView 或普通对象（配合 @ResponseBody）。 |
| **ModelAndView** | 可选的返回值类型，同时带“模型数据”和“逻辑视图名”；若直接返回对象且带 @ResponseBody，则不使用此类。 |
| **ViewResolver** | 将逻辑视图名（如 `"user/list"`）解析为具体的 View 实现，由 View 负责渲染；**返回 @ResponseBody 时不会走 ViewResolver**。 |

------

### （3）请求处理流程（步骤）

1. **客户端** 发起请求，例如 `GET /user/list`。
2. **DispatcherServlet** 收到请求，调用 **HandlerMapping** 查找能处理该 URL 和 Method 的 Handler（通常对应一个 Controller 方法）。
3. **HandlerAdapter** 根据 Handler 的方法签名解析请求参数（Query、Body、Path 等），调用 **Controller** 方法，得到返回值（视图名、ModelAndView 或对象）。
4. **若返回的是视图**：DispatcherServlet 用 **ViewResolver** 把逻辑视图名解析成 View，再渲染并写入响应；**若方法带 @ResponseBody**：返回值由 **HttpMessageConverter** 序列化（如转 JSON）后直接写回响应体。
5. **DispatcherServlet** 将响应返回客户端。

**记忆口诀**：**收请求 → 找 Handler（Mapping）→ 调 Handler（Adapter）→ 出结果（ViewResolver 或 MessageConverter）**。

> 🎯 **面试怎么说**：一次请求的**数据形态变化**——请求进来是 **HttpServletRequest**；**HandlerMapping** 根据 URL+Method 找到 **Handler**（通常是 Controller 方法）；**HandlerAdapter** 把请求里的参数（Query、Body、Path）**绑定到方法入参**，调用方法得到**返回值**（视图名 String、ModelAndView 或对象）；若 **@ResponseBody**，返回值经 **HttpMessageConverter** 转成 JSON 等写回 **HttpServletResponse**；若返回视图名，则 **ViewResolver** 解析成 View 再渲染写回。整条链由 **DispatcherServlet** 串联。

------

### （4）代码示例

```java
@Controller
@RequestMapping("/user")
public class UserController {

    @GetMapping("/list")
    @ResponseBody
    public List<User> list() {
        return userService.findAll();
    }
}
```

- `@Controller`：把该类注册为 Spring MVC 的控制器 Bean。
- `@RequestMapping("/user")`：类上的路径前缀，与下面方法上的路径拼接。
- `@GetMapping("/list")`：处理 `GET /user/list`。
- `@ResponseBody`：返回值不经过 ViewResolver，由 HttpMessageConverter（如 Jackson）序列化为 JSON 写回。

## 3️⃣ 与其他框架的对比

### （1）Spring MVC vs Struts2

| 对比项 | Spring MVC | Struts2 |
|--------|------------|---------|
| **底层** | 基于 Servlet | 基于 Filter |
| **控制器实例** | 单例（线程安全需自己保证） | 多例，每次请求一个 Action |
| **参数** | 直接绑定到方法参数 | 通过 ValueStack、OGNL 等 |
| **配置方式** | 注解 + JavaConfig 为主 | 传统上以 XML 为主 |
| **与 Spring** | 同属 Spring 生态，天然集成 | 需单独整合 |

目前企业级 Java Web 开发以 Spring（含 Spring MVC）为主流，Struts2 使用较少。

------

### （2）Spring MVC vs Spring WebFlux

| 对比项 | Spring MVC | Spring WebFlux |
|--------|------------|----------------|
| **编程模型** | 同步、阻塞（一个请求占用一个线程） | 异步、非阻塞（Reactor 响应式） |
| **运行容器** | 基于 Servlet（Tomcat、Jetty 等） | 基于 Netty、Undertow 等非 Servlet 容器 |
| **适用场景** | 常规 Web、管理后台、一般并发 | 高并发、大量 IO、流式、长连接等 |

**面试常见题**：*“Spring MVC 能支撑 10 万并发吗？”*  
答：受 Tomcat 线程池和阻塞模型限制，单机线程数有限；若追求极高并发、IO 密集，可考虑 WebFlux（非阻塞）。多数业务场景下 Spring MVC 配合合理线程池和集群即可。

### 常见坑与注意点

| 现象 / 易错点 | 原因 | 怎么改 / 怎么记 |
|---------------|------|-----------------|
| 返回 JSON 却得到 404 或视图名字符串 | 忘记加 **@ResponseBody**，返回值被当成逻辑视图名去解析 | 前后端分离、返回数据时方法上必须 **@ResponseBody** 或类上用 **@RestController**。 |
| Controller 里用成员变量存请求相关数据 | Controller 默认**单例**，多请求共享同一实例，会串数据、线程不安全 | 请求相关数据放**方法参数、局部变量**或 **ThreadLocal**，不要用实例字段。 |
| 请求路径 404 | 类上没 **@RequestMapping**、方法路径拼错、或未扫描到该 Controller | 检查包扫描、类上路径前缀、方法上路径及 HTTP 方法。 |

### 请求路径步骤与数据形态（面试可画可答）

**步骤**：① DispatcherServlet 接收请求 → ② HandlerMapping 根据 URL+Method 查 Handler → ③ HandlerAdapter 解析参数并调用 Controller 方法 → ④ 根据返回值：若视图名则 ViewResolver 解析并渲染，若 @ResponseBody 则 HttpMessageConverter 写 body → ⑤ 响应返回。

**简图**：

```mermaid
flowchart LR
    A[HttpServletRequest] --> B[DispatcherServlet]
    B --> C[HandlerMapping]
    C --> D[HandlerExecutionChain]
    D --> E[HandlerAdapter]
    E --> F[参数解析→Controller 方法]
    F --> G{返回值类型?}
    G -->|视图名| H[ViewResolver→View 渲染]
    G -->|@ResponseBody| I[HttpMessageConverter→写 body]
    H --> J[HttpServletResponse]
    I --> J
```

**各阶段数据形态**：

| 阶段 | 输入 | 输出 |
|------|------|------|
| HandlerMapping | HttpServletRequest（URL、Method 等） | HandlerExecutionChain（HandlerMethod + 拦截器链） |
| HandlerAdapter 参数解析 | Request（Query/Body/Header/Path） | 方法入参（@RequestParam、@RequestBody、Model 等） |
| Controller 执行 | 方法入参 | 返回值（String/ModelAndView/对象） |
| 视图分支 | 逻辑视图名 + Model | View 渲染 → 写入 Response |
| JSON 分支 | 返回值对象 | HttpMessageConverter 序列化 → 写入 Response body |

> 🎯 **面试怎么说**：*「请求先进 DispatcherServlet，用 HandlerMapping 根据 URL 和 Method 找到对应的 Handler（Controller 方法），再用 HandlerAdapter 把请求里的参数绑到方法入参并执行；返回值要是视图名就走 ViewResolver 渲染，要是 @ResponseBody 就走 HttpMessageConverter 转成 JSON 写回响应。」*

### 2.1 DispatcherServlet（前端控制器）

#### （1）职责与定位

**DispatcherServlet** 是 Spring MVC 的**唯一入口**，继承 `HttpServlet`，在 Servlet 容器中通常映射为 `/`，所有匹配的 HTTP 请求都会先进入它。

| 职责 | 说明 |
|------|------|
| 接收请求 | 作为前端控制器，统一接收请求并决定后续流程。 |
| 查找 Handler | 委托 **HandlerMapping** 根据 URL、Method 等找到能处理的 Handler（Controller 方法）。 |
| 执行 Handler | 委托 **HandlerAdapter** 解析参数、调用方法、处理返回值。 |
| 渲染或写回 | 若返回视图则交给 **ViewResolver** 解析并渲染；若为 @ResponseBody 则用 **HttpMessageConverter** 写响应体。 |

> 💬 **一句话**：DispatcherServlet 不干具体业务，只负责「找谁处理 → 交给谁执行 → 结果怎么出」。

#### （2）初始化与配置

- **传统 Web**：在 `web.xml` 中配置 `<servlet>` 与 `<servlet-mapping>`，`url-pattern` 一般为 `/`；启动时加载 Spring 的 Web 应用上下文（子容器），可配合 **ContextLoaderListener** 加载根容器（业务 Bean）。
- **Spring Boot**：通过 **DispatcherServletAutoConfiguration** 自动注册 `DispatcherServlet`，默认映射 `/`；容器启动时在 `onRefresh()` 中初始化各策略组件（HandlerMapping、HandlerAdapter、ViewResolver 等）。

| 配置方式 | 说明 |
|----------|------|
| web.xml | 指定 servlet-class、init-param（contextConfigLocation）、url-pattern。 |
| JavaConfig | 实现 `WebApplicationInitializer` 或继承 `AbstractAnnotationConfigDispatcherServletInitializer` 注册。 |
| Spring Boot | 无需手写，自动配置；可通过 `spring.mvc.*` 等调整。 |

------

### 2.2 HandlerMapping（处理器映射器）

#### （1）作用与实现

**HandlerMapping** 负责根据当前请求（URL、HTTP 方法、params、headers 等）找到能处理该请求的 **Handler**（在注解驱动下通常是「Controller + 方法」封装成的 **HandlerMethod**）。

| 常用实现 | 说明 |
|----------|------|
| **RequestMappingHandlerMapping** | 解析 `@RequestMapping`、`@GetMapping` 等，维护「请求条件 → HandlerMethod」的映射表，注解驱动下默认使用。 |
| BeanNameUrlHandlerMapping | 根据 Bean 名称与 URL 映射，较少用。 |

查找结果封装为 **HandlerExecutionChain**（包含 Handler 与匹配到的 **HandlerInterceptor** 列表），供 DispatcherServlet 后续调用。

#### （2）面试要点

- 映射表在应用启动时由 `RequestMappingHandlerMapping` 扫描 `@Controller` 并解析注解生成；请求进来时按条件匹配，得到唯一 Handler（或无则 404）。

------

### 2.3 HandlerAdapter（处理器适配器）

#### （1）为什么需要适配器

Handler 可能是 `@RequestMapping` 方法、`Controller` 接口实现、Servlet 等不同类型；**HandlerAdapter** 屏蔽差异，统一完成「解析请求参数 → 调用 Handler → 处理返回值」三步。

| 步骤 | 对应组件 / 行为 |
|------|------------------|
| 参数解析 | **HandlerMethodArgumentResolver** 链：将 `HttpServletRequest`、Body、Header 等解析为方法参数（支持 @RequestParam、@RequestBody、@PathVariable 等）。 |
| 调用方法 | 通过反射执行 Controller 方法。 |
| 返回值处理 | **HandlerMethodReturnValueHandler** 链：将返回值转为 ModelAndView 或直接写响应（如 @ResponseBody 走 HttpMessageConverter）。 |

#### （2）常用实现

| 实现 | 说明 |
|------|------|
| **RequestMappingHandlerAdapter** | 注解驱动下的默认适配器，配合 `RequestMappingHandlerMapping` 使用；内部注册多种 ArgumentResolver 与 ReturnValueHandler。 |

> 💡 **重点**：参数绑定、类型转换、@RequestBody / @ResponseBody 的读写，都在 HandlerAdapter 这一层完成，而不是在 DispatcherServlet 里。

------

### 2.4 Controller 与 @Controller / @RestController

#### （1）概念与区别

| 注解 | 含义 | 方法返回值默认行为 |
|------|------|---------------------|
| **@Controller** | 标识为 Spring MVC 控制器 Bean，可被 HandlerMapping 扫描。 | 返回值作为**逻辑视图名**，由 ViewResolver 解析；若方法上再加 **@ResponseBody** 则直接写响应体。 |
| **@RestController** | 等价于 @Controller + 类上默认 @ResponseBody。 | 返回值直接写响应体（通常由 HttpMessageConverter 序列化为 JSON），**不经过** ViewResolver。 |

#### （2）使用场景与示例

- **@Controller**：传统 MVC、服务端渲染页面（JSP、Thymeleaf），方法返回视图名。
- **@RestController**：前后端分离、RESTful API，统一返回 JSON。

```java
// 传统视图：返回视图名 "user/list"，由 ViewResolver 解析
@Controller
public class UserPageController {
    @GetMapping("/user/list")
    public String list(Model model) {
        model.addAttribute("users", userService.findAll());
        return "user/list";
    }
}

// RESTful API：直接返回对象，序列化为 JSON
@RestController
@RequestMapping("/api/user")
public class UserApiController {
    @GetMapping("/list")
    public List<User> list() {
        return userService.findAll();
    }
}
```

### 2.5 Model、ModelMap、ModelAndView

#### （1）作用与区别

| 类型 | 作用 | 典型用法 |
|------|------|----------|
| **Model / ModelMap** | 向视图传递数据的键值容器，可作方法参数由框架注入。 | 在方法里 `model.addAttribute("key", value)`，返回视图名后模板中通过 key 访问。 |
| **ModelAndView** | 同时封装「逻辑视图名」与「Model 数据」。 | 一次 return 既指定视图又带数据；也可先 new ModelAndView()，再 setViewName、addObject。 |

二者本质都是把数据放到请求域（request scope），供 View 渲染时使用。

#### （2）与前后端分离的关系

> 🔹 **注意**：前后端分离下接口通常用 **@ResponseBody** 或 **@RestController** 直接返回 DTO/Result，不经过视图名，因此很少再用 Model/ModelAndView；模型数据更多体现在「返回值对象」上。

------

### 2.6 ViewResolver（视图解析器）

#### （1）职责与常见实现

**ViewResolver** 将**逻辑视图名**（如 `"user/list"`）解析为具体的 **View** 实现，由 View 负责渲染（如生成 HTML、转发 JSP）。

| 实现 | 说明 |
|------|------|
| **InternalResourceViewResolver** | 将视图名解析为 JSP 或内部转发路径（prefix + viewName + suffix）。 |
| **ThymeleafViewResolver** | 配合 Thymeleaf 模板引擎，解析为 ThymeleafView。 |
| **ContentNegotiatingViewResolver** | 根据请求的 Accept 或扩展名协商使用哪个 ViewResolver。 |

#### （2）何时不经过 ViewResolver

- 方法带 **@ResponseBody** 或类为 **@RestController**：返回值由 **HttpMessageConverter** 直接写响应体，**不经过** ViewResolver。
- 返回 **ResponseEntity** 且 body 非视图名时，同样由 MessageConverter 或直接写流处理。

> 💡 **面试常问**：返回 JSON 时还会走 ViewResolver 吗？答：不会，走 HttpMessageConverter。

------

## ✅ 本章小结

| 知识点 | 面试关键词 | 实际应用 |
|--------|------------|----------|
| DispatcherServlet | 前端控制器、唯一入口、调度 Mapping/Adapter/ViewResolver | 说明一次请求从哪里进、谁在协调 |
| HandlerMapping | 根据 URL/Method 找 Handler、RequestMappingHandlerMapping、映射表 | 解释「请求如何找到对应方法」 |
| HandlerAdapter | 参数解析、调用方法、返回值处理；ArgumentResolver、ReturnValueHandler | 说明参数绑定与 @ResponseBody 在哪儿完成 |
| @Controller / @RestController | 视图名 vs 直接写 body；@RestController = @Controller + @ResponseBody | 选型：页面用 @Controller，接口用 @RestController |
| Model / ModelAndView | 传数据给视图、请求域；前后端分离少用 | 服务端渲染时向模板传数据 |
| ViewResolver | 逻辑视图名 → View；返回 JSON 不经过 ViewResolver | 区分「渲染页面」与「写 JSON」的路径 |

------

**学习要点**：

- DispatcherServlet 是唯一入口，只做调度，不写业务；初始化时加载 Web 容器并初始化各策略 Bean。
- HandlerMapping 负责「找 Handler」，HandlerAdapter 负责「解析参数、执行方法、处理返回值」；二者配合完成从请求到方法调用的桥梁。
- @Controller 与 @RestController 区别在于返回值是否作为视图名；前后端分离接口多用 @RestController，模型数据通过返回值体现。
- ViewResolver 只处理「返回视图名」的分支；返回 @ResponseBody 或 ResponseEntity 时由 HttpMessageConverter 写 body，不经过 ViewResolver。

## 🎯 面试常见追问

| 面试官提问 | 回答思路 |
|------------|----------|
| DispatcherServlet 做了什么？ | 前端控制器：接收请求 → 交给 HandlerMapping 查 Handler → 交给 HandlerAdapter 执行（参数解析、方法调用、返回值处理）→ 根据返回值走 ViewResolver 渲染视图或 HttpMessageConverter 写 body。 |
| HandlerMapping 和 HandlerAdapter 的区别？ | HandlerMapping 负责「根据请求找哪个 Handler」；HandlerAdapter 负责「怎么调这个 Handler」（解析参数、执行、处理返回值）。一个管映射，一个管执行。 |
| @Controller 和 @RestController 的区别？ | @RestController = @Controller + 类级别 @ResponseBody；前者方法默认返回视图名走 ViewResolver，后者默认直接写响应体走 HttpMessageConverter，用于 RESTful API。 |
| 返回 JSON 时还经过 ViewResolver 吗？ | 不经过。方法带 @ResponseBody 或类为 @RestController 时，返回值由 HttpMessageConverter（如 Jackson）序列化后直接写响应体，不经过 ViewResolver。 |

### 常见坑与注意点

| 现象 / 易错点 | 原因 | 怎么改 / 怎么记 |
|---------------|------|-----------------|
| 期望返回 JSON 却得到 404 或视图名 | 未加 **@ResponseBody**，返回值被当逻辑视图名交给 ViewResolver 解析 | 接口类用 **@RestController** 或方法上加 **@ResponseBody**。 |
| Controller 单例导致请求间数据串了 | 在 Controller 里用**实例变量**存请求相关数据，多请求共享 | 只用**方法参数、局部变量**或 ThreadLocal，不在单例 Controller 上存请求状态。 |
| HandlerMapping 找不到 Handler | URL 或 HTTP 方法不匹配、Controller 未被扫描、路径重复或冲突 | 检查 **@RequestMapping** 路径与 method、包扫描、是否有多个方法匹配同一 URL。 |

## 第三章：Spring MVC 核心注解与请求映射

------

> 本章对应第二章中 HandlerAdapter 的「参数解析」与「返回值处理」：掌握请求映射、参数绑定与返回值注解，即可写出规范的后台接口；与第四章数据绑定、第九章 JSON 配合使用。

------

### 3.1 控制层与请求映射

#### （1）@RequestMapping 与 HTTP 方法简写

**@RequestMapping** 可标注在类与方法上，用于指定 URL 路径、HTTP 方法、params、headers 等匹配条件；类上的路径与方法上的路径会**拼接**成最终映射路径。

| 属性 | 说明 |
|------|------|
| value / path | URL 路径，支持 Ant 风格（如 `/user/*`）与路径变量（如 `/user/{id}`）。 |
| method | 请求方法：GET、POST、PUT、DELETE 等；不指定则所有方法都接受（不推荐）。 |
| params / headers | 进一步限定请求参数或请求头，满足才匹配。 |

**@GetMapping / @PostMapping / @PutMapping / @DeleteMapping** 是 `@RequestMapping(method = xxx)` 的简写，语义更清晰，推荐使用。

#### （2）@Controller 与 @RestController

| 注解 | 说明 | 方法返回值默认处理 |
|------|------|---------------------|
| **@Controller** | 标识为控制器 Bean，可被 HandlerMapping 扫描。 | 作为逻辑视图名，走 ViewResolver。 |
| **@RestController** | 等价于 @Controller + 类级别 @ResponseBody。 | 直接写响应体，走 HttpMessageConverter（如 JSON）。 |

前后端分离接口层建议统一使用 **@RestController**。

------

### 3.2 参数绑定

#### （1）常用参数注解

| 注解 | 绑定来源 | 说明 |
|------|----------|------|
| **@RequestParam** | 查询参数、表单字段 | 可指定 name、required、defaultValue；多参数可绑到 Map 或 POJO（属性名与参数名一致）。 |
| **@PathVariable** | URI 路径变量 | 如 `/user/{id}` 中的 id，REST 风格必备。 |
| **@RequestBody** | 请求体（通常 JSON） | 由 HttpMessageConverter 反序列化为对象；Content-Type 需为 application/json 等。 |
| **@RequestHeader** | 请求头 | 如 @RequestHeader("Authorization")。 |
| **@CookieValue** | Cookie | 按 name 绑定。 |

#### （2）示例

```java
@RestController
@RequestMapping("/api/user")
public class UserController {

    @GetMapping("/{id}")
    public User get(@PathVariable Long id) {
        return userService.getById(id);
    }

    @GetMapping("/list")
    public List<User> list(@RequestParam(defaultValue = "1") int page,
                           @RequestParam(defaultValue = "10") int size) {
        return userService.page(page, size);
    }

    @PostMapping
    public User create(@RequestBody @Valid UserDTO dto) {
        return userService.create(dto);
    }
}
```

------

### 3.3 返回值处理

#### （1）@ResponseBody 与 ResponseEntity

| 方式 | 说明 |
|------|------|
| **@ResponseBody** | 方法返回值由 HttpMessageConverter 序列化后写入响应体；@RestController 下可省略。 |
| **ResponseEntity<T>** | 可同时设置状态码、响应头、body，适合需要精确控制 HTTP 响应的场景（如 201 Created、自定义 Header）。 |

#### （2）ResponseEntity 示例

```java
@PostMapping
public ResponseEntity<User> create(@RequestBody UserDTO dto) {
    User user = userService.create(dto);
    return ResponseEntity.status(HttpStatus.CREATED)
            .header("Location", "/api/user/" + user.getId())
            .body(user);
}
```

------

### 3.4 实战要点与注意

- **GET 参数**：简单参数用 @RequestParam；多参数且命名规范可用 POJO 直接接收（属性名与参数名一致）。
- **POST JSON**：方法参数用 **@RequestBody**，并保证请求 Content-Type 为 `application/json`；校验可加 **@Valid**（见第四章）。
- **REST 路径**：路径变量用 **@PathVariable**，与 @RequestMapping 中的 `{id}` 对应。
- **文件上传**：参数类型为 **MultipartFile** 或 **MultipartFile[]**（见第八章）。

> 🔹 **注意**：@RequestBody 与 @RequestParam 不能同时用于同一请求体；表单提交用参数或 POJO 绑定，JSON 用 @RequestBody。

------

## ✅ 本章小结

| 知识点 | 面试关键词 | 实际应用 |
|--------|------------|----------|
| @RequestMapping | path、method、params、headers；Get/PostMapping 简写 | 声明 URL 与 HTTP 方法 |
| 参数绑定 | @RequestParam、@PathVariable、@RequestBody、@RequestHeader | GET 参数、路径变量、POST JSON、Header |
| 返回值 | @ResponseBody、ResponseEntity、状态码与 Header | 统一 JSON 返回或精确控制响应 |
| @Controller / @RestController | 视图名 vs 直接写 body | 接口层用 @RestController |

------

**学习要点**：

- 请求映射用 @RequestMapping 或 @GetMapping 等，类路径 + 方法路径拼接为最终 URL。
- GET 用 @RequestParam 或 POJO；POST JSON 用 @RequestBody；REST 路径用 @PathVariable。
- 接口层用 @RestController，返回值由 HttpMessageConverter 写 JSON；需控制状态码或 Header 时用 ResponseEntity。

| 如何接收 REST 风格路径中的变量？ | 在 @RequestMapping 路径中用 `{变量名}`，方法参数用 @PathVariable 绑定同名变量。 |

## 第四章：数据绑定与类型转换机制

------

> 本章说明请求参数如何绑定到方法参数或命令对象、类型如何转换、以及如何做参数校验；与第三章注解配合，是「参数能正确进 Controller」的基础。

------

### 4.1 数据绑定流程

#### （1）WebDataBinder 与绑定过程

**WebDataBinder** 在 HandlerAdapter 执行时参与参数解析：将请求中的字符串参数（Query、表单）转换为方法参数或命令对象的属性类型，完成**数据绑定**。

| 环节 | 说明 |
|------|------|
| 类型转换 | 内置常见类型：String→Integer、Long、Boolean、日期等；复杂类型可注册 **Converter** 或 **Formatter**。 |
| 数据绑定 | 参数名与对象属性名一致时自动绑定；支持嵌套（如 `order.items[0].name`）。 |
| 校验（可选） | 绑定后可触发 JSR-303/380 校验（见 4.4）。 |

> 💬 **一句话**：WebDataBinder 负责「请求里的字符串 → 方法参数/对象属性」的转换与填充。

#### （2）与 @RequestBody 的区别

- **@RequestBody**：请求体由 **HttpMessageConverter** 反序列化（如 Jackson 转 JSON），不经过 WebDataBinder。
- **Query / 表单参数**：由 **HandlerMethodArgumentResolver** 配合 WebDataBinder 做类型转换与绑定。

------

### 4.2 自定义类型转换器

#### （1）Converter 与 Formatter

| 接口 | 说明 | 典型用途 |
|------|------|----------|
| **Converter<S, T>** | 任意类型 S → T | 字符串转枚举、字符串转自定义对象。 |
| **Formatter<T>** | String ↔ T，可带 Locale | 日期、金额等与 locale 相关的格式化。 |

#### （2）注册方式

- **全局**：实现 **WebMvcConfigurer**，在 `addFormatters(FormatterRegistry registry)` 中注册 Converter 或 Formatter。
- **局部**：在 Controller 方法上使用 **@InitBinder**，仅对该 Controller 生效。

```java
// 全局注册示例
@Configuration
public class WebConfig implements WebMvcConfigurer {
    @Override
    public void addFormatters(FormatterRegistry registry) {
        registry.addConverter(new StringToEnumConverter());
    }
}
```

------

### 4.3 表单与复杂类型绑定

- **嵌套对象**：如 `order.items[0].name`、`user.address.city`，属性名与参数名一致即可绑定。
- **日期**：默认格式可能不匹配，可在字段上使用 **@DateTimeFormat(pattern = "yyyy-MM-dd")** 或全局配置日期格式。
- **枚举**：默认按 name() 匹配；可自定义 Converter 实现字符串与枚举的转换。
- **集合**：如 `List<Long> ids`，请求传 `ids=1&ids=2` 即可。

------

### 4.4 数据校验（JSR-303 / JSR-380）

#### （1）常用注解与用法

在 DTO 字段上使用 **JSR-303/380**（如 **javax.validation** 或 **jakarta.validation**）注解：

| 注解 | 说明 |
|------|------|
| @NotNull、@NotBlank、@NotEmpty | 非空约束。 |
| @Size、@Min、@Max | 长度、数值范围。 |
| @Email、@Pattern | 格式校验。 |

Controller 方法参数加 **@Valid** 或 **@Validated** 触发校验；校验失败时抛出 **MethodArgumentNotValidException**（@RequestBody）或 **BindException**（表单绑定），可由全局 **@ExceptionHandler** 捕获并统一返回 400 与错误信息（见第七章）。

#### （2）BindingResult

- 若在方法参数中紧跟 @Valid 参数后声明 **BindingResult**，校验失败不会抛异常，错误信息进入 BindingResult，由业务代码自行处理；不声明则交给全局异常处理更常见。

------

## ✅ 本章小结

| 知识点 | 面试关键词 | 实际应用 |
|--------|------------|----------|
| WebDataBinder | 参数绑定、类型转换、Converter/Formatter | 理解 Query/表单参数如何变成方法参数 |
| 自定义转换 | Converter、Formatter、addFormatters、@InitBinder | 日期、枚举、自定义类型绑定 |
| 数据校验 | @Valid、JSR-303/380、MethodArgumentNotValidException | DTO 校验 + 全局异常统一返回 |

------

**学习要点**：

- 请求参数绑定与类型转换由 WebDataBinder 参与；@RequestBody 走 HttpMessageConverter，不走 WebDataBinder。
- 自定义类型转换实现 Converter 或 Formatter，通过 WebMvcConfigurer.addFormatters 或 @InitBinder 注册。
- 校验在 DTO 上写注解，方法参数加 @Valid 触发；与 @RestControllerAdvice 配合可统一返回校验错误。

------

### 常见坑与注意点

| 现象 / 易错点 | 原因 | 怎么改 / 怎么记 |
|---------------|------|-----------------|
| 日期/枚举绑定失败或格式不对 | 默认格式与前端传的不一致，或枚举按 name 匹配失败 | 用 **@DateTimeFormat** 或全局 Converter/Formatter 统一日期格式；枚举自定义 **Converter** 或传枚举 name。 |
| @Valid 不生效、校验未执行 | 未在方法参数上加 @Valid/@Validated，或依赖未引入 | 入参加 **@Valid**（或 @Validated）；确保引入 **javax.validation** / **jakarta.validation** 与 **hibernate-validator**。 |
| 嵌套对象/集合绑定为 null | 参数名或下标与请求 key 不一致（如 items[0].name） | 请求 key 与 DTO 属性名、层级一致；集合用 **ids=1&ids=2** 或 **items[0].name=xx**。 |

## 第五章：模型数据传递与作用域管理

------

> 本章说明如何向视图传递数据（Model/ModelAndView）、如何用 Session 存跨请求数据，以及前后端分离下常用的统一响应封装 Result；与第二章 ViewResolver、第七章异常处理关联。

------

### 5.1 Model、ModelMap、ModelAndView

#### （1）概念与用法

| 类型 | 说明 | 典型用法 |
|------|------|----------|
| **Model / ModelMap** | 键值容器，可作方法参数由框架注入，数据放入**请求域**。 | `model.addAttribute("key", value)`，返回视图名后模板中通过 key 访问。 |
| **ModelAndView** | 同时封装「逻辑视图名」与「Model 数据」。 | 一次 return 既指定视图又带数据；new ModelAndView("viewName").addObject("key", value)。 |

二者本质都是把数据放到 **request scope**，供 View 渲染时使用；前后端分离接口多用 @ResponseBody 直接返回对象，不经过视图名，因此较少使用。

#### （2）与第二章的对应关系

- 返回视图名（String 或 ModelAndView）时，DispatcherServlet 会交给 ViewResolver 解析，此时 Model 中的数据会随请求传给 View。

------

### 5.2 Session 与 @SessionAttributes

#### （1）@SessionAttributes

- **@SessionAttributes(names = {"key1", "key2"})**：将本次请求 Model 中指定 name 的数据**同步到 Session**，便于跨请求使用（如向导式表单、多步提交）。
- 清除：在适当时机调用 **SessionStatus.setComplete()**，清除本次 Session 中由该注解管理的属性（不销毁整个 Session）。

#### （2）前后端分离下的选择

> 🔹 **注意**：前后端分离项目通常用 **Token**（如 JWT）存登录态，少用 Session；若仍用 Session，需注意跨域与 Cookie 配置。

------

### 5.3 统一响应封装（Result）

#### （1）作用与结构

前后端分离下，接口通常统一返回固定结构，便于前端统一处理成功/失败与错误信息。

| 字段 | 说明 |
|------|------|
| code | 业务或 HTTP 状态码，如 200 成功、400 参数错误。 |
| msg | 提示信息。 |
| data | 业务数据，成功时填充。 |

#### （2）示例与配合

- Controller 正常返回：`return Result.success(data)`；失败或业务异常：`return Result.fail(code, msg)`。
- 配合 **@RestControllerAdvice + @ExceptionHandler**（见第七章）：将异常也封装为 Result，保证所有接口响应结构一致。

```java
// 统一响应示例
public class Result<T> {
    private int code;
    private String msg;
    private T data;
    public static <T> Result<T> success(T data) { ... }
    public static <T> Result<T> fail(int code, String msg) { ... }
}
```

------

## ✅ 本章小结

| 知识点 | 面试关键词 | 实际应用 |
|--------|------------|----------|
| Model / ModelAndView | 请求域、视图传参、viewName + model | 服务端渲染时向模板传数据 |
| @SessionAttributes | Model 指定 key 进 Session、SessionStatus.setComplete | 跨请求数据（少用，前后端分离多用 Token） |
| Result 封装 | code、msg、data；与全局异常配合 | 前后端分离统一响应结构 |

------

**学习要点**：

- Model/ModelMap 用于向视图传数据，数据在请求域；ModelAndView 同时带视图名与数据。
- @SessionAttributes 将 Model 中指定 key 放入 Session；前后端分离更常用 Token。
- 统一响应 Result + 全局异常处理，保证接口返回结构一致。

- **上一章**：第四章是数据绑定与校验；本章是**向视图或 Session 传数据**、以及**统一响应 Result**，与第二章 Model/ViewResolver、第七章异常返回 Result 配合。

## 第六章：拦截器（HandlerInterceptor）

------

> 本章说明如何在 Handler 执行前后插入逻辑（登录校验、权限、日志等），以及如何配置与执行顺序；与第十章 Filter、AOP 对比一起看，便于选型与面试回答。

------

### 6.1 作用与接口

#### （1）HandlerInterceptor 三个方法

| 方法 | 执行时机 | 返回值/作用 |
|------|----------|-------------|
| **preHandle** | Handler 执行**前** | 返回 true 继续链，返回 false 则中断请求（后续拦截器与 Handler 不再执行）。 |
| **postHandle** | Handler 执行**后**、视图渲染前 | 可修改 ModelAndView；若为 @ResponseBody 则无 ModelAndView。 |
| **afterCompletion** | 请求**完全结束后**（视图已渲染或 body 已写） | 可做清理、记录耗时等；无论是否异常都会执行。 |

#### （2）典型用途

- **登录/权限**：preHandle 中校验 Token 或 Session，未通过则返回 false 并写 401/403。
- **日志与耗时**：preHandle 记录开始时间，afterCompletion 计算耗时、打日志。
- **跨域、编码**：也可在 Filter 做（见第十章）。

> 💡 **重点**：登录态常放在拦截器里解析 Token，将用户信息写入 **ThreadLocal**，Controller 或 Service 中可直接获取。

------

### 6.2 配置与执行顺序

#### （1）注册方式

通过 **WebMvcConfigurer.addInterceptors** 注册，指定拦截路径与排除路径：

```java
@Configuration
public class WebConfig implements WebMvcConfigurer {
    @Override
    public void addInterceptors(InterceptorRegistry registry) {
        registry.addInterceptor(new AuthInterceptor())
                .addPathPatterns("/api/**")
                .excludePathPatterns("/api/login", "/api/register");
    }
}
```

| 方法 | 说明 |
|------|------|
| addPathPatterns | 需要拦截的路径，支持 Ant 风格（如 `/api/**`）。 |
| excludePathPatterns | 排除的路径，如登录、注册、静态资源。 |

#### （2）多拦截器执行顺序

- **preHandle**：按**注册顺序**依次执行；任一返回 false，后续拦截器与 Handler 都不执行。
- **postHandle**：按注册**逆序**执行（仅当 preHandle 都返回 true 且 Handler 已执行）。
- **afterCompletion**：按注册**逆序**执行，且只要 preHandle 返回 true 的拦截器都会执行 afterCompletion（用于清理）。

------

### 6.3 与 Filter、AOP 区别（简述）

| 维度 | Filter | HandlerInterceptor | AOP |
|------|--------|--------------------|-----|
| 层级 | Servlet 容器 | Spring MVC（DispatcherServlet 内） | Spring AOP（任意 Bean 方法） |
| 可获取 | Request/Response | Request/Response、Handler、ModelAndView | 方法参数、返回值；拿不到 Request 需 RequestContextHolder |
| 典型场景 | 编码、CORS、全局日志 | 登录、权限、请求前后逻辑 | 业务无关的日志、耗时、事务 |

详细对比与选型见第十章。

## ✅ 本章小结

| 知识点 | 面试关键词 | 实际应用 |
|--------|------------|----------|
| HandlerInterceptor | preHandle、postHandle、afterCompletion；返回 false 中断 | 登录校验、权限、日志、耗时 |
| 配置 | addInterceptors、addPathPatterns、excludePathPatterns | 指定拦截与排除路径 |
| 执行顺序 | preHandle 顺序、postHandle/afterCompletion 逆序 | 理解多拦截器时的调用顺序 |

------

**学习要点**：

- preHandle 返回 false 会中断请求；afterCompletion 无论是否异常都会执行，适合做清理与耗时统计。
- 通过 WebMvcConfigurer.addInterceptors 注册，注意排除登录等接口。
- 与 Filter、AOP 的区别：拦截器在 MVC 层，可拿 Handler 与 Model；Filter 更早；AOP 拿不到 Request 需另取。

## 第七章：异常处理机制

------

> 本章说明如何在 Controller 层与全局统一处理异常，并封装为统一响应（如 Result）；与第四章数据校验、第五章 Result 封装配合，是前后端分离项目的标配。

------

### 7.1 局部与全局异常

#### （1）@ExceptionHandler（局部）

在**单个 Controller** 内定义方法，用 **@ExceptionHandler(异常类型.class)** 标注，可处理该 Controller 中抛出的指定异常；方法可接收异常对象、Request 等，返回值与普通 Controller 方法一致（视图名或 @ResponseBody）。

#### （2）@ControllerAdvice + @ExceptionHandler（全局）

| 注解 | 说明 |
|------|------|
| **@ControllerAdvice** | 标识「全局增强类」，可配合 @ExceptionHandler、@InitBinder、@ModelAttribute 等。 |
| **@RestControllerAdvice** | 等价于 @ControllerAdvice + @ResponseBody，类内 @ExceptionHandler 方法直接写响应体（如 JSON）。 |

全局异常处理类会被 DispatcherServlet 识别，当任意 Controller 抛出异常且未被局部 @ExceptionHandler 处理时，会匹配全局类中的 @ExceptionHandler 方法。

> 💬 **一句话**：**@RestControllerAdvice + @ExceptionHandler** 是前后端分离项目里「统一把异常转成 Result JSON」的常规写法。

------

### 7.2 异常分类与统一响应

#### （1）常见异常与映射

| 异常类型 | 典型来源 | 建议处理 |
|----------|----------|----------|
| **MethodArgumentNotValidException** | @RequestBody + @Valid 校验失败 | 返回 400，body 为 Result 或错误列表（可从 BindingResult 取字段级错误）。 |
| **BindException** | 表单绑定 + @Valid 校验失败 | 同上，返回 400。 |
| **BusinessException**（自定义） | 业务逻辑抛出的业务异常 | 返回业务 code + msg，HTTP 状态码可按约定（如 200 或 4xx）。 |
| **Exception** | 未预料异常 | 兜底：打日志，对外返回友好提示，避免泄露堆栈。 |

#### （2）示例：全局类结构

```java
@RestControllerAdvice
public class GlobalExceptionHandler {

    @ExceptionHandler(MethodArgumentNotValidException.class)
    public Result<?> handleValid(MethodArgumentNotValidException e) {
        // 从 e.getBindingResult() 取字段错误，封装为 Result
        return Result.fail(400, "参数校验失败");
    }

    @ExceptionHandler(BusinessException.class)
    public Result<?> handleBusiness(BusinessException e) {
        return Result.fail(e.getCode(), e.getMessage());
    }

    @ExceptionHandler(Exception.class)
    public Result<?> handleOther(Exception e) {
        log.error("unexpected", e);
        return Result.fail(500, "系统繁忙");
    }
}
```

------

### 7.3 实战要点

- **全局类**：用 **@RestControllerAdvice**，所有方法返回 Result 或统一结构。
- **校验异常**：优先处理 **MethodArgumentNotValidException**、**BindException**，将字段错误整理后返回，便于前端展示。
- **兜底**：**@ExceptionHandler(Exception.class)** 放最后，只记录日志并返回通用错误，不暴露堆栈。
- **业务异常**：定义 **BusinessException** 与错误码枚举，在 Service 中抛出，由全局 Handler 转成 Result。

------

## ✅ 本章小结

| 知识点 | 面试关键词 | 实际应用 |
|--------|------------|----------|
| @ExceptionHandler | 局部处理指定异常 | 单 Controller 内异常处理 |
| @ControllerAdvice / @RestControllerAdvice | 全局增强、全局异常处理 | 统一把异常转成 Result |
| 异常分类 | MethodArgumentNotValidException、BusinessException、兜底 Exception | 校验失败 400、业务异常 code、未知异常 500 |

------

**学习要点**：

- 全局异常用 @RestControllerAdvice + @ExceptionHandler，返回值与正常接口一致（如 Result）。
- 校验失败（@Valid）对应 MethodArgumentNotValidException/BindException，应统一返回 400 与错误信息。
- 兜底处理 Exception，只打日志、返回友好提示，避免泄露内部信息。

| 面试官提问 | 回答思路 |
|------------|----------|
| 如何做全局异常处理？ | 定义类加 @RestControllerAdvice，方法加 @ExceptionHandler(异常类型.class)，返回值即响应体（如 Result）；可按异常类型分方法处理。 |
| @Valid 校验失败抛什么异常？ | @RequestBody 校验失败抛 MethodArgumentNotValidException；表单绑定校验失败抛 BindException；在全局 Handler 里统一捕获并返回 400 与错误信息。 |
| 统一返回和异常怎么配合？ | 正常接口返回 Result.success(data)；异常在 @ExceptionHandler 里返回 Result.fail(code, msg)，保证前端拿到的结构一致。 |

### 常见坑与注意点

| 现象 / 易错点 | 原因 | 怎么改 / 怎么记 |
|---------------|------|-----------------|
| 全局 @ExceptionHandler 不生效 | 全局类未在包扫描路径、或未加 @RestControllerAdvice、或异常被局部先处理 | 确保全局类在 **ComponentScan** 范围内并加 **@RestControllerAdvice**；局部 @ExceptionHandler 优先，可只保留全局。 |
| 校验异常返回 500 而非 400 | 未单独处理 MethodArgumentNotValidException/BindException，被兜底 Exception 捕获 | 先写 **@ExceptionHandler(MethodArgumentNotValidException.class)** 和 **BindException**，返回 400 与错误信息；再写 Exception 兜底。 |
| 异常信息直接返回堆栈给前端 | 兜底里 e.getMessage() 或 printStackTrace 暴露内部信息 | 兜底只打 **log**，对外返回统一「系统繁忙」等提示，不返回堆栈或内部异常信息。 |

## 第八章：文件上传与下载

------

> 本章说明 multipart 请求的解析、上传接口写法与安全注意点，以及如何以流式方式返回文件下载；与第三章参数绑定、第七章异常处理配合使用。

------

### 8.1 MultipartResolver 与 MultipartFile

#### （1）MultipartResolver

当请求 **Content-Type** 为 **multipart/form-data** 时，**MultipartResolver** 会解析请求体，将文件部分封装为 **MultipartFile**（或 **MultipartFile[]**），供 Controller 方法参数绑定。

| 实现 | 说明 |
|------|------|
| **StandardServletMultipartResolver** | 基于 Servlet 3.0+ 的 Part API，Spring Boot 默认使用。 |
| CommonsMultipartResolver | 依赖 commons-fileupload，需额外引入。 |

#### （2）配置（Spring Boot）

在 **application.yml** 或 **application.properties** 中配置 **spring.servlet.multipart**：

| 配置项 | 说明 |
|--------|------|
| max-file-size | 单个文件最大大小。 |
| max-request-size | 整个请求最大大小（含多文件）。 |
| enabled | 是否启用 multipart 解析，默认 true。 |

------

### 8.2 上传实现

#### （1）Controller 写法

方法参数类型为 **MultipartFile** 或 **MultipartFile[]**，参数名与表单 file 的 name 一致；可用 **@RequestParam("file")** 显式指定。

```java
@PostMapping("/upload")
public Result<String> upload(@RequestParam("file") MultipartFile file) {
    if (file.isEmpty()) return Result.fail(400, "文件为空");
    String savedPath = fileService.save(file); // 自定义：校验、重命名、落盘
    return Result.success(savedPath);
}
```

#### （2）常用 API 与安全注意

| API | 说明 |
|-----|------|
| getOriginalFilename() | 原始文件名（不可信，需校验）。 |
| getInputStream() / getBytes() | 读取内容。 |
| transferTo(File dest) | 将文件写入目标文件。 |

> ⚠️ **安全**：不要直接使用原始文件名落盘，避免路径穿越（如 `../etc/passwd`）；建议重命名为 **UUID** 或「时间戳 + 后缀」，并限制允许的后缀与 MIME 类型。

------

### 8.3 文件下载

#### （1）方式一：ResponseEntity<Resource>

返回 **ResponseEntity<Resource>**，设置 **Content-Disposition: attachment; filename="xxx"**；文件名若含中文需 **URL 编码**（如 RFC 5987）。

```java
@GetMapping("/download/{filename}")
public ResponseEntity<Resource> download(@PathVariable String filename) {
    Resource resource = fileService.load(filename);
    return ResponseEntity.ok()
            .header(HttpHeaders.CONTENT_DISPOSITION,
                    "attachment; filename=\"" + URLEncoder.encode(filename, UTF_8) + "\"")
            .body(resource);
}
```

#### （2）方式二：HttpServletResponse 写流

直接注入 **HttpServletResponse**，设置 Content-Type、Content-Disposition，用 **OutputStream** 写出文件流；适合从别处读流再写出。

|--------|------------|----------|
| MultipartResolver | multipart/form-data、MultipartFile、StandardServletMultipartResolver | 解析上传请求 |
| 上传 | getOriginalFilename、transferTo；文件名安全、重命名 | 接收文件并落盘 |
| 下载 | ResponseEntity<Resource>、Content-Disposition、filename 编码 | 返回文件流供浏览器下载 |

------

**学习要点**：

- 上传请求为 multipart/form-data，由 MultipartResolver 解析为 MultipartFile；Spring Boot 默认已配置，注意 max-file-size。
- 保存时不要信任原始文件名，重命名为 UUID 等并校验类型与大小。
- 下载用 ResponseEntity<Resource> 或 Response 写流，并设置 Content-Disposition。

## 第九章：Spring MVC 与 JSON 数据交互

------

> 本章说明 @RequestBody / @ResponseBody 背后的 **HttpMessageConverter** 与 Jackson、以及如何控制序列化行为；对应第二章「返回 JSON 不经过 ViewResolver」的实现细节。

------

### 9.1 HttpMessageConverter

#### （1）作用与链路

**HttpMessageConverter** 负责将 HTTP 请求体**读成** Java 对象（@RequestBody）、将 Java 对象**写成**响应体（@ResponseBody）；**RequestMappingHandlerAdapter** 在解析参数与处理返回值时调用相应的 Converter。

| 场景 | 使用的 Converter |
|------|-------------------|
| @RequestBody 反序列化 JSON | 根据 Content-Type 选择，通常 **MappingJackson2HttpMessageConverter**。 |
| @ResponseBody 序列化 JSON | 根据 Accept 与返回值类型选择，通常同上，依赖 **ObjectMapper**。 |

Spring Boot 默认引入 **Jackson**，自动注册 **MappingJackson2HttpMessageConverter**，因此无需额外配置即可支持 JSON。

------

### 9.2 序列化控制（Jackson 常用注解）

| 注解 | 说明 |
|------|------|
| **@JsonIgnore** | 序列化/反序列化时忽略该字段。 |
| **@JsonProperty("name")** | 指定 JSON 中字段名。 |
| **@JsonFormat** | 日期、时间等格式，如 `@JsonFormat(pattern = "yyyy-MM-dd HH:mm:ss")`。 |
| **@JsonInclude** | 控制何时包含该字段（如非空才包含）。 |

**循环引用**：双向关联时易导致无限递归；可用 **@JsonManagedReference / @JsonBackReference** 打断一方，或在一侧 **@JsonIgnore**。

------

### 9.3 全局 ObjectMapper

如需统一日期格式、空值策略、 Long 转 String 防前端精度丢失等，可自定义 **ObjectMapper** Bean 或通过 **WebMvcConfigurer.configureMessageConverters** 注册使用该 ObjectMapper 的 **MappingJackson2HttpMessageConverter**；Spring Boot 会使用你注册的 Bean 参与 JSON 读写。

```java
@Bean
public ObjectMapper objectMapper() {
    ObjectMapper om = new ObjectMapper();
    om.setDateFormat(new SimpleDateFormat("yyyy-MM-dd HH:mm:ss"));
    om.setSerializationInclusion(JsonInclude.Include.NON_NULL);
    return om;
}
```

------

| 知识点 | 面试关键词 | 实际应用 |
|--------|------------|----------|
| HttpMessageConverter | @RequestBody/@ResponseBody 的读写、MappingJackson2HttpMessageConverter | 理解 JSON 如何进、出 Controller |
| Jackson 注解 | @JsonIgnore、@JsonProperty、@JsonFormat、循环引用 | 控制字段是否输出、命名、日期格式 |
| ObjectMapper | 全局日期、空值策略、自定义 Converter | 统一 JSON 序列化行为 |

------

**学习要点**：

- @RequestBody / @ResponseBody 的读写由 HttpMessageConverter 完成，默认使用 Jackson 的 MappingJackson2HttpMessageConverter。
- 字段级控制用 Jackson 注解；全局行为通过 ObjectMapper Bean 或 configureMessageConverters 配置。
- 循环引用用 @JsonIgnore 或 @JsonManagedReference / @JsonBackReference 处理。

## 第十章：拦截器、过滤器与 AOP 对比

------

> 本章对比 **Filter、HandlerInterceptor、AOP** 三者的执行顺序、可获取的上下文与典型场景，便于选型与面试回答；与第六章拦截器配合阅读。

------

### 10.1 执行顺序与范围

#### （1）三者在请求链路中的位置

```
请求进入
    │
    ▼
[Filter]          ← Servlet 容器层，最早；可修改 Request/Response
    │
    ▼
[DispatcherServlet]
    │
    ▼
[HandlerInterceptor.preHandle]
    │
    ▼
[Controller 方法]  ← AOP 可切此处（方法前后）
    │
    ▼
[HandlerInterceptor.postHandle]
    │
    ▼
[View 渲染 或 MessageConverter 写 body]
    │
    ▼
[HandlerInterceptor.afterCompletion]
    │
    ▼
响应返回
```

#### （2）对比表

| 维度 | Filter | HandlerInterceptor | AOP |
|------|--------|--------------------|-----|
| **所属** | Servlet 规范，容器层 | Spring MVC，DispatcherServlet 内 | Spring AOP，任意 Bean 方法 |
| **可获取** | ServletRequest、ServletResponse、FilterChain | Request、Response、**Handler**、**ModelAndView** | 方法参数、返回值、异常；**拿不到 Request** 需 RequestContextHolder 或参数传入 |
| **执行时机** | 请求最早、响应最晚 | Handler 前/后/完成后 | 被代理方法前/后/异常 |
| **典型场景** | 编码、CORS、全局日志、XSS 过滤 | **登录校验、权限、请求前后逻辑** | 业务无关的日志、耗时、事务（Service 层更多） |

> 💡 **重点**：需要拿到 **Handler**（即当前要执行的 Controller 方法）或 **ModelAndView** 时，只能用 **Interceptor**；Filter 拿不到 Handler，AOP 拿不到 Request 需额外取。

------

### 10.2 协同使用建议

| 需求 | 推荐方式 | 说明 |
|------|----------|------|
| **跨域 CORS** | Filter（**CorsFilter**）或 **WebMvcConfigurer.addCorsMappings** | 在请求进入时统一加 CORS 头。 |
| **登录/权限** | **Interceptor** | 可拿 Handler、可排除路径、可写 401/403。 |
| **请求编码** | **Filter**（CharacterEncodingFilter） | 最早统一设置编码。 |
| **请求日志/耗时** | Interceptor 或 AOP | Interceptor 可拿 Request；AOP 可切 Controller 方法，需通过 RequestContextHolder 取 Request。 |

------

## ✅ 本章小结

| 知识点 | 面试关键词 | 实际应用 |
|--------|------------|----------|
| Filter vs Interceptor | 容器层 vs MVC 层；Filter 无 Handler，Interceptor 有 | 选型：要 Handler 用 Interceptor |
| Interceptor vs AOP | MVC 内 vs 任意 Bean；AOP 拿不到 Request | 登录/权限用 Interceptor，纯方法切面用 AOP |
| 协同 | CORS/编码用 Filter，登录用 Interceptor | 实际项目中的分工 |

------

**学习要点**：

- Filter 最早、在容器层；Interceptor 在 DispatcherServlet 内、可拿 Handler 与 Model；AOP 可切方法但拿不到 Request，需 RequestContextHolder。
- 登录、权限等需要「当前请求对应哪个接口」时用 Interceptor；跨域、编码用 Filter 或 WebMvcConfigurer。

## 🎯 面试常见追问

| 面试官提问 | 回答思路 |
|------------|----------|
| Filter 和 Interceptor 区别？ | Filter 是 Servlet 规范，最早执行，可改 Request/Response，拿不到 Handler；Interceptor 是 Spring MVC 的，在 DispatcherServlet 内、Handler 前后执行，可拿 Handler、Model，适合登录权限。 |
| 登录校验为什么用拦截器不用 AOP？ | 拦截器能拿到 Handler 和 Request，能根据路径排除登录接口、能直接写 401 响应；AOP 切 Controller 方法拿不到 HttpServletRequest，需 RequestContextHolder，且排除路径不如拦截器直观。 |

## 第十二章：Spring MVC 源码与执行流程解析

------

> 本章从源码角度串联 **DispatcherServlet 初始化、doDispatch 主流程、参数与返回值解析**，对应第一、二章的执行流程，便于面试口述与原理回答。

------

### 12.1 DispatcherServlet 初始化

#### （1）容器与上下文

- **DispatcherServlet** 启动时会创建或使用一个 **Web 应用上下文**（子容器），用于存放 Web 相关 Bean（Controller、HandlerMapping 等）。
- 传统 Web 中可配合 **ContextLoaderListener** 加载**根容器**（业务 Service、DAO），子容器可访问父容器 Bean，形成父子容器结构。
- Spring Boot 下通常只有一个 Web 容器，自动配置完成 DispatcherServlet 与各策略 Bean 的注册。

#### （2）onRefresh() 与策略组件

在 **onRefresh()** 中会初始化各**策略组件**（若容器中无则使用默认实现）：

| 策略 | 说明 |
|------|------|
| HandlerMapping | 从容器中获取所有 HandlerMapping Bean，用于后续根据请求查 Handler。 |
| HandlerAdapter | 同上，用于执行 Handler。 |
| ViewResolver | 同上，用于解析视图名。 |
| HandlerExceptionResolver | 处理执行过程中抛出的异常。 |

------

### 12.2 doDispatch() 核心流程

**doDispatch()** 是 DispatcherServlet 处理单次请求的核心方法，对应第一章「请求处理流程」：

1. **查 Handler**：根据当前请求调用 **HandlerMapping.getHandler(request)**，得到 **HandlerExecutionChain**（包含 Handler 与匹配的拦截器列表）。
2. **获取 HandlerAdapter**：根据 Handler 类型从 HandlerAdapter 列表中找到支持的适配器（如 RequestMappingHandlerAdapter）。
3. **拦截器 preHandle**：按顺序执行 Chain 中拦截器的 preHandle；任一返回 false 则直接 return，不执行 Handler。
4. **执行 Handler**：**HandlerAdapter.handle()** 内部完成：参数解析（ArgumentResolver）→ 调用 Controller 方法 → 返回值处理（ReturnValueHandler），得到 **ModelAndView** 或已写好的响应。
5. **拦截器 postHandle**：逆序执行 postHandle（若返回了 ModelAndView 可在此修改）。
6. **处理结果**：若返回 ModelAndView，则用 **ViewResolver** 解析并渲染视图、写响应；若已在 HandlerAdapter 中通过 MessageConverter 写了 body，则跳过 ViewResolver。
7. **拦截器 afterCompletion**：逆序执行 afterCompletion。
8. **异常**：上述任一步骤若抛异常，会交给 **HandlerExceptionResolver** 处理（如转成错误视图或 JSON）。

------

### 12.3 参数与返回值解析

| 组件 | 职责 |
|------|------|
| **HandlerMethodArgumentResolver** | 根据方法参数类型与注解（@RequestBody、@RequestParam、@PathVariable 等）解析出参数值；HandlerAdapter 内置一列 Resolver，按顺序匹配。 |
| **HandlerMethodReturnValueHandler** | 根据返回值类型与注解（@ResponseBody、视图名等）处理返回值：写 body（通过 HttpMessageConverter）或封装 ModelAndView。 |

> 💡 **面试点**：AOP 对 Controller 的代理在 Spring 容器启动时已生成，**Handler 执行的是代理对象的方法**，因此 AOP 切面可以拦截到 Controller 方法调用；执行顺序是「拦截器 preHandle → Controller 方法（被 AOP 包装）→ 拦截器 postHandle」。

------

## ✅ 本章小结

| 知识点 | 面试关键词 | 实际应用 |
|--------|------------|----------|
| 初始化 | onRefresh、HandlerMapping/Adapter/ViewResolver 策略 | 理解 DispatcherServlet 启动时做了什么 |
| doDispatch | getHandler → getAdapter → preHandle → handle → postHandle → 视图/body → afterCompletion | 口述或画图一次请求的完整链路 |
| 参数与返回值 | ArgumentResolver、ReturnValueHandler、AOP 与 Handler 关系 | 理解参数绑定与 @ResponseBody 在源码中的位置 |

------

**学习要点**：

- doDispatch 是请求处理的主线：查 Handler → 找 Adapter → 拦截器 pre → 执行 Handler → 拦截器 post → 解析视图或写 body → 拦截器 after。
- 参数解析与返回值处理在 HandlerAdapter.handle() 内部通过 ArgumentResolver 与 ReturnValueHandler 完成。
- 异常由 HandlerExceptionResolver 统一处理，对应第七章的 @ExceptionHandler 等。

------

## 🎯 面试常见追问

| 面试官提问 | 回答思路 |
|------------|----------|
| doDispatch 大致做了哪些事？ | 先 HandlerMapping 查 Handler 得到 HandlerExecutionChain，再 HandlerAdapter 执行（参数解析、调方法、处理返回值），然后根据返回值走 ViewResolver 渲染或 MessageConverter 写 body；前后有拦截器 preHandle、postHandle、afterCompletion；异常交 HandlerExceptionResolver。 |
| 参数解析是在哪里做的？ | 在 HandlerAdapter（如 RequestMappingHandlerAdapter）执行 Handler 时，通过 HandlerMethodArgumentResolver 链解析方法参数，@RequestBody、@RequestParam 等都有对应 Resolver。 |

## 第十三章：Spring MVC 项目实战案例

------

> 本章把前面各章串成一套**前后端分离项目**的实战套路：分层、统一响应、全局异常、登录校验、文件上传与参数校验；可按此模板落地或面试时概括项目结构。

------

### 13.1 分层与职责

| 层级 | 职责 | 对应章节 |
|------|------|----------|
| **Controller** | 参数接收与校验（@Valid）、调用 Service、封装返回（Result）；不写业务逻辑与事务。 | 第三、五章 |
| **Service** | 业务逻辑、事务边界（@Transactional）。 | — |
| **Mapper / Repository** | 数据访问，只做 CRUD。 | — |

> 💬 **一句话**：Controller 薄一层，只做「入参 → 调 Service → 出参」；业务与事务在 Service。

------

### 13.2 统一响应与全局异常

- **统一响应**：定义 **Result&lt;T&gt;**（code、msg、data），正常接口返回 `Result.success(data)`；见第五章。
- **全局异常**：**@RestControllerAdvice + @ExceptionHandler**，将 MethodArgumentNotValidException、BusinessException、Exception 等统一转为 Result，保证前端拿到的结构一致；见第七章。

------

### 13.3 登录校验

- **拦截器**（见第六章）：在 preHandle 中解析请求头或 Cookie 中的 **Token**（如 JWT），校验通过则将用户信息写入 **ThreadLocal**，Controller/Service 中可直接获取；未登录返回 401 或统一错误码。
- 通过 **excludePathPatterns** 排除登录、注册、静态资源等路径。

------

### 13.4 文件上传与参数校验

| 功能 | 实现要点 |
|------|----------|
| **文件上传** | 方法参数 **MultipartFile**，校验类型与大小，重命名（如 UUID）存储，返回访问 URL；见第八章。 |
| **参数校验** | DTO 上 **@Valid** 与校验注解，Controller 参数加 **@Valid**；校验失败由全局异常捕获并返回 400 与字段错误；见第四、七章。 |

------

## ✅ 本章小结

| 知识点 | 面试关键词 | 实际应用 |
|--------|------------|----------|
| 分层 | Controller 薄、Service 业务与事务、Mapper 数据 | 描述项目结构 |
| 统一响应与异常 | Result + @RestControllerAdvice + @ExceptionHandler | 前后端分离接口规范 |
| 登录 | 拦截器 + Token + ThreadLocal、excludePathPatterns | 接口鉴权 |
| 上传与校验 | MultipartFile、@Valid、全局异常返回 400 | 文件与参数安全 |

------

**学习要点**：

- 实战项目按「Controller → Service → Mapper」分层；统一 Result 与全局异常保证接口一致。
- 登录用拦截器解析 Token 写 ThreadLocal，并排除登录等路径。
- 上传与校验配合全局异常，避免在 Controller 里重复写错误返回。

## 第十四章：Spring MVC 常见问题与调优

------

> 本章汇总常见 HTTP 错误码原因、@RequestBody 与乱码问题，以及性能与监控上的注意点；便于排错与面试「遇到过什么坑」。

------

### 14.1 常见错误与原因

#### （1）HTTP 状态码与对应原因

| 状态码 | 常见原因 | 排查/解决 |
|--------|----------|-----------|
| **404** | 路径或 HTTP Method 不匹配、Controller 未被扫描（包路径、@ComponentScan） | 核对 URL、Method、启动类扫描包。 |
| **405** | Method 不允许（如接口只支持 GET 却用 POST 调） | 核对请求方法与 @GetMapping 等是否一致。 |
| **415** | Content-Type 不支持（如未传 `application/json` 却用了 @RequestBody） | 请求头加 `Content-Type: application/json`。 |
| **400** | 参数校验失败（@Valid）、类型转换失败、必填参数缺失 | 看全局异常返回的 msg 或 BindingResult；核对参数类型与名称。 |

#### （2）@RequestBody 无法解析

- 确保请求头 **Content-Type: application/json**（或其它已注册的 Converter 类型）。
- 请求体 JSON 与 DTO 字段类型、命名一致（可配合 @JsonProperty）；日期格式与 DTO 上的 @JsonFormat 或全局 ObjectMapper 一致。
- 若仍报错，看异常栈是否为 **HttpMessageNotReadableException**，根据 message 排查格式问题。

#### （3）中文乱码

- 请求/响应编码统一为 **UTF-8**：配置 **CharacterEncodingFilter**（置于 Filter 链前）或 Spring Boot 中 **server.servlet.encoding.charset=UTF-8**（默认通常已是 UTF-8）。
- 若仅响应乱码，检查 HttpMessageConverter 与 ObjectMapper 的编码配置。

------

### 14.2 性能与监控

| 方面 | 建议 |
|------|------|
| **静态资源** | 走默认静态路径（如 /static）或 CDN，避免经 DispatcherServlet，减少线程占用。 |
| **接口层** | 合理使用缓存、异步（如 @Async）降低阻塞；注意线程池与超时配置。 |
| **监控与排查** | 在拦截器或 Filter 中记录请求耗时、**TraceId**（便于链路追踪），便于发现慢请求与调用链。 |

------

## ✅ 本章小结

| 知识点 | 面试关键词 | 实际应用 |
|--------|------------|----------|
| 404/405/415/400 | 路径/Method/Content-Type/校验 | 快速定位接口报错原因 |
| @RequestBody | Content-Type、JSON 与 DTO 一致 | 联调与排错 |
| 乱码 | CharacterEncodingFilter、UTF-8 | 统一编码 |
| 性能与监控 | 静态资源、缓存、TraceId、耗时 | 优化与排查慢请求 |

------

**学习要点**：

- 404 查路径与扫描；405 查 Method；415 查 Content-Type；400 查参数与校验。
- @RequestBody 要求 Content-Type 与 body 格式正确；乱码统一 UTF-8。
- 静态资源不经过 DispatcherServlet；用 TraceId 与耗时打点便于监控与排查。

## 第十五章：Spring MVC 面试题与延伸

------

> 本章以「面试官提问 → 回答要点」的形式汇总全手册高频题，便于考前速记；详细原理与示例见对应章节。

------

### 15.1 流程与核心组件

| 问题 | 回答要点 |
|------|----------|
| Spring MVC 执行流程？ | DispatcherServlet 接收请求 → HandlerMapping 查 Handler → HandlerAdapter 执行（参数解析、调方法、处理返回值）→ 若返回视图则 ViewResolver 解析渲染，若 @ResponseBody 则 HttpMessageConverter 写 body → 响应返回。 |
| DispatcherServlet 作用？ | 前端控制器，唯一入口，负责调度 HandlerMapping、HandlerAdapter、ViewResolver、HandlerExceptionResolver；不写业务，只做协调。 |
| HandlerMapping 和 HandlerAdapter 区别？ | HandlerMapping 负责「根据请求找哪个 Handler」；HandlerAdapter 负责「怎么调这个 Handler」（解析参数、执行、处理返回值）。 |

------

### 15.2 注解与数据

| 问题 | 回答要点 |
|------|----------|
| @Controller 与 @RestController？ | @RestController = @Controller + 类级别 @ResponseBody；前者方法默认返回视图名走 ViewResolver，后者默认写响应体走 HttpMessageConverter，接口层用 @RestController。 |
| @RequestBody / @ResponseBody 实现？ | 通过 HttpMessageConverter：读请求体反序列化、写响应体序列化；默认使用 Jackson 的 MappingJackson2HttpMessageConverter。 |
| 如何支持 JSON？ | 引入 Jackson 后 Spring Boot 自动注册 MappingJackson2HttpMessageConverter；可自定义 ObjectMapper Bean 统一日期、空值等。 |
| @RequestParam 和 @RequestBody 区别？ | @RequestParam 绑查询参数/表单字段；@RequestBody 绑请求体（通常 JSON），需 Content-Type 正确，由 HttpMessageConverter 解析。 |

------

### 15.3 统一返回、异常与跨域

| 问题 | 回答要点 |
|------|----------|
| 统一返回与全局异常？ | 定义 Result（code、msg、data）；正常接口返回 Result.success(data)；用 @RestControllerAdvice + @ExceptionHandler 将各类异常转为 Result，保证前端结构一致。 |
| @Valid 校验失败怎么处理？ | 抛 MethodArgumentNotValidException（@RequestBody）或 BindException（表单）；在全局 @ExceptionHandler 中捕获，返回 400 与错误信息。 |
| 跨域怎么配置？ | @CrossOrigin 标注在方法/类上；或 WebMvcConfigurer.addCorsMappings 全局配置；生产常用网关或 Nginx 统一加 CORS 头。 |

------

### 15.4 拦截器与 Filter、AOP

| 问题 | 回答要点 |
|------|----------|
| Filter 和 Interceptor 区别？ | Filter 是 Servlet 规范，最早执行，拿不到 Handler；Interceptor 是 Spring MVC 的，在 DispatcherServlet 内、Handler 前后，可拿 Handler 与 Model，适合登录、权限。 |
| 登录校验为什么用拦截器？ | 能拿 Handler 和 Request、能按路径排除、能直接写 401；AOP 拿不到 Request 需 RequestContextHolder，排除路径不如拦截器直观。 |
