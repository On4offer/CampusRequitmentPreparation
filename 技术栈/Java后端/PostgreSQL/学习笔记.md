## 1️⃣ PostgreSQL 简介

### （1）定义与定位

**PostgreSQL**（简称 PG）是一个功能强大的**开源对象关系型数据库管理系统**（ORDBMS），以其可靠性、功能丰富和扩展性著称。

| 项目 | 说明 |
|------|------|
| **起源** | 1986年加州大学伯克利分校 POSTGRES 项目 |
| **特点** | ACID 事务、MVCC、丰富的数据类型、强大的扩展能力 |
| **协议** | PostgreSQL 许可证（类 BSD，自由使用） |
| **适用版本** | PostgreSQL 14 / 15 / 16（推荐使用最新稳定版） |

> 💬 **一句话**：PostgreSQL 是**功能最丰富的开源关系型数据库**，被称为"世界上最先进的开源数据库"。

### （2）PostgreSQL vs MySQL

| 对比维度 | MySQL | PostgreSQL |
|----------|-------|------------|
| **定位** | 纯关系型数据库 | 对象关系型数据库（ORDBMS） |
| **复杂查询** | 较弱，复杂 JOIN 性能一般 | 强大，支持复杂查询优化 |
| **数据类型** | 基本类型 | 丰富（JSON/JSONB、数组、几何、自定义类型） |
| **全文搜索** | 基础支持 | 内置强大全文搜索 |
| **GIS** | 需要额外扩展 | PostGIS 扩展功能强大 |
| **并发控制** | 表锁/行锁 | MVCC，无锁读 |
| **扩展性** | 有限 | 强大（自定义类型、函数、索引） |
| **JSON支持** | JSON 类型 | JSONB（二进制JSON，索引支持好） |
| **窗口函数** | 8.0+ 支持 | 完善支持 |
| **适用场景** | Web应用、简单事务 | 复杂查询、GIS、数据分析 |

### （3）核心特性

| 特性 | 说明 |
|------|------|
| **MVCC** | 多版本并发控制，读写不阻塞 |
| **丰富的数据类型** | JSON/JSONB、数组、范围类型、几何类型、自定义类型 |
| **全文搜索** | 内置全文搜索，支持中文分词 |
| **表继承** | 支持表继承，实现表分区 |
| **扩展机制** | 可扩展数据类型、函数、索引、操作符 |
| **CTE 递归** | 强大的 WITH 递归查询 |
| **窗口函数** | 完善的窗口函数支持 |
| **物化视图** | 支持物化视图，提升查询性能 |
| **逻辑复制** | 支持逻辑复制，灵活的数据同步 |

### （4）典型应用场景

| 场景 | 说明 |
|------|------|
| **复杂业务系统** | 复杂查询、多表关联、数据分析 |
| **GIS 应用** | PostGIS 扩展，地理信息系统 |
| **JSON 存储** | JSONB 类型，替代 MongoDB 部分场景 |
| **全文搜索** | 内置全文搜索，替代 Elasticsearch 部分场景 |
| **数据仓库** | 列存储扩展、并行查询 |
| **时序数据** | TimescaleDB 扩展 |

**核心进程说明：**

| 进程 | 作用 |
|------|------|
| **postmaster** | 主进程，监听连接，管理子进程 |
| **postgres** | 服务进程，处理客户端请求（每个连接一个） |
| **bg writer** | 后台写进程，定期将脏页刷入磁盘 |
| **WAL writer** | WAL 写进程，将 WAL 缓冲区刷入磁盘 |
| **checkpointer** | 检查点进程，执行检查点操作 |
| **autovacuum** | 自动清理进程，回收死元组空间 |
| **stats collector** | 统计信息收集进程 |

## 3️⃣ MVCC 机制

### （1）什么是 MVCC

**MVCC（Multi-Version Concurrency Control）** 是 PostgreSQL 实现并发控制的核心机制，通过维护数据的多个版本来实现：

- **读不阻塞写**：读取操作不会阻塞写入操作
- **写不阻塞读**：写入操作不会阻塞读取操作
- **快照隔离**：每个事务看到的是某个时间点的数据快照

### （2）事务隔离级别

| 隔离级别 | 脏读 | 不可重复读 | 幻读 |
|----------|------|-----------|------|
| READ UNCOMMITTED | ❌ 不允许 | ✅ 可能 | ✅ 可能 |
| READ COMMITTED | ❌ 不允许 | ✅ 可能 | ✅ 可能 |
| REPEATABLE READ | ❌ 不允许 | ❌ 不允许 | ✅ 可能 |
| SERIALIZABLE | ❌ 不允许 | ❌ 不允许 | ❌ 不允许 |

> **注意**：PostgreSQL 的 READ UNCOMMITTED 实际上等同于 READ COMMITTED。

### （3） xmin / xmax 机制

```sql
-- 每个元组（行）都有隐藏的系统列
SELECT xmin, xmax, * FROM users LIMIT 5;

-- xmin: 插入该行的事务ID
-- xmax: 删除/更新该行的事务ID（0表示未删除）

-- 更新操作实际上是：
-- 1. 将旧行的 xmax 设为当前事务ID
-- 2. 插入新行，xmin 为当前事务ID
```

### （4）Vacuum 机制

```sql
-- 手动执行 Vacuum
VACUUM users;                    -- 回收空间，更新统计信息
VACUUM FULL users;               -- 完全回收空间，锁表（慎用）
VACUUM ANALYZE users;            -- 回收空间 + 更新统计信息

-- 查看 Vacuum 状态
SELECT relname, n_live_tup, n_dead_tup, last_vacuum, last_autovacuum
FROM pg_stat_user_tables;

-- 配置自动 Vacuum
ALTER SYSTEM SET autovacuum = on;
ALTER SYSTEM SET autovacuum_vacuum_threshold = 50;
ALTER SYSTEM SET autovacuum_vacuum_scale_factor = 0.1;
```

## 1️⃣ 安装与部署

### （1）Docker 安装（推荐）

```bash
# 拉取镜像
docker pull postgres:16

# 运行容器
docker run -d \
  --name postgres \
  -e POSTGRES_USER=admin \
  -e POSTGRES_PASSWORD=123456 \
  -e POSTGRES_DB=mydb \
  -p 5432:5432 \
  -v /data/postgres:/var/lib/postgresql/data \
  postgres:16

# 进入容器
docker exec -it postgres psql -U admin -d mydb
```

### （2）Linux 安装

```bash
# 添加 PostgreSQL 仓库
sudo sh -c 'echo "deb http://apt.postgresql.org/pub/repos/apt $(lsb_release -cs)-pgdg main" > /etc/apt/sources.list.d/pgdg.list'

# 导入公钥
wget --quiet -O - https://www.postgresql.org/media/keys/ACCC4CF8.asc | sudo apt-key add -

# 安装
sudo apt-get update
sudo apt-get install -y postgresql-16

# 启动服务
sudo systemctl start postgresql
sudo systemctl enable postgresql

# 切换用户并连接
sudo -u postgres psql
```

### （3）连接 PostgreSQL

```bash
# 命令行连接
psql -h localhost -p 5432 -U admin -d mydb

# 连接字符串
psql "postgresql://admin:123456@localhost:5432/mydb"

# JDBC 连接字符串
jdbc:postgresql://localhost:5432/mydb?user=admin&password=123456
```

### （4）psql 常用命令


## 3️⃣ 表操作（DDL）

### （1）创建表

```sql
-- 基本创建
CREATE TABLE users (
  id SERIAL PRIMARY KEY,              -- 自增主键
  name VARCHAR(100) NOT NULL,
  email VARCHAR(255) UNIQUE,
  age INTEGER CHECK (age >= 0 AND age <= 150),
  status VARCHAR(20) DEFAULT 'active',
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- 完整创建（含注释、约束命名）
CREATE TABLE orders (
  id BIGSERIAL PRIMARY KEY,
  user_id INTEGER NOT NULL,
  order_no VARCHAR(50) NOT NULL,
  amount DECIMAL(10, 2) NOT NULL,
  status VARCHAR(20) NOT NULL DEFAULT 'pending',
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  
  CONSTRAINT uk_order_no UNIQUE (order_no),
  CONSTRAINT fk_user_id FOREIGN KEY (user_id) REFERENCES users(id) ON DELETE CASCADE,
  CONSTRAINT chk_amount CHECK (amount > 0)
);

COMMENT ON TABLE orders IS '订单表';
COMMENT ON COLUMN orders.order_no IS '订单编号';
COMMENT ON COLUMN orders.amount IS '订单金额';

-- 创建表（从查询结果）
CREATE TABLE users_backup AS 
SELECT * FROM users WHERE status = 'active';

-- 创建临时表
CREATE TEMP TABLE temp_users AS 
SELECT * FROM users LIMIT 100;
```

### （2）修改表

```sql
-- 添加列
ALTER TABLE users ADD COLUMN phone VARCHAR(20);

-- 删除列
ALTER TABLE users DROP COLUMN phone;

-- 修改列类型
ALTER TABLE users ALTER COLUMN age TYPE BIGINT;

-- 添加约束
ALTER TABLE users ADD CONSTRAINT uk_email UNIQUE (email);
ALTER TABLE users ADD CONSTRAINT chk_age CHECK (age >= 0);

-- 删除约束
ALTER TABLE users DROP CONSTRAINT uk_email;

-- 重命名列
ALTER TABLE users RENAME COLUMN name TO username;

-- 重命名表
ALTER TABLE users RENAME TO customers;

-- 添加/删除主键
ALTER TABLE users ADD PRIMARY KEY (id);
ALTER TABLE users DROP CONSTRAINT users_pkey;
```

### （3）删除表

```sql
-- 删除表
DROP TABLE users;

-- 条件删除
DROP TABLE IF EXISTS users;

-- 级联删除（删除依赖对象）
DROP TABLE users CASCADE;
```

## 4️⃣ 数据操作（DML）

### （1）插入数据

```sql
-- 插入单条
INSERT INTO users (name, email, age) 
VALUES ('张三', 'zhangsan@example.com', 25);

-- 插入多条
INSERT INTO users (name, email, age) VALUES
  ('李四', 'lisi@example.com', 30),
  ('王五', 'wangwu@example.com', 28),
  ('赵六', 'zhaoliu@example.com', 35);

-- 从查询插入
INSERT INTO users_backup (name, email, age)
SELECT name, email, age FROM users WHERE status = 'active';

-- 插入或更新（UPSERT）
INSERT INTO users (id, name, email) 
VALUES (1, '张三', 'new@example.com')
ON CONFLICT (id) DO UPDATE SET 
  name = EXCLUDED.name,
  email = EXCLUDED.email;

-- 插入或忽略
INSERT INTO users (id, name, email) 
VALUES (1, '张三', 'test@example.com')
ON CONFLICT (id) DO NOTHING;
```

### （2）更新数据

```sql
-- 基本更新
UPDATE users SET age = 26 WHERE name = '张三';

-- 多列更新
UPDATE users SET 
  age = 26,
  status = 'active',
  updated_at = CURRENT_TIMESTAMP
WHERE name = '张三';

-- 条件更新
UPDATE users SET status = 'inactive'
WHERE last_login < CURRENT_DATE - INTERVAL '90 days';

-- 从其他表更新
UPDATE orders o SET status = u.status
FROM users u
WHERE o.user_id = u.id AND u.status = 'inactive';

-- 返回更新的行
UPDATE users SET age = age + 1 
WHERE status = 'active'
RETURNING id, name, age;
```

### （3）删除数据

```sql
-- 条件删除
DELETE FROM users WHERE status = 'inactive';

-- 删除所有
DELETE FROM users;

-- 截断表（更快，重置序列，不能回滚）
TRUNCATE TABLE users;
TRUNCATE TABLE users RESTART IDENTITY CASCADE;

-- 返回删除的行
DELETE FROM users WHERE status = 'inactive'
RETURNING id, name;
```

## 5️⃣ 查询基础

### （1）基本查询

```sql
-- 查询所有列
SELECT * FROM users;

-- 查询指定列
SELECT name, email FROM users;

-- 别名
SELECT name AS username, email FROM users;

-- 去重
SELECT DISTINCT status FROM users;

-- 条件查询
SELECT * FROM users WHERE age >= 25 AND status = 'active';
SELECT * FROM users WHERE age BETWEEN 20 AND 30;
SELECT * FROM users WHERE status IN ('active', 'pending');
SELECT * FROM users WHERE email LIKE '%@example.com';
SELECT * FROM users WHERE email ILIKE '%@EXAMPLE.COM';  -- 不区分大小写
SELECT * FROM users WHERE email IS NULL;
```

### （2）排序与分页

```sql
-- 排序
SELECT * FROM users ORDER BY age DESC;
SELECT * FROM users ORDER BY status, age DESC;  -- 多列排序

-- 分页（LIMIT + OFFSET）
SELECT * FROM users ORDER BY id LIMIT 10 OFFSET 0;   -- 第1页
SELECT * FROM users ORDER BY id LIMIT 10 OFFSET 10;  -- 第2页

-- 分页（LIMIT + OFFSET 简写）
SELECT * FROM users ORDER BY id LIMIT 10;      -- 前10条
SELECT * FROM users ORDER BY id LIMIT 10, 10;  -- 跳过10条，取10条

-- 键集分页（性能更好）
SELECT * FROM users WHERE id > 100 ORDER BY id LIMIT 10;
```

```sql
  profile JSON,                       -- JSON 文本
  settings JSONB                      -- JSONB 二进制（推荐）
);

-- 插入 JSON
INSERT INTO users (name, profile, settings) VALUES
  ('张三', '{"age": 25, "city": "北京"}', '{"theme": "dark", "lang": "zh"}'),
  ('李四', '{"age": 30, "skills": ["java", "python"]}', '{"theme": "light"}');

-- JSON 查询操作符
SELECT profile->>'age' FROM users;              -- 提取为文本
SELECT profile->'age' FROM users;               -- 提取为 JSON
SELECT settings->'theme' FROM users;            -- 提取 JSONB 字段
SELECT settings->>'theme' FROM users;           -- 提取为文本

-- 嵌套 JSON 访问
INSERT INTO users (name, settings) VALUES
  ('王五', '{"preferences": {"notifications": {"email": true, "sms": false}}}');

SELECT settings->'preferences'->'notifications'->>'email' FROM users WHERE name = '王五';

-- JSON 路径查询（PostgreSQL 12+）
SELECT * FROM users WHERE settings @> '{"theme": "dark"}';  -- 包含
SELECT * FROM users WHERE settings ? 'theme';               -- 存在键
SELECT * FROM users WHERE settings ?| ARRAY['theme', 'lang']; -- 存在任一键
SELECT * FROM users WHERE settings ?& ARRAY['theme', 'lang']; -- 存在所有键

-- JSONB 索引
CREATE INDEX idx_settings ON users USING GIN (settings);
CREATE INDEX idx_settings_path ON users USING GIN (settings jsonb_path_ops);

-- JSON 函数
SELECT jsonb_pretty(settings) FROM users;       -- 格式化输出
SELECT jsonb_object_keys(settings) FROM users;  -- 获取所有键
SELECT jsonb_array_elements(settings->'skills') FROM users;  -- 展开数组

-- JSON 更新
UPDATE users 
SET settings = settings || '{"theme": "light"}'::jsonb  -- 合并更新
WHERE name = '张三';

UPDATE users 
SET settings = settings - 'theme'                -- 删除键
WHERE name = '张三';

UPDATE users
SET settings = jsonb_set(settings, '{preferences,theme}', '"dark"')  -- 设置嵌套值
WHERE name = '王五';
```

### （4）范围类型

```sql
-- int4range, int8range, numrange, tsrange, tstzrange, daterange

CREATE TABLE reservations (

  room_id INTEGER,
  during TSRANGE,                     -- 时间范围
  price_range NUMRANGE                -- 价格范围

-- 插入范围

SELECT * FROM reservations WHERE during && TSRANGE('[2024-01-02, 2024-01-04]');  -- 重叠
SELECT * FROM reservations WHERE during @> TIMESTAMP '2024-01-02 15:00';         -- 包含
SELECT * FROM reservations WHERE during <@ TSRANGE('[2024-01-01, 2024-01-10]');  -- 被包含

SELECT isempty(during) FROM reservations;               -- 是否为空
```

### （2）约束设计

```sql
CREATE TABLE products (
  id SERIAL PRIMARY KEY,
  name VARCHAR(200) NOT NULL,
  sku VARCHAR(50) NOT NULL,
  price DECIMAL(10, 2) NOT NULL,
  stock INTEGER DEFAULT 0,
  category_id INTEGER,
  
  -- 唯一约束
  CONSTRAINT uk_sku UNIQUE (sku),
  
  -- 检查约束
  CONSTRAINT chk_price CHECK (price >= 0),
  CONSTRAINT chk_stock CHECK (stock >= 0),
  
  -- 外键约束
  CONSTRAINT fk_category FOREIGN KEY (category_id) 
    REFERENCES categories(id) 
    ON DELETE SET NULL
    ON UPDATE CASCADE,

  EXCLUDE USING GIST (
    room_id WITH =,
    during WITH &&
  )
);

-- 延迟约束（事务提交时检查）
ALTER TABLE orders ADD CONSTRAINT fk_user 
  FOREIGN KEY (user_id) REFERENCES users(id)
  DEFERRABLE INITIALLY DEFERRED;
```

### （3）表分区

```sql
-- 范围分区（按时间）
CREATE TABLE logs (
  id BIGSERIAL,
  user_id INTEGER,
  action VARCHAR(50),
  created_at TIMESTAMP
) PARTITION BY RANGE (created_at);

-- 创建分区
CREATE TABLE logs_2024_01 PARTITION OF logs
  FOR VALUES FROM ('2024-01-01') TO ('2024-02-01');

CREATE TABLE logs_2024_02 PARTITION OF logs
  FOR VALUES FROM ('2024-02-01') TO ('2024-03-01');

-- 默认分区
CREATE TABLE logs_default PARTITION OF logs DEFAULT;

-- 列表分区
CREATE TABLE users (
  id SERIAL,
  name VARCHAR(100),
  region VARCHAR(20)
) PARTITION BY LIST (region);

CREATE TABLE users_east PARTITION OF users
  FOR VALUES IN ('上海', '北京', '广州');

CREATE TABLE users_west PARTITION OF users
  FOR VALUES IN ('成都', '重庆', '西安');

-- 哈希分区
CREATE TABLE orders (
  id BIGSERIAL,
  user_id BIGINT,
  amount DECIMAL(10, 2)
) PARTITION BY HASH (user_id);

CREATE TABLE orders_0 PARTITION OF orders
  FOR VALUES WITH (MODULUS 4, REMAINDER 0);
CREATE TABLE orders_1 PARTITION OF orders
  FOR VALUES WITH (MODULUS 4, REMAINDER 1);
CREATE TABLE orders_2 PARTITION OF orders
  FOR VALUES WITH (MODULUS 4, REMAINDER 2);
CREATE TABLE orders_3 PARTITION OF orders
  FOR VALUES WITH (MODULUS 4, REMAINDER 3);

-- 分区管理
-- 分离分区
ALTER TABLE logs DETACH PARTITION logs_2024_01;
-- 附加分区
ALTER TABLE logs ATTACH PARTITION logs_2024_01
  FOR VALUES FROM ('2024-01-01') TO ('2024-02-01');
```

### （4）表继承

```sql
-- 父表

  id SERIAL PRIMARY KEY,
  brand VARCHAR(50),
  model VARCHAR(50)

-- 子表继承
CREATE TABLE cars (
  num_doors INTEGER
) INHERITS (vehicles);

SELECT * FROM ONLY vehicles;         -- 只查询父表

-- 检查约束 + 继承实现分区
CREATE TABLE measurement (
  city_id INTEGER,

  peaktemp INTEGER

  CHECK (logdate >= '2024-01-01' AND logdate < '2024-02-01')
```

## 1️⃣ 索引类型

### （1）B-Tree 索引（默认）

```sql
-- B-Tree 适合等值查询和范围查询
CREATE INDEX idx_users_name ON users(name);
CREATE INDEX idx_users_age ON users(age);
CREATE INDEX idx_users_name_age ON users(name, age);  -- 复合索引

-- 唯一索引
CREATE UNIQUE INDEX idx_users_email ON users(email);

-- 条件索引
CREATE INDEX idx_users_active ON users(name) WHERE status = 'active';

-- 查看索引
\di
SELECT indexname, indexdef FROM pg_indexes WHERE tablename = 'users';

-- 删除索引
DROP INDEX idx_users_name;
```

### （2）Hash 索引

```sql
-- Hash 索引只支持等值查询，不常用
CREATE INDEX idx_users_email_hash ON users USING HASH (email);

-- 注意：Hash 索引不记录 WAL，崩溃后需重建
```

### （3）GIN 索引

```sql
-- GIN（Generalized Inverted Index）适合包含多个元素的值
-- 适用于数组、JSONB、全文搜索

-- 数组索引
CREATE INDEX idx_posts_tags ON posts USING GIN (tags);

-- JSONB 索引
CREATE INDEX idx_users_settings ON users USING GIN (settings);
CREATE INDEX idx_users_settings_path ON users USING GIN (settings jsonb_path_ops);  -- 更高效

-- 全文搜索索引
CREATE INDEX idx_articles_content ON articles USING GIN (to_tsvector('english', content));
```

### （4）GiST 索引

```sql
-- GiST（Generalized Search Tree）适合范围查询、几何类型

-- 范围类型索引
CREATE INDEX idx_reservations_during ON reservations USING GIST (during);

-- 几何类型索引（需 PostGIS）
CREATE INDEX idx_locations_point ON locations USING GIST (point);

-- 排他约束使用 GiST
ALTER TABLE reservations ADD CONSTRAINT ex_room_during
  EXCLUDE USING GIST (room_id WITH =, during WITH &&);
```

### （5）BRIN 索引

```sql
-- BRIN（Block Range Index）适合大表、有序数据
-- 存储每个块的摘要信息，占用空间极小

CREATE INDEX idx_logs_created_at ON logs USING BRIN (created_at);
CREATE INDEX idx_logs_user_id_created ON logs USING BRIN (user_id, created_at);

-- 适合日志表、时序数据
-- 空间占用约为 B-Tree 的 1/100
```

### （6）表达式索引

```sql
-- 对表达式结果创建索引
CREATE INDEX idx_users_lower_email ON users (LOWER(email));
CREATE INDEX idx_users_year ON users (EXTRACT(YEAR FROM created_at));

-- 查询时需使用相同表达式才能命中
SELECT * FROM users WHERE LOWER(email) = 'test@example.com';
```

### （7）部分索引

```sql
-- 只对部分数据创建索引
CREATE INDEX idx_orders_pending ON orders (created_at) 
WHERE status = 'pending';

CREATE INDEX idx_users_active ON users (last_login)
WHERE status = 'active' AND deleted_at IS NULL;
```

## 2️⃣ 执行计划分析

### （1）EXPLAIN 基础

```sql
-- 查看执行计划
EXPLAIN SELECT * FROM users WHERE name = '张三';

-- 查看实际执行统计
EXPLAIN ANALYZE SELECT * FROM users WHERE name = '张三';

-- 更详细的分析
EXPLAIN (ANALYZE, BUFFERS, FORMAT TEXT) 
SELECT * FROM users WHERE name = '张三';

-- JSON 格式输出
EXPLAIN (ANALYZE, BUFFERS, FORMAT JSON) 
SELECT * FROM users WHERE name = '张三';
```

### （2）执行计划解读

```sql
EXPLAIN ANALYZE SELECT * FROM users WHERE age > 25 ORDER BY name;

-- 输出示例：
-- Sort  (cost=xxx..xxx rows=xxx width=xxx) (actual time=xxx..xxx rows=xxx loops=1)
--   Sort Key: name
--   Sort Method: quicksort  Memory: 25kB
--   ->  Seq Scan on users  (cost=xxx..xxx rows=xxx width=xxx) (actual time=xxx..xxx rows=xxx loops=1)
--         Filter: (age > 25)
--         Rows Removed by Filter: 100
-- Planning Time: 0.123 ms
-- Execution Time: 0.456 ms
```

**关键指标说明：**

| 指标 | 说明 |
|------|------|
| **cost** | 代价估算（启动代价..总代价） |
| **rows** | 预估返回行数 |
| **width** | 预估每行字节数 |
| **actual time** | 实际执行时间（毫秒） |
| **loops** | 执行次数 |
| **Buffers** | 缓冲区命中情况（shared hit/read） |

**常见扫描类型：**

| 扫描类型 | 说明 | 性能 |
|----------|------|------|
| `Seq Scan` | 全表扫描 | 慢 |
| `Index Scan` | 索引扫描 + 回表 | 中 |
| `Index Only Scan` | 仅索引扫描 | 快 |
| `Bitmap Index Scan` | 位图索引扫描 | 中 |
| `Bitmap Heap Scan` | 位图堆扫描 | 中 |

**常见连接类型：**

| 连接类型 | 说明 | 适用场景 |
|----------|------|----------|
| `Nested Loop` | 嵌套循环连接 | 小表驱动大表 |
| `Hash Join` | 哈希连接 | 大表等值连接 |
| `Merge Join` | 归并连接 | 有序数据连接 |

### （3）优化案例

```sql
-- 案例1：避免全表扫描
EXPLAIN ANALYZE SELECT * FROM users WHERE name = '张三';
-- Seq Scan -> 创建索引 -> Index Scan

-- 案例2：索引列使用函数导致索引失效
EXPLAIN ANALYZE SELECT * FROM users WHERE UPPER(email) = 'TEST@EXAMPLE.COM';
-- Seq Scan -> 创建表达式索引 -> Index Scan

-- 案例3：复合索引顺序
-- 索引：(name, age)
SELECT * FROM users WHERE name = '张三';              -- 命中
SELECT * FROM users WHERE name = '张三' AND age > 25; -- 命中
SELECT * FROM users WHERE age > 25;                   -- 不命中
SELECT * FROM users WHERE age > 25 AND name = '张三'; -- 命中

-- 案例4：LIKE 查询优化
SELECT * FROM users WHERE name LIKE '张%';   -- 命中索引
SELECT * FROM users WHERE name LIKE '%张';   -- 不命中索引
-- 解决：使用 pg_trgm 扩展
CREATE EXTENSION pg_trgm;
CREATE INDEX idx_users_name_trgm ON users USING GIN (name gin_trgm_ops);
```

## 3️⃣ 查询优化技巧

### （1）索引优化策略

```sql
-- 1. 选择性高的列优先
-- 选择性 = DISTINCT值数 / 总行数，越接近1越好
SELECT 
  COUNT(DISTINCT status) AS distinct_values,
  COUNT(*) AS total_rows,
  COUNT(DISTINCT status)::FLOAT / COUNT(*) AS selectivity
FROM orders;

-- 2. 复合索引列顺序
-- 等值查询列在前，范围查询列在后
CREATE INDEX idx_orders_user_status_date ON orders(user_id, status, created_at);

-- 3. 覆盖索引
CREATE INDEX idx_orders_user_amount ON orders(user_id, amount);
SELECT user_id, amount FROM orders WHERE user_id = 1;  -- Index Only Scan

-- 4. 避免索引失效
-- 不要在索引列上使用函数
-- 不要使用 NOT LIKE、<>、NOT IN
-- 避免 OR 连接不同列

-- 5. 使用部分索引减少索引大小
CREATE INDEX idx_orders_recent ON orders(created_at) 
WHERE created_at > CURRENT_DATE - INTERVAL '30 days';
```

### （2）JOIN 优化

```sql
-- 1. 小表驱动大表
-- PostgreSQL 会自动选择驱动表，但可以提示
SELECT /*+ NestLoop(u o) */ u.name, o.order_no
FROM users u
JOIN orders o ON u.id = o.user_id;

-- 2. 确保 JOIN 列有索引
CREATE INDEX idx_orders_user_id ON orders(user_id);

-- 3. 减少 JOIN 表数量
-- 超过 5 个表 JOIN 考虑拆分查询

-- 4. 使用 EXISTS 替代 IN
-- 慢
SELECT * FROM users WHERE id IN (SELECT user_id FROM orders WHERE amount > 1000);
-- 快
SELECT * FROM users u WHERE EXISTS (SELECT 1 FROM orders o WHERE o.user_id = u.id AND o.amount > 1000);
```

### （3）分页优化

```sql
-- 传统分页（OFFSET 性能差）
SELECT * FROM orders ORDER BY id LIMIT 10 OFFSET 10000;

-- 键集分页（性能好）
SELECT * FROM orders WHERE id > 10000 ORDER BY id LIMIT 10;

-- 延迟关联优化
SELECT o.* FROM orders o
JOIN (SELECT id FROM orders ORDER BY created_at DESC LIMIT 10 OFFSET 10000) t
ON o.id = t.id;
```

### （4）统计信息更新

```sql
-- 手动更新统计信息
ANALYZE users;
ANALYZE users (name, age);  -- 指定列

-- 自动分析配置
ALTER SYSTEM SET autovacuum_analyze_threshold = 50;
ALTER SYSTEM SET autovacuum_analyze_scale_factor = 0.05;

-- 查看统计信息
SELECT attname, n_distinct, most_common_vals, most_common_freqs
FROM pg_stats WHERE tablename = 'users';

-- 增加统计精度
ALTER TABLE users ALTER COLUMN name SET STATISTICS 500;
ANALYZE users;
```

### （5）查询重写

```sql
-- 使用 CTE 提高可读性
WITH active_users AS (
  SELECT id, name FROM users WHERE status = 'active'
),
user_orders AS (
  SELECT user_id, SUM(amount) AS total_amount
  FROM orders
  GROUP BY user_id
)
SELECT au.name, COALESCE(uo.total_amount, 0) AS total
FROM active_users au
LEFT JOIN user_orders uo ON au.id = uo.user_id;

-- 使用 LATERAL JOIN
SELECT u.name, o.order_no
FROM users u
CROSS JOIN LATERAL (
  SELECT order_no FROM orders 
  WHERE user_id = u.id 
  ORDER BY created_at DESC 
  LIMIT 3
) o;

-- 使用 DISTINCT ON（PostgreSQL 特有）
SELECT DISTINCT ON (user_id) user_id, order_no, created_at
FROM orders
ORDER BY user_id, created_at DESC;  -- 每个用户最新订单
```

| **并行查询** | Parallel Scan | 大表查询 |

**学习要点**：
- B-Tree 是最常用的索引类型，GIN 适合 JSONB 和数组
- 使用 EXPLAIN ANALYZE 分析实际执行情况
- 复合索引遵循最左前缀原则
- OFFSET 分页性能差，优先使用键集分页

## 1️⃣ 窗口函数

### （1）窗口函数基础

```sql
-- 窗口函数语法
function_name([args]) OVER (
  [PARTITION BY partition_expression]
  [ORDER BY sort_expression [ASC|DESC]]
  [frame_clause]
)

-- 基本示例：计算每个用户的订单排名
SELECT 
  user_id,
  order_no,
  amount,
  RANK() OVER (PARTITION BY user_id ORDER BY amount DESC) AS amount_rank
FROM orders;
```

### （2）排名函数

```sql
-- 排名函数对比
SELECT 
  name,
  score,
  ROW_NUMBER() OVER (ORDER BY score DESC) AS row_num,
  RANK() OVER (ORDER BY score DESC) AS rank,
  DENSE_RANK() OVER (ORDER BY score DESC) AS dense_rank,
  PERCENT_RANK() OVER (ORDER BY score DESC) AS pct_rank
FROM students;

-- 结果示例：
-- ROW_NUMBER: 1,2,3,4,5 (连续)
-- RANK: 1,2,2,4,5 (相同分数跳号)
-- DENSE_RANK: 1,2,2,3,4 (相同分数不跳号)
```

### （3）聚合窗口函数

```sql
-- 累计求和与移动平均
SELECT 
  order_date,
  amount,
  SUM(amount) OVER (ORDER BY order_date) AS running_total,
  AVG(amount) OVER (ORDER BY order_date ROWS BETWEEN 6 PRECEDING AND CURRENT ROW) AS moving_avg,
  COUNT(*) OVER (PARTITION BY user_id) AS user_order_count
FROM orders;

-- 窗口帧语法
-- ROWS BETWEEN 6 PRECEDING AND CURRENT ROW  -- 当前行及前6行
-- ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW  -- 从开始到当前行
-- ROWS BETWEEN 1 PRECEDING AND 1 FOLLOWING  -- 前后各1行
```

### （4）偏移函数

```sql
-- LAG、LEAD、FIRST_VALUE、LAST_VALUE
SELECT 
  order_date,
  amount,
  LAG(amount, 1) OVER (ORDER BY order_date) AS prev_amount,
  LEAD(amount, 1) OVER (ORDER BY order_date) AS next_amount,
  FIRST_VALUE(amount) OVER (ORDER BY order_date) AS first_amount,
  LAST_VALUE(amount) OVER (ORDER BY order_date) AS last_amount
FROM orders;

-- 环比计算
SELECT 
  month,
  revenue,
  LAG(revenue, 1) OVER (ORDER BY month) AS prev_month_revenue,
  ROUND(
    (revenue - LAG(revenue, 1) OVER (ORDER BY month)) * 100.0 / 
    LAG(revenue, 1) OVER (ORDER BY month), 
    2
  ) AS growth_rate
FROM monthly_revenue;
```

### （5）窗口函数实战

```sql
-- 取每个用户前3笔订单

  SELECT 
    user_id,
    order_no,
    amount,
    ROW_NUMBER() OVER (PARTITION BY user_id ORDER BY created_at DESC) AS rn
  FROM orders

-- 连续登录用户（使用窗口函数）
WITH login_groups AS (
  SELECT 
    user_id,
    login_date,
    login_date - ROW_NUMBER() OVER (PARTITION BY user_id ORDER BY login_date)::INT AS grp
  FROM user_login_log

FROM login_groups
GROUP BY user_id, grp
HAVING COUNT(*) >= 3;  -- 连续3天
```

### （2）刷新物化视图

```sql

-- 并发刷新（不锁表，PG 9.4+）

-- 自动刷新（使用触发器）
CREATE OR REPLACE FUNCTION refresh_mv_user_order_stats()
RETURNS TRIGGER AS $$
BEGIN
  REFRESH MATERIALIZED VIEW CONCURRENTLY mv_user_order_stats;
  RETURN NULL;
END;
$$ LANGUAGE plpgsql;
```

```sql
CREATE OR REPLACE PROCEDURE transfer_money(
  p_from_user INTEGER,
  p_to_user INTEGER,
  p_amount DECIMAL(10, 2)
) LANGUAGE plpgsql AS $$
BEGIN
  -- 扣款
  UPDATE accounts 
  SET balance = balance - p_amount 
  WHERE user_id = p_from_user;

  UPDATE accounts 
  SET balance = balance + p_amount 
  WHERE user_id = p_to_user;
  
  -- 提交事务

EXCEPTION
  WHEN OTHERS THEN
    -- 回滚事务

    RAISE;
END;
$$;
```

### （3）触发器函数

```sql
-- 自动更新 updated_at 字段

RETURNS TRIGGER AS $$
BEGIN
  NEW.updated_at = CURRENT_TIMESTAMP;
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;

-- 创建触发器
CREATE TRIGGER trg_users_updated_at
BEFORE UPDATE ON users
FOR EACH ROW
EXECUTE FUNCTION update_modified_time();

-- 审计日志触发器
CREATE TABLE audit_log (
  id SERIAL PRIMARY KEY,
  table_name VARCHAR(100),
  operation VARCHAR(10),
  old_data JSONB,
  new_data JSONB,
  changed_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

CREATE OR REPLACE FUNCTION audit_trigger_func()
RETURNS TRIGGER AS $$
BEGIN
  IF TG_OP = 'INSERT' THEN
    INSERT INTO audit_log (table_name, operation, new_data)
    VALUES (TG_TABLE_NAME, TG_OP, to_jsonb(NEW));
    RETURN NEW;
  ELSIF TG_OP = 'UPDATE' THEN
    INSERT INTO audit_log (table_name, operation, old_data, new_data)
    VALUES (TG_TABLE_NAME, TG_OP, to_jsonb(OLD), to_jsonb(NEW));
    RETURN NEW;
  ELSIF TG_OP = 'DELETE' THEN
    INSERT INTO audit_log (table_name, operation, old_data)
    VALUES (TG_TABLE_NAME, TG_OP, to_jsonb(OLD));
    RETURN OLD;
  END IF;
END;
$$ LANGUAGE plpgsql;

-- 为表创建审计触发器
CREATE TRIGGER trg_users_audit
AFTER INSERT OR UPDATE OR DELETE ON users
FOR EACH ROW
EXECUTE FUNCTION audit_trigger_func();
```

```sql
-- 查询组织架构树
WITH RECURSIVE subordinates AS (

  FROM employees
  WHERE id = 1
  
  UNION ALL

  INNER JOIN subordinates s ON e.manager_id = s.id
```

```sql
-- 生成日期序列
WITH RECURSIVE date_series AS (
  SELECT CURRENT_DATE AS date
  UNION ALL
  SELECT date + INTERVAL '1 day'
  FROM date_series
  WHERE date < CURRENT_DATE + INTERVAL '30 days'
)
SELECT date FROM date_series;

WITH RECURSIVE category_path AS (
  SELECT id, name, parent_id, name AS path
  FROM categories
  WHERE parent_id IS NULL
  
  UNION ALL

    parent_id,

  UNION ALL
  
  SELECT 
    d.id, 
    d.name, 
    d.parent_id, 
    dh.level + 1,

  INNER JOIN dept_hierarchy dh ON d.parent_id = dh.id
```

# 🌱 第六章：事务与并发控制

> **本章在整体中解决什么问题**：深入理解 PostgreSQL 的事务机制、隔离级别、锁机制以及并发控制策略。学完本章后**第七章**会讲**备份与恢复**。

## 1️⃣ 事务基础

### （1）事务的 ACID 特性

| 特性 | 说明 | PostgreSQL 实现 |
|------|------|-----------------|
| **原子性 (Atomicity)** | 事务要么全部完成，要么全部不完成 | WAL (Write-Ahead Logging) |
| **一致性 (Consistency)** | 事务执行前后数据库保持一致性状态 | 约束、触发器、外键 |
| **隔离性 (Isolation)** | 并发事务相互隔离 | MVCC (多版本并发控制) |
| **持久性 (Durability)** | 事务提交后数据永久保存 | WAL + 检查点 |

### （2）事务控制语句

```sql
-- 基本事务控制
BEGIN;                          -- 开始事务
-- 执行 SQL 操作
INSERT INTO accounts (user_id, balance) VALUES (1, 1000);
UPDATE accounts SET balance = balance - 100 WHERE user_id = 1;
COMMIT;                         -- 提交事务

-- 回滚事务
BEGIN;
UPDATE accounts SET balance = balance - 100 WHERE user_id = 1;
-- 发现错误
ROLLBACK;                       -- 回滚，撤销所有操作

-- 保存点 (Savepoint)
BEGIN;
INSERT INTO orders (user_id, amount) VALUES (1, 100);
SAVEPOINT sp1;                  -- 设置保存点
INSERT INTO order_items (order_id, product_id) VALUES (1, 101);
-- 发现错误，回滚到保存点
ROLLBACK TO SAVEPOINT sp1;
-- 继续其他操作
INSERT INTO order_items (order_id, product_id) VALUES (1, 102);
COMMIT;
```

### （3）事务隔离级别

```sql
-- 查看当前隔离级别
SHOW transaction_isolation;

-- 设置事务隔离级别
BEGIN ISOLATION LEVEL READ COMMITTED;
BEGIN ISOLATION LEVEL REPEATABLE READ;
BEGIN ISOLATION LEVEL SERIALIZABLE;
```

| 隔离级别 | 脏读 | 不可重复读 | 幻读 | 说明 |
|----------|------|-----------|------|------|
| **READ UNCOMMITTED** | ❌ | ✅ | ✅ | PG 中等同于 READ COMMITTED |
| **READ COMMITTED** | ❌ | ✅ | ✅ | 默认级别，每次查询看到已提交数据 |
| **REPEATABLE READ** | ❌ | ❌ | ❌ | 事务内快照一致，PG 已解决幻读 |
| **SERIALIZABLE** | ❌ | ❌ | ❌ | 最高级别，完全串行化 |

## 2️⃣ 锁机制

### （1）表级锁

```sql
-- 显式表锁
LOCK TABLE users IN ACCESS SHARE MODE;          -- 最弱，只读
LOCK TABLE users IN ROW SHARE MODE;             -- 读 + 行锁
LOCK TABLE users IN ROW EXCLUSIVE MODE;         -- 写操作
LOCK TABLE users IN SHARE UPDATE EXCLUSIVE MODE; -- VACUUM, CREATE INDEX CONCURRENTLY
LOCK TABLE users IN SHARE MODE;                 -- CREATE INDEX
LOCK TABLE users IN SHARE ROW EXCLUSIVE MODE;   -- 较少用
LOCK TABLE users IN EXCLUSIVE MODE;             -- 阻塞读写
LOCK TABLE users IN ACCESS EXCLUSIVE MODE;      -- 最强，ALTER TABLE, DROP TABLE
```

**表锁兼容性矩阵：**

| 锁模式 | ACCESS SHARE | ROW SHARE | ROW EXCLUSIVE | SHARE UPDATE EXCLUSIVE | SHARE | SHARE ROW EXCLUSIVE | EXCLUSIVE | ACCESS EXCLUSIVE |
|--------|--------------|-----------|---------------|------------------------|-------|---------------------|-----------|------------------|
| ACCESS SHARE | ✅ | ✅ | ✅ | ✅ | ✅ | ✅ | ✅ | ❌ |
| ROW SHARE | ✅ | ✅ | ✅ | ✅ | ✅ | ✅ | ❌ | ❌ |
| ROW EXCLUSIVE | ✅ | ✅ | ✅ | ✅ | ❌ | ❌ | ❌ | ❌ |
| ... | ... | ... | ... | ... | ... | ... | ... | ... |

### （2）行级锁

```sql
-- FOR UPDATE: 排他锁，用于更新
BEGIN;
SELECT * FROM accounts WHERE id = 1 FOR UPDATE;
UPDATE accounts SET balance = balance - 100 WHERE id = 1;
COMMIT;

-- FOR SHARE: 共享锁，防止修改但允许读取
SELECT * FROM accounts WHERE id = 1 FOR SHARE;

-- FOR NO KEY UPDATE: 不阻塞其他 FOR KEY SHARE
SELECT * FROM accounts WHERE id = 1 FOR NO KEY UPDATE;

-- FOR KEY SHARE: 最弱行锁，只阻塞 FOR UPDATE
SELECT * FROM accounts WHERE id = 1 FOR KEY SHARE;

-- NOWAIT: 锁不到立即报错
SELECT * FROM accounts WHERE id = 1 FOR UPDATE NOWAIT;

-- SKIP LOCKED: 跳过已锁定的行（实现队列利器）
SELECT * FROM task_queue WHERE status = 'pending' 
ORDER BY created_at 
FOR UPDATE SKIP LOCKED 
LIMIT 1;
```

### （3）查看锁信息

```sql
-- 查看当前锁
SELECT 
    l.locktype,
    l.relation::regclass AS table_name,
    l.pid,
    l.mode,
    l.granted,
    a.query,
    a.state
FROM pg_locks l
LEFT JOIN pg_stat_activity a ON l.pid = a.pid
WHERE l.database IS NOT NULL
ORDER BY l.pid;

-- 查看锁等待（阻塞关系）
SELECT 
    blocked_locks.pid AS blocked_pid,
    blocked_activity.usename AS blocked_user,
    blocking_locks.pid AS blocking_pid,
    blocking_activity.usename AS blocking_user,
    blocked_activity.query AS blocked_query,
    blocking_activity.query AS blocking_query
FROM pg_catalog.pg_locks blocked_locks
JOIN pg_catalog.pg_stat_activity blocked_activity ON blocked_locks.pid = blocked_activity.pid
JOIN pg_catalog.pg_locks blocking_locks 
    ON blocking_locks.locktype = blocked_locks.locktype
    AND blocking_locks.relation = blocked_locks.relation
    AND blocking_locks.pid != blocked_locks.pid
JOIN pg_catalog.pg_stat_activity blocking_activity ON blocking_locks.pid = blocking_activity.pid
WHERE NOT blocked_locks.granted;

-- 终止进程
SELECT pg_terminate_backend(12345);
```

## 3️⃣ 并发问题与解决方案

### （1）脏读、不可重复读、幻读

```sql
-- 脏读 (Dirty Read): 读取到其他事务未提交的数据
-- PostgreSQL 的 MVCC 机制天然避免脏读

-- 不可重复读 (Non-repeatable Read): 同一事务内两次读取结果不同
-- 解决方案：使用 REPEATABLE READ 隔离级别
BEGIN ISOLATION LEVEL REPEATABLE READ;
SELECT balance FROM accounts WHERE id = 1;  -- 第一次读取: 1000
-- 其他事务修改并提交
SELECT balance FROM accounts WHERE id = 1;  -- 第二次读取: 仍是 1000（快照一致）
COMMIT;

-- 幻读 (Phantom Read): 同一事务内两次查询，结果集行数不同
-- PostgreSQL REPEATABLE READ 已解决幻读
BEGIN ISOLATION LEVEL REPEATABLE READ;
SELECT * FROM accounts WHERE balance > 500;  -- 返回 3 行
-- 其他事务插入 balance > 500 的数据并提交
SELECT * FROM accounts WHERE balance > 500;  -- 仍是 3 行（快照一致）
COMMIT;
```

### （2）丢失更新 (Lost Update)

```sql
-- 问题场景：两个事务同时读取并更新同一行
-- 事务A: 读取 balance=1000 -> 计算 1000-100=900 -> 更新为 900
-- 事务B: 读取 balance=1000 -> 计算 1000-200=800 -> 更新为 800
-- 结果：事务A的更新被覆盖，丢失 100

-- 解决方案1：SELECT FOR UPDATE
BEGIN;
SELECT balance FROM accounts WHERE id = 1 FOR UPDATE;  -- 加排他锁
UPDATE accounts SET balance = balance - 100 WHERE id = 1;
COMMIT;

-- 解决方案2：乐观锁（版本号）
ALTER TABLE accounts ADD COLUMN version INTEGER DEFAULT 0;

UPDATE accounts 
SET balance = balance - 100, version = version + 1 
WHERE id = 1 AND version = 1;  -- 版本号匹配才更新

-- 解决方案3：UPDATE WHERE 条件判断
UPDATE accounts 
SET balance = balance - 100 
WHERE id = 1 AND balance >= 100;  -- 确保余额充足
```

### （3）死锁 (Deadlock)

```sql
-- 死锁场景
-- 事务A: 更新账户1 -> 等待更新账户2
-- 事务B: 更新账户2 -> 等待更新账户1

-- 死锁检测
SHOW deadlock_timeout;  -- 默认 1s

-- 查看死锁日志
-- 日志位置: $PGDATA/log/postgresql-*.log

-- 预防死锁：统一访问顺序
BEGIN;
-- 按 ID 排序访问
SELECT * FROM accounts WHERE id IN (2, 1) ORDER BY id FOR UPDATE;
UPDATE accounts SET balance = balance - 100 WHERE id = 1;
UPDATE accounts SET balance = balance + 100 WHERE id = 2;
COMMIT;
```

### （4）SKIP LOCKED 实现任务队列

```sql
-- 任务队列表
CREATE TABLE task_queue (
    id SERIAL PRIMARY KEY,
    task_name VARCHAR(100),
    payload JSONB,
    status VARCHAR(20) DEFAULT 'pending',
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    processed_at TIMESTAMP
);

-- 消费者获取任务
WITH next_task AS (
    SELECT id 
    FROM task_queue 
    WHERE status = 'pending'
    ORDER BY created_at
    FOR UPDATE SKIP LOCKED
    LIMIT 1
)
UPDATE task_queue 
SET status = 'processing', processed_at = CURRENT_TIMESTAMP
FROM next_task
WHERE task_queue.id = next_task.id
RETURNING task_queue.*;

-- 多个消费者可以并发执行上述 SQL，每个获取不同任务
```

## 4️⃣ MVCC 深入理解

### （1）隐藏的系统列

```sql
-- 每行数据都有隐藏的系统列
SELECT 
    xmin,       -- 插入事务ID
    xmax,       -- 删除事务ID (0表示未删除)
    cmin,       -- 插入命令ID
    cmax,       -- 删除命令ID
    ctid,       -- 物理位置 (块号, 行号)
    tableoid,   -- 表OID
    *
FROM users LIMIT 5;
```

### （2）事务快照

```sql
-- 查看当前事务快照
SELECT txid_current();           -- 当前事务ID
SELECT txid_current_snapshot();  -- 当前快照

-- 快照格式: xmin:xmax:xip_list
-- xmin: 最早活跃事务ID
-- xmax: 最新已分配事务ID + 1
-- xip_list: 活跃事务ID列表
```

### （3）可见性规则

```sql
-- 行可见性判断
-- 1. xmin 对应的事务已提交
-- 2. xmax 为 0 或对应的事务未提交/已回滚
-- 3. 在快照创建时，该行版本已存在

-- 查看行的版本信息
SELECT 
    xmin,
    xmax,
    CASE 
        WHEN xmax = 0 THEN '当前版本'
        ELSE '旧版本（死元组）'
    END AS status,
    *
FROM users;
```

## 5️⃣ 并发控制最佳实践

### （1）避免长事务

```sql
-- 查看长事务
SELECT 
    pid,
    now() - xact_start AS transaction_duration,
    now() - query_start AS query_duration,
    state,
    query
FROM pg_stat_activity
WHERE state != 'idle' 
AND xact_start IS NOT NULL
AND now() - xact_start > interval '5 minutes'
ORDER BY xact_start;

-- 设置超时
SET statement_timeout = '30s';      -- 语句超时
SET lock_timeout = '10s';           -- 锁等待超时
SET idle_in_transaction_session_timeout = '1min';  -- 空闲事务超时
```

### （2）批量处理避免锁竞争

```sql
-- 分批处理大量数据
DO $$
DECLARE
    batch_size INTEGER := 1000;
    affected INTEGER;
BEGIN
    LOOP
        UPDATE large_table 
        SET processed = true 
        WHERE id IN (
            SELECT id FROM large_table 
            WHERE processed = false 
            LIMIT batch_size
            FOR UPDATE SKIP LOCKED
        );
        GET DIAGNOSTICS affected = ROW_COUNT;
        EXIT WHEN affected = 0;
        COMMIT;
    END LOOP;
END $$;
```

```sql
-- 最大连接数设置
```

## 2️⃣ 逻辑备份 (pg_dump)

### （1）基本用法

```bash
# 备份单个数据库
pg_dump -h localhost -U postgres -d mydb > mydb_backup.sql

# 压缩备份
pg_dump -h localhost -U postgres -d mydb | gzip > mydb_backup.sql.gz

# 自定义格式备份（推荐）
pg_dump -h localhost -U postgres -d mydb -Fc > mydb_backup.dump

# 备份所有数据库
pg_dumpall -h localhost -U postgres > all_databases.sql

# 只备份表结构
pg_dump -h localhost -U postgres -d mydb -s > schema_only.sql

# 只备份数据
pg_dump -h localhost -U postgres -d mydb -a > data_only.sql
```

### （2）选择性备份

```bash
# 备份指定表
pg_dump -h localhost -U postgres -d mydb -t users -t orders > backup.sql

# 备份指定 schema
pg_dump -h localhost -U postgres -d mydb -n public > backup.sql

# 排除某些表
pg_dump -h localhost -U postgres -d mydb -T 'temp_*' > backup.sql

# 只备份特定数据
pg_dump -h localhost -U postgres -d mydb \
  --table=orders \
  --where="created_at >= '2024-01-01'" > backup.sql
```

### （3）恢复操作

```bash
# 恢复 SQL 格式备份
psql -h localhost -U postgres -d mydb < mydb_backup.sql

# 恢复压缩备份
gunzip < mydb_backup.sql.gz | psql -h localhost -U postgres -d mydb

# 恢复自定义格式备份
pg_restore -h localhost -U postgres -d mydb -Fc mydb_backup.dump

# 恢复到新数据库
createdb -h localhost -U postgres newdb
pg_restore -h localhost -U postgres -d newdb -Fc mydb_backup.dump

# 只恢复表结构
pg_restore -h localhost -U postgres -d mydb -Fc -s mydb_backup.dump

# 只恢复数据
pg_restore -h localhost -U postgres -d mydb -Fc -a mydb_backup.dump

-- 恢复时排除某些表
pg_restore -h localhost -U postgres -d mydb -Fc -T 'temp_*' mydb_backup.dump
```

### （4）并行备份与恢复

```bash
# 并行备份（4 个 job）
pg_dump -h localhost -U postgres -d mydb -Fc -j 4 > mydb_backup.dump

# 并行恢复
pg_restore -h localhost -U postgres -d mydb -Fc -j 4 mydb_backup.dump
```

## 3️⃣ 物理备份 (pg_basebackup)

### （1）基本用法

```bash
# 基础物理备份

# -U: 复制用户

# -v: 详细输出

# 压缩备份

```

```sql
-- 创建复制用户
CREATE USER replicator WITH REPLICATION ENCRYPTED PASSWORD 'your_password';

-- 授予复制权限

SELECT rolname, rolreplication FROM pg_roles WHERE rolname = 'replicator';
```

```ini
host    replication     replicator      192.168.1.0/24          scram-sha-256
```

```bash

# 2. 清空数据目录
rm -rf /var/lib/postgresql/data/*
```

## 5️⃣ 时间点恢复 (PITR)

### （1）准备基础备份

```bash
# 1. 执行基础备份
pg_basebackup -h localhost -U replicator -D /backup/base -Fp -Xs -P -v

# 2. 记录备份时间
echo "Backup time: $(date)" > /backup/base/backup_time.txt
```

### （2）配置恢复

```ini
# postgresql.conf

# 恢复到指定时间
restore_command = 'cp /archive/wal/%f %p'
recovery_target_time = '2024-03-08 10:30:00'

# 或恢复到指定 LSN
recovery_target_lsn = '0/3000000'

# 或恢复到指定事务 ID
recovery_target_xid = '12345'

# 恢复后行为
recovery_target_action = 'pause'  -- pause, promote, shutdown
```

### （3）执行恢复

```bash
# 1. 停止服务
pg_ctl stop -D /var/lib/postgresql/data

# 2. 清空数据目录
rm -rf /var/lib/postgresql/data/*

# 3. 恢复基础备份
cp -r /backup/base/* /var/lib/postgresql/data/

# 4. 创建恢复信号文件
touch /var/lib/postgresql/data/recovery.signal

# 5. 启动服务（开始恢复）
pg_ctl start -D /var/lib/postgresql/data

# 6. 恢复完成后，执行 promote
pg_ctl promote -D /var/lib/postgresql/data
```

### （4）PostgreSQL 12+ 新语法

```ini
# postgresql.conf

# 使用 restore_command
restore_command = 'cp /archive/wal/%f %p'

# 恢复目标
recovery_target_time = '2024-03-08 10:30:00'
recovery_target_inclusive = on  -- 包含目标时间点

# 恢复后动作
recovery_target_action = 'promote'
```

```bash
pg_basebackup -h primary -U replicator -D /var/lib/postgresql/data -Fp -Xs -P -v

# 创建 standby.signal
touch /var/lib/postgresql/data/standby.signal

# 启动从库
pg_ctl start -D /var/lib/postgresql/data
```

### （2）自动故障转移

```bash
# 使用 repmgr 管理高可用

# 初始化集群
repmgr -f /etc/repmgr.conf primary register

# 注册从库
repmgr -f /etc/repmgr.conf standby clone -h primary
repmgr -f /etc/repmgr.conf standby register

# 查看集群状态
repmgr -f /etc/repmgr.conf cluster show

# 手动切换
repmgr -f /etc/repmgr.conf standby promote
```

## ✅ 本章小结

| 知识点 | 面试关键词 | 实际应用 |
|--------|-----------|----------|
| **逻辑备份** | pg_dump、pg_restore、自定义格式 | 小数据库、迁移 |
| **物理备份** | pg_basebackup、流式 WAL | 大数据库、生产环境 |
| **WAL 归档** | archive_mode、归档命令 | PITR、持续备份 |
| **时间点恢复** | PITR、recovery_target | 数据恢复 |
| **高可用** | 主从复制、故障转移 | 业务连续性 |
| **自动化** | Cron、监控告警 | 运维效率 |

**学习要点**：
- 小数据库用 pg_dump，大数据库用 pg_basebackup
- WAL 归档是实现 PITR 的基础
- 定期验证备份可恢复性
- 配置监控告警，及时发现备份问题
- 高可用方案需要结合业务需求选择

## 🎯 面试常见追问

| 面试官提问 | 回答思路 |
|------------|----------|
| pg_dump 和 pg_basebackup 有什么区别？ | pg_dump 是逻辑备份（SQL 语句），可跨版本但恢复慢；pg_basebackup 是物理备份（数据文件），恢复快但版本要求严格 |
| 什么是 PITR？如何实现？ | 时间点恢复，通过基础备份 + WAL 归档，恢复到指定时间点。配置 archive_mode 和 recovery_target |
| 如何验证备份是否可用？ | 使用 pg_restore -l 查看备份内容；测试恢复到临时数据库；检查备份文件大小和时间 |
| WAL 归档的配置步骤？ | 开启 archive_mode = on，配置 archive_command，创建归档目录，设置权限，验证归档正常工作 |
| 如何实现数据库高可用？ | 主从复制 + 自动故障转移。使用 pg_basebackup 初始化从库，配置复制槽，使用 repmgr 或 Patroni 管理集群 |
