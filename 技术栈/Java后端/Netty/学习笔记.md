### 1.1 Netty 简介

#### （1）定义与定位

**Netty** 是基于 **Java NIO** 的**异步事件驱动**网络应用框架，用于快速开发**高性能、高可靠**的 TCP/UDP 服务端与客户端。

| 项目 | 说明 |
|------|------|
| **来源** | 由 JBoss 提供，现由社区维护。 |
| **核心优势** | API 简洁、可扩展、成熟稳定；被大量中间件选为底层通信层。 |
| **典型使用者** | Dubbo、RocketMQ、Elasticsearch、gRPC、Zookeeper 等。 |

> 💬 **一句话**：Netty 把 NIO 的复杂与坑封装成**事件驱动 + Pipeline** 模型，让开发者专注业务逻辑，写出高性能网络程序。

#### （2）为什么需要 Netty

- **Java 原生 NIO**：需手写 Selector、ByteBuffer、线程模型，**粘包半包、边界处理、线程安全**等都要自己管，容易出错。
- **Netty**：提供 **Channel、EventLoop、Pipeline、ByteBuf** 等抽象，**内置常见编解码与线程模型**，减少样板代码与坑，同时保持高性能。

------

### 1.2 与传统 I/O 及同类框架对比

#### （1）BIO、NIO、Netty 关系

| 模型 | 说明 | 特点 |
|------|------|------|
| **BIO** | 同步阻塞 I/O，一连接一线程 | 实现简单，高并发下线程数爆炸 |
| **NIO** | 非阻塞 I/O + Selector 多路复用 | 单线程可管理多 Channel，需自己处理边界与线程 |
| **Netty** | 在 NIO 之上封装事件驱动与 Pipeline | 易用、可扩展、生产级，底层仍是 NIO |

#### （2）与同类 NIO 框架

| 框架 | 说明 |
|------|------|
| **Mina** | 同为 NIO 框架，Netty 由 Mina 主导者后来开发，API 与生态更流行。 |
| **Grizzly** | 偏 GlassFish 生态；Netty 在 Java 社区采用更广。 |

## 第二章：Netty 架构与核心组件

------

> **本章在整体中解决什么问题**：第一章讲了 Netty 的定位与场景；本章落实**核心抽象**——**Channel**（连接）、**EventLoop/EventLoopGroup**（事件循环与线程）、**Pipeline 与 Handler**（入站/出站责任链）、**ByteBuf** 与编解码（粘包半包等）。理解这些后**第三章**的 Reactor、Boss/Worker 分工与**第四章**的服务端/客户端代码才会对上号；写 Handler、排错、面试问「核心组件」都能答。

------

### 2.1 核心组件

#### （1）Channel、EventLoop、EventLoopGroup

| 组件 | 说明 |
|------|------|
| **Channel** | 对 NIO Channel 的封装，代表一条**网络连接**（或可读可写的抽象）；可获取 Pipeline、配置、远程地址等。 |
| **EventLoop** | **事件循环**，绑定一个线程与一个 Selector，负责处理注册到其上的 Channel 的 I/O 事件与提交的任务；一个 Channel 只会被一个 EventLoop 处理，保证**单线程无锁**。 |
| **EventLoopGroup** | 一组 EventLoop；**BossGroup** 通常只做 accept，将新连接注册到 **WorkerGroup** 的某个 EventLoop；**WorkerGroup** 负责已连接 Channel 的 read/write。 |

```
BossGroup(accept) → 新连接 → 注册到 WorkerGroup 的某个 EventLoop
WorkerGroup       → 每个 EventLoop 处理若干 Channel 的 I/O
```

#### （2）ChannelPipeline 与 ChannelHandler

| 组件 | 说明 |
|------|------|
| **ChannelPipeline** | **Handler 链**，入站（Inbound）与出站（Outbound）事件依次经过链上的 Handler；类似责任链，可动态增删。 |
| **ChannelHandler** | 处理连接建立/断开、读、写、异常等；**ChannelInboundHandler** 处理入站（如读到的数据、连接建立）；**ChannelOutboundHandler** 处理出站（如 write、connect、bind）。 |
| **ChannelHandlerContext** | Handler 在 Pipeline 中的上下文，可写数据（ctx.writeAndFlush）、 fire 事件、获取 Channel 等。 |

> 💡 **重点**：**入站**从网络读入 → 自上而下经过 InboundHandler；**出站**从业务写出 → 自下而上经过 OutboundHandler；编解码通常放在前面，业务 Handler 在后。

------

### 2.2 ByteBuf 与编解码

#### （1）ByteBuf

- Netty 的**字节容器**，替代 JDK **ByteBuffer**；支持**双索引**（readerIndex/writerIndex）、**池化**、**直接内存**，API 更易用。
- **池化**（PooledByteBufAllocator）减少 GC；**直接内存**（DirectBuffer）减少一次拷贝，但占用堆外内存，需注意释放与 -XX:MaxDirectMemorySize。

#### （2）编解码（Codec）

| 问题/需求 | 常见方案 |
|-----------|----------|
| **粘包半包** | **LengthFieldBasedFrameDecoder**：按长度字段拆包；或定长、分隔符等。 |
| **字节 ↔ 业务对象** | **MessageToMessageDecoder/Encoder** 或 **ByteToMessageDecoder / MessageToByteEncoder** 做协议解析与序列化。 |
| **简化“长度不足等待”** | **ReplayingDecoder**（按需读，不足时内部重试，性能略差）。 |

------

## ✅ 本章小结

| 知识点 | 面试关键词 | 实际应用 |
|--------|------------|----------|
| **Channel / EventLoop** | 连接抽象、事件循环、一个 Channel 绑定一个 EventLoop | 理解线程模型与无锁 |
| **Pipeline / Handler** | 责任链、Inbound/Outbound、编解码在前业务在后 | 定制处理逻辑与顺序 |
| **ByteBuf** | 双索引、池化、直接内存、release | 高性能与防泄漏 |
| **粘包半包** | LengthFieldBasedFrameDecoder、定长、分隔符 | 正确拆包与解码 |

------

**学习要点**：

- EventLoop 与 Channel 一一绑定，保证同一 Channel 上的 I/O 与任务在同一线程执行，无需加锁。
- Pipeline 中先解码再业务再编码；ByteBuf 谁用谁 release，或交给 SimpleChannelInboundHandler 自动释放。
- 粘包半包用基于长度、定长或分隔符的 Decoder 解决。

| 面试官提问 | 回答思路 |
|------------|----------|
| Netty 有哪些核心组件？ | Channel（连接）、EventLoop（事件循环+线程）、EventLoopGroup（一组 EventLoop）、Pipeline（Handler 链）、Handler（入站/出站处理）、ByteBuf（字节容器）。 |
| 什么是粘包半包？怎么解决？ | TCP 是流式，多次写可能被合并成一次读（粘包），一次写可能被拆成多次读（半包）；解决：定长、分隔符、或 LengthFieldBasedFrameDecoder 按长度拆包。 |
| Reactor 模型在 Netty 里怎么体现？ | BossGroup 相当于 main Reactor（accept），WorkerGroup 相当于 sub Reactor（read/write）；每个 EventLoop 对应一个线程 + Selector，即多 Reactor 多线程。 |

### 常见坑与注意点

| 现象 / 易错点 | 原因 | 怎么改 / 怎么记 |
|---------------|------|-----------------|
| ByteBuf 未 release 导致堆外内存泄漏 | 直接内存不归 GC 管，需手动 release | **谁用谁 release**，或交给 **SimpleChannelInboundHandler** 自动释放；池化下未 release 会漏回池。 |
| 粘包半包导致解析错乱 | TCP 流式，未按协议拆包 | 在 Pipeline **最前**加 **LengthFieldBasedFrameDecoder**（或定长、分隔符 Decoder），先拆包再解码业务对象。 |
| 入站出站顺序搞反 | Inbound 自上而下、Outbound 自下而上 | **入站**（读到的数据）从上往下走 InboundHandler；**出站**（write）从下往上走 OutboundHandler；编解码放前面、业务放后面。 |

## 第三章：Netty 网络通信与线程模型

------

> 本章说明 Netty 如何基于 **NIO 与 Reactor** 实现事件驱动、Boss/Worker 分工、以及异步写与 Future/Listener；与第二章组件对应。

------

### 3.1 基于 NIO 的多路复用

- **NIO**：非阻塞 I/O + **Selector** 多路复用，单线程可监听多 Channel 的读/写/连接事件。
- **Netty**：每个 **EventLoop** 绑定**一个线程**与**一个 Selector**，负责若干 Channel；新连接由 Boss 的 EventLoop accept 后，将 Channel 注册到 Worker 的某个 EventLoop，后续该 Channel 的 I/O 都在该 EventLoop 上执行。

------

### 3.2 事件驱动与线程模型

| 角色 | 职责 |
|------|------|
| **BossGroup** | 通常单线程（或少量），只做 **accept**，把新连接的 Channel 注册到 WorkerGroup 的某个 EventLoop。 |
| **WorkerGroup** | 多线程，每个 EventLoop 处理已分配 Channel 的 **read/write** 等 I/O 与提交到该 EventLoop 的任务。 |

**异步写**：`channel.writeAndFlush(msg)` 返回 **ChannelFuture**，可通过 **addListener** 或 **await** 获取结果；**不要在 EventLoop 里做阻塞操作**（如 DB、RPC），应提交到业务线程池，避免阻塞其他 Channel。

------

## ✅ 本章小结

| 知识点 | 面试关键词 | 实际应用 |
|--------|------------|----------|
| **NIO 与 EventLoop** | Selector、一个 EventLoop 一个线程、多路复用 | 理解高性能来源 |
| **Boss/Worker** | accept 与 read/write 分离、注册到 Worker | 服务端启动与连接分配 |
| **异步** | ChannelFuture、Listener、不阻塞 EventLoop | 正确使用 write 与耗时任务 |

------

**学习要点**：Boss 只做 accept，Worker 做 I/O；写操作异步，耗时任务放业务线程池，保证 EventLoop 不阻塞。

### 常见坑与注意点

| 现象 / 易错点 | 原因 | 怎么改 / 怎么记 |
|---------------|------|-----------------|
| 在 EventLoop 里做 DB/RPC 导致其他 Channel 卡顿 | EventLoop 单线程，阻塞会拖慢同组所有 Channel | **耗时操作**提交到**业务线程池**，完成后再通过 **ctx.channel().eventLoop().execute()** 或 **writeAndFlush** 写回（保证写在同一 EventLoop）。 |
| writeAndFlush 后未考虑失败 | 写是异步的，网络满或对端关闭可能失败 | 对 **ChannelFuture** 加 **addListener** 或 **await** 处理失败；在 Listener 里根据 cause 重试或关闭。 |
| Boss/Worker 线程数设过多 | 过多 EventLoop 导致上下文切换与资源占用 | Boss 通常 **1** 即可；Worker 默认 **CPU 核数×2**，I/O 密集可略增，不宜过大。 |

## 第四章：Netty 服务端与客户端开发

------

> 本章给出**服务端与客户端**的启动流程、Pipeline 初始化、Handler 编写及 Echo/简单业务示例；可直接照抄改造。

------

### 4.1 服务端流程

#### （1）步骤概览

1. 创建 **NioEventLoopGroup**：Boss（通常 1）、Worker（通常 CPU 核数 * 2 或按需）。
2. **ServerBootstrap**：group(Boss, Worker)、channel(NioServerSocketChannel.class)、childHandler(初始化 Pipeline)。
3. **bind(port)**，同步或异步；关闭时 shutdownGracefully。

#### （2）代码示例

```java
EventLoopGroup boss = new NioEventLoopGroup(1);
EventLoopGroup worker = new NioEventLoopGroup();
try {
    ServerBootstrap b = new ServerBootstrap();
    b.group(boss, worker)
     .channel(NioServerSocketChannel.class)
     .childHandler(new ChannelInitializer<SocketChannel>() {
         @Override
         protected void initChannel(SocketChannel ch) {
             ch.pipeline()
               .addLast(new SomeDecoder())
               .addLast(new SomeEncoder())
               .addLast(new MyBusinessHandler());
         }
     });
    ChannelFuture f = b.bind(8080).sync();
    f.channel().closeFuture().sync();
} finally {
    boss.shutdownGracefully();
    worker.shutdownGracefully();
}
```

------

### 4.2 客户端流程

- 使用 **Bootstrap**（无 childHandler）、group、channel(NioSocketChannel.class)、handler（初始化 Pipeline）→ **connect(host, port)**；关闭同样 shutdownGracefully。

```java
Bootstrap b = new Bootstrap();
b.group(group)
 .channel(NioSocketChannel.class)
 .handler(new ChannelInitializer<SocketChannel>() {
     @Override
     protected void initChannel(SocketChannel ch) {
         ch.pipeline().addLast(new MyClientHandler());
     }
 });
ChannelFuture f = b.connect("127.0.0.1", 8080).sync();
```

------

### 4.3 Handler 定制

| 类型 | 说明 |
|------|------|
| **ChannelInitializer** | 在连接建立时执行一次，向 Pipeline 添加各类 Handler。 |
| **ChannelInboundHandlerAdapter** | 重写 channelRead、channelActive、channelInactive、exceptionCaught 等处理入站。 |
| **SimpleChannelInboundHandler&lt;T&gt;** | 泛型指定消息类型，自动 release 入站消息，只处理该类型。 |
| **ChannelOutboundHandlerAdapter** | 重写 write、flush、connect 等处理出站。 |
| **写出数据** | **ctx.writeAndFlush(msg)**；msg 会从当前 Handler 沿 Outbound 方向传播并最终写出。 |

------

### 4.4 常见场景示例

| 场景 | 要点 |
|------|------|
| **Echo** | 入站读到 ByteBuf 后直接 **ctx.writeAndFlush(msg)** 回写（可复用同一 ByteBuf 或 copy）。 |
| **WebSocket** | 添加 **WebSocketServerProtocolHandler** 做握手与帧处理，再加业务 Handler。 |
| **HTTP** | **HttpServerCodec** + **HttpObjectAggregator**（合并 FullHttpRequest） + 业务 Handler。 |

### 常见坑与注意点

| 现象 / 易错点 | 原因 | 怎么改 / 怎么记 |
|---------------|------|-----------------|
| childHandler 与 handler 用混 | 服务端用 **childHandler**（处理**新连接**），客户端用 **handler**（处理**当前连接**） | 服务端 **ServerBootstrap.childHandler**；客户端 **Bootstrap.handler**；记「child=子 Channel」。 |
| Pipeline 里编解码顺序反了 | 解码应在**入站最前**、编码在**出站最后**，否则业务拿到未解码数据 | **先 addLast Decoder**（入站），再业务 Handler，再 **addLast Encoder**（出站）；入站从上往下、出站从下往上。 |
| 服务端未调 closeFuture().sync() | 主线程直接退出，Netty 线程随之结束 | **channel.closeFuture().sync()** 让主线程等待 Channel 关闭；或在 Spring 里用 @PreDestroy 里 shutdownGracefully 并 await。 |

## 第五章：性能优化与内存管理

------

> 本章说明 **ByteBuf 池化与直接内存、及时 release、业务线程池、连接与并发**等优化点，以及常见坑与监控建议。

------

### 5.1 内存与 ByteBuf

| 手段 | 说明 |
|------|------|
| **池化** | 使用 **PooledByteBufAllocator**（默认），减少频繁分配与 GC。 |
| **直接内存** | DirectBuffer 减少一次拷贝，提升 I/O 性能；占用堆外，需关注 -XX:MaxDirectMemorySize 与泄漏。 |
| **及时 release** | 谁用谁 release；或使用 **SimpleChannelInboundHandler**，入站消息会自动 release；**ReferenceCountUtil.release(msg)** 手动释放。 |

> ⚠️ **注意**：若把 ByteBuf 或引用传递到其他线程或异步回调，需确保引用计数正确，否则易泄漏；不要在没有 release 的情况下丢弃 ByteBuf。

------

### 5.2 连接与并发

| 方面 | 建议 |
|------|------|
| **长连接** | 客户端复用 Channel，减少建连开销。 |
| **Worker 线程数** | 默认 CPU 核数 * 2；I/O 密集可适当增加，避免过多导致上下文切换。 |
| **耗时操作** | **不要阻塞 EventLoop**；DB、RPC、复杂计算提交到**业务线程池**，完成后再通过 ctx.writeAndFlush 写回（注意线程安全）。 |
| **背压与队列** | 根据场景配置写缓冲与高低水位，避免 OOM。 |

| 知识点 | 面试关键词 | 实际应用 |
|--------|------------|----------|
| **ByteBuf** | 池化、直接内存、release、泄漏 | 高性能与防泄漏 |
| **线程** | 不阻塞 EventLoop、业务线程池 | 正确使用异步与线程池 |
| **连接与并发** | 长连接、Worker 数、背压 | 调优与稳定性 |

### 常见坑与注意点

| 现象 / 易错点 | 原因 | 怎么改 / 怎么记 |
|---------------|------|-----------------|
| 堆外内存持续增长（DirectBuffer leak） | ByteBuf 未 release 或传递到其他线程后未释放 | **谁用谁 release**；**SimpleChannelInboundHandler** 会释放入站消息；跨线程传递时用 **ReferenceCountUtil.retain/release** 或 copy 后原 buf release。 |
| -XX:MaxDirectMemorySize 设太小 | 直接内存超限抛 OOM: Direct buffer memory | 根据连接数与缓冲需求调大 **MaxDirectMemorySize**；或排查泄漏。 |
| 业务线程里直接 ctx.writeAndFlush | 写操作应在该 Channel 绑定的 EventLoop 上执行 | 在业务线程里用 **ctx.channel().eventLoop().execute(() -> ctx.writeAndFlush(msg))** 或 **channel.writeAndFlush**，保证写在同一 EventLoop。 |

### 6.1 与 Spring 集成

#### （1）生命周期管理

在 Spring 中**统一管理** EventLoopGroup 与 ServerBootstrap/Bootstrap，保证应用启动时 Netty 服务启动、应用关闭时**优雅关闭**（shutdownGracefully），避免线程池未释放或连接未断开。

**要点**：

- **启动**：在 **@PostConstruct** 或 **ApplicationRunner** 中创建 EventLoopGroup、Bootstrap，并 **bind()**；可把 ChannelFuture 或 Bootstrap 存为 Bean 供后续使用。
- **关闭**：在 **@PreDestroy** 或 **@Bean(destroyMethod)** 中先关闭 **Channel**（若有），再 **shutdownGracefully()** EventLoopGroup，并 **await** 一段时间（如 10s）再退出，避免强杀。

示例思路（服务端）：

```java
@Component
public class NettyServer {
    private EventLoopGroup bossGroup;
    private EventLoopGroup workerGroup;
    private Channel channel;

    @PostConstruct
    public void start() throws InterruptedException {
        bossGroup = new NioEventLoopGroup(1);
        workerGroup = new NioEventLoopGroup();
        ServerBootstrap b = new ServerBootstrap();
        b.group(bossGroup, workerGroup).channel(NioServerSocketChannel.class)
         .childHandler(new MyChannelInitializer());
        channel = b.bind(8080).sync().channel();
    }

    @PreDestroy
    public void stop() {
        if (channel != null) channel.close();
        if (bossGroup != null) bossGroup.shutdownGracefully();
        if (workerGroup != null) workerGroup.shutdownGracefully();
    }
}
```

#### （2）Handler 中注入 Spring Bean 与线程安全

| 方式 | 说明 | 注意 |
|------|------|------|
| **Handler 由 Spring 创建** | 在 ChannelInitializer 里用 **applicationContext.getBean(MyHandler.class)** 注入到 Pipeline；Handler 可 **@Autowired** 其他 Spring Bean。 | Handler 若为**单例**，会被多条 Channel 共享，**必须保证无实例字段或线程安全**；否则用 **prototype** 或每次 **new Handler()** 并手动注入需要的 Bean。 |
| **在 Handler 里拿 Bean** | 通过 **ApplicationContext** 或 **ObjectProvider** 在 channelRead 等回调里按需 getBean。 | 每次请求都查 Bean，注意性能与是否单例；适合无状态服务。 |

> 💡 **推荐**：业务 Handler 尽量**无状态**；若需依赖 Spring Bean，优先注入**无状态服务**（如 Service），避免在 Handler 内持有可变成员变量，否则要加锁或使用 prototype Scope。

#### （3）与 Spring Boot

- 可手写 **@Configuration** 注册上述 NettyServer/NettyClient Bean，或使用社区 **spring-boot-starter-netty** 等 Starter 简化配置与生命周期。
- 端口、线程数等可从 **application.yml** 读取，与 Spring 配置体系统一。

## 第七章：安全与容错

------

> 本章说明 **SSL/TLS 在 Pipeline 中的用法**、**重连与心跳的完整实现要点**、以及**异常处理最佳实践**，保证生产可用与面试能答。

------

### 7.1 安全

#### （1）SSL/TLS

- **SslHandler**：基于 JDK **SSLEngine**，在 Pipeline 中应放在**最前面**（或紧跟解码前的入口），先完成**握手与加解密**，后续 Handler 只处理明文。
- **服务端**：需配置 **SSLContext**（证书、私钥、信任链）；用 `SslHandler(sslEngine)` 或 Netty 提供的 `SslHandlerBuilder` 构建并 **pipeline.addLast(sslHandler)**。
- **客户端**：同样配置 SSLContext，并设置 **host/port** 用于 SNI 与证书校验。
- **应用层**：可在业务层再做**鉴权、签名、限流**等，与传输层 SSL 配合。

> 🔹 **注意**：SslHandler 是 **@Sharable** 时需保证 SSLEngine 按连接创建（每个 Channel 独立 Engine），否则并发会错乱；一般每个 Channel  new 一个 SslHandler。

#### （2）应用层安全

- 在业务 Handler 中做 **Token/签名校验**、**限流**（如按 IP 或 userId），防止未授权访问与滥用。

------

### 7.2 容错与高可用

#### （1）重连（客户端）

- **时机**：在 **channelInactive** 或 **ChannelFutureListener**（connect 失败时）中触发重连。
- **要点**：**延迟重连**（如 1s、2s、5s 指数退避），避免连接瞬间断开又立刻重连造成**重连风暴**；可设**最大重试次数**，超过后告警或降级。
- **实现思路**：用 **EventLoop.schedule(runnable, delay, unit)** 在 N 秒后再次调用 **Bootstrap.connect()**，并在 **ChannelFutureListener** 里判断成功/失败，失败则继续调度下一次重连。

```java
// 伪代码：channelInactive 中调度重连
@Override
public void channelInactive(ChannelHandlerContext ctx) {
    ctx.channel().eventLoop().schedule(this::doConnect, 2, TimeUnit.SECONDS);
}
```

#### （2）超时与心跳

- **IdleStateHandler**：在 Pipeline 中加入，用于检测**读空闲 / 写空闲 / 全空闲**；超时后触发 **IdleStateEvent**，由后续 Handler 的 **userEventTriggered** 处理。
- **常见策略**：**读空闲**（如 60s）未收到任何数据 → 认为连接假死，**关闭连接**或**发心跳包**；**写空闲**（如 30s）未写数据 → **主动发心跳**，保持连接活跃。服务端与客户端都可加，参数按协议约定（如 60s 读空闲关连接、30s 写空闲发 ping）。
- **配置示例**：`new IdleStateHandler(60, 30, 0, TimeUnit.SECONDS)` 表示读空闲 60s、写空闲 30s 触发事件；下一个 Handler 里根据事件类型发心跳或 close。

#### （3）异常处理

- 在 **exceptionCaught** 中**记录日志**（含 ctx.channel().remoteAddress()、异常栈），并 **ctx.close()** 关闭 Channel，避免异常连接占用资源。
- 可根据异常类型做**区分**：如解码异常只关当前连接；可恢复的临时错误可配合重连；**不要吞掉异常**，生产可对接告警。

| 机制 | 说明 |
|------|------|
| **连接池** | 客户端维护多个 Channel，按负载或轮询选择，避免单连接瓶颈。 |
| **重连** | channelInactive/connect 失败时**延迟重连**（指数退避），避免风暴；可设最大次数。 |
| **心跳** | IdleStateHandler 检测读/写空闲，写空闲发心跳、读空闲超时关连接或发心跳。 |
| **异常** | exceptionCaught 中打日志并 ctx.close()，按需告警或重试。 |

|--------|------------|----------|
| **SSL** | SslHandler、Pipeline 最前、每 Channel 独立 Engine | HTTPS、安全通道 |
| **容错** | 延迟重连、IdleStateHandler、心跳、exceptionCaught 打日志并关连接 | 长连接稳定与故障恢复 |

**学习要点**：SSL 放 Pipeline 最前、每连接独立 SslHandler；重连要延迟/退避；心跳用 IdleStateHandler + userEventTriggered；异常在 exceptionCaught 中记录并关闭 Channel。

## 第八章：自定义编解码与高级特性

------

> 本章说明**自定义解码/编码**的常见基类与粘包处理、以及和 RPC/自定义协议的关系。

------

### 8.1 自定义编解码

| 类型 | 基类 | 说明 |
|------|------|------|
| **解码** | **ByteToMessageDecoder** | 把 ByteBuf 转成一条条消息，需自己处理粘包半包（如先读长度再读 body）。 |
| | **MessageToMessageDecoder&lt;I&gt;** | 上一步已是消息对象，再转成另一种消息。 |
| | **ReplayingDecoder** | 按需读，长度不足时内部重试，代码简单但性能略差。 |
| **编码** | **MessageToByteEncoder&lt;I&gt;** | 把消息对象编码成 ByteBuf。 |
| | **MessageToMessageEncoder&lt;I&gt;** | 消息转消息，再交给后续 Encoder 或直接写。 |

> 🔹 **注意**：解码时若数据不足，不要消费 ByteBuf 的 readerIndex，可先 markReaderIndex，等凑够再读；或交给 LengthFieldBasedFrameDecoder 先拆帧再解码。

------

### 8.2 与 RPC、自定义协议

- **Dubbo、gRPC** 等在 Netty 之上定义自己的**协议**（魔数、长度、序列化 body 等），服务端与客户端共用一套编解码。
- 自定义微服务通信时：定义**私有协议**（如魔数 + 长度 + body），用 Netty 做服务端与客户端，配合序列化与路由即可。

| **RPC** | 私有协议、魔数+长度+body | 基于 Netty 做 RPC 传输层 |

### 常见坑与注意点

| 现象 / 易错点 | 原因 | 怎么改 / 怎么记 |
|---------------|------|-----------------|
| 解码时消费了 ByteBuf 导致半包时数据丢 | 数据不足时读了 readerIndex 又未 reset | 数据不足时 **不要** 移动 readerIndex，或先 **markReaderIndex()** 再 **resetReaderIndex()**；或交给 **LengthFieldBasedFrameDecoder** 先拆帧。 |
| 自定义 Decoder 未考虑粘包 | 一次 read 收到多条消息，只解析一条就 return | **循环**：while (in.readableBytes() >= 需要长度) 拆出一条并 out.add(msg)；或外层用 **LengthFieldBasedFrameDecoder** 再接 **MessageToMessageDecoder**。 |
| ReplayingDecoder 性能差或报错 | 内部重试机制有开销，且某些操作不支持 | 对性能敏感用 **ByteToMessageDecoder** 手动判断 **readableBytes()** 再 decode；ReplayingDecoder 仅适合协议简单的场景。 |
