# 📘《Netty 学习与实战手册》

> 系统学习见下文各章；日常常用 API、组件与场景速查见同目录《常用API与使用场景》。

------

## 第一章：Netty 概述

### 1.1 Netty 简介

- **Netty** 是基于 Java NIO 的**异步事件驱动**网络应用框架，用于快速开发高性能、高可靠的 TCP/UDP 服务端与客户端。
- 由 JBoss 提供，现为社区维护；**核心优势**：API 简洁、可扩展、成熟稳定，被 Dubbo、RocketMQ、Elasticsearch、gRPC 等大量中间件选为底层通信层。

### 1.2 与传统 I/O 及同类框架对比

- **与 Java 原生 NIO**：NIO 需手写 Selector、ByteBuffer、线程模型，Netty 封装了 **Channel、EventLoop、Pipeline**，减少样板代码与坑。
- **与 Mina、Grizzly**：同为 NIO 框架，Netty 生态与文档更丰富，企业采用更广。
- **应用场景**：RPC 框架、消息中间件、长连接推送、HTTP/WebSocket 服务、自定义协议等**高性能网络编程**。

------

## 第二章：Netty 架构与核心组件

### 2.1 核心组件

- **Channel**：对 NIO Channel 的封装，代表一条连接；**EventLoop**：事件循环，绑定线程，处理该 Channel 上的 I/O 与任务；**EventLoopGroup**：一组 EventLoop，Boss 接受连接，Worker 处理 I/O；**ChannelPipeline**：Handler 链，入站/出站事件依次经过各 Handler；**ChannelHandler**：处理连接建立、读、写、异常等。

### 2.2 ByteBuf 与编解码

- **ByteBuf**：Netty 的字节容器，支持双索引（读/写）、池化、直接内存；比 JDK ByteBuffer 更易用。**解码/编码（Codec）**：将字节与业务对象互转，如 **LengthFieldBasedFrameDecoder** 解决粘包半包，**MessageToMessageEncoder/Decoder** 做协议解析。

------

## 第三章：Netty 网络通信原理

### 3.1 基于 NIO

- **NIO**：非阻塞 I/O + **Selector** 多路复用，单线程可管理多 Channel；Netty 在 NIO 之上封装 **EventLoop**，每个 EventLoop 绑定一个线程与一个 Selector，负责若干 Channel 的读写事件。

### 3.2 事件驱动与线程模型

- **BossGroup**：通常单线程，只做 accept，将新连接注册到 **WorkerGroup** 的某个 EventLoop；**WorkerGroup**：多线程，每个 EventLoop 处理已分配 Channel 的 read/write；**异步**：write 返回 ChannelFuture，通过 Listener 或 await 获取结果，避免阻塞 EventLoop。

------

## 第四章：Netty 业务实现与使用

### 4.1 服务端与客户端流程

- **服务端**：创建 **NioEventLoopGroup**（Boss+Worker）→ **ServerBootstrap** 绑定 group、channel(NioServerSocketChannel.class)、childHandler（初始化 Pipeline）→ **bind(port)**；**客户端**：**Bootstrap**、group、channel(NioSocketChannel.class)、handler → **connect(host, port)**。

### 4.2 Handler 定制

- **ChannelInitializer** 在连接建立时向 Pipeline 添加 Handler；**ChannelInboundHandlerAdapter** 处理读、连接建立/断开、异常；**ChannelOutboundHandlerAdapter** 处理写、bind、connect 等；**ctx.writeAndFlush(msg)** 写出数据。

### 4.3 常见场景

- **Echo**：入站读 ByteBuf，直接 **ctx.writeAndFlush(msg)** 回写；**WebSocket**：添加 **WebSocketServerProtocolHandler** 与业务 Handler；**HTTP**：使用 **HttpServerCodec**、**HttpObjectAggregator** 与业务 Handler。

------

## 第五章：Netty 性能优化与调优

### 5.1 内存与 ByteBuf

- **池化 ByteBuf**（PooledByteBufAllocator）减少 GC；**直接内存**（DirectBuffer）减少拷贝，但需注意 JVM 外内存；**及时 release** 引用，避免泄漏，可用 **ReferenceCountUtil.release(msg)** 或 **SimpleChannelInboundHandler** 自动释放。

### 5.2 连接与并发

- **长连接** 复用减少建连开销；**Worker 线程数** 建议 CPU 核数 * 2 或按 I/O 密集度调整；业务耗时操作放**业务线程池**，避免阻塞 EventLoop；**背压**与队列长度根据场景配置。

------

## 第六章：Netty 与其他技术结合

### 6.1 与 Spring 集成

- Spring 中**管理** EventLoopGroup 与 Bootstrap 的生命周期（@PostConstruct 启动、@PreDestroy 关闭）；或使用 **spring-boot-starter-netty** 等 Starter；业务 Handler 可注入 Spring Bean，注意**非单例**时 Scope 与线程安全。

### 6.2 与消息队列、大数据

- **Kafka、RocketMQ** 等内部用 Netty 做 Broker 与 Client 间通信；**Zookeeper** 客户端也有基于 Netty 的实现；自定义**高性能 RPC 或网关**时可直接基于 Netty 写协议与编解码。

------

## 第七章：Netty 安全性与容错性

### 7.1 安全

- **SSL/TLS**：在 Pipeline 最前添加 **SslHandler**（基于 JDK SSLEngine），先做握手与加解密再走业务 Handler；**数据加密**可在应用层再做一层编解码。

### 7.2 容错与高可用

- **连接池**：客户端可维护多个 Channel，按负载选择；**重连**：连接断开时通过 **ChannelFutureListener** 或 **ChannelInactive** 中调度延迟重连；**超时与心跳**：**IdleStateHandler** 检测空闲，超时关连接或发心跳；异常在 **exceptionCaught** 中记录并关闭或重试。

------

## 第八章：Netty 高级特性与实践

### 8.1 自定义编解码

- 继承 **ByteToMessageDecoder** 或 **MessageToMessageDecoder** 做解码（注意粘包半包）；继承 **MessageToByteEncoder** 或 **MessageToMessageEncoder** 做编码；**ReplayingDecoder** 可简化“长度不足时等待”的逻辑。

### 8.2 分布式与 RPC

- **Dubbo、gRPC** 等 RPC 框架底层用 Netty 做传输；自定义**微服务通信**时，定义私有协议（魔数+长度+body），用 Netty 做服务端与客户端，配合序列化与路由即可。

## 第九章：Netty 发展趋势与社区生态

### 9.1 发展与生态

- Netty 4.x 为主流，持续修复与性能优化；**Netty 5** 曾探索过但未替代 4；社区活跃，文档与书籍丰富，Dubbo、RocketMQ、Elasticsearch 等源码可参考实战用法。

------

## 第十章：总结与最佳实践

### 10.1 应用场景总结

- **高并发服务端**：游戏、推送、即时通讯；**RPC/网关**：Dubbo、gRPC、Spring Cloud Gateway 等；**消息中间件**：Kafka、RocketMQ 的传输层；**HTTP/WebSocket 服务**：长连接、实时推送。

### 10.2 最佳实践

- **不在 EventLoop 中做阻塞操作**，耗时任务提交到业务线程池；**ByteBuf 谁用谁 release**，或交给 **SimpleChannelInboundHandler**；**Pipeline 顺序**：先解码再业务再编码；**异常与空闲**：exceptionCaught、IdleStateHandler 妥善处理；**压测与监控**：连接数、QPS、延迟、堆外内存。
