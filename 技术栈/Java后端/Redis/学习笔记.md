## 1.1 Redis 简介

### 🔹 1.1.1 Redis 是什么？

> **一句话**：Redis 是一个基于**内存**的 **Key-Value 数据库**，支持多种数据结构（不只有字符串），常用于**缓存、分布式锁、排行榜、消息队列**等。
> **为什么需要它**：数据库读写慢、扛不住高并发；用内存存热点数据可以大幅降低延迟、减轻 DB 压力；同时 Redis 提供过期、持久化、集群等能力，比「自己写本地缓存」更完整。
> **类比**：数据库是「大仓库」，Redis 是「前台小货架」——把最常要的东西放货架上，先看货架再看仓库，整体更快。

**Redis（REmote DIctionary Server）** 由意大利开发者 *Salvatore Sanfilippo（antirez）* 于 2009 年发布。

它支持多种数据结构（不仅仅是字符串），例如：

- **String**（字符串）
- **List**（列表）
- **Set**（集合）
- **Hash**（哈希表）
- **ZSet**（有序集合）
- **Stream、Bitmap、HyperLogLog、Geo** 等高级结构

Redis 可用作：

- 缓存系统（最常见）
- 消息队列（基于 List/Stream）
- 排行榜/计数器系统
- 分布式锁实现组件

------

### 🔹 1.1.2 Redis 的特点与优势

| 特性               | 描述                                                         |
| ------------------ | ------------------------------------------------------------ |
| **高性能**         | 基于内存操作，核心操作单线程模型避免上下文切换，速度可达 10 万+ QPS |
| **多数据结构支持** | 除 String 外，还支持 List、Set、ZSet、Hash、Stream 等        |
| **持久化**         | 提供 RDB（快照）与 AOF（追加日志）两种机制，数据可落盘       |
| **高可用性**       | 主从复制 + 哨兵机制 + 集群支持                               |
| **丰富的功能**     | 支持事务、Lua脚本、发布订阅、过期键机制                      |
| **易扩展**         | 集群模式可扩展至数百节点，支持分片与负载均衡                 |

📘 **一句话总结：**
 Redis 是一个集 **缓存 + 存储 + 消息队列 + 分布式协调器** 于一身的高性能中间件。

------

### 🔹 1.1.3 Redis 与 Memcached 的区别

| 对比项       | **Redis**                        | **Memcached**    |
| ------------ | -------------------------------- | ---------------- |
| **数据类型** | 支持多种数据结构                 | 仅支持字符串     |
| **持久化**   | 支持（RDB / AOF）                | 不支持（纯内存） |
| **集群支持** | 原生支持（Cluster）              | 需要客户端实现   |
| **线程模型** | 单线程（命令串行）               | 多线程           |
| **Lua脚本**  | 支持                             | 不支持           |
| **事务机制** | 支持（multi/exec）               | 不支持           |
| **适用场景** | 缓存、分布式锁、消息队列、排行榜 | 简单缓存场景     |

✅ **总结：**

> Redis ≈ “功能更强、可持久化的 Memcached 升级版”。

------

### 🔹 1.1.4 Redis 的常见使用场景

| 场景                    | 描述                              |
| ----------------------- | --------------------------------- |
| 🔸 **缓存热点数据**      | 提高系统性能，减少数据库压力      |
| 🔸 **分布式锁实现**      | 基于 `SETNX + EXPIRE` 或 Redisson |
| 🔸 **计数器 / 排行榜**   | 利用 ZSet 实现实时排名            |
| 🔸 **Session共享**       | 分布式部署下共享用户会话          |
| 🔸 **消息队列 / Stream** | 异步任务、日志系统、秒杀队列      |
| 🔸 **地理位置服务**      | GEO 类型计算经纬度距离            |

------

### 🔹 1.1.5 不适用场景

| 场景                 | 原因                             |
| -------------------- | -------------------------------- |
| ⚠️ 数据量超出内存容量 | Redis 以内存为主，扩容成本高     |
| ⚠️ 强一致性场景       | Redis 是最终一致性，可能短暂延迟 |

| ⚠️ 复杂查询           | 不支持多条件、联表查询，非关系型 |

## 1.2 Redis 运行原理与体系结构

### 🔹 1.2.1 Redis 单线程模型

> **一句话**：Redis 的**命令执行**（解析、执行、返回）由**一个主线程**串行完成，没有多线程争用与加锁。
> **为什么单线程还能高并发**：瓶颈在**内存访问和网络 I/O**，不在 CPU；单线程反而避免了上下文切换和锁开销，配合 **I/O 多路复用**（见下）一条线程就能处理大量连接。
> **注意**：Redis 6.0 起**网络 I/O 与部分命令**可多线程，但**命令执行**仍以单线程为主，面试时仍可说「命令单线程串行执行」。

Redis 的核心命令处理由一个主线程完成。

**设计原因：**

1. 单线程避免了多线程上下文切换的开销；
2. 基于内存操作，CPU 并不是瓶颈；
3. 使用 **I/O 多路复用(epoll)** 同时管理多个连接。

📘 **执行模型：**

```
Client 1 ─┐
Client 2 ─┼──▶ 事件队列(EventLoop) ─▶ 命令解析 ─▶ 执行 ─▶ 返回结果
Client 3 ─┘
```

Redis 的主线程处理所有客户端请求，内部通过事件循环不断轮询客户端socket。

------

### 🔹 1.2.2 I/O 多路复用机制

> **I/O 多路复用**：用**一条线程**监听**多个连接**的可读/可写事件，谁有数据谁被唤醒，避免「一个连接一个线程」导致线程数爆炸。Linux 下常用 **epoll**（高效的可读/可写事件通知机制），Redis 主线程通过 epoll 拿到就绪的客户端连接再逐个执行命令。

Redis 使用 **epoll（Linux）/ kqueue（BSD）/ select** 实现高并发 I/O。

- 每个连接都注册到内核事件（epoll）上；
- 当有事件触发时，内核通知 Redis；
- Redis 主线程从事件队列中取出任务，依次执行。

📈 **优点：**

- 高并发连接但非阻塞；
- 避免每个连接创建独立线程；
- 响应延迟低。

🧩 **事件分类：**

| 类型                       | 描述                   |
| -------------------------- | ---------------------- |
| Readable（可读事件）       | 客户端发送命令请求     |
| Writable Event（可写事件） | 返回执行结果给客户端   |
| Timer Event（定时事件）    | 定时任务（过期键检查） |

### 🔹 1.2.3 内存模型与持久化交互

#### （1）内存结构

Redis 的核心结构：

```
┌──────────────────────────——————————──┐
│ Redis Server           		       │
│  ├── Dict（哈希表）字典，存储Key-Value）  │
│  ├── Object（封装Value类型）  		    │
│  ├── SDS（简单动态字符串）       	   	  │
│  └── 内存分配器（jemalloc）    		   │
└─────────────────────────——————————───┘
```

#### （2）持久化机制交互

Redis 提供两种持久化方式：

| 持久化方式          | 描述                   | 优点           | 缺点       |
| ------------------- | ---------------------- | -------------- | ---------- |
| **RDB（快照）**     | 定期保存内存快照到磁盘 | 性能好，占用少 | 可能丢数据 |
| **AOF（追加日志）** | 记录每条写命令日志     | 更安全         | 占用空间大 |
| **混合持久化**      | 结合 RDB + AOF         | 性能与安全平衡 | 略复杂     |

🧠 **运行逻辑：**

```
客户端写入命令
      ↓
更新内存数据
      ↓
记录到 AOF / RDB
      ↓
异步刷盘
```


3. 启动服务：

   ```bash

   ```

4. 打开客户端连接：

   ```bash
   redis-cli
   ```

## 2.1 String（字符串）

### 🔹 2.1.1 基本概念

String 是 Redis 中最基础的数据结构，既可以存储普通文本，也可以存储数字、二进制数据（图片、JSON、序列化对象）。

- 最大存储大小：**512 MB**
- 实际上，所有类型最终都基于 String 实现

### 🔹 2.1.2 底层编码方式

| 编码方式   | 适用场景             | 底层结构            |
| ---------- | -------------------- | ------------------- |
| **int**    | 存储整型值           | long 类型整数       |
| **embstr** | 字符串长度 ≤ 44 字节 | 连续内存块（SDS）   |
| **raw**    | 字符串长度 > 44 字节 | 普通 SDS 动态字符串 |

🧩 **SDS（Simple Dynamic String）结构：**

```c
struct sdshdr {
    int len;      // 实际长度
    int free;     // 剩余空间
    char buf[];   // 存储内容
};
```

> **为什么用 SDS 而不是 C 字符串**：C 字符串取长度为 O(n)、不能含 `\0`、易缓冲区溢出；SDS 用 **len** 存长度 O(1)、支持二进制安全、自动扩容，更适合 Redis 的键值存储。

**面试可答：String 底层编码与选择**

- **三种编码**：能存成整数的用 **int**（如 INCR 的 key）；长度 ≤44 字节用 **embstr**（对象头+SDS 一次分配、缓存友好）；更长用 **raw**（SDS 可扩容）。可以说「Redis 会根据值和操作自动选编码，我们只要避免单 value 过大即可」。
- **List/Hash/ZSet 编码**：小数据用 **listpack**（旧版 ziplist）省内存，超过阈值切 **hashtable/skiplist** 保性能；ZSet 是 **跳表 + 字典** 实现范围与 O(1) 查分。面试追问「为什么 ZSet 用跳表」可答：有序、插入/查询 O(logN)、实现比红黑树简单、范围查询友好。

### 🔹 2.1.3 常用命令

| 命令                   | 功能       | 示例                  |
| ---------------------- | ---------- | --------------------- |
| `SET key value`        | 设置字符串 | `SET name "houziran"` |
| `GET key`              | 获取值     | `GET name`            |
| `INCR key`             | 自增       | `INCR count`          |
| `DECR key`             | 自减       | `DECR stock`          |
| `APPEND key value`     | 追加字符串 | `APPEND msg " world"` |
| `MSET key1 v1 key2 v2` | 批量设置   | `MSET k1 v1 k2 v2`    |
| `GETRANGE key 0 3`     | 获取子串   | `GETRANGE str 0 3`    |

------

### 🔹 2.1.4 应用场景

| 场景             | 描述                                  |
| ---------------- | ------------------------------------- |
| **缓存对象**     | 存储序列化后的 JSON 或字符串          |
| **计数器**       | 利用 `INCR` / `DECR` 实现点赞、浏览量 |
| **分布式锁**     | `SETNX key value PX 30000` 实现互斥锁 |
| **共享 Session** | 存储用户登录信息                      |

📘 **实战示例：分布式锁**

```bash
SET lock:order "uuid123" NX PX 30000   # 上锁，30秒超时
DEL lock:order                         # 解锁
```

**❌ 错误写法 vs ✅ 正确写法（缓存与过期）**

```bash
# ❌ 只 SET 不设过期：key 常驻内存，容易打满；或业务逻辑 Bug 导致 key 永不过期
SET user:1001 "{...json...}"

# ✅ 缓存务必带过期时间，避免内存无限增长
SET user:1001 "{...json...}" EX 3600
# 或 SETEX user:1001 3600 "{...json...}"
```

```bash
# ❌ 大 value：单 key 存几 MB 的 JSON/列表，阻塞主线程、网络传输慢、持久化压力大
SET big:key "非常长的字符串或序列化对象..."

# ✅ 拆成多个 key、或压缩、或用 Hash 按字段存；单 value 控制在 KB 级以内
HSET user:1001 name "张三" age 25 city "北京"
```

## 2.2 List（列表）

### 🔹 2.2.1 概念与特性

Redis 的 List 是一个 **有序可重复的链表结构**，支持从两端插入和弹出。

### 🔹 2.2.2 底层结构

| 版本         | 底层实现                                  | 特点                               |
| ------------ | ----------------------------------------- | ---------------------------------- |
| Redis 3.2 前 | **ziplist（压缩列表）** 或 **linkedlist** | 小数据用 ziplist，大数据用双端链表 |
| Redis 3.2+   | **quicklist**（双端链表 + ziplist）       | 节省内存 + 快速访问                |

🧩 quicklist 结构：

```
quicklist
 ├── node1(ziplist)
 ├── node2(ziplist)
 └── node3(ziplist)
```

------

### 🔹 2.2.3 常用命令

| 命令                | 功能         | 示例               |
| ------------------- | ------------ | ------------------ |
| `LPUSH key value`   | 从左插入     | `LPUSH list 1 2 3` |
| `RPUSH key value`   | 从右插入     | `RPUSH list 4 5`   |
| `LPOP key`          | 弹出左边元素 | `LPOP list`        |
| `RPOP key`          | 弹出右边元素 | `RPOP list`        |
| `LRANGE key 0 -1`   | 获取范围     | `LRANGE list 0 -1` |
| `BLPOP key timeout` | 阻塞式弹出   | `BLPOP queue 0`    |

------

### 🔹 2.2.4 应用场景

| 场景             | 说明                                   |
| ---------------- | -------------------------------------- |
| **消息队列**     | 用 `LPUSH` + `BRPOP` 实现生产者-消费者 |
| **时间线结构**   | 模拟微博关注动态流                     |
| **异步任务处理** | 定时任务、日志收集等                   |

📘 **实战案例：消息队列**

```bash
LPUSH queue task1
BLPOP queue 0
```

> 生产者 `LPUSH`，消费者 `BLPOP` 实现阻塞式消费。

## 2.3 Set（集合）

### 🔹 2.3.1 特点

- 无序、去重；
- 支持集合运算（交集、并集、差集）。

### 🔹 2.3.2 底层结构

| 编码方式      | 场景                   | 实现结构 |
| ------------- | ---------------------- | -------- |
| **intset**    | 所有元素为整数且数量小 | 整数数组 |
| **hashtable** | 元素较多或有字符串     | 哈希表   |

------

### 🔹 2.3.3 常用命令

| 命令                   | 功能         | 示例                  |
| ---------------------- | ------------ | --------------------- |
| `SADD key member`      | 添加元素     | `SADD users 1 2 3`    |
| `SREM key member`      | 移除元素     | `SREM users 2`        |
| `SMEMBERS key`         | 获取全部元素 | `SMEMBERS users`      |
| `SISMEMBER key member` | 判断是否存在 | `SISMEMBER users 3`   |
| `SINTER key1 key2`     | 求交集       | `SINTER u1 u2`        |
| `SUNION key1 key2`     | 求并集       | `SUNION u1 u2`        |
| `SDIFF key1 key2`      | 求差集       | `SDIFF u1 u2`         |
| `SRANDMEMBER key 2`    | 随机抽取     | `SRANDMEMBER users 2` |

------

### 🔹 2.3.4 应用场景

| 场景                    | 描述                             |
| ----------------------- | -------------------------------- |
| **好友列表 / 标签系统** | 用 Set 存储关注、兴趣标签        |
| **共同关注**            | SINTER 求交集                    |
| **推荐系统**            | SDIFF 求差集（“你可能认识的人”） |
| **抽奖系统**            | SRANDMEMBER 随机抽取中奖人       |

📘 **示例：共同好友**

```bash
SADD user:1 friends A B C
SADD user:2 friends B C D
SINTER user:1 user:2    # => B, C
```

------

## 2.4 Hash（哈希表）

### 🔹 2.4.1 特点

- 类似于 Java 的 `Map<String, Object>`
- 常用于缓存对象（用户信息、商品信息等）

------

### 🔹 2.4.2 底层实现

| 编码方式      | 条件         | 底层结构                 |
| ------------- | ------------ | ------------------------ |
| **ziplist**   | 字段少、值短 | 紧凑存储，节省内存       |
| **hashtable** | 字段多或值长 | 哈希表结构，支持快速查找 |

🧩 HashEntry 结构：

```
field1 → value1
field2 → value2
```

------

### 🔹 2.4.3 常用命令

| 命令                            | 功能         | 示例                            |
| ------------------------------- | ------------ | ------------------------------- |
| `HSET key field value`          | 设置字段     | `HSET user:1 name houziran`     |
| `HGET key field`                | 获取字段值   | `HGET user:1 name`              |
| `HMSET key field1 v1 field2 v2` | 批量设置     | `HMSET user:1 age 25 city XiAn` |
| `HGETALL key`                   | 获取所有字段 | `HGETALL user:1`                |
| `HDEL key field`                | 删除字段     | `HDEL user:1 age`               |
| `HINCRBY key field n`           | 字段自增     | `HINCRBY user:1 points 10`      |

------

### 🔹 2.4.4 rehash 机制

当哈希表元素增多时，Redis 会执行 **渐进式 rehash**：

1. 新建一张更大的哈希表；
2. 每次操作（读写）时，迁移部分元素（固定了迁移数量，而延迟加载是访问到了才处理自己）；
3. 避免一次性阻塞主线程。

------

### 🔹 2.4.5 应用场景

| 场景             | 说明                                           |
| ---------------- | ---------------------------------------------- |
| **用户信息缓存** | `HSET user:1 name ziran age 23`                |
| **购物车**       | `HSET cart:uid101 product:1 2`（商品1数量为2） |
| **对象属性维护** | Hash 替代 JSON 拆分存储                        |

📘 **实战示例：购物车**

```bash
HSET cart:1001 10086 2
HSET cart:1001 10087 1
HGETALL cart:1001
```

## 2.5 ZSet（有序集合）

### 🔹 2.5.1 特点

- 元素 **有序且不重复**；
- 每个元素关联一个权重 `score`；
- 自动根据 score 排序。

------

### 🔹 2.5.2 底层结构

| 编码                | 场景                 | 实现              |
| ------------------- | -------------------- | ----------------- |
| **ziplist**         | 元素数量少、score 小 | 紧凑存储          |
| **skiplist + dict** | 元素较多时           | 跳表 + 哈希表组合 |

🧩 **跳表结构示意：**

```
score(1) → userA
score(2) → userB
score(5) → userC
```

跳表是一种多层有序链表结构，查找复杂度为 **O(logN)**。

------

### 🔹 2.5.3 常用命令

| 命令                         | 功能       | 示例                         |
| ---------------------------- | ---------- | ---------------------------- |
| `ZADD key score member`      | 添加元素   | `ZADD rank 100 user1`        |
| `ZRANGE key 0 -1 WITHSCORES` | 按升序获取 | `ZRANGE rank 0 2 WITHSCORES` |
| `ZREVRANGE key 0 2`          | 降序获取   | `ZREVRANGE rank 0 2`         |
| `ZINCRBY key n member`       | 增加分数   | `ZINCRBY rank 10 user1`      |
| `ZRANK key member`           | 获取排名   | `ZRANK rank user1`           |
| `ZREM key member`            | 删除成员   | `ZREM rank user2`            |

------

### 🔹 2.5.4 应用场景

| 场景           | 描述                               |
| -------------- | ---------------------------------- |
| **排行榜系统** | 用户积分或热度排序                 |
| **延时队列**   | score 代表时间戳，定期扫描到期任务 |
| **优先级队列** | 根据权重动态出队                   |

📘 **示例：排行榜**

```bash
ZADD scoreboard 100 userA
ZADD scoreboard 200 userB
ZREVRANGE scoreboard 0 -1 WITHSCORES
```

## 2.6 高级数据结构

------

### 🔸 2.6.1 Bitmap（位图）

> 用于高效存储布尔状态（0/1）。

| 命令                      | 说明        | 示例              |
| ------------------------- | ----------- | ----------------- |
| `SETBIT key offset value` | 设置位      | `SETBIT sign 5 1` |

- 防重复访问。

### 🔸 2.6.2 HyperLogLog（基数统计）

> 用极小空间（12KB）估算集合中 **唯一元素数量（基数）**。

| 命令                     | 功能     | 示例                    |
| ------------------------ | -------- | ----------------------- |
| `PFADD key element`      | 添加元素 | `PFADD uv user1 user2`  |
| `PFCOUNT key`            | 统计数量 | `PFCOUNT uv`            |
| `PFMERGE dest key1 key2` | 合并统计 | `PFMERGE total uv1 uv2` |

📘 **应用场景：**

- 网站 UV 统计；
- 独立访客计数；
- 海量去重。

------

### 🔸 2.6.3 GEO（地理位置）

> 存储经纬度坐标，实现距离计算与范围查询。

| 命令                                   | 功能         | 示例                               |
| -------------------------------------- | ------------ | ---------------------------------- |
| `GEOADD key longitude latitude member` | 添加地理位置 | `GEOADD city 108.95 34.27 XiAn`    |
| `GEODIST key m1 m2 km`                 | 计算距离     | `GEODIST city XiAn Beijing km`     |
| `GEORADIUS key lon lat radius km`      | 查找范围内点 | `GEORADIUS city 108.9 34.2 100 km` |

📘 **应用场景：**

- 附近商家 / 附近的人；
- 距离排序推荐。

------

### 🔸 2.6.4 Stream（消息流）

> Redis 5.0 引入的新型消息队列结构。
>  相比 List，支持 **多消费者组、消息确认、ID排序**。

| 概念           | 说明           |
| -------------- | -------------- |
| Stream         | 类似消息队列   |
| Consumer Group | 消费者组       |
| Pending List   | 未确认消息列表 |
| Offset         | 消息消费位置   |

📘 **常用命令**

| 命令                                           | 说明         | 示例                    |
| ---------------------------------------------- | ------------ | ----------------------- |
| `XADD key * field value`                       | 添加消息     | `XADD mq * orderId 101` |
| `XREAD COUNT 1 STREAMS key 0`                  | 读取消息     | `XREAD STREAMS mq 0`    |
| `XGROUP CREATE key group 0`                    | 创建消费者组 | `XGROUP CREATE mq g1 0` |
| `XREADGROUP GROUP g1 c1 COUNT 1 STREAMS key >` | 消费组读取   |                         |
| `XACK key group id`                            | 确认消息     | `XACK mq g1 12345`      |

📘 **应用场景：**

- 异步任务队列；
- 订单回调；
- 消息持久化日志系统。

### 🔹 2.7 最小可运行示例（从零到一次读写）

在**已安装并启动 Redis** 的前提下（见第一章），用 **redis-cli** 或 **Java** 跑通「连接 → 写 → 读」即可验证环境与基本命令。

**方式一：redis-cli（最快）**

```bash
# 1. 连接本地 Redis（默认 6379）
redis-cli

# 2. 写入：SET key value，EX 表示过期秒数
127.0.0.1:6379> SET mykey "hello" EX 60
OK

# 3. 读取：GET key
127.0.0.1:6379> GET mykey
"hello"

# 4. 60 秒后再 GET 会得到 (nil)，因为已过期
```

**方式二：Java（Spring Boot + Lettuce/Jedis）**

```java
// 1. 依赖：spring-boot-starter-data-redis
// 2. 配置：spring.redis.host=localhost, port=6379
// 3. 注入 StringRedisTemplate 或 RedisTemplate

@Autowired
private StringRedisTemplate redisTemplate;

public void example() {
    // 写入：key, value, 过期时间, 单位
    redisTemplate.opsForValue().set("mykey", "hello", 60, TimeUnit.SECONDS);
    // 读取
    String value = redisTemplate.opsForValue().get("mykey");  // "hello"
}
```

> 🔹 **入参 → 执行 → 出参**：SET 入参为 key、value、过期（可选）；执行后 Redis 内存中有一份；GET 入参为 key，出参为 value 或 null（不存在/已过期）。后续 List/Hash/ZSet 等都用对应的 `opsForList()`、`opsForHash()` 等，思路一致。

### 常见坑与注意点

| 现象 / 易错点 | 原因 | 怎么改 / 怎么记 |
|---------------|------|-----------------|
| 缓存 key 不设过期导致内存打满 | 只 SET 不 EX/SETEX，key 常驻 | 缓存类 key **务必设过期时间**（EX 或 SETEX），并配合 maxmemory-policy。 |
| 单 key value 过大（大 key） | 存几 MB 的 JSON 或大 List，阻塞主线程、持久化慢 | 拆 key、用 Hash 按字段存、或压缩；单 value 控制在 KB 级。 |

| 排行榜用错结构 | 用 List 无法按分数排序、更新麻烦 | **有序**且按**分数**排用 **ZSet**（ZADD/ZRANGEBYSCORE）；List 适合队列、时间线。 |

# 💾 第3章：Redis 持久化机制（Persistence Mechanism）

------

> Redis 虽然是**内存数据库**，但为了防止断电或重启后数据丢失，提供了两种核心持久化方案：
>
> - **RDB**：周期性快照（全量备份）
> - **AOF**：追加日志（增量备份）
>    同时还支持 **混合持久化模式**。

------

## 3.1 RDB（快照）持久化

------

### 🔹 3.1.1 RDB 原理

RDB（Redis DataBase）通过在某个时间点将内存中所有数据保存为一个二进制快照文件（默认名：`dump.rdb`），实现持久化。

📘 **核心思想：**

> 在特定时刻生成**全量快照**，写入磁盘。

------

### 🔹 3.1.2 RDB 文件生成机制

Redis 生成 RDB 有两种方式：

| 方式            | 命令     | 说明                                      |
| --------------- | -------- | ----------------------------------------- |
| **同步 save**   | `SAVE`   | 阻塞主线程，立即生成快照                  |
| **异步 bgsave** | `BGSAVE` | fork 子进程生成快照，不阻塞主线程（推荐） |

#### 📖 执行流程（bgsave）

1. 主进程调用 `fork()` 创建子进程；
2. 子进程将内存数据写入临时文件；
3. 写入完成后替换原有 `dump.rdb`；
4. 主线程继续服务客户端请求。

🧠 **Copy-On-Write（COW）机制：**

> bgsave 使用操作系统的写时复制技术：
>
> - fork 时父子进程共享内存页；
> - 当父进程修改数据时，才复制对应内存页。
>    这样既保证快照一致，又减少性能损耗。

------

### 🔹 3.1.3 RDB 触发机制（自动快照）

在 `redis.conf` 中配置：

```bash
save 900 1      # 900秒内有1个key被修改时触发
save 300 10     # 300秒内有10个key被修改时触发
save 60 10000   # 60秒内有10000个key被修改时触发
```

还可通过命令手动触发：

```bash
SAVE
BGSAVE
```

📘 Redis 还会在以下情况下触发 RDB：

- 执行 `FLUSHALL` 且开启 `rdbcompression`；
- 主从复制（slave 启动同步时）；
- 执行 `shutdown` 时（默认）。

------

### 🔹 3.1.4 RDB 文件结构

| 段        | 内容               |
| --------- | ------------------ |
| Header    | 文件版本与标识     |
| Databases | 各数据库的数据内容 |
| EOF       | 文件结束符         |
| Checksum  | 校验值             |

------

### 🔹 3.1.5 RDB 优缺点

| 优点                 | 缺点                        |
| -------------------- | --------------------------- |
| 文件体积小，恢复快   | 会丢失最近一次快照后的数据  |
| 性能高（子进程生成） | fork 过程中可能造成短暂阻塞 |
| 适合定时备份         | 不适合高频写操作            |

✅ **适合场景：**

> - 定期备份（如每日凌晨3点）
> - 灾备恢复（RDB 文件可跨服务器复制）

------

## 3.2 AOF（追加日志）持久化

------

### 🔹 3.2.1 AOF 原理

AOF（Append Only File）通过记录 Redis 执行的所有写命令日志，实现可重放的增量持久化。

📘 **思想：**

> 不保存数据，而是**记录每次写操作命令**，重启时重放日志恢复状态。

例如：

```
SET name houziran
INCR visit
HSET user:1 age 23
```

------

### 🔹 3.2.2 AOF 写入流程

Redis 内部维护一个 **AOF 缓冲区**：

```
客户端写命令 → AOF缓冲区 → AOF文件 → 磁盘刷盘
```

刷盘策略由配置项 `appendfsync` 决定：

| 配置       | 含义             | 性能 | 安全性       |
| ---------- | ---------------- | ---- | ------------ |
| `always`   | 每次写入都同步   | 慢   | 最安全       |
| `everysec` | 每秒刷盘（默认） | 折中 | 安全性较好   |
| `no`       | 由操作系统决定   | 快   | 可能丢失数据 |

示例配置：

```bash
appendonly yes
appendfilename "appendonly.aof"
appendfsync everysec
```

------

### 🔹 3.2.3 AOF 文件重写机制（Rewrite）

> 随着时间推移，AOF 文件会越来越大，为避免过度膨胀，Redis 提供 **AOF Rewrite** 压缩机制。

📘 **原理：**

- 不影响当前写入；
- 由后台子进程执行；
- 将旧日志重写为最小化命令集。

#### 🔁 流程：

1. fork 子进程；
2. 扫描内存数据；
3. 生成最简命令（只保留最新状态）；
4. 写入临时文件；
5. 完成后替换旧 AOF。

#### 自动触发条件：

```bash
auto-aof-rewrite-percentage 100   # 文件大小增长一倍触发重写
auto-aof-rewrite-min-size 64mb    # 文件最小64MB才触发
```

------

### 🔹 3.2.4 AOF 文件恢复流程

Redis 启动时：

1. 检查 AOF 文件是否存在；
2. 顺序加载 AOF 命令；
3. 逐条执行重放；
4. 恢复数据库数据。

命令：

```bash
redis-check-aof --fix appendonly.aof
```

------

### 🔹 3.2.5 AOF 与 RDB 混合持久化（Hybrid Persistence）

Redis 4.0 起支持 **RDB + AOF 混合模式**：

📘 **机制：**

- 重写时，将当前内存数据写入 **RDB 格式**；
- 后续增量操作以 **AOF 追加**；
- 文件名仍为 `appendonly.aof`。

✅ **优点：**

- 加载速度快（基于RDB）
- 恢复精度高（增量AOF）

配置：

```bash
aof-use-rdb-preamble yes
```

------

### 🔹 3.2.6 AOF 优缺点

| 优点                       | 缺点                   |
| -------------------------- | ---------------------- |
| 数据更安全，可控制丢失范围 | 文件体积大，恢复速度慢 |
| 实时性高                   | 写放大效应，I/O 压力大 |
| 可配合 rewrite 减小文件    | 重写过程耗 CPU         |

✅ **适合场景：**

> 对数据安全性要求高的系统（如金融、订单）。

------

## 3.3 持久化策略选择与实战配置

------

### 🔹 3.3.1 常见持久化策略对比

| 模式           | 持久性                   | 性能 | 恢复速度 | 是否推荐 |
| -------------- | ------------------------ | ---- | -------- | -------- |
| **仅RDB**      | 定期快照，可能丢秒级数据 | 高   | 快       | ✔️ 常用   |
| **仅AOF**      | 实时追加，丢失极少数据   | 中   | 慢       | ✔️ 安全   |
| **混合模式**   | RDB全量 + AOF增量        | 中高 | 快       | ✅ 推荐   |
| **关闭持久化** | 全在内存中               | 最高 | 无       | ❌ 危险   |

------

### 🔹 3.3.2 实战配置模板（推荐）

📘 **redis.conf 示例：**

```bash
# 开启RDB
save 900 1
save 300 10
save 60 10000

# 开启AOF
appendonly yes
appendfilename "appendonly.aof"
appendfsync everysec
aof-use-rdb-preamble yes

# 内存与文件路径
dir /var/lib/redis/
dbfilename dump.rdb
```

------

### 🔹 3.3.3 实战：双保险模式配置

目标：数据安全优先，性能折中。

```bash
# 开启RDB定期快照
save 300 1
save 60 10000

# 开启AOF每秒刷盘
appendonly yes
appendfsync everysec
auto-aof-rewrite-percentage 100
auto-aof-rewrite-min-size 64mb
aof-use-rdb-preamble yes
```

📘 **说明：**

- RDB：周期备份，防灾；
- AOF：实时日志，数据不丢；
- 混合模式：平衡性能与可靠性。

------

### 🔹 3.3.4 实战建议

| 场景           | 推荐策略                       |
| -------------- | ------------------------------ |
| 开发测试       | 仅使用 RDB                     |
| 普通生产环境   | RDB + 每秒 AOF                 |
| 高安全金融场景 | AOF everysec + 异地RDB备份     |
| 高性能缓存场景 | 关闭持久化（如全内存临时缓存） |

| 模块         | 核心内容                            |
| ------------ | ----------------------------------- |
| **RDB**      | 定期快照，全量保存，适合备份与恢复  |
| **AOF**      | 记录操作日志，数据最安全            |
| **混合模式** | RDB 快照 + AOF 增量，性能与安全兼顾 |
| **配置建议** | RDB + everysec + rewrite，最佳实践  |

# 🧠 第4章：Redis 内存管理与淘汰策略

> Redis 的性能与稳定性，很大程度取决于**内存如何被分配、编码、回收与淘汰**。
>  本章围绕四件事：**对象如何存（编码）→ 内存如何拿（分配器）→ 何时回收（过期/删除/碎片整理）→ 不够怎么办（淘汰策略）。**

------

## 4.1 内存分配与对象编码

### 4.1.1 内存分配器

- **jemalloc**（默认）：性能优、碎片率低，按 size-class 管理小对象，减少锁竞争与系统调用次数。
- glibc/tcmalloc：可编译时切换，但生产中几乎都用 jemalloc。

**关键指标：**

- **used_memory**：已分配给 Redis 的内存（不等于 key 占用总和）。
- **allocator_frag_ratio**：分配器碎片率（~1.00 理想，>1.5 说明碎片明显）。
- **mem_fragmentation_ratio**：Redis 层总体碎片率（含页缓存等）。

```bash
INFO memory
MEMORY STATS
```

------

### 4.1.2 RedisObject 与编码（encoding）

Redis 的“值”并非直接原始字节，而是一个 **RedisObject**（类型 + 编码 + 指针）。不同类型在不同场景下会切换不同**编码**以兼顾性能与空间。

#### String 编码

| 编码       | 触发条件                    | 说明                                      |
| ---------- | --------------------------- | ----------------------------------------- |
| **int**    | 可解析为 64-bit 整数        | 直接存 long long                          |
| **embstr** | 长度 ≤ ~44B（版本略有差异） | 对象头 + SDS 一次分配，连续内存；读多写少 |
| **raw**    | 长度 > ~44B                 | 普通 SDS，支持扩容                        |

> **SDS**（Simple Dynamic String）：记录 `len/free/buf[]`，避免 `strlen` O(n) 开销，支持预分配与惰性缩容。

#### List 编码

| 版本       | 底层                                                 |
| ---------- | ---------------------------------------------------- |
| Redis 3.2+ | **quicklist**（双端链表 + listpack/ziplist 节点）    |
| Redis 7.x  | **quicklist + listpack**（ziplist 被 listpack 取代） |

- quicklist 将多段短列表（listpack）串起来，平衡**随机访问/内存占用**。

#### Hash / ZSet 编码

| 类型     | 小数据编码               | 大数据编码                  |
| -------- | ------------------------ | --------------------------- |
| **Hash** | listpack（旧称 ziplist） | hashtable                   |
| **ZSet** | listpack                 | **skiplist（跳表） + dict** |

**编码切换阈值（示例配置项）**

> 注：不同版本项名略有差异，新版以 *listpack* 结尾

```bash
# 小对象压缩阈值（满足“字段个数 ≤ entries 且 单字段长度 ≤ value”）
hash-max-listpack-entries  512
hash-max-listpack-value    64
zset-max-listpack-entries  128
zset-max-listpack-value    64
list-max-listpack-size     -2   # 单个 listpack 目标大小（负数代表按字节目标）
```

**设计要点：**

- “小而多”适合用 listpack（连续内存，节省指针开销）；
- “大而复杂”切到 hashtable/skiplist，保证 O(1)/O(logN) 查找。

------

## 4.2 内存回收机制

### 4.2.1 过期键删除（Expiration）

Redis 维护**过期字典**（expire dict），过期策略混合：

- **惰性删除（lazy）**：访问 key 时发现过期才删除（命中才清理；低开销）。
- **定期删除（active）**：后台周期采样（`serverCron`），按比例清理过期 key，防止堆积。

**相关命令/行为**

```bash
EXPIRE key seconds
PEXPIRE key milliseconds
TTL key
PERSIST key      # 取消过期
```

> 过期 ≠ 淘汰：过期是“到了寿命”，淘汰是“内存不够被赶走”。

------

### 4.2.2 同步/异步删除（lazyfree）

删除大 key（如超大 Hash/List）可能阻塞主线程，建议**异步删除**：

```bash
UNLINK key1 key2      # 异步释放内存（推荐）
DEL key               # 同步删除（阻塞风险）
FLUSHDB ASYNC         # 异步清库
FLUSHALL ASYNC        # 异步清空实例
```

**lazyfree 配置**

```bash
lazyfree-lazy-eviction yes    # 淘汰时异步释放
lazyfree-lazy-expire   yes    # 过期时异步释放
lazyfree-lazy-server-del yes  # 内部删除异步
```

------

### 4.2.3 碎片整理（Active Defrag）

长时间运行 + 频繁更新会产生碎片，Redis 支持**在线主动碎片整理**（不停止服务）：

```bash
active-defrag yes
active-defrag-ignore-bytes 100mb
active-defrag-threshold-lower 10
active-defrag-threshold-upper 100
```

- 通过后台线程搬迁小对象，降低 `mem_fragmentation_ratio`。
- 适合长生命周期、更新频繁的实例（如会话、计数类业务）。

------

## 4.3 内存淘汰策略（Eviction Policies）

当 `used_memory` 超过 `maxmemory` 时，触发**淘汰**：按策略选择 key，将其踢出内存（或直接报错）。

### 4.3.1 关键配置

```bash
maxmemory 4gb                  # 上限
maxmemory-policy allkeys-lfu   # 淘汰策略
maxmemory-samples 5            # 采样个数（影响 LRU/LFU 准确性/开销）
```

### 4.3.2 淘汰策略总览

| 策略                | 说明                               | 适用              |
| ------------------- | ---------------------------------- | ----------------- |
| **noeviction**      | 不淘汰；写命令直接返回错误         | 强一致但风险大    |
| **allkeys-lru**     | 在所有 key 中选最近最少使用的淘汰  | **通用首选**缓存  |
| **volatile-lru**    | 仅在有过期时间的 key 中按 LRU 淘汰 | 仅缓存带 TTL 的键 |
| **allkeys-random**  | 在所有 key 随机淘汰                | 简单但不稳定      |
| **volatile-random** | 在带 TTL 的 key 中随机淘汰         | 稀有场景          |
| **volatile-ttl**    | 优先淘汰“TTL 更近”的 key           | 强时效业务        |
| **allkeys-lfu**     | 在所有 key 中按 **LFU** 淘汰       | **热门命中稳定**  |
| **volatile-lfu**    | 在带 TTL 的 key 中按 LFU 淘汰      | 同上              |

> **LRU vs LFU**：
>
> - LRU：最近使用过就更难被淘汰；
> - LFU：看“使用频次”，更抗“短期热点”。

### 4.3.3 LFU 调优参数

```bash
lfu-log-factor 10     # 调整计数增长速度，越小增长越快（默认 10）
lfu-decay-time  1     # 衰减周期（分钟），越小越易“降温”
```

- 高频读写：增大 `lfu-decay-time`，让热 key 保持热度更久。
- 瞬时热点：减小 `lfu-log-factor`，让新热点更快上涨。

### 4.3.4 LRU/LFU 的“采样”机制

- 为降低开销，Redis 不会全量扫描，而是**随机采样 `maxmemory-samples` 个 key**，在样本中找“最差”的淘汰。
- `maxmemory-samples` 越大，越接近“真 LRU/LFU”，但 CPU 开销也更高（默认 5~10 较平衡）。

------

## 4.4 内存优化技巧与监控命令

### 4.4.1 Key/Value 设计与压缩

**1）减少键值元数据开销**

- **合并小键**：大量小 key → 用 **Hash** 打包（对象拆字段）
  - 例：把 10 万个 `user:uid:field`，合并为 `hash user:uid {field:value,...}`。
  - 结合 `hash-max-listpack-*`，大量节省指针与对象头内存。

**2）选对结构与编码**

- 计数/布尔：考虑 **Bitmap/Bitfield**；
- 去重 UV：**HyperLogLog**（12KB 固定）；
- 排行：**ZSet**；定时任务：ZSet（score=时间戳）。

**3）控制字符串大小**

- String 超过 ~44B 会从 embstr → raw，增加对象分配/指针开销；
- 能整型就整型（`SET key 123` 会自动用 **int 编码**）。

**4）避免“大 key”**

- 单个 value 过大（MB 级）删除阻塞、迁移困难；
- 将大列表切分分片：`list:order:202501:000~999`；
- 对超大 Hash 值，考虑拆成多 Hash。

------

### 4.4.2 过期与内存配合

- 给缓存**必须加 TTL**，避免“永不淘汰”的僵尸数据；
- 组合策略：**allkeys-lfu + TTL**，“热的常驻，冷的自然淘汰”。

------

### 4.4.3 异步删除与在线碎片整理

- 对可能阻塞的操作用 `UNLINK / ASYNC`；
- 长期在线服务开启 `active-defrag`，定期观察 `mem_fragmentation_ratio`。

------

### 4.4.4 监控与排查命令

**容量与碎片：**

```bash
INFO memory
MEMORY STATS
MEMORY DOCTOR              # 问诊建议（英文报告）
```

**找大/热 key：**

```bash
redis-cli --bigkeys        # 采样大键分布（离线、对生产慎用）
redis-cli --memkeys        # 按内存排序（新版本）
redis-cli --hotkeys        # 需要开启热点采样：CONFIG SET hotkeys-tracking yes
MEMORY USAGE key           # 单键内存占用估算
```

**配置核对：**

```bash
CONFIG GET maxmemory
CONFIG GET maxmemory-policy
CONFIG GET *lfu*
CONFIG GET *listpack*
```

------

### 4.4.5 典型配置模板（生产可参考）

```bash
# 1) 内存上限与淘汰策略
maxmemory 8gb
maxmemory-policy allkeys-lfu
maxmemory-samples 7

# 2) 小对象压缩阈值（视业务微调）
hash-max-listpack-entries 512
hash-max-listpack-value   96
zset-max-listpack-entries 256
zset-max-listpack-value   96
list-max-listpack-size    -2

# 3) 异步删除 + 碎片整理
lazyfree-lazy-eviction yes
lazyfree-lazy-expire   yes
lazyfree-lazy-server-del yes

# 4) LFU 调优（瞬时热点较多，快速响应）
lfu-log-factor 8
lfu-decay-time  1
```

------

### 4.4.6 常见坑位与规避

1. **只设置了 maxmemory，没设策略 → 写挂**

- 默认 `noeviction`，触发上限后写命令报错。务必设置 `maxmemory-policy`。

1. **只用 LRU，热点“刚冷就被踢”**

- 高频访问但间隔较长，可能被 LRU 误淘汰；换 **LFU** 更稳。

1. **大量小 key，内存飙升**

- 元数据（对象头、指针）占比极高；合并为 Hash，启用 listpack。

1. **删除大 key 卡顿**

- 使用 `UNLINK` / `FLUSHDB ASYNC`；配合 lazyfree 选项。

1. **碎片率高**

- 开启 `active-defrag`；并关注 jemalloc 版本；尽量减少“频繁扩缩容”的模式（如不断 `APPEND/SETRANGE` 超长字符串）。

1. **过期风暴**

- 大量 key 同一 TTL → 同时过期造成抖动；随机化 TTL、分散失效时间。

## 5.1 发布订阅（Pub/Sub）机制

------

### 🔹 5.1.1 概念与作用

> **发布订阅（Publish/Subscribe）** 是一种消息通信模式，允许消息发布者（Publisher）和订阅者（Subscriber）之间**解耦**。

📘 **核心思想：**

- 发布者向频道（channel）发送消息；
- 订阅者只需订阅感兴趣的频道；
- Redis 自动分发消息。

------

### 🔹 5.1.2 工作原理

Redis 在服务器端维护一个 **channel → subscribers** 的映射表：

```
"news"   → [clientA, clientB]
"order"  → [clientC]
```

发布消息时，Redis 查表推送到所有订阅者。

![Pub/Sub流程图](https://redis.io/images/redis-pubsub.png)

------

### 🔹 5.1.3 核心命令

| 命令                      | 功能     | 示例                   |
| ------------------------- | -------- | ---------------------- |
| `SUBSCRIBE channel`       | 订阅频道 | `SUBSCRIBE news`       |
| `PUBLISH channel message` | 发布消息 | `PUBLISH news "hello"` |
| `PSUBSCRIBE pattern`      | 模糊订阅 | `PSUBSCRIBE news*`     |
| `UNSUBSCRIBE channel`     | 取消订阅 | `UNSUBSCRIBE news`     |

📘 **示例：**

```bash
# 终端1（订阅者）
SUBSCRIBE order_channel

# 终端2（发布者）
PUBLISH order_channel "New order created: #1001"
```

📤 输出：

```
message
"order_channel"
"New order created: #1001"
```

------

### 🔹 5.1.4 特点与限制

| 优点           | 缺点                           |
| -------------- | ------------------------------ |
| 实时性强       | 不具备消息持久化               |
| 简单高效       | 消息不可靠（订阅者离线即丢失） |
| 适合小规模广播 | 不支持确认机制、ACK、重试      |

📌 **推荐使用：**

- 实时通知（日志、报警）
- 聊天室 / 推送系统原型
   📌 **不推荐：**
- 关键业务队列（请用 Stream / MQ）

------

## 5.2 事务（Transaction）与 Lua 脚本

------

### 🔹 5.2.1 Redis 事务概念

Redis 事务通过 **MULTI / EXEC / DISCARD / WATCH** 命令实现一组命令的原子执行。

📘 **事务特性：**

- 批量执行；
- **不支持回滚**；
- 出错时只中断当前命令，其他命令仍执行。

------

### 🔹 5.2.2 基本命令

```bash
MULTI               # 开启事务
SET key1 "A"
SET key2 "B"
INCR counter
EXEC                # 提交事务
DISCARD             # 放弃事务
```

📤 输出：

```
QUEUED
QUEUED
QUEUED
OK
```

------

### 🔹 5.2.3 乐观锁机制（WATCH）

WATCH 实现**CAS（Compare-And-Set）机制**，监控 key 是否被修改。

```bash
WATCH balance
val = GET balance
if val > 50:
    MULTI
    DECRBY balance 50
    EXEC
else:
    DISCARD
```

如果事务执行前 `balance` 被其他客户端修改，则 EXEC 返回 null，事务失败。

------

### 🔹 5.2.4 Lua 脚本（Redis Script）

> Redis 提供 **EVAL** 命令支持在服务端执行 Lua 脚本，实现原子操作。

**优势：**

- 避免多次网络往返；
- 多命令原子执行；
- 支持复杂逻辑（条件判断、循环）。

📘 **基本语法：**

```bash
EVAL "return redis.call('set', KEYS[1], ARGV[1])" 1 user ziran
```

- `KEYS[1]`：Redis key
- `ARGV[1]`：外部参数

📘 **示例：原子加减**

```bash
EVAL "
    local val = redis.call('GET', KEYS[1])
    if not val then
        redis.call('SET', KEYS[1], ARGV[1])
    else
        redis.call('INCRBY', KEYS[1], ARGV[1])
    end
" 1 count 10
```

------

### 🔹 5.2.5 脚本管理命令

| 命令                       | 功能                          |
| -------------------------- | ----------------------------- |
| `SCRIPT LOAD script`       | 将脚本加载进缓存（返回 SHA1） |
| `EVALSHA sha1 numkeys ...` | 使用脚本哈希执行              |
| `SCRIPT EXISTS sha1`       | 检查脚本是否存在              |
| `SCRIPT FLUSH`             | 清空缓存脚本                  |

✅ **实践建议：**
 频繁调用的脚本用 `EVALSHA` 提升性能，避免重复解析。

------

### 🔹 5.2.6 项目实战：库存扣减脚本

> 在秒杀/优惠券抢购中，需原子判断库存并扣减。

```lua
-- stock_decr.lua
local stock = tonumber(redis.call('get', KEYS[1]))
if stock > 0 then
  redis.call('decr', KEYS[1])
  return 1
else
  return 0
end
```

调用：

```bash
EVALSHA "sha1_of_script" 1 stock:1001
```

✅ 确保扣减逻辑在 Redis 内部原子完成，无需事务。

### 事务与 Lua：错误 vs 正确示例

| 场景 | 错误写法 / 现象 | 正确写法 / 做法 |
|------|-----------------|------------------|
| **事务中某条命令报错** | 以为 EXEC 会整体回滚；Redis 事务**不支持回滚**，出错那条不执行，其余照常执行 | 业务侧校验参数，或**用 Lua 脚本**把多步写在一起，脚本要么全成功要么全不执行；关键逻辑优先 Lua 保证原子。 |
| **WATCH 后忘记重试** | WATCH key → MULTI…EXEC 被别的客户端改 key 时 EXEC 返回 nil，直接当失败结束 | EXEC 返回 nil 时**重试**整段逻辑（重新 GET、判断、MULTI…EXEC），或改用 **Redisson 等现成锁** 简化。 |
| **Lua 里写多 key 未考虑 Cluster** | Cluster 下多 key 必须在同一槽，否则 EVAL 报错 CROSSSLOT | 多 key 用 **同一 hash tag**，如 `EVAL "..." 2 user:{uid} order:{uid} ...`，保证落在同槽；或单 key 操作。 |
| **Lua 脚本执行过久** | 脚本里循环或大范围操作，阻塞主线程，导致所有请求排队 | Lua 里避免大循环、大 key 遍历；复杂逻辑拆成多步或放应用层；可配置 **lua-time-limit** 防脚本卡死。 |
| **库存扣减用 MULTI 多条命令** | MULTI + GET + 判断 + DECR + EXEC，高并发下 GET 与 DECR 之间被其他请求插入，超卖 | **用 Lua**：GET 判断库存 + DECR 在同一个脚本里原子执行；或 **Redisson 分布式锁** 包住「查+扣」再配合 Lua/单命令。 |

------

## 5.3 Pipeline 管道机制

------

### 🔹 5.3.1 背景与原理

> 每次 Redis 命令执行都要经历：**请求 → 执行 → 响应**，存在网络往返（RTT）延迟。
>  Pipeline 将多条命令**批量发送**，减少 RTT。

📘 **区别：**

- Pipeline ≠ 事务；
- 只是客户端批量发送命令；
- 服务器仍顺序执行。

------

### 🔹 5.3.2 示例

```bash
# 常规
SET k1 v1
SET k2 v2
SET k3 v3

# Pipeline 批量发送
(redis-cli --pipe)
SET k1 v1
SET k2 v2
SET k3 v3
```

Java 实例（Jedis）：

```java
Pipeline p = jedis.pipelined();
p.set("k1", "v1");
p.set("k2", "v2");
p.incr("count");
p.sync();
```

✅ 减少网络延迟、显著提升批量写性能。

📉 **适用场景：**

- 大批量初始化；
- 批量导入/更新；
- 日志收集写入。

------

## 5.4 延迟队列实现

------

### 🔹 5.4.1 背景

延迟队列用于在**指定时间后**执行任务，例如：

- 订单超时取消；
- 支付超时提醒；
- 定时消息推送。

------

### 🔹 5.4.2 基于 ZSet 实现

> ZSet 的 score 可作为“时间戳”，实现延迟执行。

📘 **基本逻辑：**

```bash
ZADD delay_queue 1690000000 order:1001
```

后台线程定期轮询：

```bash
local now = os.time()
local tasks = redis.call('zrangebyscore', 'delay_queue', 0, now)
for i, task in ipairs(tasks) do
    redis.call('zrem', 'delay_queue', task)
    -- 执行任务（下发通知/取消订单）
end
```

**时间复杂度：**

- 插入：O(logN)
- 取出：O(logN + M)

------

### 🔹 5.4.3 优化方案

- 使用 Lua 脚本封装“取出 + 删除”原子操作；
- 利用 Stream 代替 ZSet，支持多消费者组；
- 长延时场景结合定时任务轮询。

------

## 5.5 分布式锁与 Redisson 实战

------

### 🔹 5.5.1 分布式锁需求背景

在分布式系统中，多个节点可能同时修改同一资源，需要一种跨进程的**互斥机制**。
 Redis 提供了轻量级实现。

------

### 🔹 5.5.2 简单实现：`SETNX + EXPIRE`

```bash
SET lock:order 1 NX EX 30
# 等价于：
SETNX lock:order 1       # 如果不存在才设置
EXPIRE lock:order 30     # 设置过期时间防死锁
```

📘 **问题：**

1. 两步非原子（可能在 `SETNX` 与 `EXPIRE` 之间崩溃）；
2. 解锁时必须确保**只释放自己的锁**（防误删）。

------

### 🔹 5.5.3 原子方案：`SET key value NX EX`

Redis 2.6.12+ 支持原子语法：

```bash
SET lock:order uuid NX PX 30000
```

含义：

- NX：仅当 key 不存在时设置；
- PX 30000：过期时间 30s；
- 保证原子性与自动过期。

✅ **加锁成功返回 OK，否则返回 nil。**

------

### 🔹 5.5.4 安全解锁（Lua 原子脚本）

> 只有持有锁的线程才能释放。

```lua
-- unlock.lua
if redis.call("get", KEYS[1]) == ARGV[1]
then
    return redis.call("del", KEYS[1])
else
    return 0
end
```

调用：

```bash
EVAL "if redis.call('get', KEYS[1]) == ARGV[1] then return redis.call('del', KEYS[1]) else return 0 end" 1 lock:order uuid
```

✅ 保证解锁的**原子性 + 所有权验证**。

------

### 🔹 5.5.5 Redisson 分布式锁原理

Redisson 是 Redis 官方推荐的 Java 客户端封装库，提供线程安全的分布式锁实现。

📘 **原理：**

1. 加锁：`SET lock value NX PX time`
2. 看门狗机制（WatchDog）：
   - 默认 30s 续期；
   - 线程未释放锁前自动续租；
3. 解锁：执行 Lua 脚本验证 UUID 后删除；
4. 支持可重入锁、公平锁、读写锁等。

📘 **示例代码：**

```java
RLock lock = redisson.getLock("orderLock");
try {
    lock.lock();  // 阻塞式加锁，自动续期
    // 业务逻辑
} finally {
    lock.unlock();
}
```

📘 **底层核心：**

- 加锁：SET NX + EX；
- 续期：Lua + 定时任务；
- 解锁：Lua 验证 UUID。

------

### 🔹 5.5.6 RedLock（多节点分布式锁）

> 为避免单节点宕机导致锁丢失，Redis 作者提出 **RedLock 算法**。

**步骤：**

1. 在 N 个独立节点上申请锁（建议 N=5）；
2. 获取到超过半数（N/2 + 1）则视为加锁成功；
3. 加锁与解锁使用相同唯一标识；
4. 避免单点故障。

📘 **适合多机 Redis 集群场景，但实现较复杂，一般由 Redisson 内部封装。**

# 🛡️ 第6章：Redis 高可用与集群架构

> 高可用的三层阶梯：**主从复制（Replication） → 哨兵（Sentinel） → 集群（Cluster）**
>  目标：在不同规模与一致性诉求下，选取合适方案，做到**不丢数据/少丢数据、尽快恢复、水平扩展**。

------

## 6.1 主从复制（Replication）

### 6.1.1 基本概念

- **主节点（master）** 负责读/写；**从节点（replica）** 复制主节点数据（默认只读）。
- 复制是**异步**的（可选“半同步”式约束），存在主从延迟。
- 常用于：**读写分离**、**热备**、**快速恢复**。

------

### 6.1.2 复制流程与偏移量机制（PSYNC）

Redis 2.8+ 引入 **PSYNC**（部分重同步）与 **复制偏移量**：

**关键要素：**

- **replid（复制ID）**：主节点生成的全局复制流ID；故障转移后新的主机会生成新 replid，但会保留上一个 replid 以便部分重同步。
- **offset（偏移量）**：主从维护各自的复制进度（字节偏移）。
- **backlog（复制积压缓冲）**：主节点的环形缓冲区，用于短线重连后的**增量**数据补发。

**从节点启动复制流程：**

1. 向主节点发起 `PSYNC replid offset`；
2. **能命中 backlog** → **部分重同步**（仅补缺失字节）；
3. **命不中** 或第一次同步 → **全量复制**（RDB 快照 + 期间写命令的增量缓冲）。

------

### 6.1.3 全量复制 vs 增量复制

**全量复制（第一次或条件不满足）**

1. 主节点 fork 子进程生成 **RDB**（可配置无盘复制）；
2. 将 RDB 发送给从节点，从节点加载；
3. 期间主节点将写命令写入**复制积压缓冲**；
4. RDB 传输结束后，再把积压的增量命令同步给从节点，完成对齐。

**增量复制（命中 backlog）**

- 直接从主节点 backlog 指定 offset 开始补齐缺失数据，**快**且**不阻塞**。

------

### 6.1.4 常用配置与命令

**从节点配置/命令：**

```bash
# redis.conf（从）
replicaof <master-ip> <port>       # 旧版：slaveof
masterauth <password>              # 主从鉴权（如主设了 requirepass）

# 动态命令
REPLICAOF <master-ip> <port>
REPLICAOF NO ONE                   # 取消复制（转单机）
```

**主节点写保护与一致性约束：**

```bash
# 当从节点数不足（或延迟过大）时拒绝写入（提升数据安全）
min-replicas-to-write 1
min-replicas-max-lag  3
# （旧版名：min-slaves-to-write / min-slaves-max-lag）
```

**只读从节点：**

```bash
replica-read-only yes
```

**无盘复制（减少主节点磁盘压力）：**

```bash
repl-diskless-sync yes
repl-diskless-sync-delay 5   # 延迟聚合更多从节点一起发送
```

------

### 6.1.5 读写分离与一致性

- 从节点默认**只读**，可以分担读压力。
- **一致性风险**：复制是异步的，读从可能读到旧数据（延迟/丢读）。
- 解决思路：
  - 强一致：读主（牺牲扩展性）。
  - “读己之写”：客户端做**延迟读**或**版本校验**；重要读走主。
  - 对一致性敏感的点 → **短期内强制走主**（如下单后的订单详情查询）。

------

### 6.1.6 常见指标与排障

- **主从延迟**：`INFO replication` → `master_repl_offset` / `slave_repl_offset`
- **backlog 大小**：`repl-backlog-size`（过小会导致频繁全量复制）
- **网络质量**：抖动会使从节点频繁断开/重连
- **大 key/重写**：全量复制期间，主节点 fork 可能导致瞬时卡顿

------

## 6.2 哨兵模式（Sentinel）

### 6.2.1 作用与组件

- **监控**：探测 master/replica 存活状态；
- **自动故障转移**：主挂后，选一个从升主，并通知客户端；
- **配置发布**：通过 `SENTINEL get-master-addr-by-name` 告知新主地址。

> 生产建议部署**≥3个哨兵**，分不同机器/可用区。

------

### 6.2.2 工作原理（SDOWN/ODOWN 与选主）

1. **主观下线（SDOWN）**：单个哨兵认为 master 不可达（超时 `down-after-milliseconds`）。
2. **客观下线（ODOWN）**：**达到法定票数（quorum）** 的哨兵一致认为 master 挂了。
3. **Leader 选举**：多个哨兵发起竞选，采用 epoch+投票（近似 Raft 思想）选出**故障转移主导者**。
4. **Failover**：Leader 选择优先级最高的 replica（`replica-priority`）为新主，执行：
   - 对被选中的从：`REPLICAOF NO ONE` → 升为主；
   - 其他从：`REPLICAOF <new-master>` → 重新指向新主；
   - 发布**新主地址**给客户端（通过 Pub/Sub 频道 `__sentinel__:hello` 及 API）。
5. **原主回来**：按配置降为从，指向新主。

------

### 6.2.3 关键配置（sentinel.conf）

```bash
# 1) 监控一个名为 mymaster 的主节点（IP/端口/法定票数）
sentinel monitor mymaster 10.0.0.10 6379 2

# 2) 主观下线判定时长（毫秒）
sentinel down-after-milliseconds mymaster 5000

# 3) 故障转移超时（含选举/迁移等）
sentinel failover-timeout mymaster 60000

# 4) 并行同步从节点个数（越大恢复越快；越小对新主压力小）
sentinel parallel-syncs mymaster 1

# 5) 认证（如主节点 requirepass ）
sentinel auth-pass mymaster <password>
```

**常用命令：**

```bash
SENTINEL get-master-addr-by-name mymaster   # 查询当前主地址
SENTINEL masters
SENTINEL replicas mymaster
SENTINEL failover mymaster                  # 手动触发故障转移
```

------

### 6.2.4 高可用设计要点

- **至少 3 哨兵**，分开部署，避免同机房单点故障；
- **quorum 合理设置**：建议 2（在 3 哨兵时），避免误判；
- **从节点优先级**：`replica-priority` 决定谁被优先提升为主；
- **客户端**：使用**哨兵感知**的连接驱动（Jedis/Redisson 提供 Sentinel 模式），自动感知主变更。

------

### 6.2.5 常见坑

- **SDOWN/ODOWN 误判**：网络抖动/GC 卡顿 → 增大 `down-after-milliseconds`；
- **脑裂**：业务/网络隔离导致双主 → 用**授权写**约束（min-replicas-to-write）并保证哨兵部署在不同机房；
- **复制风暴**：Failover 后所有从并发全量 → 调整 `parallel-syncs`，尽量命中 backlog（适当增大 `repl-backlog-size`）。

------

## 6.3 集群模式（Cluster）

> **Cluster = 多主多从 + 自动分片 + 自动故障转移**，追求**水平扩展**与**高可用**。
>  通过**16384 个哈希槽（hash slots）**实现数据分片；每个主节点负责一部分槽位，**从节点**为主提供冗余。

------

### 6.3.1 槽位分配与哈希取模

- **Hash 槽数**：16384
- **槽位计算**：`CRC16(key) % 16384`
- **键标签（Key Tag）**：`{...}` 中的内容参与哈希（多 key 操作需落在**同一槽位**）
  - 例：`user:{1001}:name` 与 `order:{1001}:list` → 通过 `{1001}` 绑定同槽。

------

### 6.3.2 通信与重定向

- 节点间通过 **Gossip** 协议交换状态，使用 **cluster bus 端口**（客户端端口+10000）。
- 客户端请求到达错误节点时，返回：
  - **MOVED** `<slot> <ip:port>`：告诉你正确节点（槽位已迁走——永久性）；
  - **ASK** `<ip:port>`：迁移中临时重定向，需先 `ASKING` 再发请求（临时性）。

------

### 6.3.3 故障转移与一致性

- **主观下线（PFAIL）**→ **客观下线（FAIL）**：集群中绝大多数主节点投票判定某主挂了；
- 其从节点发起**故障转移**，选举升为新主；
- 客户端需具备**重定向处理**能力（JedisCluster/Redisson 自动处理）。

------

### 6.3.4 搭建与管理（命令）

#### 方式一：命令行一键创建（最常用）

```bash
# 准备 6 个实例（3 主 3 从）
redis-server --port 7000 --cluster-enabled yes --cluster-config-file nodes-7000.conf --appendonly yes
redis-server --port 7001 ...
...

# 创建集群（每个主带一个从）
redis-cli --cluster create 127.0.0.1:7000 127.0.0.1:7001 127.0.0.1:7002 \
                        127.0.0.1:7003 127.0.0.1:7004 127.0.0.1:7005 \
                        --cluster-replicas 1
```

#### 方式二：手工分片（了解）

```bash
CLUSTER MEET ip port                      # 节点相互认识
CLUSTER ADDSLOTS 0 1 2 ... 5000           # 给节点分配槽位
CLUSTER REPLICATE <node-id-of-master>     # 指定为某主的从
```

#### 迁移/扩容/缩容（reshard）

```bash
# 线上迁移槽位给新节点（redis-cli 提供交互式：
redis-cli --cluster reshard <one-node>

# 底层命令理解
CLUSTER SETSLOT <slot> IMPORTING <source-node-id>
CLUSTER SETSLOT <slot> MIGRATING <dest-node-id>
MIGRATE <host> <port> <key> 0 <timeout> KEYS <k1> <k2> ...
```

------

### 6.3.5 集群关键配置

```bash
cluster-enabled yes
cluster-config-file nodes-7000.conf     # 持久化当前集群拓扑
cluster-node-timeout 15000              # 故障超时（毫秒）
cluster-require-full-coverage yes       # 槽位未覆盖时是否拒绝服务（默认是）
```

**容灾建议：**

- 每个主**至少一个从**；
- 主从**跨机房/可用区**；
- 关键业务对 ASK/MOVED 要求**具备自动重试**（驱动一般支持）。

------

### 6.3.6 集群中的限制与注意

- **多 key 操作**必须落在**同一槽**（用 `{tag}`）；
- **事务/脚本**若涉及多 key，需同槽，否则被拒；
- **不支持**多键跨槽的 `MGET/MSET/BLPOP` 等（除非驱动做分片聚合）；
- 单 key 的 `INCR/DECR` 等原子性不受影响；
- **keys \* / flushall** 等高危命令集群下受限或不建议使用。

------

### 6.3.7 观测与排障

```bash
CLUSTER NODES
CLUSTER INFO
CLUSTER SLOTS
```

常见问题：

- **Failover 抖动**：网络质量/心跳超时过短 → 调整 `cluster-node-timeout`；
- **MOVED/ASK 风暴**：处于 reshard 阶段或客户端未缓存路由 → 使用支持 cluster 的客户端；
- **槽位不完整**：`cluster-require-full-coverage yes` 会直接不可用 → 紧急时改为 `no`，尽快补齐槽位。

------

## 6.4 方案选型与最佳实践

| 业务规模 / 诉求     | 推荐方案       | 说明                                         |
| ------------------- | -------------- | -------------------------------------------- |
| 单机容灾 + 读写分离 | **主从复制**   | 简单易用；需自己处理主故障                   |
| 自动主备切换        | **哨兵**       | 低成本 HA；IP 不变动但需 sentinel 感知       |
| 水平扩容 + 高可用   | **集群**       | 自动分片与容灾；需客户端支持路由             |
| 强一致读            | 读主           | 或引入读一致性策略（版本/延迟屏蔽）          |
| 减少全量复制压力    | backlog 足够大 | `repl-backlog-size` 合理放大，尽量部分重同步 |
| 降低 fork 压力      | 无盘复制       | `repl-diskless-sync yes`；并控制大 key       |

------

## 6.5 实战模板（可直接落地）

### 6.5.1 主从 + 哨兵三机房部署示例

- **3 台 Redis（主1从2） + 3 台 Sentinel**，分布在 A/B/C 机房；
- `sentinel monitor mymaster <vip> 6379 2`；
- 客户端用 Sentinel 驱动连接 `mymaster` 名称，自动感知切主；
- 写安全约束：`min-replicas-to-write 1`、`min-replicas-max-lag 3`。

### 6.5.2 Cluster 3主3从

- 端口：7000–7005；
- `redis-cli --cluster create ... --cluster-replicas 1` 一键搭建；
- 扩容：`--cluster reshard` 将部分槽位迁给新主；
- 客户端：JedisCluster/Redisson（自动 MOVED/ASK 跟随）。

## 6.6 面试高频题（随手背）

1. **Redis 复制为何采用异步？如何减少丢数据？**

- 异步提升写性能；可用 `min-replicas-to-write` 类约束降低风险，但牺牲可用性。

1. **PSYNC 的部分重同步如何实现？**

- 通过 `replid + offset + backlog（复制挤压缓冲）` 命中增量区，只补缺失字节。

1. **哨兵如何判定主观/客观下线？**

- 单个哨兵超时判 SDOWN；达到 quorum 判 ODOWN → 选主 → 重新指向。

1. **Cluster 的 16384 槽位与 KeyTag 有何用？**

- 槽位用于一致分片；KeyTag `{}` 保证多 key 在同槽，支持多键原子操作/事务。

1. **MOVED 与 ASK 区别？**

- MOVED：槽位已经永久迁走；ASK：迁移中临时重定向（需先 `ASKING`）。

1. **复制积压缓冲为何要够大？**

- 小了会频繁全量复制，影响性能；根据 QPS 和网络状况估算（如 64–256MB）。

## 7.1 性能瓶颈定位（CPU / 内存 / 网络）

### 🔹 7.1.1 常见性能瓶颈

Redis 性能瓶颈通常来自以下几个方面：

#### 1) **CPU 限制**

- **计算密集型**命令（如 `SORT`）；
- **持久化过程**（RDB/AOF 保存）；
- **频繁的复制同步**（主从复制、哨兵故障转移）；
- **Lua 脚本执行**（脚本运行时间较长）。

**如何定位：**

- **INFO stats** 命令可以查看 Redis 实际的操作统计：

  ```bash
  INFO stats
  ```

  - 高 CPU 使用率通常伴随 `instantaneous_ops_per_sec` 快速增长，尤其是大规模的 `SET`、`GET` 操作。

- 查看 Redis 实例的 **CPU 占用**，使用 `top` 或 `htop`：

```bash
top -p $(pgrep redis)
```

#### 2) **内存瓶颈**

- Redis 的内存瓶颈通常由 **key-value 大小不合理**、**高碎片率**或 **大批量操作**（如批量写入）导致。
- 内存压力过大时，Redis 会触发 **淘汰策略** 或 **故障转移**。

**如何定位：**

- **INFO memory** 命令可以查看 Redis 的内存占用、碎片率与配置：

  ```bash
  INFO memory
  ```

  - **used_memory**：Redis 当前已使用的内存；
  - **mem_fragmentation_ratio**：内存碎片率；
  - **rss**：操作系统分配给 Redis 的物理内存量。

#### 3) **网络瓶颈**

- 网络带宽或延迟过高可能影响 Redis 与客户端的通信效率，尤其是大量并发请求时。
- 低延迟与高吞吐量是 Redis 性能的核心优势之一，网络问题会大大影响 Redis 的响应速度。

**如何定位：**

- 查看 **latency**（延迟）信息：

  ```bash
  INFO latency
  ```

- **慢查询命令**：监控慢查询，定位高延迟操作。

------

### 🔹 7.1.2 瓶颈排查方法

1. **CPU 排查**：
   - 使用 `INFO stats`，查看是否某个命令频繁执行，导致高 CPU 占用。
   - 使用 `top` 或 `htop` 查看 Redis 进程 CPU 使用情况。
2. **内存排查**：
   - 使用 `INFO memory`，查看内存使用情况，查找大 key 或高碎片的情况。
   - 定期清理过期数据，避免不必要的内存占用。
3. **网络排查**：
   - 查看网络延迟是否过高。
   - 使用 **Redis-cli ping** 测试网络响应时间。
   - 确保 Redis 和客户端部署在相同的高带宽网络环境中，避免跨地域/跨机房部署带来的延迟。

## 7.2 慢查询分析与优化

### 🔹 7.2.1 什么是慢查询

Redis 会记录 **慢查询**（执行时间超过阈值的命令），通过 **slowlog** 记录日志。

📘 **默认阈值：**

- `slowlog-log-slower-than 10000`（10ms）。
- **配置：** `slowlog-max-len 128`，最多记录 128 条慢查询。

------

### 🔹 7.2.2 查看慢查询日志

```bash
# 查看最新的 10 条慢查询
SLOWLOG GET 10
```

**返回格式：**

```bash
1) 1) (integer) 1234
   2) (integer) 1605757106
   3) (integer) 1000000
   4) "SET mykey value"
```

- `1234`：慢查询 ID；
- `1605757106`：查询时间戳；
- `1000000`：查询执行的时间，单位微秒；
- `"SET mykey value"`：执行的命令。

------

### 🔹 7.2.3 慢查询优化方法

1. **分析慢查询命令**：
   - 常见慢查询：`SORT`（计算密集型）、`LRANGE`（大列表操作）、`MGET`（多 key 查询）等。
   - 使用 **Redis 事务/管道（Pipelining）** 减少多个查询的网络开销。
2. **避免使用复杂操作**：
   - 避免高计算量命令：`SORT`，尽量改用 `ZSet` 替代。
   - 使用高效的数据结构，避免 `LRANGE` 处理大列表。
3. **调整慢查询阈值**：
   - 如果慢查询日志太多，可以增加 `slowlog-log-slower-than` 阈值，仅记录慢查询。
4. **优化 Redis 配置**：
   - 优化内存配置，减少碎片：`active-defrag`；
   - 增加网络带宽，降低网络延迟。
5. **使用 Lua 脚本**：
   - 把多个命令放入 Lua 脚本中，避免多次网络往返，提升效率。

------

### 🔹 7.2.4 优化案例：去除慢查询

如果发现大量 `SORT` 操作导致慢查询，可以考虑将 `SORT` 操作替换为 ZSet：

```bash
ZADD leaderboard 100 user1
ZADD leaderboard 200 user2
ZADD leaderboard 150 user3
ZRANGE leaderboard 0 2
```

改用 `ZSet` 存储并按 `score` 排序，避免了 `SORT` 的计算压力。

## 7.4 实战：百万级并发优化案例

### 🔹 7.4.1 背景

在高并发的场景下，Redis 必须处理大量请求，如何确保高效响应和稳定运行？

### 🔹 7.4.2 性能瓶颈分析

通过 **INFO stats** 和 **INFO memory** 定位瓶颈：

- **CPU**：`instantaneous_ops_per_sec` 和 `used_cpu_sys` 的波动；
- **内存**：`used_memory` 的上升与 `mem_fragmentation_ratio` 的增大；
- **网络**：使用 **latency** 命令检查响应延迟，是否有网络拥堵。

------

### 🔹 7.4.3 优化策略

1. **合理设置 `maxmemory` 和淘汰策略**：
    使用 `allkeys-lru` 或 `volatile-lru`，确保 Redis 在内存不足时能够合理清除过期数据。

2. **使用 Redis Pipeline**：
    将多个命令批量发送，减少网络延迟。

   ```bash
   redis-cli --pipe < commands.txt
   ```

3. **缓存热点数据**：
    对常用数据做预加载，避免频繁读取后端数据库。

4. **异步删除**：
    使用 `UNLINK` 而非 `DEL` 删除大对象，避免阻塞。

5. **使用集群分片**：
    将数据分片到多个节点，分担负载。

6. **合理设置 RDB/AOF 参数**：
    减少频繁的持久化操作，降低 Redis 的阻塞时间。

------

### 🔹 7.4.4 实战案例：高并发秒杀系统优化

1. **使用 Redis 作为缓存层**：
    商品库存信息存储在 Redis 中，减少数据库压力。
2. **异步扣减库存**：
    使用 `Lua 脚本` 和 `SETNX` 实现原子库存扣减，防止并发问题。
3. **延迟队列**：
    使用 `ZSet` 实现订单延迟处理，防止库存超卖。
4. **读写分离**：
    将读取请求分配给从节点，减少主节点负载。
5. **使用 Pipeline 批量下单**：
    避免单个订单的请求频繁与 Redis 通信，使用管道技术批量处理订单请求。

## **第八章：Redis 与项目实战**

------

> **本章在整体中解决什么问题**：前面学了数据结构、持久化、高可用；本章把 Redis 落到**业务场景**——**缓存穿透/击穿/雪崩**的成因与方案、**Spring Boot 集成**（RedisTemplate / Spring Cache / Redisson）、以及项目中的**缓存一致性、秒杀**等实战。面试常问「你们怎么防穿透/击穿/雪崩」「缓存和 DB 怎么一致」。

------

### 8.1 缓存穿透、击穿、雪崩

| 问题 | 定义 | 典型场景 | 防护方案 |
|------|------|----------|----------|
| **穿透** | 查询**不存在**的数据，缓存没有、请求直打 DB | 恶意查不存在的 id、爬虫扫号 | **布隆过滤器**预判 key 是否存在；或**缓存空值**（短 TTL，如 1～5 分钟），避免同一 key 反复打 DB。 |
| **击穿** | **热点 key 过期瞬间**，大量请求同时发现缓存失效，一起打 DB | 爆款商品、热门配置在过期那一刻被高并发查 | **互斥锁**：只有拿到锁的请求查 DB 并回填，其他等待或短暂走旧缓存；或**逻辑过期**（缓存不设真实过期，异步更新，读时发现逻辑过期再起线程更新）。 |
| **雪崩** | **大量 key 同时过期**或 **Redis 宕机**，流量全部打到 DB 或下游 | 批量预热时同一 TTL；单机 Redis 挂掉 | **过期时间加随机值**（如 base+random）分散失效时间；**多级缓存**（本地 + Redis）；**限流/降级**；**集群/哨兵**高可用。 |

**面试可答**：穿透用布隆或空值缓存；击穿用互斥锁或逻辑过期；雪崩用随机 TTL + 多级缓存 + 高可用。

### 8.2 Spring Boot 集成

- **RedisTemplate**：泛型序列化，可配置 Key/Value 序列化器；**StringRedisTemplate**：字符串操作，最常用。
- **Spring Cache**：`@Cacheable` / `@CacheEvict` / `@CachePut`，配合 CacheManager（Redis）；适合声明式缓存。
- **Redisson**：分布式锁、集合、队列等，**RLock** 实现可重入与**看门狗续期**，避免业务执行过长导致锁过期被他人占用。

**Spring Boot 示例：缓存 + 防击穿（互斥锁）**

```java
// 伪代码：先查缓存，未命中则加锁查 DB 并回填
public User getById(Long id) {
    String key = "user:" + id;
    User cached = (User) redisTemplate.opsForValue().get(key);
    if (cached != null) return cached;
    String lockKey = "lock:user:" + id;
    if (Boolean.TRUE.equals(redisTemplate.opsForValue().setIfAbsent(lockKey, "1", 10, TimeUnit.SECONDS))) {
        try {
            User user = userMapper.selectById(id);
            redisTemplate.opsForValue().set(key, user, 30, TimeUnit.MINUTES);
            return user;
        } finally {
            redisTemplate.delete(lockKey);
        }
    }
    Thread.sleep(50);
    return getById(id);  // 未拿到锁则短暂重试
}
```

（生产建议用 Redisson 的 RLock 替代手写 SET NX，避免锁过期与误删他人锁。）

### 8.3 黑马点评 / 苍穹外卖

- **秒杀**：缓存热点商品、库存用 **Lua 或 Redisson** 扣减，防超卖与重复下单。
- **订单状态与缓存一致性**：**先更新 DB 再删缓存**；删除失败可重试或 MQ 异步删；或 **Canal 订阅 binlog** 失效缓存。

### 8.4 常见坑与注意点

| 现象 / 易错点 | 原因 | 怎么改 / 怎么记 |
|---------------|------|-----------------|
| 先删缓存再更新 DB | 删缓存后、更新 DB 前，其他请求把旧数据又写回缓存，导致脏读 | **先更新 DB 再删缓存**；删缓存失败要重试或 MQ 补偿，避免长期不一致。 |
| 互斥锁未设过期或误删他人锁 | SET NX 后未 EXPIRE，进程挂掉锁永不释放；或过期时间短于业务执行，自己锁被释放后删了别人的锁 | 用 **SET key value NX PX 毫秒** 原子设过期；或 **Redisson 看门狗**自动续期；删除前校验 value 是否为自己持有的标识。 |
| 布隆过滤器未与 DB 同步 | 新增数据只写 DB 未加入布隆，请求仍被误判为不存在 | 写 DB 成功后**同步更新布隆**；或定期重建布隆（离线扫描 DB）。 |
| 缓存空值 TTL 过长 | 空值缓存 1 小时，期间若有真实数据写入 DB，会长时间读不到 | 空值给**较短 TTL**（如 1～5 分钟），或 key 设计区分「不存在」与「空对象」。 |

## **第九章：Redis 源码与底层机制解析**

### 9.1 事件模型

- 单线程 **事件循环**，基于 **epoll/kqueue** 做 I/O 多路复用；**文件事件** 处理客户端读写，**时间事件** 处理定时任务（如 serverCron）；命令串行执行，阻塞命令会拖垮整个实例。

- 内部 **dict**（哈希表）、**ziplist**（压缩列表）、**skiplist**（跳表）、**quicklist**（List 底层）等；命令在 **命令表** 中注册，根据名字找到处理函数，解析参数后执行并写回响应。

### 9.3 RDB/AOF 源码

- RDB 由子进程或后台线程生成快照；AOF 先写缓冲区再 fsync；混合持久化时 AOF 重写可把 RDB 与增量 AOF 合并。深入可阅读 redis 源码 src 目录。

## **第十章：Redis 常见问题与排查**

### 10.1 常见异常

- **内存不足**：检查 maxmemory、淘汰策略、大 key（redis-cli --bigkeys）；可加大内存或优化 key 设计、拆分大 key。**连接数过多**：maxclients、连接池配置；检查客户端是否泄漏连接、是否未用连接池。**主从延迟**：看 replication 的 offset、lag；网络与从库负载、慢查询影响。

### 10.2 数据不一致与缓存失效

- 先更新 DB 再删缓存；删除失败可重试或发 MQ 异步删；缓存与 DB 双写时注意顺序与并发。**崩溃恢复**：AOF 优先（若开启），否则 RDB；从节点可做备份与故障转移。

## **第十一章：Redis 安全与权限管理**

### 11.1 认证与 ACL

- **requirepass**：连接后 AUTH 密码；**ACL**（Redis 6+）：按用户配置命令与 key 权限（acl list、acl setuser 等）。

### 11.2 网络与防护

- **bind** 限制监听 IP；防火墙限制访问；**rename-command** 禁用或重命名危险命令（FLUSHALL、KEYS 等）；生产禁用 KEYS，用 SCAN 替代。
