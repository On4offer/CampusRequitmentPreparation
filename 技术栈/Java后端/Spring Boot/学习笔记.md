# 第1章：Spring Boot 概述与核心思想

------

## 1.1 Spring Boot 的起源与发展

### 🧩 背景

在 Spring Framework 早期版本（2.x、3.x）中，开发者需要手动：

- 创建繁琐的 `XML` 配置文件；
- 手动注册 Bean；
- 手动管理依赖版本；
- 编写大量样板（Boilerplate）代码。

这种复杂性被称为 **“Spring 配置地狱（Configuration Hell）”**。

------

### 🚀 诞生

- **Spring Boot 由 Pivotal 团队在 2014 年推出**（Spring Framework 4.x 之后）。
- 核心目标是**简化 Spring 应用的开发、配置与部署**。
- 它通过**自动装配（Auto Configuration）**和**约定优于配置（Convention over Configuration）**思想，
   实现“开箱即用”的开发体验。

------

### 📜 发展时间线

| 年份 | 版本 | 特性亮点                                   |
| ---- | ---- | ------------------------------------------ |
| 2014 | 1.0  | 初版发布，支持内嵌容器与自动配置           |
| 2017 | 2.0  | 基于 Spring Framework 5，全面支持 Reactive |
| 2022 | 3.0  | 兼容 Jakarta EE，全面转向 JDK 17+          |
| 2024 | 3.3+ | 提升 AOT 编译支持（GraalVM 原生镜像）      |

> ✅ Spring Boot 已成为 Spring 全家桶的核心基础，是现代企业级 Java 后端的标配框架。

## 1.2 Spring 与 Spring Boot 的关系

### 📚 基础类比

| 对比维度 | Spring Framework               | Spring Boot                |
| -------- | ------------------------------ | -------------------------- |
| 定位     | 基础框架                       | 快速开发框架（封装增强）   |
| 配置方式 | 需手动配置 Bean、XML、注解混合 | 自动装配、默认配置         |
| 启动方式 | 依赖外部容器（Tomcat、Jetty）  | 内嵌容器运行               |
| 构建方式 | 传统 Web 项目（web.xml）       | 独立运行 Jar               |
| 依赖管理 | 需手动维护版本                 | 统一版本管理（parent POM） |

------

### ⚙️ 结构关系图

```
Spring Boot
│
├── 基于 Spring Framework 核心容器（IoC + AOP）
├── 封装自动配置（spring-boot-autoconfigure）
├── 提供启动器（Starters）
├── 集成 Actuator、DevTools、Test 模块
└── 内嵌 Web 服务器（Tomcat/Jetty/Undertow）
```

> 可以理解：**Spring 是“地基”，Spring Boot 是“装修好的一栋楼”。**

## 1.3 Spring Boot 的核心目标与优势

### 🌟 三大核心目标

1. **简化配置（Simplification）**：最少的 XML / 注解。
2. **快速开发（Rapid Development）**：Starter 自动装配。
3. **独立部署（Standalone）**：Jar 打包即可运行（内嵌 Tomcat）。

------

### 💡 优势汇总表

| 优势点           | 说明                                                |
| ---------------- | --------------------------------------------------- |
| **自动装配**     | 根据依赖自动创建 Bean，无需手动配置。               |
| **内嵌容器**     | 自带 Tomcat / Jetty / Undertow，直接运行 `main()`。 |
| **统一依赖管理** | 父 POM 管理常用依赖版本，避免冲突。                 |
| **生产级特性**   | 提供 Actuator 监控、健康检查、外部化配置。          |
| **易于集成**     | 无缝整合 Spring Cloud、MyBatis、JPA、Redis 等。     |
| **微服务友好**   | 天然支持 Spring Cloud 微服务生态。                  |

## 1.4 约定优于配置（Convention over Configuration）

### 🎯 核心思想

> “让开发者关注业务，而不是框架细节。”

Spring Boot 通过定义一套合理的**默认约定（Convention）**，让系统在没有显式配置的情况下也能自动运行。

------

### 🧩 实例说明

| 功能      | 约定规则           | 示例                                                |
| --------- | ------------------ | --------------------------------------------------- |
| 配置文件  | 默认加载路径       | `application.yml` 或 `application.properties`       |
| Web 容器  | 默认使用 Tomcat    | 若引入 `spring-boot-starter-web`，即自动启用 Tomcat |
| 数据源    | 自动检测 JDBC 驱动 | 引入 MySQL 依赖 → 自动创建 DataSource               |
| JSON 转换 | 自动使用 Jackson   | 引入 `spring-boot-starter-json` 即可                |
| 端口      | 默认 8080          | 可通过配置修改 `server.port=8081`                   |

------

### 💬 举例说明

不需要手动声明：

```xml
<bean id="dataSource" class="com.zaxxer.hikari.HikariDataSource">
    <property name="driverClassName" value="com.mysql.cj.jdbc.Driver"/>
</bean>
```

只要：

```yaml
spring:
  datasource:
    url: jdbc:mysql://localhost:3306/test
    username: root
    password: 123456
```

👉 Spring Boot 自动识别并装配 HikariDataSource。

## 1.5 自动装配思想与简化开发的实现

### 🔍 核心原理概览

1. 启动类使用 `@SpringBootApplication` 注解。

2. 该注解内部包含 `@EnableAutoConfiguration`。

3. Spring Boot 启动时读取：

   ```
   META-INF/spring.factories
   ```

4. 加载匹配的自动配置类（例如 Web、DataSource、Cache）。

5. 每个自动配置类使用 `@ConditionalOnXxx` 进行条件判断，只在符合条件时生效。

------

### ⚙️ 示例：DataSource 自动装配流程

1. 引入 `spring-boot-starter-data-jdbc`
2. Spring Boot 检测到 JDBC 依赖
3. 加载 `DataSourceAutoConfiguration`
4. 条件判断：
   - 存在 `DataSource` 类
   - 存在数据库驱动
5. 自动注册 HikariDataSource Bean

------

### 📂 关键文件：`spring.factories`

```properties
org.springframework.boot.autoconfigure.EnableAutoConfiguration=\
org.springframework.boot.autoconfigure.jdbc.DataSourceAutoConfiguration,\
org.springframework.boot.autoconfigure.web.servlet.WebMvcAutoConfiguration
```

> 👉 每个 Starter 都有对应的自动配置类。

------

### 🧠 面试扩展问题

**面试官：**
 Spring Boot 的自动装配原理是什么？

**标准回答：**

> Spring Boot 的自动装配基于 `@EnableAutoConfiguration` 注解实现。
>  启动时，Spring Boot 会扫描所有依赖包下的 `META-INF/spring.factories` 文件，
>  从中加载配置的 `AutoConfiguration` 类。这些类会根据 `@ConditionalOnXxx` 条件判断当前环境是否满足，从而动态注册相应的 Bean，实现按需装配。

## 2.1 项目结构与目录说明（Maven/Gradle）

### 🏗️ Maven 项目结构

Spring Boot 项目通常使用 **Maven** 或 **Gradle** 构建，以下是典型的 Maven 项目结构：

```
.
├── pom.xml              # Maven 构建文件
├── src/
│   ├── main/
│   │   ├── java/
│   │   │   └── com/example/demo/
│   │   │       └── DemoApplication.java   # 启动类
│   │   ├── resources/
│   │   │   ├── application.properties    # 配置文件
│   │   │   └── static/                   # 静态资源（JS/CSS/图片）
│   │   │   └── templates/                # 模板文件（Thymeleaf）
│   └── test/
│       ├── java/
│       │   └── com/example/demo/
│       │       └── DemoApplicationTests.java # 单元测试类
├── target/                    # 编译输出目录
```

- **`pom.xml`**：定义项目的依赖、插件和构建配置。
- **`src/main/java`**：应用程序的主代码，包括启动类和业务逻辑类。
- **`src/main/resources`**：存放应用的配置文件、静态资源、模板等。
- **`src/test/java`**：存放测试代码和测试用例。

## 2.2 `@SpringBootApplication` 注解详解

### 📍 `@SpringBootApplication` 组成

`@SpringBootApplication` 是 Spring Boot 启动类的核心注解，包含了三个常用注解：

1. **`@SpringBootConfiguration`**：标记配置类，等效于 `@Configuration`，用于定义 Spring 配置。
2. **`@EnableAutoConfiguration`**：开启自动装配机制，Spring Boot 会根据项目依赖和配置自动加载相关的 Bean。
3. **`@ComponentScan`**：开启组件扫描，自动扫描当前包及其子包下的所有组件（如 `@Service`、`@Repository`、`@Controller` 等）。

------

### 🧩 工作原理

- 通过 `@SpringBootApplication`，Spring Boot 启动类自动启用 **组件扫描**、**自动配置** 和 **配置类加载**。
- `@EnableAutoConfiguration` 的作用是根据项目的依赖，自动配置相关组件，例如 Web 服务器、数据库连接池、消息队列等。

------

### 🧠 面试扩展

**标准回答：**

> `@SpringBootApplication` 是 Spring Boot 启动类的核心注解，作用是整合 `@SpringBootConfiguration`、`@EnableAutoConfiguration` 和 `@ComponentScan`。它不仅标记了配置类，还启用了自动装配和组件扫描功能。通过这个注解，Spring Boot 可以快速自动配置应用程序并扫描相关 Bean。

## 2.3 SpringApplication 启动流程（run 方法执行机制）

### ⚙️ 启动流程概览

Spring Boot 启动过程通过 `SpringApplication.run()` 方法实现，整个流程大致如下：

1. **创建 `SpringApplication` 实例**
   - `SpringApplication` 类负责启动应用程序，初始化应用环境。
2. **初始化环境**
   - 加载系统环境变量，读取配置文件（如 `application.properties`），确定应用程序的运行环境。
3. **创建应用上下文**
   - 根据应用类型（Web/Reactive）创建 `ApplicationContext`，默认是 `AnnotationConfigApplicationContext`（容器）。
4. **启动监听器**
   - 触发各种事件（如 `ApplicationStartedEvent`、`ApplicationReadyEvent`）。
5. **执行 `CommandLineRunner` 和 `ApplicationRunner`**
   - 如果实现了这些接口，Spring Boot 会在应用启动后执行相关的逻辑。
6. **启动嵌入式服务器**
   - 如果是 Web 应用，Spring Boot 会自动启动内嵌的 Tomcat/Jetty 服务器，并部署应用。

------

### 🔍 源码示例

```java
public static void main(String[] args) {
    SpringApplication app = new SpringApplication(DemoApplication.class);
    app.run(args);
}
```

- **步骤 1**：创建 `SpringApplication` 实例。
- **步骤 2**：调用 `run()` 方法启动应用。

`run()` 方法内部的关键步骤：

- **`prepareContext()`**：准备并加载应用上下文。
- **`refreshContext()`**：刷新容器（加载 Bean）。
- **`afterRefresh()`**：完成应用上下文的初始化。

## 2.4 自动配置加载原理（AutoConfiguration）

### 🧠 自动配置机制概述

Spring Boot 的核心特性之一就是自动配置。它通过 `@EnableAutoConfiguration` 和条件注解（`@ConditionalOn...`）来动态地装配合适的 Bean。

1. **`spring.factories` 文件**
    自动配置类由 **`spring.factories`** 文件定义，Spring Boot 会根据应用的依赖和配置选择性加载相关配置类。
2. **条件注解**
    例如：`@ConditionalOnClass`、`@ConditionalOnMissingBean`，自动配置类只有在特定条件下才会被激活。

------

### 🔍 示例：自动配置 Web 服务器

```java
@Configuration
@ConditionalOnClass(EmbeddedServletContainer.class)
public class WebAutoConfiguration {
    @Bean
    public EmbeddedServletContainer embeddedServletContainer() {
        return new TomcatEmbeddedServletContainer();
    }
}
```

- 只有当 `EmbeddedServletContainer` 存在时，Tomcat 自动配置才会生效。

# 第3章：自动装配机制（核心原理）

------

## 3.1 什么是自动装配（Auto Configuration）

### 🧩 自动装配的定义

自动装配（Auto Configuration）是 Spring Boot 的核心特性之一，旨在减少手动配置的复杂度。在 Spring Boot 中，**自动装配会根据项目的依赖、环境配置和系统条件，自动选择并加载适合的配置类**。

#### 目标

- **减少配置**：默认配置好许多常见场景，避免重复编写配置。
- **提高开发效率**：自动识别依赖并配置相关 Bean，简化开发流程。

#### 工作原理

自动装配主要通过 `@EnableAutoConfiguration` 注解结合 Spring Boot 提供的条件注解（如 `@ConditionalOnClass`）来实现，自动为依赖提供相应的配置和 Bean。

------

### 🔑 自动装配工作流程

1. **依赖加载**：Spring Boot 在应用启动时，根据项目的依赖自动判断是否需要某些配置类。
2. **配置选择**：根据条件注解（如 `@ConditionalOnClass`）动态选择并加载配置。
3. **Bean 注入**：自动注册符合条件的 Bean 到应用上下文中。

例如，当引入 `spring-boot-starter-web` 时，Spring Boot 会自动配置一个嵌入式的 Tomcat 容器，提供 Web 服务功能。

 Spring Boot 自动装配机制是如何工作的？

**标准回答：**

> Spring Boot 的自动装配通过 `@EnableAutoConfiguration` 注解实现，自动配置类会根据项目的依赖和环境条件来加载相应的配置。具体地，自动配置类通过 `@ConditionalOnXxx` 条件注解判断是否需要加载相关 Bean，简化了开发者的配置工作。比如，当引入 `spring-boot-starter-web` 时，Spring Boot 会自动配置一个嵌入式的 Tomcat。

## 3.2 `@EnableAutoConfiguration` 与 `spring.factories` 文件

### 📍 `@EnableAutoConfiguration` 注解

`@EnableAutoConfiguration` 是自动装配的核心注解，通常与 `@SpringBootApplication` 一起使用。它会触发 Spring Boot 启动时，自动加载符合条件的配置类。

#### 作用

- 启用自动装配机制。
- 让 Spring Boot 根据依赖自动配置应用的各种组件（如 Web、数据库连接池、消息队列等）。

#### 工作原理

- Spring Boot 在启动时会扫描 `spring.factories` 文件，加载其中列出的自动配置类。
- 每个自动配置类通过条件注解（如 `@ConditionalOnClass`）来判断是否需要加载。

## 3.3 条件装配注解：`@ConditionalOn...` 全家桶

### 🧩 条件装配注解概述

Spring Boot 提供了一系列条件装配注解（`@ConditionalOn...`），用于控制自动装配类的加载。常见的条件装配注解有：

1. **`@ConditionalOnClass`**
    只有在类路径中存在指定的类时，自动装配才会生效。

   ```java
   @ConditionalOnClass(DataSource.class)
   public class DataSourceAutoConfiguration {
       // 数据源自动配置
   }
   ```

2. **`@ConditionalOnMissingBean`**
    只有在容器中不存在指定类型的 Bean 时，才会注册当前 Bean。

   ```java
   @ConditionalOnMissingBean(DataSource.class)
   public class DataSourceAutoConfiguration {
       // 数据源自动配置
   }
   ```

3. **`@ConditionalOnProperty`**
    只有当指定的配置属性存在时，才会激活配置类。

   ```java
   @ConditionalOnProperty(name = "spring.datasource.enabled", havingValue = "true")
   public class DataSourceAutoConfiguration {
       // 数据源自动配置
   }
   ```

4. **`@ConditionalOnBean`**
    只有在容器中存在指定类型的 Bean 时，才会激活当前配置类。

   ```java
   @ConditionalOnBean(MyService.class)
   public class MyServiceAutoConfiguration {
       // 自动配置
   }
   ```

5. **`@ConditionalOnExpression`**
    通过 SpEL 表达式动态控制自动装配。

   ```java
   @ConditionalOnExpression("${myapp.feature.enabled:true}")
   public class FeatureAutoConfiguration {
       // 自动配置
   }
   ```

------

### 📍 示例：条件装配

```java
@Configuration
@ConditionalOnClass(MyService.class)
public class MyServiceAutoConfiguration {
    @Bean
    public MyService myService() {
        return new MyService();
    }
}
```

- 只有当 `MyService` 类存在时，才会注册该 Bean。

**标准回答：**

> `@ConditionalOn...` 注解是一系列条件注解，用于控制自动配置类是否生效。它通过判断某个条件是否成立来决定是否加载配置类。例如，`@ConditionalOnClass` 判断类路径中是否存在某个类，`@ConditionalOnProperty` 判断配置文件中某个属性的值。

## 4.1 `application.properties` 与 `application.yml` 配置文件

### 🧩 配置文件概述

Spring Boot 提供两种配置文件格式来管理应用配置：

- **`application.properties`**：基于键值对的传统配置文件格式。
- **`application.yml`**：YAML 格式的配置文件，结构清晰、可读性强。

### 📍 `application.properties` 示例

```properties
server.port=8080
spring.datasource.url=jdbc:mysql://localhost:3306/demo
spring.datasource.username=root
spring.datasource.password=secret
```

### 📍 `application.yml` 示例

```yaml
server:
  port: 8080

spring:
  datasource:
    url: jdbc:mysql://localhost:3306/demo
    username: root
    password: secret
```

- **YAML 格式**支持层级结构，适合复杂的配置，且比 `.properties` 更易于阅读和管理。

------

### ⚙️ 配置文件加载顺序

1. **`application.properties` / `application.yml`**（默认配置）
2. **命令行参数**（优先级高于配置文件）
3. **系统环境变量**
4. **`@Value` 或 `@ConfigurationProperties` 注入的属性**

Spring Boot 会自动加载 `application.properties` 或 `application.yml` 文件，默认情况下，文件存放在 `src/main/resources` 目录下。

------

### 🧠 面试扩展

**面试官：**
 `application.properties` 和 `application.yml` 配置文件有什么区别？

**标准回答：**

> `application.properties` 和 `application.yml` 都是 Spring Boot 用来管理配置的文件，主要区别在于格式。`application.properties` 使用键值对格式，适合简单配置，而 `application.yml` 使用 YAML 格式，支持层级结构，适合复杂配置。YAML 格式更加可读，也更容易管理嵌套结构的配置。

------

## 4.2 外部化配置优先级体系（命令行参数、系统环境变量等）

### 📍 外部化配置机制

Spring Boot 允许将配置从代码中分离出来，提供了一套强大的外部化配置机制，使得开发者能够根据不同环境加载不同的配置。

#### 配置优先级

Spring Boot 的配置优先级顺序如下，从低到高：

1. **默认配置文件**：`application.properties` 或 `application.yml`
2. **命令行参数**：通过 `--` 传递给 Spring Boot 的参数（例如 `--server.port=8081`）
3. **系统环境变量**：`SPRING_DATASOURCE_URL` 等
4. **配置文件中的 Profile 特定配置**（例如 `application-dev.properties`）
5. **`@Value` / `@ConfigurationProperties` 注入的属性**

Spring Boot 会按照上述优先级加载配置，后加载的配置会覆盖先加载的配置。

------

### ⚙️ 示例：命令行参数

```bash
java -jar app.jar --server.port=8081 --spring.datasource.url=jdbc:mysql://localhost:3306/mydb
```

- 命令行参数中的配置会覆盖 `application.properties` 文件中的相同配置项。

------

### 🧠 面试扩展

**面试官：**
 Spring Boot 配置的优先级体系是怎样的？

**标准回答：**

> Spring Boot 配置的优先级顺序如下：默认配置文件（`application.properties` 或 `application.yml`）最先加载，其次是命令行参数、系统环境变量、Profile 环境配置，最后是 `@Value` 和 `@ConfigurationProperties` 注入的属性。后加载的配置会覆盖先加载的配置。

## 4.3 Profile 环境隔离机制（多环境配置）

### 🧩 Profile 环境概述

Spring Boot 支持 **Profile 环境隔离**，可以根据不同的部署环境（如开发环境、测试环境、生产环境等）加载不同的配置。通过 **`@Profile` 注解** 或 **`application-{profile}.properties`** 来实现环境隔离。

#### 典型使用场景

- 开发环境使用 `application-dev.properties` 配置。
- 生产环境使用 `application-prod.properties` 配置。

### 📍 配置文件示例

#### `application-dev.properties`


#### `application-prod.properties`

```properties
spring.datasource.url=jdbc:mysql://localhost:3306/proddb
server.port=8080
```

#### 启动时指定激活的 Profile

```bash
java -jar app.jar --spring.profiles.active=dev
```

------

### ⚙️ 启动时切换 Profile

在 `application.properties` 或 `application.yml` 中配置默认激活的 Profile：

```properties
spring.profiles.active=dev
```

### 🧠 面试扩展

**面试官：**
 Spring Boot 中如何管理不同环境的配置？

**标准回答：**

> Spring Boot 通过 Profile 机制管理不同环境的配置。开发者可以在 `application-{profile}.properties` 或 `application-{profile}.yml` 文件中为不同环境指定不同的配置，并通过命令行参数（`--spring.profiles.active=dev`）或配置文件（`spring.profiles.active`）来指定当前激活的 Profile。这样，Spring Boot 会根据指定的 Profile 加载相应的配置。

## 4.4 `@ConfigurationProperties` 与 `@Value` 注入属性

### 🧩 `@Value` 注解

`@Value` 注解用于直接将配置文件中的值注入到字段中，支持基本数据类型、String、列表等。

#### 示例：

```java
@Value("${server.port}")
private int port;
```

- 这里 `server.port` 配置的值会被注入到 `port` 字段。

------

### 📍 `@ConfigurationProperties` 注解

`@ConfigurationProperties` 注解用于将一组相关的配置属性绑定到一个 POJO 类上，适用于复杂的配置结构。

#### 示例：

```java
@ConfigurationProperties(prefix = "spring.datasource")
public class DataSourceConfig {
    private String url;
    private String username;
    private String password;
    
    // getters and setters
}
```

- 上述类会将 `application.properties` 或 `application.yml` 中以 `spring.datasource` 开头的配置项绑定到 `DataSourceConfig` 类中。

------

### ⚙️ 配置绑定原理

Spring Boot 会自动扫描并将带有 `@ConfigurationProperties` 注解的类绑定到配置文件中的相应属性。

```properties
spring.datasource.url=jdbc:mysql://localhost:3306/db
spring.datasource.username=root
spring.datasource.password=secret
```

------

### 🧠 面试扩展

**面试官：**
 `@ConfigurationProperties` 和 `@Value` 注解有什么区别？

**标准回答：**

> `@Value` 注解用于将单个配置项注入到字段中，适用于简单的配置。`@ConfigurationProperties` 注解则用于将一组相关的配置项注入到一个 POJO 类中，适用于复杂的、结构化的配置。`@ConfigurationProperties` 支持批量注入，并且可以绑定多级配置属性。

## 4.5 自定义配置文件加载与参数绑定

### 📍 自定义配置文件加载

```java
@Configuration
@PropertySource("classpath:custom-config.properties")
```

### 🧩 配置文件加载顺序

1. 默认的 `application.properties` 或 `application.yml`
2. 自定义的配置文件（通过 `@PropertySource` 或命令行参数）
3. 外部配置（环境变量、命令行参数）

## 5.1 内嵌 Servlet 容器原理（Tomcat/Jetty/Undertow）

### 🧩 内嵌容器概述

- **Undertow**

#### 内嵌 Tomcat 的优点：

1. **无需外部容器**：应用打包为 Jar 文件即可运行。
2. **简化部署**：应用可以在任何地方运行，不依赖外部服务器。
3. **性能优异**：适用于微服务架构和独立部署的场景。

#### 启动示例：

```bash
java -jar my-app.jar
```

- 启动时会自动启动内嵌的 Tomcat 容器，默认端口为 8080。

------

### 🔑 Jetty 和 Undertow

- **Jetty** 和 **Undertow** 是替代 Tomcat 的轻量级容器，它们都支持 Servlet 和 WebSocket。

#### 配置 Jetty 示例：

```properties
spring.main.web-application-type=servlet
server.servlet.container=jetty
```

## 5.2 SpringMVC 自动配置与 DispatcherServlet

### 🧩 SpringMVC 自动配置

#### 核心原理

1. **`DispatcherServlet`**：作为前端控制器，拦截所有请求，负责将请求分发给合适的处理器。
2. **自动配置**：Spring Boot 自动配置了 `DispatcherServlet`，并通过 `@EnableWebMvc` 启用 SpringMVC 的功能。

#### 配置与启动

```java
    }
}
```

### 📍 `DispatcherServlet` 工作流程

1. **接收请求**：请求通过 Servlet 过滤器进入 `DispatcherServlet`。
2. **请求分发**：`DispatcherServlet` 将请求转发到适当的 **HandlerMapping**（如 `@RequestMapping`）进行处理。
3. **返回结果**：处理完成后，结果返回给 **ViewResolver**，最终将响应输出给客户端。

## 6.4 声明式事务管理与 `@Transactional` 原理

### 🧩 声明式事务管理

Spring Boot 使用 **`@Transactional`** 注解进行声明式事务管理，可以方便地控制事务的提交和回滚。

#### `@Transactional` 的原理

1. **事务管理器**：Spring Boot 默认使用 `DataSourceTransactionManager` 来管理数据库事务。
2. **配置方式**：通过 `@Transactional` 注解在方法或类级别进行事务管理。

#### 示例：`@Transactional` 注解

```java
@Transactional
public void updateUser(User user) {
    userRepository.save(user);
    // 此处若抛出异常，事务会回滚
}
```

------

### 📍 事务传播行为

Spring 提供了多种事务传播行为，常用的有：

- **`REQUIRED`**（默认）：如果当前存在事务，则加入当前事务；如果没有事务，则新建一个事务。
- **`REQUIRES_NEW`**：无论当前是否存在事务，都会创建一个新的事务。
- **`NESTED`**：如果当前存在事务，则创建一个嵌套事务；否则，新建一个事务。

------

### 🧠 面试扩展

**面试官：**
 `@Transactional` 注解如何管理事务？

**标准回答：**

> `@Transactional` 注解用于声明式事务管理。它可以应用在方法或类上，Spring 会自动管理方法的事务。若方法内发生异常，Spring 会根据事务传播行为决定是否回滚事务。Spring 默认使用 `DataSourceTransactionManager` 来管理数据库事务，并支持事务传播行为的配置。

## 6.5 多数据源配置与动态切换方案

## 7.2 缓存注解体系：`@Cacheable`、`@CacheEvict`、`@CachePut`

### 🧩 缓存注解概述

Spring 提供了三种主要的缓存注解来管理缓存：

- **`@Cacheable`**：从缓存中查找数据，如果找不到则调用方法，并将结果缓存起来。
- **`@CacheEvict`**：清除缓存中的某个条目，或者清除整个缓存。
- **`@CachePut`**：更新缓存中的数据，并继续执行方法。

#### 1. **`@Cacheable`**

用于缓存方法的返回值，如果缓存中已存在该值，则直接返回缓存值，否则调用方法执行并将结果缓存起来。

```java
@Cacheable(value = "users", key = "#id")
public User getUserById(String id) {
    return userRepository.findById(id);
}
```

#### 2. **`@CacheEvict`**

用于删除缓存中的数据，可以指定删除一个特定的缓存或清空所有缓存。

```java
@CacheEvict(value = "users", key = "#id")
public void deleteUser(String id) {
    userRepository.deleteById(id);
}
```

#### 3. **`@CachePut`**

用于更新缓存中的数据，缓存会在方法执行后更新。

```java
@CachePut(value = "users", key = "#user.id")
public User updateUser(User user) {
    return userRepository.save(user);
}
```

------

### 🧠 面试扩展

**面试官：**
 Spring Cache 中 `@Cacheable`、`@CacheEvict` 和 `@CachePut` 注解的作用是什么？

**标准回答：**

> - `@Cacheable` 用于从缓存中获取数据，如果没有则调用方法并缓存结果。
> - `@CacheEvict` 用于清除缓存中的某个条目或清空整个缓存。
> - `@CachePut` 用于更新缓存中的数据并执行方法，不会从缓存中获取数据。

## 第8章：安全与认证机制

### 8.1 Spring Security 自动配置

- 引入 **spring-boot-starter-security** 后，自动启用默认 HTTP Basic 与表单登录、CSRF 等；可通过 **SecurityFilterChain** Bean 或 **WebSecurityConfigurerAdapter**（2.x）自定义 URL 权限、登录页、放行白名单。

### 8.2 认证与权限模型

- **认证（Authentication）**：验证“你是谁”（登录）；**授权（Authorization）**：验证“你能做什么”（角色/权限）。Spring Security 通过 **UserDetailsService** 加载用户、**PasswordEncoder** 校验密码，通过 **hasRole / hasAuthority** 做方法或 URL 级权限控制。

### 8.3 Token / JWT 与拦截器

- 无状态登录：登录成功后签发 **JWT**，客户端后续在 **Authorization** 头携带；服务端用 **Filter 或 Interceptor** 解析 JWT、校验签名与过期，将用户信息写入 **SecurityContext** 或 **ThreadLocal**，供业务层获取当前用户。

### 8.4 ThreadLocal 存用户上下文

- 在拦截器 **preHandle** 中解析 token，将 UserId/UserDTO 放入 **ThreadLocal**；**afterCompletion** 中 **remove**，避免线程池复用时串数据。业务层通过工具类从 ThreadLocal 取当前用户。

### 8.5 CSRF / CORS

- **CSRF**：表单提交时需带服务端下发的 token（Session 场景）；前后端分离 + JWT 无 Session 时可禁用 CSRF。**CORS**：通过 **CorsConfigurationSource** 或 **WebMvcConfigurer.addCorsMappings** 配置允许的 origin、method、header。

### 8.6 实战：基于 Token 的鉴权

- 登录接口校验用户名密码后生成 JWT 返回；其他接口在拦截器中校验 JWT，无效或过期返回 401；白名单（登录、注册、静态资源）不校验；用户信息存 ThreadLocal，Controller/Service 按需获取。

## 10.1 Bean 的生命周期与初始化流程

### 🧩 Bean 的生命周期概述

在 Spring 中，**Bean 的生命周期**涉及从 Bean 创建、初始化、使用到销毁的全过程。Spring 管理的 Bean 生命周期通常包括以下几个阶段：

1. **实例化**：Spring 容器根据配置实例化 Bean。
2. **属性注入**：通过依赖注入（DI）注入属性（如构造器注入、字段注入、Setter 注入等）。
3. **初始化**：如果 Bean 实现了 `InitializingBean` 接口，或者使用 `@PostConstruct` 注解，则调用 `afterPropertiesSet()` 或初始化方法。
4. **销毁**：如果 Bean 实现了 `DisposableBean` 接口，或者使用 `@PreDestroy` 注解，容器在销毁前调用销毁方法。

------

### 📍 Bean 生命周期图解

```
实例化 Bean → 属性注入 → 初始化方法 → 使用 Bean → 销毁方法
```

------

### 📍 Spring Bean 初始化流程

1. **实例化 Bean**：通过反射或配置文件创建 Bean 实例。
2. **依赖注入**：根据配置将所需依赖注入 Bean。
3. **调用初始化方法**：如果 Bean 实现了 `InitializingBean` 或使用了 `@PostConstruct`，则调用初始化方法。
4. **Bean 使用**：通过 Spring 容器中的其他组件或服务使用该 Bean。
5. **销毁**：当容器销毁时，如果 Bean 实现了 `DisposableBean` 或使用了 `@PreDestroy`，则调用销毁方法。

#### 示例：`@PostConstruct` 和 `@PreDestroy`

```java
@Component
public class MyBean {

    @PostConstruct
    public void init() {
        System.out.println("Bean is initialized");
    }

    @PreDestroy
    public void destroy() {
        System.out.println("Bean is destroyed");
    }
}
```

------

### 🧠 面试扩展

 Spring Bean 的生命周期是什么样的？

**标准回答：**

> Spring Bean 的生命周期包括实例化、依赖注入、初始化、使用和销毁等阶段。Bean 在实例化后，会通过依赖注入注入属性，接着调用初始化方法（如实现 `InitializingBean` 或 `@PostConstruct`），使用完后，若实现了 `DisposableBean` 或使用 `@PreDestroy` 注解，Spring 会在销毁时调用销毁方法。

## 10.2 ApplicationContext 与 BeanFactory 区别

### 🧩 ApplicationContext 与 BeanFactory

Spring 提供了两种容器：**BeanFactory** 和 **ApplicationContext**，它们的功能相似，但 **ApplicationContext** 提供了更多的功能，如事件机制、国际化支持等。

#### **BeanFactory**

- 负责基本的容器功能，如 Bean 的管理和依赖注入。
- 使用懒加载（Lazy Initialization）加载 Bean，适合资源有限的环境。

#### **ApplicationContext**

- 扩展了 `BeanFactory`，包含更多高级功能。
- 支持 **事件机制**（`ApplicationEvent`）、**国际化支持**、**AOP** 等。
- 是大多数 Spring 应用使用的容器类型。

------

### 📍 核心区别

| 特性     | BeanFactory                   | ApplicationContext                        |
| -------- | ----------------------------- | ----------------------------------------- |
| 功能     | 基本的 Bean 管理              | 基本的 Bean 管理 + 事件机制、国际化支持等 |
| 启动时机 | 懒加载（Lazy Initialization） | 默认立即加载所有 Bean                     |
| 使用场景 | 适用于内存有限的环境          | 适用于大多数应用                          |

------

### 🧠 面试扩展

**面试官：**
 `ApplicationContext` 和 `BeanFactory` 的区别是什么？

**标准回答：**

> `ApplicationContext` 是 `BeanFactory` 的扩展，提供了更多的功能，如事件机制、国际化支持和 AOP 等。`BeanFactory` 提供最基础的 Bean 管理功能，适合内存受限的环境，采用懒加载方式。`ApplicationContext` 适用于大多数应用，且默认在启动时加载所有 Bean。

## 10.4 Starter 内部结构与 SPI 扩展点

### 🧩 Starter 内部结构

Spring Boot 提供了很多 **Starter** 依赖，用于快速集成常见的技术栈。每个 Starter 实际上是一个特定功能的自动配置类，通过 `spring.factories` 文件自动加载。

#### 示例：`spring.factories` 文件

```properties
org.springframework.boot.autoconfigure.EnableAutoConfiguration=\
com.example.demo.MyAutoConfiguration
```

------

### 📍 SPI 扩展点

Spring Boot 使用 **SPI（Service Provider Interface）** 来扩展功能，自动装配通过 `spring.factories` 文件进行管理，开发者可以通过自定义 Starter 来扩展应用。

#### 示例：自定义 Starter

```properties
org.springframework.boot.autoconfigure.EnableAutoConfiguration=\
com.example.starter.MyCustomAutoConfiguration
```

------

### 🧠 面试扩展

**面试官：**
 Spring Boot 的 Starter 是如何工作的？

**标准回答：**

> Spring Boot 的 Starter 是用于集成常见技术栈的工具，通过依赖和自动配置类的组合简化开发过程。每个 Starter 内部通过 `spring.factories` 文件声明需要自动配置的类，并在应用启动时自动加载。

### 🧠 面试扩展

**面试官：**
 你如何优化 Spring Boot 应用的启动性能？

**标准回答：**

> 可以通过以下方式优化 Spring Boot 启动性能：
>
> - 使用 **Lazy Initialization**（懒加载）来延迟 Bean 的初始化。
> - 排除不必要的自动配置类，减少启动时加载的 Bean 数量。
> - 使用 **AOT（Ahead-of-Time）编译**来提前编译代码，减少启动时的时间消耗。
> - 精简启动类，避免过多的应用初始化操作。

## 第11章：企业级项目实战（黑马点评 / 苍穹外卖）

### 11.1 黑马点评场景

- **Redis 缓存**：热点数据（如店铺、秒杀券）用 **@Cacheable** 或 **RedisTemplate** 缓存，降低 DB 压力；**缓存预热**：应用启动或定时任务中预加载热点 key；**分布式锁**：用 Redis **SET NX EX** 或 **Redisson** 做秒杀扣库存、防超卖。

### 11.2 苍穹外卖场景

- **定时任务**：**@Scheduled** 做订单超时取消、统计报表；**WebSocket**：订单状态变更推送给前端；**Spring Cache**：配合 Redis 做菜品、套餐等查询缓存，更新时 **@CacheEvict** 失效。

### 11.3 数据一致性与并发

- **缓存与 DB 一致**：先更新 DB 再删缓存（或 Canal 订阅 binlog 失效缓存）；**事务**：写操作加 **@Transactional**，避免脏读/重复扣减；**热点数据**：本地缓存 + 分布式锁限流，防止缓存击穿。

### 11.4 鉴权与 Session

- 分布式下不用 Session 存用户，改用 **JWT** 或 **Token + Redis**；拦截器/Filter 解析 token，将用户信息放 **ThreadLocal** 或 **SecurityContext**，保证同一请求内业务层可获取当前用户。

## 第12章：面试与项目延伸

### 12.1 高频面试题

- **自动装配**：@EnableAutoConfiguration + spring.factories + @ConditionalOnXxx；**配置加载**：application.yml/properties、Profile、外部化配置优先级；**IoC/AOP**：Spring 核心，Boot 在此基础上加自动配置与 Starter。

### 12.2 源码与启动流程

- **SpringApplication.run()**：创建 ApplicationContext、加载 Environment、执行 ApplicationContextInitializer、监听器、refresh()；**refresh()** 中加载 Bean、执行自动配置、启动内嵌容器。

### 12.3 项目优化与简历

- 优化：懒加载、排除无用自动配置、多数据源/缓存/异步合理使用；简历上写清楚技术栈、负责模块、解决的问题（如“基于 Redis 分布式锁解决秒杀超卖”）。

### 12.4 常见 Bug 与排查

- **Bean 冲突**：排除自动配置或 @Primary；**端口占用**：改 server.port 或杀进程；**配置不生效**：检查 Profile、配置优先级、拼写；**启动慢**：看 --debug 或 trace 日志，排除不必要 Starter。

### 12.5 Spring Boot 3.x 与 Jakarta

- 3.x 基于 **Jakarta EE 9+**，包名从 **javax.*** 改为 **jakarta.***（如 jakarta.servlet、jakarta.persistence）；需 JDK 17+；若依赖未升级需等兼容版本或继续用 2.x。
