## 1.1 Docker 的诞生与发展背景

### 💡 起源背景

- **传统部署痛点：**

  1. 环境差异大：开发环境、测试环境、生产环境不一致，出现“在我电脑上能跑”的问题。
  2. 依赖复杂：不同服务依赖不同版本的库（JDK、Redis、Nginx等），部署困难。
  3. 部署周期长：每次上线都需要重新配置环境，易出错、效率低。
  4. 资源浪费：虚拟机动辄数 GB，占用大量系统资源。

- **容器化理念的诞生：**

  > 容器 ≈ 轻量级的虚拟化技术
  >  2008年，Google 公开其内部容器管理系统 **Borg** 的思想，开启了容器化趋势。
  >  2013年，**Docker 公司（前 dotCloud）** 将容器技术标准化、开源化，极大简化了使用门槛。

### 🧭 Docker 发展历程

| 时间      | 事件                                                    |
| --------- | ------------------------------------------------------- |
| **2013**  | Docker 开源，基于 Linux 的 LXC 容器技术                 |
| **2014**  | 引入 Docker Hub，形成生态雏形                           |
| **2015**  | 发布 Docker Compose（多容器编排）                       |
| **2016**  | 推出 Docker Swarm 集群管理                              |
| **2017+** | Kubernetes 崛起，Docker 成为容器标准接口（OCI）的一部分 |
| **至今**  | Docker 已成为“云原生”技术栈的核心组件之一               |

## 1.2 虚拟机 vs 容器

### 🧱 虚拟机（VM）

- 在宿主机上运行一个完整的操作系统（如 CentOS），每个虚拟机拥有**独立的内核与资源**。
- 虚拟化层通过 **Hypervisor（虚拟机监控器）** 管理资源。

### 🧩 容器（Container）

- 共享宿主机的内核，只隔离运行环境。
- 容器使用 **Namespace + CGroup** 实现进程级隔离，资源开销更小。

### 🔍 核心对比

| 对比项   | 虚拟机（Virtual Machine）      | 容器（Docker Container）       |
| -------- | ------------------------------ | ------------------------------ |
| 启动速度 | 分钟级（需启动操作系统）       | 秒级（启动进程）               |
| 资源占用 | 高，每个虚拟机都有完整系统     | 低，共享宿主机内核             |
| 隔离性   | 强（硬件级）                   | 较强（进程级）                 |
| 镜像体积 | 几 GB                          | 几十 MB                        |
| 迁移性   | 依赖硬件与操作系统             | 跨平台、可移植                 |
| 使用场景 | 大型隔离系统、不同操作系统混合 | 微服务部署、快速交付、持续集成 |

> 🧠 面试记忆法：
>  “虚拟机隔离系统，容器隔离进程”
>  “虚拟机重，容器轻；虚拟机慢，容器快”。

## 1.3 Docker 的核心理念：一次构建，到处运行

> **Slogan：Build Once, Run Anywhere**

### 🌍 核心思想

通过将应用程序及其依赖打包为**镜像（Image）**，在任何装有 Docker 的主机上都能**快速运行一致的容器环境**。

### 📦 关键机制

1. **镜像（Image）**：应用的蓝图，包含运行所需环境与依赖。
2. **容器（Container）**：镜像的运行实例。
3. **仓库（Repository）**：集中存储镜像的地方。

### 🚀 实际意义

- 统一环境：消除“开发机 vs 生产机”的差异。
- 快速交付：镜像构建一次，多环境复用。
- 易于回滚：旧版本镜像随时恢复。
- 可重复性：任意主机部署结果一致。

------

## 1.4 Docker 的三大核心组件

### 🧩 1. 镜像（Image）

- **定义**：包含应用程序及运行环境的**静态模板**。

- **特点**：

  - 分层结构（Layered File System）
  - 只读、可复用、可缓存

- **示例**：

  ```bash
  docker pull redis:7.0
  docker images
  docker inspect redis:7.0
  ```

------

### 🧱 2. 容器（Container）

- **定义**：镜像的**运行实例**，包含独立文件系统与运行状态。

- **特性**：

  - 轻量级进程隔离
  - 可快速启动、停止、删除

- **常用命令**：

  ```bash
  docker run -d -p 6379:6379 redis:7.0
  docker ps -a
  docker exec -it <container_id> bash
  docker stop <container_id>
  ```

------

### 🏠 3. 仓库（Repository）

- **定义**：集中存放镜像的服务端。

- **分类**：

  - 公共仓库：Docker Hub、阿里云镜像服务
  - 私有仓库：Harbor、Registry

- **常用操作**：

  ```bash
  docker push myrepo/myapp:1.0
  docker pull myrepo/myapp:1.0
  ```

### 🔄 三者关系图：

```
+-------------------+
|   Dockerfile      |   ---> 构建生成
+-------------------+
          ↓
     [Image 镜像]
          ↓ 运行实例化
     [Container 容器]
          ↑
   存储在 [Repository 仓库]
```

## 1.5 Docker 的优势与适用场景

### 🏆 核心优势

| 类别       | 说明                              |
| ---------- | --------------------------------- |
| **轻量级** | 不需要完整操作系统，资源占用少    |
| **高性能** | 启动快、运行快、迁移方便          |
| **一致性** | 环境一致，消除“环境问题”          |
| **可移植** | 任意主机运行同一镜像              |
| **易集成** | 与 CI/CD、K8s、云原生生态高度融合 |

> CI/CD 是一套现代化的软件开发实践方法，通过高度自动化的工作流程，让软件能够持续、快速、可靠地交付给用户。
>
>  CI - Continuous Integration（持续集成）
>
> CD - 两个层面的含义：
>
> - a) Continuous Delivery（持续交付）
>   - **核心思想**：**自动化地准备好可发布的软件包。** 在持续集成的基础上，将代码自动构建、测试并打包成一个可以随时**手动**部署到生产环境的版本。
> - b) Continuous Deployment（持续部署）
>   - **核心思想**：**自动化地发布软件。** 这是持续交付的更高级阶段。只要代码通过所有自动化测试，系统就会**自动**将其部署到生产环境，无需任何人工干预。

### 🧰 典型使用场景

1. **微服务架构部署**
   - 每个服务一个容器，独立运行，互不干扰。
   - 示例：黑马点评中的 `Redis + MySQL + SpringBoot + Nginx` 多容器组合。
2. **持续集成 / 持续部署（CI/CD）**
   - Jenkins + Docker 实现自动化构建与发布。
3. **本地开发与测试环境统一**
   - 使用 Docker Compose 一键启动所有依赖服务。
4. **云端部署与弹性伸缩**
   - Kubernetes 集群中容器自动扩缩容。
5. **快速回滚与环境恢复**
   - 版本镜像管理使部署更可控。

## 1.6 Docker 与 DevOps、微服务的关系

### 🧱 DevOps：开发与运维一体化

- **DevOps 目标**：快速、稳定、高质量地交付软件。
- **Docker 的作用**：
  - 环境一致性 → 减少部署失败
  - 自动化构建 → 提升交付效率
  - 持续集成与持续部署（CI/CD） → 快速上线与回滚

🔧 **典型流程：**

```
代码提交 → Jenkins 自动构建 → Docker 打包镜像 → 推送仓库 → 服务器拉取镜像 → 容器化运行
```

------

### 🧩 微服务架构

- **微服务问题**：服务多、依赖复杂、部署频繁。
- **Docker 优势**：
  - 每个服务独立容器化部署
  - 易于组合、扩展与隔离
  - 与 Spring Cloud / Nginx / Redis 等完美配合

## 第2章：Docker 安装与环境配置

### 2.1 安装

- **Linux**：`curl -fsSL https://get.docker.com | sh` 或各发行版包管理；**Windows/macOS**：**Docker Desktop**（含 Engine、CLI、Compose）。**Docker Engine** 为内核，Desktop 为带 GUI 的桌面版。

### 2.2 配置与加速

- 配置文件 **daemon.json**（Linux 常为 /etc/docker/daemon.json）；**registry-mirrors** 配置国内镜像加速；**docker info** 验证、**docker run hello-world** 测试。

### 2.3 常见问题

- 权限：将用户加入 **docker** 组或 sudo；WSL2/Desktop 下网络与存储路径依文档配置。

------

## 第3章：镜像（Image）原理与操作

### 3.1 分层与缓存

- 镜像由**分层**组成（UnionFS），只读层 + 容器可写层；构建时**未改动的层可复用缓存**，加速构建。

### 3.2 常用命令

- **docker pull** 拉取、**docker images** 列表、**docker rmi** 删除、**docker inspect** 查看详情；**docker commit** 从容器打镜像（不推荐生产），**Dockerfile** 为推荐方式；**多阶段构建** 在 Dockerfile 中多 FROM，前一阶段产物拷贝到后一阶段，可显著减小最终镜像。

### 3.3 优化与清理

- 选小基础镜像（alpine）、合并 RUN、.dockerignore 减少上下文；**docker system prune -a** 清理未用镜像与缓存。

------

## 第4章：容器（Container）核心机制

### 4.1 生命周期

- **docker run** 创建并启动（-d 后台）；**docker start/stop/restart** 启停已有容器；**docker ps** 查看运行中，**docker ps -a** 含已停止；**docker exec -it 容器名 /bin/sh** 进入；**docker logs** 看日志；**docker rm** 删除容器。

### 4.2 隔离与资源

- **Namespace** 隔离进程、网络、挂载等；**CGroup** 限制 CPU、内存；**docker run --memory=512m --cpus=0.5** 限制资源。

### 4.3 持久化

- 容器删除后其可写层丢失；持久化用**数据卷（Volume）**或**绑定挂载**，见第 5 章。

------

## 第5章：数据卷与持久化存储

### 5.1 卷类型

- **匿名卷**：-v /容器路径，由 Docker 命名；**具名卷**：-v 卷名:/容器路径，便于复用；**绑定挂载**：-v 宿主机路径:/容器路径，直接映射目录。

### 5.2 使用

- **-v** 与 **--mount** 均可，--mount 类型更明确；**docker volume create** 创建具名卷；**docker volume inspect** 查看位置。MySQL 等数据目录务必挂载卷，避免容器删后数据丢失。

------

## 第6章：Docker 网络与通信机制

### 6.1 网络模式

- **bridge**（默认）：容器有独立 IP，通过宿主机端口映射对外；**host**：共用宿主机网络栈；**none**：无网络；**container**：共用某容器网络。

### 6.2 通信与端口

- **-p 宿主机端口:容器端口** 端口映射；**docker network create** 自定义网络，同网段容器可通过**容器名**互相访问；**--link** 已不推荐，用自定义网络即可。

------


## 第7章：Dockerfile 深入与镜像自动化构建

### 7.1 常用指令

- **FROM** 基础镜像；**RUN** 执行命令（可合并减少层）；**COPY/ADD** 复制文件；**WORKDIR** 工作目录；**ENV** 环境变量；**EXPOSE** 声明端口；**CMD/ENTRYPOINT** 容器启动命令（CMD 可被 run 参数覆盖，ENTRYPOINT 常作入口）。

### 7.2 构建与优化

- **docker build -t 镜像名:tag .**，`.` 为上下文，大目录用 **.dockerignore** 排除；**多阶段构建**：第一 FROM 编译，第二 FROM 只含运行时，COPY 前一阶段产物，大幅减小镜像。

### 7.3 Spring Boot 镜像

- 多阶段：第一阶段 Maven 打 jar，第二阶段 FROM eclipse-temurin:17-jre，COPY jar，ENTRYPOINT java -jar；或使用 **spring-boot-maven-plugin** 打可执行 jar 后 COPY 进镜像。

------

## 第8章：Docker Compose（多容器编排）

### 8.1 基本用法

- **docker-compose.yml** 定义 **services**（镜像、端口、环境变量、卷、网络）、**volumes**、**networks**；**docker compose up -d** 启动（或 docker-compose）；**down** 停止并删容器；**build** 构建本地镜像；同项目服务默认同网络，可用**服务名**互访。

### 8.2 实战

- 多服务（如 Nginx、Spring Boot、Redis、MySQL）各写一个 service，依赖用 **depends_on**；数据卷挂载到 MySQL/Redis 数据目录；端口映射暴露 Nginx 与后端；一键 **up** 启动全栈。

### 8.3 与 K8s

- Compose 适合单机/小规模多容器；K8s 适合集群、自愈、扩缩容、服务发现；可先用 Compose 本地联调，生产用 K8s。

------

## 第9章：Docker Registry 与镜像仓库

### 9.1 公共与私有

- **Docker Hub** 公共；国内可用**阿里云/腾讯云**镜像仓库；**私有**：**Harbor**（企业级，含鉴权、扫描）或官方 **registry** 镜像自建。

### 9.2 推送流程

- **docker tag 本地镜像 仓库地址/镜像:tag**；**docker login** 登录；**docker push** 推送；命名规范：仓库/项目/镜像:tag。

------

## 第10章：Docker 在项目中的实战应用

### 10.1 项目容器化

- 各服务编写 Dockerfile 或使用官方镜像（Redis、MySQL、Nginx）；**docker-compose** 编排，配置依赖与网络；Spring Boot 连 Redis/MySQL 时 host 填 **服务名**（Compose 网络内 DNS）。

### 10.2 CI/CD 与运维

- Jenkins/GitLab CI 中 **docker build**、**docker push** 到仓库；部署时 **docker pull**、**docker run** 或 **compose up**；**docker logs**、**restart policy**、**healthcheck** 做日志与自愈。

------

## 第11章：Docker 高级主题与生态扩展

### 11.1 安全与资源

- **Seccomp、AppArmor** 限制容器系统调用；**docker run --memory --cpus** 限制资源；镜像漏洞扫描用 **trivy**、**Harbor** 等。

### 11.2 Swarm 与 K8s

- **Docker Swarm** 为 Docker 自带集群模式，轻量；**Kubernetes** 生态更大，企业多用 K8s；容器镜像（OCI）通用，可同一镜像在 Compose/Swarm/K8s 中运行。

------

## 第12章：Docker 常见问题与面试题

### 12.1 排查思路

- **无法启动**：看 **docker logs**、**docker inspect**；**网络冲突**：改 -p 或改用其他端口；**挂载失败**：检查宿主机路径与权限；**构建失败**：看 Dockerfile 报错行、上下文与 .dockerignore；**空间不足**：**docker system prune** 清理。

### 12.2 面试要点

| 问题 | 要点 |
|------|------|
| Docker 与虚拟机区别？ | 容器共享内核、轻量、秒级启动；虚拟机独立内核、重量级。 |
| 镜像分层？ | UnionFS 分层只读 + 可写层；分层复用与缓存。 |
| Dockerfile 与 Compose？ | Dockerfile 定义单镜像构建；Compose 定义多容器编排与网络卷。 |
| 容器间通信？ | 自定义网络，容器名即主机名。 |
| 减小镜像？ | 多阶段构建、小基础镜像、合并 RUN、.dockerignore。 |
