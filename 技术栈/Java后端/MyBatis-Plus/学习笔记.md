# 🧩 第一章：MyBatis-Plus 概述与核心思想

------

> **本章在整体中解决什么问题**：建立对 MyBatis-Plus 的「第一印象」——它解决 MyBatis 哪些痛点、**与 MyBatis 如何兼容与分工**、核心模块（BaseMapper、Wrapper、拦截器）各做什么。学完本章再学**第二章**会动手搭 Spring Boot + Plus；**第三章 BaseMapper 与实体注解**、**第四章条件构造器**会展开本章提到的「单表 CRUD」与「链式条件」。

------

## 1.1 为什么需要 MyBatis-Plus：MyBatis 的不足

### （1）MyBatis 在实际开发中的痛点

| 痛点 | 说明 |
|------|------|
| **XML 与样板代码多** | 简单 CRUD 也要写 `<select>`、`<insert>`，重复率高，维护成本大。 |
| **手写 SQL 占主导** | 单表增删改查占业务大半，却要逐个写 SQL，易错且难复用。 |
| **分页需手写** | 原生 MyBatis 无自动分页，需手写 `LIMIT` 或引入 PageHelper 等。 |
| **通用能力缺失** | 无逻辑删除、乐观锁、自动填充等开箱即用能力，需自行实现。 |

------

### （2）ORM 与“半自动”的取舍

MyBatis 是**半自动 ORM**：SQL 手写、参数与结果映射由框架完成。灵活可控，但在**单表 CRUD** 场景下仍然繁琐。MyBatis-Plus 在**不替换 MyBatis** 的前提下，为单表场景提供“近乎全自动”的 CRUD，复杂 SQL 仍交给 MyBatis 处理。

> 💡 **一句话**：MyBatis-Plus 是“在 MyBatis 之上做增强”的工具，单表用 Plus，多表/复杂 SQL 用 MyBatis。

------

## 1.2 MyBatis-Plus 的定位与设计目标

| 设计目标 | 说明 |
|----------|------|
| **无侵入** | 不改变 MyBatis 核心，可混用；原有 Mapper、XML 保留，按需使用 Plus 能力。 |
| **简化 CRUD** | 提供 `BaseMapper<T>`，继承即拥有 insert、selectById、updateById、deleteById 等，无需写 XML。 |
| **条件构造** | 通过 `QueryWrapper`、`LambdaQueryWrapper` 等链式构建 WHERE 条件，替代手写动态 SQL。 |
| **分页与扩展** | 内置分页插件、乐观锁、逻辑删除、自动填充、代码生成器等，减少重复开发。 |

------

## 1.3 核心优势一览

| 优势 | 说明 |
|------|------|
| **BaseMapper** | 单表 CRUD 开箱即用，insert/selectById/updateById/deleteById/selectList 等，无需 XML。 |
| **Wrapper** | QueryWrapper、UpdateWrapper、Lambda 系列，链式 eq/like/in/orderBy，避免字段名硬编码。 |
| **分页** | `PaginationInnerInterceptor` 自动改写 SQL、统计 count，使用 `Page<T>` 即可。 |
| **逻辑删除** | `@TableLogic` 标注字段，删除变更新，查询自动加 `deleted=0`。 |
| **乐观锁** | `@Version` + 拦截器，更新时 WHERE version=旧值并 SET version+1，防并发覆盖。 |
| **自动填充** | `MetaObjectHandler` + `@TableField(fill)`，统一填充创建时间、更新时间等。 |
| **代码生成器** | 根据表结构生成 Entity、Mapper、Service、Controller 骨架。 |

------

## 1.4 架构设计与核心模块

### （1）整体架构

```
┌─────────────────────────────────────────────────────────────────┐
│  Service 层（可选）                                                │
│  继承 IService<T> / ServiceImpl<Mapper, Entity>，获得批量、分页等   │
└────────────────────────────┬────────────────────────────────────┘
                             │
                             ▼
┌─────────────────────────────────────────────────────────────────┐
│  Mapper 层                                                        │
│  继承 BaseMapper<T>，提供单表 CRUD；Wrapper 构造条件               │
└────────────────────────────┬────────────────────────────────────┘
                             │
                             ▼
┌─────────────────────────────────────────────────────────────────┐
│  MyBatis-Plus 拦截器（MybatisPlusInterceptor）                     │
│  分页、乐观锁、防全表更新删除等 InnerInterceptor                   │
└────────────────────────────┬────────────────────────────────────┘
                             │
                             ▼
┌─────────────────────────────────────────────────────────────────┐
│  MyBatis 原生：SqlSession → Executor → StatementHandler → JDBC   │
└─────────────────────────────────────────────────────────────────┘
```

------

### （2）核心模块说明

| 模块 | 职责 |
|------|------|
| **BaseMapper** | 内置单表 CRUD 的 MappedStatement，Mapper 继承后直接调用。 |
| **Wrapper** | 将 eq/like/in 等转为 SQL 片段，与 select/update 组合成完整 SQL。 |
| **InnerInterceptor** | 分页改写 SQL、乐观锁拼接 version 条件、BlockAttack 防无 where 更新/删除。 |
| **MetaObjectHandler** | 在 insert/update 时根据 `@TableField(fill)` 自动 set 字段值。 |

------

## 1.5 与 MyBatis 的兼容与分工

- **兼容**：Plus 基于 MyBatis，同一 Mapper 既可继承 `BaseMapper` 用 Plus 方法，也可在 XML 中写自定义 SQL；二者可混用。
- **分工**：单表 CRUD、简单条件查询用 BaseMapper + Wrapper；多表联查、复杂 SQL、统计用 MyBatis 的 XML。
- **依赖**：引入 `mybatis-plus-boot-starter` 会传递 MyBatis 与 MyBatis-Spring 依赖，一般无需单独再引 MyBatis。

------

## 1.6 面试高频

**Q：MyBatis 和 MyBatis-Plus 有什么区别？**

> MyBatis 是半自动 ORM，手写 SQL、框架做映射。MyBatis-Plus 在其基础上增强：提供 BaseMapper 通用 CRUD、Wrapper 条件构造、分页/乐观锁/逻辑删除/自动填充等，无侵入可混用；复杂 SQL 仍用 MyBatis 的 XML。

**Q：MyBatis-Plus 是如何增强 MyBatis 的？**

> 通过内置 MappedStatement 提供 BaseMapper 方法、通过拦截器实现分页与乐观锁、通过 Wrapper 在运行时拼接 SQL；与 MyBatis 共用同一套 SqlSession、Executor，只是多了插件与扩展点。

------

**本章小结**：MyBatis-Plus 是 MyBatis 的增强工具，在保留 MyBatis 灵活性的前提下，为单表 CRUD 提供 BaseMapper、Wrapper、分页、乐观锁、逻辑删除、自动填充等能力，无侵入、可混用，适合与项目内 MyBatis 笔记对照学习。

------

### 常见坑与注意点

| 现象 / 易错点 | 原因 | 怎么改 / 怎么记 |
|---------------|------|-----------------|
| 继承 BaseMapper 但方法报错或找不到 | 未加 **@MapperScan** 或扫描包路径不对，Mapper 未注册成 Bean | 启动类或配置类上 **@MapperScan("包路径")** 覆盖所有 Mapper 所在包。 |
| 实体字段与表列对不上（null、报错） | 表名列名**下划线**、实体**驼峰**未开启映射，或未写 **@TableName** / **@TableField** | 配置 **map-underscore-to-camel-case: true**；表名与实体不一致时用 **@TableName("表名")**。 |
| 和 MyBatis 混用时 XML 不生效 | mapper-locations 只配了 Plus 的或路径写错 | **mybatis-plus.mapper-locations** 要包含自定义 XML 所在路径，与 MyBatis 一致。 |

## 2.1 依赖与配置

```xml
</dependency>
```

Plus 已传递 MyBatis 与 MyBatis-Spring-Boot-Starter，无需再单独引入。需保证 Spring Boot、MyBatis 版本兼容（见官方文档）。

------

### （2）数据源与 Mapper 扫描

```yaml
spring:
  datasource:
    url: jdbc:mysql://localhost:3306/demo?useUnicode=true&characterEncoding=utf8
    username: root
    password: xxx
    driver-class-name: com.mysql.cj.jdbc.Driver

# 可选：MyBatis-Plus 全局配置
mybatis-plus:
  mapper-locations: classpath*:/mapper/**/*.xml
  type-aliases-package: com.example.entity
  configuration:
    map-underscore-to-camel-case: true
    log-impl: org.apache.ibatis.logging.slf4j.Slf4jImpl
  global-config:
    db-config:
      id-type: auto
      logic-delete-field: deleted
      logic-delete-value: 1
      logic-not-delete-value: 0
```

**@MapperScan**：在启动类或配置类上添加 `@MapperScan("com.example.mapper")`，扫描 Mapper 接口并注册为 Bean。

------

### （3）配置项说明

| 配置项 | 说明 |
|--------|------|
| **mapper-locations** | XML 路径，与 MyBatis 一致；Plus 方法可不写 XML。 |
| **type-aliases-package** | 实体包，XML 中可用短类名。 |
| **map-underscore-to-camel-case** | 下划线列名转驼峰。 |
| **id-type** | 主键类型：auto（数据库自增）、assign_id（雪花）、assign_uuid 等。 |
| **logic-delete-field** | 逻辑删除字段名；logic-delete-value / logic-not-delete-value 为删除/未删除时的值。 |

## 2.2 第一个 Mapper 与实体

**实体**：

```java
@TableName("tb_user")
public class User {
    @TableId(type = IdType.AUTO)
    private Long id;
    private String name;
    private Integer status;
    @TableLogic
    private Integer deleted;
}
```

**Mapper**：

```java
@Mapper
public interface UserMapper extends BaseMapper<User> {
    // 已拥有 insert、selectById、updateById、deleteById、selectList 等
}
```

**使用**：

```java
@Autowired
private UserMapper userMapper;

User user = userMapper.selectById(1L);
List<User> list = userMapper.selectList(null);  // 查全部
```

## 2.3 与 MyBatis 原生配置的兼容

- 若项目已有 `mybatis.mapper-locations`、`mybatis.configuration.*`，可逐步改为 `mybatis-plus.*`，或保留 mybatis 前缀，Plus 会兼容。
- 同一 Mapper 可既继承 `BaseMapper<User>` 又定义自定义方法，对应 XML 中的 `<select>` 等，互不冲突。

------

**本章小结**：通过 mybatis-plus-boot-starter 引入依赖，配置数据源与 `mybatis-plus.*`（含 global-config.db-config），使用 @MapperScan 扫描 Mapper。实体用 @TableName、@TableId、@TableLogic 等标注，Mapper 继承 BaseMapper<T> 即可使用单表 CRUD。

------

### 常见坑与注意点

| 现象 / 易错点 | 原因 | 怎么改 / 怎么记 |
|---------------|------|-----------------|
| 启动报错 No qualifying bean of type 'xxxMapper' | Mapper 未被扫描到，或未继承 BaseMapper | 确认 **@MapperScan** 包路径包含该 Mapper；接口要 **extends BaseMapper<实体>**。 |
| 逻辑删除配置了但删除仍是物理删除 | **logic-delete-field** 与实体字段名不一致，或未在实体字段上标 **@TableLogic** | 实体中逻辑删除字段要加 **@TableLogic**；global-config 里 logic-delete-field 与实体字段名一致。 |
| 主键插入为 null 或报错 | 未指定 **id-type** 或实体主键未标 **@TableId(type = IdType.AUTO)** | 数据库自增时实体主键加 **@TableId(type = IdType.AUTO)**；或 global-config 配 **id-type: auto**。 |

## 3.1 BaseMapper 提供的方法

### （1）插入与更新

| 方法 | 说明 |
|------|------|
| **insert(T entity)** | 插入一条，null 字段不参与插入（可配置）；主键可回填（IdType.AUTO/ASSIGN_ID）。 |
| **updateById(T entity)** | 按主键更新，null 字段不参与更新（可配置）。 |

------

### （2）删除

| 方法 | 说明 |
|------|------|
| **deleteById(Serializable id)** | 按主键删除；若配置逻辑删除，实际执行 UPDATE。 |
| **deleteByMap(Map<String, Object> columnMap)** | 按列等值条件删除。 |
| **delete(Wrapper<T> wrapper)** | 按 Wrapper 条件删除。 |
| **deleteBatchIds(Collection<?> idList)** | 按主键批量删除。 |

------

### （3）查询

| 方法 | 说明 |
|------|------|
| **selectById(Serializable id)** | 按主键查一条。 |
| **selectBatchIds(Collection<?> idList)** | 按主键批量查。 |
| **selectOne(Wrapper<T> wrapper)** | 按条件查一条，多条会抛异常。 |
| **selectList(Wrapper<T> wrapper)** | 按条件查列表。 |
| **selectPage(IPage<T> page, Wrapper<T> wrapper)** | 分页查询（需配置分页插件）。 |
| **selectCount(Wrapper<T> wrapper)** | 按条件统计条数。 |
| **selectMaps / selectObjs** | 返回 Map 列表或单列列表。 |

## 3.2 实体注解详解

### （1）@TableName

| 属性 | 说明 |
|------|------|
| **value** | 表名，默认类名转下划线（如 User → user）。 |
| **autoResultMap** | 是否自动构建 resultMap（复杂类型时有用）。 |

```java
@TableName("tb_user")
public class User { ... }
```

------

### （2）@TableId

| 属性 | 说明 |
|------|------|
| **value** | 主键列名，默认 `id`。 |
| **type** | 主键类型：AUTO（数据库自增）、INPUT（手动赋值）、ASSIGN_ID（雪花）、ASSIGN_UUID。 |

```java
@TableId(type = IdType.AUTO)
private Long id;
```

------

### （3）@TableField

| 属性 | 说明 |
|------|------|
| **value** | 列名，默认属性名转下划线。 |
| **exist** | 是否为表字段，false 表示不参与 SQL。 |
| **fill** | 自动填充策略：FieldFill.INSERT、UPDATE、INSERT_UPDATE。 |
| **select** | 是否参与 select 查询（默认 true），可设为 false 避免大字段查出。 |

```java
@TableField(exist = false)
private String remark;  // 非表字段

@TableField(fill = FieldFill.INSERT)
private LocalDateTime createTime;
```

------

### （4）@TableLogic

逻辑删除字段。删除时执行 UPDATE 将该字段设为 logic-delete-value；查询时自动加 `deleted = logic-not-delete-value`。

```java
@TableLogic
private Integer deleted;
```

------

### （5）@Version

乐观锁版本号。需配合 `OptimisticLockerInnerInterceptor` 使用，更新时 WHERE version=旧值并 SET version=version+1。

```java
@Version
private Integer version;
```

## 3.3 Service 层继承（可选）

```java
public interface IUserService extends IService<User> {}

@Service
public class UserServiceImpl extends ServiceImpl<UserMapper, User> implements IUserService {}
```

**IService** 提供 `save/update/remove/getById/list/page` 等，内部调用 BaseMapper；适合对业务层再做一层封装（如缓存、校验）。

## 3.4 面试要点

**Q：BaseMapper 有哪些常用方法？**

> insert、updateById、deleteById、selectById、selectList、selectPage、selectCount、deleteBatchIds、selectBatchIds 等，覆盖单表 CRUD 与条件查询。

**Q：@TableLogic 如何生效？**

> 配置 global-config.db-config 中的 logic-delete-field 与 value；实体字段加 @TableLogic。删除变更新，查询自动加未删除条件。

### 错误 vs 正确示例

| 场景 | 错误写法 / 现象 | 正确写法 / 做法 |
|------|-----------------|-----------------|
| **updateById 不生效** | 实体未设主键 id，Plus 不知道更新哪一行 | **updateById(entity)** 前必须给 **entity.setId(xxx)**，按主键更新。 |
| **insert 后 id 为 null** | 主键策略未配置或未用 @TableId(type = IdType.ASSIGN_ID/AUTO) | 主键用 **@TableId(type = IdType.AUTO)** 或 **ASSIGN_ID**；数据库自增用 AUTO，雪花用 ASSIGN_ID。 |
| **逻辑删除后仍查出已删数据** | 未配置 logic-delete-field 或实体未加 @TableLogic | 在 **global-config.db-config** 配 **logic-delete-field**、**logic-delete-value**、**logic-not-delete-value**；实体字段加 **@TableLogic**。 |

------

**本章小结**：BaseMapper 提供单表 CRUD 与条件查询，实体通过 @TableName、@TableId、@TableField、@TableLogic、@Version 与表结构对应；Service 可继承 IService/ServiceImpl 获得批量与分页等封装。

# 🌱 第四章：条件构造器 Wrapper

------

> **本章在整体中解决什么问题**：第三章会用 BaseMapper 的 selectList/selectPage；本章**掌握条件怎么拼**——QueryWrapper、UpdateWrapper、Lambda 系列，以及 eq/like/in/orderBy 与**动态条件**（condition 参数）。**第五章**分页会用到 **Page + Wrapper**；日常列表查询、按条件更新都依赖本章。

------

## 4.1 Wrapper 类型

| 类型 | 用途 |
|------|------|
| **QueryWrapper<T>** | 查询条件，eq/like/in/orderBy 等，用于 selectList/selectPage/selectCount。 |
| **UpdateWrapper<T>** | 更新条件 + set，可替代 entity 做部分字段更新。 |
| **LambdaQueryWrapper<T>** | 查询条件，用 Lambda 指定列（如 User::getName），避免字符串硬编码。 |
| **LambdaUpdateWrapper<T>** | 更新条件 + Lambda 指定列。 |

------

## 4.2 常用条件方法

| 方法 | 含义 | 示例 |
|------|------|------|
| **eq** | 等于 | `eq("status", 1)` |
| **ne** | 不等于 | `ne("deleted", 1)` |
| **gt / ge / lt / le** | 大于 / 大于等于 / 小于 / 小于等于 | `gt("age", 18)` |
| **like / likeLeft / likeRight** | 模糊（%val% / %val / val%） | `like("name", "张")` |
| **in / notIn** | 在/不在集合中 | `in("id", ids)` |
| **between** | 区间 | `between("age", 10, 20)` |
| **isNull / isNotNull** | 空/非空 | `isNotNull("phone")` |
| **orderByAsc / orderByDesc** | 排序 | `orderByDesc("create_time")` |
| **last** | 拼接末尾（慎用，有注入风险） | `last("LIMIT 1")` |

链式调用返回 this，可连续拼接。

------

## 4.3 示例：QueryWrapper 与 LambdaQueryWrapper

```java
// QueryWrapper：字符串列名
QueryWrapper<User> qw = new QueryWrapper<>();
qw.eq("status", 1)
  .like("name", "张")
  .orderByDesc("create_time");
List<User> list = userMapper.selectList(qw);

// LambdaQueryWrapper：方法引用，重构友好
LambdaQueryWrapper<User> lqw = new LambdaQueryWrapper<>();
lqw.eq(User::getStatus, 1)
   .like(User::getName, "张")
   .orderByDesc(User::getCreateTime);
userMapper.selectList(lqw);
```

------

## 4.4 UpdateWrapper 与 set

```java
UpdateWrapper<User> uw = new UpdateWrapper<>();
uw.eq("id", 1L).set("status", 2).set("update_time", LocalDateTime.now());
userMapper.update(null, uw);  // 第一个参数 entity 可为 null，由 wrapper 的 set 决定更新字段
```

------

## 4.5 动态条件

- **condition 参数**：`eq(condition, "col", val)`，condition 为 false 时不拼接该条件。
- **手动判断**：`if (name != null) wrapper.like("name", name);`，避免传 null 导致异常或语义不符。

```java
LambdaQueryWrapper<User> lqw = new LambdaQueryWrapper<>();
lqw.eq(status != null, User::getStatus, status)
   .like(StringUtils.hasText(name), User::getName, name);
```

------

## 4.6 面试要点

**Q：Wrapper 与 LambdaWrapper 区别？**

> 功能等价；Lambda 用方法引用指定列，避免字符串硬编码，重构时改字段名可自动同步，推荐 Lambda。

**Q：Wrapper 如何工作的？**

> Wrapper 内部维护条件表达式列表，执行时由 Plus 拼成 WHERE 片段，与 select/update 组合成完整 SQL，通过 MyBatis 执行。

### 错误 vs 正确示例

| 场景 | 错误写法 / 现象 | 正确写法 / 做法 |
|------|-----------------|-----------------|
| **last("...") 拼接用户输入** | `wrapper.last("LIMIT " + offset + "," + size)` 或拼接用户可控字符串，存在 **SQL 注入** | **last** 只拼固定或白名单值；分页用 **Page<T> + selectPage**，不要用 last 拼 LIMIT；若必须 last，参数白名单校验。 |
| **条件为 null 仍拼接** | `eq("name", name)` 当 name 为 null 时可能拼出 `name = null` 或异常 | 用 **condition 参数**：`eq(StringUtils.hasText(name), "name", name)` 或 **Lambda 版** `eq(name != null, User::getName, name)`。 |
| **UpdateWrapper 更新全表** | 未加 where 条件或条件恒真，导致全表被更新 | 使用 **BlockAttackInnerInterceptor** 防无 where 更新/删除；业务上 **update(wrapper)** 前确保 **wrapper** 有 **eq/in** 等条件。 |

------

**本章小结**：QueryWrapper/UpdateWrapper 用字符串列名；Lambda 系列用方法引用。eq/like/in/orderBy 等链式拼接；动态条件可用 condition 参数或 if 判断。UpdateWrapper 可替代 entity 做 set 更新。

------

# 🌱 第五章：分页插件与查询优化

------

> **本章在整体中解决什么问题**：第四章会用 Wrapper 做条件查询；本章**把分页接上**——配置 **PaginationInnerInterceptor**、使用 **Page<T> + selectPage**、count 优化与深分页注意点。列表接口、管理后台分页都依赖本章；**第六章**会讲自动填充与乐观锁，与分页无直接依赖。

------

## 5.1 分页配置与使用

### （1）注册分页插件

```java
@Configuration
public class MybatisPlusConfig {
    @Bean
    public MybatisPlusInterceptor mybatisPlusInterceptor() {
        MybatisPlusInterceptor interceptor = new MybatisPlusInterceptor();
        interceptor.addInnerInterceptor(new PaginationInnerInterceptor(DbType.MYSQL));
        return interceptor;
    }
}
```

------

### （2）分页查询

```java
Page<User> page = new Page<>(1, 10);  // 当前页 1，每页 10 条
userMapper.selectPage(page, queryWrapper);
// 数据
List<User> records = page.getRecords();
// 总条数
long total = page.getTotal();
// 总页数
long pages = page.getPages();
```

------

## 5.2 count 与性能

- **默认**：分页会先执行 `COUNT(*)` 再查当前页数据；大表 count 可能较慢。
- **关闭 count**：`page.setSearchCount(false)`，仅查当前页数据，不查总数；适合“只取一页、不关心总数”的场景。
- **深分页**：`LIMIT 100000, 10` 会扫描大量行，建议限制最大页码或改用游标、延迟关联等方式。

------

## 5.3 面试要点

**Q：MyBatis-Plus 分页如何实现？**

> 通过 PaginationInnerInterceptor 拦截 Executor，在原有 SQL 外包装一层 count 查询和 LIMIT，结果封装到 IPage 的 records 与 total 中。

------

**本章小结**：配置 MybatisPlusInterceptor 并添加 PaginationInnerInterceptor 后，使用 Page<T> 与 selectPage 即可分页；可按需关闭 count 或优化深分页。

------

# 🌱 第六章：自动填充与乐观锁

------

> **本章在整体中解决什么问题**：前面章节完成单表 CRUD、Wrapper、分页；本章**补上两类通用能力**——**自动填充**（创建/更新时间、操作人等）与**乐观锁**（高并发更新防覆盖）。与 **第五章** 分页、**第七章** 多表无直接依赖；**第八章** 会提到乐观锁拦截器在插件链中的位置。

------

## 6.1 自动填充

### （1）实体标注

```java
@TableField(fill = FieldFill.INSERT)
private LocalDateTime createTime;
@TableField(fill = FieldFill.INSERT_UPDATE)
private LocalDateTime updateTime;
```

------

### （2）实现 MetaObjectHandler

```java
@Component
public class MyMetaObjectHandler implements MetaObjectHandler {
    @Override
    public void insertFill(MetaObject metaObject) {
        this.strictInsertFill(metaObject, "createTime", LocalDateTime.class, LocalDateTime.now());
        this.strictInsertFill(metaObject, "updateTime", LocalDateTime.class, LocalDateTime.now());
    }
    @Override
    public void updateFill(MetaObject metaObject) {
        this.strictUpdateFill(metaObject, "updateTime", LocalDateTime.class, LocalDateTime.now());
    }
}
```

注册为 Spring Bean 即可；insert/update 时 Plus 会调用对应 fill 方法。

------

## 6.2 乐观锁

### （1）实体与配置

```java
@Version
private Integer version;
```

配置类中在 **MybatisPlusInterceptor** 里添加：

```java
interceptor.addInnerInterceptor(new OptimisticLockerInnerInterceptor());
```

------

### （2）更新逻辑

更新时 Plus 会生成：`UPDATE tb_user SET ..., version = version + 1 WHERE id = ? AND version = ?`。若影响行数为 0，表示并发下 version 已被修改，可重试或提示冲突。

------

## 6.3 面试要点

**Q：乐观锁原理？**

> 实体加 @Version 字段，更新时 WHERE 条件带 version=旧值，SET version=version+1；影响行数为 0 表示冲突，需重试或提示。

**Q：自动填充如何实现？**

> 实现 MetaObjectHandler，在 insertFill/updateFill 中根据 MetaObject 判断字段是否有 @TableField(fill)，有则 set 值；注册为 Bean，Plus 在 insert/update 前会调用。

------

**本章小结**：自动填充通过 MetaObjectHandler + @TableField(fill) 在 insert/update 时统一赋值；乐观锁通过 @Version + OptimisticLockerInnerInterceptor 实现“先查后改”的并发控制。

------

# 🌱 第七章：多表操作与自定义 SQL

------

> **本章在整体中解决什么问题**：BaseMapper 与 Wrapper 只针对**单表**；本章说明**多表联查**无内置 Join，需用 **MyBatis 的 XML 或注解** 写自定义 SQL，并与 Plus **在同一 Mapper 中混用**。**第六章** 自动填充/乐观锁与本章独立；**第八章** 插件机制是 Plus 分页/乐观锁的底层，与本章「自定义 SQL 执行」同属 MyBatis 体系。

------

## 7.1 多表与自定义 SQL

- BaseMapper 与 Wrapper **只针对单表**；多表联查需在 Mapper 中定义方法，对应 **XML 或 @Select 注解** 写 SQL。
- 参数可用 `@Param` 传入；若需条件构造，可传入 Wrapper（XML 中用 `<script>` 或 Plus 提供的 `ew.customSqlSegment` 等，需注意 SQL 注入）。

------

## 7.2 与 MyBatis 混用

同一 Mapper 既可继承 `BaseMapper<User>`，也可定义：

```java
List<OrderVO> selectOrderWithUser(@Param("userId") Long userId);
```

对应 XML：

```xml
<select id="selectOrderWithUser" resultType="OrderVO">
    SELECT o.*, u.name AS user_name
    FROM t_order o
    LEFT JOIN tb_user u ON o.user_id = u.id
    WHERE o.user_id = #{userId}
</select>
```

复杂查询、多表、统计用 XML；简单 CRUD 用 Plus。

### 多表与混用示例（同一 Mapper）

**Mapper 接口**：继承 BaseMapper，同时定义自定义方法：

```java
public interface OrderMapper extends BaseMapper<Order> {
    // Plus：单表 CRUD 直接用 baseMapper 的方法
    // 自定义：多表联查
    List<OrderVO> selectOrderWithUser(@Param("userId") Long userId);
}
```

**XML**（与 Plus 共存，同一 namespace）：

```xml
<mapper namespace="com.example.mapper.OrderMapper">
    <!-- Plus 的 BaseMapper 方法由 Plus 内置，无需在此写 -->
    <select id="selectOrderWithUser" resultType="OrderVO">
        SELECT o.id, o.order_no, o.user_id, u.name AS user_name
        FROM t_order o
        LEFT JOIN tb_user u ON o.user_id = u.id
        WHERE o.user_id = #{userId}
    </select>
</mapper>
```

**调用**：`orderMapper.selectById(1L)` 走 Plus；`orderMapper.selectOrderWithUser(userId)` 走 XML。

------

**本章小结**：多表与复杂 SQL 用 MyBatis 的 XML 或注解；Plus 负责单表 CRUD 与条件构造，二者在同一 Mapper 中可并存。

------

# 🌱 第八章：代码生成器与插件机制

------

> **本章在整体中解决什么问题**：前面章节会用 BaseMapper、Wrapper、分页、乐观锁；本章讲**代码生成器**（快速生成单表 CRUD 骨架）与 **Plus 内置插件**（分页、乐观锁、防全表更新）及自定义 Interceptor。理解插件顺序与 BlockAttack 有助于排查「为什么全表被更新」等问题。

------

## 8.1 代码生成器（AutoGenerator）

- 使用 **AutoGenerator** 配置数据源、包路径、策略（表前缀、命名、作者），可生成 Entity、Mapper、Mapper XML、Service、Controller。
- 模板可用 Velocity 或 Freemarker；通过 **StrategyConfig**、**PackageConfig**、**TemplateConfig** 控制输出。适合新表快速搭建 CRUD 骨架，生成后按业务修改。

------

## 8.2 内置插件与顺序

通过 **MybatisPlusInterceptor.addInnerInterceptor** 按需添加：

| 插件 | 作用 |
|------|------|
| **PaginationInnerInterceptor** | 分页，改写 SQL、count。 |
| **OptimisticLockerInnerInterceptor** | 乐观锁，拼接 version 条件。 |
| **BlockAttackInnerInterceptor** | 防止全表 update/delete 无 where。 |

------

## 8.3 自定义插件

实现 MyBatis 的 **Interceptor**，在 **@Intercepts** 中指定拦截 **Executor/StatementHandler** 等；可做 SQL 日志、慢 SQL 统计、租户过滤等；与 Plus 内置插件一起使用时注意添加顺序。

### 常见坑与注意点

| 现象 / 易错点 | 原因 | 怎么改 / 怎么记 |
|---------------|------|-----------------|
| 分页不生效、未拼 LIMIT | 未配置 **MybatisPlusInterceptor** 或未添加 **PaginationInnerInterceptor** | 在 **@Configuration** 里 **@Bean MybatisPlusInterceptor**，并 **addInnerInterceptor(new PaginationInnerInterceptor())**。 |
| 乐观锁更新影响行数始终 0 | 未加 **OptimisticLockerInnerInterceptor**，或 entity 的 version 未从数据库查出就更新 | 拦截器中添加 **OptimisticLockerInnerInterceptor**；**updateById** 前先 **selectById** 拿到最新 version 再 set 回 entity。 |
| 误执行全表 update/delete | Wrapper 未加条件或条件写错 | 使用 **BlockAttackInnerInterceptor** 防止无 where 的 update/delete；业务上保证 wrapper 有 eq 等条件。 |
| 多个 InnerInterceptor 顺序导致行为异常 | 分页、乐观锁、BlockAttack 顺序影响执行流程 | 一般顺序：**PaginationInnerInterceptor** → **OptimisticLockerInnerInterceptor** → **BlockAttackInnerInterceptor**（先分页再乐观锁再防全表）。 |

------

**本章小结**：代码生成器可快速生成单表 CRUD 骨架；Plus 通过 InnerInterceptor 提供分页、乐观锁、防全表更新等；自定义逻辑可基于 MyBatis Interceptor 扩展。

------

# 🌱 第九章：与 Spring Boot 整合与实战要点

------

> **本章在整体中解决什么问题**：前面各章已讲 Plus 的能力与配置；本章汇总**与 Spring Boot 整合**后的配置、事务、实战场景（缓存、自动填充、分页、乐观锁、迁移），以及**多数据源时的注意点**。与 **第八章** 插件、**第十章** 源码可结合理解。

------

## 9.1 配置与事务

- Mapper 与 Service 由 Spring 管理；**@Transactional** 与 MyBatis 一致，同一事务内共用 Connection。
- **mybatis-plus.mapper-locations**、**global-config.db-config** 在 yml 中配置；逻辑删除、主键类型、驼峰等一次配置全局生效。

------

## 9.2 实战场景小结

| 场景 | 建议 |
|------|------|
| **缓存** | Wrapper 查 DB，结果写 Redis；更新/删除时失效缓存。 |
| **自动填充** | 创建/更新时间、操作人用 MetaObjectHandler。 |
| **分页** | Page + Wrapper 做列表；大表可关 count 或限制深度。 |
| **乐观锁** | 订单状态、库存扣减等高并发更新用 @Version。 |
| **迁移** | 老项目新 Mapper 继承 BaseMapper，老 XML 保留；复杂 SQL 继续用 MyBatis。 |

### 多数据源注意点

- **多数据源**时需配置多份 **DataSource**、**SqlSessionFactory**、**TransactionManager**；Mapper 按包或 **sqlSessionFactoryRef** 绑定不同 Factory（与 MyBatis 笔记第八章一致）。
- **@Transactional** 需指定 **transactionManager**，否则用 @Primary 的，可能误走主库。
- Plus 的 **全局配置**（如 logic-delete、id-type）在**每个 SqlSessionFactory** 上均可单独指定，若主从库策略不同，在对应 FactoryBean 的 globalConfig 中分别配置。
- **分页插件**、**乐观锁** 等 InnerInterceptor 会作用于该 SqlSessionFactory 下的所有 Mapper，多数据源下每个 Factory 各有一套拦截器即可。

------

**本章小结**：Plus 与 Spring Boot 整合后配置简单；事务、缓存、分页、乐观锁、自动填充在实战中组合使用；与 MyBatis 混用可平滑迁移。

------

# 🌱 第十章：源码与执行机制（进阶）

------

## 10.1 拦截链与 BaseMapper

- Plus 通过 MyBatis 的 **Interceptor** 在 **Executor** 层拦截；分页插件改写 SQL、统计 count；乐观锁插件拼接 version。
- **BaseMapper** 的方法对应 Plus 内置的 **MappedStatement**，在启动时注入到 Configuration；调用时走 MyBatis 的 SqlSession → Executor → StatementHandler → JDBC 流程。

------

## 10.2 Wrapper 与 SQL 构建

- Wrapper 将 eq/like 等转为内部表达式（如 `ParamNameValuePair`、`Segment`）；执行时由 **SqlScript**、**AbstractWrapper** 等拼成 WHERE 片段，与 select/update 组合成完整 SQL 传给 MyBatis。

------

**本章小结**：Plus 在 MyBatis 之上通过插件与内置 MappedStatement 提供能力；Wrapper 在运行时生成 SQL 片段；理解此流程有助于排查问题与面试作答。

## 11.2 核心能力

| 问题 | 回答要点 |
