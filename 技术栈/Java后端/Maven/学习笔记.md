## 1.1 Maven 的起源与发展

### （1）背景与痛点

在早期 Java 项目开发中（J2EE 时代）：

- **依赖管理混乱**：开发者需要手动下载 `.jar` 包并维护复杂的 `lib` 目录
- **版本冲突频发**：不同模块需要的依赖版本不一致，容易出现 "Jar Hell"
- **构建复杂、不可复现**：手动编译、打包、部署命令多且容易出错
- **缺乏统一标准**：不同团队的项目结构、构建脚本不统一，协作困难

👉 为了解决这些问题，**Apache Software Foundation** 于 **2004 年发布 Maven**，目标是提供：

> "一个标准化的项目构建、依赖管理与生命周期框架"

------

### （2）发展时间线

| 年份 | 版本 | 特性亮点 |
| ---- | ---- | -------- |
| 2002-2004 | Maven 1.x | 使用 `project.xml` 管理依赖，结构简单但扩展性弱 |
| 2005 | Maven 2.x | 引入 `POM.xml`，标准化依赖管理与生命周期模型 |
| 2010 | Maven 3.x | 性能大幅提升，支持多模块构建、增量构建 |
| 2023（预览） | Maven 4.x | 完全支持 BOM 统一管理、增强缓存与并行构建 |

> ✅ Maven 已成为 Java 项目的标准构建工具，被几乎所有企业级项目采用。

------

### （3）核心目标：构建自动化与可复现

Maven 希望开发者通过一句命令完成全部操作：

```bash
mvn clean package
```

即可实现：

> 清理旧文件 → 编译源代码 → 执行测试 → 打包 jar/war → 安装到本地仓库

------

### （4）构建自动化的三大支柱

| 模块 | 含义 | 示例 |
| ---- | ---- | ---- |
| **依赖管理** | 自动下载所需 jar 包及其传递依赖 | `mvn dependency:resolve` |
| **生命周期** | 定义从编译到部署的完整流程 | `clean → compile → test → package → install → deploy` |
| **插件机制** | 通过插件扩展功能 | 如：`maven-compiler-plugin`、`maven-surefire-plugin` |

------

> 💡 **可复现构建**：只要有 `pom.xml`，任何人在任何机器上执行 `mvn clean install` 都能得到相同结果，即构建标准化 + 环境无关性。

------

## 1.2 Maven 与 Ant、Gradle 的对比

### （1）工具对比

| 特性 | Ant | Maven | Gradle |
| ---- | --- | ----- | ------ |
| **配置文件** | `build.xml`（手写命令） | `pom.xml`（声明式） | `build.gradle`（Groovy/Kotlin） |
| **依赖管理** | 手动 | 自动（中央仓库） | 自动（支持本地/远程仓库） |
| **构建模型** | 命令式（告诉怎么做） | 声明式（告诉要什么） | 混合式（声明 + DSL） |
| **扩展性** | 较弱 | 中等 | 强（支持自定义任务） |
| **构建性能** | 慢 | 一般 | 快（增量构建） |
| **主流度** | 过时 | 企业主流 | 新兴趋势（Android 官方） |

------

> 💬 **一句话**：Ant 告诉“怎么做”，Maven 告诉“要什么”，Gradle 既可声明又可编程。Spring Boot 默认 Maven 因生态成熟、插件完善、便于 CI/CD。

------

## 1.3 Maven 的核心概念

### （1）POM（Project Object Model）

> **一句话**：**POM** 是 Maven 项目的「说明书」——用 **pom.xml** 描述项目是谁（坐标）、要哪些依赖、怎么构建（插件与生命周期），一切构建行为都从 POM 推导出来。
> **为什么**：没有统一描述，依赖和构建步骤就得手写、难以复现；POM 声明式写清楚「要什么」，Maven 负责「怎么做」。

- 核心配置文件：`pom.xml`
- 描述项目的基本信息、依赖、构建方式、插件等

```xml
<project>
  <modelVersion>4.0.0</modelVersion>
  <groupId>com.example</groupId>
  <artifactId>demo</artifactId>
  <version>1.0-SNAPSHOT</version>

  <dependencies>
    <dependency>
      <groupId>org.springframework.boot</groupId>
      <artifactId>spring-boot-starter-web</artifactId>
      <version>2.7.3</version>
    </dependency>
  </dependencies>
</project>
```

作用：让项目构建信息可被自动识别、共享、继承。

------

### （2）坐标（Coordinates）

每个依赖都由唯一坐标标识：

```xml
<groupId>org.springframework.boot</groupId>
<artifactId>spring-boot-starter-web</artifactId>
<version>2.7.3</version>
```

相当于：

> Maven 的"身份证"
> → 表示组织 + 模块 + 版本

------

### （3）依赖管理

Maven 自动解析依赖树、下载 jar，支持传递依赖、作用域、版本仲裁。`mvn dependency:tree` 可查看依赖关系、分析冲突。

------

### （4）生命周期

Maven 定义了项目从创建到部署的标准阶段：

| 生命周期 | 说明 | 常用命令 |
| -------- | ---- | -------- |
| **clean** | 清理构建输出 | `mvn clean` |
| **compile** | 编译源码 | `mvn compile` |
| **test** | 执行单元测试 | `mvn test` |
| **package** | 打包 jar/war | `mvn package` |
| **install** | 安装到本地仓库 | `mvn install` |
| **deploy** | 发布到远程仓库 | `mvn deploy` |

👉 各阶段自动按顺序执行，无需手写构建逻辑

### 常见坑与注意点

| 现象 / 易错点 | 原因 | 怎么改 / 怎么记 |
|---------------|------|-----------------|
| 依赖下载很慢或超时 | 默认连国外中央仓库，网络不稳定 | 在 **settings.xml** 里配置国内**镜像**（如阿里云），`<mirrorOf>central</mirrorOf>`。 |

## 2.1 Maven 的下载与安装

- **官网**：[https://maven.apache.org/download.cgi](https://maven.apache.org/download.cgi)，下载 Binary zip archive
- **Windows**：解压到无中文路径（如 `D:\Program Files\apache-maven-3.9.x`）
- **Linux / macOS**：解压后可用 `sudo mv` 放到 `/opt` 或用户目录，或使用包管理器：`apt install maven` / `brew install maven`

------

## 2.2 环境变量配置

| 变量名 | 说明 | 示例值 |
| ------ | ---- | ------ |
| **MAVEN_HOME** | Maven 安装根目录 | `D:\Program Files\apache-maven-3.9.6` |
| **PATH** | 追加 `%MAVEN_HOME%\bin`（Windows）或 `$MAVEN_HOME/bin`（Linux/macOS） | 便于命令行直接使用 `mvn` |

配置后**重新打开终端**再验证

------

## 2.3 验证安装

```bash
mvn -v
```

应输出 Maven 版本、Java 版本等信息

------

## 2.4 Maven 安装目录结构

| 目录/文件 | 说明 |
| --------- | ---- |
| **bin/** | 可执行脚本：`mvn`、`mvn.cmd` 等 |
| **boot/** | 内置类加载器（plexus-classworlds） |
| **conf/** | 全局配置，其中 **settings.xml** 可复制到 `~/.m2/` 做用户级配置 |
| **lib/** | Maven 运行依赖的 jar 包 |

------

## 2.5 本地仓库与中央仓库

- **本地仓库**：默认在用户目录下的 `.m2/repository`，所有下载的依赖与安装的模块都存放在此
- **中央仓库**：Maven Central，默认从 [https://repo.maven.apache.org/maven2](https://repo.maven.apache.org/maven2) 拉取依赖
- 可在 `settings.xml` 中修改 `<localRepository>` 指定本地仓库路径，或配置 `<mirror>` 使用国内镜像加速

### （1）国内镜像配置示例

在 `~/.m2/settings.xml` 的 `<mirrors>` 中配置（以阿里云为例）：

```xml
<mirror>
  <id>aliyun</id>
  <mirrorOf>central</mirrorOf>
  <name>Aliyun Maven</name>
  <url>https://maven.aliyun.com/repository/public</url>
</mirror>
```

**本章小结**：Maven 需配置 JAVA_HOME，安装后设 MAVEN_HOME 与 PATH；本地仓库默认 `~/.m2/repository`，可通过 settings.xml 配置镜像与 localRepository。验证用 `mvn -v`。

### 常见坑与注意点

| 现象 / 易错点 | 原因 | 怎么改 / 怎么记 |
|---------------|------|-----------------|
| 命令行输 `mvn -v` 报错「不是内部或外部命令」 | **PATH** 未包含 Maven 的 **bin** 或未重启终端 | 确认 **MAVEN_HOME** 指向解压目录，**PATH** 中追加 `%MAVEN_HOME%\bin`（Windows）或 `$MAVEN_HOME/bin`，重新开终端。 |
| 安装路径含中文或空格 | Maven 或 JDK 对部分路径敏感 | 解压到**无中文、无空格**的目录（如 `D:\dev\apache-maven-3.9.6`）。 |
| 依赖下载失败 401/403 或一直 pending | 未配镜像、仓库地址错、或私服鉴权失败 | 检查 **settings.xml** 的 **mirror** 与 **repository**；若用私服，核对 **id** 与 **server** 里的账号密码。 |

# 🌱 第三章：Maven 项目结构与 POM 深入解析

------

## 3.1 Maven 标准目录结构

```
项目根目录/
├── pom.xml
├── src/
│   ├── main/
│   │   ├── java/           # 主代码
│   │   └── resources/      # 主资源（配置文件等）
│   └── test/
│       ├── java/           # 测试代码
│       └── resources/      # 测试资源
└── target/                 # 构建输出（可删，由 mvn clean 清理）
```

约定优于配置：无需在 POM 里写死路径，Maven 与 IDE 均能识别

------

## 3.2 POM 概念回顾

POM（Project Object Model）即项目对象模型，用 `pom.xml` 描述项目身份、依赖、构建方式与插件，是 Maven 一切行为的基础

------

## 3.3 POM.xml 关键标签

| 标签 | 含义 |
| ---- | ---- |
| **groupId** | 组织/公司维度，常为包名反写（如 `com.company`） |
| **artifactId** | 项目/模块名，仓库中对应目录名 |
| **version** | 版本号，`SNAPSHOT` 表示开发中、可被更新 |
| **packaging** | 打包类型：`jar`、`war`、`pom`（仅做父 POM 时） |
| **parent** | 继承父 POM，可继承依赖管理、插件、属性等 |
| **dependencies** | 当前项目直接依赖列表 |
| **dependencyManagement** | 统一版本与 scope，子模块引用时不写 version |
| **build** | 构建配置：源码编码、输出目录、插件等 |
| **plugins** | 在 `<build><plugins>` 中声明插件及参数 |

------

## 3.4 父子工程与继承

- **父 POM**：`packaging` 为 `pom`，不产出 jar/war，仅做依赖管理、插件管理、属性定义
- **子模块**：通过 `<parent>` 指定父 POM 的 groupId、artifactId、version 与相对路径（若父与子同仓库）
- 子模块可**继承**父的 `dependencyManagement`、`properties`、部分 `build` 配置，无需重复写版本号

### （1）父 POM 示例

```xml
<project>
  <modelVersion>4.0.0</modelVersion>
  <groupId>com.example</groupId>
  <artifactId>parent</artifactId>
  <version>1.0.0</version>
  <packaging>pom</packaging>
  
  <dependencyManagement>
    <dependencies>
      <dependency>
        <groupId>org.springframework.boot</groupId>
        <artifactId>spring-boot-dependencies</artifactId>
        <version>2.7.18</version>
        <type>pom</type>
        <scope>import</scope>
      </dependency>
    </dependencies>
  </dependencyManagement>
</project>
```

------

## 3.5 多模块与聚合（Aggregator）

- **聚合**：父 POM 的 `<modules>` 列出子模块目录，在父目录执行 `mvn clean install` 会按依赖顺序依次构建各子模块
- 多模块项目 = **继承（parent）** + **聚合（modules）**：父统一管理版本与构建，子只声明自己需要的依赖与代码

### （2）聚合示例

```xml
<modules>
  <module>module-a</module>
  <module>module-b</module>
  <module>module-c</module>
</modules>
```

------

### （2）面试要点

**Q：Maven 多模块项目的优势是什么？**

**标准回答：**

> Maven 多模块项目的优势包括：
> 1. 代码按业务或层级拆分，职责清晰
> 2. 依赖在模块间显式声明，便于复用与版本统一
> 3. 父 POM 统一管理依赖版本、插件、属性，减少重复与冲突
> 4. 可单独构建、测试、发布某一模块，或一次性聚合构建整个工程

------

**本章小结**：Maven 约定目录结构（src/main、src/test、target）；POM 用 groupId、artifactId、version、parent、dependencies、dependencyManagement、build 等描述项目与构建。父子工程通过 parent 继承、通过 modules 聚合，子模块可继承版本与插件配置。

### 常见坑与注意点

| 现象 / 易错点 | 原因 | 怎么改 / 怎么记 |
|---------------|------|-----------------|
| 子模块找不到父 POM | parent 的 relativePath 错或父未 install 到本地 | 父与子同仓库时 **relativePath** 默认 **../pom.xml**；父单独仓库时先 **mvn install** 父工程，或 relativePath 指对路径。 |
| 聚合构建顺序错或某模块未参与 | modules 顺序与依赖关系不符，或子模块未在父目录下 | **modules** 中顺序影响构建顺序；子模块目录需与 **module** 名一致且在父 POM 同目录下。 |
| 子模块写了 version 与父 dependencyManagement 不一致 | 子若显式写 version 会覆盖继承的版本 | 子模块依赖**不写 version**，由父 **dependencyManagement** 统一管理；只在特殊需求时子模块显式写 version。 |

- **下一章**：第四章 **依赖管理** 会用到坐标、scope、dependencyManagement 与冲突解决。

# 🌱 第四章：依赖管理机制详解

------

## 4.1 Maven 坐标体系

依赖由 **groupId + artifactId + version**（+ packaging + classifier）唯一确定，对应仓库路径：

```
groupId 按 / 分隔 / artifactId / version / artifactId-version.packaging
```

### （1）坐标示例

```xml
<dependency>
  <groupId>org.springframework.boot</groupId>
  <artifactId>spring-boot-starter-web</artifactId>
  <version>2.7.18</version>
  <packaging>jar</packaging> <!-- 可选，默认 jar -->
  <classifier>sources</classifier> <!-- 可选，如 sources、javadoc -->
</dependency>
```

------

## 4.2 依赖范围（scope）

| scope | 编译 | 测试 | 运行 | 传递 | 典型用途 |
| ----- | ---- | ---- | ---- | ---- | -------- |
| **compile** | ✓ | ✓ | ✓ | ✓ | 默认，业务依赖 |
| **provided** | ✓ | ✓ | ✗ | ✗ | Servlet API、Lombok（容器或编译期提供） |
| **runtime** | ✗ | ✓ | ✓ | ✓ | JDBC 驱动、日志实现 |
| **test** | ✗ | ✓ | ✗ | ✗ | JUnit、Mockito |
| **system** | ✓ | ✓ | ✓ | ✗ | 本地 jar，不推荐，不利于协作 |
| **import** | 仅用于 dependencyManagement 的 BOM | 继承版本，不引入依赖 |

### （2）面试要点

**Q：Maven 依赖的 scope 有哪些？区别是什么？**

**标准回答：**

> Maven 依赖的 scope 主要有 compile（默认，全阶段可用）、provided（编译和测试可用，运行时由容器提供）、runtime（测试和运行时可用）、test（仅测试可用）、system（本地 jar）、import（仅用于 BOM）。
> scope 决定了依赖在哪个构建阶段生效，以及是否会传递给其他模块。

------

## 4.3 依赖冲突与版本仲裁

### （1）冲突产生原因

当项目中存在同一依赖的多个版本时，会产生依赖冲突。

### （2）版本仲裁机制

1. **最短路径优先**：若同一 artifact 出现多版本，Maven 选用依赖路径最短的版本
2. **先声明优先**：路径长度相同时，POM 中先声明的优先

### （3）解决冲突的方法

- **排除传递依赖**：使用 `<exclusions>` 标签排除不需要的版本
- **统一版本**：在父 POM 或 BOM 中用 `<dependencyManagement>` 固定版本
- **分析依赖树**：使用 `mvn dependency:tree` 查看依赖关系，定位冲突

### （4）排除依赖示例

```xml
<dependency>
  <groupId>org.springframework.boot</groupId>
  <artifactId>spring-boot-starter-web</artifactId>
  <exclusions>
    <exclusion>
      <groupId>org.springframework</groupId>
      <artifactId>spring-core</artifactId>
    </exclusion>
  </exclusions>
</dependency>
```

------

## 4.4 可选依赖与传递依赖

### 🔗 传递依赖

A 依赖 B，B 依赖 C，则 A 会间接依赖 C（受 scope 影响：compile 会传递，test 不传递）

### 📍 可选依赖

通过 `optional=true` 标记的依赖不会传递给依赖本项目的其他模块，用于"可选功能"的依赖（如可选连接器）

```xml
<dependency>
  <groupId>com.example</groupId>
  <artifactId>optional-feature</artifactId>
  <version>1.0.0</version>
  <optional>true</optional>
</dependency>
```

------

## 4.5 dependencyManagement 统一版本

父 POM 或 BOM 中声明：

```xml
<dependencyManagement>
  <dependencies>
    <dependency>
      <groupId>org.springframework.boot</groupId>
      <artifactId>spring-boot-dependencies</artifactId>
      <version>2.7.18</version>
      <type>pom</type>
      <scope>import</scope>
    </dependency>
  </dependencies>
</dependencyManagement>
```

子模块中只需写 groupId、artifactId，不写 version，由父 POM 或 BOM 统一管理：

```xml
<dependency>
  <groupId>org.springframework.boot</groupId>
  <artifactId>spring-boot-starter-web</artifactId>
  <!-- 版本由 dependencyManagement 统一管理 -->
</dependency>
```

## 4.6 Spring Boot 项目依赖管理策略

### 🚀 推荐方式

1. **继承 spring-boot-starter-parent**：
   - 自动继承 Spring Boot 的依赖版本管理
   - 提供默认的构建配置

2. **使用 spring-boot-dependencies BOM**：
   - 适合已有父 POM 的项目
   - 通过 `import` 方式引入版本管理

### （2）面试要点

**Q：Spring Boot 项目如何管理依赖版本？**

**标准回答：**

> Spring Boot 项目通常通过两种方式管理依赖版本：
> 1. 继承 `spring-boot-starter-parent`，自动获得 Spring Boot 推荐的依赖版本
> 2. 使用 `spring-boot-dependencies` 的 BOM（通过 `import` 范围），适合已有父 POM 的项目
> 这样可以避免版本冲突，确保依赖的兼容性。

### 依赖冲突：错误 vs 正确示例

| 场景 | 错误写法 / 现象 | 正确写法 / 做法 |
|------|-----------------|-----------------|
| **同一依赖多版本、未统一** | 各模块或传递依赖带不同 version，运行时 ClassNotFoundException 或方法不兼容 | 父 POM **dependencyManagement** 里统一版本；子模块**不写 version**，或对冲突传递依赖用 **exclusions** 排除。 |
| **排除时只写 artifactId 未写 groupId** | exclusion 必须 groupId+artifactId 都写，否则可能无效 | **exclusion** 里 **groupId** 与 **artifactId** 都写全；**mvn dependency:tree** 确认排除生效。 |
| **provided 依赖被打进 jar** | 未设 scope，或打包插件未排除 provided | 如 Servlet API、Lombok 等设 **scope 为 provided**；spring-boot-maven-plugin 会排除 provided，普通 jar 打包需注意。 |

### 常见坑与注意点

| 现象 / 易错点 | 原因 | 怎么改 / 怎么记 |
|---------------|------|-----------------|
| 依赖下载 401/403 或找不到 | 镜像 URL 错、私服 id 与 server 不匹配、或未配置认证 | 检查 **settings.xml** 的 **mirror/repository** 与 **servers**；**distributionManagement** 的 id 与 **server id** 一致。 |
| 传递依赖带来不需要的 jar 或冲突版本 | 未排除或未在 dependencyManagement 中覆盖 | **mvn dependency:tree -Dverbose** 看冲突；用 **exclusions** 排除或父 POM **dependencyManagement** 指定版本。 |
| scope 用错导致运行时缺类或重复 | 如 JDBC 驱动应用 runtime、Servlet 应用 provided | 记清 **compile/provided/runtime/test** 的区别；容器提供的用 **provided**，仅运行需要的用 **runtime**。 |

# 🌱 第五章：Maven 的生命周期与插件机制

------

## 5.1 三大生命周期

| 生命周期 | 说明 | 常用阶段 |
| -------- | ---- | -------- |
| **clean** | 清理 | clean（删除 target） |
| **default** | 编译、测试、打包、安装、部署 | validate → compile → test → package → verify → install → deploy |
| **site** | 生成站点文档 | site、site-deploy（较少用） |

> 阶段有顺序：执行某阶段会先执行其前面所有阶段（如 `package` 会先执行到 test）

------

## 5.2 各阶段详细说明

| 阶段 | 说明 |
| ---- | ---- |
| **validate** | 校验 POM 是否有效，检查必要信息是否完整 |
| **compile** | 编译主代码，生成 target/classes 目录 |
| **test** | 编译并执行测试代码，生成测试报告 |
| **package** | 打包成 jar/war（由 packaging 决定），生成 target/*.jar |
| **verify** | 执行集成测试等校验，确保构建结果符合质量标准 |
| **install** | 安装到本地仓库，供其他项目依赖使用 |
| **deploy** | 发布到远程仓库（需配置 distributionManagement） |

### 🚀 执行顺序示例

执行 `mvn package` 时，实际执行顺序：
> validate → compile → test → package

------

## 5.3 常用插件

| 插件 | 作用 | 常用配置 |
| ---- | ---- | -------- |
| **maven-compiler-plugin** | 编译 Java | source/target（或 properties 中 maven.compiler.source/target） |
| **maven-jar-plugin** | 打 jar | mainClass、manifest 等 |
| **maven-surefire-plugin** | 运行测试 | 跳过测试、包含/排除用例、报告路径 |
| **spring-boot-maven-plugin** | Spring Boot 可执行 jar | 打 fat jar 并指定 mainClass |
| **maven-dependency-plugin** | 依赖管理 | 分析依赖、复制依赖等 |
| **maven-resources-plugin** | 资源处理 | 资源过滤、复制等 |

### （2）插件配置示例

```xml
<build>
  <plugins>
    <plugin>
      <groupId>org.apache.maven.plugins</groupId>
      <artifactId>maven-compiler-plugin</artifactId>
      <version>3.8.1</version>
      <configuration>
        <source>1.8</source>
        <target>1.8</target>
      </configuration>
    </plugin>
  </plugins>
</build>
```

## 5.4 Phase 与 Goal

### （1）核心概念

- **Phase（阶段）**：生命周期中的一个步骤，如 compile、package
- **Goal（目标）**：插件的具体任务，如 `compiler:compile`、`surefire:test`
- **绑定**：插件将 goal 绑定到 phase，执行 phase 时会触发对应的 goal

### （2）执行机制

执行 `mvn package` 时，会触发绑定到 package 及之前阶段的各 goal：
> compiler:compile → surefire:test → jar:jar

### （3）直接执行 Goal

可以直接执行插件的 goal，不通过 phase：

```bash
mvn compiler:compile
mvn dependency:tree
```

------

## 5.5 自定义插件与扩展

### （1）自定义插件开发

1. **创建 Maven 项目**，打包类型为 `maven-plugin`
2. **继承 Mojo 类**，实现 `execute()` 方法
3. **配置插件描述符**，指定 goal 和参数
4. **安装到本地仓库**，供其他项目使用

### （2）日常使用场景

日常开发中更常见的是在 POM 中配置官方或第三方插件的参数：

- **指定 JDK 版本**：使用 maven-compiler-plugin
- **跳过测试**：使用 maven-surefire-plugin
- **指定主类**：使用 maven-jar-plugin 或 spring-boot-maven-plugin

### （3）面试要点

**Q：Maven 的 Phase 和 Goal 有什么区别？**

**标准回答：**

> Phase 是生命周期中的一个步骤，如 compile、package，代表构建过程中的一个阶段；Goal 是插件的具体任务，如 compiler:compile、surefire:test，是插件执行的具体操作。
> 插件会将 Goal 绑定到 Phase，执行 Phase 时会触发对应的 Goal。也可以直接执行 Goal，不通过 Phase。

| Maven 有哪些生命周期？ | Clean、Default、Site 三大生命周期 |
| `mvn clean install` 执行了哪些阶段？ | clean → validate → compile → test → package → verify → install |
| 如何跳过测试？ | 使用 `-DskipTests` 或配置 maven-surefire-plugin |
| 如何指定 JDK 版本？ | 在 maven-compiler-plugin 中配置 source 和 target |

**本章小结**：依赖由坐标唯一确定，scope 决定编译/测试/运行与传递性；冲突时最短路径优先，可用 dependencyManagement 与 exclusions 干预。Spring Boot 项目通过 parent 或 BOM 统一版本。生命周期通过 Phase 与插件 Goal 绑定，常用插件有 compiler、surefire、spring-boot-maven-plugin 等。

### 常见坑与注意点

| 现象 / 易错点 | 原因 | 怎么改 / 怎么记 |
|---------------|------|-----------------|
| 指定了 phase 但插件未执行 | 插件未绑定到该 phase，或 phase 拼写错 | 在 **build/plugins** 里显式声明插件并绑定 **executions**；或使用插件默认绑定的 phase（如 compiler 绑 compile）。 |
| 打包后无法 java -jar 运行 | 未使用 spring-boot-maven-plugin 或未 repackage | Spring Boot 项目需 **spring-boot-maven-plugin** 的 **repackage**，生成可执行 jar；普通 jar 无 Main-Class 清单。 |
| 测试失败导致 package 中断 | 默认 package 会先执行 test 阶段 | 临时跳过：**-DskipTests**（编译测试不执行）或 **-Dmaven.test.skip=true**（不编译不执行）；修好单测后再去掉。 |

# 🌱 第六章：仓库管理与镜像配置

------

## 6.1 三种仓库类型

| 类型 | 说明 |
|------|------|
| **本地仓库** | 本机 `.m2/repository`，优先从本地取依赖，没有再请求远程 |
| **中央仓库** | Maven Central，默认远程源 |
| **远程/私服** | 公司 Nexus、Artifactory 等，可代理中央仓库并托管内部 jar，部署时用 `mvn deploy` |

## 6.2 配置镜像加速

在 `~/.m2/settings.xml` 的 `<mirrors>` 中配置（以阿里云为例）：

```xml
<mirror>
  <id>aliyun</id>
  <mirrorOf>central</mirrorOf>
  <name>Aliyun Maven</name>
  <url>https://maven.aliyun.com/repository/public</url>
</mirror>
```

`mirrorOf` 为 `*` 表示代理所有仓库，为 `central` 表示只代理默认中央仓库。

## 6.3 settings.xml 与私服

- **servers**：配置私服的 id、用户名、密码（发布时用）。
- **profiles**：可配置不同 profile 的 repository（如 release/snapshot 分离），用 `<activeProfiles>` 或 `-P` 激活。
- POM 中通过 `distributionManagement` 指定 deploy 的仓库 id，id 与 settings 中 server 的 id 对应。

## 6.4 下载依赖失败排查

- 网络/代理：检查镜像 URL、公司代理、防火墙。
- 版本不存在：在 [https://search.maven.org](https://search.maven.org) 确认 groupId/artifactId/version 是否存在。
- 本地仓库损坏：删除对应 artifact 目录后重新 `mvn dependency:resolve` 或 `mvn clean install -U`。

## 6.5 dependency:tree 分析依赖

```bash
mvn dependency:tree
```

可查看传递依赖、发现重复或冲突版本；配合 `-Dverbose` 可看冲突时的省略信息。详见同目录《常用命令与使用场景》。

------

**本章小结**：本地仓库默认 `.m2/repository`，远程有中央仓库与私服；通过 settings.xml 配置 mirror 加速、localRepository、servers（私服账号）。依赖失败可查网络、版本是否存在、本地仓库是否损坏，用 dependency:tree 分析依赖。

### 常见坑与注意点

| 现象 / 易错点 | 原因 | 怎么改 / 怎么记 |
|---------------|------|-----------------|
| mirrorOf 配 * 导致私服无法拉内部包 | * 会代理所有仓库请求，若镜像里没有内部 artifact 会 404 | 只加速中央仓库时 **mirrorOf 用 central**；要同时用多个仓库时，镜像可设 **mirrorOf 为 *,!内部仓库id** 排除内部仓库。 |
| deploy 401/403 | distributionManagement 的 id 与 settings 里 **server id** 不一致，或账号无权限 | POM 中 **repository id** 必须与 **settings.xml** 里 **server** 的 **id** 一致；检查私服账号是否有 deploy 权限。 |
| 本地仓库损坏导致构建异常 | 下载中断或磁盘问题导致 jar 不完整 | 删除 **~/.m2/repository** 下对应 groupId/artifactId/version 目录，再 **mvn clean install -U** 强制更新。 |

- **上一章**：第五章生命周期与插件；本章是**仓库类型与镜像、私服**配置。

# 🌱 第七章：多模块项目（Multi-Module Project）

------

## 7.1 为什么要使用多模块？

- 代码按业务或层级拆分，职责清晰；依赖在模块间显式声明，便于复用与版本统一。
- 父 POM 统一管理依赖版本、插件、属性，子模块只关心自身依赖，减少重复与冲突。
- 可单独构建、测试、发布某一模块，或一次性聚合构建整个工程。

## 7.2 父子关系与继承

- 父 POM：`packaging=pom`，可定义 `dependencyManagement`、`pluginManagement`、`properties`。
- 子模块：通过 `<parent>` 引用父 POM；子模块不写 version 时从父的 dependencyManagement 继承。

## 7.3 聚合构建

- 父 POM 中 `<modules><module>module-a</module><module>module-b</module></modules>`。
- 在父目录执行 `mvn clean install` 时，Maven 按模块依赖顺序构建各子模块。

## 7.4 子模块间依赖与版本统一

- 子模块 A 依赖子模块 B：在 A 的 POM 中写 B 的 groupId、artifactId，version 可与父统一或由父 dependencyManagement 管理。
- 多模块下尽量在父 POM 用 dependencyManagement 统一第三方依赖版本，子模块只声明 artifactId。

## 7.5 从单体拆分为多模块

- 先划分模块边界（如 api、service、web、common），再新建子模块目录与 pom.xml，把原包移动到对应模块；父 POM 用 modules 聚合、用 dependencyManagement 统一版本；子模块间通过 dependencies 引用。

------

**本章小结**：多模块通过 parent 继承、modules 聚合；父 POM 统一 dependencyManagement、pluginManagement、properties，子模块只声明自身依赖与代码；可单独构建某模块或在根目录一次性构建，子模块间通过坐标引用。

### 常见坑与注意点

| 现象 / 易错点 | 原因 | 怎么改 / 怎么记 |
|---------------|------|-----------------|
| 在子模块目录执行 mvn 却用到根父 POM | 未在子目录或子目录 pom 的 parent 指向错 | 在**子模块目录**下执行 **mvn** 才会用该子模块 POM；在根目录执行会按 **modules** 聚合构建。 |
| 子模块 A 依赖子模块 B，B 未先 install | 本地仓库没有 B 的 jar，A 编译找不到 | 先 **mvn install** 被依赖的模块 B，或在**根目录** **mvn clean install** 按依赖顺序整体构建。 |
| 父 POM 改了但子模块未生效 | IDE 未刷新或未重新解析 | **Maven Reload / Reimport**；或命令行 **mvn clean** 后重新 **mvn compile**，确保读到最新父 POM。 |

# 🌱 第八章：常用命令与构建流程

------

- **常用命令**：`mvn clean`、`mvn compile`、`mvn test`、`mvn package`、`mvn install`、`mvn deploy`；依赖分析 `mvn dependency:tree`、`mvn help:effective-pom`。
- **跳过测试**：`-DskipTests` 只跳过执行、仍编译测试；`-Dmaven.test.skip=true` 不编译也不执行。
- **打包类型**：`jar`（默认）、`war`（Web）、`pom`（仅父工程）。
- **Profile**：`-Pprod`、`-Pdev` 激活对应 profile。
- **其他**：`-U` 强制更新、`-o` 离线、`-q` 安静、`-X` 调试。

📌 **完整命令与参数说明见同目录《常用命令与使用场景》。**

------

**本章小结**：常用命令为 clean、compile、test、package、install、deploy；依赖分析用 dependency:tree、help:effective-pom。跳过测试用 -DskipTests 或 -Dmaven.test.skip=true；-U 强制更新、-P 激活 Profile。

# 🌱 第九章：Profiles 与环境隔离

------

## 9.1 什么是 Maven Profile？

Profile 用于在**同一 POM** 中为不同环境（dev/test/prod）提供不同配置：依赖、属性、资源过滤、插件参数等，通过激活条件或命令行切换。

## 9.2 不同环境配置切换

- 在 `<profiles>` 中定义多个 `<profile>`，每个有 `<id>`；在 profile 内可覆盖 `<properties>`、`<dependencies>`、`<build>` 等。
- 资源过滤：在 `<build><resources>` 中开启 filtering，资源文件里用 `${prop}` 引用属性，不同 profile 注入不同值。

## 9.3 pom.xml 中的 Profile 定义

```xml
<profiles>
  <profile>
    <id>prod</id>
    <properties><env>prod</env></properties>
  </profile>
</profiles>
```

## 9.4 激活方式

- **命令行**：`-P prod`。
- **默认激活**：`<activation><activeByDefault>true</activeByDefault></activation>`。
- **settings.xml**：`<activeProfiles><activeProfile>prod</activeProfile></activeProfiles>`。

## 9.5 与 Spring Boot 结合

- Spring Boot 的 `application-{profile}.yml` 与 Maven Profile 可分离：Maven Profile 负责构建时变量（如打包用哪套配置），Spring Profile 负责运行时的环境（如 `spring.profiles.active=prod`）；也可在 Maven 资源过滤中把 `env` 写入配置文件，供 Spring 使用。

------

**本章小结**：Profile 在同一 POM 中为不同环境提供不同依赖、属性、资源过滤与插件参数；通过 -P、默认激活或 settings.xml 的 activeProfiles 激活；可与 Spring Boot 的 application-{profile} 配合使用。

# 🌱 第十章：Maven 与 IDE 集成

------

## 10.1 IntelliJ IDEA

- **导入**：Open → 选择含 `pom.xml` 的目录，以 Maven 项目打开；或 Import Project → 选 pom.xml。
- **刷新**：右键项目或 pom.xml → Maven → Reload Project；或使用右侧 Maven 工具窗口的刷新按钮。
- **运行**：可配置 Maven 运行配置（如 `clean package -DskipTests`），或直接运行 Spring Boot 主类。

## 10.2 Eclipse

- 使用 M2Eclipse 插件；Import → Maven → Existing Maven Projects，选择根目录。
- 更新依赖：右键项目 → Maven → Update Project。

## 10.3 Maven Wrapper

- 项目中的 `mvnw` / `mvnw.cmd` 和 `.mvn/` 目录可在无全局安装 Maven 时使用，IDE 可配置为使用项目自带 Wrapper，保证环境一致。

## 10.4 构建失败常见 IDE 问题

- 依赖未下载完：Reload Project、检查网络与镜像、`mvn dependency:resolve -U`。
- JDK 与 POM 不一致：Project Structure → Project SDK、Modules 的 Language level；POM 中 maven.compiler.source/target。
- 缓存异常：Invalidate Caches / Restart，或删除项目下的 `.idea` 后重新打开。

------

**本章小结**：IDEA/Eclipse 通过 Open pom.xml 或 Maven 项目导入；依赖与配置变更后需 Reload/Update。Maven Wrapper 保证无全局 Maven 时也能一致构建。构建失败时检查依赖、JDK、POM 与 IDE 缓存。

# 🌱 第十一章：Maven 私服与企业级仓库搭建

------

## 11.1 Nexus 简介

Nexus Repository Manager 是常用的 Maven 私服，可代理中央仓库、托管内部 release/snapshot，统一依赖来源与发布出口。

## 11.2 安装与启动

- 从 Sonatype 或 OSS 下载 Nexus 安装包，解压后运行脚本（如 `nexus run`）；默认端口 8081，首次访问按提示完成初始化并设置 admin 密码。

### 11.3 发布仓库配置

- 在 Nexus 中创建 repository：如 maven-releases（Hosted）、maven-snapshots（Hosted）、maven-public（Group 聚合上述 + 代理的 central）。
- POM 中配置 `distributionManagement` 的 release 与 snapshot 的 url，id 与 settings.xml 中 `<server><id>` 一致；settings 中配置该 id 的用户名、密码。

## 11.4 使用 mvn deploy 发布

- 执行 `mvn clean deploy` 将当前模块发布到 distributionManagement 指定的仓库（需有权限）；快照版本会发布到 snapshot 仓库，release 到 release 仓库。

## 11.5 版本与权限

- 私服可配置哪些角色能 deploy、哪些能浏览；release 仓库一般禁止重复发布同一版本，snapshot 可覆盖。

------

**本章小结**：Nexus 等私服可代理中央仓库、托管内部 release/snapshot；POM 中配置 distributionManagement，settings.xml 中配置 server 的 id 与账号；通过 `mvn deploy` 发布，release 与 snapshot 仓库分离。

# 🌱 第十二章：Maven 常见错误与排查技巧

------

| 问题 | 排查思路 |
|------|----------|
| **依赖冲突** | `mvn dependency:tree` 看冲突路径；在 dependencyManagement 或 dependencies 中统一/排除版本。 |
| **插件版本不兼容** | 查看错误日志中的插件与 JDK/Maven 版本要求；在 pluginManagement 或 plugins 中指定兼容版本。 |
| **本地仓库损坏** | 删除 `~/.m2/repository` 下对应 artifact 目录，重新 `mvn dependency:resolve` 或 `-U`。 |
| **clean install 报错** | 先看最后一屏错误（编译失败、测试失败、找不到依赖、插件执行失败）；根据提示修代码、依赖或配置。 |
| **IDEA 构建失败** | 确认 JDK、Maven 路径、Reload Project；命令行在同一目录执行 `mvn clean install` 对比是否 IDE 特有问题。 |

更多见《常用命令与使用场景》中的「常见问题排查」。

------

**本章小结**：依赖冲突用 dependency:tree 定位后统一或排除；插件不兼容时指定版本；本地仓库损坏可删对应目录后 -U 重下；clean install 报错先看最后一屏；IDEA 问题可对比命令行是否一致。

# 🌱 第十三章：Maven 与 CI/CD 持续集成

------

- **Jenkins**：拉取代码后执行 Shell 或 Maven 步骤，如 `mvn clean package -DskipTests -Pprod`；可配置 JDK 版本、Maven 安装、归档 jar/war。
- **GitHub Actions / GitLab CI**：在 workflow/ pipeline 中安装 JDK、Maven（或使用预装镜像），执行 `mvn clean install` 或 `mvn deploy`；可加代码检查、测试、发布到私服或 Docker 镜像。
- **Docker**：镜像构建阶段用 Maven 镜像编译打包，多阶段构建中只把 target/*.jar 拷贝到运行镜像，减小镜像体积。
- Maven 在 DevOps 中提供**标准化构建入口**：一条命令完成编译、测试、打包，便于流水线重复执行与环境一致。

------

**本章小结**：Jenkins、GitLab CI、GitHub Actions 等通过执行 mvn clean install/deploy 完成构建与发布；Docker 多阶段构建中常用 Maven 镜像编译、仅拷贝 jar 到运行镜像；Maven 提供标准化构建入口便于 CI/CD 流水线。

# 🌱 第十四章：项目实战案例

------

- **单体 Spring Boot**：`spring-boot-starter-parent` 或 BOM + `spring-boot-maven-plugin`，`mvn clean package` 得到可执行 jar，`java -jar xxx.jar` 运行。
- **多模块**：父 POM 聚合各业务模块，common/api/service/web 等；父 dependencyManagement 统一 Spring Boot 与常用库版本；子模块 dependencies 只写 artifactId；构建时在根目录 `mvn clean install`，运行某一模块的 Spring Boot 主类或打出的 jar。
- **RESTful 后端 + 前端**：后端为 Maven 模块；前端可单独 npm 构建，或通过 frontend-maven-plugin 在 Maven 中执行 npm 并把产物拷贝到 resources/static，随 war 或 jar 一起发布。
- **部署**：jar 直接上传服务器 `java -jar` 或配合 systemd/Docker；war 放入 Tomcat 的 webapps 或通过 Tomcat 镜像挂载。

------

**本章小结**：单体 Spring Boot 用 parent 或 BOM + spring-boot-maven-plugin 打可执行 jar；多模块父 POM 聚合、dependencyManagement 统一版本；前后端可分离构建或通过插件集成；部署方式与 jar/war 及运行环境相关。

# 🌱 第十五章：面试题与高频考点总结

## 15.1 核心概念与基础

### 15.1 什么是 Maven？核心功能是什么？

**标准回答：**

> Maven 是一个基于项目对象模型（POM）的构建与依赖管理工具。
> 它的核心功能包括：
> 1. **依赖管理**：自动下载所需 jar 包及其传递依赖
> 2. **生命周期管理**：定义从编译到部署的完整流程
> 3. **插件机制**：通过插件扩展功能
> 4. **标准化项目结构**：提供统一的目录结构规范
> 5. **可复现构建**：确保在任何环境下都能获得相同的构建结果

------

### 15.2 Maven 的核心概念有哪些？

**标准回答：**

> Maven 的核心概念包括：
> 1. **POM（Project Object Model）**：项目对象模型，用 `pom.xml` 描述项目信息
> 2. **坐标（Coordinates）**：由 groupId、artifactId、version 组成，唯一标识一个依赖
> 3. **依赖管理**：自动解析和管理项目依赖
> 4. **生命周期**：定义项目从创建到部署的标准阶段
> 5. **插件**：提供具体的构建功能
> 6. **仓库**：存储依赖的地方，包括本地仓库和远程仓库

------

## 15.2 生命周期与插件

### 15.3 Maven 有哪些生命周期？各阶段？

**标准回答：**

> Maven 有三大生命周期：
> 1. **Clean 生命周期**：清理构建产物，主要阶段是 clean
> 2. **Default 生命周期**：核心构建过程，包括 validate、compile、test、package、verify、install、deploy
> 3. **Site 生命周期**：生成项目站点文档，包括 site、site-deploy

------

### 15.4 Phase 和 Goal 有什么区别？

**标准回答：**

> Phase 是生命周期中的一个步骤，如 compile、package，代表构建过程中的一个阶段；
> Goal 是插件的具体任务，如 compiler:compile、surefire:test，是插件执行的具体操作。
> 插件会将 Goal 绑定到 Phase，执行 Phase 时会触发对应的 Goal。也可以直接执行 Goal，不通过 Phase。

------

### 15.5 mvn clean install 执行了哪些阶段？

**标准回答：**

> `mvn clean install` 执行了以下阶段：
> 1. clean（清理构建产物）
> 2. validate（校验 POM）
> 3. compile（编译主代码）
> 4. test（执行测试）
> 5. package（打包）
> 6. verify（验证构建结果）
> 7. install（安装到本地仓库）

------

## 15.3 依赖管理

### 15.6 依赖的 scope 有哪些？区别是什么？

**标准回答：**

> Maven 依赖的 scope 主要有：
> 1. **compile**：默认，全阶段可用，会传递
> 2. **provided**：编译和测试可用，运行时由容器提供，不传递
> 3. **runtime**：测试和运行时可用，会传递
> 4. **test**：仅测试可用，不传递
> 5. **system**：本地 jar，不推荐，不传递
> 6. **import**：仅用于 dependencyManagement 的 BOM，继承版本

------

### 15.7 如何解决依赖冲突？

**标准回答：**

> Maven 解决依赖冲突的机制：
> 1. **最短路径优先**：若同一 artifact 出现多版本，Maven 选用依赖路径最短的版本
> 2. **先声明优先**：路径长度相同时，POM 中先声明的优先
> 3. **手动干预**：
>    - 使用 `<exclusions>` 排除不需要的版本
>    - 在父 POM 或 BOM 中用 `<dependencyManagement>` 固定版本
>    - 使用 `mvn dependency:tree` 分析依赖关系，定位冲突

------

### 15.8 什么是传递依赖？如何控制？

**标准回答：**

> 传递依赖是指：A 依赖 B，B 依赖 C，则 A 会间接依赖 C（受 scope 影响：compile 会传递，test 不传递）。
> 控制传递依赖的方式：
> 1. 使用 `<exclusions>` 排除不需要的传递依赖
> 2. 使用 `optional=true` 标记可选依赖，使其不传递给其他模块
> 3. 使用 `<dependencyManagement>` 统一管理依赖版本

------

### 15.9 dependencyManagement 和 dependencies 的区别？

**标准回答：**

> dependencyManagement 只声明依赖版本，不实际引入依赖；dependencies 实际引入依赖并可继承 dependencyManagement 中声明的版本。
> dependencyManagement 用于统一管理依赖版本，避免版本冲突；dependencies 用于实际引入项目需要的依赖。

------

## 15.4 多模块项目

### 15.10 多模块项目的优势是什么？

**标准回答：**

> Maven 多模块项目的优势包括：
> 1. **代码按业务或层级拆分**，职责清晰
> 2. **依赖在模块间显式声明**，便于复用与版本统一
> 3. **父 POM 统一管理依赖版本、插件、属性**，减少重复与冲突
> 4. **可单独构建、测试、发布某一模块**，或一次性聚合构建整个工程
> 5. **便于团队协作**，不同团队负责不同模块

------

### 15.11 父 POM 和子模块的关系是什么？

**标准回答：**

> 多模块项目中，父 POM 和子模块的关系是：
> 1. **继承关系**：子模块通过 `<parent>` 标签继承父 POM，可继承依赖管理、插件配置、属性等
> 2. **聚合关系**：父 POM 通过 `<modules>` 标签聚合子模块，可一次性构建所有子模块
> 3. **依赖关系**：子模块之间可以相互依赖，通过坐标引用

------

## 15.5 高级特性

### 15.12 Spring Boot 项目如何管理依赖版本？

**标准回答：**

> Spring Boot 项目通常通过两种方式管理依赖版本：
> 1. **继承 spring-boot-starter-parent**：自动获得 Spring Boot 推荐的依赖版本，提供默认的构建配置
> 2. **使用 spring-boot-dependencies BOM**：通过 `import` 范围引入，适合已有父 POM 的项目
> 这样可以避免版本冲突，确保依赖的兼容性。

------

### 15.13 Maven Profile 有什么作用？如何使用？

**标准回答：**

> Maven Profile 用于在同一 POM 中为不同环境（dev/test/prod）提供不同配置，包括依赖、属性、资源过滤、插件参数等。
> 使用方式：
> 1. 在 `<profiles>` 中定义多个 `<profile>`，每个有唯一的 `<id>`
> 2. 通过命令行激活：`-P prod`
> 3. 通过默认激活：`<activation><activeByDefault>true</activeByDefault></activation>`
> 4. 通过 settings.xml 激活：`<activeProfiles><activeProfile>prod</activeProfile></activeProfiles>`

------

### 15.14 Maven 与 Gradle 的区别是什么？

**标准回答：**

> Maven 与 Gradle 的主要区别：
> 1. **配置文件**：Maven 使用 `pom.xml`（XML 格式），Gradle 使用 `build.gradle`（Groovy/Kotlin  DSL）
> 2. **构建模型**：Maven 是声明式（告诉要什么），Gradle 是混合式（声明 + 编程）
> 3. **构建性能**：Gradle 更快，支持增量构建和缓存
> 4. **扩展性**：Gradle 更灵活，支持自定义任务
> 5. **生态**：Maven 生态更成熟，插件更丰富
> 6. **使用场景**：Maven 适合企业级项目，Gradle 适合 Android 和快速原型项目

------

## 15.6 实战与问题排查

### 15.15 如何排查 Maven 构建失败的问题？

**标准回答：**

> 排查 Maven 构建失败的问题，可按以下步骤：
> 1. **查看错误日志**：重点关注最后一屏的错误信息，确定失败原因
> 2. **依赖问题**：使用 `mvn dependency:tree` 分析依赖冲突，检查镜像配置
> 3. **编译问题**：检查 JDK 版本与 POM 中配置的 source/target 是否一致
> 4. **测试问题**：检查测试代码是否正确，可使用 `-DskipTests` 跳过测试
> 5. **插件问题**：检查插件版本是否兼容，可在 POM 中指定插件版本
> 6. **本地仓库问题**：删除损坏的依赖目录，使用 `-U` 强制更新
> 7. **IDE 问题**：在命令行执行 `mvn clean install` 确认是否为 IDE 特有问题

------

### 15.16 如何配置 Maven 镜像加速？

**标准回答：**

> 配置 Maven 镜像加速的步骤：
> 1. 找到 Maven 的 settings.xml 文件（全局：Maven 安装目录/conf/settings.xml；用户级：~/.m2/settings.xml）
> 2. 在 `<mirrors>` 标签中添加镜像配置，例如：
>    ```xml
>    <mirror>
>      <id>aliyun</id>
>      <mirrorOf>central</mirrorOf>
>      <name>Aliyun Maven</name>
>      <url>https://maven.aliyun.com/repository/public</url>
>    </mirror>
>    ```
> 3. 保存配置后，Maven 会优先从镜像仓库下载依赖

# 📎 附录

- **Maven 命令速查**：见同目录《常用命令与使用场景》。
- **常见插件**：maven-compiler-plugin（JDK 版本）、maven-surefire-plugin（测试）、spring-boot-maven-plugin（可执行 jar）；版本随 Spring Boot 或父 POM 管理即可。
- **settings.xml**：镜像、localRepository、servers（私服账号）配置示例见《常用命令与使用场景》第五节。
- **IDEA**：Maven 工具窗口 Reload、Execute Maven Goal、Run Maven 配置；Project Structure 中 JDK 与 Language level 与 POM 一致。
- **常见报错**：依赖找不到→镜像/网络/版本；编译失败→JDK 与 source/target；测试失败→单测或 -DskipTests；部署 401/403→settings 中 server 的 id 与 distributionManagement 一致且账号有权限。
