# 🧩 第1章：MyBatis 概述与核心思想

> “从 JDBC 到 ORM，再到 MyBatis，本章帮助你理解为什么要用 MyBatis，它解决了什么问题，以及它的核心设计思想是什么”

------

## 1. ORM（对象关系映射）概念与发展

### （1）什么是 ORM（Object Relational Mapping）

- **定义**：ORM 是一种 **对象与关系数据库之间的映射技术**。
   它通过把数据库表与 Java 对象进行映射，使开发者可以**面向对象操作数据库**。

| 概念层面           | 对应映射            |
| ------------------ | ------------------- |
| 数据库表（table）  | Java 类（class）    |
| 数据库行（row）    | Java 对象（object） |
| 数据库列（column） | Java 属性（field）  |

> 💡 **一句话理解：** ORM 让你用“对象”来操纵“表”，减少直接编写 SQL 的复杂度。

------

### （2）ORM 技术发展历程

| 阶段            | 特点                                 | 代表技术       |
| --------------- | ------------------------------------ | -------------- |
| 原始 JDBC 阶段  | 手动编写 SQL、参数设置、结果映射繁琐 | JDBC           |
| 半自动 ORM 阶段 | 通过配置文件映射 SQL 与对象          | MyBatis        |
| 全自动 ORM 阶段 | 自动生成 SQL，自动管理对象关系       | Hibernate、JPA |

> MyBatis 处于“**半自动 ORM**”阶段：
>  SQL 由开发者编写，但结果映射和对象管理由框架自动完成。

------

### （3）MyBatis 在 ORM 中的定位

- **MyBatis 是轻量级 ORM 框架**，介于 JDBC 与 Hibernate 之间：
  - 不完全屏蔽 SQL，让开发者保留 SQL 控制权；
  - 但自动完成参数映射、结果封装。

------

## 2. JDBC 的痛点与 MyBatis 的诞生

### （1）JDBC 开发存在的问题

传统 JDBC 开发步骤：

1. 注册驱动、建立连接；
2. 创建 Statement；
3. 手写 SQL；
4. 设置参数；
5. 执行 SQL；
6. 处理结果集；
7. 手动关闭连接。

```java
// JDBC 示例
Connection conn = DriverManager.getConnection(url, user, pwd);
PreparedStatement ps = conn.prepareStatement("select * from user where id=?");
ps.setInt(1, 1);
ResultSet rs = ps.executeQuery();
User user = new User();
while(rs.next()) {
    user.setId(rs.getInt("id"));
    user.setName(rs.getString("name"));
}
rs.close(); ps.close(); conn.close();
```

### （2）痛点总结

| 问题                 | 描述                        |
| -------------------- | --------------------------- |
| SQL 与 Java 代码耦合 | SQL 写在代码中，可维护性差  |
| 参数绑定繁琐         | 需要逐个 set 参数           |
| 结果映射复杂         | 需要手动从 ResultSet 中取值 |
| 资源管理复杂         | 每次手动关闭连接，易忘记    |
| 缺乏统一管理         | 无法集中维护 SQL            |

------

### （3）MyBatis 的诞生

- **前身：iBATIS（2002年）**，后更名为 **MyBatis（2010年）**。
- 设计目标：
  - 减少 JDBC 重复代码；
  - 提供灵活 SQL 配置；
  - 让开发者能专注于 SQL 优化；
  - 保留面向对象的数据映射机制。

> ✅ **MyBatis 解决了 JDBC 的“繁琐与重复”，但保留了 SQL 的“灵活与可控”。**

------

## 3. MyBatis、Hibernate、JPA 对比分析

| 对比项     | **MyBatis**             | **Hibernate / JPA**    |
| ---------- | ----------------------- | ---------------------- |
| SQL 控制权 | 手写 SQL（高灵活性）    | 自动生成 SQL（低控制） |
| 学习成本   | 中等                    | 高                     |
| 性能调优   | 手动优化 SQL            | 自动优化，难以微调     |
| 缓存机制   | 一级、二级缓存可自定义  | 提供完整缓存体系       |
| 数据映射   | 半自动                  | 全自动（对象全映射）   |
| 适用场景   | 对 SQL 要求高、复杂业务 | 快速原型开发           |

> 💬 实际项目中：
>
> - 电商、点评类系统 → **MyBatis**
> - 简单后台系统 → **JPA/Hibernate**

## 4. MyBatis 的核心目标与设计思想

### （1）MyBatis 的核心目标

1. 简化数据库操作；
2. 提高 SQL 管理的可维护性；
3. 保留对 SQL 的完全控制；
4. 实现对象与 SQL 的自动映射。

------

### （2）核心设计思想

| 核心思想                | 说明                                                 |
| ----------------------- | ---------------------------------------------------- |
| **配置驱动**            | 使用 XML 或注解定义 SQL、映射规则、缓存策略等        |
| **映射思想（Mapping）** | Java 对象属性 ↔ 数据库列 的双向映射                  |
| **动态 SQL**            | 通过 `<if>`、`<foreach>` 等标签动态生成 SQL          |
| **分层架构**            | 通过 Executor、StatementHandler 等抽象层解耦执行过程 |
| **插件扩展**            | 提供 Interceptor 拦截器机制，可插入自定义逻辑        |
| **与 Spring 无缝整合**  | 通过 `SqlSessionFactoryBean` 管理生命周期与事务      |

------

### （3）核心组件关系图

```text
[Configuration] --管理--> [MappedStatement(SQL配置)]
          ↓
  [SqlSessionFactory] --创建--> [SqlSession(会话)]
          ↓
     [Executor] --调用--> [StatementHandler → JDBC执行]
          ↓
     [ResultSetHandler] --封装--> [Java对象]
```

------

### （4）与 SpringBoot 的整合思想

SpringBoot 启动时自动扫描：

- Mapper 接口 → 动态代理生成 Mapper 实现类；
- 加载 XML 文件 → 生成 `MappedStatement`;
- 通过 `SqlSessionTemplate` 执行 SQL。

------

## 5. MyBatis 的执行流程概览（XML → 映射 → 执行）

### （1）整体流程图

```
XML配置文件  →  加载解析 →  MappedStatement对象
      ↓
Mapper接口  →  动态代理生成实现类
      ↓
SqlSession执行SQL
      ↓
Executor调用StatementHandler执行SQL
      ↓
ResultSetHandler封装结果为Java对象
```

------

### （2）执行过程详细步骤

1️⃣ **加载配置文件**

```java
InputStream in = Resources.getResourceAsStream("mybatis-config.xml");
SqlSessionFactory factory = new SqlSessionFactoryBuilder().build(in);
```

→ MyBatis 解析 XML 文件，创建 `Configuration` 对象。

2️⃣ **创建 SqlSession**

```java
SqlSession sqlSession = factory.openSession();
```

→ 内部创建 `Executor`、`Transaction`、`MappedStatement`。

3️⃣ **执行 SQL**

```java
User user = sqlSession.selectOne("UserMapper.selectUserById", 1);
```

→ MyBatis 根据映射 ID 找到 SQL，绑定参数并执行。

4️⃣ **结果封装**
 → 通过 `ResultSetHandler` 将结果集转为 Java 对象。

5️⃣ **关闭会话**

```java
sqlSession.close();
```

→ 释放连接、缓存更新。

------

### （3）执行原理简化图

```text
Mapper接口调用 → SqlSession → Executor → StatementHandler → JDBC
        ↑                                    ↓
   动态代理生成Mapper实例         ResultSetHandler映射结果
```

### （4）项目实战举例（黑马点评中）

> 以“查询用户详情”为例说明：

- **UserMapper.xml**

```xml
<select id="queryUserById" parameterType="Long" resultType="User">
    SELECT * FROM tb_user WHERE id = #{id}
</select>
```

- **UserMapper.java**

```java
@Mapper
public interface UserMapper {
    User queryUserById(Long id);
}
```

- **调用层**

```java
User user = userMapper.queryUserById(1L);
```

> 💡 MyBatis 自动完成：
>
> - SQL 映射；
> - 参数绑定；
> - 结果封装；
> - 异常处理与资源释放。

## 🧠 面试题模板：MyBatis 概述类问题

### Q1：MyBatis 是什么？与 JDBC 的区别？

**标准回答：**

> MyBatis 是一个半自动 ORM 框架，用于简化 JDBC 操作。
>  相比 JDBC，MyBatis 把 SQL 配置与 Java 代码分离，通过 XML 或注解管理 SQL，自动完成参数绑定与结果映射，大大提高开发效率与可维护性。

------

### Q2：MyBatis 与 Hibernate 的区别？

| 对比项   | MyBatis            | Hibernate            |
| -------- | ------------------ | -------------------- |
| SQL 编写 | 手写 SQL，灵活可控 | 自动生成 SQL         |
| 性能调优 | 手动优化           | 自动优化（可控性差） |
| 学习曲线 | 平缓               | 较陡峭               |
| 缓存     | 一级/二级          | 完善的缓存体系       |
| 适用场景 | SQL 复杂系统       | 数据模型稳定系统     |

------

### Q3：MyBatis 的核心流程是什么？

> MyBatis 的执行流程是：
>
> 1. 读取配置文件；
> 2. 生成 SqlSession；
> 3. 动态代理 Mapper；
> 4. Executor 执行 SQL；
> 5. ResultSetHandler 封装结果；
> 6. 返回 Java 对象。

------

### Q4：为什么 MyBatis 被称为“半自动 ORM 框架”？

> 因为它自动完成对象映射、参数绑定等，但 SQL 仍需开发者手动编写。
>  与 Hibernate 全自动不同，MyBatis 保留了对 SQL 的完全控制权。

------

## ⚠️ 常见坑总结

| 问题             | 说明                 | 解决方案               |
| ---------------- | -------------------- | ---------------------- |
| XML 映射路径错误 | `mappers` 未加载正确 | 使用相对路径或包扫描   |
| 参数映射异常     | 参数名不一致         | 使用 `@Param` 明确标注 |
| 多表查询报错     | resultMap 映射不全   | 明确指定列与属性映射   |
| 事务未生效       | 手动管理 SqlSession  | 交由 Spring 管理事务   |
| SQL 执行慢       | 无索引 / 未缓存      | 检查 SQL + 开启缓存    |

------

✅ **总结：**

> MyBatis 的本质是一个“映射驱动 + SQL 控制”的轻量 ORM 框架。
>  它以 XML/注解为核心，通过映射文件完成对象与 SQL 的解耦，保留 SQL 的灵活性，同时自动完成对象封装，是企业级后端中最主流的持久层框架之一。

## 第2章：MyBatis 核心组件与架构设计

### 2.1 四层架构

- **接口层**：Mapper 接口，供业务调用；**会话层**：SqlSession，封装 CRUD 与事务；**映射层**：Mapper 与 XML/注解的绑定、参数与结果映射；**执行层**：Executor、StatementHandler、ResultSetHandler，真正执行 JDBC。

### 2.2 核心组件

- **SqlSessionFactory**：全局单例，根据配置构建；**SqlSession**：一次请求或事务内使用，由工厂创建；**Executor**：执行 SQL（Simple/Reuse/Batch）；**StatementHandler**：封装 PreparedStatement；**ResultSetHandler**：结果集映射为对象。**Mapper 接口**通过 **JDK 动态代理**生成实现，代理内调用 SqlSession 的 selectOne/insert 等。

### 2.3 执行流程简述

- 业务调用 Mapper 方法 → 代理拦截 → SqlSession 根据方法名与参数找到 MappedStatement → Executor 执行 → StatementHandler 预编译、设参 → 执行 SQL → ResultSetHandler 封装结果 → 返回。

### 2.4 与 Spring 整合后

- Spring 通过 **SqlSessionTemplate** 管理 SqlSession 生命周期（通常每事务一个）；**@MapperScan** 扫描 Mapper 接口并注册为 Bean；事务由 **DataSourceTransactionManager** 管理，MyBatis 使用当前线程的 Connection。

------

## 第3章：MyBatis 配置文件详解（mybatis-config.xml）

### 3.1 根标签与结构

- **&lt;configuration&gt;** 下常见：environments、mappers、typeAliases、typeHandlers、settings、plugins。Spring Boot 下多数配置可在 application.yml 的 mybatis 节点完成，无需单独 mybatis-config.xml。

### 3.2 环境与数据源

- **&lt;environments&gt;** 可配置多环境（dev/prod），每环境指定 **&lt;transactionManager&gt;** 和 **&lt;dataSource&gt;**；与 Spring 整合后数据源由 Spring 管理，MyBatis 只做 SQL 映射。

### 3.3 映射器注册

- **&lt;mappers&gt;** 中 **&lt;mapper resource="..."/&gt;** 或 **&lt;package name="..."/&gt;**；Spring Boot 常用 **@MapperScan** 或 **mybatis.mapper-locations** 指定 XML 路径。

### 3.4 类型别名与处理器

- **typeAliases**：为实体类起短别名，如 com.example.User → User；**typeHandlers**：自定义 JDBC 类型与 Java 类型转换（如枚举、JSON）。

### 3.5 全局设置

- **cacheEnabled**：二级缓存开关；**lazyLoadingEnabled**：延迟加载；**mapUnderscoreToCamelCase**：下划线列名转驼峰；**logImpl**：日志实现（SLF4J 等）。

------

## 第4章：Mapper 映射文件详解（XML 映射文件）

### 4.1 基本结构

- 根元素 **&lt;mapper namespace="Mapper 接口全限定名"&gt;**，内为 **&lt;select&gt;**、**&lt;insert&gt;**、**&lt;update&gt;**、**&lt;delete&gt;**，id 对应接口方法名。

### 4.2 参数与结果

- **parameterType**：单参可省略；多参用 **@Param** 或 Map；**resultType**：单表实体类；**resultMap**：复杂映射、嵌套、集合。**#{}** 预编译占位符，**${}** 字符串替换（慎用，防注入）。

### 4.3 动态 SQL

- **&lt;if test="条件"&gt;**、**&lt;choose&gt;/&lt;when&gt;/&lt;otherwise&gt;**、**&lt;foreach collection="list" item="item"&gt;**、**&lt;trim&gt;**、**&lt;where&gt;** 自动去掉多余 AND/OR，避免拼接错误。

### 4.4 多表关联

- **&lt;resultMap&gt;** 中 **&lt;association&gt;** 一对一、**&lt;collection&gt;** 一对多；可嵌套 resultMap 或 **select** 子查询（N+1 时配合懒加载或批量查询优化）。

------

## 第5章：MyBatis 的运行机制与源码解析

### 5.1 SqlSession 与 Executor

- SqlSession 由工厂创建，默认每线程或每事务一个；内部持有 **Executor**，Executor 负责与 JDBC 交互；**SimpleExecutor** 每次新建 Statement，**ReuseExecutor** 复用同 SQL 的 Statement，**BatchExecutor** 批处理。

### 5.2 缓存

- **一级缓存**：SqlSession 级别，同一 Session 内相同查询直接返回缓存；**二级缓存**：Mapper 级别，需在 &lt;mapper&gt; 或 &lt;cache&gt; 开启，跨 Session 共享；注意脏读与事务边界。

### 5.3 延迟加载

- 关联对象配置 **fetchType="lazy"** 时，访问关联属性才发子查询；底层通过 CGLIB 代理或 ResultLoader 懒加载。

### 5.4 日志

- 配置 **logImpl** 或 **logging.level.xxx.mapper=DEBUG** 可打印 SQL 与参数，便于调试。

------

## 第6章：MyBatis 动态 SQL 深入与性能优化

### 6.1 动态 SQL 原理

- 基于 **OGNL** 解析 **test** 条件，在内存中拼接最终 SQL；注意 **&lt;where&gt;** 避免首 AND，**&lt;foreach&gt;** 的 open/separator/close 控制 in 列表格式。

### 6.2 批量与分页

- 批量插入用 **&lt;foreach&gt;** 拼多行 values，或 **ExecutorType.BATCH** + 循环 insert；分页可用 **PageHelper** 插件做物理分页，或手写 **LIMIT** 与 count 查询。

### 6.3 性能建议

- 避免 **${}** 拼接；合理使用二级缓存；大批量用 BatchExecutor；慢 SQL 结合数据库索引与执行计划优化。

------

## 第7章：MyBatis 缓存机制详解

### 7.1 一级缓存

- 默认开启，SqlSession 内有效；同一 Session、相同 SQL 与参数会命中；**commit/rollback、update/insert/delete、清空** 会失效；Spring 整合下通常每事务一个 SqlSession，事务结束即失效。

### 7.2 二级缓存

- 需在 Mapper XML 中 **&lt;cache&gt;** 或 **&lt;cache-ref&gt;**；跨 SqlSession、同一 Mapper 可共享；注意序列化、事务提交后才放入、多表更新时的失效范围。

### 7.3 分布式与一致性

- 单机二级缓存在多实例下会不一致；分布式场景可用 **Redis 等实现 Cache 接口** 做分布式二级缓存，或关闭二级缓存、用业务层 Redis 缓存。

------


## 第8章：MyBatis 与 Spring/Spring Boot 整合

### 8.1 MyBatis-Spring

- **mybatis-spring** 提供 **SqlSessionFactoryBean**（注入 DataSource、指定 configLocation/mapperLocations）、**SqlSessionTemplate**（替代手动 SqlSession，与 Spring 事务联动）、**MapperFactoryBean** 或 **MapperScannerConfigurer** 扫描 Mapper 接口。

### 8.2 Spring Boot 自动装配

- **mybatis-spring-boot-starter** 根据 **spring.datasource** 自动创建 SqlSessionFactory；**@MapperScan** 指定 Mapper 包；**mybatis.mapper-locations**、**mybatis.type-aliases-package**、**mybatis.configuration.map-underscore-to-camel-case** 等可在 application.yml 配置。

### 8.3 多数据源

- 定义多份 DataSource、多份 SqlSessionFactory（指定不同 mapperLocations 或包），用 **@Primary** 标主库；不同 Mapper 包或注解路由到不同 Factory；事务管理器需对应或使用 **AbstractRoutingDataSource** 动态切换。

### 8.4 事务

- **@Transactional** 由 Spring 管理；同一事务内 MyBatis 使用同一 Connection，提交/回滚由 Spring 统一处理；注意传播行为与多数据源时的事务管理器绑定。

------

## 第9章：插件与拦截器机制

### 9.1 插件原理

- 实现 **Interceptor** 接口，用 **@Intercepts** 指定拦截的 **Executor/StatementHandler/ResultSetHandler/ParameterHandler** 及方法；通过 **Plugin.wrap** 生成代理，形成责任链；在 **intercept** 中可在目标方法前后增强（如改 SQL、记日志、分页）。

### 9.2 拦截点

- **Executor**：update/query/commit 等；**StatementHandler**：prepare/parameterize 等；**ResultSetHandler**：handleResultSets。分页插件多在 Executor.query 或 StatementHandler 层改写 SQL。

### 9.3 常用插件

- **PageHelper**：物理分页；**MyBatis-Plus**：内置分页、乐观锁、逻辑删除等；**P6Spy**：SQL 日志与耗时。自定义插件需注意与现有插件的顺序与兼容性。

------

## 第10章：MyBatis-Plus 框架实战

### 10.1 概述

- MyBatis-Plus 在 MyBatis 基础上提供 **BaseMapper**（通用 CRUD）、**Wrapper** 条件构造、**分页插件**、**乐观锁**、**逻辑删除**、**自动填充**、**代码生成器**，减少 XML 与样板代码。

### 10.2 常用能力

- 继承 **BaseMapper&lt;T&gt;** 即拥有 insert/selectById/updateById/deleteById 等；**QueryWrapper/LambdaQueryWrapper** 构造条件；**@TableId**、**@TableField**、**@TableLogic**；分页 **PaginationInnerInterceptor**，乐观锁 **OptimisticLockerInnerInterceptor**。

### 10.3 与 MyBatis 关系

- 兼容 MyBatis 的 XML 与注解，可混用；复杂 SQL 仍写 XML，简单 CRUD 用 Plus；详见《MyBatis-Plus 学习笔记》及本目录《常用API与使用场景》。

------

## 第11章：MyBatis 项目实战案例

### 11.1 黑马点评

- 热点数据用 Redis 缓存，DB 用 MyBatis 查；缓存更新策略（先更新 DB 再删缓存）；多条件查询用动态 SQL（&lt;if&gt;/&lt;where&gt;）；可配合 MyBatis-Plus 做简单列表与分页。

### 11.2 苍穹外卖

- 订单等模块 Mapper 设计（订单主表、明细、状态流转）；多表联查用 resultMap 或分步查询；**@Transactional** 保证下单与库存一致性；定时任务中调用 Mapper 做超时关单、统计等。

### 11.3 分布式与缓存

- 多实例下二级缓存易不一致，通常关闭或改用 Redis 实现 Cache 接口；业务层用 Redis 做查询缓存时，写操作后主动失效 key，与 MyBatis 解耦。

## 第12章：MyBatis 面试高频题与源码剖析

| 问题 | 要点 |
|------|------|
| 与 JDBC 区别？ | MyBatis 封装连接与结果映射，SQL 可配置在 XML/注解，支持动态 SQL、缓存、与 Spring 事务整合。 |
| 执行流程？ | Mapper 代理 → SqlSession → Executor → StatementHandler → JDBC → ResultSetHandler 映射。 |
| 一级 vs 二级缓存？ | 一级 Session 级、默认开；二级 Mapper 级、需配置、跨 Session。 |
| 动态 SQL 实现？ | OGNL 解析 &lt;if&gt;/&lt;foreach&gt; 等，拼接成最终 SQL。 |
| 与 Spring 事务？ | SqlSessionTemplate 绑定当前事务的 Connection，提交/回滚由 Spring 控制。 |
| Mapper 无实现类？ | JDK 动态代理，根据方法名与参数找到 MappedStatement 并执行。 |
| 插件原理？ | Interceptor 拦截四大对象，责任链包装，在目标方法前后增强。 |

## 第13章：常见问题与调试技巧

### 13.1 参数与结果

- **参数**：多参用 **@Param**，否则取不到；**#{}** 与 **${}** 区别（预编译 vs 拼接）。**结果**：列名与属性不一致时用 **resultMap** 或 **mapUnderscoreToCamelCase**；集合用 **&lt;collection&gt;**。

### 13.2 日志与性能

- **logging.level.xxx.mapper=DEBUG** 打印 SQL；延迟加载需开启且关联访问在 Session 内；缓存脏读注意事务边界与二级缓存失效。

### 13.3 事务与慢 SQL

- 传播行为导致新事务时，不同 Connection，一级缓存不共享；慢 SQL 结合 DB 执行计划、索引、分页与批量方式优化。
