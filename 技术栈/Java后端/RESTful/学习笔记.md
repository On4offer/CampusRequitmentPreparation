## 1️⃣ 什么是 REST

### （1）REST 的定义

**REST**（Representational State Transfer，表述性状态转移）是一种**软件架构风格**，由 Roy Fielding 在 2000 年的博士论文中提出。

> 💡 **核心思想**：资源（Resource）通过 URI（统一资源标识符）进行标识，客户端通过 HTTP 协议对资源进行操作，服务器返回资源的表述（Representation，如 JSON、XML）。

### （2）REST 的核心约束

| 约束 | 说明 |
|------|------|
| **客户端-服务器分离** | 客户端和服务器职责分离，提高可移植性和可扩展性 |
| **无状态** | 服务器不保存客户端状态，每个请求都包含所有必要信息 |
| **可缓存** | 响应数据应标记是否可缓存，减少网络传输 |
| **统一接口** | 使用统一的接口规范（URI、HTTP 方法、状态码等） |
| **分层系统** | 允许中间层（代理、网关）存在，对客户端透明 |
| **按需代码**（可选） | 服务器可以临时扩展客户端功能（较少使用） |

### （3）RESTful API 的特点

| 特点 | 说明 |
|------|------|
| **基于资源** | 一切皆资源，通过 URI 唯一标识 |
| **统一接口** | 使用标准 HTTP 方法（GET、POST、PUT、DELETE 等） |
| **无状态** | 每个请求独立，不依赖上下文 |
| **表述多样** | 同一资源可有多种表述格式（JSON、XML 等） |
| **超媒体驱动**（HATEOAS） | 响应中包含相关资源的链接，客户端可动态发现 |

## 2️⃣ RESTful API 与传统 API 的对比

### （1）传统 API（RPC 风格）

```
GET /getUser?id=1
POST /createUser
POST /updateUser?id=1
POST /deleteUser?id=1
```

**特点**：
- URL 中包含动词（get、create、update、delete）
- 使用 POST 方法进行所有操作
- 语义不清晰，难以理解

### （2）RESTful API

```
GET    /users/1          # 获取用户
POST   /users            # 创建用户
PUT    /users/1          # 更新用户
DELETE /users/1          # 删除用户
```

**特点**：
- URL 中只包含名词（资源）
- 使用 HTTP 方法表示操作类型
- 语义清晰，易于理解

### （3）对比总结

| 维度 | 传统 API | RESTful API |
|------|----------|-------------|
| **URL 设计** | 包含动词 | 只包含名词（资源） |
| **HTTP 方法** | 主要用 POST | 使用 GET、POST、PUT、DELETE 等 |
| **语义** | 不清晰 | 清晰直观 |
| **可读性** | 较差 | 较好 |
| **缓存** | 难以利用 | 天然支持（GET 请求可缓存） |
| **扩展性** | 较差 | 较好 |

## 3️⃣ RESTful API 设计原则

### （1）资源命名规范

| 规则 | 示例 |
|------|------|
| **使用名词复数** | `/users`（正确）vs `/user`（不推荐） |
| **使用小写** | `/users`（正确）vs `/Users`（不推荐） |
| **使用连字符分隔** | `/user-profiles`（正确）vs `/userProfiles`（不推荐） |
| **避免层级过深** | `/users/1/orders`（推荐）vs `/users/1/orders/2024/01/01`（避免） |
| **使用查询参数过滤** | `/users?role=admin`（推荐）vs `/admin-users`（不推荐） |

### （2）HTTP 方法使用规范

| HTTP 方法 | 操作 | 幂等性 | 安全性 | 示例 |
|-----------|------|--------|--------|------|
| **GET** | 获取资源 | ✅ 是 | ✅ 是 | `GET /users/1` |
| **POST** | 创建资源 | ❌ 否 | ❌ 否 | `POST /users` |
| **PUT** | 更新资源（整体） | ✅ 是 | ❌ 否 | `PUT /users/1` |
| **PATCH** | 更新资源（部分） | ❌ 否 | ❌ 否 | `PATCH /users/1` |
| **DELETE** | 删除资源 | ✅ 是 | ❌ 否 | `DELETE /users/1` |

> 💡 **幂等性**：多次执行相同操作，结果相同。GET、PUT、DELETE 是幂等的，POST、PATCH 不是。

### （3）状态码使用规范

| 状态码 | 含义 | 使用场景 |
|--------|------|----------|
| **200 OK** | 请求成功 | GET、PUT、PATCH 成功 |
| **201 Created** | 资源创建成功 | POST 创建资源成功 |
| **204 No Content** | 请求成功但无返回内容 | DELETE 成功 |
| **400 Bad Request** | 请求参数错误 | 参数校验失败 |
| **401 Unauthorized** | 未认证 | 缺少 token 或 token 无效 |
| **403 Forbidden** | 已认证但无权限 | 权限不足 |
| **404 Not Found** | 资源不存在 | 资源 ID 错误 |
| **409 Conflict** | 资源冲突 | 资源已存在、版本冲突 |
| **500 Internal Server Error** | 服务器内部错误 | 代码异常 |

### （4）版本控制策略

| 策略 | 示例 | 优点 | 缺点 |
|------|------|------|------|
| **URL 路径** | `/api/v1/users` | 简单直观 | URL 冗长 |
| **请求头** | `Accept: application/vnd.api.v1+json` | URL 简洁 | 难以测试 |
| **查询参数** | `/users?version=v1` | 简单 | 不符合 REST 规范 |

> 💡 **推荐**：生产环境推荐使用 **URL 路径** 方式，便于理解和测试。

------

## 4️⃣ RESTful API 的优势

| 优势 | 说明 |
|------|------|
| **统一接口** | 所有 API 遵循相同规范，降低学习成本 |
| **无状态** | 服务器不保存状态，易于水平扩展 |
| **可缓存** | GET 请求可缓存，减少服务器压力 |
| **跨平台** | 基于 HTTP 协议，支持任何平台 |
| **易于测试** | 可用浏览器、curl、Postman 等工具测试 |
| **前后端分离** | 前后端通过 API 交互，解耦开发 |

| 面试官提问 | 回答思路 |
|------------|----------|
| 什么是 REST？ | REST 是一种软件架构风格，核心思想是资源通过 URI 标识，客户端通过 HTTP 协议对资源进行操作，服务器返回资源的表述。主要约束包括客户端-服务器分离、无状态、可缓存、统一接口等。 |

### 常见坑与注意点

| 现象 / 易错点 | 原因 | 正确做法 / 怎么记 |
|---------------|------|-------------------|
| URL 中包含动词 | 不符合 RESTful 规范 | URL 只包含名词（资源），用 HTTP 方法表示操作 |
| 所有操作都用 POST | 语义不清晰，无法利用缓存 | 根据操作类型使用合适的 HTTP 方法 |
| 状态码使用不当 | 客户端难以判断请求结果 | 返回符合语义的状态码（200、201、400、404 等） |
| 忽略版本控制 | API 升级时影响现有客户端 | 在 URL 路径中添加版本号（如 `/api/v1/users`） |
| 返回数据格式不统一 | 客户端解析困难 | 统一使用 JSON 格式，定义统一的数据结构 |

## 1️⃣ HTTP 方法详解

### （1）GET 方法

**用途**：获取资源

**特点**：
- 幂等性：是
- 安全性：是
- 可缓存：是
- 请求体：无

**示例**：
```http
GET /api/v1/users/1 HTTP/1.1
Host: example.com
Accept: application/json
```

**使用场景**：
- 获取单个资源：`GET /users/1`
- 获取资源列表：`GET /users`
- 带查询参数过滤：`GET /users?role=admin&page=1&size=10`

### （2）POST 方法

**用途**：创建资源

**特点**：
- 幂等性：否
- 安全性：否
- 可缓存：否
- 请求体：有

**示例**：
```http
POST /api/v1/users HTTP/1.1
Host: example.com
Content-Type: application/json

{
  "name": "张三",
  "email": "zhangsan@example.com"
}
```

**使用场景**：
- 创建资源：`POST /users`
- 执行复杂查询：`POST /users/search`

### （3）PUT 方法

**用途**：更新资源（整体更新）

**特点**：
- 幂等性：是
- 安全性：否
- 可缓存：否
- 请求体：有

**示例**：
```http
PUT /api/v1/users/1 HTTP/1.1
Host: example.com
Content-Type: application/json

{
  "name": "李四",
  "email": "lisi@example.com"
}
```

**使用场景**：
- 更新整个资源：`PUT /users/1`

### （4）PATCH 方法

**用途**：更新资源（部分更新）

**特点**：
- 幂等性：否
- 安全性：否
- 可缓存：否
- 请求体：有

**示例**：
```http
PATCH /api/v1/users/1 HTTP/1.1
Host: example.com
Content-Type: application/json

{
  "name": "王五"
}
```

**使用场景**：
- 更新资源的部分字段：`PATCH /users/1`

### （5）DELETE 方法

**用途**：删除资源

**特点**：
- 幂等性：是
- 安全性：否
- 可缓存：否
- 请求体：无

**示例**：
```http
DELETE /api/v1/users/1 HTTP/1.1
Host: example.com
```

**使用场景**：
- 删除资源：`DELETE /users/1`

### （6）其他 HTTP 方法

| 方法 | 用途 | 幂等性 | 安全性 |
|------|------|--------|--------|
| **HEAD** | 获取响应头 | ✅ 是 | ✅ 是 |
| **OPTIONS** | 获取支持的 HTTP 方法 | ✅ 是 | ✅ 是 |
| **TRACE** | 回显服务器收到的请求 | ✅ 是 | ✅ 是 |
| **CONNECT** | 建立隧道（如 HTTPS 代理） | ❌ 否 | ❌ 否 |

## 2️⃣ HTTP 状态码详解

### （1）1xx 信息响应

| 状态码 | 含义 | 使用场景 |
|--------|------|----------|
| **100 Continue** | 服务器已收到请求头，客户端应继续发送请求体 | 大文件上传 |
| **101 Switching Protocols** | 服务器同意切换协议 | WebSocket 升级 |

### （2）2xx 成功响应

| 状态码 | 含义 | 使用场景 |
|--------|------|----------|
| **200 OK** | 请求成功 | GET、PUT、PATCH 成功 |
| **201 Created** | 资源创建成功 | POST 创建资源成功，响应头包含 Location |
| **202 Accepted** | 请求已接受，但未处理完成 | 异步处理 |
| **204 No Content** | 请求成功但无返回内容 | DELETE 成功 |
| **206 Partial Content** | 部分内容 | 断点续传 |

### （3）3xx 重定向

| 状态码 | 含义 | 使用场景 |
|--------|------|----------|
| **301 Moved Permanently** | 资源永久重定向 | 域名迁移 |
| **302 Found** | 资源临时重定向 | 临时跳转 |
| **304 Not Modified** | 资源未修改，可使用缓存 | 条件请求 |

### （4）4xx 客户端错误

| 状态码 | 含义 | 使用场景 |
|--------|------|----------|
| **400 Bad Request** | 请求参数错误 | 参数校验失败 |
| **401 Unauthorized** | 未认证 | 缺少 token 或 token 无效 |
| **403 Forbidden** | 已认证但无权限 | 权限不足 |
| **404 Not Found** | 资源不存在 | 资源 ID 错误 |
| **405 Method Not Allowed** | HTTP 方法不支持 | 使用了不支持的 HTTP 方法 |
| **409 Conflict** | 资源冲突 | 资源已存在、版本冲突 |
| **422 Unprocessable Entity** | 请求格式正确但语义错误 | 业务规则校验失败 |
| **429 Too Many Requests** | 请求过多 | 触发限流 |

### （5）5xx 服务器错误

| 状态码 | 含义 | 使用场景 |
|--------|------|----------|
| **500 Internal Server Error** | 服务器内部错误 | 代码异常 |
| **502 Bad Gateway** | 网关错误 | 网关或代理服务器错误 |
| **503 Service Unavailable** | 服务不可用 | 服务器维护或过载 |
| **504 Gateway Timeout** | 网关超时 | 网关或代理服务器超时 |

## 3️⃣ HTTP 请求头与响应头

### （1）常用请求头

| 请求头 | 说明 | 示例 |
|---------|------|------|
| **Accept** | 客户端可接受的响应类型 | `Accept: application/json` |
| **Content-Type** | 请求体的类型 | `Content-Type: application/json` |
| **Authorization** | 认证信息 | `Authorization: Bearer token` |
| **User-Agent** | 客户端信息 | `User-Agent: Mozilla/5.0` |
| **If-None-Match** | 条件请求（ETag） | `If-None-Match: "33a64df551425fcc55e4d42a148795d9f25f89d4"` |
| **If-Modified-Since** | 条件请求（最后修改时间） | `If-Modified-Since: Wed, 21 Oct 2024 07:28:00 GMT` |

### （2）常用响应头

| 响应头 | 说明 | 示例 |
|---------|------|------|
| **Content-Type** | 响应体的类型 | `Content-Type: application/json` |
| **Content-Length** | 响应体的长度 | `Content-Length: 1024` |
| **ETag** | 资源的版本标识 | `ETag: "33a64df551425fcc55e4d42a148795d9f25f89d4"` |
| **Last-Modified** | 资源的最后修改时间 | `Last-Modified: Wed, 21 Oct 2024 07:28:00 GMT` |
| **Location** | 重定向或新创建资源的 URL | `Location: /api/v1/users/1` |
| **Cache-Control** | 缓存控制 | `Cache-Control: max-age=3600` |
| **WWW-Authenticate** | 认证挑战 | `WWW-Authenticate: Bearer realm="api"` |

## 4️⃣ RESTful API 的请求与响应格式

### （1）请求格式

**示例：创建用户**
```http
POST /api/v1/users HTTP/1.1
Host: example.com
Content-Type: application/json
Accept: application/json

{
  "name": "张三",
  "email": "zhangsan@example.com",
  "age": 25
}
```

### （2）响应格式

**成功响应（200 OK）**
```json
{
  "code": 200,
  "message": "success",
  "data": {
    "id": 1,
    "name": "张三",
    "email": "zhangsan@example.com",
    "age": 25,
    "createdAt": "2024-01-01T00:00:00Z"
  }
}
```

**创建成功（201 Created）**
```json
{
  "code": 201,
  "message": "User created successfully",
  "data": {
    "id": 1,
    "name": "张三",
    "email": "zhangsan@example.com",
    "age": 25,
    "createdAt": "2024-01-01T00:00:00Z"
  }
}
```

**错误响应（400 Bad Request）**
```json
{
  "code": 400,
  "message": "Validation failed",
  "errors": [
    {
      "field": "email",
      "message": "Email is required"
    },
    {
      "field": "age",
      "message": "Age must be greater than 0"
    }
  ],
  "timestamp": "2024-01-01T00:00:00Z",
  "path": "/api/v1/users"
}
```

**未找到（404 Not Found）**
```json
{
  "code": 404,
  "message": "User not found",
  "timestamp": "2024-01-01T00:00:00Z",
  "path": "/api/v1/users/999"
}
```

**服务器错误（500 Internal Server Error）**
```json
{
  "code": 500,
  "message": "Internal server error",
  "timestamp": "2024-01-01T00:00:00Z",
  "path": "/api/v1/users"
}
```

## 🎯 面试常见追问

| 面试官提问 | 回答思路 |
|------------|----------|
| GET 和 POST 的区别？ | GET 用于获取资源，POST 用于创建资源；GET 幂等且安全，POST 不幂等且不安全；GET 可缓存，POST 不可缓存；GET 请求无请求体，POST 有请求体；GET 请求参数在 URL 中，POST 在请求体中。 |
| 什么是幂等性？ | 幂等性指多次执行相同操作，结果相同。GET、PUT、DELETE 是幂等的，POST、PATCH 不是幂等的。幂等性在分布式系统中很重要，可以安全重试。 |
| 200、201、204 状态码的区别？ | 200 OK 表示请求成功；201 Created 表示资源创建成功；204 No Content 表示请求成功但无返回内容（如 DELETE）。 |
| 401 和 403 的区别？ | 401 Unauthorized 表示未认证（缺少 token 或 token 无效）；403 Forbidden 表示已认证但无权限。 |
| 如何实现 API 版本控制？ | 常见方式有 URL 路径（/api/v1/users）、请求头（Accept: application/vnd.api.v1+json）、查询参数（/users?version=v1）。推荐使用 URL 路径方式，简单直观。 |

------

### 常见坑与注意点

| 现象 / 易错点 | 原因 | 正确做法 / 怎么记 |
|---------------|------|-------------------|
| GET 请求带请求体 | 不符合 HTTP 规范 | GET 请求参数放在 URL 查询字符串中 |
| 返回 200 表示错误 | 状态码语义不清晰 | 错误时返回 4xx 或 5xx 状态码 |
| 响应格式不统一 | 客户端解析困难 | 统一响应格式，包含 code、message、data 等字段 |
| 忽略 ETag 和缓存 | 无法利用 HTTP 缓存 | 返回 ETag 和 Cache-Control 响应头，支持条件请求 |
| 错误信息不详细 | 难以排查问题 | 返回详细的错误信息，包括字段、错误原因等 |

## 1️⃣ 资源命名规范

### （1）资源命名原则

| 原则 | 说明 | 示例 |
|------|------|------|
| **使用名词** | 资源是名词，不是动词 | `/users`（正确）vs `/getUsers`（错误） |
| **使用复数** | 资源集合使用复数形式 | `/users`（正确）vs `/user`（不推荐） |
| **使用小写** | URL 使用小写字母 | `/users`（正确）vs `/Users`（错误） |
| **使用连字符** | 多个单词用连字符连接 | `/user-profiles`（正确）vs `/userProfiles`（不推荐） |
| **避免层级过深** | 层级不超过 3 层 | `/users/1/orders`（推荐）vs `/users/1/orders/2024/01/01`（避免） |

### （2）常见资源命名示例

| 资源 | URL 设计 | 说明 |
|------|----------|------|
| 用户 | `/users` | 用户集合 |
| 单个用户 | `/users/{id}` | 特定用户 |
| 用户订单 | `/users/{id}/orders` | 用户的订单 |
| 单个订单 | `/orders/{id}` | 特定订单 |
| 订单商品 | `/orders/{id}/items` | 订单的商品 |

### （3）避免的反模式

| 反模式 | 问题 | 正确做法 |
|--------|------|----------|
| `/getUsers` | URL 包含动词 | `/users` |
| `/user` | 使用单数 | `/users` |
| `/Users` | 使用大写 | `/users` |
| `/userProfiles` | 使用驼峰命名 | `/user-profiles` |
| `/api/v1/users/1/orders/2024/01/01/items` | 层级过深 | `/orders?userId=1&date=2024-01-01` |

## 5️⃣ HATEOAS（超媒体驱动）

### （1）什么是 HATEOAS

**HATEOAS**（Hypermedia as the Engine of Application State）是 REST 架构的一个约束，要求响应中包含相关资源的链接，客户端可以通过这些链接动态发现和操作资源。

### （2）HATEOAS 示例

**传统响应**：
```json
{
  "id": 1,
  "name": "张三",
  "email": "zhangsan@example.com"
}
```

**HATEOAS 响应**：
```json
{
  "id": 1,
  "name": "张三",
  "email": "zhangsan@example.com",
  "_links": {
    "self": {
      "href": "/api/v1/users/1"
    },
    "update": {
      "href": "/api/v1/users/1",
      "method": "PUT"
    },
    "delete": {
      "href": "/api/v1/users/1",
      "method": "DELETE"
    },
    "orders": {
      "href": "/api/v1/users/1/orders"
    }
  }
}
```

### （3）HATEOAS 的优缺点

| 优点 | 缺点 |
|------|------|
| 客户端不需要硬编码 URL | 响应体变大 |
| API 更新时客户端不需要修改 | 实现复杂 |
| 更符合 REST 架构 | 学习成本高 |

> 💡 **建议**：HATEOAS 是 REST 架构的理想状态，但实际项目中可以根据需求选择是否实现。

### （2）用户 Service 实现

```java
@Service
@Transactional
@RequiredArgsConstructor
public class UserServiceImpl implements UserService {
    
    private final UserRepository userRepository;
    private final ModelMapper modelMapper;
    
    @Override
    public UserDTO createUser(UserCreateRequest request) {
        User user = modelMapper.map(request, User.class);
        User savedUser = userRepository.save(user);
        return modelMapper.map(savedUser, UserDTO.class);
    }
    
    @Override
    @Transactional(readOnly = true)
    public UserDTO getUserById(Long id) {
        User user = userRepository.findById(id)
            .orElseThrow(() -> new ResourceNotFoundException("User not found"));
        return modelMapper.map(user, UserDTO.class);
    }
    
    @Override
    @Transactional(readOnly = true)
    public List<UserDTO> getAllUsers() {
        List<User> users = userRepository.findAll();
        return users.stream()
            .map(user -> modelMapper.map(user, UserDTO.class))
            .collect(Collectors.toList());
    }
    
    @Override
    @Transactional(readOnly = true)
    public Page<UserDTO> getUsers(Pageable pageable) {
        Page<User> users = userRepository.findAll(pageable);
        return users.map(user -> modelMapper.map(user, UserDTO.class));
    }
    
    @Override
    public UserDTO updateUser(Long id, UserUpdateRequest request) {
        User user = userRepository.findById(id)
            .orElseThrow(() -> new ResourceNotFoundException("User not found"));
        modelMapper.map(request, user);
        User updatedUser = userRepository.save(user);
        return modelMapper.map(updatedUser, UserDTO.class);
    }
    
    @Override
    public void deleteUser(Long id) {
        if (!userRepository.existsById(id)) {
            throw new ResourceNotFoundException("User not found");
        }
        userRepository.deleteById(id);
    }
    
    @Override
    @Transactional(readOnly = true)
    public List<UserDTO> searchUsers(String keyword) {
        List<User> users = userRepository.searchByName(keyword);
        return users.stream()
            .map(user -> modelMapper.map(user, UserDTO.class))
            .collect(Collectors.toList());
    }
}
```

## 5️⃣ Controller 层

### （1）用户 Controller

```java
@RestController
@RequestMapping("/users")
@RequiredArgsConstructor
@Tag(name = "用户管理", description = "用户相关接口")
public class UserController {
    
    private final UserService userService;
    
    @GetMapping
    @Operation(summary = "获取用户列表", description = "分页获取用户列表")
    public ApiResponse<Page<UserDTO>> getUsers(
            @RequestParam(defaultValue = "0") int page,
            @RequestParam(defaultValue = "10") int size,
            @RequestParam(defaultValue = "createdAt") String sort) {
        
        Pageable pageable = PageRequest.of(page, size, Sort.by(sort));
        Page<UserDTO> users = userService.getUsers(pageable);
        return ApiResponse.success(users);
    }
    
    @GetMapping("/{id}")
    @Operation(summary = "获取用户详情", description = "根据 ID 获取用户详情")
    public ApiResponse<UserDTO> getUserById(
```

```java
@Data
@NoArgsConstructor
@AllArgsConstructor
public class ApiResponse<T> {
    
    private int code;
    private String message;
    private T data;
    private LocalDateTime timestamp;
    
    public static <T> ApiResponse<T> success(T data) {
        return new ApiResponse<>(200, "success", data, LocalDateTime.now());
    }
    
    public static <T> ApiResponse<T> success(int code, String message, T data) {
        return new ApiResponse<>(code, message, data, LocalDateTime.now());
    }
    
    public static <T> ApiResponse<T> error(int code, String message) {
        return new ApiResponse<>(code, message, null, LocalDateTime.now());
    }
    
    public static <T> ApiResponse<T> error(int code, String message, List<FieldError> errors) {
        ApiResponse<T> response = new ApiResponse<>(code, message, null, LocalDateTime.now());
        response.setErrors(errors);
        return response;
    }
    
    private List<FieldError> errors;
    
    @Data
    @NoArgsConstructor
    @AllArgsConstructor
    public static class FieldError {
        private String field;
        private String message;
    }
}
```

## 🎯 面试常见追问

| 面试官提问 | 回答思路 |
|------------|----------|
| @RestController 和 @Controller 的区别？ | @RestController 等价于 @Controller + @ResponseBody，返回值直接写响应体（通常序列化为 JSON）；@Controller 返回视图名，由 ViewResolver 解析。 |
| @RequestMapping 和 @GetMapping 的区别？ | @RequestMapping 是通用的请求映射注解，可以指定 HTTP 方法、路径等；@GetMapping 是 @RequestMapping(method = RequestMethod.GET) 的简写，专门用于 GET 请求。 |
| 如何实现分页？ | 使用 Spring Data JPA 的 Pageable 接口，Controller 接收 page 和 size 参数，Service 调用 Repository 的 findAll(Pageable) 方法，返回 Page 对象。 |
| 如何实现统一响应封装？ | 创建 ApiResponse 类，包含 code、message、data 等字段，在 Controller 中返回 ApiResponse 对象，由 Spring MVC 序列化为 JSON。 |

------

### 常见坑与注意点

| 现象 / 易错点 | 原因 | 正确做法 / 怎么记 |
|---------------|------|-------------------|
| Controller 直接返回实体 | 响应格式不统一 | 使用 DTO 和统一响应封装 |
| 忽略事务管理 | 数据不一致 | Service 方法上添加 @Transactional |
| 参数验证失败时不返回详细错误 | 客户端难以定位问题 | 使用 @Valid 注解，返回详细的错误信息 |
| 忽略分页信息 | 客户端无法分页 | 响应中返回分页信息（page、size、total、totalPages） |
| 直接返回数据库实体 | 暴露内部结构 | 使用 DTO 进行数据传输 |

# 📎 附录：面试题速查

## 1️⃣ RESTful API 基础

| 问题 | 答案 |
|------|------|
| 什么是 REST？ | REST（Representational State Transfer）是一种软件架构风格，核心思想是资源通过 URI 标识，客户端通过 HTTP 协议对资源进行操作，服务器返回资源的表述。 |
| 什么是 RESTful API？ | RESTful API 是遵循 REST 架构风格的 API，基于资源，使用标准 HTTP 方法，统一接口，无状态。 |
| REST 的核心约束是什么？ | 客户端-服务器分离、无状态、可缓存、统一接口、分层系统、按需代码（可选）。 |
| 什么是幂等性？ | 幂等性指多次执行相同操作，结果相同。GET、PUT、DELETE 是幂等的，POST、PATCH 不是幂等的。 |

## 2️⃣ HTTP 方法与状态码

| 问题 | 答案 |
|------|------|
| GET 和 POST 的区别？ | GET 用于获取资源，POST 用于创建资源；GET 幂等且安全，POST 不幂等且不安全；GET 可缓存，POST 不可缓存。 |
| PUT 和 PATCH 的区别？ | PUT 用于整体更新资源，PATCH 用于部分更新资源；PUT 是幂等的，PATCH 不是幂等的。 |
| 200、201、204 状态码的区别？ | 200 OK 表示请求成功；201 Created 表示资源创建成功；204 No Content 表示请求成功但无返回内容（如 DELETE）。 |
| 401 和 403 的区别？ | 401 Unauthorized 表示未认证（缺少 token 或 token 无效）；403 Forbidden 表示已认证但无权限。 |
| 404 和 500 状态码的区别？ | 404 Not Found 表示资源不存在；500 Internal Server Error 表示服务器内部错误。 |

## 3️⃣ RESTful API 设计

| 问题 | 答案 |
|------|------|
| 如何设计 RESTful API 的 URL？ | 资源命名使用名词复数、小写、连字符分隔；使用合适的 HTTP 方法；嵌套资源不超过 2 层；特殊操作使用 POST + 动作名。 |
| 如何实现分页？ | 使用 page 和 size 参数（或 offset 和 limit），响应中返回分页信息（当前页、每页大小、总数、总页数）。 |
| 如何实现过滤和排序？ | 过滤使用查询参数（如 ?role=admin），排序使用 sort 参数（如 ?sort=createdAt 或 ?sort=-createdAt）。 |
| 什么是 HATEOAS？ | HATEOAS 是超媒体作为应用状态引擎，要求响应中包含相关资源的链接，客户端可以通过这些链接动态发现和操作资源。 |
| 如何实现 API 版本控制？ | 常见方式有 URL 路径（/api/v1/users）、请求头（Accept: application/vnd.api.v1+json）、查询参数（/users?version=v1）。推荐使用 URL 路径方式。 |

## 4️⃣ Spring Boot 实现

| 问题 | 答案 |
|------|------|
| @RestController 和 @Controller 的区别？ | @RestController 等价于 @Controller + @ResponseBody，返回值直接写响应体；@Controller 返回视图名，由 ViewResolver 解析。 |
| @RequestMapping 和 @GetMapping 的区别？ | @RequestMapping 是通用的请求映射注解；@GetMapping 是 @RequestMapping(method = RequestMethod.GET) 的简写，专门用于 GET 请求。 |
| 如何实现参数验证？ | 使用 JSR-303 验证注解（如 @NotNull、@NotBlank、@Email），在 Controller 方法参数上添加 @Valid 注解，Spring Boot 会自动验证参数。 |
| 如何实现全局异常处理？ | 使用 @RestControllerAdvice 注解创建全局异常处理器类，使用 @ExceptionHandler 注解处理特定类型的异常，返回统一的响应格式。 |
| 如何实现统一响应封装？ | 创建 ApiResponse 类，包含 code、message、data 等字段，在 Controller 中返回 ApiResponse 对象，由 Spring MVC 序列化为 JSON。 |
