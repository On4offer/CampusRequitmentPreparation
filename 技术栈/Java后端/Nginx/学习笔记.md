------

------

### 1.1 诞生与发展

> **一句话**：**Nginx** 是**高并发、低内存**的 Web 服务器 / **反向代理** / 负载均衡器，常放在最前面接请求，把动态请求转给后端（如 Spring Boot），自己负责静态资源或做负载分发。
> **为什么需要**：传统「一连接一线程」的服务器在高并发时线程数爆炸、内存吃紧；Nginx 用**事件驱动**（epoll）和**多 Worker 无阻塞**，少量进程就能扛大量连接。
> **类比**：传统服务器像「一个顾客一个服务员」；Nginx 像「一个服务员照看多张桌子」，谁有需求（事件就绪）就处理谁，不空等。

**Nginx** 由 Igor Sysoev 开发，2004 年发布，主打**高并发、低内存占用**；广泛用于**反向代理、负载均衡、静态资源服务、HTTPS 终结**，是互联网与云原生场景中最常见的前置 Web 服务器之一。

| 项目 | 说明 |
|------|------|
| **定位** | 高性能 Web 服务器 / 反向代理 / 负载均衡器 |
| **典型角色** | 前置层：接收客户端请求，转发给后端（Tomcat、Spring Boot、Gateway）或直接响应静态文件 |

------

### 1.2 与 Apache 对比

| 对比项 | Nginx | Apache |
|--------|--------|--------|
| **模型** | 事件驱动、多 Worker 无阻塞 | 多进程/多线程、每连接一线程或一进程 |
| **高并发** | 单机数万并发连接，内存占用低 | 连接数高时进程/线程数多，占用大 |
| **静态资源** | sendfile、高效发文件 | 支持，但高并发下不如 Nginx 轻量 |
| **动态脚本** | 需通过 FastCGI/代理转发给 PHP 等 | 模块丰富，可直接嵌 PHP 等 |
| **常见用法** | 作前置代理，动态请求转后端 | 传统 LAMP 中 Web 服务器；也可与 Nginx 组合 |

> 💬 **一句话**：Nginx 适合**高并发、静态与反向代理**；Apache 适合**模块化与动态脚本**。实践中 Nginx 常作前置，将动态请求 **proxy_pass** 给后端（如 Spring Boot、Gateway）。

------

### 1.3 核心特性

| 特性 | 说明 |
|------|------|
| **高并发** | 基于 **epoll**（Linux）/ **kqueue**（BSD），单 Worker 可处理大量连接，单机支撑数万并发。 |
| **事件驱动** | 异步非阻塞 I/O，读写未就绪时不阻塞，处理其他连接，少量进程即可应对高并发。 |
| **静态资源** | **sendfile** 零拷贝发文件，适合静态页、图片、前端资源。 |
| **模块化** | 核心 + 各功能模块（proxy、rewrite、gzip、ssl、upstream 等），可编译或动态加载。 |

------

### 1.4 工作流程（Master-Worker）

```
                    Master 进程
                         │
        读配置、信号管理、启动/回收 Worker
                         │
    ┌────────────┬────────────┬────────────┐
    ▼            ▼            ▼            ▼
 Worker 1    Worker 2    Worker 3    ...
    │            │            │
 监听端口    竞争 accept   处理请求
 处理请求    location 匹配 → 静态 root/alias 或 proxy_pass 转发
```

- **Master**：读配置、建信号管道、启动并管理 **Worker**；不处理请求。
- **Worker**：监听端口、**竞争 accept**、处理请求；经 **location** 匹配后执行**静态**（root/alias）或 **proxy_pass** 转发到后端。
- 进程间不共享连接，**无锁竞争**，利于多核与高并发。

### 2.2 安装后验证与启停

#### （1）验证安装

```bash
# 查看版本
nginx -v

# 检查配置文件语法（修改配置后必做）
nginx -t
```

若 **nginx -t** 报错，根据提示修正 **syntax error** 或 **open() failed**（路径、权限等）。

#### （2）启停与重载

| 操作 | 命令 | 说明 |
|------|------|------|
| 启动 | `nginx` 或 `systemctl start nginx` | 前台/后台由配置或 systemd 决定 |
| 立即停止 | `nginx -s stop` | 强制停止 |
| 优雅停止 | `nginx -s quit` | 处理完当前请求后退出 |
| 重载配置 | `nginx -s reload` | 不中断服务，重新加载配置 |
| 重新打开日志 | `nginx -s reopen` | 配合 logrotate 切割后使用 |

> 💡 **重点**：改完 **nginx.conf** 后先 **nginx -t**，再 **nginx -s reload**，避免配置错误导致服务异常。

### 2.3 配置文件结构

| 层级 | 说明 | 常见配置项 |
|------|------|------------|
| **main** | 全局 | worker_processes、user、error_log、pid |
| **events** | 事件模型 | worker_connections |
| **http** | HTTP 通用 | include mime.types、default_type、log_format、access_log、sendfile、keepalive_timeout 等 |
| **server** | 虚拟主机 | listen、server_name、location、root、proxy_pass 等 |
| **location** | URL 匹配与逻辑 | 匹配规则 + root/alias/proxy_pass/return 等 |
| **upstream** | 后端组（在 http 内） | server、负载策略、keepalive |

**常见路径**：Linux 主配置 **/etc/nginx/nginx.conf**，站点配置常在 **/etc/nginx/conf.d/*.conf** 并被主配置 **include**。

### 2.4 常见安装与运行问题

| 现象 | 可能原因 | 排查/处理 |
|------|----------|-----------|
| **nginx -t 报 syntax error** | 配置语法错误（少分号、括号不匹配等） | 按报错行号修改，注意 `;` 和 `{}` |
| **80 端口被占用** | 已有 Apache/其他程序占用 | `ss -tlnp \| grep :80` 或 `lsof -i :80`，改 Nginx **listen** 或停掉占用的服务 |

## 第三章：请求处理机制（Master-Worker 与事件驱动）

------

> 本章说明 **Master-Worker** 模型、**事件驱动**（epoll）与**连接数**计算，便于理解「为什么 Nginx 能高并发」并在面试中表述清楚。

------

### 3.1 Master-Worker 模型

- **Master**：读取配置、创建信号管道、**启动若干 Worker**；不处理请求；收到 **reload** 时重新加载配置并通知 Worker 优雅重启。
- **Worker**：**竞争 accept** 新连接，每个连接在事件循环中异步处理；进程间不共享连接，**无锁**，多核下可充分利用 CPU。
- Worker 数量通常设为 **worker_processes auto;**（等于 CPU 核数）或显式指定。

------

### 3.2 事件驱动（epoll / kqueue）

- Linux 下使用 **epoll**，BSD 下使用 **kqueue**，单 Worker 内**一个事件循环**处理多连接。
- **异步非阻塞**：读/写未就绪时不阻塞，注册事件后处理其他连接；就绪后再处理，从而用少量线程/进程支撑大量连接。
- 与「每连接一线程」的模型对比：Nginx 不因连接数增多而线性增加进程/线程数，内存与调度开销更小。

------

### 3.3 连接数

| 配置项 | 说明 |
|--------|------|
| **worker_connections** | 单个 Worker 可同时处理的最大连接数（在 events 块中）。 |
| **理论并发** | 约 **worker_processes × worker_connections**；实际还受 **worker_rlimit_nofile**、系统 **ulimit** 与 **file descriptor** 限制。 |

调大连接数时需同步调大 **worker_rlimit_nofile** 与系统 **nofile** 限制。

### 常见坑与注意点

| 现象 / 易错点 | 原因 | 怎么改 / 怎么记 |
|---------------|------|-----------------|
| 调大 worker_connections 仍报 too many open files | 系统 ulimit 或 worker_rlimit_nofile 未同步调大 | 在 **main** 中设 **worker_rlimit_nofile 65535;** 并保证系统 **ulimit -n** 足够；或 /etc/security/limits.conf 设 nofile。 |
| Worker 数设过多反而变慢 | 过多进程竞争 accept、上下文切换增加 | **worker_processes auto;** 或等于 CPU 核数即可；高并发主要靠事件驱动，不靠多进程堆叠。 |

## 第四章：反向代理与负载均衡

------

> 本章说明**正向代理与反向代理**的区别、**proxy_pass** 的用法与路径规则、**upstream** 与负载策略、以及一份**完整的反向代理 + 负载均衡**配置示例。

------

### 4.1 正向代理 vs 反向代理

| 类型 | 谁配置代理 | 代理谁 | 典型场景 |
|------|------------|--------|----------|
| **正向代理** | 客户端配置代理 | 代理客户端访问外网 | 科学上网、内网访问外网 |
| **反向代理** | 服务端部署 Nginx | 代理后端服务，客户端只访问 Nginx | 隐藏后端、负载均衡、统一入口 |

> 💬 **一句话**：反向代理是「**客户端 → Nginx → 后端**」；客户端不知道后端是谁，由 Nginx 转发到 **upstream** 中的某台服务器。

------

### 4.2 proxy_pass 与路径

- **proxy_pass** 将当前 **location** 匹配到的请求转发到指定 **upstream** 或地址。
- **末尾是否带 URI** 决定是否「替换」路径：
  - **proxy_pass http://backend;**（末尾无 URI）：转发时保留 **原始 URI**（如 /api/user → 后端收到 /api/user）。
  - **proxy_pass http://backend/;**（末尾带 /）：用 **/ 替换 location 匹配部分**（如 location /api/，请求 /api/user → 后端收到 /user）。
- 常用请求头设置：

```nginx
proxy_set_header Host $host;
proxy_set_header X-Real-IP $remote_addr;
proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
proxy_set_header X-Forwarded-Proto $scheme;
```

------

### 4.3 负载均衡与 upstream

| 策略 | 说明 |
|------|------|
| **默认轮询** | 依次将请求分到各 server。 |
| **weight** | 权重，如 `server 192.168.1.2:8080 weight=3;`。 |
| **ip_hash** | 同一客户端 IP 固定到同一后端，适合有状态。 |
| **least_conn** | 分配到当前连接数最少的后端。 |

**超时与失败**：**proxy_connect_timeout**、**proxy_read_timeout**、**proxy_send_timeout**；**max_fails** 与 **fail_timeout** 可做被动健康检查（连续失败若干次后暂时剔除该 server）。

------

### 4.4 完整示例：反向代理 + 负载均衡

以下示例：对外 80 端口，**/api/** 转发到 Spring Boot 集群（两台），并设置常用头与超时。

```nginx
# 在 http 块中定义后端组
upstream backend_api {
    server 192.168.1.10:8080 weight=1;
    server 192.168.1.11:8080 weight=1;
    keepalive 32;
}

server {
    listen 80;
    server_name localhost;

    location /api/ {
        proxy_pass http://backend_api/;   # 带 / 则 /api/ 被替换为 /
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
        proxy_connect_timeout 5s;
        proxy_read_timeout 60s;
        proxy_send_timeout 60s;
    }
}
```

> 🔹 **注意**：**proxy_pass http://backend_api/** 末尾的 **/** 表示把 **/api/** 前缀替换掉，后端收到的路径是 **/user/list** 而不是 **/api/user/list**；若后端需要保留 **/api**，应写 **proxy_pass http://backend_api;**（无末尾 URI）。

### proxy_pass 常见错误 vs 正确示例

| 场景 | 错误写法 / 现象 | 正确写法 / 做法 |
|------|-----------------|-----------------|
| **后端收到的路径多了或少了前缀** | location /api/ 配 proxy_pass http://backend; 后端收到 /api/user；若后端期望 /user 会 404 | 要去掉前缀：**proxy_pass http://backend/**（末尾带 /），请求 /api/user → 后端 /user；要保留前缀则**不带末尾 URI**。 |
| **upstream 名与变量混用** | proxy_pass http://$host; 等变量时，末尾 URI 规则不同（不替换） | 变量形式 proxy_pass 时，**不会**做 URI 替换；要替换需在 location 里用 rewrite 或固定 upstream 名+末尾 /。 |
| **未设 proxy_set_header Host** | 后端拿到的 Host 是 upstream 地址，虚拟主机或跨域可能错 | 设 **proxy_set_header Host $host;** 与 **X-Real-IP、X-Forwarded-For、X-Forwarded-Proto**，后端才能拿到真实 Host 与协议。 |

### 常见坑与注意点

| 现象 / 易错点 | 原因 | 怎么改 / 怎么记 |
|---------------|------|-----------------|
| 502 Bad Gateway | 后端未启动、端口错、或后端崩溃 | **curl** 直连后端地址测通；看 **error_log** 的 upstream 报错；检查 **proxy_pass** 的 upstream 与端口。 |
| 504 Gateway Timeout | 后端处理慢，超过 proxy_read_timeout | 适当调大 **proxy_read_timeout**；排查后端慢查询、阻塞；不要无脑设很大，避免长时间占连接。 |
| 后端拿不到真实 IP | 未传 X-Real-IP / X-Forwarded-For | 配置 **proxy_set_header X-Real-IP $remote_addr;** 与 **X-Forwarded-For $proxy_add_x_forwarded_for;**。 |

## 第五章：静态资源与缓存优化

------

> 本章说明 **root 与 alias** 的区别、静态资源与缓存/压缩配置、以及一份**完整的静态站点 + 缓存**示例。

------

### 5.1 root 与 alias

| 指令 | 规则 | 示例 |
|------|------|------|
| **root** | 路径 = root + **完整 URI** | location /img/ { root /data/www; } 请求 /img/a.png → /data/www/img/a.png |
| **alias** | 路径 = alias **替换** location 匹配部分 | location /img/ { alias /data/www/; } 请求 /img/a.png → /data/www/a.png |

> 💡 **重点**：**alias** 末尾通常加 **/**，替换时不会多一层目录；**root** 会把 location 前缀当作目录拼在后面。

------

### 5.2 缓存与压缩

- **缓存**：**expires 30d;** 或 **add_header Cache-Control "max-age=3600";**，让浏览器缓存静态资源。
- **压缩**：**gzip on; gzip_types text/plain text/css application/json application/javascript;**，减少传输体积。

------

### 5.3 完整示例：静态资源 + 缓存 + gzip

```nginx
server {
    listen 80;
    server_name static.example.com;
    root /var/www/static;

    location / {
        try_files $uri $uri/ /index.html;
        expires 7d;
        add_header Cache-Control "public, immutable";
    }

    location ~* \.(js|css|png|jpg|jpeg|gif|ico|woff2)$ {
        expires 30d;
        add_header Cache-Control "public, immutable";
    }

    gzip on;
    gzip_types text/plain text/css application/json application/javascript;
    gzip_min_length 1000;
}
```

------

### 5.4 防盗链（可选）

- **valid_referers** 限制 Referer；不合法时 **return 403;** 或重定向。

------

## ✅ 本章小结

| 知识点 | 面试关键词 | 实际应用 |
|--------|------------|----------|
| **root vs alias** | root=根+URI，alias=替换匹配部分 | 正确映射静态路径 |
| **缓存与压缩** | expires、Cache-Control、gzip | 前端资源加速 |

### 常见坑与注意点

| 现象 / 易错点 | 原因 | 怎么改 / 怎么记 |
|---------------|------|-----------------|
| 配了 root 仍 404 | root 会把 **完整 URI** 拼在后面，路径多一层或少一层 | 确认 **root** 目录下是否有对应路径；要「替换」location 前缀时用 **alias**，且 **alias 末尾加 /**。 |
| 静态资源不缓存或缓存不对 | 未设 expires 或 Cache-Control，或 location 未匹配到 | 对静态 location 设 **expires 30d;** 或 **add_header Cache-Control "public, max-age=xxx";**；带 hash 的前端资源可 **immutable**。 |
| gzip 未生效 | 未开 gzip、或 gzip_types 未包含该类型 | **gzip on;** 且 **gzip_types** 包含 text/css、application/javascript 等；**gzip_min_length** 避免压小文件。 |

## 第六章：重定向与重写（rewrite / return）

------

> 本章说明 **rewrite** 与 **return** 的用法、常见场景（HTTP→HTTPS、www 统一）、以及调试方式。

------

### 6.1 rewrite 与 return

| 指令 | 说明 |
|------|------|
| **rewrite regex replacement [flag]** | 按正则改写 URI；flag：**last**（重试 location）、**break**（本 location 内终止）、**redirect**（302）、**permanent**（301）。 |
| **return code [url]** | 直接返回状态码或跳转 URL，**不再执行**后续 rewrite 与 location。 |

- **return** 比 **rewrite** 更直接，适合「直接 301/302 跳转」；**rewrite** 可多次、配合 **last** 重新匹配 location。

------

### 6.2 常见场景

| 场景 | 示例 |
|------|------|
| **HTTP → HTTPS** | 在 listen 80 的 server 中：`return 301 https://$host$request_uri;` |
| **统一 www** | 若 $host 非 www，rewrite 或 return 到 https://www.$host$request_uri |
| **伪静态** | rewrite 将 /article/123 转为内部路径或 proxy_pass 参数 |

------

### 6.3 调试

- **rewrite_log on;** 配合 **error_log ... debug;** 可在错误日志中看到 rewrite 执行过程；排错后可关闭 debug 级别。

------

## ✅ 本章小结

| 知识点 | 面试关键词 | 实际应用 |
|--------|------------|----------|
| **rewrite vs return** | rewrite 可多次、last/break；return 直接返回 | 重定向与伪静态 |
| **常见场景** | HTTP→HTTPS、www 统一 | 生产必备 |

### 常见坑与注意点

| 现象 / 易错点 | 原因 | 怎么改 / 怎么记 |
|---------------|------|-----------------|
| rewrite 死循环 | rewrite 后仍匹配到同一 location，last 又重试，无限循环 | **last** 会重新走 location 匹配，避免改写后仍命中同一规则；或改用 **break** 在本 location 内终止；或 **return** 直接返回。 |
| HTTP 未跳 HTTPS、用户仍访问 80 | 80 的 server 未做 301 跳转 | 在 **listen 80** 的 server 里 **return 301 https://$host$request_uri;** 或 rewrite permanent。 |
| return 与 rewrite 顺序搞反 | 先 rewrite 再 return，return 后的不执行 | **return 会终止**后续所有 rewrite 与 location；要「先跳转再别的不执行」就把 return 放前面。 |

## 第七章：HTTPS 与安全配置

------

> 本章说明 **SSL 配置**、安全头与协议限制、以及一份**完整的 HTTPS server** 示例。

------

### 7.1 SSL 基本配置

```nginx
server {
    listen 443 ssl;
    server_name www.example.com;

    ssl_certificate     /etc/nginx/ssl/cert.pem;
    ssl_certificate_key /etc/nginx/ssl/key.pem;

    ssl_protocols TLSv1.2 TLSv1.3;
    ssl_prefer_server_ciphers on;
    ssl_session_cache shared:SSL:10m;
    ssl_session_timeout 10m;

    # 业务 location ...
}
```

### 常见坑与注意点

| 现象 / 易错点 | 原因 | 怎么改 / 怎么记 |
|---------------|------|-----------------|
| 证书路径错导致启动失败 | ssl_certificate / key 路径不存在或权限不对 | 确认路径存在、Nginx 进程用户可读；**nginx -t** 会报错；Let's Encrypt 等注意证书续期后 reload。 |
| 只开 TLSv1.0/1.1 不安全 | 旧协议有漏洞 | **ssl_protocols TLSv1.2 TLSv1.3;** 禁用 SSLv3 与 TLSv1.0/1.1；**ssl_prefer_server_ciphers on** 且用强套件。 |
| 未做 HTTP→HTTPS 跳转 | 用户仍可访问 80，未强制 HTTPS | 80 的 server 里 **return 301 https://$host$request_uri;** 或 HSTS 头 **add_header Strict-Transport-Security "max-age=31536000";**。 |

------

### 7.2 安全强化

- **HSTS**：**add_header Strict-Transport-Security "max-age=31536000";** 强制浏览器走 HTTPS。
- 禁用旧协议与弱加密；生产证书用 **Let's Encrypt**（certbot）或购买；自签仅测试用。

| **HTTPS** | listen 443 ssl、证书与 key、ssl_protocols | 生产加密 |
| **安全** | HSTS、TLS 版本、证书来源 | 安全加固 |

## 第八章：动静分离与 location 匹配顺序

------

> 本章给出**一份完整的动静分离**配置（静态前端 + /api 转发后端）、并明确 **location 匹配优先级**，以及与 Spring Boot / Gateway 的分工建议。

------

### 8.1 location 匹配优先级（重要）

| 修饰符 | 说明 | 优先级（高→低） |
|--------|------|------------------|
| **=** | 精确匹配 | 1，命中即停 |
| **^~** | 前缀匹配，命中后不再检查正则 | 2 |
| **~** / **~*** | 正则（区分/不区分大小写） | 3，按配置顺序 |
| **无修饰符** | 前缀匹配 | 4，最长前缀优先 |

> 💡 **重点**：**=** 最高；其次 **^~** 前缀；再正则 **~**；最后普通前缀。同类型时：正则按**出现顺序**，普通前缀取**最长匹配**。

------

### 8.2 完整示例：动静分离（前端静态 + /api 转 Spring Boot）

前端打包到 **/var/www/front**，后端 Spring Boot 在 **http://127.0.0.1:8080**；/api 转发到后端，其余先找静态文件再 fallback 到 index.html（前端路由）。

```nginx
upstream backend {
    server 127.0.0.1:8080;
    keepalive 32;
}

server {
    listen 80;
    server_name localhost;
    root /var/www/front;

    # 接口转发到 Spring Boot
    location /api/ {
        proxy_pass http://backend/;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }

    # 静态资源：js/css/img 等
    location ~* \.(js|css|png|jpg|jpeg|gif|ico|svg|woff2)$ {
        expires 30d;
        add_header Cache-Control "public, immutable";
    }

    # 前端路由：单页应用 fallback
    location / {
        try_files $uri $uri/ /index.html;
    }
}
```

------

### 8.3 与 Spring Cloud Gateway 的分工

| 层级 | 典型职责 |
|------|----------|
| **Nginx（前置）** | TLS 终结、静态资源、负载均衡（多实例 Gateway 或直连后端）、限流/安全头、统一入口。 |
| **Spring Cloud Gateway** | 动态路由、鉴权、限流、熔断、灰度、与注册中心集成。 |

> 🔹 **常见组合**：**Nginx** 做 80/443 入口、静态与负载；**Gateway** 做 /api 路由到各微服务；Nginx 的 **proxy_pass** 指向 Gateway 或直指后端，按架构选择。

------

### 8.4 大文件与上传

- **client_max_body_size 50m;** 放宽上传大小；**proxy_request_buffering off** 可按需关闭缓冲（注意内存）；**proxy_read_timeout** 调大长耗时接口。

### 常见坑与注意点

| 现象 / 易错点 | 原因 | 怎么改 / 怎么记 |
|---------------|------|-----------------|
| location 配了却不生效 | 被更高优先级 location 先命中（如 ^~ 或 =） | 记牢**优先级**：= > ^~ > ~ 正则 > 无修饰前缀；同前缀 **^~** 命中后不再走正则；调整顺序或改用更精确的 location。 |
| 前端路由刷新 404 | 单页应用路由在浏览器侧，/user 等直接请求会落到 Nginx，无对应文件 | 静态 location 用 **try_files $uri $uri/ /index.html;** 让不存在路径 fallback 到 index.html，由前端路由接管。 |
| /api 被静态 location 抢 | 普通前缀 / 会匹配所有请求，若 / 在 /api/ 前且能匹配 | 把 **location /api/** 写在前面，或用 **^~** 保证前缀优先；或 / 用 try_files 且 api 单独 location。 |

------

## ✅ 本章小结

| 知识点 | 面试关键词 | 实际应用 |
|--------|------------|----------|
| **location 优先级** | = > ^~ > ~ > 无修饰前缀、最长前缀 | 正确匹配与排错 |
| **动静分离** | 静态 root/try_files、/api proxy_pass | 前后端分离部署 |
| **与 Gateway** | Nginx 前置做 TLS/负载、Gateway 做路由/鉴权 | 架构分工 |

------

**学习要点**：location 优先级必会，否则易出现「配了不生效」；动静分离即静态走 Nginx、/api 走 proxy_pass；与 Gateway 可组合使用，职责分清。

## 第十四章：故障排查与常见错误码

------

> 本章汇总 **403 / 502 / 504** 等常见错误码的**原因与排查步骤**，以及 upstream 超时、权限等常见问题，便于快速排错。

------

### 14.1 常见错误码：原因与排查

| 状态码 | 常见原因 | 排查步骤 |
|--------|----------|----------|
| **403 Forbidden** | ① 目录权限：Nginx 进程用户（如 www-data）无读/执行权限；② **root** 路径错误或不存在；③ **index** 文件不存在且禁止列目录；④ SELinux 拦截 | ① 检查 **user**、目录 **ls -la**、chown/chmod；② 核对 **root**/alias 路径；③ 确认 index 存在；④ **getenforce**、**chcon** 或 setenforce 0 测试 |
| **404 Not Found** | ① 静态文件不存在；② **proxy_pass** 后端路径错误（如多/少一层路径） | ① 核对文件路径与 root/alias；② 核对 **proxy_pass** 末尾是否带 URI、后端实际路径 |
| **502 Bad Gateway** | ① **upstream** 不可达（后端未启、端口错、网络不通）；② 后端进程崩溃或超时断开 | ① **curl** 直连后端、检查防火墙与端口；② 看后端日志、**proxy_read_timeout** 是否过小 |
| **504 Gateway Timeout** | **upstream** 响应超时 | 调大 **proxy_connect_timeout**、**proxy_read_timeout**、**proxy_send_timeout**；排查后端慢 SQL、慢接口 |

------

### 14.2 其他常见问题

| 现象 | 可能原因 | 处理 |
|------|----------|------|
| **upstream 超时** | 后端慢或超时设置过小 | 调大 proxy_*_timeout |
| **proxy_buffer 过小** | 后端响应头过大 | 调大 **proxy_buffer_size**、**proxy_buffers** |
| **rewrite 不生效** | location 优先级或 flag 使用不当 | **rewrite_log on** + error_log debug 查看执行顺序 |
| **限流与防 DDoS** | 需限制请求频率与连接数 | **limit_req_zone**、**limit_conn_zone** 或配合防火墙 |

| 知识点 | 面试关键词 | 实际应用 |
|--------|------------|----------|
| **403** | 权限、root 路径、index、SELinux | 静态 403 排错 |

### 15.1 面试要点表

| 问题 | 要点 |
|------|------|
| 反向代理与负载均衡？ | 反向代理隐藏后端、统一入口；负载均衡用 upstream 将请求分到多台后端（轮询/weight/ip_hash/least_conn）。 |
| Nginx 高并发原理？ | Master-Worker、事件驱动（epoll）、异步非阻塞 I/O、少进程多连接。 |
| rewrite 与 return？ | rewrite 按正则改写 URI，可多次，flag 有 last/break/redirect/permanent；return 直接返回状态码或跳转，终止后续处理。 |
| location 优先级？ | = 精确 > ^~ 前缀 > ~ 正则 > 无修饰前缀；同类型正则按顺序，前缀取最长。 |
| 403/502/504 怎么排查？ | 403 查权限与路径；502 查后端可达性与是否崩溃；504 查超时与后端慢；结合 error_log 与 curl。 |
| 项目里 Nginx 做什么？ | 反向代理 Spring Boot、静态托管前端、HTTPS、负载均衡、限流与安全头；与 Gateway 可组合，Nginx 前置做 TLS 与负载。 |

### 15.2 学习要点

- 能说清**反向代理、负载均衡、动静分离**的配置方式（upstream、proxy_pass、root/alias、location）。
- 掌握 **location 匹配顺序** 与 **proxy_pass 末尾 URI** 对路径的影响，避免配置「不生效」。
- 能快速排查 **403/502/504**：权限、路径、后端可达、超时。
- 能说明 Nginx 与 **Spring Cloud Gateway** 的分工：Nginx 做 TLS、静态、负载；Gateway 做路由、鉴权、限流。
