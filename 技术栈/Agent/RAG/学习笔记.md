## 第1章：RAG 概念与价值

### 1.1 什么是 RAG

- **RAG**（Retrieval-Augmented Generation）是一种结合**检索**与**生成**的技术，通过在生成前检索相关文档作为上下文，增强大模型的知识与准确性。
- 核心思想：利用外部知识库补充模型参数中未包含的信息，减少幻觉，提升输出质量。

### 1.2 RAG 的价值

- 解决知识截止问题：模型参数固定，无法获取最新信息，RAG 可实时检索最新文档。
- 减少幻觉：基于检索到的文档生成，输出更准确、可溯源。
- 降低微调成本：无需频繁微调模型，通过更新知识库即可扩展模型能力。
- 领域知识适配：快速适配特定领域，如法律、医疗、企业内部知识等。

### 1.3 典型应用场景

- 企业知识库问答、文档问答、客户服务、教育辅助、法律/医疗咨询、技术支持等。

## 第2章：RAG 核心流程与组件

### 2.1 离线处理（索引构建）

1. **文档获取**：从文件系统、数据库、API 等获取源文档。
2. **文档分块**：将长文档切分为短文本片段（chunk），平衡上下文完整性与检索精度。
   - 常见分块策略：固定长度、语义分块（基于段落/章节）、重叠分块。
3. **嵌入生成**：使用嵌入模型将文本块转换为向量。
   - 常见模型：OpenAI Embedding、Hugging Face 模型（如 BGE）、国产嵌入模型。
4. **向量存储**：将向量与原文存储到向量数据库，建立索引。

### 2.2 在线推理（检索与生成）

1. **用户查询**：接收用户问题。
2. **查询嵌入**：将查询转换为向量。
3. **相似度检索**：在向量数据库中检索与查询最相似的文本块。
4. **上下文构建**：将检索到的文本块组织为上下文，与用户查询一起发送给大模型。
5. **生成回答**：大模型基于上下文生成回答，可要求引用源文档。

### 2.3 关键组件

- **嵌入模型**：负责将文本转换为向量，影响检索质量。
- **向量数据库**：存储向量并提供高效检索，影响系统性能。
- **分块策略**：影响上下文相关性与完整性。
- **检索策略**：如 top-k、重排序，影响检索结果质量。
- **Prompt 设计**：如何组织上下文与查询，影响生成质量。

## 第3章：分块策略与优化

### 3.1 分块方法

1. **固定长度分块**：按字符数或 token 数切分，简单直接。
   - 优点：实现简单；缺点：可能切断语义单元。
2. **语义分块**：基于段落、章节、句子等语义边界切分。
   - 优点：保持语义完整性；缺点：实现复杂。
3. **混合分块**：结合固定长度与语义边界，如优先按段落切分，超过长度再细分。

### 3.2 分块参数优化

- **块大小**：根据模型上下文窗口、文档类型调整，一般 256-1024 token。
- **重叠比例**：相邻块之间的重叠部分，避免信息丢失，一般 10-20%。
- **边界检测**：如检测标题、列表、代码块等特殊结构。

### 3.3 实践建议

- 文档类型不同，分块策略不同：代码文档可按函数/类分块，论文可按章节分块。
- 结合元数据：如文档标题、章节信息，增强检索相关性。

## 第4章：嵌入模型选择与优化

### 4.1 常见嵌入模型

1. **OpenAI Embedding**：text-embedding-ada-002 等，效果好但有 API 成本。
2. **Hugging Face 开源模型**：BGE、E5、Sentence-BERT 等，可本地部署。
3. **国产模型**：智谱 Embedding、通义 Embedding 等，支持中文优化。

### 4.2 模型选择考量

- **语言支持**：是否支持中文等多语言。
- **维度大小**：维度越高表达能力越强，但存储与计算成本也越高。
- **推理速度**：在线场景需考虑响应时间。
- **成本**：API 调用成本 vs 本地部署成本。

### 4.3 优化技巧

- **文本预处理**：去除噪声、标准化格式。
- **批处理**：批量生成嵌入，提高效率。
- **缓存**：缓存已生成的嵌入，避免重复计算。

------

## 第5章：向量数据库选择与使用

### 5.1 常见向量数据库

1. **云服务**：Pinecone、Weaviate Cloud、Azure Cognitive Search 等。
2. **开源本地**：Milvus、Chroma、FAISS、Qdrant、Elasticsearch（with vector plugin）等。

### 5.2 选择考量

- **规模**：数据量大小，是否需要水平扩展。
- **性能**：检索速度、并发能力。
- **功能**：是否支持元数据过滤、地理搜索等高级功能。
- **部署方式**：云服务 vs 本地部署。
- **成本**：存储与计算成本。

### 5.3 最佳实践

- **索引优化**：选择合适的索引类型（如 HNSW、IVF）。
- **批处理写入**：批量插入向量，提高写入效率。
- **元数据使用**：利用元数据过滤，提升检索精度。
- **定期维护**：如重建索引、清理过期数据。

## 第6章：检索策略与重排序

### 6.1 基础检索策略

- **Top-k 检索**：返回相似度最高的 k 个结果。
- **阈值过滤**：只返回相似度高于阈值的结果。
- **元数据过滤**：结合元数据进行条件检索。

### 6.2 重排序（Re-ranking）

- **目的**：进一步优化检索结果，提升相关性。
- **方法**：
  1. **交叉编码器**：如 BERT 交叉编码器，计算查询与文档的相关性分数。
  2. **规则重排序**：基于文档长度、位置等规则调整排序。
  3. **混合重排序**：结合多种因素进行排序。

### 6.3 实践建议

- 先使用高效的向量检索获取候选结果，再使用重排序模型精细排序。
- 根据应用场景调整 k 值与重排序策略。

## 第7章：与 LLM 集成与 Prompt 设计

### 7.1 上下文构建

- **信息组织**：将检索到的文档按相关性排序，合理组织。
- **长度控制**：确保上下文长度不超过模型最大窗口。
- **格式清晰**：使用分隔符、标题等标记，便于模型理解。

### 7.2 Prompt 设计技巧

- **明确指令**：要求模型基于提供的上下文回答，不使用外部知识。
- **引用要求**：要求模型在回答中引用源文档，增强可溯源性。
- **格式约束**：指定输出格式，如结构化回答、要点列表等。
- **示例引导**：提供示例，引导模型正确处理上下文。

### 7.3 常见模板

```
你是一个专业的问答助手，负责基于提供的文档回答用户问题。

请严格按照以下要求回答：
1. 仅基于提供的文档内容回答，不要使用外部知识
2. 回答要准确、简洁、有条理
3. 对于需要引用的内容，请在回答末尾标注来源

文档：
{{documents}}

用户问题：
{{question}}

回答：
```

## 第8章：评估与优化

### 8.1 评估指标

- **检索评估**：召回率、精确率、F1 分数。
- **生成评估**：准确率、相关性、引用正确率、用户满意度。
- **系统评估**：响应时间、吞吐量、成本。

### 8.2 优化方向

- **分块优化**：调整分块大小与策略。
- **嵌入模型优化**：选择更适合的模型或微调。
- **检索优化**：调整 k 值、重排序策略。
- **Prompt 优化**：改进 Prompt 结构与指令。
- **知识库管理**：定期更新知识库，确保信息新鲜度。

### 8.3 A/B 测试

- 对比不同配置的性能，如不同分块策略、嵌入模型、检索参数等。
- 基于实际使用数据持续优化。

## 第9章：实战与最佳实践

### 9.1 从零构建 RAG 系统

1. **步骤**：
   - 确定需求与场景
   - 准备知识库文档
   - 选择分块策略与嵌入模型
   - 选择向量数据库并构建索引
   - 实现检索与生成流程
   - 评估与优化

2. **技术栈选择**：
   - 框架：LangChain、LlamaIndex 等
   - 嵌入模型：根据语言与性能需求选择
   - 向量数据库：根据规模与部署需求选择
   - 大模型：根据功能与成本选择

### 9.2 常见问题与解决方案

- **检索结果不相关**：优化分块策略、嵌入模型、检索参数。
- **上下文过长**：调整分块大小、使用摘要压缩、滑动窗口。
- **生成内容与文档不一致**：改进 Prompt，加强引用要求。
- **响应时间长**：优化检索速度、使用流式输出、缓存热门查询。

### 9.3 面试要点

- RAG 的核心原理与价值
- 分块策略的选择与优化
- 嵌入模型的选择考量
- 向量数据库的对比与选择
- 检索策略与重排序方法
- 如何评估与优化 RAG 系统
