# 📘《操作系统学习与实战手册》

> 系统学习见下文各章；日常常用概念与使用场景速查见同目录《常用概念与使用场景》。  
> 适用：通用 OS 概念（不限定某具体 OS），面向校招与后端开发理解进程、内存、I/O 等。

------

## 第1章：操作系统概述

本章建立对操作系统的整体认知：**是什么、解决什么问题、如何与硬件/应用分层**，为后续进程、内存、死锁、同步等章节打基础。学完可继续第 2 章进程与线程。

------

### 1️⃣ 什么是操作系统

- **操作系统（OS）**：管理硬件与软件资源的程序，为应用提供**进程管理、内存管理、文件系统、设备管理**等；**为什么需要**：屏蔽硬件细节、提供统一抽象、保证安全隔离，让应用不必直接操作硬件。
- **内核态与用户态**：  
  - **用户态**：普通程序运行，权限受限，不能直接访问硬件或敏感资源。  
  - **内核态**：操作系统内核运行，可执行 I/O、内存映射、中断等敏感操作。  
  - 用户程序通过 **系统调用（System Call）** 陷入内核，由内核代为完成操作后返回用户态。

| 概念 | 说明 |
|------|------|
| 系统调用 | 用户态请求内核服务的接口（如 read/write、fork、mmap）；是用户态与内核态的分界。 |
| 中断 | 硬件或软件事件触发 CPU 暂停当前执行、转去处理中断例程，是 OS 获得控制权的关键机制。 |

### 2️⃣ 多道、分时与并发

| 概念 | 一句话 | 为什么需要 |
|------|--------|------------|
| **多道** | 多道程序同时在内存中，交替占用 CPU | 提高 CPU 与 I/O 利用率，避免单道时 CPU 在等 I/O 时空闲 |
| **分时** | 时间片轮转，多用户/多任务共享一台机器 | 让每个用户感觉“独占”机器，交互式使用 |
| **并发** | 多个任务在时间上交替执行（单核也可并发） | 逻辑上同时推进多任务 |
| **并行** | 多核/多 CPU 同时执行多条指令 | 真正同时执行，提高吞吐 |

> 💡 **重点**：并发是逻辑上的“同时”，并行是物理上的同时；单核只能并发不能并行。

------

### 📖 本章小结

| 知识点 | 面试关键词 | 实际应用 |
|--------|------------|----------|
| OS 定位 | 管理软硬件、抽象与隔离、系统调用 | 理解 JVM/应用与 OS 的关系（见第 7 章） |
| 内核态/用户态 | 权限分级、系统调用陷入内核 | 解释为什么用户程序不能直接操作硬件 |
| 多道/分时/并发/并行 | 多道提高利用率、分时共享、并发 vs 并行 | 区分高并发场景下的“并发”与“并行” |

------

### ⚠️ 常见坑与注意点

1. **现象**：把“并发”和“并行”混用。**原因**：口语中常混用。**正确做法**：单核多线程是并发；多核多线程才可能并行；面试时先说“并发是逻辑同时、并行是物理同时”。
2. **现象**：认为用户程序可以直接访问任意内存或设备。**原因**：忽略了用户态权限限制。**正确做法**：敏感操作必须通过系统调用进入内核态执行。
3. **现象**：不理解为什么要有 OS 这一层。**原因**：未建立“抽象与隔离”的概念。**正确做法**：记住 OS 提供统一接口、隔离故障与安全边界，便于多任务与多用户。

## 第2章：进程与线程

本章解决**任务如何被 OS 组织与调度**：进程是资源分配单位、线程是 CPU 调度单位，二者关系与状态转换是后续调度（第 3 章）、同步与死锁（第 5、6 章）的基础。依赖第 1 章对 OS 与系统调用的理解。

------

### 1️⃣ 进程

- **进程**：**资源分配的基本单位**，有独立地址空间、文件描述符、**PCB（Process Control Block，进程控制块）** 等；进程间隔离，通信需 **IPC（Inter-Process Communication）**。
- **为什么需要进程**：多道程序需要隔离（一崩不拖垮其他）、需要独立资源视图（地址空间、打开的文件等），进程是 OS 对“正在运行的程序”的抽象。
- **常见 IPC**：管道（匿名/命名）、消息队列、共享内存、信号量、Socket；其中共享内存最快但需自己处理同步。

### 2️⃣ 线程

- **线程**：**CPU 调度的基本单位**，同一进程内共享地址空间与资源（堆、全局变量、文件句柄等），各自有独立的栈和寄存器；切换开销比进程小。
- **为什么需要线程**：在保留进程隔离的前提下，让同一程序内多路执行（如一个线程等 I/O、另一个算逻辑），提高响应与 CPU 利用率。

### 3️⃣ 进程与线程区别

| 维度 | 进程 | 线程 |
|------|------|------|
| 资源 | 独立地址空间、独立资源 | 共享进程地址空间与资源 |
| 调度 | 进程是资源分配单位，线程才是被调度的实体 | 线程是 CPU 调度单位 |
| 切换开销 | 大（切换页表、上下文多） | 小（共享地址空间，主要切换寄存器、栈） |
| 通信 | 需 IPC | 共享内存即可，但要同步（见第 6 章） |

### 4️⃣ 进程状态与转换

典型三态：**就绪（Ready）**、**运行（Running）**、**阻塞（Blocked/Waiting）**。

```
   [新建] → 就绪 ←→ 运行 → [终止]
                ↑      ↓
                └── 阻塞
```
- **就绪 → 运行**：被调度器选中，获得 CPU。  
- **运行 → 就绪**：时间片到或被更高优先级抢占。  
- **运行 → 阻塞**：等待 I/O、信号量等事件。  
- **阻塞 → 就绪**：事件发生，重新参与调度。

> 💬 **面试怎么说**：进程在就绪队列里等 CPU，在阻塞队列里等 I/O/事件；调度器只从就绪队列里选进程上 CPU。

### 5️⃣ 用户线程与内核线程、协程

| 类型 | 说明 | 优缺点 |
|------|------|--------|
| **内核线程** | 由内核调度，一对一或多对一映射到 CPU | 调度公平、可多核并行；切换要进内核，有开销 |
| **用户线程** | 在用户态由运行时调度，多对一映射到内核线程 | 切换快、轻量；一个阻塞可能阻塞整个进程 |
| **协程** | 用户态轻量“线程”，协作式调度，遇 I/O 让出 | 极轻量、高并发 I/O；不适合 CPU 密集 |

------

### 📖 本章小结

| 知识点 | 面试关键词 | 实际应用 |
|--------|------------|----------|
| 进程 | 资源分配单位、PCB、独立地址空间、IPC | 理解一个 Java 进程 = 一个 JVM（第 7 章） |
| 线程 | CPU 调度单位、共享进程资源、切换开销小 | 多线程编程、线程池（与第 6 章同步配合） |
| 进程状态 | 就绪、运行、阻塞及转换 | 分析卡死、负载高时进程在等什么 |
| 用户/内核线程、协程 | 1:1 映射、用户态调度、协程让出 | 解释 JVM 线程与 OS 线程、Go/协程场景 |

------

### ⚠️ 常见坑与注意点

1. **现象**：认为“进程比线程快”或“线程一定比进程省资源”。**原因**：混淆了“调度单位”和“资源单位”。**正确做法**：线程切换快、共享资源方便，但多线程要处理竞态与同步；进程隔离好、但创建与切换成本高。
2. **现象**：多线程写共享变量不加锁，结果不确定。**原因**：未理解“线程共享地址空间”带来的竞态。**正确做法**：共享可写数据必须用同步手段（见第 6 章）。
3. **现象**：把“协程”当万能高并发。**原因**：协程适合 I/O 密集、协作式让出；CPU 密集仍会占满一核。**正确做法**：I/O 密集用协程/多路复用，CPU 密集用多进程/多线程并行。

------

| 面试官提问 | 回答思路 |
|------------|----------|
| 进程和线程的区别？ | 进程是资源分配单位、有独立地址空间；线程是 CPU 调度单位、共享进程资源；线程切换开销小，多线程需同步。 |
| 进程有哪些状态？如何转换？ | 就绪、运行、阻塞；就绪↔运行由调度决定，运行→阻塞因等 I/O/事件，阻塞→就绪因事件发生。若追问调度细节见第 3 章。 |

## 第3章：进程调度

本章解决**就绪队列里的进程谁先占用 CPU、占多久**的问题，与第 2 章进程状态紧密相关；理解调度算法与上下文切换有助于分析性能与“卡顿”。后续第 6 章同步会涉及线程在用户态/内核态的等待与唤醒。

------

### 1️⃣ 调度目标

| 指标 | 说明 | 典型场景 |
|------|------|----------|
| 公平性 | 各进程/用户获得合理 CPU 时间 | 多用户分时 |
| 吞吐量 | 单位时间完成作业数 | 批处理 |
| 响应时间 | 从提交到首次响应的时间 | 交互式 |
| 周转时间 | 从提交到完成的时间 | 批处理 |

批处理更关注吞吐与周转；交互式更关注响应时间。

### 2️⃣ 常见调度算法

| 算法 | 思路 | 优点 | 缺点 |
|------|------|------|------|
| **FCFS**（先来先服务） | 按到达顺序依次执行 | 简单、公平 | 护航效应（短作业等长作业）、不利于交互 |
| **SJF**（短作业优先） | 优先调度预计运行时间短的 | 平均周转时间优 | 长作业可能饿死、需要预知时间 |
| **时间片轮转（RR）** | 就绪队列轮转，每进程执行一个时间片 | 响应快、公平 | 时间片过大退化为 FCFS，过小切换开销大 |
| **多级反馈队列** | 多队列、不同时间片与优先级，进程可在队列间升降 | 兼顾响应与吞吐、综合实用 | 实现复杂 |
| **优先级调度** | 按优先级选进程 | 可体现紧急程度 | 低优先级可能饿死，需配合老化等手段 |

### 3️⃣ 多级反馈队列（面试常问）

- **思路**：设多个就绪队列，优先级从高到低，时间片从短到长；新进程进最高优先级队列；若时间片内未完成则降级到下一队列；低优先级队列中的进程可能被高优先级抢占。
- **效果**：短作业较快完成、长作业不会长期霸占、响应时间较好，是很多实际 OS 的折中方案。

> 💬 **面试怎么说**：多级反馈队列通过“多队列 + 时间片递增 + 降级 + 抢占”兼顾短作业与长作业，既照顾响应又照顾吞吐。

### 4️⃣ 上下文切换

- **上下文**：进程/线程运行所需的寄存器、栈指针、PC、页表基址等状态。
- **上下文切换**：当前进程让出 CPU 时，把上述状态保存到其 PCB；从另一进程的 PCB 恢复状态并运行。**频繁切换**会带来保存/恢复开销和 Cache 失效，线程切换因共享地址空间通常比进程切换轻量。

------

### 📖 本章小结

| 知识点 | 面试关键词 | 实际应用 |
|--------|------------|----------|
| 调度目标 | 响应时间、周转时间、吞吐、公平 | 分析系统“卡顿”或批任务完成慢 |
| FCFS/SJF/RR/多级反馈 | 护航效应、饿死、时间片 | 能说出各算法特点与适用场景 |
| 多级反馈队列 | 多队列、时间片递增、降级、抢占 | 面试标准答法：兼顾短长作业 |
| 上下文切换 | 保存/恢复 PCB、切换开销、线程更轻 | 高并发时减少线程数有时反而更快 |

------

### ⚠️ 常见坑与注意点

1. **现象**：认为“调度算法越复杂越好”。**原因**：忽略实现成本与上下文切换开销。**正确做法**：实际 OS 多用多级反馈等折中方案，而不是“纯 SJF”或“纯 RR”。
2. **现象**：时间片设得极小以为响应更快。**原因**：时间片过小导致上下文切换占比过高。**正确做法**：时间片要在响应时间和切换开销之间折中。
3. **现象**：混淆“进程调度”和“线程调度”。**原因**：线程才是调度单位，但调度器可能是内核的同一套（按线程调度）。**正确做法**：内核调度的是可运行线程/轻量进程，进程是资源的容器。

------

| 面试官提问 | 回答思路 |
|------------|----------|
| 常见调度算法有哪些？ | FCFS、SJF、时间片轮转、多级反馈队列、优先级；说出各自特点与缺点（护航、饿死、切换开销）。 |
| 多级反馈队列怎么实现的？ | 多级队列、高优先级短时间片、未完成则降级、低优先级可被抢占；兼顾响应与吞吐。若追问具体几级、时间片取值，可说“因 OS 而异，思路如上”。 |
| 什么是上下文切换？ | 保存当前进程/线程的寄存器、栈、PCB 等状态，恢复另一进程/线程的状态；有 CPU 与 Cache 开销，线程切换比进程轻量。 |

## 第4章：内存管理

本章解决**进程如何安全、高效地使用内存**：从连续分配到分页/分段，再到虚拟内存与页面置换，是理解“进程地址空间”“缺页”“抖动”的基础，也与第 7 章 JVM 堆、栈、物理内存的关系直接相关。

------

### 1️⃣ 连续分配与碎片

- **连续分配**：进程占用一块连续物理内存；**固定分区**与**动态分区**都会产生**外部碎片**（碎片在进程之间，难以利用）， compaction 可缓解但代价高。
- **为什么引入分页/分段**：消除或减少外部碎片，并支持更大的逻辑地址空间（配合虚拟内存）。

### 2️⃣ 分页

- **分页**：逻辑地址空间按**页**划分，物理内存按**页帧（Frame）**划分，大小相同；通过**页表**做逻辑页号 → 物理页帧号的映射，从而**消除外部碎片**（任一页可放任一空闲帧）。
- **多级页表**：把线性页表拆成多级，只为用到的部分分配下级页表，**节省页表占用空间**（尤其 64 位大地址空间）。
- **缺页（Page Fault）**：访问的页不在内存中时触发；OS 从外存（如 swap）调入该页（可能要先做**页面置换**），再重新执行访存指令。

```
逻辑地址 → [页号 | 页内偏移] → 页表查帧号 → 物理地址 = [帧号 | 页内偏移]
```

### 3️⃣ 分段与段页式

- **分段**：按逻辑段（代码段、数据段、栈段等）划分，每段连续但段长可变；便于共享与保护，但会产生**内部碎片**（段内未用满）。
- **段页式**：先分段再在段内分页，兼顾段式逻辑与页式物理管理，很多实际 OS 采用。

| 维度 | 分页 | 分段 |
|------|------|------|
| 划分方式 | 固定大小、物理视角 | 逻辑段、长度可变 |
| 碎片 | 内部碎片（最后一页未满） | 外部碎片（段间） |
| 地址 | 一维（页号+偏移） | 二维（段号+段内偏移） |
| 典型用途 | 通用内存管理、虚拟内存 | 代码/数据/栈分离、共享 |

### 4️⃣ 虚拟内存与页面置换

- **虚拟内存**：进程使用**逻辑地址空间**，可大于物理内存；OS 按需把用到的页调入内存，暂时不用的页可换出到磁盘，从而**支持大地址空间与多道程序**。
- **页面置换**：缺页且物理帧已满时，必须选一页换出；选谁由**置换算法**决定。
- **常见置换算法**：
  - **FIFO**：先进先出，实现简单，可能 Belady 异常。
  - **LRU**（Least Recently Used）：最近最久未用，效果好但实现成本高（近似 LRU 如 Clock）。
  - **Clock**：用使用位与指针做近似 LRU，折中。
  - **LFU**：最不经常使用，适合有明显热点时。

> 💡 **重点**：**抖动（Thrashing）**—— 缺页率过高，进程大量时间花在换页上而非执行，通常因物理帧过少或局部性差；缓解方法是增加物理内存或减少并发进程数、改进局部性。

------

### ⚠️ 常见坑与注意点

1. **现象**：把“虚拟内存”等同于“磁盘上的 swap”。**原因**：虚拟内存是地址空间抽象，swap 是换页的 backing store。**正确做法**：虚拟内存 = 逻辑地址空间 + 按需映射 + 换页；swap 是缺页时换出页的存放处。
2. **现象**：认为 LRU 一定比 FIFO 好。**原因**：LRU 要维护访问顺序，实现贵；实际常用 Clock 等近似。**正确做法**：能说出 LRU 思想与近似实现（如 Clock），以及 FIFO 的 Belady 异常。
3. **现象**：物理内存不够就无脑加 swap。**原因**：swap 过多会导致抖动，整体更慢。**正确做法**：优先加物理内存或减负载；swap 作为缓冲可接受，但不能指望靠 swap 撑过严重缺内存。

------

### 🎯 面试常见追问

| 面试官提问 | 回答思路 |
|------------|----------|
| 分页和分段有什么区别？ | 分页固定大小、按帧映射、无外部碎片有内部碎片；分段按逻辑段、可变长、便于共享与保护；可结合成段页式。 |
| 虚拟内存的作用？ | 提供大逻辑地址空间、多进程隔离、按需调入节省物理内存；通过页表与缺页、置换实现。若追问与 JVM 关系见第 7 章。 |
| LRU 怎么实现？ | 理想实现要按访问时间排序，代价高；实际用近似：Clock 算法、或硬件引用位+软件近似。 |
| 什么是缺页、抖动？ | 缺页：访问的页不在内存触发；OS 调入（或先置换）后重试。抖动：缺页率过高，大量时间在换页，需加内存或减负载。 |

## 第5章：死锁

本章解决**多进程/多线程因争抢资源而互相等待、无法推进**的问题，与第 6 章同步互斥配合：同步手段用不当会带来死锁。校招常考“四条件 + 预防/避免/检测”。

------

### 1️⃣ 死锁的四个必要条件

四者**同时满足**才可能发生死锁（缺一则可避免死锁）：

| 条件 | 含义 |
|------|------|
| **互斥** | 资源只能被一个进程/线程占用 |
| **占有且等待** | 已持有至少一个资源，并在等待其他资源 |
| **不可剥夺** | 资源不能被强行抢占，只能由持有者主动释放 |
| **循环等待** | 存在进程/线程的循环链：A 等 B、B 等 C、…、最后等 A |

> 💬 **面试怎么说**：死锁是四个条件同时成立时的现象；预防/避免的思路就是破坏其中至少一个。

### 2️⃣ 预防（破坏任一条件）

| 破坏对象 | 做法 |
|----------|------|
| 互斥 | 无法破坏（某些资源本质就是互斥的，如打印机） |
| 占有且等待 | **一次性申请全部所需资源**，不够则等待；避免“拿着 A 等 B” |
| 不可剥夺 | 允许抢占：若申请不到则释放已占资源，稍后重试（实现复杂） |
| 循环等待 | **资源有序分配**：给资源编号，进程只能按编号递增顺序申请；不会出现 A 持高号等低号、B 持低号等高号 |

### 3️⃣ 避免：银行家算法

- **思想**：分配前**模拟一次分配**，判断系统是否仍处于**安全状态**（存在一个安全序列，使所有进程能依次得到资源并完成）；若分配后不存在安全序列则**不分配**，让申请者等待。
- **安全状态**：存在一个进程执行顺序，使得按该顺序每个进程都能在“当前可用 + 前面进程释放”的资源下完成。

### 4️⃣ 检测与恢复

允许死锁发生，**定期检测**（如用资源分配图判断是否存在环路），发现后**恢复**：剥夺某进程资源、回滚、或终止进程重启。

------

### 📖 本章小结

| 知识点 | 面试关键词 | 实际应用 |
|--------|------------|----------|
| 四条件 | 互斥、占有且等待、不可剥夺、循环等待 | 分析“卡死”是否死锁、如何破 |
| 预防 | 一次性申请、资源有序分配 | 设计协议/锁顺序时避免循环等待 |
| 避免 | 银行家算法、安全状态 | 理解“分配前检测”的思路 |
| 检测与恢复 | 资源分配图、剥夺/回滚/重启 | 运维中定位死锁并恢复 |

------

### ⚠️ 常见坑与注意点

1. **现象**：只记“循环等待”一个条件。**原因**：四条件缺一不可。**正确做法**：面试时先说“四条件同时满足才死锁”，再列四个，预防时说明破坏哪一个。
2. **现象**：认为银行家算法能完全避免死锁。**原因**：银行家算法依赖已知的“最大需求”，且要实时计算，实际多用于理论或特殊系统。**正确做法**：能说出“分配前检测安全状态”的思想即可，实际工程多用预防（锁顺序）或检测恢复。
3. **现象**：多锁不加顺序，随意加锁。**原因**：容易形成循环等待。**正确做法**：多锁约定全局顺序（如按锁对象 id 排序），统一按顺序申请，破坏循环等待。

------

### 🎯 面试常见追问

| 面试官提问 | 回答思路 |
|------------|----------|
| 死锁的四个必要条件？ | 互斥、占有且等待、不可剥夺、循环等待；四者同时满足才可能死锁。 |
| 如何预防死锁？ | 破坏其一：一次性申请全部资源（破坏占有且等待）、资源有序分配（破坏循环等待）等。 |
| 银行家算法是什么？ | 分配前检查是否仍存在安全序列；若分配后不安全则不分配，避免进入死锁状态。若追问安全状态定义，可说“存在一个进程顺序都能依次获得资源并完成”。 |

## 第6章：同步与互斥

本章解决**多线程/多进程访问共享资源时的互斥与顺序协调**问题，是第 2 章线程、第 5 章死锁的延续：用锁与信号量等实现互斥与同步，用错会竞态或死锁。与 Java 的 `synchronized`、`ReentrantLock`、`wait/notify` 对应。

------

### 1️⃣ 临界区与互斥

- **临界区（Critical Section）**：访问**共享资源**的那段代码；多个执行流不能同时进入同一临界区，否则产生**竞态**、结果不可预期。
- **互斥**：保证同一时刻只有一个执行流在临界区内；常用**互斥量（Mutex）**或**信号量**实现。

### 2️⃣ 信号量与互斥量

- **信号量（Semaphore）**：一个整数 + **P（wait）** 与 **V（signal）** 操作。P：若值 > 0 则减 1 并继续，否则阻塞；V：加 1 并唤醒一个等待者。**二元信号量**（0/1）即可实现互斥；**一般信号量**可表示资源个数（如多槽缓冲区）。
- **互斥量（Mutex）**：可理解为**只用于互斥的锁**，同一时刻只允许一个持有；加锁/解锁成对，通常由同一线程释放。
- **区别**：信号量可计多资源、可被不同线程 V；互斥量一般“谁加锁谁解锁”，语义更贴近“独占临界区”。

### 3️⃣ 条件变量与管程

- **条件变量**：在“满足某条件才继续”时使用；线程在条件变量上 **wait**（释放锁并睡眠），其他线程在条件成立时 **signal** 唤醒。常与互斥量配合：先加锁，再在条件变量上 wait/signal。
- **管程（Monitor）**：把互斥与条件变量封装成一种结构，任一时刻只有一个线程在管程内执行；Java 的 `synchronized` + `wait/notify` 即管程风格。

### 4️⃣ 经典模型（流程要点）

- **生产者-消费者**：有界缓冲区；缓冲区满时生产者阻塞，空时消费者阻塞；用互斥量保护缓冲区，用两个条件变量（或两个信号量）表示“非满”“非空”。
- **读者-写者**：多读者可同时读，写者独占；可用读写锁或“读者计数 + 互斥量 + 条件变量”实现。

### 5️⃣ 锁的类型

| 类型 | 行为 | 适用 |
|------|------|------|
| **自旋锁** | 等不到就忙等（循环检查） | 临界区极短、不想进内核睡眠 |
| **睡眠锁** | 等不到就睡眠，被唤醒再试 | 临界区可能较长，避免占 CPU |
| **读写锁** | 多读可并发、写独占 | 读多写少场景 |

------

### 📖 本章小结

| 知识点 | 面试关键词 | 实际应用 |
|--------|------------|----------|
| 临界区与互斥 | 临界区、竞态、互斥量 | 多线程共享变量必须加锁 |
| 信号量 vs 互斥量 | P/V、二元信号量、谁加锁谁解锁 | 对应 Java 的 Semaphore、synchronized/Lock |
| 条件变量与管程 | wait/signal、管程 | wait/notify、ReentrantLock+Condition |
| 生产者-消费者 | 有界缓冲、满/空阻塞 | 消息队列、线程池任务队列 |
| 自旋锁 vs 睡眠锁 | 忙等 vs 睡眠 | 短临界区用自旋，长用睡眠 |

------

### ⚠️ 常见坑与注意点

1. **现象**：忘记在 wait 前释放锁或 signal 后不释放锁。**原因**：条件变量必须与互斥量配合，wait 时原子地“释放锁并睡眠”。**正确做法**：使用条件变量时严格按“加锁 → 判断条件 → wait/signal → 解锁”的规范，避免丢失唤醒或死锁。
2. **现象**：用信号量实现互斥时，P 和 V 不成对或不在同一层。**原因**：V 多了会导致多进临界区，V 少了会永久阻塞。**正确做法**：成对使用，且保证每条执行路径的 P/V 数量一致。
3. **现象**：长临界区用自旋锁，CPU 空转高。**原因**：自旋锁适合“很快能拿到锁”的场景。**正确做法**：预计等待时间较长时用睡眠锁（互斥量/信号量），让出 CPU。

------

### 🎯 面试常见追问

| 面试官提问 | 回答思路 |
|------------|----------|
| 信号量和互斥量的区别？ | 信号量可表示多资源、P/V 可被不同线程调用；互斥量一般独占、谁加锁谁解锁。二元信号量可当互斥用。 |
| 生产者-消费者怎么实现？ | 有界队列 + 互斥保护 + 两个条件变量（或信号量）：非满时生产者可放，非空时消费者可取；满则生产者 wait，空则消费者 wait。若追问死锁见第 5 章。 |
| 如何保证多线程安全？ | 识别共享可变数据与临界区；用互斥量或信号量保证互斥；需要“等条件”时用条件变量；注意锁顺序防死锁。 |

## 第7章：与 Java/后端的关系

本章把前几章的 OS 概念**落到 Java/后端**：进程与 JVM、线程与 OS 线程、内存与 GC、I/O 与多路复用，便于面试时把“操作系统”和“Java 后端”串起来。依赖第 1～6 章的概念。

------

### 1️⃣ JVM 与进程、线程

- **一个 Java 进程 = 一个 JVM 进程**：`java Main` 启动一个 OS 进程，该进程内运行 JVM，JVM 再跑字节码。
- **Java 线程与 OS 线程**：HotSpot 等主流 JVM 采用 **1:1 模型**，一个 Java 线程对应一个 OS（内核）线程；早期“绿色线程”是用户态线程，现已少见。
- **协程/虚拟线程**：Java 19+ 的 **Virtual Threads** 是用户态轻量调度，多对多映射到少量 OS 线程，适合 I/O 密集高并发（对应第 2 章协程概念）。

### 2️⃣ 内存：JVM 与 OS 虚拟内存

- JVM 的**堆、栈、方法区**等都在进程的**虚拟地址空间**里；OS 通过分页、缺页、swap 管理物理内存，JVM 不直接管物理页，但 **GC 会触发大量内存访问与可能的换页**。
- 堆过大或物理内存不足时，可能 swap 增多、GC 变慢甚至 OOM；调优时常看进程 RSS、swap 使用（与第 4 章虚拟内存、抖动关联）。

### 3️⃣ I/O：阻塞/非阻塞、同步/异步、多路复用

| 维度 | 含义 | 常见实现 |
|------|------|----------|
| **阻塞/非阻塞** | 调用 I/O 时是否立刻返回（非阻塞：无数据也返回） | 阻塞 read / NIO Channel 非阻塞模式 |
| **同步/异步** | 数据就绪后谁负责处理（异步：内核/框架回调） | 同步：应用自己轮询或阻塞等；异步：AIO、回调 |
| **多路复用** | 一个线程等多个 fd，谁就绪就处理谁 | select / poll / **epoll**（Linux） |

- **select/poll**：一次调用传入 fd 集合，内核返回就绪集合；**缺点**： fd 数量多时复制与遍历开销大、fd 上限等。
- **epoll**：内核维护就绪列表，**边缘触发/水平触发**、无 fd 数量硬限、适合高并发连接；**面试常考**：epoll 与 select 区别（数据结构、就绪通知方式、扩展性）。

> 💬 **面试怎么说**：epoll 用事件驱动、红黑树管理 fd、就绪链表，避免 select/poll 的轮询与 fd 拷贝，适合 C10K、高并发 I/O；Netty 在 Linux 上默认用 epoll。

### 4️⃣ Netty 与 OS I/O

- **Netty**：对 NIO、多路复用的封装，提供 EventLoop、Channel、Pipeline 等抽象；在 Linux 上使用 **epoll**，在 Windows 上使用 **IOCP** 等，屏蔽 OS 差异。
- 理解“Reactor 模型、单线程/多线程 EventLoop”时，可对应到 OS 的“一个线程 + 多路复用”或“多线程各自多路复用”。

------

### 📖 本章小结

| 知识点 | 面试关键词 | 实际应用 |
|--------|------------|----------|
| JVM 与进程/线程 | 一进程一 JVM、1:1 线程模型、虚拟线程 | 理解线程池、线程数与 OS 线程、协程 |
| JVM 内存与 OS | 堆/栈在虚拟地址空间、GC 与 swap | 调优堆大小、看 RSS/swap、避免抖动 |
| I/O 与多路复用 | 阻塞/非阻塞、同步/异步、select/epoll | NIO、Netty、高并发连接 |
| epoll vs select | 事件驱动、无 fd 数限、就绪链表 | 面试标准答法：为什么用 epoll |

------

### ⚠️ 常见坑与注意点

1. **现象**：认为“线程开得越多并发越高”。**原因**：OS 线程有栈与切换成本，过多反而上下文切换和内存占用大。**正确做法**：线程池大小与 CPU 密集/I/O 密集有关；I/O 密集可用 NIO/多路复用或虚拟线程减少线程数。
2. **现象**：把 NIO 说成“异步”。**原因**：NIO 常指非阻塞 + 多路复用的同步 I/O，数据就绪后仍要应用自己读。**正确做法**：区分“非阻塞”与“异步”；异步是“数据就绪后由内核/框架回调”，如 AIO、Netty 的 Future 回调。
3. **现象**：只记“epoll 比 select 快”不说原因。**原因**：面试官常追问“为什么”。**正确做法**：能说出 select/poll 的 fd 拷贝与轮询、epoll 的注册一次与就绪事件通知、可扩展性。

------

### 🎯 面试常见追问

| 面试官提问 | 回答思路 |
|------------|----------|
| 一个 Java 进程里有几个 JVM？线程和 OS 线程关系？ | 一个进程一个 JVM；HotSpot 下 Java 线程与 OS 线程 1:1。若追问绿色线程/协程，见第 2 章用户线程与协程。 |
| 用户态和内核态在 I/O 里怎么体现？ | 应用 read 时发生系统调用进入内核；内核负责把数据从设备拷到内核缓冲再拷回用户空间；多路复用（epoll）也是通过系统调用让内核通知就绪事件。 |
| epoll 和 select 的区别？ | select/poll 每次传入 fd 集合、内核轮询、有 fd 上限与拷贝开销；epoll 注册一次、事件驱动、就绪链表、无硬限，适合高并发。若追问 Reactor/Netty 见本系列 Netty/网络笔记。 |

## 第8章：I/O 管理

本章解决**设备与磁盘 I/O 的高效管理**：从磁盘调度算法到 DMA、缓冲区管理，是理解数据库索引、文件系统性能的基础，也与第 7 章的 NIO/多路复用形成互补（用户态 I/O 与内核态 I/O 的衔接）。

------

### 1️⃣ I/O 控制方式演进

| 方式 | 机制 | 特点 | 缺点 |
|------|------|------|------|
| **程序直接控制** | CPU 轮询设备状态寄存器 | 简单 | CPU 忙等，效率极低 |
| **中断驱动** | 设备就绪后发中断，CPU 响应处理 | 无需轮询 | 每次中断仍需 CPU 参与数据搬运 |
| **DMA（直接内存访问）** | DMA 控制器直接在内存与设备间传数据，完成后中断 CPU | CPU 只需发起和响应，不介入传输 | 需要 DMA 硬件支持 |
| **通道 I/O** | 专用 I/O 处理器（通道）执行通道程序，管理多设备 | CPU 干预最少，适合大型机 | 硬件复杂 |

> 💡 **重点**：现代系统普遍用 **DMA**；磁盘、网卡等高速设备都支持 DMA，大幅减少 CPU 负担。

### 2️⃣ 磁盘调度算法

磁盘是慢速设备，寻道时间占大头；调度算法决定 I/O 请求的处理顺序，目标是**减少寻道时间**。

| 算法 | 思路 | 优点 | 缺点 |
|------|------|------|------|
| **FCFS** | 先来先服务 | 公平、简单 | 可能频繁来回移动，寻道时间长 |
| **SSTF（最短寻道时间优先）** | 优先处理离当前磁头最近的请求 | 平均寻道时间短 | 可能饿死远处请求 |
| **SCAN（电梯算法）** | 磁头单向移动到底，再反向 | 无饿死，较公平 | 两端请求等待时间不均 |
| **C-SCAN（循环扫描）** | 单向移动到底后快速回到起点，再单向 | 等待时间更均匀 | 回程不处理请求 |
| **LOOK / C-LOOK** | SCAN/C-SCAN 的优化，不到底就回头 | 减少无效移动 | 实现稍复杂 |

> 💬 **面试怎么说**：SSTF 可能饿死；SCAN 像电梯来回；C-SCAN 让等待时间更均匀；实际系统常用 LOOK/C-LOOK 或结合 SSD 特性优化。

### 3️⃣ 缓冲区管理

- **为什么需要缓冲**：
  1. 缓解 **CPU 与 I/O 速度不匹配**
  2. 减少 **中断次数**（缓冲满/空才中断）
  3. 解决 **数据块大小不匹配**（如字符设备 vs 块设备）

- **单缓冲、双缓冲、循环缓冲**：
  - **单缓冲**：一个缓冲区，读写串行
  - **双缓冲**：两个缓冲区交替，读写可并行（生产者-消费者模式）
  - **循环缓冲**：多个缓冲区组成环形队列，适合高速 I/O

- **缓冲池**：系统统一管理多个缓冲区，按需分配，提高利用率。

### 4️⃣ 磁盘结构与性能

- **磁盘结构**： platter（盘片）→ track（磁道）→ sector（扇区，通常 512B 或 4KB）
- **访问时间 = 寻道时间 + 旋转延迟 + 传输时间**
  - 寻道时间：磁头移动到目标磁道（通常 3-10ms）
  - 旋转延迟：盘片旋转到目标扇区（平均半圈，2-4ms）
  - 传输时间：实际读写数据（微秒级）

> 💡 **SSD 差异**：SSD 无机械结构，无寻道时间和旋转延迟，随机读写性能接近顺序读写；但仍有写入放大、磨损均衡等问题。

| 知识点 | 面试关键词 | 实际应用 |
|--------|------------|----------|
| I/O 控制方式 | 轮询、中断、DMA、通道 | 理解 DMA 减少 CPU 负担 |
| 磁盘调度 | FCFS、SSTF、SCAN、C-SCAN | 数据库、文件系统性能优化 |
| 缓冲区 | 单/双缓冲、缓冲池 | 流式处理、生产者-消费者 |
| 磁盘性能 | 寻道、旋转、传输时间 | SSD vs HDD 选型、性能估算 |

------

### ⚠️ 常见坑与注意点

1. **现象**：认为 DMA 完全不需要 CPU 参与。**原因**：DMA 只是数据传输不经过 CPU，但发起传输、处理完成中断仍需 CPU。**正确做法**：理解 DMA 是"减少"而非"消除" CPU 介入。
2. **现象**：磁盘调度算法在 SSD 上盲目套用。**原因**：SSD 无机械延迟，随机读写快，调度算法意义不大。**正确做法**：SSD 关注磨损均衡、TRIM、写入放大；HDD 才需关注寻道优化。
3. **现象**：忽略旋转延迟。**原因**：只考虑寻道时间。**正确做法**：磁盘访问时间 = 寻道 + 旋转延迟 + 传输；旋转延迟平均是半圈时间。

| 面试官提问 | 回答思路 |
|------------|----------|
| DMA 是什么？为什么需要？ | 直接内存访问，DMA 控制器直接在设备与内存间传数据，减少 CPU 干预；适合高速设备批量传输。 |
| 磁盘调度算法有哪些？ | FCFS、SSTF、SCAN（电梯）、C-SCAN；SSTF 可能饿死，SCAN 公平，C-SCAN 等待均匀。 |
| 为什么 SSD 不需要磁盘调度？ | SSD 无机械结构，随机访问与顺序访问性能接近，无寻道时间，调度算法收益不大。 |

## 第9章：文件系统

本章解决**文件如何被组织、存储、访问**：从文件的逻辑结构到物理结构，inode、目录、链接，以及日志文件系统，是理解"ls 为什么快""rm 后能否恢复"等问题的基础。与第 8 章磁盘 I/O、第 4 章内存管理紧密相关。

------

### 1️⃣ 文件与文件系统概述

- **文件**：具有符号名的数据集合，是 OS 管理存储的基本单位。
- **文件系统**：负责文件的存储、检索、共享、保护，提供按名存取的抽象。
- **文件逻辑结构**：
  - **无结构（流式）**：字节序列，如文本文件
  - **有结构**：记录序列，如数据库表

### 2️⃣ 文件物理结构（分配方式）

| 方式 | 机制 | 优点 | 缺点 |
|------|------|------|------|
| **连续分配** | 文件占连续磁盘块 | 顺序访问快，定位简单 | 产生外部碎片，不便扩展 |
| **链接分配** | 块间用指针链接（隐式/显式 FAT） | 无外部碎片，扩展方便 | 随机访问慢，指针占空间 |
| **索引分配** | 每个文件有一个索引块，记录所有块号 | 支持直接访问，扩展灵活 | 索引块 overhead，小文件浪费 |

> 💡 **混合方案**：Unix/Linux 采用 **多级索引**（直接索引 + 间接索引），小文件直接存块号，大文件用间接块，兼顾效率与灵活性。

### 3️⃣ inode（索引节点）

- **inode**：文件的元数据结构，存储文件的属性（大小、权限、时间、块指针等），**不含文件名**。
- **文件名 → inode 号 → 文件数据**：目录是"文件名-inode号"的映射表。
- **inode 内容**：
  - 直接指针（通常 12 个）：指向数据块
  - 一级间接指针：指向一个块，该块存储数据块地址
  - 二级、三级间接指针：更大文件

```
文件名 → 目录项 → inode 号 → inode 表 → 块指针 → 数据块
```

> 💬 **面试怎么说**：inode 是文件的"身份证"，存元数据和块位置；文件名只是目录里的一个映射；ls -i 可看 inode 号。

### 4️⃣ 目录结构

- **目录本质**：特殊文件，内容是"文件名 → inode 号"的映射表。
- **目录结构**：
  - **单级目录**：所有文件放一起，易冲突
  - **两级目录**：用户目录隔离
  - **树形目录**：层级结构，有绝对/相对路径
  - **无环图目录**：支持共享（硬链接），需避免环

### 5️⃣ 硬链接与软链接（符号链接）

| 特性 | 硬链接 | 软链接（符号链接） |
|------|--------|-------------------|
| **本质** | 多个目录项指向同一 inode | 特殊文件，存目标路径 |
| **inode** | 相同 | 不同 |
| **跨文件系统** | 不可 | 可以 |
| **原文件删除** | 仍可访问（引用计数减1） | 失效（ dangling link ） |
| **类型** | 普通文件 | 链接文件（l） |

> 💡 **ln 命令**：`ln file hardlink`（硬链接），`ln -s file symlink`（软链接）

### 6️⃣ 文件存储空间管理

- **空闲块管理**：
  - **空闲表**：记录连续空闲块，适合连续分配
  - **空闲链表**：链式管理，简单但遍历慢
  - **位示图（Bitmap）**：每位表示一块是否空闲，省空间、易查找
  - **成组链接**：Unix 采用，结合链表与空闲表优点

### 7️⃣ 日志文件系统（Journaling）

- **问题**：文件系统操作常涉及多个元数据更新（如分配块、改 inode、改目录），崩溃时可能处于不一致状态。
- **日志（Journal）**：先写日志（意图），再执行实际修改；崩溃后通过日志恢复。
- **日志模式**：
  - **Writeback**：只日志元数据，性能最好，数据可能丢
  - **Ordered**：元数据日志 + 数据先写，折中
  - **Data**：元数据和数据都日志，最安全，性能最差

> 💡 **ext3/ext4**：Linux 主流日志文件系统；数据库也有类似 WAL（Write-Ahead Log）机制。

------

### 📖 本章小结

| 知识点 | 面试关键词 | 实际应用 |
|--------|------------|----------|
| 文件分配 | 连续、链接、索引、多级索引 | 理解文件大小限制、访问效率 |
| inode | 元数据、块指针、间接索引 | ls -i、df -i、理解硬链接 |
| 硬/软链接 | 同 inode vs 存路径、跨 FS | 文件共享、快捷方式 |
| 日志文件系统 | Journal、WAL、崩溃恢复 | ext4、数据库事务日志 |

------

### ⚠️ 常见坑与注意点

1. **现象**：认为删除文件就释放了空间。**原因**：硬链接存在时，inode 引用计数不为 0，空间不释放。**正确做法**：理解 rm 是减少链接计数，计数为 0 才真正释放。
2. **现象**：硬链接跨文件系统。**原因**：不同 FS 的 inode 空间独立。**正确做法**：跨 FS 只能用软链接。
3. **现象**：忽略目录也是文件。**原因**：目录存的是文件名-inode映射。**正确做法**：目录有大小（存映射表）、有 inode，是特殊类型的文件。

| 面试官提问 | 回答思路 |
|------------|----------|
| 什么是 inode？和文件名什么关系？ | inode 存文件元数据和块位置；文件名是目录里的映射；一个 inode 可被多个文件名指向（硬链接）。 |
| 硬链接和软链接区别？ | 硬链接：同 inode，不能跨 FS，原文件删了还能访问；软链接：存路径，不同 inode，可跨 FS，原文件删了失效。 |
| 文件系统如何管理空闲空间？ | 位示图、空闲表、空闲链表、成组链接；现代常用位示图或成组链接。 |
| 日志文件系统是什么？ | 先写日志再改数据，崩溃后通过日志恢复一致性；有 writeback/ordered/data 三种模式。 |

## 第11章：epoll 原理详解

本章深入**Linux 高并发 I/O 的核心机制 epoll**，对比 select/poll，理解为什么 epoll 能支撑 C10K/C100K。与第 7 章 I/O 多路复用、后端开发中的 Netty/NIO 直接相关。

------

### 1️⃣ 为什么需要 I/O 多路复用

- **问题**：一个线程处理一个连接，10K 连接需要 10K 线程，上下文切换开销巨大。
- **解决**：一个线程**同时监视多个文件描述符（fd）**，哪个就绪就处理哪个 → **I/O 多路复用**。

### 2️⃣ select 与 poll 的局限

| 特性 | select | poll |
|------|--------|------|
| **fd 上限** | 1024（FD_SETSIZE） | 无硬限（受内存限制） |
| **数据结构** | fd_set（位图） | pollfd 数组 |
| **实现机制** | 每次传入 fd 集合，内核轮询 | 类似，传入 pollfd 数组 |
| **时间复杂度** | O(n)，n 为最大 fd 号 | O(n)，n 为监控的 fd 数 |
| **缺点** | fd 上限、重复拷贝、轮询 | 重复拷贝、轮询 |

**共同问题**：
1. **每次调用都拷贝 fd 集合到内核**：用户态 ↔ 内核态拷贝开销
2. **内核轮询所有 fd 检查就绪**：O(n) 复杂度，fd 多性能下降
3. **返回后需遍历所有 fd 找就绪的**：用户态再轮询

### 3️⃣ epoll 的设计改进

epoll 通过**三个系统调用**和**内核数据结构**解决上述问题：

| 函数 | 作用 |
|------|------|
| `epoll_create` | 创建 epoll 实例，返回 epoll fd |
| `epoll_ctl` | 增删改监控的 fd（ADD/MOD/DEL） |
| `epoll_wait` | 等待事件，返回就绪的 fd 列表 |

**核心数据结构**：
- **红黑树（RB-Tree）**：存储所有监控的 fd，支持 O(log n) 增删
- **就绪链表（Ready List）**：就绪的 fd 被内核回调加入此链表
- **回调机制**：fd 就绪时，内核自动将其加入就绪链表（避免轮询）

### 4️⃣ epoll 的工作流程

```
1. epoll_create → 创建 epoll 实例（红黑树 + 就绪链表）
2. epoll_ctl(ADD) → 将 fd 加入红黑树，注册回调函数
3. 数据到达 → 网卡中断 → 内核处理 → 回调将 fd 加入就绪链表
4. epoll_wait → 返回就绪链表中的 fd（只返回就绪的）
5. 用户处理 → 可继续 epoll_wait 或 epoll_ctl 修改监控
```

**时间复杂度**：
- epoll_ctl：O(log n)，红黑树操作
- epoll_wait：O(就绪 fd 数)，只返回就绪的

### 5️⃣ 水平触发（LT）与边缘触发（ET）

| 模式 | 触发条件 | 特点 |
|------|----------|------|
| **LT（Level Trigger）** | fd 可读/可写时持续触发 | 默认模式；若一次没读完，下次继续触发；编程简单 |
| **ET（Edge Trigger）** | fd 状态变化时触发一次 | 需一次性读写完（非阻塞）；效率高，但编程复杂 |

> 💡 **ET 注意事项**：
> - 必须设为非阻塞 I/O
> - 必须循环读到 EAGAIN（读光）或写满
> - 减少 epoll_wait 返回次数，更高效

### 6️⃣ epoll vs select/poll 对比总结

| 维度 | select/poll | epoll |
|------|-------------|-------|
| **fd 数量** | select 1024 限制，poll 无硬限 | 无硬限，受内存限制 |
| **拷贝开销** | 每次调用都拷贝 fd 集合 | epoll_ctl 时注册一次 |
| **查找就绪** | 内核轮询 O(n) | 回调机制，O(1) 加入就绪链表 |
| **返回数据** | 所有 fd，需用户遍历找就绪 | 只返回就绪的 fd |
| **时间复杂度** | O(n) | O(就绪数) |
| **适用场景** | fd 少、跨平台 | fd 多、Linux 高并发 |

### 7️⃣ 与后端开发的联系

- **Netty**：在 Linux 上默认使用 epoll（EpollEventLoopGroup），比 NIO 的 poll 实现性能更好
- **Nginx**：单 master + 多 worker，每个 worker 用 epoll 处理数万连接
- **Redis**：单线程 + epoll，支撑高并发 I/O

> 💬 **面试怎么说**：epoll 用红黑树管理 fd、就绪链表存就绪事件、回调机制避免轮询；LT 编程简单，ET 更高效；支撑 C10K，是 Linux 高并发 I/O 的基础。

### ⚠️ 常见坑与注意点

1. **现象**：ET 模式下数据没读完，下次不再触发。**原因**：ET 只在状态变化时触发一次。**正确做法**：ET 必须设非阻塞，循环读到 EAGAIN。
2. **现象**：认为 epoll 完全无拷贝。**原因**：epoll_wait 返回时仍需拷贝就绪事件到用户态。**正确做法**：epoll 避免的是"每次调用都拷贝全部 fd 集合"，而非零拷贝。
3. **现象**：fd 关闭后未 epoll_ctl(DEL)。**原因**：关闭的 fd 可能被复用，导致 epoll 报告错误事件。**正确做法**：关闭 fd 前先 epoll_ctl(DEL)，或处理时检查 fd 有效性。

### 🎯 面试常见追问

| 面试官提问 | 回答思路 |
|------------|----------|
| epoll 和 select 的区别？ | select 有 fd 上限、每次拷贝、轮询 O(n)；epoll 无上限、注册一次、回调机制、只返回就绪的。 |
| epoll 的数据结构？ | 红黑树存监控的 fd，就绪链表存就绪事件，回调函数在 fd 就绪时自动加入链表。 |
| LT 和 ET 区别？ | LT 默认，持续触发直到处理完；ET 只触发一次，需一次性处理完，必须非阻塞。 |
| 为什么 epoll 能支撑 C10K？ | 无 fd 数量硬限、避免轮询、只返回就绪 fd、内核与用户态拷贝少。 |
