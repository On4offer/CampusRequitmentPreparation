> 系统学习见下文各章；日常常用概念与使用场景速查见同目录《常用概念与使用场景》。  
> 适用：通用 OS 概念（不限定某具体 OS），面向校招与后端开发理解进程、内存、I/O 等。

## 第1章：操作系统概述

本章建立对操作系统的整体认知：**是什么、解决什么问题、如何与硬件/应用分层**，为后续进程、内存、死锁、同步等章节打基础。学完可继续第 2 章进程与线程。

------

### 1️⃣ 什么是操作系统

- **操作系统（OS）**：管理硬件与软件资源的程序，为应用提供**进程管理、内存管理、文件系统、设备管理**等；**为什么需要**：屏蔽硬件细节、提供统一抽象、保证安全隔离，让应用不必直接操作硬件。
- **内核态与用户态**：  
  - **用户态**：普通程序运行，权限受限，不能直接访问硬件或敏感资源。  
  - **内核态**：操作系统内核运行，可执行 I/O、内存映射、中断等敏感操作。  
  - 用户程序通过 **系统调用（System Call）** 陷入内核，由内核代为完成操作后返回用户态。

| 概念 | 说明 |
|------|------|
| 系统调用 | 用户态请求内核服务的接口（如 read/write、fork、mmap）；是用户态与内核态的分界。 |
| 中断 | 硬件或软件事件触发 CPU 暂停当前执行、转去处理中断例程，是 OS 获得控制权的关键机制。 |

### 2️⃣ 多道、分时与并发

| 概念 | 一句话 | 为什么需要 |
|------|--------|------------|
| **多道** | 多道程序同时在内存中，交替占用 CPU | 提高 CPU 与 I/O 利用率，避免单道时 CPU 在等 I/O 时空闲 |
| **分时** | 时间片轮转，多用户/多任务共享一台机器 | 让每个用户感觉“独占”机器，交互式使用 |
| **并发** | 多个任务在时间上交替执行（单核也可并发） | 逻辑上同时推进多任务 |
| **并行** | 多核/多 CPU 同时执行多条指令 | 真正同时执行，提高吞吐 |

> 💡 **重点**：并发是逻辑上的“同时”，并行是物理上的同时；单核只能并发不能并行。

|--------|------------|----------|
| OS 定位 | 管理软硬件、抽象与隔离、系统调用 | 理解 JVM/应用与 OS 的关系（见第 7 章） |
| 内核态/用户态 | 权限分级、系统调用陷入内核 | 解释为什么用户程序不能直接操作硬件 |

### ⚠️ 常见坑与注意点

1. **现象**：把“并发”和“并行”混用。**原因**：口语中常混用。**正确做法**：单核多线程是并发；多核多线程才可能并行；面试时先说“并发是逻辑同时、并行是物理同时”。
2. **现象**：认为用户程序可以直接访问任意内存或设备。**原因**：忽略了用户态权限限制。**正确做法**：敏感操作必须通过系统调用进入内核态执行。
3. **现象**：不理解为什么要有 OS 这一层。**原因**：未建立“抽象与隔离”的概念。**正确做法**：记住 OS 提供统一接口、隔离故障与安全边界，便于多任务与多用户。

## 第2章：进程与线程

本章解决**任务如何被 OS 组织与调度**：进程是资源分配单位、线程是 CPU 调度单位，二者关系与状态转换是后续调度（第 3 章）、同步与死锁（第 5、6 章）的基础。依赖第 1 章对 OS 与系统调用的理解。

------

### 1️⃣ 进程

- **进程**：**资源分配的基本单位**，有独立地址空间、文件描述符、**PCB（Process Control Block，进程控制块）** 等；进程间隔离，通信需 **IPC（Inter-Process Communication）**。
- **为什么需要进程**：多道程序需要隔离（一崩不拖垮其他）、需要独立资源视图（地址空间、打开的文件等），进程是 OS 对“正在运行的程序”的抽象。
- **常见 IPC**：管道（匿名/命名）、消息队列、共享内存、信号量、Socket；其中共享内存最快但需自己处理同步。

### 2️⃣ 线程

- **线程**：**CPU 调度的基本单位**，同一进程内共享地址空间与资源（堆、全局变量、文件句柄等），各自有独立的栈和寄存器；切换开销比进程小。
- **为什么需要线程**：在保留进程隔离的前提下，让同一程序内多路执行（如一个线程等 I/O、另一个算逻辑），提高响应与 CPU 利用率。

### 3️⃣ 进程与线程区别

| 维度 | 进程 | 线程 |
|------|------|------|
| 资源 | 独立地址空间、独立资源 | 共享进程地址空间与资源 |
| 调度 | 进程是资源分配单位，线程才是被调度的实体 | 线程是 CPU 调度单位 |
| 切换开销 | 大（切换页表、上下文多） | 小（共享地址空间，主要切换寄存器、栈） |
| 通信 | 需 IPC | 共享内存即可，但要同步（见第 6 章） |

### 4️⃣ 进程状态与转换

典型三态：**就绪（Ready）**、**运行（Running）**、**阻塞（Blocked/Waiting）**。

```
   [新建] → 就绪 ←→ 运行 → [终止]
                ↑      ↓
                └── 阻塞
```
- **就绪 → 运行**：被调度器选中，获得 CPU。  
- **运行 → 就绪**：时间片到或被更高优先级抢占。  
- **运行 → 阻塞**：等待 I/O、信号量等事件。  
- **阻塞 → 就绪**：事件发生，重新参与调度。

> 💬 **面试怎么说**：进程在就绪队列里等 CPU，在阻塞队列里等 I/O/事件；调度器只从就绪队列里选进程上 CPU。

### 5️⃣ 用户线程与内核线程、协程

| 类型 | 说明 | 优缺点 |
|------|------|--------|
| **内核线程** | 由内核调度，一对一或多对一映射到 CPU | 调度公平、可多核并行；切换要进内核，有开销 |
| **用户线程** | 在用户态由运行时调度，多对一映射到内核线程 | 切换快、轻量；一个阻塞可能阻塞整个进程 |
| **协程** | 用户态轻量“线程”，协作式调度，遇 I/O 让出 | 极轻量、高并发 I/O；不适合 CPU 密集 |

------

### 📖 本章小结

| 知识点 | 面试关键词 | 实际应用 |
|--------|------------|----------|
| 进程 | 资源分配单位、PCB、独立地址空间、IPC | 理解一个 Java 进程 = 一个 JVM（第 7 章） |
| 线程 | CPU 调度单位、共享进程资源、切换开销小 | 多线程编程、线程池（与第 6 章同步配合） |
| 进程状态 | 就绪、运行、阻塞及转换 | 分析卡死、负载高时进程在等什么 |
| 用户/内核线程、协程 | 1:1 映射、用户态调度、协程让出 | 解释 JVM 线程与 OS 线程、Go/协程场景 |

------

### ⚠️ 常见坑与注意点

1. **现象**：认为“进程比线程快”或“线程一定比进程省资源”。**原因**：混淆了“调度单位”和“资源单位”。**正确做法**：线程切换快、共享资源方便，但多线程要处理竞态与同步；进程隔离好、但创建与切换成本高。
2. **现象**：多线程写共享变量不加锁，结果不确定。**原因**：未理解“线程共享地址空间”带来的竞态。**正确做法**：共享可写数据必须用同步手段（见第 6 章）。
3. **现象**：把“协程”当万能高并发。**原因**：协程适合 I/O 密集、协作式让出；CPU 密集仍会占满一核。**正确做法**：I/O 密集用协程/多路复用，CPU 密集用多进程/多线程并行。

## 第3章：进程调度与内核执行原理

本章将深入探讨操作系统的核心——**进程调度（Scheduling）与上下文切换（Context Switch）**的底层实现原理。结合 Linux 内核的源码思想，解决不仅“怎么调度”，还包括“内核态怎么组织进程数据结构”、“CFS（完全公平调度器）如何工作”以及“多核 CPU 的缓存亲和性”。这是高并发开发与性能调优的基石。

------

### 1️⃣ Linux 内核中的进程表示：`task_struct`

在 Linux 中，无论进程还是线程，底层都由统一的结构体 **`task_struct`** 表示。
- **万物皆任务（Task）**：Linux 不严格区分进程和线程的结构。线程只是与父进程共享了地址空间（`mm_struct`）、文件描述符表（`files_struct`）等资源的特殊任务。
- **`task_struct` 核心字段**：
  - `state`：运行状态（TASK_RUNNING, TASK_INTERRUPTIBLE, TASK_UNINTERRUPTIBLE 等）。
  - `pid`、`tgid`：进程 ID、线程组 ID（多线程中，同组的 tgid 相同，也就是对外暴露的 PID）。
  - `mm_struct *mm`：指向进程的虚拟地址空间。若为内核线程，此字段为空。
  - `fs_struct *fs` / `files_struct *files`：文件系统与打开的文件描述符表。
  - `sched_entity se`：调度实体，CFS 调度器就是基于该实体计算 `vruntime`（虚拟运行时间）的。

> 💡 **硬核扩展**：当执行 `fork()` 创建子进程时，内核实际调用 `clone()` 系统调用，复制父进程的 `task_struct`。如果是多线程的 `pthread_create()`，也是调用 `clone()`，但会带有 `CLONE_VM`、`CLONE_FILES` 等标志位，告知内核“共享内存与文件”。

### 2️⃣ 经典调度算法及演进

调度算法目标：**响应时间低（针对交互式应用）**、**吞吐量高（针对批处理）**、**公平性好**。

| 算法 | 原理与执行逻辑 | 优缺点 |
|------|---------------|------|
| **FCFS（先来先服务）** | 非抢占式。按到达就绪队列的先后顺序调度。 | 简单；但长作业会导致后面的短作业长时间等待（护航效应）。 |
| **SJF（短作业优先）** | 优先调度估计运行时间短的。可抢占或非抢占。 | 平均等待时间最短；但可能导致长作业饿死，且难以预估时间。 |
| **RR（时间片轮转）** | 抢占式。每个任务分配一个时间片，用完被放入队尾。 | 响应快，适合分时系统；时间片太大退化为 FCFS，太小则上下文切换开销过大。 |
| **多级反馈队列（MLFQ）** | 多个队列优先级递减，时间片递增。新进程入最高优先级队列，时间片耗尽则降级。I/O 阻塞返回保持或提升优先级。 | 兼顾交互式（高优短片）与批处理（低优长片）。几乎是通用 OS 之前的标配。 |

### 3️⃣ Linux 主力调度器：CFS（Completely Fair Scheduler）

现代 Linux 内核（2.6.23 起）不再使用传统 O(1) 调度器或固定时间片，而是采用 **CFS（完全公平调度器）**。针对普通进程（非实时），CFS 放弃了绝对的优先级和固定时间片，转而追求“按权重分配 CPU 时间的绝对公平”。

- **核心思想**：如果有 N 个进程，CFS 试图让每个进程得到 `1/N` 的 CPU 算力。
- **`vruntime`（虚拟运行时间）**：
  - `vruntime = 实际运行时间 * (NICE_0_LOAD / 进程权重)`
  - 权重与进程的 `nice` 值相关。优先级高的进程，权重高，其 `vruntime` 增长得慢。
- **红黑树（Red-Black Tree）调度机制**：
  - CFS 将所有就绪状态的调度实体（`sched_entity`）挂载到一棵红黑树上。
  - 树的 key 是 `vruntime`。树的**最左下角节点**就是 `vruntime` 最小的进程，也就是最需要被补偿 CPU 的进程。
  - 调度器每次挑选最左侧节点上 CPU 执行；执行一段时间后，其 `vruntime` 增加，被重新插入红黑树。
- **I/O 唤醒补偿**：如果进程因为等 I/O 长期休眠，唤醒时它的 `vruntime` 极小。CFS 会将其强制拉升到接近当前树中最小值的水平，既保证它能立刻抢占 CPU（响应 I/O），又避免它饿死别人。

### 4️⃣ 上下文切换的硬核全景（Context Switch）

上下文切换是进程调度的代价，分为**用户级上下文**（寄存器、栈）和**系统级上下文**（内核栈、页表基址寄存器 CR3）。

1. **触发时机**：中断（如时钟中断）、异常（缺页）、系统调用（如 `sleep`, `read` 等主动让出 CPU）。
2. **切换步骤**：
   - **保存 CPU 寄存器（硬件上下文）**：当前寄存器状态压入内核栈，并保存至该进程的 PCB (`task_struct->thread_struct`)。
   - **切换虚拟内存空间（针对进程切换）**：更新 **CR3 寄存器**（保存页目录表物理地址），导致 **TLB（Translation Lookaside Buffer，页表缓存）失效**。这也是**进程切换比线程切换慢的核心原因**！
   - **切换内核栈与硬件上下文**：恢复下一个进程的硬件上下文到寄存器，CPU 弹栈恢复执行。
3. **线程切换 vs 进程切换**：
   - 同属一个进程的线程共享 `mm_struct`。线程切换时，内核判断下一个线程的 `mm` 指针与当前相同，则**不切换 CR3 寄存器，不刷新 TLB**，极大保留了缓存的热度。

### 5️⃣ CPU 亲和性（Affinity）与 NUMA 架构

在高并发后端调优（如 Nginx, Redis 绑核）时，调度必须考虑硬件拓扑：
- **CPU Cache 亲和性**：频繁跨 CPU 核心调度进程会导致 L1/L2 Cache 频繁失效（Cache Miss）。Linux 调度器每个 CPU 核心维护自己的 RunQueue（就绪队列），尽量让进程在同一个核上执行。仅当负载极其不均时才触发负载均衡（Load Rebalancing）。
- **NUMA（非一致性内存访问）架构**：多路 CPU 时代，每个 CPU 有直连的本地内存。访问本地内存极快，跨 Node 访问远程内存较慢。Linux 调度与内存分配策略默认优先在当前 CPU 的本地内存节点分配。

### ⚠️ 常见坑与注意点

1. **现象**：认为“调度算法越复杂越好”。**原因**：忽略实现成本与上下文切换开销。**正确做法**：CFS 摒弃了多级反馈队列的复杂链表操作，用一棵 O(log N) 的红黑树搞定公平调度。
2. **现象**：知道线程切换比进程快，但说不出底层原因。**原因**：未深入到内存管理层。**正确做法**：面试答出“同进程的线程切换不需要更换 CR3 寄存器，不触发 TLB（快表）刷新，保留了内存映射缓存”。
3. **现象**：误以为自旋锁不会造成上下文切换就绝对好。**原因**：自旋锁不让出 CPU，但若占用过长会导致其他核白白空转。**正确做法**：短临界区用自旋锁，长临界区用睡眠锁（如 Mutex）。

| 面试官提问 | 回答思路 |
|------------|----------|
| Linux 中线程和进程在内核的区别是什么？ | 内核通过 `task_struct` 统一管理。通过 `clone()` 系统调用不同标志位决定是否共享 `mm` (内存空间) 和 `files` (文件表)。有独立资源的是进程，共享的是线程。 |

| 为什么进程切换比线程切换慢很多？ | 除了寄存器上下文，最核心是**虚拟内存空间的切换**（修改 CR3 寄存器）。这会导致 TLB 缓存失效，随后大量的内存访问会产生 Page Walk 开销。 |

## 第4章：内存管理与内核分配机制

本章解决**进程如何安全、高效地使用内存**，并深入到 Linux 内存管理的底层机制：从多级页表、TLB 快表，到内核的伙伴系统（Buddy System）与 Slab 分配器，再到 mmap 和 OOM Killer。这是理解高并发服务内存泄漏、GC 停顿与底层调优的核心。

------

### 1️⃣ 虚拟内存与多级页表映射

- **虚拟内存（Virtual Memory）**：现代 OS 的基石。进程以为自己拥有连续的地址空间，实际上 OS 通过**MMU（内存管理单元，硬件）**将虚拟地址映射到零散的物理内存（页帧）上。
  - **优势**：进程隔离、物理内存按需分配、可以换出到磁盘（Swap）以支持超大内存需求。
- **多级页表结构（Linux 4 级/5 级页表）**：
  - 如果只用一级页表，32位系统下每个进程都需要 4MB 连续内存存页表，64位下更是不敢想象。
  - Linux 采用多级页表（如 4 级：PGD -> PUD -> PMD -> PTE）。只有实际用到的虚拟地址空间才会分配对应的底层页表，极大节省了内存。
- **TLB（Translation Lookaside Buffer，快表）**：
  - 页表在内存中，MMU 每次做地址转换（Page Walk）需要多次访存，极慢。
  - TLB 是 CPU 内部的 MMU 缓存，专门缓存最近转换过的“虚拟页号 -> 物理页帧号”。TLB 命中则直接获取物理地址；TLB 未命中才去查内存中的多级页表。

### 2️⃣ 内核物理内存分配：伙伴系统与 Slab

当产生缺页中断（Page Fault）或者内核需要内存时，如何高效分配物理内存？
- **伙伴系统（Buddy System）**：
  - 负责**大块连续物理页（Page Frame）**的分配。
  - 物理内存被分为 1页、2页、4页... 1024页 的多个链表。
  - **分配**：申请 3 页，会找 4 页的链表，切出 3 页（向上取整为4页），剩下不用。
  - **回收**：释放时，如果相邻的“伙伴”也是空闲的，会自动合并成更大的块，**解决外部碎片问题**。
- **Slab 分配器（Slab / Slub）**：
  - 负责**小内存对象**的分配（比如一个几十字节的 `task_struct` 或 inode）。
  - 如果每次申请几十字节都用伙伴系统给一整页（4KB），会产生极大的**内部碎片**。
  - Slab 提前向伙伴系统申请大块页，然后将其切分成固定大小的小块（如 64B, 128B）供内核对象缓存复用，极其高效。

### 3️⃣ 用户态内存分配：malloc 与 brk/mmap

应用层通过 `malloc()` 申请内存，底层对应两个系统调用：
1. **`brk`**：当分配较小内存（通常 < 128KB）时，`brk` 将进程数据段（堆区）的指针（brk 指针）向上移动。这部分内存释放后不会立刻还给 OS，而是由 glibc 的 ptmalloc 缓存复用。
2. **`mmap`**：当分配大内存（> 128KB）时，使用匿名 `mmap` 在进程的内存映射区（Memory Mapping Segment）直接分配。释放时直接归还 OS，避免内存碎片，但有缺页开销。

### 4️⃣ Page Cache（页缓存）与 mmap

- **Page Cache**：Linux 的核心 I/O 优化。所有的文件读写，实际上都是对 Page Cache 的读写。
  - 读：先查 Page Cache，没有再读磁盘，并缓存在 Page Cache 中。
  - 写：写到 Page Cache 就返回（此时叫 Dirty Page 脏页），由内核后台线程（如 `pdflush` / `kworker`）异步刷盘。
- **`mmap` 内存映射**：
  - 传统的 `read/write` 需要从磁盘 -> Page Cache（内核态） -> 用户缓冲区（用户态），发生两次拷贝。
  - `mmap` 将用户空间的某段虚拟地址**直接映射到 Page Cache**，用户进程修改这段内存等同于修改 Page Cache，省去了一次内核到用户态的 CPU 拷贝！适合大文件处理和 Kafka/RocketMQ 等高性能中间件底层。

### 5️⃣ OOM Killer（Out Of Memory Killer）

当系统的物理内存与 Swap 全耗尽时，OS 为了自保，会触发 OOM Killer：
- **打分机制（oom_score）**：内核为每个进程计算 `oom_score`。占用内存越多、运行时间越短的进程得分越高。
- **保护机制**：可以通过调整 `oom_score_adj` 来降低某个进程（如核心 DB）被杀的概率。

| 多级页表与 TLB | PGD、PTE、Page Walk、上下文切换 | 理解线程切换比进程快（不刷新TLB）、大页（HugePage）优化 |
| 伙伴系统与 Slab | 解决外部碎片、解决内部碎片 | 分析 /proc/buddyinfo, /proc/slabinfo，理解内核级内存池 |

### ⚠️ 常见坑与注意点

1. **现象**：认为 `malloc` 返回成功物理内存就分配好了。**原因**：未理解 Linux 的**延迟分配（Demand Paging）**。**正确做法**：`malloc` 只是分配了虚拟地址空间，只有当程序第一次读写该地址时，才会触发缺页中断（Page Fault），内核此时才真正分配物理页。
2. **现象**：把“虚拟内存”等同于“磁盘上的 swap”。**原因**：概念混淆。**正确做法**：虚拟内存是地址空间的抽象；swap 只是在物理内存不够时，充当换出页的 backing store。
3. **现象**：物理内存不够就无脑加 swap。**原因**：忽略换页性能。**正确做法**：swap 频繁读写会导致“抖动（Thrashing）”，极大地拖慢系统。核心服务（如 Elasticsearch、K8s）甚至建议强制关闭 swap。

## 第5章：死锁

本章解决**多进程/多线程因争抢资源而互相等待、无法推进**的问题，与第 6 章同步互斥配合：同步手段用不当会带来死锁。校招常考“四条件 + 预防/避免/检测”。

------

### 1️⃣ 死锁的四个必要条件

四者**同时满足**才可能发生死锁（缺一则可避免死锁）：

| 条件 | 含义 |
|------|------|
| **互斥** | 资源只能被一个进程/线程占用 |
| **占有且等待** | 已持有至少一个资源，并在等待其他资源 |
| **不可剥夺** | 资源不能被强行抢占，只能由持有者主动释放 |
| **循环等待** | 存在进程/线程的循环链：A 等 B、B 等 C、…、最后等 A |

> 💬 **面试怎么说**：死锁是四个条件同时成立时的现象；预防/避免的思路就是破坏其中至少一个。

### 2️⃣ 预防（破坏任一条件）

| 破坏对象 | 做法 |
|----------|------|
| 互斥 | 无法破坏（某些资源本质就是互斥的，如打印机） |
| 占有且等待 | **一次性申请全部所需资源**，不够则等待；避免“拿着 A 等 B” |
| 不可剥夺 | 允许抢占：若申请不到则释放已占资源，稍后重试（实现复杂） |
| 循环等待 | **资源有序分配**：给资源编号，进程只能按编号递增顺序申请；不会出现 A 持高号等低号、B 持低号等高号 |

### 3️⃣ 避免：银行家算法

- **思想**：分配前**模拟一次分配**，判断系统是否仍处于**安全状态**（存在一个安全序列，使所有进程能依次得到资源并完成）；若分配后不存在安全序列则**不分配**，让申请者等待。
- **安全状态**：存在一个进程执行顺序，使得按该顺序每个进程都能在“当前可用 + 前面进程释放”的资源下完成。

### 4️⃣ 检测与恢复

允许死锁发生，**定期检测**（如用资源分配图判断是否存在环路），发现后**恢复**：剥夺某进程资源、回滚、或终止进程重启。

------

### 📖 本章小结

| 知识点 | 面试关键词 | 实际应用 |
|--------|------------|----------|
| 四条件 | 互斥、占有且等待、不可剥夺、循环等待 | 分析“卡死”是否死锁、如何破 |
| 预防 | 一次性申请、资源有序分配 | 设计协议/锁顺序时避免循环等待 |
| 避免 | 银行家算法、安全状态 | 理解“分配前检测”的思路 |
| 检测与恢复 | 资源分配图、剥夺/回滚/重启 | 运维中定位死锁并恢复 |

------

### ⚠️ 常见坑与注意点

1. **现象**：只记“循环等待”一个条件。**原因**：四条件缺一不可。**正确做法**：面试时先说“四条件同时满足才死锁”，再列四个，预防时说明破坏哪一个。
2. **现象**：认为银行家算法能完全避免死锁。**原因**：银行家算法依赖已知的“最大需求”，且要实时计算，实际多用于理论或特殊系统。**正确做法**：能说出“分配前检测安全状态”的思想即可，实际工程多用预防（锁顺序）或检测恢复。
3. **现象**：多锁不加顺序，随意加锁。**原因**：容易形成循环等待。**正确做法**：多锁约定全局顺序（如按锁对象 id 排序），统一按顺序申请，破坏循环等待。

------

### 🎯 面试常见追问

| 面试官提问 | 回答思路 |
|------------|----------|
| 死锁的四个必要条件？ | 互斥、占有且等待、不可剥夺、循环等待；四者同时满足才可能死锁。 |
| 如何预防死锁？ | 破坏其一：一次性申请全部资源（破坏占有且等待）、资源有序分配（破坏循环等待）等。 |
| 银行家算法是什么？ | 分配前检查是否仍存在安全序列；若分配后不安全则不分配，避免进入死锁状态。若追问安全状态定义，可说“存在一个进程顺序都能依次获得资源并完成”。 |

## 第6章：同步与互斥

本章解决**多线程/多进程访问共享资源时的互斥与顺序协调**问题，是第 2 章线程、第 5 章死锁的延续：用锁与信号量等实现互斥与同步，用错会竞态或死锁。与 Java 的 `synchronized`、`ReentrantLock`、`wait/notify` 对应。

------

### 1️⃣ 临界区与互斥

- **临界区（Critical Section）**：访问**共享资源**的那段代码；多个执行流不能同时进入同一临界区，否则产生**竞态**、结果不可预期。
- **互斥**：保证同一时刻只有一个执行流在临界区内；常用**互斥量（Mutex）**或**信号量**实现。

### 2️⃣ 信号量与互斥量

- **信号量（Semaphore）**：一个整数 + **P（wait）** 与 **V（signal）** 操作。P：若值 > 0 则减 1 并继续，否则阻塞；V：加 1 并唤醒一个等待者。**二元信号量**（0/1）即可实现互斥；**一般信号量**可表示资源个数（如多槽缓冲区）。
- **互斥量（Mutex）**：可理解为**只用于互斥的锁**，同一时刻只允许一个持有；加锁/解锁成对，通常由同一线程释放。
- **区别**：信号量可计多资源、可被不同线程 V；互斥量一般“谁加锁谁解锁”，语义更贴近“独占临界区”。

### 3️⃣ 条件变量与管程

- **条件变量**：在“满足某条件才继续”时使用；线程在条件变量上 **wait**（释放锁并睡眠），其他线程在条件成立时 **signal** 唤醒。常与互斥量配合：先加锁，再在条件变量上 wait/signal。
- **管程（Monitor）**：把互斥与条件变量封装成一种结构，任一时刻只有一个线程在管程内执行；Java 的 `synchronized` + `wait/notify` 即管程风格。

### 4️⃣ 经典模型（流程要点）

- **生产者-消费者**：有界缓冲区；缓冲区满时生产者阻塞，空时消费者阻塞；用互斥量保护缓冲区，用两个条件变量（或两个信号量）表示“非满”“非空”。
- **读者-写者**：多读者可同时读，写者独占；可用读写锁或“读者计数 + 互斥量 + 条件变量”实现。

### 5️⃣ 锁的类型

| 类型 | 行为 | 适用 |
|------|------|------|
| **自旋锁** | 等不到就忙等（循环检查） | 临界区极短、不想进内核睡眠 |
| **睡眠锁** | 等不到就睡眠，被唤醒再试 | 临界区可能较长，避免占 CPU |
| **读写锁** | 多读可并发、写独占 | 读多写少场景 |

------

### 📖 本章小结

| 知识点 | 面试关键词 | 实际应用 |
|--------|------------|----------|
| 临界区与互斥 | 临界区、竞态、互斥量 | 多线程共享变量必须加锁 |
| 信号量 vs 互斥量 | P/V、二元信号量、谁加锁谁解锁 | 对应 Java 的 Semaphore、synchronized/Lock |
| 条件变量与管程 | wait/signal、管程 | wait/notify、ReentrantLock+Condition |
| 生产者-消费者 | 有界缓冲、满/空阻塞 | 消息队列、线程池任务队列 |
| 自旋锁 vs 睡眠锁 | 忙等 vs 睡眠 | 短临界区用自旋，长用睡眠 |

------

### ⚠️ 常见坑与注意点

1. **现象**：忘记在 wait 前释放锁或 signal 后不释放锁。**原因**：条件变量必须与互斥量配合，wait 时原子地“释放锁并睡眠”。**正确做法**：使用条件变量时严格按“加锁 → 判断条件 → wait/signal → 解锁”的规范，避免丢失唤醒或死锁。
2. **现象**：用信号量实现互斥时，P 和 V 不成对或不在同一层。**原因**：V 多了会导致多进临界区，V 少了会永久阻塞。**正确做法**：成对使用，且保证每条执行路径的 P/V 数量一致。
3. **现象**：长临界区用自旋锁，CPU 空转高。**原因**：自旋锁适合“很快能拿到锁”的场景。**正确做法**：预计等待时间较长时用睡眠锁（互斥量/信号量），让出 CPU。

------

### 🎯 面试常见追问

| 面试官提问 | 回答思路 |
|------------|----------|
| 信号量和互斥量的区别？ | 信号量可表示多资源、P/V 可被不同线程调用；互斥量一般独占、谁加锁谁解锁。二元信号量可当互斥用。 |
| 生产者-消费者怎么实现？ | 有界队列 + 互斥保护 + 两个条件变量（或信号量）：非满时生产者可放，非空时消费者可取；满则生产者 wait，空则消费者 wait。若追问死锁见第 5 章。 |
| 如何保证多线程安全？ | 识别共享可变数据与临界区；用互斥量或信号量保证互斥；需要“等条件”时用条件变量；注意锁顺序防死锁。 |

## 第7章：Linux 网络 I/O 模型与零拷贝技术

本章将前文的 OS 概念**落到后端网络编程**的实战中，深入剖析 Linux 网络 I/O 的五种模型、零拷贝（Zero-Copy）技术的底层实现，以及网络包从网卡到应用层的全过程。这是理解 Netty、Redis、Kafka 等高性能中间件架构的核心基石。

------

### 1️⃣ Linux 五大 I/O 模型

在 Linux 中，一次网络读操作（`read`）通常包含两个阶段：
1. **等待数据准备好（Wait for data）**：数据从网卡到内核 Socket 缓冲区。
2. **将数据从内核态拷贝到用户态（Copy data from kernel to user）**。

| 模型 | 阶段1：等待数据 | 阶段2：数据拷贝 | 特点与应用 |
|------|----------------|----------------|------------|
| **阻塞 I/O (Blocking I/O)** | 阻塞 | 阻塞 | 最传统（BIO）。线程挂起，让出 CPU，直至数据拷贝完成。并发低。 |
| **非阻塞 I/O (Non-Blocking I/O)** | 非阻塞（轮询） | 阻塞 | 等待阶段一直轮询（EWOULDBLOCK），极耗 CPU；拷贝时依然阻塞。极少直接使用。 |
| **I/O 多路复用 (I/O Multiplexing)** | 阻塞在 `select/epoll` | 阻塞在 `read` | 一个线程管理多个 Socket。**核心优势**：等待阶段是阻塞在 epoll_wait 上，而不是 read 上，可以处理上万连接。NIO 的核心。 |
| **信号驱动 I/O (Signal-Driven I/O)** | 非阻塞（发信号） | 阻塞 | 内核在数据准备好时发送 SIGIO 信号，应用再发起 read。很少使用。 |
| **异步 I/O (Asynchronous I/O)** | 非阻塞 | 非阻塞 | **真正的 AIO**。应用发起 aio_read 后立刻返回，内核负责等待并**完成数据拷贝**，最后给应用发通知。Linux 原生 AIO 支持不够完美，Java AIO 使用较少。 |

### 2️⃣ 高并发网络基石：Reactor 与 Proactor 模式

基于上述 I/O 模型，演化出了高性能网络框架的两种设计模式：

- **Reactor 模式（基于 I/O 多路复用）**：
  - **核心思想**：“来了事件我通知你，你自己去读写”。
  - **组件**：Acceptor 负责连接，Reactor 负责事件分发，Handler 负责业务处理。
  - **应用**：Redis（单 Reactor 单线程）、Netty（主从 Reactor 多线程模式，BossGroup 负责 accept，WorkerGroup 负责 read/write）。
- **Proactor 模式（基于异步 I/O）**：
  - **核心思想**：“来了事件我帮你读写完，直接把数据给你”。
  - **应用**：Windows 下的 IOCP 完美实现了 Proactor；Linux 下由于 AIO 不完善，通常是用 epoll 模拟的 Reactor 模式。

### 3️⃣ 零拷贝技术（Zero-Copy）底层的硬核内幕

传统的将文件通过网络发出去（`read` + `write`）：
1. 磁盘 -> Page Cache（DMA 拷贝）
2. Page Cache -> 用户缓冲区（CPU 拷贝，状态切换：内核->用户）
3. 用户缓冲区 -> Socket 缓冲区（CPU 拷贝，状态切换：用户->内核）
4. Socket 缓冲区 -> 网卡（DMA 拷贝）
**总计：4 次拷贝（2 次 CPU + 2 次 DMA），4 次上下文切换。**

零拷贝旨在**减少 CPU 参与的拷贝次数和上下文切换次数**：

- **`mmap` + `write`（Kafka 常用）**：
  - 将文件直接映射到用户空间（Page Cache 与用户空间共享）。
  - 省去了“Page Cache -> 用户缓冲区”的 CPU 拷贝。**总计：3 次拷贝（1 次 CPU），4 次上下文切换**。
- **`sendfile` 系统调用（Nginx、Kafka 常用）**：
  - Linux 2.1 引入，直接在内核态将数据从 Page Cache 拷贝到 Socket 缓冲区。
  - **总计：3 次拷贝（1 次 CPU），2 次上下文切换**。
- **带有 SG-DMA 的 `sendfile`（真正的零拷贝）**：
  - Linux 2.4 优化，如果网卡支持 SG-DMA（Scatter-Gather DMA）。
  - `sendfile` 只把**描述符（内存地址和长度）**发到 Socket 缓冲区。网卡 DMA 直接从 Page Cache 读取数据。
  - **总计：2 次 DMA 拷贝，0 次 CPU 拷贝，2 次上下文切换！**绝对的高性能。
- **`splice`**：Linux 2.6.17 引入，通过在管道（Pipe）间移动内存页指针实现零拷贝，无需硬件 SG-DMA 支持。

### 4️⃣ 网络包的硬核旅程（从网卡到应用）

当一个数据包到达服务器时，内核发生了什么？
1. **网卡接收**：网卡收到光电信号转化为数据帧，放入网卡的 RX FIFO。
2. **硬中断**：网卡向 CPU 发送硬中断（IRQ）。CPU 挂起当前任务，执行中断处理程序，屏蔽该中断。
3. **软中断（NAPI 机制）**：硬中断例程唤醒内核的**软中断（NET_RX_SOFTIRQ）**，如 `ksoftirqd` 线程。
4. **DMA 拷贝与驱动轮询**：软中断在后台通过 `poll` 机制批量将网卡中的数据通过 DMA 拷贝到内核的 `sk_buff`（Socket Buffer）链表中。
5. **协议栈剥洋葱**：
   - 数据链路层：去 MAC 头。
   - 网络层（IP）：去 IP 头，查路由表。
   - 传输层（TCP）：去 TCP 头，验证校验和，确认序列号，触发 `epoll` 的回调函数，将对应的 fd 加入就绪链表。
6. **唤醒应用**：数据放入对应 Socket 的接收缓冲区。应用层通过 `epoll_wait` 感知到 fd 就绪，调用 `read` 从 Socket 缓冲区将数据拷贝到用户态。

| 知识点 | 深度面试关键词 | 实际应用 / 调优 |
|--------|------------|----------|
| I/O 模型 | 阻塞、非阻塞、多路复用、异步 | 理解 NIO（非阻塞+多路复用）、AIO 区别 |
| 零拷贝 | mmap、sendfile、SG-DMA、CPU/DMA 拷贝 | Kafka/RocketMQ 底层为什么快、Nginx 开启 `sendfile on` |
| 网络软中断 | 硬中断、`ksoftirqd`、`sk_buff` | CPU `si`（软中断）占用过高排查；网卡多队列绑定多核 |
| Reactor 模型 | 也是 I/O 多路复用、主从 Reactor | Netty 的 EventLoopGroup、Redis 的事件模型 |

------

### ⚠️ 常见坑与注意点

1. **现象**：认为 NIO 就是完全不阻塞的。**原因**：概念混淆。**正确做法**：Java 的 NIO（New I/O）在 Linux 下基于 epoll，属于 I/O 多路复用。它的 `read/write` 阶段如果有数据要拷贝，依然是**阻塞**的（只不过有数据时立刻返回，没有数据返回 EWOULDBLOCK）。只有真正的 AIO 才是全程非阻塞。
2. **现象**：认为零拷贝就是 0 次拷贝。**原因**：望文生义。**正确做法**：零拷贝指的是**零次 CPU 拷贝**，DMA 拷贝（硬件搬运）依然是存在的。
3. **现象**：高并发下通过 `top` 看到某个单核的 CPU `si` (软中断) 达到 100%。**原因**：网卡的中断全部打在一个 CPU 核上了。**正确做法**：开启网卡多队列（RSS/RPS），将中断分散到多个 CPU 核处理。

| 面试官提问 | 回答思路 |
|------------|----------|
| Kafka 为什么能支撑超高吞吐？ | 除了顺序写磁盘，最重要的是零拷贝。读取消息时使用 `sendfile` 系统调用，利用 DMA 直接把 Page Cache 的数据发送到网卡，不经过应用层。 |
| NIO 和 AIO 的根本区别？ | NIO 是“就绪通知”（操作系统告诉你可读了，你自己去调 read 拷贝）；AIO 是“完成通知”（操作系统帮你 read 完了，直接把填满的 buffer 交给你）。 |
| 描述一下数据包从网卡到 Socket 的过程？ | 网卡收包 -> 触发硬中断 -> 触发软中断 (ksoftirqd) -> 驱动通过 DMA 把包放到内存 sk_buff -> IP 层 -> TCP 层校验 -> 放入对应 Socket 的接收队列，触发 epoll 回调唤醒进程。 |

## 第8章：I/O 管理

本章解决**设备与磁盘 I/O 的高效管理**：从磁盘调度算法到 DMA、缓冲区管理，是理解数据库索引、文件系统性能的基础，也与第 7 章的 NIO/多路复用形成互补（用户态 I/O 与内核态 I/O 的衔接）。

------

### 1️⃣ I/O 控制方式演进

| 方式 | 机制 | 特点 | 缺点 |
|------|------|------|------|
| **程序直接控制** | CPU 轮询设备状态寄存器 | 简单 | CPU 忙等，效率极低 |
| **中断驱动** | 设备就绪后发中断，CPU 响应处理 | 无需轮询 | 每次中断仍需 CPU 参与数据搬运 |
| **DMA（直接内存访问）** | DMA 控制器直接在内存与设备间传数据，完成后中断 CPU | CPU 只需发起和响应，不介入传输 | 需要 DMA 硬件支持 |
| **通道 I/O** | 专用 I/O 处理器（通道）执行通道程序，管理多设备 | CPU 干预最少，适合大型机 | 硬件复杂 |

> 💡 **重点**：现代系统普遍用 **DMA**；磁盘、网卡等高速设备都支持 DMA，大幅减少 CPU 负担。

### 2️⃣ 磁盘调度算法

磁盘是慢速设备，寻道时间占大头；调度算法决定 I/O 请求的处理顺序，目标是**减少寻道时间**。

| 算法 | 思路 | 优点 | 缺点 |
|------|------|------|------|
| **FCFS** | 先来先服务 | 公平、简单 | 可能频繁来回移动，寻道时间长 |
| **SSTF（最短寻道时间优先）** | 优先处理离当前磁头最近的请求 | 平均寻道时间短 | 可能饿死远处请求 |
| **SCAN（电梯算法）** | 磁头单向移动到底，再反向 | 无饿死，较公平 | 两端请求等待时间不均 |
| **C-SCAN（循环扫描）** | 单向移动到底后快速回到起点，再单向 | 等待时间更均匀 | 回程不处理请求 |
| **LOOK / C-LOOK** | SCAN/C-SCAN 的优化，不到底就回头 | 减少无效移动 | 实现稍复杂 |

> 💬 **面试怎么说**：SSTF 可能饿死；SCAN 像电梯来回；C-SCAN 让等待时间更均匀；实际系统常用 LOOK/C-LOOK 或结合 SSD 特性优化。

### 3️⃣ 缓冲区管理

- **为什么需要缓冲**：
  1. 缓解 **CPU 与 I/O 速度不匹配**
  2. 减少 **中断次数**（缓冲满/空才中断）
  3. 解决 **数据块大小不匹配**（如字符设备 vs 块设备）

- **单缓冲、双缓冲、循环缓冲**：
  - **单缓冲**：一个缓冲区，读写串行
  - **双缓冲**：两个缓冲区交替，读写可并行（生产者-消费者模式）
  - **循环缓冲**：多个缓冲区组成环形队列，适合高速 I/O

- **缓冲池**：系统统一管理多个缓冲区，按需分配，提高利用率。

### 4️⃣ 磁盘结构与性能

- **磁盘结构**： platter（盘片）→ track（磁道）→ sector（扇区，通常 512B 或 4KB）
- **访问时间 = 寻道时间 + 旋转延迟 + 传输时间**
  - 寻道时间：磁头移动到目标磁道（通常 3-10ms）
  - 旋转延迟：盘片旋转到目标扇区（平均半圈，2-4ms）
  - 传输时间：实际读写数据（微秒级）

> 💡 **SSD 差异**：SSD 无机械结构，无寻道时间和旋转延迟，随机读写性能接近顺序读写；但仍有写入放大、磨损均衡等问题。

------

### 📖 本章小结

| 知识点 | 面试关键词 | 实际应用 |
|--------|------------|----------|
| I/O 控制方式 | 轮询、中断、DMA、通道 | 理解 DMA 减少 CPU 负担 |
| 磁盘调度 | FCFS、SSTF、SCAN、C-SCAN | 数据库、文件系统性能优化 |
| 缓冲区 | 单/双缓冲、缓冲池 | 流式处理、生产者-消费者 |
| 磁盘性能 | 寻道、旋转、传输时间 | SSD vs HDD 选型、性能估算 |

------

### ⚠️ 常见坑与注意点

1. **现象**：认为 DMA 完全不需要 CPU 参与。**原因**：DMA 只是数据传输不经过 CPU，但发起传输、处理完成中断仍需 CPU。**正确做法**：理解 DMA 是"减少"而非"消除" CPU 介入。
2. **现象**：磁盘调度算法在 SSD 上盲目套用。**原因**：SSD 无机械延迟，随机读写快，调度算法意义不大。**正确做法**：SSD 关注磨损均衡、TRIM、写入放大；HDD 才需关注寻道优化。
3. **现象**：忽略旋转延迟。**原因**：只考虑寻道时间。**正确做法**：磁盘访问时间 = 寻道 + 旋转延迟 + 传输；旋转延迟平均是半圈时间。

------

### 🎯 面试常见追问

| 面试官提问 | 回答思路 |
|------------|----------|
| DMA 是什么？为什么需要？ | 直接内存访问，DMA 控制器直接在设备与内存间传数据，减少 CPU 干预；适合高速设备批量传输。 |
| 磁盘调度算法有哪些？ | FCFS、SSTF、SCAN（电梯）、C-SCAN；SSTF 可能饿死，SCAN 公平，C-SCAN 等待均匀。 |
| 为什么 SSD 不需要磁盘调度？ | SSD 无机械结构，随机访问与顺序访问性能接近，无寻道时间，调度算法收益不大。 |

## 第9章：文件系统

本章解决**文件如何被组织、存储、访问**：从文件的逻辑结构到物理结构，inode、目录、链接，以及日志文件系统，是理解"ls 为什么快""rm 后能否恢复"等问题的基础。与第 8 章磁盘 I/O、第 4 章内存管理紧密相关。

------

### 1️⃣ 文件与文件系统概述

- **文件**：具有符号名的数据集合，是 OS 管理存储的基本单位。
- **文件系统**：负责文件的存储、检索、共享、保护，提供按名存取的抽象。
- **文件逻辑结构**：
  - **无结构（流式）**：字节序列，如文本文件
  - **有结构**：记录序列，如数据库表

### 2️⃣ 文件物理结构（分配方式）

| 方式 | 机制 | 优点 | 缺点 |
|------|------|------|------|
| **连续分配** | 文件占连续磁盘块 | 顺序访问快，定位简单 | 产生外部碎片，不便扩展 |
| **链接分配** | 块间用指针链接（隐式/显式 FAT） | 无外部碎片，扩展方便 | 随机访问慢，指针占空间 |
| **索引分配** | 每个文件有一个索引块，记录所有块号 | 支持直接访问，扩展灵活 | 索引块 overhead，小文件浪费 |

> 💡 **混合方案**：Unix/Linux 采用 **多级索引**（直接索引 + 间接索引），小文件直接存块号，大文件用间接块，兼顾效率与灵活性。

### 3️⃣ inode（索引节点）

- **inode**：文件的元数据结构，存储文件的属性（大小、权限、时间、块指针等），**不含文件名**。
- **文件名 → inode 号 → 文件数据**：目录是"文件名-inode号"的映射表。
- **inode 内容**：
  - 直接指针（通常 12 个）：指向数据块
  - 一级间接指针：指向一个块，该块存储数据块地址
  - 二级、三级间接指针：更大文件

```
文件名 → 目录项 → inode 号 → inode 表 → 块指针 → 数据块
```

> 💬 **面试怎么说**：inode 是文件的"身份证"，存元数据和块位置；文件名只是目录里的一个映射；ls -i 可看 inode 号。

### 4️⃣ 目录结构

- **目录本质**：特殊文件，内容是"文件名 → inode 号"的映射表。
- **目录结构**：
  - **单级目录**：所有文件放一起，易冲突
  - **两级目录**：用户目录隔离
  - **树形目录**：层级结构，有绝对/相对路径
  - **无环图目录**：支持共享（硬链接），需避免环

### 5️⃣ 硬链接与软链接（符号链接）

| 特性 | 硬链接 | 软链接（符号链接） |
|------|--------|-------------------|
| **本质** | 多个目录项指向同一 inode | 特殊文件，存目标路径 |
| **inode** | 相同 | 不同 |
| **跨文件系统** | 不可 | 可以 |
| **原文件删除** | 仍可访问（引用计数减1） | 失效（ dangling link ） |
| **类型** | 普通文件 | 链接文件（l） |

> 💡 **ln 命令**：`ln file hardlink`（硬链接），`ln -s file symlink`（软链接）

### 6️⃣ 文件存储空间管理

- **空闲块管理**：
  - **空闲表**：记录连续空闲块，适合连续分配
  - **空闲链表**：链式管理，简单但遍历慢
  - **位示图（Bitmap）**：每位表示一块是否空闲，省空间、易查找
  - **成组链接**：Unix 采用，结合链表与空闲表优点

### 7️⃣ 日志文件系统（Journaling）

- **问题**：文件系统操作常涉及多个元数据更新（如分配块、改 inode、改目录），崩溃时可能处于不一致状态。
- **日志（Journal）**：先写日志（意图），再执行实际修改；崩溃后通过日志恢复。
- **日志模式**：
  - **Writeback**：只日志元数据，性能最好，数据可能丢
  - **Ordered**：元数据日志 + 数据先写，折中
  - **Data**：元数据和数据都日志，最安全，性能最差

> 💡 **ext3/ext4**：Linux 主流日志文件系统；数据库也有类似 WAL（Write-Ahead Log）机制。

------

### 📖 本章小结

| 知识点 | 面试关键词 | 实际应用 |
|--------|------------|----------|
| 文件分配 | 连续、链接、索引、多级索引 | 理解文件大小限制、访问效率 |
| inode | 元数据、块指针、间接索引 | ls -i、df -i、理解硬链接 |
| 硬/软链接 | 同 inode vs 存路径、跨 FS | 文件共享、快捷方式 |
| 日志文件系统 | Journal、WAL、崩溃恢复 | ext4、数据库事务日志 |

------

### ⚠️ 常见坑与注意点

1. **现象**：认为删除文件就释放了空间。**原因**：硬链接存在时，inode 引用计数不为 0，空间不释放。**正确做法**：理解 rm 是减少链接计数，计数为 0 才真正释放。
2. **现象**：硬链接跨文件系统。**原因**：不同 FS 的 inode 空间独立。**正确做法**：跨 FS 只能用软链接。
3. **现象**：忽略目录也是文件。**原因**：目录存的是文件名-inode映射。**正确做法**：目录有大小（存映射表）、有 inode，是特殊类型的文件。

------

### 🎯 面试常见追问

| 面试官提问 | 回答思路 |
|------------|----------|
| 什么是 inode？和文件名什么关系？ | inode 存文件元数据和块位置；文件名是目录里的映射；一个 inode 可被多个文件名指向（硬链接）。 |
| 硬链接和软链接区别？ | 硬链接：同 inode，不能跨 FS，原文件删了还能访问；软链接：存路径，不同 inode，可跨 FS，原文件删了失效。 |
| 文件系统如何管理空闲空间？ | 位示图、空闲表、空闲链表、成组链接；现代常用位示图或成组链接。 |
| 日志文件系统是什么？ | 先写日志再改数据，崩溃后通过日志恢复一致性；有 writeback/ordered/data 三种模式。 |

## 第10章：Linux 常用命令与 OS 原理

### ⚠️ 常见坑与注意点

1. **现象**：只看 CPU 使用率不看负载。**原因**：负载（load average）包含等待 I/O 的进程。**正确做法**：`uptime` 看负载，`vmstat` 看 si/so 和 wa（I/O 等待）。
2. **现象**：僵尸进程（Z 状态）杀不掉。**原因**：僵尸进程已终止，只是父进程未收尸；杀父进程才能解决。**正确做法**：找到父进程，kill 父进程或修复父进程代码。
3. **现象**：TIME_WAIT 过多认为是问题。**原因**：主动关闭方正常状态，2MSL 后自动释放；过多可能是短连接频繁。**正确做法**：考虑连接池、长连接，或调整 tcp_tw_reuse（谨慎）。

## 第11章：epoll 源码级原理详解

本章深入**Linux 高并发 I/O 的终级杀器 epoll**，从内核源码（`eventpoll` 结构）级别对比 select/poll，理解为什么 epoll 能支撑 C100K 甚至 C1000K。

------

### 1️⃣ 为什么需要 I/O 多路复用

- **问题**：在阻塞 I/O 模型下，一个线程只能处理一个连接，10K 连接需要 10K 线程，上下文切换开销巨大。
- **解决**：一个线程**同时监视多个文件描述符（fd）**，哪个就绪就处理哪个 → **I/O 多路复用**。

### 2️⃣ select 与 poll 的局限（为什么被淘汰？）

| 特性 | select | poll |
|------|--------|------|
| **fd 上限** | 1024（由内核宏 `FD_SETSIZE` 决定） | 无硬限（链表/数组存储，仅受内存限制） |
| **数据结构** | `fd_set`（位图） | `pollfd` 数组 |
| **工作机制** | 每次传入全量 fd 集合，内核遍历轮询 | 类似，传入 pollfd 数组，内核遍历 |

**被淘汰的致命痛点**：
1. **每次调用都需要把 fd 集合从用户态全量拷贝到内核态**：在高并发下（如 10 万连接），这个全量拷贝开销极大。
2. **内核态的 O(N) 盲轮询**：内核在拿到 fd 集合后，需要遍历所有 fd，逐个调用驱动程序的 `poll` 方法检查是否就绪。
3. **用户态的 O(N) 再遍历**：`select/poll` 返回后，只告诉用户态“有几个就绪了”，但不说是谁。用户态程序必须再次遍历整个 fd 集合找就绪的 socket。

### 3️⃣ epoll 的硬核数据结构（`eventpoll`）

`epoll` 摒弃了 select/poll 的“一揽子”系统调用，拆分成三个：`epoll_create`、`epoll_ctl`、`epoll_wait`。
内核为此创建了专用的数据结构 `struct eventpoll`，其核心包含两部分：
1. **一棵红黑树（RB-Tree）**：存储所有被监控的 fd。
2. **一个双向链表（Ready List，就绪队列）**：存储已经就绪的 fd。

### 4️⃣ epoll 三大系统调用执行过程

#### Step 1: `epoll_create`
在内核中分配一个 `eventpoll` 结构体，并返回一个 epoll 专属的 fd。这就相当于在内核里开辟了一块“专区”。

#### Step 2: `epoll_ctl` (ADD / MOD / DEL)
将需要监控的 socket fd 添加到 `eventpoll` 的**红黑树**中。
- **O(log N) 高效插入/查找**：红黑树保证了在数十万 fd 下的高效管理。
- **【核心】注册等待队列回调函数 `ep_poll_callback`**：这是 epoll 高效的灵魂！在把 fd 挂到红黑树的同时，内核会向该网卡驱动/socket 注册一个回调函数。这意味着**不再需要盲轮询**了！

#### Step 3: `epoll_wait`
应用程序调用 `epoll_wait`，检查 `eventpoll` 中的**就绪链表**是否有元素：
- 如果没有，则进程让出 CPU，陷入休眠。
- **数据到达时的中断与回调（异步唤醒）**：
  1. 数据包到达网卡，触发中断，存入 socket 接收缓冲区。
  2. TCP 协议栈收到数据后，触发刚才注册的回调函数 `ep_poll_callback`。
  3. 回调函数**直接将该 socket fd 的引用封装成 `epitem`，插入到 `eventpoll` 的就绪链表中**，并唤醒阻塞在 `epoll_wait` 上的进程。
- **返回用户态**：`epoll_wait` 被唤醒后，只需将**就绪链表中的少量元素**拷贝到用户态传入的 `events` 数组中。时间复杂度是 O(就绪数)，而不是 O(总连接数 N)！

### 5️⃣ 水平触发（LT）与边缘触发（ET）

| 模式 | 触发条件 | 特点与代码要求 |
|------|----------|------|
| **LT（Level Trigger）** | 只要缓冲区还有数据可读，就会**不断触发** `epoll_wait`。 | **默认模式**。编程简单；若一次没读完，下次继续唤醒；但对内核有一定负担（频繁唤醒）。 |
| **ET（Edge Trigger）** | 只有缓冲区状态**发生变化**（如从空变非空，或新来了一批数据）时，才触发**一次**。 | 高效（减少 `epoll_wait` 唤醒次数）。**【极其严苛的编程要求】**：1. fd 必须设为**非阻塞 (O_NONBLOCK)**；2. 必须用 `while(true)` 循环 `read` 直到返回 `EAGAIN`，否则剩下的数据将永远不会再触发通知（导致消息丢失/卡死）。 |

> 💡 **惊群效应 (Thundering Herd)**：多进程/线程同时阻塞在同一个 `epoll_wait` 上等同一个 fd 时，fd 就绪会唤醒所有进程，但只有一个能 `accept` 成功，其他的会空跑。Linux 4.5+ 引入了 `EPOLLEXCLUSIVE` 标志完美解决了 epoll 的惊群问题。

### 6️⃣ epoll vs select/poll 对比总结

| 维度 | select/poll | epoll |
|------|-------------|-------|
| **底层数据结构** | 数组 / 链表 | **红黑树** + **就绪双向链表** |
| **拷贝开销** | 每次调用都全量拷贝所有 fd 到内核 | `epoll_ctl` 时注册一次，后续不需要全量拷贝 |
| **查找就绪的机制** | 内核遍历轮询 O(N) | **事件驱动（回调机制）**，自动插入就绪链表 O(1) |
| **返回数据给用户** | 返回全量 fd，需用户遍历找就绪 O(N) | **只返回就绪的 fd**，用户处理 O(就绪数) |
| **适用场景** | fd 少、需跨平台（Windows 无 epoll） | 海量连接的高并发 Linux 服务器 |

### ⚠️ 常见坑与注意点

1. **现象**：ET 模式下数据没读完，下次不再触发。**原因**：ET 只在状态变化时触发一次。**正确做法**：ET 必须设非阻塞，循环读到 `EAGAIN`。
2. **现象**：认为 epoll 完全使用了 `mmap` 实现零拷贝。**原因**：早期网文误传。**正确做法**：Linux 源码显示，`epoll_wait` 返回就绪事件时，依然使用的是 `__put_user` 等函数进行从内核态到用户态的**内存拷贝**，并未用到 `mmap`！它的快是快在不需要“全量”拷贝。
3. **现象**：fd 关闭后未 `epoll_ctl(DEL)`。**原因**：关闭的 fd 可能被系统复用，导致 epoll 报告错误事件。**正确做法**：调用 `close(fd)` 前先 `epoll_ctl(DEL)`，虽然 Linux 内核在所有文件引用释放时会自动清理，但显式处理更严谨。

### 🎯 面试常见追问

| 面试官提问 | 回答思路 |
|------------|----------|
| epoll 为什么比 select 快？ | 1. 结构：红黑树增删改 O(logN)；2. 零轮询：不用遍历全量 fd，而是通过网卡中断的回调函数将就绪 fd 挂入链表；3. 拷贝少：不需要每次调用拷贝全量 fd 集合，只返回就绪队列中的数据。 |
| epoll 的 ET 模式为什么要用非阻塞 I/O？ | ET 模式下因为不知道缓冲区有多少数据，必须循环 `read` 读到空为止。如果用阻塞 I/O，最后一次 `read` 发现没数据了，线程就会死锁阻塞在那里，整个 epoll 所在的 Reactor 线程就废了。 |
| epoll 内部有没有使用 mmap？ | **没有**。源码中向用户空间返回就绪事件数组使用的是 `put_user` 拷贝。 |
