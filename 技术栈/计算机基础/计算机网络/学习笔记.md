# 📘《计算机网络学习与实战手册》

> 系统学习见下文各章；日常常用概念与使用场景速查见同目录《常用概念与使用场景》。  
> 适用：通用计网概念（TCP/IP 为主），面向校招与后端开发（HTTP/HTTPS、TCP、DNS、常见攻击等）。

------

## 第1章：计算机网络概述

本章建立对**网络分层与各层职责**的整体认知，为后续 TCP/UDP（第 2 章）、HTTP/HTTPS（第 3 章）、DNS（第 4 章）及网络层（第 5 章）打基础。校招常从“七层/四层是什么、每层干什么”问起。

------

### 1️⃣ 分层模型

- **为什么分层**：职责分离、便于实现与替换（如换物理介质不影响上层）；每层只依赖下层服务，对上提供封装好的接口。
- **OSI 七层**（理论模型）：物理层、数据链路层、网络层、传输层、会话层、表示层、应用层。
- **TCP/IP 四层**（实际常用）：网络接口层、网际层（IP）、传输层（TCP/UDP）、应用层（HTTP、DNS、FTP 等）。校招与后端开发常重点掌握**传输层与应用层**。

| 层级（TCP/IP） | 典型协议/内容 | 关注点 |
|----------------|---------------|--------|
| 应用层 | HTTP、HTTPS、DNS、FTP、SMTP | 业务协议、状态码、方法、头 |
| 传输层 | TCP、UDP | 连接、可靠/不可靠、端口 |
| 网际层 | IP、ICMP、ARP | 寻址、路由、分片 |
| 网络接口层 | 以太网、Wi-Fi | 帧、MAC、物理介质 |

### 2️⃣ 各层职责简述

- **网络层**：**IP 寻址与路由**、分组转发；不保证可靠、不保证顺序，尽力而为。
- **传输层**：**端到端**的可靠（TCP）或不可靠（UDP）传输；用**端口**区分同一主机上的不同进程。
- **应用层**：具体协议与业务（HTTP、DNS、SMTP 等），数据格式与语义由各协议定义。

------

### 📖 本章小结

| 知识点 | 面试关键词 | 实际应用 |
|--------|------------|----------|
| 分层意义 | 职责分离、封装、便于替换 | 理解“改一层不影响其他层” |
| OSI 七层 / TCP/IP 四层 | 物理→应用、四层对应关系 | 能画出四层并说出典型协议 |
| 各层职责 | IP 寻址路由、TCP/UDP 端到端、应用层协议 | 分析问题落在哪一层（如丢包→传输/网络层） |

------

### ⚠️ 常见坑与注意点

1. **现象**：把 OSI 七层和 TCP/IP 四层一一对应错。**原因**：OSI 会话/表示层在 TCP/IP 里常归入应用层。**正确做法**：记 TCP/IP 四层及每层典型协议；被问七层时能列出来并说明“实际多用四层模型”。
2. **现象**：认为“传输层负责把数据送到对方主机”。**原因**：送到主机是网络层（IP）的职责；传输层负责端到端（进程到进程）与可靠/不可靠。**正确做法**：网络层 = 主机到主机，传输层 = 进程到进程（端口）。
3. **现象**：面试时只背层名不说协议。**原因**：层名抽象。**正确做法**：每层至少说 1～2 个协议（如传输层 TCP/UDP，应用层 HTTP/DNS），体现理解。

------

| 面试官提问 | 回答思路 |
|------------|----------|
| OSI 七层 / TCP/IP 四层分别是什么？ | 七层：物数网传会表应；四层：网络接口、网际(IP)、传输(TCP/UDP)、应用。实际开发多用四层，能说出每层典型协议即可。 |

## 第2章：TCP 与 UDP

本章解决**传输层如何提供可靠传输（TCP）与不可靠传输（UDP）**，以及连接建立与释放的流程，是“从输入 URL 到页面展示”中建立连接的基础。与第 1 章传输层职责、第 3 章 HTTP 基于 TCP 紧密相关。

------

### 1️⃣ TCP 与 UDP 对比

| 维度 | TCP | UDP |
|------|-----|-----|
| 连接 | 面向连接（先握手） | 无连接 |
| 可靠性 | 可靠（确认、重传、有序） | 不可靠、不保证顺序 |
| 数据单位 | 面向字节流 | 面向报文（整包收发） |
| 首部 | 20 字节起，选项可更长 | 固定 8 字节 |
| 典型场景 | HTTP、HTTPS、RPC、文件传输 | 实时音视频、DNS 查询、广播 |

### 2️⃣ 三次握手（建立连接）

- **目的**：双方确认**收发能力**、**协商初始序列号（ISN）**，防止旧连接报文被误认为是新连接。
- **流程**：  
  1）客户端 → 服务端：**SYN**（seq=x）；  
  2）服务端 → 客户端：**SYN+ACK**（seq=y, ack=x+1）；  
  3）客户端 → 服务端：**ACK**（ack=y+1）。  
  此后连接建立，可传数据。

```
  Client                    Server
     |  -------- SYN --------->  |
     |  <----- SYN+ACK ---------  |
     |  -------- ACK ---------->  |
     |      (连接已建立)          |
```

> 💬 **面试怎么说**：为什么不能两次？两次只能证明“客户端→服务端、服务端→客户端”单向可达，无法确认客户端收能力；且无法可靠协商序列号，旧 SYN 重传可能误建连接。三次可双向往返各一次确认 + 一次确认对方收能力。

### 3️⃣ 四次挥手（释放连接）

- **为什么是四次**：TCP 全双工；一方发 **FIN** 只表示“我不再发数据”，对方还可继续发；所以需要双方各发一次 FIN + 对方各回一次 ACK。
- **流程**：主动方 FIN → 被动方 ACK；被动方再发 FIN → 主动方 ACK。主动方收到被动方 FIN 后进入 **TIME_WAIT**（通常 2MSL）。
- **TIME_WAIT 作用**：① 保证对方收到最后一个 ACK（若丢失对方会重传 FIN，己方可再回 ACK）；② 让本连接产生的旧报文在网络中消散，避免被新连接误收。
- **CLOSE_WAIT**：被动关闭方收到 FIN 后回 ACK 并进入 CLOSE_WAIT；若应用未及时 close，会堆积 CLOSE_WAIT，常表示程序未正确关闭连接。

### 4️⃣ 流量控制与拥塞控制

- **流量控制**：**滑动窗口**，接收方通过窗口大小告诉发送方“还能收多少”，防止发送方发得过快把接收方缓冲区撑满。
- **拥塞控制**：防止网络过载；典型机制：**慢启动**（指数增窗）、**拥塞避免**（线性增）、**快重传**（收到 3 个重复 ACK 立即重传）、**快恢复**（重传后不从头慢启动）。面试能说出“慢启动、拥塞避免、快重传、快恢复”及目的即可。

------

### 📖 本章小结

| 知识点 | 面试关键词 | 实际应用 |
|--------|------------|----------|
| TCP vs UDP | 面向连接、可靠、字节流 vs 无连接、报文 | 选型：要可靠用 TCP，要低延迟/广播用 UDP |
| 三次握手 | SYN/SYN+ACK/ACK、序列号、为什么不能两次 | 画图口述、防旧连接错认 |
| 四次挥手 | FIN/ACK 双向、TIME_WAIT、CLOSE_WAIT | 排查连接泄漏、TIME_WAIT 过多 |
| 流量/拥塞控制 | 滑动窗口、慢启动、快重传 | 理解“为什么不是一发就满速” |

------

### ⚠️ 常见坑与注意点

1. **现象**：记混 SYN 和 ACK 谁先谁后。**原因**：握手是 SYN → SYN+ACK → ACK。**正确做法**：记“先 SYN 请求，再 SYN+ACK 同意并回 SYN，最后 ACK 确认”；挥手同理，先 FIN 的一方要等对方 FIN 再 ACK。
2. **现象**：服务器大量 CLOSE_WAIT。**原因**：被动关闭方在等应用 close；应用未调 close 或异常未释放。**正确做法**：代码里确保 socket/连接在 finally 或 try-with-resources 中关闭；排查未关闭的连接。
3. **现象**：认为 TIME_WAIT 越短越好。**原因**：过短可能导致旧报文干扰新连接。**正确做法**：一般保持 2MSL；若高并发短连接导致 TIME_WAIT 过多，可考虑复用连接（如 HTTP Keep-Alive）、或 SO_REUSEADDR 等（需理解语义）。

------

### 🎯 面试常见追问

| 面试官提问 | 回答思路 |
|------------|----------|
| TCP 和 UDP 区别？ | 连接性、可靠性、有序性、首部、适用场景；能举 HTTP 用 TCP、DNS 查询用 UDP 等例子。 |
| 为什么三次握手不能两次？ | 两次无法确认“客户端收能力”、无法可靠协商序列号，且旧 SYN 重传可能误建连接；三次可双向往返确认。 |
| 为什么挥手要四次？ | 全双工，双方都要发 FIN 表示不发数据，并各回 ACK；故至少两次“FIN+ACK”往返。 |

## 第3章：HTTP 与 HTTPS

本章解决**应用层最常用的协议**：HTTP 的语义、状态码、版本演进，以及 HTTPS 如何保证安全，与第 2 章 TCP（HTTP 基于 TCP）、第 6 章安全（HTTPS、攻击防护）、第 7 章“从 URL 到页面”直接相关。

------

### 1️⃣ HTTP 基础

- **HTTP**：无状态、**请求/响应**模型；基于 TCP，默认 80 端口。**为什么无状态**：简化服务端设计，状态靠 Cookie/Session 等自行维护。
- **方法**：GET（幂等、可缓存）、POST（提交数据）、PUT、DELETE、PATCH 等；**GET 与 POST**：语义上 GET 取资源、POST 提交；GET 参数在 URL、有长度与缓存；POST 在体里、更适敏感与大数据。
- **状态码**：2xx 成功、3xx 重定向、4xx 客户端错误、5xx 服务端错误；常见 301/302（重定向）、304（未修改）、400/401/403/404、500/502/503。
- **常见头**：Content-Type、Cache-Control、Cookie/Set-Cookie、Connection（Keep-Alive）、Host、Accept 等。

### 2️⃣ HTTP/1.0、1.1 与 2

| 版本 | 要点 | 问题 |
|------|------|------|
| 1.0 | 每请求一个连接 | 连接开销大 |
| 1.1 | 持久连接（Keep-Alive）、管道化（有限）、分块传输 | 同一连接上**队头阻塞**（前一个请求慢会堵住后面） |
| 2 | **多路复用**（单连接多流）、头部压缩（HPACK）、服务端推送、二进制帧 | 解决队头阻塞；仍受 TCP 队头阻塞影响 |

### 3️⃣ HTTPS 与 TLS 握手

- **HTTPS**：HTTP over **TLS/SSL**；提供机密性（加密）、完整性（防篡改）、身份验证（证书），解决 HTTP 明文与中间人风险。
- **握手流程（简述）**：① 客户端 ClientHello（支持的版本、套件等）；② 服务端 ServerHello + **证书**（公钥在内）；③ 客户端校验证书、生成预主密钥用服务端公钥加密发送；④ 双方根据预主密钥推导会话密钥；⑤ 后续用对称密钥加密通信。
- **为什么最后用对称加密**：非对称计算贵，只用于交换密钥；数据用对称加密更快。

> 💬 **面试怎么说**：HTTPS 在 TCP 之上加了一层 TLS，握手阶段用证书与公钥交换/协商出对称密钥，之后用对称加密传数据，既安全又高效。

------

### 📖 本章小结

| 知识点 | 面试关键词 | 实际应用 |
|--------|------------|----------|
| HTTP 语义 | 无状态、GET/POST、状态码、常见头 | 前后端联调、RESTful、缓存与 Cookie |
| 1.0/1.1/2 | Keep-Alive、队头阻塞、多路复用 | 理解为什么需要 HTTP/2、长连接 |
| HTTPS | TLS、证书、对称/非对称、握手 | 解释为什么 HTTPS 能防窃听与篡改 |

------

### ⚠️ 常见坑与注意点

1. **现象**：把 GET 和 POST 混用（如用 GET 传密码）。**原因**：GET 参数在 URL 里，易被日志、Referer 带走。**正确做法**：敏感数据、大体积用 POST；幂等取资源用 GET。
2. **现象**：认为“HTTPS 全程非对称加密”。**原因**：非对称只用于密钥交换。**正确做法**：握手后数据用对称加密，密钥通过非对称或 DH 等协商。
3. **现象**：忽略 301 与 302 区别。**原因**：301 永久重定向、浏览器可能缓存；302 临时。**正确做法**：换域名/永久迁用 301；临时跳转用 302（或 307/308 保方法）。

------

### 🎯 面试常见追问

| 面试官提问 | 回答思路 |
|------------|----------|
| GET 和 POST 区别？ | 语义：GET 取资源、POST 提交；参数位置、长度、缓存、安全性；幂等与副作用。 |
| 常见状态码？ | 2xx 成功、3xx 重定向(301/302)、304 未修改、4xx 客户端(400/401/403/404)、5xx 服务端(500/502/503)；各举一例含义。 |
| HTTP 和 HTTPS 区别？ | HTTPS 在 HTTP 与 TCP 间加 TLS，加密、完整性、身份验证；握手时证书与密钥交换，之后对称加密。若追问握手细节见本章 3️⃣。 |

## 第4章：DNS 与 CDN

本章解决**域名如何解析为 IP**以及**静态资源如何就近加速**，是“输入 URL 到页面展示”中第一步（DNS）和静态资源加载（CDN）的基础。与第 3 章 HTTP、第 7 章完整流程关联。

------

### 1️⃣ DNS 概述

- **DNS（Domain Name System）**：**域名到 IP 的映射**；为什么需要：人记域名、机器用 IP，DNS 做转换与分层管理。
- **分层结构**：根域 → 顶级域（如 .com、.cn）→ 权威域（如 example.com）；**递归查询**（客户端只问本地 DNS，由本地 DNS 代劳层层查）与**迭代查询**（本地 DNS 逐级向根、顶级、权威发问）。
- **记录类型**：**A**（IPv4）、**AAAA**（IPv6）、**CNAME**（别名）、**MX**（邮件）、**NS**（子域授权）等。

### 2️⃣ DNS 解析流程

1. 查**本地缓存**（浏览器、OS、 hosts）；  
2. 若无，请求**本地 DNS**（如运营商/公司 DNS）；  
3. 本地 DNS 若无缓存，则从**根 DNS** 起迭代或递归：根 → 顶级域 → 权威域，得到 A/CNAME 等记录并缓存；  
4. 结果返回给客户端。

> 💬 **面试怎么说**：先本地缓存，再问本地 DNS；本地 DNS 从根开始一层层查到权威，拿到 A 记录（或 CNAME 再查一次），返回 IP，并做缓存。

### 3️⃣ DNS 劫持与污染

- **劫持**：篡改 DNS 应答，把域名解析到错误 IP（如钓鱼、广告）。**防护**：用可信 DNS（如 DoH/DoT）、校验证书（HTTPS）。
- **污染**：在递归链路上注入错误结果，常见于某些网络环境。**防护**：加密 DNS 查询、使用可靠递归 DNS。

### 4️⃣ CDN

- **CDN（Content Delivery Network）**：把静态资源**分发到边缘节点**，用户**就近**访问，减轻源站压力、**降低延迟**。
- **流程**：用户请求资源 → DNS 解析到**边缘节点 IP**（通过 CNAME 到 CDN 域名，再由 CDN 调度返回就近节点）→ 边缘有则直接返回，无则**回源**拉取并缓存。
- **关键点**：回源策略、缓存策略（过期时间、 purge）、**命中率**；适用静态资源（图片、CSS、JS、视频等）。

------

### 📖 本章小结

| 知识点 | 面试关键词 | 实际应用 |
|--------|------------|----------|
| DNS 解析 | 本地缓存、本地 DNS、根/顶级/权威、A/CNAME | 排查“能 ping 通但打不开”、解析慢 |
| DNS 安全 | 劫持、污染、DoH/DoT、HTTPS | 选可信 DNS、全站 HTTPS |
| CDN | 边缘节点、回源、缓存、命中率 | 静态资源加速、大促扛流量 |

------

### ⚠️ 常见坑与注意点

1. **现象**：改完 DNS 记录后仍解析到旧 IP。**原因**：各级缓存（浏览器、OS、本地 DNS、权威 TTL）未过期。**正确做法**：理解 TTL；调试时可 flush 本地缓存或换 DNS，上线前可先缩短 TTL。
2. **现象**：把 CDN 当万能加速。**原因**：CDN 适合静态、可缓存内容；动态接口若走 CDN 可能拿到旧数据或需要回源。**正确做法**：静态资源走 CDN；动态 API 视情况用 CDN 或直连源站。
3. **现象**：CDN 命中率低。**原因**：缓存时间过短、URL 带随机参数、 purge 过频。**正确做法**：合理设 Cache-Control、避免 URL 带无意义变化参数、按需 purge。

------

| 面试官提问 | 回答思路 |
|------------|----------|
| DNS 解析流程？ | 本地缓存 → 本地 DNS → 根/顶级/权威 递归或迭代，拿到 A/CNAME 等并缓存；能画简图或口述步骤。 |
| DNS 劫持和污染？ | 劫持：篡改应答到错误 IP；污染：链路上注入错误。防护：可信 DNS、DoH/DoT、HTTPS。 |
| CDN 原理与适用场景？ | 边缘节点、就近访问、回源与缓存；适用静态资源、降低延迟与源站压力；动态接口需视情况。若追问命中率见本章常见坑。 |

## 第5章：网络层与路由

本章解决**网络层如何寻址与转发**：IP 地址、子网、路由表、NAT，与第 1 章网络层职责、第 2 章 TCP（IP 之上）衔接；理解 IP 与 MAC、ARP、NAT 有助于排查“能上 QQ 不能打开网页”类问题。

------

### 1️⃣ IP 概述

- **IP（Internet Protocol）**：**无连接、尽力而为**；不保证可靠、不保证顺序，只负责把分组从源传到目的（可能丢、乱序）。
- **IPv4 地址**：32 位，点分十进制；**子网与掩码**：掩码区分网络位与主机位，用于划分子网与判断是否同网段。
- **公有/私有地址**：私有（如 10.x、172.16～31.x、192.168.x）仅在内网使用；出网需 **NAT** 转成公网 IP。
- **分片与重组**：超过 MTU 的 IP 包会被分片，到达后再重组；**TTL**：每经一跳减 1，为 0 丢弃，防环路。

### 2️⃣ IP 与 MAC、ARP

- **IP 地址**：逻辑地址，用于**端到端**路由（网络层）；**MAC 地址**：物理地址，用于**同一链路内**的下一跳（数据链路层）。
- **ARP（Address Resolution Protocol）**：同一网段内，已知 IP 求 MAC；广播 ARP 请求，目标主机回 ARP 应答；结果可缓存。

### 3️⃣ 路由与 NAT

- **路由**：路由器根据**路由表**查“下一跳”，逐跳把分组转发到目的网络；路由选择算法有**距离向量**（如 RIP）、**链路状态**（如 OSPF）。
- **NAT**：内网私有 IP + 端口 与 公网 IP + 端口 的映射；内网主机访问外网时，出口设备做**源地址/端口转换**，回包时再根据映射表转回内网。**为什么需要**：节省公网 IP、隐藏内网结构。

------

| 知识点 | 面试关键词 | 实际应用 |
|--------|------------|----------|
| IP 与子网 | 无连接、尽力而为、掩码、公有/私有 | 划网段、排查“同网段不通” |
| IP vs MAC、ARP | 逻辑地址 vs 物理地址、ARP 求 MAC | 理解二层与三层分工 |
| 路由与 NAT | 路由表、下一跳、NAT 映射 | 理解“家里多设备共用一个公网 IP” |

------

### ⚠️ 常见坑与注意点

1. **现象**：配错子网掩码导致“同网段”不通。**原因**：掩码决定哪几位是网络号，错则不同网段或广播域错。**正确做法**：同一子网内主机掩码一致、网络号相同。
2. **现象**：认为“有 IP 就能上网”。**原因**：还需默认网关（路由）、DNS；内网 IP 出网需 NAT。**正确做法**：排查时看 IP、网关、DNS、NAT 是否都正常。
3. **现象**：把 NAT 和代理混为一谈。**原因**：NAT 做地址转换，通常对应用透明；代理（如 HTTP 代理）是应用层转发，可做缓存、鉴权。**正确做法**：NAT 是网络/传输层，代理是应用层；都能“让内网访问外网”，机制不同。

### 4️⃣ IPv6

**IPv4 地址耗尽**：32 位地址约 42 亿，实际可用更少，已接近耗尽。**IPv6**：128 位地址，号称"地球上每粒沙子都能分配一个 IP"。

| 特性 | IPv4 | IPv6 |
|------|------|------|
| **地址长度** | 32 位 | 128 位 |
| **地址表示** | 点分十进制（192.168.1.1） | 冒号十六进制（2001:0db8:85a3::8a2e:0370:7334） |
| **地址数量** | 约 42 亿 | 约 3.4×10³⁸ |
| **头部简化** | 可变长度选项 | 固定 40 字节，无选项 |
| **安全性** | 可选（IPSec） | 内置 IPSec |
| **广播** | 支持广播 | 不支持广播，用组播 |

**IPv6 地址类型**：
- **单播**：一对一通信
- **组播**：一对多，替代 IPv4 广播
- **任播**：一对最近，路由到最近的节点

**过渡技术**：
- **双栈**：同时支持 IPv4 和 IPv6
- **隧道**：IPv6 over IPv4（如 6to4、Teredo）
- **翻译**：NAT64，IPv6 与 IPv4 互转

### 5️⃣ ICMP 协议

**ICMP（Internet Control Message Protocol）**：用于网络诊断和错误报告。

| 类型 | 说明 |
|------|------|
| **Echo Request/Reply** | ping 命令使用 |
| **Destination Unreachable** | 目的不可达（如端口关闭、路由不可达） |
| **Time Exceeded** | TTL 耗尽（traceroute 使用） |
| **Redirect** | 路由重定向 |

**ping 原理**：发送 ICMP Echo Request，对方回 Echo Reply，测试连通性和 RTT。

**traceroute 原理**：TTL 从 1 开始递增，每跳路由器返回 ICMP Time Exceeded，显示路径。

### 6️⃣ 路由协议

**静态路由**：手动配置，适合小型网络。

**动态路由**：自动学习路由，适合大型网络。

| 协议 | 类型 | 特点 |
|------|------|------|
| **RIP** | 距离向量 | 基于跳数，最大 15 跳，收敛慢 |
| **OSPF** | 链路状态 | 基于 Dijkstra，支持区域划分，收敛快 |
| **BGP** | 路径向量 | 基于 AS 路径，用于互联网核心路由 |

**OSPF（Open Shortest Path First）**：
- 链路状态协议，每个路由器维护全网拓扑
- 使用 Dijkstra 算法计算最短路径
- 支持区域划分，减少路由表规模
- 触发更新，收敛快

**BGP（Border Gateway Protocol）**：
- 路径向量协议，用于 AS（自治系统）之间
- 基于 AS 路径和策略，不单纯基于距离
- 是互联网的核心路由协议

### 7️⃣ NAT 类型

**NAT（Network Address Translation）**：内网 IP 与公网 IP 的映射。

| NAT 类型 | 说明 |
|----------|------|
| **静态 NAT** | 一对一固定映射，内网 IP 固定对应公网 IP |
| **动态 NAT** | 从公网地址池动态分配，有数量限制 |
| **NAPT（PAT）** | 端口复用，多个内网 IP 共享一个公网 IP，最常用 |
| **锥形 NAT** | 内网主机可主动连接外网，外网也可主动连接（需端口映射） |

**NAPT（Network Address Port Translation）**：
```
内网 IP:端口 → 公网 IP:端口
192.168.1.10:1234 → 1.2.3.4:10001
192.168.1.11:5678 → 1.2.3.4:10002
```

**NAT 穿透问题**：
- P2P 应用需穿透 NAT
- 方案：STUN、TURN、ICE
- UDP 穿透相对容易，TCP 较难

## 第6章：网络安全与常见攻击

本章解决**常见安全威胁与防护思路**，与第 3 章 HTTPS、第 4 章 DNS 安全、第 7 章后端开发（防护落地）关联；校招常问 XSS/CSRF 区别、HTTPS 作用、加密基本概念。

------

### 1️⃣ 对称加密与非对称加密

| 类型 | 密钥 | 典型用途 |
|------|------|----------|
| **对称加密** | 同一密钥加解密（如 AES） | 大数据加密、HTTPS 中会话内容 |
| **非对称加密** | 公钥加密私钥解密、私钥签名公钥验签 | 密钥交换、身份认证、数字签名 |

HTTPS 中：用非对称（或 DH）协商/交换对称密钥，再用对称密钥加密数据，兼顾安全与性能。

### 2️⃣ 常见攻击与防护

| 攻击 | 含义 | 防护思路 |
|------|------|----------|
| **XSS**（跨站脚本） | 注入恶意脚本到页面，在用户浏览器执行 | 输出转义、CSP、HttpOnly Cookie、避免 innerHTML 插不可信数据 |
| **CSRF**（跨站请求伪造） | 利用用户已登录状态，在恶意站点发起伪造请求 | CSRF Token、SameSite Cookie、校验 Referer/Origin |
| **DDoS** | 大量请求/流量打满带宽或服务 | 限流、黑名单、CDN/清洗、扩容 |
| **中间人** | 窃听或篡改传输内容 | HTTPS（加密+证书校验）、证书钉扎（可选） |

> 💡 **重点**：XSS 是“在你这站执行了恶意脚本”；CSRF 是“在别的站点了你的站、借用户身份发请求”。一个偏注入、一个偏伪造请求。

### 3️⃣ HTTPS 防窃听与篡改

- **防窃听**：对称加密信道，密钥仅双方知晓。  
- **防篡改**：完整性校验（如 MAC、AEAD）；若被改则校验失败。  
- **身份验证**：证书链与公钥校验证书，防止冒充服务端（防中间人）。

### 4️⃣ SQL 注入

**SQL 注入**：通过输入恶意 SQL 语句，篡改数据库查询。

**示例**：
```sql
-- 正常查询
SELECT * FROM users WHERE username = 'admin'

-- 注入攻击（输入 ' OR '1'='1）
SELECT * FROM users WHERE username = '' OR '1'='1'
-- 返回所有用户记录
```

**防护**：
- **参数化查询**：使用预编译语句，参数与 SQL 分离
- **输入验证**：过滤特殊字符、限制输入格式
- **最小权限**：数据库账号权限最小化
- **ORM 框架**：使用 MyBatis、Hibernate 等，自动防注入

```java
// 参数化查询（安全）
String sql = "SELECT * FROM users WHERE username = ?";
PreparedStatement stmt = conn.prepareStatement(sql);
stmt.setString(1, username);
ResultSet rs = stmt.executeQuery();
```

### 5️⃣ 点击劫持

**点击劫持**：通过透明 iframe 覆盖页面，诱导用户点击不可见元素。

**原理**：
```
攻击者页面：
  <iframe src="http://bank.com/transfer" style="opacity:0"></iframe>
  <button style="position:absolute">领取红包</button>
  
用户点击"领取红包" → 实际点击了 iframe 中的转账按钮
```

**防护**：
- **X-Frame-Options**：禁止页面被 iframe 嵌入
  ```
  X-Frame-Options: DENY
  X-Frame-Options: SAMEORIGIN
  ```
- **Content-Security-Policy**：设置 frame-ancestors
  ```
  Content-Security-Policy: frame-ancestors 'none';
  ```
- **JavaScript 防御**：检测是否在 iframe 中（window.top !== window.self）

### 6️⃣ HTTPS 证书链

**证书链**：从终端证书到根证书的信任链。

```
用户浏览器
  ↓ 验证
终端证书（example.com）
  ↓ 由中间证书签发
中间证书（Intermediate CA）
  ↓ 由根证书签发
根证书（Root CA，内置在浏览器/OS）
```

**证书验证流程**：
1. **有效期**：检查证书是否在有效期内
2. **域名匹配**：证书域名与访问域名一致
3. **吊销检查**：CRL（证书吊销列表）或 OCSP（在线证书状态协议）
4. **签名验证**：用上级证书公钥验证签名，直到根证书
5. **信任锚**：根证书是否在信任存储中

**证书类型**：
- **DV（Domain Validation）**：仅验证域名所有权，适合个人博客
- **OV（Organization Validation）**：验证企业身份，适合企业网站
- **EV（Extended Validation）**：严格验证，浏览器显示绿色公司名

### 7️⃣ 常见加密算法

**对称加密**：

| 算法 | 密钥长度 | 特点 |
|------|---------|------|
| **AES** | 128/192/256 | 最常用，安全高效 |
| **DES** | 56 | 已不安全 |
| **3DES** | 168 | 已不安全 |
| **ChaCha20** | 256 | 移动设备友好 |

**非对称加密**：

| 算法 | 密钥长度 | 特点 |
|------|---------|------|
| **RSA** | 2048/4096 | 最常用，用于密钥交换和签名 |
| **ECC** | 256/384 | 更短密钥，同等安全性，计算快 |
| **DSA** | 1024/2048 | 仅用于签名，不用于加密 |

**哈希算法**：

| 算法 | 输出长度 | 特点 |
|------|---------|------|
| **MD5** | 128 位 | 已不安全，仅用于校验 |
| **SHA-1** | 160 位 | 已不安全 |
| **SHA-256** | 256 位 | 安全，常用 |
| **SHA-3** | 可变 | 最新标准 |

**密钥交换算法**：
- **RSA**：用公钥加密预主密钥
- **DH（Diffie-Hellman）**：双方协商出共享密钥，不直接传输
- **ECDH**：基于椭圆曲线的 DH，更高效

**HTTPS 中的加密组合（Cipher Suite）**：
```
TLS_ECDHE_RSA_WITH_AES_128_GCM_SHA256
  ↑        ↑    ↑        ↑            ↑
  协议    密钥交换  身份验证  对称加密    完整性
```

| 知识点 | 面试关键词 | 实际应用 |
|--------|------------|----------|
| 对称/非对称 | 单钥/双钥、加密与签名、HTTPS 中的分工 | 解释 HTTPS 为什么最后用对称加密 |
| XSS/CSRF | 注入 vs 伪造请求、转义 vs Token | 前后端防护、Cookie 属性 |
| DDoS/中间人 | 流量攻击、HTTPS 与证书 | 选型与运维防护 |

| 点击劫持 | iframe、X-Frame-Options、CSP | 页面安全 |

| 加密算法 | AES、RSA、ECC、SHA-256、DH/ECDH | 算法选型 |

------

### ⚠️ 常见坑与注意点

1. **现象**：只防 XSS 不防 CSRF 或反过来。**原因**：两者机制不同。**正确做法**：XSS 重点在“不执行不可信脚本”（转义、CSP）；CSRF 重点在“请求必须带本站凭证/Token”（CSRF Token、SameSite）。
2. **现象**：HTTPS 但证书不校验（如自签或忽略校验证书）。**原因**：中间人可替换证书。**正确做法**：生产环境必须校验证书链；自签仅限内网/测试并明确风险。
3. **现象**：把“加密”和“签名”混用。**原因**：加密为了保密、签名为了防篡改与身份。**正确做法**：加密用对方公钥（或对称密钥）；签名用己方私钥、对方用公钥验签。

| 点击劫持是什么？怎么防？ | 用透明 iframe 覆盖页面，诱导用户点击；防护 X-Frame-Options、CSP、JS 检测。 |
| HTTPS 证书链是什么？ | 终端证书→中间证书→根证书，浏览器验证到根证书信任锚。验证流程：有效期、域名、吊销、签名、信任锚。 |
| 对称加密和非对称加密区别？ | 对称：同一密钥加解密，快，适合大数据；非对称：公钥加密私钥解密，慢，用于密钥交换和签名。 |
| 常见加密算法有哪些？ | 对称：AES、ChaCha20；非对称：RSA、ECC；哈希：SHA-256；密钥交换：DH、ECDH。 |

------

### ⚠️ 常见坑与注意点

1. **现象**：只防 XSS 不防 CSRF 或反过来。**原因**：两者机制不同。**正确做法**：XSS 重点在"不执行不可信脚本"（转义、CSP）；CSRF 重点在"请求必须带本站凭证/Token"（CSRF Token、SameSite）。
2. **现象**：HTTPS 但证书不校验（如自签或忽略校验证书）。**原因**：中间人可替换证书。**正确做法**：生产环境必须校验证书链；自签仅限内网/测试并明确风险。
3. **现象**：把"加密"和"签名"混用。**原因**：加密为了保密、签名为了防篡改与身份。**正确做法**：加密用对方公钥（或对称密钥）；签名用己方私钥、对方用公钥验签。
4. **现象**：SQL 拼接导致注入。**原因**：直接拼接用户输入到 SQL 语句。**正确做法**：使用参数化查询、预编译语句、ORM 框架。
5. **现象**：页面可被 iframe 嵌入。**原因**：未设置 X-Frame-Options 或 CSP。**正确做法**：设置 X-Frame-Options: DENY 或 SAMEORIGIN，或 CSP frame-ancestors。
6. **现象**：使用 MD5 或 SHA-1 存储密码。**原因**：这些哈希算法已被破解，可彩虹表攻击。**正确做法**：使用 SHA-256、bcrypt、scrypt 或 PBKDF2，并加盐。
7. **现象**：证书链不完整。**原因**：只配置了终端证书，缺少中间证书。**正确做法**：配置完整证书链，包括终端证书和中间证书。

## 第7章：与后端开发的关系

本章把前几章**落到后端开发**：Socket 与 NIO/Netty、HTTP 服务与反向代理、以及经典题“从输入 URL 到页面展示”，把 DNS、TCP、HTTP、渲染串成一条线。依赖第 2 章 TCP、第 3 章 HTTP、第 4 章 DNS。

------

### 1️⃣ Socket 与 NIO、Netty

- **Socket 编程**：TCP 典型流程为服务端 **bind → listen → accept**，客户端 **connect**；UDP 无连接，**sendto/recvfrom**。**为什么需要**：OS 提供的网络 API，应用通过 socket 收发数据。
- **NIO**：非阻塞 I/O、多路复用（select/poll/epoll），单线程可监听多连接；适合高并发 I/O。
- **Netty**：对 NIO/多路复用的封装，提供 **EventLoop、Channel、Pipeline** 等；Linux 下用 epoll，简化异步网络编程（可与《操作系统》第 7 章 epoll 对应）。

### 2️⃣ HTTP 服务与反向代理

- **HTTP 服务**：Tomcat、Spring MVC 等监听端口，解析 HTTP 请求、路由到 Controller、返回响应。
- **反向代理（如 Nginx）**：客户端只连 Nginx，Nginx 再转发到后端多台机器；做**负载均衡**、静态资源、SSL 终结、限流等。

### 3️⃣ 从输入 URL 到页面展示（面试经典）

**流程要点**（口述顺序）：  
1）**DNS 解析**：域名 → IP（见第 4 章）；  
2）**建立 TCP 连接**：三次握手（见第 2 章）；  
3）**发 HTTP 请求**：GET/POST 等，带 Cookie、头等；  
4）**服务端处理**：后端/静态服务器返回响应；  
5）**浏览器解析**：HTML 解析、CSS、JS 执行、渲染树、布局、绘制；若遇静态资源（CSS/JS/图）可能再走 CDN、缓存。  
6）**关闭连接或 Keep-Alive**：四次挥手或复用连接。

> 💬 **面试怎么说**：先 DNS 拿 IP，再 TCP 三次握手，然后发 HTTP 请求，服务端返回后浏览器解析 HTML/CSS/JS 并渲染；静态资源可能走 CDN、有缓存；最后关连接或保持长连接。

### 4️⃣ 长连接与短连接

- **短连接**：每次请求新建 TCP 连接，响应后关闭；简单但握手/挥手开销大。
- **长连接（Keep-Alive）**：一次 TCP 连接上发多个 HTTP 请求/响应，减少连接数；HTTP/1.1 默认倾向 Keep-Alive。

------

### 📖 本章小结

| 知识点 | 面试关键词 | 实际应用 |
|--------|------------|----------|
| Socket 与 NIO/Netty | bind/listen/accept、多路复用、EventLoop | 写/调 RPC、网关、IM |
| HTTP 与反向代理 | Tomcat、Nginx、负载均衡 | 部署、高可用、静态与动态分离 |
| URL 到页面 | DNS→TCP→HTTP→解析渲染 | 面试标准答法、排查白屏/慢 |
| 长/短连接 | Keep-Alive、连接复用 | 性能调优、连接池 |

------

### ⚠️ 常见坑与注意点

1. **现象**：答“从 URL 到页面”时漏 DNS 或漏 TCP。**原因**：只记“发 HTTP”。**正确做法**：按顺序：DNS → TCP 握手 → HTTP 请求/响应 → 解析渲染；可顺带提 HTTPS 握手、CDN、缓存。
2. **现象**：高并发时每请求新建连接。**原因**：未用连接池或 Keep-Alive。**正确做法**：HTTP 客户端用连接池；服务端合理配置 Keep-Alive 与超时。
3. **现象**：把“反向代理”和“正向代理”混用。**原因**：正向代理代表客户端（如翻墙），反向代理代表服务端（如 Nginx 挡在前面）。**正确做法**：代理谁就代表谁；Nginx 做的是反向代理。

------

| 面试官提问 | 回答思路 |
|------------|----------|
| 从输入 URL 到页面展示发生了什么？ | DNS 解析 → TCP 三次握手 → 发 HTTP 请求 → 服务端响应 → 浏览器解析 HTML/CSS/JS、渲染；可提 CDN、缓存、Keep-Alive。若追问某一步见对应章。 |
| 长连接和短连接？ | 短连接：每次请求新建并关闭 TCP；长连接：复用同一 TCP 发多个 HTTP。HTTP/1.1 Keep-Alive 即长连接，减少握手开销。 |
| 反向代理做什么？ | 代表服务端接收请求并转发到后端；负载均衡、静态资源、SSL、限流等。与正向代理（代表客户端）区分。 |

## 第8章：WebSocket 与实时通信

本章介绍 **WebSocket** 协议，解决 HTTP 半双工、实时性差的局限，适用于 IM、股票行情、协同编辑等场景。与第 3 章 HTTP、第 7 章长连接紧密相关。

------

### 1️⃣ 为什么需要 WebSocket

| 特性 | HTTP/1.1 | WebSocket |
|------|----------|-----------|
| **通信模式** | 请求-响应，半双工 | 全双工，双向推送 |
| **实时性** | 轮询延迟高 | 服务器主动推送 |
| **连接** | 短连接或长连接复用 | 单一长连接 |
| **头部开销** | 每次请求带完整头部 | 首次握手后，数据帧头部极小 |

**传统方案的问题**：
- **短轮询**：客户端定时请求，实时性差、服务器压力大
- **长轮询**：保持连接直到有数据，但服务器需维护挂起连接
- **SSE（Server-Sent Events）**：服务器单向推送，客户端不能主动发

### 2️⃣ WebSocket 握手

WebSocket 基于 HTTP 协议升级：

```
客户端请求（带升级头）：
GET /chat HTTP/1.1
Host: server.example.com
Upgrade: websocket
Connection: Upgrade
Sec-WebSocket-Key: dGhlIHNhbXBsZSBub25jZQ==
Sec-WebSocket-Version: 13

服务端响应（同意升级）：
HTTP/1.1 101 Switching Protocols
Upgrade: websocket
Connection: Upgrade
Sec-WebSocket-Accept: s3pPLMBiTxaQ9kYGzzhZRbK+xOo=
```

- **101 状态码**：协议切换成功
- **Sec-WebSocket-Key**：客户端随机生成，服务端用特定算法计算 Accept 值，防止误连接

### 3️⃣ WebSocket 数据帧

握手后，数据以 **帧（Frame）** 形式传输：

| 字段 | 说明 |
|------|------|
| **FIN** | 是否为最后一帧 |
| **Opcode** | 帧类型：文本(1)、二进制(2)、关闭(8)、ping(9)、pong(10) |
| **MASK** | 客户端发的帧必须掩码，服务端可选 |
| **Payload Length** | 数据长度，支持扩展 |
| **Masking Key** | 掩码密钥（客户端必须） |
| **Payload Data** | 实际数据 |

**特点**：
- 头部最小 2 字节，远小于 HTTP
- 支持文本和二进制数据
- 内置心跳机制（ping/pong）

### 4️⃣ WebSocket vs HTTP/2 SSE

| 特性 | WebSocket | HTTP/2 SSE |
|------|-----------|------------|
| **双向通信** | ✅ 全双工 | ❌ 服务器单向推送 |
| **二进制支持** | ✅ 原生支持 | ❌ 仅文本 |
| **浏览器兼容** | 现代浏览器 | 更好 |
| **穿透代理/防火墙** | 可能困难 | 更容易（基于HTTP） |
| **适用场景** | 双向实时（IM、游戏） | 服务器推送（新闻、股票） |

### 5️⃣ 应用场景与实现

| 场景 | 说明 |
|------|------|
| **即时通讯** | 微信网页版、钉钉、Slack |
| **股票行情** | 实时价格推送 |
| **协同编辑** | 腾讯文档、Google Docs |
| **在线游戏** | 实时位置、状态同步 |
| **IoT** | 设备实时监控 |

**Java 实现**：Spring WebSocket + STOMP，或 Netty 手写 WebSocket 服务器。

------

### 📖 本章小结

| 知识点 | 面试关键词 | 实际应用 |
|--------|------------|----------|
| WebSocket 优势 | 全双工、低延迟、小头部 | 实时推送场景 |
| 握手过程 | HTTP 升级、101 状态码、Key/Accept | 协议切换原理 |
| 数据帧 | FIN、Opcode、MASK、Payload | 帧结构理解 |
| vs SSE | 双向 vs 单向、二进制 vs 文本 | 技术选型 |

------

### ⚠️ 常见坑与注意点

1. **现象**：WebSocket 连接失败。**原因**：代理/防火墙不支持 Upgrade。**正确做法**：配置 Nginx 支持 WebSocket；使用 wss（WebSocket Secure）走 443 端口。
2. **现象**：心跳机制缺失导致连接断开。**原因**：NAT 超时或防火墙切断空闲连接。**正确做法**：实现 ping/pong 心跳，或应用层心跳。
3. **现象**：混淆 WebSocket 与 Socket。**原因**：Socket 是传输层 API，WebSocket 是应用层协议。**正确做法**：WebSocket 基于 TCP，通过 HTTP 握手升级。

------

### 🎯 面试常见追问

| 面试官提问 | 回答思路 |
|------------|----------|
| WebSocket 和 HTTP 区别？ | WebSocket 全双工、服务器可主动推送、头部小；HTTP 请求-响应模式。 |
| WebSocket 握手过程？ | HTTP 请求带 Upgrade: websocket，服务端返回 101 Switching Protocols。 |
| WebSocket 适用场景？ | IM、股票行情、协同编辑、在线游戏等需要双向实时通信的场景。 |
| WebSocket 和 SSE 怎么选？ | 需要双向通信用 WebSocket；只需服务器推送用 SSE，兼容更好。 |

## 第9章：TCP 粘包与拆包

本章解决 **TCP 字节流特性带来的粘包/拆包问题**，是后端开发中网络编程的实际痛点。与第 2 章 TCP 面向字节流、第 7 章 Netty 直接相关。

------

### 1️⃣ 什么是粘包与拆包

**TCP 是面向字节流的协议**：
- 发送方：应用层发 100 字节，TCP 可能分 2 次发（50+50）
- 接收方：两次发的数据可能在同一个 TCP 包中到达

**粘包（Packet Sticking）**：
- 发送方发了两个包 A 和 B，接收方一次读到 A+B

**拆包（Packet Splitting）**：
- 发送方发了一个大包，接收方分两次读到半个包

```
发送方：       [数据A]       [数据B]
              ↓              ↓
TCP层：   [数据A部分][数据A部分+数据B][数据B部分]
              ↓              ↓           ↓
接收方：   收到A前半    收到A后半+B    收到B后半
```

### 2️⃣ 为什么 TCP 会粘包/拆包

| 原因 | 说明 |
|------|------|
| **TCP 字节流** | 无消息边界，只保证字节顺序 |
| **Nagle 算法** | 小数据包合并发送，减少网络负载 |
| **MSS/MTU 限制** | 数据包超过最大段大小会被拆分 |
| **接收缓冲区** | 应用层读取速度跟不上，数据积压 |

> 💡 UDP 无粘包问题：UDP 是面向报文的，每个包独立，有明确边界。

### 3️⃣ 解决方案

| 方案 | 原理 | 实现 |
|------|------|------|
| **固定长度** | 每个消息固定 N 字节 | 简单，但浪费空间 |
| **分隔符** | 消息间用特殊字符分隔（如 \n） | 需转义，不能含分隔符 |
| **长度字段** | 消息头存长度，先读头再读体 | 最常用，灵活高效 |
| **特殊协议** | 如 HTTP、Protobuf、JSON 自带边界 | 上层协议处理 |

**长度字段方案（推荐）**：
```
消息格式：[长度(4字节)][数据(N字节)]

发送：
  数据 = "Hello"
  长度 = 5
  发送：0x00 0x00 0x00 0x05 'H' 'e' 'l' 'l' 'o'

接收：
  先读 4 字节，得到长度 5
  再读 5 字节，得到完整数据
```

### 4️⃣ Netty 中的解决方案

Netty 提供开箱即用的编解码器：

| 编解码器 | 说明 |
|----------|------|
| **FixedLengthFrameDecoder** | 固定长度解码 |
| **LineBasedFrameDecoder** | 以换行符分隔 |
| **DelimiterBasedFrameDecoder** | 自定义分隔符 |
| **LengthFieldBasedFrameDecoder** | 长度字段解码（最常用） |

```java
// LengthFieldBasedFrameDecoder 参数：
// maxFrameLength, lengthFieldOffset, lengthFieldLength, 
// lengthAdjustment, initialBytesToStrip

pipeline.addLast(new LengthFieldBasedFrameDecoder(
    65535,   // 最大帧长度
    0,       // 长度字段偏移
    4,       // 长度字段长度（4字节int）
    0,       // 长度调整
    4        // 跳过长度字段本身
));
pipeline.addLast(new StringDecoder());
pipeline.addLast(new MyHandler());
```

------

### 📖 本章小结

| 知识点 | 面试关键词 | 实际应用 |
|--------|------------|----------|
| 粘包/拆包原因 | TCP 字节流、Nagle、MSS | 理解问题本质 |
| 解决方案 | 固定长度、分隔符、长度字段 | 长度字段最常用 |
| Netty 编解码 | FixedLength、LengthFieldBased | 实际开发直接用 |

------

### ⚠️ 常见坑与注意点

1. **现象**：用 TCP 传消息但没处理粘包。**原因**：误以为 TCP 按应用层发送边界分包。**正确做法**：必须设计消息边界协议（长度字段或分隔符）。
2. **现象**：长度字段大小端问题。**原因**：网络字节序是大端，主机可能是小端。**正确做法**：统一使用网络字节序（大端）或明确转换。
3. **现象**：最大帧长度不设限制。**原因**：攻击者可能发送超大长度导致内存溢出。**正确做法**：设置合理的 maxFrameLength。

------

### 🎯 面试常见追问

| 面试官提问 | 回答思路 |
|------------|----------|
| 什么是 TCP 粘包？ | TCP 面向字节流，无消息边界；发送方多次发送的数据可能被合并，接收方一次读到多个包。 |
| 粘包/拆包怎么解决？ | 固定长度、分隔符、长度字段；长度字段最常用：先读4字节长度，再读对应长度数据。 |
| Netty 怎么处理粘包？ | 提供编解码器：FixedLengthFrameDecoder、LengthFieldBasedFrameDecoder 等。 |
| UDP 有粘包问题吗？ | UDP 面向报文，有消息边界，无粘包问题，但不保证可靠传输。 |

## 第10章：HTTP/3 与 QUIC

本章介绍 **HTTP/3 和底层 QUIC 协议**，理解下一代 Web 协议如何解决 TCP 队头阻塞。与第 3 章 HTTP/2、第 2 章 TCP 直接相关。

------

### 1️⃣ HTTP/2 的局限

HTTP/2 解决了 HTTP/1.1 的队头阻塞（Head-of-Line Blocking）：
- **多路复用**：单连接多流，流之间互不影响
- **但仍有 TCP 队头阻塞**：TCP 层丢包会阻塞所有流

```
HTTP/2 多路复用：
Stream 1: 请求A → 响应A
Stream 2: 请求B → 响应B
Stream 3: 请求C → 响应C
         ↑ 共用一个 TCP 连接
         
问题：TCP 层丢包，所有 Stream 都等待重传
```

### 2️⃣ QUIC 协议

**QUIC（Quick UDP Internet Connections）**：
- 基于 **UDP** 实现可靠传输
- 在应用层实现类似 TCP 的拥塞控制、重传、流量控制
- **连接迁移**：连接标识不依赖 IP+端口，网络切换不断连

**QUIC 特点**：

| 特性 | TCP+TLS | QUIC |
|------|---------|------|
| **握手延迟** | 2-3 RTT | 0-1 RTT（首次1 RTT，后续0 RTT） |
| **队头阻塞** | TCP 层阻塞所有流 | 流之间独立，单流丢包不影响他流 |
| **连接迁移** | 不支持（IP变则断） | 支持（连接ID标识） |
| **安全性** | TLS 叠加在 TCP 上 | 内置 TLS 1.3 |

### 3️⃣ HTTP/3 = HTTP over QUIC

- HTTP/3 是 HTTP 语义在 QUIC 上的实现
- 保留了 HTTP/2 的头部压缩（QPACK 替代 HPACK）、服务器推送等特性
- 解决了 TCP 队头阻塞问题

**握手过程**：
```
首次连接：
  客户端 → 服务端：Client Hello（含 TLS 信息）
  客户端 ← 服务端：Server Hello + 证书 + 应用数据
  RTT = 1

后续连接（0-RTT）：
  客户端 → 服务端：Client Hello + 应用数据（基于之前会话密钥）
  RTT = 0
```

### 4️⃣ QUIC 的队头阻塞解决

QUIC 在应用层实现多流，每个流独立传输：

```
QUIC 连接：
  Stream 1: 丢包 → 仅 Stream 1 等待重传
  Stream 2: 正常传输
  Stream 3: 正常传输
  
UDP 层只管收发，可靠性和流控由 QUIC 处理
```

### 5️⃣ 部署与兼容性

| 方面 | 现状 |
|------|------|
| **浏览器** | Chrome、Firefox、Safari 已支持 |
| **服务端** | Nginx、Cloudflare、Fastly 支持 |
| **网络设备** | 部分防火墙/代理可能阻断 UDP |
| ** fallback ** | 不支持时降级到 HTTP/2 或 HTTP/1.1 |

------

### 📖 本章小结

| 知识点 | 面试关键词 | 实际应用 |
|--------|------------|----------|
| HTTP/2 局限 | TCP 队头阻塞 | 理解为什么需要 HTTP/3 |
| QUIC 特点 | 基于 UDP、0-RTT、连接迁移、内置 TLS | 下一代传输协议 |
| HTTP/3 优势 | 无队头阻塞、更快握手 | 视频、实时应用 |

------

### ⚠️ 常见坑与注意点

1. **现象**：认为 HTTP/3 只是 HTTP/2 的小改进。**原因**：HTTP/3 底层完全换为 QUIC，基于 UDP。**正确做法**：理解 QUIC 在应用层实现可靠性，解决 TCP 队头阻塞。
2. **现象**：QUIC 一定比 TCP 快。**原因**：QUIC 首次连接仍需 1 RTT，且 UDP 可能被防火墙限制。**正确做法**：QUIC 优势在连接迁移、0-RTT 恢复、多流独立。
3. **现象**：忽略 UDP 被防火墙阻断问题。**原因**：部分企业防火墙限制 UDP。**正确做法**：生产环境需支持降级到 TCP 的 fallback 机制。

------

### 🎯 面试常见追问

| 面试官提问 | 回答思路 |
|------------|----------|
| HTTP/3 相比 HTTP/2 改进？ | HTTP/3 基于 QUIC，QUIC 基于 UDP，解决 TCP 队头阻塞；支持 0-RTT、连接迁移。 |
| QUIC 是什么？为什么用 UDP？ | QUIC 在 UDP 上实现可靠传输，避免 TCP 队头阻塞，实现流级多路复用。 |
| 0-RTT 是什么？安全吗？ | 首次连接后缓存会话密钥，后续连接直接发数据；有重放攻击风险，需额外防护。 |
| 连接迁移是什么？ | TCP 用 IP+端口标识连接，IP 变则断；QUIC 用连接 ID，网络切换（如 WiFi 切 4G）保持连接。 |

### 4️⃣ 实战：分析三次握手

**过滤**：`tcp.flags.syn == 1 or tcp.flags.ack == 1`

**观察**：
```
1. 客户端 → 服务端：SYN Seq=0
2. 服务端 → 客户端：SYN, ACK Seq=0 Ack=1
3. 客户端 → 服务端：ACK Seq=1 Ack=1
```

**关键字段**：
- **Sequence Number**：序列号
- **Acknowledgment Number**：确认号
- **Window Size**：接收窗口大小
- **Options**：MSS、时间戳、SACK 等

**观察**：
```
请求：

  User-Agent: Mozilla/5.0

响应：
  HTTP/1.1 200 OK
  Content-Type: text/html
  Content-Length: 1234
```

**Follow TCP Stream**：右键 → Follow → TCP Stream，查看完整会话。

| **TLS 握手慢** | 看 Client Hello 到 Finished 时间 |
| **首字节慢（TTFB）** | 看 HTTP 请求到第一个响应字节时间 |
| **内容下载慢** | 看数据传输速率和重传情况 |

**Statistics → TCP Stream Graphs**：
- Time-Sequence Graph：看序列号随时间变化

## 第12章：网络排查方法论与常用命令

本章介绍 **网络问题的系统化排查方法**和**常用命令**，帮助快速定位网络故障。与前面各章协议理论结合，形成完整的网络问题解决能力。

------

### 1️⃣ 网络排查方法论

**分层排查法**（从下到上）：

```
1. 物理层
   - 网线/光纤是否连接
   - 网卡灯是否亮
   - 网卡状态：ip link / ifconfig

2. 数据链路层
   - MAC 地址是否正确
   - ARP 表是否正常
   - 命令：arp -a, ip neigh

3. 网络层
   - IP 地址、子网掩码、网关
   - 路由表是否正确
   - 命令：ip addr, ip route, ping

4. 传输层
   - 端口是否监听
   - 防火墙规则
   - 命令：netstat, ss, telnet, nc

5. 应用层
   - 服务是否运行
   - 应用日志
   - 命令：curl, wget, 应用日志
```

**排查流程**：
1. **确认问题范围**：单机/全网/特定服务
2. **从下到上逐层检查**：物理→数据链路→网络→传输→应用
3. **定位瓶颈**：带宽、延迟、丢包、CPU、内存
4. **验证修复**：修复后逐层验证

## 第13章：负载均衡算法与配置

本章介绍 **负载均衡的原理、算法和配置**，理解如何通过多台服务器提供高可用服务。与第 7 章反向代理、第 5 章网络层紧密相关。

------

### 1️⃣ 负载均衡概述

**负载均衡**：将请求分发到多台服务器，提高并发处理能力和可用性。

**架构层次**：
```
四层负载均衡（L4）：基于 IP + 端口
  - 工作在传输层（TCP/UDP）
  - 性能高，功能简单
  - 代表：LVS、HAProxy、Nginx stream

七层负载均衡（L7）：基于应用层协议（HTTP）
  - 工作在应用层（HTTP）
  - 功能丰富，性能略低
  - 代表：Nginx、HAProxy、云负载均衡（ALB）
```

**负载均衡类型**：
| 类型 | 说明 | 适用场景 |
|------|------|----------|
| **硬件负载均衡** | F5、A10 等专用设备 | 大型企业、金融 |
| **软件负载均衡** | Nginx、HAProxy、LVS | 中小企业、互联网 |
| **云负载均衡** | AWS ALB、阿里 SLB | 云原生应用 |
| **DNS 负载均衡** | 域名解析到多个 IP | 简单场景，无健康检查 |

### 2️⃣ 负载均衡算法

#### 随机算法

| 算法 | 原理 | 优点 | 缺点 |
|------|------|------|------|
| **随机（Random）** | 随机选择服务器 | 简单，分布均匀 | 无状态，可能不均匀 |
| **加权随机（Weighted Random）** | 按权重随机选择 | 可根据服务器性能分配 | 配置权重需要经验 |

**适用场景**：服务器性能相近，请求无状态

#### 轮询算法

| 算法 | 原理 | 优点 | 缺点 |
|------|------|------|------|
| **轮询（Round Robin）** | 按顺序轮流选择 | 简单，绝对均匀 | 不考虑服务器负载 |
| **加权轮询（Weighted Round Robin）** | 按权重轮流选择 | 可根据性能分配 | 配置复杂 |

**示例**：
```
服务器权重：A=3, B=2, C=1
请求顺序：A A A B B C A A A B B C ...
```

**适用场景**：服务器性能相近，请求无状态

#### 最少连接算法

| 算法 | 原理 | 优点 | 缺点 |
|------|------|------|------|
| **最少连接（Least Connections）** | 选择当前连接数最少的服务器 | 动态适应负载 | 需要维护连接数 |
| **加权最少连接** | 按权重和连接数选择 | 考虑性能和负载 | 配置复杂 |

**适用场景**：请求处理时间差异大，长连接场景

#### 哈希算法

| 算法 | 原理 | 优点 | 缺点 |
|------|------|------|------|
| **源地址哈希（IP Hash）** | 对客户端 IP 哈希 | 同一客户端到同一服务器 | 服务器故障影响部分用户 |
| **URL 哈希** | 对请求 URL 哈希 | 同一 URL 到同一服务器 | 不适合动态内容 |

**适用场景**：需要会话保持（Session Sticky）

#### 一致性哈希

**原理**：将服务器和数据都映射到哈希环上，顺时针查找最近的服务器。

**优点**：
- 服务器增删时，只影响部分数据
- 数据分布均匀

**适用场景**：分布式缓存、分布式存储

### 3️⃣ Nginx 负载均衡配置

#### 基础配置

```nginx
upstream backend {
    server 192.168.1.10:8080;
    server 192.168.1.11:8080;
    server 192.168.1.12:8080;
}

server {
    listen 80;
    server_name example.com;
    
    location / {
        proxy_pass http://backend;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
    }
}
```

#### 算法配置

```nginx
# 轮询（默认）
upstream backend {
    server 192.168.1.10:8080;
    server 192.168.1.11:8080;
}

# 加权轮询
upstream backend {
    server 192.168.1.10:8080 weight=3;
    server 192.168.1.11:8080 weight=2;
    server 192.168.1.12:8080 weight=1;
}

# 最少连接
upstream backend {
    least_conn;
    server 192.168.1.10:8080;
    server 192.168.1.11:8080;
}

# IP 哈希
upstream backend {
    ip_hash;
    server 192.168.1.10:8080;
    server 192.168.1.11:8080;
}
```

#### 健康检查

```nginx
upstream backend {
    server 192.168.1.10:8080 max_fails=3 fail_timeout=30s;
    server 192.168.1.11:8080 max_fails=3 fail_timeout=30s;
    server 192.168.1.12:8080 max_fails=3 fail_timeout=30s;
}

# 参数说明：
# max_fails：最大失败次数
# fail_timeout：失败后暂停时间
```

#### 备份服务器

```nginx
upstream backend {
    server 192.168.1.10:8080;
    server 192.168.1.11:8080;
    server 192.168.1.12:8080 backup;  # 备用服务器
}
```

### 4️⃣ 会话保持

**问题**：负载均衡后，用户请求可能分发到不同服务器，导致 Session 丢失。

**解决方案**：

| 方案 | 原理 | 优点 | 缺点 |
|------|------|------|------|
| **IP Hash** | 同一 IP 到同一服务器 | 简单 | 服务器故障影响部分用户 |
| **Cookie Sticky** | 在 Cookie 中标记服务器 | 精确 | Cookie 被禁用失效 |
| **Session 共享** | Redis/Memcached 存储 Session | 灵活，无状态 | 依赖外部存储 |
| **Session 复制** | 服务器间复制 Session | 不依赖外部存储 | 性能开销大 |

**Nginx Cookie Sticky**：
```nginx
upstream backend {
    server 192.168.1.10:8080;
    server 192.168.1.11:8080;
    sticky cookie srv_id expires=1h domain=.example.com path=/;
}
```

### 5️⃣ 高可用架构

**主备模式**：
```
客户端 → VIP（虚拟IP）
          ↓
    主负载均衡（Active）
          ↓
    后端服务器集群
    
    备负载均衡（Standby，监听 VIP）
```

**双主模式**：
```
客户端 → VIP1 → 主负载均衡1 → 后端服务器集群
客户端 → VIP2 → 主负载均衡2 → 后端服务器集群
```

**Keepalived 配置**：
```bash
# 主节点
vrrp_instance VI_1 {
    state MASTER
    interface eth0
    virtual_router_id 51
    priority 100
    advert_int 1
    authentication {
        auth_type PASS
        auth_pass 1234
    }
    virtual_ipaddress {
        192.168.1.100
    }
}

# 备节点
vrrp_instance VI_1 {
    state BACKUP
    interface eth0
    virtual_router_id 51
    priority 90
    advert_int 1
    authentication {
        auth_type PASS
        auth_pass 1234
    }
    virtual_ipaddress {
        192.168.1.100
    }
}
```

### 6️⃣ 性能优化

| 优化项 | 说明 | 配置 |
|--------|------|------|
| **连接数** | 增加并发连接数 | `worker_connections` |
| **缓冲区** | 调整缓冲区大小 | `client_body_buffer_size` |
| **超时** | 设置合理超时 | `proxy_read_timeout` |
| **压缩** | 启用 gzip 压缩 | `gzip on` |
| **缓存** | 缓存静态资源 | `proxy_cache` |

**示例**：
```nginx
worker_processes auto;
worker_connections 10240;

http {
    gzip on;
    gzip_types text/plain text/css application/json;
    
    proxy_cache_path /tmp/cache levels=1:2 keys_zone=my_cache:10m max_size=1g;
    
    upstream backend {
        server 192.168.1.10:8080;
        server 192.168.1.11:8080;
        keepalive 32;
    }
    
    server {
        listen 80;
        
        location / {
            proxy_pass http://backend;
            proxy_cache my_cache;
            proxy_cache_valid 200 60m;
        }
    }
}
```

### ⚠️ 常见坑与注意点

1. **现象**：负载均衡不均匀。**原因**：算法选择不当或权重配置不合理。**正确做法**：根据服务器性能和请求特点选择算法，合理配置权重。
2. **现象**：Session 丢失。**原因**：请求分发到不同服务器，Session 不共享。**正确做法**：使用 Session 共享（Redis）或会话保持（IP Hash）。
3. **现象**：单点故障。**原因**：负载均衡器本身是单点。**正确做法**：使用 Keepalived 实现主备高可用。
4. **现象**：健康检查失效。**原因**：检查路径错误或超时设置不合理。**正确做法**：配置正确的健康检查路径和超时时间。
5. **现象**：性能瓶颈。**原因**：负载均衡器成为瓶颈。**正确做法**：水平扩展负载均衡器，或使用四层负载均衡提高性能。

### 🎯 面试常见追问

| 面试官提问 | 回答思路 |
|------------|----------|
| 负载均衡算法有哪些？ | 轮询、随机、最少连接、哈希、一致性哈希；根据场景选择：无状态用轮询，长连接用最少连接，会话保持用哈希。 |
| L4 和 L7 负载均衡区别？ | L4 基于 IP+端口，性能高；L7 基于 HTTP 内容，功能丰富（路由、限流、SSL 终结）。 |
| Session 怎么保持？ | IP Hash、Cookie Sticky、Session 共享（Redis）、Session 复制；推荐 Session 共享。 |
| 负载均衡高可用怎么实现？ | 使用 Keepalived 实现主备或双主，VIP 漂移，故障自动切换。 |
| Nginx 负载均衡怎么配置？ | upstream 定义后端服务器，proxy_pass 转发，可配置权重、健康检查、算法（least_conn、ip_hash）。 |
