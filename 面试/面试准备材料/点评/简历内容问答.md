# 基于Redis的缓存层设计与实现分析

基于 Redis 设计缓存层，采用 Cache-Aside 策略，结合逻辑过期与主动更新机制，减少数据库访问压力，提升接口查询效率和系统整体响应速度

## 核心实现思路

本项目基于Redis实现了高效的缓存层，采用Cache-Aside策略，并结合逻辑过期与主动更新机制，有效减少数据库访问压力，提升接口查询效率。

### 1. 核心设计组件

#### CacheClient 工具类

CacheClient.java 是整个缓存层的核心，提供了多种缓存策略实现：

- **基本缓存操作**：提供标准的缓存读写功能
- **防缓存穿透**：通过缓存空值避免恶意请求直接访问数据库
- **逻辑过期机制**：用于热点数据的缓存管理，避免缓存击穿
- **互斥锁机制**：用于缓存重建时的并发控制

### 2. Cache-Aside 策略实现

Cache-Aside 策略的核心思想是：先操作数据库，再更新缓存。在本项目中的实现如下：

#### 更新操作（写策略）

在更新操作中，严格遵循Cache-Aside策略，先更新数据库，再删除缓存：

```java
@Override
@Transactional
public Result update(Shop shop) {
    Long id = shop.getId();
    if (id == null) {
        return Result.fail("店铺id不能为空");
    }
    // 1.更新数据库
    updateById(shop);
    // 2.删除缓存
    stringRedisTemplate.delete(CACHE_SHOP_KEY + id);
    return Result.ok();
}
```

>### 常见的缓存更新策略
>
>#### 1. Cache-Aside（旁路缓存）策略
>
>- **更新流程**：先更新数据库，再删除缓存
>- **读取流程**：先读缓存，缓存未命中则读数据库，然后更新缓存
>- **优点**：实现简单，广泛使用
>- **缺点**：在高并发下可能出现缓存不一致
>
>#### 2. Read/Write Through（读写穿透）策略
>
>- **更新流程**：只操作缓存，缓存负责同步更新数据库
>- **读取流程**：只操作缓存，缓存未命中时自动加载数据
>- **优点**：应用代码简洁，无需关心数据库操作
>- **缺点**：实现复杂度高，需要自定义缓存适配器
>
>#### 3. Write Behind（写回）策略
>
>- **更新流程**：只更新缓存，异步批量更新数据库
>- **读取流程**：与Read Through类似
>- **优点**：性能高，适合写多读少场景
>- **缺点**：数据有丢失风险，实现复杂
>
>#### 4. 主动更新策略
>
>- **定时更新**：定期从数据库读取数据更新缓存
>- **事件驱动更新**：通过消息队列或事件机制触发缓存更新
>
>### 项目中采用的缓存更新策略
>
>项目主要采用了**Cache-Aside（旁路缓存）策略**，具体体现在`ShopServiceImpl`类的`update`方法中
>
>### 采用Cache-Aside策略的原因
>
>1. **实现简单**：代码逻辑清晰，易于理解和维护
>2. **可靠性高**：相比直接更新缓存，删除缓存的方式减少了数据不一致的风险
>3. **适用场景匹配**：
>
>     - 项目中的缓存主要用于读多写少的数据（如店铺信息）
>     - 缓存删除后，下次读取时会自动重建，保证数据最终一致性
>4. **避免并发问题**：如果采用先更新数据库再更新缓存的方式，在高并发场景下可能导致更新覆盖问题
>5. **符合Redis最佳实践**：Redis官方推荐在大多数场景下使用Cache-Aside策略

### 3. 缓存问题解决方案

#### 缓存穿透解决方案

在`queryWithPassThrough`方法中实现，通过缓存空值避免数据库压力：

```java
public <R,ID> R queryWithPassThrough(
        String keyPrefix, ID id, Class<R> type, Function<ID, R> dbFallback, Long time, TimeUnit unit){
    String key = keyPrefix + id;
    // 1.从redis查询缓存
    String json = stringRedisTemplate.opsForValue().get(key);	// （null表示未引用任何对象，空值表示长度为0的对象）
    // 2.判断是否存在
    if (StrUtil.isNotBlank(json)) {		// json不为null且不是空字符串
        // 3.缓存存在，直接返回
        return JSONUtil.toBean(json, type);
    }
    // 判断命中的是否是空值。
    if (json != null) {		// json不为null，只剩下空值情况（主动写入的占位空值，之前处理过直接返回null）
        // 返回空结果
        return null;
    }

    // 4.不存在，查询数据库，json既不是null也不是空值
    R r = dbFallback.apply(id);
    // 5.数据库也不存在，缓存空值
    if (r == null) {
        // 将空值写入redis，设置较短过期时间
        // 将空值null转为特殊占位符（如空字符串""）
        stringRedisTemplate.opsForValue().set(key, "", CACHE_NULL_TTL, TimeUnit.MINUTES);
        return null;
    }
    // 6.数据库存在，写入redis
    this.set(key, r, time, unit);
    return r;
}

public void set(String key, Object value, Long time, TimeUnit unit) {
    stringRedisTemplate.opsForValue().set(key, JSONUtil.toJsonStr(value), time, unit);
}
```

>### 通用缓存查询方法(queryWithPassThrough)详解
>
>#### 执行流程
>
>- **接收参数**：`keyPrefix`: Redis键前缀，用于构建缓存键名；`id`: 查询对象的唯一标识；`type`: 返回对象的类型（Class）；`dbFallback`: 当缓存未命中时，从数据库查询的函数式接口；`time`和`unit`: 缓存过期时间及单位
>- **处理操作**：
> 1. 构建Redis缓存键：`keyPrefix + id`
> 2. 从Redis中查询缓存数据
> 3. 判断缓存是否存在且非空：（null表示未引用任何对象，空值表示长度为0的对象）
>    - 如果存在且非空，直接反序列化为指定类型并返回
> 4. 判断缓存是否存在但为空字符串（表示数据库中也不存在）：
>    - 如果是，则直接返回null
> 5. 缓存未命中，调用`dbFallback`从数据库查询
> 6. 判断数据库查询结果：
>    - 如果数据库中也不存在，将空字符串写入Redis作为空值标记，设置较短过期时间`CACHE_NULL_TTL`分钟，防止缓存穿透
>    - 如果数据库中存在，调用`set`方法将数据序列化并写入Redis，设置指定的过期时间
> 7. 返回查询结果
>
>#### 设计亮点
>
>1. **缓存穿透防护**：通过缓存空值来防止恶意请求直接穿透到数据库
>2. **泛型设计**：使用泛型使方法适用于任意类型的数据查询
>3. **函数式编程**：使用`Function<ID, R>`参数，灵活传入数据库查询逻辑
>4. **空值处理**：区分了缓存未命中和缓存命中但值为空两种情况
>5. **可配置过期时间**：不同数据类型可设置不同的缓存过期时间
>
>#### 缓存设计
>
>- **Redis数据结构**：数据类型：String；键格式：`{keyPrefix}:{id}`；值格式：JSON字符串
> - 过期策略：
>   - 正常数据：用户指定的过期时间
>   - 空值数据：较短的固定过期时间(`CACHE_NULL_TTL`分钟)
>
>#### 业务接口路径建议
>
>基于此通用方法实现的具体业务接口，路径设计应遵循RESTful风格：
>
>- **用户信息查询**：`GET /api/users/{id}`
>- **商品信息查询**：`GET /api/products/{id}`
>- **文章详情查询**：`GET /api/articles/{id}`

>#### 优化建议

>1. **缓存雪崩防护**：考虑在设置过期时间时添加随机偏移量，避免大量缓存同时过期
>2. **缓存击穿处理**：对于热点key，可考虑添加分布式锁防止缓存击穿
>3. **缓存更新策略**：结合业务需求，考虑使用Cache-Aside、Read-Through等缓存更新策略

#### 逻辑过期解决缓存击穿

针对热点数据，实现了逻辑过期机制，当缓存过期时，异步重建缓存，避免并发请求直接访问数据库：

```java
public <R, ID> R queryWithLogicalExpire(
        String keyPrefix, ID id, Class<R> type, Function<ID, R> dbFallback, Long time, TimeUnit unit) {
    String key = keyPrefix + id;
    // 1.查询缓存
    String json = stringRedisTemplate.opsForValue().get(key);
    // 2.判断是否存在，假设热点数据总是存在于缓存，只是可能过期，不处理缓存为空的情况
    if (StrUtil.isBlank(json)) {
        return null;
    }
    // 3.反序列化数据
    RedisData redisData = JSONUtil.toBean(json, RedisData.class);
    R r = JSONUtil.toBean((JSONObject) redisData.getData(), type);
    LocalDateTime expireTime = redisData.getExpireTime();
    // 4.判断是否过期
    if(expireTime.isAfter(LocalDateTime.now())) {
        // 4.1.未过期，直接返回
        return r;
    }
    // 4.2.已过期，需要缓存重建
    // 5.获取互斥锁
    String lockKey = LOCK_SHOP_KEY + id;
    boolean isLock = tryLock(lockKey);
    // 6.判断是否获取锁成功
    if (isLock){
        // 6.1.成功，异步重建缓存
        CACHE_REBUILD_EXECUTOR.submit(() -> {	// submit()有返回值，execute()没有返回值
            try {
                // 查询数据库
                R newR = dbFallback.apply(id);
                // 重建缓存
                this.setWithLogicalExpire(key, newR, time, unit);
            } catch (Exception e) {
                throw new RuntimeException(e);
            } finally {
                // 释放锁
                unlock(lockKey);
            }
        });
    }
    // 6.2.无论获取锁是否成功，都返回过期数据，先返回后更新
    return r;
}
```

>### 逻辑过期缓存查询方法(queryWithLogicalExpire)详解
>
>#### 执行流程
>
>- **接收参数**：`keyPrefix`: Redis键前缀。`id`: 查询对象的唯一标识。`type`: 返回对象的类型。`dbFallback`: 数据库查询函数。`time`和`unit`: 逻辑过期时间及单位
> - **处理操作**：
>  1. 构建Redis缓存键：`keyPrefix + id`
>  2. 从Redis中查询缓存数据
>  3. 判断缓存是否不存在：
>     - 如果不存在，直接返回null
> 4. 反序列化缓存数据：
>     - 先将JSON转为自定义的`RedisData`对象
>     - 从`RedisData`中提取业务数据并转为指定类型
>     - 获取逻辑过期时间
>  5. 判断数据是否过期：
>     - 如果未过期，直接返回数据
>     - 如果已过期，需要进行缓存重建
>  6. 获取互斥锁(`tryLock`)：
>     - 锁键格式：`LOCK_SHOP_KEY + id`
>  7. 判断是否获取锁成功：
>     - 如果成功，提交异步任务进行缓存重建：
>       * 查询数据库获取最新数据
>       * 使用`setWithLogicalExpire`方法重建缓存
>       * 任务结束后释放锁
>     - 如果失败，不进行任何操作
>  8. 返回过期数据（无论是否重建缓存）
> 
> #### 核心设计亮点
> 
> 1. **逻辑过期机制**：不依赖Redis的TTL机制，而是将过期时间存储在缓存数据中
> 2. **异步缓存重建**：缓存过期时不会阻塞用户请求，通过线程池异步更新
>3. **互斥锁防止缓存击穿**：使用锁确保同一时间只有一个线程进行缓存重建
>4. **过期数据容忍**：返回过期数据而非等待重建完成，保证高可用性
>5. **泛型设计**：适用于任意类型的数据查询
>
>#### 缓存数据结构
>
>- **Redis存储格式**：数据类型：String。键格式：`{keyPrefix}:{id}`。值格式：自定义`RedisData`对象的JSON序列化，包含：`data`: 实际业务数据（JSON对象）；`expireTime`: 逻辑过期时间（ISO日期格式）
> - 过期策略：Redis本身不设置过期时间，依赖应用层逻辑判断
>
>#### 工作原理详解
>
>1. **为什么使用逻辑过期？**
>   - 避免了传统TTL过期时的缓存雪崩问题
>   - 允许返回过期但仍然可用的数据，提高用户体验
>   - 可以在后台异步更新，不阻塞用户请求
> 
> 2. **缓存重建流程**：
>   - 检测到数据过期 → 尝试获取锁 → 成功则异步重建 → 释放锁
>  - 锁的目的是防止高并发下多个线程同时重建缓存
>  - 异步重建确保用户立即获得响应（即使是过期数据）
>
>#### 业务接口路径建议
> 
> - **店铺查询**：`GET /api/shops/{id}`
> - **商品详情**：`GET /api/products/{id}`
>- **分类信息**：`GET /api/categories/{id}`
>
> #### 关键依赖组件
> 
> - **线程池(`CACHE_REBUILD_EXECUTOR`)**: 用于异步执行缓存重建任务
>- **分布式锁**：`tryLock`和`unlock`方法实现分布式互斥
>- **RedisData类**：封装业务数据和逻辑过期时间的自定义类
>
>#### 适用场景
>
>1. **高并发读场景**：需要极高的响应速度和吞吐量
>2. **数据允许短暂不一致**：业务上能够接受短期的数据滞后
>3. **缓存击穿防护**：针对热点数据的访问保护
>4. **用户体验优先**：宁可返回稍旧的数据，也不让用户等待
>
>这种实现是一种平衡了性能、可用性和数据一致性的高级缓存策略，特别适合读多写少、对响应时间要求高的业务场景。

#### 互斥锁解决缓存击穿

提供了基于互斥锁的缓存重建机制，确保同一时间只有一个线程重建缓存：

```java
public <R, ID> R queryWithMutex(
        String keyPrefix, ID id, Class<R> type, Function<ID, R> dbFallback, Long time, TimeUnit unit) {
    String key = keyPrefix + id;
    // 查询缓存
    String shopJson = stringRedisTemplate.opsForValue().get(key);
    // 存在直接返回
    if (StrUtil.isNotBlank(shopJson)) {
        return JSONUtil.toBean(shopJson, type);
    }
    // 空值判断
    if (shopJson != null) {
        return null;
    }

    // 实现缓存重建
    // 对比上面，加锁力度更大，锁住整个缓存数据
    String lockKey = LOCK_SHOP_KEY + id;
    R r = null;
    try {
        boolean isLock = tryLock(lockKey);
        // 获取锁失败，重试
        if (!isLock) {
            Thread.sleep(50);
            return queryWithMutex(keyPrefix, id, type, dbFallback, time, unit);
        }
        // 获取锁成功，查询数据库
        r = dbFallback.apply(id);
        if (r == null) {
            stringRedisTemplate.opsForValue().set(key, "", CACHE_NULL_TTL, TimeUnit.MINUTES);
            return null;
        }
        // 写入缓存
        this.set(key, r, time, unit);
    } catch (InterruptedException e) {
        throw new RuntimeException(e);
    } finally {
        // 释放锁
        unlock(lockKey);
    }
    return r;
}
```

>### 互斥锁缓存查询方法(queryWithMutex)详解
>
>#### 执行流程
>
>- **接收参数**：`keyPrefix`: Redis键前缀。`id`: 店铺ID查询对象的唯一标识。`type`: 返回对象的类型。`dbFallback`: 数据库查询函数。`time`和`unit`: 缓存过期时间及单位
> - **处理操作**：
>  1. 构建Redis缓存键：`keyPrefix + id`
>  2. 从Redis中查询缓存数据
>  3. 判断缓存是否存在且非空：
>     - 如果存在且非空，直接反序列化并返回
> 4. 判断缓存是否存在但为空字符串（表示数据库中也不存在）：
>     - 如果是，直接返回null（防止缓存穿透）
>  5. 缓存未命中，开始缓存重建流程：
>     - 构建互斥锁键：`LOCK_SHOP_KEY + id`
>     - 尝试获取分布式锁(`tryLock`)
>     - 如果获取锁失败：
>       * 线程休眠50毫秒
>       * 递归调用自身进行重试（自旋等待）
>     - 如果获取锁成功：
>       * 调用`dbFallback`从数据库查询数据
>       * 如果数据库中不存在，缓存空字符串并设置较短过期时间`CACHE_NULL_TTL`
>       * 如果数据库中存在，将数据序列化并写入Redis，设置指定的过期时间
>  6. 在`finally`块中释放分布式锁（确保锁一定会被释放）
>  7. 返回查询结果
> 
> #### 核心设计亮点
> 
> 1. **互斥锁防缓存击穿**：使用分布式锁确保同一时间只有一个线程去数据库查询，保护数据库
> 2. **缓存穿透防护**：通过缓存空值来防止对不存在数据的恶意查询
>3. **递归重试机制**：获取锁失败时采用退避重试策略，避免线程一直阻塞
>4. **异常安全**：使用try-finally块确保锁一定会被释放，防止死锁
>5. **泛型设计**：适用于任意类型的数据查询，提高代码复用性
>
>#### 缓存数据结构
>
>- **Redis存储格式**：。数据类型：String。键格式：`{keyPrefix}:{id}`。值格式：正常数据：JSON序列化的业务对象；空值标记：空字符串`""`
> - 过期策略：正常数据：用户指定的过期时间。空值数据：较短的固定过期时间(`CACHE_NULL_TTL`分钟)

>#### 与其他缓存策略对比
>
>- **与穿透缓存(queryWithPassThrough)对比**：增加了互斥锁机制，更适合热点数据查询
>- **与逻辑过期(queryWithLogicalExpire)对比**：不返回过期数据，而是等待最新数据，但可能导致请求阻塞
>- **适用场景**：对数据实时性要求高、不允许返回过期数据的场景
>
>#### 性能影响与优化点
>
>1. **可能的性能瓶颈**：高并发下获取锁失败的线程会休眠并重试，可能导致部分请求延迟
>2. **优化建议**：
>  - 为重试添加最大次数限制，避免无限递归
>  - 考虑使用Redisson等成熟的分布式锁框架，提供更多高级特性
>  - 可以添加随机退避时间，避免线程同步重试导致的惊群效应
>  - 对热点数据考虑预热机制，减少缓存未命中的情况
>
>这种实现是一种经典的缓存击穿防护策略，通过互斥锁确保对同一资源的并发请求不会同时查询数据库，有效保护了数据库层，同时通过缓存空值防止了缓存穿透攻击。

### 4. 分布式锁实现

SimpleRedisLock.java 提供了分布式锁实现，用于缓存重建时的并发控制，使用了Lua脚本确保锁的释放是原子操作：

```java
@Override
public boolean tryLock(long timeoutSec) {
    // 获取线程标示
    String threadId = ID_PREFIX + Thread.currentThread().getId();
    // 获取锁
    Boolean success = stringRedisTemplate.opsForValue()
            .setIfAbsent(KEY_PREFIX + name, threadId, timeoutSec, TimeUnit.SECONDS);
    return Boolean.TRUE.equals(success);
}

@Override
public void unlock() {
    // 调用lua脚本
    stringRedisTemplate.execute(
            UNLOCK_SCRIPT,
            Collections.singletonList(KEY_PREFIX + name),
            ID_PREFIX + Thread.currentThread().getId());
}
```

>### 分布式锁实现(tryLock/unlock)详解
>
>#### 功能概述
>
>这是一个基于Redis实现的分布式锁组件，提供了获取锁和释放锁的核心方法，用于在分布式系统中实现资源的互斥访问。主要应用于缓存重建、库存扣减、秒杀等需要并发控制的场景。
>
>#### tryLock方法详解
>
>- **接收参数**：`timeoutSec`: 锁的过期时间（秒），防止死锁
> - **处理流程**：
> 1. 生成线程唯一标识：`ID_PREFIX + 当前线程ID`
>  2. 构建Redis锁键：`KEY_PREFIX + name`（name为锁名称）
>  3. 调用Redis的`setIfAbsent`命令尝试获取锁：
>     - 命令会原子性地检查键是否存在，不存在则设置值
>     - 同时设置过期时间，确保锁最终会被释放
>  4. 返回是否成功获取锁
> - **设计亮点**：
> - 使用线程标识确保锁的持有者唯一性
>  - 自动设置过期时间防止死锁
>  - 使用`Boolean.TRUE.equals()`安全地处理可能的null返回值
> 
>#### unlock方法详解
>
>- **处理流程**：
> 1. 执行预定义的Lua脚本`UNLOCK_SCRIPT`
>  2. 传递锁键和线程标识作为脚本参数
> - **设计亮点**：
> - 使用Lua脚本保证释放锁操作的原子性
>  - 只允许锁的持有者释放锁，防止误释放
>  - 避免了简单删除可能导致的锁误释放问题
> 
>#### Lua脚本工作原理推测
>
>基于分布式锁最佳实践，`UNLOCK_SCRIPT`应该包含以下逻辑：
>
>```lua
>-- 伪代码表示
>if redis.call('get', KEYS[1]) == ARGV[1] then
>   -- 只有持有锁的线程才能释放锁
>    return redis.call('del', KEYS[1])
> else
>   -- 不是锁的持有者，不能释放
>    return 0
> end
>```
>
>#### 技术设计要点
>
>1. **互斥性**：通过Redis的`setIfAbsent`原子命令确保同一时间只有一个线程可以获取锁
>2. **防死锁**：自动设置锁的过期时间，避免因程序异常导致锁无法释放
>3. **锁的唯一性**：使用线程标识确保锁的获取和释放对应
>4. **原子操作**：使用Lua脚本确保释放锁时的判断和删除是原子操作
>5. **高可用**：依赖Redis的高性能和可靠性，适合分布式环境
>
>#### 应用场景
>
>1. **缓存重建**：防止多个线程同时去数据库查询，保护数据库
>2. **库存扣减**：确保库存不会超卖
>3. **秒杀系统**：控制并发请求，防止系统过载
>4. **定时任务**：防止多个实例重复执行相同任务
>5. **分布式事务**：在特定场景下作为分布式事务的补充机制
>
>#### 与缓存系统的集成

>4. 获取锁失败时等待并重试或返回默认值

>#### 实现优化建议

>1. **可重入锁**：考虑增加计数器实现可重入锁机制，允许同一线程多次获取同一把锁
>2. **锁续约**：对于长时间运行的任务，可添加自动续约机制防止锁过期
>3. **公平锁**：基于Redis的List结构实现队列，提供公平锁功能

```java:com.hmdp.utils.RefreshTokenInterceptor
public class RefreshTokenInterceptor implements HandlerInterceptor {

    private StringRedisTemplate stringRedisTemplate;

    public RefreshTokenInterceptor(StringRedisTemplate stringRedisTemplate) {
        this.stringRedisTemplate = stringRedisTemplate;
    }

    @Override
    public boolean preHandle(HttpServletRequest request, HttpServletResponse response, Object handler) throws Exception {
        // 1.获取请求头中的token
        String token = request.getHeader("authorization");
        if (StrUtil.isBlank(token)) {
            return true;
        }
        // 2.基于TOKEN获取redis中的用户
        String key = LOGIN_USER_KEY + token;
        Map<Object, Object> userMap = stringRedisTemplate.opsForHash().entries(key);
        // 3.判断用户是否存在
        if (userMap.isEmpty()) {
            return true;
        }
        // 4.将查询到的hash数据转为UserDTO
        UserDTO userDTO = BeanUtil.fillBeanWithMap(userMap, new UserDTO(), false);
        // 5.存在，保存用户信息到 ThreadLocal
        UserHolder.saveUser(userDTO);
        // 6.刷新token有效期 - 滑动过期机制
        stringRedisTemplate.expire(key, LOGIN_USER_TTL, TimeUnit.MINUTES);
        // 7.放行
        return true;
    }

    @Override
    public void afterCompletion(HttpServletRequest request, HttpServletResponse response, Object handler, Exception ex) throws Exception {
        // 移除用户，防止内存泄漏
        UserHolder.removeUser();
    }
}
```

```java:com.hmdp.service.impl.UserServiceImpl
@Override
public Result login(LoginFormDTO loginForm, HttpSession session) {
    // 1.校验手机号
    String phone = loginForm.getPhone();
    if (RegexUtils.isPhoneInvalid(phone)) {
        return Result.fail("手机号格式错误！");
    }
    // 2.从redis获取验证码并校验
    String cacheCode = stringRedisTemplate.opsForValue().get(LOGIN_CODE_KEY + phone);
    String code = loginForm.getCode();
    if (cacheCode == null || !cacheCode.equals(code)) {
        return Result.fail("验证码错误");
    }

    // 3.根据手机号查询用户
    User user = query().eq("phone", phone).one();

    // 4.判断用户是否存在，不存在则创建
    if (user == null) {
        user = createUserWithPhone(phone);
    }

    // 5.保存用户信息到 redis中
    // 5.1.随机生成token，作为登录令牌
    String token = UUID.randomUUID().toString(true);
    // 5.2.将User对象转为HashMap存储
    UserDTO userDTO = BeanUtil.copyProperties(user, UserDTO.class);
    Map<String, Object> userMap = BeanUtil.beanToMap(userDTO, new HashMap<>(),
            CopyOptions.create()
                    .setIgnoreNullValue(true)
                    .setFieldValueEditor((fieldName, fieldValue) -> fieldValue.toString()));

    // 5.3.存入 Redis 的 Hash 类型
    String tokenKey = LOGIN_USER_KEY + token;
    stringRedisTemplate.opsForHash().putAll(tokenKey, userMap);
    // 5.4.设置token有效期
    stringRedisTemplate.expire(tokenKey, LOGIN_USER_TTL, TimeUnit.MINUTES);

    // 6.返回token
    return Result.ok(token);
}
```

# Redisson分布式锁实现幂等控制机制总结

通过 Redisson分布式锁，设计幂等控制机制，确保订单扣减与库存更新的一致性，避免超卖，提升了订单准确率

## 实现思路

项目通过Redisson分布式锁设计了一套完整的幂等控制机制，确保在高并发场景下订单扣减与库存更新的一致性，有效避免了超卖问题。核心实现思路如下：

### 1. 基于用户ID的分布式锁策略

- **锁粒度精确控制**：使用用户ID作为锁的唯一标识，实现对同一用户的并发请求进行精确控制
- **非阻塞获取锁**：采用tryLock()方法尝试获取锁，避免线程阻塞
- **自动释放机制**：确保锁最终被释放，避免死锁

### 2. 多层幂等校验机制

- **Redis层预校验**：在Redis中使用Set数据结构记录已购买用户，快速判断重复下单
- **分布式锁互斥校验**：同一用户的并发请求互斥执行，避免重复创建订单
- **数据库层最终校验**：在数据库层面再次查询订单是否存在，形成最后一道防线

### 3. 原子性更新策略

- **条件更新库存**：使用带条件的SQL更新库存，确保只有库存充足时才进行扣减
- **事务保证**：在事务中执行订单创建和库存扣减，确保数据一致性

```java
private void createVoucherOrder(VoucherOrder voucherOrder) {
    Long userId = voucherOrder.getUserId();
    Long voucherId = voucherOrder.getVoucherId();
    
    // 1.创建基于用户ID的分布式锁
    RLock redisLock = redissonClient.getLock("lock:order:" + userId);
    
    // 2.尝试获取锁，非阻塞方式
    boolean isLock = redisLock.tryLock();
    if (!isLock) {
        log.error("不允许重复下单！");
        return;
    }

    try {
        // 3.数据库层幂等校验：查询是否已存在订单
        int count = query().eq("user_id", userId).eq("voucher_id", voucherId).count();
        if (count > 0) {
            log.error("不允许重复下单！");
            return;
        }

        // 4.原子性库存扣减，带条件更新确保只有库存充足时才扣减，相当于一条SQL语句
        boolean success = seckillVoucherService.update()
                .setSql("stock = stock - 1") // 库存减1
                .eq("voucher_id", voucherId).gt("stock", 0) // 条件：优惠券ID匹配且库存大于0
                .update();
        
        if (!success) {
            log.error("库存不足！");
            return;
        }

        // 5.创建订单
        save(voucherOrder);
    } finally {
        // 6.确保锁释放，避免死锁
        redisLock.unlock();
    }
}
```

private class VoucherOrderHandler implements Runnable {
    @Override
    public void run() {
        while (true) {
            try {
                // 1.从消息队列获取订单信息（消费者组模式）
                // 通过 ReadOffset.lastConsumed() 确保每个消费者只获取自己未消费的消息
                // 通过 Consumer.from("g1", "c1") 明确指定了消费者组名( g1 )和消费者名( c1 )
                // 消费者组允许多个消费者实例( c1 , c2 , c3 等)并行消费，Redis会自动分发消息，避免重复消费
                List<MapRecord<String, Object, Object>> list = stringRedisTemplate.opsForStream().read(
                        Consumer.from("g1", "c1"),
                        StreamReadOptions.empty().count(1).block(Duration.ofSeconds(2)),
                        StreamOffset.create("stream.orders", ReadOffset.lastConsumed())
                );
                // 2.判断是否有消息
                if (list == null || list.isEmpty()) {
                    // 没有消息，继续下一次循环
                    continue;
                }
                // 3.解析消息数据
                MapRecord<String, Object, Object> record = list.get(0);
                Map<Object, Object> value = record.getValue();
                VoucherOrder voucherOrder = BeanUtil.fillBeanWithMap(value, new VoucherOrder(), true);
                // 4.处理订单创建
                createVoucherOrder(voucherOrder);
                // 5.确认消息处理完成
                // Redis Stream默认会自动管理消息过期，通过设置Stream的最大长度来自动清理旧消息
				// 通常的最佳实践是让Redis自动管理过期消息，而不是手动删除
                stringRedisTemplate.opsForStream().acknowledge("s1", "g1", record.getId());
            } catch (Exception e) {
                log.error("处理订单异常", e);
                // 处理失败消息
                handlePendingList();
            }
        }
    }
}

private void handlePendingList() {
    while (true) {
        try {
            // 1.从pending-list获取失败的订单信息
            // 功能语义化 ：ReadOffset.from("0") ：在消费者组模式下，"0"是一个特殊值，表示读取pending-list中的消息
            // ReadOffset.lastConsumed() ：表示从消费者上次读取的位置继续读取新消息
			// 明确的行为约定 ：这是Redis Stream API的标准约定，所有使用该API的开发者都理解这种参数的含义
            List<MapRecord<String, Object, Object>> list = stringRedisTemplate.opsForStream().read(
                    Consumer.from("g1", "c1"),
                    StreamReadOptions.empty().count(1),
                    StreamOffset.create("stream.orders", ReadOffset.from("0"))
            );
            // 2.判断是否有失败消息
            if (list == null || list.isEmpty()) {
                // 没有异常消息，结束循环
                break;
            }
            // 3.解析并处理失败消息
            // MapRecord 表示Redis中的 Hash数据结构，提供比普通Map更丰富的功能，特别是与Redis操作相关的特性
            MapRecord<String, Object, Object> record = list.get(0);
            Map<Object, Object> value = record.getValue();
            VoucherOrder voucherOrder = BeanUtil.fillBeanWithMap(value, new VoucherOrder(), true);
            createVoucherOrder(voucherOrder);
            // 4.确认消息处理完成
            stringRedisTemplate.opsForStream().acknowledge("s1", "g1", record.getId());

private void createVoucherOrder(VoucherOrder voucherOrder) {
    Long userId = voucherOrder.getUserId();
    Long voucherId = voucherOrder.getVoucherId();
    // 创建分布式锁，确保一人一单
    RLock redisLock = redissonClient.getLock("lock:order:" + userId);
    boolean isLock = redisLock.tryLock();
    if (!isLock) {
        log.error("不允许重复下单！");
        return;
    }

    try {
        // 1.再次查询订单，确保幂等性
        int count = query().eq("user_id", userId).eq("voucher_id", voucherId).count();
        if (count > 0) {
            log.error("不允许重复下单！");
            return;
        }

        // 2.扣减库存
        boolean success = seckillVoucherService.update()
                .setSql("stock = stock - 1")
                .eq("voucher_id", voucherId).gt("stock", 0)
                .update();
        if (!success) {
            log.error("库存不足！");
            return;
        }

        // 3.保存订单到数据库
        save(voucherOrder);
    } finally {
        // 释放锁
        redisLock.unlock();
    }
}

### 共同关注功能（基于Redis Set）

- **数据结构选择**：使用Redis的Set数据结构存储用户的关注列表
  - Set具有元素唯一性，确保不会出现重复关注记录
  - Set支持高效的交集运算，便于计算共同关注
- **存储设计**：一个用户对应一个Set集合，键名为`follows:{userId}`，集合元素为用户关注的其他用户ID（被关注的用户ID）
- **核心操作**：
  - 关注/取关时同步更新数据库和Redis中的Set集合
  - 使用Set的intersect命令计算两个用户之间的共同关注（`SINTER` 计算给定多个 Set 的交集，即所有给定 Set 中都包含的元素）

### 点赞排行功能（基于Redis ZSet）

- **数据结构选择**：使用Redis的ZSet数据结构存储点赞信息
  - ZSet支持分数排序，便于按时间顺序展示点赞用户
  - ZSet同样保证元素唯一性，防止重复点赞
- **存储设计**：一个博客对应一个ZSet集合，键名为`blog:liked:{blogId}`，
  - 元素：点赞用户ID
  - 分数：点赞时间戳（用于排序）
- **核心操作**：
  - 点赞/取消点赞时同步更新数据库和Redis中的ZSet集合
  - 使用ZSet的range命令获取指定范围内的元素，实现点赞排行查询

@Override
public Result likeBlog(Long id) {
    // 1.获取登录用户
    Long userId = UserHolder.getUser().getId();
    String key = BLOG_LIKED_KEY + id;
    
    // 2.检查是否已点赞
    // .score(key, userId.toString()) 调用score方法查询指定成员在ZSet中的分数值
    // key 是Redis中存储点赞信息的键，格式为 BLOG_LIKED_KEY + id ，用于标识某篇特定博客的点赞集合
    // userId.toString() 将当前用户ID转换为字符串，作为ZSet中的成员值
    Double score = stringRedisTemplate.opsForZSet().score(key, userId.toString());
    
    if (score == null) {
        // 3.未点赞：执行点赞操作
        boolean isSuccess = update().setSql("liked = liked + 1").eq("id", id).update();
        if (isSuccess) {
            // 添加到ZSet，分数为当前时间戳，score只存在Redis里面（非核心业务数据）
            stringRedisTemplate.opsForZSet().add(key, userId.toString(), System.currentTimeMillis());
        }
    } else {
        // 4.已点赞：执行取消点赞操作
        boolean isSuccess = update().setSql("liked = liked - 1").eq("id", id).update();
        if (isSuccess) {
            // 从ZSet移除
            stringRedisTemplate.opsForZSet().remove(key, userId.toString());
        }
    }
    return Result.ok();
}
```

> 该方法实现了博客点赞/取消点赞功能，具体流程如下：

> 1. 接收前端传来的博客ID参数(id)
> 2. 从UserHolder中获取当前登录用户的ID（基于ThreadLocal实现）

>    - 通过查询Redis的ZSet中是否存在该用户ID及对应的score来判断

@Override
public Result queryBlogLikes(Long id) {
    String key = BLOG_LIKED_KEY + id;
    
    // 1.查询点赞时间最早的前5个用户ID
    Set<String> top5 = stringRedisTemplate.opsForZSet().range(key, 0, 4);
    
    if (top5 == null || top5.isEmpty()) {
        return Result.ok(Collections.emptyList());
    }
    
    // 2.转换用户ID并保持原有顺序
    List<Long> ids = top5.stream().map(Long::valueOf).collect(Collectors.toList());
    String idStr = StrUtil.join(",", ids);	// 存储查询到的顺序
    
    // 3.查询用户信息，并按照ZSet中的顺序返回
    // 数据库（如MySQL）在执行 IN 查询时， 不会自动按照传入参数的顺序返回结果
    // 不指定排序时，数据库通常按照主键顺序或物理存储顺序返回数据
    // ORDER BY FIELD(id, id1, id2, id3...) 是MySQL的一个特殊语法，它可以让查询结果按照指定的ID顺序进行排序
    List<UserDTO> userDTOS = userService.query()	// query()开启链式查询
            .in("id", ids).last("ORDER BY FIELD(id," + idStr + ")").list()	// last()在SQL末尾添加自定义SQL片段，list()结束链式查询
            .stream()	// 将结合转为Stream流
            .map(user -> BeanUtil.copyProperties(user, UserDTO.class))
            .collect(Collectors.toList());
    
    return Result.ok(userDTOS);
}
```

> 该方法实现了查询指定博客点赞用户列表的功能，具体流程如下：
>
> 1. 接收前端传来的博客ID参数(id)
> 2. 构建Redis中存储点赞信息的key：BLOG_LIKED_KEY + 博客ID
> 3. 使用Redis ZSet的range方法查询点赞时间最早的前5个用户ID：
>    - 由于在likeBlog方法中使用时间戳作为score存储，range(0,4)会返回score最小的前5个元素，即最早点赞的5个用户
>    - 这里返回的是一个Set类型的数据
> 4. 处理边界情况：如果没有点赞用户，返回空列表
> 5. 数据转换与排序处理：
>    - 将Set中的用户ID字符串转换为Long类型的List
>    - 使用StrUtil.join方法将用户ID列表用逗号连接成字符串idStr

# 基于 Redis BitMap 实现用户签到

Redis BitMap 实现用户签到功能总结

## 实现思路

项目通过Redis的BitMap（位图）数据结构高效实现了用户签到功能。BitMap是Redis提供的一种位数组数据结构，每个位（bit）只能是0或1，可以用来表示某个用户在某天是否签到。这种实现方式具有以下优势：

1. **空间效率高**：一个月的签到状态只需31位，约4字节存储空间，非常节省内存
2. **操作效率高**：使用位运算进行签到统计，计算速度极快
3. **功能灵活**：支持单日签到、连续签到统计、签到日历展示等多种场景

### 核心设计

- **数据结构**：使用Redis String类型作为BitMap底层存储
- **Key设计**：`sign:{userId}:yyyyMM`，按月为单位存储每个用户的签到记录
- **Bit位设计**：位图索引从0开始，对应当月的第1天到第31天
- **统计方式**：使用位运算快速计算连续签到天数

## 核心代码实现

### 1. 签到功能实现

@Override
public Result sign() {
    // 1.获取当前登录用户
    Long userId = UserHolder.getUser().getId();
    // 2.获取当前日期
    LocalDateTime now = LocalDateTime.now();
    // 3.构建Redis key：sign:userId:yyyyMM
    String keySuffix = now.format(DateTimeFormatter.ofPattern(":yyyyMM"));
    String key = USER_SIGN_KEY + userId + keySuffix;
    // 4.获取今天是本月的第几天（从1开始）
    int dayOfMonth = now.getDayOfMonth();

> 2. 处理操作 ：
>    - 从UserHolder获取当前登录用户的ID
>    - 获取当前系统时间（LocalDateTime）
>    - 构建Redis中存储签到信息的key，格式为：USER_SIGN_KEY + userId + :yyyyMM（年月）
>    - 计算今天是本月的第几天（从1开始）
>    - 使用Redis的位操作setBit，将对应天数的二进制位设置为1，表示签到成功
>      - 注意：由于Redis的位索引从0开始，所以需要将dayOfMonth减1
> 3. 返回数据 ：返回Result.ok()，表示签到操作成功完成

### 2. 连续签到统计功能

```java:d:\code\java\hm-dianping\src\main\java\com\hmdp\service\impl\UserServiceImpl.java
@Override
public Result signCount() {
    // 1.获取当前登录用户
    Long userId = UserHolder.getUser().getId();
    // 2.获取当前日期
    LocalDateTime now = LocalDateTime.now();
    // 3.构建Redis key：sign:userId:yyyyMM
    String keySuffix = now.format(DateTimeFormatter.ofPattern(":yyyyMM"));
    String key = USER_SIGN_KEY + userId + keySuffix;
    // 4.获取今天是本月的第几天
    int dayOfMonth = now.getDayOfMonth();
    // 5.使用BITFIELD命令获取从第0位开始的连续dayOfMonth位的值
    List<Long> result = stringRedisTemplate.opsForValue().bitField(
            key,
            BitFieldSubCommands.create()
                    .get(BitFieldSubCommands.BitFieldType.unsigned(dayOfMonth)).valueAt(0)
    );
    
    // 6.处理结果为空的情况
    if (result == null || result.isEmpty()) {
        return Result.ok(0);
    }
    
    Long num = result.get(0);
    if (num == null || num == 0) {
        return Result.ok(0);
    }
    
    // 7.通过位运算计算连续签到天数
    int count = 0;
    while (true) {
        // 7.1.检查最后一个bit位是否为1（签到状态）
        if ((num & 1) == 0) {
            // 最后一位为0，说明未签到，结束循环
            break;
        } else {
            // 最后一位为1，说明已签到，计数器+1
            count++;
        }
        // 7.2.将数字右移一位，检查前一天的签到状态
        num >>>= 1;
    }
    return Result.ok(count);
}
```

> 执行流程
>
> 1. 接收数据 ：该方法不需要接收前端传来的特定参数，而是基于当前登录用户的会话状态执行操作
> 2. 处理操作 ：
>    - 从UserHolder获取当前登录用户的ID（基于ThreadLocal实现，确保线程安全）
>    - 获取当前系统时间（LocalDateTime）
>    - 构建Redis中存储签到信息的key，格式为：USER_SIGN_KEY + userId + :yyyyMM（年月）
>    - 计算今天是本月的第几天
>    - 使用Redis的BITFIELD命令获取从第0位开始的连续dayOfMonth位的值：
>      - 通过BitFieldSubCommands.create()创建子命令
>      - 使用get方法并指定无符号整数类型（unsigned(dayOfMonth)）
>      - 从位置0开始读取，一次性获取整个月的签到记录
>    - 处理边界情况：如果结果为空或为0，返回连续签到天数为0
>    - 通过位运算计算连续签到天数：
>      - 初始化计数器count为0
>      - 进入循环，不断检查num的最低位（最右边一位）
>      - 如果最低位为1（签到），计数器加1，然后将num右移一位
>      - 如果最低位为0（未签到），结束循环
>      - 这里使用无符号右移（>>>）确保符号位不影响计算
> 3. 返回数据 ：返回Result.ok(count)，其中count是连续签到的天数 连续签到天数计算细节
>    连续签到天数的计算是从当前日期往前推，直到遇到未签到的日期为止。具体算法：
>
> - 例如，假设今天是5号，且1-5号都签到了，那么bit值为0b11111（从右往左分别表示1-5号）
> - 通过num & 1检查最低位是否为1（1号是否签到）
> - 每检查一次，就将num无符号右移一位（num >>>= 1），检查下一个bit位
> - 一旦遇到0（未签到），则停止计数
> - 最终得到的count就是从今天开始向前连续签到的天数

### Bitmap位存储结构

> 在签到功能中，日期与bit位的对应关系如下：
>
> - 当月1号：bit索引0（最右边位）
> - 当月2号：bit索引1
> - 当月3号：bit索引2
> - ...以此类推
>
> **连续签到天数计算逻辑**
>
> 计算连续签到天数时的处理流程：
>
> 1. 当我们通过 **bitField** 命令获取到签到数据后，得到的是一个数字 num
> 2. 例如，如果用户1号和2号都签到了，num的值就是二进制 0b11
> 3. 在计算连续签到天数时：
>    - 我们首先检查 num & 1 ，这实际上是检查最右边的位（最低位），也就是1号的签到状态
>    - 如果最低位是1（已签到），计数器加1
>    - 然后执行 num >>>= 1 （无符号右移一位）
>    - 此时，原num的第二位变成了最右边的位，也就是检查2号的签到状态
>
> **为什么这样设计**
>
> 这种设计的巧妙之处在于：
>
> 1. 从当天向前统计 ：通过从最低位开始检查，可以自然地从1号开始统计，直到遇到未签到的日期
> 2. 计算连续天数 ：一旦遇到0，说明某个日期未签到，连续签到中断，可以立即停止计算
> 3. 高效性 ：位运算非常高效，即使对于一个月30天，最多只需要循环30次
>
> **具体例子**
>
> 假设今天是3号，且用户1-3号都签到了：
>
> - bitField获取的num = 0b111（二进制）= 7（十进制）
> - 第一次循环：num & 1 = 1 → count=1, num=0b11
> - 第二次循环：num & 1 = 1 → count=2, num=0b1
> - 第三次循环：num & 1 = 1 → count=3, num=0b0
> - 第四次循环：num & 1 = 0 → 退出循环
> - 最终返回count=3，表示连续签到3天
>   这种实现充分利用了位运算的特性，既节省了存储空间，又保证了计算效率。

## 技术亮点

### 1. 空间高效的位图存储

## 实现细节分析

### 1. 签到逻辑

签到功能的核心是`SETBIT key offset value`命令，它将指定key的offset位置设置为value。在实现中：

- `key`：`sign:{userId}:yyyyMM`格式的键名
- `offset`：当前日期在本月的第几天减1（因为位图从0开始）
- `value`：true，表示签到状态

### 2. 连续签到统计逻辑

统计连续签到天数的过程体现了位运算的巧妙应用：

1. 使用`BITFIELD key GET uN 0`命令获取从第0位开始的N个位作为无符号整数
2. 通过循环检查该整数的每一位：
   - 使用`num & 1`操作检查最低位是否为1
   - 如果最低位为1，说明签到，计数器加1
   - 使用`num >>>= 1`操作将整数右移一位，继续检查下一位
   - 直到遇到0或者所有位都检查完毕
3. 这种实现可以高效计算从当天开始向前的连续签到天数

2. **高效计算**：位运算使签到统计变得简单高效
3. **扩展性好**：易于扩展到签到日历、签到奖励等功能
4. **性能卓越**：Redis位图操作支持高并发访问
