## 一、Excel商品批量导入功能

### 核心要点
- **技术栈**：Apache POI（支持.xls/.xlsx自动识别）
- **匹配策略**：模糊+精确两步匹配
- **性能指标**：100+条数据，响应时间<2秒，提升效率60%

### 核心机制

#### 1. 文件解析
- 使用 `WorkbookFactory.create()` 自动识别格式
- 解析表头，建立字段映射（支持顺序调整）
- 逐行解析数据，处理多种单元格类型（STRING/NUMERIC/DATE）

#### 2. 商品匹配策略（两步法）
```
第一步：数据库模糊匹配
- SQL: WHERE name LIKE '%商品名称%' AND compId = ?
- 目的：快速缩小候选范围，利用数据库索引

第二步：内存精确匹配
- 去除空格后比较：trim(name).equals(trim(excelName))
- 同时匹配名称和规格
- 目的：处理空格差异，确保精确匹配

第三步：相似商品收集
- 匹配失败时收集候选商品
- 用于错误提示和推荐
```

#### 3. 数据验证
- 必填字段校验（名称、规格、单价、数量）
- 数据格式验证（数字、日期）
- 业务规则校验（价格合理性、数量范围）

### 面试问题

**Q: 为什么用两步匹配？**
- 模糊匹配：利用数据库索引，快速筛选候选商品
- 精确匹配：处理空格差异，确保准确性
- 性能优化：减少内存比较的数据量

**Q: 如何优化性能？**
- 批量查询替代逐条查询
- 使用数据库索引（name字段）
- 内存缓存常用商品信息
- 异步处理（可选）

**Q: 如何处理错误？**
- 部分成功机制：成功的数据入库，失败的数据返回错误列表
- 详细错误提示：行号+错误原因+相似商品推荐
- 支持重新导入失败数据

## 二、密码过期强制修改功能

### 核心要点
- **过期规则**：3个月（90天）未修改强制修改
- **拦截机制**：登录时校验，过期则拦截
- **修改接口**：匿名接口，无需Token，仅验证旧密码

### 核心机制

#### 1. 过期判断逻辑
```java
// 过期条件：
// 1. PasswordChangeTime 为 NULL（从未修改）
// 2. PasswordChangeTime < 当前时间 - 3个月

if (emp.PasswordChangeTime == null || 
    emp.PasswordChangeTime < DateTime.Now.AddMonths(-3)) {
    // 返回 PasswordExpired 错误码(13)
}
```

#### 2. 登录拦截流程
```
用户登录
  ↓
验证密码正确性
  ↓
密码错误 → 返回 NoAccount
  ↓
密码正确 → 检查 PasswordChangeTime
  ↓
密码过期 → 返回 PasswordExpired(13)
  ↓
前端捕获 → 跳转密码修改页面
```

#### 3. 密码修改流程
```
用户输入旧密码+新密码
  ↓
验证旧密码正确性
  ↓
验证新密码格式
  ↓
调用 ModifyPassword() 方法
  ↓
自动更新 PasswordChangeTime = 当前时间
  ↓
保存到数据库
  ↓
返回成功 → 跳转登录页
```

#### 4. 统一更新机制
- 所有密码修改路径都调用 `Employee.ModifyPassword()`
- 自动更新 `PasswordChangeTime`，避免遗漏
- 包括：强制修改、普通修改、验证码修改

### 面试问题

**Q: 为什么在登录时校验而不是修改时？**
- 登录是必经流程，确保所有用户都能被检查
- 过期后禁止登录，强制用户修改，保证安全性

**Q: 匿名接口的安全性如何保证？**
- 仅验证旧密码，不知道旧密码无法修改
- 修改后需重新登录，使用新密码
- 可加入IP限制、验证码等（可选）

**Q: NULL值如何处理？**
- NULL视为从未修改过，强制首次修改
- 首次修改后自动设置当前时间

## 三、审计日志功能

### 核心要点
- **日志类型**：登录日志（成功/失败/登出）+ 操作日志（增删改查/导出/上传）
- **存储方式**：异步存储，不阻塞主流程
- **安全机制**：敏感信息脱敏、IP地址识别

### 核心机制

#### 1. 登录日志记录
**记录时机**：
- 登录成功：验证通过后记录
- 登录失败：验证失败时记录（包含失败原因）
- 登出：调用登出接口时记录

**记录字段**：
- 操作人信息（ID、姓名、账号、门店、公司）
- 登录信息（时间、类型、IP、浏览器）
- 失败原因（仅失败时）

#### 2. 操作日志记录
**拦截机制**：
- 使用 `OperationLogInterceptor` 拦截HTTP请求
- `preHandle`：记录请求开始时间
- `afterCompletion`：记录操作日志

**操作类型判断**（优先级从高到低）：
```
删除：/delete, /remove, /del, /zuofei, /cancel
导出：/export
下载：/download
上传/导入：/upload, /import
编辑：/update, /edit, /changeStatus, /set
新增：/save, /add, /create
默认：根据HTTP方法（POST→新增，PUT→编辑，DELETE→删除）
```

**接口过滤**：
- 记录：包含操作关键字的接口
- 排除：查询接口（/index, /show, /get**, /query**）、日志查询接口、静态资源

#### 3. 异步处理机制
```java
// Java后端
CompletableFuture.runAsync(() -> {
    try {
        loginLogService.save(log);
    } catch (Exception e) {
        logger.error("记录日志失败", e);
    }
});

// .NET后端
Task.Run(() => {
    try {
        SendPostRequest(url, data);
    } catch (Exception ex) {
        logger?.Warn($"记录日志失败: {ex.Message}", ex);
    }
});
```

#### 4. 敏感信息脱敏
- 请求参数中跳过：password, pwd, token
- JSON参数脱敏：password, pwd, token, accessToken → "***"
- 限制参数长度：单个参数200字符，总长度2000字符

#### 5. IP地址识别
```
优先级从高到低：
1. X-Forwarded-For（代理场景）
2. X-Real-IP
3. Proxy-Client-IP
4. WL-Proxy-Client-IP
5. request.getRemoteAddr()（直接连接）
6. 特殊处理：IPv6 localhost → 127.0.0.1
```

### 面试问题

**Q: 为什么用异步存储？**
- 不阻塞主业务流程，提升响应速度
- 日志记录失败不影响业务操作
- 高并发场景下减少数据库压力

**Q: 异步存储失败怎么办？**
- 捕获异常，记录错误日志
- 不影响主业务流程
- 可加入重试机制或消息队列（扩展）

**Q: 如何保证日志完整性？**
- 拦截器统一拦截，覆盖所有操作接口
- 操作类型自动识别，减少遗漏
- 记录操作结果（成功/失败）和耗时

**Q: 脱敏策略是什么？**
- 密码类字段：完全脱敏（***）
- Token类字段：部分脱敏或完全脱敏
- 其他敏感信息：根据业务需求定制

## 四、Bug修复

### 核心要点
- **业绩报表移动端数据缺失**：优化SQL过滤逻辑
- **推荐类型客户保存失败**：补充接口校验与依赖注入
- **入库金额异常、明细缺失**：完善事务管理与数据验证

### 核心机制

#### 1. SQL过滤逻辑优化
**问题**：移动端查询时缺少数据过滤条件
**解决**：
- 补充门店ID、公司ID等过滤条件
- 确保移动端和PC端数据一致性
- 优化查询性能，添加必要索引

#### 2. 接口校验与依赖注入
**问题**：推荐类型客户保存时缺少必要校验
**解决**：
- 补充参数校验（@Valid、@NotNull等）
- 修复依赖注入问题（@Autowired、@Resource）
- 添加业务规则校验（推荐类型有效性）

#### 3. 事务管理与数据验证
**问题**：入库金额计算错误、明细数据缺失
**解决**：
- 使用 `@Transactional` 保证数据一致性
- 完善数据验证（金额计算、明细完整性）
- 添加异常回滚机制

### 面试问题

**Q: 如何定位Bug？**
- 查看错误日志，定位异常堆栈
- 复现问题，分析业务逻辑
- 检查数据库数据，验证数据一致性
- 使用调试工具，跟踪代码执行流程

**Q: 事务管理的关键点？**
- 事务边界：确保相关操作在同一事务中
- 异常回滚：异常时自动回滚，保证数据一致性
- 事务传播：根据业务需求选择合适的传播行为

**Q: 如何避免类似Bug？**
- 完善单元测试和集成测试
- 代码审查，检查常见问题
- 统一异常处理和日志记录
- 建立代码规范和最佳实践

## 六、常见扩展问题

### Q: 如果Excel文件很大（10万条数据）怎么办？
- 使用流式读取（SXSSFWorkbook），避免内存溢出
- 分批处理，每批1000-5000条
- 使用消息队列，异步处理
- 提供进度反馈，提升用户体验

### Q: 如何优化商品匹配性能？
- 建立商品名称+规格的联合索引
- 使用Redis缓存常用商品信息
- 批量查询替代逐条查询
- 考虑使用Elasticsearch进行全文搜索

### Q: 审计日志数据量大怎么办？
- 定期归档，历史数据迁移到冷存储
- 按时间分表，提升查询性能
- 使用消息队列，削峰填谷
- 考虑使用专门的日志系统（ELK等）

### Q: 密码过期提醒如何实现？
- 登录时检查，距离过期7天时提醒
- 定时任务扫描，发送邮件/短信提醒
- 前端显示剩余天数，引导用户主动修改
