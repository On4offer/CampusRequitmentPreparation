简历里的 **小众点评** 和 **校园订餐（苍穹外卖）** 两个项目整理成 **面试可用项目介绍**，每个都包含：**背景场景 → 技术方案 → 难点亮点 → 收获**

------

## 🏷 项目一：小众点评

这个项目类似大众点评，核心功能包括用户登录、商户查询、优惠券秒杀、用户签到、UV 统计以及附近商户。项目背景主要是要在高并发场景下保证系统的稳定性和用户体验。

在技术实现上，重点做了缓存和高并发的处理。比如在缓存层面，采用了 Redis 的 Cache-Aside 策略，并结合逻辑过期和主动更新机制来解决缓存击穿和雪崩问题，保证热点数据的可用性。用户鉴权这块，把 Token 存储在 Redis 中，结合滑动过期机制来动态延长登录态，同时用 ThreadLocal 管理用户上下文，确保了接口安全。秒杀下单环节，我用 Redisson 的分布式锁来控制库存扣减的幂等，避免出现超卖；同时借助 Redis Stream 做异步下单，把订单消息先写入队列，再由消费者落库，实现了削峰填谷。除此之外，还利用了 Redis 的多种数据结构，比如用 BitMap 做签到统计、用 HyperLogLog 做 UV 统计，以及通过 GeoHash 实现附近商户的地理位置检索。

这个项目的难点主要在于缓存和高并发处理。通过一系列优化，我解决了缓存击穿和雪崩的问题，保证了接口稳定性；同时设计了幂等控制和异步下单的流程来应对秒杀高并发。最大的收获是对 **缓存优化、分布式锁以及消息队列解耦** 这些高并发场景下的思路有了深入实践，也积累了根据不同业务场景选择合适 Redis 数据结构的经验。

## 🏷 项目二：校园订餐

这是一个面向校园场景的点餐系统，主要功能是用户点餐、商户管理以及订单的实时跟踪。做这个项目的目标是既要保证高并发下的数据一致性，又要提升用户的交互体验。

在登录模块，我是通过拦截器配合 ThreadLocal 来管理用户会话的，同时自定义注解实现了 Token 校验，解决了 HTTP 无状态带来的鉴权问题。缓存方面，在商品查询模块中引入了 Redis 和 SpringCache 来减少数据库访问，同样结合了逻辑过期和主动更新机制来防止缓存击穿。系统整体采用模块化设计，后台管理端支持独立部署和维护，保证了扩展性和可维护性。为了提升用户体验，我还基于 WebSocket 实现了订单状态的实时推送，让商户和用户都能及时收到订单的最新进展。此外，利用 SpringTask 做了定时任务，比如自动清理超时订单和更新订单状态，保证业务流程的闭环。

这个项目的亮点在于我解决了会话管理和高并发鉴权的问题，同时通过 WebSocket 把用户体验提升到了实时交互的层面。另外，系统模块化的设计让后续扩展和维护都更加方便。通过这个项目，我完整经历了 **用户鉴权、缓存优化、实时通信以及任务调度** 等多个环节的开发，对如何兼顾性能和用户体验有了更清晰的理解。

👉 面试技巧：

- 每个项目介绍时间 **2-3 分钟**，先说场景，再说方案，最后点亮难点和收获。
- 不要面面俱到，面试官会对其中某些点展开追问（比如“你们的逻辑过期是怎么实现的？”、“Redis Stream 为什么选它？”）。
