# 医疗美容行业管理系统迭代开发 - 面试深度准备文档
## 一、Excel 商品批量导入功能（核心独立开发）
### 1. 需求背景与核心诉求
- 业务痛点：医疗美容行业商品 SKU 繁多（含药品、护肤品、耗材等），门店和总部入库时手动录入效率低、易出错，单门店单次入库平均需 30 分钟，总部批量入库耗时更久
- 核心需求：支持 Excel 批量导入，兼容.xls/.xlsx 格式，适配总部入库、门店退货入库、自行采购 3 种场景
- 功能要求：
  - 文件大小限制 10MB，支持 100 + 条数据快速导入
  - 标准表头包含 11 个字段（名称 / 规格 / 单价等，必填字段 4 个）
  - 商品自动匹配，匹配失败提供相似商品推荐
  - 逐行验证数据，部分失败不影响整体导入，返回详细错误日志
  - 导入后自动更新入库清单和总额计算

### 2. 技术选型思考
- Excel 解析：选择 Apache POI（支持两种格式，生态成熟），放弃 EasyExcel（老系统兼容性不足）
- 匹配策略：设计 "模糊 + 精确" 两步匹配（医疗美容商品名称相似但规格差异大，单一匹配易误判）
- 前后端交互：采用 FormData 上传（支持大文件），避免 JSON 序列化丢失文件信息
- 性能优化：异步解析 + 批量入库，减少数据库交互次数
- 兼容性：适配 Spring Boot 后端和 Vue.js 前端，兼容原有库存管理模块数据结构

### 3. 核心业务逻辑与流程
#### 整体流程

用户选择Excel文件 → 前端校验（格式/大小）→ 构建FormData上传 → 后端接收文件 →
文件校验（格式/表头）→ 逐行解析数据 → 商品匹配（模糊→精确）→ 数据验证（必填/格式）→
批量入库处理 → 返回导入结果（成功数/失败数/错误详情）→ 前端更新入库清单

#### 关键逻辑拆解
**Excel 解析逻辑**：
- 用 WorkbookFactory 自动识别文件格式（.xls/.xlsx）
- 表头灵活匹配（支持表头顺序调整，通过字段名称映射列索引）
- 数据类型转换（数字去科学计数法、日期统一格式为 yyyy-MM-dd、空值处理）

**商品匹配策略**：
- 第一步：模糊匹配（数据库 LIKE 查询，商品名称去空格 / 特殊字符后比对）
- 第二步：精确匹配（在模糊结果中，精确比对规格 + 生产厂家）
- 匹配失败处理：收集相似商品信息（名称相似度≥80%），返回给前端供用户选择

**数据验证规则**：
- 必填项校验（名称 / 规格 / 单价 / 数量不能为空）
- 格式校验（单价 / 数量为正数、有效期为日期格式）
- 业务规则校验（进货单价优先于单价字段，小计字段与单价 × 数量一致性校验）

// 1. Excel解析工具类（ExcelImportUtils.java）
public ImportResult parseExcel(InputStream inputStream, String compId) {
    ImportResult result = new ImportResult();
    Workbook workbook = null;
    try {
        workbook = WorkbookFactory.create(inputStream); // 自动识别格式
        Sheet sheet = workbook.getSheetAt(0);
        Row headerRow = sheet.getRow(0);
        Map<String, Integer> headerMap = parseHeader(headerRow); // 表头映射
        
        // 逐行解析数据
        int lastRowNum = sheet.getLastRowNum();
        for (int i = 1; i <= lastRowNum; i++) {
            Row row = sheet.getRow(i);
            if (row == null) continue;
            
            try {
                // 解析单行数据
                Map<String, Object> goodsData = parseRow(row, headerMap, compId, i + 1);
                result.getSuccessList().add(goodsData);
            } catch (BusinessException e) {
                // 收集错误信息
                ErrorItem error = new ErrorItem(i + 1, e.getMessage());
                result.getErrorList().add(error);
            }
        }
    } catch (Exception e) {
        throw new BusinessException("Excel解析失败：" + e.getMessage());
    } finally {
        if (workbook != null) workbook.close();
    }
    return result;
}

// 2. 商品匹配核心方法
private Goods findGoodsByNameAndSpec(String name, String spec, String compId, StringBuilder candidateInfo) {
    // 模糊匹配：去空格+特殊字符
    String cleanName = name.replaceAll("[^a-zA-Z0-9\u4e00-\u9fa5]", "");
    List<Goods> fuzzyList = goodsRepository.findByNameContainingAndCompId(cleanName, compId);
    
    if (fuzzyList.isEmpty()) {
        candidateInfo.append("无相似商品");
        return null;
    }
    
    // 精确匹配：规格+名称完全一致（去空格）
    for (Goods goods : fuzzyList) {
        String cleanDbSpec = goods.getSpecification().replaceAll("[^a-zA-Z0-9\u4e00-\u9fa5]", "");
        if (cleanDbSpec.equals(spec.replaceAll("[^a-zA-Z0-9\u4e00-\u9fa5]", ""))) {
            return goods;
        }
    }
    
    // 收集相似商品
    candidateInfo.append("相似商品：");
    for (Goods goods : fuzzyList.subList(0, Math.min(3, fuzzyList.size()))) {
        candidateInfo.append(goods.getName()).append("(").append(goods.getSpecification()).append(")、");
    }
    return null;
}

// 3. 入库Service实现
@Transactional
public Result importExcel(MultipartFile file, String compId) {
    // 文件校验
    if (file.getSize() > 10 * 1024 * 1024) {
        return Result.fail("文件大小超过10MB限制");
    }
    String fileName = file.getOriginalFilename();
    if (!fileName.endsWith(".xls") && !fileName.endsWith(".xlsx")) {
        return Result.fail("仅支持.xls和.xlsx格式");
    }
    
    // 解析Excel
    ImportResult importResult = excelImportUtils.parseExcel(file.getInputStream(), compId);
    
    // 批量入库
    if (!importResult.getSuccessList().isEmpty()) {
        List<InboundItem> itemList = importResult.getSuccessList().stream()
            .map(data -> {
                InboundItem item = new InboundItem();
                item.setGoodsId(((Goods) data.get("goods")).getId());
                item.setSpec((String) data.get("specification"));
                item.setUnitPrice((BigDecimal) data.get("unitPrice"));
                item.setQuantity((Integer) data.get("quantity"));
                // 其他字段赋值
                return item;
            }).collect(Collectors.toList());
        inboundItemRepository.saveAll(itemList);
    }
    
    // 构建返回结果
    return Result.success("导入完成", ImmutableMap.of(
        "totalRows", importResult.getSuccessList().size() + importResult.getErrorList().size(),
        "successCount", importResult.getSuccessList().size(),
        "errorCount", importResult.getErrorList().size(),
        "errorList", importResult.getErrorList()
    ));
}

## 二、密码过期强制修改功能（独立开发）
### 1. 需求背景与核心诉求
- 业务痛点：医疗美容行业涉及客户隐私和财务数据，账号安全要求高，原系统无密码过期机制，存在安全风险
- 核心需求：实现密码 3 个月强制修改，保障账号安全
- 功能要求：
  - 登录时校验密码修改时间，过期则拦截登录
  - 提供匿名修改接口（无需 Token，支持过期密码修改）
  - 新密码无需短信验证，但需满足强度要求
  - 兼容 Java 和ASP.NET双后端账号体系
  - 修改后同步更新密码修改时间，支持新密码登录

### 2. 技术选型思考
- 过期判断：基于数据库新增字段`PasswordChangeTime`（datetime 类型），NULL 视为过期
- 接口设计：匿名接口（/users/password/forcechange），避免未登录状态无法访问
- 密码加密：沿用系统现有 MD5 大写加密，保证与老系统兼容
- 前后端交互：错误码 13 标识密码过期，前端捕获后自动跳转修改页面
- 事务保障：密码修改与时间更新原子性，避免数据不一致

### 3. 核心业务逻辑与流程
#### 登录流程
```
用户输入账号密码 → 前端提交登录请求 → 后端验证密码正确性 →
密码错误→返回错误提示 → 密码正确→校验PasswordChangeTime →
未过期→生成Token→登录成功 → 已过期→返回错误码13→前端跳转修改页面
```

#### 密码修改流程
```
用户进入修改页面→输入旧密码/新密码/确认密码→前端校验（格式/一致性）→
提交修改请求→后端验证旧密码→验证新密码强度→更新密码和修改时间→
返回成功→前端跳转登录页面→用户用新密码登录
```

// 1. 登录拦截校验（UserServiceImpl.java）
public Result login(LoginDTO loginDTO) {
    // 密码验证
    Employee employee = employeeRepository.findByAccountAndCompNo(loginDTO.getAccount(), loginDTO.getCompNo());
    if (employee == null || !employee.getPassword().equals(md5Encrypt(loginDTO.getPassword()))) {
        return Result.fail(ResultEnum.NO_ACCOUNT);
    }
    
    // 密码过期校验（3个月=90天）
    Date now = new Date();
    if (employee.getPasswordChangeTime() == null) {
        // 从未修改过，视为过期
        return Result.fail(ResultEnum.PASSWORD_EXPIRED);
    }
    long diffDays = (now.getTime() - employee.getPasswordChangeTime().getTime()) / (1000 * 60 * 60 * 24);
    if (diffDays > 90) {
        return Result.fail(ResultEnum.PASSWORD_EXPIRED);
    }
    
    // 生成Token，登录成功
    String token = JwtUtils.generateToken(employee.getId(), employee.getAccount());
    return Result.success(ImmutableMap.of("token", token, "employee", employee));
}

// 2. 匿名修改接口（UserController.java）
@Anonymous // 匿名访问注解
@PostMapping("/password/forcechange")
public Result forceChangePassword(@RequestBody ForceChangePasswordDTO dto) {
    // 参数校验
    if (dto.getOldPassword() == null || dto.getNewPassword() == null) {
        return Result.fail("旧密码和新密码不能为空");
    }
    if (dto.getOldPassword().equals(dto.getNewPassword())) {
        return Result.fail("新密码不能与原密码相同");
    }
    if (!PasswordUtils.checkStrength(dto.getNewPassword())) {
        return Result.fail("新密码强度不足，需包含数字、字母和特殊字符，长度≥8位");
    }
    
    // 验证旧密码
    Employee employee = employeeRepository.findByAccountAndCompNo(dto.getAccount(), dto.getCompNo());
    if (employee == null) {
        return Result.fail("账号不存在");
    }
    if (!employee.getPassword().equals(md5Encrypt(dto.getOldPassword()))) {
        return Result.fail("旧密码不正确");
    }
    
    // 更新密码和修改时间
    employee.setPassword(md5Encrypt(dto.getNewPassword()));
    employee.setPasswordChangeTime(new Date());
    employee.setUpdateTime(new Date());
    employee.setUpdateUser("system");
    employeeRepository.save(employee);
    
    // 同步更新缓存
    cacheService.updateEmployeeCache(employee.getId(), employee);
    
    return Result.success("密码修改成功，请使用新密码登录");
}

- 刚好 3 个月：密码修改时间为 90 天前，视为未过期，允许登录
- 并发修改：同一账号同时提交修改请求，通过数据库乐观锁保证数据一致性
- 网络中断：修改过程中网络中断，事务回滚，密码未更新

#### 安全测试
- 密码传输：通过 HTTPS 加密传输，抓包无法获取明文密码
- 暴力破解：接口添加限流，连续 5 次错误后锁定 10 分钟

## 三、审计日志功能（参与开发）
### 1. 需求背景与核心诉求
- 业务痛点：医疗美容行业需满足合规审计要求，原系统无完整日志记录，无法追溯用户操作行为
- 核心需求：记录登录日志和操作日志，支持审计追溯
- 功能要求：
  - 登录日志：记录成功 / 失败 / 登出状态，包含操作人、IP、浏览器信息、失败原因
  - 操作日志：记录增删改查 / 导出 / 上传等操作，包含操作人、接口路径、参数（脱敏）、结果、耗时
  - 技术要求：异步记录（不阻塞主流程）、敏感信息脱敏、IP 地址识别、跨后端（Java/.NET）同步

### 2. 技术选型思考
- 实现方式：选择 Spring 拦截器（Java）+ 过滤器（.NET），放弃 AOP（老系统兼容性不足）
- 存储方案：MySQL 数据库（sys_login_log/sys_operation_log 表），支持索引查询
- 异步处理：Java 用 CompletableFuture，.NET 用 Task.Run，避免影响主流程
- 跨系统同步：.NET 后端通过 HTTP 调用 Java 接口，统一存储日志
- 脱敏策略：密码、Token 等字段替换为 ***，手机号 / 身份证号部分隐藏

// 1. 操作日志拦截器（OperationLogInterceptor.java）
@Component
public class OperationLogInterceptor implements HandlerInterceptor {
    @Autowired
    private SysOperationLogService operationLogService;
    
    private ThreadLocal<Long> startTime = new ThreadLocal<>();
    
    @Override
    public boolean preHandle(HttpServletRequest request, HttpServletResponse response, Object handler) {
        startTime.set(System.currentTimeMillis());
        return true;
    }
    
    @Override
    public void afterCompletion(HttpServletRequest request, HttpServletResponse response,
                               Object handler, Exception ex) {
        // 异步记录日志
        CompletableFuture.runAsync(() -> {
            try {
                // 跳过查询接口和静态资源
                String requestUri = request.getRequestURI();
                if (requestUri.contains("/index") || requestUri.contains("/show") ||
                    requestUri.contains("/get") || requestUri.contains("/static")) {
                    return;
                }
                
                // 提取用户信息（从JWT Token）
                CurrentEmployeeDTO employee = CurrentEmployeeUtils.analysisEmployee(request);
                if (employee == null) return;
                
                // 构建日志对象
                SysOperationLog log = new SysOperationLog();
                log.setId(UUID.randomUUID().toString());
                log.setEmpId(employee.getId());
                log.setEmpName(employee.getEmpName());
                log.setClinicId(employee.getClinicId());
                log.setClinicName(employee.getClinicName());
                log.setCompId(employee.getCompId());
                log.setApiPath(requestUri);
                log.setApiMethod(request.getMethod());
                log.setIpAddress(getIpAddress(request));
                log.setOperationTime(new Date());
                log.setDuration(System.currentTimeMillis() - startTime.get());
                log.setOperationResult(ex == null ? 1 : 2);
                if (ex != null) {
                    log.setErrorMsg(ex.getMessage());
                }
                
                // 操作类型判断
                log.setOperationType(getOperationType(requestUri));
                // 请求参数（脱敏）
                log.setRequestParams(desensitizeParams(request));
                // 操作描述
                log.setOperationDesc(getOperationDesc(requestUri));
                
                // 保存日志
                operationLogService.save(log);
            } catch (Exception e) {
                log.error("记录操作日志失败", e);
            } finally {
                startTime.remove();
            }
        });
    }
    
    // 提取IP地址（支持代理场景）
    private String getIpAddress(HttpServletRequest request) {
        String ip = request.getHeader("X-Forwarded-For");
        if (ip == null || ip.isEmpty() || "unknown".equalsIgnoreCase(ip)) {
            ip = request.getHeader("X-Real-IP");
        }
        if (ip == null || ip.isEmpty() || "unknown".equalsIgnoreCase(ip)) {
            ip = request.getRemoteAddr();
        }
        // IPv6转IPv4
        if ("0:0:0:0:0:0:0:1".equals(ip) || "::1".equals(ip)) {
            ip = "127.0.0.1";
        }
        return ip;
    }
    
    // 敏感信息脱敏
    private String desensitizeParams(HttpServletRequest request) {
        Map<String, String[]> params = request.getParameterMap();
        Map<String, Object> paramMap = new HashMap<>();
        for (Map.Entry<String, String[]> entry : params.entrySet()) {
            String key = entry.getKey();
            String[] values = entry.getValue();
            // 密码、Token等字段脱敏
            if (key.contains("password") || key.contains("pwd") || key.contains("token")) {
                paramMap.put(key, "***");
            } else {
                paramMap.put(key, values.length == 1 ? values[0] : values);
            }
        }
        return JSON.toJSONString(paramMap);
    }
}

### 3. 入库金额异常、明细缺失问题（Bug3）
#### 问题描述

- 无异常处理：明细保存失败时，循环中断，总额未更新

触发场景：
- 场景 A（90%）：price 为 null，保存明细时抛出 DataIntegrityViolationException
- 场景 B（8%）：传入空数组（jsonArray=[]），循环不执行，总额保持 0
- 场景 C（2%）：goodsId 无效，触发外键约束异常

#### 修复思路
- 添加事务管理：确保入库单和明细保存原子性，失败回滚
- 调整执行顺序：先验证数据→计算总额→保存入库单→处理明细
- 完善数据验证：校验 goodsId（非空 + 长度 36）、price（非空）、总额（>0）
- 补充空数组判断：防止空明细提交
- 添加异常处理：捕获异常并返回明确提示

        if (jsonArray == null || jsonArray.size() == 0) {
            r.setMsg("入库明细不能为空");
            return r;
        }
        
        // 修复：先验证所有明细数据并计算总额
        double amount = 0;
        for (int i = 0; i < jsonArray.size(); i++) {
            JSONObject jsonObject = jsonArray.getJSONObject(i);
            String goodId = jsonObject.getString("goodsId");

            }
            
            // 累加总额
            amount += buyCount * price.doubleValue();
        }
        
        // 验证总额

            String errorMsg = "入库总额必须大于0，当前总额：" + amount;

        }

        Repertoryenters repertoryenter = new Repertoryenters();
        repertoryenter.setTotalAmount(amount); // 已计算的总额
        // 其他字段设置...

        return r;
    } catch (Exception e) {

        throw e; // 触发事务回滚
    }
}

#### Q2：商品匹配策略为什么设计 "模糊 + 精确" 两步？实际使用中效果如何？
A：设计两步匹配是针对医疗美容商品的特性：
- 业务痛点：医疗美容商品名称相似性高（如 "玻尿酸 A 款" 和 "玻尿酸 B 款"），但规格、厂家差异大，单一匹配易误判或漏判
- 模糊匹配（第一步）：先通过名称去特殊字符后的 LIKE 查询，快速缩小匹配范围，避免精确匹配时因微小差异（如空格、符号）导致匹配失败
- 精确匹配（第二步）：在模糊结果中，通过规格 + 厂家的精确比对，确保匹配准确性

实际效果：
- 匹配准确率从原来的 75% 提升到 98%
- 匹配失败率降低，相似商品推荐功能减少了用户手动录入的工作量
- 解决了商品名称含特殊字符、规格表述不一致导致的匹配问题

#### Q3：如何处理 Excel 导入过程中的异常？比如部分数据错误的情况？
A：采用 "逐行解析 + 错误隔离 + 详细日志" 的方案：
- 逐行解析：对 Excel 中的每一行数据单独解析和验证，一行数据错误不影响其他行
- 错误隔离：用专门的 ErrorItem 类收集错误信息（行号、错误原因、相似商品推荐）
- 事务控制：成功解析的数据批量入库，失败数据不入库，确保数据一致性
- 前端反馈：返回成功数、失败数和详细错误列表，支持弹窗查看，用户可根据错误信息修正 Excel 后重新导入

### 2. 密码过期功能相关
#### Q1：如何确保 Java 和.NET 双后端的密码状态同步？如果同步失败怎么办？
A：同步方案设计：
- 核心原则：密码修改操作统一在 Java 端完成，.NET 端通过调用 Java 接口同步数据
- 实现方式：.NET 端的密码修改请求转发到 Java 端的匿名接口，Java 端更新数据库后，同步更新缓存
- 事务保障：Java 端的密码更新和 PasswordChangeTime 修改放在同一个事务中，确保原子性

同步失败处理：
- 异常捕获：.NET 端调用 Java 接口时捕获异常，记录错误日志
- 重试机制：对暂时性错误（如网络抖动），实现 3 次重试逻辑
- 人工兜底：定时任务检测双后端密码状态不一致的账号，发送告警通知管理员手动处理
- 数据一致性：以 MySQL 数据库为准，双后端都从数据库读取密码状态，避免缓存不一致

#### Q2：密码过期判断的逻辑是什么？为什么选择 3 个月作为过期时间？
A：过期判断逻辑：
- 核心字段：依赖数据库的 PasswordChangeTime 字段，NULL 视为从未修改，直接判定过期
- 时间计算：当前时间 - PasswordChangeTime > 90 天（3 个月）则判定过期
- 判断时机：密码验证通过后、登录成功前，避免密码错误时也提示过期（提升安全性）

3 个月过期时间的选择：
- 业务层面：医疗美容行业需兼顾安全性和用户体验，3 个月是行业常见的密码过期周期
- 安全层面：过长（如 6 个月）会增加账号被盗风险，过短（如 1 个月）会频繁打扰用户
- 合规要求：满足医疗行业数据安全合规要求，定期密码修改可降低风险

#### Q3：匿名修改接口如何保证安全？防止恶意修改他人密码？
A：安全保障措施：
- 身份校验：必须提供 "公司编号 + 账号 + 旧密码" 三重验证，旧密码需与数据库存储的 MD5 值一致
- 接口限流：通过 IP 和账号维度限流，连续 5 次错误后锁定 10 分钟，防止暴力破解
- 日志审计：记录所有匿名修改操作，包含 IP 地址、操作时间、账号信息，便于追溯
- 密码强度校验：新密码必须满足 "长度≥8 位 + 包含数字、字母、特殊字符"，防止弱密码
- 传输加密：所有请求通过 HTTPS 传输，避免数据被抓包窃取

### 3. 审计日志功能相关
#### Q1：为什么选择拦截器 / 过滤器实现日志记录，而不是 AOP？
A：技术选型考量：
- 兼容性：项目是老系统迭代，部分模块未使用 Spring AOP，拦截器 / 过滤器无需修改原有代码，兼容性更好
- 易用性：拦截器可直接获取 HttpServletRequest 对象，方便提取 IP、接口路径、参数等信息，无需额外配置
- 性能：拦截器在 Spring MVC 流程中执行，比 AOP 的动态代理性能损耗更低，尤其适合高并发接口
- 功能适配：日志记录需要区分请求类型（HTTP 请求），拦截器天然适合处理 Web 请求，AOP 更适合业务方法的横切逻辑

如果是新项目，可能会选择 AOP（如 @AspectJ），因为它更灵活，可通过注解精准控制需要记录日志的方法，但在老系统迭代场景下，拦截器 / 过滤器是更稳妥的选择。

#### Q2：异步记录日志如何保证不丢失？如果日志保存失败怎么办？
A：不丢失保障方案：
- 队列缓冲：使用内存队列（如 LinkedBlockingQueue）暂存日志，异步线程从队列中消费，避免直接操作数据库导致的阻塞
- 批量提交：积累一定数量（如 100 条）或固定时间（如 5 秒）批量插入数据库，提升性能
- 失败重试：日志保存失败时，放入重试队列，重试 3 次，仍失败则写入本地文件
- 文件兜底：本地文件按日期分割，定时任务扫描文件，将未入库的日志重新导入数据库

异常处理：
- 日志记录失败不影响主业务：异步线程捕获所有异常，不抛出到主流程
- 告警通知：连续失败 5 次以上，发送邮件告警给管理员
- 日志分级：操作日志为普通级别，登录日志为重要级别，重要日志失败时重试次数更多

#### Q3：敏感信息脱敏的具体实现？如何避免脱敏过度或不足？
A：脱敏实现方案：
- 字段识别：通过关键字匹配（如 password、pwd、token、mobile、idcard）识别敏感字段
- 脱敏策略：
  - 密码 / Token：直接替换为 "***"
  - 手机号：保留前 3 位和后 4 位，中间替换为 "**"（如 138**1234）
  - 身份证号：保留前 6 位和后 4 位，中间替换为 "*********"
  - 地址：保留省份和城市，详细地址替换为 "***"
- 实现方式：自定义脱敏工具类，在日志记录前对请求参数进行处理，支持 JSON 格式和普通参数

避免脱敏过度 / 不足：
- 配置化：脱敏规则通过配置文件定义，可灵活调整
- 分级脱敏：不同环境（开发 / 测试 / 生产）使用不同脱敏级别，生产环境脱敏更严格
- 日志预览：开发环境支持日志预览，验证脱敏效果
- 定期审计：检查生产环境日志，确保敏感信息未泄露，同时不影响审计分析

### 4. Bug 修复相关
#### Q1：业绩报表移动端数据缺失的根本原因是什么？如何定位到这个问题的？

#### Q2：修复入库金额异常问题时，为什么要调整执行顺序（先验证再保存入库单）？

- 原流程：先保存入库单（TotalAmount=0）→ 处理明细 → 更新入库单总额

调整后的优势：
- 先验证所有明细数据，确保无格式错误、必填项完整，避免后续异常
- 先计算总额，确保总额 > 0 后再保存入库单，避免无效入库单
- 入库单和明细在同一个事务中，要么同时成功，要么同时回滚，保证数据一致性
- 减少数据库操作次数（先计算总额，避免后续更新），提升性能
