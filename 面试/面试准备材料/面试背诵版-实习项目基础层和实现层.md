#### 题目1：请简单介绍一下Excel商品批量导入功能的核心需求是什么？这个功能主要解决什么业务问题？

**解决的业务问题：**
1. **效率问题**：传统手动录入效率低，批量导入一次性处理多条数据，提升效率60%

3. **操作便利性**：供应商通常提供Excel格式，直接导入避免二次录入

#### 题目3：为什么选择支持.xls和.xlsx两种格式？这两种格式有什么区别？

**背诵答案：**

**为什么支持两种格式：**
1. **用户兼容性**：不同用户使用不同Excel版本（Office 2003用.xls，2007+用.xlsx）
2. **技术便利性**：Apache POI的`WorkbookFactory.create()`自动识别格式，无需手动区分
3. **业务场景**：不同门店、供应商可能使用不同版本

**格式区别：**
| 对比维度 | .xls（Excel 97-2003） | .xlsx（Excel 2007+） |
|---------|---------------------|---------------------|
| 文件格式 | 二进制格式（BIFF） | 基于XML的压缩格式（OOXML） |
| 行数限制 | 65536行 | 1048576行 |
| POI实现 | `HSSFWorkbook` | `XSSFWorkbook` |

**技术实现：** 使用`WorkbookFactory.create(inputStream)`自动识别，统一接口处理

#### 题目4：你提到的"模糊+精确两步商品匹配策略"是什么意思？为什么要设计两步匹配？

**背诵答案：**

**两步匹配策略：**

**第一步：数据库模糊匹配（缩小范围）**
- SQL: `WHERE name LIKE '%商品名称%' AND compId = ?`
- 目的：快速缩小候选范围，利用数据库索引

**第二步：内存精确匹配（去除空格）**
- 去除空格后比较：`trim(name).equals(trim(excelName))`
- 同时匹配名称和规格
- 目的：处理空格差异，确保精确匹配

**第三步：相似商品收集**
- 匹配失败时收集候选商品，用于错误提示

**为什么设计两步：**
1. **性能优化**：先用LIKE缩小范围（如10万条→10条），再内存匹配，避免全表扫描
2. **容错性**：处理Excel中的空格差异（首尾空格、多个连续空格）
3. **准确性**：保证商品名称和规格完全匹配，避免错误商品入库
4. **用户体验**：匹配失败时提供相似商品推荐

#### 题目5：如果用户上传的Excel文件格式不正确（比如表头不对、列顺序不对），系统应该如何处理？

**背诵答案：**

**表头识别机制（顺序无关）：**

**容错性：** 支持表头小幅差异（空格、大小写等），通过trim()和equalsIgnoreCase()提高匹配成功率

#### 题目6：在导入过程中，如果100条数据中有10条匹配失败，系统应该如何处理？是全部回滚还是部分成功？

**背诵答案：**

**处理策略：部分成功模式（逐行验证，单行错误不影响其他行）**

**核心原则：**
1. 逐行解析和验证：每行数据独立处理
2. 单行错误不影响其他行：某行失败不影响其他正常数据
3. 收集所有错误：统一返回给前端
4. 返回成功和失败列表：分别返回成功列表和失败列表

**数据结构：**
```java
ImportResult {
    successList: 成功导入的商品列表
    errorList: 失败的商品列表（包含错误信息）
    totalRows: 总行数
    successCount: 成功数量
    errorCount: 失败数量
}
```

**为什么部分成功而不是全部回滚：**
- **用户体验**：90条成功导入，只需处理10条错误，效率高
- **业务场景**：批量导入数据相对独立，部分失败不会导致数据不一致
- **实际应用**：供应商清单可能有部分商品不存在，允许部分成功可快速完成大部分工作

### 实现层：技术选型与代码实现

#### 题目10：为什么选择Apache POI来处理Excel文件？有没有考虑过其他方案（如EasyExcel、JXL）？它们的优缺点是什么？

**背诵答案：**

**技术选型对比：**

**1. Apache POI（最终选择）**
- **优点**：功能全面、格式支持完整（.xls+.xlsx）、成熟稳定、企业级应用广泛
- **缺点**：内存占用高（全量加载）、性能相对较差（大文件）、API相对复杂
- **适用场景**：中小型文件（<10万行）、需要完整Excel功能支持

**2. EasyExcel（阿里巴巴开源）**
- **优点**：内存占用低（流式读取）、性能优秀（大文件）、API简洁
- **缺点**：功能相对有限、社区相对较小
- **适用场景**：大文件导入导出（>10万行）、对内存有严格要求

**3. JXL（Java Excel API）**
- **缺点**：只支持.xls格式、项目已停止维护（2010年后）
- **结论**：不推荐使用，已过时

**为什么选择Apache POI：**
1. **业务需求匹配**：需要支持.xls和.xlsx，数据量100-1000条
2. **技术成熟度**：业界标准，团队熟悉度高，文档完善
3. **项目实际情况**：项目已使用Apache POI，保持技术栈统一
4. **功能需求**：未来可能需要导出功能，POI功能完整性有保障

#### 题目11：Apache POI处理.xls和.xlsx格式有什么区别？在代码中是如何区分和处理的？

**背诵答案：**

**底层实现区别：**

**1. .xls格式（Excel 97-2003）**
- 实现类：`HSSFWorkbook`（Horrible Spreadsheet Format）
- 文件格式：二进制格式（BIFF）
- 依赖包：`poi`（核心包）
- 行数限制：65536行

**2. .xlsx格式（Excel 2007+）**
- 实现类：`XSSFWorkbook`（XML Spreadsheet Format）
- 文件格式：基于XML的压缩格式（OOXML）
- 依赖包：`poi-ooxml`（需要额外依赖）
- 行数限制：1048576行

**代码处理方式：使用WorkbookFactory自动识别（推荐）**

```java
// 自动识别格式，无需手动区分
Workbook workbook = WorkbookFactory.create(inputStream);

// 后续代码完全统一，无需区分格式
Sheet sheet = workbook.getSheetAt(0);
Row headerRow = sheet.getRow(0);
// 统一的解析逻辑
```

**WorkbookFactory工作原理：**
1. 读取文件头（Magic Number）判断格式
2. 根据文件头自动选择HSSFWorkbook或XSSFWorkbook
3. 返回统一的Workbook接口

**设计优势：**
- 代码统一：一套代码处理两种格式
- 自动识别：无需手动判断文件类型
- 易于维护：只需维护一套解析逻辑

#### 题目12：你的代码架构是怎么设计的？Controller、Service、工具类各层的职责是什么？为什么要这样分层？

**背诵答案：**

**整体架构：**
```
Controller层 → Service层 → 工具类层 → Repository层 → 数据库
```

**各层职责：**

**1. Controller层（控制器层）**
- **职责**：接收HTTP请求、参数校验、Token验证和权限检查、调用Service层、统一返回结果
- **不应该做**：业务逻辑处理、Excel解析、数据库操作
- **示例**：文件上传接收、IsGeneral权限检查、调用Service

**2. Service层（业务逻辑层）**
- **职责**：业务逻辑处理、文件校验、调用工具类解析Excel、构建返回结果、异常处理
- **不应该做**：HTTP请求处理、Excel底层解析、数据库直接操作
- **示例**：文件格式和大小校验、调用ExcelImportUtils、结果转换

**3. 工具类层（ExcelImportUtils）**
- **职责**：Excel文件解析、表头识别和验证、数据行解析和转换、商品匹配逻辑、数据验证
- **特点**：可复用，总部和门店入库共用

**4. Repository层（数据访问层）**
- **职责**：商品查询、数据库操作

**为什么这样分层：**
1. **单一职责原则**：每层只负责自己的职责，代码清晰
2. **依赖倒置原则**：上层依赖下层接口，降低耦合
3. **代码复用**：工具类层可被多个Service复用
4. **易于测试**：每层可独立测试
5. **易于维护**：修改某层不影响其他层

**设计优势：**
- 职责清晰，易于理解和维护
- 代码复用，降低维护成本
- 便于扩展和测试

#### 题目15：商品匹配时如果数据库中有多个相似商品（名称相同但规格不同，或名称相似），如何选择最匹配的那个？

**背诵答案：**

**匹配策略（两步匹配）：**

**第一步：数据库模糊匹配（缩小范围）**
```sql
WHERE name LIKE '%商品名称%' AND compId = ?
```
- 返回所有名称包含关键字的商品（可能有多个）

**第二步：内存精确匹配（精确筛选）**
- **匹配规则**：同时匹配名称和规格
- **匹配条件**：`trim(name).equals(trim(excelName)) && trim(spec).equals(trim(excelSpec))`
- **结果**：只有名称和规格都完全匹配的商品才会被选中

**如果多个商品都匹配：**
- **理论上不应该出现**：如果商品名称和规格都相同，说明是同一个商品
- **如果出现**：可能是数据重复，系统会选择第一个匹配的商品
- **实际处理**：通常商品名称+规格是唯一组合，不会出现多个匹配

**如果匹配失败（没有完全匹配的商品）：**
- **收集相似商品**：收集模糊匹配的候选商品
- **错误提示**：返回"未找到匹配商品"，并提供相似商品列表供用户参考
- **用户操作**：用户需要手动选择或创建新商品

**匹配优先级：**
1. **完全匹配**：名称和规格都完全匹配（去除空格后）
2. **相似匹配**：名称相似但规格不同（用于错误提示）
3. **无匹配**：返回错误，提示用户

#### 题目16：如果导入过程中系统崩溃或网络中断，如何保证数据一致性？有没有使用事务？

**背诵答案：**

**当前实现：**
- **导入阶段**：只进行商品匹配和数据验证，**不涉及数据库写入**
- **保存阶段**：用户确认后，前端提交数据，后端保存入库单及明细
- **事务使用**：在**保存入库单**的Service方法上使用`@Transactional`

**数据一致性保证：**

**阶段1：Excel解析和匹配（无事务）**
```
Excel解析 → 商品匹配 → 数据验证 → 返回结果给前端
```
- 只查询数据库，不修改数据
- 不涉及事务，即使中断也不影响数据

**阶段2：用户确认后保存（有事务）**

```java
@Transactional(rollbackFor = Exception.class)

    repertoryenterService.save(repertoryenter);
    
    // 保存入库明细

}
```

**异常处理：**
- **导入阶段异常**：只影响本次导入，不影响数据库
- **保存阶段异常**：事务回滚，入库单和明细都不保存，保证数据一致性

**设计优势：**
- **分离关注点**：导入和保存分离，导入失败不影响数据库
- **用户体验**：用户可以查看匹配结果，确认后再保存
- **数据安全**：保存阶段使用事务，保证原子性

**如果系统崩溃：**
- **导入阶段崩溃**：用户重新上传即可，无数据丢失
- **保存阶段崩溃**：事务回滚，数据不会部分保存，用户重新提交即可

#### 题目1：密码过期强制修改功能的核心需求是什么？这个功能主要解决什么安全问题？

3. **内部威胁风险**：降低离职员工和内部泄露的风险

#### 题目6：密码加密方式是什么？为什么使用MD5？有没有考虑过其他加密方式？

**背诵答案：**

**当前加密方式：MD5（大写）**
- **加密算法**：MD5（Message Digest Algorithm 5）
- **处理方式**：MD5加密后转大写
- **存储方式**：数据库中存储MD5加密后的字符串

**为什么使用MD5：**
1. **历史原因**：系统早期设计使用MD5，保持一致性
2. **简单快速**：MD5计算速度快，适合登录验证场景
3. **项目现状**：已有大量用户数据使用MD5，迁移成本高

**MD5的局限性：**
1. **安全性不足**：MD5已被破解，存在碰撞攻击风险
2. **不可逆但可破解**：虽然不可逆，但可以通过彩虹表等方式破解
3. **不适合新项目**：现代项目不推荐使用MD5

**其他加密方式对比：**

| 加密方式 | 安全性 | 性能 | 适用场景 |
|---------|--------|------|---------|
| **MD5** | ⚠️ 低（已破解） | ✅ 快 | 不推荐新项目使用 |
| **SHA-256** | ✅ 高 | ✅ 快 | 单向加密，适合密码存储 |
| **BCrypt** | ✅ 高 | ⚠️ 慢（可配置） | 推荐，带盐值，抗彩虹表 |
| **Argon2** | ✅ 最高 | ⚠️ 慢 | 最新标准，抗GPU破解 |

**优化建议（如果未来需要）：**
- **短期**：继续使用MD5，但添加盐值（Salt）增强安全性
- **长期**：逐步迁移到BCrypt或Argon2
- **迁移策略**：新用户使用新算法，旧用户登录时逐步迁移

- **含义2**：密码已过期，需要强制修改
- **含义3**：系统升级前的历史数据（兼容性考虑）

**判断逻辑：**
```csharp
// 过期条件：
// 1. PasswordChangeTime == null（从未修改过，视为过期）
// 2. PasswordChangeTime < DateTime.Now.AddMonths(-3)（超过3个月）

if (emp.PasswordChangeTime == null || 
    emp.PasswordChangeTime < DateTime.Now.AddMonths(-3)) {
    // 密码已过期
}
```

**设计优势：**
- **语义清晰**：NULL明确表示"从未修改过"
- **逻辑简单**：一个条件判断即可，无需额外字段
- **兼容性好**：兼容历史数据，无需数据迁移

**如果设计为不可空：**
- 需要为历史数据设置默认值（如创建时间）
- 无法区分"从未修改过"和"已修改过但过期"
- 需要额外的字段或逻辑判断

#### 题目10：如果系统中有多个后端（.NET和Java），密码过期功能如何保证一致性？数据如何同步？

**背诵答案：**

**系统架构：**
- **.NET后端**：处理登录接口、密码修改接口
- **Java后端**：处理其他业务接口
- **共享数据库**：MySQL数据库，两个后端共享

**一致性保证机制：**

**1. 数据库层面（核心保障）**
- **共享数据库**：两个后端访问同一个MySQL数据库
- **同一张表**：`employees`表，包含`PasswordChangeTime`字段
- **数据一致性**：数据库层面保证数据一致性，两个后端读取的是同一份数据

**2. 业务逻辑层面**
- **.NET后端负责**：登录验证、密码过期判断、密码修改
- **Java后端**：不涉及密码过期逻辑，只读取用户信息
- **职责清晰**：密码相关逻辑集中在.NET后端，避免逻辑分散

**数据同步机制：**

**场景1：密码修改后，两个后端如何同步？**
```
用户修改密码（.NET后端）
    ↓
更新employees表（PasswordChangeTime = 当前时间）
    ↓
Java后端下次查询时，自动获取最新数据
    ↓
无需额外同步，数据库保证一致性
```

**场景2：登录验证时的一致性**
```
.NET后端登录验证
    ↓
查询employees表（包含PasswordChangeTime）
    ↓
判断密码是否过期
    ↓
Java后端不涉及密码过期判断
    ↓
数据一致性由数据库保证
```

**设计优势：**
- **数据一致性**：数据库层面保证，两个后端读取同一份数据
- **逻辑集中**：密码相关逻辑集中在.NET后端，易于维护
- **无需额外同步**：不需要消息队列或缓存同步，简单可靠

**潜在问题和解决方案：**

**问题1：缓存不一致（如果使用缓存）**
- **问题**：如果.NET后端更新了密码，Java后端的用户缓存可能过期
- **解决**：密码修改时清除相关缓存，或设置较短的缓存过期时间

**问题2：分布式事务（如果需要）**
- **当前场景**：密码修改只涉及单表操作，无需分布式事务
- **如果未来需要**：可以使用分布式事务或最终一致性方案

**最佳实践：**
- ✅ **共享数据库**：两个后端共享数据库，保证数据一致性
- ✅ **职责分离**：密码相关逻辑集中在.NET后端
- ✅ **避免缓存**：密码相关数据不缓存，或设置短过期时间
- ✅ **统一接口**：密码修改统一通过.NET后端接口

#### 题目2：审计日志功能记录哪些类型的日志？登录日志和操作日志有什么区别？分别记录什么信息？

**背诵答案：**

**记录两种类型的日志：**
1. **登录日志（Login Log）**：记录用户的登录行为（成功/失败/登出）
2. **操作日志（Operation Log）**：记录用户的业务操作（新增/编辑/删除/导出/下载/上传）

**登录日志 vs 操作日志：**

| 维度 | 登录日志 | 操作日志 |
|------|---------|---------|
| **记录对象** | 身份认证行为 | 业务操作行为 |
| **记录时机** | 登录/登出时 | 业务操作时 |
| **记录方式** | 主动调用 | 拦截器自动记录 |
| **主要字段** | 登录账号、登录类型、登录IP、浏览器信息 | 操作类型、接口路径、操作描述、操作结果 |

**登录日志记录信息：**
- 用户信息：操作人ID、姓名、登录账号
- 门店信息：门店ID、门店名称
- 登录状态：登录类型（1-成功，2-失败，3-登出）
- 登录环境：IP地址、浏览器信息
- 失败信息：失败原因（仅登录失败时）

**操作日志记录信息：**
- 用户信息：操作人ID、姓名
- 门店信息：门店ID、门店名称
- 操作信息：操作类型（1-新增，2-编辑，3-删除，5-导出，6-下载，7-上传）
- 接口信息：接口路径、HTTP方法
- 操作结果：操作结果（1-成功，2-失败）、错误信息
- 性能信息：操作耗时

#### 题目3：登录日志记录成功、失败、登出三种状态，这三种状态分别在什么场景下触发？如何区分？

**背诵答案：**

**三种状态的触发场景：**

**1. 登录成功（login_type = 1）**
- **触发时机**：登录接口验证通过后
- **触发条件**：账号存在、密码正确、账号未锁定
- **记录信息**：完整的用户信息、门店信息、登录环境

**2. 登录失败（login_type = 2）**
- **触发时机**：登录接口验证失败时
- **触发条件**：密码错误、账号不存在、账号锁定等
- **记录信息**：账号、失败原因、登录环境（不包含用户信息）

**3. 登出（login_type = 3）**
- **触发时机**：登出接口被调用时
- **触发条件**：用户主动退出系统
- **记录信息**：完整的用户信息、门店信息、登出环境

**如何区分：**
- **通过login_type字段**：1-成功，2-失败，3-登出
- **通过数据完整性**：登录失败时缺少用户信息（emp_id、emp_name为NULL）
- **通过fail_reason字段**：登录失败时有失败原因，成功和登出为NULL

#### 题目4：操作日志记录增删改查、导出、上传等操作，这些操作是如何被识别和记录的？是自动记录还是手动记录？

- 无侵入性，业务代码完全不需要修改

**操作类型识别逻辑（优先级从高到低）：**
1. **删除操作**：包含/delete, /remove, /del, /zuofei, /cancel
2. **导出操作**：包含/export
3. **下载操作**：包含/download
4. **上传/导入操作**：包含/upload, /import
5. **编辑操作**：包含/update, /edit, /changeStatus, /set
6. **新增操作**：包含/save, /add, /create
7. **默认**：根据HTTP方法判断（POST→新增，PUT→编辑，DELETE→删除）

**接口过滤：**
- **需要记录**：包含操作关键字的接口
- **排除**：查询接口（/index, /show, /get**, /query**）、日志查询接口、静态资源

#### 题目5：你提到的"敏感信息脱敏"是什么意思？哪些信息需要脱敏？脱敏的规则是什么？

**背诵答案：**

**敏感信息脱敏的含义：**
- 在记录操作日志时，对包含敏感信息的字段进行隐藏或替换处理
- 避免敏感信息泄露，保护用户隐私和系统安全

**需要脱敏的信息：**
1. **密码类**：password, pwd, oldPassword, newPassword
2. **令牌类**：token, accessToken, refreshToken

**脱敏规则：**
1. **URL参数和表单参数**：直接跳过敏感字段，不记录到日志中
2. **JSON请求体**：将敏感字段的值替换为掩码（如"******"）
3. **限制长度**：单个参数值限制200字符，总长度限制2000字符

**脱敏示例：**
- 脱敏前：`{"password": "123456", "token": "abc123"}`
- 脱敏后：`{"password": "******", "token": "******"}`

#### 题目6：为什么使用异步存储？异步存储失败怎么办？

```java
// Java后端
CompletableFuture.runAsync(() -> {

        operationLogService.save(log);

});
```

**异步存储失败处理：**
- **捕获异常**：使用try-catch捕获异常
- **记录错误日志**：记录到系统日志，便于排查
- **不影响业务**：日志记录失败不影响主业务流程
- **可加入重试机制**：未来可以加入重试或消息队列（扩展）

#### 题目7：操作日志是如何通过拦截器自动记录的？拦截器的工作流程是什么？

**背诵答案：**

**拦截器实现：**
- 使用`OperationLogInterceptor`实现`HandlerInterceptor`接口
- 在`preHandle`中记录请求开始时间
- 在`afterCompletion`中记录操作日志

**工作流程：**
```
1. preHandle: 记录请求开始时间
2. Controller处理业务逻辑
3. afterCompletion: 记录操作日志
   ├─ 判断是否需要记录（排除查询接口）
   ├─ 获取用户信息（从JWT Token）
   ├─ 识别操作类型（根据接口路径和HTTP方法）
   ├─ 生成操作描述
   ├─ 获取IP地址
   ├─ 获取请求参数（脱敏处理）
   ├─ 计算操作耗时
   ├─ 判断操作结果（成功/失败）
   └─ 异步保存到数据库
```

**拦截器配置：**
- 拦截路径：`/manager/**`, `/intranet/**`
- 排除路径：查询接口、日志查询接口、静态资源

1. X-Forwarded-For（代理场景）

#### 题目9：登录日志是如何记录的？.NET后端和Java后端是如何协作的？

**背诵答案：**

**记录流程：**
```
1. .NET后端登录接口
2. 登录成功/失败
3. 调用LoginLogHelper.RecordLoginSuccess/Fail()
4. 异步发送HTTP请求到Java后端
5. Java后端接收请求，调用LoginLogUtils.recordLoginSuccess/Fail()
6. 异步查询员工姓名和门店名称
7. 构建SysLoginLog对象
8. 保存到数据库
```

**跨后端通信：**
- .NET后端通过HTTP POST调用Java后端接口
- 接口路径：`/syslog/loginLog/record`
- 使用`Task.Run`异步发送请求，不阻塞登录流程

**设计原因：**
- 登录接口在.NET后端，但日志存储在Java后端的数据库
- 通过HTTP调用实现跨后端通信
- 异步处理保证不阻塞登录流程

#### 题目11：如果日志数据量很大（比如每天产生10万条日志），如何保证查询性能？

**背诵答案：**

**性能优化策略：**

**1. 数据库索引优化（基础）**
```sql
-- 登录日志表索引
CREATE INDEX idx_login_time ON sys_login_log(login_time);
CREATE INDEX idx_emp_id ON sys_login_log(emp_id);
CREATE INDEX idx_login_type ON sys_login_log(login_type);

-- 操作日志表索引
CREATE INDEX idx_operation_time ON sys_operation_log(operation_time);
CREATE INDEX idx_emp_id ON sys_operation_log(emp_id);
CREATE INDEX idx_operation_type ON sys_operation_log(operation_type);
CREATE INDEX idx_comp_id ON sys_operation_log(comp_id);
```
- **时间索引**：支持按时间范围查询
- **用户索引**：支持按用户查询
- **类型索引**：支持按操作类型查询

**2. 分表策略（数据量大时）**
- **按时间分表**：按月或按年分表（如`sys_operation_log_2024_01`）
- **按公司分表**：不同公司的日志存储在不同表
- **优点**：减小单表数据量，提升查询性能
- **缺点**：查询逻辑变复杂，需要跨表查询

**3. 数据归档策略**
- **热数据**：最近3个月的日志保留在业务表
- **冷数据**：3个月前的日志归档到历史表或文件
- **查询策略**：优先查询热数据，需要历史数据时再查询归档数据

- 只记录错误日志，不中断业务

#### 题目15：为什么选择Spring拦截器而不是AOP切面来实现操作日志？这两种方案有什么区别？

**背诵答案：**

**技术方案对比：**

| 对比维度 | Spring拦截器 | AOP切面 |

| **精确控制** | ⚠️ 相对粗粒度（按路径拦截） | ✅ 精确（可以精确到方法） |
| **性能** | ✅ 性能好（拦截器链） | ⚠️ 相对较慢（代理对象） |
| **适用场景** | ✅ HTTP请求拦截 | ✅ 方法调用拦截 |

**为什么选择Spring拦截器：**

**1. 无需新依赖**
- Spring MVC自带拦截器功能
- 无需引入额外的AOP依赖
- 减少项目依赖复杂度

**2. 配置简单**
```java
// 拦截器配置（简单）
registry.addInterceptor(operationLogInterceptor)
    .addPathPatterns("/manager/**", "/intranet/**")
    .excludePathPatterns("/**/index", "/**/show");
```

**3. 与现有模式一致**
- 项目中已有其他拦截器（如认证拦截器）
- 使用拦截器保持技术栈统一
- 团队熟悉度高

**4. 性能优势**
- 拦截器在请求处理链中，性能开销小
- 无需创建代理对象，性能更好

**AOP切面的优势（如果使用）：**
- **精确控制**：可以精确到方法级别
- **灵活配置**：可以使用注解方式标记需要记录日志的方法
- **业务解耦**：日志逻辑与业务逻辑完全分离

**当前选择的原因：**
- ✅ **项目现状**：项目已使用拦截器模式
- ✅ **需求匹配**：按路径拦截已满足需求
- ✅ **简单可靠**：配置简单，易于维护
- ✅ **性能考虑**：拦截器性能更好

**如果使用AOP切面（示例）：**
```java
// 使用注解标记需要记录日志的方法
@OperationLog(operationType = OperationType.ADD, description = "新增预约")

    // 业务逻辑
}

// AOP切面自动拦截
@Aspect
@Component

    @Around("@annotation(operationLog)")
    public Object around(ProceedingJoinPoint point, OperationLog operationLog) {
        // 记录操作日志
    }
}
```

**总结：**
- **选择拦截器**：简单、无需新依赖、性能好、与现有模式一致
- **AOP切面**：更精确、更灵活，但需要额外依赖和配置

#### 题目2：Cache-Aside策略是什么？为什么选择这个策略而不是Write-Through或Write-Back？

**背诵答案：**

**Cache-Aside策略（旁路缓存策略）：**
- **读流程**：
  1. 先查缓存，缓存命中直接返回
  2. 缓存未命中，查数据库
  3. 将数据库结果写入缓存
- **写流程**：
  1. 先更新数据库
  2. 删除缓存（让下次查询时重建缓存）

**为什么选择Cache-Aside：**
1. **实现简单**：逻辑清晰，易于理解和维护
2. **灵活性高**：可以灵活控制缓存更新时机
3. **适合读多写少场景**：店铺信息读多写少，Cache-Aside性能好
4. **数据一致性**：通过删除缓存保证一致性，避免脏数据

**与其他策略对比：**
| 策略 | 写流程 | 优点 | 缺点 | 适用场景 |
|------|--------|------|------|----------|
| **Cache-Aside** | 更新DB → 删除缓存 | 实现简单，灵活性高 | 可能短暂不一致 | 读多写少 ✅ |
| **Write-Through** | 更新DB → 更新缓存 | 数据一致性强 | 写性能差 | 写多读少 |
| **Write-Back** | 更新缓存 → 异步写DB | 写性能好 | 数据可能丢失 | 写多读少，可容忍丢失 |

**代码示例：**
```java
// 读流程（Cache-Aside）
public Shop queryById(Long id) {
    // 1. 先查缓存
    String json = stringRedisTemplate.opsForValue().get(CACHE_SHOP_KEY + id);
    if (StrUtil.isNotBlank(json)) {
        return JSONUtil.toBean(json, Shop.class);  // 缓存命中
    }
    // 2. 缓存未命中，查数据库
    Shop shop = getById(id);
    if (shop != null) {
        // 3. 写入缓存
        stringRedisTemplate.opsForValue().set(CACHE_SHOP_KEY + id, 
            JSONUtil.toJsonStr(shop), CACHE_SHOP_TTL, TimeUnit.MINUTES);
    }
    return shop;
}

// 写流程（Cache-Aside）
public Result update(Shop shop) {
    // 1. 更新数据库
    updateById(shop);
    // 2. 删除缓存
    stringRedisTemplate.delete(CACHE_SHOP_KEY + shop.getId());
    return Result.ok();
}
```

### 实现层：技术实现与核心逻辑

#### 题目3：缓存穿透是什么？你是如何解决的？缓存空值的TTL如何设置？

**背诵答案：**

**缓存穿透：**
- **定义**：查询不存在的数据，缓存和数据库都没有，导致每次请求都查数据库
- **危害**：大量无效请求直接打到数据库，可能导致数据库压力过大甚至崩溃
- **场景**：恶意请求不存在的店铺ID（如负数、超大ID）

**解决方案：缓存空值（Null Object Pattern）**
- **核心思路**：即使数据库查询为空，也写入缓存（空值），避免重复查询数据库
- **实现逻辑**：
  1. 查询缓存，命中则返回（包括空值）
  2. 缓存未命中，查数据库
  3. 数据库为空，写入空值到缓存（设置较短TTL）
  4. 数据库有数据，写入正常数据到缓存

**代码实现：**
```java
public <R, ID> R queryWithPassThrough(
        String keyPrefix, ID id, Class<R> type, 
        Function<ID, R> dbFallback, Long time, TimeUnit unit) {
    String key = keyPrefix + id;
    // 1. 从Redis查询缓存
    String json = stringRedisTemplate.opsForValue().get(key);
    // 2. 判断是否存在
    if (StrUtil.isNotBlank(json)) {
        // 缓存命中，直接返回
        return JSONUtil.toBean(json, type);
    }
    // 3. 判断命中的是否是空值（空字符串）
    if (json != null) {  // json == "" 表示缓存了空值
        return null;  // 返回null，表示数据不存在
    }
    
    // 4. 缓存未命中，查询数据库
    R r = dbFallback.apply(id);
    // 5. 数据库不存在，缓存空值
    if (r == null) {
        // 将空值写入Redis，设置较短TTL（2分钟）
        stringRedisTemplate.opsForValue().set(key, "", 
            CACHE_NULL_TTL, TimeUnit.MINUTES);  // CACHE_NULL_TTL = 2
        return null;
    }
    // 6. 数据库存在，写入正常缓存
    this.set(key, r, time, unit);
    return r;
}
```

**空值TTL设置：**
- **设置较短TTL**：通常2-5分钟（`CACHE_NULL_TTL = 2`）
- **原因**：
  1. 防止恶意请求长期占用缓存空间
  2. 如果数据后来被创建，较短TTL可及时更新
  3. 平衡缓存效果和存储成本

**其他防护措施（可选）：**
- **参数校验**：前端和后端都校验ID合法性（如ID > 0）
- **布隆过滤器**：使用布隆过滤器预判数据是否存在（适合大规模场景）

#### 题目4：缓存击穿是什么？你使用了哪两种方案解决？它们有什么区别？

**背诵答案：**

**缓存击穿：**
- **定义**：热点数据缓存过期，大量请求同时查询数据库
- **危害**：瞬间大量请求打到数据库，可能导致数据库压力过大
- **场景**：热门店铺缓存过期，大量用户同时访问

**解决方案1：互斥锁（Mutex Lock）**
- **核心思路**：缓存过期时，只允许一个线程查数据库，其他线程等待
- **实现逻辑**：
  1. 缓存未命中，尝试获取分布式锁
  2. 获取锁成功，查数据库并重建缓存
  3. 获取锁失败，等待后重试或返回旧数据
- **优点**：保证数据一致性，逻辑简单
- **缺点**：等待的线程可能阻塞，用户体验稍差

**解决方案2：逻辑过期（Logical Expiration）**
- **核心思路**：缓存不设置TTL，而是存储逻辑过期时间，过期后异步重建
- **实现逻辑**：
  1. 缓存中存储数据 + 逻辑过期时间
  2. 查询时判断是否逻辑过期
  3. 未过期直接返回
  4. 已过期，获取锁后异步重建缓存，返回旧数据
- **优点**：用户无需等待，体验好
- **缺点**：可能返回过期数据（短暂不一致）

**代码实现对比：**

**互斥锁方案：**
```java
public <R, ID> R queryWithMutex(
        String keyPrefix, ID id, Class<R> type, 
        Function<ID, R> dbFallback, Long time, TimeUnit unit) {
    String key = keyPrefix + id;
    // 1. 查缓存
    String json = stringRedisTemplate.opsForValue().get(key);
    if (StrUtil.isNotBlank(json)) {
        return JSONUtil.toBean(json, type);  // 缓存命中
    }
    
    // 2. 缓存未命中，获取互斥锁
    String lockKey = LOCK_SHOP_KEY + id;
    boolean isLock = tryLock(lockKey);
    if (!isLock) {
        // 获取锁失败，休眠后重试
        Thread.sleep(50);
        return queryWithMutex(keyPrefix, id, type, dbFallback, time, unit);
    }
    
    try {
        // 3. 获取锁成功，查数据库
        R r = dbFallback.apply(id);
        if (r == null) {
            // 缓存空值
            stringRedisTemplate.opsForValue().set(key, "", 
                CACHE_NULL_TTL, TimeUnit.MINUTES);
            return null;
        }
        // 4. 写入缓存
        this.set(key, r, time, unit);
        return r;
    } finally {
        // 5. 释放锁
        unlock(lockKey);
    }
}
```

**逻辑过期方案：**
```java
public <R, ID> R queryWithLogicalExpire(
        String keyPrefix, ID id, Class<R> type, 
        Function<ID, R> dbFallback, Long time, TimeUnit unit) {
    String key = keyPrefix + id;
    // 1. 从Redis查询缓存
    String json = stringRedisTemplate.opsForValue().get(key);
    if (StrUtil.isBlank(json)) {
        return null;  // 缓存不存在，返回null（需要预热）
    }
    
    // 2. 反序列化为RedisData对象（包含数据和逻辑过期时间）
    RedisData redisData = JSONUtil.toBean(json, RedisData.class);
    R r = JSONUtil.toBean((JSONObject) redisData.getData(), type);
    LocalDateTime expireTime = redisData.getExpireTime();
    
    // 3. 判断是否逻辑过期
    if (expireTime.isAfter(LocalDateTime.now())) {
        // 未过期，直接返回
        return r;
    }
    
    // 4. 已过期，需要缓存重建
    String lockKey = LOCK_SHOP_KEY + id;
    boolean isLock = tryLock(lockKey);
    if (isLock) {
        // 获取锁成功，开启独立线程异步重建缓存
        CACHE_REBUILD_EXECUTOR.submit(() -> {
            try {
                // 查询数据库
                R newR = dbFallback.apply(id);
                // 重建缓存
                this.setWithLogicalExpire(key, newR, time, unit);
            } finally {
                unlock(lockKey);
            }
        });
    }
    // 5. 返回过期的数据（用户无需等待）
    return r;
}
```

**两种方案对比：**
| 对比维度 | 互斥锁方案 | 逻辑过期方案 |
|---------|-----------|------------|
| **用户体验** | 可能等待，体验稍差 | 无需等待，体验好 ✅ |
| **数据一致性** | 强一致性 ✅ | 短暂不一致 |
| **实现复杂度** | 简单 ✅ | 较复杂 |
| **适用场景** | 对一致性要求高 | 对用户体验要求高 ✅ |
| **缓存预热** | 不需要 | 需要（缓存必须预先存在） |

**项目选择：**
- **实际使用**：互斥锁方案（`queryWithMutex`）
- **原因**：保证数据一致性，实现简单，适合店铺信息查询场景

#### 题目5：缓存雪崩是什么？你是如何通过动态TTL解决的？

**背诵答案：**

**缓存雪崩：**
- **定义**：大量缓存同时过期，导致大量请求同时查询数据库
- **危害**：瞬间大量请求打到数据库，可能导致数据库崩溃
- **场景**：系统启动时批量加载缓存，设置相同TTL，同时过期

**解决方案：动态TTL（随机过期时间）**
- **核心思路**：为每个缓存设置不同的过期时间（基础TTL + 随机值），避免同时过期
- **实现逻辑**：
  1. 设置基础TTL（如30分钟）
  2. 添加随机值（如0-10分钟）
  3. 实际TTL = 基础TTL + 随机值（30-40分钟）

**代码实现：**
```java
// 设置缓存时，使用动态TTL
public void set(String key, Object value, Long time, TimeUnit unit) {
    // 基础TTL
    long baseTTL = unit.toSeconds(time);
    // 随机值（0-10分钟）
    long randomTTL = ThreadLocalRandom.current().nextLong(0, 600);
    // 实际TTL = 基础TTL + 随机值
    long actualTTL = baseTTL + randomTTL;
    
    stringRedisTemplate.opsForValue().set(key, 
        JSONUtil.toJsonStr(value), actualTTL, TimeUnit.SECONDS);
}
```

**动态TTL的优势：**
1. **避免同时过期**：不同缓存的过期时间分散，不会同时失效
2. **实现简单**：只需在设置缓存时添加随机值
3. **效果明显**：有效降低缓存雪崩风险

**其他防护措施（可选）：**
- **多级缓存**：本地缓存 + Redis缓存，双重保护
- **限流降级**：缓存失效时，对数据库查询进行限流
- **缓存预热**：系统启动时提前加载热点数据

**项目实践：**
- ✅ **动态TTL**：设置缓存时使用随机TTL
- ✅ **互斥锁**：缓存击穿时使用互斥锁，进一步保护数据库
- ✅ **空值缓存**：缓存穿透时缓存空值，减少无效查询

#### 题目2：滑动过期机制是什么？与固定过期有什么区别？为什么选择滑动过期？

**背诵答案：**

**滑动过期机制（Sliding Expiration）：**
- **定义**：每次用户请求时，如果Token有效，就自动刷新Token的过期时间
- **原理**：用户活跃时，Token有效期向后延长；用户不活跃时，Token自然过期
- **效果**：活跃用户无需重新登录，不活跃用户自动退出

**固定过期机制（Fixed Expiration）：**
- **定义**：Token在创建时设置固定过期时间，之后不再改变
- **原理**：无论用户是否活跃，Token都在固定时间后过期
- **效果**：用户可能在使用过程中突然被要求重新登录

**两种机制对比：**
| 对比维度 | 滑动过期 | 固定过期 |
|---------|---------|---------|
| **用户体验** | 活跃时无需重新登录 ✅ | 可能在使用中过期 |
| **安全性** | 不活跃用户自动退出 ✅ | 固定时间后强制退出 |
| **实现复杂度** | 需要每次请求刷新 | 实现简单 ✅ |
| **适用场景** | 用户活跃度高的应用 ✅ | 安全性要求极高的应用 |

**为什么选择滑动过期：**
1. **用户体验好**：用户在使用过程中不会突然被要求重新登录
2. **安全性合理**：不活跃用户自动退出，避免Token长期有效
3. **符合业务场景**：点评类应用用户可能长时间浏览，滑动过期更合适

**实现方式：**
- **每次请求时**：在拦截器中检查Token，如果有效则刷新过期时间
- **刷新时机**：在`preHandle`方法中，验证Token有效后立即刷新
- **刷新命令**：`stringRedisTemplate.expire(key, LOGIN_USER_TTL, TimeUnit.MINUTES)`

#### 题目3：Token是如何存储在Redis中的？为什么使用Hash结构而不是String？

**背诵答案：**

**Token存储方式：**
- **Key格式**：`login:token:{token值}`

- **存储内容**：用户信息（userId、nickName、icon等）

**代码实现：**
```java
// 登录时存储Token
public Result login(LoginFormDTO loginForm) {
    // ... 验证用户 ...
    
    // 1. 生成Token（使用UUID）
    String token = UUID.randomUUID().toString(true);

    UserDTO userDTO = BeanUtil.copyProperties(user, UserDTO.class);

        new HashMap<>(), CopyOptions.create()
            .setIgnoreNullValue(true)

                // 将Long类型转为String，避免序列化问题
```

**为什么使用Hash而不是String：**
1. **部分更新**：Hash支持部分字段更新，String需要整体替换

3. **查询效率**：Hash支持按字段查询，不需要反序列化整个对象
4. **灵活性**：可以单独更新某个字段（如用户昵称），无需重新存储整个对象

**Hash vs String对比：**
| 对比维度 | Hash | String（JSON） |
|---------|------|---------------|
| **部分更新** | 支持 ✅ | 不支持 |
| **内存占用** | 更节省 ✅ | 较大 |
| **查询效率** | 按字段查询 ✅ | 需要反序列化 |
| **实现复杂度** | 需要转换 | 简单 ✅ |

#### 题目4：滑动过期机制是如何实现的？拦截器的执行顺序是如何设计的？

**背诵答案：**

**滑动过期实现：**
- **实现位置**：`RefreshTokenInterceptor`拦截器
- **执行时机**：每次请求的`preHandle`方法中
- **刷新逻辑**：验证Token有效后，调用`expire`方法刷新过期时间

**完整实现代码：**
```java
public class RefreshTokenInterceptor implements HandlerInterceptor {
    private StringRedisTemplate stringRedisTemplate;
    
    @Override
    public boolean preHandle(HttpServletRequest request, 
                            HttpServletResponse response, 
                            Object handler) {
        // 1. 获取请求头中的Token
        String token = request.getHeader("authorization");
        if (StrUtil.isBlank(token)) {
            return true;  // 无Token，放行（由登录拦截器处理）
        }
        
        // 2. 基于Token获取Redis中的用户信息
        String key = LOGIN_USER_KEY + token;  // login:token:xxx
        Map<Object, Object> userMap = stringRedisTemplate.opsForHash().entries(key);
        
        // 3. 判断用户是否存在
        if (userMap.isEmpty()) {
            return true;  // Token不存在或已过期，放行
        }
        
        // 4. 将Hash数据转为UserDTO对象
        UserDTO userDTO = BeanUtil.fillBeanWithMap(userMap, new UserDTO(), false);
        
        // 5. 保存用户信息到ThreadLocal
        UserHolder.saveUser(userDTO);
        
        // 6. 刷新Token有效期（滑动过期核心逻辑）
        stringRedisTemplate.expire(key, LOGIN_USER_TTL, TimeUnit.MINUTES);
        
        // 7. 放行
        return true;
    }
    
    @Override
    public void afterCompletion(HttpServletRequest request, 
                               HttpServletResponse response, 
                               Object handler, 
                               Exception ex) {
        // 请求处理完成后，清理ThreadLocal，防止内存泄漏
        UserHolder.removeUser();
    }
}
```

**拦截器执行顺序设计：**
- **两个拦截器**：
  1. `RefreshTokenInterceptor`（order=0）：刷新Token，设置ThreadLocal
  2. `LoginInterceptor`（order=1）：验证用户是否登录

**执行顺序：**
```
请求到达
  ↓
RefreshTokenInterceptor.preHandle()  (order=0)
  - 获取Token
  - 从Redis查询用户信息
  - 设置ThreadLocal
  - 刷新Token有效期
  ↓
LoginInterceptor.preHandle()  (order=1)
  - 从ThreadLocal获取用户信息
  - 判断是否需要登录
  ↓
Controller方法执行
  ↓
LoginInterceptor.afterCompletion()  (order=1)
  ↓
RefreshTokenInterceptor.afterCompletion()  (order=0)
  - 清理ThreadLocal
```

**为什么这样设计：**
1. **RefreshTokenInterceptor先执行**：确保用户信息已设置到ThreadLocal
2. **LoginInterceptor后执行**：可以直接从ThreadLocal获取用户信息，无需重复查询Redis
3. **清理顺序相反**：后进先出，确保ThreadLocal正确清理

#### 题目5：ThreadLocal是如何管理用户上下文的？为什么要使用ThreadLocal？如何防止内存泄漏？

**背诵答案：**

**ThreadLocal实现：**
- **工具类**：`UserHolder`封装ThreadLocal操作
- **存储内容**：`UserDTO`对象（用户ID、昵称、头像等）
- **生命周期**：请求开始时设置，请求结束时清理

**代码实现：**
```java
public class UserHolder {
    // 使用ThreadLocal存储用户信息
    private static final ThreadLocal<UserDTO> tl = new ThreadLocal<>();
    
    // 保存用户信息
    public static void saveUser(UserDTO user) {
        tl.set(user);
    }
    
    // 获取用户信息
    public static UserDTO getUser() {
        return tl.get();
    }
    
    // 移除用户信息（防止内存泄漏）
    public static void removeUser() {
        tl.remove();
    }
}
```

**为什么使用ThreadLocal：**
1. **线程隔离**：每个请求在独立线程中处理，ThreadLocal保证线程安全
2. **代码简化**：业务代码无需传递用户参数，直接通过`UserHolder.getUser()`获取
3. **性能优势**：避免在方法参数中层层传递用户信息
4. **设计优雅**：符合"上下文模式"，代码更清晰

**使用示例：**
```java
// Controller层
@GetMapping("/me")
public Result me() {
    // 直接从ThreadLocal获取用户信息，无需从参数传递
    UserDTO user = UserHolder.getUser();
    return Result.ok(user);
}

// Service层
public void doSomething() {
    Long userId = UserHolder.getUser().getId();  // 直接获取用户ID
    // ... 业务逻辑 ...
}
```

**ThreadLocal内存泄漏问题：**
- **问题原因**：
  1. ThreadLocal的Key是弱引用，但Value是强引用
  2. 线程池中的线程长期存活，ThreadLocal的Value不会被GC
  3. 如果忘记调用`remove()`，Value会一直占用内存

**防止内存泄漏：**
1. **必须调用remove()**：请求处理完成后，在`afterCompletion`中清理
2. **使用try-finally**：确保即使发生异常也能清理
3. **拦截器统一管理**：在拦截器中统一设置和清理，避免遗漏

**正确的清理方式：**
```java
@Override
public void afterCompletion(HttpServletRequest request, 
                           HttpServletResponse response, 
                           Object handler, 
                           Exception ex) {
    // 关键：无论是否发生异常，都要清理ThreadLocal
    try {
        UserHolder.removeUser();
    } catch (Exception e) {
        log.error("清理用户上下文失败", e);
    }
}
```

**内存泄漏示例（错误）：**
```java
// ❌ 错误：忘记调用remove()
public void processRequest() {
    UserHolder.saveUser(userDTO);
    // ... 业务逻辑 ...
    // 忘记调用 UserHolder.removeUser();
    // 如果线程被线程池复用，userDTO会一直占用内存
}
```

**正确示例：**
```java
// ✅ 正确：使用try-finally确保清理
public void processRequest() {
    try {
        UserHolder.saveUser(userDTO);
        // ... 业务逻辑 ...
    } finally {
        UserHolder.removeUser();  // 确保清理
    }
}
```

**项目实践：**
- ✅ **拦截器统一管理**：在`RefreshTokenInterceptor`的`afterCompletion`中清理
- ✅ **异常安全**：使用try-catch确保即使异常也能清理
- ✅ **工具类封装**：`UserHolder`提供统一的get/set/remove方法

**背诵答案：**

**核心需求：**
1. **防止重复下单**：同一用户在同一秒杀活动中只能下单一次

- **场景**：秒杀活动中，用户快速点击多次，可能创建多个订单

  1. 网络延迟导致用户重复提交

  3. 没有加锁保护，查询和插入之间存在时间窗口

  1. 高并发下，多个线程同时读取库存（如都是10）
  2. 多个线程同时扣减库存（都执行 stock - 1）
  3. 最终库存可能变成负数
- **危害**：库存数据错误，无法履约，影响用户体验

**业务场景：**
- 秒杀活动：高并发下单场景
- 优惠券抢购：限量优惠券抢购
- 库存扣减：任何涉及库存扣减的场景

**解决方案：**
- **分布式锁**：防止重复下单（用户维度锁）
- **乐观锁**：防止超卖（数据库乐观锁）
- **原子操作**：Lua脚本保证操作的原子性

#### 题目3：用户维度的分布式锁是什么意思？锁的粒度是如何设计的？为什么选择用户维度？

**背诵答案：**

**用户维度的分布式锁：**
- **定义**：锁的Key以用户ID为维度，同一用户的操作串行化，不同用户的操作可以并行
- **锁Key设计**：`lock:order:{userId}`
- **效果**：同一用户同时只能有一个下单操作在执行

**锁粒度设计：**
- **用户维度锁**：`lock:order:{userId}` - 防止同一用户重复下单
- **商品维度锁**：`lock:stock:{voucherId}` - 防止同一商品库存并发扣减（可选）
- **全局锁**：`lock:seckill` - 所有操作串行化（性能差，不推荐）

**为什么选择用户维度：**
1. **业务需求**：业务规则是"一人一单"，用户维度锁正好满足需求
2. **性能优势**：不同用户的操作可以并行，性能好
3. **锁粒度合适**：既保证业务正确性，又不过度影响性能
4. **实现简单**：锁Key设计简单，易于理解和维护

**代码实现：**
```java
private void createVoucherOrder(VoucherOrder voucherOrder) {
    Long userId = voucherOrder.getUserId();
    Long voucherId = voucherOrder.getVoucherId();
    
    // 创建用户维度的锁对象
    RLock redisLock = redissonClient.getLock("lock:order:" + userId);
    
    // 尝试获取锁（非阻塞，立即返回）
    boolean isLock = redisLock.tryLock();
    if (!isLock) {
        log.error("不允许重复下单！");
        return;
    }
    
    try {
        // 1. 查询订单（检查是否重复下单）
        int count = query()
            .eq("user_id", userId)
            .eq("voucher_id", voucherId)
            .count();
        if (count > 0) {
            log.error("不允许重复下单！");
            return;
        }
        
        // 2. 扣减库存（乐观锁）
        boolean success = seckillVoucherService.update()
            .setSql("stock = stock - 1")
            .eq("voucher_id", voucherId)
            .gt("stock", 0)  // 乐观锁：库存必须大于0
            .update();
        if (!success) {
            log.error("库存不足！");
            return;
        }
        
        // 3. 创建订单
        save(voucherOrder);
        
    } finally {
        // 释放锁
        redisLock.unlock();
    }
}
```

**锁粒度对比：**
| 锁粒度 | 锁Key | 优点 | 缺点 | 适用场景 |
|--------|-------|------|------|---------|
| **用户维度** ✅ | `lock:order:{userId}` | 性能好，不同用户并行 | 同一用户串行 | 一人一单业务 ✅ |
| **商品维度** | `lock:stock:{voucherId}` | 库存扣减安全 | 所有用户串行，性能差 | 库存扣减 |
| **全局锁** | `lock:seckill` | 实现简单 | 性能极差，所有操作串行 | 不推荐 |

**设计优势：**
- ✅ **业务匹配**：用户维度锁正好满足"一人一单"的业务需求
- ✅ **性能优化**：不同用户的操作可以并行，不影响整体性能
- ✅ **实现简单**：锁Key设计简单，易于理解和维护

#### 题目4：Redis原子操作和乐观锁策略是如何结合使用的？它们分别解决什么问题？

**背诵答案：**

**两种策略的结合：**
- **分布式锁（Redisson）**：解决重复下单问题（用户维度锁）
- **乐观锁（数据库）**：解决超卖问题（库存扣减）

**分布式锁解决重复下单：**
- **问题**：同一用户可能创建多个订单
- **解决方案**：用户维度分布式锁，保证同一用户的操作串行化
- **实现**：Redisson的`RLock`，锁Key为`lock:order:{userId}`

**乐观锁解决超卖：**
- **问题**：库存可能被扣成负数
- **解决方案**：数据库乐观锁，使用`stock > 0`条件判断
- **实现**：MyBatis-Plus的`update().setSql("stock = stock - 1").gt("stock", 0)`

**完整实现代码：**
```java
private void createVoucherOrder(VoucherOrder voucherOrder) {
    Long userId = voucherOrder.getUserId();
    Long voucherId = voucherOrder.getVoucherId();
    
    // ========== 分布式锁：防止重复下单 ==========
    RLock redisLock = redissonClient.getLock("lock:order:" + userId);
    boolean isLock = redisLock.tryLock();
    if (!isLock) {
        return;  // 获取锁失败，可能是重复请求
    }
    
    try {
        // 1. 检查是否重复下单（在锁保护下）
        int count = query()
            .eq("user_id", userId)
            .eq("voucher_id", voucherId)
            .count();
        if (count > 0) {
            return;  // 已存在订单，重复下单
        }
        
        // ========== 乐观锁：防止超卖 ==========
        // 2. 扣减库存（乐观锁：原子操作 + 条件判断）
        boolean success = seckillVoucherService.update()
            .setSql("stock = stock - 1")  // 原子操作：库存减1
            .eq("voucher_id", voucherId)  // 条件：指定商品
            .gt("stock", 0)              // 乐观锁：库存必须大于0
            .update();
        
        if (!success) {
            // 扣减失败：可能是库存不足（stock <= 0）
            // 或者并发冲突（其他线程已扣减）
            return;
        }
        
        // 3. 创建订单
        save(voucherOrder);
        
    } finally {
        redisLock.unlock();  // 释放锁
    }
}
```

**乐观锁的工作原理：**
```sql
-- MyBatis-Plus生成的SQL
UPDATE seckill_voucher 
SET stock = stock - 1 
WHERE voucher_id = ? 
  AND stock > 0;  -- 乐观锁：只有库存大于0才能扣减

-- 如果多个线程同时执行：
-- 线程1：stock = 10, 执行后 stock = 9, 影响行数 = 1 ✅
-- 线程2：stock = 10, 执行后 stock = 9, 影响行数 = 1 ✅
-- 线程3：stock = 9, 执行后 stock = 8, 影响行数 = 1 ✅
-- ...
-- 线程10：stock = 1, 执行后 stock = 0, 影响行数 = 1 ✅
-- 线程11：stock = 0, 执行后 影响行数 = 0 ❌（扣减失败，库存不足）
```

**两种策略的配合：**
1. **分布式锁**：保证同一用户的操作串行化，防止重复下单
2. **乐观锁**：保证库存扣减的原子性，防止超卖
3. **配合效果**：既防止重复下单，又防止超卖

**为什么需要两种策略：**
- **分布式锁**：解决用户维度的并发问题（重复下单）
- **乐观锁**：解决商品维度的并发问题（超卖）
- **两者结合**：全面解决并发问题

**其他可选方案：**
- **悲观锁**：`SELECT ... FOR UPDATE`，性能差，不推荐
- **Lua脚本**：Redis原子操作，适合纯Redis场景
- **数据库唯一索引**：防止重复下单，但无法防止超卖

#### 题目5：Redisson分布式锁的底层实现原理是什么？如何保证加锁和释放锁的原子性？

**背诵答案：**

**Redisson分布式锁实现原理：**
- **底层命令**：使用Redis的`SET key value NX EX timeout`命令
- **NX**：只有当key不存在时才设置（原子操作）
- **EX**：设置过期时间（秒）
- **看门狗机制**：自动续期锁的过期时间

**加锁流程：**
1. **尝试加锁**：执行Lua脚本，使用`SET key value NX EX timeout`
2. **判断结果**：如果返回OK，表示加锁成功；否则加锁失败
3. **启动看门狗**：加锁成功后，启动看门狗线程，定期续期

**释放锁流程：**
1. **判断锁归属**：使用Lua脚本判断锁的value是否匹配
2. **删除锁**：如果匹配，删除锁；否则不删除（防止误删其他线程的锁）
3. **停止看门狗**：释放锁后，停止看门狗线程

**Lua脚本实现（Redisson底层）：**
```lua
-- 加锁脚本
if (redis.call('exists', KEYS[1]) == 0) then
    redis.call('hset', KEYS[1], ARGV[2], 1);
    redis.call('pexpire', KEYS[1], ARGV[1]);
    return nil;
end;
if (redis.call('hexists', KEYS[1], ARGV[2]) == 1) then
    redis.call('hincrby', KEYS[1], ARGV[2], 1);
    redis.call('pexpire', KEYS[1], ARGV[1]);
    return nil;
end;
return redis.call('pttl', KEYS[1]);

-- 释放锁脚本
if (redis.call('hexists', KEYS[1], ARGV[3]) == 0) then
    return nil;
end;
local counter = redis.call('hincrby', KEYS[1], ARGV[3], -1);
if (counter > 0) then
    redis.call('pexpire', KEYS[1], ARGV[2]);
    return 0;
else
    redis.call('del', KEYS[1]);
    return 1;
end;
return nil;
```

**看门狗机制：**
- **作用**：自动续期锁的过期时间，避免业务未完成锁就过期
- **实现**：后台线程定期（默认每10秒）检查锁是否还存在，如果存在则续期
- **续期时间**：默认30秒（可配置）
- **停止条件**：锁被释放或业务完成

**原子性保证：**
1. **Lua脚本**：Redis执行Lua脚本是原子性的，保证加锁和释放锁的原子性
2. **单线程模型**：Redis单线程执行命令，不会出现并发问题
3. **锁值匹配**：释放锁时检查锁的value，防止误删其他线程的锁

**代码示例：**

- **缺点**：可能增加响应时间，需要合理设置重试次数和间隔

```java
while (!isLock && retryCount < maxRetries) {

if (!isLock) {
    return Result.fail("系统繁忙，请稍后重试");
```

**方案3：阻塞等待（Redisson支持）**
- **实现**：使用`lock()`方法，阻塞等待直到获取锁
- **优点**：保证最终能获取到锁

#### 题目8：分布式锁在高并发场景下会不会成为性能瓶颈？如何优化？

**背诵答案：**

**可能的性能瓶颈：**
1. **锁竞争激烈**：大量请求竞争同一把锁，导致大量请求等待
2. **Redis压力**：频繁的加锁和释放锁操作，增加Redis压力
3. **网络延迟**：每次加锁和释放锁都需要网络通信，增加延迟

**性能优化策略：**

**1. 减小锁粒度（最重要）**
- **问题**：锁粒度太大，导致大量请求竞争同一把锁
- **优化**：使用用户维度锁，而不是全局锁
- **效果**：不同用户的操作可以并行，大幅提升性能

**对比：**
```java
// ❌ 全局锁：所有用户串行，性能差
RLock lock = redissonClient.getLock("lock:seckill");

// ✅ 用户维度锁：不同用户并行，性能好
RLock lock = redissonClient.getLock("lock:order:" + userId);
```

**2. 减少锁持有时间**
- **问题**：锁持有时间过长，其他请求等待时间长
- **优化**：只对关键操作加锁，减少锁内业务逻辑
- **效果**：锁持有时间短，吞吐量提升

**优化示例：**
```java
// ❌ 锁内包含大量业务逻辑
try {
    lock.lock();
    // 大量业务逻辑（查询、计算、保存等）
    // 锁持有时间长
} finally {
    lock.unlock();
}

// ✅ 只对关键操作加锁
// 1. 先执行非关键操作（不加锁）
Shop shop = shopService.getById(shopId);
// 2. 只对关键操作加锁
try {
    lock.lock();
    // 只执行关键操作（检查、扣减、保存）
    checkAndSave();
} finally {
    lock.unlock();
}
```

**3. 使用非阻塞锁（tryLock）**
- **问题**：阻塞锁（lock）会导致大量线程等待
- **优化**：使用非阻塞锁（tryLock），获取失败立即返回
- **效果**：避免线程阻塞，提高响应速度

**对比：**
```java

// ✅ 非阻塞锁：立即返回

    return;  // 获取失败，直接返回
}
```

**4. Redis连接池优化**
- **问题**：Redis连接数不足，导致加锁操作等待
- **优化**：增加Redis连接池大小，使用连接池复用连接
- **效果**：减少连接创建和销毁的开销

**5. 使用本地锁 + 分布式锁（两级锁）**
- **问题**：所有请求都访问Redis，网络开销大
- **优化**：先使用本地锁（synchronized），再使用分布式锁
- **效果**：大部分请求在本地锁层面就被过滤，减少Redis访问

**代码示例：**
```java

        // 2. 分布式锁（跨JVM）

            if (lock.tryLock()) {
                // 业务逻辑
            }
        } finally {
            lock.unlock();
        }
    }
}
```

**6. 异步处理 + 消息队列**
- **问题**：同步加锁导致请求阻塞
- **优化**：使用消息队列，异步处理订单
- **效果**：请求快速返回，后台异步处理，提升用户体验

**项目实践：**
- ✅ **用户维度锁**：减小锁粒度，不同用户并行
- ✅ **非阻塞锁**：使用`tryLock()`，获取失败立即返回
- ✅ **锁持有时间短**：只对关键操作加锁
- ✅ **消息队列**：下单和订单落库解耦，异步处理

**性能指标：**
- **锁竞争**：用户维度锁，锁竞争大幅减少
- **响应时间**：非阻塞锁，响应时间快

**监控建议：**

#### 题目9：分布式锁可能出现死锁吗？如何避免？Redisson是如何防止死锁的？

**背诵答案：**

**分布式锁的死锁场景：**
1. **锁未释放**：业务执行异常，锁未释放，导致其他线程永远无法获取锁
2. **锁过期时间设置不当**：锁过期时间过短，业务未完成锁就过期，可能导致并发问题
3. **网络分区**：Redis与客户端网络断开，锁无法释放

**避免死锁的方法：**

**1. 使用try-finally确保锁释放（最重要）**

- **原理**：无论业务是否异常，都确保锁被释放
- **实现**：在finally块中释放锁

**代码示例：**
```java
RLock lock = redissonClient.getLock("lock:order:" + userId);
try {
    boolean isLock = lock.tryLock();
    if (!isLock) {
        return;
    }
    // 业务逻辑（可能抛出异常）
    doBusiness();
} finally {
    // 确保锁被释放
    if (lock.isHeldByCurrentThread()) {
        lock.unlock();
    }
}
```

**2. 设置合理的过期时间**
- **原理**：即使业务异常，锁也会在过期时间后自动释放
- **实现**：设置足够长的过期时间，或使用看门狗机制

**3. 使用可重入锁**
- **原理**：同一线程可以多次获取锁，避免嵌套调用时死锁
- **实现**：Redisson的RLock是可重入锁

**代码示例：**
```java
RLock lock = redissonClient.getLock("lock:order:" + userId);
try {
    lock.tryLock();
    // 方法1调用方法2，方法2也需要获取锁
    method1();  // 可重入，不会死锁
} finally {
    lock.unlock();
}

void method1() {
    // 同一线程，可以再次获取锁（可重入）
    lock.tryLock();  // ✅ 成功，不会阻塞
    try {
        method2();
    } finally {
        lock.unlock();
    }
}
```

**4. 避免嵌套锁**
- **原理**：避免多个锁的嵌套使用，可能导致死锁
- **实现**：尽量使用单把锁，或确保锁的获取顺序一致

**Redisson如何防止死锁：**

**1. 可重入锁支持**
- **实现**：Redisson的RLock是可重入锁，同一线程可以多次获取
- **原理**：使用Hash结构存储锁信息，记录持有线程和重入次数

**2. 看门狗机制**
- **实现**：自动续期锁的过期时间，避免业务未完成锁就过期
- **原理**：后台线程定期检查锁是否存在，如果存在则续期

**3. 锁值匹配**
- **实现**：释放锁时检查锁的value，只有匹配才释放
- **原理**：防止误删其他线程的锁，避免死锁

**4. Lua脚本原子性**
- **实现**：加锁和释放锁使用Lua脚本，保证原子性
- **原理**：避免加锁和释放锁过程中的并发问题

**项目实践：**
- ✅ **try-finally**：确保锁释放
- ✅ **可重入锁**：Redisson默认支持可重入
- ✅ **看门狗机制**：自动续期，避免锁提前过期
- ✅ **异常处理**：捕获异常，确保锁释放

**死锁检测和监控：**
1. **监控锁的持有时间**：如果锁持有时间过长，可能是死锁
2. **监控锁的获取失败率**：如果获取失败率过高，可能是锁未释放
3. **设置锁的最大持有时间**：超过最大持有时间，强制释放锁（需谨慎）

**最佳实践：**
- ✅ **必须使用try-finally**：确保锁释放
- ✅ **设置合理的过期时间**：或使用看门狗机制

#### 题目10：如何保证分布式锁的高可用性？如果Redis宕机了怎么办？

**背诵答案：**

**高可用性保证：**

**1. Redis主从复制**
- **原理**：主节点负责写，从节点负责读，主节点故障时自动切换
- **优点**：提高可用性，数据备份
- **缺点**：主从切换时可能丢失部分数据

**2. Redis哨兵模式（Sentinel）**
- **原理**：监控Redis主从节点，主节点故障时自动选举新的主节点

#### 题目1：请简单介绍一下Redis Stream消息队列的核心需求是什么？为什么需要将秒杀下单与订单落库解耦？

**背诵答案：**

**核心需求：**
1. **异步处理**：秒杀下单成功后，异步处理订单落库，提升响应速度
2. **解耦设计**：下单流程和订单落库流程解耦，降低系统耦合度
3. **削峰填谷**：高并发下单时，消息队列缓冲请求，平滑处理数据库压力
4. **可靠性保证**：消息持久化，支持消息确认和异常重试

**为什么需要解耦：**
1. **性能问题**：同步落库时，数据库成为瓶颈，响应时间长

#### 题目3：Redis Stream是如何实现消息发送和消费的？消费者组的作用是什么？

**背诵答案：**

**消息发送（生产者）：**
- **命令**：`XADD stream.orders * userId 123 voucherId 456 id 789`
- **实现位置**：Lua脚本中，秒杀验证通过后发送消息
- **特点**：原子操作，保证消息发送和业务逻辑的原子性

**Lua脚本中的消息发送：**
```lua
-- 秒杀验证通过后，发送消息到Stream
redis.call('xadd', 'stream.orders', '*', 
    'userId', userId, 
    'voucherId', voucherId, 
    'id', orderId)
```

**Java代码实现：**
```java
@Override
public Result seckillVoucher(Long voucherId) {
    Long userId = UserHolder.getUser().getId();
    long orderId = redisIdWorker.nextId("order");
    
    // 1. 执行Lua脚本（验证库存、重复下单、扣库存、发送消息）
    Long result = stringRedisTemplate.execute(
            SECKILL_SCRIPT,
            Collections.emptyList(),
            voucherId.toString(), 
            userId.toString(), 
            String.valueOf(orderId)
    );
    
    int r = result.intValue();
    if (r != 0) {
        // 验证失败
        return Result.fail(r == 1 ? "库存不足" : "不能重复下单");
    }
    
    // 2. 立即返回订单ID（消息已发送到Stream，后台异步处理）
    return Result.ok(orderId);
}
```

**消息消费（消费者）：**
- **命令**：`XREADGROUP GROUP g1 c1 COUNT 1 BLOCK 2000 STREAMS stream.orders >`
- **实现位置**：后台线程，持续消费消息
- **特点**：阻塞读取，有新消息立即处理

**消费者组的作用：**
1. **负载均衡**：多个消费者可以并发消费，提高处理速度
2. **消息分发**：同一个消息只被一个消费者处理，避免重复处理（Redis Stream 主要通过 **消费者组（Consumer Group）** + **消息状态追踪** 的组合机制来实现这一目标）
3. **故障转移**：某个消费者故障，其他消费者可以继续处理
4. **消息确认**：支持消息确认机制，保证消息不丢失

**消费者组示例：**
```
Stream: stream.orders
  ├── Consumer Group: g1
  │   ├── Consumer: c1 (处理消息1, 3, 5...)
  │   └── Consumer: c2 (处理消息2, 4, 6...)
  └── Consumer Group: g2
      └── Consumer: c1 (处理所有消息，用于备份)
```

**完整消费代码：**
```java
private class VoucherOrderHandler implements Runnable {
    @Override
    public void run() {
        while (true) {
            try {
                // 1. 从消费者组读取消息
                List<MapRecord<String, Object, Object>> list = 
                    stringRedisTemplate.opsForStream().read(
                        Consumer.from("g1", "c1"),  // 消费者组g1，消费者c1
                        StreamReadOptions.empty()
                            .count(1)  // 每次读取1条消息
                            .block(Duration.ofSeconds(2)),  // 阻塞2秒
                        StreamOffset.create("stream.orders", 
                            ReadOffset.lastConsumed())  // 从上次消费的位置继续
                    );
                
                // 2. 判断是否有消息
                if (list == null || list.isEmpty()) {
                    continue;  // 没有消息，继续循环
                }
                
                // 3. 解析消息
                MapRecord<String, Object, Object> record = list.get(0);
                Map<Object, Object> value = record.getValue();
                VoucherOrder voucherOrder = BeanUtil.fillBeanWithMap(
                    value, new VoucherOrder(), true);
                
                // 4. 处理订单（落库）
                createVoucherOrder(voucherOrder);
                
                // 5. 确认消息（ACK）
                stringRedisTemplate.opsForStream().acknowledge(
                    "stream.orders", "g1", record.getId());
                    
            } catch (Exception e) {
                log.error("处理订单异常", e);
                handlePendingList();  // 处理异常消息
            }
        }
    }
}
```

**消费者组的关键概念：**
- **Consumer Group**：消费者组，多个消费者可以属于同一个组
- **Consumer**：消费者，组内的一个消费者实例
- **Pending List**：待处理消息列表，消息被读取但未确认
- **ACK**：消息确认，处理完成后确认消息

#### 题目4：消息确认（ACK）机制是如何实现的？如果消息处理失败怎么办？

**背诵答案：**

**消息确认机制：**
- **原理**：消息被消费者读取后，进入Pending List，处理完成后发送ACK确认
- **命令**：`XACK stream.orders g1 messageId`
- **作用**：确认消息已处理，从Pending List中移除

**消息状态流转：**
```
新消息
  ↓
被消费者读取（XREADGROUP）
  ↓
进入Pending List（待确认）
  ↓
处理订单
  ↓
发送ACK（XACK）
  ↓
从Pending List移除 ✅
```

**代码实现：**
```java
// 1. 读取消息（进入Pending List）
List<MapRecord<String, Object, Object>> list = 
    stringRedisTemplate.opsForStream().read(
        Consumer.from("g1", "c1"),
        StreamReadOptions.empty().count(1).block(Duration.ofSeconds(2)),
        StreamOffset.create("stream.orders", ReadOffset.lastConsumed())
    );

// 2. 处理消息
try {
    createVoucherOrder(voucherOrder);
    
    // 3. 处理成功，确认消息
    stringRedisTemplate.opsForStream().acknowledge(
        "stream.orders", "g1", record.getId());
} catch (Exception e) {
    // 处理失败，消息仍在Pending List中
    log.error("处理订单异常", e);
}
```

**消息处理失败的处理：**

**方案1：Pending List重试（项目使用）**
- **原理**：处理失败的消息留在Pending List中，定期重新处理
- **实现**：单独线程处理Pending List中的消息
- **优点**：自动重试，保证消息不丢失
- **缺点**：如果消息一直失败，会一直重试

**代码实现：**
```java
private void handlePendingList() {
    while (true) {
        try {
            // 1. 读取Pending List中的消息（从0开始，表示未确认的消息）
            List<MapRecord<String, Object, Object>> list = 
                stringRedisTemplate.opsForStream().read(
                    Consumer.from("g1", "c1"),
                    StreamReadOptions.empty().count(1),
                    StreamOffset.create("stream.orders", 
                        ReadOffset.from("0"))  // 0表示Pending List
                );
            
            // 2. 判断是否有待处理消息
            if (list == null || list.isEmpty()) {
                break;  // 没有待处理消息，结束循环
            }
            
            // 3. 重新处理消息
            MapRecord<String, Object, Object> record = list.get(0);
            Map<Object, Object> value = record.getValue();
            VoucherOrder voucherOrder = BeanUtil.fillBeanWithMap(
                value, new VoucherOrder(), true);
            
            // 4. 处理订单
            createVoucherOrder(voucherOrder);
            
            // 5. 确认消息
            stringRedisTemplate.opsForStream().acknowledge(
                "stream.orders", "g1", record.getId());
                
        } catch (Exception e) {
            log.error("处理Pending List异常", e);
            // 继续处理下一条消息
        }
    }
}
```

**方案2：死信队列**
- **原理**：重试多次后仍失败，将消息移到死信队列
- **实现**：记录重试次数，超过阈值后移到死信队列
- **优点**：避免无限重试，便于人工处理
- **缺点**：需要额外的死信队列管理

**方案3：消息过期**
- **原理**：设置消息过期时间，过期后自动删除
- **实现**：Redis Stream支持消息过期（需配置）
- **优点**：自动清理过期消息
- **缺点**：可能丢失重要消息

**项目实践：**
- ✅ **Pending List重试**：处理失败的消息自动重试
- ✅ **异常处理**：捕获异常，记录日志，继续处理
- ✅ **监控告警**：监控Pending List长度，及时发现异常

**最佳实践：**
1. **及时ACK**：处理成功后立即ACK，避免消息堆积
2. **异常处理**：捕获所有异常，避免消费者线程崩溃
3. **重试机制**：处理失败的消息自动重试
4. **监控告警**：监控Pending List长度，及时发现异常

#### 题目5：Redis Stream消息队列的性能如何？如何保证高并发场景下的稳定性？

- **效果**：减少连接创建开销，提高性能

- **监控告警**：监控消费者状态，及时发现故障

- **限流**：限制消息发送速度，避免消息堆积
- **降级**：消息队列不可用时，降级到同步处理
- **熔断**：连续失败时，暂时停止消费，避免雪崩

  - Stream长度（消息堆积情况）
  - Pending List长度（待处理消息数）

  - 消息处理失败率
- **告警**：指标异常时及时告警

**项目实践：**

#### 题目2：Feed流推送机制的业务流程是什么？推送时机和查询流程是如何设计的？

**业务流程：**

**1. 内容发布流程（推送）：**

```
  ↓

  ↓

  ↓

  ↓

  ↓

```

```
  ↓

  ↓

```

**推送时机：**
- **发布时推送**：用户发布内容成功后，立即推送给所有粉丝

- **实时性**：内容发布后，粉丝的Feed流立即更新

**查询流程设计：**
- **滚动分页**：使用ZSet的`reverseRangeByScoreWithScores`实现滚动分页

- **批量查询**：先获取内容ID列表，再批量查询内容详情

### 实现层：技术实现与核心逻辑

#### 题目3：Feed流推送机制是如何使用Redis ZSet实现的？为什么选择ZSet而不是List或Set？

**背诵答案：**

**Redis ZSet实现：**
- **数据结构**：每个用户有一个ZSet，Key为`feed:userId`
- **存储内容**：内容ID作为member，时间戳作为score
- **排序方式**：按score（时间戳）倒序排列，最新内容在前

**为什么选择ZSet：**
1. **有序性**：ZSet天然有序，按时间戳排序，无需额外排序
2. **范围查询**：支持按分数范围查询，实现滚动分页
3. **去重**：自动去重，同一内容不会重复推送
4. **性能好**：ZSet的查询和插入性能都很好

**与其他数据结构对比：**

| 对比维度 | ZSet ✅ | List | Set |
|---------|--------|------|-----|
| **有序性** | 支持（按分数排序） ✅ | 支持（按插入顺序） | 不支持 |
| **范围查询** | 支持（按分数范围） ✅ | 支持（按索引） | 不支持 |
| **去重** | 支持 ✅ | 不支持 | 支持 ✅ |
| **时间排序** | 支持（分数=时间戳） ✅ | 需要额外排序 | 不支持 |
| **滚动分页** | 支持（高效） ✅ | 支持（需计算索引） | 不支持 |
| **性能** | 高 ✅ | 中等 | 高 ✅ |

**ZSet的关键操作：**
```java
// 1. 推送内容ID（添加元素）
stringRedisTemplate.opsForZSet().add(
    "feed:userId",           // Key
    "blogId",                // Member（内容ID）
    System.currentTimeMillis() // Score（时间戳）
);

// 2. 查询Feed流（按时间倒序，滚动分页）
Set<ZSetOperations.TypedTuple<String>> tuples = 
    stringRedisTemplate.opsForZSet()
        .reverseRangeByScoreWithScores(
            "feed:userId",   // Key
            0,               // 最小分数
            max,             // 最大分数（上次查询的最小时间戳）
            offset,          // 偏移量（跳过相同时间戳的数量）
            2                // 查询数量
        );

// 3. 删除内容（取消推送）
stringRedisTemplate.opsForZSet().remove(
    "feed:userId", 
    "blogId"
);
```

**滚动分页实现原理：**
- **首次查询**：`max = Long.MAX_VALUE`，从最新内容开始查询
- **后续查询**：`max = 上次查询的最小时间戳`，`offset = 相同时间戳的数量`
- **优势**：避免数据重复和遗漏，支持高效分页

**代码示例：**
```java

    String key = FEED_KEY + userId;
    
    // 查询ZSet（按时间倒序，范围查询）

        stringRedisTemplate.opsForZSet()
            .reverseRangeByScoreWithScores(
                key, 0, max, offset, 2

        ids.add(Long.valueOf(tuple.getValue()));
        long time = tuple.getScore().longValue();
        if (time == minTime) {
            os++;  // 相同时间戳的数量

            minTime = time;  // 更新最小时间戳
            os = 1;
        }
    }
    
    // 返回结果和下次查询参数
    return Result.ok(blogs, minTime, os);
}
```

**项目优势：**
- ✅ **有序性**：ZSet天然有序，按时间戳排序
- ✅ **范围查询**：支持高效的滚动分页
- ✅ **去重**：自动去重，避免重复推送
- ✅ **性能好**：ZSet查询和插入性能都很好

#### 题目4：如果用户有很多粉丝（比如百万粉丝），推送性能如何优化？有没有考虑过推拉结合模式？

**背诵答案：**

**推模式的性能问题：**
- **问题**：如果用户有百万粉丝，发布内容时需要推送百万次，性能差
- **影响**：推送操作耗时，影响内容发布的响应时间
- **存储成本**：每个粉丝的Feed流都存储，存储成本高

**优化策略：**

**1. 异步推送（最重要）**
- **原理**：内容发布后，异步推送给粉丝，不阻塞主流程
- **实现**：使用线程池或消息队列异步处理推送
- **效果**：内容发布立即返回，推送在后台进行

**代码示例：**
```java
@Override
public Result saveBlog(Blog blog) {
    // 1. 保存内容
    save(blog);
    
    // 2. 异步推送（不阻塞主流程）
    CompletableFuture.runAsync(() -> {
        pushToFollowers(blog);
    }, pushExecutor);
    
    // 3. 立即返回
    return Result.ok(blog.getId());
}

private void pushToFollowers(Blog blog) {
    List<Follow> follows = followService.query()
        .eq("follow_user_id", blog.getUserId())
        .list();
    
    for (Follow follow : follows) {
        String key = FEED_KEY + follow.getUserId();
        stringRedisTemplate.opsForZSet().add(
            key, 
            blog.getId().toString(), 
            System.currentTimeMillis()
        );
    }
}
```

**2. 批量推送**
- **原理**：使用Redis Pipeline批量推送，减少网络往返
- **实现**：收集所有推送操作，一次性执行
- **效果**：大幅提升推送性能

**代码示例：**
```java

    .eq("follow_user_id", blog.getUserId())
    .list();

stringRedisTemplate.executePipelined(new RedisCallback<Object>() {
    @Override
    public Object doInRedis(RedisConnection connection) {
        for (Follow follow : follows) {
            String key = (FEED_KEY + follow.getUserId()).getBytes();
            String value = blog.getId().toString().getBytes();
            double score = System.currentTimeMillis();
            connection.zAdd(key, score, value);
        }
        return null;
    }
});
```

**3. 推拉结合模式**
- **原理**：粉丝数少的用户使用推模式，粉丝数多的用户使用拉模式
- **实现**：设置阈值（如1万），超过阈值使用拉模式
- **效果**：兼顾性能和实时性

**推拉结合实现：**
```java

        new QueryWrapper<Follow>()
            .eq("follow_user_id", blog.getUserId())

        // 粉丝数少，使用推模式

    } else {
        // 粉丝数多，使用拉模式（不推送，查看时再拉取）
        // 或者标记为大V，使用特殊处理
    }
    
    return Result.ok(blog.getId());
}
```

**4. 分片推送**
- **原理**：将粉丝列表分片，多线程并发推送
- **实现**：使用线程池，每个线程处理一部分粉丝
- **效果**：提高推送速度

**代码示例：**
```java

List<Follow> follows = followService.query()
    .eq("follow_user_id", blog.getUserId())
    .list();

int batchSize = 1000;  // 每批处理1000个
List<List<Follow>> batches = Lists.partition(follows, batchSize);

for (List<Follow> batch : batches) {
    CompletableFuture.runAsync(() -> {
        for (Follow follow : batch) {
            pushToFollower(blog, follow.getUserId());
        }
    }, pushExecutor);
}
```

**项目实践：**
- ✅ **异步推送**：内容发布后异步推送，不阻塞主流程
- ✅ **批量推送**：使用Pipeline批量推送，提高性能
- ⚠️ **推拉结合**：当前未实现，但可以考虑优化

- **用户体验**：内容发布立即返回，用户体验好

#### 题目5：Feed流的滚动分页是如何实现的？与传统的分页有什么区别？

**背诵答案：**

**滚动分页（Cursor-based Pagination）：**
- **原理**：使用时间戳和偏移量作为游标，每次查询返回下次查询的参数
- **优势**：避免数据重复和遗漏，性能好
- **适用场景**：Feed流、时间线等按时间排序的场景

**传统分页（Offset-based Pagination）：**
- **原理**：使用页码和每页数量，通过LIMIT和OFFSET查询
- **问题**：数据新增或删除时，可能出现重复或遗漏
- **适用场景**：数据相对稳定的场景

**两种分页对比：**
| 对比维度 | 滚动分页 ✅ | 传统分页 |
|---------|-----------|---------|
| **实现方式** | 时间戳+偏移量 | 页码+每页数量 |
| **数据一致性** | 好（避免重复遗漏） ✅ | 差（可能重复遗漏） |
| **性能** | 好（范围查询） ✅ | 差（OFFSET性能差） |
| **适用场景** | 动态数据（Feed流） ✅ | 静态数据 |
| **用户体验** | 好（无限滚动） ✅ | 一般（页码跳转） |

**滚动分页实现：**
```java

    Long userId = UserHolder.getUser().getId();
    String key = FEED_KEY + userId;

                key, 
                0,              // 最小分数
                max,            // 最大分数（上次查询的最小时间戳）
                offset,         // 偏移量（跳过相同时间戳的数量）
                2               // 查询数量
            );
    
    if (typedTuples == null || typedTuples.isEmpty()) {
        return Result.ok();
    }
    
    // 2. 解析内容ID和时间戳
    List<Long> ids = new ArrayList<>();
    long minTime = 0;  // 本次查询的最小时间戳
    int os = 1;        // 相同时间戳的数量

        ids.add(Long.valueOf(tuple.getValue()));
        long time = tuple.getScore().longValue();
        
        if (time == minTime) {
            os++;  // 相同时间戳，偏移量+1
        } else {
            minTime = time;  // 更新最小时间戳
            os = 1;          // 重置偏移量
        }
    }
    
    // 3. 批量查询内容详情
    List<Blog> blogs = listByIds(ids);
    
    // 4. 返回结果（包含下次查询的max和offset）
    ScrollResult scrollResult = new ScrollResult();
    scrollResult.setList(blogs);
    scrollResult.setMinTime(minTime);  // 下次查询的max
    scrollResult.setOffset(os);        // 下次查询的offset
    
    return Result.ok(scrollResult);
}
```

**滚动分页的关键点：**
1. **max参数**：上次查询的最小时间戳，本次查询早于该时间的内容
2. **offset参数**：跳过相同时间戳的数量，避免重复
3. **返回参数**：返回本次查询的最小时间戳和偏移量，供下次查询使用

- ✅ **数据一致性**：避免数据重复和遗漏

| **共同关注** | Set ✅ | 支持交集运算，性能好 | SINTER（交集） |

| **用户签到** | BitMap ✅ | 节省内存，适合布尔值存储 | SETBIT、BITFIELD（位操作） |

**业务场景：**
- **共同关注**：用户查看与好友的共同关注，发现共同兴趣

- **用户签到**：用户每日签到，获得积分奖励，提升活跃度

**数据结构选择原则：**
1. **根据业务需求选择**：需要交集用Set，需要排序用ZSet，需要布尔值用BitMap
2. **考虑性能**：不同数据结构的操作性能不同
3. **考虑内存**：BitMap最省内存，Set和ZSet相对占用更多

```java
@Override
public Result followCommons(Long id) {

    Long userId = UserHolder.getUser().getId();

    // 2. 获取目标用户
    String key2 = "follows:" + id;

        .intersect(key1, key2);

        .collect(Collectors.toList());

}
```

#### 题目4：点赞排行功能是如何使用Redis ZSet实现的？为什么使用时间戳作为分数？

**背诵答案：**

**Redis ZSet实现：**

**点赞操作：**
```java
@Override
public Result likeBlog(Long id) {
    Long userId = UserHolder.getUser().getId();
    String key = BLOG_LIKED_KEY + id;  // blog:liked:blogId
    
    // 1. 查询是否已点赞
    Double score = stringRedisTemplate.opsForZSet()
        .score(key, userId.toString());
    
    if (score == null) {
        // 2. 未点赞，执行点赞
        // 2.1. 数据库点赞数+1
        boolean isSuccess = update()
            .setSql("liked = liked + 1")
            .eq("id", id)
            .update();
        
        // 2.2. 添加到ZSet，分数为时间戳
        if (isSuccess) {
            stringRedisTemplate.opsForZSet().add(
                key, 
                userId.toString(), 
                System.currentTimeMillis()  // 时间戳作为分数
            );
        }
    } else {
        // 3. 已点赞，取消点赞
        // 3.1. 数据库点赞数-1
        boolean isSuccess = update()
            .setSql("liked = liked - 1")
            .eq("id", id)
            .update();
        
        // 3.2. 从ZSet移除
        if (isSuccess) {
            stringRedisTemplate.opsForZSet().remove(key, userId.toString());
        }
    }
    
    return Result.ok();
}
```

**查询点赞排行：**
```java
@Override
public Result queryBlogLikes(Long id) {
    String key = BLOG_LIKED_KEY + id;
    
    // 1. 查询前5名点赞用户（按时间倒序）
    Set<String> top5 = stringRedisTemplate.opsForZSet()
        .range(key, 0, 4);  // 按分数升序，取前5个
    
    if (top5 == null || top5.isEmpty()) {
        return Result.ok(Collections.emptyList());
    }
    
    // 2. 解析用户ID
    List<Long> ids = top5.stream()
        .map(Long::valueOf)
        .collect(Collectors.toList());
    String idStr = StrUtil.join(",", ids);
    
    // 3. 批量查询用户信息（保持顺序）
    List<UserDTO> userDTOS = userService.query()
        .in("id", ids)
        .last("ORDER BY FIELD(id," + idStr + ")")  // 保持ZSet的顺序
        .list()
        .stream()
        .map(user -> BeanUtil.copyProperties(user, UserDTO.class))
        .collect(Collectors.toList());
    
    return Result.ok(userDTOS);
}
```

**为什么使用时间戳作为分数：**
1. **时间排序**：时间戳天然有序，可以实现按点赞时间排序
2. **唯一性**：时间戳唯一，不会重复
3. **范围查询**：支持按时间范围查询，如查询某时间段内的点赞用户
4. **业务需求**：点赞排行通常按时间排序，最早点赞的用户在前

**时间戳的优势：**
- ✅ **天然有序**：时间戳递增，自然实现排序
- ✅ **唯一性**：每个点赞时间不同，避免分数冲突
- ✅ **业务匹配**：符合"按点赞时间排序"的业务需求
- ✅ **范围查询**：支持查询某时间段内的点赞用户

**其他可选方案：**
- **自增ID**：可以实现排序，但无法体现时间信息
- **点赞数**：如果按点赞数排序，但同一博客的点赞数都是1，无法区分
- **组合分数**：时间戳+其他信息，但实现复杂

**项目优势：**
- ✅ **有序存储**：ZSet天然有序，按时间戳排序

#### 题目5：用户签到功能是如何使用Redis BitMap实现的？如何计算连续签到天数？

**背诵答案：**

**Redis BitMap实现：**

**签到操作：**
```java
@Override
public Result sign() {
    // 1. 获取当前登录用户
    Long userId = UserHolder.getUser().getId();
    
    // 2. 获取日期
    LocalDateTime now = LocalDateTime.now();
    
    // 3. 拼接key（按月存储）
    String keySuffix = now.format(DateTimeFormatter.ofPattern(":yyyyMM"));
    String key = USER_SIGN_KEY + userId + keySuffix;  // sign:userId:yyyyMM
    
    // 4. 获取今天是本月的第几天
    int dayOfMonth = now.getDayOfMonth();
    
    // 5. 设置对应位为1（SETBIT）
    stringRedisTemplate.opsForValue().setBit(
        key, 
        dayOfMonth - 1,  // 位偏移量（从0开始）
        true             // 设置为1（已签到）
    );
    
    return Result.ok();
}
```

**查询连续签到天数：**
```java
@Override
public Result signCount() {
    // 1. 获取当前登录用户
    Long userId = UserHolder.getUser().getId();
    
    // 2. 获取日期
    LocalDateTime now = LocalDateTime.now();
    String keySuffix = now.format(DateTimeFormatter.ofPattern(":yyyyMM"));
    String key = USER_SIGN_KEY + userId + keySuffix;
    
    // 3. 获取今天是本月的第几天
    int dayOfMonth = now.getDayOfMonth();
    
    // 4. 获取本月截止今天为止的所有签到记录（BITFIELD）
    List<Long> result = stringRedisTemplate.opsForValue().bitField(
        key,
        BitFieldSubCommands.create()
            .get(BitFieldSubCommands.BitFieldType.unsigned(dayOfMonth))
            .valueAt(0)
    );
    
    if (result == null || result.isEmpty()) {
        return Result.ok(0);
    }
    
    Long num = result.get(0);
    if (num == null || num == 0) {
        return Result.ok(0);
    }
    
    // 5. 计算连续签到天数（从低位到高位）
    int count = 0;
    while (true) {
        // 5.1. 与1做与运算，得到最后一个bit位
        if ((num & 1) == 0) {
            // 如果为0，说明未签到，结束
            break;
        } else {
            // 如果不为0，说明已签到，计数器+1
            count++;
        }
        // 5.2. 右移一位，继续下一个bit位
        num >>>= 1;
    }
    
    return Result.ok(count);
}
```

**BitMap存储原理：**
- **存储方式**：每个位代表一天的签到状态
- **位偏移量**：第1天对应位0，第2天对应位1，以此类推
- **Key设计**：按月存储，`sign:userId:yyyyMM`
- **内存占用**：一个月最多31天，只需31位（约4字节）

**计算连续签到天数的逻辑：**
1. **获取签到数据**：使用BITFIELD获取本月签到记录（二进制数）
2. **从低位到高位遍历**：从第1天开始，逐位检查
3. **连续判断**：如果某位为0（未签到），停止计数
4. **返回结果**：返回连续签到的天数

**BitMap的优势：**
- ✅ **节省内存**：每个用户每月只需4字节，比Set或List节省大量内存
- ✅ **性能好**：位操作性能高，查询速度快
- ✅ **操作简单**：SETBIT和BITFIELD操作简单
- ✅ **适合布尔值**：适合存储大量布尔值（签到/未签到）

**与其他数据结构对比：**
| 对比维度 | BitMap ✅ | Set | List |
|---------|----------|-----|------|
| **内存占用** | 极小（4字节/月） ✅ | 大（每个ID占用空间） | 大（每个ID占用空间） |
| **查询性能** | 高（位操作） ✅ | 高 | 中等 |
| **适用场景** | 布尔值存储 ✅ | 去重集合 | 有序列表 |
| **操作复杂度** | 简单 ✅ | 简单 ✅ | 简单 ✅ |

**项目优势：**
- ✅ **内存效率**：BitMap最省内存，适合大量用户签到
- ✅ **性能好**：位操作性能高，查询速度快
- ✅ **实现简单**：SETBIT和BITFIELD操作简单
- ✅ **业务匹配**：完美匹配签到场景（布尔值存储）

**当前优化：**

- **实现**：使用Redis Cluster，数据分片

**高并发场景优化：**
- ✅ **多级缓存**：本地缓存+Redis缓存，减少网络延迟
- ✅ **缓存预热**：系统启动时预热热点数据

#### 题目4：消息队列如何保证消息不丢失？如果Redis宕机了怎么办？

**背诵答案：**

**消息不丢失的保证：**

**1. 消息持久化**
- **Redis持久化**：开启AOF持久化，每次写操作都记录
- **消息存储**：Redis Stream支持持久化，消息存储在磁盘
- **配置**：`appendonly yes`，保证消息不丢失

**2. 消息确认机制（ACK）**
- **原理**：消息被消费后，发送ACK确认
- **实现**：处理成功后ACK，失败后不ACK，消息留在Pending List
- **重试**：处理Pending List中的消息，保证最终处理

**3. 消费者组**
- **原理**：多个消费者并发消费，某个消费者故障不影响整体
- **实现**：消费者故障后，消息自动分配给其他消费者
- **故障转移**：支持消费者故障转移，保证消息处理

**Redis宕机的处理：**

**1. 高可用架构**
- **主从复制**：Redis主从复制，主节点故障自动切换
- **哨兵模式**：Redis Sentinel监控，自动故障转移
- **集群模式**：Redis Cluster，多节点部署，高可用

**2. 降级策略**
- **降级到数据库**：Redis不可用时，降级到数据库处理
- **消息持久化**：消息持久化到数据库，Redis恢复后同步
- **熔断机制**：Redis连续失败时，暂时不使用Redis

**3. 数据备份**
- **定期备份**：定期备份Redis数据，防止数据丢失
- **数据恢复**：Redis恢复后，从备份恢复数据

#### 题目4：Feed流推送机制在大V用户（百万粉丝）场景下如何优化？

**背诵答案：**

**大V用户的挑战：**
- **推送性能**：百万粉丝，推送耗时，影响响应时间
- **存储成本**：每个粉丝的Feed流都存储，存储成本高
- **查询性能**：Feed流查询可能变慢

**优化方案：**

**1. 推拉结合（最重要）**
- **原理**：粉丝数少的推模式，粉丝数多的拉模式
- **实现**：设置阈值（如1万），超过阈值使用拉模式
- **效果**：大V用户不推送，查看时再拉取，节省推送成本

**2. 异步推送**
- **原理**：发布内容后，异步推送给粉丝
- **实现**：使用消息队列，后台异步推送
- **效果**：不阻塞内容发布，提升响应速度

**3. 分片推送**
- **原理**：将粉丝列表分片，多实例并发推送
- **实现**：使用分布式任务调度，每个实例处理一部分粉丝
- **效果**：提高推送速度，分散压力

**4. 延迟推送**
- **原理**：非实时推送，延迟推送，降低峰值压力
- **实现**：使用消息队列，延迟推送
- **效果**：平滑推送压力，避免系统峰值

**5. 智能推送**
- **原理**：只推送给活跃粉丝，不活跃粉丝不推送
- **实现**：根据用户活跃度，选择性推送
- **效果**：减少推送量，节省存储成本

**6. 存储优化**
- **原理**：Feed流数据分片存储，按时间或用户分片
- **实现**：使用Redis Cluster，数据分片
- **效果**：分散存储压力，提升查询性能

**架构设计：**
```
发布内容
  ↓
判断粉丝数
  ↓
粉丝数 < 1万：推模式（异步推送）
  ↓
粉丝数 >= 1万：拉模式（不推送，查看时拉取）
  ↓
分片推送（多实例并发）
  ↓
更新Feed流
```

**项目实践：**
- ✅ **当前实现**：推模式，适合中小规模
- ✅ **大V优化**：推拉结合、异步推送、分片推送
- ✅ **可扩展性**：支持千万级用户，适应大规模场景
