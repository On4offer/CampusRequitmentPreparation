## 二、业务场景

### 2.1 一句话介绍

「小众点评是一个本地生活点评项目，类似大众点评的简化版，包含商户展示、优惠券秒杀、探店笔记、关注 Feed 流等，我主要负责缓存、秒杀和 Feed 流相关的设计与实现。」

### 2.2 核心功能模块（对应简历）

1. **商户查询**：按类型、地理距离查店铺，店铺详情带缓存。
2. **短信登录**：验证码存 Redis，登录态用 Token + Redis，滑动过期。
3. **优惠券秒杀**：限时抢券、一人一单、防超卖、异步下单。
4. **探店笔记（达人探店）**：发笔记、点赞（ZSet 排行榜）、关注、Feed 流。
5. **附近商户**：GEO 地理坐标查询（讲义有，简历未写可简要带过）。

### 2.3 用户主流程（便于讲业务）

- **游客**：看店铺列表、店铺详情、热门笔记。
- **登录**：手机号+验证码 → Token 存 Redis，请求头带 Token，拦截器校验并刷新 TTL。
- **秒杀**：选秒杀券 → 校验库存与一人一单（Lua）→ **发 RocketMQ 消息** → 异步消费者落库，返回订单号。
- **Feed**：关注达人 → 达人发笔记 → 笔记 ID 推送到粉丝收件箱（ZSet）→ 下拉滚动分页查收件箱。

### 3.1 核心表及职责

| 表名 | 用途 | 关键字段 |
|------|------|----------|
| **tb_user** | 用户 | id, phone(唯一), nick_name, icon |
| **tb_shop** | 商户 | id, name, type_id, area, address, x/y(经纬度), avg_price, sold, score |
| **tb_shop_type** | 商户类型 | id, name, icon, sort |
| **tb_blog** | 探店笔记 | id, shop_id, user_id, title, images, content, liked, comments |
| **tb_follow** | 关注关系 | user_id, follow_user_id（谁关注了谁） |
| **tb_voucher** | 优惠券（普通/秒杀） | id, shop_id, pay_value, actual_value, type(0普通 1秒杀) |
| **tb_seckill_voucher** | 秒杀券库存与时间 | voucher_id, stock, begin_time, end_time |
| **tb_voucher_order** | 秒杀订单 | id(业务主键), user_id, voucher_id, status |

### 3.2 面试可说的设计点

- **tb_voucher_order.id**：不用自增，用 **Redis 生成的全局唯一 ID**（时间戳+日期+当日自增），便于在 Redis 里先产生订单号再异步落库。
- **tb_seckill_voucher**：与 tb_voucher 一对一，把「库存、开抢/结束时间」拆出来，方便秒杀时只扣库存、不频繁改主表。
- **tb_follow**：user_id + follow_user_id 表示「谁关注谁」，Feed 推送到粉丝收件箱时查的就是 follow_user_id = 当前作者。
- **tb_blog**：liked 存数据库，点赞列表和是否点赞用 Redis ZSet（blog:liked:{blogId}），member 为 userId，score 为点赞时间，便于按时间取 TopN 和判断是否点赞。

## 四、技术选型

### 4.1 为什么用 Redis 做缓存？

- 商户详情读多写少，DB 压力大；Redis 内存读写快，减轻 MySQL 压力。
- 项目里用 **Cache-Aside**：先查缓存，未命中再查 DB 并回写缓存；更新时先更新 DB 再删缓存。

### 4.2 为什么用 Redisson 做分布式锁？

- 秒杀要保证「同一用户不能重复下单」，多机多线程下需要分布式锁。
- Redisson 提供可重入、看门狗续期，避免业务执行时间超过锁 TTL 导致锁失效；比手写 setNX+过期更稳妥。

### 4.3 为什么用 Lua 做秒杀资格校验？

- 「扣库存 + 记录用户已购」需要原子性，Lua 在 Redis 里单线程执行，避免多命令之间的并发问题。
- 一次网络往返完成：判库存、判一人一单、扣库存、SADD 用户；**消息发送在 Java 侧用 RocketMQ 完成**，Lua 只负责 Redis 内原子操作（新版 seckill.lua 已去掉 XADD）。

### 4.4 为什么用 ZSet 做 Feed 流？

- 需要按**时间顺序**拉取关注人的笔记，ZSet 的 score 用时间戳，天然支持按时间范围/分页。
- 推模式：发笔记时直接写入每个粉丝的收件箱（ZSet），拉取时按 score 倒序 + 滚动分页即可。

### 4.5 为什么用 RocketMQ 做异步下单？

- **新版实现**：Lua 只做库存扣减 + SADD 用户，返回 0 后由 **RocketMQ** 发送 `SeckillOrderMessage`（voucherId、userId、orderId）到 topic `seckill-order-topic`，消费者 `SeckillOrderConsumer` 异步落库。
- 解耦秒杀接口与写库，接口只做 Redis + 发消息，响应快；RocketMQ 持久化、消费 ACK、重试机制保证消息不丢，与简历表述一致。

## 五、代码实现（核心流程）

### 5.1 商户缓存查询（Cache-Aside + 三大问题）

- **入口**：`ShopServiceImpl#queryById` → **默认** `CacheClient#queryWithPassThrough`（防穿透）；注释中可选 `queryWithMutex`（互斥锁）、`queryWithLogicalExpire`（逻辑过期）。
- **流程**：  
  1）查 Redis key = `cache:shop:{id}`；  
  2）有则反序列化返回；若用逻辑过期则判断逻辑过期时间，过期则抢锁 `lock:shop:{id}`，拿到则异步线程查 DB 并 `setWithLogicalExpire` 回写；  
  3）无则查 DB，空则写空串+短 TTL（防穿透），有则写缓存。
- **可强调**：封装在 `CacheClient`，方法名为 `queryWithLogicalExpire`、`setWithLogicalExpire`、`unlock`；泛型 + `Function<ID, R> dbFallback` 做模板方法，店铺、类型等都可复用。

### 5.1.1 附近商户（GEO）

- **入口**：`ShopServiceImpl#queryShopByType(typeId, current, x, y)`。
- **流程**：当 x、y 不为空时，用 Redis GEO：key = `shop:geo:{typeId}`，以 (x,y) 为中心、半径 5000m 搜索，分页截取 from~end，得到 shopId 与距离；再按 id 查 Shop 并设置 distance。

### 5.2 Token 校验与滑动过期

- **入口**：`RefreshTokenInterceptor#preHandle`。
- **流程**：从请求头拿 `authorization`（Token）→ Redis key `login:token:{token}` 用 Hash 存用户信息 → 存在则转 UserDTO 放入 `UserHolder`（ThreadLocal），并 `expire(key, TTL)` 刷新过期时间 → `afterCompletion` 里 `UserHolder.removeUser()` 防止泄漏。
- **可强调**：无状态、多机共享 Session、滑动过期每次访问刷新，响应快。

### 5.3 秒杀（Lua + RocketMQ + 分布式锁 + 异步落库）

- **入口**：`VoucherOrderServiceImpl#seckillVoucher`。
- **流程**：  
  1）执行 Lua：校验库存 `seckill:stock:{voucherId}`、一人一单 `seckill:order:{voucherId}`（Set）、扣库存、SADD 用户；**Lua 不再写 Stream**，返回 0 表示有资格；  
  2）Lua 返回 0 后，封装 `SeckillOrderMessage(voucherId, userId, orderId)`，通过 **RocketMQTemplate** 发送到 topic `seckill-order-topic`；  
  3）若**发送失败**：回滚 Redis——`increment(stockKey)`、`Set.remove(orderKey, userId)`，再返回「系统繁忙」；  
  4）消费者 **SeckillOrderConsumer**（`@RocketMQMessageListener` topic=`seckill-order-topic`，consumerGroup=`seckill-consumer-group`）：解析 MessageExt body 为 SeckillOrderMessage → 获取 Redisson 锁 `lock:order:{userId}`（tryLock(3, 10, SECONDS)）→ 幂等检查（查库是否已有该 user+voucher 订单）→ 乐观锁扣库存 → insert 订单；若主键冲突（DuplicateKeyException）且为同一用户+优惠券则幂等返回。
- **可强调**：资格校验在 Redis 内原子完成；落库用 RocketMQ 解耦，发送失败回滚 Redis 保证一致性；消费者内 Redisson 锁 + 幂等 + 乐观锁防超卖与重复下单。

### 5.4 Feed 流（推模式 + 滚动分页）

- **写**：`BlogServiceImpl#saveBlog` — 发笔记后查 `tb_follow` 中 follow_user_id = 当前用户，对每个粉丝的 key `feed:{userId}` 执行 ZADD，member 为 blogId，score 为当前时间戳。
- **读**：`queryBlogOfFollow(Long max, Integer offset)` — 用 `ZREVRANGEBYSCORE feed:{userId} 0 max LIMIT offset 2`，得到 (blogId, score)，用 score 做下一次的 max，相同 score 用 offset 跳过，实现滚动分页；再按 id 查 Blog、填用户、填是否点赞。

## 六、高并发与性能（对应简历数据）

### 6.1 「数据库查询减少约 97%、缓存命中率 95%」

- **含义**：大部分请求命中缓存，只有少量请求打到 DB。
- **可答**：通过 Cache-Aside + 逻辑过期/互斥锁 + 空值缓存，把热点店铺都缓存在 Redis，实测或压测中统计「请求总数 vs 查 DB 次数」得到约 97% 减少；命中率通过监控 Redis 命中次数/总请求得到 95% 左右。（若没真实压测，就说「根据缓存策略和热点数据比例，理论上可达到这一量级，实际需压测验证」。）

### 6.2 「响应时间 <5ms、成功率 100%」

- **含义**：Token 校验只做 Redis 查询和 ThreadLocal 存取，延迟低；逻辑正确时校验成功即放行。
- **可答**：登录态放在 Redis，拦截器内一次 Hash 查询 + 刷新 TTL，无 DB，所以响应时间很短；只要 Redis 正常、Token 有效，就放行，成功率 100% 指业务逻辑上的校验成功。

### 6.3 「重复下单率与超卖率均为 0」

- **可答**：Lua 内 SADD 保证一人一单，落库时 Redisson 锁 + 再次查订单表 + 乐观锁 `stock > 0` 扣库存，多重保障，所以重复下单和超卖都避免。

### 6.4 「TPS 提升约 15 倍、平均响应时间减少约 94%」

- **可答**：秒杀请求只做 Lua 校验 + 发 RocketMQ 消息，不做同步写库，写库由 **RocketMQ 消费者**异步处理，所以接口 TPS 明显提升、响应时间大幅下降；具体倍数和百分比可压测对比「同步落库 vs RocketMQ 异步落库」得到。

## 七、一致性 & 可靠性

### 7.1 缓存和数据库一致性

- **策略**：Cache-Aside，更新店铺时先 `updateById(shop)`，再 `delete(CACHE_SHOP_KEY + id)`，下次读时回源并写缓存。
- **可答**：我们采用「先更新 DB 再删缓存」，读时总是以 DB 为准回填，避免写缓存失败导致长期脏读；强一致场景可以再考虑加锁或延迟双删，我们场景是最终一致即可。

### 7.2 消息会不会丢？

- **新版（RocketMQ）**：Broker 持久化消息；消费者正常处理完后 ACK，异常抛出则触发重试（最多 16 次）；发送端若发送失败会**回滚 Redis**（库存 +1、Set 移除用户），保证 Redis 与最终 DB 一致。
- **可答**：RocketMQ 持久化 + 消费 ACK + 重试保证消息不丢；发送失败时我们主动回滚 Redis，避免「Redis 已扣库存但消息未发出」导致数据不一致。

### 7.3 幂等

- **秒杀**：Lua 里 SADD 用户到 `seckill:order:{voucherId}` 天然幂等；消费者内先查库判断是否已有该 user+voucher 订单，有则直接返回；insert 时若发生 **DuplicateKeyException**（消息重复消费或订单 ID 冲突），判断是否为同一用户+优惠券，是则视为幂等成功并返回。

## 八、异常与兜底

### 8.1 缓存穿透

- **现象**：查不存在的数据，缓存没有，每次都打到 DB。
- **实现**：`queryWithPassThrough` 里，DB 查不到时往 Redis 写空串（或占位值），TTL 设短（如 2 分钟），下次同一 key 直接命中空值不再查 DB。

### 8.2 缓存击穿

- **现象**：热点 key 过期瞬间大量请求打到 DB。
- **实现**：  
  - **互斥锁**：`queryWithMutex`，未命中时先抢锁 `lock:shop:{id}`，抢到再查 DB 并写缓存，抢不到则休眠重试或直接返回旧数据；  
  - **逻辑过期**：value 里带过期时间，不设 Redis TTL；过期后由拿到互斥锁的线程异步重建，其它线程仍返回旧数据，避免同时打 DB。

### 8.3 缓存雪崩

- **现象**：大量 key 同时过期，请求集中打 DB。
- **实现**：动态 TTL，在基础 TTL 上加随机值（如 30 + random(10) 分钟），打散过期时间；或逻辑过期不依赖 Redis TTL，由业务时间控制。

### 8.4 分布式锁过期了怎么办？

- **可答**：Redisson 有看门狗，默认 30s 续期，业务没执行完会自动续期，避免「业务执行时间 > 锁 TTL」导致锁被释放；自己实现的 SimpleRedisLock 要设合理 TTL 或实现续期逻辑。

## 九、项目难点与亮点（STAR）

### 9.1 难点示例

- **场景**：秒杀要同时保证高性能、不超卖、一人一单。  
- **做法**：Lua 在 Redis 内原子完成库存与一人一单校验；通过 **RocketMQ** 异步落库，发送失败时回滚 Redis；消费者内 Redisson 锁 + 幂等检查 + 乐观锁扣库存 + DuplicateKey 幂等处理。  
- **结果**：接口只做 Redis + 发消息，TPS 高；超卖和重复下单通过 Lua + RocketMQ 解耦 + 消费者锁与乐观锁多重保障为 0。

### 9.2 亮点示例

- **Feed 流**：用 ZSet 做推模式收件箱，时间戳做 score，滚动分页用 `ZREVRANGEBYSCORE` + offset，支持万级粉丝 1 秒内触达（发笔记时 N 次 ZADD，N 为粉丝数）。  
- **缓存模板**：`CacheClient` 抽象出 PassThrough / Mutex / LogicalExpire 三种策略，泛型 + 回调，店铺、类型等都可复用，便于维护和扩展。

## 十、扩展与演进

### 10.1 如何压测

- 用 JMeter 或 ab 对「店铺详情」「秒杀接口」压测，观察 QPS、RT、DB 连接数、Redis 命中率；对比开/关缓存、同步/异步落库的差异，得到简历上的百分比数据。

### 10.2 如何扩容

- Redis 做主从/集群；MySQL 做主从读分离；秒杀可多实例 + Redis 集中存库存；Feed 推模式粉丝量特别大时可考虑拉模式或混合（大 V 用拉，普通用户用推）。

### 10.3 重做会怎么改

- 缓存：可考虑多级缓存（本地 + Redis）、缓存预热。  
- 秒杀：可把库存提前放到 Redis，全程在 Redis 扣减，定时或批量同步 DB；MQ 用 RocketMQ/Kafka 做持久化和集群。  
- Feed：粉丝量极大时推模式写放大明显，可按粉丝数分档（如 >N 用拉、≤N 用推）。

## 十一、八股追问速记

### 11.1 Redis 数据结构在你项目里的用法

- **String**：缓存店铺 JSON、验证码、库存计数（seckill:stock:{voucherId}）、分布式锁 key、全局 ID 日期序列。  
- **Hash**：登录用户信息（login:token:xxx）。  
- **Set**：秒杀已购用户集合（seckill:order:{voucherId}）。  
- **ZSet**：Feed 收件箱（feed:{userId}）、点赞列表（blog:liked:{blogId}），score 为时间戳。  
- **GEO**：附近商户（shop:geo:{typeId}），按类型存店铺坐标，按距离分页查询。  
- **消息队列**：订单异步落库使用 **RocketMQ**（seckill-order-topic），不在 Redis。

### 11.2 为什么用 Lua？

- 多个命令要原子执行（判库存、判一人一单、扣库存、SADD），Lua 在 Redis 单线程执行，无中间态，避免并发问题；减少多次网络往返。消息发送放在 Java 侧用 RocketMQ 完成，Lua 只负责 Redis 内操作。

### 11.3 Redisson 锁原理

- 底层用 Lua 加锁（存在则续期或等待），key 默认 30s，看门狗每 10s 续期；解锁时校验线程标识，避免误删别人锁；可重入用 Hash 存重入次数。

### 11.4 ThreadLocal 作用与注意点

- **作用**：在当前线程内保存用户信息（UserDTO），Controller/Service 随时取，避免层层传参。  
- **注意**：请求结束必须在 `afterCompletion` 里 `remove`，否则线程复用时可能串用户或内存泄漏。

### 11.5 雪花算法 / 全局 ID

- 项目用 **Redis 自增 + 时间戳**：`RedisIdWorker#nextId`，key 为 `icr{prefix}:yyyy:MM:dd`，incr 得到当日序列号，再与「当前秒级时间戳 - 起始时间戳」左移 32 位拼接，保证趋势递增、同一天内不重复，且便于从 ID 反推时间。
