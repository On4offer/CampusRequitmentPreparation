## 二、业务场景

### 考察点
- 能否说清需求的来源、用户、价值。
- 能否区分「功能需求」和「非功能需求」（性能、安全、体验）。

### 回答要点（按功能准备 1 分钟版本）

**Excel 商品批量导入**
- **场景**：总部/门店入库时，商品条目多，手工录入慢且易错。
- **用户**：仓库/门店管理员。
- **价值**：支持 .xls/.xlsx、100+ 条导入、模糊+精确两步匹配、部分成功部分失败仍可导入，提升入库效率。

**密码过期强制修改**
- **场景**：满足安全合规，定期强制改密。
- **用户**：所有登录员工。
- **价值**：3 个月过期、登录时校验、过期跳转修改页、修改后更新 `PasswordChangeTime`，兼顾安全和体验。

**审计日志**
- **场景**：满足审计要求，可追溯谁在何时做了什么。
- **用户**：审计/管理员查看。
- **价值**：登录日志（成功/失败/登出）+ 操作日志（增删改、导出下载上传），异步记录、敏感信息脱敏，不阻塞主流程。

**入库单标识**
- **场景**：业务需要对部分入库单做标记便于筛选和跟进。
- **用户**：仓库/门店操作人员。
- **价值**：创建/修改时可勾选标识，列表按「已标识/未标识」筛选，列表用星标展示，支持直接入库、退货入库、自行采购。

**Bug 修复（业绩报表 / 客户推荐）**
- **场景**：报表缺移动端消耗数据；推荐类型客户保存时报「请先匹配推荐人」。
- **价值**：报表数据完整、客户录入流程可走通。

### 常见追问
- **Q：需求是谁提的？有没有和产品/业务确认过？**  
  A：可以答：需求来自产品/业务，自己负责技术方案和实现，关键逻辑（如 3 个月过期、标识默认值）与文档/同事确认过。
- **Q：如果业务后来要改成「6 个月过期」或「支持按门店配置」？**  
  A：过期时间可做成配置项（库表或配置中心）；按门店配置可加「门店-策略」表或配置结构，登录校验时按当前门店取策略。

## 三、表设计

### 考察点
- 字段是否合理（类型、可空、默认值）。
- 是否考虑历史数据、索引、扩展。

### 回答要点

**密码过期**
- 表：`employees`，新增 `PasswordChangeTime`（datetime, NULL）。
- NULL 表示从未修改视为过期；默认不设 0，用 NULL 语义更清晰。
- 判断：`PasswordChangeTime == null || PasswordChangeTime < Now().AddMonths(-3)`。

**审计日志**
- `sys_login_log`：id, emp_id, emp_name, account, clinic_id, clinic_name, comp_id, login_time, login_type(1成功/2失败/3登出), login_ip, user_agent, fail_reason 等；按 emp_id、login_time、login_type 建索引便于查询。
- `sys_operation_log`：id, emp_id, operation_type(增/删/改/导出/下载/上传), api_path, operation_desc, request_params(脱敏), operation_result, duration, operation_time 等；按 emp_id、operation_time、api_path 建索引。
- 请求参数做长度限制和脱敏再存，避免敏感信息和大字段拖垮性能。

**入库单标识**
- 门店表 `repertoryenters`、总部表 `tbl_repertoryenter` 均新增 `isMarked` TINYINT(1) DEFAULT 0，0=未标识、1=已标识。
- 历史数据默认 0，兼容老数据；若筛选频繁可在 `isMarked` 上建索引。

### 常见追问
- **Q：为什么登录日志和操作日志分两张表？**  
  A：读写场景不同（登录按人/时间查，操作按人/时间/接口查）；字段差异大；分表便于按表做归档或清理策略。
- **Q：日志表数据量大了怎么办？**  
  A：按时间分区或归档历史数据；只查近期可加时间范围索引；考虑冷热分离或写入 MQ 再落库，减轻主库压力。

## 四、技术选型

### 考察点
- 为什么用 A 不用 B，有无对比。
- 是否了解所选技术的适用场景和局限。

### 回答要点

**Excel 解析：Apache POI**
- 选型：Java 生态常用，支持 .xls/.xlsx，`WorkbookFactory.create()` 可自动识别格式。
- 对比：EasyExcel 更省内存，适合超大文件；当前需求 10MB、百条级，POI 足够，且项目已有依赖。

**审计日志：Spring 拦截器 vs AOP**
- 选型：用拦截器（Interceptor）统一记录操作日志。
- 理由：与现有请求处理链路一致、不引入新 AOP 依赖、可针对路径配置拦截/排除规则（如排除 query/list）。

**密码加密：MD5**
- 说明：现有系统沿用 MD5（大写），自己实现时保持一致。
- 若被问「MD5 是否安全」：可答已知有碰撞风险，新系统更推荐 bcrypt/Argon2；改造需统一加密方式与历史数据迁移策略。

**前端请求：FormData 上传**
- 选型：Excel 上传用 FormData，Content-Type multipart/form-data。
- 注意：拦截器里若对 data 做 `Qs.stringify` 会破坏 FormData，需判断 `config.data instanceof FormData` 时跳过。

### 常见追问
- **Q：Excel 再大一点（如 10 万行）怎么搞？**  
  A：换流式解析（如 EasyExcel）、分批入库、异步任务+进度查询；前端可做分片上传或先传后查结果。
- **Q：审计日志为什么不用 MQ？**  
  A：当前量级异步写库可接受；若量很大或要解耦，可以改为发 MQ，消费者落库，便于削峰和扩展。

## 五、代码实现

### 考察点
- 分层是否清晰，职责是否单一。
- 关键逻辑能否讲清（匹配策略、异常处理、复用）。

### 回答要点

**Excel 导入分层**
- Controller：接收文件、Token/权限校验、调 Service。
- Service：文件格式与大小校验、调工具类解析、拼装返回结果（successList + errorList）。
- 工具类 ExcelImportUtils：解析表头、逐行解析、商品匹配、数据校验；总部入库与门店入库共用同一工具类，避免重复逻辑。

**商品匹配策略（两步）**
- 第一步：按名称 LIKE 查库，缩小候选。
- 第二步：内存里按「名称+规格」去空格后精确匹配；若无匹配，收集相似商品用于错误提示（相似商品推荐）。

**审计日志异步**
- Java：`CompletableFuture.runAsync(() -> { 记录日志 });`，主流程不等待。
- .NET 调 Java 记登录日志：`Task.Run(() => { 发 HTTP 到 Java 接口 });`，失败只打日志，不影响登录结果。

**入库单标识**
- 后端：实体加 `isMarked`；save/update 从请求取 `isMarked` 写入；index 查询支持 `isMarked` 条件与返回字段。
- 前端：创建/编辑用 `el-switch` 绑定 `temp.isMarked`；列表用 `el-select` 筛选项 + 表格列用星标图标展示。

### 常见追问
- **Q：工具类为什么没放在 Service 里？**  
  A：Excel 解析和匹配是通用能力，多处 Service 复用；独立工具类便于单测和复用，符合单一职责。
- **Q：异步写日志失败了怎么办？**  
  A：catch 里打错误日志，不抛出去，保证主流程不受影响；若要更可靠可后续改为写 MQ 或本地队列再落库。

## 六、难点与坑点（问题排查与解决）

### 考察点
- 是否真动手排查过问题。
- 思路是否清晰（复现 → 定位 → 根因 → 方案）。

### 回答要点（挑 2～3 个讲透）

**1. Excel 上传 501：FormData 被 Qs.stringify**
- 现象：门店入库导入 Excel 报 501。
- 排查：对比总部/门店请求差异，发现 request 拦截器对 body 统一做了 `Qs.stringify`，FormData 被转成字符串，文件丢失。
- 解决：在拦截器里判断 `if (!(config.data instanceof FormData))` 再 `Qs.stringify`。

**2. 业绩报表移动端消耗数据缺失**
- 现象：消耗管理有移动端数据，报表里没有。
- 排查：沿调用链找到报表 SQL 的过滤条件，发现 `isExclusionChannels=1` 时有一段 not exists 子查询，未区分移动端（如 `is_network_order`），把移动端数据一起过滤掉了。
- 解决：在子查询中加上对移动端渠道的判断，保留移动端消耗数据。

**3. 客户推荐保存失败：「请先匹配推荐人信息」**
- 现象：客户来源为推荐类型且已匹配推荐人，保存仍报错。
- 排查：在 update 方法里找到该提示，发现只校验了「需要推荐人」，但没有正确依赖「是否已绑定推荐人」的数据或接口，导致逻辑错误。
- 解决：补充推荐人绑定校验（如查 `customers_recommends` 或调用 bindingRecommend 的校验逻辑），并注入所需 Mapper/Service。

**4. 入库单查询报 Unknown column 'isMarked'**
- 原因：只给总部表加了字段，门店表 `repertoryenters` 未加。
- 解决：两个表都执行 ALTER 添加 `isMarked`，并做历史数据默认值更新。

**5. ExcelImportUtils 未被 Spring 扫描**
- 现象：工具类加了注解但注入失败。
- 解决：在启动类的 `@ComponentScan` 中增加对应包（如 `cnbwx.util`）。

### 常见追问
- **Q：你是怎么定位到是拦截器的问题的？**  
  A：对比能成功的请求（总部）和失败的请求（门店），看请求体、Content-Type、拦截器配置差异，再单步或打日志确认拦截器对 data 的处理。
- **Q：如果线上没有日志，你怎么查？**  
  A：先复现（用相同参数、环境）；加日志或远程调试再抓一次；查 DB 和中间件状态；看监控和错误码。

## 七、性能与扩展

### 考察点
- 是否考虑过大流量、大数据量、超时。
- 是否有异步、批量、限流等意识。

### 回答要点

**Excel 导入**
- 当前：10MB、百条级，POI 解析 + 逐行匹配，响应控制在 10 秒内。
- 扩展：文件更大或行数更多时，可改为 EasyExcel 流式读、分批查商品与写入、异步任务+任务 ID 查询结果；前端可限制单次文件大小或提示「大批量请分批」。

**审计日志**
- 当前：异步写库，不阻塞主流程。
- 扩展：若 QPS 很高，可先写内存队列或 MQ，再批量落库；或按时间分表/分区，便于归档和查询。

**密码修改**
- 无高并发瓶颈点；若做「修改密码后踢掉其他端登录」等，再考虑 token 失效与缓存更新即可。

### 常见追问
- **Q：导入时如何避免把数据库打满？**  
  A：限制单次行数或文件大小；批量查商品时用 IN 分批、控制每批条数；必要时加限流或队列排队。
- **Q：日志表写太多会不会影响业务库？**  
  A：已用异步；若仍担心可独立库/表、或写从库；或先写 MQ 再消费落库，与业务库隔离。

## 八、安全与合规

### 考察点
- 敏感信息处理、权限、审计、密码策略。

### 回答要点

**审计日志**
- 请求参数落库前脱敏：password、pwd、token 等不记录或打星号；参数长度限制，防止大 body 拖垮存储。
- 只记录必要字段，避免把完整请求体都存下来。

**密码过期**
- 过期判断在密码校验通过之后，避免「密码错误也提示过期」的信息泄露。
- 强制修改接口为匿名接口，只校验旧密码+公司+账号，不依赖登录态；修改后更新 `PasswordChangeTime` 并更新缓存。

**Excel 导入**
- 接口做 Token 校验和权限（如 IsGeneral）；文件类型、大小校验，防止恶意上传。

### 常见追问
- **Q：匿名修改密码如何防暴力破解？**  
  A：可加图形验证码、同一账号短时间失败次数限制、IP 限流；必要时记录失败日志便于风控。
- **Q：操作日志被人篡改怎么办？**  
  A：库权限控制、只追加不提供普通用户改日志的接口；若需更强可考虑写链或只写型存储。

## 九、前后端协作与接口设计

### 考察点
- 接口约定、错误码、文件上传、异常返回。

### 回答要点

**统一返回结构**
- 如：`{ code, msg, data }`，其中 data 里可有 successList、errorList、totalRows 等；登录接口用 success=0 表示成功、13 表示密码过期，前端据此跳转修改密码页。

**文件上传**
- Content-Type：multipart/form-data；参数名 file；前端用 FormData  append 文件，拦截器不对 FormData 做 stringify。

**错误码**
- 密码错误与密码过期区分（如 2 vs 13），前端可分别提示和跳转；Excel 导入返回逐行错误信息与相似商品推荐，便于用户修正。

### 常见追问
- **Q：你们前后端接口文档怎么维护？**  
  A：可答 Swagger/ Apifox 等；或按模块维护 Markdown/Wiki，写清路径、参数、返回示例和错误码。
- **Q：跨域和 Token 怎么处理？**  
  A：Token 放 Header（如 Authorization）；跨域由后端 CORS 或网关配置；登录后前端存 Token，请求拦截器统一带上。

## 十、事务与数据一致性

### 考察点
- 何时开事务、边界在哪、失败如何回滚或补偿。

### 回答要点

**入库单保存**
- 若一次保存涉及主表 + 明细表多条插入/更新，应在 Service 方法上加 `@Transactional`，保证主表与明细表同时成功或同时回滚。
- 简历提到「完善入库模块的事务管理和数据校验」：可展开为「保存前校验数量、金额等，保存时用事务包住主从表写操作」。

**Excel 导入**
- 当前是「解析 + 匹配 + 返回结果」，真正写入入库单由用户在前端确认后再调保存接口；保存接口内部用事务保证一致性。若未来做成「解析后直接落库」，应在单次落库范围内加事务，并考虑部分失败是否回滚整批或只回滚失败行。

**审计日志**
- 日志异步写，失败不影响主流程，不纳入业务事务；若要求「操作必留痕」，可先同步写日志再提交业务事务，或使用异步+可靠投递（如 MQ）保证最终写入。

### 常见追问
- **Q：事务里调了异步方法，异步方法里的异常会导致回滚吗？**  
  A：不会，事务只管理当前线程的 Connection；异步在另一线程，需在异步逻辑里自己做补偿或记录失败。
- **Q：你们入库有没有做库存扣减？**  
  A：若做了，可答库存扣减与入库单在同一事务中，或先扣减再生成入库单，按实际项目说明即可。

## 十一、跨系统与遗留系统（.NET + Java）

### 考察点
- 多技术栈如何协作、接口契约、故障边界。

### 回答要点

**双后端**
- 登录、密码修改在 .NET；部分业务在 Java（如报表、入库、审计日志存储）。前端根据模块调不同 BaseURL（如 request3 / request4 对应不同后端）。
- 审计日志：.NET 登录/登出成功后，通过 HTTP 调 Java 的 `/syslog/loginLog/record` 记日志；Java 侧异步落库，.NET 不关心是否写成功。

**适配注意点**
- 时间、枚举、字符集要统一（如密码 MD5 大写、日期格式）。
- Java 侧若读 .NET 库表（如 employees），需与 .NET 约定字段名和类型，避免 NULL 或类型不一致导致 NPE 或错误。

### 常见追问
- **Q：.NET 调 Java 超时或失败怎么办？**  
  A：记日志日志用 Task.Run 异步调，超时或失败不影响登录结果；若后续要保证「必记」，可加重试或 MQ 补偿。
- **Q：为什么登录在 .NET 而日志在 Java？**  
  A：可答历史原因或团队分工，日志统一在 Java 侧存储和查询，.NET 只负责发一条 HTTP 记录。

## 十二、测试与验证

### 考察点
- 是否主动设计用例、边界、数据校验。

### 回答要点

**Excel 导入**
- 格式：.xls/.xlsx 可导入，非 Excel 或超 10MB 拒绝并提示。
- 数据：必填项缺失、数字/日期格式错误、商品不存在或名称规格不匹配，返回错误列表和相似商品；部分行成功部分失败时，成功行加入列表，失败行在弹窗中展示。
- 性能：100 条、500 条在预期时间内完成（如 5～10 秒）。

**密码过期**
- 未过期可登录；过期则返回 13 并跳转修改页；密码错误不提示过期；修改后时间更新并可登录。
- 边界：PasswordChangeTime 为 NULL、刚好 3 个月（按你们实现是未过期还是过期要一致）。

**审计日志**
- 登录成功/失败/登出各记一条；操作类型（增/删/改/导出/下载）与接口对应；参数中无明文密码/token；异步写失败不影响接口返回。

**入库单标识**
- 创建/修改后列表显示正确；筛选「已标识/未标识/全部」结果正确；历史数据 isMarked 为 0 显示正常。
