2. **Token 校验与用户上下文**

   - 本机测试：吞吐量（TPS）提升约 **15 倍**，平均响应时间减少约 **94%**。

5. **Feed 流推模式**

   - 支持**万级粉丝**用户内容 **1 秒内触达**。

**可补充的技术细节**  
- 秒杀链路：Redis Lua 原子校验与扣库存 → 发 RocketMQ → 消费者内 Redisson 锁 + 乐观锁扣 DB + 写订单；MQ 发送失败时对 Redis 做回滚，保证一致性。  
- 若被问「为什么用 RocketMQ」：解耦、削峰，本机测 TPS 提升约 15 倍、响应时间降约 94%，且支持事务消息，适合下单与落库的最终一致。

### 一、实习项目：门店 Excel 导入报 501，文件上传失败

**现象**  
- 门店侧：在「入库管理」里选择 Excel 文件点击导入，请求报 **501**，后端收不到文件，导入失败。  
- 总部侧：同样的 Excel、同样的接口逻辑，在总部入库里导入是**正常**的。

**复现与对比**  
- 在本地/测试环境用门店账号复现：确认只有「门店入库导入」会 501，总部导入正常。  
- 对比两边请求：接口路径不同（总部走一套前端、门店走另一套），请求体格式不同——总部上传时 body 是正常的 **multipart/form-data**（带文件），门店上传时 body 被改成了**字符串**，文件没了。

**排查过程**  
1. 先看后端：接口期望的是 `multipart/form-data`，参数名 `file`；501 多为「未实现」或请求格式不对，怀疑请求没按约定传。  
2. 看前端：总部和门店分别用了不同的 axios 实例或请求封装（例如 `request3` / `request4`），请求前会走**请求拦截器**。  
3. 在拦截器里发现：对 **所有请求** 的 `config.data` 做了 `Qs.stringify`（把对象序列化成 query 字符串），目的是统一处理普通表单或 JSON 参数。  
4. 门店的导入请求里，`config.data` 是 **FormData**（里面 append 了文件）。一旦被 `Qs.stringify`，FormData 会被转成字符串，**文件内容丢失**，后端拿到的就不是文件流，导致 501 或解析失败。

**根因**  
- 请求拦截器**没有区分**「普通对象」和「FormData」：对 FormData 也执行了 `Qs.stringify`，破坏了文件上传的 body。

**解决方案**  
- 在请求拦截器里，**先判断** `config.data` 是否为 FormData，若是则**不再**对 `data` 做 `Qs.stringify`，直接发原始 FormData。  
- 代码层面：`if (!(config.data instanceof FormData)) { ... Qs.stringify(config.data) ... }`，只有非 FormData 才做序列化。

**验证与收获**  
- 修改后门店导入 Excel 正常，与总部行为一致。  
- 收获：文件上传必须用 `multipart/form-data`，且前端不能对 FormData 做「对象序列化」；以后写拦截器要区分「普通 JSON/表单」和「FormData」，避免类似问题。

**若被追问「怎么定位到是拦截器的」**  
答：通过对比能成功的请求（总部）和失败的请求（门店），看请求体、Content-Type、以及各自走的拦截器/axios 实例差异，再在拦截器里打日志或单步调试，确认对 `config.data` 的处理导致 FormData 被改写。

**实习项目其他问题（可作补充）**  
- **业绩报表缺移动端消耗**：报表 SQL 里 `not exists` 子查询未区分移动端渠道，把移动端消耗一起排除；在子查询中加上对移动端渠道的判断后修复。  
- **客户推荐保存报错**：update 只校验「需要推荐人」未校验「已绑定推荐人」；补充推荐人绑定校验（查关联表或调校验接口）后修复。

### 二、小众点评：秒杀场景的并发与一致性设计（最大难点，对应简历7中的 Redisson + Redis 原子操作 + 乐观锁 + RocketMQ）

**问题与为什么难**  
秒杀要同时满足：**库存不超卖**、**同一用户不能重复下单**、**高并发下不能把数据库打满**。若用传统「查库存 → 扣库存 → 写订单」在应用里顺序执行，多线程下会出现超卖和重复下单；若所有请求都直接打数据库，瞬时 QPS 会压垮 DB。

**解决思路与具体实现（与简历7表述一致）**  

1. **Redis 原子操作（Lua 脚本）**  
   - 在 Redis 里用 Lua 原子执行：查库存、判用户是否已下单（Set）、扣库存、记录用户，避免多步 Redis 调用之间的并发窗口。  
   - 对应简历：**Redis 原子操作**，保证「校验 + 扣减 + 记录」一次完成，不满足条件直接返回（库存不足 / 不能重复下单）。

2. **Redisson 用户维度分布式锁 + 乐观锁**  
   - 消费者里用 **Redisson** 实现**用户维度**的分布式锁（key 如 `lock:order:{userId}`），同一用户只会被一个线程处理；在锁内再查 DB 是否已有订单，有则不再创建。  
   - 扣减数据库库存用**乐观锁**：`update ... set stock = stock - 1 where ... and stock > 0`，防止超卖。  
   - 对应简历：**重复下单率与超卖率均为 0**。

3. **RocketMQ 解耦下单与落库**  
   - Lua 通过后发消息到 RocketMQ，接口立即返回，订单在消费者里创建，请求不直接写库，起到削峰作用。  
   - 对应简历：**TPS 提升约 15 倍，平均响应时间减少约 94%**。

4. **消息发送失败时的回滚**  
   - 若 MQ 发送失败，Redis 已扣库存、已记录用户，但订单未创建，会造成不一致。代码里在 catch 中对 Redis 做回滚（库存加回、Set 移除用户），用户可重试，数据一致。

**验证与收获**  
- 本地/本机压测验证：重复下单率与超卖率均为 0，TPS 与响应时间改善与简历数据一致。  
- 收获：高并发秒杀要把校验与预扣减放在 Redis 并保证原子性，落库通过 MQ 解耦削峰，配合 Redisson 锁与乐观锁防重复与超卖，并设计好失败回滚。

## 3. 项目运行的整个过程，从前端到后端，涉及哪些协议

**说明**：两个项目各答一条完整链路（从前端到后端），并写出涉及的协议；面试时按问到的项目选答。

### 一、实习项目：从前端到后端的一条完整链路（以「门店 Excel 导入」为例）

**流程**  
1. **用户 → 浏览器**：在门店入库页面选择 Excel 文件，点击「导入」。  
2. **前端（Vue）**：表单用 `FormData` 把文件 append 进去，通过 axios 发请求（如门店用 `request4`）；请求前经过**请求拦截器**（这里要对 FormData 做区分，不能 Qs.stringify）。  
3. **前端 → 后端**：  
   - **协议**：**HTTP/HTTPS**，方法 POST，URL 如 `/repertoryenters/importExcel`。  
   - **请求体**：**multipart/form-data**（因为带文件），参数名一般为 `file`；Header 里带 **Authorization**（Token），用于登录态校验。  
4. **后端（Java Spring Boot）**：  
   - Controller 接收 `MultipartFile file`，做权限校验（如 IsGeneral），调 Service。  
   - Service 里做文件格式与大小校验（.xls/.xlsx、如 10MB 限制），调 **ExcelImportUtils** 解析：读表头、逐行解析、按名称模糊查库再内存里精确匹配商品、校验数据，得到 successList 和 errorList。  
   - 若需要落库，再调入库单保存接口（主表+明细表，通常在同一事务内）。  
5. **后端 → 前端**：HTTP 响应，body 为 **JSON**，统一结构如 `{ code, msg, data }`，data 里包含成功条数、失败列表、相似商品推荐等。  
6. **前端 → 用户**：根据 code 提示成功或失败，展示失败行和原因。

**涉及协议小结**  
- **用户到后端**：**HTTP(S)**，上传为 **multipart/form-data**，认证用 Header 里的 Token。  
- **后端内部**：Java 调 **MySQL** 用 **TCP + MySQL 协议**（JDBC）；若 .NET 调 Java 记审计日志，则为 **HTTP**（如 POST JSON）。  
- **响应**：**HTTP + JSON**。

**若选「登录 + 审计日志」这条链路**  
- 用户登录在 **.NET** 完成（HTTP 请求到 .NET，表单或 JSON）；登录成功后 .NET 侧用 **HTTP** 调 Java 的日志接口（如 `/syslog/loginLog/record`），Java 异步写库；前端拿到的仍是 .NET 返回的 Token 和登录结果。协议：浏览器 ↔ .NET 为 HTTP；.NET ↔ Java 为 HTTP；Java ↔ MySQL 为 TCP/MySQL 协议。

### 二、小众点评：从前端到后端的一条完整链路（以「秒杀」为例，对应简历7技术栈）

**流程**  
1. **用户 → 浏览器**：在秒杀活动页点击「立即抢购」。  
2. **前端（Vue 等）**：发 **HTTP** 请求，如 `POST /voucher-order/seckill/{id}`，Header 里带 **Authorization**（Token），body 可为空或简单 JSON。  
3. **前端 → 后端**：  
   - **协议**：**HTTP/HTTPS**，**RESTful**，数据格式一般为 **JSON**（Content-Type: application/json）；这里秒杀可能无 body，参数在路径里。  
4. **后端（Spring Boot）**：  
   - 拦截器/过滤器做 **Token 校验**，取出 userId。  
   - 执行 **Redis Lua 脚本**：与 **Redis** 通信，协议为 **RESP（Redis 协议）**，底层 **TCP**；Lua 里完成库存判断、扣 Redis 库存、记录用户到 Set。  
   - Lua 返回 0 表示有资格：生成订单 ID（如用 Redis 自增或雪花算法），把消息发到 **RocketMQ**（**TCP + RocketMQ 协议**），接口立即返回「排队中」。  
   - 若发 MQ 失败，对 Redis 做 **回滚**（INCRBY 库存、SREM 用户）。  
5. **RocketMQ 消费者**：  
   - 从 MQ 拉取消息（**TCP + MQ 协议**），拿到 voucherId、userId、orderId。  
   - 用 **Redisson** 抢 **分布式锁**（与 **Redis** 通信，RESP），锁内：查 **MySQL**（**TCP + MySQL 协议**）是否已有订单，若无则乐观锁扣减库存、插入订单表。  
6. **后端 → 前端**：秒杀接口的 HTTP 响应为 **JSON**，如 `{ code, msg }` 表示成功排队或失败原因（库存不足/重复下单）。  
7. **前端 → 用户**：展示「抢购成功排队中」或「库存不足」等；订单结果可轮询或通过 WebSocket/推送另行通知（本项目多为轮询或刷新订单列表）。

**涉及协议小结**  
- **用户到后端**：**HTTP(S)**，**JSON**，Token 在 Header。  
- **后端 ↔ Redis**：**TCP + RESP（Redis 协议）**。  
- **后端 ↔ RocketMQ**：**TCP + RocketMQ 协议**。  
- **消费者 ↔ MySQL**：**TCP + MySQL 协议**。  
- **响应**：**HTTP + JSON**。

## 4. 讲一下 IO 多路复用

- **含义**：在一个（或少量）线程里，同时监听多个 IO 描述符（如多个 socket），哪个就绪就处理哪个，避免为每个连接开一个线程，从而节省资源、提高并发。
- **常见实现**：select、poll、epoll（Linux）。
- **典型应用**：Redis 单线程高并发、Nginx 高并发，都依赖 IO 多路复用（Linux 下多用 epoll）在单进程/单线程里处理大量连接。
- **和你的项目**：小众点评里用 Redis 做缓存和秒杀校验，Redis 之所以单线程还能支撑高并发，原因之一就是用了 **IO 多路复用（epoll）**，网络 IO 不阻塞工作线程，请求在队列里依次执行，避免锁竞争。

---

## 5. select、poll、epoll 有什么区别

| 维度       | select                    | poll                      | epoll                         |
|------------|---------------------------|---------------------------|-------------------------------|
| 数据结构   |  fd_set 位图              |  pollfd 数组              | 内核维护的「就绪链表」+ 红黑树 |
| 最大 fd 数 | 一般 1024                 | 无硬编码上限              | 无硬编码上限                  |
|  fd 拷贝   | 每次调用都拷贝到内核      | 同上                      | 注册一次，不用每次拷贝        |
| 就绪方式   | 遍历所有 fd 看是否就绪    | 同上                      | 只返回就绪的 fd，O(1)         |
| 触发方式   | 水平触发                  | 水平触发                  | 支持水平/边缘触发             |
| 性能       |  fd 多时线性下降          | 同上                      | 活跃连接少时更高效            |

- **结论**：Linux 下高并发网络服务多用 **epoll**；Redis、Nginx 在 Linux 上都用 epoll。面试可答：Redis 单线程 + epoll 多路复用，能高效处理大量连接。

## 6. 讲一下 JVM 虚拟机

- **定位**：JVM 是 Java 程序的运行环境，负责加载 class、执行字节码、管理内存（堆、栈、方法区等）、垃圾回收等。
- **内存结构（常问）**：
  - **程序计数器**：线程私有，存当前字节码指令地址。
  - **虚拟机栈**：线程私有，方法调用对应栈帧（局部变量、操作数栈、返回地址等），栈溢出会 StackOverflowError。
  - **本地方法栈**：为 native 方法服务。
  - **堆**：线程共享，放对象实例，GC 主要区域，OOM 常见于 heap space。
  - **方法区（JDK8 后为元空间 Metaspace）**：线程共享，存类元信息、常量池、静态变量等，在本地内存不在堆里。
- **类加载**：加载 → 验证 → 准备 → 解析 → 初始化，双亲委派模型（先让父加载器加载，避免重复加载核心类）。
- **垃圾回收**：对象主要在堆上；判断可回收（如引用计数、可达性分析）；常见算法有标记-清除、复制、标记-整理；分代收集（新生代、老年代），常见器有 Serial、Parallel、CMS、G1 等。

结合项目可说：实习/个人项目都是 Spring Boot，跑在 JVM 上，线上可配合 -Xms/-Xmx 等做堆大小设置，避免 OOM。

## 7. 虚拟内存是什么（及常见追问）

- **虚拟内存**：每个进程看到的是独立的虚拟地址空间，由操作系统和 MMU 映射到物理内存（或磁盘 swap）。进程用虚拟地址访问，内核负责把虚拟页映射到物理页或换页。
- **作用**：进程隔离、扩大「可用内存」（用磁盘做扩展）、统一编址、支持按需分页和换页。
- **常见追问**：
  - **和物理内存区别**：物理内存是真实 RAM；虚拟内存是抽象，可能映射到物理内存或磁盘。
  - **缺页**：访问的页不在物理内存时发生缺页，由内核调入再访问。
  - **页面置换**：物理页不够时，把部分页换出到磁盘（如 LRU），腾出物理页。

## 8. Java 虚拟机里的内存和操作系统的虚拟内存是什么关系

- **层次关系**：  
  - 操作系统的**虚拟内存**：给整个进程（包括 JVM 进程）提供虚拟地址空间，JVM 作为普通进程运行在这一层。  
  - **JVM 内存**：是 JVM 在进程的虚拟地址空间里再划分出来的区域（堆、栈、元空间等），仍然是在 OS 的虚拟地址空间内，由 JVM 管理和使用。
- **对应关系**：
  - JVM 的堆、栈、元空间等，最终都落在进程的虚拟地址上，再由 OS 映射到物理内存（或 swap）。
  - 例如 -Xmx 设置的堆最大容量，就是 JVM 向 OS 申请/使用的虚拟地址范围，实际占用的物理内存在使用时才通过缺页等方式分配。
- **一句话**：JVM 内存是「进程虚拟地址空间里由 JVM 规划的那一块」，和 OS 的虚拟内存是「被包含与包含」的关系，不是两套独立体系。

## 10. 代码题：Hot100 原题——岛屿数量

**题意**：二维网格 `grid`，`'1'` 表示陆地，`'0'` 表示水，求岛屿数量（上下左右连通的 `'1'` 算同一座岛）。

**思路**：DFS 或 BFS。遍历每个格子，遇到 `'1'` 就计数一次并把整座岛「淹没」（DFS/BFS 把相连的 `'1'` 全标记掉），这样每个岛只被计一次。

**DFS 示例（Java）：**

```java
public int numIslands(char[][] grid) {
    if (grid == null || grid.length == 0) return 0;
    int rows = grid.length, cols = grid[0].length;
    int count = 0;
    for (int r = 0; r < rows; r++) {
        for (int c = 0; c < cols; c++) {
            if (grid[r][c] == '1') {
                count++;
                dfs(grid, r, c, rows, cols);
            }
        }
    }
    return count;
}

private void dfs(char[][] grid, int r, int c, int rows, int cols) {
    if (r < 0 || r >= rows || c < 0 || c >= cols || grid[r][c] == '0') return;
    grid[r][c] = '0';
    dfs(grid, r - 1, c, rows, cols);
    dfs(grid, r + 1, c, rows, cols);
    dfs(grid, r, c - 1, rows, cols);
    dfs(grid, r, c + 1, rows, cols);
}
```

- **时间**：O(M×N)  
- **空间**：O(M×N)（递归栈最坏情况）

**面试可说**：遍历网格，遇到陆地就岛屿数 +1，并用 DFS 把整块陆地标记成水，避免重复计数。
