# 📌 面试题

Spring 中的循环依赖是如何解决的？三级缓存的作用是什么？

------

### ✅ 一、概念回答

- **循环依赖**：多个 Bean 在创建过程中互相依赖，导致无法完成实例化。
  - 例如：A 依赖 B，B 又依赖 A。
- Spring 为了支持这种场景，引入了 **三级缓存机制**，保证单例 Bean 的依赖注入可以完成。

------

### ✅ 二、Spring 中循环依赖的分类

1. **构造器循环依赖**：
   - A 构造函数需要 B，B 构造函数又需要 A → **无法解决**（会直接抛异常）。
   - 解决方案：调整设计 / 使用 `@Lazy` 延迟注入。
2. **Setter/Field 循环依赖（常见）**：
   - A 注入 B，B 注入 A → Spring 可以通过 **三级缓存** 提前暴露 Bean 引用来解决。

------

### ✅ 三、Spring 解决循环依赖的原理

在 **单例 Bean 的创建流程**中，Spring 用了三级缓存：

1. **singletonObjects（一层缓存，成品池）**
   - 已经完成实例化和属性注入的 **完整 Bean**。
2. **earlySingletonObjects（二层缓存，半成品池）**
   - 提前曝光的 **成品 Bean 引用**，还没填充属性，但可被依赖。
3. **singletonFactories（三层缓存，工厂池）**
   - 存放一个 **ObjectFactory**，可以生成该 Bean 的早期引用（可能包含 AOP 代理）。

------

### ✅ 四、三级缓存解决流程举例

1. Spring 创建 A → 发现 A 需要注入 B。
2. 容器发现 B 还没创建 → 去创建 B。
3. 创建 B 时，发现依赖 A → 从三级缓存获取 A 的**早期引用**（ObjectFactory 生成）。
4. B 成功注入 A（早期引用），完成初始化，放入一级缓存。
5. 回到 A，继续完成 A 的属性注入，把最终成品放入一级缓存。

👉 **三级缓存的关键作用**：

- **解决 setter 循环依赖**：提前暴露 Bean 的早期引用，避免死锁。
- **支持 AOP 代理**：三级缓存中保存的是 `ObjectFactory`，可以在需要时返回代理对象。
- **分层设计**：保证性能（一级）、一致性（一级和二级）、扩展性（三级支持代理）。

------

### ✅ 五、结合项目经验

- 在 **苍穹外卖** 项目中，我们有多个 Service 层相互调用的情况（如订单服务依赖用户服务，用户服务又依赖消息服务），Spring 默认的三级缓存机制解决了循环依赖问题。
- 如果使用的是 **构造器注入**，就会抛出循环依赖异常，因此我们一般采用 **Setter 注入** 或 **@Lazy** 来规避。

------

### ✅ 六、扩展追问（常见面试深挖）

1. 为什么需要三级缓存，二级缓存不行吗？
   - 二级缓存只能存放早期 Bean，无法解决 AOP 代理场景；三级缓存通过 `ObjectFactory` 延迟生成代理对象。
2. 构造器循环依赖为什么解决不了？
3. SpringBoot 中如果用了 `@Scope("prototype")`，循环依赖还能解决吗？（答：不能，prototype Bean 不会放入缓存池）。
4. 如何手动关闭 Spring 的循环依赖处理？（`spring.main.allow-circular-references=false`）。
5. 如果业务中确实出现构造器循环依赖，该怎么解决？（重构依赖关系、用 Setter/@Lazy、拆分职责）。

------

⚡ **标准回答模板（面试时可直接说）：**

> Spring 中的循环依赖主要出现在单例 Bean 的 Setter 注入场景。Spring 通过 **三级缓存**来解决：一级缓存放完整 Bean，二级缓存放半成品 Bean，三级缓存存放 ObjectFactory 以便生成 Bean 的早期引用。这样当 A 依赖 B，B 又依赖 A 时，可以先把 A 的早期引用暴露给 B，等 B 创建完成后再回填到 A，从而解决循环依赖。需要注意，构造器循环依赖是无法解决的。在实际项目中，比如苍穹外卖，我遇到过服务之间互相依赖的问题，Spring 的三级缓存机制就帮助我们解决了。
