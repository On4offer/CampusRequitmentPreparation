# 面试题：MVCC（多版本并发控制）的原理是什么？在什么场景下使用？

------

## 1. 概念

- **MVCC（Multi-Version Concurrency Control，多版本并发控制）**
  - 是 InnoDB 实现 **事务隔离** 的重要机制。
  - 核心思想：**读操作不阻塞写，写操作也不阻塞读**。
  - 通过保存数据的多个版本（快照），让事务在并发场景下也能读到符合隔离级别要求的数据。

------

## 2. 原理

- **依赖的数据结构**
  1. **隐藏字段**：每行数据都有两个隐藏列：
     - `trx_id`：记录最近一次修改该行的事务 ID。
     - `roll_pointer`：指向 Undo Log 中的旧版本。
  2. **Undo Log（回滚日志）**：存储数据的历史版本，用于回滚和快照读。
  3. **Read View（读视图）**：事务开始时生成，记录当前活跃事务 ID，用来判断哪些数据对当前事务可见。
- **实现方式**
  - **快照读（Snapshot Read）**：普通 `SELECT`，通过 MVCC 读取历史版本，不加锁。
  - **当前读（Current Read）**：`SELECT ... FOR UPDATE`、`UPDATE`、`DELETE`，读取最新数据并加锁。
- **不同隔离级别下的表现**
  - **Read Committed**：每次查询都会生成新的 Read View。
  - **Repeatable Read**（MySQL 默认）：事务第一次查询时生成 Read View，全程复用，保证可重复读。

------

## 3. 案例

- 事务 A：查询 `余额=100`
- 事务 B：更新余额为 `200` 并提交
- 事务 A 在 **可重复读（RR）** 下再次查询 → 依然看到 `100`（快照读）。
- 事务 A 如果执行 `SELECT ... FOR UPDATE` → 会看到 `200`（当前读）。

------

## 4. 使用场景

- **高并发读写场景**（电商库存、订单查询）：避免加锁阻塞，提高性能。
- **金融交易系统**：保证可重复读，防止读到未提交数据。
- **日志查询**：历史快照读，避免被写操作干扰。

------

## 5. 面试模板回答

> MySQL InnoDB 使用 **MVCC** 来解决读写冲突问题。
>  它的实现依赖于 **隐藏列（事务 ID、回滚指针）、Undo Log 和 Read View**。
>  普通查询使用快照读，不加锁，通过比较事务 ID 来判断数据版本的可见性；而更新和加锁查询属于当前读，需要拿到最新数据。
>  在 **读已提交（RC）** 下，每次读都会生成新快照，而在 **可重复读（RR）** 下，同一事务始终复用一个快照，从而避免不可重复读。
>  这样做的好处是 **读写不阻塞，大幅提升并发性能**。
