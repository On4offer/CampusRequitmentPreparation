# 🎯 面试题

**synchronized 锁升级的过程（无锁 → 偏向锁 → 轻量级锁 → 重量级锁）？**

------

## ✅ 标准回答

### 1. 背景

- 在 JDK 1.6 之后，JVM 对 `synchronized` 做了大量优化，引入了 **偏向锁、轻量级锁、重量级锁**，通过“锁升级”来减少性能开销。
- **核心思想**：根据竞争程度，逐步升级锁的级别。

------

### 2. 锁升级过程

1. **无锁状态**
   - 对象头的 Mark Word 存放哈希值、GC 信息等。
2. **偏向锁（Biased Lock）**
   - 第一次加锁时，把线程 ID 写入对象头 Mark Word，之后该线程再次进入同步块时，无需 CAS，直接进入。
   - **特点**：适合只有一个线程访问的场景，几乎零开销。
   - **撤销**：如果有其他线程竞争，偏向锁会被撤销，升级为轻量级锁。
3. **轻量级锁（Lightweight Lock）**
   - 线程通过 CAS 尝试将对象头的 Mark Word 替换为指向自己栈中锁记录的指针。
   - 如果 CAS 成功，获得锁；如果失败（表示存在竞争），线程会**自旋**等待。
   - **特点**：适合短时间的多线程竞争，避免线程阻塞和唤醒。
4. **重量级锁（Heavyweight Lock）**
   - 当自旋超过一定次数或线程竞争激烈时，锁会膨胀为重量级锁。
   - 线程会挂起，进入操作系统的阻塞状态，等待唤醒。
   - **特点**：性能差，但保证了线程安全。

------

### 3. 总结顺序

👉 **无锁 → 偏向锁 → 轻量级锁 → 重量级锁**
 （只能升级，不能降级，除非对象被 GC 回收）

------

## 📌 使用场景

- **偏向锁**：单线程访问对象时，例如方法内局部对象。
- **轻量级锁**：多线程交替访问临界区，竞争不激烈时（比如读写小数据）。
- **重量级锁**：高并发竞争激烈时，保证安全但牺牲性能。
- **黑马点评/苍穹外卖**：
  - 缓存重建时，多个线程可能同时尝试加载热点数据，轻量级锁可减少阻塞。
  - 秒杀下单高并发时，可能会升级为重量级锁，通常要用分布式锁来避免单机锁瓶颈。

------

## ⚠️ 易错点

- **锁不会降级**：不会从重量级回到轻量级或偏向锁。
- **偏向锁默认开启**（JDK 1.6+），但在 JDK 15 后被废弃。
- **自旋次数**过多会浪费 CPU，因此 JVM 会自适应决定是否自旋或直接膨胀为重量级锁。

------

## 🔍 追问

1. **为什么偏向锁能提高性能？**
    👉 避免 CAS 和阻塞/唤醒开销，单线程重入时直接进入。
2. **轻量级锁为什么要自旋？**
    👉 因为大多数锁竞争时间很短，用自旋等待比线程挂起/唤醒开销小。
3. **重量级锁为什么慢？**
    👉 涉及操作系统的阻塞/唤醒（用户态 ↔ 内核态切换），上下文切换开销大。
