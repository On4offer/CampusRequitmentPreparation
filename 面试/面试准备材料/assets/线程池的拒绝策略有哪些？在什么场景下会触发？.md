# 🎯 面试题

**线程池的拒绝策略有哪些？在什么场景下会触发？**

------

## ✅ 标准回答

### 1. 触发时机

当线程池同时满足以下条件时，会触发 **拒绝策略**：

1. 当前运行的线程数 = `maximumPoolSize`（线程数已满）；
2. 任务队列 `workQueue` 已经放不下新任务。

此时线程池无法接收新的任务，交给 **RejectedExecutionHandler** 处理。

------

### 2. JDK 内置的四种拒绝策略

1. **AbortPolicy（默认）**
   - **表现**：直接抛出 `RejectedExecutionException`。
   - **场景**：默认策略，适合对任务结果有强一致性要求的场景（不能丢任务）。
2. **CallerRunsPolicy**
   - **表现**：由提交任务的线程自己执行该任务。
   - **场景**：可以减缓任务提交速度，起到“削峰”作用。比如 Web 请求高峰期，主线程执行任务，间接限流。
3. **DiscardPolicy**
   - **表现**：直接丢弃新提交的任务，不抛异常。
   - **场景**：适合对实时性要求高、允许丢任务的场景（如日志收集）。
4. **DiscardOldestPolicy**
   - **表现**：丢弃队列中最旧的任务，然后尝试重新提交新任务。
   - **场景**：适合需要优先处理最新任务的场景（如消息通知，只关心最新消息）。

------

## 📌 使用场景（结合项目）

- **黑马点评**：
  - 异步记录用户操作日志 → 可用 `DiscardPolicy`，允许丢少量日志。
- **苍穹外卖**：
  - 下单核心流程 → 适合 `AbortPolicy`，不能丢任务，必须报警。
  - 短信通知 → 可用 `DiscardOldestPolicy`，优先保证最新通知。
  - 高并发限流 → 可用 `CallerRunsPolicy`，让主线程自己跑，降低提交速率。

------

## ⚠️ 易错点

- 忘记设置有界队列（`ArrayBlockingQueue`），配合拒绝策略使用。
- 使用无界队列（`LinkedBlockingQueue`）时，**最大线程数参数失效**，拒绝策略几乎不会触发，但可能 OOM。
- 盲目使用 `DiscardPolicy`，可能导致关键任务丢失。

------

## 🔍 追问

1. **如果不希望任务被丢弃，该怎么办？**
    👉 可以用 `CallerRunsPolicy` 或者自己实现拒绝策略，持久化任务到数据库/消息队列。
2. **为什么不推荐 Executors 默认线程池？**
    👉 因为 `newFixedThreadPool` 默认用无界队列，拒绝策略形同虚设，可能导致内存溢出。
