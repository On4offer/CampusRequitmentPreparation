## ❓ RocketMQ 如何避免消息重复消费？常见解决方案有哪些？

### 1. 概念

在分布式环境下，消息传输过程中 **“至少一次投递”** 是常见语义。RocketMQ 默认保证消息不会丢，但可能会 **重复投递**。因此，需要业务端保证 **幂等性**，即相同的消息被多次消费，结果与一次消费一致。

------

### 2. 重复消费的原因

- **生产者重试**：发送超时，但消息其实已写入 Broker，导致重复发送。
- **Broker 重投递**：消费者处理失败或未 ACK 确认，Broker 会重新投递。
- **网络抖动/故障转移**：Consumer 重启或负载均衡时，可能再次拉取已消费的消息。

------

### 3. 常见解决方案（幂等性保障）

1. **唯一业务 ID 去重**
   - 每条消息绑定唯一业务键（如订单号、流水号）。
   - 消费端维护一个去重表（Redis/DB），若已处理则直接丢弃。
2. **数据库唯一约束**
   - 利用数据库的唯一索引防止插入重复数据。
   - 如 `orderId` 做唯一键，重复执行插入语句会直接失败。
3. **乐观锁 / 状态校验**
   - 更新前先校验数据状态，只有符合预期才更新。
   - 如扣减库存时先判断 `stock > 0 AND version = ?`。
4. **分布式锁**
   - 利用 Redis 或 Zookeeper 分布式锁，保证同一条业务消息只有一个消费者成功执行。

------

### 4. 项目案例

在 **黑马点评秒杀项目**：

- 用户下单消息可能会被重复投递。
- 解决办法：下单时使用 **订单号 + 唯一索引**，即使消费者重复消费，数据库只会插入一次。
- 结合 **Redis 去重 Key**（`order:uid:pid`），保证幂等性。

------

### 5. 面试标准回答（背诵版）

“RocketMQ 本身只能保证消息 **至少一次投递**，因此可能会产生重复消费。常见的解决方案是 **业务端保证幂等性**，比如给每条消息绑定唯一业务 ID，在消费端通过 Redis/DB 去重，或者利用数据库唯一索引来防止重复插入。这样即使消息被重复投递，也不会影响最终结果。”
