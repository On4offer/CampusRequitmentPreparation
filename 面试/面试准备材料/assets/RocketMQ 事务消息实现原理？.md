## ❓ RocketMQ 的事务消息原理是什么？如何保证最终一致性？

### 1. 概念

事务消息是 **RocketMQ 独有的分布式事务方案**，用于解决 **本地事务（数据库操作）和消息发送** 的一致性问题。
 目标：确保 **要么数据库和消息都成功，要么都失败**，保证 **最终一致性**。

------

### 2. 原理（两阶段提交 + 回查机制）

RocketMQ 事务消息流程类似 **两阶段提交**：

1. **准备阶段（半消息 / Half Message）**
   - 生产者先向 Broker 发送一条 **半消息**，Broker 先持久化但不会投递给消费者。
2. **执行本地事务**
   - 生产者执行本地事务（如扣减账户余额）。
3. **提交/回滚阶段**
   - 如果本地事务成功 → 发送 **Commit**，Broker 将消息投递给消费者。
   - 如果本地事务失败 → 发送 **Rollback**，Broker 删除该半消息。
4. **事务回查机制**
   - 如果生产者宕机或超时未回应，Broker 会主动回查本地事务状态。
   - 生产者实现 `TransactionListener` 接口，返回 **Commit / Rollback / Unknown**。
   - 如果返回 Unknown，Broker 会多次回查，直到拿到明确结果。

------

### 3. 案例

在 **苍穹外卖支付场景**：

- 用户支付成功 → RocketMQ 先写入半消息。
- 系统执行本地事务：更新订单状态为“已支付”。
- 本地事务成功 → 发送 Commit，MQ 通知骑手端、商家端。
- 如果服务宕机，Broker 会发起回查，确保订单和消息状态一致。

------

### 4. 使用场景

- **电商订单**：支付成功 → 更新订单状态 → 通知物流/库存。
- **金融转账**：扣款成功 → 发送转账消息，保证资金链一致。
- **跨服务交易**：多个服务协作更新，必须保证一致性。

------

### 5. 面试标准回答（背诵版）

“RocketMQ 的事务消息采用 **两阶段提交 + 回查机制**。生产者先发送半消息，再执行本地事务，最后根据事务结果决定提交还是回滚；如果超时未回应，Broker 会回查本地事务状态。这样可以保证本地事务与消息的一致性，实现 **最终一致性**，常用于订单、支付等场景。”
