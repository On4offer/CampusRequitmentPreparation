## 📌 面试题：常见的垃圾回收器有哪些？

### ⭐ 标准回答

JVM 提供了多种垃圾回收器，不同 GC 适合不同的场景，常见的有：

------

### 1. **Serial 收集器**

- **特点**：单线程，进行 GC 时会 Stop-The-World。
- **优点**：简单高效（单核机器下），内存占用小。
- **缺点**：停顿时间长，不适合多核服务器。
- **应用场景**：适用于单核、小内存环境。

------

### 2. **ParNew 收集器**

- **特点**：Serial 的多线程版本，用于新生代。
- **优点**：多线程并行，利用多核 CPU。
- **应用场景**：常与 CMS 搭配使用。

------

### 3. **Parallel Scavenge 收集器（吞吐量优先收集器）**

- **特点**：新生代并行收集器，追求 **高吞吐量**。
- **调优目标**：可通过 `-XX:MaxGCPauseMillis` 和 `-XX:GCTimeRatio` 设置。
- **应用场景**：批处理、科学计算等注重吞吐量的后台任务。

------

### 4. **CMS（Concurrent Mark-Sweep）收集器**

- **特点**：基于 **标记-清除算法**，并发收集，减少停顿时间。
- **优点**：低延迟，用户线程和 GC 线程并发工作。
- **缺点**：
  - 会产生内存碎片。
  - 需要更多 CPU 资源。
  - 可能发生 Concurrent Mode Failure。
- **应用场景**：对响应时间要求高的 Web 系统。

------

### 5. **G1（Garbage-First）收集器**

- **特点**：基于 Region 的分区收集器，标记-整理+复制算法。
- **优点**：
  - 可预测停顿时间（`-XX:MaxGCPauseMillis`）。
  - 减少内存碎片。
  - 更适合大堆内存（>4GB）。
- **应用场景**：JDK9 之后的默认 GC，通用高并发场景。

------

### 6. **ZGC（Z Garbage Collector）**

- **特点**：JDK11 引入，超低延迟 GC。
- **优点**：
  - 停顿时间通常 **<10ms**。
  - 支持超大堆（TB 级）。
- **应用场景**：需要极低延迟的系统，如金融交易。

------

### 7. **Shenandoah 收集器**

- **特点**：RedHat 推出，JDK12 后支持。
- **优点**：
  - 停顿时间与堆大小无关。
  - 与 ZGC 类似，专注低延迟。
- **应用场景**：实时系统、对响应延迟敏感的应用。

------

### 📊 对比总结表

| 收集器            | 代别   | 算法           | 优点         | 缺点               | 场景             |
| ----------------- | ------ | -------------- | ------------ | ------------------ | ---------------- |
| Serial            | 新生代 | 复制           | 简单高效     | 单线程，停顿长     | 单核、小内存     |
| ParNew            | 新生代 | 复制           | 多线程并行   | 吞吐量不高         | 搭配 CMS         |
| Parallel Scavenge | 新生代 | 复制           | 吞吐量优先   | 停顿不可控         | 批处理、后台任务 |
| CMS               | 老年代 | 标记-清除      | 并发低延迟   | 内存碎片、失败风险 | Web 系统         |
| G1                | 整堆   | 标记-整理+复制 | 可预测停顿   | 实现复杂           | 默认 GC，大内存  |
| ZGC               | 整堆   | 标记-整理      | 超低延迟     | 新 GC，配置复杂    | 金融、高实时性   |
| Shenandoah        | 整堆   | 标记-整理      | 停顿与堆无关 | 新 GC，兼容性问题  | 实时应用         |

------

### 🔍 面试追问

1. 为什么 JDK9 默认选择 G1，而不是 CMS？
    👉 G1 可预测停顿时间、无碎片，更适合大内存。
2. ZGC 和 G1 的区别是什么？
    👉 G1 停顿在毫秒级，ZGC 停顿在 **亚毫秒/微秒级**，支持 TB 级堆。
3. CMS 为什么会产生 “Concurrent Mode Failure”？
    👉 因为并发清理时老年代空间不足，只能触发 Full GC。
