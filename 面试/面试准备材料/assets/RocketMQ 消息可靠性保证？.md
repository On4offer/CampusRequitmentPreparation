## ❓ RocketMQ 如何保证消息的可靠传输？

### 1. 概念

消息可靠传输指的是 **消息从生产端发送 → Broker 存储 → 消费端处理** 的整个链路中，消息不会因为网络抖动、Broker 宕机、消费失败等原因而丢失或漏处理。RocketMQ 提供了多层机制保证消息的 **不丢失、不乱序、可恢复**。

------

### 2. 原理（关键环节）

1. **生产者端**
   - 支持 **同步发送（SendResult 确认）**、**异步回调**、**单向发送**。
   - 一般业务用 **同步发送 + 重试机制**（默认重试 2 次）。
   - Producer 内部有 **失败重试、容错机制**，可以切换 Broker。
2. **Broker 端**
   - **刷盘机制**：
     - 同步刷盘：写入 CommitLog 后立刻刷盘，可靠性高。
     - 异步刷盘：先写 PageCache，稍后批量刷盘，性能高。
   - **主从复制**：
     - 同步复制：消息写入主节点同时写入从节点，主从都成功才确认。
     - 异步复制：先写主节点，再异步复制到从节点。
3. **消费者端**
   - **消费确认机制（ACK）**：只有消费者处理成功并返回 ACK，Broker 才会删除消费位点。
   - **失败重试机制**：如果消费失败，消息会进入 **重试队列**，默认最多重试 16 次。
   - **死信队列（DLQ）**：超过重试次数仍失败，消息会进入死信队列，供人工排查。

------

### 3. 项目案例

在 **苍穹外卖项目** 的支付成功通知场景：

- 生产者使用 **同步发送** 保证消息一定发到 Broker。
- Broker 开启 **同步刷盘**，即使宕机也不会丢失数据。
- 消费者（骑手端、商家端）必须 **ACK 确认**，否则消息会进入重试队列，保证不会遗漏推送。

------

### 4. 使用场景

- **订单系统**：支付成功后必须可靠通知库存、物流。
- **秒杀系统**：必须保证下单消息不丢，避免用户成功下单但订单没生成。
- **金融系统**：跨服务的交易流水，要求严格的可靠性。

------

### 5. 面试标准回答（背诵版）

“RocketMQ 从 **生产端、存储端、消费端** 三个环节保证消息可靠性。生产端通过同步发送 + 重试机制确保消息一定到 Broker；Broker 通过 **刷盘机制和主从复制** 保证消息持久化；消费端通过 **ACK 确认、失败重试和死信队列** 确保消息不会丢失。这样可以保证端到端的消息可靠传输。”
