## 📌 提交任务到线程池的执行流程？

**标准回答思路（四步口诀）**
 👉 **先核心 → 再排队 → 后扩容 → 最后拒绝**

------

### 🔹 1. 流程概览

1. **判断核心线程数**
   - 若运行线程数 < `corePoolSize`，**直接创建新线程执行任务**（即便有空闲线程也不会复用）。
2. **任务入队**
   - 否则，尝试放入 `workQueue`。
   - 如果放成功，任务等待被空闲线程取走。
3. **扩容非核心线程**
   - 如果队列已满，并且当前线程数 < `maximumPoolSize`，**创建非核心线程执行任务**。
4. **拒绝策略**
   - 如果线程数已达 `maximumPoolSize` 且队列已满，触发 `RejectedExecutionHandler`（拒绝策略）。

------

### 🔹 2. 底层执行原理

- 提交任务调用 `execute()` → 内部调用 `addWorker()` 创建 `Worker` → `Worker` 包装了线程和 `runWorker()` 方法。
- `runWorker()` 会不断从 **workQueue** 拉取任务并执行（通过 `getTask()`）。
- 线程池运行状态和线程数量通过一个原子变量 `ctl` 管理（高位表示状态，低位表示线程数）。

------

### 🔹 3. 案例应用

- **黑马点评**：热点缓存重建任务提交到线程池时，
  - 核心线程先处理；
  - 大量并发时排队（避免一瞬间打爆 DB）；
  - 超过阈值再扩容少量非核心线程；
  - 最后超限触发拒绝（可选择 `CallerRunsPolicy`，让请求线程自己执行任务，做自然限流）。
- **苍穹外卖**：短信发送/推送通知，若短信网关压力大：
  - 队列兜底，避免一瞬间全部线程爆满；
  - 队列满时让调用线程自己执行，避免消息丢失。

------

### 🔹 4. 面试答题模板

> **回答示例：**
>
> “线程池接收任务后有一个典型的四步流程：
>  首先，如果运行线程数小于核心线程数，会直接创建新线程来执行；
>  否则尝试进入任务队列，如果队列没满就排队等待；
>  如果队列已满，但线程数还没到最大，就会扩容线程来执行；
>  最后如果线程数已经到达最大并且队列也满了，就会触发拒绝策略。
>  底层是通过 `execute()` → `addWorker()` → `runWorker()` 循环执行任务来实现的。”

### 🔹 5. 追问点（面试官常问）

- 如果队列是无界队列（如 `LinkedBlockingQueue`），会发生什么？
- `submit()` 提交任务时和 `execute()` 有何不同？
