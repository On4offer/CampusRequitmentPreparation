# 🎯 面试题

**synchronized 和 ReentrantLock 的区别？底层实现？**

------

## ✅ 标准回答

1. **基本区别**

- `synchronized`：Java **关键字**，隐式加锁、自动释放，语法简单。
- `ReentrantLock`：**JUC 提供的显式锁**，功能更强大（可中断、可超时、公平锁、多条件队列）。

1. **主要区别点**

| 对比项              | synchronized                                                 | ReentrantLock                                                |
| ------------------- | ------------------------------------------------------------ | ------------------------------------------------------------ |
| **实现层面**        | JVM 内置关键字，基于对象头的 Mark Word 和 Monitor 实现（重量级锁可能依赖 OS Mutex） | JDK 提供的 API，基于 AQS（AbstractQueuedSynchronizer）实现   |
| **可重入性**        | 支持                                                         | 支持                                                         |
| **公平性**          | 不支持（默认非公平，JVM 控制）                               | 支持，构造时可选公平锁/非公平锁                              |
| **可中断性**        | 不支持，线程阻塞后不可中断                                   | 支持 `lockInterruptibly()`，可响应中断                       |
| **超时获取**        | 不支持，只能阻塞直到拿到锁                                   | 支持 `tryLock(long, TimeUnit)`，可限时尝试                   |
| **条件队列**        | 只有一个隐式条件队列，依赖 `wait/notify`/`notifyAll`         | 可创建多个 `Condition` 对象，支持精细化的线程调度            |
| **锁释放**          | 自动释放（同步代码块/方法执行完毕后由 JVM 自动释放）         | 需手动调用 `unlock()`，否则可能死锁                          |
| **可见性/内存语义** | 内置支持，释放锁时会自动清空工作内存并刷新主内存（happens-before 保障） | 需要依赖 `volatile` + AQS 保证内存可见性                     |
| **适用场景**        | 简单同步场景，代码少且容易维护                               | 高级并发控制，需灵活控制锁获取、超时、中断、多个条件队列时更适合 |

------

## ⚙️ 底层实现

1. **synchronized**

- **对象头 (Mark Word)** 存储锁信息。
- **锁优化**（JDK1.6 引入）：
  - **偏向锁**：CAS 写入线程 ID，无竞争时几乎无开销。
  - **轻量级锁**：自旋 + CAS 尝试获取锁。
  - **重量级锁**：膨胀成 OS 互斥量，阻塞/唤醒。
- **释放锁**：线程退出同步代码块时，JVM 自动释放。

1. **ReentrantLock**

- 基于 **AQS (AbstractQueuedSynchronizer)**。
- **state** 变量记录锁状态和重入次数。
- **CLH 队列**维护等待线程。
- **LockSupport.park/unpark** 实现线程的阻塞与唤醒。
- `unlock()` 时 CAS 修改 `state`，并唤醒队列中下一个节点。

------

## 📌 使用场景（结合项目）

- **synchronized**：简单同步场景（黑马点评缓存重建时，防止并发击穿）。
- **ReentrantLock**：需要**可中断、超时、公平性**（苍穹外卖骑手抢单等待、商户出餐等待场景，可以用不同 `Condition` 分组）。

------

## ⚠️ 易错点

- `ReentrantLock` 必须 `try-finally` 保证释放锁，否则会死锁。
- 滥用公平锁会降低吞吐量。
- synchronized 在高竞争场景下性能差于 ReentrantLock，但低竞争时有锁优化，性能不弱。

------

## 🔍 追问

1. **偏向锁和轻量级锁的区别？**
    👉 偏向锁几乎无竞争，直接 CAS 写线程 ID；轻量级锁有短暂竞争，用自旋获取。
2. **为什么 ReentrantLock 可以支持多个 Condition？**
    👉 AQS 维护多个等待队列，每个 Condition 都有独立队列，`await/signal` 精确唤醒。
3. **锁升级过程能降级吗？**
    👉 偏向锁在全局安全点（即所有线程都被暂停的时刻，以确保数据一致性）可以撤销，降级回无锁；重量级锁不会降级。
