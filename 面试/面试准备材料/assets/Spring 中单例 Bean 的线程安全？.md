# 📌 面试题

Spring 中单例 Bean 是如何保证线程安全的？一定安全吗？

------

### ✅ 一、概念回答

- **单例 Bean**：在 Spring 容器中，`singleton` 是默认作用域。整个容器中只有一个 Bean 实例，所有请求共享。
- **线程安全性**：Spring 容器本身并不会为单例 Bean 提供额外的并发控制，它只是保证了 **Bean 的实例化过程是线程安全的**，但并不能保证 Bean 内部的业务逻辑线程安全。

------

### ✅ 二、原理剖析

1. **实例化过程线程安全**
   - Spring 容器在创建单例 Bean 时，使用了 **同步锁（synchronized）** 保证实例化过程只有一个线程能进入。
   - 这避免了 **重复创建 Bean** 的并发问题。
2. **运行时不一定线程安全**
   - 单例 Bean 是 **多线程共享** 的，如果 Bean 内部包含 **可变的全局变量/状态**，多个线程同时修改会产生并发问题。
   - 如果 Bean 是 **无状态的（纯粹依赖方法参数，内部不保存共享变量）**，一般是线程安全的。

------

### ✅ 三、是否一定安全？

- **不一定安全** ❌
- **安全场景**：
  - Controller、Service、Dao 类，一般不保存可变成员变量，仅依赖方法参数处理业务。
  - 无状态类（比如工具类、配置类）。
- **不安全场景**：
  - Bean 内部定义了可变的共享成员变量（如 `List`、`Map`、计数器），且多个线程同时修改。
  - 在高并发下可能出现数据覆盖、脏读等问题。

👉 解决办法：

- 避免在单例 Bean 中使用可变的共享变量。
- 使用 `ThreadLocal` 隔离线程变量。
- 或者将有状态的 Bean 设置为 `prototype`，每次请求创建新对象。
- 必要时用并发工具（如 `ConcurrentHashMap`、锁机制）保证线程安全。

------

### ✅ 四、结合项目经验

- 在 **黑马点评** 项目中：
  - Controller、Service 层 Bean 默认是单例的，但因为方法只依赖请求参数，不存共享状态，所以是线程安全的。
  - 如果我们在 Service 中定义了一个成员变量缓存用户信息（比如 `private User currentUser;`），那在高并发下会出现用户信息串用的问题，这就是线程不安全的典型例子。
- 正确做法是用 `ThreadLocal` 保存用户信息，保证每个线程独立存储，不会互相干扰。

------

⚡ **标准回答模板（面试时可直接说）：**

> Spring 单例 Bean 的实例化过程是线程安全的，容器通过同步锁保证只会创建一个实例。但单例 Bean 本身并不一定线程安全，因为它会被多个线程共享。如果 Bean 内部是无状态的（只依赖方法参数），一般是安全的；但如果包含可变的成员变量，在高并发下可能出现线程安全问题。在实际项目中，比如黑马点评，我们用单例 Controller/Service 没问题，但用户信息这类需要隔离的变量就放在 ThreadLocal 里，避免多线程串用。
