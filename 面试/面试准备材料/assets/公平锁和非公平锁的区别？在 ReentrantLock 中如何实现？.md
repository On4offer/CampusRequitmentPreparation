# 🎯 面试题

**公平锁和非公平锁的区别？在 ReentrantLock 中如何实现？**

------

## ✅ 标准回答

### 1. 公平锁（Fair Lock）

- **定义**：按照线程请求锁的先后顺序来获取锁（FIFO）。
- **优点**：避免线程饥饿，保证相对公平。
- **缺点**：吞吐量较低，频繁切换上下文，性能差。

### 2. 非公平锁（Nonfair Lock）

- **定义**：新来的线程可以直接尝试抢占锁，不必排队。
- **优点**：吞吐量高，减少线程挂起和唤醒开销。
- **缺点**：可能导致部分线程长期得不到锁（饥饿）。

------

### 3. 在 ReentrantLock 中的实现

- **构造方法**：

  ```java
  // 默认是非公平锁
  ReentrantLock lock1 = new ReentrantLock();
  // 显式指定公平锁
  ReentrantLock lock2 = new ReentrantLock(true);
  ```

- **底层实现**：

  - `ReentrantLock` 基于 **AQS** 实现。

  - **非公平锁**：`tryAcquire()` 直接通过 CAS 尝试获取锁，如果失败再进入队列排队。

  - **公平锁**：在 CAS 前会先判断 **等待队列是否为空**，如果有线程排队则乖乖排队，不插队。

  - 公平锁内部调用：

    ```java
    if (hasQueuedPredecessors()) {
        return false; // 有线程在排队，不能插队
    }
    ```

------

## 📌 使用场景

- **公平锁**：适合对响应时间要求高、不能出现线程饥饿的场景，比如收费系统、排队系统。
- **非公平锁**（默认）：适合追求高吞吐量的场景，比如高并发下单、缓存更新。
- **黑马点评**：缓存重建锁用非公平锁，保证热点 key 能尽快刷新。
- **苍穹外卖**：骑手抢单可以用非公平锁，提高整体吞吐量。

------

## ⚠️ 易错点

- 以为公平锁一定更好 → 实际性能差很多。
- 在无锁竞争的情况下，公平锁和非公平锁性能几乎一样，但在高竞争下非公平锁更高效。
- 公平锁只能保证 **线程获取锁的公平性**，不能保证 **任务执行顺序的公平性**。

------

## 🔍 追问

1. **synchronized 是公平锁吗？**
    👉 不是，`synchronized` 是非公平锁，无法指定公平性。
2. **公平锁为什么性能差？**
    👉 因为每次获取锁前都要检查队列，保证 FIFO，增加了上下文切换。
3. **如何判断当前锁对象是公平还是非公平？**
    👉 `ReentrantLock` 提供了 `isFair()` 方法。
