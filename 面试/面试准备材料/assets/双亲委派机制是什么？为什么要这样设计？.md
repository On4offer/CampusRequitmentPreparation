## 📌 面试题：双亲委派机制是什么？为什么要这样设计？

### ⭐ 标准回答

**打破：重写 ClassLoder 的 loadClass 方法；使用线程上下文加载器，父类请求子类去加载实现逆向委托**

### 1. **定义**

双亲委派机制（Parent Delegation Model）是 **类加载器加载类** 时采用的一种层级委派模式：

- 每个类加载器在加载某个类时，**先把请求交给父加载器**。
- 父加载器找不到，才由当前加载器自己去加载。

**类加载器层级结构**（JDK8 为例）：

1. **Bootstrap ClassLoader**（启动类加载器，C++ 实现，加载 `rt.jar`、核心类库）。
2. **Extension ClassLoader**（扩展类加载器，加载 `ext` 目录下的类）。
3. **App ClassLoader**（应用类加载器，加载 classpath 下的类）。
4. **自定义 ClassLoader**（用户自定义）。

------

### 2. **工作流程**

- 加载类时，先向上请求父加载器。
- 父加载器无法加载（返回 `ClassNotFoundException`）时，才由子加载器尝试。

------

### 3. **为什么要这样设计？（好处）**

1. **避免类的重复加载**
    👉 保证同一个类在 JVM 中只会被加载一次，减少资源浪费。
2. **保证 Java 核心类的安全性**
    👉 即使用户自己写了一个 `java.lang.String` 类，也不会被随便加载，因为会优先由 Bootstrap ClassLoader 加载 JDK 自带的核心类。
3. **保证类的唯一性和一致性**
    👉 不同 ClassLoader 加载的类即使字节码相同，也会被认为是不同类。双亲委派能避免出现多个版本的核心类。

------

### 4. **例外情况**

虽然默认是双亲委派，但有些场景会打破：

- **SPI（Service Provider Interface）机制**
   👉 例如 JDBC 驱动，核心类需要调用第三方实现类，这里需要 **线程上下文类加载器** 来打破双亲委派。
- **OSGi、Tomcat、Spring Boot** 等框架
   👉 为了支持模块化/插件化，会设计自己的类加载机制。

------

### 📊 总结对比

| 机制     | 优点                                 | 问题/例外                                  |
| -------- | ------------------------------------ | ------------------------------------------ |
| 双亲委派 | 安全、避免重复加载、保证核心类一致性 | 难以支持插件化，需要打破（SPI、Tomcat 等） |

------

### 🔍 面试追问

1. 如果用户自己写了一个 `java.lang.Object` 类，能被加载吗？
    👉 不会，优先由 Bootstrap ClassLoader 加载 JDK 核心类。
2. 为什么 JDBC 要打破双亲委派？
    👉 因为核心类库需要调用第三方驱动（子加载器加载），必须反向委派。
3. 如何自定义类加载器？重写哪个方法？
    👉 继承 `ClassLoader`，重写 `findClass()`，一般不要重写 `loadClass()`。
