# 面试题：Git 中常见的分支管理策略有哪些？

### 1. 概念解释

分支管理策略（Branching Strategy）是团队在多人协作开发时，**如何划分分支、如何合并分支** 的一套规范。
 常见目标：保证主干稳定、支持多人并行开发、简化上线流程。

------

### 2. 常见策略

#### （1）Git Flow

- **分支类型**：
  - `master`：线上代码，始终保持可发布状态。
  - `develop`：日常开发主干。
  - `feature/*`：新功能分支，从 `develop` 拉取，完成后合并回 `develop`。
  - `release/*`：预发布分支，用于测试和修复。
  - `hotfix/*`：线上紧急修复分支，从 `master` 拉取。
- **优点**：流程清晰，适合大型团队，功能、修复、上线流程区分明确。
- **缺点**：分支繁多，流程复杂，小团队会觉得冗余。

------

#### （2）GitHub Flow

- **分支类型**：
  - `main`：唯一主分支，直接对应线上环境。
  - `feature/*`：新功能或修复分支，从 `main` 拉取，完成后通过 PR（Pull Request）合并。
- **优点**：轻量、简单，结合 CI/CD 可快速上线。
- **缺点**：分支少，对测试和预发布环境要求较高。

------

#### （3）GitLab Flow

- **特点**：结合 Git Flow 和 GitHub Flow，强调环境分支。
- **典型分支**：
  - `production`：线上分支。
  - `pre-production`：预发布分支。
  - `feature/*`：需求开发分支。
- **优点**：兼顾环境部署和功能开发。

------

### 3. 案例说明（结合项目）

在 **黑马点评项目**：

- 一般采用 **GitHub Flow**：
  - `main` 分支代表线上稳定版本。
  - 每个新功能（如“店铺缓存优化”）开一个 `feature/shop-cache` 分支，开发完后通过 PR 合并到 `main`，触发测试和部署。

在 **苍穹外卖项目**：

- 如果团队较大，可以采用 **Git Flow**：
  - 开发阶段在 `develop` 分支迭代。
  - 功能用 `feature` 分支独立开发。
  - 上线前从 `develop` 切出 `release`，测试无误后再合并到 `master`。

------

### 4. 使用场景

- **Git Flow**：适合大中型企业项目、对版本管理要求严格的团队。
- **GitHub Flow**：适合中小团队、敏捷开发、持续集成。
- **GitLab Flow**：适合需要多环境部署（开发/测试/生产）的团队。

------

### 5. 面试标准回答模板

> Git 中常见的分支管理策略有 **Git Flow、GitHub Flow 和 GitLab Flow**。
>  Git Flow 分支较多，适合大团队、版本管理严格的场景；GitHub Flow 简单，只有主分支和功能分支，结合 CI/CD 更灵活；GitLab Flow 结合环境部署，适合有多环境需求的团队。
>  在我的项目中（如黑马点评），我们采用 GitHub Flow，新功能以 feature 分支开发，完成后合并到 main 分支，触发自动化测试和上线，既保证了稳定性，也能快速迭代。
