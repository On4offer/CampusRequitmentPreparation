# 面试题：MySQL 的事务隔离级别有哪些？分别会产生哪些并发问题？

------

## 1. 概念

- **事务隔离级别**：数据库为了解决并发事务带来的数据不一致问题，定义了不同的隔离级别。
- SQL 标准定义了 **四种隔离级别**：
  1. **读未提交（Read Uncommitted）**
  2. **读已提交（Read Committed，RC）**
  3. **可重复读（Repeatable Read，RR）**
  4. **串行化（Serializable）**

------

## 2. 各隔离级别与并发问题

| 隔离级别     | 可能出现的问题                                              | 特点                 |
| ------------ | ----------------------------------------------------------- | -------------------- |
| **读未提交** | 脏读、不可重复读、幻读                                      | 几乎无隔离，性能最好 |
| **读已提交** | 不可重复读、幻读                                            | Oracle 默认级别      |
| **可重复读** | 幻读（MySQL InnoDB 已通过 **MVCC+Next-Key Lock** 解决幻读） | MySQL 默认级别       |
| **串行化**   | 无问题，但性能最差（事务串行执行）                          | 适合强一致场景       |

------

## 3. 并发问题解释

1. **脏读（Dirty Read）**
   - 读到了其他事务未提交的数据。
   - 例：A 修改余额但未提交，B 查询到了该数据。
2. **不可重复读（Non-Repeatable Read）**
   - 同一事务中多次读取同一行，结果不一致。
   - 例：A 两次读余额，中间 B 提交更新，结果不同。
3. **幻读（Phantom Read）**
   - 同一事务中多次查询某个范围，结果集不一致。
   - 例：A 查询 `WHERE age > 20`，第一次 5 行，第二次 6 行。

------

## 4. 案例

**库存扣减**：

- 使用 **读未提交** → 可能读到未提交的库存，超卖。
- 使用 **读已提交** → 避免脏读，但可能两次查询库存不一致。
- 使用 **可重复读** → 解决不可重复读，防止超卖。
- 使用 **串行化** → 串行执行，安全但性能低。

------

## 5. 使用场景

- **读未提交**：几乎不用。
- **读已提交（RC）**：大多数系统（Oracle 默认），读写并发高。
- **可重复读（RR）**：MySQL 默认，电商/支付系统常用。
- **串行化**：银行转账、财务记账等对一致性要求极高的场景。

------

## 6. 面试模板回答

> MySQL 定义了 4 种事务隔离级别：读未提交、读已提交、可重复读、串行化。
>
> - 读未提交可能产生脏读、不可重复读和幻读；
> - 读已提交避免了脏读，但仍可能产生不可重复读和幻读；
> - 可重复读避免了脏读和不可重复读，在 MySQL InnoDB 中通过 MVCC 和间隙锁解决了幻读；
> - 串行化完全避免并发问题，但性能最差。
>    MySQL 默认隔离级别是 **可重复读（RR）**。
