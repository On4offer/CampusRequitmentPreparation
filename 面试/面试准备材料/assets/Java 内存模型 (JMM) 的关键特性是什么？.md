# 🎯 面试题

**Java 内存模型 (JMM) 的关键特性是什么？**

------

## ✅ 标准回答

**JMM（Java Memory Model）** 规定了 **多线程如何通过内存交互数据**，主要包含以下关键特性：

1. **可见性**
   - 一个线程对共享变量的修改，能被其他线程及时看到。
   - `volatile`、`synchronized`、`final` 都能保证可见性。
2. **原子性**
   - 一个操作不可被分割，要么全部执行，要么全部不执行。
   - `synchronized`、`Lock` 可以保证临界区内的原子性。
   - `volatile` 不能保证复合操作的原子性。
3. **有序性**
   - 程序执行的先后顺序与代码顺序一致。
   - 实际运行中，编译器和 CPU 可能进行 **指令重排**。
   - `volatile`、`synchronized`、`Lock` 可以通过 **内存屏障** 保证一定的有序性。

------

## ⚙️ 底层机制

- **主内存 vs 工作内存**
  - 变量存放在 **主内存**（堆/方法区），每个线程有自己的 **工作内存（CPU 缓存/寄存器）**。
  - 线程对变量的操作必须在工作内存中进行，最终刷新回主内存。
- **happens-before 规则**（核心保证可见性/有序性的约束）
  1. **程序次序规则**：单线程内，按代码顺序执行。
  2. **监视器锁规则**：解锁 `unlock` 先于后续加锁 `lock`。
  3. **volatile 规则**：写 volatile 变量先于读操作。
  4. **传递性**：A happens-before B，B happens-before C → A happens-before C。
  5. **线程启动/终止规则**：`Thread.start()` 先于子线程所有操作，子线程结束先于 `join()` 返回。

------

## 📌 使用场景（结合项目）

- **黑马点评**：缓存重建时必须保证线程间可见性，可以用 `volatile` 标志控制线程停止。
- **苍穹外卖**：多线程生成订单 ID，需要保证原子性（用 `synchronized` 或分布式锁），同时避免指令重排（DCL 单例）。

------

## ⚠️ 易错点

- 以为 `volatile` 能保证原子性（其实只能保证可见性和有序性）。
- 忘记 `happens-before`，以为多线程看到的一定是最新值。
- 忽略 CPU 指令重排，导致 DCL 单例失效。

------

## 🔍 追问

1. **volatile 能不能保证原子性？**
    👉 不能，只能保证单次读写的原子性。
2. **为什么要有 happens-before 规则？**
    👉 因为指令可能乱序，CPU 有缓存，必须有一套约束来保证线程安全。
3. **final 关键字在 JMM 中的作用？**
    👉 保证初始化安全性，final 字段在对象构造完成后对其他线程可见。
