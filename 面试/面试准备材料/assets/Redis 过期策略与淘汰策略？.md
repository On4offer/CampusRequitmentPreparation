# 面试题

**Redis 的过期策略和淘汰策略有哪些？适合什么场景？**

------

## 1️⃣ 概念

- **过期策略**：决定 Redis 如何删除已过期的 Key。
- **淘汰策略**：当内存不足时，决定 Redis 删除哪些 Key，以腾出空间。

------

## 2️⃣ 过期策略（Expiration Policy）

Redis 采用 **惰性删除 + 定期删除** 两种策略结合：

1. **惰性删除**：访问 Key 时才检查是否过期，如果过期就删除。
   - 优点：节省 CPU，不访问就不检查。
   - 缺点：大量过期 Key 不访问时，会堆积占内存。
2. **定期删除**：Redis 定期抽样部分 Key 检查并删除过期 Key。
   - 优点：能清理掉长期未访问的过期 Key。
   - 缺点：周期性任务消耗 CPU，如果设置不合理可能影响性能。

⚡ Redis 默认采用 **惰性 + 定期** 结合，平衡性能和内存占用。

------

## 3️⃣ 淘汰策略（Eviction Policy）

当内存达到上限（`maxmemory`）时，Redis 会触发淘汰策略。

常见策略（Redis 4.0+）：

1. **noeviction（默认）**：不淘汰，直接报错（适合纯缓存不可丢失的场景）。
2. **volatile-xxx（只在设置了过期时间的键中淘汰）**
   - `volatile-lru`：淘汰最近最少使用的 Key。
   - `volatile-lfu`：淘汰访问次数最少的 Key。
   - `volatile-ttl`：淘汰即将过期的 Key。
   - `volatile-random`：随机淘汰。
3. **allkeys-xxx（所有 Key 都可能被淘汰）**
   - `allkeys-lru`：淘汰最近最少使用的 Key（最常用）。
   - `allkeys-lfu`：淘汰访问次数最少的 Key（推荐热点数据场景）。
   - `allkeys-random`：随机淘汰。

------

## 4️⃣ 使用场景

- **过期策略**：
  - 惰性删除：节省 CPU，适合数据量大、读写频繁场景。
  - 定期删除：适合需要清理长期不用数据的场景（如会话缓存）。
- **淘汰策略**：
  - `noeviction`：金融、支付等不能丢数据的场景。
  - `allkeys-lru`：通用缓存系统，热点数据留存，冷数据淘汰。
  - `allkeys-lfu`：高并发、热点集中的场景（如推荐、秒杀系统）。
  - `volatile-ttl`：消息、验证码等自然过期业务。
  - `random`：测试或低要求场景。

------

## 5️⃣ 项目实践（黑马点评 / 苍穹外卖）

- **黑马点评**：
  - 店铺缓存设置 TTL（防止击穿、脏数据），Redis 定期删除 + 惰性删除保证空间释放。
  - 高频店铺数据：采用 **allkeys-lru** 策略，保证热点数据常驻内存。
- **苍穹外卖**：
  - 验证码、Token：短 TTL，依赖定期删除和惰性删除自动清理。
  - 秒杀库存：不能丢失 → 只允许主动删除，不用淘汰策略。

------

## 6️⃣ 面试标准回答模板

> Redis 的 **过期策略**有惰性删除和定期删除，结合使用：访问时删除过期 Key，同时后台定期抽查清理。
>  当内存不足时，Redis 提供多种 **淘汰策略**：
>
> - `noeviction` 不淘汰；
> - `volatile-lru/lfu/ttl/random` 在设置了过期时间的 Key 中淘汰；
> - `allkeys-lru/lfu/random` 在所有 Key 中淘汰。
>    其中 **allkeys-lru 和 allkeys-lfu** 是最常用的策略，适合缓存系统保证热点数据。
>    在项目中，我会结合业务场景选择，比如验证码用 TTL 自动清理，热点缓存用 LRU 保证高命中率。
