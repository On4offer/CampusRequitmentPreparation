# 🎯 面试题

**MyBatis 在高并发下如何保证性能和安全？**

------

## 一、概念

- **高并发问题**：大量请求同时访问数据库，可能导致 **连接耗尽、性能下降、数据不一致、脏读/幻读、SQL 注入** 等问题。
- **MyBatis 保证性能和安全的目标**：
  1. **性能优化**：减少数据库压力，提高查询效率。
  2. **安全保障**：防止数据错乱和安全漏洞。

------

## 二、原理 & 关键机制

1. **连接池管理**
   - MyBatis 内置连接池（一般推荐配合 HikariCP、Druid 使用）。
   - 保证在高并发下复用连接，避免频繁创建销毁连接。
2. **缓存机制**
   - 一级缓存：`SqlSession` 内部 Map，减少同一会话的重复查询。
   - 二级缓存：跨 `SqlSession` 共享，可配置 Redis 等分布式缓存，减轻数据库压力。
3. **Executor 执行器选择**
   - `SimpleExecutor`：每次新建 Statement。
   - `ReuseExecutor`：复用 Statement，减少开销。
   - `BatchExecutor`：批量执行，提高写操作性能。
4. **SQL 预编译（防 SQL 注入）**
   - `#{}` 使用 PreparedStatement，占位符方式，避免拼接 SQL 注入风险。
5. **分页插件**
   - MyBatis-Plus 分页拦截器在执行前改写 SQL，加上 `limit`，避免全表扫描。
6. **事务 & 并发控制**
   - 通过 Spring 事务（AOP + ThreadLocal）保证一致性。
   - 支持悲观锁（for update）、乐观锁（@Version）。

------

## 三、案例示例

```java
// 乐观锁控制并发更新
@TableName("product")
public class Product {
    private Long id;
    private String name;
    private Integer stock;

    @Version // 乐观锁注解
    private Integer version;
}

// 更新库存时，MyBatis-Plus 自动在 WHERE 条件加 version
UPDATE product SET stock = stock - 1, version = version + 1
WHERE id = ? AND version = ?
```

------

## 四、使用场景

- **苍穹外卖项目**：
  - 用户下单 → 多人同时扣减库存，需用 **乐观锁/悲观锁** 保证数据正确。
  - 热门菜品查询 → 使用 **Redis + 二级缓存** 提升性能。
- **黑马点评项目**：
  - 商户详情、点评查询 → 使用缓存 + 分页优化。
  - 秒杀优惠券 → 使用 **悲观锁 / 分布式锁（Redis/Redisson）** 避免超卖。

------

## 五、面试模板（标准作答）

> 在高并发场景下，MyBatis 主要通过 **连接池、缓存机制、SQL 预编译、分页插件、事务管理** 来保证性能和安全。
>
> **性能方面**：利用连接池复用数据库连接，减少开销；通过一级缓存和二级缓存减少数据库访问；执行器选择和批量操作提升效率；分页插件避免全表扫描。
>
> **安全方面**：`#{}` 使用预编译防止 SQL 注入；Spring 事务和 MyBatis 的 Executor 保证数据一致性；在并发写入时，可用乐观锁/悲观锁避免超卖问题。
>
> 在真实项目中（如外卖下单、秒杀场景），通常会结合 **Redis 缓存、分布式锁、消息队列**，进一步提升并发处理能力。

------

## 六、扩展追问

1. MyBatis 的一级缓存为什么在高并发场景下效果有限？
    👉 因为作用域是 SqlSession 级别，不同请求的 SqlSession 无法共享。
2. 如果 MyBatis 的二级缓存和 Redis 同时使用，如何保证一致性？
    👉 需要在更新数据库时同步删除缓存（Cache Aside 策略）。
3. 在秒杀/抢购场景下，仅靠 MyBatis 能解决超卖吗？
    👉 不行，需要结合 Redis 分布式锁或消息队列进行削峰。
4. 乐观锁在高并发下会遇到什么问题？
    👉 **冲突率过高**导致频繁重试，此时可能更适合悲观锁或分布式锁。
