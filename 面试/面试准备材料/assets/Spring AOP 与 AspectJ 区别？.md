# 🎯 面试题：Spring AOP 和 AspectJ 的区别是什么？Spring 为什么没有采用完全编译期织入？

## 1. 概念解释

- **Spring AOP**：
  - 基于 **代理模式**（JDK 动态代理 / CGLIB），在 **运行时** 为目标对象创建代理。
  - 只支持 **方法级别** 的切点。
- **AspectJ**：
  - 一个 **独立的 AOP 框架**，功能更强大，支持 **编译期 / 类加载期 / 运行期织入**。
  - 提供 `ajc` 编译器，能在 **字节码层面**直接修改目标类，支持构造器、字段访问等切点。

------

## 2. 区别对比表格

| 对比点       | **Spring AOP**                     | **AspectJ**                      |
| ------------ | ---------------------------------- | -------------------------------- |
| **织入时机** | 运行时（基于代理）                 | 编译期、类加载期（LTW）、运行期  |
| **实现方式** | JDK 动态代理 / CGLIB               | 字节码修改（ajc、ASM）           |
| **切点粒度** | 仅支持方法级别                     | 支持方法、构造器、字段、静态块等 |
| **性能**     | 运行时代理，有一定性能损耗         | 编译期织入，性能更高             |
| **复杂度**   | 使用简单，零侵入，整合 Spring 容易 | 功能强大，但需额外编译器、插件   |
| **应用场景** | 日志、事务、安全等常见场景         | 框架开发、性能要求极高的系统     |

------

## 3. 为什么 Spring 没有采用完全编译期织入？

1. **易用性优先**
   - Spring 目标是让企业开发者快速上手，而不是学习复杂的 AspectJ 编译器（ajc）、IDE 插件。
   - 运行时代理对开发者几乎透明，只需要加 `@EnableAspectJAutoProxy` 和 `@Aspect` 就能用。
2. **兼容性和灵活性**
   - 编译期织入需要改造编译流程，引入 `ajc` 或 LTW（Load Time Weaving）代理，增加学习成本。
   - 运行时代理更灵活，Spring 只需基于 Bean 生命周期生成代理对象，不依赖特定编译工具。
3. **性能与需求权衡**
   - 绝大多数企业场景只需要 **方法级别增强**（事务、日志、权限校验）。
   - Spring AOP 已经能满足需求，不需要 AspectJ 那样复杂的切点。
   - 真正需要字节码级别增强的，通常是中间件或框架开发，不是 Spring 的目标用户群。

------

## 4. 面试标准回答（简洁版）

> Spring AOP 和 AspectJ 的区别在于：Spring AOP 基于动态代理，只能在运行时对方法进行增强，简单易用但功能有限；AspectJ 基于字节码增强，支持编译期/类加载期织入，能拦截方法、构造器、字段访问，功能更强大但复杂度高。
>  Spring 没有采用编译期织入的原因是为了保持使用的简洁性和灵活性，避免开发者依赖额外的 ajc 编译器和插件，同时大多数业务场景只需要方法级别增强，Spring AOP 已经足够。
