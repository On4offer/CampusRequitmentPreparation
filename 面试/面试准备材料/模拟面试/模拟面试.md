# 模拟大厂 Java 开发实习面试（2025 年场景，时长≥1 小时）

## 面试规则

1. 评分标准：总分 100 分，分项目深挖（40 分）、场景设计（30 分）、技术原理（20 分）、代码 / 算法（10 分）四个维度，60 分合格，85 分及以上优秀。
2. 答题要求：需结合简历项目细节与技术原理作答，禁止泛泛而谈，可主动提出优化方案，面试官会随时追问与 “拷打”。
3. 流程安排：先项目深挖，再场景题，接着技术原理，最后代码 / 算法，全程计时，超时会影响评分。

## 一、实习项目深挖（医疗美容行业管理系统，40 分）

### 核心功能追问（20 分）

1. 你做的 Excel 商品批量导入用了 Apache POI，.xls 和.xlsx 解析的区别是什么？模糊匹配和精确匹配的具体实现逻辑是怎样的？如果导入 10 万行数据出现 OOM，你怎么优化？（4 分）
2. 密码过期强制修改功能，3 个月过期校验是在登录拦截器里怎么实现的？如何避免时钟不一致导致的校验错误？密码修改接口如何防止 CSRF 攻击？（4 分）
3. 审计日志的敏感信息脱敏用了什么策略？IP 地址识别是调用第三方接口还是本地库？异步存储日志用了线程池还是 Spring 的 @Async？线程池的核心参数是怎么设置的？（4 分）
4. 入库单标识功能的数据库字段设计包含哪些字段？字段类型和索引怎么选？前后端接口的幂等性如何保证？UI 交互中筛选功能的性能优化做了哪些？（4 分）
5. 修复移动端业绩报表数据筛选问题时，定位问题的流程是什么？客户推荐参数校验补充了哪些规则？入库模块事务管理用了 Spring 的哪种事务传播机制？（4 分）

### 项目架构与问题排查（20 分）

1. 系统适配原有[asp.net](https://asp.net)系统，你在开发时遇到了哪些兼容性问题？比如接口调用、数据格式、权限认证方面，具体怎么解决的？（5 分）
2. 入库模块的事务管理和数据校验完善后，如何验证事务的一致性？有没有做过压测？压测数据如何？（5 分）
3. 异步存储日志时，若异步线程抛出异常导致日志丢失，你怎么处理？如何保证日志的最终一致性？（5 分）
4. 这个系统支撑总部与门店双业务场景，门店网络不稳定时，如何保证数据同步的可靠性？有没有考虑过离线操作与数据补传方案？（5 分）

## 一、实习项目深度复盘：医疗美容行业管理系统（40 分）

**面试官视角**：你的实习经历虽然只有两个月，但作为 211 硕士，我对你的代码质量和解决问题的思路要求会比本科生更高。

### 1. 关于 Excel 批量导入与性能优化（10 分）

- **Q1**: 你提到使用了 Apache POI。请描述一下 `UserModel` 和 `EventModel` 的区别。你在处理 10 万行数据时，具体使用的是哪种模式？如果不使用 EasyExcel 等封装框架，如何通过原生 POI 避免 OOM？
- **Q2**: 批量导入涉及到 “模糊匹配”，这个逻辑是在内存中做的，还是查库做的？如果是查库，10 万条数据每条都查一次数据库，会有什么问题？如何优化？（考察：SQL 优化、批量查询、内存缓存）。
- **Q3**: 导入过程中如果第 5000 行数据格式错误，你的系统是如何处理的？是回滚前 4999 条，还是跳过错误继续执行？你是如何记录错误日志以便用户排查的？

### 2. 关于审计日志与异步处理（10 分）

- **Q1**: 你提到使用 “异步方式存储日志”。如果主线程处理完了，用户看到了成功的反馈，但异步线程保存日志失败了（比如磁盘满了或数据库连接池耗尽），怎么保证日志不丢失？
- **Q2**: 你是用 Spring 的 `@Async` 还是自己定义的线程池？如果用 `@Async`，默认的线程池配置（SimpleAsyncTaskExecutor）有什么隐患？在你的项目中，线程池的核心线程数、最大线程数、队列长度是如何配置的？为什么？
- **Q3**: 日志中包含 “IP 地址识别”，你是怎么获取客户端真实 IP 的？如果系统前面有 Nginx 反向代理，X-Forwarded-For 头可能会被伪造，你怎么处理？

### 3. 关于密码安全与 Web 安全（10 分）

- **Q1**: 密码过期校验在拦截器中实现。拦截器（Interceptor）和过滤器（Filter）的执行顺序是怎样的？如果你的系统集成了 Spring Security，你会把这个逻辑放在哪里？
- **Q2**: 密码存储在数据库中是明文吗？如果不是，用的什么加密算法（MD5, SHA, BCrypt）？BCrypt 的工作原理是什么？为什么它比 MD5 更安全？
- **Q3**: 你提到修复了 “移动端业绩报表数据筛选问题”。这个问题的根源是什么？是 SQL 写错了，还是前端传参格式不对？你是如何复现并定位这个 Bug 的？（考察：Debug 能力）。

### 4. 关于新旧系统兼容（10 分）

- **Q1**: 系统需要适配原有[ASP.NET](https://ASP.NET)系统。如果 Java 系统需要调用[ASP.NET](https://ASP.NET)的老接口，你是用什么方式调用的（HttpClient, RestTemplate, OpenFeign）？
- **Q2**: 涉及到双系统数据同步，如果 Java 这边更新了数据，[ASP.NET](https://ASP.NET)那边读取不到（延迟），你考虑过分布式事务吗？或者你们是如何保证最终一致性的？

## 二、个人项目深挖：小众点评（30 分）

**面试官视角**：这个项目是典型的 “高并发练习册”，我要看的不是你 “用了什么”，而是你 “为什么这么用” 以及 “有没有坑”。

### 1. Redis 与缓存架构（15 分）

- **Q1**: 你用 “逻辑过期” 解决缓存击穿。请详细描述一下逻辑过期的读写流程。相比于 “互斥锁” 方案，逻辑过期的优缺点是什么？在你的 Feed 流场景中，为什么选择逻辑过期？
- **Q2**: 基于 Redis Stream 的消息队列。Stream 相比于 List（LPUSH/RPOP）和 Pub/Sub 有什么优势？你在代码中是如何处理 Stream 的消费者组（Consumer Group）的？如果消费者宕机了，消息会发生什么？
- **Q3**: 你用 Redisson 实现分布式锁。如果业务逻辑执行时间超过了锁的 TTL（看门狗时间），会发生什么？Redisson 的 “看门狗” 机制是如何自动续期的？
- **Q4**: Feed 流推送，“将内容 ID 推送给所有粉丝”。如果一个大 V 有 1000 万粉丝，这行代码执行会发生什么？（考察：大 Key 问题、阻塞主线程）。你有什么优化方案？

### 2. 数据库与并发控制（15 分）

- **Q1**: 解决 “超卖” 问题。除了 Redis 分布式锁，数据库层面你做了什么防护？比如利用 MySQL 的乐观锁（CAS），请写出那条核心的 UPDATE SQL 语句。
- **Q2**: 订单落库是异步的。如果 Redis 消息队列积压了 10 万条消息，消费者处理不过来，导致用户下单后很久看不到订单，你怎么解决？
- **Q3**: 点赞排行用 ZSet。如果数据量有上亿条，ZREVRANGE 命令的性能如何？如何优化热榜的查询速度？

### 面试建议（2025 大厂趋势）：

1. **不要背书**：比如问 Redis 缓存穿透，不要只说 “布隆过滤器”，要说 “在我的项目中，因为数据量不大，我用了缓存空对象；但如果是亿级数据，我会考虑布隆过滤器，因为...”。
2. **关注稳定性**：现在的面试非常看重 “异常处理”、“重试机制”、“限流熔断”。
3. **数据驱动**：回答性能优化问题时，尽量带上数据（比如：优化前响应时间 500ms，优化后 50ms）。
