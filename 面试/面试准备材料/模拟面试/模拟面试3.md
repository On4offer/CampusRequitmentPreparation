**面试官**：嗯，直接查库重建简单直接，适合数据不复杂的场景。那重建完缓存后，这个分布式锁是手动释放还是框架自动释放的？  
**回答**：这个我感觉就是框架它自动释放了，我好像没有手动的显示去释放这个锁。

**面试官**：Redisson 的锁在 try-with-resources 或者锁对象被回收时会自动释放，这样也能避免遗漏释放的问题。  
**回答**：好的，还有下一个问题吗？

**面试官**：那聊聊缓存穿透的问题吧，比如有人恶意请求不存在的商品 ID，缓存和数据库都查不到，怎么处理？  
**回答**：这种情况下的话可以考虑使用缓存一个空值，就是查询缓存不存在，同时查询数据库也不存在的时候，就返回一个空值占位符，然后并对它设置一个比较短的一个 TTL 过期时间，这样的话后续的线程或者下一次再查询的话，就能查询到这个空值返回，避免这个恶意请求大量或者重复的打到数据库。第二种方案的话就是使用布隆过滤器来进行判断这个缓存是否存在。

**面试官**：这两种方案都常用，缓存空值实现简单，布隆过滤器能从源头拦截。你在 "小众点评" 里选的哪种？  
**回答**：我这选择的是缓存空值，因为这个毕竟它的业务体量不大，然后也不涉及到对安全性，像像支付或者订单类的特别安全性、实时性要求比较高的场景。

**面试官**：对于小众点评这种初期业务，缓存空值确实更轻量，不用维护布隆过滤器的复杂度。那这个空值的 TTL 你设了多久？  
**回答**：大概设置的是 2~5 秒，嗯，时间太短的话就会起球过多，时间太长了的话就会影响那个正确的缓存，正确的数据，嗯。

**面试官**：2 到 5 秒的 TTL 挺合适，既能挡掉短时间的恶意请求，又不会影响真实数据的更新。那如果后续这个不存在的商品 ID 突然被创建了，缓存里的空值会不会导致新数据查不到？  
**回答**：会，会有这么几秒钟的时间，那个短 TTL 的一个延迟。但是因为商品的 ID 创建，商品更新的话，毕竟频率比较低，所以的话我觉得这个问题是可以接受的。

**面试官**：嗯，这种低频更新的场景，几秒延迟确实在可接受范围内。那除了缓存穿透，你有没有考虑过缓存雪崩的情况？  
**回答**：缓存雪崩的情况的话，就是说它缓存大量缓存同时失效，同时过期，这个东西的话，就是说我们在缓存预热的时候，或者怎么说的时候，就不要去给那个设置一个统一的逻辑过期时间，就因为它是同时加载进内存，加载进缓存嘛？我们要给那个缓存过期时间加一个随机的一个抖动，加一个随机时间。

**面试官**：给过期时间加随机抖动是个简单有效的办法，能避免大量缓存集中失效。你在项目里给这个随机抖动设的范围大概是多少？  
**回答**：是 0~30 分钟的一个随机数，我记得好像是。

**面试官**：0 到 30 分钟的随机抖动，能有效打散缓存过期时间，避免集中失效。对于点评类系统的商品详情缓存来说，这个范围也比较合理，不会让缓存生命周期差异过大。那在缓存预热的时候，你是怎么把商品数据加载到 Redis 的？  
**回答**：我靠，怎么把缓存预热的时候，怎么把缓存数据加载到 Redis 这一块？我一是使用的那个脚本把它加载进去的。

**面试官**：用脚本批量加载挺方便的，适合初始化场景。那脚本里是直接查数据库全量导出，还是按一定规则筛选热门商品加载的？  
**回答**：这个的话就是筛选的热门商品，或者别的业务规则，或者别的判断方法，不是全量加入。

**面试官**：只加载热门商品能节省 Redis 空间，也更贴合实际访问需求。那这个热门商品的筛选规则，是按点击量还是评分来定的？  
**回答**：日本鬼子的话，因为我们这种预热的话是进行活动的，所以的话，点击量的话是主要的一个判断方式，然后评分高的话我们也会把它加载进去，保证质量。

**面试官**：结合点击量和评分来选热门商品，既能保证流量又兼顾质量，挺全面的。那活动期间如果有新的商品突然变热，脚本预热可能覆盖不到，这种情况你有考虑过动态补充缓存吗？  
**回答**：哦，这一块的话就确实没有考虑到就商品突然变热，如果要完成这种逻辑的话就需要加一个判断，就是说判断有那个商品变成热门商品的话，加一个判断逻辑，然后就把它加进那个预热，就对它进行预热，加进 Redis 缓存，需要补充一个这么多的逻辑，这是我的一个想法。

**面试官**：这个思路可行，比如可以监控商品的实时访问量，当短时间内点击量超过阈值时，自动触发缓存加载。  
**回答**：还，还有需要提问的吗？

**面试官**：那如果要实现这个动态补充缓存的逻辑，你觉得用定时任务去扫描访问日志，还是在接口层实时统计点击量更合适？  
**回答**：一个是扫描日志，一个是在接口层统计点击量。嗯，日志的话，如果现，现有的框架已经有了那个成熟的日志系统的话，并且日志的那个时效性比较好的话，那么我觉得可以去采用后台的定时任务去扫描日志，和主流程主线上进行分开。嗯，对于接口层实时统计点击量的话，这个东西的话我感觉可能会切入代码，因为代码影响那个主要的流程。这个方案的话其实也可以接受，因为它的业务量不大，但是还是得去仔细的去考虑一下。
