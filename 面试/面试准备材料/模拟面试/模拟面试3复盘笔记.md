## 二、 实习项目：医疗美容行业管理系统

### 1. Excel 商品批量导入

- **表现分析**：清晰地答出了 `WorkbookFactory.create` 自动识别格式、流式读取防 OOM（EasyExcel）等关键技术点。对“两步匹配策略”的解释方向正确（先找出候选，再内存比对）。
- **不足之处**：在回答“比对了哪些字段”时带上了价格（文档中核心匹配是基于**名称和规格**）；在说应对5万行数据时，提到了批量查询，但这属于“优化建议”，目前系统中的实际做法是**逐行解析并收集错误列表统一返回**。
- **🔥 标准话术**：
  > **模糊+精确策略**：第一步，利用 `LIKE` 关键字按**商品名称**去数据库做模糊查询，把候选集加载到内存；第二步，在内存中对候选数据去除空格和特殊字符后，严格比对**商品名称和规格**，完全一致才算匹配成功，否则就收集相似商品信息作为前端的友好提示。
  > **大文件与解析优化**：代码中利用了 `WorkbookFactory.create()`，它会自动读取文件头魔数来区分 `.xls` 和 `.xlsx`，无需硬编码区分。如果遇到几万行的大文件，传统 POI 一次性加载会引发 OOM，我会采用两种优化方案：业务上采用分批导入；技术上替换为基于事件驱动模型解析的 EasyExcel 框架。

### 2. 密码过期强制修改

- **表现分析**：知道 `PasswordChangeTime` 字段存在数据库里。
- **致命缺陷**：回答“在登录拦截器里面判断，如果过期直接跳转到密码修改页面”。**这在前后端分离架构下是错误的！** 后端不能直接控制浏览器跳转页面。
- **🔥 标准话术**：
  > **过期校验与拦截逻辑**：3个月过期的字段 `PasswordChangeTime` 存在 MySQL 的员工表里。校验并非在全局拦截器里做，而是在**登录接口的业务逻辑**中。为了安全，必须是**先校验用户密码正确**后，才去检查时间。如果发现时间为空或超过3个月，后端返回一个约定的错误码（如 `13` PasswordExpired）。前端捕获到这个特定错误码后，再由 Vue Router 跳转到密码强制修改页。
  > **安全校验**：这是一个匿名接口（因为用户还没登录拿不到Token）。所以安全上除了校验账号、门店公司ID、比对原密码的MD5外，还会强制校验新密码不能与原密码相同。

### 3. 审计日志

- **表现分析**：答出了对于不同字段的脱敏规则（替换星号、截断删除）。但表达不够结构化，显得有些语无伦次。
- **🔥 标准话术**：
  > **脱敏字段与规则**：为了防止日志泄露安全信息，在拦截器获取请求参数（如 requestBody 或 ParameterMap）时做了脱敏。主要针对涉及账号安全的字段，如 `password`、`pwd`、`token` 等。对于普通表单或JSON中的密码，将其值替换为 `*`** 星号；对于 HTTP Header 里的 Token，则直接不予记录或截断删除。
  > **日志落地机制**：整个采集在 Spring 拦截器的 `afterCompletion` 中进行，提取完信息后，利用 `CompletableFuture.runAsync` 交给异步线程池处理，绝不阻塞主业务流程。

## 三、 个人项目：小众点评

### 1. 缓存与数据库一致性 (Cache-Aside)

- **表现分析**：面试官问“如果删除缓存后没来得及更新数据库...”，顺着面试官的话说“确实会查到旧数据，只能用延迟双删”。
- **避坑指南**：面试官在**挖坑**！标准的 Cache-Aside 策略是**先更新数据库，再删除缓存**。如果是先删缓存再更新数据库，确实会产生脏数据，这本身就是一个反模式。
- **🔥 标准话术**：
  > “面试官，我们在项目里使用的标准 Cache-Aside 策略其实是**先更新数据库，再删除缓存**。这样读请求回源时拿到的永远是 DB 里最新的数据。您刚才提到的‘先删缓存后更新数据库’确实会导致高并发下的脏数据问题。如果业务强制要求必须先删缓存，我才会采用**延迟双删**策略（删缓存 -> 更新DB -> 睡眠500ms -> 再删缓存）。但在我这个项目里，我们严格采用先写 DB 后删缓存，结合最终一致性已经能满足绝大多数场景了。”

### 2. 逻辑过期与分布式锁

- **表现分析**：对逻辑过期和分布式锁（Redisson 看门狗机制）的掌握非常透彻。回答得很完美。
- **进阶建议**：可以顺带提一句代码封装设计，体现工程能力。
  > *“我在项目中用泛型和函数式接口（`Function<ID, R> dbFallback`）封装了一个 `CacheClient`，把缓存击穿（逻辑过期）和穿透（缓存空值）都做成了模板方法，极大提高了代码复用性。”*

### 3. 缓存穿透与雪崩

- **表现分析**：准确说出空值 TTL 设短（2-5秒），雪崩 TTL 加随机数（0-30分钟）。数字说得非常落地，有实际开发经验的即视感。

### 4. 场景题：动态补充缓存

- **表现分析**：面对没有准备过的场景题，给出了两种 Trade-off 方案（扫描日志 vs 接口层统计），展现了很好的技术思维。
- **🔥 进阶优化思路（向高级工程师靠拢）**：
  > “如果要做动态热点探测，我会倾向于**异步解耦**的方案。如果在接口层实时统计（比如用 Redis 做访问计数器），对主流程性能有一定侵入；我更倾向于使用异步方案：在应用层把访问事件通过 MQ 或直接记录到本地日志中，后台由一个流计算任务（比如 Flink 或简单的定时任务）去滑动窗口分析（比如最近1分钟访问超1000次）。一旦发现热点，立刻触发预热脚本主动把数据推入 Redis。这样既保证了主业务极高的性能，又能实现自动的动态缓存补充。”

## 四、 面试演练核心建议

1. **小心面试官的“引导性挖坑”**
  面试官常常会故意说错一个前置条件（比如“你先删了缓存…”、“你主从同步断了锁丢了…”），千万不要盲目顺着答。要先**纠正正确的前提**（“我们的做法其实是先写DB”），然后再回答“即便出现了极端情况，我的兜底方案是什么”。
2. **多说技术决策（Trade-off）**
  在被问到为什么选某种方案时，多用对比。比如：“当时针对缓存击穿我考虑过互斥锁和逻辑过期，互斥锁能保证强一致但高并发下线程阻塞严重；因为点评属于读多写少的 C 端展示，可用性大于强一致性，所以我最终选择了逻辑过期。” 这种回答非常加分。
3. **前后端分离思维要牢记**
  校招后端开发最容易犯的错就是把后端和前端揉在一起说。凡是涉及到“跳转页面、弹出提示”，一定要说是**后端返回约定的状态码（Status Code/Result Enum），由前端（Vue）去控制路由渲染**。
