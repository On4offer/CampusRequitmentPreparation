### 2. 线程池核心参数、拒绝策略

#### 核心参数（5 个）：

```
ThreadPoolExecutor(int corePoolSize,    // 核心线程数（常驻线程，不会回收）
                   int maximumPoolSize, // 最大线程数（核心+临时线程上限）
                   long keepAliveTime,  // 临时线程空闲存活时间
                   TimeUnit unit,       // 存活时间单位
                   BlockingQueue<Runnable> workQueue, // 任务阻塞队列（核心线程满后存任务）
                   ThreadFactory threadFactory, // 线程创建工厂（自定义线程名、优先级等）
                   RejectedExecutionHandler handler) // 拒绝策略（队列+最大线程都满时）
```

#### 拒绝策略（4 种默认实现）：

1. AbortPolicy（默认）：直接抛出RejectedExecutionException，中断任务提交；

1. CallerRunsPolicy：由提交任务的主线程自己执行（减缓提交速度，避免任务丢失）；

1. DiscardPolicy：静默丢弃无法处理的任务，无任何提示；

1. DiscardOldestPolicy：丢弃队列头部（最旧）的任务，尝试提交当前任务。

### 3. 怎么设计一个线程池？

核心思路：**根据业务场景（CPU 密集 / IO 密集）确定参数，兼顾性能、稳定性和可监控性**

1. **确定核心参数**：

- CPU 密集型（如计算）：核心线程数 = CPU 核心数 + 1（减少线程切换开销）；

- IO 密集型（如 DB 查询、网络请求）：核心线程数 = CPU 核心数 × 2（利用 IO 等待时间并行处理）；

- 队列选择：无界队列（LinkedBlockingQueue）适合任务量稳定，有界队列（ArrayBlockingQueue）避免内存溢出；

- 拒绝策略：核心业务用AbortPolicy（快速失败），非核心用DiscardOldestPolicy或自定义（如持久化任务到 MQ）。

1. **增强功能**：

- 自定义ThreadFactory：设置线程名（便于排查日志）、设置为守护线程 / 非守护线程；

- 增加监控：统计线程池活跃数、队列任务数、完成任务数（结合 Spring Boot Actuator）；

- 优雅关闭：重写shutdown()，确保任务执行完再关闭（配合awaitTermination）。

### 4. CMS、G1（JVM 垃圾收集器）

#### CMS（Concurrent Mark Sweep，并发标记清除）

- 目标：**低延迟**（适合响应时间敏感的业务，如 Web 应用）；

- 核心机制：并发执行（标记、清除阶段与用户线程并行），分 4 步：

1. 初始标记（STW，暂停用户线程，标记 GC Roots 直接关联的对象，快）；

1. 并发标记（与用户线程并行，遍历可达对象）；

1. 重新标记（STW，修正并发标记期间用户线程修改的对象，快）；

1. 并发清除（与用户线程并行，清除不可达对象）；

- 缺点：产生内存碎片、占用 CPU 资源（并发阶段与用户线程抢资源）、无法处理浮动垃圾（并发清除时新产生的垃圾）。

#### G1（Garbage First，垃圾优先）

- 目标：**平衡延迟与吞吐量**（替代 CMS，适合大堆场景，如 8G + 堆）；

- 核心机制：

1. 分区化：将堆划分为多个大小相等的 Region（年轻代、老年代 Region 混合分布）；

1. 优先级：优先回收垃圾最多的 Region（“垃圾优先”）；

1. 复制算法：回收时将存活对象复制到空 Region，无内存碎片；

1. 停顿预测模型：根据用户设置的最大停顿时间（-XX:MaxGCPauseMillis）动态调整回收 Region 数量；

- 优点：无内存碎片、延迟可控、吞吐量均衡；缺点：小堆场景下性能不如 CMS。

### 5. CMS 配合的新生代算法是什么？

CMS 是**老年代收集器**，必须搭配**新生代收集器**使用，默认搭配：ParNew（并行新生代收集器）。

- 原因：ParNew 是 Serial 收集器的并行版本，支持多线程回收，与 CMS 的并发特性契合，且能与 CMS 共享 GC Roots 扫描结果，减少开销；

- 其他搭配：理论上可搭配 Serial，但 Serial 是单线程，性能差，实际很少用。

### 6. MySQL 死锁的情况

死锁是**两个或多个事务互相持有对方需要的锁，且都不释放**，导致循环等待。常见场景：

1. 事务顺序不一致：

   - 事务 A：更新 A 表 → 更新 B 表；

   - 事务 B：更新 B 表 → 更新 A 表；

2. 主键 / 唯一索引并发插入：两个事务同时插入相同唯一键值（如并发注册同一手机号），互相等待对方释放意向锁；

3. 范围锁冲突：使用SELECT ... FOR UPDATE（悲观锁）查询范围数据，两个事务锁定的范围重叠（如事务 A 锁 id 1-5，事务 B 锁 id 3-7）；

4. 外键约束：事务 A 删除父表数据，事务 B 同时删除子表数据，外键检查导致锁等待循环。

- 解决：统一事务操作顺序、避免长事务、用乐观锁（version）替代悲观锁、开启死锁检测（innodb_deadlock_detect=ON）。

### 7. ThreadLocal

#### 核心定义：

线程本地存储，为**每个线程提供独立的变量副本**，线程间互不干扰，避免线程安全问题。

#### 底层原理：

- 每个 Thread 对象持有一个ThreadLocalMap（哈希表），key 是 ThreadLocal 实例（弱引用），value 是线程本地变量；

- ThreadLocal 本身不存数据，仅作为ThreadLocalMap的 key，间接访问线程的本地变量；

- 弱引用原因：避免 ThreadLocal 实例未被回收时，导致ThreadLocalMap内存泄漏（若 ThreadLocal 是强引用，即使外部无引用，ThreadLocalMap仍持有，导致内存无法释放）。

#### 注意事项：

- 内存泄漏风险：线程池场景下，线程复用导致ThreadLocalMap中的 value 未被清除（线程一直存活），需手动调用remove()清理；

- 适用场景：存储线程上下文（如用户登录信息、数据库连接、事务上下文）。

### 8. MySQL 索引

#### 索引本质：

排好序的数据结构（如 B + 树），用于快速查找数据，减少磁盘 IO。

#### 常见索引类型：

1. 按数据结构：

- B + 树索引（默认）：聚簇索引（主键索引，叶子节点存整行数据）、非聚簇索引（二级索引，叶子节点存主键值，需回表查询）；

- 哈希索引：适合等值查询，不支持范围查询、排序；

- 全文索引：适合文本模糊查询（如MATCH AGAINST）。

1. 按功能：

- 主键索引：唯一且非空，默认创建聚簇索引；

- 唯一索引：值唯一，允许空值；

- 普通索引：无约束，加速查询；

- 联合索引：多列组合索引，遵循 “最左前缀原则”（如(a,b,c)，仅支持a、a+b、a+b+c查询）。

#### 索引失效场景：

- 用函数 / 表达式操作索引列（如where SUBSTR(name,1,2)='张'）；

- 隐式类型转换（如索引列是 int，查询条件是where id='123'）；

- 联合索引不满足最左前缀；

- like以 % 开头（如where name like '%三'）；

- or连接非索引列（如where id=1 or age=20，age 无索引）。

### 9. MySQL 隔离级别

#### 事务 ACID：

原子性（Atomicity）、一致性（Consistency）、隔离性（Isolation）、持久性（Durability）。

#### 隔离级别（由低到高，解决脏读、不可重复读、幻读问题）：

| 隔离级别                     | 脏读（读未提交） | 不可重复读（读已提交） | 幻读（重复读时数据新增 / 删除） |
| ---------------------------- | ---------------- | ---------------------- | ------------------------------- |
| Read Uncommitted（读未提交） | 允许             | 允许                   | 允许                            |
| Read Committed（读已提交）   | 禁止             | 允许                   | 允许                            |
| Repeatable Read（可重复读）  | 禁止             | 禁止                   | 禁止（InnoDB 通过 MVCC 实现）   |
| Serializable（串行化）       | 禁止             | 禁止                   | 禁止（加表锁，性能差）          |

- 默认隔离级别：InnoDB 默认Repeatable Read（RR），通过 MVCC（多版本并发控制）实现可重复读，避免幻读；

- 隔离级别越高，并发性能越差，需根据业务权衡（如互联网业务常用 RR）。

### 10. 给个 SQL 语句问加了什么锁（示例）

#### 核心原则：

锁类型取决于**隔离级别、SQL 操作（读 / 写）、索引是否命中**。

#### 示例 1：RR 隔离级别，主键等值更新（命中索引）

```
update user set name='test' where id=1; -- id是主键
```

- 加锁：**行锁（Record Lock）**，仅锁定 id=1 的行，不影响其他行。

#### 示例 2：RR 隔离级别，非主键非索引更新（未命中索引）

```
update user set name='test' where age=20; -- age无索引
```

- 加锁：**表锁（Table Lock）** 或 **间隙锁（Gap Lock）+ 临键锁（Next-Key Lock）**（InnoDB 会扫描全表，锁定所有行和间隙，避免幻读）。

#### 示例 3：RR 隔离级别，范围查询（命中联合索引）

```
select * from user where id between 1 and 10 for update; -- id是主键
```

- 加锁：**临键锁（Next-Key Lock）**，锁定 id 1-10 的行及相邻间隙（如 0-1、10-11），防止插入新数据（幻读）。

#### 总结：

- 写操作（update/delete/insert）：命中索引→行锁，未命中索引→表锁 / 临键锁；

- 读操作：RR 隔离级别下，普通查询（快照读）不加锁，for update/lock in share mode（当前读）加锁。

### 12. Redis 内存淘汰

当 Redis 内存达到maxmemory阈值时，触发内存淘汰策略，删除部分键释放内存：

#### 淘汰策略分类：

1. 只淘汰过期键：

   - volatile-lru（默认）：淘汰过期键中最近最少使用的；

   - volatile-ttl：淘汰过期键中剩余过期时间最短的；

   - volatile-random：随机淘汰过期键；

   - volatile-lfu：淘汰过期键中最不常使用的（Redis 4.0+）。

1. 淘汰所有键（不管是否过期）：

   - allkeys-lru：淘汰所有键中最近最少使用的；

   - allkeys-random：随机淘汰所有键；

   - allkeys-lfu：淘汰所有键中最不常使用的（Redis 4.0+）。

1. 不淘汰（默认禁用）：
   - noeviction：不淘汰任何键，新增数据时返回错误（OOM）。

#### 选择建议：

- 核心业务（如缓存用户信息）：用volatile-lru（只淘汰过期键，避免误删有效数据）；

- 非核心业务（如临时缓存）：用allkeys-lru（优先保留常用数据）。

### 13. 进程和线程区别

| 维度     | 进程                             | 线程                                                         |
| -------- | -------------------------------- | ------------------------------------------------------------ |
| 定义     | 操作系统资源分配的最小单位       | 操作系统调度的最小单位（进程内的执行单元）                   |
| 资源占用 | 独立资源（内存、文件句柄）       | 共享进程资源（同一进程的线程共享堆、方法区，私有栈、程序计数器） |
| 切换开销 | 大（需切换进程上下文）           | 小（仅切换线程上下文）                                       |
| 通信方式 | 复杂（管道、Socket、共享内存）   | 简单（直接访问进程共享资源）                                 |
| 稳定性   | 独立运行，一个进程崩溃不影响其他 | 共享进程资源，一个线程崩溃可能导致整个进程崩溃               |
| 并发粒度 | 粗（进程间并发）                 | 细（线程间并发）                                             |

### 14. AOP 和 IOC（Spring 核心）

#### IOC（Inverse of Control，控制反转）

- 核心思想：**将对象的创建、依赖注入交给 Spring 容器管理**，而非手动new对象；

- 目的：解耦（对象无需关心依赖对象的创建）、便于测试和维护；

- 实现方式：DI（Dependency Injection，依赖注入），包括构造器注入、Setter 注入、字段注入（@Autowired）；

- 容器：Spring 的ApplicationContext（应用上下文），负责对象的初始化、配置、销毁。

#### AOP（Aspect Oriented Programming，面向切面编程）

- 核心思想：**分离核心业务逻辑（如用户登录）和横切逻辑（如日志、事务、权限校验）**，横切逻辑封装为 “切面”，动态织入核心逻辑；

- 关键概念：

- 切面（Aspect）：横切逻辑的封装（如日志切面）；

- 切入点（Pointcut）：定义切面作用的方法（如所有@Controller注解的方法）；

- 通知（Advice）：切面的执行时机（@Before前置、@After后置、@Around环绕、@AfterReturning返回后、@AfterThrowing异常后）；

- 实现方式：JDK 动态代理（基于接口）、CGLIB 动态代理（基于子类，无接口时使用）。

### 15. 双亲委派

#### 核心定义：

Java 类加载器的加载机制：**当一个类加载器收到类加载请求时，先委托给父类加载器加载，只有父类加载器无法加载（找不到类）时，才由自身加载**。

#### 类加载器层级（从父到子）：

1. Bootstrap ClassLoader（启动类加载器）：加载JAVA_HOME/lib下的核心类（如java.lang.String），用 C++ 实现，无父类；

1. Extension ClassLoader（扩展类加载器）：加载JAVA_HOME/lib/ext下的扩展类；

1. Application ClassLoader（应用类加载器）：加载应用classpath下的类（自己写的代码）；

1. 自定义类加载器：继承ClassLoader，重写findClass()（自定义加载逻辑，如加载加密类）。

#### 目的：

- 沙箱安全：防止核心类被篡改（如自定义java.lang.String，父类加载器已加载核心类，不会再加载自定义类）；

- 类唯一性：同一个类由同一个类加载器加载，避免类重复（如不同类加载器加载同一类，会被认为是不同类）。

### 16. JVM 内存区域

根据《Java 虚拟机规范》，JVM 内存区域分为**线程共享区**和**线程私有区**：

#### 线程共享区（所有线程共用，OOM 可能发生）：

1. 堆（Heap）：最大内存区域，存储对象实例和数组，垃圾收集器主要回收区域；

1. 方法区（Method Area）：存储类信息（类名、字段、方法）、常量、静态变量、即时编译后的代码（JDK 8 后用元空间 Metaspace 实现，物理内存中，默认无上限）；

1. 运行时常量池（Runtime Constant Pool）：方法区的一部分，存储编译期生成的常量、符号引用，运行时可动态添加（如String.intern()）。

#### 线程私有区（每个线程独立，栈溢出可能发生）：

1. 程序计数器（Program Counter Register）：记录当前线程执行的字节码指令地址，无 OOM（唯一不会 OOM 的区域）；

1. 虚拟机栈（VM Stack）：存储方法调用的栈帧（局部变量表、操作数栈、方法出口等），栈深度超过阈值会抛出StackOverflowError；

1. 本地方法栈（Native Method Stack）：类似虚拟机栈，为 Native 方法（如System.currentTimeMillis()）提供内存空间。

### 17. 堆是怎么划分的（JVM 堆内存结构）

堆按**对象生命周期**划分，目的是优化垃圾回收效率（分代回收算法）：

#### 年轻代（Young Generation）：

- 存储新创建的对象（朝生夕死），垃圾回收频繁（Minor GC）；

- 分为：Eden 区（伊甸园，对象初始分配区）、Survivor 0 区（S0）、Survivor 1 区（S1）；

- 比例：默认 Eden:S0:S1 = 8:1:1（可通过-XX:SurvivorRatio调整）；

- 回收流程：Eden 满→Minor GC→存活对象复制到 S0→Eden 清空；下次 Eden 满→Minor GC→存活对象复制到 S1→S0 清空；反复几次后，存活对象晋升到老年代。

#### 老年代（Old Generation）：

- 存储存活时间长的对象（如缓存对象），垃圾回收频率低（Major GC/Full GC）；

- 回收算法：CMS 用并发标记清除，G1 用复制算法；

- 晋升条件：对象年龄达到阈值（默认 15，-XX:MaxTenuringThreshold）、大对象直接进入老年代（-XX:PretenureSizeThreshold）、Survivor 区对象放不下时晋升。

#### 永久代 / 元空间（PermGen/Metaspace）：

- JDK 7 及以前：永久代（PermGen），属于堆的一部分，存储方法区数据；

- JDK 8 及以后：元空间（Metaspace），脱离堆，使用物理内存，默认无上限（可通过-XX:MetaspaceSize和-XX:MaxMetaspaceSize限制）。

### 18. HashMap、ConcurrentHashMap

#### HashMap（非线程安全）

- 底层结构：JDK 8 前（数组 + 链表），JDK 8 后（数组 + 链表 + 红黑树）；

- 核心参数：初始容量（16）、负载因子（0.75），容量达到16×0.75=12时扩容（2 倍）；

- 红黑树转换：链表长度≥8 且数组长度≥64 时，链表转红黑树（查询时间复杂度从 O (n)→O (logn)）；

- 线程安全问题：多线程并发 put 时，可能导致链表循环（JDK 8 前）或数据覆盖（JDK 8 后）。

#### ConcurrentHashMap（线程安全，替代 Hashtable）

- 底层结构：JDK 8 前（分段锁Segment，每个 Segment 是一个 HashMap），JDK 8 后（数组 + 链表 + 红黑树 + CAS+synchronized）；

- 线程安全实现：

- JDK 8 前：分段锁（减少锁竞争，多个线程可操作不同 Segment）；

- JDK 8 后：对数组元素（链表头 / 红黑树根节点）加 synchronized，配合 CAS（无锁操作），锁粒度更细（仅锁定当前操作的节点）；

- 优势：并发性能优于 Hashtable（Hashtable 是全局锁），支持高并发读写；

- 常用方法：putIfAbsent()（原子操作，不存在则插入）、size()（无锁统计，通过 CounterCell 数组）。

### 19. RDB 和 AOF（Redis 持久化机制）

Redis 通过持久化将内存数据写入磁盘，避免重启后数据丢失，两种机制可单独使用或同时使用。

| 特性     | RDB（Redis DataBase）                         | AOF（Append Only File）                            |
| -------- | --------------------------------------------- | -------------------------------------------------- |
| 原理     | 定时生成内存数据的快照（二进制文件，.rdb）    | 记录每一条写命令（文本文件，.aof），重启时重放命令 |
| 优点     | 恢复速度快、文件体积小、对性能影响小          | 数据完整性高（默认每秒同步，最多丢 1 秒数据）      |
| 缺点     | 数据一致性差（定时快照，可能丢快照后的数据）  | 恢复速度慢、文件体积大、写命令多时有性能开销       |
| 触发方式 | 手动（save/bgsave）、自动（配置save 900 1等） | 手动（bgrewriteaof）、自动（配置appendfsync）      |
| 适用场景 | 数据允许少量丢失、追求恢复速度                | 数据不允许丢失、追求高一致性                       |

#### 最佳实践：

- 核心业务：RDB+AOF（AOF 保证数据完整性，RDB 用于快速恢复）；

- 非核心业务：仅 RDB（简化配置，降低性能开销）。

### 20. synchronized 和 ReentrantLock 的区别

| 维度     | synchronized                                             | ReentrantLock（可重入锁）                                    |
| -------- | -------------------------------------------------------- | ------------------------------------------------------------ |
| 底层实现 | JVM 层面（监视器锁monitorenter/monitorexit）             | JDK 层面（基于 AQS 框架，Java 代码实现）                     |
| 锁类型   | 非公平锁（默认），JDK 6 后优化为偏向锁→轻量级锁→重量级锁 | 公平锁 / 非公平锁（可指定，默认非公平）                      |
| 可重入性 | 支持（同一线程可多次获取锁）                             | 支持（可重入锁）                                             |
| 锁释放   | 自动释放（代码块执行完 / 异常时）                        | 手动释放（必须在finally中调用unlock()）                      |
| 功能扩展 | 无（仅基础加锁解锁）                                     | 支持中断锁（lockInterruptibly()）、超时锁（tryLock(long)）、条件变量（Condition） |
| 性能     | JDK 6 后优化（偏向锁、轻量级锁），性能接近 ReentrantLock | 并发场景下性能稳定，功能更灵活                               |
| 适用场景 | 简单加锁场景（如方法、代码块）                           | 复杂并发场景（如超时等待、中断、多条件唤醒）                 |

### 21. synchronized 底层

synchronized 的底层实现依赖**监视器锁（Monitor）**，JDK 6 对其进行了优化（锁升级），从低开销到高开销逐步升级：

#### 锁升级流程：

1. 无锁状态：对象刚创建，未被任何线程锁定；

1. 偏向锁：同一线程多次获取锁时，仅记录线程 ID（无 CAS 操作，开销最低）；

- 触发条件：只有一个线程竞争锁；

- 撤销：有其他线程竞争时，偏向锁撤销为轻量级锁；

1. 轻量级锁：多个线程交替获取锁（无激烈竞争），通过 CAS 操作修改对象头的锁记录（自旋锁，避免线程阻塞）；

- 自旋次数：默认 10 次（可通过-XX:PreBlockSpin调整），自旋失败则升级为重量级锁；

1. 重量级锁：多个线程激烈竞争，依赖操作系统的互斥锁（Mutex Lock），线程阻塞（开销最高）；

- 底层：通过对象头的Mark Word存储锁状态，Monitor 对象管理锁的所有者、等待队列。

#### 关键：对象头（Mark Word）

存储对象的锁状态、线程 ID、哈希码等信息，是锁升级的核心载体（不同锁状态下，Mark Word 的结构不同）。

### 22. 最长不重复子串（算法题，LeetCode 3）

#### 题目描述：

给定一个字符串s，请找出其中不含有重复字符的**最长子串**的长度。

#### 示例：

- 输入：s = "abcabcbb" → 输出：3（最长子串是"abc"）；

- 输入：s = "bbbbb" → 输出：1（最长子串是"b"）；

- 输入：s = "pwwkew" → 输出：3（最长子串是"wke"或"kew"）。

#### 最优解法：滑动窗口（双指针 + 哈希表）

- 思路：用滑动窗口维护当前不重复子串的范围[left, right]，用哈希表存储字符最后出现的索引；

- 步骤：

1. 初始化left=0、maxLen=0，哈希表map；

1. 遍历right（右指针），对于每个字符s[right]：

- 若s[right]在map中且map.get(s[right]) ≥ left（字符在当前窗口内重复），更新left = map.get(s[right]) + 1（左指针右移，排除重复字符）；

- 更新map中s[right]的索引为right；

- 计算当前窗口长度right - left + 1，更新maxLen；

- 时间复杂度：O (n)（每个字符遍历一次）；

- 空间复杂度：O (k)（k 为字符集大小，如 ASCII 码最多 128）。

#### 代码实现（Java）：

```
public int lengthOfLongestSubstring(String s) {
    Map<Character, Integer> map = new HashMap<>();
    int maxLen = 0;
    int left = 0;
    for (int right = 0; right < s.length(); right++) {
        char c = s.charAt(right);
        // 字符在当前窗口内重复，左指针右移
        if (map.containsKey(c) && map.get(c) >= left) {
            left = map.get(c) + 1;
        }
        map.put(c, right);
        maxLen = Math.max(maxLen, right - left + 1);
    }
    return maxLen;
}
```
