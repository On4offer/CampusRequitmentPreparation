# 面试题

**Redis 的持久化机制 RDB 和 AOF 有什么区别？各自适用什么场景？**

------

## 1️⃣ 概念

- **RDB（Redis DataBase）**：通过生成快照（Snapshot）的方式，将某一时刻内存中的数据写入磁盘。
- **AOF（Append Only File）**：以日志的形式记录 Redis 的每一个写操作，追加到文件中，重启时回放日志恢复数据。

------

## 2️⃣ 原理

- **RDB**：
  - 触发方式：手动 `SAVE` / `BGSAVE`，或配置规则定期触发。
  - 底层：fork 出子进程，将内存数据序列化写入临时文件，写完后替换旧文件。
  - 优点：恢复速度快，文件小。
  - 缺点：两次快照间如果宕机，可能丢失数据。
- **AOF**：
  - 记录所有写命令（类似 MySQL binlog），支持三种写回策略：
    - `always`：每次写命令都刷盘，最安全但性能差。
    - `everysec`：每秒刷盘一次（默认），最多丢失 1 秒数据。
    - `no`：由操作系统决定，性能高但可能丢失较多数据。
  - 优点：数据更完整，可通过 `rewrite` 压缩文件。
  - 缺点：文件体积较大，恢复速度慢。

------

## 3️⃣ 对比分析

| 对比项         | **RDB**                               | **AOF**                        |
| -------------- | ------------------------------------- | ------------------------------ |
| **持久化方式** | 周期性快照                            | 写命令追加日志                 |
| **数据安全性** | 可能丢失最近一次快照后的数据          | 最多丢失 1 秒数据（默认）      |
| **性能**       | 高（fork 子进程写磁盘，不影响主线程） | 较低（需要频繁写日志）         |
| **文件大小**   | 紧凑                                  | 较大                           |
| **恢复速度**   | 快                                    | 慢（需逐条回放命令）           |
| **应用场景**   | 大规模数据备份、快速重启              | 数据安全要求高、实时性高的业务 |

------

## 4️⃣ 使用场景

- **RDB 适用场景**：
  - 灾备：用 RDB 文件做远程容灾备份（文件小，易传输）。
  - 快速重启：恢复速度快，适合对启动时间敏感的系统。
- **AOF 适用场景**：
  - 数据安全性要求高：比如金融、电商订单系统。
  - 需要尽量减少数据丢失的场景。
- **混合持久化（Redis 4.0+）**：
  - 先用 RDB 生成快照，再用 AOF 追加增量日志。
  - 结合了 **RDB 快速恢复 + AOF 高可靠性** 的优点。

------

## 5️⃣ 项目实践（黑马点评 / 苍穹外卖）

- **黑马点评**：
  - 店铺数据主要来自数据库，缓存丢失可重建 → **不强依赖持久化**，一般只保留默认 RDB。
- **苍穹外卖**：
  - 订单、秒杀库存要求高可靠 → 可开启 AOF（everysec），避免丢单。
  - 为避免日志过大，需定期 AOF rewrite。

------

## 6️⃣ 面试标准回答模板

> Redis 提供两种持久化机制：
>
> - **RDB**：通过快照保存内存数据，文件小，恢复快，但可能丢失最近一次快照后的数据；
> - **AOF**：记录写命令日志，数据更安全，但文件大，恢复慢。
>    实际上两者可以结合使用：RDB 用于快速备份和恢复，AOF 保证数据安全。
>    在项目中，如果缓存数据可以从数据库重建，我一般只保留 RDB；如果是订单等关键数据，会开启 AOF 以减少丢失风险。
