# 面试题

**Redis 分布式锁如何实现？会有哪些问题？**

------

## 1️⃣ 概念

- **分布式锁**：在分布式系统中，用于保证同一时刻只有一个节点能操作某个共享资源。
- **Redis 实现思路**：利用 Redis 的 **原子性命令 + 过期时间** 来实现互斥访问。

------

## 2️⃣ 实现方式

### 基于单节点

1. **SETNX + EXPIRE（早期做法）**
   - `SETNX key value`：只有当 key 不存在时才能写入（加锁）。
   - `EXPIRE key time`：设置过期时间，避免死锁。
   - 问题：不是原子操作，可能 `SETNX` 成功后宕机，导致锁无过期时间。
2. **SET key value NX EX time（推荐）**
   - Redis 2.6.12+ 提供单条命令，保证 **加锁 + 设置过期时间** 的原子性。
   - value 一般设置为 **唯一标识（如 UUID）**，用于区分不同客户端。
3. **释放锁**
   - 不能直接 `DEL key`，要先判断 **value 是否是自己设置的**，否则可能误删他人锁。
   - 一般用 **Lua 脚本** 实现“检查 value 并删除”的原子操作。

------

### 基于多节点（Redlock 算法）

- 思路：在多个独立 Redis 节点上申请锁，超过一半节点成功则认为加锁成功。
- 优点：提高容错性（单节点宕机不影响）。
- 缺点：实现复杂，性能不如单机锁。

------

## 3️⃣ 存在的问题

1. **锁超时释放**
   - 如果业务执行时间 > 锁的 TTL，可能锁提前释放，导致并发问题。
   - 解决：**看门狗机制**（如 Redisson），定时延长锁的过期时间。
2. **锁误删**
   - 客户端 A 获得锁，执行时间过长 → 锁过期 → 客户端 B 获得锁 → A 执行完后删掉了 B 的锁。
   - 解决：释放锁时必须校验 value，一般用 **Lua 脚本**保证原子性。
3. **Redis 单点故障**
   - 如果 Redis 宕机，可能导致锁丢失。
   - 解决：采用 **Redlock 多节点方案**，提高可用性。
4. **不可重入**
   - 同一线程/请求需要多次加锁时会阻塞。
   - 解决：使用 Redisson 提供的可重入锁实现。

------

## 4️⃣ 使用场景

- 秒杀/库存扣减：保证同一商品库存操作的串行化，避免超卖。
- 订单处理：避免重复下单或并发支付。
- 定时任务：保证分布式环境下任务不被多个节点同时执行。

------

## 5️⃣ 项目实践（黑马点评 / 苍穹外卖）

- **黑马点评**：
  - 店铺缓存重建：用分布式锁保证同一热点 key 只有一个线程去回源数据库。
  - 使用 Lua 脚本 + SET NX EX，避免误删。
- **苍穹外卖**：
  - 秒杀库存预减：使用 Redisson 的分布式锁，解决超卖和重复下单问题。

------

## 6️⃣ 面试标准回答模板

> Redis 分布式锁通常用 `SET key value NX EX` 实现，保证加锁和过期时间设置的原子性。
>  value 设为唯一标识，释放时用 Lua 脚本检查 value 是否一致，避免误删。
>  在复杂场景下，可以用 **Redisson** 提供的可重入锁、看门狗机制，或使用 **Redlock** 算法实现跨节点锁，提高可靠性。
>  不过 Redis 分布式锁仍有一些问题，比如锁超时、误删、单点故障，需要结合业务选择方案。
