# 📌 面试题

Spring 的事务失效的常见场景？

------

### ✅ 一、概念回答

- Spring 的事务本质上是通过 **AOP 动态代理 + @Transactional 注解** 实现的。
- 如果事务没有被正确代理、传播或拦截，事务就会**失效**，导致方法执行时不会开启事务或事务无法回滚。

------

### ✅ 二、常见事务失效场景

1. **方法不是 public 的**
   - Spring 的事务是基于代理的，只有 `public` 方法才会被代理增强。
   - 如果加在 `private`、`protected`、`default` 方法上，事务不会生效。
2. **同类内部方法调用（自调用）**
   - 事务依赖代理对象调用，如果同一个类中 A 方法调用 B 方法，B 上的 `@Transactional` 不会生效，因为调用走的是 **this（本类）** 而不是代理对象。
3. **未被 Spring 管理的类调用**
   - 如果类没有交给 Spring 容器管理（如 `new` 出来的对象），`@Transactional` 不会生效。
4. **异常未抛出或类型不匹配**
   - 默认情况下，Spring 事务只在 **运行时异常（RuntimeException）和 Error** 时回滚。
   - 如果抛出的是受检异常（如 IOException），事务不会回滚，除非显式配置 `rollbackFor = Exception.class`。
   - 如果异常被 **捕获并处理**（没有抛出到代理层），事务也不会回滚。
5. **数据库不支持事务**
   - 如果操作的数据库表引擎是 MyISAM（不支持事务），即使加了事务也不会回滚。
   - 必须用支持事务的存储引擎（如 InnoDB）。
6. **事务方法内部开启了新线程**
   - 事务绑定在当前线程，子线程不会继承事务上下文。
   - 新线程里的数据库操作不会受外层事务控制。
7. **@Transactional 注解位置错误**
   - 只能加在 **类或方法** 上，如果加在接口上（JDK 动态代理不会识别），有可能失效。
8. **事务传播属性设置不当**
   - 比如外层事务是只读的，而内层需要写操作，可能导致事务不生效。
   - 或者 REQUIRES_NEW 等传播机制误用，导致事务不按预期工作。

------

### ✅ 三、结合项目经验

- 在 **黑马点评** 项目中：
  - 我们在 Service 层加了 `@Transactional` 来保证下单操作的原子性。
  - 一开始出现过事务不生效的问题，原因是我们在同一个类中调用了另一个事务方法（自调用），后来改为通过 **注入代理对象** 调用，才解决问题。
  - 另外，为了保证订单异常时能回滚，我们加了 `rollbackFor = Exception.class`，避免 checked 异常不回滚的问题。

### ✅ 四、扩展追问（可能会问到）

1. 为什么事务必须加在 `public` 方法上？
2. 如何解决事务在同类内部调用时失效的问题？（答：注入自身代理 `AopContext.currentProxy()` 或者通过其他 Bean 调用）
3. 默认情况下事务为什么只对 RuntimeException 回滚？
4. 如果我想对自定义异常也回滚该怎么做？（`rollbackFor`）
5. 如果数据库层用了 MyISAM，会发生什么？为什么？

------

⚡ **标准回答模板（面试时可直接说）：**

> Spring 的事务常见失效场景包括：方法不是 public、自调用导致代理失效、类未被 Spring 管理、异常未抛出或类型不匹配、数据库不支持事务、在新线程中执行事务方法、注解加在接口上、以及传播属性设置不当。比如在黑马点评项目中，我们就遇到过事务失效的问题，原因是同类内部方法调用，后来通过 AopContext 获取代理对象调用才解决。在面试中一定要强调：Spring 的事务依赖 AOP 代理，失效的根本原因就是**没有走到代理增强**。
