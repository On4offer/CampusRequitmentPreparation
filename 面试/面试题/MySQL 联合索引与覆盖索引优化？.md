# 面试题：覆盖索引、联合索引的优化策略是什么？

------

## 1. 概念

- **覆盖索引（Covering Index）**
  - 查询所需的字段 **都在索引里**，无需回表。
  - 优点：减少一次随机 I/O（避免回表），提高性能。
- **联合索引（Composite Index / 多列索引）**
  - 一个索引包含多个字段（如 `(a, b, c)`）。
  - 遵循 **最左前缀原则**：查询条件必须从索引的最左列开始连续匹配，索引才能生效。

------

## 2. 原理

- **覆盖索引**
  - InnoDB 的二级索引只存索引列和主键 ID。
  - 如果查询只涉及索引列 → 直接返回，避免“回表”。
- **联合索引**
  - 索引 `(a, b, c)` 本质是按照 `(a) → (a,b) → (a,b,c)` 排序。
  - `WHERE a=1 AND b=2` 可用到 `(a,b)` 部分索引。
  - 但如果 `WHERE b=2`，因为跳过了最左的 `a`，则索引失效。

------

## 3. 案例

**覆盖索引**：

```sql
-- 索引：idx_name (name)
SELECT name FROM user WHERE name = 'Tom'; 
-- 只查 name，直接命中索引，避免回表
```

**联合索引**：

```sql
-- 联合索引：idx_user (age, city)
SELECT * FROM user WHERE age=20 AND city='Beijing';  -- 命中
SELECT * FROM user WHERE city='Beijing';             -- 不命中
```

------

## 4. 使用场景

- **覆盖索引**：高频查询场景，如搜索列表页，只返回部分字段。
- **联合索引**：多条件组合查询，如电商系统 `(user_id, product_id)`，同时过滤多个字段。
- **优化策略**：
  - 索引列尽量选择 **区分度高** 的列放在前面。
  - 避免在索引列上做函数计算/隐式类型转换。
  - 尽量让查询用到覆盖索引，减少回表。

------

## 5. 面试模板回答

> MySQL 常见优化策略包括覆盖索引和联合索引。
>
> - 覆盖索引指查询字段完全包含在索引里，避免回表，提高性能。
> - 联合索引指多个字段组成的索引，遵循最左前缀原则，适合多条件组合查询。
>    在实际优化中，常把区分度高的字段放在联合索引前面，并通过覆盖索引减少磁盘 I/O。
