## ❓ RocketMQ 与 Kafka、RabbitMQ 相比有什么区别？

### 1. 概念

市面上常见的消息队列中间件有 **RocketMQ（阿里开源）**、**Kafka（LinkedIn 开源，后由 Apache 维护）** 和 **RabbitMQ（Erlang 语言开发，AMQP 协议实现）**。
 它们都提供 **异步解耦、削峰填谷、可靠传输** 的能力，但在架构设计、功能特性、适用场景上各有侧重。

------

### 2. 对比分析

| 对比项        | **RocketMQ**                               | **Kafka**                                  | **RabbitMQ**                              |
| ------------- | ------------------------------------------ | ------------------------------------------ | ----------------------------------------- |
| **语言/架构** | Java 实现，轻量，单机性能百万级 TPS        | Scala/Java 实现，高吞吐，日志型存储        | Erlang 实现，基于 AMQP 协议，功能丰富     |
| **消息模型**  | Pub/Sub + 队列模式，支持顺序消息、事务消息 | 典型的 **发布订阅模型**，分区存储          | 队列模型，路由灵活，支持多种交换机        |
| **可靠性**    | 支持 **事务消息、死信队列、延时消息**      | 高吞吐，但事务支持弱                       | ACK 机制成熟，可靠性高                    |
| **性能**      | 单机百万级 TPS，延时低                     | 极致高吞吐，适合大数据日志收集             | 吞吐量一般（万级），延时低                |
| **顺序消息**  | 支持（队列级别有序消费）                   | 分区内有序，全局有序需特殊设计             | 默认不保证顺序                            |
| **事务支持**  | 独有事务消息机制，保证最终一致性           | 弱事务（幂等/补偿）                        | 支持事务，但性能较差                      |
| **运维难度**  | 中等，配置简单，阿里系系统常用             | 依赖 ZooKeeper（新版本用 KRaft），集群复杂 | 功能多但复杂，学习成本高                  |
| **典型场景**  | 电商订单、支付、秒杀等 **强一致性场景**    | 大数据实时日志、埋点分析、流计算           | 金融、IM、物联网消息推送等 **高可靠场景** |

------

### 3. 案例

- **RocketMQ**：在 **苍穹外卖项目** 中用于 **订单支付成功 → 通知商家端和骑手端**，需要事务消息保证一致性。
- **Kafka**：在大数据项目中用于 **日志采集、埋点上报**，追求高吞吐。
- **RabbitMQ**：在金融系统中用于 **交易流水处理、消息推送**，追求高可靠、低延时。

------

### 4. 使用场景

- **RocketMQ**：适合金融、电商订单、支付、秒杀等 **要求高一致性、支持事务的业务**。
- **Kafka**：适合日志收集、实时分析、流式计算等 **大数据场景**。
- **RabbitMQ**：适合消息通知、IM 聊天、物联网等 **可靠性要求高、并发量适中** 的业务。

------

### 5. 面试标准回答（背诵版）

“RocketMQ、Kafka 和 RabbitMQ 都是常见的消息中间件。RocketMQ 在事务消息、顺序消息、延时消息方面功能更强，适合电商、支付等一致性要求高的场景；Kafka 追求高吞吐，常用于大数据日志收集；RabbitMQ 基于 AMQP 协议，可靠性高，适合金融、消息通知等场景。因此我们在项目里选择 RocketMQ 来保证订单和支付消息的可靠一致性。”
