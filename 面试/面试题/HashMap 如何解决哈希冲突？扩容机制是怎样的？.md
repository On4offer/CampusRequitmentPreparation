# 📌 面试题：HashMap 如何解决哈希冲突？扩容机制是怎样的？

## 一、核心回答

1. **哈希冲突解决方式**
   - HashMap 采用 **链地址法（拉链法）**：
     - 数组元素存放的是链表或红黑树的头节点。
     - 当多个 key 经过哈希函数计算后得到相同索引时，会挂到该索引的链表（或红黑树）上。
     - JDK1.8 引入红黑树优化：当链表长度超过 **8** 且数组容量大于 **64** 时，会将链表转换为红黑树，查询复杂度从 O(n) 降到 O(log n)。
2. **扩容机制**
   - 默认初始容量为 16，负载因子默认 0.75。
   - 当 **元素数量 > 阈值（容量 \* 负载因子）** 时触发扩容。
   - 扩容步骤：
     1. 容量翻倍（newCap = oldCap * 2）。
     2. 创建新数组。
     3. 节点重新分配位置：
        - JDK1.7：所有节点 **重新计算 hash 值（rehash）**，再放入新数组。
        - JDK1.8：优化了迁移逻辑，每个节点只需判断是否在 **原索引** 或 **原索引 + oldCap**，减少 rehash 计算，提高效率。

------

## 二、扩展回答（面试官可能追问）

- **为什么要用红黑树？**
  - 链表查询是 O(n)，在冲突严重时性能差；红黑树能稳定在 O(log n)。
- **为什么是阈值 8 和 64？**
  - 链表过长才值得转红黑树，避免小数据量时维护红黑树带来的额外开销。
- **扩容有什么缺点？**
  - 扩容过程耗时，且会引发频繁的 GC；因此在高并发场景下扩容是性能瓶颈。

------

## 三、项目实战回答示例

- 在 **黑马点评项目** 中，热点商户数据存储在本地缓存的 HashMap 里，如果热点数据量很大，容易触发扩容，所以实际使用时会考虑设置合理的初始容量，避免频繁扩容带来的性能损耗。
- 在 **苍穹外卖项目** 中，菜单类别、订单状态映射等用 HashMap 保存配置数据，因为数据量相对固定，不会频繁触发扩容。

## 四、常见追问

1. HashMap 的负载因子为什么是 0.75？
2. 扩容是怎么保证线程安全的吗？（答案：HashMap 本身不保证线程安全，多线程下可能死循环或数据丢失）。
3. ConcurrentHashMap 的扩容机制与 HashMap 有什么不同？

✅ **一分钟口述版总结**
 HashMap 通过链地址法解决哈希冲突：数组存放链表或红黑树，冲突时挂在链表，链表长度超过 8 且数组容量大于 64 时转为红黑树。扩容时，当元素数量超过容量*负载因子，容量翻倍。JDK1.7 需要重新计算 hash，JDK1.8 优化后节点只需判断在原索引或原索引+oldCap，减少开销。
