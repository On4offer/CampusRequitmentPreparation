# 面试题：索引的底层结构是什么？为什么 MySQL 使用 B+ 树？

------

## 1. 概念

- **索引**：加速数据检索的数据结构，相当于“书的目录”。
- MySQL 常见索引结构：
  - **B+ 树索引**（InnoDB 主流，默认）
  - **Hash 索引**（Memory 引擎支持，适合等值查询）
  - **全文索引**（MyISAM/InnoDB，适合文本检索）
  - **R-Tree 索引**（空间索引，GIS 应用场景）

------

## 2. 原理 & 为什么用 B+ 树

- **B+ 树特性**
  1. 多路平衡搜索树，所有数据存放在 **叶子节点**，非叶子节点只存索引。
  2. 叶子节点通过链表相连 → **高效范围查询**。
  3. 树的高度较低，通常 2~4 层即可存储百万级数据，**磁盘 IO 次数少**。
- **为什么不用其他结构？**
  - **二叉搜索树（BST/AVL/红黑树）**：树高过深，磁盘 IO 次数过多。
  - **Hash 表**：只支持等值查询，不支持范围查询、排序。
  - **B 树 vs B+ 树**：B 树非叶子节点也存数据，范围查询效率低；B+ 树所有数据在叶子节点，天然支持区间扫描。

------

## 3. 案例

- 查询 `SELECT * FROM user WHERE id BETWEEN 100 AND 200;`
  - **Hash 索引**：只能逐个匹配，不适合范围查询。
  - **B+ 树索引**：通过叶子节点链表可以快速遍历 `[100,200]` 范围内的数据。

------

## 4. 使用场景

- B+ 树索引 → 主键、普通索引、大部分 OLTP 场景。
- Hash 索引 → 内存表、缓存类业务，只适合等值查询。
- 全文索引 → 搜索引擎/模糊匹配场景。

------

## 5. 面试模板回答

> MySQL 索引底层主要采用 **B+ 树** 实现。
>  相比于 Hash 和 B 树，B+ 树的优势在于：
>
> - 树高更低，磁盘 IO 次数少；
> - 所有数据都存储在叶子节点，叶子节点通过链表相连，支持高效范围查询；
> - 非叶子节点只存索引，提高缓存命中率。
>    因此，InnoDB 默认使用 B+ 树作为索引结构。
