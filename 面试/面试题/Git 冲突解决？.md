# 面试题：如何解决 Git 冲突？

### 1. 概念解释

- **Git 冲突**：当多个分支修改了同一文件的同一区域，Git 无法自动合并，就会提示“冲突（conflict）”。
- 冲突不是错误，而是 Git 要求开发者明确“最终结果”应该是什么。

------

### 2. 冲突出现的场景

1. **合并分支（merge）** 时出现不同修改。
2. **变基（rebase）** 时，当前分支提交和基线分支提交冲突。
3. **多人协作** 同时修改同一个文件/同一行代码。

------

### 3. 解决冲突的步骤

1. **查看冲突文件**

   - Git 会提示 `CONFLICT`，并在文件中标记：

   ```text
   <<<<<<< HEAD
   当前分支的内容
   =======
   合并进来的分支内容
   >>>>>>> feature
   ```

2. **手动修改文件**

   - 删除冲突标记 `<<<<<<< ======= >>>>>>>`，根据业务逻辑保留或合并两边的内容。

3. **标记冲突已解决**

   ```bash
   git add <file>
   ```

4. **完成合并**

   - 如果是 merge：`git commit`
   - 如果是 rebase：`git rebase --continue`

5. **验证代码**

   - 编译运行 + 测试，确保逻辑正确。

------

### 4. 案例说明（结合项目）

在 **苍穹外卖** 项目中：

- 我在 `feature/order` 分支新增了订单支付逻辑，另一个同学在 `feature/coupon` 分支修改了同一行 `OrderService.java` 的 `updateStatus()` 方法。
- 合并时 Git 提示冲突，我们打开文件，看到两边修改了同一行：
  - 我的逻辑是更新订单状态并写入支付时间；
  - 他的是更新订单状态并写入优惠券使用信息。
- 最终我们手动合并，把两部分代码都保留，并通过单元测试验证。

------

### 5. 使用场景

- **日常开发**：个人分支 rebase main 时可能冲突。
- **团队协作**：合并 PR 时最常见，需要团队沟通确定最终逻辑。
- **CI/CD**：冲突必须先解决，否则流水线无法继续。

------

### 6. 面试标准回答模板

> Git 冲突通常发生在 **merge 或 rebase** 时，因为多个分支修改了同一文件的同一区域。
>  解决步骤是：先查看冲突文件，手动编辑并删除冲突标记，保留需要的逻辑，然后 `git add` 标记已解决，最后提交或继续 rebase。
>  在项目中（比如苍穹外卖），我遇到过多人同时修改订单服务类的情况，我们沟通后把两边逻辑合并，保证支付和优惠券功能都正确执行。
