# 🎯 面试题

**乐观锁和悲观锁的区别？在 Java 中如何实现？**

## ✅ 标准回答（一句话速背）

- **悲观锁**：先上锁再干活，默认“会冲突”，靠**阻塞/串行**保证安全（如 `synchronized`、`ReentrantLock`、DB 的 `SELECT … FOR UPDATE`）。
- **乐观锁**：先干活再提交，默认“很少冲突”，靠**版本号/CAS 比对**失败重试（如 `Atomic*`、版本字段 `version`）。

------

## ⚙️ 原理要点

- **悲观锁**：进入临界区前获取独占/共享锁，其他线程等待；吞吐量低但**强一致**、实现简单。
- **乐观锁**：读取→计算→提交前比对（期望值=当前值）→一致则提交，不一致则**失败或重试**；吞吐量高但对**冲突敏感**。

------

## 🛠 在 Java 中如何实现

### 1) 悲观锁实现

**代码（内存级）：**

```java
// 互斥：标准悲观锁
private final ReentrantLock lock = new ReentrantLock(true); // 可选公平
public void update() {
    lock.lock();
    try {
        // 临界区
    } finally {
        lock.unlock();
    }
}
```

**数据库级：**

```sql
-- 行级锁（事务内使用）
BEGIN;
SELECT stock FROM t_item WHERE id = ? FOR UPDATE; -- 加排他锁
UPDATE t_item SET stock = stock - 1 WHERE id = ?;
COMMIT;
```

### 2) 乐观锁实现

**CAS（内存级）：**

```java
AtomicInteger stock = new AtomicInteger(100);
boolean ok = false;
while (!ok) {
    int cur = stock.get();
    if (cur <= 0) break;
    ok = stock.compareAndSet(cur, cur - 1); // 失败则自旋重试
}
```

**版本号（数据库级）：**

```sql
-- 表含 version 列
UPDATE t_item 
SET stock = stock - 1, version = version + 1
WHERE id = ? AND stock > 0 AND version = ?;  -- 命中1行表示成功
// 失败返回0行 → 重读version并重试/失败回退
```

------

## 📌 适用场景（结合你的项目）

- **悲观锁**：高冲突区、强一致交易（如外卖“余额扣减+写订单”一步内必须原子）；热点资源串行化。
- **乐观锁**：读多写少、冲突低（如黑马点评**点赞数、阅读数**，外卖**店铺浏览统计**）；高并发秒杀下的**扣库存**可用版本号+有限次重试+限流（冲突激烈时配合队列/MQ）。

------

## ⚠️ 易错点 & 对策

- **乐观锁活锁/饿死**：冲突极高自旋过久 → 设**最大重试次数/退避**，退回队列。
- **CAS 的 ABA 问题**：A→B→A 被误判未变 → 用 `AtomicStampedReference`（版本/时间戳）或**版本号列**。
- **悲观锁吞吐低**：长事务/大粒度锁导致阻塞 → **缩短临界区、细粒度分段锁**，只锁必要数据。
- **DB 乐观锁失败后误覆盖**：忽略 `UPDATE` 影响行数 → 必须检查 `rows==1` 才算成功。
- **分布式场景**：JVM 锁仅进程内有效 → 用 **Redis 分布式锁/数据库行锁**；注意锁续期与原子解锁。

## ⛳ 对比速表

| 维度     | 悲观锁                                      | 乐观锁                       |
| -------- | ------------------------------------------- | ---------------------------- |
| 冲突假设 | 一定会冲突                                  | 很少冲突                     |
| 开销     | 阻塞/上下文切换                             | 重试/自旋                    |
| 一致性   | 强                                          | 需处理失败重试               |
| 吞吐     | 低-中（高竞争差）                           | 中-高（低竞争优）            |
| 典型实现 | `synchronized`/`ReentrantLock`/`FOR UPDATE` | `CAS`/`Atomic*`/`version` 列 |

## 🔍 追问备选

1. **CAS 一定比锁快吗？**
    不是。**高冲突**时 CAS 反复失败，自旋成本高，可能**慢于**悲观锁。
2. **数据库乐观锁适合什么场景？**
    读多写少、低冲突的**属性更新**（积分、点赞、库存预减等）。
3. **如何降低乐观锁冲突？**
    分片键/热点分离、排队（MQ/队列）、**随机退避**、限频限流。
