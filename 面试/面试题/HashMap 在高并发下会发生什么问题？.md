# 📌 面试题：HashMap 在高并发下会发生什么问题？

## 一、核心回答

1. **线程不安全**
   - HashMap 本身不是线程安全的，在多线程环境下会出现以下问题：
2. **具体问题**
   - **数据丢失/覆盖**：多个线程同时 put 时，可能互相覆盖，导致数据不一致。
   - **扩容死循环（JDK1.7 特有）**：
     - HashMap 在扩容时采用 **头插法**。
     - 多线程同时 rehash 时可能会形成 **环形链表**，导致 get 时 CPU 飙升至 100%（死循环）。
   - **元素丢失**：扩容迁移过程中，如果多个线程同时操作，可能导致节点丢失。
   - **读到脏数据**：遍历时结构被修改，可能导致读取异常或抛出 `ConcurrentModificationException`。

------

## 二、扩展回答

- **为什么会死循环？**
  - JDK1.7 扩容时，rehash 会把链表反转。
  - 多线程同时迁移，可能形成环形链表，导致 get 时无限循环。
- **JDK1.8 改进**
  - 改用 **尾插法**，解决了环形链表问题。
  - 但仍然可能出现 **数据覆盖、丢失**，因为没有加锁。

------

## 三、项目场景举例

- 在 **黑马点评项目** 中，如果把热点商户缓存存在 HashMap 里，并且多个线程同时写入，就可能出现数据错乱或丢失，因此需要用 `ConcurrentHashMap` 或加锁来保证安全。
- 在 **苍穹外卖项目** 中，如果用 HashMap 存储订单状态，在高并发下更新可能出现状态丢失，导致订单流程混乱。

------

## 四、常见追问

1. **如何解决 HashMap 的线程安全问题？**
   - 使用 `Collections.synchronizedMap(new HashMap<>())`。
   - 使用 `ConcurrentHashMap`（推荐）。
2. **ConcurrentHashMap 为什么不会出现死循环？**
   - JDK1.8 采用 CAS + synchronized，扩容时多线程协作迁移，避免了环形链表问题。
3. **HashTable 和 ConcurrentHashMap 的区别？**
   - HashTable 整个表加锁，性能差；ConcurrentHashMap 锁粒度更细，性能更高。

------

✅ **一分钟口述版总结**
 HashMap 在高并发下不安全，可能出现数据丢失、覆盖，甚至在 JDK1.7 中扩容时形成环形链表导致死循环。JDK1.8 解决了死循环问题，但依然存在线程不安全。因此在并发场景下应使用 ConcurrentHashMap。
