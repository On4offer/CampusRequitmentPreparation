# 📌 面试题：HashMap 的底层实现原理？JDK 1.7 和 1.8 有哪些变化？为什么要引入红黑树？

## 一、核心回答（面试官想听的点）

1. **底层结构**
   - HashMap 底层是 **数组 + 链表 + 红黑树（JDK 1.8 引入）**。
   - 元素通过 **key 的 hash 值 + 扰动函数** 定位数组索引。
2. **JDK 1.7 实现**
   - 结构：**数组 + 链表**。
   - 冲突解决：采用拉链法，冲突的元素存储在链表中。
   - 扩容机制：达到阈值时（负载因子 * 容量），数组容量翻倍，并进行 rehash。
   - 链表插入：采用 **头插法**（可能导致链表反转 + 并发下死循环问题）。
3. **JDK 1.8 实现**
   - 结构：**数组 + 链表 + 红黑树**。
   - 链表长度超过 8 且数组容量大于 64 时，链表会转为 **红黑树**，提升查询效率。
   - 链表插入：采用 **尾插法**，避免并发下环形链表问题。
   - 扩容时：链表节点可能直接分为 **原索引** 和 **原索引+oldCap** 两组，减少 rehash 开销。
4. **引入红黑树的原因**
   - 链表查询时间复杂度是 O(n)，当哈希冲突严重时性能下降。
   - 红黑树能保证查询复杂度稳定在 O(log n)，大大优化了性能。

------

## 二、扩展回答（可展开细讲）

- **Hash 函数优化**
  - JDK 1.8 使用了 `hashCode ^ (hashCode >>> 16)` 的扰动函数，减少低位冲突。
- **负载因子 (LoadFactor)**
  - 默认是 0.75，在时间和空间利用率之间做平衡。
- **红黑树退化**
  - 当红黑树节点数量小于 6 时，会退回链表，避免维护树的额外开销。
- **线程安全性**
  - HashMap 本身是非线程安全的。多线程扩容时可能死循环，因此在并发场景下用 `ConcurrentHashMap`。

## 三、项目实战（结合黑马点评/苍穹外卖）

- **黑马点评**：商户缓存时可能把热点商户列表放在 HashMap 里，本地缓存查询很快。
- **苍穹外卖**：用户订单状态枚举或菜品类别映射，通常会用 HashMap 来存储和快速查找。

✅ **一分钟口述版总结**：
 HashMap 底层是数组+链表，在 JDK1.8 之后引入了红黑树。1.7 用头插法 + rehash，1.8 改为尾插法，链表长度超过 8 时转红黑树，查询复杂度从 O(n) 优化为 O(log n)。这样避免了哈希冲突严重时性能下降的问题。
