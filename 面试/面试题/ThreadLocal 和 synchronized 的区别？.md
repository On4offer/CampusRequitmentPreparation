## ✅ 面试题：ThreadLocal 和 synchronized 的区别？

### 一、核心区别

| 对比点   | **ThreadLocal**                                       | **synchronized**                                          |
| -------- | ----------------------------------------------------- | --------------------------------------------------------- |
| 设计目的 | **线程隔离**：为每个线程提供独立副本                  | **线程同步**：保证多个线程串行访问共享资源                |
| 关注点   | 数据副本的“独立性”                                    | 临界区代码/共享资源的“互斥性”                             |
| 底层实现 | 每个线程维护一个 `ThreadLocalMap`，存储属于自己的变量 | 基于 **JVM 对象头 + Monitor（重量级锁）** 或偏向/轻量级锁 |
| 使用场景 | 保存用户信息、事务上下文、数据库连接等线程独享的数据  | 对共享变量的读写加锁，保证原子性和可见性                  |
| 性能表现 | 无需加锁，线程间无竞争                                | 存在锁竞争，可能导致阻塞和上下文切换                      |
| 数据共享 | 线程之间**互不可见**                                  | 线程之间**共享同一份数据**                                |

------

### 二、类比理解

- **ThreadLocal**：好比**每个线程一人一份试卷**，各写各的，不会互相影响。
- **synchronized**：好比**一支笔大家轮流用**，必须排队，保证不会同时写坏。

------

### 三、实际场景举例

1. **ThreadLocal**：
   - 在 Web 项目中保存**当前用户信息（UserContext）**，避免多层方法传参。
   - 在数据库操作中存放**当前线程持有的 Connection**，保证事务一致性。
2. **synchronized**：
   - 订单扣减库存时，保证同一商品的库存更新操作**串行化**，避免超卖。

------

### 四、标准回答（面试模板）

> ThreadLocal 和 synchronized 的本质区别是：**ThreadLocal 解决的是线程间的数据隔离问题，而 synchronized 解决的是线程间的并发冲突问题**。
>  ThreadLocal 为每个线程提供独立的变量副本，各线程互不干扰；synchronized 是让多个线程排队访问同一个共享资源。
>  在实际项目中，我会用 ThreadLocal 保存用户信息或数据库连接，而在涉及共享资源写操作时，用 synchronized 或锁机制来保证一致性。
