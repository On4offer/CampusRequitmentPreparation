# 📌 面试题

Spring 中的事务传播机制有几种？具体应用场景是什么？

------

### ✅ 一、概念回答

- **事务传播机制**：当一个事务方法被另一个事务方法调用时，Spring 如何管理这两者之间事务关系。
- 通过 `@Transactional(propagation = Propagation.XXX)` 指定传播行为。
- 主要有 **7 种传播机制**。

------

### ✅ 二、7 种事务传播机制

1. **REQUIRED（默认）**
   - 如果当前存在事务，则加入该事务；如果没有事务，则新建事务。
   - **场景**：大多数业务逻辑，保证整体一致性。
   - 例：下单时，扣库存、减余额、写订单表都在同一事务中。
2. **REQUIRES_NEW**
   - 无论是否存在事务，都新建一个事务，原事务挂起。
   - **场景**：日志记录、发送消息，不希望受主事务回滚影响。
   - 例：下单主事务失败了，但操作日志事务仍然单独提交。
3. **SUPPORTS**
   - 如果有事务，则加入事务；如果没有事务，则以非事务方式运行。
   - **场景**：可有可无的读操作。
   - 例：查询用户信息时，如果外层有事务，就参与；否则直接执行。
4. **NOT_SUPPORTED**
   - 总是以非事务方式执行，如果存在事务，则挂起。
   - **场景**：执行一些大查询，不希望占用事务资源。
5. **MANDATORY**
   - 必须在事务中运行，如果没有事务则抛出异常。
   - **场景**：依赖上层必须开启事务的模块，如财务系统中的结算方法。
6. **NEVER**
   - 必须在非事务状态下运行，如果存在事务则抛出异常。
   - **场景**：不允许在事务中运行的操作，比如某些只读、对性能敏感的统计方法。
7. **NESTED**
   - 如果当前有事务，则在事务内再开启一个“嵌套事务”（保存点机制）；外层回滚可以影响内层，但内层失败不会强制外层回滚。
   - **场景**：局部操作需要回滚，但不影响整体事务。
   - 例：下单流程中，优惠券扣减失败时可以只回滚优惠券，不影响主订单事务。

------

### ✅ 三、项目结合（黑马点评 / 苍穹外卖）

- **黑马点评**：用户下单 → 扣库存 + 写订单 + 记录操作日志
  - 订单、库存：`REQUIRED`（同一事务）。
  - 日志记录：`REQUIRES_NEW`（即使订单失败也要写日志）。
- **苍穹外卖**：用户支付 → 扣余额 + 更新订单状态 + 发送 MQ 消息
  - 扣余额、更新订单：`REQUIRED`。
  - 发送 MQ：`REQUIRES_NEW`，保证消息可靠投递，不被主事务影响。

### ✅ 四、扩展追问（可能被问到）

1. `REQUIRES_NEW` 和 `NESTED` 的区别？
   - 前者挂起外事务，完全独立；后者依赖外事务，基于保存点。
2. 在 SpringBoot + JTA 分布式事务中，传播机制是否完全生效？
3. 如果在同一个类中调用事务方法，传播机制为什么可能失效？（**事务是通过代理实现的**，自调用不会生效）
4. @Transactional 注解为什么常失效？（如修饰 private 方法 / 代理失效 / rollbackFor 配置问题）
5. MyBatis-Plus 的 Service 层事务默认传播机制是什么？

------

⚡ **标准回答模板（面试时可直接说）：**

> Spring 的事务传播机制有 7 种，默认是 `REQUIRED`。常用的是 `REQUIRED`（加入当前事务）、`REQUIRES_NEW`（新开事务）、`NESTED`（嵌套事务）。比如在黑马点评里，订单和库存用 `REQUIRED` 保证一致性，但日志记录用 `REQUIRES_NEW`，避免被主事务回滚影响。要注意的是，传播机制是通过代理实现的，如果在同类方法内自调用，事务可能不生效。
