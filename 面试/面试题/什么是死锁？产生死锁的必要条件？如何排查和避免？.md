# 🎯 面试题

**什么是死锁？产生死锁的必要条件？如何排查和避免？**

------

## ✅ 标准回答

### 1. 死锁的定义

- **死锁**是指两个或多个线程在执行过程中，因争夺资源而造成的一种互相等待的现象，导致线程永久阻塞，程序无法继续推进。

------

### 2. 死锁产生的四个必要条件（操作系统经典理论）

1. **互斥条件**：资源一次只能被一个线程占用。
2. **请求与保持条件**：线程至少持有一个资源，同时请求新的资源时不释放已有资源。
3. **不可剥夺条件**：线程已获得的资源在使用完之前，不能被强制剥夺。
4. **循环等待条件**：存在一个线程资源的循环等待链。

👉 **四个条件同时满足 → 才会死锁。**

------

### 3. 如何排查死锁

1. **线程堆栈分析**
   - 使用 `jstack <pid>` 查看线程堆栈，是否有 `Found one Java-level deadlock`。
2. **监控工具**
   - `jconsole`、`jvisualvm`，能直观显示死锁线程。
3. **日志排查**
   - 在关键加锁/解锁位置打印日志，辅助定位死锁链路。

------

### 4. 如何避免死锁

1. **减少锁的持有时间**：尽快释放锁，避免长时间占用。
2. **固定加锁顺序**：多个线程获取多个资源时，按照相同顺序加锁，避免循环等待。
3. **尝试锁**：用 `tryLock(timeout)` 代替 `lock()`，获取不到锁时超时退出。
4. **细化锁粒度**：避免大范围锁定资源。
5. **使用并发容器**：用 `ConcurrentHashMap`、`BlockingQueue` 代替手写锁。

------

## 📌 使用场景（结合项目）

- **黑马点评项目**：
  - 如果缓存重建逻辑中，线程 A 获取店铺锁，又去获取用户锁，而线程 B 先拿到用户锁，再请求店铺锁 → 可能导致死锁。
- **苍穹外卖项目**：
  - 如果骑手分单时一个线程锁住订单表，另一个线程锁住骑手表，互相等待，也可能产生死锁。

------

## ⚠️ 易错点

- 以为加了超时就一定能避免死锁 → 如果没有释放已有资源，仍可能形成死锁。
- 用 `synchronized` 嵌套时未注意顺序，容易引发循环等待。

## 🔍 追问

1. **死锁和活锁的区别？**
    👉 死锁：线程永久阻塞；活锁：线程不断切换状态，但任务始终无法推进。
2. **死锁和饥饿的区别？**
    👉 死锁：互相等待；饥饿：低优先级线程长期得不到资源。
3. **如何用代码演示死锁？**
    👉 两个线程分别获取两个对象的锁，交叉持有再请求。
