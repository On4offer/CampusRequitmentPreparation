## 📌 为什么不建议自己创建线程，而是用线程池？

### 🔹 1. 核心结论

> **线程池能避免频繁创建/销毁线程的开销，统一管理线程资源，提升性能和稳定性，并且提供任务调度、限流和拒绝策略等能力。**

------

### 🔹 2. 原理细节

1. **性能开销大**
   - 频繁 `new Thread().start()` → 每次都要创建 OS 级线程，涉及用户态/内核态切换，开销很大。
   - 线程销毁 → 资源回收也有成本。
2. **缺乏统一管理**
   - 手动创建线程不受控，可能无限增加 → **OOM 或 CPU 上下文切换过多**。
   - 无法复用线程，导致资源浪费。
3. **无法提供高级功能**
   - 无法限制并发数（容易打爆数据库/Redis）。
   - 没有任务排队、优先级、拒绝策略等机制。
   - 无法做监控和统计（活跃线程数、队列长度）。
4. **线程池的优势**
   - **线程复用**：避免反复创建和销毁。
   - **任务调度**：支持队列、定时、周期任务。
   - **限流保护**：通过核心线程数、最大线程数、队列容量控制并发规模。
   - **统一管理**：集中管理线程生命周期，支持监控。

------

### 🔹 3. 项目实践

- **黑马点评**
  - 缓存重建 / 异步秒杀下单，如果直接创建线程处理 → 高并发时可能瞬间开启几千线程，直接 OOM。
  - 使用线程池（固定核心数 + 有界队列 + CallerRunsPolicy），既能限流保护 DB，又能避免线程暴增。
- **苍穹外卖**
  - 异步发短信 / 推送消息，如果每条短信都 `new Thread()`，在高峰期会造成线程爆炸。
  - 用线程池 → 保证短信任务排队执行，系统平稳运行。

------

### 🔹 4. 面试答题模板

> “不建议自己频繁创建线程，主要有三点原因：
>  ① 性能开销大：频繁创建和销毁线程成本高；
>  ② 不可控：线程数量无限增长，容易导致 OOM 或上下文切换过多；
>  ③ 功能不足：缺乏队列、限流、拒绝策略等机制。
>  使用线程池可以做到线程复用、集中管理、限流保护，并且提供监控和拒绝策略，是生产级别的最佳实践。”
