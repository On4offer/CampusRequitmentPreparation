## 📌 面试题：JVM 常见的垃圾回收器有哪些？使用场景分别是什么？

### ⭐ 标准回答

JVM 提供了多种垃圾回收器，主要分为 **新生代收集器、老年代收集器、整堆收集器**，常见的有：

------

### 1. **Serial 收集器**

- **特点**：单线程 GC，发生 GC 时 Stop-The-World。
- **优点**：实现简单，内存占用小。
- **缺点**：停顿时间长，不适合并发环境。
- **使用场景**：单核 CPU、小内存、客户端应用。

------

### 2. **ParNew 收集器**

- **特点**：Serial 的多线程版本，常用在新生代。
- **优点**：利用多核 CPU 并行 GC。
- **缺点**：在高并发下可能有锁竞争。
- **使用场景**：搭配 CMS 收集器使用，Web 服务常见。

------

### 3. **Parallel Scavenge 收集器（吞吐量优先）**

- **特点**：新生代多线程收集器，目标是高吞吐量（尽量减少 GC 时间占比）。
- **优点**：可以设置 `-XX:GCTimeRatio` 和 `-XX:MaxGCPauseMillis`。
- **缺点**：停顿时间不可预测。
- **使用场景**：后台任务、科学计算、批处理系统。

------

### 4. **CMS（Concurrent Mark-Sweep）收集器**

- **特点**：基于标记-清除，老年代并发回收，减少停顿。
- **优点**：低延迟，适合对响应时间要求高的应用。
- **缺点**：
  - 可能产生内存碎片。
  - 并发回收时 CPU 占用高。
  - 有 “Concurrent Mode Failure” 风险。
- **使用场景**：对响应速度敏感的 Web 系统（如电商）。

------

### 5. **G1（Garbage First）收集器**

- **特点**：分 Region 管理堆内存，混合使用标记-整理+复制算法。
- **优点**：
  - 可预测停顿时间（`-XX:MaxGCPauseMillis`）。
  - 避免内存碎片。
  - 更适合大堆（>4GB）。
- **缺点**：实现复杂，调优门槛高。
- **使用场景**：JDK9 之后的默认 GC，适合大内存、高并发服务。

------

### 6. **ZGC**

- **特点**：JDK11 引入的低延迟收集器，基于 Region + 并发标记整理。
- **优点**：
  - 停顿时间通常 **<10ms**。
  - 支持 TB 级别大堆。
- **缺点**：较新，参数调优复杂。
- **使用场景**：金融交易、广告系统等极低延迟场景。

------

### 7. **Shenandoah**

- **特点**：RedHat 推出，低延迟 GC。
- **优点**：停顿时间与堆大小无关。
- **缺点**：兼容性和成熟度略逊。
- **使用场景**：实时系统、响应延迟要求极高的应用。

------

### 📊 总结对比表

| 收集器            | 代别   | 算法           | 优点         | 缺点           | 适用场景         |
| ----------------- | ------ | -------------- | ------------ | -------------- | ---------------- |
| Serial            | 新生代 | 复制           | 简单高效     | 单线程，停顿长 | 单核、小内存     |
| ParNew            | 新生代 | 复制           | 多线程并行   | 吞吐量一般     | 搭配 CMS         |
| Parallel Scavenge | 新生代 | 复制           | 吞吐量优先   | 停顿不可控     | 批处理、后台任务 |
| CMS               | 老年代 | 标记-清除      | 并发、低延迟 | 碎片、失败风险 | Web 系统         |
| G1                | 整堆   | 标记-整理+复制 | 可预测停顿   | 调优复杂       | 大内存、高并发   |
| ZGC               | 整堆   | 并发标记-整理  | 超低延迟     | 新 GC，复杂    | 金融、低延迟     |
| Shenandoah        | 整堆   | 并发标记-整理  | 停顿与堆无关 | 成熟度不足     | 实时应用         |

------

### 🔍 面试追问

1. 为什么 JDK9 默认使用 G1？
    👉 G1 可预测停顿，适合大内存，避免 CMS 碎片问题。
2. CMS 为什么会发生 “Concurrent Mode Failure”？
    👉 老年代空间不足，导致并发清理失败，触发 Full GC。
3. 如何选择 GC 收集器？
    👉 响应时间优先 → CMS/G1/ZGC；吞吐量优先 → Parallel Scavenge。
