# 面试题：在高并发场景下，MySQL 如何进行性能优化？

------

## 1. 概念

- 高并发场景：电商秒杀、热点商品查询、支付系统等。
- 挑战：数据库成为瓶颈（QPS/TPS 限制、锁竞争严重、IO 压力大）。
- 目标：**提升吞吐量、降低延迟、避免数据库崩溃**。

------

## 2. 优化思路

### 🔹 1）SQL 与索引优化

- 使用 **合适的索引**（联合索引、覆盖索引、前缀索引）。
- 避免索引失效（`like '%xx'`、函数操作、隐式转换）。
- 只查必要字段，避免 `SELECT *`。

### 🔹 2）缓存与降级

- **本地缓存**（Caffeine、Guava）+ **分布式缓存**（Redis） → 抗高 QPS。
- 缓存策略：逻辑过期、异步更新、热点 key 预热。
- 缓存穿透/击穿/雪崩的防护措施（布隆过滤器、互斥锁、随机 TTL）。

### 🔹 3）读写分离

- 主库写、从库读，提升读性能。
- 配合 **Binlog → Relay Log → 从库同步**。

### 🔹 4）分库分表

- 水平分表：按用户 ID、订单 ID hash 拆分。
- 垂直分库：业务维度拆分。
- 避免单表过大（行数千万级别后性能急剧下降）。

### 🔹 5）连接池与事务优化

- 使用 **连接池（HikariCP）**，减少频繁建连。
- 事务尽量小，减少锁持有时间，避免长事务。

### 🔹 6）架构层优化

- **消息队列（MQ）削峰填谷**（RocketMQ / Kafka / Redis Stream）。
- **异步化**：下单写入消息队列，后台异步落库。
- **限流/降级**：保护数据库。

------

## 3. 案例

**电商秒杀**：

- 请求先打到 Redis，做库存预减。
- 订单写入消息队列，异步落库。
- MySQL 仅承受顺滑的写请求，避免直接被“锤爆”。

------

## 4. 使用场景

- **电商订单系统**：热点查询走缓存，写请求走队列。
- **支付场景**：事务小化 + 乐观锁，避免长事务。
- **日志/报表**：分库分表，冷热数据分离。

------

## 5. 面试模板回答

> 在高并发场景下，MySQL 的优化通常分为几个层面：
>
> - **SQL 与索引优化**（联合索引、覆盖索引，避免全表扫描）；
> - **缓存机制**（Redis 缓存热点数据，防止击穿/雪崩/穿透）；
> - **架构优化**（读写分离、分库分表、主从复制）；
> - **事务与连接池优化**（减少锁竞争，缩短事务时间）；
> - **异步化与削峰填谷**（MQ 处理秒杀/突发流量）。
>    实际项目中会综合应用，例如电商秒杀场景一般是 “Redis 预减库存 + MQ 异步落库 + MySQL 分表存储”。
