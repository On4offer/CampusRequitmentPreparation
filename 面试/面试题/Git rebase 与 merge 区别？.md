# 面试题：Git rebase 和 merge 的区别？

### 1. 概念解释

- **merge**：把两个分支的最新提交合并在一起，保留分支合并的历史记录，生成一个新的“合并提交（merge commit）”。
- **rebase**：将一个分支的提交“重新应用”到另一个分支的最新提交上，相当于“变基”，会重写提交历史，使历史记录更直线化。

------

### 2. 核心区别（对比分析）

| 对比点       | `merge`                      | `rebase`                       |
| ------------ | ---------------------------- | ------------------------------ |
| **历史记录** | 保留分支合并点，历史有分叉   | 重写提交历史，历史干净、直线   |
| **提交ID**   | 不会改变已有提交的 commit ID | 会改变提交的 commit ID         |
| **操作结果** | 合并后生成新的 merge commit  | 把分支提交依次放到目标分支之后 |
| **安全性**   | 相对安全，不会破坏历史       | 改写历史，不适合多人共享分支   |
| **常用场景** | 保留分支开发痕迹             | 让提交历史简洁、直观           |

------

### 3. 示例说明

假设有两个分支：

- `main`: A — B — C
- `feature`: D — E

#### 使用 `merge`

```
git checkout main
git merge feature
```

结果：

```
A — B — C —— M(merge)
          \ 
           D — E
```

> 历史保留了分叉点和合并提交 M。

#### 使用 `rebase`

```
git checkout feature
git rebase main
```

结果：

```
A — B — C — D’ — E’
```

> `feature` 的提交被“挪”到 `main` 之后，看起来像是一直在 `main` 上开发。

------

### 4. 案例说明（结合项目）

在 **黑马点评项目**：

- 如果团队开发一个新功能（如“店铺缓存优化”），开发者在 `feature/cache` 分支上工作：
  - 用 **merge** 合并到 `main`，可以保留团队协作的痕迹，方便回溯问题。
  - 用 **rebase**，可以让提交历史直观清晰，适合个人开发者整理自己的分支再提交 PR。

------

### 5. 使用场景

- **merge**：适合团队协作，保证提交历史完整，不会影响其他人。
- **rebase**：适合个人分支整理历史，让代码提交看起来更干净，通常在合并到公共分支之前使用。

------

### 6. 面试标准回答模板

> Git 的 `merge` 和 `rebase` 都是用来整合分支的。
>  **merge** 会保留分支历史，生成一个新的合并提交，安全但历史可能比较复杂；
>  **rebase** 会把分支的提交“搬到”目标分支上，历史更直观干净，但会改写提交 ID，不适合多人共享分支。
>  在实际开发中，我一般在 **本地开发分支用 rebase 整理历史**，在 **合并到主分支时用 merge**，这样既保证了历史清晰，也保证了团队协作安全。
