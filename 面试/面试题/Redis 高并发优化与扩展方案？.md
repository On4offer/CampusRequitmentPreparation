# 面试题

**在高并发场景下，Redis 如何避免成为瓶颈？**

------

## 1️⃣ 概念

- **Redis 本身是单线程处理命令**（网络 I/O + 命令执行），在高并发场景下，如果不合理使用，可能成为系统瓶颈。
- 瓶颈常见表现：QPS 降低、延迟升高、CPU 或网络 IO 占满、热点 Key 争抢严重。

------

## 2️⃣ 优化方向

### （1）架构层面

1. **读写分离**
   - 部署主从架构，主库负责写，从库负责读，分担读压力。
2. **分片集群（Cluster）**
   - 使用哈希槽分布数据，避免单点性能瓶颈。
3. **多级缓存**
   - JVM 本地缓存（Caffeine、Guava Cache）+ Redis 二级缓存，减少 Redis 压力。
4. **热点 Key 分散**
   - 大 Key 拆分（如排行榜拆分为多个区段）。
   - 热点 Key 分片（如 hash tag 分布式锁）。

------

### （2）命令与数据结构优化

1. **避免大 Key / 慎用阻塞命令**
   - 如 `KEYS`、`HGETALL`、`SMEMBERS`，应替换为 `SCAN`、分页查询。
   - 避免存储大 List/Hash/Set（一次取出耗时长）。
2. **合理设置过期时间**
   - 防止大量 Key 同时过期导致雪崩。
3. **管道（Pipeline）批量执行**
   - 一次网络请求中发送多条命令，减少 RTT。
4. **Lua 脚本**
   - 将多条操作合并到服务器端执行，避免多次网络往返。

------

### （3）运维层面

1. **持久化优化**
   - 关闭 AOF 或设置 `everysec`，避免频繁刷盘影响性能。
2. **内存管理**
   - 合理设置淘汰策略（allkeys-lru/lfu），避免频繁 OOM。
3. **网络优化**
   - 使用长连接（连接池），减少连接建立开销。
4. **监控与限流**
   - 监控热点 Key/QPS，必要时限流或降级。

------

## 3️⃣ 项目实践（黑马点评 / 苍穹外卖）

- **黑马点评**：
  - 热点店铺详情缓存 → 使用逻辑过期 + 分布式锁，避免击穿。
  - UV 统计用 HyperLogLog，减少存储空间，避免大 Key 压力。
- **苍穹外卖**：
  - 秒杀库存扣减用 Lua 脚本，保证原子性，避免多次往返。
  - 高峰期下单请求，通过 Redis 主从读写分离 + 分片集群支撑高并发。

------

## 4️⃣ 面试标准回答模板

> Redis 在高并发场景下可能成为瓶颈，我一般从三方面优化：
>
> - **架构层面**：主从读写分离、Cluster 分片、多级缓存，避免单点瓶颈；
> - **命令层面**：避免大 Key 和阻塞命令，使用 Pipeline、Lua 脚本减少网络开销；
> - **运维层面**：优化持久化、合理设置淘汰策略，使用连接池、限流和降级。
>    在项目中，比如黑马点评的店铺缓存，我用了逻辑过期防击穿，UV 统计用 HyperLogLog 降低内存开销；苍穹外卖的秒杀场景，用 Lua 脚本+主从架构保证高并发下的性能。
