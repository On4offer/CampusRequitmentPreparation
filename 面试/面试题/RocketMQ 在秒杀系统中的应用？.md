## ❓ 在高并发秒杀场景中，RocketMQ 如何防止超卖？

### 1. 概念

**超卖问题**：指在高并发秒杀中，多个用户同时下单，库存未及时扣减或并发不安全，导致实际售出数量 > 库存数量。
 RocketMQ 作为消息中间件，可以帮助削峰填谷、异步下单，但要结合缓存和锁机制防止超卖。

------

### 2. 超卖产生的原因

- **并发写数据库**：多个请求同时查询库存，均判断“有库存”，并发扣减导致超卖。
- **缓存与数据库不一致**：缓存未及时更新，多个请求同时命中旧库存值。
- **数据库行锁竞争激烈**：导致并发控制失效或超时重试。

------

### 3. RocketMQ 引入后的解决方案

1. **削峰填谷**
   - 用户请求先写入 MQ，不直接打数据库，避免瞬时高并发压垮 DB。
2. **顺序消息 / 单队列串行消费**
   - 将同一商品的秒杀消息路由到同一个队列，按顺序逐条消费。
   - 确保同一时间只有一个线程修改库存，避免并发写冲突。
3. **库存预减（Redis） + 异步落库**
   - 下单时先在 Redis 进行库存预扣减。
   - 再通过 RocketMQ 异步写订单，最终落库。
   - Redis + MQ 保证高并发下的性能和一致性。
4. **幂等性保证**
   - 每个用户+商品生成唯一订单号。
   - 消费端处理时先判断是否已有订单，避免重复下单。

------

### 4. 案例

在 **黑马点评/苍穹外卖秒杀项目**：

- 用户秒杀请求 → 进入 RocketMQ → 投递到“秒杀队列”。
- 该队列按 **商品 ID 进行路由**，同一商品的请求进入同一队列，消费者 **串行扣减库存**。
- 扣减库存成功 → 生成订单 → 返回结果；库存不足 → 直接丢弃消息。
- 同时结合 **Redis 预减库存**，大部分用户在入口处就会被拦截，减少无效请求进入 MQ。

------

### 5. 面试标准回答（背诵版）

“在秒杀场景下，RocketMQ 主要通过 **削峰填谷** 和 **顺序消息** 来防止超卖。请求先进入 MQ，避免数据库被瞬间冲击；同一商品的秒杀请求路由到同一队列，由消费者串行处理，保证库存扣减操作有序。同时结合 Redis 预减库存和幂等性保证，可以避免并发写库和重复下单，实现高并发下的安全秒杀。”
