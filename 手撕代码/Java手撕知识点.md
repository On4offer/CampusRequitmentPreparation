# Java 手撕知识点（对应题目原理）

> 本文件整理手撕题背后的**原理与考点**，便于理解「为什么这样写」、面试时能讲清。

---

## 一、单例模式 → 对应「单例 DCL / 饿汉 / 静态内部类」

### 1. 为什么需要 volatile（DCL）

`instance = new Singleton()` 在 JVM 中并非原子指令，大致分三步：

1. 分配内存
2. 初始化对象（调用构造方法）
3. 将引用指向内存

**指令重排**：JIT 可能把 2、3 调换顺序，导致「引用已赋值但对象未初始化」。此时另一线程拿到非 null 的 instance 去用，会得到半初始化对象。

**volatile 作用**：禁止重排，保证「写 happens-before 读」，读到引用时对象一定已完成初始化。

### 2. 为什么两次 if (instance == null)

- **第一次**：大部分情况下 instance 已创建，直接返回，避免每次进入 synchronized，减少开销。
- **第二次**：在锁内，防止多线程同时通过第一次检查后，重复创建实例。

### 3. 静态内部类为什么是懒加载的

- 类加载时机：`Holder` 只有在被调用时才会加载（如 `getInstance()` 里访问 `Holder.INSTANCE`）。
- 类加载过程由 JVM 保证线程安全，`Holder.INSTANCE` 的初始化只会执行一次。

### 4. 饿汉式线程安全吗

类加载时 `INSTANCE` 由 JVM 完成初始化，天然线程安全，但无法懒加载。

## 二、生产者-消费者（阻塞队列）→ 对应「BlockingQueue + Lock/Condition」

### 1. 为什么用 while 而不是 if 做 await 判断

**虚假唤醒**：`await()` 被唤醒时，条件可能仍不满足（如被别的线程抢先消费了）。用 `while` 会再次检查，不满足则继续 await；用 `if` 可能直接往下执行，导致越界或空指针。

### 2. Condition 与 wait/notify 的区别

| 对比 | 一个锁 | 多个条件 |
|------|--------|----------|
| wait/notify | 一个 Object 锁，只能有一个等待队列 | 无法区分「满」和「空」 |
| Condition | 一个 Lock 可配多个 Condition | notFull、notEmpty 分开，唤醒更精确 |

### 3. signal 和 signalAll

- `signal()`：只唤醒一个等待线程，效率高，但可能唤醒「错误」的线程（如满时唤醒生产者）。
- `signalAll()`：唤醒全部，被唤醒的线程会重新竞争锁，再通过 while 检查条件；实现简单，通常用 `signalAll` 更安全。

## 三、LRU → 对应「手写 LRU」

### 1. 为什么需要 HashMap + 双向链表

- **HashMap**：O(1) 按 key 查找节点。
- **双向链表**：O(1) 删除、插入、移动节点（已知前后节点）。

单链表无法 O(1) 删除当前节点（需要前驱）；只用 HashMap 无法维护访问顺序。

### 2. 为什么 LinkedHashMap 的 accessOrder=true 能实现 LRU

- `accessOrder=false`：按插入顺序。
- `accessOrder=true`：每次 `get` 会把节点移到队尾。

重写 `removeEldestEntry`，当 `size > capacity` 时返回 true，会删除最老的节点（队头）。

### 3. 淘汰策略

- LRU：最近最少使用，删除最久未访问的。
- LFU：最不经常使用，按访问频率淘汰。
- FIFO：先进先出。

## 四、交替打印 / 顺序打印 → 对应「两线程交替、三线程 ABC」

### 1. 为什么用 while 而不是 if 判断唤醒条件

同「生产者-消费者」：被唤醒后条件可能仍不满足（如轮到 B 打印，却唤醒了 A），用 `while` 会再次检查，不满足则继续 await。

### 2. 为什么用 state 变量控制顺序

用共享变量标记「当前该谁执行」，每个线程只在自己的 state 时执行，否则 await，执行完后更新 state 并 signal 下一个线程。这样可扩展到 N 个线程顺序执行。

### 3. 为什么用 Condition 而不是一个 wait

多个 Condition（condA、condB、condC）可以精确唤醒对应线程，避免「唤醒所有人再竞争」带来的无效唤醒。

## 五、ArrayList / HashMap → 对应「手写 ArrayList、HashMap」

### 1. ArrayList 扩容

- 默认容量 10，首次 add 时若不足则扩容。
- 新容量 = `oldCapacity + (oldCapacity >> 1)`，即 1.5 倍。
- 使用 `Arrays.copyOf` 复制到新数组。

### 2. HashMap 结构

- 数组 + 链表（红黑树）：`table.length` 为 2 的幂，`index = (n-1) & hash`。
- 哈希：`hash = key.hashCode() ^ (hashCode >>> 16)`，高 16 位参与运算，减少碰撞。
- 链表长度 ≥ 8 且 数组长度 ≥ 64 时转红黑树；树节点 ≤ 6 时退化为链表。

### 3. 负载因子 0.75

- 太小：浪费空间，扩容频繁。
- 太大：链表/树变长，查询变慢。
- 0.75 是空间与时间的折中。

### 4. 为什么容量是 2 的幂

- `index = (n-1) & hash` 等价于 `hash % n`，但位运算更快。
- 扩容时，旧节点要么在原位置，要么在「原位置 + oldCap」，只需判断 `(hash & oldCap) == 0`。

## 六、死锁 → 对应「死锁示例」

### 1. 死锁四个必要条件

- 互斥：资源只能被一个线程持有。
- 占有且等待：持有资源的同时等待其他资源。
- 不可抢占：资源不能被强制剥夺。
- 循环等待：存在 A→B→C→…→A 的等待链。

### 2. 如何避免

- 统一加锁顺序：所有线程按相同顺序获取锁（如先 A 后 B）。
- 超时：`tryLock(timeout)`，超时则释放已持有的锁并重试。
- 死锁检测：定期检测并中断/回滚。

### 3. 如何排查

- `jstack <pid>` 查看线程栈，搜索 `deadlock` 或 `BLOCKED` 的互相等待。
- JConsole 可检测死锁。

## 七、wait / notify → 对应「wait/notify 生产者消费者」

### 1. 为什么必须在 synchronized 块内调用

- `wait()` 会释放锁，被唤醒后需要重新竞争锁才能继续。
- 若不在同步块内，没有「当前持有锁」的前提，无法正确释放和唤醒。

### 2. wait 和 sleep 的区别

| 对比 | wait | sleep |
|------|------|-------|
| 所属 | Object | Thread |
| 释放锁 | 释放 | 不释放 |
| 唤醒 | notify/notifyAll | 超时或 interrupt |
| 使用场景 | 条件等待 | 单纯延时 |

### 3. 为什么推荐用 Lock + Condition 而不是 wait/notify

- 支持多个条件、可中断、可超时、公平/非公平。
- API 更清晰，`lock()` / `unlock()` 显式控制。

## 八、线程池 → 对应「手写简单线程池」

### 1. 核心参数

- 核心线程数：常驻线程，即使空闲也不回收（除非 `allowCoreThreadTimeOut`）。
- 最大线程数：任务队列满时，可创建到最大线程数。
- 存活时间：非核心线程空闲多久后被回收。
- 工作队列：任务排队，如 `ArrayBlockingQueue`、`LinkedBlockingQueue`。
- 拒绝策略：队列满且线程满时的处理（Abort、CallerRuns、Discard、DiscardOldest）。

### 2. 执行流程

1. 线程数 < 核心数 → 新建核心线程执行。
2. 否则 → 任务入队。
3. 队列满且线程数 < 最大数 → 新建非核心线程。
4. 否则 → 执行拒绝策略。

### 3. 为什么用 BlockingQueue

- `take()` 阻塞：队列空时 worker 阻塞等待，不占用 CPU。
- `put()` 阻塞：队列满时 submit 可阻塞（取决于队列类型）。

## 九、限流器 → 对应「令牌桶、滑动窗口」

### 1. 令牌桶 vs 漏桶

| 对比 | 令牌桶 | 漏桶 |
|------|--------|------|
| 思想 | 固定速率放令牌，取到令牌才通过 | 固定速率流出，请求先入桶 |
| 突发 | 允许突发（桶内有令牌时） | 平滑输出，不允许突发 |
| 典型 | Guava RateLimiter、网关限流 | 消息队列削峰 |

### 2. 滑动窗口 vs 固定窗口

- 固定窗口：如 [0,1s) 内最多 100 次，可能跨边界突发（如 0.9s 和 1.1s 各 100 次）。
- 滑动窗口：按当前时间往前推窗口，更平滑，但实现更复杂。

### 3. 限流算法选择

- 简单计数：固定窗口，实现简单，有边界问题。
- 令牌桶：允许突发，适合 API 限流。
- 漏桶：严格平滑，适合消息队列。

## 十、带过期缓存 → 对应「ExpireCache + DelayQueue」

### 1. DelayQueue 原理

- 内部用 `PriorityQueue` 按过期时间排序，队首是最早过期的。
- `take()` 会阻塞直到队首元素过期，返回该元素。
- 用于定时任务、过期清理等场景。

### 2. 为什么用 ConcurrentHashMap + DelayQueue

- `ConcurrentHashMap`：并发安全的 key-value 存储。
- `DelayQueue`：按过期时间有序，清理线程只需 `take()` 即可拿到过期元素。

### 3. 其他实现思路

- 惰性删除：get 时检查过期，过期则删除并返回 null。
- 定时扫描：后台线程定期扫描过期 key 删除。
- 红黑树：按过期时间排序，定期取最小过期 key 删除。

## 十一、公平锁 vs 非公平锁 → 对应「ReentrantLock(true/false)」

### 1. 非公平锁

- `lock()` 时先 CAS 抢一次，抢到直接执行，不抢到再入队。
- 吞吐高：刚释放锁的线程可能再次抢到，减少上下文切换。
- 可能饥饿：队列中的线程可能一直等不到。

### 2. 公平锁

- 严格按 FIFO 顺序，新来的线程直接入队，不抢。
- 避免饥饿，但吞吐一般更低（更多上下文切换）。

### 3. AQS 简要

- `ReentrantLock` 内部用 AQS（AbstractQueuedSynchronizer）实现。
- 状态变量 state：0 表示未占用，1 表示持有者重入次数。
- 等待队列：CAS 失败后入队，前驱节点 release 时唤醒后继。
