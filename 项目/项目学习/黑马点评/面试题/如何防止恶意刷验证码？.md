# 面试题：如何防止恶意刷验证码？

## 1. 场景&风险（先定性）

- **风险类型**：撞库/撞码、批量注册、短信轰炸（攻击别人手机号）、羊毛党薅券、接口压测拖垮网关/短信通道。
- **目标**：在不伤害真实用户体验的前提下，**限流 + 风控 + 溯源 + 审计**闭环。

------

## 2. 总体策略（八字诀）

> **前置验证、一次一用、频率受控、错误惩罚、绑定场景、溯源审计、灰黑名单、联动封禁**

------

## 3. 设计方案（分层防护）

### A. 前置校验（挡掉大部分“白嫖请求”）

1. **图形/滑块验证码**：首次/异常频次时先过人机校验；失败不落库、不发短信。
2. **签名与时间戳**：`appId + ts + nonce + HMAC`，**防重放**（`nonce`入库短 TTL），过期/验签失败直接拒绝。
3. **基本校验**：手机号格式、业务白名单渠道、Referer/UA 合规、跨域限制。

### B. 发送频控（多维度+滑动窗口）

- **手机号维度**：60s 1次，10分钟≤N次，24小时≤M次。
- **IP/设备维度**：10分钟≤K次，超过阈值触发图形验证码/限流。
- **接口级熔断**：短信通道异常或系统高压时，整体降级（提示“系统繁忙”）。

**示例（手机号 60s 1次）**

```java
String key = "login:sms:send:" + phone;
Boolean ok = redis.opsForValue().setIfAbsent(key, "1", 60, TimeUnit.SECONDS);
if (Boolean.FALSE.equals(ok)) return fail("发送频繁，请稍后再试");
```

**示例（IP 滑动窗口 / ZSET）**

```java
String zkey = "ratelimit:sms:ip:" + ip;
long now = System.currentTimeMillis();
redis.opsForZSet().add(zkey, String.valueOf(now), now);
redis.opsForZSet().removeRangeByScore(zkey, 0, now - 10*60_000);
Long cnt = redis.opsForZSet().zCard(zkey);
redis.expire(zkey, 15, TimeUnit.MINUTES);
if (cnt != null && cnt > 20) return fail("请求过多，请稍后再试");
```

### C. 验证码“一次一用”与**原子校验**

- **短 TTL**（如 2 分钟），**一次性**：校验成功**原子删除**，防重放。
- 可与**场景绑定**（`login:code:{scene}:{phone}`）。

**Lua 原子 GETDEL**

```java
DefaultRedisScript<String> GETDEL = new DefaultRedisScript<>(
 "local v=redis.call('GET',KEYS[1]); if v then redis.call('DEL',KEYS[1]); end; return v;", String.class);
String cache = redis.execute(GETDEL, Collections.singletonList("login:code:"+scene+":"+phone));
if (cache == null || !cache.equals(code)) return fail("验证码错误或已失效");
```

### D. 错误次数惩罚（防撞码/穷举）

- 维护 `login:code:err:{phone}`，**5分钟内≥N次**：临时冻结 10~30 分钟。
- 冻结期间继续请求 → 强制走图形验证码/直接拒绝。

**示例**

```java
String ek = "login:code:err:" + phone;
Long c = redis.opsForValue().increment(ek, 1);
if (c == 1) redis.expire(ek, 5, TimeUnit.MINUTES);
if (c != null && c > 5) return fail("尝试过多，请稍后再试");
```

### E. 业务绑定 & 最小必要信息

- 验证码与**场景**/渠道绑定：注册、登录、改密、支付校验**互不通用**。
- 下发短信模板**不回显完整验证码**于日志；线上日志脱敏。

### F. 账号/设备画像 & 灰黑名单

- 维护 `risk:gray:{ip/device}`，分数超阈值 → 要求前置人机校验；
- 命中**黑名单**（代理池、恶意号段、异常设备指纹）→ 直接拒绝并告警。

### G. 供应商侧防护联动

- 用**金融级短信通道**支持**回执**，异常号段/大量退订/投诉自动降配；
- 失败重试需**幂等**，避免“雪上加霜”的二次轰炸。

### H. 审计与告警

- 指标：发送量/成功率/唯一手机号数、IP 分布、图形验证码命中率、错误次数分布、黑名单触发量；
- **阈值报警**：单 IP/单设备/单号段短时激增，自动拉高防护等级（策略开关走配置中心）。

------

## 4. “黑马点评”可落地最小闭环（够用且易实现）

1. 首次/异常频率 → **滑块验证码**通过后才允许发短信；
2. **手机号 60s 1次** + **10分钟≤3次**；**IP 10分钟≤20次**（ZSET 窗口）；
3. **验证码 2 分钟有效**，**Lua 原子 GETDEL**；
4. **错误 5 次**锁 15 分钟；
5. 日志脱敏、指标埋点、可配置阈值（Nacos）；
6. 后台支持手机号、IP、设备指纹**临时加黑**。

------

## 5. 标准回答（30 秒版）

> 我们对短信验证码做了**多维度风控**：
>  ① **前置人机校验** + 请求签名/时间戳/nonce 防重放；
>  ② **手机号/ IP /设备**多维限频，IP 用 **ZSET 滑动窗口**；
>  ③ 验证码**短 TTL + 一次一用**，用 **Lua GETDEL** 原子校验防重放；
>  ④ **错误次数惩罚**与临时冻结；
>  ⑤ 与业务**场景绑定**，日志脱敏并全量埋点；
>  ⑥ 灰黑名单与供应商告警联动。
>  这样能在不影响正常用户体验的前提下，有效抵御刷码、轰炸与撞库。

------

## 6. 面试延伸追问

- **为什么要用 Lua 原子 GETDEL？**（避免并发下“读到旧码又被复用”）
- **如果攻击者换 IP 怎么办？**（设备指纹/账号画像/号段库/代理池识别+联动图形验证码）
- **高并发下 Redis 写放大怎么控？**（限频键过期+窗口清理；图形验证码门槛降低总体请求量）
- **如何不打扰真实用户？**（分级策略：先人机→再短信；按风险分层，不“一刀切”）
