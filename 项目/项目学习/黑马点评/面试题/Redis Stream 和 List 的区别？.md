## 🎯 标准面试回答

**Redis List 和 Stream 的区别主要有以下几点：**

1. **数据结构不同**
   - **List**：简单的链表结构（先进先出），常用 `LPUSH` / `RPOP` 模拟队列。
   - **Stream**：Redis 5.0 新增的专用消息队列结构，底层是日志型存储，支持消息 ID、有序存储。
2. **消费模型**
   - **List**：只能 **点对点消费**，谁取走消息，其他人就拿不到。
   - **Stream**：支持 **消费组（Consumer Group）**，一个消息可以分发给多个消费者，并行消费。
3. **可靠性**
   - **List**：消息取出来就没了，如果消费者挂了，消息会丢失。
   - **Stream**：有 **确认机制（XACK）**，未确认的消息会进入 **Pending List**，可重试，保证消息不丢。
4. **持久化能力**
   - **List**：只是普通 Redis 数据结构，消息不会自动留存。
   - **Stream**：带有 **日志存储 + 消息 ID**，可以回溯历史消息，甚至用来做消息追踪。
5. **适用场景**
   - **List**：简单异步任务、延迟不敏感的小功能（如邮件通知队列）。
   - **Stream**：高并发、可靠性要求高的消息场景（如黑马点评秒杀下单）。

------

## ✅ 面试扩展追问

- **Q1：为什么不用 List？**
   → 因为 List 没有消费确认和 Pending List，消费者挂了就丢消息，不满足秒杀场景下的可靠性需求。
- **Q2：Stream 和专业 MQ 比呢？**
   → Stream 更轻量，适合中小型场景；Kafka/RabbitMQ 功能更丰富，适合分布式事务、大规模日志处理。

------

## 📌 面试模板总结

> Redis **List** 只是个简单队列，没有确认和重试机制，适合轻量级异步任务；
>  Redis **Stream** 才是真正的消息队列，支持 **消费组、确认机制、Pending List**，能保证消息不丢失、不重复，是黑马点评秒杀场景的最佳选择。
