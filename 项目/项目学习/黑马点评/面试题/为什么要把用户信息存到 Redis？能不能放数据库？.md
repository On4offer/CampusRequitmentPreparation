下面按“面试题风格 + 黑马点评项目”来答：

# 题干

**为什么把用户信息放到 Redis？能不能直接放数据库（MySQL）？**

------

# 一、核心结论（起手式30秒）

- 在黑马点评，**放 Redis 不是替代数据库**，而是为了**会话/登录态**与**高频读**的**性能与可用性**：低延迟、自动过期、横向扩展、支持集群无状态。
- **用户主数据**（注册资料、手机号、头像等）依然在 **MySQL**；**Redis 只放登录态/高频字段**（UserDTO）与少量热点衍生信息。

------

# 二、为什么选 Redis（按面试高频点逐条打）

1. **性能**：内存读写，**亚毫秒延迟**，单机可达十万级 QPS；MySQL 每次鉴权都查库，RT↑、连接池压↑。
2. **自动过期 TTL**：会话/验证码/临时令牌天然有时效，用 Redis 设置 `EX/TTL` 一键搞定，MySQL 需要额外清理任务。
3. **无状态集群**：把登录态放 Redis，应用实例**无会话粘滞**，随便水平扩容；如果放本机内存或MySQL，很难既快又无状态。
4. **数据结构友好**：`HASH/SET/ZSET` + 原子操作（Lua、INCR、EXPIRE）满足**滑动过期**、**黑名单/踢人**、**并发计数**等场景。
5. **隔离读写压力**：把高频鉴权读流量**挡在 Redis**，保护 MySQL 做“真源数据”的增删改查。

------

# 三、能不能放数据库？可以，但不优（说清取舍）

- **能**：小流量后台、管理端单体应用，用 MySQL 存会话表也能跑。
- **不优**：
  - **时效管理麻烦**：过期清理依赖定时任务；
  - **吞吐瓶颈**：鉴权每请求一次查库，连接池和索引压力大；
  - **一致性/扩展性差**：多机房/多实例下共享会话不如 Redis 简洁。

> 面试金句：**MySQL 是事实存储，Redis 是会话与热点加速层**；二者职责不同，不是二选一。

------

# 四、项目落地（黑马点评登录链路）

- **登录**：
  1. 验证码校验通过 → 生成 **UUID token**；
  2. `HMSET login:token:<uuid> -> {id,nick,icon,...}` + `EX=30min`（或业务TTL）；
  3. token 返回给前端（放 `Authorization` 头）。
- **鉴权拦截**：
  - `RefreshTokenInterceptor` 读取 token → `HGETALL login:token:<uuid>` 命中 → 放入 `ThreadLocal`；
  - **滑动过期**：命中则 `EXPIRE +30min`，持续活跃就不断续命；
  - 未命中走匿名/401。
- **用户主档**：仍在 MySQL 的 `tb_user`；当用户资料改动（昵称/头像）时：**写库 → 删 Redis 登录态或合并更新**。

------

# 五、关键设计 & 细节（面试可展开）

- **Key 设计**：`login:token:{uuid}`，Hash 存 **UserDTO（最小必要集）**，避免敏感信息（手机号可脱敏或不放）。
- **TTL 策略**：登录态 30min~2h；验证码 2~5min；可按活跃度续期（滑动过期）。
- **并发安全**：Redis 原子命令/Lua；必要时“单用户单端登录”可用 `SETNX login:user:{uid}`。
- **登出/踢下线**：`DEL login:token:{uuid}` 或加入 `deny:token` 集合做黑名单。
- **监控**：命中率、鉴权耗时、key 数量、内存占用、过期率。

------

# 六、风险与对策（面试官会问坑）

- **一致性**：用户资料更新后 Redis 里还是旧值？
  - 策略：**写库 → 删登录态Key**（下次用新的Token重新登录）或合并更新 Hash；
- **雪崩/穿透**：
  - 策略：合理 TTL + 随机抖动、空值缓存、布隆过滤器（针对用户画像类热点）。
- **Redis 故障**：
  - 策略：部署主从/哨兵/集群，鉴权接口降级（短时允许旧态/本地缓存兜底或提示重登）。

------

# 七、标准回答模板（90 秒背诵）

> 我们把**用户登录态与高频字段**放 Redis，原因是**低延迟、高吞吐、自动过期、天然支持无状态集群**，可以把鉴权这类高频读压在缓存层，保护 MySQL 作为用户主数据的事实来源。
>  用户主档仍在 MySQL，Redis 只存 **UserDTO** 和会话相关信息，并设置 TTL/滑动过期。
>  如果只放数据库也能做，但会带来 RT 上升、连接池压力、过期清理复杂等问题。
>  资料更新时我们**写库再删/更新 Redis**，确保最终一致；Redis 故障有主从/哨兵保障并做合理降级。
>  综上，**Redis 是会话/热点加速层，MySQL 是持久真源**，二者分工明确。

------

# 八、扩展追问（帮你准备二面）

1. **JWT vs Redis 会话**：为何黑马点评选择“UUID+Redis”而不是全 JWT？（答：易控会话、可踢人、可续期、可失效；JWT 适合无状态但不好做强制失效/滑动过期）
2. **滑动过期如何实现？**（答：拦截器命中即 `EXPIRE` 续期；失败不续）
3. **多端登录如何限制？**（答：`SET user_session:{uid} -> token`，新登录替换旧token并 `DEL` 旧Key）
4. **哪些字段不应该放 Redis？**（答：敏感/隐私、低频/强一致字段不放；放最小必要集）
