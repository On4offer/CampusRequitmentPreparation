# 面试题：黑马点评项目为什么用 UUID 作为 Token，而不是 JWT？

## 1. 背景

- 在黑马点评项目中，用户登录成功后需要下发一个 **Token**，用于后续请求的身份校验。
- 常见的 Token 实现方式有两种：
  - **UUID（随机字符串）**
  - **JWT（JSON Web Token）**

------

## 2. UUID 方案

- **原理**：登录成功后生成一个随机的 UUID 作为 Token，写入 Redis（`token → 用户信息`），并设置 TTL。
- **使用方式**：客户端请求时带上这个 Token，服务端通过 Redis 查询用户信息完成认证。

------

## 3. JWT 方案

- **原理**：将用户信息（如 userId、过期时间等）加密签名，生成一个自包含的 Token，直接返回给客户端。
- **使用方式**：客户端请求时带上 JWT，服务端通过解析 Token 校验签名并读取用户信息。

------

## 4. 为什么选择 UUID 而不是 JWT？

### （1）**用户信息变更的问题**

- JWT 自包含，一旦下发给客户端，就无法单独失效。
  - 如果用户改了头像、昵称，或者账号被封禁，JWT 内的数据仍然是旧的。
  - 只能等到 JWT 过期才会更新。
- UUID 存在 Redis 中，Redis 里的用户数据可以实时更新，保证一致性。

### （2）**Token 主动失效**

- JWT 无法“强制下线”用户，因为已经发给客户端，除非服务端维护黑名单。
- UUID 存在 Redis，可以直接删除对应的 key，让用户立即失效。

### （3）**系统复杂度**

- JWT 需要配置秘钥、加密解密算法，解析时也要消耗 CPU。
- UUID 只依赖 Redis 存取，逻辑更简单。

### （4）**项目体量与场景**

- 黑马点评属于中小型项目，用户量有限，完全可以依赖 Redis 做状态存储。
- 如果是大型分布式系统、微服务架构，JWT 去中心化的优势才更明显。

------

## 5. 结论（标准回答）

> 在黑马点评项目中，选择 **UUID 作为 Token**，是因为它依赖 Redis 存储用户会话，可以保证用户信息的实时性和一致性，支持主动下线与信息更新，逻辑简单，性能损耗小。而 **JWT 虽然去中心化、无状态，但难以处理用户信息变更与主动失效的问题**，对本项目来说并不合适。
