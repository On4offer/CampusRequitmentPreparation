这个问题考察的是 **消费端幂等性设计**。RocketMQ 本身不能保证“消息只消费一次”，只能保证“至少一次投递”。所以我们必须从 **业务逻辑** 层面来设计。下面是详细回答：

------

## 🎯 面试追问：RocketMQ 如何避免重复下单？

### 1. **问题背景**

- RocketMQ 的消息投递语义是 **At-Least-Once**，也就是“至少一次”。
- 在以下场景可能导致重复消费：
  - 消费者消费成功，但 ACK 确认前宕机，Broker 认为消费失败，消息会被重新投递。
  - 网络抖动、超时，导致 Broker 没收到 ACK，也会重复投递。

👉 所以“重复下单”不是 MQ 的 bug，而是设计原则。要靠消费端业务幂等来保证。

------

### 2. **解决思路：消费端幂等**

常见做法：

#### （1）数据库唯一约束

- 在订单表上加唯一索引 `(user_id, voucher_id)`。
- 即使同一个用户的同一张优惠券消息被重复投递，数据库插入时会报唯一约束冲突，从而避免重复下单。
- 代码上要捕获异常，判定为重复消费直接忽略。

#### （2）业务侧去重（Redis / 分布式锁）

- 在消费前，先用 Redis 的 `SETNX`（或 Redisson 分布式锁）做去重。
- Key 可以设计成 `"order:userId:voucherId"`，存在即表示已处理过。
- 避免并发情况下多次插入。

#### （3）消息唯一 ID 去重表

- 每条消息都有唯一 MessageId 或自定义业务 ID（比如 `userId+voucherId`）。
- 消费端维护一张“去重表”或 Redis Set，记录已处理过的 ID。
- 新消息先检查是否存在，存在则直接丢弃。

------

### 3. **秒杀场景结合 RocketMQ**

在黑马点评秒杀优化里：

- 用户请求先通过 Redis 预扣库存，保证不会超卖。
- 异步下单消息进入 RocketMQ。
- Consumer 消费时：
  1. 先检查数据库唯一索引（`userId+voucherId`），避免重复下单。
  2. 如果重复消费，直接捕获异常，忽略该消息。
  3. RocketMQ 仍然认为消费成功（ACK），不会再重试。

------

### 4. **总结一句话**

> RocketMQ 本身保证“消息至少投递一次”，不能避免重复；
>  我们需要在 **消费端实现幂等性**，典型做法是：数据库唯一约束、Redis 去重、或者基于消息唯一 ID 的幂等校验，从而避免重复下单。

------

### 📌 标准面试回答（30 秒版）

> RocketMQ 默认是至少一次投递，所以必须在消费端保证幂等。
>  在秒杀下单场景，我会在订单表加唯一索引 `(userId, voucherId)`，这样即使消息被重复投递，也只会成功一次。
>  同时可以结合 Redis `SETNX` 或去重表来进一步保证幂等性。
>  这样就能避免重复下单问题。
