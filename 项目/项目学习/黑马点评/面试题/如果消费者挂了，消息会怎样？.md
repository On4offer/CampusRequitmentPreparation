这个问题考察你对 **Redis Stream 的 Pending List 机制** 是否理解清楚。回答的时候要把 **消费者挂了之后的消息流转** 讲完整。

------

## 🎯 标准面试回答

如果消费者在处理过程中挂了，Redis Stream 的行为是：

1. **消息不会丢失**
   - 消息已经被 `XREADGROUP` 拉取，但没有 `XACK`，会进入 **Pending List**。
   - Pending List 记录了该消息被分配给哪个消费者，但还没确认。
2. **消费者恢复后**
   - 可以继续从 Pending List 里把未确认的消息重新拿出来消费。
   - 典型做法：定期执行 `XREADGROUP ... STREAMS s1 0`，从头拉取未确认消息。
3. **其他消费者处理**
   - 如果原消费者彻底挂掉，也可以由同组内的其他消费者用 **`XCLAIM` 命令** 抢占 Pending List 中的消息，继续消费。

------

## ✅ 面试扩展追问

- **Q1：消息会不会丢？**
   → 不会。只要 Redis 开启持久化，消息在 Stream 里和 Pending List 都有存储。
- **Q2：消息会不会被重复消费？**
   → 可能，因为 Pending List 的消息可能被重新拉取。所以需要业务层幂等性（如订单唯一索引）。
- **Q3：如果整个 Redis 宕机呢？**
   → 开启 AOF/RDB 持久化 + 主从/集群架构，确保消息不会因为单点故障而丢失。

------

## 📌 面试模板总结

> 如果消费者挂了，Redis Stream 的消息会进入 **Pending List**，不会丢失。
>  消费者恢复后可以重新处理这些消息，或者由其他消费者通过 `XCLAIM` 接手。
>  这就是 Redis Stream 提供的 **At-Least-Once 投递语义**，再结合数据库唯一索引保证幂等，就能避免重复下单。
