## 🎯 一、秒杀系统四大核心问题

> 高并发、超卖、一人多单、数据库压力大

------

## ✅ 1️⃣ 高并发（瞬时流量暴增）

### 📌 问题：

秒杀开始瞬间，大量请求涌入，可能直接把数据库、接口、应用打挂。

### 📉 原因：

- 所有请求都直接打到数据库；
- 无缓存、无限流保护；
- 并发量远超数据库 QPS。

### ✅ 解决方案：

| 层面         | 技术手段                             | 说明                       |
| ------------ | ------------------------------------ | -------------------------- |
| **前端层**   | 页面静态化、CDN缓存、接口隐藏        | 减少动态请求数量           |
| **网关层**   | 限流（Nginx、Sentinel、RateLimiter） | 丢弃超出流量，保护后端     |
| **应用层**   | 异步下单（Redis Stream / MQ）        | 请求先入队列，后台慢慢处理 |
| **缓存层**   | Redis 缓存热点商品信息               | 避免数据库压力             |
| **数据库层** | 分库分表、读写分离                   | 支撑高并发写入             |

**黑马点评实战：**
 👉 使用 **Redis + Lua 脚本** 实现库存扣减 + 异步入队（Stream 队列）。
 前端秒杀请求全部打 Redis，不直接访问数据库。

------

## ✅ 2️⃣ 超卖（库存为负）

### 📌 问题：

高并发时多线程同时扣库存，可能出现负数库存（超卖）。

### 📉 原因：

- 并发更新库存字段；
- 检查库存与扣减分两步执行（非原子操作）；
- 缓存和数据库不同步。

### ✅ 解决方案：

| 层面           | 技术手段               | 说明                      |
| -------------- | ---------------------- | ------------------------- |
| **原子操作**   | Redis Lua 脚本         | 一次性完成库存检查 + 扣减 |
| **分布式锁**   | RedissonLock / SETNX   | 保证同一商品串行扣减      |
| **数据库约束** | update where stock > 0 | 防止负库存                |
| **异步校验**   | 订单消费者二次检查     | 保证最终一致性            |

**黑马点评实战：**
 👉 秒杀 Lua 脚本核心逻辑：

```lua
if (stock <= 0) then
  return 1  -- 库存不足
end
stock = stock - 1
-- 添加消息到 stream 队列
```

确保库存检查与扣减在 Redis 层原子执行，不存在并发冲突。

------

## ✅ 3️⃣ 一人多单（重复下单）

### 📌 问题：

同一用户快速点击，可能生成多个订单。

### 📉 原因：

- 无幂等控制；
- 多次请求并发进入；
- 数据库唯一约束缺失。

### ✅ 解决方案：

| 层面               | 技术手段                    | 说明                 |
| ------------------ | --------------------------- | -------------------- |
| **Redis 原子校验** | Lua 脚本中判断是否已下单    | 避免重复入队         |
| **唯一索引**       | userId + voucherId 唯一索引 | 防止数据库层重复插入 |
| **幂等设计**       | 接口幂等性（token、锁）     | 保证每次请求结果一致 |

**黑马点评实战：**
 👉 Lua 脚本中增加判断：

```lua
if (已下单) then
  return 2  -- 重复下单
end
```

👉 MySQL 层加唯一约束：

```sql
UNIQUE KEY uk_user_voucher (user_id, voucher_id)
```

双保险保证一人只能下一单。

------

## ✅ 4️⃣ 数据库压力大

### 📌 问题：

秒杀时所有请求写数据库，DB 承压严重甚至宕机。

### 📉 原因：

- 并发写操作多；
- 热点表、热点行（同一商品）；
- 无缓存或异步分流。

### ✅ 解决方案：

| 层面         | 技术手段                     | 说明                         |
| ------------ | ---------------------------- | ---------------------------- |
| **异步削峰** | Redis Stream / MQ 异步下单   | 实时请求先缓存，后台慢慢写库 |
| **缓存预热** | 秒杀前预加载商品信息到 Redis | 降低数据库读取压力           |
| **分库分表** | 按用户或订单分库             | 降低单库写入压力             |
| **批量写入** | 异步消费者批量落库           | 提高写入效率                 |

**黑马点评实战：**

- Redis Stream 队列存储订单消息；
- 异步线程池消费消息并写入数据库；
- 数据库只承受异步写压力，峰值被平滑化。

------

## ✅ 五、总结一句话（面试标准口述版）

> 秒杀系统的核心问题是高并发、超卖、一人多单和数据库压力大。
>  我在项目中主要通过 Redis + Lua 实现原子扣库存防止超卖，
>  使用唯一索引和脚本判断防止重复下单，
>  通过 Redis Stream 实现异步下单削峰，
>  再结合限流、缓存、消息队列共同保障系统高可用和一致性。

## ✅ 六、附带扩展：面试官可能追问的问题

| 追问问题                          | 回答方向                                        |
| --------------------------------- | ----------------------------------------------- |
| 为什么用 Redis 而不是直接 MySQL？ | Redis 内存级别性能更高，支持原子 Lua 脚本操作。 |
| Redis 宕机怎么办？                | 启用持久化 AOF + 主从复制。                     |
| 队列堆积怎么处理？                | 增加消费者线程 / 分区消费 / 限流。              |
| 如何保证消息不丢？                | Stream + 消费组 + pending-list 重试机制。       |
| 如果库存和订单不一致？            | 定期校对库存与订单表，保证最终一致性。          |

## ✅ 七、完整设计总览图（逻辑概括）

```
前端限流/CDN缓存
        ↓
请求进入 Redis
        ↓
Lua 脚本原子扣库存 + 防重
        ↓
消息写入 Stream 队列
        ↓
消费者异步下单写数据库
        ↓
数据库唯一约束防止多单
```
