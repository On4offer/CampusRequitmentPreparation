## 🎯 面试追问：为什么要从 Redis Stream 换成 RocketMQ？

### 1. **先承认 Redis Stream 的价值**

Redis Stream 在黑马点评项目里已经解决了最基本的问题：

- 秒杀请求先经过 **Redis Lua 预扣库存**，避免直接冲击数据库；
- 下单请求进入 **Redis Stream 队列**，再由消费者异步落库；
- 它运维简单，依赖已有 Redis 就能用，适合教学和中小业务场景。

👉 **问题来了**：随着并发量上升，Redis Stream 的短板会越来越明显。

------

### 2. **Redis Stream 的局限性分析**

1. **堆积能力有限**
   - Redis 是内存数据库，消息存放在内存中。
   - 一旦出现大促、秒杀，瞬间几百万请求打进来，消息堆积会吃掉大量内存，影响缓存服务本身。
   - 虽然可以持久化，但 Redis 的设计核心是“缓存”，而不是“消息堆积”。
2. **可靠性不足**
   - Redis Stream 的持久化依赖 RDB/AOF，高可用要靠主从+哨兵。
   - 跨机房部署、网络抖动、宕机恢复时，可能出现消息丢失/重复的问题。
   - Stream PendingList 虽然能追踪未确认消息，但容灾和消息一致性保障能力不如专业 MQ。
3. **事务一致性缺失**
   - 秒杀场景里，订单落库和库存扣减必须和消息队列保持一致。
   - Redis Stream 本身不支持 **事务消息**，没办法解决“消息发出成功但本地事务失败”的问题。
   - 最终可能导致 **超卖** 或 **丢单**。
4. **消费模型过于简单**
   - Stream 只支持 **消费者组 + PendingList**，功能偏弱。
   - 不支持顺序消费（如同一用户的订单必须顺序执行）、不支持广播消费、多维度订阅。
   - 很难满足复杂的企业级业务需求。

------

### 3. **RocketMQ 的优势解释**

RocketMQ 作为专业消息中间件，正好弥补了以上缺陷：

- **高性能高吞吐**：百万级 TPS，支持大规模秒杀场景。
- **可靠存储**：消息落盘，Broker 多副本机制保证消息不丢；支持海量消息堆积，不受内存限制。
- **事务消息**：天然支持事务消息，能保证“扣库存 + 写订单 + 发消息”的一致性。
- **丰富消费模型**：支持集群消费（削峰）、广播消费（通知类业务）、顺序消费（保证用户下单顺序）。
- **跨机房容灾**：多副本、异步复制、自动 failover，远比 Redis Stream 强。

------

### 4. **总结对比**

- Redis Stream：轻量、上手快，适合教学和中小型场景。
- RocketMQ：企业级高并发场景的标配，能解决 **削峰填谷、可靠性、事务一致性、堆积能力** 等核心问题。

👉 **一句话总结**：Redis Stream 更像是“过渡方案”，RocketMQ 才能真正支撑企业级秒杀。

### 📌 面试官加分回答（带分析）

> 在黑马点评中，我们先用 Redis Stream 来实现异步削峰，但它在可靠性、消息堆积、事务一致性上有先天不足。
>  比如，秒杀场景下订单量暴增，Redis 内存容易被挤爆；订单写库失败时也没法保证消息和数据库一致。
>  所以在企业级场景，我会引入 RocketMQ：它支持事务消息，能保证“扣库存 + 下单 + 消息发送”的强一致；同时消息落盘，堆积能力强，不怕流量洪峰。
>  这样不仅解决了削峰填谷，还提升了系统的可靠性和扩展性。
