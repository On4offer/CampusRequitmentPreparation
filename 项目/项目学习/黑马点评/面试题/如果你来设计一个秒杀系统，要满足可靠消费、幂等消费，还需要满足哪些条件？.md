## 🎯 一、面试题原型

> “如果你来设计一个秒杀系统，要满足可靠消费、幂等消费，还需要满足哪些条件？”

面试官考察点：

- 你是否能覆盖秒杀的**三大核心问题**：**超卖、重复下单、系统崩溃**
- 是否能讲出完整的技术链条（缓存、消息队列、锁、事务等）

------

## ✅ 二、系统设计必须满足的关键条件（核心 6 点）

| 分类           | 目标                 | 说明                                                         |
| -------------- | -------------------- | ------------------------------------------------------------ |
| 1️⃣ 数据一致性   | **防止超卖、超发**   | 必须保证库存扣减和订单创建的一致性，使用原子操作（Lua 脚本 / 分布式锁 / 事务）。 |
| 2️⃣ 请求削峰     | **防止系统崩溃**     | 使用消息队列（如 Redis Stream / MQ）异步下单，削峰填谷。     |
| 3️⃣ 幂等性       | **防止重复下单**     | 用户维度加唯一约束（userId + voucherId），Redis + 数据库双重防重。 |
| 4️⃣ 可靠消费     | **防止消息丢失**     | MQ 需支持确认机制（ACK / PendingList），消费者异常可恢复。   |
| 5️⃣ 一致可用性   | **防止接口雪崩**     | Redis 缓存热点数据、限流 + 熔断 + 降级，保证秒杀活动不被打挂。 |
| 6️⃣ 高可用与扩展 | **支撑高并发高可用** | 服务分层：前端缓存、限流、异步消息队列、数据库分库分表。     |

------

## ✅ 三、系统整体设计目标（面试表述要连贯）

> 我们在设计秒杀系统时，需要从**一致性、可靠性、性能和扩展性**四个维度考虑：
>
> - 一致性：防止超卖、保证库存扣减原子性。
> - 可靠性：防止消息丢失、重复消费。
> - 性能：通过缓存、限流、消息队列削峰。
> - 扩展性：服务分层与异步化，支持水平扩展。

------

## ✅ 四、每个维度下的典型技术方案

| 目标           | 技术手段                                     | 实现要点                     |
| -------------- | -------------------------------------------- | ---------------------------- |
| **防超卖**     | Redis Lua 原子扣库存 / 分布式锁 / 数据库行锁 | 保证库存扣减和订单创建一致。 |
| **防重复下单** | 唯一索引 + Redis Set + 幂等校验              | userId + voucherId 唯一键。  |
| **请求削峰**   | Redis Stream / MQ / 异步任务队列             | 先写入队列，后异步下单。     |
| **可靠消费**   | ACK 机制 + PendingList 重试                  | 防止消费者宕机丢消息。       |
| **幂等消费**   | 幂等标识 + 数据库约束                        | 消费重复消息也不影响结果。   |
| **限流保护**   | Redis + Lua 限流 / Guava RateLimiter         | 防止恶意刷接口。             |
| **热点缓存**   | Redis 缓存库存与商品信息                     | 防止数据库被打爆。           |
| **降级熔断**   | Hystrix / Sentinel                           | 秒杀活动异常时快速失败。     |

## ✅ 五、扩展性设计（答得更“架构化”）

| 层次   | 模块                          | 功能                        |
| ------ | ----------------------------- | --------------------------- |
| 前端层 | 静态页 + 限流脚本             | 缓存商品页、JS 限制点击频率 |
| 网关层 | Nginx + Sentinel              | 限流与黑名单拦截            |
| 应用层 | Redis 秒杀接口 + Lua 脚本     | 原子扣库存 + 消息入队       |
| 消费层 | Stream 消费者线程 / MQ 消费者 | 异步下单，保证可靠消费      |
| 数据层 | MySQL + 唯一索引              | 保障幂等与一致性            |
| 异常层 | PendingList + 补偿任务        | 宕机重试、消息补偿          |

## ✅ 六、总结一段标准面试回答模板（30 秒口述版）

> 在设计秒杀系统时，除了要保证可靠消费和幂等消费外，还要同时满足**高并发安全性**和**系统可用性**。
>  我会从四个方面设计：
>
> - **一致性**：通过 Redis Lua 原子扣库存防止超卖；
> - **可靠性**：消息队列 + 消费组确保消息不丢；
> - **幂等性**：数据库唯一约束 + 消息去重防止重复下单；
> - **高可用**：前端限流、缓存预热、异步下单、降级熔断防止雪崩。
>
> 整体实现上采用“请求入队、异步下单、可靠消费”的架构，既能保证系统稳定，又能应对高并发。

## ✅ 七、扩展追问方向（建议你准备）

| 追问                         | 你该怎么答                                 |
| ---------------------------- | ------------------------------------------ |
| 如果 Redis 崩了怎么办？      | 启用 MQ 或持久化、自动重试机制；业务回滚。 |
| 如何防止缓存穿透/击穿/雪崩？ | 布隆过滤器 + 逻辑过期 + 分布式锁重建。     |
| 并发太高下单失败多怎么办？   | 使用限流、排队预扣库存、异步通知。         |
| 订单创建慢影响队列怎么办？   | 消费者多线程、分区消费、异步批处理。       |

## ✅ 八、一句话总结

> **可靠消费 + 幂等消费只是保证“下单安全”，
>  但一个真正健壮的秒杀系统，还要满足：防超卖、防重复、削峰填谷、限流降级、高可用扩展。**
