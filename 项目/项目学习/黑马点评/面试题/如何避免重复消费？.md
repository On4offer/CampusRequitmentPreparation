这个问题是 **“消息可靠性三连”** 之一，面试官问你就是要考察：你知道**消息可能会重复投递/消费**，并且能从架构和业务上保证幂等性。

------

## 🎯 标准面试回答

在黑马点评项目中，避免 Redis Stream 的重复消费主要靠 **消费确认机制 + 幂等性设计**：

1. **消息可能重复的原因**
   - 消费者处理完业务，但在 `XACK` 前宕机 → 消息还留在 Pending List，下次会被再次消费。
   - 消费者主动重试时，也可能重复拉取同一条消息。
2. **技术措施**
   - **消费确认（XACK）**：只有业务处理成功才确认，避免“消息丢失”。
   - **Pending List**：未确认的消息会留存，系统定期扫描并重试。
3. **业务侧幂等性保证**
   - 数据库层加唯一约束：比如 **用户 ID + 优惠券 ID 建唯一索引**，保证一个用户同一优惠券只能下一单。
   - 代码层先查再写，或利用分布式锁，确保订单只生成一次。

------

## ✅ 面试扩展追问

- **Q1：如果 Pending List 被多次重试，会不会重复写订单？**
   → 不会，因为数据库唯一索引保证幂等，即使多线程并发插入，也只会成功一次。
- **Q2：幂等性除了唯一索引还有什么手段？**
   → 可以使用 **幂等 Token**（下单前生成唯一流水号），或在 Redis 里做去重标记。
- **Q3：如果消息必须严格一次消费（Exactly Once），能做到吗？**
   → Redis Stream 天然是 **At-Least-Once** 语义，严格一次需要引入分布式事务或 Kafka + 幂等处理，Redis 自身无法保证。

------

## 📌 面试模板总结

> Redis Stream 本身只能保证消息 **至少一次（At-Least-Once）** 被消费，因此避免重复消费必须依赖业务幂等。
>  在黑马点评秒杀下单里，我们通过 **数据库唯一索引（userId + voucherId）** 来保证一人只能下一单，即使消息被重复投递，也不会生成重复订单。
>  这样结合 **XACK + Pending List + 唯一索引**，就能做到“不丢消息、不重下单”。
