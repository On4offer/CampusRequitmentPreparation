# 面试题：如何保证“只有一个线程”去重建缓存？

## 1）场景设定（面试官口吻）

热点 Key 过期的瞬间有大量请求同时到来，**如何确保只有一个线程回源 DB 重建缓存**，其余线程不把数据库打爆？

------

## 2）核心思路（一句话版）

**“单飞（SingleFlight）+ 双重校验（Double-Check）+ 有界等待/降级 + 安全解锁（Lua）+ 超时与续期（看门狗）+ 细粒度粒度（按Key加锁）”**。

------

## 3）可背标准回答（结构化）

### A. 设计目标

- **单飞**：同一 Key 同时只有一个线程回源；
- **可用优先**：没抢到锁的线程要么**快速返回旧值**，要么**短暂等待并重读**；
- **健壮**：防死锁、解锁安全、超时续期、限流兜底。

### B. 方案清单（从简到强）

1. **Redis 分布式互斥锁（SET NX PX） + 双重校验**

- 加锁：`SET lock:shop:{id} uuid NX PX 10s`
- 回源前 **再查一次缓存**（Double-Check）
- 解锁：**Lua 脚本比对 uuid** 再 `DEL`，避免删别人锁
- 未拿到锁：**睡眠50ms重试（带抖动）** 或 **直接返回旧值/短路降级**

1. **Redisson RLock（看门狗续期）**

- `tryLock(waitTime, leaseTime, TimeUnit)`；长耗时重建由 **看门狗**自动续期
- 公平锁可选（一般不需要，降低吞吐）

1. **本机 SingleFlight（JVM 内合并） + 跨机 Redis 锁（全局合并）**

- 先用本机 `ConcurrentHashMap<key, CompletableFuture<T>>` 合并同机并发
- 若首次命中失败，再用 Redis 锁做跨实例合并
- 这样**双层合并**吞吐更好

1. **异步重建队列 + 去重集合**

- 读线程只**提交重建任务**（如 Redis `SETNX rebuild:{key}` 作为去重标记）
- 由后台线程池统一按速率重建（削峰）

1. **版本号/栅栏（Fencing Token）+ CAS 写回**

- 回源时从 DB 读出 `version`
- 写回缓存时仅当 `version` **更新**或 **token 最大** 才覆盖
- 即使“多线程重建”，也能保证**最终只生效最新者**

### C. 关键细节（一定要讲的“避坑点”）

- **双重校验**：抢到锁后**再查一次缓存**，避免重复回源；
- **解锁安全**：**Lua 比对** value 再删，防止误删他人锁；
- **锁粒度**：**按 Key（id）加锁**，而非全局锁；
- **超时策略**：`PX` 必须 ≥ 回源 P95；不确定时用 **Redisson 看门狗**；
- **失败策略**：拿不到锁的线程 **短睡+限次重试** 或 **直接返回旧值**（配合逻辑过期更顺滑）；
- **限流/降级**：热点爆发时，适当 **限流** 或 **返回骨架/旧值**，保护 DB；
- **监控**：锁等待时长、重建耗时、失败率、热点 TopN。

## 4）最小可用代码（面试可写）

### 4.1 Redis 原生锁 + 双重校验 + Lua 解锁

```java
public <T> T queryWithMutex(String key, Supplier<T> dbFallback, long ttl, TimeUnit unit) {
    String json = redis.opsForValue().get(key);
    if (StrUtil.isNotBlank(json)) return JSONUtil.toBean(json, new TypeReference<T>() {}, false);
    if (json != null) return null; // 命中空值

    String lockKey = "lock:" + key;
    String lockId  = UUID.randomUUID().toString(true);
    boolean locked = tryLock(lockKey, lockId, 10, TimeUnit.SECONDS);
    if (!locked) {
        // 没拿到锁：短睡+重试 或 直接返回（推荐短睡+限次）
        sleep(50);
        return queryWithMutex(key, dbFallback, ttl, unit);
    }
    try {
        // Double-Check：拿到锁后再查一次，避免重复回源
        json = redis.opsForValue().get(key);
        if (StrUtil.isNotBlank(json)) return JSONUtil.toBean(json, new TypeReference<T>() {}, false);
        if (json != null) return null;

        T db = dbFallback.get();
        if (db == null) {
            redis.opsForValue().set(key, "", 2, TimeUnit.MINUTES); // 空值短TTL
            return null;
        }
        redis.opsForValue().set(key, JSONUtil.toJsonStr(db), ttl, unit);
        return db;
    } finally {
        unlockByLua(lockKey, lockId);
    }
}

private boolean tryLock(String key, String val, long ttl, TimeUnit unit) {
    Boolean ok = redis.opsForValue().setIfAbsent(key, val, ttl, unit);
    return BooleanUtil.isTrue(ok);
}

private static final DefaultRedisScript<Long> UNLOCK_SCRIPT;
static {
    UNLOCK_SCRIPT = new DefaultRedisScript<>();
    UNLOCK_SCRIPT.setScriptText(
        "if redis.call('get', KEYS[1]) == ARGV[1] then " +
        "  return redis.call('del', KEYS[1]) " +
        "else return 0 end");
    UNLOCK_SCRIPT.setResultType(Long.class);
}
private void unlockByLua(String key, String val) {
    redis.execute(UNLOCK_SCRIPT, Collections.singletonList(key), val);
}

private void sleep(long ms) {
    try { Thread.sleep(ms); } catch (InterruptedException e) { Thread.currentThread().interrupt(); }
}
```

### 4.2 Redisson（更省心）

```java
RLock lock = redisson.getLock("lock:" + key);
if (lock.tryLock(100, 0, TimeUnit.MILLISECONDS)) { // waitTime=100ms, leaseTime=0(看门狗)
    try {
        // Double-Check 再查缓存
        // 回源&写回
    } finally {
        lock.unlock();
    }
} else {
    // 没拿到锁：短等待后重读缓存/返回旧值
}
```

### 4.3 本机 SingleFlight（合并同机并发）

```java
private final ConcurrentHashMap<String, CompletableFuture<Object>> inflight = new ConcurrentHashMap<>();

public <T> T singleFlight(String key, Supplier<T> supplier) {
    CompletableFuture<Object> future = new CompletableFuture<>();
    CompletableFuture<Object> prev = inflight.putIfAbsent(key, future);
    if (prev != null) {
        return (T) prev.join(); // 等“先行者”结果
    }
    try {
        T res = supplier.get();
        future.complete(res);
        return res;
    } catch (Throwable t) {
        future.completeExceptionally(t);
        throw t;
    } finally {
        inflight.remove(key);
    }
}
```

> 实战常用：“**先本机SingleFlight**，失败后再**Redis锁**”，双层合并抖动更小。

## 5）与“逻辑过期”的配合（高可用实践）

- 缓存值带 `expireTime`，即使过期也**先返回旧值**，并 **后台异步抢锁刷新**；
- 好处：用户侧**无抖动**，没抢到锁的请求**不阻塞**；
- 结论：**强一致首选互斥锁**，**可用优先首选逻辑过期**，**大促/热点**二者结合最佳。

## 6）一段话总结（面试收尾）

> 我会采用**按 Key 细粒度的分布式互斥锁**保证“单飞”，并配合**双重校验**避免重复回源；解锁用 **Lua 比对**保证只删自己的锁；锁 **TTL ≥ 回源 P95** 或直接用 **Redisson 看门狗**防止过期释放；没拿到锁的并发请求**短暂重试或返回旧值**（逻辑过期）以保障可用性。更进一步，用 **SingleFlight** 合并同机并发、**版本号/CAS**保证只有最新结果写回、**限流/降级**保 DB 安全，整体实现高性能与稳定的一致性控制。
