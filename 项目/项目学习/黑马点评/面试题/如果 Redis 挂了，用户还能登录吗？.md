这是面试官很喜欢抛的**追问**，考察你对**单点故障、系统高可用和容错机制**的理解。你要分场景回答：

------

## 1. 直白回答

如果 **Redis 挂了**，系统将无法：

- 存验证码（验证码登录失效）；
- 存 Token 信息（Token 无法校验、续期）。

所以，用户**无法完成登录**，已经登录的用户请求也会因为无法校验 Token 而被拦截。

------

## 2. 深入分析

### （1）为什么会挂

- Redis 是**单点**（单实例部署，没有哨兵/集群）。
- 硬件故障 / 进程异常 / 网络抖动。

### （2）影响范围

- **验证码登录流程**：验证码无法写入 → 登录入口全挂。
- **Token 校验**：Redis 不可用 → 拦截器查不到用户信息 → 认为未登录。
- 等于整个用户系统“挂”了。

------

## 3. 工程上的解决方案

### （1）高可用 Redis 部署

- **主从 + 哨兵**（Sentinel）：保证 Redis 节点宕机后能自动切换主节点。
- **Redis Cluster**：分片 + 高可用，抗单点。
- **云服务 Redis**（如阿里云/腾讯云）：一般 SLA 99.9%，自带主从。

### （2）本地/降级方案

- **验证码**：在极端情况下，可考虑 **直接写数据库（MySQL）作为兜底**，但性能差、用作紧急兜底方案。
- **Token**：可以在登录后同时 **签发 JWT（带签名、不依赖 Redis）**，JWT 校验走本地逻辑，不必依赖 Redis。Redis 仅作为**Session增强**（滑动过期、强退）。
  - JWT 的缺点：无法即时强退、无法滑动过期。
  - 所以可以混合：Redis 挂了时 → 系统切到 JWT-only 模式，保证**核心业务可用**。

### （3）业务容错

- 在拦截器里加**降级逻辑**：Redis 挂了时 → 允许短时白名单接口通过，或提示“系统维护中，请稍后重试”，避免大规模 500 错误。
- 在验证码接口加 **限流和 fallback**：Redis 不可用 → 暂时禁止发送，防止短信费用爆炸。

------

## 4. 面试标准回答

> 如果 Redis 挂了，用户是无法登录的，因为验证码和 Token 校验都依赖 Redis。
>  在实际项目中我们会通过 **Redis 高可用集群（主从 + 哨兵/Cluster）** 来保证 Redis 不会单点故障。
>  此外还会设计**兜底方案**：验证码可在极端情况下回退到数据库存储；Token 校验可以采用 **JWT + Redis 混合模式**，即便 Redis 不可用，用户凭 JWT 也能短期访问，保证系统的**可用性和用户体验**。
>  总结就是：业务层面要有降级兜底，架构层面要有 Redis 高可用。
