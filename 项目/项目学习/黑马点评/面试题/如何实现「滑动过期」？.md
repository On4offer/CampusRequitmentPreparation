# 面试题：如何实现「滑动过期」(Sliding Expiration)？

## 1. 核心概念（10 秒说清）

- **定义**：用户会话在 Redis 里设置 TTL；**只要用户持续活跃，就在临近过期时刷新 TTL**；若长时间不访问，**自动过期下线**。
- **适用场景**：本项目使用 **UUID Token + Redis 会话**（有状态），不使用自包含的 JWT。

## 2. 两种实现思路（说出取舍）

- **A. Redis 会话滑动 TTL（本项目用）**：`login:token:{uuid} -> UserDTO(Hash)`，每次请求**接近过期时** `EXPIRE` 续命。
- **B. AT/RT 双 Token（大厂常见）**：短效 AccessToken + 长效 RefreshToken，快过期时用 RT 换新 AT/RT。更复杂，适合无状态 JWT 或统一认证中心。

## 3. 关键流程（顺序与职责）

1. **登录成功**：生成 `token=UUID`；写入 `login:token:{token}`（Hash 存精简 `UserDTO`），`EXPIRE=30min`。
2. **拦截器顺序**：
   - `RefreshTokenInterceptor`（order=0）：校验并**按阈值续期** + 将 `UserDTO` 放入 `ThreadLocal`。
   - `LoginInterceptor`（order=1）：鉴权（需要登录的接口）。
3. **请求到达**：命中 Redis → 注入用户 → 如 **TTL < 阈值** 才续期；未命中 → 401。

## 4. 代码（可直接用）

### 4.1 拦截器（阈值续期，降写放）

```java
public class RefreshTokenInterceptor implements HandlerInterceptor {
    private final StringRedisTemplate redis;
    private static final long TTL_SEC = 30 * 60;                 // 30min
    private static final long REFRESH_THRESHOLD_SEC = 10 * 60;   // 剩余 <10min 再续

    public RefreshTokenInterceptor(StringRedisTemplate redis) { this.redis = redis; }

    @Override
    public boolean preHandle(HttpServletRequest req, HttpServletResponse resp, Object handler) {
        String token = req.getHeader("authorization");
        if (token == null || token.isEmpty()) return true;

        String key = "login:token:" + token;
        Map<Object,Object> map = redis.opsForHash().entries(key);
        if (map == null || map.isEmpty()) return true;

        // 1) 注入用户
        UserDTO dto = BeanUtil.fillBeanWithMap(map, new UserDTO(), false);
        UserHolder.saveUser(dto);

        // 2) 临近过期再续
        Long ttl = redis.getExpire(key, TimeUnit.SECONDS);
        if (ttl != null && ttl > 0 && ttl < REFRESH_THRESHOLD_SEC) {
            redis.expire(key, TTL_SEC, TimeUnit.SECONDS);
        }
        return true;
    }

    @Override
    public void afterCompletion(HttpServletRequest req, HttpServletResponse resp, Object handler, Exception ex) {
        UserHolder.removeUser();
    }
}
```

### 4.2 可选加分：Lua 原子“存在才续期”

```lua
-- KEYS[1]=key  ARGV[1]=ttlSeconds
if redis.call('EXISTS', KEYS[1]) == 1 then
  return redis.call('EXPIRE', KEYS[1], ARGV[1])
else
  return 0
end
```

> 在 Java 中用 `DefaultRedisScript<Long>` 执行，避免并发下“读到就没了”的竞态续期。

### 4.3 拦截器注册（确保续期优先）

```java
registry.addInterceptor(new RefreshTokenInterceptor(stringRedisTemplate)).order(0);
registry.addInterceptor(new LoginInterceptor())
        .excludePathPatterns("/user/code","/user/login","/shop/**","/shop-type/**","/upload/**","/voucher/**")
        .order(1);
```

## 5. 面试可讲的优化点（点到即止）

- **限频续期**：只在 TTL < 阈值（1/3~1/2）续，避免每次请求都写 Redis（写放大）。
- **精简会话**：只存 `UserDTO` 必要字段；资料变更时按用户维度批量刷新其活跃 token 的 Hash。
- **多端配额/强制下线**：维护 `login:user:{uid}`（ZSET 记录 token 与登录时间），控制最大会话数，后台一键踢人。
- **可观测性**：埋点续期次数、命中率、401 比例，配置化 TTL/阈值（Nacos）。

## 6. 标准回答模板（30 秒版）

> 我用 **UUID+Redis** 存会话并设置 30 分钟 TTL；在 `RefreshTokenInterceptor` 中先从 Redis 取用户并放入 `ThreadLocal`，**仅当剩余 TTL 低于阈值（如 10 分钟）时才 `EXPIRE` 续期**，从而实现滑动过期并减少写放大。
>  拦截器顺序是**续期优先、鉴权其次**；同时我们支持多端会话配额与后台强制下线，并对续期做埋点和配置化，保证体验与安全的平衡。
