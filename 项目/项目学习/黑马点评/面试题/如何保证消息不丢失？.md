## 🎯 标准面试回答

在黑马点评项目里，我们主要通过以下机制保证 Redis Stream 的消息不丢失：

1. **生产端：写入保证**
   - 使用 `XADD` 写入消息到 Redis Stream。
   - Redis 自身支持 **RDB / AOF 持久化**，即使 Redis 异常宕机，也能依靠 AOF 追加日志恢复消息。
2. **存储端：持久化存储**
   - Stream 本质上是 Redis 的一种持久化数据结构，不会像内存队列那样掉电即丢失。
   - 可以配置 **AOF everysec**，保证 1s 内写入的消息能落盘。
3. **消费端：确认机制**
   - 消费者读取消息时用 `XREADGROUP`，只有在成功处理订单并落库后，才调用 `XACK` 确认消费。
   - 如果消费者未确认，消息会进入 **Pending List**。
4. **异常兜底：Pending List 重试**
   - 我们有专门的线程定期扫描 Pending List（未确认消息列表），对失败消息进行重新消费。
   - 这样即使消费者宕机、异常退出，消息依然能被重新处理。

## ✅ 面试追问扩展

- **Q1：Redis 挂了怎么办？**
   → 开启持久化（AOF/RDB），并且在生产环境部署 **Redis 主从 + 哨兵 / 集群**，避免单点故障。
- **Q2：消费失败会不会丢消息？**
   → 不会。消息会留在 Pending List，等消费者恢复或补偿任务执行时重新消费。
- **Q3：如果消息被重复消费呢？**
   → 数据库侧保证幂等性，例如下单表 `userId + voucherId` 建唯一索引，确保一个用户只能下一单。

## 📌 面试模板总结

> 在我们项目中，Redis Stream 保证消息不丢失主要依靠 **三层机制**：
>
> - **Redis 持久化**（防止宕机丢失）；
> - **XACK 确认机制**（保证消费成功才删除消息）；
> - **Pending List 兜底**（处理未确认消息，防止遗漏）。
>
> 再结合数据库的幂等约束，可以做到“消息不丢、消费不重”。
