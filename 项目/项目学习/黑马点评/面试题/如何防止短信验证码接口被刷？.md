# 开门结论（30s 背诵版）

> 防短信验证码被刷要用**多层防护**：1）请求端做验证码/风控挑战（CAPTCHA、行为验证）；2）边缘限流（IP/设备/手机号/签名/模板限速）；3）后端基于 Redis 做滑动窗口/令牌桶限流与冷却；4）对高风险触发更严格挑战；5）结合短信供应商配额、黑名单、监控告警与补偿策略。既要保证用户体验，又要把滥发成本和恶意行为压到可控范围。

# 1. 为什么单点措施不够？

攻击者可用分布式 botnet、代理池、重放、脚本并发或大量伪造手机号来刷接口。单靠 IP 限流会被代理绕过；单靠验证码又影响转化率。需要**多维度联合防护**。

------

# 2. 五层防护体系（逐层说明 + 黑马点评落地）

### A. 前端 & 客户端挑战（入口抉择）

- 在“请求发送短信”前强制 **图片/滑动/行为型验证码**（人机校验）。
- 对可疑用户或高频请求升级为更复杂挑战（风险评分策略）。
- 黑马点评落地：在小程序/WEB端先展示滑动验证码或触发图形验证码（降低自动化请求）。

**优点**：把大部分自动化脚本一步挡掉，体验影响可控（只在高风险时弹）。

### B. 边缘限流（网关/Nginx/WAF）

- 在接入层做**IP 速率限制**（例如 10 次/分钟），并结合 **geo / ASN / IP reputation**。
- 对同一 IP 发往不同手机号但短时间大量请求要限流（防代理现场分发）。
- 黑马点评落地：网关层先做粗粒度限流与恶意 IP 黑名单。

### C. 后端精细限流（Redis 为主）

- **按手机号限流**（必须）——例如：`1 次/分钟, 5 次/小时, 10 次/日`。
- **按 IP 限流**（次优）——例如：`20 次/分钟`；和手机号组合使用更强。
- **按设备指纹/UA/Session** 限流。
- **按短信模板/签名/业务场景** 限制（防止某一模板被滥用）。
- **实现方案**（常用三种）：
  1. **固定窗口计数（简单）**：`INCR key; EXPIRE if first`。适合低精度场景。
     - Key: `sms:phone:138xxxx:minute:20250929-1503`
  2. **滑动窗口（准确）**：使用 Redis Sorted Set，记录时间戳，删旧、计数。
     - Key: `sms:sliding:phone:138xxx`，ZADD timestamp member，ZREMRANGEBYSCORE … ZCOUNT …
  3. **令牌桶/漏桶（平滑突发）**：用 Lua 实现原子化加/减，允许短时突发但长期平均受限。
- 黑马点评落地：对 `login/send_code` 接口在 Service 层用 Redis ZSET 做手机号滑动窗口 + IP 固定窗口限流，失败返回统一错误码和延迟提示（不暴露细节）。

### D. 风险识别与挑战升级（分级验证）

- **风险评分**：结合手机号历史、请求频次、IP 地理、UA、设备指纹、是否有账号等，得分超过阈值则：
  - 要求更强验证（图片验证码/行为验证/短信图形验证码）或直接阻断并人工审核。
- 黑马点评落地：对高频手机号做“二次挑战”，例如出现频繁重试则弹 CAPTCHA。

### E. 第三方供应商与限额控制

- 与 SMS 供应商协同：设置每天/每小时的发送上限、告警阈值；对异常流量自动降速或暂停投放。
- 使用多供应商策略；当一个供应商出现异常高成功率（说明可能被滥用）时自动切换并上报。
- 黑马点评落地：设置供应商配额与回退通道（备用厂商）。

# 3. 具体 Redis 实现示例（简洁可背诵）

## 固定窗口（快速实现）

```text
KEY: sms:count:phone:138xxxx:20250929-15-03
INCR -> if ==1 then EXPIRE 60
if value > 1 return 429
```

## 滑动窗口（伪代码，精准）

```text
# key = sms:sliding:phone:138xxxx
now = currentMillis()
redis.zremrangebyscore(key, 0, now - WINDOW_MS)
count = redis.zcard(key)
if count >= LIMIT: reject
else redis.zadd(key, now, now)
redis.expire(key, WINDOW_MS/1000 + 1)
```

## 令牌桶（Lua 原子）

- 存 `tokens` 与 `last_refill_ts`，计算可用 token，原子取走 1 个，否则拒绝。

# 4. 体验与误判控制（重要）

- 对真正用户：**短信流程要尽量顺畅**，限制不宜过严导致短信无法下发。
- 提示语要通用（不要泄露“被限流/被封禁”的细节），例如：**“操作过于频繁，请稍后再试或使用图形验证码”**。
- 对被误判的号码要有人工申诉或客服通道。

------

# 5. 监控与告警（必须）

- 指标：`sms_sent_total`、`sms_reject_total`、`sms_fail_total`、`rate_limit_hits_by_phone`、`rate_limit_hits_by_ip`、`captcha_challenge_rate`。
- 告警场景：短时间内 `sms_reject_total` 激增、供应商成功率异常、单手机号被频繁拒绝。
- 日志采样：保存部分被拒请求的 IP/UA/时间窗，便于溯源与封禁。

------

# 6. 进阶与对抗（面对复杂攻击）

- **Device Fingerprinting + behavioral**：使用 JS 指纹或 SDK 信息，结合鼠标/触屏行为识别脚本。
- **图灵测试 + Proof-of-Work**：在极端场景下要求计算小量工作量（如算数题/微小哈希）以增加攻击成本。
- **号码信誉/黑灰名单**：维护可疑号码池（例如被频繁注册/频繁回退的号码）。
- **Honeypot / Honeytoken**：隐蔽接口用于检测自动化脚本，一旦被触发立即拉黑。

# 7. 面试官常问的追问 & 推荐回答要点

1. **“IP 代理如何绕过？”** → 用多维度（手机号+设备+IP+行为）联合判定，并对疑似代理加严挑战。
2. **“误伤合法用户怎么办？”** → 提供滑动阈值、验证码退路、客服/申诉通道与人工白名单。
3. **“Redis 限流如何原子化？”** → 使用 Lua 脚本或 ZSET 操作保证读写原子性。
4. **“短信成本怎么控制？”** → 限制发送成功率、预估成本许可、设定阈值自动降级（只给高价值用户发）。

# 8. 标准回答模板（90 秒面试版）

> 为防止短信验证码接口被刷，我们采用多层防护：入口做 CAPTCHA/行为验证；网关做 IP 限流；后端用 Redis 做手机号/IP/设备的滑动窗口或令牌桶限流；对高风险升级挑战或拒绝；同时结合短信供应商配额与多供应商切换。所有限流与拒绝都配合监控、告警与人工申诉流程，既把恶意流量挡在外面，又尽可能保证真实用户体验。
