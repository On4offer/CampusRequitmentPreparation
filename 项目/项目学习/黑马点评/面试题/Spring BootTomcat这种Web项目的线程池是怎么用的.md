明白啦 ✅ 你其实就想知道：**在库存扣减的场景里，这些“多线程”到底是怎么来的**。

在黑马点评 / 苍穹外卖这类 Spring Boot 项目里，**多线程的来源就是服务端处理请求时用的线程池**：

------

## 1. HTTP 请求入口 → Web 容器线程池

- 你的后端服务跑在 **Tomcat（默认）、Undertow 或 Jetty** 上。
- 每一个前端请求（比如一个用户点“下单”按钮发起 HTTP 调用），都会被容器分配到 **线程池里的一个工作线程** 来执行 `Controller → Service → Dao` 这条调用链。
- **并发产生**：如果同一时刻有很多用户下单，就会有很多请求同时进入，容器就会调度**多个线程**并行执行这段扣库存的代码。
- 默认配置：Tomcat 最大工作线程数一般是 `200`，这意味着最多同时有 200 个线程在跑业务逻辑。

------

## 2. 消息队列消费者线程（如果用了 MQ）

- 在秒杀优化里，如果下单请求写入 MQ（比如 RocketMQ / Kafka），会有消费者程序去订阅队列。
- MQ 客户端库会**启动消费线程池**，并发地拉取消息和处理订单逻辑。
- 所以即使入口是串行写入 MQ，消费端也可能并发处理。

------

## 3. 业务代码里的线程池（可选）

- 有时候项目里会显式用到 `@Async`、`CompletableFuture`、或者自建 `ThreadPoolTaskExecutor`。
- 这些地方也会额外起线程并发跑任务，比如缓存重建、异步下单等。

------

## 🔑 关键点总结

- **不是一个用户一个线程**，而是**一个请求一个线程**（线程来自容器的线程池，执行完会被复用）。
- 当多个请求在**同一时间点**落到你的应用里，就会触发多个线程**同时执行扣库存逻辑**，这就是所谓“并发下的多线程”。
- 所以库存超卖问题，本质上就是：
  - **容器分配了多个线程**并发跑代码 → 每个线程都读到库存大于 0 → 都执行扣减 → 超卖。
