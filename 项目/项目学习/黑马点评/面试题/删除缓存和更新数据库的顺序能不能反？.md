结论先说：**在黑马点评我们坚持“先更新数据库，再删除缓存（Cache-Aside）”，不建议反过来。**
 原因是两种顺序的“竞争窗口”风险不一样——**删缓存→更库**的脏读更难消除。

# 一、两种顺序的时序对比（面试要点）

## 方案A：**更新DB → 删除缓存**（项目采用）

```
T1：写请求  UPDATE DB 成功提交
T2：DEL cache:shop:123
T3：读请求 命中旧缓存（介于T1~T2之间）→ 短暂读旧
T4：缓存被删 → 下次读 miss → 回源读“新DB” → 回填新缓存
```

- **风险**：T1~T2 的极短窗口内仍可能读到旧缓存；
- **后果**：只是**暂时读旧**，不会把旧值再写回去；
- **修复**：若 DEL 失败，用**重试/延迟双删 + CDC 补偿**即可“最终一致”。

## 方案B：**删除缓存 → 更新DB**（不推荐）

```
T1：DEL cache:shop:123
T2：读请求 缓存 miss → 回源读“旧DB” → 把旧值回填进缓存
T3：写请求 UPDATE DB 提交新值
T4：缓存里却是“旧值”且可能 TTL 很长 → 长时间“脏缓存”
```

- **风险**：T1~T3 之间，读线程会**把旧DB数据重新写回缓存**；
- **后果**：形成**真正脏缓存**，除非再删一次或等TTL过期；
- **补救**：必须加**延迟双删**（删一次→更库→sleep→再删一次）或异步失效，复杂度更高。

# 二、为什么黑马点评选 A（结合项目实际）

- **读多写少**：我们以“店铺详情/探店笔记”为主，**短暂读旧**可接受；
- **A 的失败面更可控**：即便第一次 `DEL` 失败，也能靠
  - **重试 + 延迟双删**（sleep≈一次回源+回填时延，如 300~500ms）；
  - **CDC/Canal 订阅 binlog 异步失效**、**Outbox 补偿**
     做到“最终一致”。
- **B 的失败面更棘手**：删完缓存到更新落库之间，任何读都会**把旧值重新写回缓存**，导致**长时间脏缓存**，需要额外一次删除（延迟双删）才能兜住。

# 三、面试加分：可选的“反顺序”如何自救（若被追问）

只有在你**强约束读路径**或**写后主动写新缓存**时，删→更才有机会安全：

- **删→更→再删（延迟双删）**：
   `DEL → UPDATE DB → sleep 300~500ms → DEL`，避免读线程把旧值写回；
- **写穿/直写（Write-Through）**：
   `UPDATE DB 成功后，直接 set 新缓存`（替代删除），但要考虑**并发覆盖**与**事务一致性**；
- **写事务消息/CDC**：
   更新完成后由 MQ/Canal 通知缓存层**再次失效**（至少一次）；
- **读写互斥（不推荐）**：
   更新时给 `shop:123` 加**写锁**、读路径尊重锁，代价大、影响吞吐。

> 面试表述建议
>  “**删→更**除非配**延迟双删/CDC**，否则极易把旧值回填成**脏缓存**；而**更→删**即使 `DEL` 失败也能通过**补偿**实现最终一致，所以我们在黑马点评统一采用**更→删**。”

# 四、标准回答（30 秒背诵）

> 我们项目采用 **“先更新数据库，再删除缓存”**。理由是这种顺序的竞争窗口只会造成**短暂读旧**，并且可以用**重试+延迟双删**以及 **Canal/MQ 异步失效**做“至少一次”补偿，最终一致性可保障。
>  反之如果“先删缓存再更库”，读线程会把**旧DB数据重新写回缓存**，形成真正的**脏缓存**，除非再删一次或等TTL，治理成本更高。因此在黑马点评里不建议反过来。
