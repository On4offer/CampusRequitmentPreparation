# 1. 手机验证码登录

## （1）背景与需求

- **业务场景**：用户无需注册即可通过手机号登录，输入验证码即可完成身份验证。
- **目标**：提升用户体验，避免繁琐的注册流程。
- **安全性要求**：验证码必须具备时效性、唯一性，并防止恶意刷接口。接口安全（验证码不能明文存储在客户端）。

------

## （2）技术方案

1. **验证码生成**
   - 使用工具类（如 `RandomUtil`）生成 6 位数字验证码。存入 Redis 时，设置 TTL（过期时间）
2. **验证码存储**
   - 存入 Redis（带 TTL），避免数据库压力。（Redis读写快适合高并发，Redis支持TTL自动过期无需额外清理）
   - key 设计：`login:code:手机号`，value = 验证码，过期时间 LOGIN_CODE_TTL 分钟。
3. **验证码验证**
   - 前端输入验证码 → 后端取 Redis 验证 → 成功后生成登录凭证。
   - 校验成功 → 生成登录 Token。
4. **Token 管理**
   - 登录成功后，生成一个随机 Token（UUID），作为用户凭证，将用户信息存入 Redis（Hash 结构，存储精简版用户对象（UserDTO），避免存放敏感字段）。
   - key：`login:token:uuid`，value：用户对象序列化后的 Hash。
   - TTL：30 分钟，用户活跃时刷新。
5. **拦截器 + ThreadLocal**
   - 拦截需要登录的请求，校验 Token。通过配置两个拦截器：LoginInterceptor（order=1）：用于登录验证，只拦截需要登录的接口，排除指定路径（如登录、验证码等接口）；RefreshTokenInterceptor（order=0）：用于刷新token，拦截所有请求，校验 Token 是否存在，若存在则刷新 Redis 中 Token 的 TTL。
   - 通过order(0)和order(1)设置执行顺序，RefreshTokenInterceptor优先执行
   - 使用 ThreadLocal 保存当前用户信息，供后续业务调用。

------

### 校验验证码并登录

```java
// 校验验证码 → 查询或创建用户 → 生成 token → 存 Redis
// 校验手机号
String phone = loginForm.getPhone();
if(RegexUtils.isPhoneInvalid(phone)){
    return Result.fail("手机号格式错误！");
}
// 从 Redis 获取验证码
String cacheCode = stringRedisTemplate.opsForValue().get(LOGIN_CODE_KEY + phone);
String code = loginForm.getCode();
if (cacheCode == null || !cacheCode.equals(code)) {
    return Result.fail("验证码错误");

// 判断用户是否存在

    // 不存在，创建新用户并保存
    user = createUserWithPhone(phone);
}

// 生成 Token 保存用户信息到Redis

// 将用户信息保存到 Redis（Hash）
UserDTO userDTO = BeanUtil.copyProperties(user, UserDTO.class);
// // 把 UserDTO 转为 Map<String, Object>，用于 Redis 的 Hash 存储。
Map<String, Object> userMap = BeanUtil.beanToMap(userDTO, 
    new HashMap<>(),
            CopyOptions.create().setIgnoreNullValue(true)
            .setFieldValueEditor((field, value) -> value.toString()));
            /* beanToMap的转换原理
            1. 反射获取 UserDTO.class 的所有方法
            2. 筛选出 getter 方法：getId(), getPhone(), getNickName()
            3. 通过方法名推导属性名：id, phone, nickName
            4. 调用 getter 方法获取值：1L, "13812345678", "user123"
            5. 应用 CopyOptions 规则处理值
            6. 放入 Map：{"id" → "1", "phone" → "13812345678", "nickName" → "user123"}
            */

// 存储
String tokenKey = LOGIN_USER_KEY + token;
stringRedisTemplate.opsForHash().putAll(tokenKey, userMap);
// 设置token有效期
stringRedisTemplate.expire(tokenKey, LOGIN_USER_TTL, TimeUnit.MINUTES);
// 返回token
return Result.ok(token);

// 用户不存在，用户第一次登录时自动创建账户并存储手机号到数据库中
private User createUserWithPhone(String phone) {
    // 1.创建用户
    User user = new User();
    user.setPhone(phone);
    user.setNickName(USER_NICK_NAME_PREFIX + RandomUtil.randomString(10));
```

### 登录拦截器

#### RefreshTokenInterceptor：滑动过期续期，思路类似“看门狗”：只要用户活跃，就不断续命。

这段 `RefreshTokenInterceptor` 实现的是 **基于 Redis 的“滑动过期续期”**，每次请求刷新 Redis 的 TTL，**保持原 Token 不变**；而我们前面说的 **Refresh Token 方案**是**颁发新的 Token 对**（AT+RT），解决纯 JWT 无状态下的会话续期。

❌ 不同点

1. **本代码：滑动过期**
   - Redis 里存的是 `login:token:{token}` → 用户信息（带 TTL，比如 30 分钟）。
   - 每次请求都会 `expire()` 重置 TTL。
   - Token 本身不变，只是 Redis 的有效期被延长。
   - 适合 **服务端存状态**（Redis Session）。
2. **标准 Refresh Token 机制**
   - Access Token（AT）短期有效，比如 15 分钟；Refresh Token（RT）长期有效，比如 7 天。
   - 当 AT 快过期/已过期 → 用 RT 调接口 `/auth/refresh` → 签发一对新的 AT + RT（RT 也会轮换）。
   - Token 本身会变（AT、RT 都有版本/轮换）。
   - 适合 **无状态 JWT** 或 **跨系统认证**。

```java
public class RefreshTokenInterceptor implements HandlerInterceptor {

    private StringRedisTemplate stringRedisTemplate;

    public RefreshTokenInterceptor(StringRedisTemplate stringRedisTemplate) {
        this.stringRedisTemplate = stringRedisTemplate;
    }

     /**
     * 在请求处理之前进行拦截处理
     *
     * @param request  HTTP请求对象，用于获取请求头信息和会话信息
     * @param response HTTP响应对象，用于设置响应状态码
     * @param handler  处理器对象，表示将要执行的处理器方法
     * @return boolean 返回true表示放行请求，返回false表示拦截请求
     * @throws Exception 处理过程中可能抛出的异常
     */
    @Override
    public boolean preHandle(HttpServletRequest request, HttpServletResponse response, Object handler) throws Exception {
        // 基于token进行用户身份验证和会话管理
        // 1. 获取请求头中的token
        String token = request.getHeader("authorization");
        if (token == null || token.isEmpty()) {
            return true;
        }
        // 2. 基于token获取redis中的用户
        String key = LOGIN_USER_KEY + token;
        Map<Object, Object> userMap = stringRedisTemplate.opsForHash().entries(key);
        // 3.判断用户是否存在
        if (userMap.isEmpty()) {
            return true;
        }
        // 5. 将查询到的hash数据转为UserDTO对象
        UserDTO userDTO = BeanUtil.fillBeanWithMap(userMap, new UserDTO(), false);
        // 6. 存在，保存用户信息到 ThreadLocal
        UserHolder.saveUser(userDTO);
        // 7. 刷新token有效期
        stringRedisTemplate.expire(key, LOGIN_USER_TTL, TimeUnit.MINUTES);
        // 8. 放行
        return true;
    }

    @Override
    public void afterCompletion(HttpServletRequest request, HttpServletResponse response, Object handler, Exception ex) throws Exception {
        // 移除用户
        UserHolder.removeUser();
    }
}
```

```
@Configuration
public class MvcConfig implements WebMvcConfigurer {

    @Resource
    private StringRedisTemplate stringRedisTemplate;

    @Override
    public void addInterceptors(InterceptorRegistry registry) {
        registry.addInterceptor(new LoginInterceptor())
                .excludePathPatterns(
                        "/user/code",
```

## （4）难点与优化

1. **验证码安全**

   - 限制验证码发送频率（如 60 秒内同一手机号只能获取一次）。

   - 使用 Redis 的 `INCR` 实现 IP/手机号限流

   - 代码示例：

     ```java
     String key = "login:sms:count:" + phone;
     Long count = stringRedisTemplate.opsForValue().increment(key, 1);
     if (count == 1) {
         stringRedisTemplate.expire(key, 1, TimeUnit.MINUTES);
     }
     if (count > 3) {
         return Result.fail("操作频繁，请稍后再试");
     }
     ```

     

1. **Token 续期**
   - 用户只要活跃，就会刷新 Token TTL，避免被踢下线。
2. **分布式部署问题**
   - Session 共享困难 → 使用 Redis 统一存储 Token。
3. **内存压力**
   - 存储用户信息时用 **Hash + 精简对象（UserDTO）**，避免直接序列化大对象。

## （5）项目实战经验

- 验证码发送接口加了 **限流**，防止被恶意刷短信。
- 登录拦截器用 **ThreadLocal** 管理用户信息，避免重复查 Redis。
- Redis TTL **滑动过期**：用户每次访问接口都会刷新 token 过期时间，提升用户体验。

# 2. 商户查询缓存

### 一、功能目标

- 店铺详情（`/shop/{id}`）是**高频热点接口**，必须命中缓存以降低 DB 压力、提升 P99 延迟。
- 在**异常/极端**流量下，仍保持稳定（不把数据库打爆）。
- 读多写少，采用**Cache-Aside（旁路缓存）**为主的读写路径。

#### 读写路径（Cache-Aside）

- **读**：`GET shop:id → Redis 命中 ? 命中返回 : 查DB并回写缓存`
- **写**：`UPDATE DB → 删除缓存 key（延迟双删可选）`

------

### 二、技术点 & 方案设计

#### 1）缓存更新策略

1. ##### 背景与问题

- 缓存的本质：加速读，减轻数据库压力。
- 但更新带来问题：
  - **缓存和数据库数据不一致**；
  - **内存淘汰策略**不足以保证强一致（Redis 会基于 LRU/TTL 主动删除，或者内存不足淘汰 key）；
  - **高并发写**容易导致缓存、数据库错乱。

> 因此要设计“缓存更新策略”，保证热点数据既快又准。

2. ##### 主动更新策略分类

| 策略     | 内存消耗                     | 一致性 | 维护成本 | 说明                                     |
| -------- | ---------------------------- | ------ | -------- | ---------------------------------------- |
| 内存淘汰 | 无需额外代码，Redis 自动删除 | 弱     | 低       | 靠 Redis TTL/LRU，命中低，无法保证一致性 |
| 超时剔除 | 设置 TTL，到期重新加载       | 中     | 中       | 有空窗期，可能读到旧值                   |
| 主动更新 | 写时主动删除/更新缓存        | 好     | 高       | 开发复杂度高，但一致性最好               |

3. ##### 不一致解决方案（常见三种）

1. **Cache-Aside Pattern（旁路缓存）**
   - 读：先查缓存，缓存没有 → 查数据库 → 回写缓存。
   - 写：先更新数据库，再删除缓存。
   - 优点：常用、简单，读多写少场景适合。
2. **Read/Write Through Pattern**
   - 应用只操作缓存，由缓存框架负责同步数据库。
   - 适合写多的场景，但实现复杂。
3. **Write Behind Caching Pattern**
   - 写请求先写缓存，再异步批量写数据库。
   - 写性能高，但一致性较差。

#### 2）缓存穿透（查询永远不存在的数据）

**现象**：攻击/异常流量不断请求**不存在的 `id`**，每次都穿透到 DB。
 **方案**（二选一或叠加）：强烈建议“**布隆过滤器 + 空值缓存**”双保险，既挡住**不存在**，也降低**误报带来的 DB 查询概率**。

- **缓存空值**：对“不存在”的结果也写入缓存（空串/特殊标记），设置**短 TTL**（如 2~5 分钟），避免重复打 DB。

- **布隆过滤器**（BF）：将**“可能存在”的 id 放入 BF，**查询前先 `BF.EXISTS`，否就**直接拦截**。
  
  - 误判率可控（1e-3 ~ 1e-6），需要**定期同步**数据增量，或用 RedisBloom 的 `CF`/`BF.RESERVE`。
  
  - 布隆过滤器**不会漏报**，即如果判断“不存在”一定正确。但可能出现**误报（False Positive）**，即元素实际上没存过，却因为哈希冲突导致相关 bit 位都为 1，从而返回“存在”。误判概率与位数组大小、哈希函数个数、插入数据量相关。
  
  - **如果布隆说“不存在”** → 一定正确（绝对没存过）。**如果布隆说“存在”** → 不一定正确（可能误判）。
  
  - **误报存在**会导致：布隆说“可能有”，系统继续查数据库。结果 DB 里查不到 → 这类请求还是会落到数据库。所以：
  
    **布隆能拦截掉所有“不存在”的确定情况**（减少大部分穿透流量）。
  
    但对少量“误报”的 Key，数据库还是要兜底查一次。
  
    为了防止重复穿透，常配合 **空值缓存**（短 TTL 缓存 null 结果）。

#### 3）缓存击穿（热点 key 过期瞬间被并发“锤爆”）

**1）定义 & 现象**

- **定义**：单个**超级热点 Key**在高并发场景下**刚好过期**，大量请求同时穿透到数据库“回源重建”，造成瞬时 DB 压力飙升甚至雪崩。
- **典型表现**：缓存命中率突然跌落、DB RT/连接数/CPU 飙升、接口P99抖动。

**2）两大主线方案总览**

1. **互斥锁（Mutex/SingleFlight）**
   - **目标**：只允许**一个**请求去**回源重建缓存**，其他并发请求**等待/快速失败/降级**。
   - **一致性**：强（重建完成前不会读到旧数据）。
   - **风险**：重建慢则队列堆积；实现需注意锁粒度/超时/防死锁。
2. **逻辑过期（Logical Expiration）**
   - **目标**：缓存值携带 `expireTime`；即使过期也**先返回旧值**（可用优先），再**异步刷新**。
   - **一致性**：最终一致；适用于**读多、对强一致不敏感**的详情页/榜单等。
   - **优势**：没有瞬时抖动；用户侧无感刷新。

> 实战常见：**读多详情页** → 逻辑过期；**金融账单/下单等强一致场景** → 互斥锁（或短期降级+闸流）

#### 4）缓存雪崩（大量 key 同时过期）

**1. 现象与本质**

- **现象**：大量缓存 Key 在同一时刻过期，或 Redis 整体不可用，导致请求同时回源数据库，形成“流量洪峰”，数据库/下游系统可能被瞬间压垮。
- **本质**：缓存作为“削峰器”失效，原本平滑的高并发直接打到数据库，引发级联故障。

**2. 典型诱因**

- **统一 TTL**：所有 Key 设为 24h，凌晨 0 点集体过期。
- **批量加载**：预热脚本、批量写入同时设定相同 TTL。
- **发布/重启**：清空缓存，大量 Key 一次性失效。
- **Redis 故障**：集群宕机或网络分区，等价于“全体过期”。

**3. 防护与优化策略**

**（1）预防措施 —— 避免“同一时刻过期”**

- **TTL 加随机抖动**
   给 Key 的过期时间增加随机值，如 `baseTTL + [0, 300]s`，让 Key 在不同时间散落失效。
- **错峰过期**
   按 Key 的 hash 值分批设置不同 TTL，避免批量聚集。
- **热点数据预热**
   发布或大促前，将高频访问数据提前写入缓存，避免“冷启动”时瞬时击穿。
- **多级缓存**
   JVM 本地缓存（Caffeine/Guava）作为 L1，Redis 作为 L2。即使 Redis 瞬时失效，L1 也能兜住一部分请求。
- **逻辑过期**
   在 Value 内部加“逻辑过期时间”，过期后仍返回旧值，同时后台异步刷新，避免出现大量请求并发击穿。

**（2）缓解措施 —— 雪崩发生时降低冲击**

- **互斥锁/SingleFlight**
   同一 Key 只有一个线程能回源，其余等待或返回旧值，避免重复查询数据库。
- **限流与降级**
  - 超过阈值 QPS 时，部分请求被快速失败/返回兜底数据（如骨架屏）。
  - 熔断策略：下游压力过大时，自动熔断一部分请求，保护核心服务。
- **后台刷新**
   请求量大时，集中由后台线程池按速率刷新，避免所有前端请求直接触发 DB。

**（3）恢复措施 —— 雪崩后修复与防复发**

- **热点快速回填**：根据近期访问日志或 TopN 热点榜，脚本批量加载缓存。
- **配置复盘**：检查是否存在统一 TTL、批量任务、Redis 驱逐策略等风险点。
- **演练机制**：在压测环境中模拟雪崩，验证限流、熔断、降级的效果。

**4. 实际案例场景**

#### 3.1 通用 CacheClient（含三种策略）

- `queryWithPassThrough`：缓存穿透（空值）
- `queryWithMutex`：互斥锁重建
- `queryWithLogicalExpire`：逻辑过期 + 异步重建

```java
// CacheClient.java
@Component
@RequiredArgsConstructor
public class CacheClient {

    private final StringRedisTemplate stringRedisTemplate;

    private static final ExecutorService CACHE_REBUILD_EXECUTOR =
            new ThreadPoolExecutor(2, 8, 60, TimeUnit.SECONDS,
                    new LinkedBlockingQueue<>(1024),
                    new ThreadFactoryBuilder().setNameFormat("cache-rebuild-%d").build());

    // ------------ 1) 穿透（查询不存在的id）：缓存空值 ------------
    public <R, ID> R queryWithPassThrough(
            String keyPrefix, ID id, Class<R> type,
            Function<ID, R> dbFallback,
            long ttl, TimeUnit unit) {

        String key = keyPrefix + id;
        // 从Redis中查询商铺缓存
        String json = stringRedisTemplate.opsForValue().get(key);
        if (StrUtil.isNotBlank(json)) {
            // StrUtil.isNotBlank()会检查字符串是否不为null、不为空字符串、不为空白字符
            // 存在有效数据，将JSON字符串反序列化为指定类型的对象并返回
            return JSONUtil.toBean(json, type);
        }
        
        // 判断命中的是否是空值（解决缓存穿透的关键步骤
        if (json != null) { // 命中 "" 空值
            return null;
        }
        // 查库，这行代码正是用来解决"数据库有数据而Redis没有数据"这种情况的关键步骤。
        R r = dbFallback.apply(id);
        if (r == null) {	 // 数据库没有数据
            // 写空值 ""，短TTL
            stringRedisTemplate.opsForValue().set(key, "", 2, TimeUnit.MINUTES);
            return null;
        }
        // 回写缓存
		this.set(key, r, time, unit);
        return r;
    }

    // ------------ 2) 击穿（热点key过期）：互斥锁 ------------
    public <R, ID> R queryWithMutex(
            String keyPrefix, String lockPrefix, ID id, Class<R> type,
            Function<ID, R> dbFallback,
            long ttl, TimeUnit unit) {

        String key = keyPrefix + id;
        // 先查缓存
        String json = stringRedisTemplate.opsForValue().get(key);
        if (StrUtil.isNotBlank(json)) return JSONUtil.toBean(json, type);
        if (json != null) return null; // 命中空值

        String lockKey = lockPrefix + id;
        R r = null;
        try {
            if (!tryLock(lockKey)) {
                // 没拿到锁，短睡+重试 / 直接返回空 / 返回本地缓存
                Thread.sleep(50);
                return queryWithMutex(keyPrefix, lockPrefix, id, type, dbFallback, ttl, unit);
            }
            // 二次查缓存（double check）
            json = stringRedisTemplate.opsForValue().get(key);
            if (StrUtil.isNotBlank(json)) return JSONUtil.toBean(json, type);
            if (json != null) return null;

            // 回源，获取锁成功，根据id查询数据库
            r = dbFallback.apply(id);
            if (r == null) {	// 不存在
                stringRedisTemplate.opsForValue().set(key, "", 2, TimeUnit.MINUTES);
                return null;
            }
            // 存在，写入Redis
            stringRedisTemplate.opsForValue().set(key, JSONUtil.toJsonStr(r), ttl, unit);
        } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
        } finally {
            unlock(lockKey);
        }
        return r;
    }

    // ------------ 3) 击穿：逻辑过期 + 异步重建 ------------
    @Data
    public static class RedisData {
        private LocalDateTime expireTime;
        private Object data;
    }

    public <R, ID> R queryWithLogicalExpire(
            String keyPrefix, String lockPrefix, ID id, Class<R> type,
            Function<ID, R> dbFallback,
            long ttl, TimeUnit unit) {

        String key = keyPrefix + id;
        String json = stringRedisTemplate.opsForValue().get(key);
        if (StrUtil.isBlank(json)) {
            return null;  // 或者回退到 passThrough
        }
        RedisData redisData = JSONUtil.toBean(json, RedisData.class);
        R r = JSONUtil.toBean((JSONObject) redisData.getData(), type);
        // 未过期，直接返回
        if (redisData.getExpireTime().isAfter(LocalDateTime.now())) {
            return r;
        }
        // 过期，尝试重建
        String lockKey = lockPrefix + id;
        if (tryLock(lockKey)) {
            CACHE_REBUILD_EXECUTOR.submit(() -> {
                try {
                    R fresh = dbFallback.apply(id);	// 查询数据库
                    this.setWithLogicalExpire(key, fresh, ttl, unit);	// 写入Redis
                } finally {
                    unlock(lockKey);
                }
            });
        }
        // 返回旧值兜底
        return r;
    }

    public void setWithLogicalExpire(String key, Object value, Long time, TimeUnit unit) {
        // 创建RedisData对象，用于包装缓存数据和逻辑过期时间
        RedisData redisData = new RedisData();
        // 设置要缓存的实际数据
        redisData.setData(value);
        // 设置逻辑过期时间：当前时间加上指定的过期时长
        redisData.setExpireTime(LocalDateTime.now().plusSeconds(unit.toSeconds(time)));

        // 将RedisData对象转换为JSON字符串并写入Redis
        // JSONUtil.toJsonStr()将对象序列化为JSON格式字符串
        // opsForValue().set()是RedisTemplate的操作方法，用于设置字符串类型的键值对
        stringRedisTemplate.opsForValue().set(key, JSONUtil.toJsonStr(redisData));
    }

    // ------------- 公共：分布式锁 -------------
    private boolean tryLock(String key) {
        Boolean ok = stringRedisTemplate.opsForValue()
                .setIfAbsent(key, "1", 10, TimeUnit.SECONDS);
        return BooleanUtil.isTrue(ok);
    }
    private void unlock(String key) {
        stringRedisTemplate.delete(key);
    }
}
```

#### 3.2 Redis 分布式锁（更健壮版本）

- 采用 **SET NX PX** 原子加锁，释放锁时校验**只有持有者才能删**（Lua）。

```java
public class RedisLock {
    private final StringRedisTemplate redis;
    private final String lockKey;
    private final String lockId = UUID.randomUUID().toString(true);

    public RedisLock(StringRedisTemplate redis, String name) {
        this.redis = redis;
        this.lockKey = "lock:" + name;
    }

    public boolean tryLock(long ttl, TimeUnit unit) {
        Boolean ok = redis.opsForValue().setIfAbsent(lockKey, lockId, ttl, unit);
        return BooleanUtil.isTrue(ok);
    }

    private static final DefaultRedisScript<Long> UNLOCK_SCRIPT;
    static {
        UNLOCK_SCRIPT = new DefaultRedisScript<>();
        UNLOCK_SCRIPT.setScriptText(
                "if redis.call('get', KEYS[1]) == ARGV[1] then " +
                "  return redis.call('del', KEYS[1]) " +
                "else return 0 end");
        UNLOCK_SCRIPT.setResultType(Long.class);
    }

    public void unlock() {
        redis.execute(UNLOCK_SCRIPT, Collections.singletonList(lockKey), lockId);
    }
}
```

### 四、缓存一致性策略（写路径）

#### 总体模式：Cache-Aside（旁路缓存）

在黑马点评的 **店铺详情查询** 中，采用的是 **Cache-Aside 模式**：

- **读流程**
  1. 查询 Redis（`cache:shop:{id}`）
  2. 未命中 → 查询 MySQL
  3. 将结果写入 Redis（带 TTL）
- **写流程**
  1. 更新数据库（`UPDATE shop ...`）
  2. 删除缓存中的店铺数据
  3. 保证数据一致性
     - 如果 **先删缓存，再更新数据库**：
       - A 线程先删缓存。
       - B 线程立刻读数据，缓存 miss → 去 DB，读到旧值。
       - B 再把旧值写回缓存。
       - 接着 A 更新数据库。
          缓存里是旧值，出现 **脏数据**。
     - 如果 **先更新数据库，再删缓存**：
       - 即使有并发读，读到的最多是旧值，但缓存会被删掉。
       - 下次请求重新从 DB 加载时，一定是新值。
          最终缓存与数据库能保持一致。
  4. 删除而不是更新缓存
     - **更新缓存成本高**：需要精确知道哪些 Key 受影响（可能有多个维度：`shop:1`、`shop:hot`、`shop:list`）。
     - **删除缓存更简单**：只删掉对应 Key，让后续请求自然触发缓存重建。

这样能保证 **读多写少** 的场景下缓存命中率高，同时避免更新缓存时字段不全的问题。

#### 写路径策略：逻辑过期刷新（热点商户，项目采用）

- **应用**：某些热点店铺（例如“西安热门小吃店”），访问量极高
- **思路**：
  - 缓存中存放 `data + expireTime`
  - 如果过期：
    - **先返回旧值**，保证接口可用
    - **后台线程拿锁刷新**缓存数据
- **优势**：避免热点缓存过期瞬间的大量回源请求（缓存击穿）。
- **适用场景**：黑马点评的明星店铺详情页，能容忍短时旧数据，能保证最终一致性。

#### 延迟双删（简单方案）

- **应用**：当商户后台更新店铺信息时

- **流程**：

  ```
  UPDATE shop SET ... WHERE id = ?
  DEL cache:shop:{id}
  sleep 300~500ms
  DEL cache:shop:{id}
  ```

- **原因**：如果在删除缓存后，恰好有请求回源查 DB 并把旧值写入缓存，第二次删除可以清理掉旧缓存。

- **适用场景**：黑马点评的后台商户更新功能，写操作频率不高，但要防止偶尔的并发脏数据。

------

#### 事件驱动失效（更稳妥）

- **应用**：商户管理端批量修改店铺、营销活动批量更新店铺数据
- **流程**：
  1. DB 更新成功后，发送 **缓存失效事件**（MQ）。
  2. 订阅者消费事件，删除或刷新缓存。
- **优势**：即使应用宕机，删除缓存的动作也不会丢失，可重试。
- **适用场景**：大促活动、大规模店铺批量更新，保证缓存最终一致。

### 五、Key 设计 & TTL 建议

- **命名**：`cache:shop:{id}`、`lock:shop:{id}`、`bf:shop`
- **TTL**：
  - 详情：`30m ~ 2h`（再 + 随机抖动 `0~10m`）
  - 空值：`2~5m`
- **序列化**：JSON（可观测、易演进）；体量大时考虑压缩（GZIP）或省字段（DTO）

### 六、监控、压测与排错

- **监控指标**：
  - 缓存命中率、DB QPS、锁等待时长、缓存重建耗时、热点 Key 列表
  - 错误率（缓存回源失败数、Lua 执行失败数）
- **压测方法**：JMeter / wrk 对 `/shop/{hotId}` 施压，观测 P95/P99、回源比例
- **常见坑**：
  1. 忘记 **double-check** → 抢到锁后重复回源
  2. 锁未设置 TTL → 死锁
  3. 逻辑过期返回 null → 记得**兜底返回旧值**
  4. 空值 TTL 太长 → 数据“假死”
  5. 布隆过滤器不更新 → 新增 id 永远判不存在

### 七、面试高频问答

**Q1：[击穿时为什么选“逻辑过期”而不是“互斥锁”？](面试题/击穿时为什么选“逻辑过期”而不是“互斥锁”？)**
 A：逻辑过期能**先返回旧值**，用户无感知，适合读多一致性要求较低的详情页；互斥锁会让未拿到锁的请求阻塞/重试，峰值时可能出现抖动。线上可**热点用逻辑过期**，普通用互斥锁。

**Q2：[如何保证“只有一个线程”去重建？](面试题/如何保证“只有一个线程”去重建？)**
 A：通过 `SET NX PX` 拿锁 + **double-check**。也可在应用层做 **SingleFlight**（同一 key 合并回源）。

**Q3：[空值缓存会不会导致“脏缓存”？](面试题/空值缓存会不会导致“脏缓存”？)**
 A：给空值设置**短 TTL**（2~5 分钟），并结合布隆过滤器降低误存风险；若业务明确“id 不存在即永远不存在”，可更长 TTL。

**Q4：[如何定位雪崩和击穿？](面试题/如何定位雪崩和击穿？)**
 A：监控同时刻多 key 同时过期、DB QPS 突刺、热点 key 重建次数异常，结合日志（重建打点）和 Redis 命令 `OBJECT idletime/ttl`。

**Q5：[如果删除缓存失败怎么办？](面试题/如果删除缓存失败怎么办？)**
 → MQ 补偿机制 / 重试机制。

**Q6：[删除缓存和更新数据库的顺序能不能反？](面试题/删除缓存和更新数据库的顺序能不能反？)**
 → 应该先更新 DB，再删缓存，避免读到旧值。

**Q7：[如果并发下读写同时发生，如何保证一致性？](面试题/如果并发下读写同时发生，如何保证一致性？)**
 → 延迟双删、加锁、逻辑过期等手段。

------

### 九、总结（一句话记忆）

> **穿透**：空值 / 布隆；
>  **击穿**：互斥锁 or 逻辑过期 + 异步重建；
>  **雪崩**：随机 TTL + 预热 + 多级缓存 + 限流；
>  写路径用 **Cache-Aside + 删缓存**，必要时**延迟双删/MQ**做最终一致。

# 3. 优惠券秒杀功能

## 3.1 背景与需求

- **业务场景**：用户在活动期间抢购优惠券（秒杀券）。
- **目标**：保证高并发场景下系统稳定，同时满足业务约束：
  1. **一人一单**（同一用户只能成功抢购一次）。
  2. **库存不能超卖**（扣减库存必须原子化）。
  3. **削峰填谷**（避免瞬时并发把数据库打垮）。
  4. 全局唯一ID生成。在黑马点评的**优惠券秒杀**或 **订单系统** 中，每次下单都需要生成一个**全局唯一的订单 ID**：
     - **唯一性**：避免重复 ID（多个节点同时生成必须不冲突）。
     - **有序性**：希望 ID 大体有递增趋势，便于数据库分页和索引优化。
     - **高可用**：在分布式部署、多实例服务中也能正确生成。
     - **高性能**：在高并发场景下生成效率不能成为瓶颈。

------

## 3.2 技术难点

1. **库存超卖问题**

   - 多线程同时判断库存大于 0，然后并发扣减，可能出现超卖。

2. **一人一单问题**

   - 同一用户发起多次请求，可能重复下单。

3. **高并发流量冲击**

   - 秒杀瞬间流量陡增，数据库、应用服务器容易被打挂。

4. **全局唯一ID生成方案**

   | 方案                      | 特点                                            | 优缺点                                                      |
   | ------------------------- | ----------------------------------------------- | ----------------------------------------------------------- |
   | **数据库自增 ID**         | 利用 MySQL 自增主键                             | 简单；单点瓶颈，扩展性差，不适合高并发                      |
   | **UUID**                  | `32` 位字符串，随机生成                         | 保证唯一；无序，冗长，占用空间大，索引性能差                |
   | **雪花算法（Snowflake）** | Twitter 开源，64 位整数，含时间戳+机器号+序列号 | 高效、本地生成、趋势递增；依赖时钟，时钟回拨可能导致重复    |
   | **Redis 生成 ID**         | 利用 `INCR` 实现分布式自增                      | 保证唯一、趋势递增；需要 Redis 依赖，但实现简单，高并发友好 |

------

## 3.3 方案设计

### 3.3.1 基础思路

- **请求入口限流**：Nginx + 网关限流，避免过载。
- **资格校验 & 扣减库存**：在 Redis 中完成，保证高性能 & 原子性。
- **下单逻辑异步化**：请求合格 → 写入消息队列 / 队列缓存，由后台线程或 MQ 异步创建订单。

### 3.3.2 关键点

- **Redis + Lua 脚本**

  - 保证“校验资格 + 扣减库存”原子执行。
  - 如果失败，直接返回，不访问数据库。

- **异步下单**

  - 请求只负责写入队列，不直接写数据库。
  - 消费者线程从队列取任务，执行数据库操作（扣库存 + 创建订单）。

- **Redis 全局唯一 ID 设计**

  ID 结构（64 位）：

  ```
  +-------------------------------------------------------------------+
  | 1bit 符号位 | 31bit 时间戳（秒级） | 23bit 序列号 | 9bit 机器 ID |
  +-------------------------------------------------------------------+
  ```

  - **符号位**：固定为 0。
  - **时间戳部分**：当前时间戳（相对自定义起始时间），保证有序性。
  - **序列号部分**：当天的自增序列，保证同一秒内的并发请求不冲突。
  - **机器 ID 部分**：区分不同服务实例或节点，防止多节点生成冲突。

  3.2 Redis Key 设计：

  - `icr:order:20250101` → 1、2、3…
  - 按日期作为 Key 前缀，保证每天的序列号从 0 开始。

# 4. 分布式锁总结（Redis 分布式锁 & Redisson）

## 4.1 背景与需求

在黑马点评项目里，商户详情是高频接口，常见问题是**缓存击穿**：

- 热点 key 过期瞬间，大量请求同时回源数据库 → 导致 DB 被“锤爆”。

**解决思路**：

- 只有**一个线程**去回源重建缓存，其他线程要等待或直接返回旧值。
- 这就需要 **分布式锁** 来保证互斥。

------

## 4.2 Redis 分布式锁（原生实现）

### 4.2.1 基本原理

- **加锁**：

  ```
  SET lockKey lockValue NX PX 10000
  ```

  - `NX`：只在 key 不存在时才设置，保证互斥。
  - `PX`：设置过期时间，避免死锁。
  - `lockValue`：唯一标识（通常是 UUID + 线程 ID），用于解锁时比对。

- **解锁**：Lua 脚本保证原子性：

  ```lua
  -- 这里的 KEYS[1] 就是锁的key，这里的ARGV[1] 就是当前线程标示
  -- 获取锁中的标示，判断是否与当前线程标示一致
  if (redis.call('GET', KEYS[1]) == ARGV[1]) then
    -- 一致，则删除锁
    return redis.call('DEL', KEYS[1])
  end
  -- 不一致，则直接返回
  return 0
  ```

### 4.2.2 Java 示例

```
public class RedisLock {
    private final StringRedisTemplate redisTemplate;
    private final String lockKey;
    private final String lockValue;

    public RedisLock(StringRedisTemplate redisTemplate, String name) {
        this.redisTemplate = redisTemplate;
        this.lockKey = "lock:" + name;
        this.lockValue = UUID.randomUUID().toString(true) + "-" + Thread.currentThread().getId();
    }

    // 加锁
    public boolean tryLock(long timeoutSec) {
        Boolean ok = redisTemplate.opsForValue()
                .setIfAbsent(lockKey, lockValue, timeoutSec, TimeUnit.SECONDS);
        return Boolean.TRUE.equals(ok);		// 防止空指针异常
    }

    // 解锁（Lua 脚本执行）
    private static final DefaultRedisScript<Long> UNLOCK_SCRIPT;
    static {
        UNLOCK_SCRIPT = new DefaultRedisScript<>();
        UNLOCK_SCRIPT.setScriptText(
            "if redis.call('get', KEYS[1]) == ARGV[1] then " +
            "   return redis.call('del', KEYS[1]) " +
            "else return 0 end"
        );
        UNLOCK_SCRIPT.setResultType(Long.class);
    }

    public void unlock() {
        redisTemplate.execute(UNLOCK_SCRIPT, Collections.singletonList(lockKey), lockValue);
    }
}
```

## 4.3 Redisson 分布式锁（封装实现）

## 4.3.1 为什么用 Redisson（相对原生 Redis 锁）

- **可重入**、**自动续期（看门狗）**、**原子解锁**、**多锁模型**（公平锁、读写锁、联锁、红锁）开箱可用。
- 容错好：业务执行超时不会“提前放锁”（看门狗会延长租期）。
- API 贴近 `java.util.concurrent`，使用成本低，避免自己写 Lua。

> 结论：黑马点评这种**读多写少**、锁持有时间不确定的场景（缓存重建、下单幂等），Redisson 更稳更省心。

## 4.3.3 常用锁型 & API 速记

- `RLock lock = redisson.getLock(key)`：**可重入互斥锁**（默认看门狗自动续期 30s，轮询续期每 10s）。
  - `lock.lock()`：阻塞加锁（自动续期）。
  - `lock.lock(leaseTime, unit)`：**不续期**，到期自动释放（适合短任务）。
  - `lock.tryLock(waitTime, leaseTime, unit)`：**等待 + 自动释放**，常用于避免无限阻塞。
  - `lock.unlock()`：安全解锁。
- `RReadWriteLock rw = redisson.getReadWriteLock(key)`：**读写锁**（读-读并行，写互斥）。
  - `rw.readLock().lock()` / `rw.writeLock().lock()`
- `RLock fair = redisson.getFairLock(key)`：**公平锁**（先到先得，吞吐略降）。
- 其他：`RSemaphore`、`RCountDownLatch`、`RMultiLock`、`RedLock`（跨实例容错，更重）。

- `tryLock(wait, lease)`：防止长时间阻塞；**设置 leaseTime** 防止忘记解锁。
- 下单一致性：**Redis Lua** 先行校验（库存+一人一单） → 队列异步落单（削峰）。

- 语义化：`lock:shop:{id}`、`lock:order:user:{uid}`、`lock:coupon:{voucherId}`
- **粒度尽量小**（按资源维度：店铺/用户/券），降低竞争
- 统一前缀，便于监控与清理

## 4.3.6 选型建议 & 常见坑

**选型**

- **缓存重建**：`RLock.tryLock()` + **逻辑过期**（热点无抖动）
- **用户幂等等**：`tryLock(wait, lease)`；或 `RReadWriteLock`（读多写少）
- **严格顺序/排队场景**：`getFairLock()`（吞吐略降）
- **跨集群高容错**：`RedLock`（成本高，谨慎使用）

**坑点**

1. **误用 `lock.lock()` 无 leaseTime**：强依赖看门狗续期；若线程阻塞/停在断点可能长时间持锁。
    → 优先 `tryLock(wait, lease)`，业务可控时长设 **leaseTime**。

2. 看门狗只在**未指定 leaseTime**的加锁方式下生效；你显式传了 `leaseTime`，就**按你的租期到点自动释放**，不会续期。

   | 调用方式                                  | 是否启用看门狗 | 说明                       |
   | ----------------------------------------- | -------------- | -------------------------- |
   | `lock.lock()`                             | ✅              | 无 leaseTime，自动续期     |
   | `lock.tryLock()`（无参或只传 waitTime）   | ✅              | 等待期间或持有期间自动续期 |
   | `lock.lock(leaseTime, unit)`              | ❌              | 固定租期，到点释放，不续期 |
   | `lock.tryLock(waitTime, leaseTime, unit)` | ❌              | 固定租期，不续期           |

3. **锁内抛异常未解锁**：务必 `try/finally unlock()`。

4. **锁粒度过大**：全局锁导致吞吐暴跌。按资源颗粒度设计 Key。

5. **非幂等的“重建缓存”**：如果重建失败没监控，会长期返回旧值。加**日志/告警**与**重试**。

6. **集群模式下节点时间回拨**（少见）：影响不大，Redisson靠 TTL + 续期兜底。

# 5. 秒杀优化

## 一、面试高频考点

1. **秒杀的核心问题**：高并发、超卖、一人多单、数据库压力大。
2. **优化思路分层**：
   - **请求层**：限流、Nginx、验证码
   - **缓存层**：Redis库存预减 + Lua 保证原子性
   - **异步层**：消息队列削峰（RabbitMQ / Kafka / 内存阻塞队列）
   - **数据库层**：唯一索引 + 乐观锁
3. **黑马点评项目中的实践**：
   - 使用 **Redis 预减库存**。
   - 使用 **Redis Set / Lua 脚本** 保证一人一单。
   - 使用 **阻塞队列 + 异步线程池** 实现削峰。

------

## 二、优化分层讲解

### 1. 请求拦截与削峰

- **问题**：瞬间大量请求直接打到数据库。
- **优化措施**：
  - **Nginx 限流**：限制单位时间的请求数。
  - **验证码 / Token**：防止恶意请求。
  - **消息队列 / 队列削峰**：请求进入队列，后台异步下单。

📌 **面试回答要点**：

> 首先在请求入口做削峰处理，避免瞬时流量直接冲击数据库。

------

### 2. Redis 缓存预减库存

- **思路**：把商品库存预先放到 Redis，用户下单时先扣 Redis。
- **实现**：
  - `DECR key` → 判断库存是否大于 0。
  - **原子操作**：Lua 脚本同时完成库存扣减 + 用户校验。
- **优点**：性能极高（内存操作），防止超卖。

📌 **面试回答要点**：

> Redis 的预减库存可以避免数据库热点行锁冲突，并通过 Lua 脚本实现原子性，保证不会出现超卖和一人多单。

------

### 3. 异步下单（消息队列）

- **问题**：即使 Redis 扛住了，但如果每个请求立刻写数据库，仍可能压力过大。
- **优化**：Redis 扣减成功后 → 将下单请求放入队列 → 后台异步消费，写入数据库。
- **黑马点评**：使用 **阻塞队列 + 线程池**（简化版 MQ）实现。

📌 **面试回答要点**：

> 异步下单把高并发写库变成异步队列消费，起到削峰填谷作用。

------

### 4. 数据库层兜底

- **一人一单**：数据库表加唯一索引 `(user_id, voucher_id)`。
- **超卖问题**：更新库存时 `set stock = stock-1 where stock>0`。

📌 **面试回答要点**：

> 即使 Redis 层出问题，数据库唯一索引和库存校验依然能兜底。

```
        VoucherOrder voucherOrder = new VoucherOrder();

         proxy = (IVoucherOrderService)AopContext.currentProxy();
```

# 6. Redis消息队列

## 一、为什么需要消息队列？

- **场景问题**：
  - 秒杀时，用户请求量巨大，如果直接落数据库，容易压垮系统。
  - Redis 库存预减可以抗住并发，但**写订单**依然可能拖垮数据库。
- **目标**：
  - 削峰填谷：把瞬时流量转为平滑流量。
  - 异步解耦：下单请求快速返回，订单持久化后台异步处理。
  - 提高可靠性：利用 MQ 的持久化、确认机制，防止订单丢失。

------

## 二、Redis Stream 方案

> 黑马点评使用 **Redis Stream + 消费组** 来实现消息队列。

### 特点：

1. **Stream**：Redis 5.0 新数据类型，天然支持消息队列。
2. **消费者组**：支持多消费者并行消费。
3. **持久化**：消息存储在 Redis，防丢失。
4. **确认机制**：`XACK` 确认消费，支持 Pending List 处理异常消息。

------

## 三、实现流程（结合代码）

1. **下单时写入消息队列**
   - 将订单信息写入 Redis Stream（`XADD stream.orders * ...`）。
   - Redis 作为队列，高并发写入。
2. **消费者线程处理消息**
   - 线程池/单线程不断 `XREADGROUP` 从 Stream 中拉取消息。
   - 解析消息 → 转换为 `VoucherOrder` 对象。
   - 调用 `createVoucherOrder` 落库。
3. **确认消息**
   - 成功处理后调用 `XACK` 确认消费，避免重复消费。
4. **异常处理（Pending List）**
   - 如果消费失败，消息进入 Pending List。
   - 系统定期扫描 Pending List（`XREADGROUP ... 0`），重新消费异常消息，保证可靠性。

------

## 四、核心代码讲解

### 消费订单线程

```
public void run() {
    while (true) {
        try {
            // 1. 从Stream读取消息
            List<MapRecord<String, Object, Object>> list = stringRedisTemplate.opsForStream().read(
                Consumer.from("g1", "c1"),
                StreamReadOptions.empty().count(1).block(Duration.ofSeconds(2)),
                StreamOffset.create("stream.orders", ReadOffset.lastConsumed())
            );

            if (list == null || list.isEmpty()) {
                continue; // 没有消息，继续循环
            }

            // 2. 解析订单
            MapRecord<String, Object, Object> record = list.get(0);
            VoucherOrder order = BeanUtil.fillBeanWithMap(record.getValue(), new VoucherOrder(), true);

            // 3. 处理订单（写数据库）
            createVoucherOrder(order);

            // 4. 确认消费
            stringRedisTemplate.opsForStream().acknowledge("s1", "g1", record.getId());
        } catch (Exception e) {
            log.error("处理订单异常", e);
            handlePendingList(); // 异常处理
        }
    }
}
```

### Pending List 处理

```java
private void handlePendingList() {
    while (true) {
        try {
            // 1. 从Pending List拉取未确认的消息
            List<MapRecord<String, Object, Object>> list = stringRedisTemplate.opsForStream().read(
                Consumer.from("g1", "c1"),
                StreamReadOptions.empty().count(1),
                StreamOffset.create("stream.orders", ReadOffset.from("0"))
            );

            if (list == null || list.isEmpty()) {
                break; // 没有异常消息，结束
            }

            // 2. 解析并重新处理
            MapRecord<String, Object, Object> record = list.get(0);
            VoucherOrder order = BeanUtil.fillBeanWithMap(record.getValue(), new VoucherOrder(), true);
            createVoucherOrder(order);

            // 3. 再次确认
            stringRedisTemplate.opsForStream().acknowledge("s1", "g1", record.getId());
        } catch (Exception e) {
            log.error("处理Pending消息异常", e);
            try { Thread.sleep(20);
            } catch (Exception e) {
            	e.printStackTrace();
            }
        }
    }
}
```

# 7. RocketMQ 优化秒杀

## 一、为什么引入 RocketMQ？

- **痛点回顾**
  - Redis Stream：轻量级，但可靠性和功能仍有限（跨机房、事务、顺序性、消息堆积能力不足）。
  - 秒杀业务：订单量暴增，要求 **削峰填谷、保证可靠投递、支持事务消息**，避免超卖、丢单。
- **RocketMQ 优势**
  1. **高性能高吞吐**：百万级 TPS。
  2. **分布式存储**：Topic 多队列，支持水平扩展。
  3. **可靠性**：Broker 持久化，消息不丢失。
  4. **消费模型**：支持广播/集群，支持顺序消费。
  5. **事务消息**：下单场景可保证数据库与消息一致性。

------

## 二、整体架构设计（秒杀场景）

1. **入口**：
   - 用户请求 → Redis Lua 脚本：校验库存 >0 & 一人一单 → 返回成功。
   - 成功后异步调用 RocketMQ **Producer** → 发送下单消息。
2. **MQ 层**：
   - 消息进入 `topic_voucher_order`。
   - Broker 持久化消息，等待消费者处理。
3. **消费层**：
   - 多个 **ConsumerGroup** 实例并发订阅消费。
   - 消息拉取 → 幂等校验（DB唯一索引）→ 创建订单。
   - 消费成功 → ACK，失败 → 重试 / DLQ（死信队列）。
4. **异常保障**：
   - RocketMQ 自带消息重试 & 死信队列。
   - 消费端幂等保证避免重复下单。

# 8. 达人探店

## 一、功能概念

- **达人探店**：允许用户发布图文笔记，其他用户可以浏览、点赞、关注达人，形成社区内容流。
- **目标**：提升用户活跃度和粘性，形成“UGC + 社交”的闭环。

------

## 二、核心问题

1. **笔记发布**：带图片的笔记如何存储与展示？
2. **笔记查询**：分页查询 & 关联用户信息。
3. **点赞功能**：高并发下保证点赞数准确，避免重复点赞。
4. **笔记推送**：达人发布的笔记要推送给粉丝，形成 Feed 流。
5. **点赞排行榜**：高频热度笔记排序。

------

## 三、架构设计 & 技术点

### 1. 发布探店笔记

- **流程**：前端上传图片 → 存储到 OSS/本地 → 返回 URL；保存笔记数据到 MySQL（title、images、shopId、userId、createTime）
- **实现要点**：`@RequestBody Note` 对象接收数据；返回笔记 ID，前端跳转详情页

------

### 2. 查询探店笔记

- **需求**：分页查询 + 携带用户昵称、头像
- **实现**：`Page<Note>` 分页查询 MySQL；再根据 `userId` 查询用户信息，封装成 DTO

------

### 3. 点赞功能

- **问题**：单表更新会有热点写问题；用户点赞状态需要快速判断
- **优化**：
  - Redis **Set** 存储点赞用户ID：`note:like:noteId`
    - `SADD` 点赞
    - `SREM` 取消点赞
    - `SCARD` 点赞总数
    - `SISMEMBER` 判断是否点赞
- **定时任务**：Redis → DB 同步点赞数，保证最终一致性

------

### 4. 点赞排行榜

- **Redis ZSet**：
  - key：`note:hot`
  - value：笔记ID
  - score：热度（点赞数 + 时间衰减公式）
- **支持分页**：`ZREVRANGEBYSCORE` 实现 Top-N 热门笔记

------

### 5. 笔记推送（Feed 流）

- **需求**：达人发布笔记后，推送给粉丝，形成个性化信息流。
- **常见方案**：
  1. **推模式（Fan-out on write）**：
     - 达人发布时，把笔记 ID **写入所有粉丝的收件箱**（Redis List：`feed:userId`）。
     - 优点：用户读取快；缺点：写扩散大，粉丝数多时压力大。
  2. **拉模式（Fan-out on read）**：
     - 只存储笔记本身，用户读取时再去查关注列表 + 笔记表，合并分页。
     - 优点：写入轻量；缺点：读扩散，性能差。
  3. **混合模式（常用）**：
     - 活跃用户用 **推模式**（写入收件箱），非活跃用户用 **拉模式**（按需读取）。
- **技术实现**：
  - Redis List 作为收件箱：`feed:userId = [noteId...]`
  - 达人发笔记时：`LPUSH feed:粉丝ID noteId`，只保留最新 N 条（防止过大）。
  - 用户查看时：`LRANGE feed:userId start end` 分页获取，再批量查询笔记详情。
- **优化点**：
  - **异步化**：发布笔记后通过 MQ（如 RocketMQ）异步推送到粉丝收件箱，避免写阻塞。
  - **分页游标**：基于时间戳或笔记ID作为游标，支持下拉刷新。
  - **冷热分离**：大V → 活跃粉丝用推送，普通用户 → 直接拉取。

------

```java
        // 3.查询blog是否被点赞
        isBlogLiked(blog);
        return Result.ok(blog);
    }

     /**
     * 检查当前博客是否被登录用户点赞
     * @param blog 需要检查的博客对象
     */
    private void isBlogLiked(Blog blog) {
        // 1.获取登录用户
        UserDTO user = UserHolder.getUser();
        if (user == null) {
            // 用户未登录，无需查询是否点赞
            return;
        }
        Long userId = user.getId();
        // 2.判断当前登录用户是否已经点赞
        String key = "blog:liked:" + blog.getId();
        Double score = stringRedisTemplate.opsForZSet().score(key, userId.toString());
        blog.setIsLike(score != null);
    }


    /**
     * 对博客进行点赞或取消点赞操作
     * @param id 博客ID
     * @return 操作结果
     */
    @Override
    public Result likeBlog(Long id) {
        // 1.获取登录用户
        Long userId = UserHolder.getUser().getId();
        // 2.判断当前登录用户是否已经点赞
        String key = BLOG_LIKED_KEY + id;
        Double score = stringRedisTemplate.opsForZSet().score(key, userId.toString());
        if (score == null) {
            // 3.如果未点赞，可以点赞
            // 3.1.数据库点赞数 + 1
            boolean isSuccess = update().setSql("liked = liked + 1").eq("id", id).update();
            // 3.2.保存用户到Redis的set集合  zadd key value score
            if (isSuccess) {
                stringRedisTemplate.opsForZSet().add(key, userId.toString(), System.currentTimeMillis());
            }
        } else {
            // 4.如果已点赞，取消点赞
            // 4.1.数据库点赞数 -1
            boolean isSuccess = update().setSql("liked = liked - 1").eq("id", id).update();
            // 4.2.把用户从Redis的set集合移除
            if (isSuccess) {
                stringRedisTemplate.opsForZSet().remove(key, userId.toString());
            }
        }
        return Result.ok();
    }

    /**
     * 查询博客点赞用户列表
     * @param id 博客ID
     * @return 返回博客点赞数前5的用户信息列表
     */
    @Override
    public Result queryBlogLikes(Long id) {
        String key = BLOG_LIKED_KEY + id;
        // 1.查询top5的点赞用户 zrange key 0 4
        Set<String> top5 = stringRedisTemplate.opsForZSet().range(key, 0, 4);
        if (top5 == null || top5.isEmpty()) {
            return Result.ok(Collections.emptyList());
        }
        // 2.解析出其中的用户id
        List<Long> ids = top5.stream().map(Long::valueOf).collect(Collectors.toList());
        String idStr = StrUtil.join(",", ids);
        // 3.根据用户id查询用户 WHERE id IN ( 5 , 1 ) ORDER BY FIELD(id, 5, 1)
        List<UserDTO> userDTOS = userService.query()
                .in("id", ids).last("ORDER BY FIELD(id," + idStr + ")").list()
                .stream()
                .map(user -> BeanUtil.copyProperties(user, UserDTO.class))
                .collect(Collectors.toList());
        // 4.返回
        return Result.ok(userDTOS);
    }

     /**
     * 保存博客并推送给粉丝
     * @param blog 博客对象，包含博客的基本信息
     * @return Result 返回操作结果，成功时返回博客ID，失败时返回错误信息
     */
    @Override
    public Result saveBlog(Blog blog) {
        // 1.获取登录用户
        UserDTO user = UserHolder.getUser();
        blog.setUserId(user.getId());
        
        // 2.保存探店笔记
        boolean isSuccess = save(blog);
        if(!isSuccess){
            return Result.fail("新增笔记失败!");
        }
        
        // 3.查询笔记作者的所有粉丝 select * from tb_follow where follow_user_id = ?
        List<Follow> follows = followService.query().eq("follow_user_id", user.getId()).list();
        
        // 4.推送笔记id给所有粉丝
        for (Follow follow : follows) {
            // 4.1.获取粉丝id
            Long userId = follow.getUserId();
            // 4.2.推送
            String key = FEED_KEY + userId;
            stringRedisTemplate.opsForZSet().add(key, blog.getId().toString(), System.currentTimeMillis());
        }
        
        // 5.返回id
        return Result.ok(blog.getId());
    }

    /**
     * 查询当前用户关注的博主所发布的博客（滚动分页）
     * 该方法基于 Redis 中存储的用户收件箱（feed）实现滚动分页查询。
     * 收件箱中保存了用户关注的博主发布的博客 ID 和发布时间（作为分数），按时间倒序排列。
     * 每次查询会根据上一次查询的最大时间戳和偏移量获取下一批数据，并更新最小时间戳及新的偏移量。
     * @param max 上次查询的最小时间戳，本次查询将返回早于该时间的博客；首次查询传入 Long.MAX_VALUE 或其他极大值
     * @param offset 当前偏移量，用于跳过相同时间戳的数据，保证分页准确性
     * @return Result 包含博客列表、下次查询的时间戳和偏移量的滚动分页结果
     */
    @Override
    public Result queryBlogOfFollow(Long max, Integer offset) {
        Long userId = UserHolder.getUser().getId(); 
        String key = FEED_KEY + userId; 
        Set<ZSetOperations.TypedTuple<String>> typedTuples = stringRedisTemplate.opsForZSet() 
                .reverseRangeByScoreWithScores(key, 0, max, offset, 2); 
        if (typedTuples == null || typedTuples.isEmpty()) { 
            return Result.ok(); 
        }
        List<Long> ids = new ArrayList<>(typedTuples.size());
        long minTime = 0;
        int os = 1;
        for (ZSetOperations.TypedTuple<String> tuple : typedTuples) {
            ids.add(Long.valueOf(tuple.getValue()));
            long time = tuple.getScore().longValue();
            if(time == minTime){
                os++;
            }else{
                minTime = time;
                os = 1; 
            }
        }

        String idStr = StrUtil.join(",", ids);
        List<Blog> blogs = query().in("id", ids).last("ORDER BY FIELD(id," + idStr + ")").list();

        for (Blog blog : blogs) {
            queryBlogUser(blog);
            isBlogLiked(blog); 
        }

        ScrollResult r = new ScrollResult();
        r.setList(blogs);
        r.setOffset(os);
        r.setMinTime(minTime);

        return Result.ok(r);

    /**
     * 查询博客对应的用户信息并设置到博客对象中
     * @param blog 博客对象，用于获取用户ID并设置用户相关信息
     */
    private void queryBlogUser(Blog blog) {
        Long userId = blog.getUserId();
        User user = userService.getById(userId);
        
        blog.setName(user.getNickName());
        blog.setIcon(user.getIcon());
    }
```

### 3.2 核心实现（MyBatis-Plus + StringRedisTemplate）

```java
@Service
@RequiredArgsConstructor
public class FollowServiceImpl extends ServiceImpl<FollowMapper, Follow>
        implements IFollowService {

    private final StringRedisTemplate stringRedisTemplate;
    private static final String FOLLOWS_KEY = "follows:"; // follows:{uid}
    private static final String FANS_KEY = "fans:";       // fans:{uid}

    @Override
    @Transactional
    public void follow(Long followUserId, boolean isFollow) {
        Long userId = UserHolder.getUser().getId();
        if (isFollow) {
            // 1. 落库（带唯一键，幂等）
            Follow f = new Follow();
            f.setUserId(userId);
            f.setFollowUserId(followUserId);
            this.saveOrUpdate(f, new LambdaQueryWrapper<Follow>()
                    .eq(Follow::getUserId, userId)
                    .eq(Follow::getFollowUserId, followUserId));
            // 2. 写缓存集合
            stringRedisTemplate.opsForSet().add(FOLLOWS_KEY + userId, String.valueOf(followUserId));
            stringRedisTemplate.opsForSet().add(FANS_KEY + followUserId, String.valueOf(userId));
        } else {
            // 1. 先删库
            this.remove(new LambdaQueryWrapper<Follow>()
                    .eq(Follow::getUserId, userId)
                    .eq(Follow::getFollowUserId, followUserId));
            // 2. 再删缓存
            stringRedisTemplate.opsForSet().remove(FOLLOWS_KEY + userId, String.valueOf(followUserId));
            stringRedisTemplate.opsForSet().remove(FANS_KEY + followUserId, String.valueOf(userId));
        }
    }

    @Override
    public boolean isFollow(Long followUserId) {
        Long userId = UserHolder.getUser().getId();
        // 读缓存优先，其次回源 DB
        Boolean isMember = stringRedisTemplate.opsForSet()
                .isMember(FOLLOWS_KEY + userId, String.valueOf(followUserId));
        if (Boolean.TRUE.equals(isMember)) return true;

        long count = this.count(new LambdaQueryWrapper<Follow>()
                .eq(Follow::getUserId, userId)
                .eq(Follow::getFollowUserId, followUserId));
        if (count > 0) {
            // 缓存回填（可加过期）
            stringRedisTemplate.opsForSet().add(FOLLOWS_KEY + userId, String.valueOf(followUserId));
            return true;
        }
        return false;
    }

    @Override
    public List<UserDTO> followCommons(Long otherId) {
        Long userId = UserHolder.getUser().getId();
        String key1 = FOLLOWS_KEY + userId;
        String key2 = FOLLOWS_KEY + otherId;
        // Redis 端求交集（SINTER）
        Set<String> intersect = stringRedisTemplate.opsForSet().intersect(key1, key2);
        if (intersect == null || intersect.isEmpty()) return Collections.emptyList();

        List<Long> ids = intersect.stream().map(Long::valueOf).toList();
        // 批量查用户（保持展示顺序可按关注时间或 ids 顺序）
        List<User> users = userService.listByIds(ids);
        return users.stream().map(UserDTO::from).toList();
    }
}
```

## 四、Feed 流（推模式 & 滚动分页）

### 4.1 发布博文 → 推送到粉丝收件箱

**思路**：作者发布成功后，查出所有粉丝 `fanId`，对每个 `fanId` 往 `ZSET feed:{fanId}` 添加一条记录：`(score=System.currentTimeMillis(), value=blogId)`。

```java
@Service
@RequiredArgsConstructor
public class BlogServiceImpl extends ServiceImpl<BlogMapper, Blog> implements IBlogService {
    private final StringRedisTemplate stringRedisTemplate;
    private final IFollowService followService;
    private static final String FEED_KEY = "feed:"; // feed:{uid}

    @Override
    @Transactional
    public Long saveBlog(Blog blog) {
        Long userId = UserHolder.getUser().getId();
        blog.setUserId(userId);
        // 1. 落库
        this.save(blog);
        Long blogId = blog.getId();

        // 2. 查粉丝（走索引 idx_follow_user）
        List<Long> fanIds = this.baseMapper.queryFanIds(userId); // SELECT user_id FROM tb_follow WHERE follow_user_id=?
        if (fanIds == null || fanIds.isEmpty()) return blogId;

        // 3. 推送到粉丝收件箱（ZSET）
        long now = System.currentTimeMillis();
        ZSetOperations<String, String> zset = stringRedisTemplate.opsForZSet();
        for (Long fanId : fanIds) {
            zset.add(FEED_KEY + fanId, String.valueOf(blogId), now);
        }
        return blogId;
    }
}
```

> 面试补充：
>
> - 粉丝千万级时避免**全量 fanout**：可用**异步 MQ** + **批量/分片**推送，或切换为**拉模式**（时间线按作者聚合），或**冷热分层**（仅近7天活跃粉丝推送，沉默粉丝改为拉取）。
> - 幂等：同一 blogId 对同一 fanId 只添加一次（ZSET 本身对 value+score 组合可视为幂等，或先 `ZSCORE` 判断）。

### 4.2 滚动分页（基于时间戳 + 偏移量）

**为什么不用页码？**
 Feed 是“插队”型数据（新内容随时插入最前），固定页码会重复/漏读。**滚动分页**用上一页最小时间 `minTime(lastId)` + **同分数 offset**保证稳定续读。

#### 请求参数

- `lastId`: 上一次结果最小的 `score`（时间戳）
- `offset`: 与 `lastId` 相同 `score` 的记录数偏移（处理同分数多条）

#### 代码实现

```java
@Data
public class ScrollResult<T> {
    private List<T> list;
    private Long minTime;  // 本次返回中的最小score
    private Integer offset; // 下次查询时的偏移
}

@Service
@RequiredArgsConstructor
public class FeedService {
    private final StringRedisTemplate stringRedisTemplate;
    private final BlogMapper blogMapper;
    private static final String FEED_KEY = "feed:";

    public ScrollResult<BlogDTO> queryFeed(Long lastId, Integer offset) {
        Long userId = UserHolder.getUser().getId();
        String key = FEED_KEY + userId;

        // 1. 构造范围： (min, +inf] 用 ZREVRANGEBYSCORE 降序取
        if (lastId == null || lastId == 0) lastId = Long.MAX_VALUE; // 首次

        // 2. 读取固定大小窗口
        int limit = 10;
        Set<ZSetOperations.TypedTuple<String>> tuples =
                stringRedisTemplate.opsForZSet().reverseRangeByScoreWithScores(
                        key, 0, lastId, offset == null ? 0 : offset, limit);

        if (tuples == null || tuples.isEmpty()) {
            ScrollResult<BlogDTO> empty = new ScrollResult<>();
            empty.setList(Collections.emptyList());
            empty.setMinTime(0L);
            empty.setOffset(0);
            return empty;
        }

        // 3. 解析 blogId、记录本次的 minTime 与同分数数量
        List<Long> ids = new ArrayList<>(tuples.size());
        long minTime = 0;
        int os = 0;
        for (ZSetOperations.TypedTuple<String> t : tuples) {
            ids.add(Long.valueOf(Objects.requireNonNull(t.getValue())));
            long time = Objects.requireNonNull(t.getScore()).longValue();
            if (minTime == 0 || time < minTime) {
                minTime = time;
                os = 1;
            } else if (time == minTime) {
                os++;
            }
        }

        // 4. 批量查 DB，按 ZSET 顺序返回
        String idStr = ids.stream().map(String::valueOf).collect(Collectors.joining(","));
        List<BlogDTO> list = blogMapper.queryByIdsInOrder(idStr); // SELECT ... WHERE id IN (...) ORDER BY FIELD(id, ...)

        ScrollResult<BlogDTO> r = new ScrollResult<>();
        r.setList(list);
        r.setMinTime(minTime);
        r.setOffset(os);
        return r;
    }
}
```

## 五、为什么这么设计（面试可讲的“合理性”）

1. **Redis Set 求交集**：共同关注是典型集合运算，`SINTER` O(N) 级别，远快于 DB 自连接。
2. **推模型 ZSET**：写扩散、读便宜；时间线排序天然由 `score` 保证。
3. **滚动分页**：解决“插队导致的页码不稳定”；`lastId + offset` 兼顾**同分值**稳定性。
4. **一致性顺序**：**先 DB 后 Redis**，失败可重试；对外接口保持幂等。
5. **可扩展性**：
   - 热点作者发文：引入 MQ 批推 / 分片 Key（`feed:{uid % 64}:{uid}`）减少单 Key 热点；
   - 超大粉丝量：活跃粉丝推送，沉默粉丝拉取（混合模式）；
   - 多机：用户维度做**一致性哈希**路由 feed key，提升伸缩性。

## 六、常见坑 & 排查建议

- **共同关注为空**：先检查 `follows:{uid}` 是否构建；导数据时也要同步构建缓存或做懒加载回填。
- **Feed 重复**：使用同一 `blogId` 推送幂等；或消费者端去重（本地 Set + 过期）。
- **顺序混乱**：DB 批量查询要用 `ORDER BY FIELD(id, ...)` 按 ZSET 里的顺序返回。
- **滚动分页漏读**：`lastId` 用本批次**最小 score**，`offset` 统计**等于最小 score**的条数。
- **大 Key**：定期清理 `feed:{uid}` 只保留近 N 天/N 条（如 `ZREMRANGEBYRANK`）。
- **缓存与库不一致**：采用**延迟双删**或**消息补偿**；读多写少场景下影响小，可接受最终一致。

## 七、可以直接用的“标准面试回答模板”

**Q：好友关注与共同关注怎么实现？**
 A：关注关系落 MySQL 表 `tb_follow`，同时在 Redis 维护 `follows:{uid}` 集合。关注/取关时**先操作数据库，再增删 Redis** 保证一致性。共同关注直接用 Redis 的 `SINTER` 求交集，最后按 ID 批量查用户信息即可。

**Q：Feed 流为什么选推模式？如何分页？**
 A：拉模式读放大，推模式把写放大到作者，**更适合读多写少**的时间线。发文后把 `blogId` 以时间戳为 score 推到粉丝的 `feed:{uid}` ZSET。分页使用**滚动分页**：前端携带 `lastId(上次最小时间戳)` 和 `offset(同分数偏移)`，后端用 `ZREVRANGEBYSCORE` 继续读取，避免页码不稳定。

**Q：粉丝很多怎么办？**
 A：引入 MQ 异步化、分片推送；区分活跃粉丝与沉默粉丝（活跃推、沉默拉）；或按业务转为作者分桶拉取混合方案；并对 `feed` 做 TTL 或条数裁剪避免 BigKey。

# 10. 附近商户

> 场景：用户在App/小程序内查看“附近美食/酒吧/KTV”等商户列表，并按距离升序分页展示。技术栈：Spring Boot + MyBatis-Plus + Redis（GEO）。

------

## 1. 核心概念速记

- **Redis GEO 本质**：基于 **Sorted Set** 存储。`member` 是业务ID（如 shopId），`score` 实际是 **经纬度经 geohash 编码后的 52bit 值**。
- **常用命令**
  - `GEOADD key lon lat member [lon lat member ...]`：写入（注意**先经度、后纬度**）。
  - `GEOPOS key member`：查坐标。
  - `GEODIST key m1 m2 [m|km|ft|mi]`：两点距离。
  - `GEOHASH key member`：geohash 字符串（可做粗粒度网格）。
  - `GEOSEARCH key FROMMEMBER <m>|FROMLONLAT <lon> <lat> BYRADIUS <r> m|km [ASC] [WITHDIST] [COUNT n [ANY]]`：**推荐**，6.2+ 统一查询语义；可排序、带距离、限返回条数。
  - `GEOSEARCHSTORE`：把结果落到新key（做快照/二次筛选）。
- **分页要点**：Redis GEO **没有 OFFSET**，只能 `COUNT page*size` 后再在应用层 `subList(from, to)`。
- **距离排序**：`ASC` 按距离近→远；结合 `WITHDIST` 返回距离值。
- **键设计**：`shop:geo:{typeId}`，把「类型」作为**分片维度**，降低一个大Key热点与扫描量。

------

## 2. 业务背景 & 需求拆解

- **功能**：选中商品类别（如“美食”）+ 当前定位（lon/lat）→ 返回附近商户，**按距离升序**，支持**分页**。
- **非功能**：QPS 高、延迟低、可水平扩展；位置数据更新较少（商户坐标变动很少）。
- **为什么选 Redis GEO**：
  - 内存结构 + 近似球面距离算法，**查询快**（毫秒级）；
  - 仅需经纬度与ID即可构建索引；
  - 代码与部署简单，适合“附近搜索”这类读多写少的热点场景。
- **可替代方案**：
  - MySQL GIS（`SPATIAL INDEX` + `ST_Distance_Sphere`）：语义强，但在应用层面更重，且热点查询延迟通常高于内存型缓存。
  - ES geo_* 查询：适合复杂过滤/聚合；运维成本更高。

------

 **易错点**：**经度在前，纬度在后**；member 放 `shopId` 字符串。

## 4. 查询实现（带分页与距离）

### 4.2 Service 实现要点与优化

- 使用 `GEOSEARCH FROMLONLAT (x,y) BYRADIUS r WITHDIST ASC COUNT N`
- **分页**：取前 `current * PAGE_SIZE` 条，然后在内存里 `skip(from)`；
- 拿到 member 列表的 shopId，再 **一次性批量查库**（避免N+1），按返回顺序回填距离。

```java
@Override
    public Result queryShopByType(Integer typeId, Integer current, Double x, Double y) {
        // 1.判断是否需要根据坐标查询
        if (x == null || y == null) {
            // 不需要坐标查询，按数据库查询
            Page<Shop> page = query()
                    .eq("type_id", typeId)
                    .page(new Page<>(current, SystemConstants.DEFAULT_PAGE_SIZE));
            // 返回数据
            return Result.ok(page.getRecords());
        }

        // 2.计算分页参数
        int from = (current - 1) * SystemConstants.DEFAULT_PAGE_SIZE;
        int end = current * SystemConstants.DEFAULT_PAGE_SIZE;

        // 3.查询redis、按照距离排序、分页。结果：shopId、distance
        String key = SHOP_GEO_KEY + typeId;
        GeoResults<RedisGeoCommands.GeoLocation<String>> results = stringRedisTemplate.opsForGeo() // GEOSEARCH key BYLONLAT x y BYRADIUS 10 WITHDISTANCE
                .search(
                        key,
                        GeoReference.fromCoordinate(x, y),
                        new Distance(5000),
                        RedisGeoCommands.GeoSearchCommandArgs.newGeoSearchArgs().includeDistance().limit(end)
                );
        // 4.解析出id
        if (results == null) {
            return Result.ok(Collections.emptyList());
        }
        List<GeoResult<RedisGeoCommands.GeoLocation<String>>> list = results.getContent();
        if (list.size() <= from) {
            // 没有下一页了，结束
            return Result.ok(Collections.emptyList());
        }
        // 4.1.截取 from ~ end的部分
        List<Long> ids = new ArrayList<>(list.size());
        Map<String, Distance> distanceMap = new HashMap<>(list.size());
        list.stream().skip(from).forEach(result -> {
            // 4.2.获取店铺id
            String shopIdStr = result.getContent().getName();
            ids.add(Long.valueOf(shopIdStr));
            // 4.3.获取距离
            Distance distance = result.getDistance();
            distanceMap.put(shopIdStr, distance);
        });
        // 5.根据id查询Shop
        String idStr = StrUtil.join(",", ids);
        List<Shop> shops = query().in("id", ids).last("ORDER BY FIELD(id," + idStr + ")").list();
        for (Shop shop : shops) {
            shop.setDistance(distanceMap.get(shop.getId().toString()).getValue());
        }
        // 6.返回
        return Result.ok(shops);
    }
```

**补充解释**

- `limit(end)` 不是“跳过 from 条”的意思，它只是**取前 N 条**；因此需要在应用层 `subList(from, end)`。
- 距离单位默认米（m），如果前端要 km，可 `/1000.0`。
- `ORDER BY FIELD(id, ...)` 保证与 Redis 返回顺序一致（即按距离排序）。
- 半径 `5km` 可根据城市密度动态调整，或让前端传递。

------

## 5. 典型边界 & 易错点

1. **经纬度顺序**：`GEOADD` 与 `Point` 构造均是 **lon, lat**。
2. **分页性能**：页码越大，`limit(end)` 越大，会拖慢查询。解决思路：
   - 限制最大页（如 50 页）；
   - 结合 **地理栈式分页（带最后一条距离 + member 作为游标）** 做“滚动加载”。
3. **坐标更新**：商户换地址 → 需执行 `GEOADD` 覆盖；或先 `ZREM` 再 `GEOADD`。
4. **多维筛选**：如果再叠加评分、价格等排序，GEO 不支持复合排序 → 先按距离取一小批，再在内存/DB二次排序。
5. **跨城市**：**半径** + **类型Key分片** 是主要“范围剪枝”。如需城市级限制，可在DB层按 cityId 过滤后再写入对应 key。
6. **一致性**：商户基本信息仍在 MySQL，GEO 只做“坐标索引”；主数据变更要同步更新 Redis（事件驱动或定时校准）。

## 7. 与官方代码相比的**关键补充**

- **使用 `GEOSEARCH`（Redis 6.2+）统一替代旧的 `GEORADIUS`**，参数更清晰，也支持 `COUNT`、`ASC`、`WITHDIST` 等。
- **解释分页策略的“LIMIT 不是 OFFSET”**，避免误会。
- **加上 `ORDER BY FIELD`** 保证 DB 批量查询后仍保持 Redis 距离顺序。

- **Key 分片策略建议**：至少 `typeId` 维度；大型城市可叠加 `cityId` 维度减少一个Key过大。

## 8. 面试标准回答模板（1~2 分钟）

> **题目**：如何实现“附近商户按距离排序分页展示”？

1. **思路**：使用 Redis GEO 存商户坐标，以 `shop:geo:{typeId}` 为 Key，将 `shopId` 作为 member。查询时用 `GEOSEARCH FROMLONLAT ... BYRADIUS ... WITHDIST ASC`，取前 `page*size` 条，再在应用层做 `subList` 完成分页。
2. **关键点**：
   - 经纬度顺序是 lon,lat；
   - Redis 没有 OFFSET，只能“取前 N 再截断”；
   - 批量按 id 回库查询并用 `ORDER BY FIELD` 保序；
   - 半径和最大页做保护，避免深分页拖垮性能。
3. **优化**：
   - Key 按 `typeId`/`cityId` 分片；
   - 游标式滚动分页；
   - 商户坐标更新时同步 `GEOADD`。
4. **对比**：MySQL GIS/ES 也能做地理查询，但 Redis GEO 延迟更低、实现更轻，适合读多写少的“附近”类查询。

## 9. 可延展的追问（附要点）

1. **为什么 Redis GEO 不直接支持 OFFSET？**
   - 底层是跳表的有序集合，按距离排序后再偏移会导致较高的遍历成本，官方未提供通用 OFFSET；推荐应用层截断或游标式分页。
2. **如何做“综合排序”（距离 + 评分）？**
   - 先按距离取一小批（如前 200），再在内存按 `f(distance, score)` 计算综合分排序；或预先离线求分在 DB 中排序。
3. **高并发下的缓存穿透/击穿？**
   - GEO 本身只存坐标，不涉及详情；详情可做缓存＋逻辑过期或互斥锁，GEO 查询结果 id 再批量查缓存/DB。
4. **商户坐标更新一致性怎么保证？**
   - 事件驱动：修改商户后发 MQ/订阅，消费者更新 `GEOADD`；或定时校验 DB 与 Redis 坐标差异。
5. **跨城或超大半径的性能问题？**
   - 通过 Key 分片（`cityId`）、半径上限、前N限制，避免扫描过多元素。
6. [Redis GEO 的底层原理是什么？为什么说 GEO 是基于 Sorted Set 实现的？](面试题/)

# 11. 用户签到

> 目标：支持“每日签到、展示当月签到日历、统计连续签到天数/本月累计签到数、发放积分/连续奖励”。
>  方案核心：**Redis Bitmap（位图）** + 少量 MySQL（用于流水/积分账变）。

------

## 1. 背景与痛点

- 早期做法：一张 `tb_sign(uid, date, ...)` 表，**一天一行**。
  - 缺点：行数随用户×天数线性增长；统计“本月/连续天数”需要聚合或多次查询，开销大。
- 位图做法：**一个用户一个月一个 Key**，用 **1 bit 表示某天是否签到**。
  - 优点：**存储极省**（31 天≈4B）、**查询 O(1)**、天然适合“日历 + 连续统计”。

------

## 2. 核心数据结构与 Key 设计

- Key：`sign:uid:{userId}:{yyyyMM}`
- 偏移（offset）：`dayOfMonth - 1`（第 1 天用偏移 0）。
- 命令速记
  - `SETBIT key offset 1` 设置当天已签到（返回值是**旧 bit**，可用于幂等判断）。
  - `GETBIT key offset` 查询当天是否签到。
  - `BITFIELD key GET u{N} 0` 一次取 **[0..N-1] 共 N 位** 的无符号整数。
  - `BITCOUNT key` 统计 1 的个数（本月总签到数）。
  - `BITPOS key 0|1` 查找首个 0/1 的位置（偶尔用于扫描）。

> 为什么选 Bitmap：用位来存日状态，天然契合“日历天粒度”；相较 Set/List/Hash 更省空间、算子丰富且**原子**。

------

## 3. 核心接口（面试级别）

### 3.1 签到（当日）

```java
public Result signToday(Long userId) {
    LocalDate today = LocalDate.now(ZoneId.of("Asia/Shanghai"));
    String key = String.format("sign:uid:%d:%s", userId, DateTimeFormatter.ofPattern("yyyyMM").format(today));
    int offset = today.getDayOfMonth() - 1;

    // SETBIT 原子返回旧值：0->首次签到；1->重复点击
    Boolean already = stringRedisTemplate.opsForValue().setBit(key, offset, true);
    if (Boolean.TRUE.equals(already)) {
        return Result.fail("今日已签到");
    }

    // 可选：设置过期（例如 420 天，保留一年多的查询窗口）
    stringRedisTemplate.expire(key, Duration.ofDays(420));

    // 积分/成长（写 MySQL 流水，避免仅在 Redis）
    // pointService.addPoint(userId, SignPointRule.SIGN_DAILY);

    return Result.ok();
}
```

**补充**

- 为何不用锁？`SETBIT` 是**单条原子命令**，且返回旧值，天然实现幂等。
- 为何要设过期？历史不做长期统计时可控空间；如要年度排行可不设过期或做冷归档。

### 3.2 本月签到日历

```java
public Result monthCalendar(Long userId, YearMonth ym) {
    // 1) 当月天数
    int days = ym.lengthOfMonth();
    // 2) 取整月位图
    String key = String.format("sign:uid:%d:%s", userId, ym.format(DateTimeFormatter.ofPattern("yyyyMM")));
    List<Long> rs = stringRedisTemplate.opsForValue().bitField(
            key,
            BitFieldSubCommands.create().get(BitFieldSubCommands.BitFieldType.unsigned(days)).valueAt(0)
    );
    long bits = (rs == null || rs.isEmpty() || rs.get(0) == null) ? 0L : rs.get(0);

    // 3) 还原为布尔日历（LSB 对应1号）
    List<Boolean> calendar = new ArrayList<>(days);
    for (int i = 0; i < days; i++) {
        calendar.add(((bits >> i) & 1) == 1);
    }
    return Result.ok(calendar); // [true,false,true,...]，前端按日期渲染
}
```

**解释**

- `BITFIELD GET u{days} 0` 一次拿到**从 1 号起的 days 位**；最低位是 1 号。
- 需要把 long 的每一位拆成布尔数组供前端画日历。

### 3.3 本月总签到数

```java
public Result monthCount(Long userId, YearMonth ym) {
    String key = String.format("sign:uid:%d:%s", userId, ym.format(DateTimeFormatter.ofPattern("yyyyMM")));
    Long count = stringRedisTemplate.execute((RedisCallback<Long>) con -> con.bitCount(key.getBytes()));

}
```

> 为何选 `BITCOUNT` 而不是遍历日历？`BITCOUNT` 是 C 层高效指令，对大对象也快。

### 3.4 连续签到天数（截止“今天”）

```java
public Result signStreak(Long userId) {
    LocalDate today = LocalDate.now(ZoneId.of("Asia/Shanghai"));
    YearMonth ym = YearMonth.from(today);
    int day = today.getDayOfMonth();

    String key = String.format("sign:uid:%d:%s", userId, ym.format(DateTimeFormatter.ofPattern("yyyyMM")));

    // 取 [0..day-1] 共 day 位
    List<Long> rs = stringRedisTemplate.opsForValue().bitField(
            key,
            BitFieldSubCommands.create().get(BitFieldSubCommands.BitFieldType.unsigned(day)).valueAt(0)
    );
    long val = (rs == null || rs.isEmpty() || rs.get(0) == null) ? 0L : rs.get(0);

    // 统计“从今天往前连续的 1 的个数”（即统计尾部连续的 1）
    int streak = 0;
    for (int i = 0; i < day; i++) {
        if ((val & 1) == 1) { // 今天为 1 才能累计
            streak++;
            val >>>= 1;
        } else {
            break;
        }
    }

    // 跨月续签（可选）：如果今天为月初且昨天也签到，可再向上月 key 继续接力统计
    // streak += carryFromPrevMonthIfNeeded(userId, today, streak);

    return Result.ok(streak);
}
```

**补充：跨月“连续”**

- 有需求时，**若今天是 1 号且当前 streak 覆盖到 1 号**，就到上月 `key` 再取末尾连续 1；以此**向前“接力”**。
- 上月连续统计也可用 `BITFIELD GET u{N} (offset)` + 尾 1 计数。

## 4. 典型业务规则与实现要点

1. **时区与日期边界**
   - 一律用 `Asia/Shanghai` 或用户绑定的时区；**避免服务器默认时区导致“跨零点错签”**。
2. **补签**
   - 若允许补签：**不能直接改位图**（会影响“当日奖励”逻辑与运营数据）；建议落一张 `tb_sign_patch`，对“连续数”算法可配置“忽略某些 0”。
3. **积分发放**
   - 签到产生的积分/成长建议**落 MySQL 账变**（可用事务 outbox/MQ），防丢失；Redis 仅作为签到状态。
4. **并发 & 幂等**
   - 重复点击由 `SETBIT` 的**旧值返回**天然处理。
5. **大 Key 与过期**
   - 一个用户每月一个 Key，天然拆分；可设置过期天数，例如 420 天（保证最近一年统计）。
6. **统计扩展**
   - **任意区间签到数**：`BITFIELD GET u{len} {offset}` 取出后自己 `bitCount`；
   - **某周签到**：计算本周起止 day-of-month，截取对应位段即可。

------

## 5. 复杂度与存储评估（面试说法）

- **空间**：每月最多 31 bit ≈ 4B / 用户；1,000 万用户 ≈ 40MB（未计 Redis 元数据）——非常省。
- **时间**：签到 `SETBIT` **O(1)**，统计 `BITCOUNT/尾 1 计数` **O(1)**；日历 `BITFIELD` **O(1)**。
- **可靠性**：签到状态保在 Redis，积分流水在 DB；可通过定时任务把 Bitmap 汇总到离线仓库。

------

## 6. 与官方实现相比的**关键补充**

- 利用 `SETBIT` **旧值**实现“幂等 + 防重复提示”。
- 提供了**当月日历**与**跨月连续**的两种实现思路。
- 统一 **Key 规范** 与 **ZoneId**，避免时区 bug。
- 明确“**积分写库，签到在 Redis**”的分层，避免仅内存账。
- 给出**过期策略**和**估算**，方便面试时做容量评估。

------

## 7. 面试标准回答模板（1 分钟）

> **题目**：如何实现高效的用户签到、当月日历和连续天数统计？

- **存储**：用 **Redis Bitmap**，Key= `sign:uid:{uid}:{yyyyMM}`，当月每天 1 bit。
- **签到**：`SETBIT key offset 1`（返回旧值做幂等）。
- **当月累计**：`BITCOUNT key`。
- **当月日历**：`BITFIELD GET u{days} 0` 拿整月，再按位还原。
- **连续天数**：同样 `BITFIELD` 取到**[1..today]**位，**从低位开始统计连续 1**；若需跨月连签，再到上月 key 做接力。
- **细节**：注意 **时区**、**key 过期**、**积分落库**、最大并发（命令原子，O(1)）。
- **评估**：空间极省，千万级用户月级别 < 百 MB；性能毫秒级。

------

## 8. 延展追问 & 思路要点

1. **如何支持“补签但不影响连续奖励”？**
   - 额外记录 `tb_sign_patch`，前端展示日历时把补签覆盖显示；连续奖励计算时**不把补签位当 1**（或策略化开关）。
2. **如何做“任意日期区间的签到数”？**
   - 用 `BITFIELD GET u{len} offset` 拿到位段，应用层 `Long.bitCount()`。
3. **如何做“周连续”或“周奖励”？**
   - 计算本周 offset+len，取段判断是否全 1 或计数≥阈值。
4. **如何防刷？**
   - 登录态校验、设备指纹/频控、同 IP 限流；签到动作绑定用户真实登录态且落库积分。
5. **如何做大促统计 & 排行**？
   - 周/月 `BITCOUNT` + MySQL/OLAP 汇总；也可把 `BITCOUNT` 的结果按用户落入 `zset` 做**在线排行**。

# 12. UV 统计（独立访客）

------

## 1. 概念速记

- **UV (Unique Visitor)**：独立访客量，统计**一天内访问某站点的不同用户数**（按账号/IP/设备去重）。
- **PV (Page View)**：页面浏览量，统计**访问次数**（不去重，刷新也+1）。
  - 举例：同一用户浏览 5 次 → PV=5, UV=1。

> 面试常考：**UV 去重场景下为什么不用 Set？**

------

## 2. 为什么不用 Set 存用户ID？

- **直接存 Redis Set**
  - 优点：精确无误差。
  - 缺点：存储消耗大（比如 1 亿用户，每个 userId 8 字节，占 ~0.8GB，Redis 压力大）。
- **解决思路**：采用近似去重算法。
  - **HyperLogLog（HLL）**：基于概率统计，存储空间固定 ≤ **12KB/16KB**，统计误差 **0.81%**。

**面试答法**：

> 对 UV 统计，精度允许有小误差（0.8% 内），相比存 Set，大幅节省内存，是业界通用方案。

------

## 3. Redis HyperLogLog

- **基于 String 底层实现**，Redis 2.8.9+ 支持。
- **API 命令**：
  - `PFADD key element [element ...]`：添加元素（不返回去重结果，仅返回是否修改）。
  - `PFCOUNT key [key ...]`：返回基数（近似值）。
  - `PFMERGE destkey sourcekey [sourcekey ...]`：合并多个 HLL。

------

## 4. 代码示例（测试百万用户）

```java
@Test
void testHyperLogLog() {
    // 模拟 100 万用户 ID
    String[] users = new String[1000];
    int index = 0;

    for (int i = 1; i <= 1_000_000; i++) {
        users[index++] = "user_" + i;

        // 每 1000 个批量写入一次
        if (i % 1000 == 0) {
            index = 0;
            stringRedisTemplate.opsForHyperLogLog().add("hll1", users);
        }
    }

    // 获取 UV 数
    Long size = stringRedisTemplate.opsForHyperLogLog().size("hll1");
    System.out.println("UV count = " + size);
}
```

**解释**

- **批量提交**：每次存 1000 个，避免单条 `PFADD` 过多网络交互。
- `size()` 内部就是 `PFCOUNT`。
- 实测：真实 100 万，结果 ≈ 992k~1.008M（误差 < 1%）。

------

## 5. 使用场景

- **UV 统计**：日活、月活、活动页独立访客数。
- **大数据场景**：去重统计（手机号、设备ID、IP）。
- **日志分析**：统计独立 IP 数。
- **合并统计**：`PFMERGE` 跨天/跨服务 UV 统计。

------

## 6. 优缺点总结

| 方案        | 精确度    | 内存                 | 使用场景                 |
| ----------- | --------- | -------------------- | ------------------------ |
| Redis Set   | 100%      | 大，随用户数线性增长 | 小规模精确统计           |
| HyperLogLog | 约 ±0.81% | 永远 ≤16KB           | 大规模 UV 去重，实时场景 |

------

## 7. 常见面试问答

1. **Q：HyperLogLog 为什么能用 16KB 表示亿级基数？**
   - A：内部使用稀疏/稠密桶存储 + 基于「最大前导零数」的概率算法，近似估算基数。
2. **Q：HyperLogLog 的误差能接受吗？**
   - A：误差 ≤0.81%，对于 UV 统计这种对精确度要求没那么高的场景完全可接受。
3. **Q：能否统计用户活跃的交集？**
   - A：不支持交集，只支持基数估计。交集/并集需用 `Bitmap` 或 Bloom Filter。
4. **Q：如果我既要 UV 去重又要后续精确分析？**
   - A：在线 UV 用 HLL，离线分析用日志全量数据 + Hive/ES。

------

## 8. 标准回答模板（面试速答）

> **题目**：如何实现高性能 UV 统计？

1. 直接存用户 ID 到 Set 精确统计，但内存消耗大。
2. 采用 Redis HyperLogLog，固定 16KB 内存，可承载上亿用户去重，误差 0.81% 内。
3. API：`PFADD` 插入、`PFCOUNT` 查询、`PFMERGE` 合并。
4. 实测：100 万用户只占 16KB，统计结果在误差范围内。
5. 适用于 UV/日活/月活、独立 IP 等大规模去重场景。

------

## 9. 一句话总结

> **“UV 统计用 HyperLogLog，空间固定 ≤16KB，误差 ≤0.81%，适合大规模独立访客统计。”**

# 项目难点总结

## 一、整体架构难点

- **技术栈**：Spring Boot + MyBatis-Plus + Redis + MySQL + Nginx + MQ（可选扩展 RocketMQ/Kafka）
- **典型业务**：登录 + 商户查询 + 缓存优化 + 秒杀抢购 + 用户签到 + 附近商户 + 达人探店 + UV统计
- **项目定位**：电商类“点评+秒杀”平台，覆盖缓存、高并发、分布式锁、地理位置检索、统计分析等**面试高频考点**。

------

## 二、核心难点模块

### 1. **缓存与一致性**

- **场景**：商户详情、热点数据。
- **难点**：缓存穿透、缓存击穿、缓存雪崩。
- **解决方案**：
  - 穿透 → **布隆过滤器** 或 空值缓存。
  - 击穿 → **逻辑过期**（异步刷新） + **互斥锁**。
  - 雪崩 → **随机过期时间**，热点数据预热。
- **面试延伸**：Redis 缓存一致性策略？（延迟双删 / 异步消息 / Binlog 同步）

------

### 2. **分布式锁**

- **场景**：秒杀场景、库存扣减。
- **难点**：单机锁不可靠，需跨节点同步。
- **实现**：
  - 简单版：`SETNX key value EX seconds` 实现。
  - 高级版：Redisson 提供的 **可重入锁 / 公平锁 / 读写锁 / 信号量**。
- **面试追问**：
  - 如何保证锁不会误删？（加唯一标识 + Lua 脚本解锁）
  - RedLock 算法原理？（多节点半数以上获取成功才算加锁成功）

------

### 3. **全局唯一ID**

- **场景**：订单 ID、秒杀 ID。
- **难点**：MySQL 自增ID有单点问题。
- **方案**：基于 Redis + 时间戳 + 递增序列，保证趋势递增、分布式唯一。
- **面试延伸**：
  - 雪花算法（Snowflake）原理？
  - Redis 自增 ID 的高可用问题？

------

### 4. **秒杀优化**

- **场景**：高并发下的优惠券秒杀。
- **难点**：超卖、重复下单、库存一致性。
- **优化点**：
  - 预减库存（Redis 原子操作）
  - 异步下单（Stream/MQ）
  - 一人一单（Lua 脚本校验）
- **面试延伸**：
  - 为什么用 Redis Stream 而不是消息队列？
  - 如果使用 RocketMQ/Kafka 如何设计消费端幂等？

------

### 5. **附近商户（GEO）**

- **场景**：用户搜索附近美食/KTV。
- **难点**：按经纬度 + 半径搜索，结果按距离排序并分页。
- **方案**：
  - Redis GEO（底层是 ZSet + Geohash 编码）
  - 使用 `GEOSEARCH` + 应用层分页
- **问题点**：Redis GEO 不支持 offset → **取前 N 再截断**。
- **面试延伸**：MySQL GIS vs Redis GEO 的对比？

------

### 6. **用户签到（Bitmap）**

- **场景**：用户连续签到、积分奖励。
- **难点**：高效存储与连续签到统计。
- **方案**：
  - `SETBIT` 标记签到
  - `BITCOUNT` 统计本月总签到
  - `BITFIELD` 提取日历
  - 连续签到：从低位连续统计 1 的数量
- **面试延伸**：跨月连续签到怎么实现？（接力统计）

------

### 7. **UV 统计（HyperLogLog）**

- **场景**：统计独立访客。
- **难点**：去重统计在大规模用户下性能与内存消耗。
- **方案**：
  - Redis HyperLogLog（误差 0.81%，内存固定 16KB）。
- **对比**：
  - Set：精确，耗内存大。
  - HLL：近似，超省空间。
- **面试延伸**：HLL 适合场景？交集能否计算？（不行，只能并集）

------

### 8. **达人探店（Feed 流）**

- **场景**：关注用户后，首页展示关注用户动态。
- **难点**：Feed 流推送/拉取模型的选择。
- **方案**：
  - 拉模式：查询关注用户的内容拼接 → 性能差。
  - 推模式：写动态时推送到粉丝的收件箱（Redis ZSet 按时间排序）。
- **面试延伸**：推/拉模型优缺点？如何应对大 V 问题？
