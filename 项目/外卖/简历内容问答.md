# 拦截器token+ThreadLocal

- **登录模块使用拦截器，自定义线程变量完成用户鉴权，通过 ThreadLocal 进行 token 校验，判断用户是否登录状态，解决了 http 请求无状态的问题。** 

## 项目问答

### 1. **功能是如何设计的？**

**面试问题：** *请解释一下你们项目中使用拦截器和ThreadLocal实现用户鉴权的设计？*

**回答：**
 该系统使用 **拦截器** 来拦截所有的 HTTP 请求，并利用 **ThreadLocal** 来存储当前用户的上下文信息，尤其是用来存储验证后的 token。在请求到达业务逻辑之前，拦截器会验证请求头中的 token 是否有效，若有效，会从 token 中提取用户的 ID 信息。然后将该用户的 ID 存入 **ThreadLocal** 中，这样在当前线程内的任何地方都可以通过 `ThreadLocal` 访问到该用户的信息，确保在整个请求处理过程中都能正确的识别当前用户。

### 2. **流程是什么？**

**面试问题：** *能否详细描述一下整个用户登录模块的流程，从请求到验证用户的过程？*

**回答：**

1. **拦截器触发**：每当一个请求到达时，拦截器首先检查请求头中是否包含 token。
2. **token 校验并提取信息**：如果 token 存在，系统会进行有效性验证。验证通过后，提取 token 中的用户信息（通常是用户 ID）。
3. **使用 ThreadLocal 存储用户信息**：验证成功后，将用户 ID 存储到 `ThreadLocal` 中，这样整个请求过程中都能使用 `ThreadLocal` 获取到当前用户的相关信息。
4. **业务逻辑处理**：在后续的服务层代码中，可以通过 `ThreadLocal` 获取到用户信息，进行权限校验和业务逻辑处理。

### 3. **为什么使用 ThreadLocal？**

**面试问题：** *为什么选择使用 ThreadLocal 来存储用户的上下文信息，而不是直接存储在请求或 session 中？*

**回答：**
 `ThreadLocal` 提供了每个请求线程独立的存储空间，保证了线程安全性。在多线程环境中，它可以确保每个请求对应的用户信息是隔离的，不会互相干扰。与传统的存储在 session 或请求上下文中不同，`ThreadLocal` 在性能和内存管理上更有优势，因为它避免了不必要的对象传递，也避免了 session 在分布式环境中的同步问题。

### 4. **可以替换掉 ThreadLocal 吗？**

**面试问题：** *是否有其他技术可以替代 ThreadLocal 来实现类似的功能？*

**回答：**
 除了 `ThreadLocal`，我们也可以使用 **Spring Security** 的上下文机制，它也能够实现用户信息的存储和访问。Spring Security 提供的 `SecurityContextHolder` 类可以用来存储用户的认证信息，不过在一些轻量级的应用中，直接使用 `ThreadLocal` 更加简便且效率高。此外，还可以通过 **请求参数** 或 **会话（session）** 来传递用户信息，但这会增加服务器的负担，并可能导致跨服务器的同步问题，尤其是在分布式环境中。

### 5. **代码如何实现？**

**面试问题：** *请展示一下这段认证模块的核心代码实现？*

**回答：**以下是ThreadLocal存储用户信息相关的完整代码：

> **1. 核心实现原理**
>
> - 项目通过**自定义ThreadLocal**配合**JWT拦截器**，优雅地解决了HTTP请求无状态的问题
>
> - 使用两个关键拦截器分别处理管理端和用户端的身份验证
>
> **2. ThreadLocal实现**
> 
>- 在`BaseContext`类中定义`ThreadLocal<Long> threadLocal` 对象存储线程上下文用户ID
> 
>- 提供三个核心方法：
>   - `setCurrentId(Long id)`: 设置当前线程的用户ID
>   - `getCurrentId()`: 获取当前线程的用户ID
>   - `removeCurrentId()`: 移除当前线程的用户ID
> - 每个HTTP请求线程拥有独立的用户ID存储，避免了线程安全问题
> 
> **3. JWT拦截器配置**
>
> - **管理端拦截器**：`JwtTokenAdminInterceptor`
>  - 拦截路径：`/admin/**`
>   - 排除路径：`/admin/employee/login`
>   - 解析token获取`EMP_ID`并存入ThreadLocal
> 
> - **用户端拦截器**：`JwtTokenUserInterceptor`
>  - 拦截路径：`/user/**`
>   - 排除路径：`/user/user/login`、`/user/shop/status`
>   - 解析token获取`USER_ID`并存入ThreadLocal
> 
> **4. Token生成与验证流程**
>
> - **登录时生成token**：
>  - 管理员登录：`EmployeeController.login()`生成包含员工ID的token
>   - 用户登录：`UserController.login()`生成包含用户ID的token
>   - 使用`JwtUtil.createJWT()`方法创建token并使用，设置过期时间
> 
> - **请求时验证token**：
>  - 拦截器从请求头获取token
>   - 调用`JwtUtil.parseJWT()`解析token
>   - 验证通过后将用户ID设置到ThreadLocal中
>   - 验证失败返回401状态码
> 
> **5. 实际应用**
>
> - 业务层通过`BaseContext.getCurrentId()`获取当前登录用户ID
>- 无需在每个方法中传递用户身份信息
> - 保证了用户操作的安全性和准确性
> - 解决了HTTP请求无状态的特性，实现了有状态的用户会话管理

1. **BaseContext.java (ThreadLocal的核心实现)**

```java
package com.sky.context;

public class BaseContext {

    public static ThreadLocal<Long> threadLocal = new ThreadLocal<>();

    public static void setCurrentId(Long id) {
        threadLocal.set(id);	
    }

    public static Long getCurrentId() {
        return threadLocal.get();
    }

    public static void removeCurrentId() {
        threadLocal.remove();
    }
}
```

**JwtTokenAdminInterceptor.java (管理员JWT拦截器)**

```java
package com.sky.interceptor;

/**
 * jwt令牌校验的拦截器
 */
@Component
@Slf4j
public class JwtTokenAdminInterceptor implements HandlerInterceptor {

    @Autowired
    private JwtProperties jwtProperties;

    /**
     * 校验jwt
     *
     * @param request
     * @param response
     * @param handler
     * @return
     * @throws Exception
     */
    public boolean preHandle(HttpServletRequest request, HttpServletResponse response, Object handler) throws Exception {
        //判断当前拦截到的是Controller的方法还是其他资源
        if (!(handler instanceof HandlerMethod)) {
            //当前拦截到的不是动态方法，直接放行
            return true;
        }

        //1、从请求头中获取令牌
        String token = request.getHeader(jwtProperties.getAdminTokenName());

        //2、校验令牌
        try {
            log.info("jwt校验:{}", token);
            Claims claims = JwtUtil.parseJWT(jwtProperties.getAdminSecretKey(), token);
            Long empId = Long.valueOf(claims.get(JwtClaimsConstant.EMP_ID).toString());
            log.info("当前员工id：", empId);
            BaseContext.setCurrentId(empId);
            //3、通过，放行
            return true;
        } catch (Exception ex) {
            //4、不通过，响应401状态码
            response.setStatus(401);
            return false;
        }
    }
}
```

4. **WebMvcConfiguration.java (拦截器配置)**

```java
package com.sky.config;

/**
 * 配置类，注册web层相关组件
 */
@Configuration
@Slf4j
public class WebMvcConfiguration extends WebMvcConfigurationSupport {

    @Autowired
    private JwtTokenAdminInterceptor jwtTokenAdminInterceptor;
    @Autowired
    private JwtTokenUserInterceptor jwtTokenUserInterceptor;

    /**
     * 注册自定义拦截器
     * @param registry
     */
    protected void addInterceptors(InterceptorRegistry registry) {
        log.info("开始注册自定义拦截器...");
        registry.addInterceptor(jwtTokenAdminInterceptor)
                .addPathPatterns("/admin/**")
                .excludePathPatterns("/admin/employee/login");

        registry.addInterceptor(jwtTokenUserInterceptor)
                .addPathPatterns("/user/**")
                .excludePathPatterns("/user/user/login")
                .excludePathPatterns("/user/shop/status");
    }
}
```

**工作流程总结：**

1. 用户/员工登录成功后，系统生成包含用户ID的JWT令牌
2. 后续请求携带令牌，JWT拦截器从令牌中解析出用户ID

5. 请求结束后，应该在拦截器的postHandle方法中调用BaseContext.removeCurrentId()清理ThreadLocal，防止内存泄漏

### 6. **如何确保线程安全？**

**面试问题：** *如何保证 `ThreadLocal` 在多线程环境下的安全性？*

**回答：**
 `ThreadLocal` 自身通过将每个线程的变量保存在本地内存中，避免了不同线程之间的数据竞争，因此可以确保线程安全。然而，使用完 `ThreadLocal` 后需要及时清除其中存储的用户信息，以避免内存泄漏。在请求结束时，可以调用 `UserContext.clear()` 来清除当前线程中的用户信息。

### 7. **如何处理多用户并发请求时的性能瓶颈问题？**

**面试问题：** *在高并发的环境下，使用 `ThreadLocal` 存储用户信息是否会引发性能问题？如何优化这个性能瓶颈？*

**回答：**
 `ThreadLocal` 本身是线程安全的，但在高并发环境下，每个线程都有自己独立的存储空间，这可能导致内存消耗增大。如果线程数非常多，内存消耗会显著增加。此外，过多的线程切换和上下文切换也可能影响性能。为了优化：

1. **线程池**：使用线程池来限制线程的数量，避免过多线程创建带来的性能损耗。
2. **清理 ThreadLocal**：及时清除 `ThreadLocal` 中的数据，避免内存泄漏和不必要的资源占用。
3. **局部缓存**：对于需要频繁访问的用户信息，可以考虑缓存机制，减少每次请求都从数据库或 token 中获取信息的操作。
4. **异步请求**：对于不需要同步执行的部分，采用异步请求处理，减少主线程的阻塞。

### 8. **如何确保用户信息不被篡改？**

**面试问题：** *在 token 校验和用户信息提取的过程中，如何确保传输过程中的用户信息不被篡改？*

**回答：**
 为了确保用户信息不被篡改，建议采取以下措施：

1. **Token 加密**：使用 **JWT（JSON Web Token）** 或 **OAuth** 等安全协议对 token 进行签名和加密，确保 token 在传输过程中不可篡改。签名部分使用强加密算法（如 HMAC SHA256）确保数据完整性。
2. **HTTPS 加密传输**：确保所有请求都通过 HTTPS 协议传输，这样可以防止中间人攻击和数据篡改。
3. **Token 过期机制**：设置合理的 token 过期时间，并在用户身份验证时每次生成新的 token，避免 token 被盗用后长时间有效。
4. **双重验证**：除了使用 token，还可以在敏感操作中加入二次验证（如验证码、短信验证等），增加安全性。

### 9. **如何防止用户在一个 token 下被多个设备同时登录？**

**面试问题：** *如何处理用户 token 被多个设备同时使用的情况？如果希望一个用户只允许一个设备登录，应该如何设计？*

**回答：**
 为了防止 token 被多个设备同时使用，可以采用以下几种策略：

1. **每个设备唯一 token**：为每个设备分配唯一的 token，例如基于设备的唯一标识符（如设备 ID、MAC 地址等）生成不同的 token。当同一用户在多个设备上登录时，生成多个独立的 token，而不是复用一个 token。
2. **服务器端存储设备信息**：将每个用户的登录状态（包括设备 ID、登录时间等）保存在服务器端，并在验证 token 时检查是否有其他设备正在使用该用户的 token。
3. **单点登录（SSO）机制**：实现单点登录机制，在用户在新设备上登录时，先将其他设备上的 token 标记为失效，确保用户只能在一个设备上登录。

### 10. **Token 验证失败时，如何处理错误响应？**

**面试问题：** *在 token 验证失败时，如何设计错误处理和响应机制？*

**回答：**
 当 token 验证失败时，应该提供清晰、友好的错误响应，以便客户端进行正确的处理。设计错误处理机制时，可以考虑以下几点：

1. **统一错误处理**：使用全局异常处理器（如 Spring 的 `@ControllerAdvice`）统一捕获 token 验证失败的异常（如 `ExpiredJwtException` 或 `SignatureException`），并返回标准化的错误响应。

2. **明确错误信息**：返回错误响应时，除了状态码外，还需要返回明确的错误信息，例如：

   ```json
   {
     "status": 401,
     "message": "Token 已过期，请重新登录",
     "errorCode": "TOKEN_EXPIRED"
   }
   ```

3. **提供刷新 token 的机制**：如果采用 JWT，可以考虑设计 **刷新 token** 的机制，当主 token 过期时，客户端可以通过刷新 token 获取新的主 token，避免用户频繁登录。

4. **记录和监控**：在出现 token 错误时，记录日志并进行监控，以便及时发现和解决潜在的安全问题。

### 11. **如何实现 Token 的黑名单机制？**

**面试问题：** *如何设计 token 的黑名单机制，确保某些 token 在被吊销后无法再次使用？*

**回答：**
 为了确保某些 token 在吊销后无法再次使用，可以实现 **token 黑名单** 机制。常见的设计方案包括：

1. **存储黑名单**：使用 Redis 或数据库存储被吊销的 token 列表，每次请求验证 token 时，先检查该 token 是否在黑名单中。
2. **Redis 存储**：利用 Redis 的快速读写特性，将失效的 token 存入 Redis，并设置较短的过期时间（例如 token 的生命周期）。当用户登出或密码修改时，可以将当前的 token 加入黑名单。
3. **过期时间检查**：除了黑名单机制，还可以结合 **短时间内的 token 过期时间**，确保即使攻击者获得了 token，也能在短时间内被自动失效。
4. **高效查找**：使用哈希数据结构来存储黑名单，保证查询操作的高效性。

### 12. **如何设计 Token 刷新机制，避免频繁登录？**

基于 Redis + SpringCache 优化商品模块缓存，降低数据库访问压力，提高了系统查询性能和响应速度。  

**面试问题：** *如何设计一个有效的 Token 刷新机制，避免用户每次登录时都需要重新认证？*

**回答：**
 **Token 刷新机制** 主要通过两个 token 实现：**访问 token** 和 **刷新 token**。设计方案如下：

1. **访问 Token**：每次用户登录时，生成一个访问 token，它的有效期较短（如 15 分钟），用于进行日常的身份验证。
2. **刷新 Token**：在用户登录时，还需要生成一个刷新 token，它的有效期较长（如 7 天）。当访问 token 过期时，客户端可以通过刷新 token 获取新的访问 token，无需重新登录。
3. **刷新流程**：
   - 用户的访问 token 过期时，客户端会使用刷新 token 向服务器请求新的访问 token。
   - 服务器验证刷新 token 是否有效，如果有效，则颁发新的访问 token。
   - 刷新 token 过期时，要求用户重新登录。
4. **刷新 token 吊销机制**：如果用户登出或密码更改时，应当将刷新 token 吊销，从而强制用户重新登录。

通过这种方式，用户不需要频繁登录，同时又能确保系统的安全性。

## 延伸场景

### 场景一：**高并发下的 Token 校验性能问题**

**问题：**
 在电商平台中，用户量和请求量极大，系统需要高效的鉴权机制来确保用户请求的合法性。你采用了 **ThreadLocal** 来存储每个请求的用户信息（如用户 ID），并通过拦截器配合 **Token 校验** 来判断用户是否已登录。但在高并发情况下，大量请求同时到达，可能导致 **ThreadLocal** 的性能瓶颈或线程隔离问题。你如何保证高并发下的鉴权性能，避免系统性能下降？

**回答：**
 为确保高并发下的鉴权性能，可以采取以下措施：

1. **线程池隔离**：通过使用线程池来处理每个请求，确保每个请求的 ThreadLocal 数据不相互干扰。使用合适的线程池配置来避免线程过多造成系统资源的耗尽。
2. **减少 ThreadLocal 使用的场景**：对于一些不需要频繁变动的用户信息（如用户角色、权限等），尽量避免使用 ThreadLocal 存储。可以考虑使用 **缓存** 或 **请求上下文** 来存储这些信息，减少 ThreadLocal 的压力。
3. **异步处理**：对于一些非实时的鉴权操作（如频繁的权限检查），可以将这些操作放入异步任务中处理，避免同步任务造成主线程阻塞。
4. **Token 验证优化**：Token 校验可以通过 **缓存机制** 来优化。将已经验证过的 Token 存入缓存（如 Redis），下次相同的 Token 请求直接从缓存中读取验证结果，避免每次都进行数据库查询或 Token 解码。
5. **WebSocket 或长连接优化**：在高并发环境下，如果有使用 WebSocket 或长连接的场景，可以考虑对这些连接进行定期的 Token 校验，避免每次请求都进行同步校验。

------

### 场景二：**跨服务鉴权问题**

**问题：**
 在微服务架构中，系统需要跨多个服务进行用户身份验证。例如，用户登录后，通过鉴权拦截器验证 Token 来判断是否已登录，并获取用户信息，但在微服务架构中，每个服务都需要通过独立的拦截器进行鉴权。如何确保跨多个服务时的用户信息一致性，避免在每个服务中都进行重复的 Token 校验？

**回答：**
 为了确保跨服务时用户信息的一致性，避免重复的 Token 校验，可以采用以下解决方案：

1. **统一认证服务**：在微服务架构中，可以设计一个统一的 **认证服务**（例如，基于 OAuth2 的认证服务），所有微服务都通过调用认证服务来进行 Token 校验和用户信息获取。认证服务会验证用户 Token，并返回用户信息（如用户 ID、角色等）。其他服务只需要从认证服务获取用户信息，避免在每个服务中进行 Token 校验。
2. **JWT（JSON Web Token）**：使用 **JWT** 来进行跨服务认证。每个微服务可以通过解析 JWT 中的 Payload 来获取用户的身份信息和权限，无需每个服务单独查询数据库或进行 Token 校验，减少性能开销。
3. **API Gateway 鉴权**：在微服务架构中，通常会使用 **API Gateway** 来做统一的入口，所有请求首先通过 Gateway，Gateway 会做 Token 校验和用户鉴权，然后将用户信息附带到请求头或请求体中，传递到后端服务。这样每个微服务都可以从请求中获取用户信息，避免重复验证。
4. **共享缓存**：在多个微服务间共享缓存（如 Redis）来存储已验证的用户信息，避免重复的 Token 校验。每个服务可以从缓存中读取用户信息，减少了 Token 解码的频率和延迟。

### 场景三：**ThreadLocal 的内存泄漏问题**

**问题：**
 在长时间运行的系统中，使用 **ThreadLocal** 存储用户鉴权信息（如用户 ID）可能会导致内存泄漏，特别是在多线程环境中，ThreadLocal 变量在请求结束后没有及时清除，导致对象无法回收。你如何防止 **ThreadLocal** 引发内存泄漏问题，确保系统的稳定性？

**回答：**
 为了避免 **ThreadLocal** 引发内存泄漏，可以采取以下措施：

1. **手动清除 ThreadLocal**：每次请求处理完毕后，显式调用 `ThreadLocal.remove()` 清除存储在 ThreadLocal 中的用户信息，确保线程能够正确回收资源。尤其是在请求结束或线程处理完任务后，及时清理 ThreadLocal 中的数据。

   ```
   @After
   public void clearThreadLocal() {
       UserContext.clear(); // 手动清除ThreadLocal中的数据
   }
   ```

2. **使用 ThreadPoolExecutor 配合 ThreadLocal**：如果系统中使用了线程池来处理请求，确保线程池的线程在处理请求时不会长时间保持用户信息。可以通过自定义的线程池或任务执行器，确保每个任务执行结束后清除其对应的 ThreadLocal 数据。

3. **使用拦截器管理 ThreadLocal 生命周期**：可以通过 **拦截器** 来确保每个请求的开始和结束时都处理 ThreadLocal 数据。请求开始时设置 ThreadLocal 数据，请求结束时清理数据，避免线程对象一直保持对用户信息的引用，造成内存泄漏。

4. **监控与调试**：定期进行 **内存泄漏检测**，通过 **JVM 内存分析工具**（如 **jvisualvm** 或 **YourKit**）检查是否存在 ThreadLocal 导致的内存泄漏问题，及时发现并处理。

5. **使用其他替代方案**：如果 ThreadLocal 的管理较复杂，考虑使用其他方法来存储用户上下文信息，例如将用户信息存储在请求对象中或使用 **请求上下文**，这可以在拦截器和过滤器中传递，而不依赖于 ThreadLocal。

# Redis + SpringCache 优化缓存

- **基于 Redis + SpringCache 优化商品模块缓存，降低数据库访问压力，提高了系统查询性能和响应速度。**  

## 项目问答

### 1. **这个功能是怎么设计的？**

**面试问题：** *请简要描述一下商品模块缓存优化的设计？你是如何使用 Redis 和 SpringCache 来优化查询性能的？*

**回答：**
 为了降低数据库的访问压力并提高查询性能，我们使用了 **Redis** 作为缓存层，并结合 **SpringCache** 来管理缓存。设计思路是：

1. 商品模块的查询请求首先会尝试从 Redis 中获取缓存数据，避免每次都访问数据库。
2. 如果缓存中没有数据，则从数据库查询商品信息，并将查询结果放入 Redis 中，设置合理的缓存过期时间，确保缓存数据的时效性。
3. 我们使用 **SpringCache** 来简化缓存的管理，通过注解实现缓存的存取操作，减少手动管理的复杂度。
4. 还设计了缓存失效的策略，例如商品信息更新时，及时清除相关的缓存数据，避免过期缓存的影响。

### 2. **流程是什么？**

**面试问题：** *缓存优化的具体流程是怎样的？能否详细描述一下从请求到响应的完整流程？*

**回答：**

1. **用户请求到达**：当用户请求商品数据时，首先会触发商品查询功能。
2. **查询缓存**：系统通过 **SpringCache** 检查 Redis 中是否存在缓存。如果缓存中存在数据，则直接从 Redis 返回，避免了数据库查询。
3. **缓存不存在**：如果 Redis 缓存未命中，则查询数据库获取商品数据。
4. **更新缓存**：数据库查询后，将数据放入 Redis 中，并设置合理的过期时间（例如 5 分钟）。下次相同的查询请求就会命中缓存。
5. **返回响应**：最终，将缓存数据或数据库查询结果返回给前端。

    private DishService dishService;

     * 根据分类id查询菜品
     *
     * @param categoryId
     * @return
     */

    @ApiOperation("根据分类id查询菜品")

        // 手动控制   查询查询redis中是否存在菜品数据
        List<DishVO> list = (List<DishVO>) redisTemplate.opsForValue().get(key);	
        if(list != null && list.size() > 0){
            //如果存在，直接返回，无须查询数据库

        }

        Dish dish = new Dish();

        //如果不存在，查询数据库，将查询到的数据放入redis中
        list = dishService.listWithFlavor(dish);
        redisTemplate.opsForValue().set(key, list);

        return Result.success(list);
    }

### 5. **有没有别的技术替换？**

**面试问题：** *除了 Redis + SpringCache 以外，你是否考虑过其他缓存技术？如果有，为什么最终选择了 Redis？*

**回答：**
 除了 Redis，我们还可以考虑使用以下几种技术：

1. **本地缓存（如 Guava Cache）**：适用于单机应用，避免了网络延迟的开销，但它的缺点是无法在分布式环境下共享缓存，并且数据丢失时无法恢复。
2. **Memcached**：也可以作为缓存解决方案，具有较高的性能，适合大规模缓存，但它的功能相对较少，没有 Redis 丰富的特性（如持久化和数据结构支持）。
3. **数据库查询缓存**：数据库本身可以配置缓存，例如 MySQL 的查询缓存，但它的粒度较粗，无法像 Redis 那样细粒度地控制缓存数据，并且通常无法适应频繁的缓存更新。

**最终选择 Redis 的原因**：

- Redis 是一个高性能的分布式缓存，支持丰富的数据结构（如字符串、哈希、集合等），非常适合我们这种需要快速读写并且有复杂数据结构需求的场景。
- Redis 提供了 **持久化机制**，在系统重启时可以恢复缓存，避免了丢失数据的风险。
- 它也有 **集群化** 支持，能够在高并发场景下扩展应用。

### 6. **如何保证缓存一致性？**

**面试问题：** *你是如何保证缓存和数据库之间的一致性的？是否有过期或更新策略？*

**回答：**
 为了保证缓存和数据库之间的数据一致性，我们设计了以下几种策略：

1. **缓存过期时间**：为缓存设置合理的过期时间（例如 5 分钟）。当缓存过期时，下一次请求会重新查询数据库并更新缓存，避免缓存和数据库的数据不一致。
2. **缓存更新策略**：当商品信息更新时，我们使用 `@CachePut` 注解来更新缓存，确保缓存中的数据与数据库同步。删除商品时，使用 `@CacheEvict` 注解清除缓存，确保不会返回已经删除的数据。
3. **主动清除缓存**：对于一些敏感数据（如库存量、商品价格等），在更新数据库时会主动清除缓存，防止缓存中的数据和数据库中的数据不同步。
4. **缓存穿透**：使用布隆过滤器（Bloom Filter）（或者缓存空值）防止无效的查询请求直接访问数据库，避免频繁的缓存失效造成数据库负载。

### 7. **如何处理缓存穿透问题？**

**面试问题：** *在你使用 Redis 缓存商品数据时，如何避免缓存穿透的问题？*

**回答：**
 缓存穿透指的是请求查询的数据在缓存中不存在且数据库中也没有，导致每次都查询数据库并增加数据库负担。为解决这个问题，常见做法是：

1. **使用布隆过滤器（Bloom Filter）**：布隆过滤器能有效避免查询无效数据。我们可以将商品 ID 预先存储到布隆过滤器中，每次查询时，先通过布隆过滤器检查商品是否存在。如果商品不存在，直接返回空数据，避免不必要的数据库查询。
2. **缓存空值**：对于查询结果为空的数据，可以将空值缓存一段时间，防止频繁查询数据库。例如，某些商品被删除或不存在时，返回空数据，并设置短时间的缓存过期，避免每次都查询数据库。
3. **避免过度缓存无效数据**：只缓存有效且常访问的数据，避免缓存过期的或无效的内容。

### 8. **如何处理缓存雪崩问题？**

**面试问题：** *在高并发情况下，多个缓存同时失效会导致缓存雪崩，你是如何避免这种情况的？*

**回答：**
 缓存雪崩是指在高并发情况下，多个缓存同时失效，导致大量请求直接访问数据库，增加数据库的负载。为避免缓存雪崩，可以采取以下策略：

1. **设置不同的过期时间**：对于不同的缓存数据，设置不同的过期时间，避免所有缓存数据在同一时刻失效。可以采用随机的过期时间，增加缓存失效的时间差。
2. **使用热点数据缓存预热**：对于频繁访问的数据，可以在系统启动时提前加载缓存，避免缓存首次失效时直接访问数据库。
3. **引入缓存更新机制**：使用缓存的更新机制（如定时刷新、逻辑过期+异步刷新）代替直接失效，防止缓存数据集中失效时造成的流量激增。

### 9. **如何确保 Redis 集群在高并发场景下的稳定性和高可用性？**

**面试问题：** *当你在系统中使用 Redis 集群时，如何确保 Redis 在高并发场景下的稳定性和高可用性？*

**回答：**
 为了确保 Redis 在高并发场景下的稳定性和高可用性，我们采取了以下措施：

2. **启用 Redis 哨兵模式**：通过 Redis 哨兵机制实现高可用性，确保当主节点出现故障时，能够自动切换到从节点，保证系统的持续可用。
3. **连接池管理**：使用 Redis 连接池（如 Jedis 或 Lettuce）来管理 Redis 连接，避免每次请求都重新建立连接，减少连接开销，提高并发处理能力。

### 10. **如何设计缓存更新策略，保证数据一致性？**

**面试问题：** *在数据库更新时，如何设计缓存的更新策略以确保缓存和数据库之间的一致性？*

**回答：**
 为了保证缓存和数据库的一致性，我们采用以下几种策略：

1. **写入时更新（Cache Aside Pattern）**：在数据更新时，先更新数据库，然后更新或删除缓存中的数据。这样可以确保缓存中始终存储最新的数据。
2. **过期时间管理**：为缓存设置合理的过期时间，避免缓存数据长期存留。当数据更新时，清除过期数据，强制下一次查询从数据库重新获取并更新缓存。
3. **使用 `@CachePut` 注解**：SpringCache 提供的 `@CachePut` 注解可以在数据更新时强制更新缓存，确保缓存中的数据与数据库保持一致。
4. **事务级缓存控制**：在数据库操作时，将缓存操作与数据库操作放在同一事务中，确保数据一致性，防止数据库更新成功而缓存未更新的情况。

**2. WebSocket服务实现 (WebSocketServer.java)**

```java
package com.sky.websocket;

/**
 * WebSocket服务
 */
@Component
@ServerEndpoint("/ws/{sid}")
public class WebSocketServer {

    //创建一个HashMap 存放会话对象（客户端会话连接）
    private static Map<String, Session> sessionMap = new HashMap();

    /**
     * 连接建立成功调用的方法
     */
    @OnOpen
    public void onOpen(Session session, @PathParam("sid") String sid) {
        System.out.println("客户端：" + sid + "建立连接");
        sessionMap.put(sid, session);
    }

    /**
     * 收到客户端消息后调用的方法
     *
     * @param message 客户端发送过来的消息
     */
    @OnMessage
    public void onMessage(String message, @PathParam("sid") String sid) {
        System.out.println("收到来自客户端：" + sid + "的信息:" + message);
    }

    /**
     * 连接关闭调用的方法
     *
     * @param sid
     */
    @OnClose
    public void onClose(@PathParam("sid") String sid) {
        System.out.println("连接断开:" + sid);
        sessionMap.remove(sid);
    }

    /**
     * 群发
     *
     * @param message
     */
    public void sendToAllClient(String message) {
        // 用集合存储所有连接
        Collection<Session> sessions = sessionMap.values();
        for (Session session : sessions) {
            try {
                //服务器向客户端发送消息
                session.getBasicRemote().sendText(message);
            } catch (Exception e) {
                e.printStackTrace();
            }
        }
    }
}
```

### 5. **有没有别的技术替换？**

**面试问题：** *除了 WebSocket，还有哪些技术可以用来实现订单状态的实时推送？为什么选择 WebSocket 而不是其他技术？*

**回答：**
 除了 WebSocket，还可以使用以下技术来实现实时推送：

1. **HTTP 长轮询**：客户端定期发送请求询问服务器是否有新的数据更新。虽然能提供实时性，但相对于 WebSocket，长轮询会导致大量的 HTTP 请求和响应，效率较低且对服务器资源要求较高。
2. **Server-Sent Events（SSE）**：SSE 是一种服务器推送数据到客户端的技术，基于 HTTP 协议，通过事件流将信息推送到浏览器。它比长轮询更高效，但仅支持单向通信，不如 WebSocket 支持双向通信。
3. **Push Notification（推送通知）**：适用于移动端，系统可以通过推送服务将订单状态更新消息推送到用户的手机或桌面端。但推送通知需要额外的第三方服务支持，并且需要处理复杂的推送机制。

**为什么选择 WebSocket**：

- WebSocket 提供了双向通信和持久连接，能够实时且高效地推送数据。
- 对比长轮询和 SSE，WebSocket 更适合频繁的数据推送和交互应用，尤其是在订单状态更新这种需要快速响应的场景。

### 6. **如何保证 WebSocket 的连接稳定性和高可用性？**

**面试问题：** *在高并发的场景下，如何保证 WebSocket 连接的稳定性和高可用性？*

**回答：**
 为确保 WebSocket 连接的稳定性和高可用性，我们可以采取以下措施：

1. **负载均衡**：通过 **Nginx** 或其他负载均衡器，确保多个 WebSocket 服务器实例之间的请求分配和高可用性。如果一个 WebSocket 服务节点宕机，负载均衡器会自动将请求转发到其他节点。
2. **Redis Pub/Sub**：在多服务器环境中，通过 Redis 的发布/订阅功能来实现消息的广播。这样，当某个服务器推送消息时，可以通过 Redis 将消息推送到所有连接的 WebSocket 实例。
3. **心跳检测**：定期发送心跳包来检查 WebSocket 连接是否正常，及时关闭异常连接并重连。可以通过在 WebSocket 协议中设置心跳帧来保持连接活跃。
4. **断线重连机制**：在客户端和服务端都实现断线重连机制，确保在网络中断或服务重启时，WebSocket 连接能够恢复。

这些措施能有效提高系统的稳定性和可用性，尤其在高并发和分布式环境下，保证用户和商户端实时接收订单状态更新。

### 8. **如何保证消息的可靠性，防止消息丢失或重复推送？**

1. **消息确认机制**：实现消息发送后的确认机制，客户端在接收到消息后会返回一个确认响应。如果服务器没有收到确认响应，可以重新推送消息。
2. **消息队列**：将需要推送的消息存入消息队列（如 Kafka、RabbitMQ），确保消息在推送过程中不会丢失。如果 WebSocket 连接断开，服务器可以从消息队列中重新推送未成功的消息。
3. **幂等性设计**：确保每条消息的发送操作是幂等的，即即使重复推送也不会对系统状态产生不一致的影响。这通常可以通过在消息中加入唯一标识符来实现。

5. **事务管理**：对于重要的操作（如订单状态变更），可以将**消息发送操作和数据库更新操作**放入同一个事务中，确保一致性。

### 10. **WebSocket 消息推送的性能瓶颈有哪些？如何优化？**

**面试问题：** *WebSocket 消息推送可能遇到的性能瓶颈有哪些？你是如何优化这些瓶颈的？*

**回答：**
 WebSocket 消息推送的性能瓶颈主要集中在以下几个方面：

1. **连接数过多**：每个 WebSocket 连接都需要占用一定的资源，如果连接数非常大，服务器可能会遇到性能瓶颈。为此，我们可以使用 **连接池管理**，并通过负载均衡分担压力。
2. **消息处理延迟**：当服务器需要推送大量消息时，可能会产生处理延迟。为此，可以使用 **异步消息处理**（如基于 Netty 的事件驱动模型）来提高并发处理能力。
3. **广播性能问题**：当消息需要推送到多个客户端时，可能会出现广播的延迟和性能问题。使用 **Redis Pub/Sub** 或 **消息队列** 来**分发消息**，可以确保多个 WebSocket 实例之间共享消息，优化广播性能。
4. **网络带宽**：如果消息体积过大，可能会导致网络带宽成为瓶颈。可以通过压缩消息内容，减少每个消息的大小来缓解带宽压力。
5. **负载均衡**：在高并发场景下，单一 WebSocket 服务器可能无法处理所有请求。通过 **分布式部署** 和 **负载均衡** 可以水平扩展 WebSocket 服务，确保系统稳定性和高可用性。

/**

@Component
@Slf4j
public class OrderTask {

    @Autowired
    private OrderMapper orderMapper;

    /**
     * 处理超时订单的方法
     * 每分钟触发一次
     */
    @Scheduled(cron = "0 * * * * ? ")
    public void processTimeoutOrder(){
        log.info("定时处理超时订单：{}", LocalDateTime.now());

        // 计算15分钟前的时间
        LocalDateTime time = LocalDateTime.now().plusMinutes(-15);

        // 查询状态为待付款且下单时间超过15分钟的订单
        List<Orders> ordersList = orderMapper.getByStatusAndOrderTimeLT(Orders.PENDING_PAYMENT, time);

        // 批量处理超时订单
        if(ordersList != null && ordersList.size() > 0){
            for (Orders orders : ordersList) {
                orders.setStatus(Orders.CANCELLED);
                orders.setCancelReason("订单超时，自动取消");
                orders.setCancelTime(LocalDateTime.now());
                orderMapper.update(orders);
            }
        }
    }

    /**
     * 处理一直处于派送中状态的订单
     * 每天凌晨1点触发一次
     */
    @Scheduled(cron = "0 0 1 * * ?")
    public void processDeliveryOrder(){
        log.info("定时处理处于派送中的订单：{}",LocalDateTime.now());

        // 计算60分钟前的时间
        LocalDateTime time = LocalDateTime.now().plusMinutes(-60);

        // 查询状态为派送中且时间超过60分钟的订单
        List<Orders> ordersList = orderMapper.getByStatusAndOrderTimeLT(Orders.DELIVERY_IN_PROGRESS, time);

        // 批量处理超时派送订单
        if(ordersList != null && ordersList.size() > 0){

### 6. **如何保证定时任务的高可用性？**

**面试问题：** *如果系统出现宕机，如何保证定时任务能够继续执行？*

**回答：**
 为保证定时任务的高可用性，可以考虑以下策略：

1. **高可用服务器部署**：将应用部署到高可用的集群中，通过负载均衡器将流量分发到不同的节点，确保系统在部分节点宕机时仍能继续执行定时任务。
2. **分布式任务调度**：对于大型分布式系统，可以使用 **Quartz** 与 **Zookeeper** 配合，确保任务在集群中的唯一性，避免多个节点同时执行相同的任务。
3. **任务持久化**：在任务执行之前，可以将任务状态持久化到数据库中（如执行时间、任务标识等），确保在系统重启后可以恢复任务执行。
4. **重试机制**：定时任务失败时，可以设计重试机制，例如失败后自动重试 N 次，确保任务不会因临时故障丢失。
5. **监控与报警**：设置监控，实时检查定时任务的执行状态。如果任务执行失败或未按时执行，系统会及时发送告警信息，以便运维人员及时处理。

通过这些措施，确保定时任务的高可用性和可靠性，保障系统稳定运行。

### 7. **如何确保定时任务的执行不受到系统负载过高的影响？**

**面试问题：** *如果系统在高负载的情况下，定时任务可能会被延迟执行，如何保证定时任务的及时性和准确性？*

**回答：**
 为了确保定时任务在高负载情况下的准确性和及时性，我们可以采取以下措施：

1. **任务优先级控制**：设置任务的优先级，确保高优先级任务能够在系统负载较高时优先执行。可以通过线程池管理实现。
2. **分布式调度**：使用分布式任务调度框架（如 Quartz 集成 Zookeeper），确保任务在多个节点之间分配，不会因单个节点的负载过高而影响任务执行。
3. **任务隔离**：将定时任务的执行与主业务流程隔离，使用独立的线程池来处理定时任务，避免与主应用的请求处理混杂。
4. **合理配置任务执行频率**：对于高负载环境下的任务，可以适当延长任务的执行间隔，避免过于频繁地执行定时任务，减轻系统负担。
5. **任务回退机制**：当任务因高负载被延迟执行时，设计回退机制，使得任务可以在后续某个时间点补充执行，保证不丢失业务逻辑。

### 8. **定时任务的任务失败和异常处理是如何设计的？**

**面试问题：** *当定时任务执行失败或者发生异常时，如何设计异常处理和任务回退机制，确保数据的准确性和业务逻辑的完整性？*

**回答：**
 定时任务的失败和异常处理设计是确保系统可靠性的重要环节。我们通常采取以下策略：

1. **任务重试机制**：当定时任务失败时，我们会设计重试机制，最多重试 N 次，避免由于临时的网络问题或数据库问题导致任务失败。
2. **事务管理**：使用 **Spring 的事务管理**，确保定时任务中的数据库操作和业务操作要么全部成功，要么全部回滚，避免出现数据不一致的情况。
3. **异常日志记录**：对任务中的异常进行详细的日志记录，及时发现并处理潜在问题。使用 **日志框架**（如 Logback 或 SLF4J）记录任务执行时的异常信息，方便后期分析。
4. **告警机制**：结合监控系统（如 **Prometheus** 或 **Grafana**）监控定时任务的执行状态，及时发现任务失败的情况，并通过邮件、短信等方式发送告警通知。
5. **失败数据补偿**：对于任务失败的数据，可以设计数据补偿机制。在任务失败时，将失败的任务记录到数据库中，后续重新执行时可以进行补偿处理。

### 9. **定时任务执行的准确性和容错性如何保证？**

**面试问题：** *如何确保定时任务在高并发和系统复杂情况下的准确性和容错性？*

**回答：**
 为了保证定时任务在高并发和复杂环境下的准确性和容错性，我们可以采取以下措施：

1. **任务粒度控制**：将大的任务拆分为小的粒度任务，每次执行少量的数据操作。这样可以减少任务执行时的负载，提高任务的执行精度。
2. **容错设计**：每个定时任务都应设计合理的容错机制。例如，在数据操作失败时，可以通过捕获异常并回退操作来保证任务的容错性。任务可以通过 **补偿机制** 来恢复到一致性状态。
3. **任务幂等性**：确保任务操作具有幂等性，即无论任务执行多少次，结果都是一样的，这样即使任务由于某些原因被重复执行，也不会产生不一致的结果。
4. **分布式锁**：在分布式系统中，使用分布式锁（如 **Redis** 的 `SETNX`）来避免多个节点同时执行相同的任务，确保任务执行的唯一性。
5. **检查点和数据快照**：在长时间运行的定时任务中，可以设置检查点，在任务失败时从最近的检查点恢复，而不是从头开始执行，避免重复工作。

### 10. **如何解决定时任务与系统其它任务（如用户请求）之间的资源竞争问题？**

**面试问题：** *如何确保定时任务和系统其他任务（如用户请求）之间的资源不会竞争，影响系统性能？*

**回答：**
 为了解决定时任务与系统其他任务的资源竞争问题，我们可以采取以下措施：

1. **任务隔离**：将定时任务与用户请求的处理逻辑进行隔离。可以使用 **独立线程池** 来处理定时任务，避免与用户请求共享资源。
2. **任务优先级管理**：根据业务需求，对不同任务设置不同的优先级。例如，定时任务可以设置较低的优先级，确保用户请求在高并发情况下得到优先处理。
3. **资源配额管理**：通过合理的资源配额来管理定时任务和用户请求的资源使用。比如设置定时任务的 CPU 使用时间限制或内存占用限制，确保不会因定时任务过多占用系统资源而影响用户请求。
4. **负载均衡**：使用 **负载均衡** 来分散请求压力，在不同服务器节点上均衡分配用户请求和定时任务。可以使用 **Nginx** 或 **HAProxy** 等负载均衡工具。
5. **异步任务执行**：对于不影响系统实时性的任务，可以考虑将任务异步执行，避免阻塞主线程的执行（简单异步任务线程池；复杂异步任务链式操作CompletableFuture；Spring框架@Async注解；高并发计算ForkJoinPool；周期定时ScheduledExecutorService）。

### 11. **定时任务中的数据库查询如何优化，避免对数据库产生过高的负载？**

**面试问题：** *如何在定时任务中优化数据库查询，避免频繁查询对数据库造成过高的负载？*

**回答：**
 为了优化定时任务中的数据库查询，避免对数据库造成过高的负载，我们可以采取以下措施：

1. **分页查询**：对于需要查询大量数据的定时任务，采用分页查询的方式，每次查询一小批数据，避免一次性加载大量数据对数据库造成压力。
2. **索引优化**：确保定时任务查询的字段上有适当的索引（例如订单表中的订单状态、创建时间等），加速查询速度，减少数据库扫描时间。
3. **缓存机制**：对于频繁查询的数据，可以使用 **Redis** 等缓存系统缓存查询结果，减少数据库的查询压力。例如，可以将订单状态存储在 Redis 中，每次查询时先从缓存获取，缓存未命中再查询数据库。
4. **数据预聚合**：对于一些需要统计的操作，可以使用定时任务提前将数据预聚合并存储在专门的统计表中，避免每次查询时都进行复杂的聚合操作。
5. **查询限流**：在高并发场景下，定时任务查询可以采取限流措施，限制每秒查询的数量，避免瞬时大量的查询请求压垮数据库。

5. **重试机制**：当任务执行失败时，自动触发重试机制，最多重试 N 次，避免因为临时故障导致任务丢失。
