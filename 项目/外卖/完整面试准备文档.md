## 二、技术点一：JWT + 拦截器 + ThreadLocal 鉴权

### 问题1：详细说一下你的鉴权方案是如何实现的？

**回答思路（结合代码）**：

```
【整体流程】
1. 用户登录：验证用户名密码后，生成JWT token返回给前端
2. 后续请求：前端在请求头中携带token
3. 拦截器校验：拦截器从请求头获取token，解析JWT，提取用户ID
4. ThreadLocal存储：将用户ID存入ThreadLocal，供后续使用

【代码实现】

1. 登录生成token（EmployeeController.login）：
   - 验证用户名密码
   - 生成JWT，包含用户ID
   - 返回token给前端

2. 拦截器校验（JwtTokenAdminInterceptor.preHandle）：
   - 从请求头获取token
   - 解析JWT，验证签名和过期时间
   - 提取用户ID，存入ThreadLocal
   - 返回true放行，或false拒绝（401）

3. ThreadLocal使用（BaseContext）：
   - setCurrentId()：设置当前用户ID
   - getCurrentId()：获取当前用户ID
   - 在Service层和AOP中直接使用，无需传参
```

**关键代码位置**：
- 登录：`EmployeeController.login()` 第45-66行
- 拦截器：`JwtTokenAdminInterceptor.preHandle()` 第35-59行
- ThreadLocal：`BaseContext` 类

### 问题2：为什么用JWT而不是Session？

**回答思路**：
1. JWT优势：
   - 无状态，服务器不存储，适合分布式
   - 可跨域，适合前后端分离
   - 包含用户信息，减少数据库查询
2. Session劣势：
   - 需要服务器存储，集群需要共享Session
   - 跨域需要额外处理
   - 增加服务器内存压力

### 问题3：ThreadLocal的原理是什么？为什么用它？

**回答思路**：
1. 原理：
   - 每个线程有独立的ThreadLocalMap
   - 通过线程隔离存储数据
   - 避免线程安全问题
2. 为什么用：
   - 每个HTTP请求在独立线程中处理
   - 避免在方法间传递用户ID
   - 性能好，无需查询数据库

### 问题4：ThreadLocal会导致内存泄漏吗？如何避免？

**回答思路**：
1. 会泄漏：
   - 线程池复用线程，ThreadLocal可能未清理
   - Entry的value是强引用，无法被GC
2. 当前问题：
   - 拦截器中没有清理ThreadLocal
3. 解决方案：
   ```java
   @Override
   public void afterCompletion(...) {
       BaseContext.removeCurrentId(); // 清理ThreadLocal
   }
   ```

### 问题5：如果用户同时登录多个设备，你的方案如何处理？

**回答思路**：
1. 当前方案：每个设备独立token，可以同时登录
2. 如果需要单点登录：
   - 在Redis中存储 userId -> token 映射
   - 新登录时，使旧token失效
   - 拦截器中检查token是否在Redis中

### 问题6：JWT的secretKey泄露了怎么办？

**回答思路**：
1. 问题：攻击者可以伪造任意token
2. 解决方案：
   - 使用环境变量或配置中心管理密钥
   - 定期轮换密钥
   - 实现token黑名单（Redis存储已注销的token）
   - 使用RS256非对称加密（更安全）

## 三、技术点二：Redis + SpringCache 缓存优化

### 问题1：详细说一下你的缓存实现方案

**回答思路（结合代码）**：

```
【实现方式】
项目中使用两种方式：
1. SpringCache注解方式（套餐查询）
2. 手动RedisTemplate方式（菜品查询）

【方案一：SpringCache注解】
- 使用@Cacheable查询缓存
- 使用@CacheEvict清除缓存
- 自动管理缓存生命周期

【方案二：手动RedisTemplate】
- 查询前先查Redis
- 缓存未命中时查数据库
- 将结果存入Redis
- 更新/删除时清除相关缓存
```

**关键代码位置**：
- SpringCache：`SetmealController.list()` 第33行
- 手动缓存：`DishController.list()` 第36-56行
- 缓存清除：`DishController.cleanCache()` 第151-154行

### 问题2：为什么有些地方用@Cacheable，有些用RedisTemplate？

**回答思路**：
1. @Cacheable优势：
   - 代码简洁，注解即可
   - 自动管理缓存
2. 手动RedisTemplate优势：
   - 更灵活，可以自定义key规则
   - 可以精确控制缓存逻辑
   - 支持通配符删除（如 `dish_*`）
3. 项目中的原因：
   - 套餐缓存：使用固定key，适合注解
   - 菜品缓存：需要通配符删除，手动控制更灵活

### 问题3：你的缓存更新策略是什么？

**回答思路**：
1. 策略：Cache-Aside（旁路缓存）
   - 查询：先查缓存，未命中查数据库，再写入缓存
   - 更新：先更新数据库，再删除缓存
2. 代码实现：
   - 新增套餐：删除特定key的缓存
   - 修改/删除：清空整个缓存（allEntries = true）
3. 优化建议：
   - 可以只删除相关key，而不是清空所有

### 问题4：如何防止缓存穿透、击穿、雪崩？

**回答思路**：

**缓存穿透**：
- 问题：查询不存在的数据，每次都查数据库
- 解决：缓存空值（设置短过期时间）

**缓存击穿**：
- 问题：热点key过期，大量请求同时访问数据库
- 解决：使用分布式锁
```java
String lockKey = "lock:dish:" + categoryId;
Boolean lock = redisTemplate.opsForValue()
    .setIfAbsent(lockKey, "1", 10, TimeUnit.SECONDS);
if (lock) {
    try {
        // 再次检查缓存
        // 查询数据库
        // 写入缓存
    } finally {
        redisTemplate.delete(lockKey);
    }
}
```

**缓存雪崩**：
- 问题：大量key同时过期
- 解决：设置随机过期时间（如30-60分钟）

### 问题5：你的缓存没有设置过期时间，会导致什么问题？

**回答思路**：
1. 问题：
   - 数据可能一直不过期，导致数据不一致
   - 占用内存空间
2. 当前代码问题：
   ```java
   redisTemplate.opsForValue().set(key, list); // 没有过期时间
   ```
3. 改进方案：
   ```java
   int expireTime = 30 + new Random().nextInt(30); // 30-60分钟随机
   redisTemplate.opsForValue().set(key, list, expireTime, TimeUnit.MINUTES);
   ```

### 问题6：使用Redis的keys命令有什么问题？

**回答思路**：
1. 问题：
   - `keys` 命令是阻塞的，会扫描所有key
   - 在生产环境可能导致Redis阻塞
2. 当前代码：
   ```java
   Set keys = redisTemplate.keys(pattern); // 使用keys命令
   ```
3. 优化方案：
   - 使用 `SCAN` 命令替代 `KEYS`
   - 或者维护一个key集合，记录需要删除的key

### 问题2：为什么选择WebSocket而不是HTTP轮询？

2. WebSocket优势：

   - 实时性好，延迟低
   - 减少网络开销

**回答思路**：
1. 问题：
   - HashMap不是线程安全的
   - 多客户端同时连接/断开时，可能出现并发问题
2. 改进方案：
   ```java
   private static ConcurrentHashMap<String, Session> sessionMap = 
       new ConcurrentHashMap<>();
   ```

### 问题4：如何实现点对点推送，而不是群发？

**回答思路**：
1. 当前问题：所有客户端都收到消息
2. 改进方案：
   ```java
   // 根据用户ID存储Session
   sessionMap.put("merchant:" + merchantId, session);
   
   // 点对点推送
   public void sendToClient(String userId, String message) {
       Session session = sessionMap.get(userId);
       if (session != null && session.isOpen()) {
           session.getBasicRemote().sendText(message);
       }
   }
   ```

### 问题5：如果客户端断开连接后重连，消息会丢失吗？

**回答思路**：
1. 会丢失：客户端断开时，服务器发送消息会失败
2. 解决方案：
   - 消息队列：发送消息时先存入队列，客户端重连后获取
   - 数据库存储：将消息存入数据库，客户端重连后查询
   - Redis存储：使用Redis的List存储未读消息

### 问题6：如何实现WebSocket的负载均衡？

**回答思路**：
1. 问题：多台服务器部署时，连接分布在不同服务器
2. 解决方案：
   - Sticky Session：Nginx配置ip_hash
   - 消息队列：服务器发送消息到队列，所有服务器监听
   - Redis Pub/Sub：服务器发布消息，所有服务器订阅

## 五、技术点四：SpringTask 定时任务

**回答思路（结合代码）**：

**关键代码位置**：

### 问题2：为什么超时订单每分钟检查，而派送中订单每天检查？

**回答思路**：
1. 业务需求不同：
   - 超时订单：需要及时处理，避免用户等待
   - 派送中订单：异常情况较少，可以低频检查
2. 性能考虑：
   - 超时订单数量可能较多，需要频繁检查
   - 派送中订单异常情况少，低频检查即可

### 问题3：定时任务在集群环境下会重复执行吗？

**回答思路**：
1. 问题：多实例部署时，每个实例都会执行
2. 解决方案：为了解决定时任务在集群环境下重复执行的问题，常见的做法是使用分布式锁机制。通过分布式锁，可以确保同一时刻只有一个实例执行任务，其他实例会等待或者跳过任务的执行。
   ```java
   @Scheduled(cron = "0 * * * * ?")  // 定时任务每分钟执行一次
   public void processTimeoutOrder(){
       String lockKey = "lock:processTimeoutOrder";  // 锁的key
       // 尝试设置分布式锁，60秒后过期
       Boolean lock = redisTemplate.opsForValue()
           .setIfAbsent(lockKey, "1", 60, TimeUnit.SECONDS);
       
       // 如果设置锁成功，表示当前实例获得了锁
       if (lock) {
           try {
               // 执行任务逻辑，例如处理超时订单
           } finally {
               // 任务执行完成后删除锁，释放锁资源
               redisTemplate.delete(lockKey);
           }
       }
   }
   ```

## 九、常见追问问题

### 追问1：能画一下订单支付的流程图吗？

**回答思路**：
```
用户点击支付
    ↓
前端调用支付接口
    ↓
后端生成预支付订单（调用微信支付API）
    ↓
返回支付参数给前端
    ↓
前端调起微信支付
    ↓
用户完成支付
    ↓
微信调用回调接口
    ↓
后端解密数据、更新订单状态、推送WebSocket
    ↓
返回成功响应给微信
```

### 追问2：能说一下JWT的生成和解析过程吗？

**回答思路**：
1. 生成：
   - 设置Header（算法、类型）
   - 设置Payload（用户ID、过期时间）
   - 使用secretKey签名
   - Base64编码生成token
2. 解析：
   - Base64解码
   - 验证签名
   - 检查过期时间
   - 提取用户信息

### 追问3：Redis的缓存更新策略有哪些？

**回答思路**：
1. Cache-Aside（旁路缓存）：先更新数据库，再删除缓存
2. Write-Through（写透）：先更新缓存，再更新数据库
3. Write-Back（写回）：先更新缓存，异步更新数据库
4. 项目使用：Cache-Aside

## 十、面试技巧总结

### 1. 回答原则
- STAR法则：Situation（背景）、Task（任务）、Action（行动）、Result（结果）
- 结合代码：提到具体类名、方法名、行号
- 承认不足：说明当前实现的问题和改进方案
- 展示思考：说明为什么这样设计，有什么考虑

### 2. 常见陷阱
- 不要只说"我用了XX技术"，要说"为什么用"和"怎么用"
- 不要只说优点，要承认问题和改进方向
- 不要编造数据，可以说相对数据
- 不要只说技术，要结合业务场景

### 3. 加分项
- 能说出当前实现的问题
- 能提出改进方案
- 能考虑分布式、高并发场景
- 能结合业务场景说明技术选型
