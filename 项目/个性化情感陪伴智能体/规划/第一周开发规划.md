# 第一周开发规划（V0→V1 基座）：可演示 + 可观测 + 不像玩具

> 本周目标：端到端跑通，并且**每轮对话都有 trace 可回放**；同时把“陪伴感”最关键的 **情绪识别 + 状态机雏形**立起来（策略先简单也没关系，关键是可控、可解释、可回归）。

---

## 0. 本周验收标准（Definition of Done）

- **对话可用**：同一会话连续 20+ 轮稳定，不崩、不丢 `session_id`。
- **可观测**：每次请求返回 `trace_id`，可通过 `GET /trace/{trace_id}` 查看链路记录：
  - **耗时拆分**：LLM/预处理/状态机/记忆组装（后两者可先占位，但字段必须有）
  - **成本字段**：`token_in/token_out`（拿不到就先估算或记录 prompt 字符数/消息条数）
  - **关键决策**：情绪结果、状态机 mode、是否触发安全模式、是否调用工具（本周工具可先占位）
- **可演示**：最小 UI（Gradio 或其他）至少能展示：聊天 + 右侧诊断面板（情绪/模式/trace_id/耗时/成本）。
- **像陪伴**：情绪识别输出 `label + intensity + evidence`；状态机能稳定区分 **闲聊/倾听/安慰/工作** 四类之一，并写入 trace。

---

## 1. Day 1：项目骨架与最小可运行链路

### 1.1 任务
- 初始化目录结构（建议）
  - `app/`（服务入口）
  - `app/api/`（路由）
  - `app/core/`（配置/依赖注入/通用工具）
  - `app/llm/`（模型客户端）
  - `app/trace/`（trace 结构与存储）
  - `app/memory/`（STM/LTM，先放 STM 占位）
  - `app/policy/`（状态机/策略）
  - `ui/`（Gradio 或 Web UI）
  - `data/`（trace/评测集/样例数据）
- 配置 `.env`（模型 key、base_url、model 名等）
- FastAPI 起服务：`GET /health`、`POST /chat`（先回模型回复即可）
- 写 `LLMClient`：只负责把 messages 发给模型并返回文本

### 1.2 验收
- `POST /chat` 输入 `message` 能返回模型回复
- 服务能正常启动/重启

---

## 2. Day 2：会话与短期记忆（STM v0）

### 2.1 任务
- 定义请求必须带 `user_id`、`session_id`
- 会话隔离：按 `session_id` 维护对话历史
- STM v0：滑动窗口（按轮次或粗 token 预算裁剪）
- 提供重置接口：`POST /sessions/reset`（清空某会话的 STM）

### 2.2 验收
- 同一 `session_id` 多轮对话上下文生效
- 不同 `session_id` 的上下文完全隔离

---

## 3. Day 3：Trace 体系（V0 核心，不做这个就像玩具）

### 3.1 任务
- 每次 `/chat` 生成 `trace_id`
- 定义 trace 最小结构（建议字段）
  - `request`: `user_id, session_id, message, timestamp`
  - `pipeline.steps[]`: 每一步包含 `name, start_ms, end_ms, input_summary, output_summary, error`
  - `metrics`: `latency_ms, token_in, token_out, model, degraded, safety_mode`
  - `decision`: `emotion, mode, mode_reason`
- 存储：先用本地 JSONL/SQLite 均可（优先简单可靠）
- 提供查询：
  - `GET /trace/{trace_id}`：返回完整 trace
  - 可选 `GET /traces?session_id=...`：列表

### 3.2 验收
- 任意一次对话都能查到 trace
- trace 里至少包含：总耗时、情绪字段占位、mode 字段占位、token/成本字段（可先估算）

---

## 4. Day 4：情绪识别（Emotion v0）+ 安全模式占位

### 4.1 任务
- 实现 `emotion_analyzer`，输出：
  - `label`（如：开心/平静/低落/焦虑/愤怒/疲惫）
  - `intensity`（0-3 或 0-100）
  - `evidence`（触发依据：关键词/句子片段/LLM解释）
- 加入风险分层字段：一般/关注/高风险
- 安全模式 v0：
  - 高风险时使用固定的安全策略模板（更克制、更安全）
  - 在 trace 标注 `safety_mode=true` 和触发原因

### 4.2 验收
- 典型输入（低落/焦虑/愤怒/自伤暗示）输出稳定
- trace 中可看到情绪与风险分层结果

---

## 5. Day 5：状态机（Policy/State v0）与共情策略骨架

### 5.1 任务
- 状态机输出 `mode`：
  - 闲聊 / 倾听 / 安慰 / 工作（工具先占位）
- 判定依据：意图 + 情绪 + 用户显式指令（例如“别安慰我，我只想吐槽”）
- 共情策略骨架：
  - 为每个 mode 准备一套 prompt 模板（至少 4 套）
  - trace 记录：命中的 mode 与依据（`mode_reason`）

### 5.2 验收
- 相同上下文下 mode 稳定，不要每次乱跳
- trace 中能解释“为什么进入该模式”

---

## 6. Day 6：UI 最小可演示（Chat + 诊断面板）

### 6.1 任务
- Gradio 最小页面：
  - 中间：对话区
  - 右侧：本轮诊断面板（emotion、mode、trace_id、latency、token）
  - session 切换与 reset
  - 复制/打开 trace（或直接在 UI 展示 trace JSON）

### 6.2 验收
- UI 可跑完“多轮对话 → 查看诊断 → 查 trace → 切 session”

---

## 7. Day 7：演示脚本 + 回归用例（把“像业务”坐实）

### 7.1 任务
- 写 `demo_script.md`：准备固定输入串，保证演示可复现
- 最小回归集（建议 20 条 JSON）：
  - 情绪识别稳定性（同样输入输出一致）
  - 状态机 mode 稳定性
  - 安全模式触发
  - session 隔离
- 记录本周 badcase（后续做 V2/V4 会用）

### 7.2 验收
- 2 分钟演示闭环：
  - 聊天 → 诊断面板 → trace 回放 → 安全模式触发（可控且可解释）

---

## 8. 本周结束的“作品集观感”要求（防玩具化）

- 打开 UI：不只是聊天框，而是**聊天 + 诊断面板 + trace 回放**。
- 面试一句话能立住：
  - “我做了一个可观测、可控策略的陪伴型 Agent 基座，后续所有 RAG/工具/评测都挂在这条 trace 链路上做回归。”
