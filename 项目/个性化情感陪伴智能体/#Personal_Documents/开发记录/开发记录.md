## 2026-03-19 Day5：状态机（Policy v0）与共情策略骨架

### 完成内容
- **状态机**（`app/policy/state_machine.py`）
  - `decide_mode(message, emotion_label, emotion_intensity)` → (mode, mode_reason)
  - mode：闲聊 / 倾听 / 安慰 / 工作（工具占位）
  - 判定优先级：用户显式指令（如「别安慰我，只想吐槽」）> 情绪规则；默认 倾听
- **共情策略**：四个 mode 各一套 system prompt 模板（`MODE_PROMPTS`），`get_system_prompt_for_mode(mode)` 返回对应模板
- **/chat**：在情绪识别后调用 `decide_mode`，非安全模式下用 `get_system_prompt_for_mode(mode)` 作为 system；trace 中写入 `decision.mode`、`decision.mode_reason`
- **测试**：`tests/test_day5_policy.py`（trace 含 mode/mode_reason、显式指令覆盖情绪、四套 prompt 存在）

### 涉及文件
- 新增：`app/policy/__init__.py`、`app/policy/state_machine.py`、`tests/test_day5_policy.py`
- 修改：`app/api/routes.py`（接入 decide_mode、get_system_prompt_for_mode，trace 写入 mode/mode_reason）

### 验收
- pytest 10 条全过
- 相同输入 mode 稳定；trace 可解释「为什么进入该模式」（mode_reason）

## 2026-03-19 情绪识别改为 LLM 为主 + 关键词兜底

### 完成内容
- **情绪识别**：由纯关键词改为 **LLM 优先、规则兜底**
  - **风险分层**：仍用关键词（`_risk_from_keywords`），保证高危不漏、可解释
  - **情绪 label/intensity/evidence**：优先调用 LLM 单轮（固定 prompt 要求只输出 JSON）；解析失败或请求异常时回退到规则（`_emotion_from_rules`）
- **接口**：`analyze_emotion` 改为 async，入参增加 `llm: LLMClient`，便于复用同一 client
- **/chat**：先 `llm = _build_llm_client()`，再 `await analyze_emotion(req.message, llm=llm)`，主对话复用该 `llm`
- **测试**：Day2 因每轮 /chat 多一次情绪 LLM 调用，将「LLM 调用次数」断言改为每轮 2 次（情绪 + 对话），并调整取第二轮对话的索引

### 涉及文件
- 修改：`app/emotion/analyzer.py`（重写为 LLM 主 + 规则兜底）、`app/api/routes.py`（提前构建 llm、await analyze_emotion）、`tests/test_day2_sessions.py`（调用次数与索引）

### 验收
- pytest 7 条全过
- 情绪泛化更好（换说法可由 LLM 识别）；风险仍由关键词保证；LLM 失败时规则兜底不影响主对话

- **配置**：`app/core/settings.py` 新增 `trace_dir`
