# 常用命令与说明

项目根目录下执行。端口、路径等可按需修改。

---

## 1. 虚拟环境与依赖（首次必做）

若尚未创建环境或安装依赖：

```powershell
python -m venv .venv
.\.venv\Scripts\Activate.ps1
pip install -r requirements.txt
```

`requirements.txt` 已包含 **pytest**；运行测试前**必须**激活上述 venv（否则会报 `No module named pytest`）。

**拉取新代码或 `requirements.txt` 有变更后**（例如三期增加了 `sqlalchemy` / `pymysql` / `redis`），在**已激活的 `.venv` 里**再执行一次：

```powershell
pip install -r requirements.txt
```

未更新时若 `.env` 里配置了 `DATABASE_URL`，启动可能报 `No module named 'sqlalchemy'`。

**PowerShell（日常使用）：**

```powershell
.\.venv\Scripts\Activate.ps1
```

**CMD：**

```cmd
.\.venv\Scripts\activate.bat
```

- **说明**：使用项目自带的 `.venv`，保证依赖与解释器一致。激活后终端前会显示 `(.venv)`。
- **注意**：若同时装了 Conda，先 `conda deactivate` 再激活 venv，避免用错 Python。

更完整的首次上手步骤见**项目根目录 `README.md`**。

## 2. 启动服务（FastAPI + uvicorn）

```powershell
python -m uvicorn app.main:app --reload --host 127.0.0.1 --port 8076
```

- **说明**：以开发模式启动，`--reload` 下改代码会自动重启。Swagger 文档：`http://127.0.0.1:8076/docs`。
- **端口**：8076 可改为 8075、8077、8000 等（避免与其它程序占用冲突）。
- **前提**：已激活 venv；项目根目录下有 `.env` 且配置好 `LLM_API_KEY` 等（否则 `/chat` 会 500）。V1 可选配置（共情策略路径、情绪落盘、策略开关）见 `.env.example` 注释，修改后需重启后端。

**探测 MySQL（`DATABASE_URL`）是否连通**（不启动 uvicorn）：

```powershell
python scripts/check_db.py
```

未配置 `DATABASE_URL` 时脚本直接退出并提示；失败时打印异常，退出码为 `1`。

**探测 Redis（`REDIS_URL`）是否连通**（不启动 uvicorn）：

```powershell
python scripts/check_redis.py
```

未配置 `REDIS_URL` 时脚本直接退出并提示；失败时退出码为 `1`。

## 3. 运行测试（pytest）

**全量测试（推荐日常用）：**

```powershell
python -m pytest -q
```

- **说明**：运行 `tests/` 下所有用例，`-q` 为简洁输出。覆盖 Day1/Day2/Day3 等。**须已 `pip install -r requirements.txt` 且已激活 `.venv`**。
- **`tests/conftest.py`**：默认会把 **`DATABASE_URL` / `REDIS_URL` 置空**，避免本地 `.env` 指向真实 MySQL/Redis 时单测写脏数据或行为不一致；并 **autouse mock 主链 `ChatOpenAI`**（`tests/orchestrator_mocks.FakeLCMainChatModel`），情绪/抽取等仍由各用例 patch **`routes._build_llm_client`**。若要带着真实库跑集成测试，可设 **`PYTEST_USE_REAL_DB=1`** 后再执行 pytest。

**只跑某文件：**

```powershell
python -m pytest -q tests\test_day3_trace.py
python -m pytest -q tests\test_day2_sessions.py
```

**带打印/详细输出：**

```powershell
python -m pytest -v
python -m pytest -v -s
```

- `-v`：每条用例一行；`-s`：不捕获 stdout，方便看 `print`。

**说明**：测试会 mock LLM，不发起真实 API 请求；Trace 测试使用内存 store，不写磁盘。

## 7. 二期 Web 控制台（Vite + React，M1 连接探针）

**前提**：后端已启动（同上，默认 `127.0.0.1:8076`）；根目录 `.env` 中 `CORS_ORIGINS` 可留空（内置允许 `5173/4173`）。

```powershell
cd web
npm install
npm run dev
```

- 浏览器打开终端提示的本地地址（一般为 `http://127.0.0.1:5173`）；默认进入 **`/chat` 对话页**；`/probe` 为连接探针；**`/memory`** 为长期记忆 Studio；**`/eval`** 为 Eval Lab（需与 `/admin/*` 相同鉴权逻辑，本机无 `ADMIN_CONFIG_TOKEN` 时通常可直接调用）；**`/emotion`** 为情绪仪表盘（依赖 `EMOTION_LOG_ENABLED` 与 `data/emotion_log.jsonl`）；**`/trace`** 为 Trace 回放（可 `?id=trace_id`）；**`/ops`** 为运营台（热参数、配额、反馈、风险事件、**合规**导出 `GET /users/{id}/export` 与遗忘 `POST /users/{id}/forget`，与 Gradio 运营台同源 admin 鉴权）。
- **`/ops` 热参数**：除 RAG/配额/STM/安全外，可改 **隐式 LTM** 四项（`ltm_extract_enabled`、`ltm_extract_every_n_turns` 0–20、`ltm_extract_dedup_enabled`、`ltm_extract_count_toward_quota`），与 `PATCH /admin/config` 及 `data/hot_config.json` 白名单一致。
- 页面通过 **Vite 代理** 将 `/api` 转发到后端（目标可在 `web/.env` 里设 `VITE_DEV_API_TARGET`，默认 `http://127.0.0.1:8076`）。
- **连接探针**：`GET /health`（redis/database/`ltm_extract_enabled`）、`POST /chat`；`user_id` / `session_id` / 可选 `X-Admin-Token` 写入 localStorage；错误区展示 FastAPI 结构化 `detail`（含 422 校验列表）。

生产构建：

```powershell
cd web
npm run build
```

若部署到与 API 不同源，在构建前设置 `VITE_API_BASE` 指向后端根 URL（见 `web/.env.example`）。

## 8. Redis（STM）与 MySQL（LTM）：怎么产生数据、怎么查看

**前提**：根目录 `.env` 已配置 **`REDIS_URL`** / **`DATABASE_URL`**，且 `GET /health` 里对应项为 **`ok`**（否则 STM/LTM 会退回内存，外置库里没有业务数据）。**V1.1**：`GET /health` 的 **`ltm_extract_enabled`** 与 **`LTM_EXTRACT_ENABLED`** 一致。开启后，`POST /chat` / **`/chat/stream`** 在 STM 写入成功后按 **`LTM_EXTRACT_EVERY_N_TURNS`**（默认每 3 条 assistant）触发抽取；**高风险安全模式**下会跳过抽取。响应体 **`ltm_extract_written`** / **`ltm_extract_updated`**（去重合并更新）与 Trace **`decision`** 同名字段对应；Trace 步骤 **`ltm_extract`** 的 **`output_summary`** 含 `written` / `updated` / `skipped_dup`。去重参数见 `.env.example` 中 **`LTM_EXTRACT_DEDUP_*`**。若开启 **`QUOTA_ENABLED`** 与 **`LTM_EXTRACT_COUNT_TOWARD_QUOTA`**（默认 true），抽取会在调用前做日预算预检，成功返回后按「抽取 prompt 字符 + 模型输出字符」扣减；不足时 Trace 步骤 **`ltm_extract`** 为 **`skipped_quota`**，主对话仍正常。  

**V1.1 P2 异步抽取**：**`LTM_EXTRACT_ASYNC=true`** 且 **`REDIS_URL`** 已配置时，隐式抽取入 Redis 队列由进程内 worker 执行；响应字段 **`ltm_extract_async_pending`** 为真时，二期 Web **`/chat`** 会 Toast 提示，并**自动轮询**拉取合并后的 Trace（亦可手动点侧栏 **「刷新 Trace」**）；合并后出现 **`ltm_extract_new_ids`** 时再使用「撤销本轮隐式记忆」。未配 Redis 时异步开关不生效（回退同步抽取）。  

**撤销隐式写入**：`POST /memory/ltm/undo_extract`，body `{"trace_id":"…","user_id":"…"}`（与 Trace 归属一致）；仅软删 **`source=dialogue_extract`** 且列在 Trace **`decision.ltm_extract_new_ids`** 中的新建行，并从 RAG 索引移除。Web 聊天侧栏「反馈」区有「撤销本轮隐式记忆」按钮（**`ltm_extract_new_ids` 非空**时可用；**异步抽取**场景请先侧栏 **「刷新 Trace」** 再撤销）。

### 8.1 Redis — 短期会话（STM）

| 动作 | 说明 |
|------|------|
| **写入** | 调用 **`POST /chat`**（或前端聊天页发消息），请求体里带上 **`user_id`**、**`session_id`**（可固定一对用于测试）。每轮会在 STM 里追加 user/assistant 等消息。 |
| **Key 形态** | 字符串：`companion:stm:{user_id}:{session_id}`（其中的 `:` 会替换成 `_`）；另有集合 `companion:stm:useridx:{user_id}` 记录该用户下有哪些 `session_id`。 |
| **用 Redis 客户端看** | 连上虚拟机/本机 Redis 后：`KEYS companion:stm*` → 对某个 key 做 **`GET`**（值为 JSON 数组消息）；`SMEMBERS companion:stm:useridx:你的user_id` 看会话列表。 |
| **用 API 看（不写 redis-cli）** | **`GET /sessions?user_id=...`** 列会话；**`GET /sessions/{session_id}/messages?user_id=...`** 读该会话消息（读的是当前 STM 实现，Redis 与内存接口一致）。 |

### 8.2 MySQL — 长期记忆（LTM）

| 动作 | 说明 |
|------|------|
| **写入** | **`POST /memory/ltm`**（Swagger `/docs` 或 Web **`/memory`** Memory Studio），提交类型、内容等；**V1.1**：`LTM_EXTRACT_ENABLED=true` 时 **`POST /chat`** 亦可写入（`source` 多为 **`dialogue_extract`**）。 |
| **表名** | **`ltm_items`**（首次成功连库时由 SQLAlchemy **`create_all`** 建表）。`created_at` / `updated_at` 存的是 **毫秒时间戳**（约 13 位数字），必须用 **BIGINT**；若你第一次建表时代码把这两列建成了 **INT**，写入会报 **`Out of range value for column 'created_at'`**（MySQL 1264）。 |
| **用 MySQL 客户端看** | `USE companion;` → **`SELECT * FROM ltm_items ORDER BY created_at DESC LIMIT 20;`** |

**何时要做 `ALTER TABLE`**：只有在你已经用「旧表结构」建过 `ltm_items`、且写 LTM 时出现 **1264 / Out of range** 时才需要。**全新库**或 **删表后让应用重新 `create_all`** 建出的表，一般已是 BIGINT，可不做。

**在哪里做**：在能连上你 `.env` 里 **`DATABASE_URL` 同一台 MySQL、同一库** 的客户端里执行（任选其一即可）：

- **MySQL Shell / 命令行**：本机安装了 MySQL 客户端时，PowerShell 里例如：  
  `mysql -h 127.0.0.1 -P 3306 -u root -p`，登录后选库执行下面 SQL。  
- **MySQL Workbench / Navicat / DBeaver / DataGrip**：新建连接（主机、端口、用户、密码与 `DATABASE_URL` 一致），打开 **`companion`** 库，在查询窗口执行下面 SQL。  
- **phpMyAdmin** 等网页工具：同样连到该实例与 **`companion`** 库后执行。

**具体执行什么**（在 **`companion`** 库下）：

```sql
USE companion;

-- 先看当前类型（可选）：DESC ltm_items;
-- 若 created_at / updated_at 是 int(11) 等，再执行：

ALTER TABLE ltm_items
  MODIFY COLUMN created_at BIGINT NOT NULL,
  MODIFY COLUMN updated_at BIGINT NOT NULL DEFAULT 0;
```

执行成功后无需改代码，重启 uvicorn 再试 **`POST /memory/ltm`** 即可。

| **用 API 看** | **`GET /memory/ltm?user_id=...`** 分页列表（可选 **`source=dialogue_extract`** 只看隐式写入）；**`GET /memory/ltm/{id}`** 单条。 |

### 8.3 小结

- **多轮对话**始终更新 **STM**。**`LTM_EXTRACT_ENABLED=false`**（默认）时**不会**从对话隐式写入 **LTM**；为 `true` 时按 **`LTM_EXTRACT_EVERY_N_TURNS`** 触发抽取（见上）。Memory Studio 仍可能为空，直到有显式写入或满足轮次触发。  
- **在 Memory Studio / `POST /memory/ltm` 里新增记忆** → **MySQL** 里会出现对应行；**开启 RAG** 后对话才会**引用**这些条目。  
- 连通性探测（不启动 uvicorn）：**`python scripts/check_db.py`**（MySQL）、**`python scripts/check_redis.py`**（Redis）。
