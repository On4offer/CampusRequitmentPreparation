# V1 开发规划：情绪与策略增强 + 工具占位 + 为 V2 铺路

> V0 已完成：多轮对话、Trace、情绪识别、状态机（闲聊/倾听/安慰/工作）、安全模式、Gradio 诊断、演示脚本与回归集。  
> V1 不按“周/天”排期，按**模块与里程碑**推进；每块可独立验收、合并顺序可调。

---

## 0. V1 目标与验收总览

- **目标**：在 V0 基座上增强“像陪伴、可配置、可扩展”，并预留工具与长期记忆接口，为 V2（LTM + RAG）和 V3（工具系统）打基础。
- **总验收**：
  - 共情策略可配置（至少支持从配置/文件加载模板与规则权重）。
  - 状态机明确包含“工具”分支，且 trace 可记录“拟调用工具/占位结果”。
  - 情绪结果可落库或落文件，支持后续“情绪曲线/回访”使用（占位即可）。
  - 长期记忆（LTM）数据模型与写入/查询接口占位就绪，不在主对话链路强依赖。

---

## 1. 共情策略可配置化

### 1.1 内容
- 将当前硬编码的 **mode 对应 system prompt** 抽离为可配置（如 YAML/JSON 或环境指向的配置文件）。
- 可选：状态机规则（显式指令关键词、情绪→mode 映射）也做成可配置，便于调参不改代码。
- 保留代码内默认配置，当无外部配置时行为与 V0 一致。

### 1.2 验收
- 修改配置文件后重启服务，共情话术与 mode 映射生效。
- 现有回归用例（`tests/test_day7_regression.py`）仍全过或仅因默认配置等价而无需改断言。

---

## 2. 状态机“工具”分支与占位

### 2.1 内容
- 状态机输出扩展为五类：**闲聊 / 倾听 / 安慰 / 工作 / 工具**。
- “工具”触发条件：用户显式请求（如“查一下天气”“记一下明天开会”）或意图识别为工具类；先做**关键词/简单规则**占位，不接真实工具 API。
- 进入“工具”时：
  - system 使用**工具占位 prompt**（说明当前为工具模式、后续将接入具体工具）。
  - trace 的 `decision` 中增加字段：`intended_tool`（如 `weather`/`memo`/`time` 或 `unknown`）、`tool_params_placeholder`（可选，空或示例结构），便于后续 V3 真实接入时复用同一决策点。

### 2.2 验收
- 若干条“工具向”输入（如“帮我查天气”“记个事”）稳定进入 mode=工具，并在 trace 中看到 `intended_tool` 或等价信息。
- 回归集中补充 2～3 条“期望 mode=工具”的用例并纳入 pytest。

---

## 3. 情绪结果持久化占位（为情绪曲线/回访做准备）

### 3.1 内容
- 在 trace 已有 `decision.emotion` 基础上，增加**可选落库/落文件**：按 `user_id + session_id + timestamp` 写入一条情绪记录（label、intensity、risk_tier、trace_id）。
- 存储形态：先文件（如 `data/emotion_log.jsonl`）或 SQLite 表均可；不要求实现查询 API，只要求“写得出、格式固定”，便于后续做日/周/月曲线与回访逻辑。

### 3.2 验收
- 对话若干轮后，能在指定路径或表中看到对应条数的情绪记录，且字段与 trace 中 emotion 一致。

---

## 4. 长期记忆（LTM）数据模型与接口占位

### 4.1 内容
- **数据模型**：定义 LTM 条目类型与最小字段（与整体规划一致），例如：
  - 类型：Preference / Profile / Event / Constraint
  - 字段：id, user_id, type, content, created_at, source, confidence, tags（可选）
- **占位存储**：内存或单文件/单表即可；实现 `put(user_id, item)`、`get_by_id(id)`、`list_by_user(user_id, type=None, limit= N)` 等最小接口（可先不接向量，只按 user_id + type 过滤）。
- **API 占位**：
  - `POST /memory/ltm`：写入一条 LTM（body 含 type、content、source、confidence 等）。
  - `GET /memory/ltm?user_id=xxx&type=Preference&limit=10`：列表查询。
  - 不在 `/chat` 主链路中调用 LTM（V2 再做检索与注入）。

### 4.2 验收
- 能通过 API 写入并查出 LTM 条目；数据结构与文档一致，为 V2 的 RAG 与检索预留扩展点。

---

## 5. 可观测与配置外置（轻量）

### 5.1 内容
- **Trace 字段**：若尚未包含，补充 `intended_tool` / `tool_params_placeholder`（见第 2 块）；确保情绪持久化时 trace_id 可关联。
- **配置外置**：关键阈值与开关（如安全模式是否强制倾听、关注档是否优先倾听、情绪落盘开关）从环境变量或配置文件读取，避免写死在代码里。
- **结构化日志（可选）**：为 `/chat` 关键步骤打一条结构化日志（trace_id、latency_ms、mode、risk_tier、error），便于后续接日志系统。

### 5.2 验收
- 通过改配置/环境变量即可切换“情绪是否落盘”“安全/关注档策略”等，无需改代码。
- 若有结构化日志，能在本地或文件中看到 trace_id 与关键字段。

## 7. 建议实施顺序（可调）

| 顺序 | 模块 | 说明 |
|------|------|------|
| 1 | 共情策略可配置化 | 先打通“改配置生效”，后续调参都受益 |
| 2 | 状态机工具分支 + trace 占位 | 与 V0 状态机同处，一次改完再补回归 |
| 3 | 情绪持久化占位 | 独立小模块，不阻塞其他 |
| 4 | LTM 数据模型 + API 占位 | 为 V2 必须，接口稳定即可 |
| 5 | 可观测与配置外置 | 可与 1 合并做，或放在 2/3 之后 |
| 6 | 回归与演示更新 | 每块做完都可补一点，最后统一过一遍 |

---

## 8. 与后续版本的衔接

- **V2（LTM + 最小 RAG）**：直接复用 V1 的 LTM 模型与 `GET/POST /memory/ltm`；在 `/chat` 中接入“按 query 检索 LTM → 注入 prompt”；本规划不要求实现检索与注入。
- **V3（工具系统）**：直接复用 V1 的 mode=工具 与 trace 中 `intended_tool`；实现真实 time/memo/weather 与护栏即可。
- **V4/V5**：情绪曲线、回访、评测、运营台等，在 V1 情绪落盘与可配置基础上继续扩展。
