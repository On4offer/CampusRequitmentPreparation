# 三期迭代规划：界面标杆化 · 功能去占位 · 数据层与中间件落地

> **前提**：一期（V1～V5）与二期（Web 全链路）已交付；《整体检查后迭代规划》已完成契约与文档收口。  
> **三期总目标（三条同时推进）**  
> 1. **界面**：对标优秀前端产品与同类陪伴 / 助手类应用的体验与信息架构，在**视觉层次、交互细节、空态与加载、响应式**上明显超越「能用的中后台」。  
> 2. **功能与逻辑**：参考同类开源/商业项目在 **RAG、记忆、会话、评测、运营** 上的常见做法，把当前代码中仍为**内存占位、文件凑合、字符近似**的能力，升级为**可持久、可查询、可多实例**的实现。  
> 3. **中间件**：按需引入 **PostgreSQL（+ pgvector 可选）**、**Redis**、（可选）**消息队列/任务队列**，与 **Docker Compose** 一键环境对齐；**不实现**需外接资质或第三方平台的接口（见 §7）。  
>
> **本文用途**：三期排期、选型答辩与简历描述；每个模块均建议写明 **验收标准** 与 **本期不做**。

## 1. 界面：可以从哪些方面思考（参考维度，非抄袭）

以下仅作**审美与交互的参照系**，实现时须自有设计语言与品牌色，避免像素级照搬。

### 1.1 同类与优秀前端作品的共性（可抽象成验收项）

| 参照类型 | 可借鉴的点 | 映射到本项目 |
|----------|------------|----------------|
| 通用 AI 对话产品 | 流式输出、输入区固定底栏、停止生成、复制/重试、会话列表分组 | **SSE/WebSocket**、聊天页布局、错误与重试已有则加强 |
| 高质量 Dashboard | 信息密度与留白平衡、卡片层级、图表与表格主题统一 | **情绪仪表盘、Eval Lab、运营台** 的组件化与暗色主题一致性 |
| 文档/知识类产品 | 双向链接、引用预览、侧栏导航 | **citations → Memory Studio** 深链已具备，可加强「悬停摘要」 |
| 陪伴 / 角色类应用（合规前提下） | 温和反馈、减少冰冷错误码、渐进披露高级功能 | **429/502 文案**、诊断信息默认折叠、新手引导（可选） |

### 1.2 三期界面硬指标（建议写进验收）

- **流式对话**：主路径支持流式展示；非流式降级可保留。  
- **会话**：**重命名**、可选 **置顶/归档**；侧栏 **宽度可调**（拖拽或预设断点）。  
- **移动端**：至少 **可用**（单栏 + Tab），不必与桌面完全等价。  
- **动效与反馈**：列表增删、发送中、骨架屏（关键路由）；**150～300ms** 级过渡。  
- **可访问性**：焦点环、表单 label、主要对比度保持二期水准并查漏补缺。  

### 1.3 工程侧支撑界面

- **设计令牌**：色板、圆角、间距、字体层级**文档化**（可简化为 `CSS 变量表 + 1 页 Figma/截图说明`）。  
- **组件库策略**：在现有 Tailwind 基础上，**收敛**重复样式为可复用组件（Button、Card、Modal、DataTable 等）。  
- **参考实现来源**：可研读 **shadcn/ui、Radix、TanStack Table、React Aria** 等**模式**（不必整套引入，避免过重）。

## 2. 功能与逻辑：参考同类项目的常见做法

### 2.1 对话与会话

- **粘性会话**：刷新页面后恢复当前会话与未发送草稿（`sessionStorage` 或短期服务端草稿，二选一）。  
- **并发与顺序**：多 Tab 同时发消息时的 **session 版本号或队列**（避免乱序覆盖 UI）。  
- **系统提示**：静默失败重试策略与**用户可见**失败原因分层（网络 / 模型 / 配额）。  

### 2.2 RAG（在现有混合检索之上）

- **索引与存储分离**：向量与元数据进 **pgvector / 专用向量库**；文本块可存 PG JSONB 或对象存储。  
- **写入路径**：LTM 变更 → **异步或同步**更新向量（失败可 `embedding_status=failed` 与重试任务）。  
- **检索链**：召回 → **可选 rerank**（交叉编码器或小模型）→ 去重与最大 token 预算（已有则参数化）。  
- **可观测**：Trace 中保留 **query、改写 query、命中 id 列表、分数**（已部分具备，与 DB 侧日志对齐）。  

### 2.3 记忆（LTM）

- **显式 vs 隐式**：三期至少保证 **显式 CRUD** 全走持久化；**隐式从对话抽取**可作为增强项（需抽取管道与置信度，单独立项）。  
- **冲突与更新**：同主题多条 Preference 的合并策略（最新覆盖 / 人工确认）在文档与 API 层说清。  

### 2.4 评测与运营

- **Eval Job**：进程内 `_jobs` dict → **Redis + 独立 worker** 或 **DB 任务表 + 轮询**，避免重启丢任务、支持多 worker 扩展。  
- **反馈与情绪日志**：JSONL 可保留为**导出格式**，线上查询走 **DB 表** 或至少 **按用户索引的文件分片**（降低全表扫）。

## 3. 当前代码中的「占位 / 简化」与三期落地对照

| 模块 | 现状 | 三期目标形态 | 典型中间件 |
|------|------|----------------|------------|
| **STM** | 默认 `InMemorySessionStore`；**已支持** `REDIS_URL` → `RedisSessionStore` | 外置存储 + TTL 或会话表 | **Redis**（首选低延迟）或 **PostgreSQL** |
| **LTM** | 默认 `InMemoryLTMStore`；**已支持** `DATABASE_URL` → `SQLLTMStore`（SQLAlchemy，如 MySQL） | 关系表 + 索引；与 API 契约不变或可版本化迁移 | **PostgreSQL** / **MySQL** |
| **向量索引** | `InMemoryVectorIndex`（内存稀疏向量） | **pgvector** 或 **Milvus / Qdrant**；嵌入模型调用结果持久化 | PG 扩展或向量库 |
| **配额 / QPS** | 进程内计数器 | 按用户 + 日期的原子计数 | **Redis** `INCR` + 过期 或 PG 聚合 |
| **Eval 任务** | 内存 `_jobs` | 持久 job + 状态机 | **Redis** 队列 或 **PG + Celery/RQ**（按需） |
| **Trace** | 文件 `FileTraceStore`（单写者友好） | 多实例时：**单写卷** 或迁 **OpenSearch/Elastic** 或 **PG JSONB**（按查询需求选） | 卷 / PG / 日志栈 |
| **情绪 / 反馈** | JSONL 追加 | 查询频繁则 **表结构**；导出仍支持 JSONL | **PostgreSQL** |
| **Token 统计** | 字符长度近似 | **tiktoken** 或厂商返回的 usage（若 API 提供） | 无或轻量库 |
| **工具** | 时间/天气等 | 参数校验与超时已有；可扩展 **备忘录类** 需 DB 表 | PostgreSQL |

## 4. 推荐技术栈（三期默认组合，可按规模裁剪）

| 层级 | 推荐 | 说明 |
|------|------|------|
| 应用 | FastAPI + Uvicorn（保持） | 多 worker 时注意 STM/配额已外置 |
| 关系库 | **PostgreSQL 15+** | LTM、用户、会话元数据、Eval job、反馈表等 |
| 向量 | **pgvector**（同 PG）或 **Milvus** | 求简单运维选 pgvector；海量向量再拆 Milvus |
| 缓存 / 会话 | **Redis 7+** | STM、配额、分布式锁、可选 SSE 连接元数据 |
| 前端 | React + Vite（保持） | 流式、`openapi-typescript`、Vitest/Playwright |
| 部署 | **Docker Compose** | `api` + `web` + `postgres` + `redis` +（可选）`nginx` |

**SQLite** 仅建议作为「单机演示降级」，与「多实例」目标冲突时以 PG 为准。

### 4.1 连接信息与协作约定（Redis / MySQL 或 PostgreSQL）

- **代码已接入（可选）**：在 `.env` 中配置 **`REDIS_URL`** 后，短期记忆（STM）使用 Redis；配置 **`DATABASE_URL`**（如 `mysql+pymysql://user:pass@host:3306/db`）后，长期记忆（LTM）使用关系库表 `ltm_items`（启动时 `create_all`）。未配置时仍为进程内实现，便于 pytest 与本地零依赖运行。  
- **`GET /health`** 响应体含 **`redis`** / **`database`**：`skipped`（未配置）、`ok`（探活成功）、`error`（配置了但连不上）。  
- 真实主机、端口、库名、账号密码仍**须由维护者自行提供**并只写入本地 `.env`；**切勿**把凭证提交进仓库或长期留在聊天里。  
- 选型上与 §4 表一致时优先 **PostgreSQL**；若你指定 **MySQL**，以你的环境为准（当前 SQL 层为通用 SQLAlchemy，MySQL 已验证路径）。

## 5. 里程碑建议（界面与数据层交错推进）

| 阶段 | 主题 | 产出 |
|------|------|------|
| **P3-M1** | 体验基线 | 流式对话 + 会话重命名 + 侧栏可调；设计令牌文档 |
| **P3-M2** | 数据骨架 | Compose 起 PG + Redis；**LTM/STM 迁移**最小闭环（迁移脚本 + 双写/读新优先可选） |
| **P3-M3** | RAG 真持久 | 向量进 **pgvector**（或选定向量库）；`embedding_status` 与重试任务 |
| **P3-M4** | 运营与评测可靠 | Eval job 持久化；反馈/情绪**可查表**；配额 Redis 化 |
| **P3-M5** | 观测与部署文档 | Trace 多实例策略定稿；CI（pytest + build）；性能基线一页纸 |

不必严格顺序；若团队偏前端，可先 **M1** 再并行 **M2**。

## 6. 风险与范围控制

- **大迁移**：LTM 从内存迁 PG 需 **导出/导入** 工具与回归测试，避免演示数据丢失。  
- **向量维度变更**：嵌入模型更换需 **全量重嵌入** 任务与版本字段。  
- **界面范围**：「像头部产品」易无限细化——以 **§1.2 硬指标** 为合同，其余进 backlog。
