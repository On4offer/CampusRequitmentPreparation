# V1.2 迭代规划（总体规划对齐 · JD 叙事补强 · 通向三期）

> **定位**：V1.1 已闭环「隐式 LTM + 异步抽取 + 可选远程 embedding + Web/运营热配」后，用 **小而实** 的增量对齐《整体规划》叙事与常见 **校招 JD 关键词**，避免与 **三期**（pgvector、Compose 全栈、界面标杆）抢体量。  
> **用途**：简历/面试话术、下一迭代排期、与《整体规划》《三期迭代规划》交叉索引。

---

## 1. 总体规划符合度（诚实对照）

《整体规划》里的 **终局能力** 与 **当前仓库** 对比如下（≠ 失败，而是 **分阶段交付**）。

| 总体规划要点 | 当前版本 | 说明 |
|--------------|----------|------|
| 多轮对话 + 策略/情绪驱动 | **符合** | STM、共情策略、安全模式、工具路由等已落地。 |
| 双层记忆 STM + LTM | **符合** | 内存默认；**可选 Redis（STM）+ DATABASE_URL（LTM）** 与规划一致。 |
| RAG：召回 + 证据注入 | **符合** | `RAG_ENABLED` 时检索 LTM；citations、query 改写、混合（向量+BM25）已有。 |
| 向量索引形态 | **部分符合** | 使用 **进程内 `InMemoryVectorIndex`**（稀疏字符/词 + 可选 OpenAI 兼容 embedding 压成稀疏维度字典），**不是** FAISS / Milvus / **pgvector** 持久向量库。 |
| 切分 + chunk 元数据 | **简化** | 以 **整条 LTM 为检索单元**，非通用文档分块 pipeline；与「陪伴记忆」场景匹配，但与规划 §5.1 通用 RAG 叙述有差距。 |
| Rerank | **未做** | 三期建议项。 |
| 离线评测 + 指标 | **部分** | Eval Lab、builtin JSONL、trace；**检索 Recall@K 等系统化报表**仍弱。 |
| 可部署 Docker | **部分** | 本地 uvicorn + Vite；**Compose 一键环境**在三期规划，仓库可 V1.2 先补 **最小 compose**。 |
| 关系进度/时间线/纪念日 | **未做** | 明确长期项，归三期或独立产品迭代。 |

**结论**：当前版本在 **「Agentic RAG 最小闭环 + 工程化后端 + 可观测」** 上与总体规划 **方向一致**；在 **持久向量库、通用切块、重排、评测指标体系** 上 **尚未达到终局**，应在面试中主动说明 **选型与 trade-off**（见 §4）。

## 2. RAG /「向量库」到底用到了什么

避免简历写「用了向量库」而面试穿帮，建议统一话术：

| 组件 | 实现 | 是否独立向量数据库 |
|------|------|---------------------|
| 向量相似度 | `InMemoryVectorIndex`：稀疏 `dict` 点积（归一化后≈余弦） | **否**，随进程内存，重启需 **warm** 重建。 |
| 文本词面 | `InMemoryBM25` + RRF/加权，**混合检索** | **否**。 |
| Embedding | 默认 **本地稀疏**；可选 **`RAG_EMBEDDING_API_*`** 调 OpenAI 兼容 `/embeddings`，再映射为稀疏维度 | **否**，未写入 pgvector/Milvus。 |
| 数据落库 | LTM 元数据在 **MySQL 等**（若配 `DATABASE_URL`）；**向量不单独落库** | 与三期「索引与存储分离」仍有差距。 |

**一句话**：**RAG 已用于生产路径，但向量索引是应用内存形态；不是 Milvus/pgvector 那种「向量库服务」。**

## 3. 校招 JD 常见技术 · 对照与可补强项

下列为大陆「后端 / 后端+AI」JD 高频词与仓库现状；**可加入** 指适合放进 **V1.2** 的低成本项。

| 技术/关键词 | 现状 | V1.2 是否建议纳入 |
|-------------|------|-------------------|
| Python / FastAPI / Pydantic | **有** | 维持；可补 **API 版本前缀或 OpenAPI 标签整理**（可选）。 |
| Redis | **可选接入**（STM、隐式 LTM 异步队列） | **文档 + 简历 bullet 写清场景**；可选 **compose 带 redis**。 |
| MySQL / PostgreSQL / SQLAlchemy | **LTM 可选 SQL** | 同左；V1.2 可 **示例 `docker-compose` 只起 MySQL+Redis**。 |
| RAG / LangChain | **自研管线，无 LangChain** | 面试说明「手搓 RAG」；V1.2 **不强行引入 LangChain**（避免臃肿），除非明确学习目标。 |
| 向量数据库（Milvus / pgvector） | **未用** | **归三期**；V1.2 仅在文档写「三期路线图」。 |
| Docker / Compose | **弱** | **V1.2 P0 推荐**：`docker-compose.dev.yml`（api + redis + mysql 可选）。 |
| 消息队列 Kafka/RabbitMQ | **无**（LTM 异步用 **Redis list**） | **不强制**；JD 若强依赖 MQ，可 V1.2 写设计对比说明即可。 |
| pytest | **有** | V1.2：**关键路径覆盖率**或 **CI 示例 job**（GitHub Actions 片段写入文档）。 |
| OpenAPI → TS | README 有示例 | V1.2：`web/package.json` **脚本** `gen:api` + 说明何时跑。 |
| 压测 locust | **无** | V1.2：**极小 locustfile**（只打 `/health` + `/chat` mock）或 **文档占位**。 |
| 结构化日志 structlog / loguru | **标准 logging** | V1.2：**可选** `structlog` 或 JSON 一行日志开关（环境变量）。 |
| Kubernetes | **无** | 归三期/实习公司栈，**不纳入 V1.2**。 |

## 4. 面试话术建议（总体规划 + 现状）

- **为什么不用 pgvector 先做**：单机演示与 pytest 零依赖优先；LTM 条数可控时 **内存索引 + 启动 warm** 足够；**三期**再按规模拆「索引与元数据分离」。  
- **RAG 优化做过什么**：混合检索、query 改写、阈值与 `rag_max_chars`、Trace 里命中与改写可观测；**未做** cross-encoder rerank。  
- **记忆污染怎么控**：显式 CRUD + 隐式抽取 **默认关**、每 N 轮、置信度、去重/合并、撤销、`source=dialogue_extract` 可筛可删。

## 5. V1.2 版本目标（建议边界）

**主题**：**「对齐叙事 + JD 可写 + 不打开三期闸门」**

### P0（强烈建议，1～2 周量级）

| ID | 项 | 验收 |
|----|-----|------|
| V12-01 | **docker-compose**（`api` + 可选 `redis` + 可选 `mysql`）+ `.env.example` 注释对齐 | 新人 `docker compose up` 能起依赖并连上现有 `DATABASE_URL`/`REDIS_URL` |
| V12-02 | **OpenAPI → TS** 写入 `package.json` 脚本 + 《常用命令》一条 | 后端启动后一条命令生成 `schema.d.ts`，减少类型漂移 |
| V12-03 | **技术栈说明页**：`README` 或 `#Personal_Documents` 短篇 **「与整体规划 / JD 对照」**（可链接本文 §1～§3） | 投递不同 JD 时可快速改简历 bullet |

### P1（增强可信度）

| ID | 项 | 验收 |
|----|-----|------|
| V12-04 | **Eval / RAG**：对 builtin 或小型集输出 **HitRate@K 或 Recall@K**（规则：相关记忆 id 是否出现在 topK） | 文档或脚本打印 Before/After 表（哪怕样本 20 条） |
| V12-05 | **locust** 极简压测或 k6 单文件 | 能演示「并发下 /health、限流行为」 |
| V12-06 | **CI 示例**：pytest on push（文档化即可，不要求你必须开 GitHub） | `README` 或 `.github/workflows` 样例 |

### P2（可选）

| ID | 项 | 验收 |
|----|-----|------|
| V12-07 | **结构化日志**（JSON 行 + `trace_id` 字段） | 开关由环境变量控制，默认保持现状 |
| V12-08 | **Web Vitest** 抽 1～2 个纯函数/ hook 测试 | 证明前端工程化意识，不做全量 E2E |

### 明确不纳入 V1.2（避免范围膨胀）

- **pgvector / Milvus / 通用文档切块 pipeline** → **三期**  
- **Trace/Eval/配额多实例强一致** → **三期**  
- **关系时间线、纪念日、画像卡片** → **整体规划长期项 / 独立版本**

若岗位 JD **强要求 LangChain**：另见 **`框架化重构指南_LangChain向.md`**（主链已统一 LCEL；文档含迁移史与验收项，可与 V1.2 并行排期）。
