# 二期迭代规划：功能完善 · 界面美观 · 真实可用

> **前提**：一期（V1～V5）已交付，详见《V1-V5一期验收说明.md》《V5开发规划.md》。  
> **二期总要求（三条同时满足）**  
> 1. **功能尽可能完善**：在不做多租户计费/企业 SSO 的前提下，把《整体规划》§12 中的「主界面 + 记忆 + 评测 + 运营/合规」做成**完整闭环**，少留「仅 Swagger 可用」的缺口。  
> 2. **界面美观**：有统一视觉体系（色板、字体、间距、动效），**拒绝**廉价默认皮肤；深色模式、空状态、关键反馈都要有设计。  
> 3. **真实可用**：主链路**全部对接真实后端 API**，日常可自己拿来连续使用；含加载/错误/限流等体验，不是「只能录屏」的静态 demo。  
>
> **仍不包含（三期/独立文档）**：**部署与运维交付**——Docker/Compose、K8s、生产 CI/CD、域名与 HTTPS 上线、Nginx 网关等；与「本地真实可用」不矛盾。  
> 本文可作为排期、选型答辩与简历「下一阶段」描述素材。

---

## 0. 二期定位与边界

| 维度 | 二期要做（高标准） | 二期不做（可放三期） |
|------|-------------------|----------------------|
| **产品完成度** | 聊天 / 记忆 / 评测 / 情绪与 Trace / 运营与合规在 UI 层**可独立完成全流程** | 多租户计费、复杂 RBAC、企业 SSO |
| **用户体系** | **建议**：简单登录（JWT）或「设置页持久化 user_id + Token」；与后端隔离策略一致 | 手机号实名、OAuth 全家桶 |
| **数据与状态** | 会话列表、最近会话、关键设置 **localStorage 持久化**；错误与 429 可读 | 容器编排、多实例一致性（属部署域） |
| **评测** | Eval Lab：**选集 → 跑批 → 进度 → 结果落表**，可导出；失败可点进 trace | 线上 A/B 分流 |
| **部署** | **不在范围内** | 见上文「仍不包含」 |

## 1. 建议技术栈（对齐「美观 + 真实可用」）

### 1.1 二期前端路径（强烈建议）

| 推荐 | 说明 |
|------|------|
| **主推：Vite + React 18 + TypeScript** | 组件生态成熟，易做精致 UI 与复杂交互；**二期默认选此**，除非团队只会 Vue（可换 Vue3 + Vite 同等标准）。 |
| **备选：Next.js + Tailwind** | 需要更强路由/布局或后续要 SEO 时；本地 `next dev` 即「真实可用」。 |
| **不推荐作为二期主产品**：仅加强 Gradio | 难同时满足「美观 + 复杂信息架构」；可**保留**作内部调试，与主站并存时职责要划清。 |

### 1.2 后端（在现有 FastAPI 上扩展）

| 能力 | 推荐技术 | 说明 |
|------|-----------|------|
| 主服务 | **FastAPI + Uvicorn**（保持） | 增量 API，OpenAPI 与前端类型同步 |
| 配置 | **pydantic-settings** + 热配置（保持） | 运营参数仍走现有机制 |
| 长任务 | **BackgroundTasks** + 简易 job 状态（内存 dict 或文件） | 评测跑批必须异步 + 可轮询进度；**不强制** Redis |
| 鉴权 | **JWT** 或沿用 **Admin Token + user_id** | 真实可用至少要有「我是谁」与运营接口保护 |
| 类型同步 | **openapi-typescript** | 减少前后端字段漂移 |

### 1.3 UI 与动效（美观硬指标）

- **样式**：**Tailwind CSS** + **shadcn/ui**（Radix）或 **Ant Design 5**（需自定义 token 色板，避免默认蓝灰敷衍）  
- **字体**：**必须**引入一组 Display + 一组 UI 正文字体（Google Fonts 或国内合规 CDN），禁用纯系统字体凑合  
- **图标**：Lucide / Phosphor，风格统一  
- **图表**：**ECharts** 或 **Recharts**，主题与全局色板联动  
- **动效**：页面切换与列表 **150～300ms** 过渡；按钮与卡片 **hover/focus** 态完整  
- **主题**：**浅色 + 深色**双主题，由用户切换并持久化  
- **Markdown**：助手消息 **GFM**（列表、表格、代码块、引用等）已接入 `react-markdown` + `remark-gfm`

### 1.4 工程与质量（不含部署发布）

| 项 | 建议 |
|----|------|
| 前端测试 | Vitest + RTL：**本期未接入**；后端 **pytest** 已覆盖主链与合规/配额等（建议后续补前端单测） |
| E2E | Playwright：**本期未接入**（建议三期或与部署一并补） |
| 格式化 | Ruff + Black；Prettier + ESLint |

## 2. 功能模块（二期尽量「做全」，按块验收）

### 2.1 聊天主界面（真实可用核心）

- [x] **会话列表**：`GET /sessions`；新建、切换、删除（可选 **重命名** 未做，仍显示 `session_id` 短码）  
- [x] **消息区**：气泡、发送中加载态、**长消息折叠/展开**、**复制全文**、时间戳；助手 **Markdown（GFM）** 渲染；**流式输出**未做（仍为建议项 / 可放三期）  
- [x] **证据与工具**：`citations` → 可点击跳转 **Memory Studio**（`?ltm_id=` 深链打开编辑）；`tool_summary` 分字段卡片展示  
- [x] **诊断侧栏**：情绪、模式、trace_id、latency、token；复制 trace_id；链接 **Trace 回放**；配额省流提示  
- [x] **反馈**：点赞/点踩 + 纠错 → `POST /feedback`；**全局 Toast** 成功提示  
- [x] **异常**：429/502/401 **分层中文说明** + **重试上一次** + 关闭；不白屏  

### 2.2 Memory Studio（记忆管理）

- [x] 列表：类型筛选、搜索、分页，对接 LTM API  
- [x] CRUD：创建 / 编辑 / 软删除，表单校验与提示  
- [x] 展示：`confidence`、`source`、`tags`、时间；空状态与加载；**对话 citation 深链打开单条**  

### 2.3 Eval Lab（评测）

- [x] 数据集：JSONL / 内置集与校验提示  
- [x] 运行：`POST /admin/eval/run` → `job_id` → 轮询进度  
- [x] 结果：指标 + 案例表；导出；失败关联 trace（视后端字段）  

### 2.4 情绪仪表盘（统计向）

- [x] `GET /emotion/stats` + 日/周；免责声明  
- [x] 图表/表格与无数据引导  

### 2.5 Trace 回放

- [x] `trace_id` 拉取完整 trace；步骤时间线 + **RAG/工具/配额/安全** 筛选  
- [x] decision/metrics **折叠** JSON；**复制完整 JSON**  

### 2.6 运营与合规（与一期 API 对齐）

- [x] **运营台页**：热参数、配额快照、反馈列表、风险事件——**全部走真实接口**（`/admin/*`、`/feedback/recent` 等），布局清晰，不单依赖 Gradio  
- [x] **导出/删除**：`GET /users/{id}/export`（JSON：LTM+STM+反馈）、`POST /users/{id}/forget`（`confirm` + 可选 `clear_stm`）；Web `/ops`「合规」Tab；操作前勾选确认，完成后 Toast  

### 2.7 配额降级（补一期能力）

- [x] 超额时：**可配置** `quota_degrade_on_exhaust`：去掉本轮 RAG 注入并追加省流 system 提示（先「提示+无 RAG」再「仅无 RAG」重试预算）；关闭则仍 429。  
- [x] trace：`quota_exceeded` + `degraded_mode`；`/chat` 响应 `quota_degraded`；热配置 `quota_degrade_system_hint`（空则内置短句）。

## 3. 「真实可用」体验清单（与美观并列验收）

| 类别 | 要求 |
|------|------|
| 数据真实 | 主流程 **禁止** 长期用假数据；开发期 mock 仅限非主链路口 |

| 可访问性基础 | 焦点可见、表单 label、主要对比度达标 |
| 安全提示 | 管理 Token、**user_id 变更影响**已在聊天侧栏提示 |

## 4. 界面美化（设计方向补充）

- **气质**：温暖、克制、可信；避免「AI Slop」通用渐变与 Inter 堆叠。  
- **布局**：聊天三栏 **比例可调**（拖拽或折叠侧栏）；移动端可降级为 Tab（二期若只做桌面可先标注）  
- **组件密度**：运营与记忆页偏 **中后台信息密度**；聊天区偏 **留白与阅读舒适**  
- **设计资产**：至少 **一版主色与深色** 的 CSS 变量表；可选 Figma 组件库与两页高保真

## 5. 后端 API 增量清单（须与前端合同同步）

| 方法 | 路径 | 用途 |
|------|------|------|
| GET | `/sessions` | 用户会话列表与元数据 |
| GET | `/emotion/stats` | 仪表盘聚合 |
| POST | `/admin/eval/run` | 启动评测任务 |
| GET | `/admin/eval/jobs/{id}` | 任务状态与结果 |
| GET | `/users/{id}/export` | 合规导出（LTM+STM+反馈 JSON，admin 鉴权）✓ |
| POST | `/users/{id}/forget` | 批量软删 LTM + 可选清 STM（`confirm` 必选）✓ |

*字段、鉴权与错误码在 OpenAPI 中写死，前端生成类型。*
