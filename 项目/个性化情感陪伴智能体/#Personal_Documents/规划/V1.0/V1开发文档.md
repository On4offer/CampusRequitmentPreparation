# V1 开发文档（学习记忆 · 设计框架 · 代码结构）

> 面向“理解与记忆”的 V1 技术文档：设计依据、整体架构、目录与代码结构、六模块设计、配置与扩展点。便于复习和后续 V2/V3 开发时快速定位。

---

## 1. 定位与目标

### 1.1 V1 在版本中的位置

- **V0**：多轮对话、Trace、情绪识别、状态机（闲聊/倾听/安慰/工作）、安全模式、Gradio 诊断、演示与回归。
- **V1**：在 V0 上做“可配置、可扩展、为后续铺路”，**不改变主链路语义**（默认行为与 V0 一致），新增均为可选或占位。
- **V2**：LTM + 最小 RAG（检索注入 /chat）；**V3**：真实工具（time/memo/weather）+ 护栏。

### 1.2 V1 要达成的三件事

1. **可配置**：共情策略（mode 话术与规则）、情绪落盘、安全/关注档策略开关均可通过配置或环境变量切换，无需改代码。
2. **可扩展**：状态机增加“工具”分支并在 trace 中记录 intended_tool；LTM 数据模型与 API 占位，为 RAG 与工具接入留好接口。
3. **可观测**：情绪可落盘（为曲线/回访预留）；/chat 成功打结构化日志；Trace 含工具占位字段。

## 2. 设计框架与原则

### 2.1 设计原则（便于记忆）

| 原则 | 含义 |
|------|------|
| **默认等价 V0** | 不配任何 V1 选项时，行为与 V0 完全一致，回归用例全过。 |
| **配置优于硬编码** | 策略、开关、路径等从 settings/env 或配置文件读，避免写死在业务代码里。 |
| **占位先行** | 工具分支、LTM、情绪落盘先做“能写能查、结构固定”的占位，真实实现后直接替换或扩展。 |
| **主链路不绑 LTM** | /chat 不调用 LTM；LTM 仅通过独立 API 读写，V2 再在 /chat 中做检索与注入。 |
| **可观测可回放** | 每轮有 trace_id，decision 含 emotion/mode/intended_tool；关键步骤打日志，便于排障与复盘。 |

### 2.2 请求主链路（/chat）概览

```
请求入参 (message, user_id?, session_id?)
    → 规范化 user_id / session_id
    → 情绪识别 (analyze_emotion) → emotion_result, safety_mode
    → 状态机 (decide_mode) → mode, mode_reason, intended_tool?, tool_params_placeholder?
    → [可选] 安全/关注档覆盖 mode（受 safety_force_listen / watch_tier_force_listen 控制）
    → 选 system prompt（安全模式用 SAFE，否则 get_system_prompt_for_mode(mode)）
    → [可选] 情绪落盘 append_emotion_record（受 emotion_log_enabled 控制）
    → 组装 messages = [system] + history(STM) + user
    → LLM chat → reply
    → 写回 session_store（STM）
    → 写 trace（含 decision.emotion/mode/intended_tool 等）
    → 打日志 chat_done
    → 返回 reply, trace_id, user_id, session_id
```

- **LTM 不在此链路**：只通过 POST/GET /memory/ltm 独立使用。

## 4. 六模块设计依据与结构

### 4.1 模块一：共情策略可配置化

- **设计依据**：话术与规则常需调参，若写死在代码里则每次改都要发版；抽成配置后运营/产品可改 JSON，重启即生效。
- **结构**：
  - **配置形态**：JSON 文件，含 `mode_prompts`（mode → system 文案）、`explicit_rules`（关键词 → mode/reason）、`emotion_to_mode`（情绪标签 + min_intensity → mode/reason）。
  - **加载时机**：`create_app()` 中根据 `settings.policy_config_path` 解析路径并调用 `reload_policy_config(path)`；路径为空或文件不存在则用代码内 `_DEFAULT_*`。
  - **使用方式**：`decide_mode` / `get_system_prompt_for_mode` 内优先读 `_policy_config`，缺省再回退默认。
- **关键类型**：无新增 Pydantic；配置为普通 `dict`，键与 `config/policy_config.json` 一致。

### 4.2 模块二：状态机“工具”分支与占位

- **设计依据**：真实工具（V3）需要“先识别意图、再执行”，先在状态机里占位“工具”分支并落 trace，后续 V3 只需接执行层与护栏。
- **结构**：
  - **Mode**：由四类扩展为五类，增加 `"工具"`。
  - **优先级**：工具关键词 > 显式指令 > 情绪规则；工具关键词命中则返回 `(mode, reason, intended_tool, tool_params_placeholder)`，否则后两为 `None`。
  - **工具关键词**：代码内 `_TOOL_KEYWORDS`，如 天气/查天气 → weather，记一下/备忘 → memo，几点了/时间 → time。
  - **Trace**：`TraceDecision` 增加 `intended_tool`、`tool_params_placeholder`；仅当 mode=工具时非空，安全/关注档覆盖 mode 时清空。
- **关键类型**：`Mode`（Literal 五选一）；`decide_mode` 返回 `tuple[Mode, str, str | None, dict | None]`。

### 4.3 模块三：情绪结果持久化占位

- **设计依据**：情绪曲线、回访等需要“按时间/用户聚合情绪”，先有稳定写入格式与开关，后续只加查询与展示。
- **结构**：
  - **写入时机**：在得到 `emotion_for_trace` 和 `trace_id` 后，若 `emotion_log_enabled` 且 `emotion_log_path` 非空，调用 `append_emotion_record(path, record)`。
  - **记录格式**：单行 JSON，字段 `user_id, session_id, timestamp_ms, label, intensity, risk_tier, trace_id`，与 trace 中 emotion 对齐。
  - **实现**：追加写 JSONL；目录不存在则创建；写失败静默忽略，不打断主链路。
- **关键类型**：无 Pydantic；`record` 为 `dict`，由 routes 拼好传入。

### 4.4 模块四：长期记忆（LTM）数据模型与接口占位

- **设计依据**：整体规划中 LTM 按类型建模（Preference/Profile/Event/Constraint），V2 RAG 需要“能写、能按 user/type 查”；先定模型与 API，占位存储，V2 再换向量/DB 并接入 /chat。
- **结构**：
  - **模型**：`LTMItem`（id, user_id, type, content, created_at, source, confidence, tags）；`type` 为 Literal 四选一。
  - **存储**：`InMemoryLTMStore`，`put(user_id, item)` 返回 id，`get_by_id(id)`，`list_by_user(user_id, type=None, limit=N)`；按 user 索引，列表按写入顺序倒序。
  - **API**：`POST /memory/ltm`（body 含 user_id, type, content, source, confidence, tags）→ `{ id }`；`GET /memory/ltm?user_id=&type=&limit=` → `{ items: LTMItemOut[] }`。不接入 /chat。
- **关键类型**：`LTMItem`（Pydantic）, `LTMType`（Literal）, `LTMWriteRequest` / `LTMItemOut` / `LTMListResponse`（schemas）。

### 4.5 模块五：可观测与配置外置

- **设计依据**：策略开关与落盘开关若写死，则 A/B 或关停某功能必须改代码；外置后通过 env 即可切换。可观测需要每轮有可检索的日志。
- **结构**：
  - **配置外置**：`safety_force_listen`、`watch_tier_force_listen` 在 settings 中，routes 中“是否用安全/关注档覆盖 mode”依此判断。
  - **结构化日志**：/chat 成功返回前 `logger.info("chat_done trace_id=... latency_ms=... mode=... risk_tier=... user_id=...")`，便于 grep 或接入日志系统。
- **关键类型**：无新增；依赖 `settings` 与标准库 `logging`。

### 4.6 模块六：回归与演示更新

- **设计依据**：每版都要“可回归、可演示”，避免功能堆上去却无法验证。
- **结构**：
  - **回归**：`data/regression_v0.jsonl` 增加工具用例（expect_mode=工具、expect_intended_tool）；`test_day7_regression` 断言 intended_tool；`test_day5_policy` 覆盖五类 mode 与 4 元组。
  - **演示**：`demo_script.md` 增加“工具占位演示”一节；验收项增加“工具向输入 mode=工具、trace 含 intended_tool”。

## 5. 配置体系（V1 相关）

- **来源**：环境变量 + `.env`（pydantic-settings 加载到 `app.core.settings.settings`）。
- **V1 相关项**（均为可选，不配则用代码默认）：

| 变量 | 含义 | 默认 | 生效时机 |
|------|------|------|----------|
| POLICY_CONFIG_PATH | 共情策略 JSON 路径（相对项目根） | 空 | 进程启动时在 create_app 中加载 |
| EMOTION_LOG_ENABLED | 是否情绪落盘 | false | 每轮 /chat 判断 |
| EMOTION_LOG_PATH | 情绪 JSONL 路径 | data/emotion_log.jsonl | 同上 |
| SAFETY_FORCE_LISTEN | 高风险是否强制 mode=倾听 | true | 每轮 /chat 判断 |
| WATCH_TIER_FORCE_LISTEN | 关注档是否优先 mode=倾听 | true | 每轮 /chat 判断 |

- **修改配置后**：改 .env 或 policy 配置文件后需**重启后端**才能生效（策略文件在 create_app 时读，env 在进程启动时读）。

## 6. 关键代码路径（便于记忆与排障）

- **策略加载**：`main.create_app()` → `reload_policy_config(project_root / settings.policy_config_path)`。
- **单轮决策**：`routes.chat` → `analyze_emotion` → `decide_mode` → [安全/关注档覆盖] → `get_system_prompt_for_mode(mode)`。
- **工具命中**：`decide_mode` 内先遍历 `_TOOL_KEYWORDS`，命中即返回 `("工具", reason, tool_id, {})`。
- **情绪落盘**：`routes.chat` 中 `if settings.emotion_log_enabled and settings.emotion_log_path` → `append_emotion_record(...)`。
- **LTM 写入**：`POST /memory/ltm` → `LTMItem(..., created_at=now_ms())` → `ltm_store.put(user_id, item)`。
- **Trace 写入**：所有 `trace_store.put(TraceRecord(..., decision=TraceDecision(..., intended_tool=..., tool_params_placeholder=...)))` 均带 V1 字段。

## 7. 与 V0 / V2 / V3 的衔接

- **V0**：V1 默认行为与 V0 一致；仅在使用可选配置或命中工具分支时才有可见差异。
- **V2**：复用 `LTMItem`、`ltm_store` 接口与 `POST/GET /memory/ltm`；在 /chat 中增加“用 query 检索 LTM → 注入 prompt”即可，无需改 LTM 模型与 API 契约。
- **V3**：复用 mode=工具 与 `decision.intended_tool`；实现真实 time/memo/weather 与最大步数/超时/熔断等护栏，替换占位执行即可。
