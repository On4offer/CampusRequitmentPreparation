# 整体检查后迭代规划（质量 · 工程化 · 体验收口）

> **来源**：在二期功能闭环与 Web 控制台交付之后，对代码与使用路径做一次**整体检查**（主链路逻辑、前后端一致性、默认配置与文档、API 边界行为）。  
> **定位**：本文是**二期验收之后**的增量规划，不改变《二期迭代规划.md》中「二期已完成」的结论；与《整体规划.md》中三期/部署域区分——**仍以本地与单实例真实可用为主**，部署与多实例一致性可引用本文「长期项」并另文展开。  
> **原则**：优先 **P0 低成本高收益**（文档与契约修正、易踩坑 API）；**P1** 工程化与测试；**P2** 架构与成本优化按需排期。

## 1. 迭代目标与边界

### 1.1 本期要做

- 消除「按文档操作却联调失败」类问题（**端口与代理约定统一**）。
- 收紧 **API 契约**：对明显误用返回明确错误，避免静默空结果。
- 补齐 **新人上手路径**：根目录或 `web/` 的简明 README，与《常用命令.md》交叉引用。
- 建立**最小可重复**的质量门槛：依赖说明 + 关键用例可本地执行。

### 1.2 明确不做（或仅列为长期 backlog）

- 不改变二期已交付的**产品功能合同**（除非 P0 为纯修复类）。
- **生产部署**（Docker/K8s/HTTPS）仍不在本文排期内。
- **STM 换 Redis/DB** 属架构升级，列入 P2/三期，不强制与本文同里程碑完成。

## 4. 风险与依赖

- **P0-2 为行为变更**：若有外部脚本依赖「不传 session_id」的旧行为（虽无意义），需在 **OpenAPI 描述** 中写明必填，并在变更说明中一笔带过。
- **端口统一**涉及个人本地习惯：以「文档自洽 + 示例一致」为准，不必强制全局改端口数字，但必须**写明对应关系**。
- **P2 成本优化**可能触及安全与策略（情绪档位），需与《policy_config》与合规表述一致，避免「为省钱削弱风险识别」的误解。
