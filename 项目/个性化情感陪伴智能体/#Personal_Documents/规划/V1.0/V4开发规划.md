# V4 开发规划：RAG 深度优化 + 离线评测

> 当前基线：V3 已完成工具链路、护栏、trace、UI 工具摘要。V2 具备单路向量检索（稀疏 TF）、证据注入、citations。  
> V4 目标：**RAG 可量化优化**，做到 **优化前后有数字、退化可定位、评测可复现**。  
> 本文按模块组织，可直接作为执行与验收文档。

---

## 0. V4 目标与总验收（DoD）

- **目标**
  - 在 V2 单路向量检索基础上，引入至少 2 项优化：混合检索、Query 改写重试、Rerank
  - 构建 benchmark 与指标计算，产出「优化前后对比表」
  - UI 有评测页，一键跑评测并展示指标与典型案例

- **总验收（DoD）**
  1. 混合检索（向量 + BM25）或 Query 改写重试 至少实现 2 项；
  2. 能产出「baseline vs 优化版」指标对比表（至少 Recall@K 或 MRR）；
  3. 能定位退化 case 并解释原因（trace / 评测报告）；
  4. UI 评测页可一键跑 benchmark，展示指标与典型 case；
  5. 新增 V4 回归用例，pytest 全过，且不破坏 V1/V2/V3 关键用例。

---

## 1. 范围与边界

### 1.1 V4 要做

- 混合检索：向量 + BM25 融合（RRF 或加权）
- Query 改写/重试：首次召回弱/为空时，用 LLM 改写 query 再检索一次
- Rerank（可选）：对 TopK 做精排（可先用简单规则或轻量模型）
- Benchmark 数据集与指标计算
- 评测 API（`POST /eval/run`）与评测报告
- UI 评测页（Gradio 新增 tab 或独立页）

### 1.2 V4 不做

- 真实 embedding 模型替换（可后续单独迭代）
- 大规模 chunk 策略重构（LTM 仍按条存储，chunk 优化做最小增量）
- 复杂 A/B 分桶与线上实验（V5）
- 精细成本治理看板（V5）

---

## 2. 模块一：混合检索（向量 + BM25）

### 2.1 内容

- 新增 `app/rag/bm25.py`：基于 LTM 内容构建 BM25 索引（内存版）
- 扩展 `app/rag/retriever.py`：支持 `retrieve_hybrid`（向量 TopK + BM25 TopK → RRF 融合）
- 配置项：`rag_use_hybrid`、`rag_bm25_top_k`、`rag_fusion_method`（rrf/weighted）

### 2.2 验收

- 开启 `rag_use_hybrid` 时，检索结果融合向量与 BM25；
- 关闭时行为与 V2 一致；
- 对「问得具体却没命中」的 case 有改善（可量化）。

---

## 3. 模块二：Query 改写/重试

### 3.1 内容

- 新增 `app/rag/query_rewrite.py`：`rewrite_for_retrieval(query, context)`，调用 LLM 生成检索友好 query
- 在 retriever 中：首次召回为空或 score 均低于阈值时，自动改写 query 再检索一次
- trace 记录 `retrieve_ltm` 的 `rewrite_triggered`、`rewritten_query`

### 3.2 验收

- 首次召回弱时触发改写，trace 可看到改写前后 query；
- 改写后命中率提升（benchmark 可验证）；
- 改写失败或超时时降级为原 query 结果，不中断主链路。

---

## 4. 模块三：Rerank（可选）

### 4.1 内容

- 新增 `app/rag/rerank.py`：对 TopK 结果做精排
- 方案 A：简单规则（如关键词重叠加分、时间/置信度加权）
- 方案 B：轻量 cross-encoder 或 LLM 打分（可选，成本较高）
- 配置项：`rag_rerank_enabled`、`rag_rerank_top_n`

### 4.2 验收

- 开启时对召回结果重排，输出更相关的 TopN；
- 关闭时跳过 rerank，行为与未开启一致。

---

## 5. 模块四：Benchmark 与指标计算

### 5.1 内容

- 新增 `data/rag_benchmark.jsonl`：每条 `{query, user_id, preload_memories, expect_hit_ids, expect_reply_contains, expect_citation_type}`
- 新增 `app/eval/`：`run_rag_eval()` 跑 benchmark，计算 Recall@K、MRR、HitRate@K
- 输出报告：`data/eval_report_{timestamp}.json` 或 `.md`

### 5.2 验收

- 能对 baseline（单路向量）与优化版（混合/改写/rerank）分别跑评测；
- 产出对比表：Recall@3、MRR、HitRate@3 等；
- 能输出典型退化 case 列表（优化后变差的样本）。

---

## 6. 模块五：评测 API 与 UI

### 6.1 内容

- 新增 `POST /eval/run`：接收配置（如 `rag_use_hybrid`、`rag_rewrite_enabled`），跑 benchmark，返回指标与报告摘要
- Gradio：新增「评测页」tab，一键跑评测，展示指标表格与典型 case（成功/退化）

### 6.2 验收

- 不打开 Swagger 也能在 UI 跑评测、看结果；
- 2～3 分钟演示可完整复现「baseline vs 优化版」对比。

## 9. 风险与回滚策略

- **风险 1：混合检索/改写增加延迟**
  - 策略：配置开关控制，默认关闭；超时降级

- **风险 2：Query 改写依赖 LLM，成本与稳定性**
  - 策略：仅当首次召回弱时触发；改写失败直接降级

- **风险 3：Benchmark 过小或标注不准**
  - 策略：先做 20～50 条小规模集，保证可复现；后续扩充

- **风险 4：优化后指标反而下降**
  - 策略：保留 baseline 配置，可随时回退；报告中标明退化 case
