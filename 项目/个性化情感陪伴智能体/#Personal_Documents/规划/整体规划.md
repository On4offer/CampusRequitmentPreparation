# 面试版：个性化情感陪伴智能体（Agentic RAG + 工程化后端）开发规划

> 目标：做一个“能演示、能量化评测、能服务化部署、能讲清楚 trade-off”的真实落地型项目，而不是只会调 API 的 Demo。

------

## 0. 项目定位（建议你在简历/面试的第一句话）

**个性化情感陪伴智能体**：围绕“长期关系 + 情绪理解 + 可控工具执行”的需求，搭建一套可迭代的 **Agentic RAG** 系统。

- **核心逻辑手搓**：记忆管理（短期/长期）、RAG 检索管线（切分/召回/融合/改写）、工具编排（路由/执行/护栏）、对话状态机（策略切换）、评测与可观测（离线指标 + 在线链路日志）。
- **基础设施用成熟库**：Web 服务、向量索引、数据校验、UI 演示等，保证工程质量和可维护性。

面试官最爱追问的点，你要能用项目回答：
- 为什么需要双层记忆？各自存什么？何时写入/读取？如何避免污染？
- RAG 检索为什么不准？你做过哪些可量化优化？指标怎么证明？
- Agent 怎么避免死循环/乱调用工具/安全风险/成本失控？
- 这套系统怎么服务化、怎么观测、怎么在并发下稳定？

## 1. 最终能力清单（最终验收）

### 1.1 用户侧体验
- **多轮对话**：不丢上下文，支持“闲聊/倾听/安慰/工作模式/工具模式”切换。
- **情绪理解**：识别情绪与强度（如：低落/焦虑/愤怒/疲惫/开心），并驱动回复策略。
- **长期个性化**：能记住偏好、禁忌、重要事件、时间线（“上次你说…”），并在合适时机自然提取。
- **工具能力**：查时间/天气/备忘录，支持失败处理与解释。

### 1.2 情感陪伴特色（让它不像玩具）
- **情绪曲线与回访**：为每次对话输出 `emotion(label,intensity)`，形成“日/周/月”情绪曲线；在合适时机触发回访（“你前两天提到…最近好些了吗？”）。
- **共情策略库（可解释）**：把“倾听/反映/澄清/边界确认/行动建议”做成可配置策略（规则 + LLM），并在 trace 中记录“为什么这样回”（证据：情绪/意图/用户偏好）。
- **关系进度与成长档案**：把长期记忆按主题聚类（工作/家庭/健康/爱好），形成“用户画像卡片”和“重要事件时间线”，让陪伴有连续性。
- **纪念日与提醒**：从对话中抽取日期类信息（生日/面试/DDL），写入日程工具；在 UI 中可看到“即将到来”列表。
- **危机干预与安全模式**：检测自伤/他伤/极端情绪时进入“安全模式”（降低自由发挥、给出求助建议、引导联系现实支持），并在日志中标注。
- **边界与偏好一致性**：用户明确的边界（不要鸡汤/不要说教/称呼偏好）进入 `Constraint/Preference`，生成时作为强约束；可在 UI 一键查看与编辑。

### 1.3 工程侧能力
- **完整后端服务**：FastAPI 提供会话接口、记忆接口、评测接口、管理接口。
- **可观测性**：每次对话链路可回放（检索命中、工具调用、耗时、token 成本、失败原因）。
- **离线评测**：至少一套小规模 benchmark（例如 50~200 条）衡量检索与回答质量。
- **可部署**：本地一键启动（可选 Docker），环境变量管理密钥。
- **可运营**：关键参数可配置（策略阈值、写入规则、TopK、重试次数、限流配额），无需改代码就能调优。

## 3. 总体架构（你要能画出来，并能对每一步解释“为什么”）

### 3.1 一次对话的链路（端到端）
1. **输入预处理**：长度限制、敏感内容与自伤风险检测（最小规则也可）
2. **对话状态机**：根据意图 + 情绪 + 历史状态决定当前策略（闲聊/倾听/安慰/工具/工作）
3. **短期记忆（STM）**：滑窗 + 关键摘要（受 token 预算约束）
4. **长期记忆（LTM）检索（RAG）**：
   - query 生成/改写（必要时）
   - 召回（向量/混合）
   - 过滤与去重（时间、类型、冲突）
   - 可选 rerank（TopK 精排）
5. **工具路由（可选）**：是否需要工具；参数校验；超时/重试/熔断；结果落盘
6. **prompt 组装**：系统提示词 + 状态策略 + STM + LTM 证据 + 工具结果
7. **生成与后处理**：引用证据/表达不确定性；风格控制与安全规则
8. **写入与观测**：选择性写 LTM（带置信度/来源/时间）；写 trace 日志与评测样本

### 3.2 “交给 LLM” vs “必须硬控”
- **交给 LLM**：意图/情绪推断、query 改写建议、回复措辞、工具是否需要的软判断（但要护栏）
- **必须硬控**：工具参数、最大步骤数、超时、重试策略、token 预算、数据写入规则、安全策略

## 4. 记忆系统设计（面试高频）

### 4.1 短期记忆 STM（解决连贯性与成本）
**目的**：保证最近对话连贯，控制 token 成本。

- **机制**：滑动窗口 + 摘要压缩（阈值触发）
- **验收**：同一话题持续 20+ 轮仍不丢关键约束（例如“不要鸡汤”“叫我 X”“我的目标是 Y”）

### 4.2 长期记忆 LTM（RAG，解决跨会话个性化）
**目的**：跨会话长期个性化，且可检索、可解释、可删除。

建议把记忆按类型建模（讲起来更像业务系统）：
- **Preference**：偏好/厌恶（食物、称呼、风格）
- **Profile**：背景信息（工作、专业、关系）
- **Event**：重要事件与时间线
- **Constraint**：边界与禁忌（不可触碰话题/输出限制）

每条记忆最小字段建议：
- `id, user_id, type, content, created_at, source, confidence, tags`

**写入策略（避免污染）**：
- 只有满足“可复用 + 稳定”的信息才写入 LTM
- “情绪化临时表述/不稳定观点”降低 `confidence` 或只写 STM
- 支持“用户纠错/更新/删除”（合规意识加分）

## 5. RAG 设计与优化（必须能量化对比）

### 5.1 最小可用 RAG（MVP）
- 切分：按段落/标点递归切分 + overlap；保留 chunk 元数据（来源、时间、类型）
- 召回：向量 TopK
- 融合：将命中记忆作为“证据块”注入 prompt
- 输出：要求模型引用证据（记忆 id/时间/类型），降低幻觉

### 5.2 进阶优化（建议至少做到 2 项）
- **混合检索**：向量 + BM25（提高召回，减少“问得具体却没命中”）
- **Query 改写/重试**：首次召回弱/为空时自动改写 query 再检索一次（典型 Agentic RAG 思路）
- **Rerank**：对 TopK 做精排（提升相关性，代价是延迟/成本）
- **冲突处理**：同类记忆冲突时按时间/置信度/用户确认优先级决策

### 5.3 离线评测（你必须拿得出数字）
准备一个小评测集（建议 100 条以内也行）：
- **检索层指标**（至少 1 个）：Recall@K / HitRate@K / MRR / NDCG
- **生成层指标**（至少 1 个）：Faithfulness（证据一致性）、Correctness、Relevance

最低要求：能展示“优化前 vs 优化后”的对比（表格 + 结论），并能回归检测退化。

## 6. 工具调用（Function Calling）与护栏（面经必问）

### 6.1 工具系统最小集合
- `time`：获取当前时间
- `memo`：增删查改备忘录
- `weather`：天气查询

### 6.2 必须有的护栏（否则会被追问到崩）
- **最大迭代步数**：最多 N 次工具调用/反思循环
- **超时与重试**：超时、指数退避、失败超过阈值熔断
- **参数强校验**：Pydantic 校验，不通过就拒绝执行并要求模型重给参数
- **权限边界**：工具只允许读写允许的资源（不要“任意文件/任意网络”）
- **成本控制**：token 预算、上下文压缩阈值、工具优先级（低价值工具不调用）

------

## 7. 情绪识别与对话状态机（“陪伴”差异化点）

### 7.1 情绪识别（从轻到重都能讲）
- 轻量：规则/词典 + LLM 分类（输出 `label + intensity + evidence`）
- 进阶：将情绪写入状态与记忆（例如连续 3 轮低落触发“倾听模式”）
- 业务化：引入“情绪风险分层”（一般/关注/高风险），高风险触发安全模式与人工求助提示（不做诊断，不装专业）

### 7.2 状态机（必须可解释、可复现）
- 状态：闲聊 / 倾听 / 安慰 / 工作 / 工具
- 触发条件：意图、情绪、工具需求、用户显式指令
- 日志中记录：从哪个状态到哪个状态、依据是什么（why）

验收：给定同一段对话，状态机输出应稳定（不要每次跑都变）。

## 8. 后端服务化与“像业务”的工程化（JD 高频）

### 8.1 API 设计（建议）
- `POST /chat`：`user_id, session_id, message` → `reply + trace_id + (可选)debug`
- `GET /trace/{trace_id}`：查看一次对话链路（检索/工具/耗时/token）
- `POST /memory/write`：写入长期记忆（内部/管理员接口）
- `POST /eval/run`：跑离线评测集并产出报告（JSON/Markdown）
- `POST /sessions/reset`：清空 STM/LTM（合规与可控）
- `GET /profile/{user_id}`：返回用户画像卡片（偏好/边界/主题聚类/近期情绪摘要）
- `POST /feedback`：用户对某条回复点赞/点踩/纠错（进入闭环迭代与离线集）

### 8.2 可观测性（非常加分）
- 结构化日志字段建议：`trace_id, latency_ms, token_in/out, rag_hit_ids, tool_calls, errors`
- 关键耗时拆分：embedding、检索、rerank、LLM、工具调用
- 业务指标：日活会话数、平均轮次、情绪改善趋势（仅统计、不过度承诺）、高风险触发次数、工具调用成功率

### 8.3 并发与稳定性（轻量实现也要有策略）
- 会话状态外置（可选 Redis），服务尽量无状态
- 缓存：embedding 缓存、常用检索结果缓存（`user_id + query_hash`）
- 限流：按 user_id/IP 限制 QPS（避免成本爆炸）
- 降级策略：RAG 失败→纯对话；工具失败→解释+替代方案；LLM 超时→短回复+稍后重试提示

### 8.4 业务功能（让项目“像产品”）
- **用户体系**：最小化也要有 `user_id/session_id`，支持多用户隔离；可选登录（JWT/Session）。
- **配额与成本**：按用户/按天 token 配额与限流；超额提示与降级（只用 STM、降低 TopK、关闭 rerank）。
- **内容安全**：敏感词/暴力/自伤风险检测与输出约束；记录触发原因（可审计）。
- **运营配置台（轻量版）**：可在 UI 调整关键参数（写入阈值、TopK、重试次数、策略模板），并立即生效。
- **反馈闭环**：点踩/纠错进入 `feedback`，自动沉淀为评测样本（离线集）+ 线上规则（如黑名单短语/偏好更新）。
- **A/B 与回归**：同一评测集对比“baseline vs 改进版”（如混合检索/改写/精排），输出报告；线上可按用户分桶。

## 10. 版本路线（不绑定时间：按“可验收版本”迭代）

> 原则：每个版本都要“可演示 + 可观测 + 可回归”，否则只是堆功能。

### V0（能跑通，不丢链路）
- **内容**：多轮对话（STM 滑窗）+ FastAPI `POST /chat` + trace_id + 基础日志
- **验收**：连续 20+ 轮对话稳定；每轮都有 trace_id；能在日志里定位一次请求的耗时与 token
- **演示**：页面只有聊天也行，但必须能打开“trace 回放”

### V1（开始像陪伴：情绪与策略）
- **内容**：情绪识别（label/intensity/evidence）+ 状态机（闲聊/倾听/安慰/工作/工具）+ 共情策略库（可配置）
- **验收**：同一段对话状态切换稳定；trace 里能看到“为什么进入某状态”
- **演示**：UI 展示“当前情绪 + 当前模式 + 切换原因”

### V2（开始像个性化：LTM + 最小 RAG）
- **内容**：长期记忆模型（Preference/Profile/Event/Constraint）+ 写入策略（置信度/来源）+ 向量召回 + 证据引用输出
- **验收**：跨会话命中偏好/边界；回答能引用记忆 id；支持用户编辑/删除记忆
- **演示**：UI 有“记忆面板”（可查看/搜索/编辑/删除）+ “证据引用”

### V3（开始像工程：工具系统 + 护栏）
- **内容**：time/memo/weather 工具 + 参数强校验 + 最大步数 + 超时重试熔断 + 降级策略
- **验收**：工具失败不死循环；错误可解释；日志可审计（工具名/参数/耗时/结果/错误）
- **演示**：UI 展示“工具调用卡片”（输入参数、返回值、失败原因）

### V4（开始像落地：RAG 深度优化 + 离线评测）
- **内容**：chunk 策略升级 +（混合检索/改写重试/rerank 至少选 2）+ benchmark + 指标计算 + 回归
- **验收**：能产出“优化前后对比表”；能定位退化 case 并解释原因
- **演示**：UI 有“评测页”（一键跑评测 + 展示指标/典型案例）

### V5（开始像产品：业务功能与运营闭环）
- **内容**：用户隔离 + 配额/限流 + 内容安全 + 反馈闭环 + 运营配置（阈值/模板/策略可调）
- **验收**：不同用户记忆严格隔离；超额/高风险有明确行为；反馈能沉淀到离线集
- **演示**：UI 有“运营台”（参数开关、配额、反馈列表、风险事件列表）

## 11. 面试讲解提纲（你照这个讲就像“做过线上系统”）

7. 工程化：日志/trace、缓存、限流、并发与成本控制
8. 安全：隐私与自伤风险处理策略

### 12.1 界面展示效果（建议做成“你一眼就像在看 SaaS”）
- **Chat 主界面**：
  - 左侧会话列表（按日期/主题），支持搜索
  - 中间对话区（消息气泡 + 引用证据块 + 工具调用卡片）

- **记忆管理页（Memory Studio）**：
  - 画像卡片（偏好/边界/主题聚类）

- **情绪仪表盘（Emotion Dashboard）**：
  - 日/周/月情绪曲线与分布（只做统计，不做诊断）

- **评测页（Eval Lab）**：
  - 选择基线/实验配置，跑 benchmark
  - 展示 Recall@K/MRR/NDCG/Faithfulness 等 + 典型 case 对比

### 12.2 演示脚本（让面试官 2 分钟看懂“不是玩具”）
- **场景 1：边界一致性**：用户说“别鸡汤、叫我X”，第二天再来仍然遵守
- **场景 2：长期记忆命中**：提到“上次面试/DDL/重要事件”，系统检索并引用证据
- **场景 3：情绪回访**：连续低落→进入倾听模式→一段时间后主动回访
- **场景 4：工具与护栏**：天气工具超时→重试→熔断→降级说明；无死循环
- **场景 5：评测与优化**：展示“优化前后指标提升 + 退化 case 分析”

## 13. 简历写法（Java 后端转 Agent/大模型应用：可直接粘贴再改）

> 建议你投递时的岗位方向写法：**大模型应用开发 / AI 工程（后端+AI）/ Agent 应用开发**。  
> 你不需要先学 SpringAI：它是 Java 生态的“框架封装层”。面试更看重你是否掌握 **RAG 链路、Agent 护栏、评测与可观测、成本与稳定性**。这些能力与语言无关，你做出来后再迁移到 Java/Spring 只是工程量问题。

### 13.1 项目名称（一句话）
**个性化情感陪伴智能体（Agentic RAG）**：基于 LLM + 双层记忆 + 可控工具调用，构建可观测、可评测、可运营的陪伴型对话服务。

### 13.2 简历项目描述（3~6 条，建议按投递方向取舍）
- 设计并实现 **STM/LTM 双层记忆**：STM 采用滑窗+摘要压缩控制 token；LTM 按 `Preference/Profile/Event/Constraint` 建模，支持纠错/更新/删除，降低长期记忆污染风险。
- 落地 **Agentic RAG**：实现切分与索引、向量召回，并引入（混合检索 / Query 改写重试 / rerank）中至少两项优化；回答强制引用证据（记忆 id/类型/时间），降低幻觉。
- 自研 **工具调用与护栏**：工具注册/路由/执行全链路可审计；Pydantic 强校验参数；支持超时、指数退避、熔断、最大步数与降级策略，避免死循环与成本失控。
- 构建 **可观测性与回放**：为每轮对话生成 `trace_id`，记录检索命中、工具调用、耗时、token 成本与错误原因；支持 Trace 回放页用于定位 badcase。
- 搭建 **离线评测与回归**：构造小规模 benchmark，计算 Recall@K/MRR/NDCG/Faithfulness 等指标，输出“优化前后对比 + 典型退化 case 分析”，形成可持续迭代闭环。
- 产品化增强：实现用户隔离、配额/限流、内容安全、安全模式（自伤风险提示）、反馈闭环（点赞/点踩/纠错）与轻量运营配置台，提升可上线性。

### 13.3 技术栈关键词（对齐 JD，写在简历“技术栈/关键词”里）

### 13.4 量化指标模板（没有数字=像玩具；有数字=像业务）
> 下面括号里的数字你后续用实际跑出来的结果替换。
- RAG 检索：Recall@5 从（A）提升到（B）；MRR 从（C）提升到（D）。
- 生成质量：Faithfulness 从（E）提升到（F）；badcase 率下降（G%）。
- 性能成本：P95 延迟（X ms）；平均 token/轮（Y）；工具调用成功率（Z%）。
- 稳定性：工具失败熔断阈值（N 次）后降级，避免死循环；线上高风险触发（K 次/周）均进入安全模式并可审计。
