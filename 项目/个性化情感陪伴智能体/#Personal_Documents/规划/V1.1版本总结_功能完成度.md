# V1.1 版本总结：已完成功能 · 未完成项 · 占位说明 · 完成度

> **文档目的**：以 **V1.1 发布视角**概括「到当前仓库为止」项目做到了什么、还缺什么、哪些是工程占位，并给出一句话 **完成度** 判断，便于答辩、简历与排期。  
> **说明**：V1.1 在规划上主要是 **隐式 LTM + 相关工程补强**；整体产品还包含一期～二期已交付的大量能力，下文分层表述，避免把「整个项目」与「仅 V1.1 增量」混为一谈。

---

## 1. 版本与范围怎么理解

| 名称 | 含义（本文用法） |
|------|------------------|
| **一期（V1～V5）** | 对话、情绪与策略、RAG 最小闭环、工具、LTM API、安全、配额、Trace、反馈等后端能力基线。 |
| **二期** | Vite/React 控制台：聊天、记忆、Trace、情绪、Eval、运营/合规等与真实 API 联调闭环（部署不在二期范围）。 |
| **V1.1** | 在上一行基线上，增加 **可选「对话 → 隐式 LTM」**（开关、抽取、去重、撤销、异步队列、可选远程 embedding）及若干小改（见《V1.1开发文档.md》）。 |
| **三期** | 界面标杆、pgvector/持久向量、Compose、Eval/Trace 多实例、流式主路径强化、前端测试与 E2E 等（见《三期迭代规划.md》）。 |

## 2. 到当前为止：已完成的主要功能

### 2.1 后端与主链路（一期～V5 + 持续修补）

- **多轮对话**：`POST /chat`、`POST /chat/stream`（SSE），主回复走 **LangChain LCEL**（`app/langchain/`）。  
- **短期记忆 STM**：进程内或 **Redis**（`REDIS_URL`）；会话列表与消息读取。  
- **长期记忆 LTM**：显式 `POST /memory/ltm` 等；可选 **MySQL/SQLAlchemy**（`DATABASE_URL`）。  
- **情绪与策略**：`analyze_emotion`、`policy_config`、安全扫描与输出脱敏、模式切换。  
- **RAG（可选）**：混合检索（稀疏向量 + BM25）、query 改写、阈值与注入预算；**默认进程内索引**，启动 warm。  
- **工具（可选）**：`TOOL_ENABLED` 时时间/天气等（LangChain `StructuredTool`）。  
- **配额与运营**：日预算、QPS、热配置 `PATCH /admin/config`、探针 `GET /health`（含 redis/database/ltm_extract 等字段）。  
- **Trace**：落盘文件存储、与 decision 字段联动。  
- **评测**：Eval Lab 异步 job、builtin 样本等（任务状态 **进程内**，见下节「占位」）。  
- **工程**：`pytest` 主链与合规等回归（当前规模约 **100+** 用例量级，以本机 `pytest` 为准）。

### 2.2 二期 Web（产品壳）

- 聊天、Memory Studio、Trace、情绪仪表盘、Eval、运营/合规等 **与后端真实联调** 的主路径（细节以《二期迭代规划》与代码为准）。  
- 流式对话消费、错误与重试、深色模式等二期要求中的大部分已落地；**前端 Vitest/E2E** 规划为后续（三期或未接入）。

### 2.3 V1.1 专项（相对 V1～V5 的增量）

- **`LTM_EXTRACT_ENABLED`**：关闭时与「仅显式记忆」时代行为一致；开启后按轮次 **独立 LLM 调用** 抽取结构化条目写入 LTM（`source=dialogue_extract`）。  
- **去重/合并**：`ltm_dedup` + 环境变量阈值；**撤销** `POST /memory/ltm/undo_extract`（针对本轮新建 id）。  
- **Trace / 响应字段**：`ltm_extract_*`、`TraceDecision` 对齐；异步 pending 与 Web 侧刷新/Toast（见《常用命令》§8）。  
- **异步抽取**：`LTM_EXTRACT_ASYNC` + Redis 队列 **`companion:ltm_extract:queue`** + `lifespan` worker。  
- **可选远程 embedding**：`RAG_EMBEDDING_*`，失败回退稀疏向量。  
- **韧性**：`DATABASE_URL` 异常时 LTM 回退内存、Redis STM 僵尸 session 清理、`/health` 探活不阻塞事件循环等（见《整体检查后迭代规划》与代码）。

## 3. 未完成或仅部分完成（相对「终局 / 三期」）

下列 **不** 视为 V1.1 必须交付，但属于「项目整体仍可进步」的空间：

| 类别 | 内容 |
|------|------|
| **部署与编排** | Docker Compose 一键环境、生产 CI/CD、HTTPS/Nginx 等（二期/三期文档边界）。 |
| **向量与 RAG 终局** | **pgvector / Milvus** 等持久向量库；通用文档切块 pipeline、cross-encoder rerank（三期）。 |
| **多实例一致性** | Redis STM 并发 append 强一致、配额跨进程、Trace/Eval 共享存储（三期）。 |
| **数据层** | 反馈/情绪等大规模仍以 **JSONL/文件** 为主；Eval **job 内存字典** 重启丢进度（三期）。 |
| **数据库迁移** | 主要依赖 `create_all`，**无 Alembic** 版本链（三期/工程债）。 |
| **前端测试** | **Vitest/RTL、Playwright E2E** 未作为硬门禁接入（二期规划已写明）。 |
| **OpenAPI → TS** | README 示例命令有，**未强制脚本化/CI**（V1.2 规划项）。 |
| **成本策略** | 情绪 LLM 与配额顺序优化等（《整体检查后迭代规划》P2-2，三期或按需）。 |
| **界面标杆** | 全路由骨架屏、虚拟滚动、移动端专项等（三期 §1）。 |

## 4. 占位 / 简化实现（诚实描述，避免简历过度承诺）

| 项 | 当前形态 | 面试/简历建议说法 |
|----|----------|-------------------|
| **向量检索** | **进程内** `InMemoryVectorIndex` + 可选远程 embedding 映射到稀疏维度 | 「单机/演示规模够用；向量未进独立向量库」 |
| **Embedding 默认** | **字符级稀疏** 为主 | 「默认可零依赖；可接 OpenAI 兼容 embeddings」 |
| **RAG 单元** | 以 **整条 LTM** 为检索单元 | 「陪伴记忆场景，非通用知识库切块」 |
| **Eval 任务** | **内存 dict** 存 job | 「异步跑批可用；重启丢任务，待持久化」 |
| **Trace** | **本地文件** | 「单机可观测；多实例需共享卷或换存储」 |
| **配额** | 逻辑在应用内；多实例 **不** 自动共享同一计数 | 「单实例演示；多副本需 Redis 化配额」 |
| **主路径流式** | 已有 `/chat/stream`；三期还可强化「停止生成、统一指标」等 | 按实际代码说「SSE 已接」即可 |

## 5. 完成度怎么看（「完成长度」）

这里用 **三层口径**，避免一个百分比糊弄：

| 口径 | 判断 | 说明 |
|------|------|------|
| **V1.1 规划内交付** | **高（约 90%～100%）** | P0～P2 在代码与 `test_v11_*` 中均有对应；默认关开关不影响旧行为。剩余主要是文档/个人手测习惯差异，而非缺模块。 |
| **二期产品合同（Web + 后端闭环）** | **高** | 主功能可在 UI 走完；明确未做的主要是 **前端自动化测试、部分体验标杆**。 |
| **《整体规划》终局 + 三期目标** | **中偏低（约 35%～50%）** | 持久向量库、Compose、多实例、E2E、评测体系报表化等多在三期或未启动。 |

**一句话**：**V1.1 该做的开发在仓库里已闭环；整个项目已是「可日常自用的本地产品 + 较完整后端」；距离「可对外宣称的生产级多实例中台」还差三期那一类工程。**

## 6. 自动化测试能否代表「逻辑正确」？

- **能代表**：在 **mock LLM** 前提下，主链编排、契约、大量边界与 V1.1 抽取相关用例与当前代码一致。  
- **不能代表**：真实供应商延迟/限流、真实 Redis/MySQL 故障组合、浏览器端到端、未写用例的开关组合。

**建议**：对外说「**pytest 全绿 + 主路径手测**」比单独说「测试过了」更严谨。
