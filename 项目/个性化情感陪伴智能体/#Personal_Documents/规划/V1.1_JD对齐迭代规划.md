# V1.1 迭代规划（JD 对齐学习路径驱动 · 收口与可讲）

> **依据**：《JD对齐学习路径_项目驱动.md》§2～§4（对照学、补学清单、阶段 B/C）。  
> **与 V1.0 文件夹关系**：`V1.0/V1.1迭代规划.md` 描述的是**产品向**「隐式 LTM + Trace + 异步 + 远程 embedding」等能力；**当前仓库已大体实现**（见 `app/memory/ltm_extract*.py`、`GET /health` 的 `ltm_extract_enabled`、`app/langchain/`、`README.md`）。本文不重复抄写产品 PRD，而是回答：**在 JD 视角下，V1.1 还能做什么、如何验收、与 V1.2/三期如何切分。**

---

## 1. 定位：V1.1 在版本线里是什么

| 维度 | 说明 |
|------|------|
| **产品** | 原 V1.1 的 P0～P2（开关、抽取、去重、异步队列、可选 embedding、配额与 Trace）→ **以代码为准做验收清单**，缺什么补什么，不另起大盘。 |
| **JD / 面试** | 对齐《JD 对齐学习路径》**阶段 A～B 的「能讲 + 小改证明」**：数据流、RAG 设计笔记、只读诊断或契约级单测；**不把 Docker/pgvector/压测**塞进 V1.1（归入 **V1.2** 或 **三期**）。 |
| **边界** | V1.1 = **记忆叙事闭环 + 可演示 + 可测试 + 文档自洽**；V1.2 = **Compose、OpenAPI→TS 脚本化、评测/压测/CI 等工程与 JD 关键词补强**。 |

## 2. 当前基线（相对 JD 对照表）

结合仓库现状与《JD 对齐学习路径》§2，**已强对齐**项（面试可直接引用模块名）：

- **Python 异步 / FastAPI**：`routes.py`、`chat_turn` 全链路。  
- **Redis / MySQL**：`redis_stm.py`、`ltm_sql.py`、`.env.example`。  
- **RAG**：`app/rag/*` + `app/langchain/rag_lcel.py`（混合检索 + LCEL）。  
- **LLM**：主链 LangChain（`main_chain.py`）；情绪与隐式抽取 **httpx** `LLMClient`（与文档「分工」叙述一致）。  
- **Agent/工具轻量**：`app/langchain/tools_lc.py`、`decide_mode`。  
- **可观测**：`app/trace/*`、LangChain `LlmProviderIdCallback`。  
- **配额 / 热配**：`QUOTA_*`、`hot_config`、策略 JSON。  
- **测试**：`tests/test_langchain_*.py`、`test_v11_ltm_extract_async.py` 等。

**JD §3 中明确留给后续版本的弱项**（**不要**在 V1.1 强行吃完）：

- Docker Compose、locust、structlog、OpenAPI→TS 进脚本、pgvector → **V1.2 / 三期**，与 `V1.0/V1.2迭代规划.md`、`三期迭代规划.md` 一致。

## 3. V1.1 可做事项（按优先级）

### P0 — 验收与「能讲」（无大功能，1～3 天）

| ID | 项 | 产出 / 验收 |
|----|-----|-------------|
| **V11-JD-P0-1** | **隐式 LTM 端到端手测** | 按《常用命令》§8：`LTM_EXTRACT_ENABLED=true`、可选 `LTM_EXTRACT_ASYNC` + `REDIS_URL`；确认 Trace 有 `ltm_extract`、Memory Studio 可筛 `dialogue_extract`、撤销路径可用。 |
| **V11-JD-P0-2** | **一张数据流图 + 口述稿** | 用户一句话 → 情绪/安全 → RAG/工具 → STM → 配额 → 主链 LLM → 落盘 / 抽取 / Trace；**3 个追问**（主链为何 LangChain、向量存哪、多实例怎么办）与 JD §4 阶段 D 一致。 |
| **V11-JD-P0-3** | **pytest 门禁** | 本地 `python -m pytest -q` 全绿；CI 可后移到 V1.2，但 V1.1 收口至少**文档写明**前置条件（根 `README` 已有，保持与《常用命令》一致）。 |

### P1 — 阶段 B「作业级」工程（可选代码，3～7 天）

| ID | 项 | 说明 |
|----|-----|------|
| **V11-JD-P1-1** | **RAG 调试笔记** | JD 要求：读 `retriever.py`、`query_rewrite.py`、`warm_index.py`，打印**改写前后 query + topK id**（可用临时日志或仅开发开关），半页纸写清设计 trade-off。 |
| **V11-JD-P1-2** | **只读诊断增强（小）** | 在**不膨胀 OpenAPI 面**的前提下：例如 `GET /health` 或现有 admin 诊断增加 **RAG 是否启用、索引是否已 warm、embedding 模式（sparse / remote）** 等只读字段之一 + **一条 pytest**（对齐《整体检查后迭代规划》P1 精神）。 |
| **V11-JD-P1-3** | **LangChain 进阶（选做）** | JD §3：Callback 与 `LlmProviderIdCallback` 对照笔记；或 **独立 20 行 Retriever demo**（`tests/` 或 `scripts/`），用于面试「不是只会调包」。 |

### P2 — 文档与矩阵同步（持续）

| ID | 项 | 说明 |
|----|-----|------|
| **V11-JD-P2-1** | 更新 `V1.0/V1.1迭代规划.md` §4 **待办矩阵**：已交付项打 ✅，避免与代码漂移。 |
| **V11-JD-P2-2** | 在《JD 对齐学习路径》§5 索引中可**增加本文链接**，形成「学习路径 ↔ 版本规划」闭环。 |

### 明确不纳入 V1.1（避免与 V1.2 抢范围）

- `docker-compose`、OpenAPI→TS `package.json` 脚本、locust、Eval 报表化、pgvector、E2E。  
- STM 多实例强一致、Trace/Eval 持久化、情绪与配额顺序大改 → **三期** 或《整体检查后迭代规划》P2。
