# 面试题：如何设计一个电商系统的秒杀系统？

------

## 一、背景与问题描述

- **秒杀系统**：在电商大促（如双11、618、秒杀专场）中，商品在极短时间内以超低价出售，常常瞬间涌入大量用户请求。
- **问题挑战**：
  1. **高并发**：同一时刻可能有几十万甚至百万用户同时请求。
  2. **超卖问题**：库存有限，但高并发下可能被多次扣减，导致超卖。
  3. **缓存穿透/击穿/雪崩**：热点商品缓存瞬间被打爆。
  4. **公平性**：如何保证下单公平，不被黄牛或刷单。
  5. **系统限流与降级**：避免请求洪峰冲垮数据库。

------

## 二、核心设计思路

### 1. 架构分层

```
用户请求 → Nginx 限流 → 秒杀网关（接口防刷） 
→ Redis 预减库存 + 请求排队 
→ MQ 异步下单 
→ MySQL 落库
```

### 2. 技术要点

1. **前端层**：静态化页面、CDN 缓存、JS 倒计时，减少后端压力。

2. **限流防刷**：

   - 接口级限流（令牌桶/漏桶算法）。
   - 验证码/滑块校验。
   - 用户资格校验（如提前发放秒杀资格 token）。

3. **库存预减**：

   - 请求进入 Redis，库存 `decr`。
   - 若库存 < 0，则直接失败（避免打到数据库）。

4. **请求削峰**：

   - 将请求写入 MQ（如 RabbitMQ/RocketMQ/Redis Stream）。
   - 后台异步消费者串行下单，避免瞬时写 DB。

5. **数据库层**：

   - 下单逻辑加 **唯一索引（user_id+voucher_id）** 防止重复下单。

   - 扣减库存时加 **乐观锁（version 字段）** 或 SQL 条件更新：

     ```sql
     update t_stock set stock = stock - 1 
     where item_id = ? and stock > 0;
     ```

6. **防超卖 & 保证一致性**：

   - Redis 原子操作 (`decr`) 保证单机正确。
   - 分布式下需 Redisson 锁或 Lua 脚本。

7. **防黄牛**：

   - 接口限频（同一用户/设备 1s 内只能请求一次）。
   - 黑名单机制。
   - 秒杀资格随机化（抽签/令牌桶）。

------

## 三、案例：Redis + MQ + MySQL 实现秒杀

1. **用户请求**：
   - 校验用户是否登录 & 是否有秒杀资格。
   - Redis `decr` 库存：库存不足立即返回失败。
2. **异步下单**：
   - 将下单消息放入 MQ。
   - 消费者从 MQ 消费消息，写入数据库。
3. **数据库写入**：
   - 利用唯一索引防止同一用户重复下单。
   - 利用乐观锁防止库存超卖。

- **游戏道具/门票秒杀**
- **限时优惠券抢购**

## 五、面试模板回答

> **回答示例**：
>  在秒杀场景下，系统需要解决高并发、超卖和防刷等问题。我会设计一个 **前端静态化 + 限流防刷 + Redis 预减库存 + MQ 削峰 + 数据库唯一约束** 的架构。
>  具体做法是：用户请求先经过 Nginx 和网关限流，Redis 中预减库存，库存不足直接返回，库存充足则将请求放入 MQ，后台异步消费者负责真正下单并写数据库，数据库层通过唯一索引和乐观锁避免超卖。这样能保证系统的高可用与一致性。

## 六、扩展追问

1. 如果 Redis 挂了怎么办？
   - 本地缓存 + 降级兜底，或者快速失败。
2. MQ 堆积怎么办？
   - 消费者多线程消费，消息分区。
3. 如何保证公平性？
   - 秒杀资格 token，或者排队等待。
4. 如何防止库存回滚失败？
   - 事务 + 补偿机制（定时任务扫描异常订单）。
5. 和普通下单系统相比，秒杀系统的差异是什么？
   - 秒杀系统以“削峰限流、快速失败”为核心目标。
