# 场景题描述（可直接念）

> 做一个“联系方式校验”能力，支持 **手机号、邮箱、座机、国际电话、可选的企业邮箱规则**。
>  需要在 **前端实时校验** 并在 **后端强校验**；校验通过后再入库/发送验证码。要求考虑 **国际化、异常与安全**（长度限制、ReDoS、Unicode 混淆等）。

------

# 需求拆解与规则

**1）基本规则（示例）：**

- 中国大陆手机号：`^1[3-9]\\d{9}$`
- 邮箱（简化版）：`^[A-Za-z0-9._%+-]+@[A-Za-z0-9.-]+\\.[A-Za-z]{2,}$`
- 国际电话（E.164 形态）：`^\\+?[1-9]\\d{1,14}$`
- 座机（大陆常见）：`^0\\d{2,3}-?\\d{7,8}$`

**2）规范化（Normalization）：**

- 去除空格、全角字符转半角、统一大小写（邮箱域名不区分大小写）
- 手机去掉分隔符 `-`、空格
- 国际电话尽量转成 **E.164**（带国码）

**3）前后端分层：**

- 前端：即时提示（正则 + 交互 UX）
- 后端：最终可信校验（JSR-380 + 自定义校验器），避免绕过

**4）验证 vs 校验：**

- **校验（validation）**：只检查格式
- **验证（verification）**：发验证码/邮件点链接才算真实可用

# 后端实现（Spring Boot / Java）

## A. DTO + Bean Validation（快速上手）

```java
import jakarta.validation.constraints.*;
import lombok.Data;

@Data
public class ContactDTO {
    @Pattern(regexp = "^1[3-9]\\d{9}$", message = "非法的中国大陆手机号")
    private String phoneCN;

    @Pattern(regexp = "^\\+?[1-9]\\d{1,14}$", message = "非法的国际手机号")
    private String phoneIntl;

    @Email(message = "非法邮箱格式")
    @Size(max = 254, message = "邮箱长度过长")
    private String email;

    @Pattern(regexp = "^0\\d{2,3}-?\\d{7,8}$", message = "非法的座机号码")
    private String landline;
}
```

## B. 自定义注解（支持多国家/策略切换）

```java
// 1) 注解
@Target({ElementType.FIELD})
@Retention(RetentionPolicy.RUNTIME)
@Constraint(validatedBy = PhoneValidator.class)
public @interface Phone {
    String message() default "非法手机号";
    Class<?>[] groups() default {};
    Class<? extends Payload>[] payload() default {};
    // 指定类型：CN / INTL
    String type() default "CN";
}

// 2) 校验器
public class PhoneValidator implements ConstraintValidator<Phone, String> {
    private String type;
    private static final Pattern CN = Pattern.compile("^1[3-9]\\d{9}$");
    private static final Pattern INTL = Pattern.compile("^\\+?[1-9]\\d{1,14}$");

    @Override
    public void initialize(Phone anno) { this.type = anno.type(); }

    @Override
    public boolean isValid(String value, ConstraintValidatorContext ctx) {
        if (value == null || value.isBlank()) return true; // 交给 @NotBlank 控制必填
        String v = normalize(value);
        return "INTL".equalsIgnoreCase(type) ? INTL.matcher(v).matches()
                                             : CN.matcher(v).matches();
    }

    private String normalize(String s) {
        return s.trim().replaceAll("\\s|-", "");
    }
}
```

# 前端（简要思路）

- 输入时节流校验（300ms），红/绿状态提示
- 对手机号：自动去空格与 `-`，国家区号下拉
- 对邮箱：blur 后提示；域名白名单（企业场景）
- 若要“验证”，则走验证码通道 + 限流

# 单元测试与边界用例

**手机号（CN）：**

- ✅ `13800138000`、`19912345678`
- ❌ 长度不对、前缀不对、含空格/破折号（规范化后再测）

**邮箱：**

- ✅ `alice.bob+test@sub.example.com`
- ❌ `a@b`（后缀太短）、含中文全角、连续两个点 `a..b@x.com`

**国际电话：**

- ✅ `+8613800138000`、`+1 650 555 1234`（规范化后）
- ❌ 前导 0、长度>15

**座机：**

- ✅ `010-88886666`、`02988886666`
- ❌ 区号/位数不匹配

# 安全与鲁棒性（面试加分点）

- **正则 ReDoS 防御**：避免回溯爆炸的复杂写法；固定上限长度，例如 `@Size(max=30)`。
- **输入长度与字符集限制**：先做长度/字符集白名单，再用正则。
- **Unicode 混淆**：全角/零宽字符清洗；邮箱域名转小写。
- **限流与风控**：验证码发送频率限制，防止轰炸；同 IP/同账号阈值。
- **国际化**：推荐使用业界库（如 libphonenumber）做精细规则与格式化（面试说到库名即可）。

# 标准面试回答（≈1–2分钟模板）

> 我会把联系方式校验拆成 **格式校验** 和 **实际验证** 两层。
>  **后端**用 **JSR-380 Bean Validation** 做基础校验，手机号（大陆/国际）、邮箱、座机分别给出正则，并通过**自定义注解 + ConstraintValidator**支持不同国家/策略；同时做**规范化**（去空格、去`-`、大小写统一）与**长度限制**，避免 ReDoS。
>  **前端**做实时提示，但后端是最终可信；若需要“可用性验证”，通过验证码/邮件链接并**限流**。国际化方面，手机号统一向 **E.164** 规范靠拢，必要时引入专业库提升准确率。
>  测试覆盖正常/异常/边界场景；安全上防输入超长、Unicode 混淆、验证码轰炸等。

# 扩展追问（准备要点）

1. **为什么不用一个超级邮箱正则？** —— 复杂易错、ReDoS 风险，业务用简化版+验证链路更可靠。
2. **如何做企业邮箱白名单？** —— 解析域名并白名单匹配，注意多级域与大小写。
3. **如何处理存量数据清洗？** —— 批量规范化到 E.164/小写域名，建立唯一索引前先去重。
4. **如何设计验证码安全？** —— 限流、图形/行为验证码、滑动窗口、IP+账号联合策略、灰度熔断。
