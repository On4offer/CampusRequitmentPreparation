# 场景题：验证码设计

## 题目描述

> 系统需要实现一个 **短信/邮箱验证码功能**，要求：
>
> 1. 验证码必须具备 **时效性**（过期后无效）；
> 2. 必须具备 **唯一性**（同一手机号/邮箱短时间内只能存在一个有效验证码）；
> 3. 防止用户或攻击者 **恶意刷接口**（频繁请求验证码或暴力破解）。

请你设计系统方案，包括 **存储、校验、限流、异常场景**。

------

## 参考答案结构

### 1. **时效性**

- 验证码存储在 Redis，设置过期时间（例如 5 分钟）。
- Key 设计：`verify:code:{phone}` → Value: 随机 6 位数。
- Redis `EXPIRE` 保证过期自动删除。

### 2. **唯一性**

- 同一手机号请求验证码时，旧验证码自动覆盖。
- 或者 **加锁**：5 分钟内只能有一个有效验证码。
- Redis `SETNX`/`SETEX` 实现：若已存在且未过期，拒绝重复发送。

### 3. **防刷机制**

- **接口防刷**：
  - Redis 计数器实现限流：
    - Key: `verify:count:{ip}`，TTL=1分钟，超过阈值拒绝。
  - 针对手机号/账号维度也做限流（比如 1小时最多5次）。
- **验证码防爆破**：
  - 校验时失败次数计数：超过 3 次即失效。
  - Key: `verify:fail:{phone}`。

### 4. **安全加固**

- 验证码不应明文存数据库，只放 Redis。
- 可加 **雪花 ID/Token** 映射：前端提交验证码时附带 Token，服务端验证对应关系，避免枚举手机号。

## 代码伪例

```java
// 发送验证码
String key = "verify:code:" + phone;
Boolean exists = stringRedisTemplate.hasKey(key);
if (Boolean.TRUE.equals(exists)) {
    throw new BizException("验证码未过期，请稍后再试");
}

// 限流：一分钟内最多1次
String limitKey = "verify:limit:" + phone;
Long count = stringRedisTemplate.opsForValue().increment(limitKey);
if (count == 1) {
    stringRedisTemplate.expire(limitKey, Duration.ofMinutes(1));
} else if (count > 1) {
    throw new BizException("请求过于频繁");
}

// 生成验证码
String code = RandomUtil.randomNumbers(6);
stringRedisTemplate.opsForValue().set(key, code, Duration.ofMinutes(5));
// 调用短信网关发送
```

## 面试追问点

1. **如果 Redis 挂了怎么办？**
   - 答：短信验证码场景允许降级，短时失败可直接报错，或 fallback 到 DB（带过期字段）。
2. **如何防止验证码被撞库？**
   - 答：增加 Token、图形验证码、人机验证；错误次数阈值。
3. **如何兼顾并发与幂等？**
   - 答：生成验证码用 Redis `SETNX` + 过期时间，保证同手机号幂等。

------

✅ 总结一句面试用话术：

> 验证码系统的设计要保证三点：
>
> - **时效性**：Redis 设置 TTL；
> - **唯一性**：同手机号覆盖或 SETNX；
> - **防刷**：接口限流 + 校验失败次数限制。
>    同时注意安全加固，如 Token 映射、人机验证，避免暴力破解。
