**Next-Key Lock（临键锁）** 是 InnoDB 存储引擎中独有的一种复合锁，**由[记录锁](记录锁)（Record Lock）+ [间隙锁](间隙锁)（Gap Lock）组合而成**，用于在可重复读（`REPEATABLE READ`）隔离级别下**防止幻读**。

------

### 一、Next-Key Lock 是什么？

- 它不是单独的一种锁，而是：

  > **“锁住某条记录本身 + 它之前的间隙”**

- 作用：**防止其他事务修改当前记录**，也**防止在该记录前插入新记录**。

------

### 二、锁定范围示意

假设有如下索引列数据：`[10, 20, 30]`
 若加 Next-Key Lock 于 `id = 20`，它锁住的实际范围是：

```
(10, 20]   ← 锁住的是这个区间（包含 20，不包含 10）
```

这样就能防止：

- 修改或删除 `20`（记录锁的作用）
- 插入 `id = 15` 等位于 `(10, 20)` 的记录（间隙锁的作用）

------

### 三、什么时候触发 Next-Key Lock？

- 在 **`REPEATABLE READ`** 隔离级别下；

- 执行如下语句时：

  ```sql
  SELECT ... FOR UPDATE;
  SELECT ... LOCK IN SHARE MODE;
  UPDATE ... WHERE ...;
  DELETE ... WHERE ...;
  ```

- **查询条件使用范围（<、>、BETWEEN）且命中了索引**。

------

### 四、为什么需要它？

主要目的是：**防止幻读**

#### 幻读例子：

```sql
-- 事务 A
START TRANSACTION;
SELECT * FROM users WHERE age BETWEEN 20 AND 30 FOR UPDATE;

-- 事务 B
INSERT INTO users (id, name, age) VALUES (5, 'Tom', 25);  -- 会被阻塞
```

→ 因为事务 A 对 `(20, 30]` 加了 Next-Key Lock，**B 不能在这段区间插入新记录**，避免了“幻行”出现。

------

### 五、特殊情况：间隙锁优化（Next-Key Lock 退化）

如果你使用的是 **等值查询** 且命中了唯一索引，InnoDB 会自动优化为只加**记录锁**，不加间隙锁。

------

### 六、与其它锁对比

| 锁类型          | 锁定范围             | 用途                 |
| --------------- | -------------------- | -------------------- |
| 记录锁          | 精确锁某一行         | 防止改写该行         |
| 间隙锁          | 锁定两个记录间的空隙 | 防止插入“幻行”       |
| **Next-Key 锁** | 间隙 + 行本身        | 同时防插入和更新该行 |

------

### 七、小结一句话：

> **Next-Key Lock 是 InnoDB 为防止幻读而设计的锁机制，锁住记录本身及其前间隙，避免其他事务插入或更改范围内的数据。**
