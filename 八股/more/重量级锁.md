### **重量级锁（Heavyweight Lock）简介**

**背景**：**重量级锁**是 Java 虚拟机（JVM）中 `synchronized` 机制在并发竞争激烈时自动升级后的最终形态。它是一种基于操作系统**互斥量（Mutex）\**实现的线程同步方式。当多个线程\**同时竞争一个锁对象**，且偏向锁、轻量级锁都无法满足同步需求时，锁会膨胀为**重量级锁**，涉及线程阻塞和唤醒，性能开销较大。

------

### **一、重量级锁的触发条件**

- 初始状态：无锁
- 如果某个对象被一个线程获取锁，则可能偏向该线程（**偏向锁**）
- 如果有另一个线程参与竞争，偏向锁撤销，升级为**轻量级锁**
- 如果多个线程**同时**竞争轻量级锁，CAS 失败，JVM 会将其升级为**重量级锁**

------

### **二、重量级锁的实现机制**

当锁升级为重量级锁时，JVM 会：

1. 使用 **操作系统的 Monitor（互斥量）机制**；
2. 将无法获取锁的线程挂起（阻塞）；
3. 等待锁释放后再唤醒阻塞线程；
4. 阻塞/唤醒线程需要用户态和内核态切换，**代价高、性能差**。

------

### **三、重量级锁的影响**

| 特性                 | 描述                                     |
| -------------------- | ---------------------------------------- |
| **线程阻塞**         | 获取锁失败的线程将被阻塞                 |
| **上下文切换开销大** | 阻塞/唤醒线程需操作系统介入              |
| **性能最差**         | 在所有锁类型中性能最低，但功能最完整     |
| **可重入性**         | 支持可重入，即同一线程可以多次获得同一锁 |

------

### **四、重量级锁的锁标志（对象头 Mark Word）**

在对象头中，锁标志位的值表示当前锁的状态：

| 状态     | 标志位         | 含义                       |
| -------- | -------------- | -------------------------- |
| 无锁     | 01             | 没有加锁                   |
| 偏向锁   | 01（特定标志） | 偏向线程 ID                |
| 轻量级锁 | 00             | 指向 Lock Record           |
| 重量级锁 | 10             | 指向 Monitor（互斥量）结构 |

------

### **五、重量级锁的适用场景**

尽管性能差，但在以下情况下是必要的：

- 多线程**同时访问同一资源**（高并发竞争激烈）；
- 执行**时间较长**的同步块；
- 保证**线程阻塞**后数据的准确性和安全性；
- 在 JVM 无法通过轻量级机制解决竞争问题时。

------

### **六、优化建议**

为了避免频繁进入重量级锁状态，应：

1. **减少锁的粒度**：只在必须同步的代码区域加锁；
2. **减少锁竞争**：使用局部变量或 ThreadLocal 替代共享变量；
3. **避免长时间持有锁**：快速释放锁，缩短锁定时间；
4. **使用并发类库**：如 `ReentrantLock`、`ConcurrentHashMap` 等提供更高性能的同步工具；
5. **启用 JVM 锁优化**：如偏向锁、轻量级锁（默认开启）。

------

### **七、锁状态演化流程（总结图）**

```text
无锁
 ↓
偏向锁（无竞争）
 ↓（有线程参与）
轻量级锁（CAS）
 ↓（多线程同时竞争）
重量级锁（阻塞 + 唤醒）
```

------

### **八、小结一句话**

> **重量级锁是 JVM 在高并发竞争下自动膨胀出的同步机制，基于操作系统互斥量实现，性能较差，但保证线程安全，是偏向锁和轻量级锁无法胜任时的最后手段。**
