### **读写锁（ReadWriteLock）简介**

**背景**：在多线程并发场景中，如果一个共享资源**读操作远多于写操作**，使用普通互斥锁（如 `synchronized` 或 `ReentrantLock`）会限制读操作的并发性。为了解决这个性能瓶颈，Java 提供了 **读写锁机制**，使得**多个线程可以同时读数据，但写操作必须独占**。

------

### **一、什么是读写锁？**

**读写锁（ReadWriteLock）** 是一种并发控制策略，允许：

- **多个线程同时读取共享资源**；
- **写操作必须互斥执行**，即同一时刻只能有一个写线程，且写时不能读。

在 Java 中，读写锁由接口 **`java.util.concurrent.locks.ReadWriteLock`** 定义，常用实现类是：

> **`ReentrantReadWriteLock`**

------

### **二、读写锁的核心原则**

| 操作          | 是否可并发 |
| ------------- | ---------- |
| 多个读-读线程 | 是         |
| 读-写线程     | 否         |
| 写-写线程     | 否         |

> 即：**读读共享，读写互斥，写写互斥**

------

### **三、使用示例：`ReentrantReadWriteLock`**

```java
import java.util.concurrent.locks.ReentrantReadWriteLock;

public class Cache {
    private final ReentrantReadWriteLock rwLock = new ReentrantReadWriteLock();
    private int data;

    // 读操作：可以并发
    public int read() {
        rwLock.readLock().lock();  // 加读锁
        try {
            return data;
        } finally {
            rwLock.readLock().unlock();  // 解锁
        }
    }

    // 写操作：排它
    public void write(int value) {
        rwLock.writeLock().lock();  // 加写锁
        try {
            data = value;
        } finally {
            rwLock.writeLock().unlock();  // 解锁
        }
    }
}
```

------

### **四、读写锁的适用场景**

- 缓存读写（如 LRU 缓存）；
- 配置中心（多数读取，偶尔修改）；
- 数据采集系统（后台写入数据，前台频繁读取）；
- 日志或监控系统（写入较少、查询较多）。

------

### **五、读写锁的优点与缺点**

#### **优点：**

- 提高并发性能（读读不阻塞）；
- 减少不必要的锁竞争；
- 提高 CPU 利用率，特别适合“读多写少”的业务场景。

#### **缺点：**

- 不适用于“写多读少”的场景；
- 复杂度更高，可能导致**写线程饥饿（写永远排队）**；
- 需要开发者细心管理，防止死锁或不一致。

------

### **六、与 ReentrantLock 的区别**

| 对比项           | `ReentrantLock`        | `ReentrantReadWriteLock` |
| ---------------- | ---------------------- | ------------------------ |
| 是否支持读写分离 | 否（读写互斥）         | 是（读读可并发）         |
| 性能             | 适用于高写并发或通用锁 | 适用于读多写少场景       |
| 锁的种类         | 单一互斥锁             | 读锁 + 写锁              |
| 可重入性         | 是                     | 是                       |

------

### **七、小结一句话**

> **读写锁是一种在读多写少场景下提升并发性能的锁机制，允许多个线程同时读取，写操作时则完全互斥，常用于缓存、配置中心等高并发读取场景。**
