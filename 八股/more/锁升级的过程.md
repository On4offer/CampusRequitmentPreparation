### **锁升级的过程（JVM 中对象锁的升级机制）**

**背景**：在 Java 中，`synchronized` 是一种重量级锁的代名词，但为提升并发性能，JVM 实现了**锁的优化机制**，支持锁的**升级（或称“膨胀”）\**过程，根据线程竞争程度动态调整锁的类型，以\**平衡性能与安全**。

------

### **一、锁的四种状态**

对象在运行时可以处于以下四种锁状态（按开销从小到大排列）：

1. **无锁（No Lock）**
2. **偏向锁（Biased Lock）**
3. **轻量级锁（Lightweight Lock）**
4. **重量级锁（Heavyweight Lock）**

------

### **二、锁升级的完整流程**

#### **1. 无锁 → 偏向锁**

- 默认情况下，对象是无锁状态。
- 当**一个线程第一次访问该对象的同步代码块**，JVM 会给该对象加上**偏向锁**，并将线程 ID 记录到对象头的 Mark Word 中。

#### **2. 偏向锁 → 轻量级锁**

- 如果**另一个线程也尝试加锁**这个对象，JVM 认为偏向失效。
- JVM 会撤销偏向锁，将其升级为**轻量级锁**，进入 CAS 竞争阶段。

#### **3. 轻量级锁 → 重量级锁**

- 如果**有多个线程同时竞争轻量级锁**（CAS 自旋失败），JVM 会将锁**膨胀为重量级锁**。
- 后续线程再请求这个锁时，会被**阻塞挂起**，直到锁被释放。

------

### **三、流程图示意**

```text
无锁
 ↓
偏向锁（线程独占，快速）
 ↓（检测到其他线程竞争）
轻量级锁（CAS + 自旋）
 ↓（自旋失败）
重量级锁（阻塞 + 唤醒）
```

------

### **四、锁升级特性**

- **锁升级是单向的**，只会从低到高（无锁 → 偏向 → 轻量级 → 重量级），**不会降级**。
- JVM 自动管理，无需开发者干预；
- 升级时机由 JVM 内部策略决定，比如偏向锁被撤销、自旋失败次数等。

------

### **五、锁升级的实用意义**

- **偏向锁、轻量级锁**避免了线程阻塞，提高性能；
- **重量级锁**保证安全性，在高竞争环境下保持正确性；
- 通过锁升级，JVM 在**性能与线程安全之间找到平衡点**。

------

### **六、小结一句话**

> **锁升级是 JVM 根据锁竞争强度自动执行的优化过程，先尝试使用偏向锁、轻量级锁避免阻塞，只有竞争激烈时才升级为重量级锁，以提升并发性能并确保线程安全。**
