**MVCC（多版本并发控制，Multiversion Concurrency Control）** 是 InnoDB 引擎中实现**高并发读写性能和事务隔离**的核心机制之一。它允许**事务在不加锁的情况下读取一致性数据快照**，避免读写冲突，从而大大提高数据库的并发能力。

------

### 一、MVCC 的核心目标

- 实现 **[非阻塞的读操作](非阻塞的读操作)**（快照读），提升性能
- 保证读到的数据是事务一致的（符合隔离级别）
- 避免读锁，减少死锁概率

------

### 二、MVCC 如何工作？

MVCC 通过**为每行记录维护多个版本**来实现。InnoDB 每条记录**隐式维护两个隐藏字段**：

| 字段                     | 作用                                 |
| ------------------------ | ------------------------------------ |
| **创建版本（trx_id）**   | 标记该行是哪个事务插入的             |
| **删除版本（roll_ptr）** | 指向该行旧版本的回滚信息（undo log） |

此外，每个事务也有一个**当前版本号（事务ID）**，用来决定它能看到哪些数据版本。

------

### 三、MVCC 的两种读操作

#### 1. **快照读（Snapshot Read）** — 不加锁

- 普通 `SELECT` 查询使用；
- 读取的是某个事务启动时刻的**数据快照**；
- 支持多版本，依赖 undo log；
- **MVCC 起作用的场景**。

示例：

```sql
SELECT * FROM orders WHERE id = 1;
```

#### 2. **当前读（Current Read）** — 加锁

- 显式加锁或修改操作；
- 会读取最新版本，**并加行锁**；
- MVCC **不适用**，常用于更新或排它读取。

包括语句：

```sql
SELECT ... FOR UPDATE;
SELECT ... LOCK IN SHARE MODE;
UPDATE / DELETE
```

------

### 四、不同隔离级别下 MVCC 表现

| 隔离级别             | MVCC 行为                                    |
| -------------------- | -------------------------------------------- |
| **READ COMMITTED**   | 每次查询都读取当前已提交数据快照（最新版本） |
| **REPEATABLE READ**  | 多次查询都读取事务开始时的快照（默认级别）   |
| **SERIALIZABLE**     | 强制加锁，**不使用 MVCC**，性能最低          |
| **READ UNCOMMITTED** | 读取未提交数据，MVCC不生效，可能脏读         |

------

### 五、MVCC 优势与局限

#### 优势：

- 提高读性能，无需加锁；
- 降低死锁概率；
- 保证读一致性。

#### 局限：

- 增加 undo log 空间和管理开销；
- 大事务长时间存在会导致版本链变长，影响性能；
- 写操作之间仍然存在锁竞争。

------

### 六、总结一句话：

> **MVCC 让读取数据既“看到旧版本”又“避免加锁”，实现高并发下的事务一致性，是 InnoDB 性能和事务隔离的核心机制。**
