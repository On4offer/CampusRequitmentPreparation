## ✅ 二、什么是 ACID？

ACID 是数据库事务应具备的四大特性，是确保数据**可靠性与一致性**的基础。

| 字母 | 含义        | 中文含义 |
| ---- | ----------- | -------- |
| A    | Atomicity   | 原子性   |
| C    | Consistency | 一致性   |
| I    | Isolation   | 隔离性   |
| D    | Durability  | 持久性   |

------

## ✅ 三、ACID 各部分详解

### 🔹 A：原子性（Atomicity）

事务中所有操作**要么全部执行，要么全部不执行**，中途失败会被回滚。

📌 例如：银行转账，要么扣钱+加钱都成功，要么全部失败。

数据库实现方式：

- 利用日志（如 redo/undo log）实现事务回滚

------

### 🔹 C：一致性（Consistency）

事务执行前后，数据库必须从一个**一致状态变为另一个一致状态**。

📌 例如：转账总金额不变，不能“凭空多出”或“少了”钱。

数据库实现方式：

- 通过约束（外键、唯一性）、触发器、事务控制保证一致性。

------

### 🔹 I：隔离性（Isolation）

多个事务并发执行时，彼此**互不干扰**，避免脏读、不可重复读、幻读等问题。

数据库支持的隔离级别：

| 级别             | 说明                       |
| ---------------- | -------------------------- |
| Read Uncommitted | 可读未提交（可能脏读）     |
| Read Committed   | 读已提交（防止脏读）       |
| Repeatable Read  | 可重复读（防止不可重复读） |
| Serializable     | 可串行化（最严格）         |

数据库通过 **锁机制（行锁、表锁）或 MVCC** 实现隔离。

------

### 🔹 D：持久性（Durability）

事务一旦提交，其结果将**永久写入数据库**，即使系统宕机也不会丢失。

数据库实现方式：

- 提交事务时写入 redo log
- 崩溃恢复机制从日志中重建已提交事务状态

------

## ✅ 四、实战案例举例

### 🧩 银行转账操作（A + C）

```sql
BEGIN;
UPDATE account SET balance = balance - 1000 WHERE id = 1;
UPDATE account SET balance = balance + 1000 WHERE id = 2;
COMMIT;
```

- 如果在第二条 SQL 执行前宕机，事务未提交，数据库自动回滚，保持一致性。

------

### 🧩 并发写入订单（I）

两个用户同时提交订单，使用隔离级别 `REPEATABLE READ` 可避免读取同一个库存并同时扣减，防止超卖。

## ✅ 五、ACID 与数据库引擎的关系

| 数据库引擎      | 是否支持 ACID           | 特性说明               |
| --------------- | ----------------------- | ---------------------- |
| InnoDB（MySQL） | ✅ 支持全部              | 支持事务、行级锁、MVCC |
| MyISAM（MySQL） | ❌ 不支持事务            | 不支持 ACID            |
| PostgreSQL      | ✅ 全部支持              | 强一致事务支持         |
| MongoDB         | ✅（4.0+）支持多文档事务 | 之前只支持单文档事务   |
