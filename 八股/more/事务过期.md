在数据库或分布式系统中，**“事务的过期与淘汰策略”**并不是标准术语，但你的问题非常好 —— 它触及的是事务在长时间未提交、资源竞争激烈、或超时场景下，**系统如何“清理”或“终止”事务**的策略。

下面从**MySQL、分布式事务（如 Seata）、缓存事务场景（如 Redis）、以及事务管理器层**几个角度，详细说明事务的“过期/淘汰”策略或行为：

------

## ✅ 一、MySQL 层的事务超时/淘汰机制

### 1. **InnoDB 中长事务的淘汰影响**

- InnoDB 会保留 undo log 和行锁，直到事务提交/回滚；
- 长时间不提交的事务会：
  - 阻塞 purge 线程（无法清理历史版本）；
  - 导致其他事务死锁或等待；
  - 占用资源，降低性能。

虽然 InnoDB **不会自动淘汰事务**，但可以通过配置超时机制来“驱动回滚”：

### 2. **显式配置事务超时**

```sql
SET innodb_lock_wait_timeout = 50;  -- 单个语句锁等待超时，默认 50 秒
SET innodb_rollback_on_timeout = ON;  -- 超时后自动回滚整个事务
```

### 3. **会话超时**

```sql
SET wait_timeout = 28800;  -- 连接空闲超时
```

⏱️ 如果客户端长时间连接不活跃，MySQL 会断开连接，未提交事务将自动回滚。

------

## ✅ 二、Spring / JTA 事务管理器层的过期策略

Spring 提供了事务超时控制：

```java
@Transactional(timeout = 10) // 秒
```

- 超过 10 秒事务未提交，抛出异常；
- 事务管理器会强制回滚；
- 常用于防止**人为写死循环**或外部接口超时导致事务卡死。

------

## ✅ 三、分布式事务中事务超时（如 Seata、RocketMQ、TCC）

在分布式事务中，**全局事务的超时控制非常关键**，否则资源会长时间锁定：

### 1. **Seata 全局事务过期控制**

```yaml
client.tm.default-global-transaction-timeout=60000 # 60 秒
```

- 超时后，全局事务管理器（TC）会发起回滚；
- 各参与者收到通知，释放资源。

### 2. **RocketMQ 事务消息**

- 半消息未提交超时，则会触发事务回查；
- 如果 Producer 不响应，就回滚消息。

------

## ✅ 四、缓存/数据库混合事务策略（如 Redis）

Redis 本身没有传统事务，但你可能在 Lua 脚本或 WATCH/MULTI 机制中模拟事务：

- Redis 没有事务超时；
- 但可以通过脚本执行时间限制防止长期占用。

------

## ✅ 五、其他淘汰策略间接关联

| 场景           | 描述                                                         |
| -------------- | ------------------------------------------------------------ |
| **慢事务清理** | DBA 人工 kill 长事务（通过 `SHOW ENGINE INNODB STATUS` 查看） |
| **连接池驱逐** | 长时间占用连接（如 Druid、Hikari）会被踢出                   |
| **死锁回滚**   | MySQL 检测到死锁会自动中断其中一个事务                       |

------

## ✅ 总结：事务过期/淘汰的策略来源

| 层级           | 策略/配置                            | 效果                   |
| -------------- | ------------------------------------ | ---------------------- |
| MySQL          | `innodb_lock_wait_timeout`、连接超时 | 防止死锁、阻塞         |
| Spring         | `@Transactional(timeout = X)`        | 主动控制事务生命周期   |
| 分布式事务框架 | 全局事务 TTL、自动回滚               | 释放资源，防止悬挂事务 |
| 运维层（手动） | kill session、慢 SQL 清理            | 人工清理不规范事务     |

------

## 💬 面试标准答题模板

> 虽然事务没有标准的“过期淘汰”术语，但各层都提供了事务生命周期的控制机制。例如 MySQL 提供了 `innodb_lock_wait_timeout` 来避免长事务阻塞，Spring 支持通过 `@Transactional(timeout)` 限定事务执行时长，分布式事务如 Seata 也有全局事务超时设置。一旦超时，事务将被强制回滚，保障系统资源及时释放，防止数据不一致或性能下降。

------

## ✅ 一句话总结

> 事务“过期淘汰”= 各系统通过超时配置 + 自动回滚 + 连接驱逐等方式，防止事务无限期占用资源。
