你这个问题问得非常好：**除了 Redis，还能把锁状态保存在哪？**

锁状态的本质是：**某个资源是否被占用、谁占用的、多久占用**。只要能记录这些信息，理论上很多系统都可以用来做分布式锁。

------

## 一、除了 Redis，还可以保存在哪里？

### 1. **Zookeeper**

- 利用临时有序节点来表示锁状态
- **优点**：天然支持**强一致性 + 自动释放**
- **适用**：主备切换、任务调度、配置变更等一致性敏感场景

------

### 2. **数据库（如 MySQL / PostgreSQL）**

- 使用一张表专门维护锁记录，比如：

```sql
CREATE TABLE distributed_lock (
  lock_key VARCHAR(64) PRIMARY KEY,
  lock_owner VARCHAR(64),
  expires_at DATETIME
);
```

- 加锁时使用：

```sql
INSERT INTO distributed_lock VALUES ('myLock', 'service1', NOW() + INTERVAL 10 SECOND);
```

- **优点**：易于实现、无需引入新组件
- **缺点**：性能差、并发低、释放失败可能导致死锁

------

### 3. **Etcd（键值对数据库）**

- 和 Zookeeper 类似，支持租约机制
- **适用**：Kubernetes 集群管理、配置中心等

------

### 4. **Consul**

- 由 HashiCorp 提供的服务发现 + KV 存储系统
- 支持 Session + TTL 实现锁
- **适用**：服务发现、配置同步、基础设施管理

------

### 5. **本地内存 / 缓存系统**

- 使用如 `ConcurrentHashMap`、Guava Cache 等管理锁状态
- 只能用于**单机锁**，无法跨服务生效

------

## 二、这些方案之间的对比

| 存储方案  | 是否分布式 | 是否强一致性 | 可靠性 | 性能 | 使用复杂度 | 是否常用        |
| --------- | ---------- | ------------ | ------ | ---- | ---------- | --------------- |
| Redis     | 是         | 否（AP）     | 中     | 高   | 低         | 非常常用        |
| Zookeeper | 是         | 是（CP）     | 高     | 中   | 中高       | 常用            |
| MySQL     | 是         | 是           | 中     | 低   | 低         | 可用            |
| Etcd      | 是         | 是           | 高     | 中   | 中         | Kubernetes 常用 |
| Consul    | 是         | 是           | 中     | 中   | 中         | 较少            |
| 内存      | 否（单机） | 是           | 低     | 高   | 非常低     | 本地锁用        |

------

## 三、总结一句话：

> 分布式锁的状态除了 Redis，还可以保存在 Zookeeper、MySQL、Etcd、Consul 等具备“共享 +一致性 + 过期控制”的系统中，具体选谁要看你更看重**性能、可靠性**还是**一致性**。
