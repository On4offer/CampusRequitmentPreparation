这个问题非常好，**HashMap 和 Hashtable 的扩容机制不完全一样**，虽然它们底层都是基于数组 + 链表/桶 的哈希结构，但在**触发条件、扩容方式、再哈希逻辑、线程安全处理**等方面存在明显差异。

下面我将从底层源码角度详细讲解 HashMap 和 Hashtable 的扩容机制差异，并提供面试回答模板。

------

## ✅ 一、先说结论

| 比较点           | HashMap                                                    | Hashtable                        |
| ---------------- | ---------------------------------------------------------- | -------------------------------- |
| **扩容触发条件** | size ≥ capacity × loadFactor（默认 0.75）                  | size ≥ capacity（无 loadFactor） |
| **默认初始容量** | 16                                                         | 11                               |
| **每次扩容倍数** | *2 倍扩容*                                                 | *2 倍 + 1*                       |
| **是否线程安全** | ❌ 否，扩容时需注意并发问题（JDK 7 有死链风险）             | ✅ 所有方法加锁，线程安全         |
| **扩容方式**     | 重新分配数组，**链表可能重排**（Java 8 起需重新计算 hash） | 重建新数组并复制旧 Entry         |

------

## ✅ 二、HashMap 的扩容机制（重点）

### ⚙️ 触发扩容条件：

```java
if (size >= threshold) // threshold = capacity * loadFactor
    resize();
```

### ⚙️ resize() 扩容步骤（Java 8）：

1. 新数组大小 = 旧数组大小 × 2
2. 创建新数组
3. 遍历旧数组中每个桶，将链表/红黑树重新计算 hash 后放入新桶（**重新映射**）
4. 替换旧数组引用

> ⚠️ Java 7 中扩容顺序插入，容易形成**死循环链表**；Java 8 改为头插+新链表/红黑树处理，解决了这个 bug

------

## ✅ 三、Hashtable 的扩容机制

### ⚙️ 触发扩容条件：

```java
if (count >= threshold) // threshold = initial capacity
    rehash();
```

### ⚙️ rehash() 过程：

1. 创建新数组，大小 = 旧数组大小 × 2 + 1（比如 11 → 23）
2. 所有旧 Entry 重新计算 hash 并放入新数组中
3. 无并发问题，因整个方法加了 `synchronized`（低并发性能）

------

## ✅ 四、线程安全带来的差异

| 扩容期间是否允许并发？ | HashMap                                     | Hashtable                      |
| ---------------------- | ------------------------------------------- | ------------------------------ |
| ❌ 不安全（需要避免）   | ✔ 可能导致数据丢失、死循环（特别是 Java 7） | ✔ 完全串行化，线程安全但性能差 |

------

## ✅ 五、标准面试回答模板

> 以下是你在面试中可以使用的一段结构化、准确的回答：

------

### 🎯 回答模板：

**“HashMap 和 Hashtable 的扩容机制在触发条件和扩容方式上都不一样。”**

**“HashMap 的扩容触发是基于容量 × 负载因子（默认 0.75），每次扩容为原容量的两倍；而 Hashtable 的扩容触发是满容量，扩容为原容量的 2 倍 + 1。”**

**“HashMap 是非线程安全的，因此在并发环境下扩容过程中可能出现链表死循环（Java 7）或数据覆盖；而 Hashtable 所有方法都加了 synchronized，因此扩容过程线程安全，但性能较低。”**

**“在实际开发中，HashMap 的扩容和 rehash 是性能热点之一，JDK 8 对其做了优化，比如链表树化（转为红黑树）来提升查询性能。”**

------

## ✅ 六、总结建议

- **单线程场景**：用 HashMap，注意负载因子与初始容量配置，避免频繁扩容
- **多线程场景**：不要用 Hashtable，推荐使用 **ConcurrentHashMap**，扩容机制更优化（分段/桶迁移）
