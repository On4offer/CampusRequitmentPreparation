行锁（Row Lock）是 MySQL 中 **粒度最小、并发性最高** 的锁类型，主要由 **InnoDB 存储引擎**实现。它用于锁定数据表中某一行或多行记录，防止多个事务同时修改相同的数据，从而保证数据的一致性和隔离性。

------

### 一、行锁的特点

- **粒度小**：只锁定需要操作的数据行，不影响其他行。
- **并发性高**：多个事务可同时操作不同的行。
- **自动加锁**：不需要用户显式指定，一般通过 `SELECT ... FOR UPDATE` 或 `UPDATE` 语句自动加锁。
- **仅 InnoDB 支持**：MyISAM 不支持行锁，只支持表锁。

------

### 二、行锁的类型（InnoDB）

1. **共享锁（S 锁，Shared Lock）**

   - 多个事务可以读取数据（共享访问），但不能修改。

   - 语法：

     ```sql
     SELECT * FROM table_name WHERE id = 1 LOCK IN SHARE MODE;
     ```

2. **排他锁（X 锁，Exclusive Lock）**

   - 一个事务独占某行数据，其他事务不能读也不能写。

   - 语法：

     ```sql
     SELECT * FROM table_name WHERE id = 1 FOR UPDATE;
     ```

------

### 三、行锁的工作原理

- InnoDB 通过 **索引** 来实现行锁，**必须走索引**才能锁定特定的行。
- 如果查询条件**没有用到索引**，InnoDB 会退化成 **表锁**。
- 使用主键或唯一索引能有效减少锁的范围。

------

### 四、行锁的典型语句

```sql
-- 加共享锁
SELECT * FROM users WHERE id = 5 LOCK IN SHARE MODE;

-- 加排他锁（用于更新）
SELECT * FROM users WHERE id = 5 FOR UPDATE;

-- 更新操作自动加排他锁
UPDATE users SET age = 30 WHERE id = 5;
```

------

### 五、注意事项

- 行锁容易引发**死锁**，应合理设计事务顺序。
- 大量小事务比一个大事务更安全。
- 行锁依赖索引，如果误用会降低并发性能。
