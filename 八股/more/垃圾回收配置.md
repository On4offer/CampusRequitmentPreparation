## 一、JVM垃圾回收基本原理

Java程序中的对象分配在**堆内存**，生命周期不同。GC的目标是**自动回收无用对象所占内存**，避免内存泄露和溢出。

内存分代思想：

- **新生代（Young Generation）**：大多数新对象，生命周期短，频繁GC。
- **老年代（Old Generation）**：长生命周期对象，GC较少。
- **元空间（MetaSpace）**：类元信息，JDK8后代替了永久代。

------

## 二、主流GC算法与回收器

| GC回收器                      | 特点                            | 适合场景                          |
| ----------------------------- | ------------------------------- | --------------------------------- |
| **Serial GC**                 | 单线程、Stop-The-World          | 小型应用，Client模式              |
| **Parallel GC（吞吐量优先）** | 多线程新生代GC，适合高CPU利用率 | 中大型批量任务                    |
| **CMS GC（响应时间优先）**    | 并发标记清除，减少停顿          | Web服务，需要低延迟（已被G1取代） |
| **G1 GC（均衡）**             | 分区管理，支持大堆，低停顿      | 大内存应用，推荐默认GC（JDK9+）   |
| **ZGC** / **Shenandoah GC**   | 极低延迟GC，支持TB级堆          | 高吞吐+超低停顿系统（如实时交易） |

## 三、常用垃圾回收配置参数

### 1. **选择GC类型**

```bash
-XX:+UseSerialGC        # 启用Serial GC
-XX:+UseParallelGC      # 启用Parallel GC（新生代）+ ParallelOldGC（老年代）
-XX:+UseConcMarkSweepGC # 启用CMS GC（JDK9废弃）
-XX:+UseG1GC            # 启用G1 GC（JDK9+默认）
-XX:+UseZGC             # 启用ZGC（JDK11+）
```

### 2. **GC行为调优**

| 参数                               | 作用                           | 示例                            |
| ---------------------------------- | ------------------------------ | ------------------------------- |
| `-XX:MaxGCPauseMillis`             | G1/ZGC 最大GC停顿时间目标      | `-XX:MaxGCPauseMillis=200`      |
| `-XX:GCTimeRatio`                  | 吞吐比目标（用于Parallel GC）  | `-XX:GCTimeRatio=9` 表示GC占10% |
| `-XX:+ExplicitGCInvokesConcurrent` | System.gc()触发并发GC          | 避免Full GC停顿                 |
| `-XX:+UseStringDeduplication`      | G1下去重字符串对象             | 减少老年代压力                  |
| `-XX:+AlwaysPreTouch`              | 启动时分配全部堆，提升预热速度 | 大堆推荐                        |

### 3. **GC日志相关**

JDK8:

```bash
-XX:+PrintGCDetails -XX:+PrintGCDateStamps -Xloggc:/logs/gc.log
```

JDK9+（统一日志）:

```bash
-Xlog:gc*:file=/logs/gc.log:time,uptime,level,tags
```

## 四、实际业务场景与调优示例

### **场景 1：电商秒杀系统（低延迟，高并发）**

**目标**：低GC停顿、稳定响应

```bash
-XX:+UseG1GC
-XX:MaxGCPauseMillis=100
-XX:+ParallelRefProcEnabled
-XX:+UnlockExperimentalVMOptions
-XX:+UseStringDeduplication
```

### **场景 2：后台大数据任务处理（高吞吐）**

**目标**：最大吞吐，容忍停顿

```bash
-XX:+UseParallelGC
-XX:+UseParallelOldGC
-XX:GCTimeRatio=9
```

### **场景 3：实时金融服务（极低延迟）**

**目标**：极低GC延迟，不影响交易

```bash
-XX:+UseZGC
-XX:ZUncommitDelay=300 # 空闲内存释放延迟
-XX:MaxHeapSize=16g
```

## 五、GC日志分析核心指标

使用 `gc.log` 或 `jstat`, `jmc`, `VisualVM`，关注以下内容：

| 指标           | 意义                               |
| -------------- | ---------------------------------- |
| GC次数频率     | 是否频繁触发？                     |
| GC停顿时间     | 是否影响响应？                     |
| 老年代增长速度 | 是否频繁Full GC？                  |
| Promotion失败  | 新生代对象无法晋升老年代，易OOM    |
| GC类型占比     | Minor GC / Full GC / Mixed GC 比例 |

------

## 六、小结与建议

- 小应用可用 `SerialGC`，中大型服务推荐 `G1GC`。
- 容器部署时要配合 `-XX:+UseContainerSupport`（JDK10+）。
- 调优GC时应以**业务SLA（响应时间）和吞吐**为导向。
- 持续监控GC日志、堆使用趋势、OOM风险，是稳定运行的关键。
