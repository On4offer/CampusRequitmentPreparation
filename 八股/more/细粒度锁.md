“**细粒度锁（Fine-Grained Lock）**”是一种**控制锁粒度更小、更精确**的并发控制策略，它与“**粗粒度锁（Coarse-Grained Lock）**”相对，是提升并发性能的重要手段之一。

------

### 一、细粒度锁的概念

细粒度锁的核心思想是：

> **将大范围的锁拆分成多个小的锁，多个线程可以并发地访问不同的数据或资源，从而提高并发度和程序效率。**

------

### 二、细粒度锁的示意

#### 假设一个共享数组资源：

- 粗粒度锁：对整个数组加锁，任何读写都要等待。
- 细粒度锁：对数组的每个元素、每个区块或每个桶加锁，不同线程可同时操作不同部分。

------

### 三、Java 中细粒度锁的例子

#### 1. `ConcurrentHashMap` 的锁设计

- JDK7：将整张表划分为多个“段”（Segment），每段独立加锁。
- JDK8：取消了 Segment，用 **桶级锁 + CAS** 实现更高效的细粒度同步。

#### 2. 数据库连接池中的资源控制

- 每个连接对象可能被单独加锁，避免全局等待。

#### 3. 锁住对象的某个字段或方法

```java
synchronized(objA) {
    // 只锁住 objA，不影响 objB
}
```

------

### 四、细粒度锁 vs 粗粒度锁

| 对比项     | 粗粒度锁           | 细粒度锁                   |
| ---------- | ------------------ | -------------------------- |
| 锁范围     | 大（比如整个容器） | 小（比如容器的某部分）     |
| 并发性能   | 低                 | 高                         |
| 实现复杂度 | 简单               | 高（要管理多个锁）         |
| 死锁风险   | 小                 | 大（可能多个线程循环等待） |

------

### 五、细粒度锁常见应用场景

| 场景                           | 说明                                |
| ------------------------------ | ----------------------------------- |
| 并发容器如 `ConcurrentHashMap` | 不同桶用不同锁，实现高并发插入/读取 |
| 多线程缓存系统                 | 每个缓存条目加锁，避免影响整体性能  |
| 分布式系统资源控制             | 每个节点或资源使用独立锁            |

------

### 六、总结

细粒度锁虽然性能高，但要注意：

- **避免死锁**（多个线程依赖多个锁时）
- **控制锁数量**（太多锁会增加维护成本）
- **适度使用 CAS 或无锁编程** 做性能补充
