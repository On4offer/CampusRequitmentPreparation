## ✅ 二、什么是 Redo Log？

**Redo Log（重做日志）** 是 **InnoDB 存储引擎** 提供的一种 **物理日志**，用于记录事务对数据页的修改操作，以确保**事务的持久性（Durability）**。

> 它是实现数据库 **崩溃恢复** 和 **WAL（Write-Ahead Logging）策略** 的关键组件。

------

## ✅ 三、为什么需要 Redo Log？

数据库崩溃后（如宕机、断电），内存中的数据（Buffer Pool）会丢失。

🧩 解决方案：

- 提前将数据变更记录到 Redo Log（磁盘）
- 系统恢复时读取 Redo Log，重新应用变更，恢复数据一致性

------

## ✅ 四、Redo Log 的工作机制（WAL 原理）

**WAL（Write-Ahead Logging）**：先写日志，再写数据。

流程如下：

```text
1. 客户端执行更新 SQL
2. InnoDB 修改 Buffer Pool 中的页
3. 同时将操作写入 Redo Log（先写磁盘，刷盘）
4. 事务提交（commit） → fsync 日志落盘
5. 后台线程异步将脏页刷入磁盘（Checkpoint）
```

🟩 如果崩溃，只需根据 Redo Log 中的操作，重做未刷入的数据页。

## ✅ 五、Redo Log 的两部分结构

| 组成部分                                         | 说明                 |
| ------------------------------------------------ | -------------------- |
| **Redo Log Buffer**                              | 位于内存，写入速度快 |
| **Redo Log File** (`ib_logfile0`, `ib_logfile1`) | 持久化到磁盘         |

------

## ✅ 六、Redo Log 的两种写入操作

| 类型        | 时机       | 用途                 |
| ----------- | ---------- | -------------------- |
| **Prepare** | 事务执行中 | 写入日志缓冲区       |
| **Commit**  | 事务提交时 | 刷盘，保证事务持久性 |

🔄 Redo Log 采用**循环写入（Ring Buffer）**，用 `write_pos` 和 `checkpoint` 控制回收。

------

## ✅ 七、Redo Log 与其他日志的区别

| 日志类型     | 作用                         | 作用范围                    |
| ------------ | ---------------------------- | --------------------------- |
| **Redo Log** | 实现事务持久性和恢复         | InnoDB 引擎私有             |
| **Undo Log** | 实现事务回滚、MVCC           | InnoDB 引擎私有             |
| **Binlog**   | 记录所有事务操作（逻辑日志） | MySQL Server 层，全引擎共享 |

> ✅ Redo Log 是物理级别，Binlog 是逻辑级别。

------

## ✅ 八、实战案例说明

### 🎯 场景：系统宕机恢复

1. 用户执行更新操作 → Redo Log 已写入并 fsync，但数据页尚未刷盘；
2. 此时数据库宕机 → 启动恢复流程；
3. InnoDB 扫描 Redo Log → 找到提交事务记录；
4. 将变更再次应用到数据页 → 数据恢复成功，满足持久性。
