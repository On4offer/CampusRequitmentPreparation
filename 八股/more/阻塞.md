**阻塞（Blocking）** 在数据库和并发编程中，指的是**一个操作因为资源被其他操作占用而必须等待**的现象。它是并发控制中常见的问题，尤其在涉及**锁、事务或 I/O 等共享资源**时更容易发生。

------

### 一、数据库中的阻塞

在 MySQL 等关系型数据库中，阻塞通常发生在事务之间，表现为：

- 一个事务持有锁（如行锁或表锁）；
- 另一个事务试图访问相同数据（读或写）；
- 第二个事务**不能继续执行，只能等待第一个事务释放锁**。

#### 举例：

```sql
-- 事务 A
BEGIN;
UPDATE accounts SET balance = balance - 100 WHERE id = 1;
-- 此时事务 A 未提交，持有 id = 1 的行锁

-- 事务 B
SELECT * FROM accounts WHERE id = 1 FOR UPDATE;
-- 被阻塞，等待事务 A 提交或回滚
```

------

### 二、阻塞的常见原因

| 原因         | 描述                                   |
| ------------ | -------------------------------------- |
| **锁冲突**   | 两个事务对同一行或表加不同类型的锁     |
| **锁等待**   | 一个事务已持有锁，另一个事务试图获取   |
| **死锁**     | 双方互相等待对方释放资源，永远无法继续 |
| **资源不足** | 如连接池满、磁盘 I/O 饱和等            |

------

### 三、阻塞与锁的关系

| 锁类型          | 是否可能阻塞其他操作 | 举例                                 |
| --------------- | -------------------- | ------------------------------------ |
| **共享锁（S）** | 阻塞写，不阻塞读     | 两个事务可共享读取数据               |
| **排他锁（X）** | 阻塞读和写           | 一个事务更新数据时，其他事务必须等待 |

------

### 四、阻塞与非阻塞的对比

| 特性             | 阻塞操作                | 非阻塞操作（MVCC快照读） |
| ---------------- | ----------------------- | ------------------------ |
| 是否加锁         | 是                      | 否                       |
| 是否等待         | 是                      | 否                       |
| 是否看到最新数据 | 是（当前读）            | 否（看到旧版本数据）     |
| 示例语句         | `SELECT ... FOR UPDATE` | `SELECT ...`（不加锁）   |

------

### 五、如何避免或减少阻塞

- 使用 **MVCC** 实现快照读，避免不必要的锁；
- 控制事务范围，**快速提交或回滚**；
- 优化索引，**精确锁定数据范围**；
- 调整隔离级别，减少锁范围；
- 检查慢查询与锁等待日志，定位热点行或慢事务；
- 使用 `NOWAIT` / `SKIP LOCKED` 选项（如在 PostgreSQL 或 MySQL 8.0+ 中）

------

### 六、小结一句话：

> **阻塞是事务或线程在等待锁或资源释放时被挂起的状态，是并发环境中导致性能下降和死锁的重要因素之一。**
