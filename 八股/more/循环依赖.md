### **Spring 中 Bean 的循环依赖**

**背景**：在使用 Spring 的依赖注入（DI）功能时，有时会出现多个 Bean **相互依赖**的情况，即：

> A 需要注入 B，B 也需要注入 A，这种情况就叫 **循环依赖（Circular Dependency）**。

Spring 框架对循环依赖做了专门处理，**可以解决大多数单例 Bean 的循环依赖问题**，但不是所有情况都能处理。

------

### **一、示例：构造器循环依赖（Spring 无法解决）**

```java
@Component
public class A {
    public A(B b) {}
}

@Component
public class B {
    public B(A a) {}
}
```

- 这是**构造器注入**引发的循环依赖。
- Spring 在创建 A 的时候需要先创建 B，创建 B 又需要 A。
- **结果：StackOverflowError（栈溢出）**
- **结论：构造器循环依赖无法解决！**

------

### **二、示例：字段注入循环依赖（Spring 可解决）**

```java
@Component
public class A {
    @Autowired
    private B b;
}

@Component
public class B {
    @Autowired
    private A a;
}
```

- 这是**属性（字段）注入**。
- Spring 会通过**三级缓存机制**（后面详细讲）提前暴露部分 Bean 引用，**延迟注入**，从而解决循环依赖。

------

### **三、Spring 如何解决循环依赖？（三级缓存）**

Spring 使用了三个缓存来处理 Bean 的创建流程：

| 缓存名                    | 作用                                                      |
| ------------------------- | --------------------------------------------------------- |
| **singletonObjects**      | 一级缓存：存放完全初始化好的单例 Bean                     |
| **earlySingletonObjects** | 二级缓存：存放提前暴露的半成品 Bean                       |
| **singletonFactories**    | 三级缓存：存放可以生成半成品 Bean 的工厂（ObjectFactory） |

#### 简化流程：

1. 创建 Bean A，先放入三级缓存；
2. 创建 Bean A 的依赖 Bean B；
3. B 依赖 A，尝试获取 A；
4. 从三级缓存获取 A 的 ObjectFactory，转为 A 半成品放入二级缓存；
5. 完成 B 的注入；
6. 回来完成 A 的初始化，至此循环依赖解决。

------

### **四、哪些循环依赖不能解决？**

| 场景                           | 能否解决                     |
| ------------------------------ | ---------------------------- |
| 构造器注入循环依赖             | 否                           |
| 原型（prototype）Bean 循环依赖 | 否（Spring 不缓存原型 Bean） |
| 字段/Setter 注入的单例         | 是                           |

------

### **五、如何避免或解决循环依赖？**

1. **使用 Setter 注入代替构造器注入**；
2. **重构设计，消除紧耦合**；
3. **拆分 Bean 的职责**；
4. 使用 `@Lazy` 注解让某个依赖懒加载，打破初始化顺序。

```java
@Component
public class A {
    @Autowired
    @Lazy
    private B b;
}
```

------

### **六、小结一句话**

> **Spring 使用三级缓存机制解决了大多数单例 Bean 的循环依赖问题，但对构造器注入和原型作用域无能为力，建议通过设计优化避免循环依赖。**
