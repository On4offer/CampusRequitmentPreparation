### **显式锁（Explicit Lock）简介**

**背景**：在并发编程中，**显式锁**（也称为 **显示锁**）是指由开发者显式地使用特定的锁对象来控制线程同步的机制。与 Java 的 **内置锁（synchronized）** 自动加锁不同，显式锁需要开发者手动创建和释放锁。

Java 中常见的显式锁有 **`ReentrantLock`** 和 **`ReadWriteLock`**，它们位于 `java.util.concurrent.locks` 包中。显式锁提供了更多的灵活性和控制，可以精细控制锁的获取、释放及其超时、可中断等特性。

------

### **一、`ReentrantLock`（可重入锁）**

`ReentrantLock` 是 Java 中最常用的显式锁。它是一个**可重入的互斥锁**，允许同一个线程重复获取同一个锁，不会导致死锁。

#### **特点：**

1. **可重入性**：同一个线程可以多次获取同一把锁。
2. **可中断**：可以在等待锁的过程中响应中断。
3. **公平锁与非公平锁**：`ReentrantLock` 支持公平锁（FIFO 排队）和非公平锁（JVM 默认使用非公平锁）。
4. **锁超时**：可以通过 `tryLock()` 方法指定锁的最大等待时间。

#### **使用示例：**

```java
import java.util.concurrent.locks.ReentrantLock;

public class ReentrantLockExample {
    private final ReentrantLock lock = new ReentrantLock();
    
    public void performTask() {
        lock.lock();  // 加锁
        try {
            // 执行同步代码
            System.out.println("Task is being performed.");
        } finally {
            lock.unlock();  // 确保在任务完成后释放锁
        }
    }

    public static void main(String[] args) {
        ReentrantLockExample example = new ReentrantLockExample();
        example.performTask();
    }
}
```

#### **优点：**

- 显式锁提供了 **比 `synchronized` 更细粒度的控制**。
- 支持 **可中断锁**，可以响应线程中断。
- 提供 **公平锁**，确保按请求顺序分配锁。

------

### **二、`ReadWriteLock`（读写锁）**

`ReadWriteLock` 是一种用于优化读操作的显式锁。它允许多个线程同时读取共享资源，但**只有一个线程可以进行写操作**，并且**写操作会独占资源**。

#### **特点：**

1. **读锁（`readLock()`）**：多个线程可以同时持有读锁，只要没有线程持有写锁。
2. **写锁（`writeLock()`）**：写锁是排他的，持有写锁的线程会阻塞其他所有读锁和写锁。
3. **优化读操作**：当读操作远多于写操作时，使用读写锁可以显著提高并发性能。

#### **使用示例：**

```java
import java.util.concurrent.locks.ReadWriteLock;
import java.util.concurrent.locks.ReentrantReadWriteLock;

public class ReadWriteLockExample {
    private final ReadWriteLock lock = new ReentrantReadWriteLock();
    
    public void read() {
        lock.readLock().lock();  // 获取读锁
        try {
            System.out.println("Reading...");
        } finally {
            lock.readLock().unlock();  // 释放读锁
        }
    }

    public void write() {
        lock.writeLock().lock();  // 获取写锁
        try {
            System.out.println("Writing...");
        } finally {
            lock.writeLock().unlock();  // 释放写锁
        }
    }

    public static void main(String[] args) {
        ReadWriteLockExample example = new ReadWriteLockExample();
        
        // 模拟读写操作
        example.read();
        example.write();
    }
}
```

#### **优点：**

- 在多读少写的情况下，**显著提高并发性能**。
- 读锁可以共享，写锁是独占的，**提高了读操作的并发性**。

#### **缺点：**

- 由于写锁是独占的，当写操作频繁时，**写操作会阻塞其他操作**。
- 相比普通锁，**实现复杂**，使用时需要考虑读写锁之间的协调。

------

### **三、显式锁与内置锁（synchronized）的对比**

| 特性               | `synchronized`（内置锁）   | `ReentrantLock`（显式锁）                              |
| ------------------ | -------------------------- | ------------------------------------------------------ |
| **加锁方式**       | 自动加锁，JVM 管理         | 显式调用 `lock()` 和 `unlock()`                        |
| **是否支持可中断** | 否                         | 支持 `tryLock()`、可中断的 `lock()`                    |
| **是否公平**       | 否                         | 支持公平锁，`ReentrantLock` 可以通过构造函数设置公平性 |
| **锁的粒度控制**   | 只能锁定方法或代码块       | 锁定任意对象，可以灵活控制锁粒度                       |
| **性能**           | 对于竞争较小的场景效果较好 | 对于复杂场景具有更好的灵活性和性能                     |
| **实现复杂度**     | 简单                       | 相对复杂，需要手动管理锁的获取与释放                   |

------

### **四、使用显式锁的注意事项**

1. **死锁问题**：显式锁需要手动管理锁的获取与释放，如果不同线程的锁获取顺序不一致，可能会导致死锁。因此，显式锁需要特别小心设计锁的顺序。
2. **必须保证锁的释放**：显式锁使用时必须在 **`finally` 块中**确保调用 `unlock()` 释放锁，避免锁没有被释放导致其他线程无法获取锁。
3. **性能问题**：显式锁虽然提供了更多的灵活性，但频繁地获取和释放锁会增加系统的开销。在某些高并发场景下，过度使用显式锁可能会影响性能。

------

### **五、小结一句话**

> **显式锁提供了比内置锁更细粒度的控制和灵活性，适用于高并发场景下对锁的精确管理，能够解决更多复杂的同步问题，但需要开发者手动管理锁的获取与释放。**
