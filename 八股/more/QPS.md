  ## ✅ 二、什么是 Redis 的 QPS？

  **QPS（Queries Per Second）**：每秒处理请求的数量，是衡量 Redis 性能的核心指标之一。

  > **Redis 的 QPS = 客户端请求总数 / 运行时间（秒）**

  ------

  ## ✅ 三、Redis QPS 的参考数值

  | Redis 部署方式          | 理想 QPS 上限（粗略估算） |
  | ----------------------- | ------------------------- |
  | 单机部署（普通硬件）    | 5 万 ~ 10 万 / 秒         |
  | 高性能 SSD / 多核 CPU   | 20 万 ~ 50 万 / 秒        |
  | Redis Cluster（分布式） | 可达到百万级 QPS          |

  > ⚠️ 实际 QPS 取决于操作类型、硬件配置、网络延迟、命令复杂度等因素。

  ------

  ## ✅ 四、影响 Redis QPS 的关键因素

  | 因素                         | 说明                                                       |
  | ---------------------------- | ---------------------------------------------------------- |
  | **命令复杂度**               | `GET/SET` 是 O(1)，性能高；如 `SORT`、`ZRANGE` 为 O(NlogN) |
  | **网络延迟**                 | Redis 是典型 IO 密集型应用，网络瓶颈会限制 QPS             |
  | **客户端连接数**             | 每个连接会占用 Redis I/O 线程资源                          |
  | **内存/CPU**                 | 单线程 Redis 对 CPU 敏感，内存命中率也影响吞吐             |
  | **是否使用管道（pipeline）** | pipeline 可以大幅减少网络交互，提升 QPS                    |
  | **AOF / RDB 持久化**         | 开启 AOF 会略微影响写入性能（尤其是 `appendfsync always`） |
  | **慢查询命令**               | 大 key、阻塞命令（如 `BLPOP`）拉低整体性能                 |

  ## ✅ 五、如何测量 Redis QPS

  ### 方式一：`INFO` 命令

  ```bash
  redis-cli info stats
  ```

  输出示例：

  ```txt
  instantaneous_ops_per_sec:75634
  ```

  > 表示当前 Redis 实例每秒处理大约 75,634 个命令

  ------

  ### 方式二：`redis-benchmark` 压测工具

  ```bash
  redis-benchmark -n 100000 -c 100 -q
  ```

  参数说明：

  - `-n`：请求总数
  - `-c`：并发连接数
  - `-q`：简洁输出

  结果示例：

  ```txt
  SET: 113636 qps
  GET: 125000 qps
  ```

  ## ✅ 六、Redis QPS 优化实战建议

  | 优化手段            | 说明                                     |
  | ------------------- | ---------------------------------------- |
  | 使用 Pipeline       | 批量写入可提升 5~10 倍 QPS               |
  | 拆分大 Key          | 将大 Hash / List 拆分为多个小结构        |
  | 合理设置 maxclients | 避免连接打满导致拒绝服务                 |
  | 降低持久化影响      | 设置 AOF 为 `everysec` 而非 `always`     |
  | 使用 Redis Cluster  | 水平扩展 Redis 实例，提升整体吞吐        |
  | 客户端限流          | 保证 Redis 服务不会因突发流量雪崩        |
  | 配置优化            | 调整 TCP backlog、BIO 缓冲区、线程亲和等 |
