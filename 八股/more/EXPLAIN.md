------

## ✅ 什么是 `EXPLAIN`？

> `EXPLAIN` 是 MySQL 提供的一种**查询执行计划分析工具**，用于展示 **SQL 语句在执行时的具体执行路径**，包括是否使用索引、扫描方式、连接类型、预估行数等。

通过它可以判断：

- SQL 有没有走索引？
- 用了什么索引？
- 是全表扫描还是索引扫描？
- 哪一步最耗资源？

------

## 🧪 使用方式

```sql
EXPLAIN SELECT * FROM user WHERE id = 1;
```

或使用更详细的新语法：

```sql
EXPLAIN FORMAT=JSON SELECT ...
```

------

## 🔍 `EXPLAIN` 输出结果字段详解（经典 12 列）

| 字段名            | 含义                                                  |
| ----------------- | ----------------------------------------------------- |
| **id**            | 查询中执行的步骤 ID，越大越先执行（子查询或联合查询） |
| **select_type**   | 查询类型（如 SIMPLE、PRIMARY、SUBQUERY、UNION）       |
| **table**         | 当前操作的表或临时表名                                |
| **partitions**    | 涉及的分区（如果有）                                  |
| **type**          | **连接类型（重点）**，越靠左越好                      |
| **possible_keys** | 查询可能使用的索引                                    |
| **key**           | **实际使用的索引**（为空代表未使用索引）              |
| **key_len**       | 索引使用的字节数                                      |
| **ref**           | 哪个列与索引进行匹配（如 const、func、列名）          |
| **rows**          | **预估扫描行数**（越少越好）                          |
| **filtered**      | 行过滤比例，越高代表越精准                            |
| **Extra**         | 额外信息（是否排序、是否回表、是否使用临时表等）      |

------

## 📌 `type` 字段的常见取值（性能从好到差）

| type 值  | 描述                         | 是否好   |
| -------- | ---------------------------- | -------- |
| `system` | 表仅有一行                   | ✅ 最优   |
| `const`  | 用主键或唯一索引查一行       | ✅ 优秀   |
| `eq_ref` | 唯一索引连接，返回一行       | ✅ 好     |
| `ref`    | 普通索引连接，返回多行       | ✅ 可接受 |
| `range`  | 范围查询（BETWEEN, <, > 等） | ✅ 一般   |
| `index`  | 遍历整个索引（不回表）       | ⚠️ 较差   |
| `ALL`    | 全表扫描                     | ❌ 最差   |

------

## 🔎 常见 Extra 信息及含义

| Extra 内容            | 含义说明                              |
| --------------------- | ------------------------------------- |
| Using index           | 覆盖索引，无需回表，性能好            |
| Using where           | 使用了 where 过滤条件                 |
| Using temporary       | 使用了临时表，常见于 group by、排序等 |
| Using filesort        | 需要额外排序（不是索引顺序），性能差  |
| Using index condition | ICP（索引条件下推），提升效率         |
| NULL                  | 没额外操作，通常代表最优路径          |

------

## 📚 实际例子分析

```sql
EXPLAIN SELECT name FROM user WHERE id = 10;
```

输出结果可能是：

| id   | select_type | table | type  | possible_keys | key     | key_len | rows | Extra       |
| ---- | ----------- | ----- | ----- | ------------- | ------- | ------- | ---- | ----------- |
| 1    | SIMPLE      | user  | const | PRIMARY       | PRIMARY | 4       | 1    | Using index |

✅ 表示：

- 使用了主键索引；
- 只查一行；
- 是覆盖索引（只查 name，不用回表）；
- 非常优秀。

## 💬 面试答题模板

> `EXPLAIN` 是 MySQL 提供的 SQL 执行计划分析工具，可以用来查看 SQL 是否使用了索引、扫描了多少行、是否使用临时表或排序等信息。通过分析 `type`、`key`、`rows` 和 `Extra` 字段，可以判断语句的性能瓶颈。常见的索引失效、全表扫描、排序等都可以通过 EXPLAIN 定位，是 SQL 优化的重要手段。

> `EXPLAIN` 是 SQL 的“透视眼”，它告诉你执行过程的**每一步**，让你找出性能瓶颈并进行优化。
