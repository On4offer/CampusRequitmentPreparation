### **ZooKeeper 简介**

**背景**：在分布式系统中，节点之间需要协调一致地完成任务（如服务注册、选主、分布式锁等）。**ZooKeeper** 是 Apache 提供的一款**高性能、分布式协调服务框架**，专门用于处理这些“分布式一致性”问题。

------

### **一、ZooKeeper 是什么？**

ZooKeeper 是一个**开源的分布式协调服务**，本质上是一个提供 **数据一致性、高可用性** 和 **顺序性保证** 的中心节点服务，客户端通过它来实现对分布式节点的统一管理。

------

### **二、ZooKeeper 的核心特点**

| 特性           | 描述                                            |
| -------------- | ----------------------------------------------- |
| **高可用性**   | 通过集群部署，允许部分节点宕机仍可服务          |
| **顺序一致性** | 所有操作按顺序执行，支持版本控制                |
| **原子性**     | 所有操作要么成功执行，要么全部失败              |
| **实时性**     | 客户端能及时感知数据变更（基于 Watch 机制）     |
| **持久性**     | 数据存储在内存 + 日志 +快照中，服务重启后仍保留 |

------

### **三、ZooKeeper 的应用场景**

1. **服务注册与发现**
   - 微服务集群中，服务注册到 ZooKeeper，其他服务可通过监听实时获取可用服务列表。
2. **分布式锁**
   - 通过创建临时顺序节点，谁编号最小谁获锁，其他节点监听上一个节点，达到锁的协调控制。
3. **主从选举**
   - 各节点尝试创建同名临时节点，谁先成功谁就是主节点（Master）。
4. **配置中心**
   - 全局配置项存入 ZooKeeper，客户端监听数据变更自动更新配置。
5. **命名服务**
   - 类似于 DNS 的作用，为服务提供统一命名和访问路径。

------

### **四、ZooKeeper 的核心概念**

#### 1. **数据节点（ZNode）**

- ZooKeeper 中所有数据以“树形结构”存储，称为 **ZNode**。
- 类似于文件系统：`/config/serviceA/version`

#### 2. **节点类型**

- **持久节点**（Persistent）：创建后永久存在，除非主动删除；
- **临时节点**（Ephemeral）：会话断开自动删除；
- **顺序节点**（Sequential）：系统自动为节点添加序号（用于选主或加锁）；
- **组合节点**：临时+顺序、持久+顺序。

#### 3. **Watch 机制**

- 客户端可对节点注册监听（`watch`），当节点发生变更时，ZooKeeper 会通知对应客户端。

#### 4. **版本控制**

- 每个 ZNode 都有版本号，更新操作需指定版本，防止并发覆盖。

------

### **五、ZooKeeper 架构设计**

ZooKeeper 集群通常由多个服务器节点组成：

- **Leader（主）**：处理所有写请求，负责数据同步；
- **Follower（从）**：负责读请求和转发写请求；
- **Observer（观察者）**：不参与选主，减轻主从负载。

#### ZooKeeper 使用 **ZAB 协议（ZooKeeper Atomic Broadcast）** 保证数据一致性。

------

### **六、ZooKeeper 的优点与缺点**

#### **优点：**

- 分布式一致性强，CAP 中偏向 CP；
- 实现简单，API 接口轻量；
- 广泛应用于 Hadoop、Kafka、Dubbo 等系统中；
- 支持事件监听，响应及时。

#### **缺点：**

- 数据量大时不适合存储大数据；
- 并发写性能有限（集中处理写操作）；
- 部署维护需要额外运维能力；
- 新手学习曲线稍高。

------

### **七、小结一句话**

> **ZooKeeper 是一个用于构建分布式系统中协调、注册、选主、配置同步等场景的强一致性服务框架，是许多中间件的基础组件。**
