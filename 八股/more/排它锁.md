**排它锁（Exclusive Lock，简称 X 锁）** 是 MySQL InnoDB 存储引擎中一种**最强的行锁类型**，用于确保在加锁期间，**只有当前事务能够读写该行数据，其他事务既不能读也不能写**。

------

### 一、排它锁的作用

- 确保被锁住的行在事务提交之前不会被其他事务读取或修改。
- 常用于数据更新、删除等需要独占访问的操作。

------

### 二、加锁方式（两种）

1. **自动加锁**：

   - 执行 `UPDATE`、`DELETE`、`INSERT` 等语句时，InnoDB 会自动对相关行加排它锁。

   ```sql
   UPDATE users SET age = 25 WHERE id = 1;
   ```

2. **显式加锁**：

   - 手动通过 `SELECT ... FOR UPDATE` 加排它锁（必须在事务中）。

   ```sql
   START TRANSACTION;
   SELECT * FROM users WHERE id = 1 FOR UPDATE;
   ```

------

### 三、排它锁的特性

| 特性               | 描述                                                   |
| ------------------ | ------------------------------------------------------ |
| 锁粒度             | 行级锁（依赖索引），未命中索引可能锁全表               |
| 与共享锁兼容性     | 不兼容，阻塞 S 锁                                      |
| 与其他排它锁兼容性 | 不兼容，只能一个事务加 X 锁                            |
| 加锁范围           | 精确锁住索引对应的记录，默认是 Next-Key Lock（含间隙） |
| 锁释放             | 事务提交或回滚后自动释放                               |

------

### 四、示例说明

#### 表：`orders`

```sql
+----+--------+---------+
| id | user   | status  |
+----+--------+---------+
| 1  | Tom    | pending |
+----+--------+---------+
```

#### 事务 A：

```sql
START TRANSACTION;
SELECT * FROM orders WHERE id = 1 FOR UPDATE;
-- 对 id=1 加排它锁
```

#### 事务 B：

```sql
UPDATE orders SET status = 'shipped' WHERE id = 1;
-- 被阻塞，直到事务 A 提交或回滚
```

------

### 五、排它锁 vs 共享锁（对比）

| 比较点     | 排它锁（X）                   | 共享锁（S）                    |
| ---------- | ----------------------------- | ------------------------------ |
| 是否允许读 | 否（其他事务不能读）          | 是（多个事务可以同时读）       |
| 是否允许写 | 否（其他事务不能写）          | 否（自己不能写，别人也不能写） |
| 使用场景   | 修改/删除数据                 | 仅读数据，要求一致性           |
| SQL语法    | `FOR UPDATE` 或写操作自动加锁 | `LOCK IN SHARE MODE`           |

------

### 六、注意事项

- 排它锁是**行锁**，但如果查询语句未走索引，可能**退化为表锁**。
- 在并发场景中使用排它锁，容易**引发锁等待或死锁**，需要注意事务设计。
