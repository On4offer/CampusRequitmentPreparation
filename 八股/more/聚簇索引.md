### **聚簇索引（Clustered Index）简介**

**背景**：聚簇索引是 InnoDB 存储引擎中最核心的数据结构之一，是理解 MySQL 查询优化、主键设计和性能调优的基础知识点。

------

### **一、什么是聚簇索引？**

聚簇索引是一种**将数据行和主键索引存储在一起**的索引方式。它决定了表中数据的物理存储顺序：

> **表的数据存储与主键索引是一体的，数据按主键排序存储。**

在 InnoDB 中，**每张表只能有一个聚簇索引**，通常就是**主键索引**。

------

### **二、聚簇索引 vs 非聚簇索引（辅助索引）**

| 比较项       | 聚簇索引（主键）                 | 非聚簇索引（普通索引）       |
| ------------ | -------------------------------- | ---------------------------- |
| 数据存储     | 索引结构中直接包含完整数据       | 索引结构中只包含主键值       |
| 数量限制     | 每张表只有一个                   | 可以有多个                   |
| 查找过程     | 直接查找到数据行                 | 先查主键，再回表（回表查询） |
| 顺序访问性能 | 高效，适合范围查询（BETWEEN 等） | 回表多，效率稍差             |

------

### **三、聚簇索引的结构（B+ 树）**

在 InnoDB 中，聚簇索引使用 **B+ 树** 存储结构：

- 每个叶子节点存放的是整行数据；
- 内部节点只存储索引（主键）；
- 数据天然按主键有序排列；
- 查找主键非常高效，O(log n) 时间复杂度。

------

### **四、创建聚簇索引的规则**

InnoDB 引擎自动选择聚簇索引：

1. **如果有主键（PRIMARY KEY）** → 作为聚簇索引；
2. **如果没有主键，但有唯一 NOT NULL 的索引** → 选其中一个作为聚簇索引；
3. **否则** → InnoDB 自动生成一个隐藏的行 ID 作为聚簇索引。

------

### **五、[回表操作](回表操作)（使用辅助索引时）**

当使用非聚簇索引（如普通字段索引）查询时：

- **非聚簇索引存的是主键值**；
- 查询时需“回表”通过主键再查真实数据；
- 这就是为什么 [`SELECT *` 比 `SELECT id` 慢](SELECT 星 比 SELECT id)，**回表次数多了**。

------

### **六、聚簇索引的优劣势**

#### 优点：

- **主键查询快**；
- **范围查询快**（如 `BETWEEN`、`<`、`>`）；
- 数据物理顺序与索引一致，提升磁盘预读效率。

#### 缺点：

- 插入新的主键可能导致页面分裂（尤其是非自增主键）；
- 更新主键代价大（相当于删除+插入）；
- 辅助索引会更大，因为它们要存主键值（而不是 RowID）。

------

### **七、设计建议**

- 尽量使用 **自增主键**，避免频繁的页分裂；
- 主键越小越好（如 `BIGINT` 优于 `UUID`），减少 B+ 树层级；
- 频繁按范围或顺序读取的字段应设计为主键或前缀索引。

------

### **八、小结一句话**

> **聚簇索引是 InnoDB 存储引擎中“数据即索引”的结构，主键与数据一体存储，查询主键或范围性能极高，但每表只能有一个聚簇索引。**
