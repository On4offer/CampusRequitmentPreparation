### **[HashMap](HashMap) 的扩容机制详解**

**背景**：`HashMap` 是 Java 中最常用的键值对集合之一，其扩容机制直接影响性能和效率。面试中常考底层细节，如负载因子、扩容时机、再哈希（rehash）过程等。

------

### **一、[HashMap](HashMap) 的基本结构回顾**

- 底层结构是：**数组 + 链表（+ 红黑树）**
- 每个键通过 `hashCode()` 计算索引，定位数组中的桶（bucket）
- 同一桶中发生冲突时，采用链表或树存储

------

### **二、扩容的触发条件**

HashMap 扩容是在**“当前容量 × 负载因子 < 元素数量”**的情况下自动触发的。

- **默认初始容量**：16
- **默认负载因子**：0.75
- **触发阈值**：`threshold = capacity × loadFactor`

#### 示例：

```java
capacity = 16;
loadFactor = 0.75;
threshold = 16 × 0.75 = 12;

当元素数 > 12，就会触发扩容
```

------

### **三、扩容过程详解**

1. **创建一个新数组，容量是原数组的 2 倍**
2. **重新计算每个元素在新数组中的位置**
   - 这是因为扩容后，*数组长度变了，哈希结果对应的索引也会变*（`(n - 1) & hash`）
3. **将老数组中的元素转移到新数组**
   - 原链表结构保持顺序
   - 如果链表长度 ≥ 8 且数组容量 ≥ 64，链表会转成红黑树

------

### **四、[Rehash](Rehash) 过程的优化（Java 8）**

在 Java 8 中，扩容时的节点迁移**避免了重新计算 hash 值**，通过如下方式优化：

- 对每个桶中的节点，根据原位置 `index` 和 `index + oldCapacity` 决定是留在原位置，还是迁移到新的位置。

------

### **五、树化与取消树化**

- 当**链表长度 ≥ 8 且数组容量 ≥ 64**，链表会转换为红黑树，提升查询性能。
- 当树中元素数 < 6，结构会退化回链表。

------

### **六、扩容的影响**

| 优点               | 缺点                         |
| ------------------ | ---------------------------- |
| 保证哈希表查找效率 | 扩容过程性能开销大，可能阻塞 |
| 减少哈希冲突       | 迁移过程涉及重新哈希和复制   |

因此在高性能场景中，建议：

- **提前指定合适初始容量**；
- 尽量避免频繁扩容（特别是大数据量下）；
- 考虑使用 `ConcurrentHashMap` 替代 `HashMap`（线程安全）。

------

### **七、小结一句话**

> HashMap 的扩容通过“容量翻倍 + 重新分布桶元素”实现，触发点是元素数超过 `threshold = capacity × loadFactor`，并会导致较大的性能开销。
