**表锁（Table Lock）** 是 MySQL 中一种 **锁定整个表** 的锁机制，在执行期间会阻止其他会话对该表的读或写操作，直到锁被释放。

------

### 一、表锁的特点

- **作用范围大**：一次锁定整个表。
- **加锁速度快、开销小**。
- **并发能力低**：在高并发场景下性能较差。
- **适用于 MyISAM、MEMORY 等存储引擎**，InnoDB 也支持，但通常不推荐使用。

------

### 二、表锁的类型

1. **读锁（共享锁 / READ Lock）**

   - 多个线程可以同时读取该表。

   - 加了读锁后，不能写，只能读。

   - 语法：

     ```sql
     LOCK TABLES table_name READ;
     ```

2. **写锁（排他锁 / WRITE Lock）**

   - 只有一个线程可以读或写该表。

   - 加了写锁后，其他线程既不能读也不能写。

   - 语法：

     ```sql
     LOCK TABLES table_name WRITE;
     ```

3. **解锁**

   - 使用以下语句释放表锁：

     ```sql
     UNLOCK TABLES;
     ```

------

### 三、使用表锁的场景

- **批量数据处理**（插入、删除、更新）时，使用写锁可以防止其他线程干扰。
- **数据备份**时加读锁，防止数据被修改。
- 在使用 MyISAM 或 MEMORY 引擎时，由于不支持行锁，表锁是唯一的锁方式。

------

### 四、注意事项

- 表锁必须**显式加锁和释放**，在事务中使用时要小心死锁风险。
- 加锁顺序不当容易产生阻塞。
- InnoDB 虽然支持表锁，但通常推荐使用其更细粒度的 **行锁**。

------

### 一、表锁 vs 行锁 对比表

| 对比项       | 表锁（Table Lock）            | 行锁（Row Lock）            |
| ------------ | ----------------------------- | --------------------------- |
| 锁定范围     | 整张表                        | 一行或多行                  |
| 并发性能     | 低                            | 高                          |
| 开销         | 较小                          | 较大                        |
| 加锁方式     | 显式（LOCK TABLES）           | 自动（基于 SQL 语句和索引） |
| 死锁可能     | 小                            | 大                          |
| 使用场景     | 小数据量、批处理、MyISAM 引擎 | 高频并发读写、InnoDB 引擎   |
| 锁释放       | 需要显式释放（UNLOCK TABLES） | 事务提交或回滚后自动释放    |
| 是否依赖索引 | 否                            | 是，索引不命中可能锁全表    |

------

### 二、锁冲突示例

#### 情景描述：

两个事务对同一张用户表 `users` 进行操作。

#### 假设数据：

```sql
+----+--------+
| id | name   |
+----+--------+
| 1  | Alice  |
| 2  | Bob    |
+----+--------+
```

#### 示例 1：表锁冲突（MyISAM 表）

**事务 A：**

```sql
LOCK TABLES users WRITE;
UPDATE users SET name = 'AliceA' WHERE id = 1;
-- 未执行 UNLOCK TABLES
```

**事务 B：**

```sql
SELECT * FROM users; -- 会阻塞，直到事务 A 执行 UNLOCK
```

→ 因为 A 锁住整张表，B 不能读或写。

------

#### 示例 2：行锁冲突（InnoDB 表）

**事务 A：**

```sql
START TRANSACTION;
SELECT * FROM users WHERE id = 1 FOR UPDATE;
-- 此时 id=1 行被排他锁锁住
```

**事务 B：**

```sql
START TRANSACTION;
UPDATE users SET name = 'AliceB' WHERE id = 1; -- 阻塞
UPDATE users SET name = 'BobB' WHERE id = 2;   -- 成功（id=2 没锁）
```

→ 只有试图访问 id=1 的操作被阻塞，其他行不受影响，体现了行锁的高并发性。
