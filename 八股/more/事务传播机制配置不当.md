你问得很好！**事务传播机制配置不当** 是导致 Spring 项目中事务失效、数据不一致、回滚失败等问题的**根本原因之一**。我们来系统讲清楚它是什么、会引发什么问题、有哪些典型误区以及如何正确配置。

------

## ✅ 什么是“事务传播机制配置不当”？

> 指的是在一个方法调用链中，**传播属性设置不合理**，导致事务边界、回滚逻辑、隔离性无法按预期工作。

------

## 📌 常见的配置不当情况（重点场景）

### ❌ 场景 1：**子方法用 `REQUIRES_NEW`，主方法回滚但子事务未回滚**

```java
@Service
public class OrderService {
    @Transactional
    public void createOrder() {
        orderDao.insert();               // 主事务
        try {
            paymentService.pay();        // 使用 REQUIRES_NEW
        } catch (Exception e) {
            log.error("支付异常", e);    // 异常被捕获
        }
        throw new RuntimeException("主事务失败"); // 主事务回滚
    }
}

@Service
public class PaymentService {
    @Transactional(propagation = Propagation.REQUIRES_NEW)
    public void pay() {
        paymentDao.insert(); // 单独新事务，已提交
        throw new RuntimeException("支付失败");
    }
}
```

**后果**：

- 支付记录写进去了（已提交）；
- 订单却失败回滚；
- 数据不一致！

✅ **修复建议**：

- 不要轻易使用 `REQUIRES_NEW`，除非你明确要“强制提交”；
- 子方法应抛出异常，让主方法一起回滚。

------

### ❌ 场景 2：**父方法无事务，子方法用 REQUIRED，以为能一起回滚**

```java
public void registerUser() {
    userService.save(); // 无事务
    roleService.assign(); // @Transactional(propagation = REQUIRED)
    throw new RuntimeException(); // 没有任何事务控制
}
```

**后果**：

- `save()` 和 `assign()` 各自执行并提交；
- `registerUser()` 出现异常，却**不能回滚**；
- 事务传播失败！

✅ **修复建议**：

- 外层必须加事务；
- 或使用 `MANDATORY` 强制校验外层必须有事务。

------

### ❌ 场景 3：**使用 `NESTED` 却以为它是新事务**

```java
@Transactional
public void outer() {
    saveMain();               // 主事务
    try {
        nestedSave();         // NESTED
    } catch (Exception e) {
        // swallow
    }
    throw new RuntimeException();
}

@Transactional(propagation = Propagation.NESTED)
public void nestedSave() {
    saveDetail();             // 保存点创建
    throw new RuntimeException();
}
```

**后果**：

- `nestedSave()` 虽然抛异常并部分回滚；
- 但外层事务失败仍导致**全部回滚**；
- 并**不是新事务**！

✅ **正确理解**：

- `NESTED` 创建的是 **保存点 SavePoint**；
- 外层事务失败时，一样全部撤销。

------

## 🧠 配置不当的核心后果

| 问题表现               | 实质原因                                      |
| ---------------------- | --------------------------------------------- |
| 数据不一致             | 子事务提交了，主事务失败                      |
| 事务不生效             | 外层没事务，子事务 `REQUIRED` 实际无效        |
| 回滚不彻底             | 使用 `REQUIRES_NEW` 或错误 swallow 异常       |
| 异常不抛出，事务不回滚 | 异常被捕获或是 checked 异常未设置 rollbackFor |
| 调用自己类方法不生效   | Spring AOP 代理无法拦截本类内部调用           |

------

## ✅ 配置传播机制的正确建议

| 场景                       | 建议传播行为             |
| -------------------------- | ------------------------ |
| 普通业务方法嵌套调用       | 默认 `REQUIRED` ✅        |
| 日志、审计、通知等独立提交 | `REQUIRES_NEW` 🟡（慎用） |
| 仅在事务中允许调用的方法   | `MANDATORY`              |
| 不需要事务或调用慢的操作   | `NOT_SUPPORTED`          |
| 希望子方法失败不影响主流程 | 可结合 `NESTED + try`    |
| 控制台查阅、统计分析等     | `SUPPORTS`               |

------

## 💬 面试答题模板

> 事务传播机制配置不当主要体现在对子方法事务行为判断错误，如 `REQUIRES_NEW` 提交导致主事务回滚无效，或主方法无事务导致子事务自动提交。在 Spring 中常见问题包括异常未抛出、同类内部调用无事务、嵌套事务误解等。建议根据业务实际合理使用传播属性，并配合日志、异常、测试等手段保障事务的一致性。

------

## ✅ 一句话总结

> 事务传播机制配置不当会导致事务不生效、回滚异常、数据不一致，必须结合调用链清晰设计事务边界。
