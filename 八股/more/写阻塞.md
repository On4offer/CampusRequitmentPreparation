## ✅ 什么是写阻塞（Write Blocking）？

**写阻塞**是指：

> 一个事务尝试对某行数据进行写操作（如 `UPDATE` / `DELETE`）时，发现该行已经被**其他事务加了锁（通常是共享锁或排他锁）**，就必须**等待释放锁后才能继续执行**，从而导致阻塞。

------

## 🔐 本质原因

写操作加的是 **排他锁（X锁）**，但：

- 如果该行已经被其他事务加了 **共享锁（S锁）或排他锁**，就不能再加 X 锁；
- 所以当前事务只能等待，这就形成了 **写阻塞**。

------

## 📚 场景举例

### 示例 1：被读锁阻塞写操作

```sql
-- 事务 A：
START TRANSACTION;
SELECT * FROM user WHERE id = 1 LOCK IN SHARE MODE; -- 加 S 锁

-- 事务 B：
START TRANSACTION;
UPDATE user SET name = 'Tom' WHERE id = 1; -- 尝试加 X 锁，阻塞！
```

事务 B 想写，但事务 A 持有 S 锁（共享锁），不释放，事务 B 就只能“等”。

------

### 示例 2：写写冲突

```sql
-- 事务 A：
START TRANSACTION;
UPDATE product SET price = price + 1 WHERE id = 10; -- 加 X 锁

-- 事务 B：
START TRANSACTION;
UPDATE product SET price = price - 1 WHERE id = 10; -- 阻塞等待事务 A 释放
```

两个事务都想修改同一行，MySQL 会让后来的事务阻塞，直到前一个提交或回滚。

------

## ⚠️ 写阻塞 vs 死锁的区别

| 概念   | 描述                                                         |
| ------ | ------------------------------------------------------------ |
| 写阻塞 | 写操作等待其他事务释放锁，**有机会继续执行**                 |
| 死锁   | 多个事务互相等待，永远无法执行，**最终数据库会中断一个事务** |

------

## 🚦 如何判断和处理写阻塞？

### ✅ 检查方法：

1. 查看当前等待锁的事务：

   ```sql
   SHOW ENGINE INNODB STATUS\G
   ```

2. 使用 `information_schema.innodb_locks`（MySQL 5.7+）查看等待队列。

3. 设置合理的 **锁等待超时**（默认 50 秒）：

   ```sql
   SET innodb_lock_wait_timeout = 10;
   ```

## 🧠 面试回答模板

> 写阻塞是指当一个事务对某条记录尝试写入时，发现该记录被其他事务加锁（如共享锁或排他锁），就会被阻塞等待。MySQL 使用排他锁控制写操作的并发安全，因此只允许一个事务修改同一条数据。如果其他事务没释放锁，写操作只能等待，造成写阻塞。这是事务隔离和一致性保障的体现，但也需要合理控制锁粒度和事务时长，避免性能问题。

------

## ✅ 总结一句话

> 写阻塞是写操作因锁冲突被挂起等待的现象，是并发控制中正常但需要优化的重要一环。
