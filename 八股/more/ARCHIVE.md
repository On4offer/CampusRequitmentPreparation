### **ARCHIVE 存储引擎简介**

**背景**：`ARCHIVE` 是 MySQL 提供的一种**专为归档和压缩存储设计的存储引擎**，适用于**只写不改，偶尔查询的大量历史数据**，如日志、审计记录、统计数据等。

------

### **一、ARCHIVE 的核心特点**

| 特性                         | 描述                                   |
| ---------------------------- | -------------------------------------- |
| **压缩存储**                 | 使用 zlib 压缩算法，节省磁盘空间。     |
| **仅支持 INSERT 和 SELECT**  | 不支持 `UPDATE` 和 `DELETE`。          |
| **表级锁**                   | 插入和查询都使用表级锁。               |
| **不支持索引（仅支持主键）** | 不支持二级索引、全文索引，查询效率低。 |
| **数据持久化**               | 数据存储在磁盘上，支持持久化。         |
| **支持自动增长列**           | 可用于主键字段的自增 ID。              |

------

### **二、使用方式**

#### 创建 ARCHIVE 表：

```sql
CREATE TABLE logs (
    id INT AUTO_INCREMENT PRIMARY KEY,
    message TEXT,
    created_at TIMESTAMP
) ENGINE=ARCHIVE;
```

#### 插入数据：

```sql
INSERT INTO logs (message, created_at) VALUES ('user login', NOW());
```

#### 查询数据：

```sql
SELECT * FROM logs WHERE id = 1000;
```

------

### **三、优缺点对比**

| 优点                         | 缺点                                 |
| ---------------------------- | ------------------------------------ |
| 高压缩率，节省磁盘空间       | 不支持 `UPDATE`/`DELETE` 操作        |
| 插入性能优异，写入快         | 查询效率低（无索引）                 |
| 数据持久化，适合长期存储     | 不支持外键、事务、全文索引等高级功能 |
| 可作为审计或归档数据存储方案 | 不适合实时查询或频繁访问的业务场景   |

------

### **四、适用场景**

- 日志归档（如用户访问日志、系统日志）
- 数据仓库的历史快照存储
- 法规要求的长期数据保留（如订单记录、监控数据）
- 数据分析前的冷数据保留

------

### **五、与其他引擎对比**

| 特性     | ARCHIVE                | InnoDB            | MyISAM           |
| -------- | ---------------------- | ----------------- | ---------------- |
| 支持事务 | 否                     | 是                | 否               |
| 支持索引 | 仅主键                 | 支持多索引        | 支持多索引       |
| 查询性能 | 差                     | 优（有索引）      | 较好             |
| 写入性能 | 快（顺序写）           | 一般（写锁+事务） | 较快（表锁）     |
| 适合场景 | 日志归档、数据压缩存储 | 通用事务处理      | 简单读多写少场景 |

------

### **六、小结一句话**

> **ARCHIVE 存储引擎适合只追加写入、不修改的数据归档场景**，具备高压缩率和写入效率，但查询能力有限，不能用于实时或频繁访问的数据存储。
