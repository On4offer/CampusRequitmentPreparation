### Redis 的持久化机制介绍

**背景**：Redis 是一个基于内存的高性能数据库，为了防止断电或重启导致数据丢失，提供了持久化机制。**持久化是 Redis 在实际项目中非常关键的功能**，特别是在数据安全性要求较高的场景下。

------

### Redis 的两种持久化方式：

#### 1. RDB（Redis DataBase 快照）

**原理**：

- Redis 在指定时间间隔内，将内存中的数据**生成快照（Snapshot）写入磁盘**。
- 是**压缩后的二进制文件**，文件名如 `dump.rdb`。

**特点**：

- 启动恢复速度快，占用空间小；
- 适合灾备和定期备份；
- 持久化时间不确定，容易丢数据；
- 执行时会 fork 子进程，占用 CPU 和内存；

**配置**（在 redis.conf）：

```bash
save 900 1   # 900秒内有1个写操作则触发
save 300 10  # 300秒内有10个写操作则触发
```

------

#### 2. AOF（Append Only File）

**原理**：

- Redis **将所有写命令按顺序追加到日志文件中（appendonly.aof）**，可通过重放日志恢复数据。

**特点**：

- 数据更完整，可配置写入策略（always、everysec、no）；
- 文件较大，恢复速度慢；
- 有日志重写机制（AOF rewrite）防止文件无限增大；
- 实时性更强，适合要求**高可靠性**的系统；

**配置**：

```bash
appendonly yes
appendfsync everysec   # 每秒写入一次，兼顾性能和安全
```

------

### 3. 混合持久化（AOF + RDB）

从 Redis 4.0 起，支持 **混合持久化**：

- 启动恢复时，先加载 **RDB 快照**，再加载后续 **AOF 修改日志**，兼顾速度和完整性；
- 推荐在生产中开启混合模式。

------

### 项目使用建议

- **普通缓存型业务**：可只用 RDB，提高性能；
- **重要数据系统（如订单、交易）**：建议启用 AOF，并设置 `appendfsync everysec`；
- **关键系统**：使用 **AOF + RDB 混合持久化**，保证数据完整性与恢复效率；
