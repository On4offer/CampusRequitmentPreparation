------

## ✅ 什么是联合索引（Composite Index）

**联合索引**是指：

> 在一个索引中包含 **多个列** 的索引，常用于多个字段组合查询场景。

它在数据库中生成的是一个多列组合键的 **B+ 树结构**，每个索引条目都包含这几列的值。

### 📌 示例：

```sql
CREATE INDEX idx_name_age ON user(name, age);
```

表示创建了一个以 `name` 为第一列、`age` 为第二列的联合索引。

------

## 🧠 为什么使用联合索引？

- 减少多个单列索引的维护成本；
- 提高多字段组合查询的效率；
- 能够覆盖多个查询条件，减少回表；
- 支持**覆盖索引优化**（即查询字段全部在索引中）。

------

## 🔍 工作原理（B+ 树结构）

以 `(name, age)` 为例，索引是按如下顺序排列的：

```
(name1, age1)
(name1, age2)
(name2, age1)
...
```

这是一个“字典排序”结构，B+树先按 `name` 排序，再在 `name` 相同的情况下按 `age` 排序。

------

## 📌 最重要的原则：**最左前缀原则**

联合索引遵循“**最左前缀匹配**”规则：

| 查询条件                       | 是否命中索引 (`name, age`) |
| ------------------------------ | -------------------------- |
| `WHERE name = 'Tom'`           | ✅ 是                       |
| `WHERE name = 'Tom' AND age=3` | ✅ 是                       |
| `WHERE age = 3`                | ❌ 否                       |
| `WHERE age = 3 AND name='Tom'` | ✅ 是（MySQL 会优化顺序）   |
| `WHERE name LIKE 'T%'`         | ✅ 是（前缀匹配）           |
| `WHERE name LIKE '%om'`        | ❌ 否（不是前缀）           |

只有从最左边的列开始连续使用，才能用上联合索引。

------

## 🚀 优点

- ✅ **提升多列条件查询性能**
- ✅ **支持排序优化**（比如 `ORDER BY name, age`）
- ✅ **支持覆盖索引**，减少回表
- ✅ 比多个单列索引更高效（MySQL 不会同时使用多个单列索引做“与”查询）

------

## ⚠️ 注意事项 / 缺点

- ❌ 最左前缀不匹配时无效
- ❌ 过多列会增加维护成本、占用空间
- ❌ 非常频繁变更的字段不适合建联合索引

------

## 💬 面试回答模板（结构化）

> 联合索引是指包含多个字段的索引，它在底层是按多个列的组合进行排序的 B+ 树结构。在查询时遵循最左前缀原则，也就是说查询条件必须从索引的第一列开始，才能命中索引。相比多个单列索引，联合索引查询效率更高，还支持覆盖索引和排序优化。但它也有最左匹配限制，列数太多也会增加维护开销。

------

## ✅ 典型应用场景

- `WHERE user_id = ? AND status = ?`
- `ORDER BY create_time DESC, id DESC`
- `GROUP BY region, category`

------

## ✍️ 补充建议

- 常将**过滤性高的字段放前面**；
- 建索引前最好配合**慢查询日志**和 **EXPLAIN 分析**判断查询走没走索引；
- 配合 **覆盖索引** 可以极大提高查询效率（避免回表）；
