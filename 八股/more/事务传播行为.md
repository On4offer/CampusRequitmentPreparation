  ## ✅ 二、什么是事务传播行为？

  > 在 Spring 中，一个方法在事务中被调用时，它该不该开启新事务，或者加入现有事务，由 **事务传播行为（Propagation）** 决定。

  换句话说，它定义了**当前方法执行时是否需要事务**、**是否复用调用方事务**、**事务边界怎么处理**等问题。

  ------
  
  ## ✅ 三、Spring 事务传播类型（7种）
  
  Spring 定义在枚举类 `org.springframework.transaction.annotation.Propagation` 中：
  
  | 类型               | 含义                       | 场景说明                   |
  | ------------------ | -------------------------- | -------------------------- |
  | `REQUIRED`（默认） | 有事务则加入，无则新建     | 常用；保证操作在一个事务中 |
  | `REQUIRES_NEW`     | 总是新建事务，挂起原事务   | 独立事务，例如日志记录     |
  | `NESTED`           | 嵌套事务，内层失败可回滚   | 底层数据库需支持保存点     |
  | `SUPPORTS`         | 有事务就加入，没有就非事务 | 只读查询                   |
  | `NOT_SUPPORTED`    | 不支持事务，挂起当前事务   | 调用第三方接口             |
  | `NEVER`            | 不允许事务，有事务则抛异常 | 强制无事务                 |
  | `MANDATORY`        | 必须在事务中运行，否则异常 | 强依赖调用方事务           |
  
  ------
  
  ## ✅ 四、事务传播类型详解（重点）
  
  ### 🔸 1. `REQUIRED`（默认）

  ```java
  @Transactional(propagation = Propagation.REQUIRED)
  ```

  - 如果当前线程已有事务，则加入；
  - 否则创建新事务。
  
  🧩 **常见场景**：多个业务操作需要在同一事务中完成。
  
  ------
  
  ### 🔸 2. `REQUIRES_NEW`
  
  ```java
  @Transactional(propagation = Propagation.REQUIRES_NEW)
  ```
  
  - **总是新建一个事务**
  - 如果存在外层事务，会**挂起外层事务**

  🧩 **常见场景**：日志、消息记录不受主业务失败影响。

  ------

  ### 🔸 3. `NESTED`

  ```java
  @Transactional(propagation = Propagation.NESTED)
  ```

  - 内部事务依赖外部事务，使用**保存点（savepoint）**
  - 内部事务异常时，可以只回滚自己，不影响外部事务
  
  🧩 **常见场景**：复杂流程中的子模块局部失败但不影响整体提交。
  
  ⚠️ 注意：数据库需支持保存点，如 MySQL 的 InnoDB。
  
  ------
  
  ## ✅ 五、事务传播行为对回滚的影响
  
  | 外层方法   | 内层传播       | 内层异常处理   | 是否回滚外层事务 |
  | ---------- | -------------- | -------------- | ---------------- |
  | `REQUIRED` | `REQUIRED`     | 捕获异常       | ❌ 不回滚         |
  | `REQUIRED` | `REQUIRES_NEW` | 捕获异常       | ❌ 不回滚         |
  | `REQUIRED` | `NESTED`       | 手动回滚保存点 | ❌ 不回滚外层事务 |

  建议：在捕获异常时，如需回滚事务，可通过编程式 `TransactionAspectSupport.currentTransactionStatus().setRollbackOnly();`

  ## ✅ 六、实战案例：订单与日志独立提交
  
  ```java
  @Service
  public class OrderService {
  
      @Autowired
      private OrderRepository orderRepository;
  
      @Autowired
      private LogService logService;
  
      @Transactional
      public void createOrder() {
          orderRepository.save(new Order());
  
          try {
              logService.saveLog(); // 使用 REQUIRES_NEW
          } catch (Exception e) {
              // 不影响主业务
          }
  
          // 如果这里异常，整个事务回滚（不包括日志）
      }
  }
  
  @Service
  public class LogService {
      @Transactional(propagation = Propagation.REQUIRES_NEW)
      public void saveLog() {
          // 保存操作日志
      }
  }
  ```

  ## ✅ 九、总结建议
  
  - 默认使用 `REQUIRED` 即可覆盖大多数业务场景
  - 对于日志、消息等副作用操作，建议使用 `REQUIRES_NEW`
  - 多模块调用中注意事务传播组合的异常传播与回滚策略
  - 使用 `@Transactional` 时避免同一个类中方法互调（自调用无效）
