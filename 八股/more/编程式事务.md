## ✅ 什么是编程式事务？

> **编程式事务**是指通过**代码显式地控制事务的开始、提交、回滚逻辑**，通常使用 Spring 的 `TransactionTemplate` 或 `PlatformTransactionManager` 进行操作。

通俗来说：

> 你在代码中手动写出“开始事务 ➜ 执行业务 ➜ 成功就提交 ➜ 失败就回滚”的流程，完全由你控制。

------

## 📦 实现方式

Spring 提供两种方式实现编程式事务：

### 1. 使用 `TransactionTemplate`（推荐）

```java
@Autowired
private TransactionTemplate transactionTemplate;

public void doBiz() {
    transactionTemplate.execute(status -> {
        try {
            dao.save1();
            dao.save2();
            return true;
        } catch (Exception e) {
            status.setRollbackOnly(); // 显式回滚
            return false;
        }
    });
}
```

### 2. 使用 `PlatformTransactionManager`（更底层）

```java
@Autowired
private PlatformTransactionManager txManager;

public void doBiz() {
    DefaultTransactionDefinition def = new DefaultTransactionDefinition();
    TransactionStatus status = txManager.getTransaction(def);
    try {
        dao.save1();
        dao.save2();
        txManager.commit(status);
    } catch (Exception e) {
        txManager.rollback(status);
    }
}
```

## ✅ 编程式事务 vs 声明式事务（@Transactional）

| 对比项     | 编程式事务                           | 声明式事务（AOP）              |
| ---------- | ------------------------------------ | ------------------------------ |
| 控制粒度   | 精细，完全自定义                     | 容易用，适合常规业务           |
| 入侵性     | 高，需要写事务控制代码               | 低，只需加注解                 |
| 适合场景   | 异常捕获复杂、跨方法逻辑、条件事务等 | 普通业务方法、一致性事务       |
| 容易出错点 | 写错事务边界、忘记回滚               | AOP 无法代理自身内部方法调用等 |
| AOP 依赖   | ❌ 不依赖 AOP                         | ✅ 依赖 Spring AOP              |

------

## 🧠 编程式事务的典型使用场景

| 场景                                 | 原因                                     |
| ------------------------------------ | ---------------------------------------- |
| 需要条件判断是否提交或回滚事务       | 比如判断某个接口返回值才能决定提交事务   |
| 事务逻辑跨多个方法，不能通过注解控制 | 用编程方式手动写事务边界                 |
| 想把事务逻辑与业务逻辑分离           | 将事务处理抽象出来（模板 + 回调）        |
| 处理多数据源、嵌套事务等复杂场景     | 手动控制多个事务管理器，精细化提交和回滚 |

------

## ❗常见误区

| 误区                                 | 正确做法                                     |
| ------------------------------------ | -------------------------------------------- |
| 捕获异常但忘记手动回滚               | 调用 `status.setRollbackOnly()`              |
| 在 Lambda 表达式中抛出受检异常       | 使用包装 RuntimeException，或 try-catch 包住 |
| 编程式事务和 @Transactional 混用不清 | 避免一个方法里混用，会导致事务行为难以预测   |

## 💬 面试标准答题模板

> 编程式事务是指在代码中手动控制事务的开始、提交和回滚，常用的工具有 `TransactionTemplate` 和 `PlatformTransactionManager`。相比于声明式事务，它提供了更高的灵活性，适用于事务逻辑复杂、跨方法调用或需要根据运行时判断是否回滚的场景。但它入侵性强、代码冗长，日常开发中更推荐使用声明式事务，编程式事务适合特殊需求。

------

## ✅ 一句话总结

> 编程式事务给你事务的“遥控器”，适合定制化需求，但要用得准、控制得稳。
