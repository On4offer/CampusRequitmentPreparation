**InnoDB** 是 MySQL 默认和最常用的**事务型存储引擎**，以其**支持事务、行级锁定和外键**等高级功能而著称。它适用于数据一致性要求高、并发访问频繁的业务场景。

------

### 一、InnoDB 的核心特点

| 特性             | 描述                                                         |
| ---------------- | ------------------------------------------------------------ |
| **支持事务**     | 遵循 ACID 原则，支持 `BEGIN` / `COMMIT` / `ROLLBACK`         |
| **行级锁**       | 高并发场景下性能更优，支持 S 锁、X 锁、意向锁等              |
| **崩溃恢复**     | 使用重做日志（Redo Log）和撤销日志（Undo Log）保证数据完整性 |
| **外键支持**     | 可定义 `FOREIGN KEY` 约束，保证表间数据一致性                |
| **聚簇索引**     | 主键即为数据存储的顺序方式，主键越短越好                     |
| **[MVCC](MVCC)** | 多版本并发控制，实现一致性读而不加锁（适用于快照读）         |
| **自动崩溃恢复** | 使用双写缓冲区和 WAL（预写日志）机制，保障异常恢复           |

------

### 二、InnoDB 的重要机制

#### 1. **事务（Transaction）**

- 支持四大特性：**原子性（A）、一致性（C）、隔离性（I）、持久性（D）**

- 常用语法：

  ```sql
  START TRANSACTION;
  UPDATE ...;
  COMMIT;
  ```

#### 2. **锁机制**

- 支持行锁、表锁，默认使用**行级锁**（高并发友好）
- 支持共享锁（S）、排他锁（X）、意向锁（IS/IX）、间隙锁、Next-Key Lock 等

#### 3. **MVCC（多版本并发控制）**

- 实现非锁定读：`SELECT` 查询通常不会加锁
- 通过隐藏列（事务ID、回滚指针）判断数据可见性

#### 4. **Redo Log（重做日志）**

- 保证 **事务提交后的数据不会丢失**，即使崩溃也能恢复
- 与 **WAL（预写日志）机制**配合使用

#### 5. **Undo Log（撤销日志）**

- 用于支持事务回滚、MVCC版本回溯
- 保留历史版本供快照读使用

#### 6. **聚簇索引（Clustered Index）**

- 数据按主键顺序存储，主键即数据地址
- 副索引（Secondary Index）存储主键作为引用

------

### 三、InnoDB 与 MyISAM 对比

| 特性         | InnoDB             | [MyISAM](MyISAM)   |
| ------------ | ------------------ | ------------------ |
| 是否支持事务 | 是                 | 否                 |
| 锁粒度       | 行级锁             | 表级锁             |
| 是否支持外键 | 是                 | 否                 |
| 并发性能     | 高（多读写场景）   | 一般（写冲突较大） |
| 是否崩溃恢复 | 是（自动恢复机制） | 否（数据容易损坏） |
| 是否默认引擎 | 是（MySQL 5.5+）   | 否                 |

------

### 四、典型使用场景

- 银行、订单系统等要求**数据强一致性**的应用
- 多并发事务同时操作的场景
- 外键关联密集的复杂数据库结构
