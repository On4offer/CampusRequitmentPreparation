------

## ✅ 什么是事务的传播行为？

> **事务传播行为**定义了**当一个方法在事务中运行时，另一个方法是否要共享当前事务**，或**开启新事务**的策略。

通俗来说：

> A 方法有事务，调用了 B 方法，**B 是用 A 的事务？还是自己开一个新的？还是挂起 A 的？** —— 这就是“传播行为”。

------

## 🧠 Spring 定义了 7 种事务传播行为（Propagation）

| 枚举常量           | 含义简述                                 |
| ------------------ | ---------------------------------------- |
| `REQUIRED`（默认） | 有事务就加入，没有就新建一个 ✅（最常用） |
| `REQUIRES_NEW`     | 总是新建事务，原事务挂起                 |
| `NESTED`           | 嵌套事务（开启保存点，可部分回滚）       |
| `SUPPORTS`         | 有事务就加入，没有就不开事务             |
| `NOT_SUPPORTED`    | 总是不开事务，若当前有事务就挂起         |
| `MANDATORY`        | 必须在事务中运行，否则抛异常             |
| `NEVER`            | 必须在非事务中运行，否则抛异常           |

------

## 🔍 详细解释每种行为

### 1. `REQUIRED`（默认）

- **当前有事务就加入**；
- 没有就开启新事务。

🔹 场景：绝大多数业务逻辑都适合使用这个模式。

------

### 2. `REQUIRES_NEW`

- **始终开启新事务**；
- 当前事务（如果有）会被**挂起**。

🔹 场景：如日志记录、短信发送，即使主业务回滚也要提交。

------

### 3. `NESTED`

- 如果当前有事务，就**开启一个嵌套事务（保存点）**；
- 如果没有事务，表现和 `REQUIRED` 一样。

🔹 场景：需要**部分回滚**（例如 A 方法中调用 B，B 出错回滚但 A 继续执行）。

⚠️ 注意：嵌套事务依赖底层数据库支持 `SAVEPOINT`（MySQL InnoDB 支持）。

------

### 4. `SUPPORTS`

- 如果有事务，就加入；
- 如果没有，也**正常执行，但无事务保障**。

🔹 场景：读操作，**不强制事务**，但又希望在事务中时能加入。

------

### 5. `NOT_SUPPORTED`

- 永远不使用事务；
- 如果当前有事务，**挂起它**，等当前方法执行完再恢复。

🔹 场景：某些操作执行时间长，不适合加事务，如批量导入日志。

------

### 6. `MANDATORY`

- 要求必须在事务中执行；
- 如果没有事务，直接抛异常。

🔹 场景：希望被调用方法**只能由事务方法调用**（如某些数据库强一致性逻辑）。

------

### 7. `NEVER`

- 不能在事务中执行；
- 如果当前有事务，抛异常。

🔹 场景：安全操作、外部接口调用，不希望被事务控制。

------

## ⚠️ 常见误区

| 问题                            | 说明                                            |
| ------------------------------- | ----------------------------------------------- |
| 调用本类方法事务不生效          | 因为代理对象未生效，Spring AOP 无法拦截         |
| 异常未抛出事务不回滚            | 默认只回滚 RuntimeException，检查异常需手动配置 |
| `REQUIRES_NEW` 不真正开启新事务 | 如果数据源不支持嵌套或没配置好事务管理器        |

------

## 🧾 示例代码（Spring 注解）

```java
@Transactional(propagation = Propagation.REQUIRES_NEW)
public void saveLog() {
    // 无论调用方事务如何，这里都会开新事务
}
```

## 💬 面试标准答题模板

> Spring 提供了 7 种事务传播行为，最常用的是 `REQUIRED`，它会加入当前事务或新建事务；而 `REQUIRES_NEW` 会挂起当前事务并重新创建一个事务；`NESTED` 支持事务嵌套和局部回滚。传播行为控制了事务的传播方式和边界，是实现复杂业务事务控制的重要机制，尤其在多方法调用、日志分离、异步执行等场景中非常关键。

------

## ✅ 一句话总结

> 事务传播行为定义了“一个事务中方法调用另一个方法时，事务是否共用、挂起、强制或禁止”，是实现业务一致性和灵活控制的关键。
