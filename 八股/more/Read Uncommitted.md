`READ UNCOMMITTED` 是数据库事务的**最低隔离级别**，允许一个事务读取另一个事务**尚未提交的数据**。这种隔离级别提供了**最强的并发性能**，但也是**最不安全的**，容易导致各种并发异常。

------

### 一、特点

| 特性                   | 说明                           |
| ---------------------- | ------------------------------ |
| **是否允许脏读**       | 是，**允许读取未提交数据**     |
| **是否允许不可重复读** | 是，可能同一行两次读出不同数据 |
| **是否允许幻读**       | 是，行数和内容都可能变         |
| **性能**               | 高（几乎无锁）                 |
| **安全性**             | 低（可能读取错误或无效数据）   |

------

### 二、脏读示例

```sql
-- 事务 A
START TRANSACTION;
UPDATE accounts SET balance = balance - 100 WHERE id = 1;
-- 未提交

-- 事务 B
SELECT balance FROM accounts WHERE id = 1;  -- 能看到事务 A 修改后的数据（脏读）

-- 如果事务 A 回滚了，事务 B 的读取结果就是错误的
```

------

### 三、使用场景

- 非关键性数据分析，如日志、报表缓存等；
- 允许偶尔读到不一致数据，追求极致性能；
- 适合只读查询、且对精确性要求不高的系统。

------

### 四、设置方式（MySQL）

```sql
SET SESSION TRANSACTION ISOLATION LEVEL READ UNCOMMITTED;
```

------

### 五、对比其他隔离级别

| 隔离级别             | 是否可能脏读 | 不可重复读 | 幻读             | 适用场景             |
| -------------------- | ------------ | ---------- | ---------------- | -------------------- |
| **READ UNCOMMITTED** | 是           | 是         | 是               | 极致性能、数据可容错 |
| READ COMMITTED       | 否           | 是         | 是               | 多数数据库默认       |
| REPEATABLE READ      | 否           | 否         | 可能/InnoDB 中否 | MySQL 默认（推荐）   |
| SERIALIZABLE         | 否           | 否         | 否               | 强一致性、低并发场景 |

------

### 六、小结一句话：

> `READ UNCOMMITTED` 牺牲数据一致性换取并发性能，**允许事务读取“别人还没提交的数据”**，是最快也最危险的隔离级别。
