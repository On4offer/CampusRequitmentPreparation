“**组合优于继承**”（**Composition over Inheritance**）是面向对象设计的经典原则之一，主张**通过“拥有”而不是“是一个”**来复用功能。

它的核心思想是：

> **与其继承一个类，不如在类中组合（包含）一个对象来使用它的功能。**

------

### 一、为什么组合优于继承？

| 继承问题                     | 组合优势                 |
| ---------------------------- | ------------------------ |
| 强耦合：子类依赖父类实现     | 松耦合：只依赖接口或行为 |
| 易破坏封装：暴露不该用的方法 | 封装好：只暴露需要的功能 |
| 改父类可能影响多个子类       | 组合关系更可控、更灵活   |
| 继承只能“单继承”             | 组合可以包含多个功能组件 |

------

### 二、Java 标准库中的经典例子

#### 1. `HashSet` 使用 `HashMap`（你提到的）

- 不是继承 `HashMap`，而是**组合 `HashMap` 实现功能**。

#### 2. `PriorityQueue` 使用 `Heap`（隐式）

- 内部用数组和堆结构处理优先级，不暴露细节。

#### 3. `ArrayList` 使用数组 `Object[]`

- 并没有继承数组类型，而是组合数组进行管理。

#### 4. `PrintWriter` 使用了 `Writer`

- 包含一个 `Writer` 对象（如 `BufferedWriter`），进行委托。

------

### 三、自定义代码中的应用示例

#### 错误方式（继承）：

```java
class Engine {
    void start() {}
}

class Car extends Engine {  // 不合理，“Car is-a Engine”？不是
}
```

#### 推荐方式（组合）：

```java
class Car {
    private Engine engine = new Engine();
    
    void start() {
        engine.start(); // 委托调用
    }
}
```

这样：

- Car 的功能更灵活；
- 更好地隐藏细节；
- 更容易维护和扩展。

------

### 四、适合使用组合的典型场景

- **你只想用另一个类的功能，而不是它的“身份”**
- **你不想暴露或继承不需要的方法**
- **你需要切换或扩展某个行为（面向接口）**
