## ✅ 二、什么是 AOF？

**AOF（Append Only File）** 是 Redis 提供的一种基于日志的**持久化机制**：

> Redis 将所有对数据库执行的**写命令**（如 `SET`、`DEL`）**按顺序写入到 AOF 文件中**，通过重放这些命令可恢复数据。

生成的文件名通常为：

```
appendonly.aof
```

------

## ✅ 三、AOF 的写入流程（原理）

1. 客户端执行写操作（如 `SET key value`）
2. Redis 将命令追加到 AOF 缓冲区
3. 根据策略决定是否立即写入磁盘（见下方）
4. AOF 文件增长
5. Redis 启动时读取 AOF 文件，**顺序回放命令**恢复数据

------

## ✅ 四、AOF 的写入策略（配置项：`appendfsync`）

| 模式             | 配置值                                 | 特点                      |
| ---------------- | -------------------------------------- | ------------------------- |
| always           | 实时写入磁盘（每次写操作都调用 fsync） | 最安全，性能最低          |
| everysec（默认） | 每秒写入一次                           | 较高性能，最多丢 1 秒数据 |
| no               | 由操作系统决定写入时机                 | 性能最高，最不安全        |

配置示例：

```ini
appendonly yes
appendfsync everysec
```

------

## ✅ 五、AOF 重写机制（避免文件无限增长）

AOF 文件会随着写操作不断增长，Redis 提供 **AOF 重写（rewrite）机制**：

> Redis 会根据当前内存中的数据结构，生成一个最小命令集覆盖原文件，**达到压缩文件体积的目的。**

### AOF 重写触发方式：

- 自动触发（默认当 AOF 文件大小是上次 rewrite 的 2 倍）
- 手动执行命令：`bgrewriteaof`

> 重写不会中断写操作，它会 fork 子进程异步完成，主进程继续服务。

------

## ✅ 六、AOF 的优缺点

| 优点                                  | 缺点                         |
| ------------------------------------- | ---------------------------- |
| 数据更安全，丢失最少（支持 everysec） | 写入频繁，性能略低于 RDB     |
| 支持命令回放，易于调试                | 文件体积更大（多条重复命令） |
| 支持自动重写机制                      | 启动恢复比 RDB 慢            |

------

## ✅ 七、RDB vs AOF（面试高频对比）

| 对比项     | RDB                 | AOF                  |
| ---------- | ------------------- | -------------------- |
| 持久化方式 | 快照                | 写日志               |
| 写入频率   | 间隔写（秒/分钟级） | 实时/每秒            |
| 数据丢失   | 可丢失几分钟数据    | 最多丢 1 秒          |
| 文件体积   | 小                  | 大                   |
| 启动速度   | 快                  | 慢（需 replay 命令） |
| 性能       | 高                  | 稍低（频繁写磁盘）   |

👉 **推荐实际项目使用：RDB + AOF 混合持久化**

## ✅ 八、实际使用场景与案例

### 🧩 场景：金融/电商系统中关键数据保存

> 某支付系统使用 Redis 保存订单流水、支付状态，为避免关键数据丢失，开启 `appendonly yes` + `everysec`，并启用定期 AOF 重写。

### 🧩 场景：Redis 宕机恢复

> 重启 Redis 时加载 AOF 文件进行命令回放，可恢复最近数据，**比仅用 RDB 更可靠**。

## ✅ 十一、最佳实践建议

- 启用 `appendonly yes` + `appendfsync everysec` 是推荐配置
- 定期执行 AOF 重写，或配置触发条件
- 搭配 RDB 使用，实现快速恢复 + 高可靠性
- 部署 Redis Sentinel / Cluster 做高可用保障
- 备份 AOF 文件到外部存储，防止本地磁盘故障
