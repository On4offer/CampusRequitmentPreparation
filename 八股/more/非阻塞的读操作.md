## 一、什么是“阻塞”？

**阻塞（Blocking）** 指的是一个事务或线程**因为资源被其他事务占用而不得不等待**，无法继续执行。

在数据库中最常见的阻塞情况是：

- 事务 A 正在写某行数据（加了排它锁）
- 事务 B 试图读取或写同一行
- 由于锁冲突，**事务 B 被挂起（阻塞）**，直到事务 A 提交或回滚释放锁

------

## 二、非阻塞的读操作：快照读（Snapshot Read）

**非阻塞读操作**指的是在读取数据时，不会等待其他事务释放锁。InnoDB 通过 **MVCC（多版本并发控制）** 实现了这一点。

### 特点：

- **不加锁、不等待**
- **读到的是事务一致性的快照版本**
- **即使目标数据正在被其他事务修改，也不会受影响**

### 示例：

```sql
-- 假设事务 A 正在更新某一行
START TRANSACTION;
UPDATE accounts SET balance = balance - 100 WHERE id = 1;

-- 事务 B 同时执行这个查询
SELECT * FROM accounts WHERE id = 1;  -- 读取旧版本，非阻塞！
```

此时事务 B **不会等待**事务 A 结束，它通过 **Undo Log** 读取那行数据的历史版本。

------

## 三、对比：阻塞 vs 非阻塞读

| 类型       | 是否加锁 | 是否等待其他事务 | 读取的是               | 用于语句类型                                |
| ---------- | -------- | ---------------- | ---------------------- | ------------------------------------------- |
| **快照读** | 否       | 否               | 一致性视图（历史版本） | `SELECT ...`                                |
| **当前读** | 是       | 是（可能阻塞）   | 当前最新版本           | `SELECT ... FOR UPDATE`、`UPDATE`、`DELETE` |

------

## 四、总结一句话：

> **阻塞** 是由于锁冲突导致的等待，而 **非阻塞读** 是 InnoDB 利用 MVCC 技术实现的“读旧版本不等人”，让读操作既快又安全。
