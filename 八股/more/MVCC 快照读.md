------

## ✅ 什么是 MVCC？

**MVCC（Multi-Version Concurrency Control，多版本并发控制）** 是 InnoDB 存储引擎中实现高并发读操作的核心机制。

> 它允许多个事务**并发读取**同一份数据，而不会互相阻塞。核心思想是：**读操作不加锁，而是读“过去某一时刻的数据快照”**。

------

## ✅ 什么是“快照读”？

快照读（Snapshot Read）是指：

> 在使用 **普通的 `SELECT` 语句**（不加锁）时，MySQL 返回的是该事务**开始时看到的数据版本**，而不是最新被其他事务修改后的版本。

这就是“**读快照，而不是最新值**”。

------

## 📚 举个例子（快照读 vs 当前读）

假设两个事务：

```text
T1: 开始事务，执行 SELECT * FROM account WHERE id = 1;

T2: 开始事务，UPDATE account SET balance = balance - 100 WHERE id = 1; COMMIT;
```

T1 是快照读：

- 它会读到事务开始前的 `balance` 值；
- 即使 T2 已经提交修改，T1 仍然读的是“旧版本”数据。

这避免了读阻塞，提升了性能。

------

## 🔧 MVCC 的实现机制（InnoDB）

InnoDB 使用 **隐藏字段 + Undo Log** 实现多版本：

每条记录背后隐含着两个字段：

- `trx_id`: 最近一次修改该行的事务 ID
- `roll_pointer`: 指向 undo log（存放旧版本数据）

在快照读时：

- MySQL 根据当前事务的 **Read View（可见性视图）**，
- 决定当前事务**能否看到某条记录的版本**。

------

## ✅ 哪些语句属于“快照读”？

| 语句                             | 是否快照读     |
| -------------------------------- | -------------- |
| `SELECT * FROM table WHERE ...;` | ✅ 是           |
| `SELECT ... LOCK IN SHARE MODE`  | ❌ 否（加锁读） |
| `SELECT ... FOR UPDATE`          | ❌ 否（加锁读） |
| `UPDATE / DELETE / INSERT`       | ❌ 否（当前读） |

------

## ✅ 快照读的优势

| 优点             | 描述                                           |
| ---------------- | ---------------------------------------------- |
| ✅ 非阻塞读       | 不加锁，不阻塞写操作                           |
| ✅ 高并发性能     | 支持读写并行                                   |
| ✅ 保证一致性     | 事务读到的数据版本一致，满足事务隔离要求       |
| ✅ 避免幻读（RR） | 在可重复读下可避免幻读（通过快照和间隙锁配合） |

------

## ❗ 快照读的限制（注意）

- 快照读只能用于 **事务隔离级别 ≥ Read Committed**；
- 快照读下读到的数据不一定是“最新提交的”，而是**事务开始时的数据视图**；
- 写操作仍然使用加锁的“当前读”机制，必须进行版本检查和加锁。

------

## 💬 面试回答模板

> 快照读是 InnoDB 实现 MVCC 的核心特性，指的是事务在执行普通 SELECT 查询时，不直接读取当前最新数据，而是读取事务开始时可见的数据快照。这样可以避免读写冲突，提高并发性能。快照读依赖隐藏的版本控制字段和 undo log 来实现，是 InnoDB 默认的读方式，在事务隔离级别如 RC 和 RR 下都有效。

------

## ✅ 总结一句话

> 快照读 = 不加锁的读，用“过去的数据副本”实现高并发 + 一致性，是 InnoDB 的核心能力之一。
