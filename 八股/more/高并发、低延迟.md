“**高并发**”与“**低延迟**”是现代系统（特别是互联网、大型分布式系统）中的两个关键性能指标，它们经常一起提，但本质不同：

------

## 一、什么是高并发（High Concurrency）？

### **定义**：

系统**能够同时处理大量请求/任务**的能力。

### **目标**：提高**吞吐量（TPS/QPS）**，不轻易崩溃或卡顿。

### **举例**：

- 淘宝双十一秒杀活动：每秒上百万用户访问下单。
- 微信聊天：几亿人同时在线发消息。

### **实现策略**：

| 技术手段         | 简要说明                       |
| ---------------- | ------------------------------ |
| 多线程 / 协程    | 并行处理多个任务               |
| 异步非阻塞       | 避免线程等待 I/O               |
| 线程池 / 连接池  | 控制资源数量，避免频繁创建销毁 |
| 负载均衡         | 多台服务器分摊请求             |
| 分布式架构       | 拆分系统，提高系统整体承载能力 |
| 缓存（如 Redis） | 缓解数据库压力，提升响应速度   |

------

## 二、什么是低延迟（Low Latency）？

### **定义**：

指系统从**接收到请求到返回响应的时间**非常短。

### **目标**：尽可能缩短**响应时间（Response Time）**。

### **举例**：

- 用户点击“下单”按钮，系统 100ms 内返回成功。
- 实时语音识别延迟 < 50ms 才不会让人感到卡顿。

### **实现策略**：

| 技术手段              | 简要说明                         |
| --------------------- | -------------------------------- |
| 缓存命中              | 内存级读写代替磁盘/数据库访问    |
| CDN / 静态资源加速    | 靠近用户部署节点，减少网络跳数   |
| 减少网络跳数/压缩数据 | 减少数据传输时间                 |
| 紧凑的数据格式        | 使用 Protobuf、MsgPack 替代 JSON |
| 服务间调用链优化      | 减少微服务层级调用或并行调用     |
| 数据预处理 / 预加载   | 提前准备好用户需要的数据         |

------

## 三、对比总结

| 属性 | 高并发                         | 低延迟                       |
| ---- | ------------------------------ | ---------------------------- |
| 指标 | 吞吐量（QPS / TPS）            | 响应时间（ms）               |
| 重点 | 系统能“扛住多大压力”           | 系统能“反应多快”             |
| 场景 | 秒杀、抢购、弹幕、爬虫、网关等 | 即时聊天、推荐、游戏、交易等 |
| 难点 | 控制资源、线程调度、限流、扩容 | 避免阻塞、网络优化、快速计算 |

------

## 四、系统设计中如何兼顾两者？

- **缓存优先**（如 Redis 缓存热点数据）
- **服务拆分**（垂直+水平）保持轻量
- **异步解耦**（消息队列 MQ）
- **限流 + 熔断 + 降级**（防止雪崩）
- **连接复用**（HTTP Keep-Alive，数据库连接池）

------

## 五、一句话总结：

> **高并发让系统“不垮”，低延迟让用户“爽快”。**
