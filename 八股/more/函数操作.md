你说得很对，这是**索引失效**中非常典型的一种情况。下面是对“函数操作导致索引失效”的详细解释与示例：

------

### **一、为什么函数操作会导致索引失效？**

索引基于“原始字段值”构建的是**有序结构（如 B+ 树）**。

函数会**改变字段的原始值**，使其与索引结构不匹配。

数据库无法“跳过”非命中的数据，只能**全表扫描并逐行运算对比**。

索引的本质是将列中的值进行排序后，建立一个结构（如 B+ 树），从而能快速定位到对应的记录。

**当你在 WHERE 子句中对字段使用函数时，数据库无法直接使用已有的索引值进行比对**，必须先对每一行的数据执行函数，再判断是否符合条件，这就变成了**全表扫描**，从而导致索引失效。

------

### **二、典型示例**

#### 示例 1：对字符串字段使用函数

```sql
-- 有索引：name
SELECT * FROM user WHERE UPPER(name) = 'TOM';
```

**失效原因**：`UPPER(name)` 需要对每一行执行函数操作，无法使用 `name` 上的索引。

#### 示例 2：对日期字段使用函数

```sql
-- 有索引：created_at
SELECT * FROM orders WHERE DATE(created_at) = '2023-05-01';
```

**失效原因**：`DATE(created_at)` 会把时间截断为日期部分，B+ 树中的排序结构无法命中原始时间字段。

------

### **三、优化方法**

为了避免索引失效，应该**尽量将函数操作挪到字段之外或使用等价的区间查询**。

#### 优化写法示例：

**写法错误（函数在字段上）**

```sql
SELECT * FROM orders WHERE DATE(created_at) = '2023-05-01';
```

**推荐写法（函数移到等值比较外）**

```sql
SELECT * FROM orders 
WHERE created_at >= '2023-05-01' AND created_at < '2023-05-02';
```

这样写能完整使用 `created_at` 上的索引，查询效率大幅提升。
