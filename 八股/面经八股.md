### 1. **[HTTPS](more/HTTPS)和[HTTP](more/HTTP)有哪些区别？**

**答案**：

| 特性         | **HTTP**                 | **HTTPS**                             |
| ------------ | ------------------------ | ------------------------------------- |
| **是否加密** | 不加密（明文传输）       | 加密（[SSL/TLS](more/SSL和TLS) 加密） |
| **端口**     | 80                       | 443                                   |
| **安全性**   | 不安全，容易被中间人攻击 | 安全，可防止劫持、监听、篡改          |
| **证书**     | 不需要                   | 需要 SSL 数字证书                     |
| **SEO 影响** | 无                       | 对搜索排名友好（Google 优先 HTTPS）   |
| **性能**     | 快，但不安全             | 稍慢（握手加密耗时），但现在几乎无感  |
| **适用场景** | 内网通信、测试环境       | 生产环境、涉及账号/支付/隐私的系统    |

**背景**：

- [**HTTP 和 HTTPS**](more/HTTP 和 HTTPS概念介绍) 都是用于在客户端（如浏览器）和服务器之间传输数据的协议。HTTP 是互联网通信的基础协议，用于浏览器访问网页等；然而，由于 HTTP 是明文传输，可能会有数据被第三方窃取或篡改的风险。
- **HTTPS** 在 **HTTP** 的基础上加入了加密机制，通过 [SSL/TLS](more/SSL和TLS) 协议对数据进行加密传输，因此比 HTTP 更安全。

**问题背景**：

- HTTPS 的出现是为了保护用户数据，尤其是保护隐私敏感信息，如银行账号、密码等。
- 使用 HTTPS 时，数据在传输过程中会进行加密，只有目标服务器可以解密，防止中间人攻击。

### 2. **[父子进程](more/父子进程)会共享一部分数据，那复制了什么？**

**答案**：

- 当[父进程](more/父进程)通过 [`fork()`](more/ `fork()` ) 创建子进程时，子进程会复制父进程的**内存空间**（如数据段、堆栈等），但操作系统通过 **[写时复制](more/写时复制)（Copy-on-Write, COW）** 技术，只在父进程或子进程对某一内存区域进行写操作时才进行实际复制，避免浪费内存和提高性能。

**背景**：

- 在操作系统中，[进程](more/进程)是一个正在执行的程序实例。[父子进程](more/父子进程)指的是通过系统调用 [`fork()`](more/fork（）概念)创建的子进程，子进程会复制父进程的某些资源。
- 但是，操作系统并不会完全复制父进程的所有资源，而是通过某些技术优化资源共享，确保系统性能。

**问题背景**：

- 理解父子进程之间的资源共享和复制机制对于深入理解操作系统中的进程管理至关重要。

### 3. **[Redis](more/Redis)和[MySQL](more/MySQL)的区别**

**答案**：

- **Redis** 是一个内存数据库，主要用于缓存，支持高性能的键值对存储，支持多种[数据类型](more/有哪些数据类型)（如字符串、哈希、列表等），适合[高并发、低延迟](more/高并发、低延迟)的[应用场景](more/有哪些实际场景)。
- **MySQL** 是[关系型数据库](more/关系型数据库)，使用 SQL 进行查询，适合结构化数据存储和复杂查询，支持[事务](more/事务)、[ACID 属性](more/ACID 属性)等特性，适用于对数据一致性要求较高的场景。

**背景**：

- **Redis** 和 **MySQL** 都是常见的数据存储系统，但它们的应用场景、数据存储方式和使用方式大不相同。
- **Redis** 是一个基于内存的键值数据库，适用于快速缓存和高并发数据操作；而 **MySQL** 是一个关系型数据库，适用于结构化数据的存储和复杂查询。

**问题背景**：

- Redis 作为 NoSQL 数据库，通常用于缓存、消息队列等实时性高的应用场景；MySQL 作为关系型数据库，适合用来存储事务性、结构化的数据。

### 4. **[HashMap](more/HashMap)、[HashTable](more/HashTable介绍)、[ConcurrentHashMap](more/ConcurrentHashMap)三者的共同点和区别**

**答案**：

- **共同点**：都使用[哈希表](more/哈希表)实现，提供通过[哈希值](more/哈希值)快速查找键值对的功能，都是无序的。
- **区别**：
  - **HashMap**：
    - 非线程安全，适合单线程或通过外部同步来保证线程安全的场景。
    - 支持 `null` 键和 `null` 值。
  - **HashTable**：
    - 线程安全，所有方法都进行了同步操作，因此性能较低，适用于较老的线程安全需求。整体锁（synchronized 锁整个表；过时，不推荐使用）
    - 不支持 `null` 键和 `null` 值。
  - **ConcurrentHashMap**：
    - 线程安全（分段锁或 [CAS](more/CAS)，性能高），采用[分段锁](more/分段锁)机制，通过[细粒度锁](more/细粒度锁)提高并发性能。高并发读写场景
    - 支持 `null` 键，但不支持 `null` 值。

**背景**：

- 在 Java 中，`HashMap`、`HashTable` 和 `ConcurrentHashMap` 都是 [`Map` 接口](more/`Map` 接口)的实现，主要用于存储键值对，[它们的底层](more/怎么实现)都是基于哈希表的结构。
- 它们之间的区别主要体现在线程安全、性能和空值支持等方面。

**问题背景**：

- 理解这些集合类的实现方式、优缺点和适用场景，特别是多线程环境中的使用方式，对提高代码性能和稳定性至关重要。

### 5. **HashMap的底层实现、扩容、哈希碰撞处理**

**答案**：（[hashmap和hashtable 的扩容机制](more/hashmap和hashtable 的扩容机制对比)不一样）

- **底层实现**：[HashMap](more/HashMap ) 的底层是一个数组，当发生哈希冲突时，冲突的元素会被存储在链表中。当链表的长度过长时，会优化为红黑树（当链表长度超过 8 时）。
- **扩容**：当元素个数超过容量的 0.75 时，会触发扩容，新的数组大小是原数组的两倍，并且需要重新计算每个元素的哈希位置。底层结构：数组 + 链表 + 红黑树（当链表长度 > 8 且数组大小 ≥ 64）**尾插法**（保持原顺序）
- **[哈希碰撞处理](more/哈希碰撞处理)**：当多个键计算出相同的哈希值时，会存储在相同位置，通常使用链表（或[红黑树](more/红黑树)）来存储冲突的元素，保证 O(log N) 或 O(1) 的查找效率。

**背景**：

- [`Map`](more/map) 是常见的数据结构之一，特别是在大规模数据处理时，性能优化非常关键。**[哈希表](more/HashTable)**是实现 `Map` 接口的基础数据结构，理解其底层实现、扩容机制和碰撞处理机制对于性能优化非常重要。

**问题背景**：

- 在实现大型分布式系统或高并发应用时，了解数据结构的底层实现有助于避免性能瓶颈。

### 6. **[B+树](more/B+树)和[B树](more/B树)的区别**

**答案**：

- **B树**：
  - 每个节点都包含数据，内部节点也存储**数据项（Key + Value）**，查找时会从根节点到叶节点进行。
  - 支持范围查询，但效率较低。
- **B+树**：
  - 所有数据（值）存储在叶子节点，内部节点仅存储键，起到索引作用。
  - 叶子节点通过指针相连，形成一个有序链表（通常是升序），支持高效的范围查询和按序遍历。（大多数数据库（如MySQL的InnoDB）中，叶子节点是**双向链表**（每个节点保存前驱和后继指针），便于向前或向后遍历。）

**背景**：

- [**B树** 和 **B+树**](more/B和B+树) 都是自平衡的多路查找树，广泛应用于数据库索引和文件系统中。它们能够在大数据集上提供高效的查找、插入和删除操作。

**问题背景**：

- 理解 B 树和 B+ 树的结构和性能特点对于数据库和文件系统的优化非常重要，特别是在高并发环境下，如何设计高效的索引结构非常关键。

### 7. **[Redis分布式锁](more/redis分布式锁)；[SetNX](more/SetNX)是针对单个资源的，如果有100个资源，怎么上锁？**

**答案**：

- Redis 分布式锁是一种通过 Redis 实现的分布式锁机制，通常用于保证在分布式环境下**对共享资源的互斥访问**。在分布式系统中，多个进程或服务器可能会同时访问某个资源，而分布式锁用于确保同一时间只有一个客户端能够访问共享资源，防止竞态条件和数据不一致性问题。
- 对于多个资源，可以为每个资源创建独立的锁，使用 `SetNX` 对每个资源进行上锁。对于多个资源的并发控制，可以使用 [**Redisson** 或 **Zookeeper**](more/Redisson 或 Zookeeper) 等工具，它们提供了更复杂的分布式锁管理功能，避免[死锁和资源竞争](more/死锁和资源竞争)。

**背景**：

- **Redis** 是一个高性能的内存数据库，广泛应用于缓存和分布式锁等场景。分布式锁在分布式系统中用于确保多个客户端之间的操作互斥
- `SetNX` 是 Redis 提供的一种原子操作，可以用于实现分布式锁，但当资源较多时，如何管理多个资源的锁就需要更复杂的策略。

**问题背景**：

- 分布式锁是解决分布式系统中资源争用问题的关键技术，了解如何高效地使用 Redis 实现分布式锁有助于提升[系统的并发性和稳定性](more/并发性和稳定性)。

### 9. **IOC是怎么实现的**

**答案**：

- 在实现IOC时，通常使用[依赖注入](more/依赖注入)（DI，Dependency Injection）这一技术。具体来说，IOC有以下几种实现方式：

- Spring 的 IOC 是通过“**解析 Bean 定义 + 构建 [BeanFactory](more/BeanFactory) + 执行依赖注入**”来实现的，依赖注入是它的核心机制。
  1. **构造函数注入（Constructor Injection）**： 通过构造函数将依赖关系注入到对象中，确保对象在创建时所依赖的资源或服务已经提供。
  
  1. **setter注入（Setter Injection）**： 通过公共的 setter 方法注入依赖，这种方式允许在对象创建后再注入依赖。
  
  1. **接口注入（Interface Injection）**： 依赖通过接口的方法注入，调用类实现特定接口来完成依赖注入。
  
     补充：**为什么不是字段注入？**因为字段注入（Field Injection）**不是标准 DI 模型的一部分**，而是 Spring 特有的简化手段。而接口注入是 DI 理论（如 Martin Fowler 提出）中的第三种经典形式。字段注入是 Spring 特有、语法层面的简化方式，不属于传统 DI 理论。
  
- IOC容器（如Spring框架中的[ApplicationContext](more/ApplicationContext)）会管理这些依赖关系，负责对象的创建和生命周期的管理。当一个对象依赖其他对象时，IOC容器会自动将这些依赖注入到对象中，而不是由对象本身创建。

**背景**：

- **[IOC（控制反转）](more/ IOC（控制反转）)** 是一种设计模式，它将对象的创建和管理交给框架，而不是由对象本身管理。[Spring 的核心特性](more/Spring 的核心特性)之一就是 IOC，它是实现依赖注入（DI）的基础，能有效解耦业务逻辑。

**问题背景**：

- 通过 IOC，Spring 可以控制对象的生命周期和依赖关系。理解 IOC 的实现方式可以帮助开发者理解 Spring 的核心原理。

### 10. **如果不用扫描，怎么发现相关的bean**

**答案**：

- **手动注册 [Bean](more/Bean)**：可以通过 `AnnotationConfigApplicationContext` 或 `GenericWebApplicationContext` 等类来[手动注册 Bean](more/手动注册Bean)。例如，使用 `context.register(Class<?>... annotatedClasses)` 方法注册 Bean。

- **Java 配置类**：通过 `@Configuration` 注解的配置类，显式地通过 `@Bean` 注解来注册 Bean。

  ```java
  @Configuration
  public class AppConfig {
      @Bean
      public MyBean myBean() {
          return new MyBean();
      }
  }
  ```

- **ApplicationContext**：通过 `ApplicationContext` 的 `registerBean` 方法手动注册 Bean。

**背景**：

- 在 Spring 中，通常会使用 **[类路径扫描](more/类路径扫描)**（通过注解或 XML 配置）来自动发现并注册 Bean。但如果不使用扫描，可以通过[显式的配置或手动注册 Bean](more/显式配置或手动注册) 来完成 Bean 的管理。

**问题背景**：

- 了解如何手动注册 Bean，有助于在不使用自动扫描的情况下灵活配置 Bean，特别是在某些特殊场景下，可能无法使用自动扫描。

### 11. **private 的对象可以自动注入吗**

**答案**：

- **可以自动注入**：即使字段是 `private` 的，Spring 依然可以通过反射机制进行注入。这是因为 **Spring 使用反射访问字段并进行赋值，而不受访问修饰符的限制。**
- **通过反射注入**：Spring 会**绕过 Java 的访问控制符（如 `private`），直接通过反射机制注入依赖。**可以通过 `@Autowired` 注解（字段注入）或者构造器注入等方式进行自动注入。

**背景**：

- 在 Spring 中，自动注入（Dependency Injection，DI）是通过[反射机制](more/反射机制)实现的。Spring 支持通过注解或 XML 配置将对象的依赖自动注入到目标对象中。

**问题背景**：

- 了解 Spring 是否能注入 `private` 成员，可以帮助我们理解 Spring 如何通过反射机制进行依赖注入，以及如何处理 Java 的访问修饰符。

### 12. **Redis：原来访问量1000w [QPS](more/QPS)，已经最高，现在2000w，怎么优化**

**答案**：

- **数据结构优化**：选择合适的数据结构，避免使用复杂的命令操作来提高效率，例如避免在高并发时频繁使用 `GET` 和 `SET`。
- **主从复制**：利用 Redis 的 **[主从复制](more/主从复制)** 提升读取性能，多个从节点可以分担读取请求，减少主节点的压力。
- **分片**：使用 Redis [**分片**（Cluster）](more/分片)将数据分布到多个节点，减少单个节点的负载，提升系统扩展性。
- **使用 Pipeline**（管道）：通过 Redis 的 [Pipeline 技术](more/Pipeline 技术)将多个命令打包发送，减少网络往返次数，从而提高性能。
- **持久化优化**：调整 [Redis 的持久化策略](more/Redis 的持久化策略)（如 [RDB](more/RDB ) 和 [AOF](more/AOF)），根据需求选择合适的持久化策略来降低持久化的性能开销。

**背景**：

- **Redis** 是一个高性能的键值对存储系统，广泛应用于[缓存、消息队列](more/缓存和消息队列)等场景。当 [Redis 的访问量达到高峰时](more/Redis 的访问量达到高峰)，如何优化其性能成为一个重要问题。

**问题背景**：

- 在高并发场景下，Redis 的性能瓶颈可能出现在内存、CPU、网络带宽、IO 等多个方面。优化 Redis 的性能不仅需要调整 Redis 本身的配置，还需要优化应用层的访问模式。

### 13. **除了[读写分离](more/读写分离)（Redis的[主从复制](more/主从复制)可以被称为读写分离）呢？（可以加资源）**

**答案**：

- **水平扩展**：通过 **Redis Cluster** 进行水平扩展，将数据分布到多个 Redis 节点上，分摊负载，提升性能。
- **增加缓存层**：利用 Redis 作为一级缓存，在数据库之前加一层缓存，避免数据库的高负载。
- **使用 Redis 的多种数据类型**：选择合适的数据结构和操作方式来优化性能，比如使用 `Bitmap`、`HyperLogLog` 等。
- **异步操作**：尽量减少同步操作的使用，采用 **异步** 方式进行数据更新或读取，避免阻塞。

**背景**：

- 在分布式系统中，读写分离是一种常见的优化策略，通过将读取操作分配给从节点、写操作分配给主节点，从而提升系统的吞吐量和响应速度。

**问题背景**：

- 了解如何在 Redis 高并发场景下通过其他手段来优化性能，有助于设计更高效的系统架构。

14.**数组和链表的区别，体现在内存读取和 CPU 计算上**

**答案**：

- **数组**：
  - **内存布局**：数组存储在连续的内存空间中，支持快速随机访问（O(1) 时间复杂度）。
  - **CPU计算**：由于内存是连续的，CPU 可以高效地加载数据，尤其是对于大的数组，CPU 缓存命中率较高，访问速度更快。
  - **缺点**：数组大小固定，插入和删除操作需要移动大量元素，时间复杂度为 O(n)。
- **链表**：
  - **内存布局**：链表的元素不一定在连续的内存空间中，节点间通过指针链接。
  - **CPU计算**：由于链表节点分散存储，CPU 缓存命中率较低，访问速度慢。每次访问需要跳转到下一个节点，增加了访问的时间。
  - **优点**：动态扩展，插入和删除操作非常高效（O(1) 时间复杂度），特别是对于中间位置的插入和删除。

**背景**：

- **数组** 和 **链表** 都是常见的数据结构，各自有不同的优势。数组通过连续内存位置存储数据，而链表则通过节点连接的方式存储数据。

**问题背景**：

- 理解数组和链表在内存布局、访问速度、CPU 计算等方面的区别，能够帮助开发者在设计算法时做出更合适的数据结构选择。

### 15. **一个巨大的列表 L 有非常多 [URLs](more/URLs)，然后一大堆请求 URL 来判断列表中是否存在请求 URL 的前缀，怎么做？**

**答案**：

- **哈希表优化**：[通过构建哈希表存储 URL 的前缀](more/通过哈希表存储URL前缀)，能够在 O(1) 时间内进行快速查找。

**背景**：

- 在高性能的场景下，如何高效判断某个请求 URL 是否在一个巨大的列表中存在前缀，直接使用列表进行匹配的方式可能效率较低。
- 这是一个**高性能字符串前缀匹配问题**，适用于大规模数据、频繁查询的场景。你可以使用**高效的数据结构和算法**来实现快速前缀判断

**问题背景**：

- 判断 URL 是否存在前缀，涉及字符串的匹配，传统的线性查找效率较低。考虑使用数据结构或算法来加速这种匹配操作。

### 20. **[Bean](more/ Bean)的作用域有哪些？[BeanFactory](more/BeanFactory)和[FactoryBean](more/FactoryBean)有什么区别？**

**答案**：

- **Bean的作用域**：
  1. **[singleton](more/singleton)**（默认）：单例模式，Spring 容器中只会有一个 Bean 实例。
  2. **[prototype](more/prototype)**：每次请求都会创建一个新的 Bean 实例。
  3. **request**：每个 HTTP 请求都会创建一个新的 Bean 实例，仅在 Web 环境中有效。（[request和prototype的区别](more/request和prototype的区别)）
  4. **session**：每个 HTTP 会话会创建一个新的 Bean 实例，仅在 Web 环境中有效。（[HTTP会话和HTTP请求的区别](more/HTTP会话和HTTP请求的区别)）
  5. **application**：在整个 Web 应用中共享一个 Bean 实例，仅在 Web 环境中有效。（[application 和 singleton的区别](more/application 和 singleton的区别)）
- **BeanFactory 和 FactoryBean 的区别**：
  - **[BeanFactory](more/BeanFactory)**：是 **Spring 最基本的容器接口**，负责管理和创建 Bean。它是所有容器的基础接口。
  - **[FactoryBean](more/FactoryBean)**：是一个接口，用于**提供自定义的 Bean 创建逻辑**。当我们需要创建一些复杂的 Bean，或者通过 FactoryBean 动态创建 Bean 时，使用该接口。`FactoryBean` 的实现类通常会返回一个对象，而不是一个简单的 Bean。

**背景**：

- Spring 提供了不同的 Bean 作用域来控制 Bean 的生命周期。作用域决定了 Bean 实例的创建时机和销毁时机。`BeanFactory` 和 `FactoryBean` 都是 Spring 中用于创建和管理 Bean 的接口，但它们的作用不同。

**问题背景**：

- 了解 Bean 的作用域及其管理机制，有助于更好地控制应用中的资源和性能。同时，理解 `BeanFactory` 和 `FactoryBean` 的区别，也有助于更加灵活地创建和管理 Bean。

### 23. **Spring事务中有哪些[事务传播行为](more/事务传播行为)？**

**答案**：

- **PROPAGATION_REQUIRED**：如果当前存在事务，则加入该事务，否则新建一个事务。默认的事务传播行为，也是使用最广泛的一种。
- **PROPAGATION_SUPPORTS**：如果当前存在事务，则加入该事务，否则以非事务方式执行。
- **PROPAGATION_MANDATORY**：必须存在事务，如果当前没有事务，则抛出异常。
- **PROPAGATION_REQUIRES_NEW**：无论当前是否存在事务，都会新建一个事务。
- **PROPAGATION_NOT_SUPPORTED**：以非事务方式执行，如果当前存在事务，则挂起当前事务。
- **PROPAGATION_NEVER**：以非事务方式执行，如果当前存在事务，则抛出异常。
- **PROPAGATION_NESTED**：如果当前存在事务，则开启一个嵌套事务，否则和 `PROPAGATION_REQUIRED` 一样新建一个事务。

**背景**：

- 事务传播行为是指当多个事务相互嵌套时，外部事务如何与内部事务交互的规则。Spring 提供了多种事务传播行为，适用于不同的场景。

**问题背景**：

- 了解 Spring 提供的事务传播行为，可以帮助开发者根据业务需求选择合适的事务策略。

### 24. **[＠Transactional](more/＠Transactional)的实现原理？**

**答案**：

- **代理机制**：`@Transactional` 是基于 Spring AOP 实现的。Spring 会为带有 `@Transactional` 注解的类或方法创建[代理对象](more/代理对象)，在方法执行前开启事务，在方法执行后提交事务，出错时回滚事务。
- **事务管理器**：Spring 使用 [`PlatformTransactionManager`](more/ `PlatformTransactionManager`) 管理事务，`@Transactional` 会通过它来控制事务的生命周期。

**背景**：

- `@Transactional` 注解用于声明事务管理，它能够简化事务管理的代码，自动处理事务的提交、回滚等操作。

**问题背景**：

- 理解 `@Transactional` 的实现原理可以帮助开发者更加高效地使用 Spring 的事务管理，避免事务管理上的问题。

### 28. **[HashSet](more/HashSet)底层原理**

**答案**：

- **底层数据结构**：`HashSet` 底层是基于 [`HashMap`](more/HashMap) 实现的。它使用一个 `HashMap` 来存储数据，`HashSet` 的每个元素作为 `HashMap` 的 key，value 为一个固定的常量（如 `Boolean.TRUE`）。
- **哈希存储**：`HashSet` 使用哈希表来存储元素，元素的哈希值决定了它存储在哈希表的哪个位置。当插入新元素时，`HashSet` 会计算元素的哈希值，并尝试将其插入到相应的桶中。如果桶中已经有元素，则使用链表或红黑树（在哈希冲突较严重时）来解决哈希冲突。
- **查找性能**：由于使用哈希表，`HashSet` 提供了 O(1) 的平均时间复杂度来检查元素是否已存在。

**背景**：

- **HashSet** 是 Java 中的一个集合类，它实现了 `Set` 接口，表示一个不包含重复元素的集合。它背后依赖于 `HashMap`，因此理解 `HashSet` 的底层实现需要了解 `HashMap` 的工作原理。

**问题背景**：

- HashSet 是通过哈希表来实现集合元素的存储和查找。它的核心目标是去重，通过哈希值快速判断元素是否存在于集合中。

### 29. **[JVM](more/JVM)调优命令、调优参数，平时用什么调优**

**答案**：

- **常见的 JVM 调优参数**：
  1. **[堆内存配置](more/堆内存配置)**：
     - `-Xms`：设置 JVM 启动时堆内存的初始大小。
     - `-Xmx`：设置堆内存的最大值。
  2. **[垃圾回收配置](more/垃圾回收配置)**：
     - `-XX:+UseG1GC`：使用 G1 垃圾回收器。
     - `-XX:+UseConcMarkSweepGC`：使用 CMS（并发标记清除） 垃圾回收器。
     - `-XX:MaxGCPauseMillis`：设置最大 GC 暂停时间。
  3. **JVM 输出日志**：
     - `-XX:+PrintGCDetails`：打印详细的 GC 日志。
     - `-XX:+PrintGCDateStamps`：打印 GC 时间戳。
  4. **JVM 性能分析**：
     - `-Xloggc:gc.log`：将 GC 日志输出到指定文件。
  5. **JVM 线程配置**：
     - `-XX:ThreadStackSize`：设置线程栈大小。
- **平时用的调优**：
  - 一般情况下，最常见的调优是调整堆内存大小（`-Xms` 和 `-Xmx`）以及选择合适的垃圾回收器（如 G1 GC 或 CMS）。在高并发场景下，可以通过调整 JVM 的线程池、GC 的参数来优化性能。

**背景**：

- **JVM（Java Virtual Machine）** 是 Java 程序的运行环境，JVM 的性能直接影响到 Java 程序的执行效率。JVM 提供了很多参数用于调优，它们可以用来优化堆内存、垃圾回收（GC）、线程等方面的性能。

**问题背景**：

- JVM 调优是提高 Java 应用性能的关键步骤，尤其是当程序面对大量数据和高并发时，合适的 JVM 参数可以显著提升应用的吞吐量和响应时间。

### 30. **[AOP](more/AOP)事务实现方式？**

**答案**：

- **[声明式事务](more/声明式事务)**：
  - 使用 [`@Transactional`](more/@Transactional) 注解来定义事务的边界，Spring 会在方法执行前开启事务，执行完毕后提交事务，发生异常时回滚事务。
  - Spring 利用 AOP 的[代理机制](more/代理机制)，在方法调用时插入事务管理逻辑。
  - 适合绝大多数情况，因为它简化了事务管理，不需要手动控制事务的开启和提交。
  - 基于 XML 配置的声明式事务：使用 `<tx:advice>` 配合 `<aop:config>` 配置事务逻辑；适用于老项目或不方便使用注解的场景
- **[编程式事务](more/编程式事务)**：
  - 开发者手动使用 `PlatformTransactionManager` 接口来控制事务的开始、提交和回滚。
  - 适用于需要复杂事务控制的场景，如事务的条件判断、动态事务管理等。

**背景**：

- **AOP（面向切面编程）** 是一种编程范式，用于分离横切关注点（如日志记录、事务处理等）。在 Spring 中，事务管理常常通过 AOP 来实现，使得事务的管理和业务逻辑解耦。

**问题背景**：

- Spring 提供了两种主要的事务管理方式：声明式事务和编程式事务。声明式事务主要通过 AOP 实现，通过注解或 XML 配置来简化事务的管理。

### 31. **[事务的过期](more/事务过期)淘汰策略**有哪些

**答案**：

- **事务过期处理**：
  - 一般情况下，事务的超时设置可以通过配置来控制。例如，`@Transactional` 注解的 **[`timeout` 参数](more/`timeout` 参数)**可以设置事务的超时时间。
  - 锁等待超时，事务申请锁的时候等待时间超过设定值，就会放弃申请，可能回滚事务
  - 如果事务未在指定时间内完成，**Spring 会自动回滚事务**，并且事务将被视为“过期”。
  - 在分布式系统中，事务超时可能会导致部分事务没有提交，因此需要使用事务管理系统（如分布式事务框架）来处理事务超时和失败。
  - 常见的超时策略有：**自动回滚**、**强制回滚**（在设置的时间限制内未提交的事务会被回滚）。

**背景**：

- 在一些分布式系统或高并发环境下，事务的过期淘汰是为了防止系统中有过多未完成的事务，导致资源被占用，进而影响系统的性能。

**问题背景**：

- 事务过期指的是事务在一定时间内未能完成或未被提交，这时需要采取措施来处理这些过期事务。

### 32. **什么时候[事务](more/事务)失效（[@Transactional](more/@Transactional) 会失效）**

**答案**：

- **事务失效的原因**：
  1. **`private`/`final`/`static` 修饰的方法**：如果事务方法是 `private`/`final`/`static` 的，Spring 无法通过代理来拦截这个方法，因为代理只能作用于 public 方法。**失效原因**：无法生成[代理对象](more/代理对象)。只有 `public` 实例方法，且是通过 Spring 容器管理的代理对象调用时，`@Transactional` 才能生效。
  2. **[事务方法调用自身失效](more/事务方法调用自身失效)（this调用）（同一类中的方法）**：如果方法 A 调用类中的方法 B，且方法 B 被 `@Transactional` 注解标注，被A调用的B的事务无法生效，因为 Spring AOP 只能拦截外部调用，如果是类内部调用，事务不会被应用。
  3. **[事务的传播行为](more/事务的传播行为)**：如果一个方法内已经有一个事务存在，而另一个方法被调用时具有不同的事务传播行为（例如 `PROPAGATION_REQUIRES_NEW`），那么事务可能会失效。
  4. **异常被捕获了，没有抛出**：Spring 就无法感知这个异常，因此不会触发[事务回滚](more/事务回滚)
  5. **未被 Spring 管理的类或对象调用事务方法：**只有被 Spring 容器管理的 Bean（即加了 @Component、@Service 等注解并被扫描注册）才能享受到 Spring 的事务增强。Spring 会为带有 `@Transactional` 的 Bean 生成一个代理对象；如果一个类**不是 Spring 托管的 Bean**，那么 Spring 根本不会对它做 AOP 处理，也就不会织入事务逻辑；调用的是**普通对象**，而不是[代理对象](more/代理对象)，事务自然无效。
  6. **方法执行成功但未提交（没有配置正确的[事务传播行为](more/事务传播行为)）：**通常出现在**多层嵌套事务调用**的情况下，属于**[事务传播机制配置不当](more/事务传播机制配置不当)**所引起的问题。**传播行为配置不当，可能导致方法 B 根本不参与事务、或提前提交、或在异常后仍被提交**。表面看“执行成功”，但实际上事务逻辑没有完整封装，从而导致事务“失效”。

**背景**：

- 事务失效的原因可能是由于方法调用的上下文发生了变化或事务的代理被绕过。Spring AOP 是基于代理模式的，如果事务的控制方法是 `private` 或者在同一类中调用，可能会导致事务失效。

**问题背景**：

- 事务管理通常依赖 AOP 代理，在 Spring 中，事务是通过动态代理来实现的。因此，如果事务方法是 `private`，或者是内部方法调用，会绕过代理，导致事务失效。

### 33. **[索引](more/索引)什么时候会失效**

| 编号 | 原因                                                  | 说明 / 示例                                                  |
| ---- | ----------------------------------------------------- | ------------------------------------------------------------ |
| ①    | **不满足最左前缀原则**                                | 联合索引 `(a, b)`，只查 `b` 无法命中                         |
| ②    | **字段运算或函数包裹**                                | `WHERE DATE(create_time) = '2024-01-01'`，索引失效           |
| ③    | **隐式类型转换**                                      | `WHERE phone = 12345678901`，字段为字符串，右侧是数字，会导致全表扫描 |
| ④    | **使用 `%xxx` 模糊匹配开头**                          | `WHERE name LIKE '%abc'`，无法用到索引（不是前缀匹配）       |
| ⑤    | **OR 条件中部分字段无索引**                           | `WHERE a = 1 OR b = 2`，a 有索引，b 没有，也失效             |
| ⑥    | **对索引字段使用 `IS NULL`、`IS NOT NULL`**（视情况） | 有时优化器判断为全表扫描                                     |
| ⑦    | **数据分布极度不均衡**                                | 索引选择度太差，优化器认为索引比全表扫描更慢                 |
| ⑧    | **使用 `!=`、`<>`、`NOT IN`、`NOT LIKE` 等否定条件**  | 无法使用索引快速定位，索引失效                               |
| ⑨    | **排序字段不符合索引顺序**                            | 有索引，但 `ORDER BY` 不按顺序或混合排序方向，会失效         |
| ⑩    | **MySQL 判断索引不如全表快**                          | 即使有索引，但小表+选择度差，优化器自动放弃使用索引          |

💬 面试标准答题模板

> 索引失效是指查询未使用预期的索引，通常由多种原因引起，包括违反最左前缀原则、字段运算、类型不一致导致的隐式转换、使用不等号或模糊匹配、数据分布不均等。要避免索引失效，需要从字段设计、SQL 写法、索引选择度等方面综合优化，必要时可结合 [`EXPLAIN`](more/EXPLAIN) 分析执行计划。

✅ 一句话总结

> **索引不是一定生效**，写对 SQL + 设计好字段 + 避免踩雷，才能真正让索引起作用。

### 38. **[三次握手与四次挥手](more/三次握手与四次挥手)**

**背景**：网络基础题，重点考察 TCP 建立连接的流程。TCP 中常见[标志位](more/标志位)（如 SYN、ACK、FIN 等）

**📦 三次握手过程**

1️⃣ **客户端发送 SYN （Synchronize同步序列号）包（同步请求）**

- 表示：客户端希望建立连接；
- 包含：SYN 标志位、初始序列号 `seq（Sequence Number序列号） = x`。

2️⃣ **服务器返回 SYN + ACK（Acknowledgment确认） 包**

- 表示：服务器接受请求，准备好连接；
- 包含：`SYN=1`：回复同步请求；`ACK=1`：确认客户端的 `SYN`；`seq = y`，`ack（Acknowledgment Number确认号） = x + 1`。

3️⃣ **客户端发送 ACK 包**

- 表示：客户端确认收到服务器的响应；
- 包含：`ACK=1`，`ack = y + 1`。

> 💡 此时连接建立完成，双方可以开始传输数据。

**📦 四次挥手过程**

1️⃣ **客户端发送 FIN（Finish完成） 包**

- 表示：客户端不再发送数据了，但还能接收；
- 进入 **FIN_WAIT_1** 状态。

2️⃣ **服务器回复 ACK 包**

- 表示：收到客户端关闭请求；
- 客户端进入 **FIN_WAIT_2**；
- 服务器仍可能继续发送剩余数据。

3️⃣ **服务器发送 FIN 包**

- 表示：服务器数据发完，准备断开连接；
- 进入 **LAST_ACK** 状态。

4️⃣ **客户端发送 ACK 包**

- 确认服务器关闭请求，进入 **TIME_WAIT** 状态；
- 2MSL（Maximum Segment Lifetime最大报文段寿命） 后彻底关闭连接。

### **46. [HashMap 的扩容机制](more/HashMap 的扩容机制)介绍一下**

### **47. MySQL 中的 [MVCC](more/MVCC ) 是怎么实现的？**

  - **READ COMMITTED**：每次查询生成一个快照；

### **50. [B+ 树](more/B+树)的特点**

**背景**：B+树是 MySQL 索引结构的核心。

**答案**：

- 所有数据都存储在**叶子节点**；
- 叶子节点**按链表有序连接**，支持范围查询；
- 内部节点只存储索引键，不存数据；
- 比二叉树高效、层级浅，适合磁盘 I/O。

### **53. [锁升级的过程](more/锁升级的过程)了解过吗？**

**背景**：考查你对 JVM 锁优化的理解。

**答案**：

Java 对象锁的升级路径：

1. **无锁**
2. → **[偏向锁](more/偏向锁)**：只有一个线程访问，加锁极快；
3. → **[轻量级锁](more/轻量级锁)**：多个线程竞争时，使用自旋方式尝试获取；
4. → **[重量级锁](more/重量级锁)**：自旋失败，进入阻塞等待。

锁是**逐级升级、不可降级**的。

### **55. Spring 如何实现[自动装配](more/自动装配)？**

**背景**：自动装配是 Spring 实现依赖注入（DI）的一种方式，容器会根据类型或名称自动将所需的 Bean 注入目标对象，简化了配置、降低了耦合。是 [Spring 核心能力](more/Spring 核心能力)。

### 56. 怎么在Java中操作redis

**1. 使用 Jedis（老牌 Redis 客户端）**

- `Jedis` 是 Redis 官方推荐的早期 Java 客户端，使用简单、入门方便。
- 它是**非线程安全的**，所以通常在连接池中使用。
- 使用场景：小型项目或非并发敏感的环境。

```java
Jedis jedis = new Jedis("localhost", 6379);
jedis.set("name", "Alice");
String name = jedis.get("name");
jedis.close();
```

------

**2. 使用 Lettuce（Spring Boot 默认集成）**

- `Lettuce` 是基于 Netty 的异步、线程安全的客户端，支持响应式编程。
- 在 Spring Boot 中是默认 Redis 客户端，性能更好，适合高并发场景。
- 支持同步、异步和反应式 [API](more/API)。

```java
RedisClient client = RedisClient.create("redis://localhost:6379");
RedisCommands<String, String> commands = client.connect().sync();
commands.set("user", "Bob");
String user = commands.get("user");
```

------

**3. 使用 Spring Data Redis（企业项目首选）**

- Spring Boot 中集成了 `spring-boot-starter-data-redis`，提供了 `RedisTemplate` 和 `StringRedisTemplate` 两个封装类。
- 支持丰富的数据结构（List、Hash、Set 等），还能结合缓存注解、事务等高级特性。
- 推荐使用 `StringRedisTemplate` 处理字符串操作。

```java
@Autowired
private StringRedisTemplate redisTemplate;

redisTemplate.opsForValue().set("key", "value");
String val = redisTemplate.opsForValue().get("key");
```

------

**补充说明：**

- 我们一般在生产环境中选用 **Lettuce + Spring Data Redis**，这样既线程安全又易于与 Spring 体系集成。
- 项目中我还使用 Redis 做过缓存、分布式锁、延迟队列等功能。

------

总结：

> Java 操作 Redis 的方式主要有 Jedis、Lettuce、Spring Data Redis，建议根据项目规模与并发需求选择合适的方案。在实际开发中，我倾向使用 Spring Redis + Lettuce，配合注解和缓存机制更高效、可维护性更强。
