# 0）一页速查表

| 池类型      | 由谁创建         | 关键类/组件                                                  | 典型场景                            | 要点/坑位                                                    |
| ----------- | ---------------- | ------------------------------------------------------------ | ----------------------------------- | ------------------------------------------------------------ |
| 线程池      | 业务代码或框架   | `ThreadPoolExecutor` / `ScheduledThreadPoolExecutor` / `ForkJoinPool` | Web请求、异步任务、定时任务、批处理 | **避免**`Executors.*`默认无界队列；设置**有界队列**、命名线程、拒绝策略；记得`shutdown()` |
| DB连接池    | 框架或你通过配置 | **[HikariCP](more/HikariCP)**（推荐）、Druid                 | 关系型数据库访问                    | 连接泄漏检测、超时与最大连接数匹配数据库上限                 |
| Redis连接池 | 客户端库         | Jedis：`JedisPool`；Lettuce：Netty事件循环                   | 高并发Redis访问                     | Jedis用池；Lettuce通常**不需要**池（异步/多路复用）          |
| HTTP连接池  | HTTP客户端       | OkHttp `ConnectionPool`、Apache HttpClient Pooling           | 调用外部HTTP服务                    | 连接复用、超时与最大并发；DNS/Keep-Alive配置                 |
| 通用对象池  | 你或中间件       | Apache **commons-pool2**                                     | 代价昂贵/不可频繁创建的对象         | 谨慎使用：容易状态脏、复杂度高；现代GC下仅在昂贵对象上收益明显 |
| 内存/缓冲池 | 框架/你          | Netty `PooledByteBufAllocator`、Direct `ByteBuffer`池        | 高频网络IO、序列化/反序列化         | 避免频繁分配回收；注意**引用释放**，否则内存泄漏             |
| 常量池      | 编译器/JVM       | **Class文件常量池**、**运行时常量池**、**String Pool**       | 字面量、符号引用、字符串驻留        | `String#intern()`节省内存但可能放大元数据占用；理解`==`与`equals()`差异 |

# 1）线程池（最常考）

## 1.1 谁创建？

- **你在业务代码中手动创建**（推荐）
- 或由**框架**（如Tomcat/Netty/Spring）创建并注入使用

## 1.2 关键代码（推荐写法）

```java
BlockingQueue<Runnable> queue = new ArrayBlockingQueue<>(1000);
ThreadFactory tf = r -> {
    Thread t = new Thread(r);
    t.setName("biz-pool-" + t.getId());
    t.setDaemon(false);
    return t;
};

RejectedExecutionHandler reject = new ThreadPoolExecutor.CallerRunsPolicy(); 
// 或者 DiscardPolicy / AbortPolicy / DiscardOldestPolicy，按需选择

ThreadPoolExecutor pool = new ThreadPoolExecutor(
    8,                 // corePoolSize
    16,                // maximumPoolSize
    60, TimeUnit.SECONDS, // keepAliveTime
    queue,
    tf,
    reject
);

// 提交任务
Future<Integer> f = pool.submit(() -> {
    // 任务逻辑
    return 42;
});

// 关闭
pool.shutdown();
```

**队列选择：**

- `ArrayBlockingQueue`（有界）：可控背压（常用）
- `LinkedBlockingQueue`（建议自定容量）：默认无界，易OOM
- `SynchronousQueue`：无缓冲，适合突发但可扩容的场景（配合较大`maxPoolSize`）

**规模估算：**

- CPU密集：≈ `CPU核数 + 1`
- IO密集：≈ `CPU核数 * (1 + IO耗时/计算耗时)`，通常可大于CPU核数

## 1.3 使用场景

- Web请求处理、异步下单/发券、日志落盘、定时任务（`ScheduledThreadPoolExecutor`）
- 批量并行（`invokeAll`）/分治计算（`ForkJoinPool`）
- `CompletableFuture` 默认使用 `ForkJoinPool.commonPool()`，需要隔离时显式指定自定义池

## 1.4 易错点

- 直接用 `Executors.newFixedThreadPool()`（默认无界队列）→ **风险：堆积/OOM**
- 忘记关闭线程池；没有命名线程导致排障困难
- 小心阻塞操作（DB/HTTP）把线程占满，应设置超时与限流

# 2）连接池

## 2.1 数据库连接池（HikariCP）

**谁创建**：通常由Spring Boot自动配置，也可手动。

```java
HikariConfig cfg = new HikariConfig();
cfg.setJdbcUrl("jdbc:mysql://host:3306/db?characterEncoding=utf8");
cfg.setUsername("user");
cfg.setPassword("pwd");
cfg.setMaximumPoolSize(20);
cfg.setMinimumIdle(5);
cfg.setConnectionTimeout(3000);
cfg.setIdleTimeout(600000);

HikariDataSource ds = new HikariDataSource(cfg);

// 使用
try (Connection conn = ds.getConnection();
     PreparedStatement ps = conn.prepareStatement("SELECT 1")) {
    ps.execute();
} // try-with-resources 自动归还连接
```

**场景与要点**：高并发DB访问；合理设置最大连接数与数据库上限匹配；开启泄漏检测（`leakDetectionThreshold`）。

## 2.2 Redis连接池

- **Jedis**（阻塞IO）：使用 `JedisPool`

```java
GenericObjectPoolConfig<?> pc = new GenericObjectPoolConfig<>();
pc.setMaxTotal(50);
JedisPool pool = new JedisPool(pc, "redis-host", 6379);
try (Jedis jedis = pool.getResource()) {
    jedis.set("k", "v");
}
```

- **Lettuce**（Netty异步、多路复用）：**通常不需要**连接池；用单连接/连接共享即可（除非大量阻塞命令）。

## 2.3 HTTP连接池

- **OkHttp**

```java
ConnectionPool cp = new ConnectionPool(100, 5, TimeUnit.MINUTES);
OkHttpClient client = new OkHttpClient.Builder()
    .connectionPool(cp)
    .connectTimeout(2, TimeUnit.SECONDS)
    .readTimeout(3, TimeUnit.SECONDS)
    .build();
```

要点：复用长连接、设置合理超时与最大空闲连接。

# 3）通用对象池（commons-pool2）

**何时使用**：构造/销毁成本极高、且对象可安全复用（且有限）的场景。
 **不建议**：普通POJO/轻量对象（现代GC足够快，池化反而复杂且易出错）。

```java
class ExpensiveObject {
    void reset() { /* 清理状态 */ }
}

class ExpensiveFactory extends BasePooledObjectFactory<ExpensiveObject> {
    @Override public ExpensiveObject create() { return new ExpensiveObject(); }
    @Override public PooledObject<ExpensiveObject> wrap(ExpensiveObject obj) { return new DefaultPooledObject<>(obj); }
    @Override public void passivateObject(PooledObject<ExpensiveObject> p) { p.getObject().reset(); }
}

GenericObjectPool<ExpensiveObject> pool =
    new GenericObjectPool<>(new ExpensiveFactory());
pool.setMaxTotal(20);

try (ExpensiveObject obj = pool.borrowObject()) {
    // 使用obj
} finally {
    // 归还：commons-pool通常是显式 returnObject(obj)，
    // 若想try-with-resources，需要自行封装AutoCloseable代理
}
```

**风险**：状态未清理→脏数据；借还不平衡→枯竭；并发下活跃检查/驱逐策略要配置好。

# 4）内存/缓冲池

## 4.1 Netty缓冲池

高频网络IO与序列化适合池化 `ByteBuf`，减少堆外/堆内分配开销。

```java
ByteBufAllocator alloc = PooledByteBufAllocator.DEFAULT;
ByteBuf buf = alloc.buffer(1024); // 申请
try {
    // 写读字节
} finally {
    buf.release(); // 必须释放，避免泄漏
}
```

要点：**手动释放**引用计数；诊断内存泄漏可开启`-Dio.netty.leakDetection.level=paranoid`。

## 4.2 `ByteBuffer`池（自建或第三方）

适合高频DirectBuffer操作；注意复用时清空状态与线程安全。

# 5）常量池（JVM层面）

## 5.1 分类

- **Class文件常量池**：字面量与符号引用保存在`.class`内部。
- **运行时常量池**：类加载后进入JVM（元空间/堆等），供解析/链接使用。
- **String常量池（String Pool）**：驻留（interned）字符串的共享区域（JDK7起在**堆**中）。

## 5.2 关键点与代码

- 编译期常量（`final static` 且可折叠）直接进入常量池。
- `String#intern()` 手动驻留，可减少重复字符串内存，但对**大量动态值**不建议滥用。

```java
String a = "abc";
String b = "ab" + "c";  // 编译期折叠
System.out.println(a == b); // true

String x = new String("abc");
System.out.println(a == x); // false
System.out.println(a == x.intern()); // true
```

**面试易错**：`==`比较引用，`equals()`比较内容；JDK版本差异（String Pool位置）。

# 6）线程本地“池化”（ThreadLocal复用）

适合**线程私有、可复用且非线程安全**的对象（如`SimpleDateFormat`），减少创建开销与锁竞争。
 注意**容器环境（线程池复用）下的泄漏风险**，请求结束要清理。

```java
private static final ThreadLocal<DateFormat> SDF =
    ThreadLocal.withInitial(() -> new SimpleDateFormat("yyyy-MM-dd"));

String s = SDF.get().format(new Date());
// 请求完成后（如Web过滤器）：
SDF.remove();
```

# 7）如何回答“什么时候用池化？”

**判定四问：**

1. 创建/销毁是否昂贵？（昂贵→考虑池化）
2. 资源是否稀缺/受配额限制？（DB连接/Socket→是）
3. 对象是否可安全复用且易于重置状态？（是→考虑）
4. 不池化是否会导致吞吐/延迟明显劣化？（是→考虑）

**不建议池化的情况**：对象很轻、生命周期短、状态复杂难清理、会增加Bug概率。

------

# 8）面试标准回答（可背）

> **池化的本质**是用“少量可复用实例”服务“多次请求”，以摊薄创建销毁成本、控制资源上限并提高吞吐。Java里最常见的是**线程池**和各类**连接池**；此外有**对象池**与**缓冲/内存池**以及JVM层面的**常量池**。
>  线程池用`ThreadPoolExecutor`并配**有界队列**和**拒绝策略**；DB用HikariCP；Redis用JedisPool或Lettuce（多路复用不必池）；HTTP用OkHttp/HttpClient连接池；Netty用`PooledByteBufAllocator`。常量池则由编译器/JVM维护，`String#intern()`可做驻留但需谨慎。是否池化取决于**创建成本**、**资源稀缺**与**复用安全性**。

# 🎯 案例背景

电商秒杀场景：

- 系统需要同时处理 **1000 个用户请求**，每个请求都要写订单日志（耗时任务）。
- 我们对比两种方式：**不使用线程池** vs **使用线程池**。

------

## 1）❌ 不使用线程池（new Thread）

```java
public class NoThreadPoolDemo {
    public static void main(String[] args) {
        for (int i = 0; i < 1000; i++) {
            new Thread(() -> {
                processOrder();
            }).start();
        }
    }

    private static void processOrder() {
        try {
            // 模拟写数据库/日志，耗时100ms
            Thread.sleep(100);
        } catch (InterruptedException e) {
            e.printStackTrace();
        }
    }
}
```

### 📌 特点

- **每个请求都 new 一个线程** → 1000 个线程同时创建。
- 线程频繁创建和销毁，**内存抖动**，上下文切换消耗大。
- 极端情况：线程数过多，导致 **CPU 调度开销 > 业务逻辑**，甚至 OOM。

👉 在面试中，面试官会追问：如果并发 1w 个请求呢？显然撑不住。

------

## 2）✅ 使用线程池（ThreadPoolExecutor）

```java
import java.util.concurrent.*;

public class ThreadPoolDemo {
    public static void main(String[] args) {
        ExecutorService pool = new ThreadPoolExecutor(
                10,                     // 核心线程数
                20,                     // 最大线程数
                60, TimeUnit.SECONDS,   // 空闲线程存活时间
                new ArrayBlockingQueue<>(100), // 任务队列（有界，防止OOM）
                Executors.defaultThreadFactory(),
                new ThreadPoolExecutor.CallerRunsPolicy() // 拒绝策略
        );

        for (int i = 0; i < 1000; i++) {
            pool.execute(() -> {
                processOrder();
            });
        }

        pool.shutdown();
    }

    private static void processOrder() {
        try {
            Thread.sleep(100);
        } catch (InterruptedException e) {
            e.printStackTrace();
        }
    }
}
```

### 📌 特点

- 系统只创建 **固定数量线程（≤20）**，线程可复用。
- 超过 20 的任务会进入队列排队。
- 拒绝策略保证系统不会崩溃（例如 CallerRunsPolicy → 任务在调用方线程执行）。
- 内存和 CPU **稳定可控**，避免了线程爆炸。

# ⚖️ 对比总结

| 对比点     | 无线程池（new Thread） | 有线程池（ThreadPoolExecutor） |
| ---------- | ---------------------- | ------------------------------ |
| 线程创建   | 每次请求都 new         | 线程复用，最多 20 个           |
| 内存消耗   | 极高，容易 OOM         | 可控，受队列和参数限制         |
| 上下文切换 | 线程数过多，频繁切换   | 线程数有限，切换成本小         |
| 稳定性     | 高并发下不稳定         | 高并发下稳定                   |
| 管理性     | 无法监控、调优         | 可配置队列大小、超时、拒绝策略 |

------

# 🎤 面试标准回答（案例化）

> 在秒杀下单场景，如果不使用线程池，每个请求都会 new 一个线程，高并发下会产生大量线程，造成上下文切换频繁、内存飙升甚至 OOM。
>  使用线程池后，线程数量可控，任务复用线程执行，并且可以通过队列和拒绝策略进行流量削峰和限流，系统稳定性更强。
>  所以线程池的本质价值就是**复用线程、控制并发规模、避免系统崩溃**。
