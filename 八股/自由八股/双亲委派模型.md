------

## 1. 术语解释

**双亲委派模型**是 **Java 类加载器（ClassLoader）** 的一种工作机制。
 核心思想：
 👉 当一个类加载器接收到类加载请求时，它不会自己马上去加载，而是**先把请求委派给父类加载器**，父加载器如果找不到，才由自己尝试加载。

这样设计的好处是：

- 保证了 **Java 核心类库的唯一性**（避免用户自己写个 `java.lang.String` 干扰系统）。
- 提高了 **类加载的安全性和稳定性**。

------

## 2. 原理剖析

Java 类加载过程分为 **加载、链接（验证、准备、解析）、初始化**。
 而 **加载阶段** 就涉及到类加载器的选择，双亲委派模型正是此时的机制。

类加载器分层次结构：

- **Bootstrap ClassLoader（启动类加载器）**
  - C++ 实现，不是 Java 类，加载 `rt.jar` 等核心库。
- **Extension ClassLoader（扩展类加载器）**
  - 加载 `jre/lib/ext` 下的扩展类库。
- **Application ClassLoader（应用类加载器）**
  - 默认加载 `classpath` 下的类。
- **自定义 ClassLoader**
  - 用户可以继承 `ClassLoader` 实现。

**执行流程**（伪代码）：

```java
classLoader.loadClass(String name) {
    // 先委派给父加载器
    if (parent != null) {
        return parent.loadClass(name);
    } else {
        return findBootstrapClassOrNull(name);
    }
    // 如果父类加载器无法完成，则自己尝试加载
    return findClass(name);
}
```

------

## 3. 使用方式

一般情况下我们不用手动操作，JVM 自动完成。
 如果要打破双亲委派（比如 SPI、热部署、Tomcat 等场景），需要**自定义类加载器**，覆盖 `findClass()` 方法。

------

## 4. 对比分析

| 机制             | 优点                             | 缺点                                 | 使用场景           |
| ---------------- | -------------------------------- | ------------------------------------ | ------------------ |
| **双亲委派模型** | 保证核心类库唯一性，安全         | 灵活性差，难以加载同名但不同版本的类 | 一般应用           |
| **破坏双亲委派** | 可以隔离不同模块的类，支持热加载 | 可能导致类冲突，内存泄漏             | Web 容器、插件系统 |

举例：

- **Tomcat**：每个 Web 应用一个类加载器，可以加载不同版本的 `servlet-api.jar`。
- **JDBC SPI**：通过 `Thread Context ClassLoader` 打破双亲委派。

------

## 5. 项目实践

在分布式项目或微服务场景下：

- 我们通常依赖第三方框架（Spring、Dubbo），它们底层都和类加载有关。
- 插件化系统（比如 IDE 插件、游戏引擎扩展）会大量用到自定义类加载器，避免不同插件之间冲突。

## 6. 标准回答（面试版）

面试官问：**什么是双亲委派模型？为什么要这样设计？**

你可以答：

> 双亲委派模型是 Java 类加载器的一种工作机制。当一个类加载器接收到类加载请求时，会优先把请求交给父加载器，逐层向上，最终由启动类加载器尝试加载。如果父加载器无法完成，才由子加载器自己尝试。这样保证了 Java 核心类库不会被用户自定义类替代，提升了安全性和稳定性。
>  例如 `java.lang.String` 类一定由启动类加载器加载，即使我们自己写了同名类也不会被错误加载。
>  当然，在一些框架中会破坏双亲委派，比如 JDBC 的 SPI、Tomcat 的类隔离机制，这样可以支持插件化和模块化的需求。
