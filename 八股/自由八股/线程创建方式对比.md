下面把 **三种创建/提交并发任务的方式** 放在一起对比：`Thread + Runnable`、`Callable + FutureTask`、**线程池（Executor/ThreadPoolExecutor）**。

# 概览对比表

| 维度          | Thread + Runnable          | Callable + FutureTask                     | 线程池（Executor/ThreadPoolExecutor）        |
| ------------- | -------------------------- | ----------------------------------------- | -------------------------------------------- |
| 返回值        | ❌ 无                       | ✅ 有（`V call()`）                        | ✅ 有（提交 `Callable` 返回 `Future<V>`）     |
| 异常传递      | ❌ 无法向调用方抛出受检异常 | ✅ `call()` 可抛异常，`future.get()` 感知  | ✅ 同左：`future.get()` 感知                  |
| 取消任务      | ❌                          | ✅ `future.cancel()`                       | ✅ `future.cancel()`                          |
| 任务提交方式  | `new Thread(r).start()`    | `new Thread(new FutureTask<>(c)).start()` | `executor.execute(r)` / `executor.submit(c)` |
| 线程复用      | ❌ 不复用                   | ❌ 不复用                                  | ✅ 复用线程，开销小                           |
| 并发数控制    | ❌ 难以控制                 | ❌ 难以控制                                | ✅ 由池参数精确控制                           |
| 定时/周期任务 | ❌                          | ❌                                         | ✅ `ScheduledExecutorService`                 |
| 适用场景      | 简单一次性异步、教学演示   | 需要**返回值/异常**的一次性异步           | 生产环境：高并发、批量任务、可控资源         |
| 典型问题      | 线程爆炸、难管理           | 仍然是“一个任务=一个线程”                 | 需正确配置参数与拒绝策略                     |

------

# 各自最小示例

**1) Thread + Runnable**

```java
Runnable task = () -> System.out.println("work");
new Thread(task, "t1").start();
```

**2) Callable + FutureTask**

```java
Callable<Integer> c = () -> 42;
FutureTask<Integer> ft = new FutureTask<>(c);
new Thread(ft).start();
Integer res = ft.get(); // 阻塞拿结果
```

**3) 线程池（推荐生产用 ThreadPoolExecutor）**

```java
ExecutorService pool = new ThreadPoolExecutor(
    2, 4, 60, TimeUnit.SECONDS,
    new LinkedBlockingQueue<>(100),
    Executors.defaultThreadFactory(),
    new ThreadPoolExecutor.AbortPolicy()
);
Future<Integer> f = pool.submit(() -> 42);
Integer res = f.get();
pool.shutdown();
```

------

# 什么时候用哪一个？

- **只为快速起一条线程做点小事** → `Thread + Runnable` 足够。
- **需要拿返回值 / 感知异常 / 可取消** → `Callable + FutureTask`。
- **实际项目（高并发/批量任务/可控资源/定时任务）** → **线程池**：
  - 普通并发：`ThreadPoolExecutor`
  - 定时/周期：`ScheduledThreadPool`（或 `ScheduledExecutorService`）
  - 串行执行：`newSingleThreadExecutor()`
  - 短平快、IO 密集：`newCachedThreadPool()`（注意风险）

------

# 线程池配置要点（面试常问）

- **核心参数**：`corePoolSize`、`maximumPoolSize`、`keepAliveTime`、`workQueue`、`threadFactory`、`handler(拒绝策略)`
- **拒绝策略**：`AbortPolicy`（默认异常）/ `CallerRunsPolicy` / `DiscardPolicy` / `DiscardOldestPolicy`
- **为何不建议直接用 `Executors`**：
  - `newFixedThreadPool` + 无界队列 → 任务堆积风险
  - `newCachedThreadPool` → 最大线程数 `Integer.MAX_VALUE`，可能 OOM
     ✅ 建议用 `ThreadPoolExecutor` 明确队列与上限

------

# 常见坑 & 提示

- `start()` vs `run()`：调用 `run()` 只是普通方法，不会起新线程。
- `Future.get()` 是**阻塞**的；若需要非阻塞回调，可用 `CompletableFuture`。
- 线程池要记得 `shutdown()` 或 `shutdownNow()`，避免进程无法退出。
- CPU 密集 vs IO 密集：
  - CPU 密集：线程数≈CPU核数或略多
  - IO 密集：线程数可大于核数（考虑 IO 等待时间）
- 结合 **异常处理**：线程池中的异常不会冒泡到主线程，需通过 `Future.get()`/自定义 `ThreadFactory` 的 `UncaughtExceptionHandler` 处理。

------

## 结论（一句话背诵）

- **Runnable 简单但无结果**，
- **Callable + FutureTask 有结果可取消**，
- **线程池可复用线程、控并发、拿结果、可定时，是生产首选**。
