这个问题非常关键 👍，不论在数据库还是多线程编程里，**数据不一致**都是常见风险。我们分成两部分来讲：

------

# 🔹 一、导致数据不一致的原因

## 1. 并发读写问题（多线程 / 多事务）

- **线程本地缓存**：线程对共享变量的修改未及时写回主内存，其他线程看到旧值 → **可见性问题**。
- **非原子操作**：例如 `i++`，实质是三步（读 → 加一 → 写），多个线程同时执行会丢失更新。
- **指令重排序**：编译器/CPU 优化导致操作顺序和期望不一致。

## 2. 数据库并发问题

多个事务同时操作相同数据时会出现：

- **脏读（Dirty Read）**：读到别的事务未提交的数据。
- **不可重复读（Non-repeatable Read）**：同一事务多次读到的结果不一致。
- **幻读（Phantom Read）**：同一事务中多次查询，数据条数不同（如 `count(*)` 结果不一样）。
- **丢失更新（Lost Update）**：两个事务都修改同一行，后提交的覆盖前一个。

## 3. 分布式场景

- **网络延迟/分区**：节点间数据同步不及时。
- **副本不一致**：主从复制滞后。
- **CAP 权衡**：网络分区下无法同时保证一致性和可用性。

------

# 🔹 二、保证数据一致性的方法

## 1. 在多线程/单机层面

- **volatile**：保证共享变量的可见性。
- **synchronized / Lock**：保证临界区的互斥和可见性，避免多个线程同时修改同一数据。
- **原子类（AtomicInteger, AtomicLong 等）**：通过 CAS 保证原子性。
- **ThreadLocal**：线程隔离，避免共享数据冲突。

## 2. 在数据库层面

- **事务 ACID**：原子性、一致性、隔离性、持久性。
- **事务隔离级别**：
  - Read Uncommitted → 避免不了脏读
  - Read Committed → 避免脏读
  - Repeatable Read → 避免不可重复读（MySQL 默认）
  - Serializable → 避免幻读，但效率低
- **锁机制**：行锁、表锁、MVCC（多版本并发控制）。
- **两阶段提交（2PC）、三阶段提交（3PC）**：分布式事务一致性。
- **补偿事务（TCC 模式）**：Try → Confirm → Cancel，保证最终一致性。

## 3. 在分布式系统层面

- **强一致性**：Paxos、Raft 等一致性协议。
- **最终一致性**：消息队列（Kafka、RocketMQ）+ 重试机制，保证数据最终同步。
- **幂等性设计**：即使请求多次执行，结果也一致（如通过唯一流水号避免重复扣款）。
- **CAP & BASE 理论**：
  - CAP：一致性、可用性、分区容错性三者不可兼得。
  - BASE：Basically Available, Soft state, Eventual consistency → 放弃强一致，追求最终一致性。

------

# 🔹 三、总结（面试标准回答）

- **导致数据不一致的原因**：并发读写、事务隔离不足、分布式延迟与网络分区等。
- **保证一致性的方法**：
  - 单机层面：volatile、synchronized、锁、原子类
  - 数据库层面：事务 ACID、隔离级别、MVCC、锁机制
  - 分布式层面：分布式事务（2PC/3PC）、TCC、消息队列、幂等性、共识协议（Raft/Paxos）、BASE 最终一致性
