## 🎯 单例模式核心概念

### 定义
单例模式是一种创建型设计模式，它保证一个类只有一个实例，并提供一个全局访问点来获取这个实例。

### 三个要素
1. **私有化构造函数**：防止外部通过 `new` 创建实例
2. **内部创建唯一实例**：在类内部创建并持有唯一实例
3. **提供公共访问方法**：提供静态方法返回唯一实例

### 应用场景
- 数据库连接池（避免频繁创建连接）
- 配置管理类（全局共享配置）
- 日志记录器（统一日志输出）
- 线程池（资源管理）
- Spring 中的 Bean（默认单例）

## 🔍 七种实现方式详解

### 方式1：饿汉式（静态常量）⭐

```java
private static final Singleton INSTANCE = new Singleton();
```

**关键步骤：**
- 私有化构造函数
- 使用 `static final` 创建唯一实例（类加载时初始化）
- 提供公共静态方法返回实例

**优点：**
- ✅ 实现简单
- ✅ 线程安全（JVM类加载机制保证）

**缺点：**
- ❌ 没有延迟加载，可能造成内存浪费
- ❌ 无法处理需要参数初始化的情况

---

### 方式2：饿汉式（静态代码块）

```java
static {
    INSTANCE = new Singleton();
}
```

**关键步骤：**
- 声明静态变量但不立即赋值
- 在静态代码块中初始化实例

**优点/缺点：** 与方式1相同，只是实现方式略有不同

---

### 方式3：懒汉式（线程不安全）❌

```java
public static Singleton getInstance() {
    if (INSTANCE == null) {
        INSTANCE = new Singleton();
    }
    return INSTANCE;
}
```

**关键问题：**
- 多线程环境下，两个线程可能同时通过 `INSTANCE == null` 检查
- 导致创建多个实例，违反单例原则

**适用场景：** 仅限单线程环境（不推荐）

---

### 方式4：懒汉式（同步方法）

```java
public static synchronized Singleton getInstance() {
    if (INSTANCE == null) {
        INSTANCE = new Singleton();
    }
    return INSTANCE;
}
```

**关键步骤：**
- 使用 `synchronized` 修饰整个方法

**优点：**
- ✅ 线程安全
- ✅ 延迟加载

**缺点：**
- ❌ 效率低，每次调用都要同步（实际上只需要第一次创建时同步）
- ❌ 不推荐使用

---

### 方式5：双重检查锁定（Double-Check Locking）⭐⭐

```java
private static volatile Singleton INSTANCE;

public static Singleton getInstance() {
    if (INSTANCE == null) {                    // 第一次检查
        synchronized (Singleton.class) {
            if (INSTANCE == null) {            // 第二次检查
                INSTANCE = new Singleton();
            }
        }
    }
    return INSTANCE;
}
```

**关键步骤解析：**

1. **第一次检查（不加锁）**
   - 如果实例已存在，直接返回
   - 避免不必要的同步，提高性能

2. **同步代码块**
   - 只同步创建实例的关键部分
   - 减少同步范围，提高效率

3. **第二次检查（加锁后）**
   - 防止多个线程同时通过第一次检查后，重复创建实例

4. **volatile 关键字（JDK 5+必须）**
   - **作用1：保证可见性** - 一个线程修改后，其他线程立即可见
   - **作用2：防止指令重排序** - 禁止 JVM 对 `new` 操作的指令重排序
   - **为什么需要？** 
     - `new` 操作分为三步：
       1. 分配内存空间
       2. 初始化对象
       3. 将引用指向内存地址
     - 如果不加 `volatile`，JVM 可能重排序为 1→3→2
     - 其他线程可能看到未完全初始化的对象（引用不为null，但对象未初始化）

**优点：**
- ✅ 线程安全
- ✅ 延迟加载
- ✅ 性能较好（只在第一次创建时同步）

**缺点：**
- ❌ 实现相对复杂
- ❌ 需要注意 JDK 版本（JDK 5+ 才完全安全）

---

### 方式6：静态内部类（推荐）⭐⭐⭐

```java
private static class StaticInnerClass {
    private static final Singleton INSTANCE = new Singleton();
}

public static Singleton getInstance() {
    return StaticInnerClass.INSTANCE;
}
```

**关键步骤解析：**

1. **静态内部类特性**
   - 外部类加载时，静态内部类不会立即加载
   - 只有访问静态内部类时才会加载

2. **类加载机制保证线程安全**
   - JVM 在类加载时初始化静态成员变量
   - JVM 保证类加载过程的线程安全性

3. **延迟加载实现**
   - 只有调用 `getInstance()` 时才加载内部类
   - 加载内部类时创建实例

**优点：**
- ✅ 线程安全（JVM保证）
- ✅ 延迟加载
- ✅ 实现简单，无需同步
- ✅ **推荐使用**

**缺点：**
- ❌ 可以通过反射破坏（可以通过反射创建新实例）

---

### 方式7：枚举（最推荐）⭐⭐⭐⭐

```java
enum Singleton {
    INSTANCE;
    
    public void doSomething() {
        // 业务方法
    }
}
```

**关键特性：**

1. **JVM 层面的保证**
   - 枚举实例在类加载时创建，且只创建一次
   - JVM 保证枚举实例的唯一性

2. **天然防止反射攻击**
   - `Constructor.newInstance()` 会检查是否为枚举类型
   - 如果是枚举，直接抛出异常

3. **天然防止反序列化创建新实例**
   - 枚举类有特殊的序列化机制
   - 反序列化时返回已存在的枚举实例

**优点：**
- ✅ 实现最简单
- ✅ 线程安全
- ✅ 防止反射攻击
- ✅ 防止反序列化破坏
- ✅ **最推荐使用**（Joshua Bloch 在《Effective Java》中推荐）

**缺点：**
- ❌ 不能延迟加载（枚举在类加载时初始化）

## 💡 面试高频问题

### Q1: 单例模式有几种实现方式？各有什么优缺点？

**回答要点：**
1. 列举7种方式（按上述表格）
2. 重点说明**双重检查锁定**、**静态内部类**、**枚举**三种
3. 对比线程安全、性能、实现复杂度

**参考答案：**
> 常见的单例实现方式有7种：饿汉式（静态常量、静态代码块）、懒汉式（线程不安全、同步方法）、双重检查锁定、静态内部类、枚举。
> 
> 推荐使用**枚举**或**静态内部类**：
> - **枚举**：实现最简单，天然防止反射和反序列化破坏
> - **静态内部类**：延迟加载，线程安全，实现简单
> - **双重检查锁定**：性能好，但需要注意 volatile 关键字和 JDK 版本

---

### Q2: 双重检查锁定为什么要用 volatile？

**回答要点：**
1. 防止指令重排序
2. 保证可见性
3. 解释 new 操作的三个步骤和可能的指令重排序问题

**参考答案：**
> `volatile` 有两个重要作用：
> 
> 1. **防止指令重排序**：
>    - `new Singleton()` 分为三步：①分配内存 ②初始化对象 ③引用指向内存地址
>    - 如果不加 `volatile`，JVM 可能重排序为 ①③②
>    - 其他线程可能看到未初始化的对象（引用不为 null，但对象未初始化）
> 
> 2. **保证可见性**：
>    - 一个线程修改 INSTANCE 后，其他线程立即可见最新值
> 
> JDK 5+ 的 JMM 增强了 volatile 的内存语义，确保双重检查锁定的正确性。

---

### Q3: 为什么枚举方式是最好的单例实现？

**回答要点：**
1. 实现简单
2. 线程安全
3. 防止反射攻击
4. 防止反序列化破坏
5. 引用《Effective Java》

**参考答案：**
> 枚举方式是最佳单例实现，原因包括：
> 
> 1. **实现简单**：代码最简洁
> 2. **线程安全**：JVM 保证枚举实例的唯一性
> 3. **防止反射攻击**：`Constructor.newInstance()` 会检查枚举类型并抛出异常
> 4. **防止反序列化破坏**：枚举有特殊的序列化机制，保证反序列化返回同一实例
> 
> Joshua Bloch 在《Effective Java》中明确推荐使用枚举实现单例。

---

### Q4: 静态内部类是如何实现延迟加载和线程安全的？

**回答要点：**
1. 静态内部类的加载时机
2. JVM 类加载机制保证线程安全
3. 延迟加载的实现原理

**参考答案：**
> 静态内部类实现延迟加载的原理：
> 
> 1. **延迟加载**：
>    - 外部类加载时，静态内部类不会立即加载
>    - 只有调用 `getInstance()` 访问内部类的静态成员时，才会加载内部类
>    - 加载内部类时才创建实例
> 
> 2. **线程安全**：
>    - JVM 在类加载时初始化静态成员变量
>    - JVM 保证类加载过程的线程安全性（每个类只加载一次）
>    - 因此静态内部类的实例创建是线程安全的

---

### Q5: 如何防止单例被反射破坏？

**回答要点：**
1. 枚举方式天然防止（最佳方案）
2. 其他方式需要在构造函数中判断

**参考答案：**
> 1. **枚举方式**（推荐）：
>    - 枚举天然防止反射破坏，`Constructor.newInstance()` 会检查枚举类型并抛出异常
> 
> 2. **其他方式**（不推荐，仅作了解）：
>    - 在构造函数中添加标志位判断
>    - 使用第一次调用时设置的标志，再次调用构造函数时抛出异常
>    - 但这种方式可以被反射绕过（反射可以修改标志位）

---

### Q6: 如何防止单例被反序列化破坏？

**回答要点：**
1. 枚举方式天然防止
2. 实现 `readResolve()` 方法

**参考答案：**
> 1. **枚举方式**（推荐）：
>    - 枚举有特殊的序列化机制，反序列化时返回已存在的枚举实例
> 
> 2. **实现 `readResolve()` 方法**：
>    ```java
>    private Object readResolve() {
>        return INSTANCE;
>    }
>    ```
>    - 反序列化时会调用此方法，返回已存在的实例
>    - 注意：新创建的对象会被 GC 回收

---

### Q7: 单例模式的适用场景有哪些？

**回答要点：**
1. 需要唯一实例的场景
2. 资源管理类
3. 配置管理类
4. 工具类

**参考答案：**
> 单例模式适用于以下场景：

## ⚠️ 注意事项

1. **JDK 版本**：双重检查锁定在 JDK 5+ 才完全安全（volatile 语义增强）
2. **反射攻击**：除了枚举，其他方式都可能被反射破坏
3. **序列化**：需要序列化的单例类，除了枚举，需要实现 `readResolve()` 方法
4. **多类加载器**：不同类加载器加载的同一个类，会产生不同的单例实例

## 📝 总结

- **生产环境推荐**：**枚举** > **静态内部类** > **双重检查锁定**
- **面试重点**：理解每种方式的原理，特别是 **volatile 的作用**和**静态内部类的延迟加载机制**
- **实践建议**：优先使用枚举，如果必须延迟加载，使用静态内部类
